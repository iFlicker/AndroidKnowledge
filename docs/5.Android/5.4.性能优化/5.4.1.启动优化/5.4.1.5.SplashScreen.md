# Android SplashScreen 详解与启动优化实践

在 Android 应用性能优化中，启动速度是衡量用户体验最直接的黄金指标。而在冷启动过程中，用户点击桌面图标到应用首帧内容完整呈现之间的这一段“时间空白”，是性能优化中兵家必争之地。

从 Android 12 (API 31) 开始，Google 引入了全新的系统代管 SplashScreen 机制，以规范化应用的启动视觉体验。与此同时，官方推出了 `androidx.core:core-splashscreen` 兼容库，使得 API 23 及以上的设备都能获得一致的冷启动体验。

本篇将从底层物理本源、系统窗口机制、XML 核心属性、兼容库源码实现、实战代码方案以及常见避坑误区等多个维度，对 Android SplashScreen 进行深入剖析，帮助开发者构建一套工业级、无闪烁、物理衔接完美的启动优化方案。

---

## 1. 启动期黑白屏现象的物理本源与 Starting Window 机制

### 1.1 冷启动的物理阶段与性能瓶颈

冷启动（Cold Start）是指应用从零开始启动，即在系统进程中没有该应用的任何运行信息（如 `ProcessRecord` 为空）。当用户在 Launcher 桌面点击应用图标那一刻起，系统需要经历一条漫长的物理流水线才能将界面呈现在用户眼前：

1. **Launcher 进程的响应**：用户触摸屏幕，Launcher 捕获触控事件，通过 Binder 向 `system_server` 中的 `ActivityTaskManagerService (ATMS)` 发送 `startActivity` 请求。
2. **系统进程 WMS 与 ATMS 的决策**：ATMS 评估目标应用进程是否存在。当判定为冷启动时，ATMS 通知 `WindowManagerService (WMS)` 为即将启动的 Activity 准备窗口令牌（Window Token）及相关的系统元数据。
3. **Zygote 进程 fork 应用进程**：ATMS 通过 Socket 向 `Zygote` 进程发出创建新进程的请求。`Zygote` 接收请求并 `fork` 出应用的目标进程。
4. **应用进程的初始化**：新 fork 出来的应用进程初始化 ART 虚拟机，创建 Binder 线程池，反射执行 `ActivityThread.main()`。随后，主线程的 `H`（一个 Handler）开始处理消息，实例化 `Application` 并调用其 `onCreate()`，在此阶段通常会伴随大量的第三方 SDK 初始化。
5. **Activity 生命周期的调度**：在 `Application` 初始化完成后，主线程继续收到启动 Activity 的指令。它实例化目标 Activity，依次触发 `onCreate()` -> `onStart()` -> `onResume()`。
6. **布局树的解析与测量**：在 `onCreate` 阶段调用 `setContentView` 会触发 `LayoutInflater` 对 XML 布局文件进行解析，并将其反射转换为内存中的 View 对象树。随后在 `onResume()` 之后，`WindowManagerImpl` 引导 `ViewRootImpl` 绑定 PhoneWindow 的根视图 `DecorView`。
7. **VSYNC 信号与 Choreographer 传导机制**：
   在根视图挂载后，应用不会立刻开始绘制，而是需要等待系统的 VSYNC（垂直同步）信号。
   * 当 VSYNC 脉冲发生时，系统底层的 `DisplayEventReceiver` 会捕获此信号，并通过 Native 层的 `FrameDisplayEventReceiver` 向上分发，触发 Java 层 `Choreographer` 的 `doFrame()` 流程。
   * `Choreographer` 在接收到消息后，依次分发并发起三大物理阶段：**输入事件处理（Input）**、**动画执行（Animation）** 和 **遍历绘制（Traversal）**。
8. **首帧绘制与上屏**：`ViewRootImpl` 通过调用 `performTraversals()` 发起测量（Measure）、布局（Layout）与绘制（Draw）的指令。主线程将绘制指令存入 `DisplayList`，随后唤醒 `RenderThread`（渲染线程）。`RenderThread` 与 GPU 协同，将渲染后的图像 Buffer 提交给系统的合成器 `SurfaceFlinger`。最终，`SurfaceFlinger` 将图像数据同步输出 to 物理显示屏幕上。

从用户点击图标，到 `SurfaceFlinger` 硬件合成首帧并上屏，这中间涉及跨进程通信（IPC）、进程 fork、虚拟机初始化、I/O 读取、主线程排队、GPU 渲染等多重物理操作。即使是在性能强劲的旗舰机上，这个过程也需要 300ms ~ 500ms，而在系统负载高或配置低下的设备上，这一时间会被拉长到数秒。

### 1.2 为什么会出现短暂的“黑屏/白屏”？

在前文所述的冷启动物理阶段中，应用进程的创建与首帧的绘制是无法瞬间完成的。这意味着，在用户点击图标后的前几百毫秒内，屏幕上没有任何来自应用进程的像素 Buffer 可以供给系统进行显示。

为了让用户在点击屏幕后能够立刻获得“即时反馈”（这也是人机工效学中的核心设计原则，要求系统对用户操作的物理反馈时间控制在 100ms 以内，以维持操作的流畅感），系统会在收到 `startActivity` 请求后，由 WMS 立即决策，在系统进程中为目标应用强行叠加一个临时的占位窗口——**`Starting Window`**（启动占位窗口）。

系统在绘制这个占位窗口时，只能依赖解析应用 Manifest 文件时读取到的静态元数据（即当前 Activity 在 Manifest 中配置的主题 Theme）。如果当前配置的主题使用的是系统默认背景，且 `android:windowBackground` 为纯白或纯黑，系统进程就会用该单色填充并绘制 Starting Window。

当应用进程在后台历经千辛万苦，终于完成了 Application 初始化、Activity 生命周期流转、首帧绘制并向 WMS 汇报“首帧 Buffer 已就绪”（即调用 `reportDraw`）后，WMS 会在下一个 VSYNC 信号到来时移除 Starting Window，展现应用真实绘制的 Window。

如果应用默认 Theme 的 `windowBackground` 为纯白，而应用的真实首帧界面（如带有深色顶栏的首页，或者整体呈暗黑色调的布局）与该纯白背景存在巨大的色彩、亮度或几何图形的视觉反差，用户在启动页切换到首帧的瞬间，其视网膜就会经历一次强烈的亮度骤变和布局跃动。这种不协调的瞬时跳变，在视觉上会被强烈地感知为“闪烁”、“黑白屏”或“冷启动卡顿”，极大地破坏了应用的品质感。

### 1.3 深度解密 WMS 的 Starting Window 机制

Starting Window（启动占位窗口）并非应用进程的产物，而是 WMS 的一项系统级物理设计。其核心目的在于：**实现冷启动期间应用进程创建与视觉反馈的完全异步解耦**。

```
[Launcher 点击事件]
       │
       ├─────────────────────────────────────────┐ (异步并行)
       ▼                                         ▼
 WMS (SystemServer)                       Zygote fork App 进程
       │                                         │
 读取 Activity Theme                            创建 JVM、初始化 ART
       │                                         │
 强制创建 TYPE_APPLICATION_STARTING 窗口         执行 Application.onCreate()
       │                                         │
 渲染 windowBackground (毫秒级响应)             执行 Activity.onCreate() -> 绘制首帧
       │                                         │
       ▼                                         ▼
 屏幕呈现 Starting Window                  App 进程上报首帧就绪 (reportDraw)
       │                                         │
       └──────────────────┬──────────────────────┘
                          ▼
                  WMS 执行窗口替换
                          │
                  移除 Starting Window
                          │
                  显示 App 真实 Window
```

#### 1.3.1 窗口类型的物理特殊性

WMS 为启动占位窗口分配了特殊的窗口类型：`WindowManager.LayoutParams.TYPE_APPLICATION_STARTING`。
这个窗口与普通的 Activity 窗口（`TYPE_BASE_APPLICATION`）有着本质的不同：
* **不依赖应用主线程**：它的生命周期和渲染行为完全受控于系统进程 `system_server` 中的 `WindowManagerService`。即使应用进程的主线程因为发生了严重的卡顿（如在 `Application.onCreate()` 中执行了同步的 I/O 操作或数据库升级），Starting Window 依然可以稳定、流畅地显示在屏幕上，不受应用卡顿的影响。
* **独立的图层合成（Layer Composition）**：在系统的合成器 `SurfaceFlinger` 中，Starting Window 占用了独立的图层（Layer）。硬件合成器（HWC - Hardware Composer）会为其分配专属的合成管道。在应用首帧就绪前，`SurfaceFlinger` 只需要将系统的 Starting Window 图层与系统的状态栏、导航栏图层进行物理叠加合成，无需消耗任何与应用进程交互的 CPU/GPU 周期。

#### 1.3.2 Starting Window 的加载决策与生命周期

当 AMS 执行 `startActivityLocked` 时，会通过 `ActivityRecord` 检查该应用是否已存在对应的进程。如果不存在，则判定为冷启动，并触发 `ActivityRecord.showStartingWindow()`。

1. **StartingData 的构建**：WMS 内部的 `StartingData` 负责携带启动页的展示参数。WMS 最终会通过 `WindowManagerPolicy.addStartingWindow()`，传入当前 Activity 的 Theme 资源 ID 以及系统配置。在底层，`PhoneWindowManager` 负责协调创建这个预览视图，并计算它的 Window Type 和 LayoutParams。
2. **Starting Window 的绘制**：在系统进程中，WMS 实例化一个特化的 Window 对象，并通过底层的 `WindowAnimator` 对该窗口进行管理。WMS 此时并不知道应用主界面的结构，它仅仅是拉取 Theme 中的 `windowBackground`、状态栏颜色等基础属性，并在系统内存中绘制一片纯色区域（或者一张静态图片）。
3. **窗口替换的物理衔接**：当应用进程完成了主线程的所有初始化，且 `ViewRootImpl` 完成了首帧的 `draw` 动作，它会通过 Binder 调用 WMS 的 `relayoutWindow` 接口。WMS 判定应用的第一个真实 Window 已经“画好”（`mReportDrawStatus` 为就绪），就会触发 `ActivityRecord.removeStartingWindow()`。此时，系统会以极其平滑的过渡方式，将 `TYPE_APPLICATION_STARTING` 窗口的 Alpha 值渐变至 0，同时将 `TYPE_BASE_APPLICATION` 窗口的 Alpha 值渐变至 1，完成图层上的物理替换。

### 1.4 历史优化“偏方”的致命缺陷

为了消除因 Theme 不一致导致的“闪烁”与“黑白屏”，早期的 Android 开发者为了追求视觉上的绝对统一，经常使用两种广为流传的“优化偏方”，但它们都存在严重的致命缺陷。

#### 1.4.1 缺陷分析一：禁用预览窗口 (`windowDisablePreview = true`)

```xml
<item name="android:windowDisablePreview">true</item>
```

* **物理原理**：此配置会显式地向 WMS 声明：“该 Activity 在启动时，不要为其创建任何类型的 Starting Window。”
* **致命副作用**：当用户在 Launcher 桌面点击应用图标后，系统收到了指令不创建 Starting Window。由于此时应用进程尚未创建，屏幕上将不会出现任何变化，依然停留在 Launcher 界面，整个系统呈现出一种“无响应”的状态。直到 1~2 秒后，应用进程初始化完毕，首帧界面才突然在屏幕上蹦出来。
* **人机工效学灾难**：这严重违背了交互设计的“即时反馈”原则。在无视觉反馈的情况下，用户会以为自己没有点中图标，从而去二次、三次点击图标，甚至怀疑手机系统卡死、触屏失灵。这种“假死”体验比短暂的黑白屏闪烁对用户的伤害更大。

#### 1.4.2 缺陷分析二：设置窗口透明 (`windowIsTranslucent = true`)

```xml
<item name="android:windowIsTranslucent">true</item>
```

* **物理原理**：将 Starting Window 的背景以及窗口属性强行设置为完全透明，以此在视觉上隐匿冷启动期间的黑白屏过渡。
* **致命体验副作用**：与禁用预览窗口类似，当点击图标后，因为 Starting Window 是透明的，用户看到的依然是桌面，产生长达数秒的“假死无响应”体验。
* **致命生命周期与性能副作用**：
  在 Android 的 Activity 堆栈管理与窗口调度机制中，当系统启动一个新 Activity 时， AMS 会检查新 Activity 是否具有透明属性（`windowIsTranslucent` 或 `windowIsFloating`）。
  * 如果新 Activity 是**不透明**的，系统会判定旧 Activity（如桌面 Launcher）将被完全遮挡。此时，在首帧就绪后，系统会向旧 Activity 发送生命周期指令，触发其 `onPause()` 并最终执行 `onStop()`，从而将旧 Activity 标记为不可见。一旦进入 `onStop()`，旧 Activity 拥有的 View 树、加载的 Bitmap 内存、临时动画资源等都将**允许被系统回收**。
  * 如果新 Activity 被声明为**透明**的，AMS 则判定旧 Activity 依然是“可见的”（透过透明窗口可见）。因此，**下方的旧 Activity（如桌面 Launcher，或者上一个应用的 Activity）将永远不会执行 `onStop()` 生命周期，而是被迫一直停留在 `onPause()` 状态！**
  这就意味着，在冷启动这个对 CPU 和内存带宽极度敏感的阶段，系统不得不强行在主存中维持两个活跃状态的 Activity 内存。如果前一个 Activity 内存占用很大（国内各种复杂的 Launcher 经常占用数几百 MB 内存），在冷启动时极易触发底层的 LMKD（Low Memory Killer Daemon）强杀后台进程。不仅如此，两个 Activity 同时处于活跃状态，还会引发频繁的 GC 内存抖动，占用宝贵的主线程 CPU 周期，导致应用的冷启动总耗时大幅增加。

## 2. Android 12 (API 31) SplashScreen 革命与系统代管机制

面对 Android 碎片化的启动视觉体验，以及开发者滥用 `windowIsTranslucent` 和 `windowDisablePreview` 带来的系统级性能下降，Google 在 Android 12 (API 31) 中对 Starting Window 进行了彻底的革命，推出了系统强制代管的全新的 **SplashScreen** 机制。

### 2.1 核心设计思想

从 Android 12 开始，系统接管了所有的冷启动 Starting Window，并统一规范为带有应用 Icon、Icon 矢量动画、图标背景以及品牌图的标准 SplashScreen。
这一革命彻底规范了冷启动初期的体验，使之具有以下三大设计特征：
1. **强制执行**：开发者无法再通过 `windowDisablePreview` 强行绕过，任何冷启动都必须展示 SplashScreen，保证了触控响应。
2. **高度可定制的系统代管**：系统在 `system_server` 中，根据开发者提供的 XML 属性自动组装并渲染启动页，依然保持毫秒级响应的独立性。
3. **无缝的物理衔接能力**：提供了 `KeepOnScreenCondition`（挂起控制）与 `setOnExitAnimationListener`（退出衔接动画）两大核心 API，将启动页的生命周期与应用自身的预初始化、首帧物理上屏进行了深度融合，彻底消灭闪烁。

---

### 2.2 核心定制属性与 XML 配置

开发者可以通过配置 Activity 的 Theme 属性，来定制由系统托管的 SplashScreen 画面。

#### 2.2.1 属性详细参数表

| XML 属性 | 物理含义 | 物理规范与适配陷阱 |
| :--- | :--- | :--- |
| `android:windowSplashScreenBackground` | 启动页的背景颜色。 | 必须是单一颜色。如果不配置，系统会默认读取 `windowBackground`。若首帧也是纯色，建议两者配置完全一致以消灭闪烁。 |
| `android:windowSplashScreenAnimatedIcon` | 启动页中心的图标。 | 支持普通的静态图片（Drawable、SVG），但**强烈推荐**配置 `AnimatedVectorDrawable`。系统会在 Starting Window 上屏后自动播放此图标动画。 |
| `android:windowSplashScreenAnimationDuration` | 告知系统中心图标的动画时长。 | **非常核心的参数**。该属性**不会**自动压缩或拉伸矢量动画本身的实际播放时间。它仅仅是告诉 WMS 系统：“我的图标动画将播放 X 毫秒，请以此为参考来决定 Starting Window 的生命周期调度”。建议控制在 1000ms 以内。 |
| `android:windowSplashScreenIconBackgroundColor` | 中心图标的背景色。 | 用于在透明图标（如没有底色的 App Logo）下方垫一层圆形色块，以提高图标在不同背景色下的对比度和易读性。 |
| `android:windowSplashScreenBrandingImage` | 启动页底部的品牌图（Logo）。 | 可选项。展示企业品牌。但在折叠屏、平板以及小屏幕设备上存在拉伸适配问题。**Google 官方不建议过度使用此属性**。 |

#### 2.2.2 尺寸规范与几何裁剪规则（极其严格）

由于系统会将中心图标裁剪进一个圆形的遮罩中，因此设计图标时必须遵循以下物理尺寸：
* **包含图标背景（带有底色圆圈）**：
  图标的最大物理直径必须限制在 **108dp** 以内。在这其中，核心的图案（前景）必须局限在一个直径为 **80dp** 的同心圆（安全区域）中，超出此范围的内容在渲染时会被系统圆形遮罩强行裁剪掉。
* **不含图标背景（透明底色）**：
  图标的最大物理直径可以放宽到 **288dp** 空间，但是为了保证在任何设备上都不被圆形裁剪，核心的图案区域应始终保持在直径为 **160dp** 的安全圆形范围内。

#### 2.2.3 Adaptive Icon（自适应图标）的合成机制
在 API 26 之后，Android 引入了自适应图标（Adaptive Icon），它将应用图标拆分为 Foreground（前景）与 Background（背景）两层。
* 当我们在 `windowSplashScreenAnimatedIcon` 中指定了一个 Adaptive Icon 时，WMS 会读取其 XML 中定义的前景与背景层。
* 在系统进行图层渲染时，WMS 会自动将背景层平铺到由 `windowSplashScreenIconBackgroundColor` 配置的区域中，接着把前景层叠加在正上方，最后套上系统自带的圆形遮罩。
* **避坑警告**：如果你的图标前景图没有在其内部预留足够的安全边距（Padding），在圆形裁剪时边缘会被直接削去，导致 Logo 变形。因此，UI 设计师在输出启动页前景 Logo 时，必须严格保证图形的所有有效视觉信息均收拢在 80dp（含背景）或 160dp（不含背景）的同心圆安全红线以内。

#### 2.2.4 RenderThread 动画的流畅度保障
为什么冷启动时，系统自带的矢量动画（AVD）能够如此流畅，完全不卡顿？
这是因为 Android 12 SplashScreen 充分利用了 **RenderThread（渲染线程）** 的 **"RT Animations" 机制**。当 WMS 启动 Starting Window 时，系统进程会把 `AnimatedVectorDrawable` 的帧数据和变化轨迹通过 IPC 提交给 RenderThread。
RenderThread 在 Native 层直接调用 GPU 驱动进行硬件级别的插值计算和帧提交。由于这个阶段应用进程的主线程（UI Thread）依然在进行繁重的 `Application.onCreate` 初始化和反射加载，主线程即使发生 100% 的 CPU 占用（也就是主线程卡死），RenderThread 依然可以独立、稳定地向物理屏幕渲染 60fps 或 120fps 的矢量动画，从而给用户带来极其丝滑的响应体验。

### 2.3 KeepOnScreenCondition 延迟控制逻辑的艺术

在以往的 Starting Window 机制中，只要应用的首帧就绪并完成 `reportDraw`，WMS 就会无条件地撤销占位窗口。
然而，在实际的商业级应用中，应用首帧就绪时，首页往往由于核心数据（如网络配置、推荐列表、用户状态）还未加载完成，从而显示为一个空白的“骨架屏”或者光秃秃的空白壳子。用户在经历了精美的启动图后，却要面对一个空白的页面二次等待，体验依旧糟糕。

Android 12 推出了 `setKeepOnScreenCondition` API，赋予了应用**挂起启动页生命周期**的权力。

#### 2.3.1 挂起逻辑的底层原理

当我们在 Activity 中调用 `splashScreen.setKeepOnScreenCondition { viewModel.isDataLoaded }` 时，我们实际上是将一个 Lambda 闭包注册到了应用进程的 `ViewTreeObserver` 链条中。

在底层，兼容包和系统会将其封装进 `ViewTreeObserver.OnPreDrawListener`。在应用的 `ViewRootImpl.performTraversals()` 流程中，测量与布局执行完毕、准备触发绘制（Draw）的前一刻，系统会回调 `OnPreDrawListener.onPreDraw()`。

如果我们在 Lambda 闭包中判定数据尚未加载完成，闭包评估结果为 `true`。此时，`onPreDraw()` 会直接返回 `false`。
这会在物理上**强行挂起（Suspend）Activity 真实首帧的绘制**，阻止渲染线程向 SurfaceFlinger 提交第一帧 Buffer。因为 WMS 没有收到 `reportDraw` 信号，系统会认为应用首帧依然没有准备好，所以系统级的 `Starting Window` 会继续强行停留在屏幕上。

#### 2.3.2 挂起期间的 CPU 资源倾斜

这种“时间换空间”的挂起机制具有极高的物理优化价值：
在首帧绘制被挂起期间，应用主线程虽然在重复执行“VSYNC 唤醒 -> 评估 OnPreDraw -> 返回 false 挂起” the 轻量循环，但它**不会**触发复杂的 View 树绘制、Bitmap 的 GPU 上传以及 Native 层的 RenderNode 构建。
这使得主线程和 CPU 能够腾出宝贵的计算带宽，集中算力去执行后台的数据读取、热修复解压以及网络请求分发，从而间接地加快了后台预初始化的速度。

#### 2.3.3 超时与防 ANR 设计的必要性

但是，主线程的挂起是一把双刃剑。如果由于网络不可达、服务器宕机或热修复损坏，导致 `isDataLoaded` 永远不为 `true`，主线程将无限期地处于“拦截绘制 -> 重新发起 Traversals”的死循环中。
此时，如果用户对屏幕进行触摸或物理按键操作，这些输入事件会被发送到应用的 `InputStage` 管道。由于应用的首帧一直无法绘制并呈现，且主线程的 VSYNC 循环处于高频拦截状态，输入事件无法得到正确的消费和绘制反馈。当系统检测到输入事件在管道中堆积超过 **5 秒**（ANR 阈值），就会毫不留情地抛出 **ANR 崩溃**。

因此，**在任何工业级实践中，都必须对 `KeepOnScreenCondition` 的闭包加入强物理超时保护**（例如设置最大挂起时长为 2500ms），时间一到，无论后台初始化是否成功，都必须强制放行。

### 2.4 setOnExitAnimationListener 物理衔接动画的几何奥秘

在 Android 12 之前，启动页的消失是瞬间的、粗暴的，或者是通过简单的 Activity 之间渐变实现。
Android 12 的 `setOnExitAnimationListener` 能够让我们拿到系统的 `SplashScreenView` 实例。这个 View 在物理上其实已经成为了我们 Activity 的 DecorView 树中最顶层的一个子 View。

通过这个 View，我们可以设计极具“物理感”和“流动感”的过渡动效：
1. **背景分屏淡出**：我们可以对 `SplashScreenView` 本身执行 Alpha 渐变动画，使其从 1.0 平滑过渡到 0.0，使下方的 Activity 内容像晨雾消散般显现。
2. **Icon 的空间几何变换**：我们可以拿到 `SplashScreenViewProvider.iconView`。利用属性动画（`ObjectAnimator`），控制 Icon 的 `scaleX`、`scaleY` 逐渐缩小，同时计算首页真实布局中顶部 Logo 的物理坐标，让 Icon 执行一个向上的平移（`translationY`），实现物理上的“图标飞向目标位置”的无缝衔接。
3. **动画终点的物理释放**：
   在编写自定义退出动画时，**必须显式调用 `SplashScreenViewProvider.remove()`**。如果不调用，这个 SplashScreenView 即使 Alpha 已经变为了 0，它依然作为最顶层的 Child 驻留在 DecorView 树中。在 Android 的事件分发机制中，它依然会占用当前 Window 的 Touch Target 焦点，导致屏幕下方的所有真实组件（按钮、列表）都**无法接收到任何点击事件**。同时，它包含的 Icon 资源也无法被 GC 垃圾回收，造成严重的物理内存泄漏。

## 3. 官方兼容包 core-splashscreen 底层兼容实现与反射自愈

在低于 Android 12 的设备（API 23 ~ 30）上，为了能够用同一套属性和逻辑享受一致的启动体验，官方提供了 `androidx.core:core-splashscreen` 官方兼容库。其核心思路是：**在 Android 12+ 上直接透传给系统 API，而在 Android 12 以下设备上，利用应用层自身的 Theme 劫持与 DecorView 挂载，完美模拟出一致的行为。**

### 3.1 兼容包的核心架构设计

在兼容包中，`SplashScreen` 的基类会根据设备的 API 版本，在初始化时通过策略模式分发并创建不同的实现类 `Impl`：

```
       [SplashScreen.installSplashScreen(Activity)]
                          │
            ┌─────────────┴─────────────┐
            ▼ (API >= 31)               ▼ (API < 31)
        [Impl31]                    [Impl23]
            │                           │
  直接桥接系统 Native API      · 读取 postSplashScreenTheme
  WMS 系统进程代管             · 动态 activity.setTheme 恢复主题
                              · 创建自定义 SplashScreenViewProvider
                              · addView 劫持 DecorView 树
                              · 注入 OnPreDrawListener 挂起首帧
```

### 3.2 深度剖析 installSplashScreen(Activity) 源码

我们来详细解剖核心入口 `SplashScreen.installSplashScreen(Activity)` 的底层调用链。

```kotlin
// androidx.core:core-splashscreen 源码简化剖析
public class SplashScreen private constructor(activity: Activity) {

    private val impl: Impl = when {
        Build.VERSION.SDK_INT >= 31 -> Impl31(activity)
        Build.VERSION.SDK_INT >= 23 -> Impl23(activity)
        else -> Impl(activity) // 降级垫片
    }

    public companion object {
        @JvmStatic
        public fun Activity.installSplashScreen(): SplashScreen {
            val splashScreen = SplashScreen(this)
            // 核心！在 Activity.onCreate 之前执行核心劫持
            splashScreen.install()
            return splashScreen
        }
    }

    internal fun install() {
        impl.install()
    }
}
```

我们重点走读 API 23 ~ 30 的实现类 `Impl23`，它是如何在没有 WMS 原生支持的情况下模拟整套逻辑的：

#### 3.2.1 步骤一：劫持并恢复主题 (Theme Recovery)

在 `Impl23.install()` 中，首要的任务是读取并重写当前 Activity 的主题。

```kotlin
// Impl23.kt 源码核心逻辑走读
open class Impl23(val activity: Activity) : Impl() {
    
    protected var postThemeId: Int = 0

    override fun install() {
        val theme = activity.theme
        val typedValue = TypedValue()
        
        // 1. 从当前配置 of Theme 中寻找自定义属性 postSplashScreenTheme 的资源 ID
        if (theme.resolveAttribute(R.attr.postSplashScreenTheme, typedValue, true)) {
            postThemeId = typedValue.resourceId
        } else {
            Log.w(TAG, "postSplashScreenTheme not found, you must set it in your theme!")
        }
        
        // 2. 在 super.onCreate 之前，动态将 Activity 主题恢复为正常的业务主题
        if (postThemeId != 0) {
            activity.setTheme(postThemeId)
        }
    }
}
```

**物理深意**：如果兼容包不在此处将主题切换为 `postThemeId`，那么当 Activity 渲染真实的 View 树时，这些 View 就会继续沿用启动主题（`Theme.SplashScreen`）下的某些强力属性（如隐藏了 TitleBar，或者特殊的 WindowInsets 设置）。这会导致首页布局错位，甚至导致某些依赖特定 Material 主题属性的自定义 View 发生 Crash。

#### 3.2.2 步骤二：构建 SplashScreenViewProvider 并劫持 DecorView

当 Activity 完成了其 `setContentView()` 后，兼容库会通过 `ContentFrameLayout` 的加载时机或者首帧渲染周期，在应用进程中动态构建出一个模拟的 Splash 视图并进行挂载：

```kotlin
// Impl23.kt 中对 DecorView 挂载的底层逻辑
override fun adjustInsets(
    activity: Activity,
    splashScreenViewProvider: SplashScreenViewProvider
) {
    val decorView = activity.window.decorView as ViewGroup
    
    // 1. 动态创建一个 FrameLayout 并解析 windowSplashScreenBackground 与 Icon
    val splashScreenView = splashScreenViewProvider.view
    
    // 2. 强行将该 View 作为最顶层的 Child 塞入 DecorView 中，MatchParent 填满窗口
    val lp = FrameLayout.LayoutParams(
        ViewGroup.LayoutParams.MATCH_PARENT,
        ViewGroup.LayoutParams.MATCH_PARENT
    )
    decorView.addView(splashScreenView, lp)
}
```

#### 3.2.3 步骤三：利用 OnPreDrawListener 拦截 VSYNC 信号

挂起首帧的关键就在于 `OnPreDrawListener` 的注入：

```kotlin
override fun setKeepOnScreenCondition(condition: KeepOnScreenCondition) {
    val decorView = activity.window.decorView
    
    // 注册首帧前监听器
    decorView.viewTreeObserver.addOnPreDrawListener(object : ViewTreeObserver.OnPreDrawListener {
        override fun onPreDraw(): Boolean {
            // 实时回调开发者定义的挂起闭包条件
            if (condition.shouldKeepOnScreen()) {
                // 返回 false：强行中断本次 VSYNC 渲染流程！
                // 这会导致本次底层的 performDraw() 不被调用，GPU 不会渲染出真实的 Activity
                return false
            }
            
            // 条件释放，允许真实布局开始绘制并显示
            decorView.viewTreeObserver.removeOnPreDrawListener(this)
            
            // 触发自定义的退出过渡动画
            dispatchExitAnimation()
            return true
        }
    })
}
```

在 Android 系统的 `ViewRootImpl.performTraversals()` 中，绘制流程的基石代码如下：

```java
// ViewRootImpl.java 简化源码逻辑
private void performTraversals() {
    // ... 执行 measure ...
    // ... 执行 layout ...
    
    boolean cancelDraw = mAttachInfo.mTreeObserver.dispatchOnPreDraw();
    if (!cancelDraw) {
        // 如果 OnPreDrawListener 返回 false，则 cancelDraw 为 true
        // 此时会直接中断本次 performTraversals 流程，不调用 performDraw()
        // 并且会重新调用 scheduleTraversals() 发起下一次 VSYNC 绘制请求
        scheduleTraversals();
        return; 
    }
    
    performDraw(); // 真正将渲染 Buffer 送入 SurfaceFlinger
}
```

通过这种极为巧妙的物理拦截，在低版本上，应用虽然进程已经启动，Activity 已经就绪，但**下方的真实内容就是不进行最终的物理渲染上屏**。屏幕上显示的依然是覆盖在最顶层的 `SplashScreenView`，完美地模拟了 Android 12 原生系统的“拦截挂起”体验。

### 3.3 反射自愈与兼容细节踩坑

由于国内厂商对 Android 系统的深度定制，很多厂商 of ROM（如 MIUI、Flyme 等）修改了 `DecorView` 的默认层次结构，或者在 Activity 初始化时对 `Window` 进行了一些非标准的操作。这导致兼容包在低版本上运行时容易遇到以下致命问题：

1. **DecorView 结构被修改导致 View 被遮挡或失效**：
   部分厂商会将特定的系统状态栏占位 View 或悬浮 View 插在 DecorView 的最顶层。如果兼容库盲目地通过 `addView` 将 Splash 布局加入，可能会将系统的安全区视图遮挡，或者因为厂商重写了 `addView` 导致 ClassCastException。
   * **兼容库的自愈逻辑**：`androidx.core:core-splashscreen` 内部在操作 `DecorView` 时，大量采用了防御式编程，捕获所有可能的异常，并通过反射获取 `Window` 的内部私有字段，如果常规 `addView` 失败，会回退到通过 `PhoneWindow.getDecorView()` 获取原始对象，并通过底层的 `WindowManagerGlobal` 重新计算窗口层级。

2. **低版本上的沉浸式状态栏与导航栏抖动问题**：
   在 Android 12 以下，如果将一个铺满屏幕的 View 强行插入 DecorView，该 View 默认是不感知 `WindowInsets` 的。这会导致应用图标在渲染时，无法感知状态栏和导航栏的存在，从而**向下方偏移**，使得图标并没有处于屏幕真正的物理几何中心。当 Splash 退出时，真实的 Activity 布局又使用了沉浸式状态栏，这会导致用户看到图标产生一次**剧烈的上下抖动**。
   * **自愈机制**：兼容库内部实现了专用的 `onApplyWindowInsets` 劫持。它在 Splash 视图的测量阶段，主动将 `WindowInsets` 拦截并将其内部的值进行差值补偿，使 Icon 在定位时减去状态栏与导航栏的高度，确保其无论在何种沉浸式设置下，都处于屏幕绝对的几何中心。

3. **低版本模拟 Icon 动效的局限性与反射补偿**：
   在 Android 12 以下的设备上，系统本身是不支持通过 `windowSplashScreenAnimatedIcon` 自动播放 `AnimatedVectorDrawable` (AVD) 动画的。为了在低版本上实现动效对齐，兼容包在 `SplashScreenViewProvider` 内部的 `ViewImpl23` 类中，不得不利用反射强行提取 AVD 动画的 `VectorDrawableCompat` 实例，在布局被 addView 到 DecorView 后的第一个 VSYNC 周期，由兼容包的主线程代码**手动调用 `animation.start()`**。如果 AVD 编写使用了不支持低版本的底层属性（如特殊的 PathInterpolator），兼容包会自动捕获 ClassCastException 并自动降级为静态 Icon 渲染，保证应用绝对不发生 Crash。

4. **防范 ViewTreeObserver 泄漏引起的内存崩溃**：
   在老旧设备上，如果 Activity 频繁销毁重建（如屏幕旋转），向 `DecorView` 的 `ViewTreeObserver` 注册的 `OnPreDrawListener` 如果没有在合适的时间被移除，可能会导致 `DecorView` 持有外部 Activity 实例而产生严重的**内存泄漏**。兼容库在 `Impl23` 内部实现了一个生命周期感应垫片，在 `onDestroy()` 触发时，会自动调用 `ViewTreeObserver.removeOnPreDrawListener()`，确保在任何极端生命周期波动下均无内存泄漏风险。

### 4.2 核心 Kotlin 源码实现（含极详尽行级中文注释与高级三维过渡特效）

下面给出一段在企业生产中经过多重优化的核心 Activity 源码，在常规 Alpha + Scale 基础上引入了更具质感的 Camera 3D 三维翻转效果，同时展示了如何高精度计算物理目标位置进行“飞入”衔接。

```kotlin
package com.demo.splash

import android.animation.AnimatorListenerAdapter
import android.animation.AnimatorSet
import android.animation.ObjectAnimator
import android.animation.PropertyValuesHolder
import android.os.Bundle
import android.view.View

import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow

/**
 * 工业级超时防御状态包装器
 * 核心目的：防止后台网络请求或热修复解压由于异常挂起，导致 KeepOnScreenCondition 永久卡死触发 ANR
 */
class SafeSplashScreenLoader(private val maxWaitTimeMs: Long) {
    
    private val startTime = System.currentTimeMillis()

    /**
     * 实时检查是否可以放行启动页
     * @param isBusinessReady 业务层的数据加载是否已完成
     * @return true 代表需要继续保留启动页，false 代表释放拦截，允许首帧上屏
     */
    fun shouldKeepOnScreen(isBusinessReady: Boolean): Boolean {
        val elapsedTime = System.currentTimeMillis() - startTime
        // 如果物理耗时超过了预设的最大安全等待时间，强制放行以防止 ANR
        if (elapsedTime >= maxWaitTimeMs) {
            return false
        }
        // 如果还没超时，则完全服从业务层的数据加载状态
        return !isBusinessReady
    }
}

/**
 * 负责在冷启动时进行业务数据异步装载的 ViewModel

class SplashViewModel : ViewModel() {

        // 模拟后台异步网络请求、热修复包加载、或者本地核心配置数据库读写
        viewModelScope.launch {
            // 真实生产环境在此进行高优先级的初始化工作
            delay(1500) 
            _isDataLoaded.value = true // 业务数据装载完毕，通知 Activity
        }

    private val safeLoader = SafeSplashScreenLoader(maxWaitTimeMs = 2500)

    override fun onCreate(savedInstanceState: Bundle?) {

        // 【极重要】步骤 1：必须在 super.onCreate 之前调用 installSplashScreen()
        // 兼容库会在此处读取主题中的 postSplashScreenTheme，并在内部自动执行 activity.setTheme(postThemeId)
        // ────────────────────────────────────────────────────────────────────────
        val splashScreen = installSplashScreen()

        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // ────────────────────────────────────────────────────────────────────────
        // 步骤 2：使用 KeepOnScreenCondition 挂起 Activity 真实首帧的绘制上屏
        // 底层机制：在 onPreDrawListener 中返回 false，挂起渲染，直至安全释放
        // ────────────────────────────────────────────────────────────────────────
        splashScreen.setKeepOnScreenCondition {
            // 实时评估是否应该释放启动页
            safeLoader.shouldKeepOnScreen(isBusinessReady = viewModel.isDataLoaded.value)
        }

        // ────────────────────────────────────────────────────────────────────────
        // 步骤 3：配置 OnExitAnimationListener，编写高品质的“无缝物理衔接”过渡动画
        // ────────────────────────────────────────────────────────────────────────
        splashScreen.setOnExitAnimationListener { splashScreenViewProvider ->

            val splashScreenView = splashScreenViewProvider.view
            // iconView 代表了启动页正中央的应用 Logo 矢量图视图
            val iconView = splashScreenViewProvider.iconView

            // A. 创建中心图标 (Icon) 的物理空间几何过渡动画
            // 模拟高阶 3D 轴向空翻效果，同时向上位移并缩小
            iconView.cameraDistance = 8000f * iconView.resources.displayMetrics.density // 设置投影相机视距，防 3D 裁剪锯齿
            
            // 组合属性：位移 + 缩放 + 3D 沿 X 轴翻转
            val pvhY = PropertyValuesHolder.ofFloat(View.TRANSLATION_Y, 0f, -iconView.height.toFloat() * 1.5f)
            val pvhScaleX = PropertyValuesHolder.ofFloat(View.SCALE_X, 1f, 0.4f)
            val pvhScaleY = PropertyValuesHolder.ofFloat(View.SCALE_Y, 1f, 0.4f)
            val pvhRotationX = PropertyValuesHolder.ofFloat(View.ROTATION_X, 0f, 90f) // 向上旋转 90 度呈“掀起”效果
            
            val iconAnimator = ObjectAnimator.ofPropertyValuesHolder(
                iconView, pvhY, pvhScaleX, pvhScaleY, pvhRotationX
            ).apply {
                // 使用 AnticipateInterpolator 创造拉起回弹感
                interpolator = AnticipateInterpolator(1.4f)
            }

            // B. 启动页背景的扩散式渐变淡出动画
            // 随着背景变透明，下方的 Activity 真实主界面会自然而然地显露出来，物理衔接极佳
            val alphaOut = ObjectAnimator.ofFloat(
                splashScreenView,
                View.ALPHA,
                1f,
                0f
            )

            // C. 将所有物理动效组合进 AnimatorSet 并行触发
            AnimatorSet().apply {
                // 动画时长严格控制在 500ms 黄金范围内，防止拖泥带水，保证用户交互的紧凑性
                duration = 500
                playTogether(iconAnimator, alphaOut)
                
                // ────────────────────────────────────────────────────────────────────
                // 【最关键】步骤 4：必须在动画结束时显式调用 splashScreenViewProvider.remove()
                // 如果不调用，此透明 View 依然会以 MatchParent 的大小遮挡在 DecorView 顶端，
                // 导致屏幕下方的真实布局无法接收到任何 Touch 事件，并发生内存泄漏！
                // ────────────────────────────────────────────────────────────────────
                doOnEnd {
                    splashScreenViewProvider.remove()
                }
                
                start()
            }
        }
    }
}
```

### 4.3 双入口启动、冷启动与热启动状态判断

WMS 所代管的 SplashScreen，**仅仅会在应用冷启动（Cold Start）和温启动（Warm Start）时，才由系统进程强制创建并渲染。**
在热启动（Hot Start）状态下（如用户点击 Back 键退回桌面，或者通过 Home 键切出后，短时间内再次点击图标重新进入），应用进程依然处于活跃存活状态，且 Activity 栈中的实例依然在内存中挂载。此时，WMS 不会渲染 SplashScreen，而是直接将内存中已经绘制好的 Activity 窗口拉到最前显示。

#### 4.3.1 热启动下的业务闪屏适配与 LaunchCoordinator 架构设计

对于很多需要在启动时展示开屏广告（Splash Ads）或运营海报的应用，在冷启动时，可以通过系统代管的 SplashScreen 挂起，然后在 `setOnExitAnimationListener` 动画结束后，再向首帧无缝插入广告。
但是在**热启动**下，系统完全不会弹出 `SplashScreen`，这意味着应用直接进入了 Activity 页面。
为了实现热启动时依然展示广告且不与冷启动逻辑发生冲突，企业级开发中通常会设计一个启动控制中枢 `LaunchCoordinator`：

```
                              [LaunchCoordinator 统一分发]
                                            │
                             检测到冷启动? ─┴─ 检测到热启动?
                                   │                  │
                ┌──────────────────┴──┐               ▼
                ▼                     ▼           计算退台时长
             WMS 托管             业务预加载           超过 5 分钟?
         SplashScreen 挂起      数据就绪释放             │
                │                     │               ▼ (若是)
                └─────────┬───────────┘          直接动态挂载广告 Overlay
                          ▼                      (不依赖 SplashScreen 机制)
                  执行退出平滑动画
                          │
                      广告上屏
```

* **退台时间判定**：在 `Application` 中注册 `ActivityLifecycleCallbacks`，监听全局 Activity 的 `onStop` 与 `onStart`。当所有 Activity 均进入后台，记录当前时间戳 `backgroundTimestamp`。
* **分发逻辑**：当下一次 Activity 触发 `onStart` 时，对比当前时间戳。如果时间差超过设定阈值（如 300 秒），判定为需要进行热启动广告展示。此时，在主 Activity 中直接动态挂载一个全屏的广告视图（`Overlay View`），由应用层代码自行控制其生命周期，彻底避免与系统的 `SplashScreen` 产生生命周期重叠冲突。

### 4.4 性能度量指标（TTID 与 TTFD）在新版 SplashScreen 场景下的精准测量

在启动优化中，一切的优化结果都必须用客观的数据说话。Android 引入了两个用于精确刻画启动速度的黄金物理指标：

#### 4.4.1 TTID（Time to Initial Display，首次显示时间）

* **定义**：从用户点击图标，到屏幕上**首次出现任何有效视觉元素**（即系统进程渲染出 SplashScreen 纯色背景与应用图标）的时间差。
* **测量方法**：系统会自动在日志中打印其耗时。开发者可以在 Logcat 中过滤 `ActivityTaskManager` 标签，会输出如下日志：
  ```
  ActivityTaskManager: Displayed com.demo.splash/.MainActivity: +380ms
  ```
  使用新版 SplashScreen 方案后，由于系统在进程 fork 完成前就在 `system_server` 中直接把占位窗口渲染上屏，这个 TTID 的耗时通常在旗舰机上会缩短到 100ms 以内。这是最快速的用户心理物理反馈。

#### 4.4.2 TTFD（Time to Full Display，完全呈现并可交互时间）

* **定义**：从用户点击图标，到应用**核心业务数据全部装载完毕、真实界面已经完全绘制到屏幕上且用户可以进行滑动点击操作**的总时间差。
* **测量机制**：在过去，系统无法获知应用什么时候才算“真正准备好”。在新版 SplashScreen 中，由于我们使用 `KeepOnScreenCondition` 挂起了首帧，直到后台初始化完毕才释放并执行退出动画。
* **精准上报**：在 Activity 内部，当 ViewModel 数据就绪、`KeepOnScreenCondition` 判定释放，且首页 `RecyclerView` 列表完成了第一轮完美的渲染并显示后，我们应当在主线程安全地主动调用：
  ```kotlin
  activity.reportFullyDrawn()
  ```
  一旦调用该 API，系统会在 Logcat 中以如下标签上报精准的 TTFD：
  ```
  ActivityTaskManager: Fully drawn com.demo.splash/.MainActivity: +1850ms
  ```
  通过监控这个时间的涨落，才能客观判定我们在 `KeepOnScreenCondition` 挂起期间放入的初始化操作是否影响了应用的最终体验。

## 5. 高级适配场景与多设备挑战

在商业级的 Android 项目开发中，除了常规的手机冷启动外，我们还会经常面临分屏多任务、应用多开双开、以及单 Activity 多 Fragment 导航架构等复杂物理环境，这些环境对系统的 `SplashScreen` 提出了极高的适配挑战。

### 5.1 双开与分身应用（Dual Apps）下的 Multi-User 权限劫持

在全球各大厂商的定制系统中，多开应用在底层是通过 Android 的 `Multi-User`（多用户机制）实现的。每一个应用分身都运行在隔离的 `userId` 下（如 `UserHandle.USER_SYSTEM` 之外的子 User）。
* **物理挑战**：当分身应用冷启动时，`ATMS` 与 `WMS` 运行在系统的进程中，必须动态获取该分身所属用户的资源权限以加载其对应的 Theme 背景。
* **厂商崩溃踩坑**：在某些旧版定制系统上，由于多开用户没有读写主用户 APK Theme 资源的直接权限，WMS 在为分身应用创建 Starting Window 时，无法加载 Theme 中的 `windowSplashScreenAnimatedIcon`。这会导致系统进程抛出 `SecurityException` 或 `Resources.NotFoundException`，直接退化为“黑屏无响应”或导致应用冷启动瞬间直接闪退。
* **规避防范**：在兼容库的 `postSplashScreenTheme` 中，建议**避免使用只有主用户私有目录下才存在的外部动态资源**，尽量配置静态打包进 APK 内部的基础 Color 和 XML 资源，确保分身用户能被系统进程无障碍地解析与渲染。

### 5.2 分屏与自由窗口（Multi-Window）模式下的几何拉伸灾难

在折叠屏（Foldables）与平板（Tablets）设备上，应用经常处于分屏（Split-screen）或者自由窗口（Freeform window）的物理状态下被唤醒启动。
* **几何裁剪抖动**：当应用被限制在 1:2 或 1:3 的细长分屏窗口中冷启动时，系统依然会在该小窗口中绘制 Starting Window。如果配置的 Icon 没有良好的安全边距，或者没有设置自适应大小，WMS 会在合成图层时强制按照小窗的 Boundary 尺寸对 Splash 界面进行缩放。这会导致中心图标被压扁，或者在首帧渲染就绪、小窗展开的瞬间，由于 Insets 物理高度剧烈拉伸而产生一次极其突兀的闪烁。
* **优化策略**：在为折叠屏进行启动优化时，启动背景图绝对不能使用带有特定方向、非对称图形的背景。必须使用由 `windowSplashScreenBackground` 决定的绝对单色背景，中心图标则严格控制在 80dp 安全圆内，依靠单色背景的延展性完美自适应任何物理分屏 Boundary。

### 5.3 单 Activity 多 Fragment (Single Activity) 架构下的启动状态追踪

现代许多项目采用 Single Activity 架构。在此架构中，`MainActivity` 作为全局唯一的宿主壳，其 onCreate 仅仅触发一次。而真正的首页逻辑分散在如 `HomeFragment` 等组件中。
* **挂起时机迷茫**：在 Single Activity 下，如果仅在 `MainActivity` 级别注册 `setKeepOnScreenCondition` 并根据其主线程加载状态放行，那么当 `MainActivity` 判定数据就绪释放后，启动页移除了，但此时顶层的 `HomeFragment` 里的 `RecyclerView` 还没有进行任何数据绑定。用户依然会看到 200~300ms 的空白 Fragment 占位视图。
* **完美追踪方案**：我们需要构建一个全局的 `LaunchStateTracker`。在 `MainActivity` 中挂起：
  ```kotlin
  splashScreen.setKeepOnScreenCondition {
      // 必须同时满足全局业务就绪，且首页核心 Fragment 的 View 树已经至少完成了一次 layout 绘制
      !LaunchStateTracker.isHomeFragmentViewReady()
  }
  ```
  在 `HomeFragment.onViewCreated` 中，向 RecyclerView 的 LayoutManager 注册一个一次性的 `onGlobalLayoutListener`，在其触发 layout 回调时将 `isHomeFragmentViewReady` 置为 `true`。通过这种跨组件的联合状态追踪，才能实现 Single Activity 架构下完美的启动页“真数据物理衔接”。
* **深层 Fragment 路由拦截器的拦截等待**：在许多企业级 App 中，冷启动还需要根据用户的登录状态来决定是路由到登录页还是主页。这个路由分发的决策计算（如读取加密的 SP 或 KeyStore）通常有数十毫秒的主线程延迟。通过将路由框架的拦截决策结果（如 `isNavigationSettled`）也绑定至 `LaunchStateTracker` 中，可以确保启动页被拉下时，屏幕上展露的就是最终路由成功的目标 Fragment 页面，消灭了先展示主页然后瞬间跳向登录页的“闪现”视觉错乱。

## 6. 常见误区与方案权衡

### 6.1 误区一：未重置系统主题导致后续 Dialog/PopWindow 崩溃

* **现象描述**：应用使用了 `androidx.core:core-splashscreen`。在 Activity 启动后，后续弹出 `AlertDialog`、系统 `PopupMenu` 或者自定义弹窗时，应用突然发生 Crash，报错提示 `java.lang.IllegalArgumentException: You need to use a Theme.AppCompat theme (or descendant) with this activity.`，或者弹窗的背景显示为巨大的纯色。
* **物理本源**：开发者虽然在 Theme 中配置了 `postSplashScreenTheme`，但要么是配置错误（配置为了一个非 AppCompat/MaterialTheme 的裸主题），要么是在 `AndroidManifest.xml` 中将 Activity 的默认 Theme 设为了 `Theme.App.Starting`，却**遗漏**了在 Activity 初始化时调用 `installSplashScreen()`。
  如果没有调用 `installSplashScreen()`，兼容库就不会去执行 `activity.setTheme(postThemeId)`，导致 Activity 在其整个生命周期中，都将持续使用 `Theme.SplashScreen` 主题。该启动主题为了防止 Starting Window 被拉伸，隐藏了大量系统的标准组件样式属性，从而导致后续依赖这些样式的弹窗组件在反序列化反射解析时直接 Crash。
* **规避方案**：任何使用了 `Theme.App.Starting` 主题的 Activity，其 `onCreate` 之前**必须且只能**调用一次 `installSplashScreen()`，以保证主题能够安全自愈并被还原为正常的 `postSplashScreenTheme`。

---

### 6.2 误区二：在 Splash 过渡动效期间触发 Fragment 事务提交导致 Crash

* **现象描述**：在 `setOnExitAnimationListener` 的退出动画执行期间，后台的网络数据包返回。开发者顺理成章地在回调中通过 `supportFragmentManager` 执行 `FragmentTransaction.commit()` 来动态切换首页的 Tab，导致应用直接崩溃，抛出 `java.lang.IllegalStateException: Can not perform this action after onSaveInstanceState`。
* **物理本源**：
  在 Activity 启动时，如果有退出动画的执行，`setOnExitAnimationListener` 的触发时机是在应用的 Activity 已经执行完了 `onResume()`，且首帧已经就绪之后。
  如果在此期间，用户的设备发生了横竖屏旋转，或者因为弱网卡顿用户直接按下了 Home 键，Activity 已经在系统层执行了 `onSaveInstanceState()` 状态保存，并进入了后台。
  此时，退出动画的 Callback 还在主线程的消息队列（Handler Queue）中排队等待执行，当它一旦被回调并试图执行 `commit()` 时，就会直接触碰 Android 系统的状态完整性防线，引发 Crash。
* **规避方案**：在退出动画的回调内部，如果涉及到 Fragment 的事务提交或敏感的生命周期转换，一律使用 `commitAllowingStateLoss()` 进行提交，或者借助 `Lifecycle` 的状态评估，确保 Activity 处于 `STARTED` 状态以上时才进行页面的深度事务变更。

---

### 6.3 误区三：在 installSplashScreen 之前执行了过多重度初始化逻辑

* **现象描述**：冷启动时，黑白屏现象消失了，但点击图标后，依然会卡顿相当长的一段时间，且系统的 Starting Window Icon 动画播放出现了严重的掉帧。
* **物理本源**：
  开发者虽然引入了兼容包，但把大量的 SDK 初始化动作（如某些大厂的 APM 组件初始化、动态资源包加载等）同步堆放在了 `installSplashScreen()` 这一行代码之上。
  在 `installSplashScreen()` 执行之前，Activity 尚未将主题从 `Starting Theme` 恢复为 `postSplashScreenTheme`。由于这些重度初始化动作阻塞了主线程，阻碍了 Choreographer 向系统申请 VSYNC 信号，使得 `setTheme` 无法被及时执行。这不仅让应用失去了快速展示占位窗口的最佳物理时机，还会使得系统的 Starting Window 以为进程已经卡死，从而在播放矢量动画时发生严重掉帧，甚至触发系统级的 ANR。
* **规避方案**：**`installSplashScreen()` 必须是 Activity.onCreate() 中的第一行物理代码**。在其之上，除了部分极轻量的、非主线程不可且必须在 Theme 切换前执行的基础统计组件外，任何重度的业务逻辑都应该抛到 `installSplashScreen()` 之后、或者放入子线程中异步进行。

### 6.4 低端机型上的卡顿与掉帧调优实践

低端设备由于 CPU 核心主频较低且小核性能孱弱，在冷启动释放 SplashScreen 的那一瞬间，由于主线程正疯狂反序列化首页复杂的 XML View 树并触发大量的 `measure` / `layout`，容易与顶层的 Splash 退出动画产生**严重的抢占 CPU 带宽竞争**，进而导致退出动画卡死或严重掉帧（瞬间帧率跌至 10 帧以下）。
* **调优策略一：线程优先级（Thread Priority）抑制**：
  在退出动画播放的 500ms 黄金期间，人为降低后台初始化线程的优先级（如调用 `Process.setThreadPriority(Process.THREAD_PRIORITY_BACKGROUND)`），确保前台 RenderThread 和渲染主线程能垄断 80% 以上的 CPU 物理算力。动画结束后，再恢复正常优先级。
* **调优策略二：应用层异步预充气（AsyncLayoutInflater）**：
  将主 Activity 下方被遮挡的复杂布局，在启动页挂起（KeepOnScreen）期间，使用 `AsyncLayoutInflater` 在后台的 I/O 线程中提前进行 XML 充气（Inflate）反序列化。当主页放行时，直接将后台充气好的 View 挂载到容器中，彻底避免了退出动画播放时因主线程解析 XML 导致主线程卡顿的弊端。

### 6.5 方案权衡对比表：系统代管 Splash 方案 vs. 传统自定义 Activity 方案

在实际的 Android 工程架构中，特别是国内的巨型生态 App，针对启动页的设计往往存在巨大的商业利益博弈（如广告展现、合规协议首次强弹等）。下面给出两种主流方案在多个维度上的物理权衡对比，帮助架构师做出正确的方案决策：

| 物理度量指标 | 系统代管 + core-splashscreen 方案 (推荐) | 传统自定义 SplashActivity 方案 |
| :--- | :--- | :--- |
| **进程创建前响应速度** | **极快（100ms 以内）**。<br/>WMS 直接在系统进程完成 Starting Window 渲染，彻底解放应用进程。 | **慢（300ms - 800ms 以上）**。<br/>必须等 zygote 完成 fork、JVM 初始化、SplashActivity 生命周期完整分发后才能展示画面。 |
| **系统级 Binder IPC 消耗** | **仅 1 次**。<br/>直接启动核心 MainActivity，省去了一次 Activity 栈调度。 | **至少 2 次**。<br/>启动 SplashActivity 后，还需要再次通过 Binder 请求 AMS 启动 MainActivity，多出一次进程内的生命周期转换。 |
| **冷启动物理内存消耗** | **极低**。<br/>主 Activity 直接作为宿主承载，没有额外 Activity 的进出栈和 View 树创建销毁开销。 | **较高**。<br/>在冷启动内存敏感期，由于 SplashActivity 是透明的，会迫使 Launcher 一直保留在内存中，极易引起 GC 和 LMK 强杀。 |
| **视觉物理衔接度** | **完美**。<br/>可通过 `setOnExitAnimationListener` 操纵 `SplashScreenView` 进行多维度的空间缩放、平移，实现无缝物理流动。 | **较差**。<br/>由于跨越了两个 Activity，只能依赖系统的 Activity 过渡动画，无法实现图标元素物理层面的连贯流动。 |
| **隐私合规弹窗支持** | **支持，但略显复杂**。<br/>需在 MainActivity 启动时，通过 `KeepOnScreenCondition` 挂起首帧，以 DialogFragment 形式在 DecorView 顶层进行弹窗。 | **非常简单**。<br/>直接在 `SplashActivity` 的 Layout XML 中静态配置并进行逻辑阻断。 |
| **开屏广告 (Splash Ads)** | **支持，需业务侧适配**。<br/>在 `setOnExitAnimationListener` 结束时，不执行 remove，而是将启动页 View 作为底板，动态在顶层展示开屏广告。 | **原生支持**。<br/>直接在 `SplashActivity` 内部生命周期中接入各种广告平台的 SDK，展示逻辑极其简单粗暴。 |

#### 总结性方案建议：
1. **对于大部分中小型 App**：应该毫不动摇地选择 **系统代管 + core-splashscreen 方案**。这能以最低的工程维护成本，换取最流畅、最符合 Google 标准人机交互规范的无缝启动体验。
2. **对于有强商业化开屏广告或极度频繁隐私强弹的巨型 App**：如果必须要使用独立的 `SplashActivity`，也务必在 `SplashActivity` 中调用 `installSplashScreen()` 并将其主题与系统的 `Theme.SplashScreen` 保持一致，保证进程创建前依然由 WMS 提供即时无闪烁的 Starting Window 反馈。同时在广告加载完毕跳转主页时，采用共享元素（Shared Element）或自定义 Activity 过渡动画，尽可能减缓跨 Activity 切换带来的视觉撕裂。

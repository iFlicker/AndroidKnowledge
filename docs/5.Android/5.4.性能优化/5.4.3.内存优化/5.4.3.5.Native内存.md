# 5.4.3.5 Native内存

在 Android 性能治理的深水区，开发者面临的内存挑战已不再仅仅是 Java 虚拟机堆空间的溢出。随着 Android 系统架构的演进、跨平台框架（如 Flutter、React Native）的普及以及音视频、AI 边缘计算等重型业务的接入，由 C/S 架构底座、JNI 调用以及 C++ Native 堆分配引发的 **Native 内存泄漏与溢出**，已经成为 APM 卡死和稳定性卡死的核心症结。

本章将系统性解密 Android 历史版本中 Bitmap 物理像素宿主地址的演进与底层 `NativeAllocationRegistry` 的自动回收机制，深度剖析 JNI 全局引用与 C++ `malloc` 内存泄漏的微观本质，并探讨基于 `PLT/Inline Hook` 线上拦截 libc 以及官方 `heapprofd` 零侵入 Native 内存采样的 APM 核心技术。

---

## 一、 Bitmap 像素宿主物理演进与 NativeAllocationRegistry 机制

Bitmap 像素数据的存储位置是 Android 系统设计史上最著名的演进之一。理解这一物理变迁，对治理大图 OOM 和 Native 内存有着极其重要的指引意义。

### 1. Bitmap 像素数据物理宿主的三代变迁
```mermaid
grid
    android-2-x : Android 2.x - 3.0 之前
    android-3-0 : Android 3.0 (API 11) - 7.x 时代
    android-8-0 : Android 8.0 (API 26) 至今
```

*   **第一代：Android 2.x - 3.0 之前（Native 宿主时代）**：
    *   *物理布局*：Java 层的 `Bitmap` 实例极其轻量，仅持有一个指向 Native 像素数据数组（`SkPixelRef`）的 C++ 物理指针。像素矩阵直接分配在 C++ 堆空间中。
    *   *致命硬伤*：Native 内存的垃圾回收并不是由虚拟机 GC 主动触发的。当 Java 虚拟机堆空间还有大量余量时，系统即使检测到 Native 内存已彻底耗尽，也**不会主动触发 Java 垃圾回收器**来扫除无用的 Java Bitmap 对象。这导致系统高频发生因 Native 内存溢出导致的 Crash，要求开发者必须在代码中显式、手动地调用 `bitmap.recycle()` 进行 Native 内存的强制擦除，开发体验极差。
*   **第二代：Android 3.0 - 7.x 时代（Java 堆宿主时代）**：
    *   *物理布局*：Google 强行将 `SkPixelRef` 的像素字节数组分配到了 Java 虚拟机堆中的 **`byte[]` 数组**里。
    *   *好处*：图片的生命周期与普通 Java 对象 100% 融合，能无缝享受 GC 的自动标记与清理，彻底告别了手动 `recycle()`。
    *   *缺陷*：当时手机进程默认的 Java 堆最大上限极小（多为 192MB 左右）。多张大图同时解码瞬间就会把 Java 堆撑爆，极大增加了 Java OOM 的发生频率。
*   **第三代：Android 8.0 至今（Native 物理重归宿与 NativeAllocationRegistry 联动，最核心）**：
    *   *物理布局*：Google 重新将像素数据移出了 Java 堆，直接分配在 **Native 堆中（C++ 匿名共享内存 ashmem 或 C++ malloc 堆）**。
    *   *自愈飞跃*：为了杜绝第一代的“Native 内存无法随 Java GC 回收”的大坑，Google 引入了 **`NativeAllocationRegistry`** 桥接机制。

### 2. NativeAllocationRegistry 自动垃圾回收机制
`NativeAllocationRegistry` 解决了 Java 垃圾回收器与 Native C++ 物理指针释放的级联协调问题：
当 Java 层通过 Native 方法成功在 C++ 堆中创建了一个 `SkPixelRef`（像素存储结构）时，系统会紧接着调用 `NativeAllocationRegistry.registerNativeAllocation()`：

```kotlin
// Android SDK 内部伪代码逻辑
val registry = NativeAllocationRegistry(
    classLoader, 
    freeFunctionPointer, // 指向 C++ 底层释放内存函数（如 free 或 SkPixelRef::unref）的函数指针
    nativeByteSize       // 对应的 Native 字节大小，用于告诉 JVM 这一步消耗了多少 Native 空间
)
// 注册当前 Java 对象 (this) 与底层 Native 物理指针 (nativePtr) 的绑定
registry.registerNativeAllocation(this, nativePtr)
```

#### 底层物理机制：
1.  **动态感知**：虚拟机在进行 GC 运算时，会将注册传入的 `nativeByteSize` 累加到 Java 进程的总内存开销中，从而使得 **GC 决策算法能精确感知到 Native 空间的内存压力**，当 Native 空间吃紧时，能主动触发 Java 层的 GC 动作。
2.  **异步安全释放 (Cleaner 线程)**：
    当 JVM 进行垃圾标记，判定 Java 层的 `Bitmap` 对象已无任何 GC Root 可达、准备将其物理回收时：
    *   JVM 内部的垃圾回收器会将该 `Bitmap` 关联的 `Cleaner` 对象加入到 `ReferenceQueue` 中。
    *   专门的系统守护线程 `FinalizerDaemon` 或 `Cleaner` 会从队列中取出记录，直接根据注册时传入的 `freeFunctionPointer`（C++ 底层释放函数指针），通过 JNI 跨越用户态，反向调用 C++ 的 `free(nativePtr)` 或删除 `SkPixelRef` 实例。
    这一机制使得 **Native 内存完美融入了 Java GC 的自动回收网图中**，在不压迫 Java 堆上限的前提下实现了自动回收，是 Android 内存治理的重大技术革命。

## 二、 三大 Native 内存泄漏与溢出隐患

尽管有了自动清理机制，但在 Native 堆和 JNI 边界，依然存在三个因为开发不规范而引发物理泄漏的雷区。

### 1. JNI 全局引用（Global Reference）泄漏
在 Java 字节码与 Native C/C++ 通信时，如果需要在 C++ 对象的生命周期内长久持有一个 Java 对象，开发者需要通过 JNI 的 `NewGlobalRef` 来升级引用：
```cpp
// C++ 代码：创建一个全局引用以防 Java 对象被 GC
jobject globalActivityRef = env->NewGlobalRef(localActivityRef);
```
*   **物理危害与 OOM 原理**：
    JNI 的全局引用（Global Reference）在 JVM 的垃圾回收模型中，是**直接作为 GC Root** 存在的。
    只要这个 C++ 的 `globalActivityRef` 没有在不需要时手动调用 `DeleteGlobalRef` 进行擦除：
    ```cpp
    env->DeleteGlobalRef(globalActivityRef); // 物理擦除
    ```
    那么对应的 Java Activity 对象及其底下的整个 View 树将**永远无法被 JVM GC 标记和回收**，从而发生严重的 Java 内存泄漏；同时，Native 层也会因为未能释放底层结构体而发生 Native 内存的同步泄漏。
*   *治理手段*：通过线上 hook 系统的 JNI 方法，对 `NewGlobalRef` 与 `DeleteGlobalRef` 执行计数统计与匹配校验。

### 2. C++ Native 层的 malloc/new 物理未释放
这是最传统的 C++ 内存泄漏。在 NDK 动态链接库（.so）中，通过 `malloc()` 或 `new` 分配的内存，处于 Linux 进程的用户空间堆中，**虚拟机 GC 对其完全无感。**
```cpp
char* buffer = (char*) malloc(1024 * 1024 * 10); // 申请 10MB Native 空间
// 如果函数退出前没有执行 free(buffer)，这 10MB 空间将彻底从 Linux 进程可用空间中消失
```
当此类未释放的内存不断累积时，进程的物理内存占用（RSS）会直线冲顶，最终物理触发 Linux 系统的内核 LMK（Low Memory Killer）直接杀掉应用进程。

### 场景3：Native 线程泄露与虚拟内存空间耗尽（Thread Stack Leak）
在 Android 64 位或 32 位系统下，线程的创建和栈空间分配同样是 Native OOM 的高发地。
*   **物理本质**：
    当我们在 Java 层执行 `new Thread().start()`，或者在 C++ 层调用 `pthread_create()` 创建一个新线程时，底层都是通过 **`clone()`** 系统调用克隆一个轻量级进程（LWP）。
    Linux 系统为了保证该线程能够正常运行，会为其分配一个独立的**线程栈空间（Stack Space）**。在 Android 中，这个栈空间通常默认为 **1MB** 的虚拟内存物理页映射（Virtual Memory Area）。
*   *OOM 原理*：
    如果应用由于线程池调配失控，或者后台线程陷入无限等待而无法销毁，导致高频创建了 1000+ 个活跃线程：
    1.  仅仅线程栈虚拟空间就会吞噬近 **1GB** 的虚拟内存（VSS）。对于 32 位设备（虚拟寻址上限为 4GB，用户空间实际仅剩 3GB 左右）来说，会瞬间耗尽可用的虚拟内存地址空间。
    2.  再次调用 `pthread_create` 时，由于系统找不到连续 1MB 的空闲虚拟地址来分配栈空间，会导致底层的物理映射失败，抛出致命的 `java.lang.OutOfMemoryError: pthread_create failed (1048576B stack space)` 崩溃，直接终止进程。

## 三、 高阶线上 Native 内存监控与泄漏定位

针对 Native 内存泄露的黑盒状态，大厂 APM 监控通常采用 PLT Hook 拦截 `libc.so` 或者借助 Perfetto 进行底层采样检测。

### 1. 方案一：基于 PLT Hook 拦截 libc 的 Native 内存监控组件（APM 核心）
在 Linux 设备中，一切底层的 C++ 内存申请与释放动作，最终都会通过动态链接（Dynamic Linking）调用 `libc.so` 中的 `malloc`、`calloc`、`realloc` 和 `free`。
*   **监控原理**：
    利用 **PLT Hook（Procedure Linkage Table，基于 ELF 文件的过程链接表劫持）** 技术，动态拦截目标 `.so` 对 `libc.so` 的函数调用，将真实的 `malloc` 和 `free` 重定向到自研 APM 的监控代理函数中。
*   *引用计数匹配模型*：
    1.  当发生 `proxy_malloc(size)` 时，拦截并调用 `backtrace()` 提取当前 C++ 运行时的 Native 方法调用栈（获取 IP 指针并进行符号反解）；
    2.  以返回的物理内存指针（`pointer`）为 Key，将 `(size, stack_trace)` 作为一个 Entry 存入一个高并发安全的 `HashMap` 容器中；
    3.  当发生 `proxy_free(pointer)` 时，在拦截中拦截并执行 `HashMap.remove(pointer)`；
    4.  当应用运行一段时间（或用户执行特定场景后），**留在 HashMap 中始终未被 remove 的记录，就是发生 Native 内存泄露的真实嫌疑堆栈和行号。**

#### 跨层级劫持与栈回溯原理：
*   **DWARF 栈回溯（Unwind Stack）**：
    在 64 位 Android 系统上，直接使用系统的 `backtrace()` 可能会因为帧指针（Frame Pointer）被编译器优化抹去而失败。APM 组件通常会加载 `.eh_frame` 和 DWARF 格式数据，读取其中的调用现场上下文（CFA），实现零卡顿的高精度 Native 栈解析。

下面是基于 ELF 劫持 Native `malloc` 并过滤回溯的 C++ 底层设计伪代码：

```cpp
#include <jni.h>
#include <malloc.h>
#include <unordered_map>
#include <mutex>
#include <vector>
#include <unwind.h>
#include <dlfcn.h>
#include <android/log.h>

#define TAG "NativeAPM"
#define LOGI(...) __android_log_print(ANDROID_LOG_INFO, TAG, __VA_ARGS__)

// 保存未释放 Native 内存的全局哈希容器
struct AllocRecord {
    size_t size;
    std::vector<uintptr_t> backtrace;
};

std::unordered_map<uintptr_t, AllocRecord> g_alloc_map;
std::mutex g_map_mutex;

// 保存 libc.so 原始函数的函数指针
extern "C" {
    void* (*orig_malloc)(size_t) = nullptr;
    void (*orig_free)(void*) = nullptr;
}

// 栈回溯的回调函数
struct BacktraceState {
    void** current;
    void** end;
};

_Unwind_Reason_Code unwind_callback(struct _Unwind_Context* context, void* arg) {
    auto* state = static_cast<BacktraceState*>(arg);
    uintptr_t pc = _Unwind_GetIP(context);
    if (pc) {
        if (state->current == state->end) {
            return _URC_END_OF_STACK;
        } else {
            *state->current++ = reinterpret_cast<void*>(pc);
        }
    }
    return _URC_NO_REASON;
}

// 获取当前 Native 线程的方法调用栈
size_t capture_backtrace(void** buffer, size_t max_depth) {
    BacktraceState state = {buffer, buffer + max_depth};
    _Unwind_Backtrace(unwind_callback, &state);
    return state.current - buffer;
}

/**
 * 劫持拦截后的自定义 malloc
 */
void* proxy_malloc(size_t size) {
    // 1. 调用原始的 malloc 申请物理空间
    void* ptr = orig_malloc(size);
    if (ptr != nullptr) {
        void* buffer[30];
        // 2. 高精度捕获当前 C++ 栈帧
        size_t depth = capture_backtrace(buffer, 30);
        
        std::vector<uintptr_t> bt_pcs;
        for(size_t i=0; i<depth; ++i) {
            bt_pcs.push_back(reinterpret_cast<uintptr_t>(buffer[i]));
        }

        // 3. 将 (指针 -> 记录) 登记至全局 Map 中，使用互斥锁保护线程安全
        std::lock_guard<std::mutex> lock(g_map_mutex);
        g_alloc_map[reinterpret_cast<uintptr_t>(ptr)] = {size, bt_pcs};
    }
    return ptr;
}

/**
 * 劫持拦截后的自定义 free
 */
void proxy_free(void* ptr) {
    if (ptr != nullptr) {
        // 从未释放登记簿中剔除
        std::lock_guard<std::mutex> lock(g_map_mutex);
        g_alloc_map.erase(reinterpret_cast<uintptr_t>(ptr));
    }
    // 调用原始的 free 物理清除内存
    orig_free(ptr);
}

/**
 * 打印未释放嫌疑堆栈的报表
 */
void dump_native_leaks() {
    std::lock_guard<std::mutex> lock(g_map_mutex);
    LOGI("======================================");
    LOGI("检测到未释放的 Native 内存泄漏报表:");
    for (const auto& entry : g_alloc_map) {
        LOGI("泄露内存指针: 0x%lx, 泄露大小: %zu 字节", entry.first, entry.second.size);
        LOGI("回溯堆栈地址清单:");
        for (const auto& pc : entry.second.backtrace) {
            // 通过 dladdr 可将物理 PC 指针动态翻译为 SO 名称与符号行号
            Dl_info info;
            if (dladdr(reinterpret_cast<void*>(pc), &info) && info.dli_sname) {
                LOGI("  at %s (%s + 0x%lx)", info.dli_sname, info.dli_fname, pc - (uintptr_t)info.dli_fbase);
            } else {
                LOGI("  at 0x%lx (unknown)", pc);
            }
        }
    }
    LOGI("======================================");
}
```

### 2. 方案二：Android 官方 heapprofd 零侵入内存采样分析
在 Android 10 (API 29) 之后，系统引入了全新一代卡顿与内存诊断工具 **`Perfetto`**。针对 Native 内存，Perfetto 内置了 **`heapprofd`** 守护进程，这是目前业界最先进、低损耗的官方 Native 分析利器。

#### heapprofd 物理工作原理：
*   **Bionic libc 的内置钩子（Bionic Instrumentation）**：
    Android 系统的标准 C 库 `bionic libc` 原生支持内存选项调试。当开启 `heapprofd` 时，Perfetto 的 daemon 进程会通过修改目标应用进程的系统属性（配置 `libc.debug.malloc.options` 包含 `heapprofd`），或者向目标进程动态发送 `SIGRTMIN + 4` 中断信号。
*   **动态注入劫持**：
    目标进程接收到信号后，内部的 `bionic libc` 会在运行时自动加载 `/system/lib64/libheapprofd.so`。该 so 库会将底层的 `malloc`/`free` 分发重定向至 `heapprofd`。
*   **高性能进程间共享内存 IPC（Ring Buffer）**：
    为了规避传统的 dump 文件读写卡死（如 300MB 的内存堆栈遍历导致的严重停顿），`heapprofd` 通过 **虚拟共享内存（Shared Memory）与无锁环形队列（Lock-free Ring Buffer）** 进行跨进程通信：
    1.  当应用进程调用 `malloc` 分配内存时，`libheapprofd.so` 仅将“当前时间戳、分配大小、线程 TID 以及原始调用栈 IP 数组”打包。
    2.  利用无锁环形队列，将该极其轻量级的二进制数据块瞬间写入与 `heapprofd` daemon 共享的匿名内存页中。
    3.  外部的 `heapprofd` 独立守护进程以高频采样率从共享内存中读取数据并持久化为 Protobuf 数据包，在**主进程零阻塞、CPU 损耗小于 5%** 的极致性能下完成 Native 堆的完全分析。
*   *优势*：极度适用于排查生产环境下的中重度 NDK 代码内存泄露，是进行 NDK 库深度优化的统治级诊断工具。

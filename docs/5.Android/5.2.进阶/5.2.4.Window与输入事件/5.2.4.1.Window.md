# 5.2.4.1 Window

Android 的 Window（窗口）体系是整个系统 UI 呈现的核心幕后功臣。无论是 Activity 的界面展现、Dialog 的悬浮弹出，还是 Toast 的提示，乃至状态栏、锁屏界面的绘制，其底层都离不开 Window 体系的支撑。然而，Window 本身是一个十分抽象的逻辑概念，它并不直接参与 UI 的测量、布局和绘制工作。

本文将从核心概念、窗口管理、添加流程、生命周期绑定及典型问题等多个维度，对 Android Window 体系进行深度的机制剖析。

---

## 1. 核心概念（是什么）

### 1.1 Window 的定义与本质
在 Android 系统中，许多开发者容易混淆 `Window`、`View` 和 `Surface` 的概念。要理解 Window 的本质，我们需要理清这三者在架构设计上的职责分工：

*   **View（视图）**：是 UI 的基本构建单元。它承载了具体的测量（Measure）、布局（Layout）和绘制（Draw）逻辑。所有的 Button、TextView、LinearLayout 等，在客户端进程中都以 View 树的形式组织。View 并不直接与底层的显示系统交互，它是一个纯粹在应用进程中运作的绘制逻辑单元。
*   **Surface（画布）**：是 Android 渲染架构中的一块原始图像缓冲区。它对应着底层共享内存（Anonymous Shared Memory），专门用于存储像素数据。当应用需要绘制界面时，它会向 Surface 请求一块 Canvas，然后通过 Skia 或 OpenGL/Vulkan 硬件加速将像素写入缓冲。最后，系统进程的 `SurfaceFlinger` 服务会将这些 Surface 的缓冲数据进行合成，并最终输出到屏幕上。
*   **Window（窗口）**：是一个**逻辑上的虚拟窗口管理器或视口**。Window 本身既不是 View，也不是 Surface。它在面向对象设计中是一个抽象类（`android.view.Window`），定义了一套关于窗口样式、特征、背景、输入事件分发的标准接口。

**三者的协作关系可以总结为**：Window 作为逻辑管理者，持有一个 `DecorView`（即 View 树的根节点）。当 Window 需要被显示时，应用进程会向系统服务申请创建一个与之对应的 `Surface`。接着，通过逻辑通道，Window 会协调 `ViewRootImpl` 驱动 DecorView 树在 Surface 上进行绘制，并将绘制完成的像素数据提交给系统渲染管道。

```
+-------------------------------------------------------------+
|                     应用进程 (App Process)                  |
|                                                             |
|   +------------------+     持有     +------------------+    |
|   |  PhoneWindow     | -----------> |    DecorView     |    |
|   |  (Window 唯一实现) |             |   (View 树根节点)  |    |
|   +------------------+              +------------------+    |
|            |                                 |              |
|            v 桥接                            v 驱动绘制      |
|   +------------------+              +------------------+    |
|   |  ViewRootImpl    | -----------> |     Surface      |    |
|   |  (连接 WMS 的桥梁) |   请求Canvas | (图像缓冲区)      |    |
|   +------------------+              +------------------+    |
+------------|---------------------------------|--------------+
             | IPC (Binder)                    | 共享内存 / 渲染流
             v                                 v
+------------|---------------------------------|--------------+
|     系统进程 (System Server)                 | SurfaceFlinger|
|                                              |              |
|   +------------------+                       |  +--------+  |
|   |  WindowManager-  |                       |  | Screen |  |
|   |  Service (WMS)   |                       |  +--------+  |
|   +------------------+                       +--------------+
+-------------------------------------------------------------+
```

### 1.2 PhoneWindow：Window 接口的唯一具体实现
在 Android 框架中，`android.view.Window` 类是一个抽象类，其唯一的具体实现类是 `com.android.internal.policy.PhoneWindow`。

#### 1.2.1 PhoneWindow 的诞生与挂载
当 Activity 启动时，会在其生命周期的初始化阶段通过 `Activity.attach()` 方法创建对应的 Window 实例：

```java
// Activity.java 源码片段（简写）
final void attach(Context context, ActivityThread thread,
        Instrumentation instr, IBinder token, int ident,
        Application application, Intent intent, ActivityInfo info,
        CharSequence title, Activity parent, String id,
        NonConfigurationInstances lastNonConfigurationInstances,
        Configuration config, String referrer, IVoiceInteractor voiceInteractor,
        Window window, ActivityConfigCallback activityConfigCallback, IBinder assistToken) {
    
    // ...
    // 实例化 PhoneWindow
    mWindow = new PhoneWindow(this, window, activityConfigCallback);
    mWindow.setWindowControllerCallback(mWindowControllerCallback);
    mWindow.setCallback(this); // 将 Activity 设为 Window 的 Callback（用于分发按键、触摸事件等）
    mWindow.setOnWindowDismissedCallback(this);
    // ...
    // 为 Window 设置 WindowManager
    mWindow.setWindowManager(
            (WindowManager)context.getSystemService(Context.WINDOW_SERVICE),
            mToken, mComponent.flattenToString(),
            (info.flags & ActivityInfo.FLAG_HARDWARE_ACCELERATED) != 0);
    // ...
}
```

#### 1.2.2 PhoneWindow 的职责范围
PhoneWindow 承载了 Android 系统风格与交互的默认实现策略，其主要职责包括：
1.  **管理 DecorView 的创建**：当我们在 Activity 中调用 `setContentView()` 时，该调用最终会委托给 PhoneWindow，由其创建并初始化 DecorView 树。
2.  **分发输入事件**：触摸、键盘、轨迹球等硬件输入事件首先会到达 ViewRootImpl，然后传递给 PhoneWindow，PhoneWindow 再通过它设置的 Callback（即 Activity）将事件分发出去。
3.  **处理系统风格与特征**：包括是否启用 Action Bar、设置 Title、控制状态栏和导航栏的背景颜色、处理转场动画样式等。

### 1.3 DecorView：根视图的组织架构
`DecorView` 是 PhoneWindow 的一个内部类，它继承自 `FrameLayout`，作为整个 Window 视图树的根节点。

#### 1.3.1 DecorView 的层次结构
DecorView 本质上是一个 FrameLayout，但它内部的子视图会根据系统版本、主题样式（Theme）以及启用的 Feature（如 `FEATURE_NO_TITLE`、`FEATURE_ACTION_BAR`）动态装配。在最经典且常见的配置下，DecorView 的子视图结构通常如下：

```
DecorView (FrameLayout)
 └── LinearLayout (Vertical)
      ├── ViewStub (用于加载 ActionBar / TitleBar)
      └── FrameLayout (ID 为 android.R.id.content)
           └── 用户自定义布局 (即我们在 setContentView 中传入的 Layout)
```

#### 1.3.2 setContentView 的具体链路
当开发者调用 `Activity.setContentView(layoutResID)` 时，底层的调用流转如下：
1.  `Activity.setContentView()` 委托给 `PhoneWindow.setContentView()`。
2.  PhoneWindow 首先检查 `mContentParent`（即上面结构中的 `android.R.id.content`）是否为空。如果为空，则调用 `installDecor()` 来创建并初始化 DecorView。
3.  `installDecor()` 中会通过 `generateDecor(-1)` 生成 DecorView，再通过 `generateLayout(mDecor)` 加载对应的主题布局（如 `screen_simple.xml` 或 `screen_action_bar.xml`），并获取到 ID 为 `content` 的 `FrameLayout` 容器。
4.  最终，通过 `mLayoutInflater.inflate(layoutResID, mContentParent)`，将开发者定义的布局填充到 `mContentParent` 容器中。

### 1.4 窗口类型与层级（WindowManager.LayoutParams.type）
在 Android 的窗口体系中，WMS 管理着数个甚至数十个窗口。为了决定哪些窗口覆盖在哪些窗口之上，WMS 引入了**窗口类型（Type）**与**窗口层级（Z-Order）**的概念。

根据 `WindowManager.LayoutParams` 中的 `type` 字段，Android 窗口可以被划分为以下三大类：

#### 1.4.1 应用窗口（Application Window）
*   **Type 数值范围**：`1 ~ 99`。
*   **特点**：对应着一个具体的 Activity。在创建和添加时，必须提供一个由 Activity 对应在 WMS 端的 `AppWindowToken`，否则 WMS 校验会失败。
*   **典型值**：
    *   `TYPE_BASE_APPLICATION` (1)
    *   `TYPE_APPLICATION` (2)：普通的 Activity 窗口。
    *   `TYPE_APPLICATION_STARTING` (3)：应用启动时的 Starting Window（即闪屏预览窗口），用于在应用主线程尚未初始化完成前占位。

#### 1.4.2 子窗口（Sub Window）
*   **Type 数值范围**：`1000 ~ 1999`。
*   **特点**：不能独立存在，必须依附于一个已存在的父窗口（通常是应用窗口）。子窗口的 Z-Order 层级依赖于父窗口，生命周期也随父窗口消亡而消亡。
*   **典型值**：
    *   `TYPE_APPLICATION_PANEL` (1000)：最常见的子窗口，例如 `PopupWindow`。
    *   `TYPE_APPLICATION_MEDIA` (1001)：用于视频渲染的 `SurfaceView`。
    *   `TYPE_APPLICATION_SUB_PANEL` (1002)：依附于 Panel 窗口的二级 Panel。
    *   `TYPE_APPLICATION_ATTACHED_DIALOG` (1003)：依附于父窗口的 Dialog（普通的 `android.app.Dialog` 本质上不属于此类，下面会详细解析其机制）。

#### 1.4.3 系统窗口（System Window）
*   **Type 数值范围**：`2000 ~ 2999`。
*   **特点**：由系统直接创建或需要特殊权限才能创建的全局窗口。它们不依赖于 Activity Token，可以在屏幕的任意位置呈现，通常拥有极高的显示层级。
*   **典型值**：
    *   `TYPE_STATUS_BAR` (2000)：系统状态栏。
    *   `TYPE_SEARCH_BAR` (2001)：搜索栏。
    *   `TYPE_PHONE` (2002)：电话/来电通话界面（已被限制/废弃）。
    *   `TYPE_SYSTEM_ALERT` (2003)：系统警告窗口（已被限制/废弃）。
    *   `TYPE_KEYGUARD` (2004)：锁屏界面。
    *   `TYPE_TOAST` (2005)：Toast 提示窗口。
    *   `TYPE_APPLICATION_OVERLAY` (2038)：Android 8.0 引入的统一悬浮窗类型，替换了以前的 `TYPE_PHONE`、`TYPE_SYSTEM_ALERT` 等。

#### 1.4.2 Z-Order 层级决定机制
WMS 在确定窗口的绘制顺序时，会利用窗口类型进行初次定位。Z-Order 越高的窗口越优先显示。对于子窗口，它们会被放置在父窗口的上一层或下一层（例如 SurfaceView 会被默认放置在宿主 Window 的下方）。WMS 内部通过 `WindowAnimator` 和 `DisplayContent` 统一管理这些窗口层级的动态计算与合成。

#### 1.4.3 系统窗口权限校验机制演进
系统悬浮窗长期以来是 Android 生态中恶意软件进行钓鱼、强制弹广告的重灾区。因此，Google 对系统窗口权限（特别是悬浮窗权限）进行了多次收紧，详细适配变更可参考 [AndroidVersionChangeLog.md](../../../../../AndroidVersionChangeLog.md)：

1.  **Android 6.0 (API 23) 之前**：
    应用只需在 `AndroidManifest.xml` 中声明 `<uses-permission android:name="android.permission.SYSTEM_ALERT_WINDOW" />` 权限，即可任意使用 `TYPE_SYSTEM_ALERT` 或 `TYPE_PHONE` 创建悬浮窗口，甚至有一些应用利用 `TYPE_TOAST`（当时无需权限）在后台弹出全局弹窗。
2.  **Android 6.0 (API 23) 引入动态权限**：
    `SYSTEM_ALERT_WINDOW` 被归类为“特殊权限（Special Permission）”。应用不能像普通权限那样在运行时弹窗申请，而是必须通过 `Settings.canDrawOverlays(Context)` 检测，若无权限，需引导用户跳转到系统设置的“显示在其他应用上”页面手动授权。
3.  **Android 8.0 (API 26) 窗口类型重构**：
    为了规范乱象，Android 8.0 废弃了 `TYPE_PHONE`、`TYPE_SYSTEM_ALERT`、`TYPE_SYSTEM_OVERLAY`、`TYPE_PRIORITY_PHONE` 等老旧类型。
    *   **统一引入 `TYPE_APPLICATION_OVERLAY` (2038)** 窗口类型。
    *   如果应用 targetSdkVersion >= 26 并尝试 addView 老旧的系统窗口类型，WMS 会直接抛出 `BadTokenException`。
    *   所有非系统核心组件的第三方悬浮窗，都必须使用 `TYPE_APPLICATION_OVERLAY`，并且其层级低于系统关键状态栏与输入法，确保不会遮挡系统核心交互。

## 2. 窗口的添加与管理原理（怎么做 / 为什么）

### 2.1 WindowManager 家族的设计与职责分工
窗口的管理接口是 `WindowManager`，其继承关系如下：

```
                    ViewManager (Interface)
                 (定义: addView, updateViewLayout, removeView)
                              ^
                              |
                    WindowManager (Interface)
                 (定义: getDefaultDisplay, removeViewImmediate)
                              ^
                              |
                   WindowManagerImpl (Class)
                 (WindowManager 接口的客户端具体实现)
```

在具体的代码架构中，`WindowManagerImpl` 并没有实现真正的管理逻辑。它采用了**桥接/代理模式**，将所有的工作委托给了一个进程内的单例对象 `WindowManagerGlobal`：

*   **WindowManagerImpl**：与具体的 Context 绑定（例如 Activity 或 Application）。这使得它持有的 `WindowManager.LayoutParams` 能够继承 Context 的一些特征（比如 Activity 的 Token），它是多例的。
*   **WindowManagerGlobal**：进程内的唯一全局单例。它集中管理应用进程中所有的 Window 状态、对应的 ViewRootImpl、DecorView 树以及对应的 LayoutParams，它是无状态且与上下文无关的。

#### WindowManagerGlobal 的三大核心容器
WindowManagerGlobal 内部维护了三个 `ArrayList`，用于保持数据的一致性：
1.  `private final ArrayList<View> mViews = new ArrayList<View>();`
    存储所有被添加的窗口根视图（即 DecorView 实例）。
2.  `private final ArrayList<ViewRootImpl> mRoots = new ArrayList<ViewRootImpl>();`
    存储与每个窗口绑定的 `ViewRootImpl` 实例。
3.  `private final ArrayList<WindowManager.LayoutParams> mParams = new ArrayList<WindowManager.LayoutParams>();`
    存储每个窗口添加时对应的布局参数配置。

所有的操作（如 `addView`、`removeView`、`updateViewLayout`）都会在这三个集合上同步增删改，如果集合之间状态失联，将直接抛出系统级致命异常。

### 2.2 深度解剖：从 WindowManagerGlobal.addView() 到 WMS
当我们在 Activity 的 `handleResumeActivity()` 阶段或者通过 `windowManager.addView(decorView, layoutParams)` 往屏幕上添加窗口时，整个添加流程经历了客户端进程初始化与跨进程 Binder 通信的复杂链路。

#### 2.2.1 第一阶段：WindowManagerGlobal 内的组装与验证
当调用进入 `WindowManagerGlobal.addView()` 时：

```java
// WindowManagerGlobal.java 源码核心逻辑简化版
public void addView(View view, ViewGroup.LayoutParams params,
        Display display, Window parentWindow, int userId) {
    // 1. 参数与类型校验
    if (view == null) {
        throw new IllegalArgumentException("view must not be null");
    }
    if (!(params instanceof WindowManager.LayoutParams)) {
        throw new IllegalArgumentException("Params must be WindowManager.LayoutParams");
    }

    final WindowManager.LayoutParams wparams = (WindowManager.LayoutParams) params;
    
    // 2. 如果存在父窗口，根据父窗口 of LayoutParams 调整属性
    if (parentWindow != null) {
        parentWindow.adjustLayoutParamsForSubWindow(wparams);
    }

    ViewRootImpl root;
    View panelParentView = null;

    synchronized (mLock) {
        // ...
        // 防止重复添加同一个 View
        int index = findViewLocked(view, false);
        if (index >= 0) {
            throw new IllegalStateException("View " + view + " has already been added to the window manager.");
        }

        // 3. 实例化 ViewRootImpl ── 客户端 UI 管理的核心大脑
        root = new ViewRootImpl(view.getContext(), display);

        view.setLayoutParams(wparams);

        // 4. 同步更新三大核心容器
        mViews.add(view);
        mRoots.add(root);
        mParams.add(wparams);

        try {
            // 5. 核心跨进程挂载：通过 ViewRootImpl 将 View 绑定到窗口
            root.setView(view, wparams, panelParentView);
        } catch (RuntimeException e) {
            // 发生异常时，回滚三大容器的状态
            final int viewIndex = index >= 0 ? index : mViews.size() - 1;
            if (viewIndex >= 0) {
                removeViewLocked(viewIndex, true);
            }
            throw e;
        }
    }
}
```

#### 2.2.2 第二阶段：ViewRootImpl 的初始化与 W Binder 通道创建
在上面第 3 步中，`new ViewRootImpl` 内部创建了几个极为重要的通信成员：
*   **`mWindowSession`**：它是 `IWindowSession` 类型的 Binder 代理对象，代表了应用进程与 WMS 通信的会话通道。它是通过 `WindowManagerGlobal.getWindowSession()` 跨进程获取的系统服务单例代理。
*   **`mWindow` (即类 `W`)**：它是 `IWindow.Stub` 的实现类，是一个 Binder 本地对象（Stub）。当 WMS 需要反向通知应用进程（例如：窗口大小改变、焦点丢失、按键事件回调）时，会通过这个 Binder 通道调用到应用端。
*   **`Choreographer`**：监听系统的 VSYNC 信号，用于驱动界面的测量、布局和绘制周期。

#### 2.2.3 第三阶段：ViewRootImpl.setView() 内部的双轨并行
在 `root.setView(view, wparams, panelParentView)` 执行中，主要完成了两件大事：

##### 轨道一：触发本地首次 UI 渲染流程（同步）
在调用 `setView()` 之后，会立即执行 `requestLayout()`：
```java
// ViewRootImpl.java
@Override
public void requestLayout() {
    if (!mHandlingLayoutInLayoutRequest) {
        checkThread(); // 验证当前是否为主线程
        mLayoutRequested = true;
        scheduleTraversals(); // 申请 VSYNC 信号，启动 measure/layout/draw 流程
    }
}
```
这确保了在窗口正式被 WMS 接收并显示之前，本地的 View 树已经准备好了首次测量的尺寸与布局。

##### 轨道二：通过 IWindowSession 进行 IPC 通信将窗口推送到 WMS（异步/同步响应）
在 `setView` 内部，紧接着会通过 IPC 调用向 WMS 申请注册：
```java
// ViewRootImpl.java 源码片段
try {
    // ...
    // 调用 Session 的 Binder 接口将窗口信息发送给 WMS
    res = mWindowSession.addToDisplayAsUser(mWindow, mSeq, mLayout,
            getHostVisibility(), mDisplay.getDisplayId(), userId,
            mInsetsController.getRequestedVisibilities(), inputChannel,
            mTempInsets, mTempControls);
} catch (RemoteException e) {
    throw new RuntimeException("Adding window failed", e);
}
```
**关键参数解析**：
*   `mWindow`：将客户端的 `W` 传递过去，WMS 就能持有该窗口对应的客户端回调代理。
*   `mLayout`：即 `WindowManager.LayoutParams`，包含了窗口类型、尺寸、标志（Flags）等关键渲染参数。
*   `inputChannel`：用于接收输入事件的通道。在跨进程注册成功后，WMS 会对这个 `inputChannel` 进行初始化。

#### 2.2.4 第四阶段：WMS 侧的窗口收容与注册
当 IPC 调用到达系统服务进程的 `WindowManagerService.addWindow()` 时，WMS 将执行严格的控制策略：
1.  **权限与安全校验**：
    WMS 会检查该窗口的 `type`。如果是系统窗口（如悬浮窗），会检查当前调用进程是否声明并被授予了 `SYSTEM_ALERT_WINDOW` 或相关系统特权权限；如果是应用窗口，会验证 `token` 是否合法且与一个处于活动状态 of `ActivityRecord` 绑定。
2.  **创建 WindowState**：
    WMS 会在内部为该窗口创建并维护一个 `WindowState` 对象。`WindowState` 在系统服务端代表了一个具体的窗口，它聚合了该窗口的尺寸限制、Z-Order 层级、客户端 Binder 引用 `IWindow` 以及与之关联的宿主进程状态。
3.  **计算 Z-Order 排序**：
    WMS 决定该窗口在屏幕上的绝对叠加层级，并通知底层的显示系统。
4.  **构建 InputChannel 通道**：
    为了使新窗口能够接收物理屏幕的触摸和按键事件，WMS 会通过 `InputManagerService` 创建一对输入通道 `InputChannel`（内部基于 Unix Socket 实现）。其中一个 InputChannel 会注册到系统进程的 `InputDispatcher` 中，另一个则通过 Binder 调用传回给应用进程的 `ViewRootImpl`。
5.  **返回结果**：
    若添加成功，WMS 会返回 `ADD_OKAY` 状态码，并附带最新的系统配置参数（如屏幕方向、状态栏避让区域 Insets 等）。

```mermaid
    Activity->>PhoneWindow: 实例化 PhoneWindow

    deactivate VRI
```

## 3. 生命周期绑定与资源清理（适用边界与典型崩溃）

### 3.1 Activity, Window, View, Surface, Canvas, Layer 的全景关系
为了防止内存泄露和系统崩溃，我们必须理解这些渲染实体在物理和逻辑上的对应关系与生命周期：

| 实体 | 所在进程 | 生命周期绑定方 | 本质与核心职责 |
| :--- | :--- | :--- | :--- |
| **Activity** | 应用进程 | 系统的 ActivityStack / ActivityRecord | 控制组件的生命周期与业务交互逻辑。 |
| **Window (PhoneWindow)** | 应用进程 | Activity 的生命周期 | 作为逻辑容器，封装系统特征、DecorView 样式与输入事件分发代理。 |
| **DecorView** | 应用进程 | Window 的生命周期 | View 树的根节点。容器布局，承载用户真正填充的 UI。 |
| **ViewRootImpl** | 应用进程 | 窗口的添加/删除状态 (addView/removeView) | 界面渲染的控制器。驱动 View 树遍历，也是与 WMS 进行 Binder 交互的窗口代理。 |
| **Surface** | 应用进程 & 系统进程 | WMS 窗口可见性 / 缓冲区申请周期 | 承载像素数据的图像共享内存段，供应用进程绘制并供合成器读取。 |
| **Canvas** | 应用进程 | 仅存在于单次绘制周期中 | 绘图接口，向 Surface 写入绘制指令与渲染命令。 |
| **Layer (图层)** | SurfaceFlinger | 系统的 WindowState 生命周期 | Surface 在底层的实体形式。WMS 最终告诉 SurfaceFlinger 将哪些 Layer 进行硬件层级的合成展示。 |

### 3.2 典型崩溃一：`WindowManager$BadTokenException` 剖析
在日常开发中，围绕 Window 弹出最常见的崩溃莫过于 `BadTokenException`。导致这个崩溃的核心原因是**窗口添加时验证的 Token 无效或不匹配**。

#### 场景 1：使用 Application Context 弹出 Dialog
*   **崩溃代码**：
    ```java
    Dialog dialog = new Dialog(getApplicationContext()); // 传入了全局 Application 上下文
    dialog.setContentView(R.layout.dialog_layout);
    dialog.show(); // 崩溃抛出 BadTokenException
    ```
*   **崩溃根源分析**：
    普通的 Dialog 在 `show()` 时，其内部会创建一个 `PhoneWindow`，并通过 `WindowManagerImpl.addView()` 添加到屏幕。
    因为 Dialog 的默认窗口类型是 `TYPE_APPLICATION`（普通应用窗口），根据 WMS 的窗口添加规则，**所有应用窗口在 WMS 侧都必须与一个合法的 Activity Token（对应系统中的 ActivityRecord）相关联**。
    然而，`ApplicationContext` 内部并没有绑定到 WMS 端的 `AppWindowToken`，它的 Token 在 WMS 看来是一个空的或者是无效的。当 WMS 在 `addWindow` 进行安全校验时，发现该窗口类型是应用窗口，但缺乏合法的 `AppWindowToken` 凭证，于是当即拒绝，并抛出 `BadTokenException`。
*   **防范方案**：
    Dialog 的上下文必须使用 `Activity` 的 Context。若在特殊后台服务中确实需要弹出全局 Dialog，需申请悬浮窗权限，并将 Window 类型的 type 显式修改为 `TYPE_APPLICATION_OVERLAY`（系统窗口不要求 Activity Token 绑定）。

#### 场景 2：在 Activity 已经 Finish 后尝试弹出/更新 Dialog
*   **崩溃代码**：
    ```java
    // 异步网络请求回调
    runOnUiThread(() -> {
        if (!isFinishing()) {
            dialog.show(); // 如果在此之前 Activity 已经销毁，可能会发生崩溃
        }
    });
    ```
*   **崩溃根源分析**：
    当 Activity 调用了 `finish()` 并且系统已经开始销毁流程时，WMS 会将该 Activity 对应的 `AppWindowToken` 标记为失效（Disabled），甚至将其从 WMS 的 Token 管理列表中移除。
    如果在此时，异步任务的回调触发，尝试调用 `dialog.show()`。虽然我们使用的是 Activity 的 Context，但由于 Activity 在 WMS 侧对应的 Token 已经失效，WMS 在校验 Token 时会判定此 Token 已经过期。为了防止“僵尸窗口”悬浮于屏幕上，WMS 会拒绝添加并抛出 `BadTokenException`。
*   **防范方案**：
    在显示任何 Dialog 或更新窗口前，必须严格验证 Activity 的状态：
    ```java
    if (context instanceof Activity) {
        Activity activity = (Activity) context;
        if (!activity.isFinishing() && !activity.isDestroyed()) {
            dialog.show();
        }
    }
    ```

### 3.3 典型崩溃二：`WindowLeaked` (窗口泄漏)
窗口泄漏（Window Leaked）是 Android 开发中特有的一类内存泄漏形式，在 Logcat 中通常表现为 `WindowManagerGlobal: Activity ... has leaked window ... that was originally added here` 异常警告。

```
E/WindowManager: android.view.WindowLeaked: Activity com.example.MainActivity has leaked window DecorView@26c8b09[MainActivity] that was originally added here
    at android.view.ViewRootImpl.<init>(ViewRootImpl.java:623)
    at android.view.WindowManagerGlobal.addView(WindowManagerGlobal.java:389)
    ...
```

#### 3.3.1 崩溃发生的底层逻辑
1.  当我们展示一个 Dialog 或 PopupWindow 时，`WindowManagerGlobal` 内部的 `mViews`、`mRoots`、`mParams` 列表会添加并持有 Dialog 的 `DecorView` 与 `ViewRootImpl`。
2.  这些 View 的 Context 引用的是宿主 Activity。因此，`WindowManagerGlobal`（作为一个全局单例）就间接通过 Dialog 的 View 树对宿主 Activity 产生了**强引用链**。
3.  当 Activity 销毁（onDestroy）时，正常的流程要求开发者必须先手动让 Dialog 消失（调用 `dialog.dismiss()`），这会触发 `WindowManagerGlobal.removeView()`，将引用从列表清除。
4.  如果开发者忘记了 `dismiss()`，Activity 销毁了，但 `WindowManagerGlobal` 依然强引用着 Dialog 的 View，Dialog 的 View 又强引用着 Activity。这就导致整个 Activity 树以及底层的 Window 资源无法被 GC 回收，发生了严重的内存泄露。
5.  系统（`ActivityThread`）在检测到 Activity 退出后，会主动遍历并强行清理该 Activity 关联的未关闭窗口，同时在日志中抛出上述 `WindowLeaked` 异常，警示开发者。

#### 3.3.2 防范方案
在 Activity 的 `onDestroy()` 生命周期中，确保主动销毁所有附属于它的 Dialog 或 PopupWindow：
```java
@Override
protected void onDestroy() {
    super.onDestroy();
    if (mMyDialog != null && mMyDialog.isShowing()) {
        mMyDialog.dismiss(); // 主动断开引用，释放 WMS 中的 WindowState 与 ViewRootImpl
        mMyDialog = null;
    }
}
```

### 3.4 Toast 的窗口机制演进与典型 Bug 剖析
`Toast` 作为系统提供的一种轻量级气泡提示，其底层也是基于 Window 体系实现的。但与普通的 Dialog 不同，Toast 并不是直接由应用管理的，而是由系统进程的 `NotificationManagerService` (NMS) 统一排队和控制调度。

在不同的 Android 版本中，Toast 的底层窗口机制经历了数次重大重构，导致了不同版本的适配 Bug。

#### 3.4.1 Android 7.1 (API 25) 上的 `BadTokenException` 系统 Bug
在 Android 7.1 上，开发者们经常遇到非主动引起的 `Toast$BadTokenException` 崩溃，这是 Android 系统的设计缺陷。

##### 底层发生机制：
1.  应用调用 `Toast.show()`，该请求通过 IPC 传递给系统的 `NMS`。
2.  `NMS` 会把这个 Toast 加入展示队列，并在轮到该 Toast 时，为应用进程临时生成并授予一个特殊的 `Binder Token`，然后通过回调（`TN` 接口）把 Token 发送回应用进程的 `TN` 对象。
3.  应用进程的 `TN` 对象收到回调后，在主线程（UI 线程）通过 `Handler` 投递一个显示任务，调用 `WindowManagerImpl.addView()` 展示 Toast，并将从系统获得的这个临时 Token 作为 LayoutParams 的 token 传给 WMS。
4.  同时，WMS 侧有一个超时销毁机制。为了防止 Toast 异常驻留屏幕，系统会在指定时间（比如 2 秒或 3.5 秒）后使这个临时 Token 失效。
5.  **Bug 的触发点**：如果应用的主线程此时发生了卡顿（例如执行了复杂的数据库操作或复杂的 UI 绘制），导致 `TN` 在主线程派发的显示任务迟迟得不到执行。
6.  当卡顿结束，主线程终于开始执行 `TN` 的显示任务并 addView 时，系统端的超时计时早已结束，NMS 早已让那个临时 Token 失效。
7.  WMS 收到带有过期 Token 的 Toast addView请求时，判定为非法添加，抛出 `BadTokenException` 导致应用 Crash。

##### 解决方案（Android 8.0 之前的通用适配）：
由于这是系统底层的 Handler 消息调度逻辑，开发者通常采用**动态反射代理 TN 中的 Handler** 的方案：在调用 addView 时捕获 `BadTokenException` 异常，避免程序直接崩溃。

```java
// 核心伪代码：对 TN 的 mHandler 进行包装
try {
    Field tnField = Toast.class.getDeclaredField("mTN");
    tnField.setAccessible(true);
    Object tnInstance = tnField.get(toast);
    
    Field handlerField = tnInstance.getClass().getDeclaredField("mHandler");
    handlerField.setAccessible(true);
    Handler originalHandler = (Handler) handlerField.get(tnInstance);
    
    // 注入自定义的安全包装 Handler
    handlerField.set(tnInstance, new SafeHandler(originalHandler));
} catch (Exception e) {
    // 忽略异常
}

// SafeHandler 内部逻辑：
class SafeHandler extends Handler {
    private final Handler mDecHandler;
    SafeHandler(Handler handler) { mDecHandler = handler; }
    
    @Override
    public void handleMessage(Message msg) {
        try {
            mDecHandler.handleMessage(msg);
        } catch (WindowManager.BadTokenException e) {
            // 捕获系统在 API 25 的 Crash
        }
    }
}
```

#### 3.4.2 Android 8.0 (API 26) 及之后的修复
在 Android 8.0 之后，Google 在系统层层面对 Toast 窗口添加逻辑进行了彻底重构：
*   **由 WMS 侧分配统一的 Token**：不再由 NMS 在每次显示时动态管理临时 Token，而是为 Toast 分配一种特定的、拥有单独校验通道的系统级别 Token。
*   **客户端异常捕获**：在 `ViewRootImpl` 客户端的核心代码里，对特定的 `TYPE_TOAST` 窗口进行了更友好的异常捕获和规避，避免由于主线程排队卡顿引起前台崩溃。
*   同时，在较新版本中（Android 11+），系统对非前台应用弹出带有自定义视图的 Custom Toast 进行了严格限制，仅允许在应用处于前台时渲染自定义 View，后台仅能展示标准的系统文本 Toast，具体行为变化参见 [AndroidVersionChangeLog.md](../../../../../AndroidVersionChangeLog.md)。

## 4. 总结与最佳实践

掌握 Android Window 体系的管理与运作细节，是通往 Android 高级开发的必经之路。在日常工程实践中，我们需要牢记以下几条核心法则：
1.  **Context 的精准选择**：凡是涉及界面展示的逻辑，如 Dialog、PopupWindow、自定义 FloatView，必须确保传入带有 ActivityToken 的 Activity Context。除非是声明了全局 Overlay 权限的系统浮窗。
2.  **生命周期的对称性**：在 `Activity` 的销毁（`onDestroy`）甚至在 `Fragment` 的解挂（`onDestroyView`）中，确保所有动态 addView 或 show 的 Dialog 进行同步的 `removeView` 或 `dismiss`，避免发生 `WindowLeaked`。
3.  **遵循最新的权限策略**：针对全局悬浮窗，一律适配 `TYPE_APPLICATION_OVERLAY` 窗口类型，并且一定要做好 `Settings.canDrawOverlays(Context)` 的动态引导与降级体验。

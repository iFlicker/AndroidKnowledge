# 3.1.6 设计模式

## 定位

设计模式是对常见设计问题的命名和经验总结。它不是 Java 语法特性，也不是必须照抄的代码模板，而是在反复出现的约束下组织对象、职责、依赖关系和变化点的一组惯用方案。一个模式之所以成立，关键不在于类名是否叫 `Factory`、`Strategy` 或 `Observer`，而在于它是否解决了某类稳定问题：对象创建过程过于复杂、调用方依赖了太多具体实现、算法分支散落在多个位置、子系统协作流程难以复用、状态变化导致条件判断膨胀，或者横向能力反复侵入业务逻辑。

学习设计模式的重点不是背诵类图，而是理解每个模式背后的判断：它试图隔离什么变化，它引入了哪一层间接性，它把复杂度从哪里移动到了哪里，它让调用方少知道了什么，又让维护者多承担了什么成本。一个成熟的 Java 设计并不会把所有模式都摆在代码表面，很多时候它只保留模式的核心思想。例如简单的策略可以用函数式接口表达，固定集合的策略可以用枚举常量表达，单例可以用枚举或静态内部类表达，迭代器已经成为集合框架的一部分，模板方法可能隐藏在抽象基类中，装饰器可能表现为一组同接口包装类。

设计模式通常被分为创建型、结构型和行为型。创建型模式关注对象如何创建，目标是让调用方不被构造细节、具体类型选择和初始化顺序绑死。结构型模式关注类和对象如何组合，目标是让调用方看到稳定接口，同时让内部复用、适配、增强或简化复杂结构。行为型模式关注算法、流程和协作职责如何分配，目标是让可变行为从稳定主流程中分离出来。真实项目中这些分类只是理解入口，模式经常组合出现，也可能在一个类中只体现一部分思想。

在 Java 中讨论设计模式，还要结合语言本身提供的表达能力。接口和抽象类可以建立稳定契约，泛型可以保留类型信息，枚举可以限定有限集合并承载行为，lambda 和方法引用可以表达轻量回调，记录类适合表达不可变数据载体，标准库中的集合、流、并发工具和 I/O 装饰体系已经内置了很多模式思想。越熟悉这些语言能力，就越能用自然的 Java 写出恰当的模式，而不是为了符合某张类图制造额外类层级。

## 为什么需要设计模式

软件设计面对的核心问题不是“如何写出能运行的代码”，而是“当需求、规则、依赖和规模变化时，代码是否还能被理解、修改和验证”。没有任何模式可以消除变化，模式能做的是把变化限制在更小的范围内，让稳定部分和变化部分之间的依赖关系更清楚。创建型模式把“怎么创建”从“怎么使用”中拆出来；结构型模式把“不兼容的接口”“额外能力”“复杂子系统”包装成更合适的形状；行为型模式把“算法、动作、状态转换、通知、访问顺序”变成可替换或可组合的对象。

设计模式带来的第一个价值是命名。团队成员说“这里适合策略”，通常比说“把这些 if 分支拆成若干实现类，再由一个上下文按规则选择其中一个执行”更高效。模式名称是一种压缩后的设计语言，它让沟通聚焦在意图、边界和代价上，而不是只讨论代码形状。但命名只是一种辅助，不能替代判断。一个错误使用的模式，即使名字标准，也会让代码更难读。

第二个价值是依赖方向。好的模式通常会让高层策略依赖抽象而不是具体细节，让调用方依赖稳定契约而不是依赖一组易变类。这样当具体实现变化时，修改可以停留在实现层、工厂层或注册层，不必扩散到所有调用点。依赖方向清晰之后，单元测试也更容易编写，因为测试可以替换协作者、构造边界条件，并验证某个对象对契约的使用方式。

第三个价值是控制重复。重复不只指代码文本重复，也包括设计决策重复。如果多个地方都在根据类型选择算法，真正重复的是“类型到算法的映射规则”；如果多个调用方都按相同步骤协调若干组件，真正重复的是“流程编排”；如果多个类都在调用前后做相同校验、日志或缓存，真正重复的是“横向职责”。设计模式把这些重复的决策提取出来，让它们拥有单一的维护位置。

第四个价值是表达约束。构造器、工厂、建造者、抽象工厂可以表达对象创建时的合法性和一致性；代理可以表达访问控制；外观可以表达一段协作流程的推荐入口；状态模式可以表达状态转换边界；责任链可以表达处理顺序和中断规则。很多设计问题并不是代码不知道怎么写，而是约束没有被代码结构明确表达，导致调用者可以轻易绕过规则。

但设计模式也有成本。每引入一个接口、抽象类、包装类或注册表，读者都需要理解它存在的原因。调试时调用链可能更长，定位行为需要跳转更多文件，简单需求可能被间接层包裹得过度复杂。因此模式的收益必须来自真实问题，而不是来自“看起来更高级”。如果一个分支很短、变化少、只出现一次，直接 `if` 或 `switch` 可能比策略类更合适。如果一个对象只用一行构造方法就能创建，工厂可能只是噪声。如果只有一个实现且没有替换需求，接口可能只是形式主义。

## 使用设计模式的基本边界

判断是否使用设计模式，可以先问四个问题。第一，这个变化点是否真实存在，还是只是想象出来的未来扩展？第二，这个变化点是否会出现在多个位置，或者是否已经让调用方知道了太多细节？第三，引入间接层后，调用方是否更简单，依赖是否更稳定，测试是否更容易？第四，新结构带来的类数量、跳转成本和调试成本是否可以接受？只有这些问题的答案整体为正，模式才值得引入。

模式适合解决重复出现的结构性问题，不适合替代清晰的业务建模。很多代码难以维护，并不是因为缺少模式，而是因为概念命名不准确、职责边界混乱、数据结构不合适、异常语义不清晰或并发约束没有定义。此时盲目引入工厂、策略和管理类，只会把原本的问题包在新层级里。设计模式应该服务于清晰模型，而不是掩盖模型贫血。

模式也不应该破坏 Java 的惯用表达。简单数据对象可以用记录类或不可变类，不必为了“建造者”写一套冗长结构；简单回调可以用函数式接口，不必总是定义命令类；明确有限的类型分支可以用枚举，不必过早拆成一组类；标准库已经提供的能力应优先使用，不必重复造轮子。所谓“面向对象设计”不等于“类越多越好”，更不等于“所有逻辑都不能写在方法里”。

在维护已有代码时，模式更适合作为重构结果，而不是一开始的默认形状。先写直接的代码，等到重复、变化和边界真正显现，再提取策略、工厂、外观或责任链，往往比预先铺满抽象更稳妥。过早设计容易让抽象与真实变化错位，后续修改不仅要改业务逻辑，还要拆掉错误抽象。一个好的设计模式落点，通常能让人读到代码时自然地说“这里确实需要这一层”，而不是“这里为什么这么绕”。

## 创建型模式

创建型模式处理对象创建问题。它们关心的不是简单的 `new` 关键字，而是创建过程中的选择、校验、默认值、一致性和依赖隐藏。对象创建一旦包含条件分支、多个可选参数、跨对象一致性、缓存复用或生命周期控制，就可能影响调用方代码的稳定性。创建型模式通过工厂、建造者、单例和原型等方式，把这些创建细节从使用场景中拆出来。

### 简单工厂与静态工厂方法

简单工厂不是经典 GoF 模式之一，但在 Java 中非常常见。它把根据输入选择具体实现的逻辑集中到一个方法或类中，让调用方只关心抽象类型。简单工厂适合创建分支较少、选择规则稳定、具体类型不希望散落在调用处的场景。

```java
interface Parser {
    Object parse(String text);
}

final class JsonParser implements Parser {
    public Object parse(String text) {
        return Map.of();
    }
}

final class CsvParser implements Parser {
    public Object parse(String text) {
        return List.of();
    }
}

final class ParserFactory {
    private ParserFactory() {
    }

    static Parser create(String type) {
        return switch (type) {
            case "json" -> new JsonParser();
            case "csv" -> new CsvParser();
            default -> throw new IllegalArgumentException("unsupported type: " + type);
        };
    }
}
```

简单工厂的收益是调用方不再到处写 `new JsonParser()`、`new CsvParser()` 和重复的类型判断。它的代价是新增类型通常要修改工厂，因此它并不完全符合“对扩展开放、对修改关闭”。这不是绝对缺点。如果类型集合由当前模块控制，新增类型本来就需要统一注册和校验，那么修改工厂反而能让选择规则集中可见。只有当实现来自多个独立模块，或者新增实现频繁到不适合修改中心工厂时，才需要考虑注册表、服务发现或工厂方法等更开放的机制。

Java 标准库和常见类库中还有大量静态工厂方法。静态工厂方法可以拥有名字，可以返回子类型或缓存实例，可以根据参数选择实现，也可以隐藏构造方法。与简单工厂类相比，静态工厂方法通常放在产品类自身或相关工具类中，适合创建规则与某个类型高度相关的场景。它的局限是不能被子类多态覆盖，因此如果创建逻辑需要由不同子类决定，就要转向工厂方法。

### 工厂方法

工厂方法把对象创建延迟到子类或具体工厂中。调用方依赖一个创建者抽象，具体创建者决定生产哪种产品。它适合产品使用流程稳定，但产品具体类型需要由不同创建者扩展的场景。

```java
interface Exporter {
    void export(String text, Path target);
}

abstract class ExportJob {
    public final void run(String text, Path target) {
        Exporter exporter = createExporter();
        exporter.export(text, target);
    }

    protected abstract Exporter createExporter();
}

final class TextExportJob extends ExportJob {
    protected Exporter createExporter() {
        return new PlainTextExporter();
    }
}
```

工厂方法的核心不是“把 `new` 包一层”，而是让稳定流程和可变产品解耦。上例中 `run` 的流程固定，具体导出器由子类决定。如果不存在稳定流程，只是为了避免写构造方法而创建一个工厂层，工厂方法就容易变成形式主义。它还依赖继承，扩展点要提前设计好；如果创建规则需要运行期组合，使用注入、注册表或简单工厂可能更直接。

### 抽象工厂

抽象工厂用于创建一组相互匹配的产品族。它的重点不是创建一个对象，而是保证一组对象来自同一实现体系，避免调用方随意混搭。只要多个协作对象之间存在一致性约束，抽象工厂就有表达价值。

```java
interface ReportFactory {
    Parser parser();

    Formatter formatter();

    Exporter exporter();
}

final class CsvReportFactory implements ReportFactory {
    public Parser parser() {
        return new CsvParser();
    }

    public Formatter formatter() {
        return new CsvFormatter();
    }

    public Exporter exporter() {
        return new PlainTextExporter();
    }
}
```

抽象工厂适合“产品族”稳定、“产品等级结构”清晰的场景。例如某个处理流程需要成套的解析器、格式化器和导出器，这些对象必须遵循同一数据约定。使用抽象工厂后，调用方只选择一个工厂，就能拿到一组匹配对象。它的代价是扩展新的产品种类比较麻烦：如果接口上新增一个产品方法，所有具体工厂都要跟着修改。因此抽象工厂适合产品族扩展较多、产品种类相对稳定的情况；如果产品种类频繁增加，组合式注册表可能比抽象工厂更灵活。

### 建造者

建造者适合构造参数多、可选项多、默认值多、校验规则集中、目标对象希望保持不可变的场景。它把“逐步收集参数”和“最终构造合法对象”分开，让调用方不用记住长构造器中每个参数的位置，也减少传入大量 `null` 或布尔值造成的歧义。

```java
final class Query {
    private final String table;
    private final List<String> columns;
    private final int limit;
    private final boolean distinct;

    private Query(Builder builder) {
        this.table = builder.table;
        this.columns = List.copyOf(builder.columns);
        this.limit = builder.limit;
        this.distinct = builder.distinct;
    }

    static Builder builder(String table) {
        return new Builder(table);
    }

    static final class Builder {
        private final String table;
        private final List<String> columns = new ArrayList<>();
        private int limit = 100;
        private boolean distinct;

        private Builder(String table) {
            this.table = Objects.requireNonNull(table);
        }

        Builder column(String column) {
            columns.add(Objects.requireNonNull(column));
            return this;
        }

        Builder limit(int limit) {
            this.limit = limit;
            return this;
        }

        Builder distinct() {
            this.distinct = true;
            return this;
        }

        Query build() {
            if (table.isBlank()) {
                throw new IllegalStateException("table required");
            }
            if (limit <= 0) {
                throw new IllegalStateException("limit must be positive");
            }
            return new Query(this);
        }
    }
}
```

建造者的收益在于可读性和合法性。`Query.builder("orders").column("id").limit(50).build()` 比一个包含多个字符串、整数和布尔值的构造器更容易读。最终校验集中在 `build` 中，可以保证创建出的对象满足不变量。不可变对象还能避免构造后被外部修改，降低并发和共享场景中的理解成本。

建造者的代价是代码量增加，并且可能让对象创建过程显得比实际复杂。两个必填参数的简单值对象通常不需要建造者。还有一种常见误区是把建造者做成没有任何校验的可变数据袋，这样它只是在构造器外面复制了一份字段，并没有维护不变量。真正有价值的建造者应该让非法对象尽量无法被创建，至少在 `build` 阶段明确失败。

### 单例

单例保证某个类型在一个类加载上下文中只有一个公开实例。Java 中常见实现包括饿汉式静态字段、静态内部类延迟加载、双重检查锁和枚举单例。枚举单例写法简洁，并且天然处理序列化时的实例唯一性。

```java
enum ClockRegistry {
    INSTANCE;

    private Clock clock = Clock.systemUTC();

    Clock clock() {
        return clock;
    }

    void use(Clock clock) {
        this.clock = Objects.requireNonNull(clock);
    }
}
```

单例适合无状态服务、进程内唯一注册表、全局配置视图或确实代表唯一资源的对象。但它也是最容易被滥用的模式之一。单例会引入全局状态和隐藏依赖，调用方可以在任何地方直接访问它，导致对象之间的真实依赖关系不再通过构造器或方法参数体现。测试时如果单例持有可变状态，还容易造成用例之间互相污染。

因此使用单例要区分“唯一实例”和“全局访问”。某个对象在运行时只需要一个实例，并不意味着所有代码都应该静态访问它。更清晰的做法通常是由应用启动层创建唯一实例，再通过构造器显式传给需要的对象。这样既保留了实例唯一性，又让依赖关系可见。只有当对象确实无状态、访问成本低、替换需求很弱时，全局单例才更容易被接受。

### 原型

原型模式通过复制已有对象来创建新对象，适合对象构造成本高、初始状态复杂、或希望以某个模板为基础生成多个变体的场景。Java 提供了 `Cloneable` 和 `clone`，但这套机制有不少历史包袱：`Cloneable` 没有声明 `clone` 方法，浅拷贝和深拷贝容易混淆，继承层级中也难以保证每个字段都被正确复制。

在现代 Java 代码中，更推荐显式复制构造器、静态复制方法或 `with` 风格方法。这样复制语义更清楚，也可以明确哪些字段共享、哪些字段重新创建。

```java
final class RenderOptions {
    private final String theme;
    private final int width;

    RenderOptions(String theme, int width) {
        this.theme = theme;
        this.width = width;
    }

    RenderOptions withWidth(int width) {
        return new RenderOptions(theme, width);
    }
}
```

原型模式的关键边界是复制语义。浅拷贝只复制对象引用，多个对象可能共享同一个可变成员；深拷贝会递归复制成员，但成本更高，也可能遇到循环引用或资源对象。只要对象包含集合、缓存、文件句柄、连接、锁或线程相关状态，就必须谨慎定义复制行为。没有清晰复制语义的原型，比直接构造更危险。

## 结构型模式

结构型模式关注对象如何组合。它们解决的是接口不匹配、职责叠加、复杂结构简化、对象访问控制、整体与部分统一处理、共享对象复用等问题。结构型模式往往让调用方看到一个稳定表面，而把复杂适配、包装、委托和组合隐藏在内部。

### 适配器

适配器把一个已有接口转换成调用方期望的另一个接口。它适合已有类功能可用，但方法名、参数模型、返回类型或异常语义与当前抽象不一致的场景。适配器的重点是隔离不匹配，让调用方依赖自己的稳定契约，而不是被外部或历史接口牵着走。

```java
interface TextReader {
    String read();
}

final class SupplierReader implements TextReader {
    private final Supplier<String> supplier;

    SupplierReader(Supplier<String> supplier) {
        this.supplier = Objects.requireNonNull(supplier);
    }

    public String read() {
        return supplier.get();
    }
}
```

适配器通常有两种形态。对象适配器通过组合持有被适配对象，灵活性更好，也是 Java 中更常见的方式。类适配器通过继承实现适配，但 Java 单继承限制使它的适用范围较小。适配器不应该承载太多业务决策，它主要负责翻译接口。如果适配层开始包含复杂流程、缓存、权限和统计，就要重新判断这些职责是否属于代理、装饰器或外观。

适配器也常用于隔离第三方库或遗留代码。调用方定义自己需要的接口，适配器负责把外部对象转换进来。这样外部 API 变化时，修改集中在适配层；测试时也可以用简单假实现替代真实外部依赖。不过适配器不能神奇消除语义差异，如果外部行为和内部契约本质上不一致，就必须在适配层明确处理失败、默认值和不支持操作，而不是假装完全兼容。

### 桥接

桥接模式把抽象层次和实现层次分离，使两者可以独立变化。它适合一个概念存在两个以上变化维度，如果用继承组合所有维度，会导致类数量膨胀的场景。桥接通过“抽象对象持有实现接口”的方式，把继承层级转化为组合关系。

```java
interface MessageSender {
    void send(String title, String body);
}

abstract class Notification {
    private final MessageSender sender;

    protected Notification(MessageSender sender) {
        this.sender = sender;
    }

    public final void notify(String body) {
        sender.send(title(), body);
    }

    protected abstract String title();
}

final class WarningNotification extends Notification {
    WarningNotification(MessageSender sender) {
        super(sender);
    }

    protected String title() {
        return "warning";
    }
}
```

如果没有桥接，可能会出现 `EmailWarningNotification`、`SmsWarningNotification`、`EmailInfoNotification`、`SmsInfoNotification` 这样的组合爆炸。桥接把“通知类型”和“发送方式”拆成两个维度，新增通知类型不必新增所有发送方式子类，新增发送方式也不必修改通知类型层级。

桥接的边界在于变化维度是否真的独立。如果两个维度强耦合，强行拆开会让对象之间频繁互相询问，反而增加复杂度。桥接也不是所有组合关系的别名，它强调的是抽象与实现分离，并且两边都有独立扩展的需要。

### 组合

组合模式把单个对象和对象集合放在同一接口下处理，适合树形结构。目录与文件、表达式节点、菜单项、组织结构、规则树都可以用组合思想建模。它让调用方可以对叶子节点和组合节点使用统一操作。

```java
interface Node {
    int size();
}

final class TextNode implements Node {
    private final String text;

    TextNode(String text) {
        this.text = text;
    }

    public int size() {
        return text.length();
    }
}

final class GroupNode implements Node {
    private final List<Node> children;

    GroupNode(List<Node> children) {
        this.children = List.copyOf(children);
    }

    public int size() {
        return children.stream().mapToInt(Node::size).sum();
    }
}
```

组合模式的价值在于递归一致性。调用方不需要判断当前对象是单个节点还是一组节点，只调用同一个接口即可。它的风险是把不适合叶子节点的操作也塞进统一接口。例如 `addChild` 对组合节点有意义，对叶子节点没有意义。如果统一接口包含太多只对部分节点有效的方法，调用方就会遇到不支持操作异常，组合模式的简洁性会下降。更稳妥的做法是让不可用操作不进入公共接口，或者在构建阶段而不是运行阶段处理树结构变更。

### 装饰器

装饰器在不改变原对象类型的前提下叠加行为。它通常与共同接口配合使用：装饰器实现同一接口，内部持有同接口对象，并在调用前后增加逻辑。装饰器适合日志、计时、缓存、校验、重试、压缩、加密等横向能力，这些能力可以独立叠加，也可以按顺序组合。

```java
final class TimingParser implements Parser {
    private final Parser delegate;

    TimingParser(Parser delegate) {
        this.delegate = Objects.requireNonNull(delegate);
    }

    public Object parse(String text) {
        long start = System.nanoTime();
        try {
            return delegate.parse(text);
        } finally {
            System.out.println(System.nanoTime() - start);
        }
    }
}
```

装饰器和继承都能扩展行为，但装饰器更适合运行期组合。继承是在类定义阶段固定扩展关系，多个扩展维度会造成子类组合爆炸；装饰器则可以把多个能力按对象链叠起来。Java I/O 中大量输入输出流包装类就是装饰思想的典型体现。

装饰器的代价是调用链变长，调试时要沿着包装层逐级进入。多个装饰器叠加时，顺序也可能影响结果，例如先缓存后计时与先计时后缓存统计含义不同。设计装饰器时应保持单一职责，避免一个装饰器同时做校验、缓存、重试和日志；否则它就不再是可组合的小能力，而变成难以复用的混合对象。

### 代理

代理控制对真实对象的访问。它和装饰器结构相似，都是实现同一接口并持有目标对象，但意图不同：装饰器强调增强职责，代理强调访问控制。代理可以用于延迟创建、权限校验、远程调用、缓存、事务边界、统计采样或失败隔离。静态代理由手写类实现，动态代理可以在运行期生成代理对象。

```java
interface Repository {
    String findById(String id);
}

final class LazyRepositoryProxy implements Repository {
    private final Supplier<Repository> factory;
    private Repository target;

    LazyRepositoryProxy(Supplier<Repository> factory) {
        this.factory = factory;
    }

    public String findById(String id) {
        if (target == null) {
            target = factory.get();
        }
        return target.findById(id);
    }
}
```

Java 动态代理要求目标抽象是接口，适合为一组接口方法统一增加横向处理。基于字节码生成的代理可以代理类，但会受到 `final` 类、`final` 方法、构造过程和可见性等限制。无论采用哪种方式，代理都应尽量保持对外契约一致。调用方使用代理时，不应该在正常情况下感知自己面对的是代理还是真实对象，除非代理本身明确改变了访问语义。

代理的误区是把所有横向逻辑都塞进一个庞大拦截器。访问控制、缓存、重试、事务和日志虽然都可以通过代理实现，但它们的失败语义、执行顺序和异常处理不同。代理层越通用，越需要清楚定义规则，否则问题会从业务代码转移到难以调试的代理链中。

### 外观

外观为复杂子系统提供简化入口。它不一定隐藏所有能力，而是把常用流程收束到更稳定的高层 API 中，让调用方不必了解内部多个组件的协作顺序。外观适合子系统内部对象较多、协作流程固定、调用方重复编排相同步骤的场景。

```java
final class ReportService {
    private final Parser parser;
    private final Formatter formatter;
    private final Exporter exporter;

    ReportService(Parser parser, Formatter formatter, Exporter exporter) {
        this.parser = parser;
        this.formatter = formatter;
        this.exporter = exporter;
    }

    void generate(String input, Path target) {
        Object data = parser.parse(input);
        String text = formatter.format(data);
        exporter.export(text, target);
    }
}
```

外观的收益是降低调用方认知负担，并把流程一致性集中到一个入口。调用方只需要知道“生成报告”，不需要每次都记住先解析、再格式化、最后导出的顺序。外观也可以作为模块边界，对外提供少量稳定方法，对内允许子系统重构。

外观的风险是膨胀成上帝类。如果外观不断收纳所有方法，最终会变成一个巨大服务，既了解所有子系统，又承载大量业务分支。好的外观应该围绕明确用例或稳定流程组织，而不是成为所有对象的转发器。外观也不应该彻底封死底层能力；对于高级调用方，可以保留更细粒度接口，但推荐普通调用方使用外观入口。

### 享元

享元模式通过共享大量细粒度对象来降低内存占用。它适合对象数量巨大、对象中有大量可共享的不可变内部状态，而外部状态可以由调用方传入的场景。享元的关键是区分内部状态和外部状态：内部状态存放在共享对象中，必须不可变或等效不可变；外部状态由调用方在每次使用时提供。

```java
final class Symbol {
    private final String name;

    Symbol(String name) {
        this.name = name;
    }

    String render(int position) {
        return name + "@" + position;
    }
}

final class SymbolPool {
    private final Map<String, Symbol> symbols = new HashMap<>();

    Symbol get(String name) {
        return symbols.computeIfAbsent(name, Symbol::new);
    }
}
```

Java 中的字符串常量池、装箱类型缓存和枚举都体现了共享思想，但业务代码不应盲目实现对象池。现代 JVM 的对象分配和垃圾回收已经很高效，手写对象池可能引入锁竞争、生命周期错误和内存泄漏。享元真正适合的是对象数量极大且共享状态明显的场景。若只是为了避免少量对象创建，通常不值得引入池化结构。

## 行为型模式

行为型模式关注算法、流程、状态和对象协作。它们解决的是“谁来做”“何时做”“按什么顺序做”“行为如何替换”“请求如何流转”“状态如何影响行为”等问题。行为型模式通常比创建型和结构型更贴近业务规则，因此更需要谨慎命名和边界控制。

### 策略

策略模式把可替换算法封装成独立对象。调用方依赖策略接口，在运行时选择具体策略。它适合算法分支较多、分支逻辑复杂、同一分支规则出现在多个位置、或算法需要独立测试和演进的场景。

```java
interface DiscountPolicy {
    BigDecimal discount(BigDecimal amount);
}

final class NoDiscount implements DiscountPolicy {
    public BigDecimal discount(BigDecimal amount) {
        return BigDecimal.ZERO;
    }
}

final class RateDiscount implements DiscountPolicy {
    private final BigDecimal rate;

    RateDiscount(BigDecimal rate) {
        this.rate = rate;
    }

    public BigDecimal discount(BigDecimal amount) {
        return amount.multiply(rate);
    }
}
```

策略的收益是把“选择算法”和“执行算法”分开。算法实现可以独立测试，调用方可以通过配置、映射或构造器注入选择策略。它也能消除散落在多个类中的同类 `switch`。如果每个分支只有一行、只出现一次且变化很少，策略类可能比直接分支更难读。判断策略是否值得引入，核心看分支逻辑是否有独立生命周期。

现代 Java 中，简单策略也可以用函数式接口表达。例如 `Function<Order, BigDecimal>` 可以承载价格计算，`Predicate<User>` 可以承载条件判断。函数式写法简洁，但复杂逻辑不应藏在很长的 lambda 里。只要策略需要命名、复用、持有依赖、记录状态或单独测试，就应该提取为具名类或具名方法。

### 模板方法

模板方法把稳定流程放在父类方法中，把变化步骤延迟给子类实现。它适合流程骨架稳定、步骤差异有限、类族关系明确的场景。模板方法常把主流程方法声明为 `final`，避免子类破坏执行顺序。

```java
abstract class Task {
    final void run() {
        prepare();
        execute();
        cleanup();
    }

    void prepare() {
    }

    abstract void execute();

    void cleanup() {
    }
}
```

模板方法的优势是可以明确表达流程顺序，并复用公共步骤。它的代价是依赖继承，子类与父类耦合较强。父类一旦包含太多钩子方法，子类就需要理解复杂的调用时机；父类一旦在构造过程中调用可覆盖方法，还可能导致对象尚未初始化完成就执行子类逻辑。模板方法适合稳定类族，不适合运行期频繁切换行为。如果变化点需要组合和替换，策略通常比模板方法灵活。

### 观察者

观察者模式定义对象间的一对多通知关系。主题状态变化时通知订阅者，订阅者不需要被主题直接硬编码。它适合事件生产者和消费者数量不固定、双方生命周期相对独立、或希望新增消费者时不修改生产者的场景。

```java
interface Listener {
    void onChanged(String value);
}

final class Subject {
    private final List<Listener> listeners = new ArrayList<>();

    void addListener(Listener listener) {
        listeners.add(listener);
    }

    void removeListener(Listener listener) {
        listeners.remove(listener);
    }

    void update(String value) {
        for (Listener listener : List.copyOf(listeners)) {
            listener.onChanged(value);
        }
    }
}
```

观察者的价值是解耦通知发送者和接收者。主题只知道监听器接口，不知道具体监听者数量和类型。新增监听者通常不需要修改主题逻辑。但观察者也带来隐式控制流：一个 `update` 可能触发很多对象执行代码，调用者不一定直观看到后续影响。事件过多时，系统行为会变得难以追踪。

实现观察者时要明确几个细节。通知顺序是否有保证？监听者抛出异常时，是中断后续通知，还是隔离失败继续通知？监听者能否在通知过程中新增或移除订阅？主题是否需要弱引用避免订阅者泄漏？通知是同步执行还是异步投递？这些问题如果不定义清楚，观察者模式会从解耦工具变成排查困难的隐式依赖网络。

### 责任链

责任链把请求沿处理器链传递，每个处理器决定处理、修改、拒绝或继续传递。它适合校验、过滤、拦截、分级处理和可插拔流程。责任链让每个处理器只关注一小段逻辑，链条负责组织执行顺序。

```java
interface Handler {
    boolean handle(Request request);
}

final class Pipeline {
    private final List<Handler> handlers;

    Pipeline(List<Handler> handlers) {
        this.handlers = List.copyOf(handlers);
    }

    boolean handle(Request request) {
        for (Handler handler : handlers) {
            if (!handler.handle(request)) {
                return false;
            }
        }
        return true;
    }
}
```

责任链的关键是协议清晰。处理器返回 `true` 表示继续还是处理成功？返回 `false` 表示拒绝还是已经处理完毕？处理器能否修改请求？异常是否由链条统一捕获？链条顺序由谁决定？如果这些规则模糊，链条变长后会很难定位请求在哪里被改变、拦截或终止。

责任链适合线性流程，但不适合复杂分支流程。如果处理器之间存在大量条件跳转、回退和交叉依赖，责任链会变成隐藏的流程图，不如显式流程编排清楚。链条中的处理器也要保持粒度适中，过细会造成跳转过多，过粗又失去可组合价值。

### 命令

命令模式把一次操作封装成对象，使操作可以排队、撤销、记录、重试、延迟执行或跨线程传递。Java 标准库中的 `Runnable` 和 `Callable` 都体现了命令思想：调用方把要执行的动作作为对象交给执行器。

```java
interface Command {
    void execute();
}

final class RenameCommand implements Command {
    private final Document document;
    private final String name;

    RenameCommand(Document document, String name) {
        this.document = document;
        this.name = name;
    }

    public void execute() {
        document.rename(name);
    }
}
```

命令对象比直接调用方法多了一层包装，因此只有当操作需要作为数据被传递时才有明显价值。比如任务队列需要保存待执行操作，撤销栈需要记录反向动作，审计系统需要记录操作元数据，重试机制需要保存参数和失败原因。若只是立即执行的一行代码，命令模式可能过重。

命令模式常与备忘录、责任链、组合模式一起使用。撤销功能需要保存执行前状态或反向命令；批处理可以把多个命令组合成一个宏命令；队列执行可以在命令外层加代理或装饰器处理重试、日志和限流。命令的边界是副作用管理：命令执行是否幂等，失败后能否重试，部分执行后如何回滚，都需要显式定义。

### 状态

状态模式把对象在不同状态下的行为分散到状态对象中，让状态转换替代大量条件判断。它适合状态数量明确、每个状态下行为差异明显、转换规则需要集中表达的场景。

```java
interface OrderState {
    OrderState pay();

    OrderState cancel();
}

final class CreatedState implements OrderState {
    public OrderState pay() {
        return new PaidState();
    }

    public OrderState cancel() {
        return new CancelledState();
    }
}

final class PaidState implements OrderState {
    public OrderState pay() {
        throw new IllegalStateException("already paid");
    }

    public OrderState cancel() {
        throw new IllegalStateException("cannot cancel paid order");
    }
}
```

状态模式和策略模式结构相似，但意图不同。策略关注可替换算法，策略之间通常彼此独立；状态关注对象生命周期，不同状态之间存在转换关系。状态对象不仅决定当前行为，也决定下一状态。使用状态模式后，原本集中在一个类中的 `if (status == ...)` 可以移动到各个状态类中，每个状态只描述自己允许的行为。

状态模式的风险是类数量增加，并且状态转换可能分散。如果转换规则需要全局审计，可以通过状态机表、枚举加转换矩阵或集中转换服务表达。状态模式适合行为复杂的状态，不适合只有少量状态码和简单展示逻辑的场景。若状态没有行为差异，只是数据字段，拆成状态类反而会增加负担。

### 迭代器

迭代器提供顺序访问聚合对象元素的方式，而不暴露内部表示。Java 集合框架已经把迭代器作为基础抽象，`Iterable`、`Iterator`、增强 `for` 循环和流式操作都建立在类似思想上。

```java
final class Lines implements Iterable<String> {
    private final List<String> values;

    Lines(List<String> values) {
        this.values = List.copyOf(values);
    }

    public Iterator<String> iterator() {
        return values.iterator();
    }
}
```

迭代器的价值是隔离遍历方式。调用方不需要知道集合底层是数组、链表、树还是懒加载序列，只要通过统一协议取下一个元素即可。自定义聚合对象如果希望支持增强 `for`，实现 `Iterable` 往往比暴露内部集合更合适。

使用迭代器要注意修改语义。遍历过程中修改集合可能触发快速失败，也可能产生弱一致视图，具体取决于集合实现。自定义迭代器还要明确是否支持 `remove`、是否惰性计算、是否可重复遍历、是否需要关闭底层资源。迭代器看似简单，但一旦涉及资源和并发，契约必须写清楚。

### 中介者

中介者模式用一个对象封装多个对象之间的交互，避免对象之间形成网状依赖。它适合组件数量较多、交互规则集中、对象彼此直接引用会导致耦合爆炸的场景。每个组件只与中介者通信，中介者负责协调其他组件。

中介者的收益是降低组件之间的直接依赖，让交互流程有统一入口。它常用于表单组件联动、工作流协调、模块间消息调度等场景。风险是中介者膨胀。如果所有规则都堆到一个中介者里，它会成为新的复杂中心。好的中介者应该围绕一组相关交互建模，必要时拆分为多个协作者，而不是承担所有对象之间的全部通信。

中介者和观察者都能解耦对象关系，但方向不同。观察者强调事件发布与订阅，发送者不关心谁接收；中介者强调协调规则，组件把交互请求交给中介者，由中介者决定影响哪些对象。选择哪个模式，取决于核心复杂度是在“通知扩散”还是“协作编排”。

### 备忘录

备忘录模式在不暴露对象内部结构的前提下保存和恢复对象状态。它适合撤销、回滚、快照和临时试算。备忘录通常由发起者创建，外部管理者只负责保存，不理解其中细节。

```java
final class Editor {
    private String text = "";

    Snapshot snapshot() {
        return new Snapshot(text);
    }

    void restore(Snapshot snapshot) {
        this.text = snapshot.text();
    }

    record Snapshot(String text) {
    }
}
```

备忘录的关键是封装和成本。保存完整快照简单可靠，但可能占用大量内存；保存增量变化更节省，但恢复逻辑复杂。快照对象应该尽量不可变，避免保存后又被外部修改。对于包含外部资源、线程状态或大型对象图的对象，备忘录并不一定适合，可能需要更明确的事务、日志或持久化机制。

### 访问者

访问者模式把作用于对象结构中各元素的操作封装成访问者，使得可以在不修改元素类的情况下增加新操作。它适合对象结构稳定、元素类型较少且不常变化，但需要频繁增加新操作的场景。典型例子是语法树、表达式树或规则树。

```java
interface Expr {
    <R> R accept(ExprVisitor<R> visitor);
}

interface ExprVisitor<R> {
    R visitNumber(NumberExpr expr);

    R visitAdd(AddExpr expr);
}

record NumberExpr(int value) implements Expr {
    public <R> R accept(ExprVisitor<R> visitor) {
        return visitor.visitNumber(this);
    }
}

record AddExpr(Expr left, Expr right) implements Expr {
    public <R> R accept(ExprVisitor<R> visitor) {
        return visitor.visitAdd(this);
    }
}
```

访问者解决的是“操作多、结构稳定”的问题。新增一个求值访问者、打印访问者或校验访问者，不需要修改每个节点类。但它的反面代价很明显：新增一个节点类型时，所有访问者接口和实现都要修改。因此如果元素类型频繁变化，访问者并不合适。

访问者还有一个现实成本是样板代码较多，双分派理解门槛较高。Java 近年来的模式匹配和密封类能力可以在某些场景中替代传统访问者，让有限类型集合的分支处理更直接。是否使用访问者，要看操作扩展和元素扩展哪个更频繁，而不是看它是否“更面向对象”。

## Java 实现中的取舍

Java 中实现设计模式时，最重要的取舍是接口、继承、组合、枚举、lambda 和标准库之间的选择。接口是多数模式的基础，但接口不是越多越好。只有当存在多个实现、需要测试替换、需要隔离模块边界、需要稳定契约或需要延迟绑定时，接口才有明显收益。如果一个接口只有一个实现，且没有替换计划，读者就会怀疑它是否只是为了形式而存在。

组合通常比继承更灵活。策略、装饰器、代理、桥接和责任链都依赖对象组合，可以在运行期调整协作关系。模板方法依赖继承，适合稳定类族，但扩展自由度较低。继承会把父类实现细节暴露给子类，也会让父类演进更困难，因为子类可能依赖了某个受保护方法的调用顺序。除非确实需要表达“是一种”的类型关系和复用稳定流程，否则优先考虑组合。

枚举可以承载有限策略和状态。当策略集合固定、每个策略行为简单、且不需要复杂依赖时，枚举常量重写方法可以减少类数量。

```java
enum RoundingPolicy {
    UP {
        BigDecimal apply(BigDecimal value) {
            return value.setScale(2, RoundingMode.UP);
        }
    },
    DOWN {
        BigDecimal apply(BigDecimal value) {
            return value.setScale(2, RoundingMode.DOWN);
        }
    };

    abstract BigDecimal apply(BigDecimal value);
}
```

枚举策略的优点是类型集合封闭、分支集中、调用简单。缺点是每个常量不适合持有复杂可变依赖，也不适合由外部模块扩展。若策略需要读取配置、访问仓储、调用服务或拥有较多内部状态，独立策略类更清晰。

lambda 能简化函数式策略、回调和命令，但不应让复杂逻辑匿名化。短小、无状态、语义明显的行为适合 lambda；长逻辑、有分支、有异常处理、有复用需求或需要单独测试的行为，应提取为具名类型或方法。lambda 的简洁性来自省略结构，不来自省略命名。没有名字的复杂逻辑会降低可读性。

泛型可以让模式更类型安全。例如通用工厂、访问者返回值、处理器链上下文都可以用泛型表达。但泛型也会增加签名复杂度。若泛型参数只是为了追求抽象而没有实际类型约束收益，就会让调用方困惑。好的泛型设计应该让错误在编译期暴露，而不是让方法签名变成难以理解的符号堆叠。

反射和动态代理可以实现高度通用的模式框架，但要谨慎使用。反射会绕过一部分编译期检查，动态代理会让调用路径不再直接可见。它们适合框架层、基础设施层和横向能力，不适合普通业务逻辑随手使用。能用普通对象组合表达的设计，通常不需要反射参与。

异常语义也是模式实现的一部分。工厂创建失败应该抛出什么异常？责任链某个处理器失败是否继续？观察者监听器异常是否影响其他监听器？命令执行失败能否重试？装饰器是否应该吞掉异常？这些行为如果不定义清楚，模式结构再漂亮也无法保证可维护性。

## 模式重构的识别信号

设计模式最好从真实重复中长出来。若同一个 `switch` 按类型分支出现在多个位置，并且每个分支都执行不同算法，策略或多态可能更合适。若对象构造参数越来越多，并且调用处充满 `null`、默认值或难以理解的布尔参数，建造者可能更清晰。若调用方需要按固定顺序协调多个子组件，外观可以收束流程。若对象创建要保证一组协作者一致，抽象工厂可能比散落构造更可靠。

另一个信号是测试困难。如果一个类在构造时直接创建大量具体依赖，调用方无法替换外部协作者，工厂、依赖注入或接口抽象可能有价值。如果测试必须了解很多内部步骤才能驱动一个行为，说明职责边界可能需要重分配。如果一个方法既选择算法、又执行算法、又记录日志、又处理异常，策略、装饰器或模板方法可能帮助拆分职责。

第三个信号是变化扩散。新增一种类型时，如果要修改十几个调用点，说明选择逻辑可能没有集中。新增一个横向能力时，如果要在大量方法开头和结尾复制代码，说明装饰器、代理或责任链可能更合适。新增一个状态时，如果要到处搜索状态码分支，说明状态转换规则可能需要更明确的模型。

也要识别不该引入模式的信号。只有一个实现且短期没有变化计划时，接口可能只是噪声。只有两个简单分支时，`if` 或 `switch` 可能比策略类更易读。对象创建只有一行构造方法时，工厂会让代码跳转更多。没有树形结构时，不要为了统一接口引入组合模式。状态没有行为差异时，不要拆状态类。模式的收益应来自变化隔离、职责清晰和依赖稳定，而不是来自名称本身。

重构到模式时应小步进行。先用测试固定现有行为，再提取接口或类；先消除重复选择逻辑，再决定是否需要注册表；先把长方法拆成有意义的步骤，再判断模板方法是否合适；先把横向逻辑包成一个装饰器，再决定是否需要通用代理框架。模式是重构后的结构，不应成为一次性大改的借口。

## 小型示例：从分支到策略

假设一段代码根据计费类型计算价格：

```java
BigDecimal price(Order order) {
    return switch (order.type()) {
        case NORMAL -> order.amount();
        case VIP -> order.amount().multiply(new BigDecimal("0.9"));
        case WHOLESALE -> order.amount().multiply(new BigDecimal("0.8"));
    };
}
```

如果这段逻辑只出现一次，分支清晰且稳定，保留 `switch` 没有问题。`switch` 的优势是局部可读，所有规则一眼可见。如果折扣算法开始出现在预估、结算、报表等多个位置，且每类算法不断变化，就可以抽出策略。

```java
interface PricingPolicy {
    BigDecimal price(Order order);
}

final class VipPricing implements PricingPolicy {
    public BigDecimal price(Order order) {
        return order.amount().multiply(new BigDecimal("0.9"));
    }
}
```

再由一个映射负责选择策略：

```java
final class PricingPolicies {
    private final Map<OrderType, PricingPolicy> policies;

    PricingPolicies(Map<OrderType, PricingPolicy> policies) {
        this.policies = Map.copyOf(policies);
    }

    PricingPolicy get(OrderType type) {
        PricingPolicy policy = policies.get(type);
        if (policy == null) {
            throw new IllegalArgumentException("unsupported order type: " + type);
        }
        return policy;
    }
}
```

这个重构的收益是算法变化集中到策略类，选择逻辑集中到注册表，调用方只依赖 `PricingPolicy`。代价是类型数量增加、跳转更多、初始化时需要注册策略。是否值得，取决于变化频率和复用范围。如果新增类型仍然需要修改枚举、注册表、配置和多处文档，那么策略并没有让扩展完全自动化；它只是把算法实现从选择逻辑中拆开。这个拆分是否有价值，要看算法本身是否足够复杂、是否需要独立测试、是否被多个场景复用。

进一步演进时，还可以把策略注册从构造器参数改为模块装配，把策略选择从简单 `Map` 改为按条件匹配，把策略执行外层套上装饰器统计耗时或处理异常。但每一步都应有真实需求。不要在只有三种稳定计费类型时提前做复杂的插件机制，否则新增结构会比原始分支更难维护。

## 常见误区

第一个误区是为了使用模式而设计。模式应该从问题中长出来，而不是先选模式再找用处。没有变化点的抽象会增加阅读成本；没有复用需求的工厂会增加跳转；没有复杂状态行为的状态类会分散逻辑；没有大量对象共享的享元会带来池化复杂度。设计模式不是代码成熟度的装饰品。

第二个误区是把模式名称当成正确性的证明。一个类叫 `Factory`，不代表它的创建职责清晰；一个类叫 `Manager`，常常反而意味着职责模糊；一个类叫 `Strategy`，也不代表算法边界合理。判断一个模式是否正确，要看它是否让变化集中、依赖稳定、调用方简化、测试容易，而不是看命名是否像教科书。

第三个误区是过度集中。工厂、外观、中介者和管理类都容易膨胀成上帝对象。它们一开始可能只是集中创建或协调流程，后来不断吸收条件分支、缓存、权限、日志和异常处理，最终变成任何需求都要修改的中心。集中并不等于清晰，合理的集中应该围绕一个稳定决策或流程，而不是把所有知识放到一个类里。

第四个误区是把所有条件分支都替换成多态。条件分支并不天然低级，多态也不天然高级。少量稳定分支用 `switch` 更直观，特别是枚举分支能够让编译器帮助检查覆盖情况。当分支逻辑复杂、频繁扩展、散落多处或需要独立测试时，再考虑策略、状态或多态。优秀设计不是消灭所有分支，而是把分支放在最合适的位置。

第五个误区是忽略异常、线程和生命周期。观察者如果不取消订阅，可能造成对象长期被引用；单例如果持有可变状态，可能造成测试污染和并发问题；责任链如果没有异常策略，可能在某个处理器失败后留下半处理状态；命令如果没有幂等设计，重试可能产生重复副作用；装饰器如果顺序不清，缓存和统计结果可能失真。模式描述的是结构，工程代码还必须定义运行时语义。

第六个误区是把框架习惯等同于模式本身。很多框架会大量使用代理、工厂、观察者和模板方法，但业务代码不应机械模仿框架内部结构。框架追求通用性、扩展点和运行时装配，普通业务模块更需要局部清晰和直接表达。能用简单构造器和组合解决的问题，不必升级为复杂的框架式模式。

## 实践判断清单

使用创建型模式时，先判断创建复杂度来自哪里。如果只是构造器参数多，建造者可能合适；如果是根据类型选择实现，简单工厂或注册表可能合适；如果是一组协作者必须匹配，抽象工厂可能合适；如果是唯一实例，先判断是否真的需要全局访问；如果是复制模板对象，先定义清楚浅拷贝和深拷贝语义。不要把所有创建问题都交给同一个 `Factory`。

使用结构型模式时，先判断调用方真正需要看到什么接口。如果已有对象接口不匹配，用适配器；如果要叠加可组合能力，用装饰器；如果要控制访问，用代理；如果要简化子系统，用外观；如果有两个独立变化维度，用桥接；如果是树形结构，用组合；如果有大量可共享不可变状态，用享元。结构型模式的关键是让表面稳定，内部组合清楚。

使用行为型模式时，先判断变化的是算法、流程、状态还是协作方式。算法可替换，用策略；流程骨架稳定、步骤可变，用模板方法；一对多通知，用观察者；线性处理和拦截，用责任链；操作需要排队、撤销或记录，用命令；行为随状态变化，用状态；遍历聚合对象，用迭代器；多个组件交互复杂，用中介者；需要保存恢复点，用备忘录；对象结构稳定但操作频繁增加，用访问者。

落地到 Java 代码时，还要检查几个细节。抽象是否有足够清晰的名字？接口方法是否表达稳定契约，而不是泄露实现细节？异常是否有统一语义？对象是否不可变或线程安全？默认值和校验是否集中？装配关系是否容易在测试中替换？新增一个变化点时，需要修改的文件是否合理？调用方是否比之前更少知道内部细节？如果这些问题没有改善，模式就没有发挥作用。

最后，设计模式不应与简单代码对立。很多优秀 Java 代码看起来并不“模式化”，只是类名准确、方法短小、依赖清晰、数据不可变、异常明确、测试容易。模式是帮助达到这些目标的工具之一，而不是目标本身。真正有效的模式使用，通常让调用方知道更少，让变化集中在更小范围，让代码结构表达业务约束，同时仍然保持足够直接。

## 小结

设计模式是可复用的设计语言，不是固定代码模板。创建型模式管理构造复杂度和实例选择，结构型模式组织对象组合和接口表面，行为型模式分离流程、算法、状态和协作变化。在 Java 中应用模式时，应结合接口、组合、枚举、lambda、记录类、泛型和标准库能力，追求职责清晰、依赖稳定、异常语义明确和测试友好。

判断一个模式是否值得使用，不能只看它是否符合教科书类图，而要看它是否解决了当前代码的真实问题。它应该减少重复决策、隔离变化点、表达约束、稳定调用方，而不是制造额外跳转。面对简单稳定的代码，保持直接往往是最好的设计；面对重复扩散和变化频繁的代码，恰当的模式可以把复杂度压回可维护的边界内。

# 3.1.2 基础语法

## 定位

Java 基础语法是理解后续集合、并发、I/O、反射和工程代码的入口。它包含类型系统、变量与作用域、运算符、流程控制、方法、类成员、包和访问控制等规则。基础语法看似零散，但核心目标一致：让代码在编译期尽可能明确类型、边界和执行路径，从而减少运行期错误。

Java 是静态类型语言。变量、字段、参数和返回值在编译期都有确定类型，编译器会据此检查赋值、方法调用、重载选择和可见性。Java 也是区分大小写的语言，标识符、关键字和字面量写法都需要严格遵守语法规则。理解这些规则，有助于读懂编译错误背后的真实原因，而不是只靠 IDE 自动修复。

## 基本类型与引用类型

Java 类型分为基本类型和引用类型。基本类型包括 `byte`、`short`、`int`、`long`、`float`、`double`、`char`、`boolean`。引用类型包括类、接口、数组、枚举、注解等。基本类型变量直接保存值，引用类型变量保存对象引用。

整数默认字面量类型是 `int`，小数默认字面量类型是 `double`。给 `long` 赋较大字面量时应使用 `L` 后缀；给 `float` 赋小数字面量时应使用 `F` 后缀。

```java
long count = 3_000_000_000L;
float ratio = 0.75F;
double price = 19.99;
```

数字字面量中的下划线只用于提升可读性，不改变数值。它不能出现在数字开头、结尾、小数点旁或类型后缀旁。

基本类型之间存在自动类型提升和强制类型转换。容量较小的整数类型参与算术运算时通常会提升为 `int`。因此 `byte + byte` 的结果不是 `byte`，而是 `int`。

```java
byte a = 1;
byte b = 2;
int c = a + b;
```

强制转换可能丢失精度或发生溢出。`(int) 3.9` 得到 `3`，不是四舍五入；超出目标类型范围的整数转换会截断高位。涉及金额、计数、容量和边界条件时，类型选择应保守。

引用类型可以为 `null`，基本类型不能为 `null`。自动装箱会在基本类型和包装类型之间转换，例如 `int` 与 `Integer`。装箱对象可能为 `null`，拆箱时若值为 `null` 会抛出 `NullPointerException`。

## 变量、作用域与初始化

变量按位置可分为局部变量、实例字段、静态字段和参数。局部变量定义在方法、构造方法或代码块中，必须先显式赋值再使用。字段有默认值：数值类型为零值，`boolean` 为 `false`，`char` 为 `\u0000`，引用类型为 `null`。

作用域决定变量名在哪里可见。局部变量的作用域从声明位置开始，到所在代码块结束。较小作用域能减少误用，也能让读者更快判断变量生命周期。变量应尽量靠近首次使用位置声明。

`final` 修饰变量表示只能赋值一次。对基本类型而言，值不可再变；对引用类型而言，引用不可再指向别的对象，但对象自身是否可变取决于对象类型。

```java
final List<String> names = new ArrayList<>();
names.add("Alice");      // 合法，列表对象仍可变
// names = List.of();    // 非法，引用不能重新赋值
```

`var` 可以用于局部变量类型推断，编译器根据初始化表达式推断静态类型。`var` 不等于动态类型，也不能用于字段、方法参数和返回值。使用 `var` 时应确保右侧表达式能让读者清楚看出类型，否则会降低可读性。

## 标识符、关键字与语句边界

Java 程序由包声明、导入声明、类型声明、成员声明、语句和表达式逐层组成。语法学习不能只记住某个关键字的写法，还要理解它可以出现在哪里。`class`、`interface`、`enum`、`record` 用于声明类型；`if`、`switch`、`for`、`while` 用于组织执行路径；`public`、`private`、`static`、`final` 等修饰符用于约束成员或类型；`new`、`return`、`throw` 等关键字则直接参与表达式或控制转移。不同语法位置的规则不同，把只能用于声明位置的修饰符写到语句中，或把只能用于方法体内的控制语句写到类体中，都会产生编译错误。

标识符用于命名包、类型、方法、字段、局部变量和参数。Java 标识符区分大小写，`count` 与 `Count` 是两个不同名字。标识符不能与关键字相同，不能以数字开头，也不能包含空格和普通运算符。虽然 Java 支持较宽的 Unicode 字符范围，但在通用工程代码中，标识符通常应使用稳定、易输入、易搜索的英文单词。命名的核心不是炫技，而是让调用方和维护者能从名字中推断职责、类型和边界。

语句通常以分号结束，代码块使用花括号组织。分号不是装饰字符，它代表一条语句的结束。多写或少写分号都会改变语法结构，最典型的错误是在条件或循环后意外写出空语句。

```java
if (ready); {
    run(); // 这个代码块总会执行
}
```

上面代码能通过语法解析，但含义几乎一定不是作者想要的。`if (ready);` 已经是一条完整的空语句，后面的花括号只是普通代码块。类似问题也可能出现在 `while` 和 `for` 后面。基础语法阶段应养成一个习惯：控制语句即使只有一行，也优先写花括号，让代码结构显式可见。

表达式语句是能作为独立语句执行的表达式，常见形式包括赋值、自增自减、方法调用和对象创建。不是所有表达式都能单独成为语句，例如单独写一个比较表达式没有意义，编译器会拒绝。这个限制体现了 Java 对“产生结果”和“产生效果”的区分：表达式可以计算值，但只有带副作用或控制意义的表达式才适合作为独立语句存在。

空白符和换行大多不影响语义，但它们影响阅读。Java 不依赖缩进决定代码块，所以错误缩进不会改变编译结果，却会误导读者。花括号代表真实结构，缩进应跟随花括号，而不是反过来让读者从缩进去猜结构。阅读陌生代码时，优先看括号和语句边界，再看视觉排版。

## 声明、赋值与初始化顺序

声明负责引入名字，赋值负责给名字绑定值，初始化则强调对象或变量进入可用状态。三者经常写在同一行，但概念并不相同。

```java
int size;      // 声明
size = 10;     // 赋值
int limit = 20; // 声明并初始化
```

局部变量没有默认值，是 Java 基础语法中非常重要的规则。编译器会做确定赋值分析，确认局部变量在每条可能执行到的路径上都已赋值。只在某个分支中赋值，随后在分支外读取，可能无法通过编译。这个规则让许多低级错误提前暴露。

```java
int result;
if (success) {
    result = 1;
}
// System.out.println(result); // success 为 false 时没有赋值
```

字段不同于局部变量。实例字段在对象创建时获得默认值，静态字段在类初始化时获得默认值，然后再按声明顺序执行显式初始化和初始化块。默认值规则能保证对象内存有确定状态，但不代表默认值一定符合业务含义。一个引用字段默认为 `null`，只能说明它尚未指向对象，不能说明“空集合”“未知”“未加载”或“无结果”这些语义。基础代码中应把语义状态建模清楚，而不是让调用方猜测某个默认值代表什么。

同一个作用域内不能声明两个同名局部变量，内部块也不能重新声明已经在外部局部作用域中存在的同名变量。字段与局部变量可以同名，但局部变量会遮蔽字段。遮蔽不是重写，也不是复制，只是名字解析优先级发生变化。需要明确访问当前对象字段时，可以使用 `this`。

```java
class Counter {
    private int value;

    void setValue(int value) {
        this.value = value;
    }
}
```

`this` 表示当前对象引用，只能在实例上下文中使用。静态方法和静态初始化块没有当前对象，因此不能直接使用 `this`，也不能直接访问实例字段。这个限制来自成员归属：静态成员属于类，实例成员属于对象。若静态方法需要使用实例状态，必须先获得某个对象引用，再通过该引用访问。

初始化顺序是理解字段值和构造方法行为的基础。对象创建时，先为对象分配空间并设置字段默认值，然后执行父类初始化，再按源码顺序执行当前类的实例字段初始化和实例初始化块，最后执行构造方法体。静态字段和静态初始化块在类初始化时按源码顺序执行一次。不要在初始化表达式中写过于复杂的逻辑，尤其不要依赖后面才初始化的字段，否则读者很难判断对象何时进入可用状态。

数组初始化也有声明、创建和赋值的区别。`int[] a;` 只是声明引用变量，`new int[3]` 才创建数组对象。数组初始化列表能在创建时给出元素值，但数组长度仍然固定。理解这一点可以避免把“数组引用可重新赋值”和“数组对象长度可变化”混为一谈。

```java
int[] numbers = {1, 2, 3};
numbers = new int[] {4, 5}; // 引用指向了新数组，原数组长度没有改变
```

`final` 局部变量必须在使用前且只在一条确定路径上赋值一次。空白 final 字段可以在构造方法中赋值，但必须保证每个构造路径都完成赋值。这个规则常用于表达对象创建后某些核心状态不可再替换。不可再赋值不是深度不可变，若 final 字段指向可变对象，对象内部状态仍可能变化。

## 运算符

Java 运算符包括算术、比较、逻辑、位运算、赋值、条件和类型判断等。算术运算要注意整数除法和溢出。`5 / 2` 结果是 `2`，若需要小数结果，至少一个操作数应是浮点类型。整数溢出不会自动报错，而是按二进制补码规则回绕。

比较运算符 `==` 对基本类型比较值，对引用类型比较引用是否指向同一对象。对象内容比较通常应使用 `equals`。字符串、包装类型和自定义值对象尤其要避免误用 `==`。

```java
String a = new String("java");
String b = new String("java");
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true
```

逻辑运算符 `&&` 和 `||` 具有短路行为。`a && b` 中若 `a` 为 false，不再计算 `b`；`a || b` 中若 `a` 为 true，不再计算 `b`。短路常用于空值保护。

```java
if (name != null && !name.isBlank()) {
    System.out.println(name);
}
```

位运算符作用于整数类型，包括 `&`、`|`、`^`、`~`、`<<`、`>>`、`>>>`。它们适合处理掩码、标志位和底层编码，但在普通业务代码中应避免为了“简短”而牺牲可读性。

条件运算符 `condition ? a : b` 适合表达简单二选一。若表达式过长或含副作用，使用 `if` 更清晰。

## 流程控制

`if` 用于条件分支。多个条件分支应按最容易理解或最常见的路径组织，避免深层嵌套。提前返回可以减少缩进层级，但也要保证资源释放和状态一致。

`switch` 可用于 `byte`、`short`、`char`、`int`、包装类型、`String` 和枚举等类型。传统 `switch` 语句需要注意 `break`，否则会继续执行后续分支。较新的 `switch` 表达式可以使用箭头形式返回值，减少遗漏 `break` 的风险。

```java
String label = switch (status) {
    case 0 -> "new";
    case 1 -> "running";
    case 2 -> "done";
    default -> "unknown";
};
```

循环包括 `for`、增强 `for`、`while` 和 `do while`。普通 `for` 适合需要索引的场景；增强 `for` 适合只遍历元素；`while` 适合循环次数不明确、由条件驱动的场景。修改集合时要理解迭代器规则，不能在增强 `for` 中直接调用集合的结构性修改方法。

`break` 终止当前循环或 `switch`，`continue` 跳过本轮循环后续语句。带标签的 `break` 和 `continue` 可以作用于外层循环，但可读性较差，应谨慎使用。

## 方法与参数

方法由修饰符、返回类型、方法名、参数列表、异常声明和方法体组成。方法名应表达动作或查询意图。返回 `void` 的方法通常表示执行命令；有返回值的方法通常表示查询或转换。把命令和查询混在一起会增加调用方理解成本。

Java 参数传递只有值传递。传递基本类型时复制值；传递引用类型时复制引用值。方法内部可以通过这个引用修改对象状态，但不能让调用方变量重新指向另一个对象。

```java
void change(List<String> list) {
    list.add("x");
    list = new ArrayList<>();
    list.add("y");
}
```

调用 `change(names)` 后，原列表会增加 `"x"`，但调用方变量不会指向新列表。这个规则是理解 Java 方法副作用的关键。

可变参数使用 `...` 声明，本质上是数组。一个方法最多只能有一个可变参数，并且必须位于参数列表最后。可变参数适合少量便利调用，不适合替代清晰的数据结构。

方法重载要求方法名相同、参数列表不同。编译器选择重载方法时会考虑精确匹配、类型提升、装箱和可变参数，复杂重载可能导致调用结果不直观。对外 API 应避免只靠细微参数类型差异区分语义。

方法签名中的返回类型不参与重载区分。两个方法如果只有返回类型不同，参数列表完全相同，不能同时存在。原因很直接：调用表达式可以忽略返回值，编译器无法仅凭调用语句判断应该选择哪一个方法。方法重载应服务于同一个语义动作在不同输入形态下的便利表达，而不是把完全不同的行为塞进同一个名字。

`return` 用于结束方法并返回结果。非 `void` 方法必须保证所有正常执行路径都返回与声明类型兼容的值；`void` 方法可以使用不带值的 `return` 提前结束。`return` 不是简单跳出当前代码块，而是结束整个方法执行。因此在多层循环或复杂条件中使用提前返回时，要确认它不会绕过必要的状态处理。若方法使用 `try finally`，即使在 `try` 中执行 `return`，`finally` 仍会在方法真正返回前执行。

方法参数是局部变量的一种，它们在方法调用开始时被初始化。参数名只在方法体内有效，不属于方法签名的一部分。调用方只根据参数位置和类型传值，不能依赖被调用方法内部的参数名。连续多个同类型参数容易造成调用错误，例如 `resize(10, 20)` 很难从语法上看出哪个是宽度、哪个是高度。基础语法允许这种写法，但良好的 API 会尽量让调用点具备自解释能力。

递归方法是方法调用自身的形式。语法上递归没有特殊关键字，关键在于必须有明确终止条件，并且每次递归都应向终止条件推进。递归适合表达树形结构、分治计算等天然递归的问题，但每次调用都会消耗调用栈。基础阶段不需要把递归神秘化：它只是方法调用，只是调用目标恰好还是自己。

```java
int sum(int n) {
    if (n <= 0) {
        return 0;
    }
    return n + sum(n - 1);
}
```

方法体中的局部变量生命周期由执行路径决定，作用域由代码块决定。一个变量离开作用域后名字不可再访问，但对象是否被回收不是由作用域单独决定，而取决于是否还有可达引用。基础语法只定义名字可见性，不直接定义内存回收时刻。不要把“变量出了作用域”误解为“对象立刻销毁”。

## 构造方法、对象创建与成员访问

构造方法用于初始化新对象。它的名字必须与类名相同，没有返回类型，连 `void` 也不能写。使用 `new` 创建对象时，构造方法会被调用，创建表达式的结果是新对象引用。若类没有显式声明任何构造方法，编译器会提供一个无参构造方法；一旦声明了任意构造方法，默认无参构造方法就不再自动生成。

```java
class User {
    private final String name;

    User(String name) {
        this.name = name;
    }
}
```

构造方法可以重载。一个构造方法可以使用 `this(...)` 调用同类中的另一个构造方法，也可以使用 `super(...)` 调用父类构造方法。二者若出现，必须是构造方法体中的第一条语句。这个限制保证父类部分和委托构造逻辑在当前构造方法继续执行前已经完成。构造方法不是普通方法，不能被继承，也不能被重写。

对象成员访问使用点号。`object.field` 访问字段，`object.method()` 调用方法。字段访问在编译期根据声明类型解析，实例方法调用则具有动态分派特性，会在运行期根据实际对象类型选择重写后的实现。基础语法层面要先区分“变量的静态类型”和“对象的运行期类型”。变量声明为某个父类型或接口时，只能直接访问该静态类型暴露的成员；对象实际类型可能更具体，但编译器不会因为运行期可能性放宽静态检查。

```java
CharSequence text = "hello";
int length = text.length();
// text.isBlank(); // CharSequence 没有声明 isBlank
```

字段和方法同名不冲突，因为访问语法不同，但这样命名会严重干扰阅读。局部变量、字段、方法、类型都共享读者的认知空间，基础代码应尽量避免靠语法上下文才能勉强区分的名字。语法允许不代表设计合理。

成员访问还涉及空引用。对 `null` 引用访问实例字段或调用实例方法会抛出 `NullPointerException`。静态成员应通过类名访问，而不是通过对象引用访问。通过对象引用访问静态成员虽然可能通过编译，但会让读者误以为访问依赖对象状态。清晰写法是 `TypeName.staticMember`。

## 类、包与访问控制

包用于组织类型命名空间。源文件顶部使用 `package` 声明所属包，使用 `import` 引入其他包中的类型。`import` 只是简化类型书写，不改变类的依赖关系。静态导入可以直接使用静态成员，但过度使用会让成员来源不清晰。

一个源文件最多只能有一个 public 顶级类，且文件名必须与 public 类名一致。源文件也可以包含非 public 顶级类，但为了可维护性，通常一个主要类型对应一个文件更清楚。

访问控制是 Java 模块化的基础之一。`public` 暴露范围最大，应谨慎使用；`private` 用于隐藏实现细节；包访问级别适合包内协作；`protected` 同时开放给同包和子类，语义比很多初学者想象得更宽。

成员修饰符还包括 `static`、`final`、`abstract`、`synchronized`、`native`、`strictfp` 等。基础阶段最常见的是 `static` 和 `final`。`static` 表示成员属于类而不是对象；`final` 表示不可再次赋值、不可重写或不可继承，具体含义取决于修饰目标。

包名本质上是类型全限定名的一部分。两个简单类名相同的类型只要包名不同，就代表不同类型。源码中的 `import` 只是在当前编译单元中建立简写，不会把被导入类型“复制”进来，也不会影响运行期加载规则。当两个导入类型简单名冲突时，不能同时用简单名区分，只能至少对其中一个使用全限定名。

```java
java.util.Date utilDate = new java.util.Date();
java.sql.Date sqlDate = java.sql.Date.valueOf("2026-01-01");
```

访问控制要从“谁能看见这个声明”来理解。顶级类只能是 `public` 或包访问级别；成员可以使用四种访问级别。`private` 只在当前顶级类型内部可见，包括同一个顶级类型中的嵌套类型彼此访问私有成员。包访问级别没有关键字，表示同一个包内可见。`protected` 包含同包可见和子类访问两层规则，不是简单的“只对子类可见”。`public` 对所有可访问到该类型的位置开放，但如果外层类型本身不可见，内部 public 成员也无法被外部直接访问。

访问控制表面上是语法修饰符，实际影响代码边界。字段通常不应直接暴露为 public，因为字段一旦成为外部契约，后续就很难在不影响调用方的情况下加入校验、延迟计算或表示方式调整。方法暴露也要克制：能保持 private 的辅助逻辑不要提前扩大可见性；确实需要包内协作时，再使用包访问级别。基础语法中的可见性选择，会长期影响代码能否演进。

`static` 成员不依赖对象实例。静态字段常用于类级共享状态或常量，静态方法常用于不读取实例状态的工具行为或工厂行为。但共享可变静态字段会让状态来源变得分散，也会让测试和并发场景更难推断。基础阶段应把 `static` 理解为归属关系，而不是“调用方便”的同义词。

`final` 修饰类表示不能被继承，修饰方法表示不能被子类重写，修饰字段或变量表示只能赋值一次。`abstract` 修饰类表示不能直接实例化，修饰方法表示只有声明没有方法体，必须由具体子类实现。`final` 与 `abstract` 在同一目标上通常语义冲突：一个表示不允许改变或扩展，一个表示需要后续补全。理解修饰符之间的组合限制，比单独背诵每个关键字更重要。

类体中的成员声明顺序不会改变成员是否存在，但会影响初始化顺序和阅读体验。通常把常量、字段、构造方法、公开方法、私有辅助方法按稳定顺序组织，能让读者从对象状态一路读到行为边界。语法不强制这种顺序，工程可维护性需要这种秩序。

## 数组与基础泛型使用

数组是固定长度、同类型元素的容器。数组对象创建后长度不可变，`length` 是数组字段，不是方法。数组元素会按类型获得默认值，因此引用类型数组创建后元素初始为 `null`。

```java
int[] numbers = new int[3];
numbers[0] = 10;
System.out.println(numbers.length);
```

数组访问会进行边界检查，越界会抛出 `ArrayIndexOutOfBoundsException`。数组适合长度固定、按下标访问频繁、结构简单的场景。若需要频繁增删、去重、按键查找或表达更丰富的集合语义，应使用集合框架。

数组具有协变特性，`String[]` 可以赋给 `Object[]`，但这也带来运行期风险。

```java
String[] strings = new String[1];
Object[] objects = strings;
// objects[0] = 1; // 运行期抛出 ArrayStoreException
```

泛型集合在编译期提供类型检查，通常比原始类型更安全。不要使用原始 `List`、`Map`，除非在处理非常旧的 API。泛型类型参数能让编译器阻止错误元素放入集合，也能减少读取时的强制转换。

```java
List<String> names = new ArrayList<>();
names.add("Alice");
String first = names.get(0);
```

基础阶段需要记住：泛型主要是编译期约束，运行期存在类型擦除。不能直接创建 `new List<String>[10]` 这样的泛型数组，也不能用 `instanceof List<String>` 判断具体类型参数。更深入的泛型边界会在集合与泛型主题中展开。

## 表达式求值与副作用

表达式不仅有结果，也可能有副作用。赋值、自增、自减、方法调用都可能改变状态。把多个副作用塞进同一行会让执行顺序难以阅读。

```java
int i = 0;
int value = i++ + ++i;
```

这类代码虽然有明确语法规则，但维护价值很低。更好的写法是把状态变化拆成几行，让每一步的意图清楚。基础语法允许简写，不代表简写就是好代码。

Java 会按规定顺序求值方法调用的目标对象和参数。参数从左到右求值，但不应依赖复杂副作用顺序表达业务逻辑。若某个方法调用依赖前一个方法调用改变状态，应拆开写，并给中间结果命名。

布尔表达式也要注意副作用。短路运算会跳过右侧表达式，如果右侧表达式包含状态修改，就可能出现条件不同导致副作用不同的情况。除非意图非常明确，否则条件表达式中应只放判断，不放修改。

## 字面量、转义与文本表达

字符字面量使用单引号，字符串字面量使用双引号。常见转义包括 `\n` 换行、`\t` 制表符、`\"` 双引号、`\\` 反斜杠。字符字面量只能表示一个 `char`，而不是任意用户感知字符。某些 Unicode 字符可能需要代理项对，不能放入单个 `char`。

```java
char lineBreak = '\n';
String path = "C:\\temp\\data.txt";
```

文本块使用三个双引号，适合多行文本。文本块会处理公共缩进，末尾换行也有规则。它能减少转义噪声，但不应把大段外部模板硬塞进源码。若文本需要由非开发人员维护，配置文件或模板文件通常更合适。

数字字面量还可以使用二进制、八进制和十六进制形式。八进制以前导 `0` 表示，容易误读，应谨慎使用。

```java
int mask = 0b0101;
int color = 0xFF00AA;
```

## 编译期常量与常量折叠

被 `static final` 修饰且初始化为编译期常量表达式的基本类型或字符串，常被称为编译期常量。编译器可能把它的值直接内联到使用方字节码中。修改公共常量值后，如果使用方没有重新编译，可能仍然使用旧值。

```java
public static final int DEFAULT_LIMIT = 100;
```

因此公共常量应表示真正稳定的值，例如数学常量、协议中不会改变的固定标记。可配置的默认值不应伪装成公共编译期常量，更适合通过配置、构造参数或方法返回。

常量折叠也会影响字符串拼接。只由编译期常量组成的表达式可能在编译期合并，而包含变量的方法调用会在运行期执行。理解这一点有助于解释某些 `==` 比较结果，但业务代码不应依赖这种细节判断字符串内容。

## main 方法与程序入口

标准 Java 程序常用 `main` 方法作为入口。它的典型签名是 `public static void main(String[] args)`。`public` 允许启动器访问，`static` 表示不需要先创建对象，`void` 表示入口方法不通过返回值把结果交给启动器，`String[] args` 保存命令行参数。

```java
public class Application {
    public static void main(String[] args) {
        System.out.println(Arrays.toString(args));
    }
}
```

理解入口方法有助于区分“类加载”“创建对象”和“执行实例方法”。`main` 只是普通静态方法的一种约定，不代表所有 Java 代码都必须从这里开始组织。库代码通常没有入口方法，而是提供类型和方法供其他程序调用。

命令行参数全部是字符串。若参数表达数字、布尔值或路径，需要在入口附近解析和校验。不要把未校验的原始参数一路传入深层逻辑，否则错误会在离输入源很远的位置暴露。入口层适合完成参数解析、默认值处理和错误提示，核心逻辑则应接收已经建模好的类型。

## 嵌套类与局部类基础

Java 支持在类内部声明嵌套类。静态嵌套类使用 `static` 修饰，不持有外部类实例引用；非静态内部类会关联一个外部类实例，可以访问外部实例成员。基础阶段最常见、也最推荐优先掌握的是静态嵌套类，例如建造者、结果类型或只服务于外部类的辅助类型。

```java
class Page {
    private final int number;

    static class Request {
        private final int pageSize;

        Request(int pageSize) {
            this.pageSize = pageSize;
        }
    }
}
```

如果嵌套类型不需要访问外部对象状态，应声明为静态嵌套类。非静态内部类隐式持有外部对象引用，这会影响对象生命周期和构造方式，也会让依赖关系更不明显。

方法内部还可以声明局部类，表达只在当前方法中使用的辅助类型。现代 Java 中，许多简单回调和小行为可以用 lambda 或匿名类表达，但当逻辑需要命名字段和多个方法时，局部类仍有可读性价值。无论使用哪种形式，都应避免把复杂业务逻辑塞进难以复用和测试的匿名结构里。

## 类型判断与模式匹配基础

`instanceof` 用于判断对象是否是某个类型。传统写法需要先判断再强制转换。较新的 Java 版本支持 `instanceof` 模式变量，可以减少重复转换。

```java
Object value = "java";
if (value instanceof String text) {
    System.out.println(text.length());
}
```

类型判断适合处理真实的类型边界，例如解析外部对象、实现通用容器或处理继承层级中的少数差异。若业务代码大量依赖 `instanceof` 分支，往往说明多态、接口或数据建模可以重新审视。基础语法允许运行期判断类型，但面向对象设计通常希望把差异行为放到类型自己的方法中。

强制类型转换失败会抛出 `ClassCastException`。为了避免运行期错误，转换前应有明确来源保证或使用 `instanceof` 判断。泛型集合、接口抽象和清晰的方法签名能减少显式强转出现的频率。

模式变量有作用域限制。只有在编译器能确定类型判断成立的路径上，模式变量才可用。这个规则与短路逻辑结合时尤其有用：`value instanceof String s && s.length() > 0` 中，右侧能使用 `s`，因为执行到右侧时左侧必然为 true；而在 `||` 的右侧直接使用同一个模式变量通常不成立，因为左侧为 false 时才会执行右侧。模式匹配减少了样板代码，但没有取消静态类型检查。

`instanceof` 对 `null` 总是返回 false，因此先写 `value instanceof String s` 不需要额外判断 `value != null`。这与强制转换不同，直接把 `null` 转换成某个引用类型是允许的，结果仍然是 `null`；真正抛出异常的是把非目标类型对象转换成不兼容类型，或随后对空引用调用实例成员。基础语法中很多运行期错误并不来自单个关键字，而来自多个表达式组合后的边界。

## 异常处理语法基础

异常处理是 Java 控制流的一部分。`try` 包围可能失败的代码，`catch` 捕获并处理特定异常，`finally` 表示无论是否发生异常都尽量执行的收尾逻辑，`throw` 用于主动抛出异常，`throws` 用于在方法声明中说明可能向调用方传播的受检异常。异常不是普通返回值，它会打断当前正常执行路径，沿调用栈寻找匹配的处理逻辑。

```java
try {
    int value = Integer.parseInt(text);
    System.out.println(value);
} catch (NumberFormatException e) {
    System.out.println("invalid number");
}
```

异常分为受检异常和非受检异常。受检异常通常继承自 `Exception` 但不继承 `RuntimeException`，编译器要求捕获或声明抛出。非受检异常包括 `RuntimeException` 及其子类，编译器不强制处理。这个划分不是“重要”和“不重要”的划分，而是 API 设计者对调用方是否必须显式面对该失败模式的选择。文件不存在、输入格式不符合约定、网络中断等可恢复或可预期失败常被设计为受检异常；空指针、越界、非法状态等程序错误常表现为非受检异常。

`catch` 分支按从上到下匹配，子类型异常应写在父类型异常前面。若先捕获 `Exception`，后面的具体异常分支就不可达，编译器会报错。多 catch 可以用 `|` 合并多个处理方式相同的异常类型，但这些异常类型之间不能存在父子关系。异常变量在 catch 块中可用，通常不应被忽略；即使当前层只能转换错误信息，也应保留原始异常作为原因，避免丢失定位线索。

`finally` 常用于释放资源或恢复状态，但它不是万能保障。进程直接终止、虚拟机崩溃等极端情况可能让 finally 没有机会执行。更常见的问题是在 `finally` 中写 `return` 或再次抛出异常，这会覆盖 `try` 或 `catch` 中原本要返回的值和异常。基础代码应避免在 finally 中改变控制流，让它专注于收尾动作。

`try-with-resources` 是更适合资源关闭的语法。实现 `AutoCloseable` 的资源可以放在 try 后的括号中，语句结束时自动关闭。多个资源按声明的逆序关闭。这个语法减少了手写 finally 的样板，也能正确处理关闭时的异常抑制关系。

```java
try (BufferedReader reader = Files.newBufferedReader(path)) {
    return reader.readLine();
}
```

`throw` 后面必须是 `Throwable` 的实例。主动抛出异常时，应选择能表达错误性质的异常类型，并写清楚消息。不要用异常替代普通条件分支，也不要吞掉异常后继续执行一个已经不可信的状态。异常语法的价值在于把异常路径从正常路径中分离出来，同时让失败能被上层有意识地处理。

## 枚举、记录与注解的基础语法位置

枚举用于声明一组有限且命名稳定的常量。枚举类型使用 `enum` 声明，枚举常量写在枚举体开头。与普通整数常量相比，枚举有独立类型，能让编译器阻止把无关值传入方法。枚举可以拥有字段、构造方法和方法，但枚举构造方法不能公开调用，枚举实例由语言运行时按声明创建。

```java
enum Direction {
    NORTH, EAST, SOUTH, WEST
}
```

枚举适合表达封闭集合，例如方向、状态、级别、策略名称等。若值集合需要由外部配置动态扩展，枚举就不是合适选择。枚举与 `switch` 搭配时，可读性通常比裸整数和字符串更好，因为每个分支面对的是明确的类型成员。需要注意的是，枚举常量名也是公开契约的一部分，若被序列化或持久化，重命名会影响外部数据兼容。

记录类使用 `record` 声明，适合表达以数据为中心的浅不可变载体。记录头中的组件会自动生成私有 final 字段、访问器、构造方法以及基于组件的 `equals`、`hashCode` 和 `toString`。记录不是语法糖意义上的普通类替换，它表达的是“这个类型主要由这些数据组件定义身份”。当类型需要复杂生命周期、可变状态或隐藏大量派生状态时，普通类更合适。

```java
record Point(int x, int y) {
}
```

注解使用 `@` 语法附着在类型、方法、字段、参数、局部变量等声明位置，具体可用位置由注解自身的元信息决定。注解本身不直接改变 Java 基础语句的执行顺序，它通常为编译器、工具或运行期框架提供元数据。基础阶段需要知道注解是声明的一部分，不是注释；注释会被编译器忽略，而注解可能参与编译检查、代码生成或运行期反射读取。

这些类型形式都属于 Java 语法系统的一部分，但它们服务的目标不同：类表达状态与行为的组合，接口表达能力契约，枚举表达有限命名实例，记录表达数据聚合，注解表达元数据。基础语法学习应先掌握它们在源码中的位置、声明方式和边界，再深入各自的设计模式和运行期机制。

## 注释与代码风格

Java 支持单行注释、多行注释和文档注释。注释应解释意图、约束和非显然原因，而不是复述代码字面行为。公开 API 的文档注释应说明参数含义、返回值、异常条件和边界行为。

命名是基础语法中最影响可读性的部分。类名通常使用大驼峰，方法和变量使用小驼峰，常量使用全大写下划线。布尔变量和方法应能自然表达真假，例如 `enabled`、`isEmpty`、`hasNext`。

代码块应保持缩进一致。复杂表达式应拆成中间变量，让条件和计算过程更容易阅读。语法允许的写法不一定是适合维护的写法，基础代码也应追求清晰。

文档注释使用 `/** ... */`，通常放在公开类型、方法、字段或构造方法之前。它可以使用 `@param`、`@return`、`@throws` 等标签描述契约。好的文档注释不重复方法名已经表达清楚的内容，而是补充调用前置条件、边界值、失败模式和线程安全等语法本身看不出来的信息。若一个私有方法需要大段注释才能说明，也可能说明方法名、参数或拆分方式还不够清晰。

代码风格不是语法规则，但它能降低语法误读。统一的大括号位置、缩进宽度、空行分组和导入排序，让读者把注意力放在逻辑上，而不是不断重新适应排版。基础语法越熟悉，越应该克制使用冷僻写法。能用直接控制流表达清楚的问题，不必故意压缩成一行；能用明确中间变量命名的计算，不必把所有表达式嵌套在方法调用参数里。

注释还要与代码同步。过期注释比没有注释更危险，因为它会给读者一个错误的确定感。修改条件、默认值、异常行为或边界语义时，应同时检查附近注释和文档注释。对基础语法而言，最可靠的说明仍然是清晰的类型、名字和结构；注释负责解释这些结构背后的原因。

## 常见误区与实践要点

不要把包装类型当成基本类型随意使用。包装类型可能为 `null`，比较时也可能因为对象身份和缓存机制造成误解。需要数值计算时优先基本类型，需要集合元素、泛型参数或可空语义时再使用包装类型。

不要依赖整数溢出的“碰巧结果”。计数、容量、时间和金额相关计算应选择足够范围的类型，并在必要时使用 `Math.addExact` 等方法显式检查溢出。

不要用 `==` 比较对象内容。除非明确要判断两个引用是否指向同一对象，否则应使用 `equals`，并注意左侧可能为 `null` 时的写法。

不要让方法参数承担过多含义。多个同类型参数连续出现时，调用处很难看出每个值的语义。可以考虑引入命名清晰的值对象，或拆分方法。

不要为了减少行数牺牲分支清晰度。三元表达式、链式调用和嵌套 lambda 都可能让基础逻辑变得难读。代码首先是给人读，其次才是给编译器执行。

不要混淆编译期类型和运行期对象。编译期类型决定你能直接调用哪些成员，运行期类型决定重写方法的实际执行版本。很多“明明对象是某个具体类，为什么不能调用某方法”的疑问，都来自变量声明类型不包含该方法。解决方式不是随意强转，而是重新审视方法签名是否应该暴露更准确的类型或抽象。

不要把 `null` 当成所有缺省情况的统一答案。`null` 只能表达“没有引用”，无法区分未传入、未找到、尚未初始化、计算失败、结果为空等不同语义。基础语法允许引用为空，但 API 设计应尽量让空值语义明确。集合返回值通常可以用空集合表达“没有元素”；必须允许为空的参数和返回值，应在文档或类型约定中说清楚。

不要忽略字符和字符串的边界。`char` 表示 UTF-16 代码单元，不等同于用户看到的一个完整字符；`String.length()` 返回代码单元数量，不一定等同于用户感知字符数量。处理普通英文标识和简单协议字段时这个差异常常不可见，但涉及多语言文本、符号和组合字符时就会暴露。基础语法中的 `char` 很简单，真实文本处理并不总是简单。

不要用导入和包结构掩盖依赖关系。`import` 只是简写类型名，代码依赖并不会因为使用通配导入就变少。通配导入还可能让同名类型来源不明显。对于核心类型、容易冲突的类型和公开 API 中出现的类型，清晰导入比少写几行更重要。

不要在构造方法中让对象过早暴露。构造期间对象还未完全初始化，如果把 `this` 传给外部代码、启动回调或调用可被子类重写的方法，外部逻辑可能看到半初始化状态。基础语法允许在构造方法中写很多语句，但构造方法最稳妥的职责是校验参数、建立不变量、完成必要字段赋值。

不要把异常处理写成空 catch。空 catch 会让失败静默消失，后续代码可能在错误状态下继续运行。若异常确实可以忽略，也应通过注释或明确条件说明原因。更常见的做法是转换为更贴近当前抽象层的异常，或返回能表达失败的结果类型。

## 小结

Java 基础语法建立在静态类型、明确作用域、可见性控制和结构化流程之上。掌握基本类型与引用类型、运算符规则、流程控制、方法调用和包组织，能帮助读者准确判断代码在编译期和运行期分别发生什么。基础语法不是孤立知识点，而是后续理解对象设计、集合行为、异常传播、并发可见性和 API 边界的共同前提。

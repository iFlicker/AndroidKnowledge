# 3.1.1 面向对象

## 定位

面向对象是 Java 组织程序的核心方式。Java 程序中的业务概念、状态、行为、访问边界和扩展点通常都落在类、对象、接口、继承关系和组合关系上。学习面向对象不是只背“封装、继承、多态”三个词，而是理解 Java 如何把数据与行为放进可命名、可复用、可约束的类型中，并通过类型系统降低大型程序的理解和修改成本。

面向对象解决的主要问题有三个。第一是建模：用类描述一类对象共同拥有的字段、方法和不变量。第二是隔离变化：通过访问控制、接口和抽象类把稳定的契约与可变化的实现分开。第三是替换与扩展：调用方依赖父类或接口时，可以在不改调用方代码的前提下换入不同实现。

Java 的面向对象并不等于“所有代码都必须继承”。继承只是类型复用和多态的一种机制，很多场景下组合、接口和委托更清晰。判断设计是否合理，应看类型边界是否表达了真实职责，是否减少了调用方需要知道的细节，而不是看类的数量或继承层级是否复杂。

## 类与对象

类是对象的模板，定义字段、构造方法、实例方法、静态成员和初始化逻辑。对象是类在运行时创建出的实例，拥有独立的实例字段。引用变量保存的是对象引用，而不是对象本体，因此同一个对象可以被多个引用变量指向。

```java
class Account {
    private final String id;
    private long balance;

    Account(String id, long openingBalance) {
        if (id == null || id.isBlank()) {
            throw new IllegalArgumentException("id must not be blank");
        }
        if (openingBalance < 0) {
            throw new IllegalArgumentException("balance must not be negative");
        }
        this.id = id;
        this.balance = openingBalance;
    }

    void deposit(long amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("amount must be positive");
        }
        balance += amount;
    }

    long balance() {
        return balance;
    }
}
```

这个类不只是“字段加方法”的容器。构造方法保证对象创建后拥有合法状态，`private` 字段避免外部绕过规则直接修改余额，方法负责维护状态变化的不变量。面向对象设计首先要关心对象是否始终处于有效状态，而不是先关心 getter、setter 是否齐全。

字段分为实例字段和静态字段。实例字段属于某个对象，静态字段属于类本身，所有实例共享。方法也有实例方法和静态方法之分：实例方法可以访问当前对象的 `this`，静态方法没有当前对象。把状态放入静态字段会让数据变成全局共享，必须谨慎使用，尤其要考虑初始化顺序、并发访问和测试隔离。

理解对象时要同时区分“变量”“引用”“对象”三个层次。局部变量、字段或参数只是保存引用的槽位，赋值改变的是槽位中保存的引用；对象本身存在于运行时内存中，只有通过引用才能被访问。把一个引用赋给另一个变量不会复制对象，只会让两个变量指向同一实例。因此，当对象是可变对象时，多个引用会共享同一份状态，任何一个入口的修改都会被其他入口观察到。面向对象设计中的很多问题，本质上都来自没有意识到共享可变状态的范围。

类还承担“命名概念”的作用。一个好的类名不只是给字段集合起名字，而是告诉读者这个类型代表什么概念、可以保持什么状态、允许发生哪些行为。例如 `Money`、`Range`、`OrderLine`、`UserName` 这类类型往往比散落的基本类型更能表达含义。把一组值提升为类的价值不在于增加文件数量，而在于把校验、比较、格式化、边界条件和不变量收拢到一个明确位置。相反，如果类名很宽泛，字段彼此没有内聚关系，方法只是对外部数据的薄包装，那么这个类就没有真正帮助程序表达问题。

对象也不是越“真实世界映射”越好。面向对象建模需要服务于程序要解决的问题，而不是机械模拟现实。现实中的一个概念可能在不同程序中被拆成多个类型，也可能被简化成一个值对象。例如同样是“地址”，在物流系统中可能包含校验、分区、可配送范围和格式化规则；在简单联系人程序中可能只是展示文本。判断类是否合理，应看它是否承载了当前程序中稳定的规则，而不是看它是否完整描述了现实事物。

## 封装

封装的直接表现是访问控制，但它的目标是隐藏实现细节、保护对象不变量、缩小调用方需要依赖的知识范围。Java 提供四类访问级别：`private` 仅当前类可见；包访问级别没有修饰符，仅同包可见；`protected` 对同包和子类可见；`public` 对所有可访问该类型的位置可见。

好的封装不是把所有字段都机械地配一组 public getter/setter。setter 往往会暴露对象内部状态，使调用方可以在任意时刻把对象改到不合理状态。更好的做法是用表达业务意图的方法描述状态变化，例如 `deposit`、`withdraw`、`rename`、`activate`，并在方法内部统一校验规则。

封装还包括返回值的保护。若对象内部维护可变集合，直接返回集合引用会让外部绕过对象方法修改内部状态。

```java
class Team {
    private final List<String> members = new ArrayList<>();

    void addMember(String name) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("name must not be blank");
        }
        members.add(name);
    }

    List<String> members() {
        return List.copyOf(members);
    }
}
```

这里 `List.copyOf` 返回不可变快照，调用方可以读取成员列表，但不能通过返回值破坏 `Team` 的内部约束。封装的边界越稳定，类的实现越容易重构。

封装的核心对象是不变量。不变量是对象在对外可见的稳定时刻必须始终满足的条件，例如金额不能为负、起始时间不能晚于结束时间、标识符不能为空、集合中不能出现重复成员。构造方法负责建立初始不变量，实例方法负责在每次状态变化后继续维持不变量。若一个对象只能在“调用者按照正确顺序调用多个 setter 后”才合法，它就把不变量维护责任转移给了外部，封装质量很低。

暴露字段的 getter 并不一定破坏封装，关键看返回的是什么以及调用方能否借此绕过规则。返回不可变值、计算结果、只读视图或防御性副本，通常仍然可以保持封装；返回内部可变对象引用，则会让外部拥有修改内部状态的后门。对于数组、集合、日期时间对象、自定义可变类型等成员，尤其要明确是返回快照、不可变视图，还是有意共享同一个对象。如果选择共享，就应把共享语义写进类型设计，而不是让它成为偶然副作用。

封装还包含异常边界。对象方法发现输入不满足前置条件时，应尽早失败，并给出能定位问题的异常信息。沉默修正错误输入、返回含糊的特殊值、让对象进入半合法状态，都会让错误在更远的位置爆发。Java 中常见做法是对调用者错误抛出 `IllegalArgumentException` 或 `IllegalStateException`，对不可恢复的编程错误保持快速失败。异常类型本身也是 API 的一部分，公开方法不应随意泄漏内部实现层的异常细节。

良好的封装并不等于类对外什么都不暴露。对象必须提供足够清晰的行为入口，否则调用方只能绕开对象自己拼装逻辑。真正的问题不是 public 方法数量，而是 public 方法是否围绕同一职责组织，是否让调用方用更少的知识完成正确操作。一个类如果把规则藏得过深，导致每个调用方都要猜测内部状态，也会降低可维护性。封装需要在“隐藏细节”和“表达能力”之间取得平衡。

## 继承

继承使用 `extends` 表达“子类是父类的一种”。子类继承父类的可访问字段和方法，并可以新增成员或重写方法。继承适合表达稳定的 is-a 关系，例如一个更具体的领域类型确实能在所有父类使用位置被替代。

继承最大的价值是多态，最大的风险是耦合。子类不仅依赖父类公开 API，还会依赖父类的初始化顺序、可重写方法调用时机、受保护成员和隐含约定。父类修改内部实现时，子类可能受到影响。因此可继承类的设计要求很高：父类应清晰说明哪些方法允许重写、重写后必须满足什么条件、父类构造和回调顺序如何。

Java 中所有类都直接或间接继承 `Object`。`Object` 提供 `toString`、`equals`、`hashCode` 等基础方法。重写 `equals` 时必须同时重写 `hashCode`，否则对象放入哈希集合或哈希映射时会出现查找不一致。`equals` 还应满足自反、对称、传递、一致和非空比较为 false 的契约。

继承不适合只为“复用几行代码”而使用。若两个类只是共享某段实现，但概念上不存在稳定的父子类型关系，组合通常更好。组合把可复用行为放入独立对象，再由使用方持有它。这样依赖的是明确协作关系，而不是继承层级中的隐式约束。

判断继承关系是否成立，可以从替换角度思考：任何接受父类的位置，是否都可以放心接收子类，并且不需要知道它是子类。如果答案是否定的，继承就很可疑。典型问题是父类承诺了一组行为，而子类为了表达自身特殊性不得不禁用某些方法、抛出“不支持”异常或改变返回含义。此时子类虽然在语法上继承了父类，语义上却不能承担父类契约，调用方会在运行期才发现替换失败。

继承还会形成脆弱基类问题。父类为了优化或重构而调整内部方法调用关系，可能意外改变子类行为；子类重写某个方法后，也可能破坏父类原本依赖的内部假设。尤其当父类在构造方法、公共方法或模板方法中调用可重写方法时，父类实际上把执行权交给了未知子类。可继承类若没有明确设计和文档约束，就容易让父类与子类互相依赖对方的内部细节。

Java 提供 `final` 来限制继承或重写。`final class` 表示类不能被继承，`final method` 表示方法不能被重写。使用 `final` 不是保守，而是在表达设计边界：这个类型没有承诺可扩展性，或者这个方法承载了必须保持的流程和不变量。对外发布的类如果没有经过继承设计，默认不鼓励被继承；需要扩展时，可以优先暴露接口、组合点或受控的模板方法。

父类字段通常应保持 `private`，由受控方法管理。把字段设为 `protected` 看似方便子类复用，实际会让子类直接依赖父类内部表示，父类之后几乎无法更换字段结构。更稳妥的做法是让父类提供少量语义明确的 `protected` 方法作为扩展点，并保持内部状态封装。继承中的“少暴露”比普通封装更重要，因为子类与父类生命周期和行为调用顺序紧密交织。

## 多态

多态指同一调用在运行时根据实际对象类型执行不同实现。Java 的实例方法默认具有动态分派：变量的声明类型决定编译期能调用哪些方法，对象的实际类型决定运行期执行哪个重写实现。

```java
interface Formatter {
    String format(Object value);
}

class PlainFormatter implements Formatter {
    public String format(Object value) {
        return String.valueOf(value);
    }
}

class QuotedFormatter implements Formatter {
    public String format(Object value) {
        return "\"" + value + "\"";
    }
}

class Printer {
    void print(Object value, Formatter formatter) {
        System.out.println(formatter.format(value));
    }
}
```

`Printer` 不需要知道格式化的具体算法，只依赖 `Formatter` 契约。调用方可以传入不同实现改变行为。这就是多态带来的开放性：稳定部分依赖抽象，变化部分封装在实现类里。

需要区分重写和重载。重写发生在继承关系中，方法签名相同，运行期根据实际对象类型选择实现。重载发生在同一个类或继承层级中，方法名相同但参数列表不同，编译期根据静态类型和参数类型选择目标方法。返回值不同不能单独构成重载。

字段没有多态。若父类和子类声明同名字段，访问哪个字段由引用的静态类型决定，这种隐藏通常会造成困惑，应避免。静态方法也不参与实例方法式的动态分派，子类中同签名静态方法只是隐藏父类静态方法。

动态分派让调用方可以稳定地面向父类或接口编程，但它也意味着阅读代码时不能只看变量声明类型。一个接口方法的具体行为，取决于运行期传入的实现对象。好的多态设计会让不同实现共享同一语义，只在算法、策略、存储方式或表现形式上变化；差的多态设计会让同一个方法名在不同实现中表达完全不同的事情，调用方虽然依赖了抽象，实际仍然需要了解每个具体类的特殊规则。

重载与多态经常混在一起造成误解。重载选择发生在编译期，依据的是表达式的静态类型，而不是对象的运行期类型。若同时存在 `print(Object)` 和 `print(String)`，传入一个静态类型为 `Object`、实际对象为 `String` 的变量时，编译器会选择 `print(Object)`。之后如果该方法内部调用可重写实例方法，才会进入运行期动态分派。理解这条边界，有助于解释很多“为什么没有调用我以为的方法”的问题。

多态并不要求每个变化点都使用继承层级。Java 的函数式接口、枚举策略、组合对象、方法引用都可以表达可替换行为。若变化只是一个小算法，定义一个庞大的抽象类层级反而会放大复杂度。面向对象中的多态本质是“调用方依赖稳定契约，具体行为可替换”，实现方式可以很轻，也可以很重，应根据变化范围选择。

多态还要求调用方不要反复向下转型。若代码中频繁出现 `instanceof` 判断具体子类，然后执行不同分支，说明抽象可能没有把共同操作表达出来。适量的类型判断有其用途，例如边界层解析不同输入、实现 `equals`、处理异构数据结构；但在核心业务流程中大量按类型分支，通常意味着应该把差异行为移动到各实现类中，让动态分派承担选择责任。

## 抽象类

抽象类用 `abstract` 修饰，可以包含抽象方法、具体方法、字段和构造方法。抽象类不能直接实例化，子类必须实现未实现的抽象方法，除非子类仍然是抽象类。

抽象类适合抽取一组相关类的共同状态和基础流程。当多个实现不仅共享契约，还共享字段、构造参数、校验逻辑或模板流程时，抽象类比纯接口更直接。

```java
abstract class FileExporter {
    public final void export(String target) {
        validate(target);
        write(target);
    }

    private void validate(String target) {
        if (target == null || target.isBlank()) {
            throw new IllegalArgumentException("target must not be blank");
        }
    }

    protected abstract void write(String target);
}
```

这里 `export` 是稳定流程，`write` 是变化点。`export` 标记为 `final` 可以防止子类破坏流程顺序。抽象类的设计要避免把太多内部细节暴露给子类，`protected` 不是“安全的 public”，它仍然扩大了依赖面。

## 接口

接口描述一组行为契约。类通过 `implements` 实现接口，一个类可以实现多个接口。接口字段默认是 `public static final` 常量，接口方法可以是抽象方法、默认方法、静态方法和私有辅助方法。默认方法用于在不破坏已有实现类的前提下为接口增加可复用行为，但它不应承载过多状态相关逻辑。

接口适合定义能力边界，例如比较、格式化、转换、存储、通知等。调用方依赖接口时，可以隔离具体实现，便于测试和替换。接口越小，职责越聚焦，越容易组合。过大的接口会迫使实现类依赖自己并不需要的方法，违反接口隔离原则。

接口和抽象类的选择可以从几个问题判断：是否需要共享实例状态；是否要求实现类只能选择一个基类；是否更关注能力契约而不是类族关系；未来是否需要多个能力组合。若只是声明能力，优先接口；若需要共同状态和基础流程，再考虑抽象类。

接口的粒度非常重要。过细的接口会让系统充满只被一个类实现、只被一个调用方使用的抽象，阅读时需要不断跳转；过粗的接口又会迫使实现类提供无关方法，调用方也无法只依赖自己真正需要的能力。较好的接口通常来自已经出现的稳定协作关系：多个实现确实共享同一调用方式，或者调用方确实需要隔离一个可替换的外部能力。接口不是为了展示“面向对象”，而是为了让依赖边界更清楚。

默认方法让接口具备一定演进能力，但也要控制使用范围。它适合提供基于其他抽象方法推导出来的便利方法，或者为已有接口添加不破坏兼容性的行为。若默认方法开始依赖复杂状态、隐藏多步流程或承担核心业务规则，就会让接口从契约变成半个基类，而接口又没有实例字段和构造过程来支撑这些规则，最终容易出现职责混乱。

接口也可以作为类型语言。`Comparable<T>` 表示对象之间存在自然顺序，`AutoCloseable` 表示对象持有需要关闭的资源，`Iterable<T>` 表示对象可以被迭代。实现这些标准接口时，类会自动融入 Java 语言和库的生态。但实现标准接口必须严格遵守其契约，例如比较结果要保持传递性和一致性，关闭方法要能处理重复或异常边界。标准接口带来便利，也会把类型暴露在更广泛的使用方式下。

当接口只用于测试替换时，也要判断是否值得。对于复杂外部依赖、耗时操作、随机性、时钟、网络或文件系统访问，接口可以让测试更稳定；对于普通纯计算类，强行抽接口可能只是增加间接层。测试友好性应该来自清晰依赖和可控边界，而不是每个实现类都配一个空泛接口。

## 构造、初始化与不可变对象

对象创建过程包括分配内存、字段默认初始化、执行父类初始化、执行当前类字段初始化和构造方法体。构造方法不能被继承，但子类构造方法必须直接或间接调用父类构造方法。若父类没有无参构造方法，子类必须显式使用 `super(...)`。

构造期间应避免调用可被子类重写的方法。因为父类构造方法执行时，子类字段可能还没有完成初始化，动态分派到子类方法会观察到半初始化对象，容易产生难以定位的问题。

不可变对象是构造后状态不再改变的对象。常见做法是类标记为 `final` 或谨慎控制继承，字段使用 `private final`，构造时完成所有校验，不暴露可变内部对象引用。不可变对象天然更容易推理，也更适合在并发和缓存场景中复用。

构造方法的职责是把对象从“不存在”带到“可用且合法”的状态。构造方法不宜做过多与对象合法性无关的重操作，例如启动后台线程、访问远程资源或把 `this` 注册到全局容器。原因是构造期间对象还没有完全交给调用方，异常处理和生命周期管理都比较脆弱。如果确实需要复杂创建过程，可以考虑静态工厂、构建器或单独的初始化服务，把参数收集、校验、资源准备和对象创建分成更清晰的步骤。

`final` 字段有助于表达对象状态在构造后不再重新绑定，但它不自动让引用指向的对象变得不可变。`private final List<String> names` 只能保证 `names` 字段不再指向另一个列表，不能阻止列表内容变化。构造不可变对象时，必须同时处理内部可变成员：传入集合要拷贝，返回集合要提供不可变视图或快照，成员对象自身也应是不可变的，或者至少不能被外部继续持有并修改。

不可变对象也不是所有场景的默认答案。大量频繁变化的状态如果每次都创建新对象，可能带来额外分配和复制成本；需要维护身份、生命周期或增量更新的对象，也可能更适合可变模型。实践中常见做法是让值对象不可变，让聚合根、缓存、连接、构建器等具有明确生命周期的对象可变。关键是可变性必须被边界控制，而不是让任意调用方在任意时刻修改任意字段。

静态初始化也需要谨慎。静态字段在类初始化阶段赋值，初始化顺序受类加载和首次主动使用影响。若静态初始化逻辑依赖外部环境、复杂对象图或可变全局状态，问题会很难复现。面向对象设计中，应尽量让对象依赖通过构造参数显式传入，而不是隐藏在静态单例或静态工具类内部。静态方法适合无状态的纯工具、命名构造入口或简单工厂，不适合承载大量可变化策略。

## Object 契约与对象身份

Java 对象既有身份，也可能有值语义。对象身份指两个引用是否指向同一个实例，使用 `==` 判断。值语义指两个对象虽然不是同一实例，但在业务意义上表示相同值，通常使用 `equals` 判断。没有重写 `equals` 的类会继承 `Object.equals`，其默认行为等同于引用相等。

值对象应认真设计 `equals` 和 `hashCode`。例如金额、坐标、配置键、范围、标识符包装类等类型，调用方通常关心内容是否相等，而不是是否同一个实例。重写时要选择稳定字段，避免把可变字段作为哈希依据。对象放入 `HashSet` 或作为 `HashMap` 键后，如果参与哈希计算的字段发生变化，集合可能再也找不到这个对象。

```java
final class Point {
    private final int x;
    private final int y;

    Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public boolean equals(Object other) {
        if (this == other) {
            return true;
        }
        if (!(other instanceof Point point)) {
            return false;
        }
        return x == point.x && y == point.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }
}
```

`toString` 不参与对象相等契约，但它对调试和日志很重要。一个好的 `toString` 应包含能识别对象状态的关键信息，同时避免输出敏感字段。`clone` 和 `finalize` 虽然来自 `Object`，但现代 Java 代码很少主动依赖它们。对象复制更推荐使用拷贝构造方法、静态工厂或明确的复制方法；资源释放应使用显式关闭和 try-with-resources，而不是依赖终结机制。

设计 `equals` 时，最困难的部分常常不是写代码，而是确定相等语义。实体对象通常有身份语义，例如数据库中的一条记录、一个会话、一个正在运行的任务；值对象通常有内容语义，例如坐标、金额、区间、名称包装类型。实体在生命周期中可能发生状态变化，但仍被视为同一个对象；值对象只要内容相同就可替换。若把实体设计成基于全部可变字段相等，集合行为和缓存行为都会变得危险。若把值对象保留默认引用相等，调用方又无法自然表达内容一致。

继承会让 `equals` 更复杂。父类和子类之间是否允许相等，直接影响对称性和传递性。若父类 `Point` 只比较坐标，子类 `ColoredPoint` 又比较颜色，那么 `point.equals(coloredPoint)` 和 `coloredPoint.equals(point)` 很容易得到不同结果。对于有值语义的类，常见做法是避免可继承，或者在相等判断中严格要求运行时类型一致。可继承层级中的相等语义需要非常谨慎，不能只靠 IDE 自动生成后就认为正确。

`hashCode` 的稳定性与集合使用密切相关。对象作为哈希键时，参与哈希计算的字段在键存在期间不应变化。若业务上确实需要修改对象状态，应避免把该对象本身作为哈希键，可以使用不可变标识符作为键，或者在修改前移除、修改后重新放入。这个约束不是集合实现的细节，而是对象相等契约与哈希结构共同工作的基础。

`toString` 应面向诊断，而不是面向序列化。它的输出格式通常不应被其他程序逻辑解析，因为未来为了调试可读性可能调整格式。需要稳定交换格式时，应使用明确的序列化机制或专门方法。把 `toString` 当成业务协议，会把一个本该自由演进的调试方法变成隐式公共契约。

## 组合、委托与依赖方向

组合表示一个对象持有另一个对象，并把部分工作委托给它。组合的依赖关系更显式：类的构造方法、字段和方法签名能直接说明它需要哪些协作者。继承把父类实现带入子类，组合则把协作者作为独立对象保留边界。

```java
class PasswordService {
    private final PasswordHasher hasher;

    PasswordService(PasswordHasher hasher) {
        this.hasher = Objects.requireNonNull(hasher);
    }

    boolean matches(String raw, String encoded) {
        return hasher.hash(raw).equals(encoded);
    }
}
```

这里 `PasswordService` 不继承某个哈希基类，而是依赖 `PasswordHasher`。这种设计的好处是替换算法不需要改服务类，也不要求不同算法共享父类实现。组合还让测试更容易，因为调用方可以传入简单的测试实现。

依赖方向应指向稳定抽象，而不是易变细节。高层策略、业务流程或公共 API 不应被具体实现反向牵制。接口不是唯一手段，构造参数中的具体值对象、标准库类型、函数式接口都可以成为稳定依赖。关键在于调用方知道的细节是否足够少，变化发生时影响范围是否足够小。

## 可见性与包边界

Java 的包不仅是命名空间，也能形成轻量封装边界。包访问级别适合把一组协作类放在同一包内，只对外暴露少量 public 类型。这样可以让内部类之间协作不必全部公开，同时减少外部依赖内部实现的机会。

public 类型越多，长期维护成本越高。公开类名、方法签名、异常类型、返回集合是否可变等细节都会成为调用方可能依赖的契约。设计 API 时应优先暴露稳定抽象，隐藏构造细节和可变实现。若一个类只服务于当前包内的另一个类，就不应为了“以后可能用到”而设为 public。

`protected` 经常被误用。它不是只对子类可见，还对同包类可见；同时它让子类依赖父类内部扩展点。除非明确设计为可继承 API，否则更推荐使用 `private` 或包访问级别。可继承 API 需要文档说明调用顺序、重写限制和线程安全预期。

包边界应服务于概念内聚。把所有类按技术后缀分成 `model`、`service`、`util` 并不一定错误，但如果一个功能的核心类型被拆散到多个宽泛包中，包访问级别就很难发挥封装作用。更好的组织方式通常是让一组密切协作、共同维护某个概念不变量的类型位于同一包内，对外只公开入口类型、接口或少量值对象。这样内部辅助类可以保持包可见，外部代码也不会被鼓励直接依赖细节。

嵌套类也是表达边界的工具。若一个类型只服务于外部类的实现，且与外部类概念强绑定，可以考虑使用私有静态嵌套类。静态嵌套类不持有外部类实例引用，边界更清晰；非静态内部类会隐式持有外部对象，适合需要访问外部对象状态的场景，但也更容易造成生命周期和可见性上的隐含耦合。选择嵌套类不是为了减少文件数量，而是为了表达“这个类型属于哪个上下文”。

公开 API 还包括构造方法。一个 public 类如果暴露 public 构造方法，调用方就能直接创建它，并依赖构造参数含义。若创建过程需要命名、缓存、校验分支或返回子类型，静态工厂方法有时更合适。构造方法必须与类名同名，无法表达创建意图；静态工厂可以命名为 `of`、`from`、`parse`、`empty`、`copyOf` 等，让调用方更清楚创建语义。不过静态工厂也可能隐藏实现，使用时要保持文档清楚。

访问控制不能替代模块和团队边界。`public` 只是语言层面的可见性，实际项目中还会受到构建模块、发布包、版本兼容和调用约定影响。即使一个类在语言上可访问，也不代表它应该被外部业务依赖。文档、包命名、模块导出和测试覆盖共同构成 API 边界。面向对象设计需要同时考虑语言机制和工程组织，而不是只看修饰符。

## 泛型与面向对象类型关系

Java 泛型让类和接口可以在保持类型安全的同时复用结构。`List<String>` 和 `List<Integer>` 使用同一套集合逻辑，但编译器会限制元素类型，减少运行期强制转换错误。泛型并不是独立于面向对象之外的语法，它直接影响抽象边界、接口设计和多态使用方式。一个设计良好的泛型类型，应让类型参数表达真实变化点，而不是把所有内容都退化成 `Object`。

泛型类型参数通常代表“这个类型操作的元素是什么”“这个策略输入输出什么”“这个容器持有什么”。例如 `Repository<T>` 表示仓库围绕某类对象工作，`Converter<S, T>` 表示从源类型转换为目标类型。类型参数名不必长，但语义要清楚。若一个类有多个类型参数，却很难解释每个参数的职责，说明抽象可能过宽，或者应该拆分成更小的类型。

泛型中的子类型关系容易误解。`String` 是 `Object` 的子类，但 `List<String>` 不是 `List<Object>` 的子类。否则如果把 `List<String>` 当成 `List<Object>`，就可以向其中加入任意 `Object`，破坏原列表的类型安全。Java 使用通配符表达受控的协变和逆变：读取一组某种上界类型时，可以使用 `? extends T`；向结构中写入某种下界类型时，可以使用 `? super T`。常见经验是“生产者使用 extends，消费者使用 super”，本质是根据对象在方法中扮演的读写角色决定边界。

泛型与继承结合时，应避免制造难以理解的层级。例如 `BaseService<T>`、`AbstractManager<T, ID, R>` 这类抽象如果只是为了复用少量样板代码，可能会让调用者面对复杂的类型推导和错误信息。泛型抽象应优先服务于清晰契约：输入输出类型是否更安全，调用方是否少写强制转换，错误是否能在编译期暴露。如果泛型让 API 更难读、更难定位问题，就需要重新评估抽象收益。

类型擦除是 Java 泛型的重要边界。运行期通常无法直接知道 `List<String>` 的具体类型参数，不能用 `new T()` 创建泛型实例，也不能对普通泛型参数做完整运行期类型判断。需要运行期类型信息时，可以显式传入 `Class<T>`、类型描述对象或工厂。面向对象设计中不要假设泛型能保存所有运行期元数据，它主要是编译期类型约束工具。

## 对象协作与职责分配

面向对象程序不是孤立类的集合，而是一组对象通过方法调用完成协作。设计对象时，要考虑某个职责应该放在哪个类型中。一个常用判断是“谁拥有完成这件事所需的数据和规则”。若对象自己拥有状态和不变量，那么相关行为通常应靠近该对象；若行为需要协调多个对象，则可以放在专门的服务或协调者中。职责放错位置，会导致对象贫血、服务臃肿或循环依赖。

贫血模型指类型只有数据，没有行为，所有规则都堆在外部服务里。它并非永远错误：数据传输对象、配置载体、序列化结构可以很薄；但如果核心领域对象也只剩 getter/setter，规则散落各处，维护成本会随功能增长而上升。充血对象也不是把所有逻辑都塞进实体里。对象应承担与自身状态强相关的行为，而跨对象流程、事务边界、权限判断、外部系统交互等职责，往往更适合放在应用服务或协调对象中。

方法设计应尽量表达意图，而不是暴露步骤。`account.withdraw(amount)` 比 `account.setBalance(account.getBalance() - amount)` 更好，因为前者把校验、状态变化和异常处理封装成一个语义操作。调用方不需要知道余额字段叫什么，也不需要重复判断金额是否有效。对象方法的粒度应与业务动作或概念动作相匹配：太细会泄漏内部步骤，太粗会让方法承担过多不相关职责。

对象协作还要避免过长调用链。如果调用方经常写 `a.getB().getC().doSomething()`，说明它了解了过多内部结构。这样的代码一旦中间结构变化，调用方会大量修改。可以考虑让 `a` 提供更直接的行为，或者把需要协作的对象作为依赖显式传入。并不是所有链式调用都坏，流式 API 和不可变构建器可以很清晰；问题在于调用方是否穿透了对象边界去操作内部成员。

循环依赖是职责边界不清的信号。两个对象如果互相持有、互相调用，很容易产生初始化困难、测试困难和状态同步问题。解决方式可能是提取第三个协调者、反转依赖方向、引入事件或回调接口，也可能只是重新划分类的职责。循环依赖并非绝对禁止，但它需要明确理由，不能成为对象随意互相访问的结果。

## 可变性、别名与并发视角

即使不讨论复杂并发，面向对象也必须面对可变状态。对象状态一旦可变，就要回答三个问题：谁可以修改，什么时候可以修改，修改后谁能观察到。引用别名会让这些问题变得复杂，因为多个变量、集合或对象字段可能指向同一可变实例。封装的许多规则，例如私有字段、防御性拷贝、不可变返回值，都是为了控制别名带来的影响范围。

可变对象需要清楚的生命周期。有些对象在构建阶段可变，构建完成后不可变，例如构建器和最终产品；有些对象在整个生命周期中可变，例如缓存、计数器、会话状态；有些对象逻辑上可变但对外提供快照，例如历史记录、统计结果。把这些生命周期混在一起，会导致调用方不知道什么时候读取才可靠，也不知道能否长期保存某个引用。

并发会放大封装问题。一个对象如果会被多个线程访问，就必须定义线程安全策略：完全不可变、外部同步、内部同步、线程封闭，或只允许在特定阶段发布。面向对象层面的设计应尽量让线程安全策略可见，而不是让调用方猜测。不可变对象和局部对象天然更容易共享；可变对象若要共享，必须保护复合操作的不变量，而不仅仅是给单个 getter/setter 加锁。

对象发布也很重要。构造期间不要让 `this` 逃逸到其他线程、全局集合或回调中，因为其他代码可能观察到尚未完成初始化的状态。构造完成后再把对象交给外部，能让不变量建立过程更清楚。对于复杂对象图，最好通过工厂或组装层统一创建，再以稳定依赖形式传给使用者，避免对象在半成品状态下互相暴露。

可变性并不是坏事，未受控制的可变性才是坏事。面向对象的目标不是消灭状态，而是把状态放在能维护规则的边界内。只要修改入口少、语义清楚、不变量集中、共享范围明确，可变对象同样可以设计得可靠。

## SOLID 原则的 Java 化理解

单一职责原则强调一个类应只有一个主要变化原因。它不是要求类越小越好，而是要求职责聚合合理。一个对象可以有多个方法，但这些方法应围绕同一个概念的不变量和行为展开。

开闭原则强调对扩展开放、对修改关闭。Java 中常见实现方式是接口、多态、组合和注册表。但开闭原则不是拒绝修改旧代码。若变化尚未稳定，直接修改可能比提前抽象更清晰；当同类变化重复出现时，再抽取扩展点更稳妥。

里氏替换原则要求子类能替代父类而不破坏调用方预期。违反它的常见表现是子类重写方法后拒绝父类允许的输入、返回更弱的结果、抛出调用方不预期的异常，或改变父类承诺的状态变化。

接口隔离原则要求调用方不依赖自己不用的方法。Java 中可以通过多个小接口表达不同能力，例如 `Readable`、`Closeable`、`Comparable`。小接口更容易组合，也更容易为测试构造替身。

依赖倒置原则强调高层模块不依赖低层细节，二者依赖抽象。这里的抽象可以是接口，也可以是稳定的值对象或函数。重点是依赖方向服务于变化隔离，而不是机械地给每个类生成接口。

## 常见误区与实践要点

不要把面向对象理解成“名词建类”。类应该承担清晰职责，而不是把每个名词都变成贫血数据容器。若一个类只有字段和 getter/setter，所有规则都散落在外部服务中，这个类没有真正封装行为。

不要滥用继承。继承层级越深，理解一个方法的实际行为越需要在父子类之间跳转。优先使用组合表达“拥有某能力”或“委托某组件完成工作”，只在存在稳定替换关系时使用继承。

不要让接口过早抽象。接口应来自明确的变化点、测试替换需求或模块边界，而不是为了“面向接口编程”而给每个类配一个接口。无意义的接口会增加文件数量和跳转成本。

重写方法时要维护父类契约。子类不能比父类要求更强的前置条件，也不能给出更弱的结果承诺，否则调用方以父类视角使用子类时会被破坏。

访问修饰符应尽量收窄。能 `private` 就不要包可见，能包可见就不要 `protected` 或 `public`。公开 API 一旦被外部依赖，修改成本会显著增加。

## 小结

Java 面向对象的重点是用类型表达稳定边界，用封装保护对象状态，用多态隔离变化，用接口和抽象类组织契约与复用。继承、抽象类和接口都是工具，不是目标。好的对象设计应让调用方少知道内部细节，让状态变化集中在可信入口，让替换实现不影响稳定代码，并让类型关系符合真实概念关系。

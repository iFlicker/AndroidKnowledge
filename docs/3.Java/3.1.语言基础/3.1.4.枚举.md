# 3.1.4 枚举

## 定位

枚举是 Java 用来表达“一组固定、有限、具名取值”的语言机制。它看起来只是把若干常量放在一起，但真正的价值并不只是少写几个 `public static final`，而是把取值集合、类型约束、实例唯一性、字段、方法、行为和标准库支持放进同一个类型中。只要一个概念天然只有有限个稳定选项，并且调用方不应该随意创造新值，就可以考虑用枚举来建模。

常见例子包括方向、状态、等级、排序方式、开关模式、日期中的星期、处理结果、计算符号、日志级别、协议中的已知类型等。这些概念的共同点是：取值范围不是无限开放的，程序中很多判断都依赖“它到底是哪一个值”。如果用整数或字符串表示，编译器无法阻止非法值进入系统；如果用枚举表示，方法签名本身就告诉调用方只能传入这一组合法常量。

枚举不是为了替代所有常量。`MAX_SIZE`、`DEFAULT_TIMEOUT`、`PI` 这类单个数值常量不需要枚举；运行期从配置中心、数据库或用户输入中动态扩展的类别也通常不适合枚举。枚举更适合表达程序编译时就能列举清楚的概念边界。判断一个模型是否适合枚举，可以先问三个问题：取值集合是否有限；这些取值是否有清晰名字；调用方是否应该被限制在这些名字之内。如果答案基本为是，枚举往往比字符串和整数常量更稳妥。

在 Java 语言体系里，枚举还承担了一个重要角色：它让“值域”成为类型系统的一部分。一个参数写成 `OrderState`，就比写成 `int state` 或 `String state` 更能表达业务语义；一个返回值写成 `ParseResult`，也比返回 `0`、`1`、`2` 更容易被阅读和维护。类型越准确，调用方越不容易误用，重构工具也越能可靠地发现影响面。

本文只从通用 Java 技术视角讨论枚举，不把它映射到任何特定平台或框架。重点不是背语法，而是理解枚举解决了什么问题、它在编译器和运行时层面有什么特性、哪些场景适合把行为放进枚举、哪些场景应该让枚举保持轻量，以及在序列化、持久化、外部协议和版本演进中需要注意哪些边界。

## 为什么需要枚举

在没有枚举的写法中，开发者通常会使用整数常量或字符串常量表达有限集合。

```java
final class StateConstants {
    static final int NEW = 0;
    static final int RUNNING = 1;
    static final int DONE = 2;

    private StateConstants() {
    }
}
```

这种代码短期看很直接，但它的类型约束很弱。任何 `int` 都可以传入期望状态的方法，`-1`、`99` 甚至另一个常量类中的值在编译期都不会报错。调用方必须依靠约定记住哪些数字是合法状态，维护者也必须在阅读时不断把数字翻译成人类可理解的含义。一旦系统变大，非法值进入核心流程后往往很难定位来源。

字符串常量比整数稍微可读一些，但问题仍然存在。`"running"`、`"RUNNING"`、`"runing"` 都是普通字符串，编译器不知道哪个代表合法状态。字符串还容易在大小写、分隔符、国际化命名和外部协议编码之间混用。把字符串常量集中到一个类里可以减少拼写错误，却不能阻止调用方直接手写一个新字符串。

枚举把问题从“靠约定使用常量”提升为“靠类型表达约束”。

```java
enum TaskState {
    NEW, RUNNING, DONE
}

void update(TaskState state) {
    // 调用方只能传入 TaskState 中声明的常量，或者显式传入 null。
}
```

这段代码的关键不在于语法更短，而在于非法状态无法自然通过编译。调用方不能把任意整数传进 `update`，也不能把另一个枚举类型的常量误传进来。即使两个枚举常量名字相同，只要类型不同，编译器也会把它们区分开。这种类型隔离是枚举相对常量类最重要的优势。

枚举还提升了可读性。`TaskState.RUNNING` 比 `1` 更容易理解，`SortOrder.ASC` 比 `"asc"` 更容易被 IDE 重命名和查找引用。枚举常量作为真正的对象，还可以携带字段和方法，把与取值强相关的知识收拢到类型内部。例如等级不仅有名字，还可以有权重；错误类型不仅有名称，还可以有稳定编码；运算符不仅有符号，还可以有计算行为。

更进一步，枚举给“穷尽处理”提供了条件。对一个枚举做 `switch` 时，编译器和 IDE 可以知道当前枚举有哪些常量，也能在新增常量后提示相关分支可能需要更新。整数和字符串很难做到这一点，因为它们的取值空间过大，编译器不知道哪些值才是业务上关心的有限集合。

因此，枚举解决的问题可以概括为四类：第一，让有限集合成为类型，而不是散落的约定；第二，让非法值尽量在编译期暴露；第三，让常量携带与自身强绑定的数据和行为；第四，让代码阅读、搜索、重构和分支处理更可靠。

## 基本声明与使用

最简单的枚举只需要使用 `enum` 关键字声明常量列表。

```java
enum Direction {
    NORTH, EAST, SOUTH, WEST
}
```

枚举常量之间使用逗号分隔。如果枚举体中除了常量列表没有其他成员，最后一个常量后的分号可以省略；如果后面还要声明字段、构造方法或方法，则常量列表后必须写分号。

```java
enum Priority {
    LOW, NORMAL, HIGH;

    boolean isHigh() {
        return this == HIGH;
    }
}
```

枚举类型可以像普通类一样作为变量类型、参数类型、返回值类型和字段类型。

```java
Direction direction = Direction.NORTH;

Direction turnRight(Direction current) {
    return switch (current) {
        case NORTH -> Direction.EAST;
        case EAST -> Direction.SOUTH;
        case SOUTH -> Direction.WEST;
        case WEST -> Direction.NORTH;
    };
}
```

枚举变量可以为 `null`，这点容易被忽略。枚举限制的是非空值必须来自声明的常量集合，并不是自动消除空指针问题。是否允许 `null` 应该由方法契约明确决定。对于必须传入枚举值的参数，实践中可以用 `Objects.requireNonNull` 在入口处快速失败；对于可选值，可以使用 `Optional`、单独的字段状态，或者在枚举中定义一个业务上真实存在的 `UNKNOWN`、`UNSPECIFIED`、`NONE` 等常量，但不要把这些常量当成逃避建模的万能选项。

枚举可以用于 `if` 判断，也可以用于 `switch`。比较枚举常量时通常使用 `==`，因为每个枚举常量在同一个类加载器范围内是唯一实例。使用 `equals` 也能得到正确结果，但 `==` 更直接，并且左侧为 `null` 时不会因为调用方法而抛出异常。当然，如果变量可能为 `null`，无论使用 `==` 还是 `switch`，都要明确处理空值；对空枚举变量执行 `switch` 通常会导致空指针异常。

```java
boolean isTerminal(TaskState state) {
    return state == TaskState.DONE;
}
```

在较新的 Java 版本中，`switch` 表达式可以直接返回结果，使枚举分支更集中。对于所有常量都被列出的枚举 `switch` 表达式，可以不写 `default`。这样当未来新增枚举常量时，编译器更容易暴露未覆盖分支。相反，如果总是写一个宽泛的 `default`，新增常量可能被悄悄归入默认分支，导致行为没有经过有意识的设计。

`default` 并不是绝对不能写。处理外部输入、兼容旧版本数据、保护库代码边界时，默认分支可能有价值。关键是理解它的代价：它会降低穷尽处理的可见性。如果一个 `switch` 的目标是表达“每个枚举常量都必须被明确处理”，最好列出所有常量并避免用 `default` 吞掉未来变化。

## 枚举常量是对象

Java 枚举的一个核心事实是：枚举常量不是简单的数字或字符串，而是该枚举类型的固定实例。每个常量在枚举类初始化时创建，之后被全局共享。可以把它理解为语言受控的一组单例对象。

下面的枚举不仅有名字，还携带权重字段。

```java
enum Priority {
    LOW(1),
    NORMAL(5),
    HIGH(10);

    private final int weight;

    Priority(int weight) {
        this.weight = weight;
    }

    int weight() {
        return weight;
    }
}
```

枚举构造方法不能是 `public` 或 `protected`。即使不显式写 `private`，它也只能在枚举内部被调用。外部代码不能 `new Priority(...)`，也不能继承某个枚举再添加常量。这保证了枚举的取值集合只能由声明处控制。

构造参数写在常量名后面，每个常量都必须满足构造方法要求。字段一般应设计为 `private final`，并通过方法暴露只读含义。由于枚举常量是全局共享实例，可变字段会变成全局共享状态，容易制造隐藏耦合和并发问题。例如在枚举里放一个可变 `List`、计数器或当前用户上下文，往往会让枚举承担不该承担的运行期状态。

不可变字段则非常适合枚举。稳定编码、展示文本、排序权重、数学符号、默认参数、能力标记等都可以作为枚举字段。这样调用方不需要维护平行数组或工具类映射，也不需要在多个地方写重复 `switch`。

```java
enum FileType {
    TEXT("txt", true),
    CSV("csv", true),
    BINARY("bin", false);

    private final String extension;
    private final boolean readableAsText;

    FileType(String extension, boolean readableAsText) {
        this.extension = extension;
        this.readableAsText = readableAsText;
    }

    String extension() {
        return extension;
    }

    boolean readableAsText() {
        return readableAsText;
    }
}
```

这类字段应该表达枚举常量本身的稳定属性，而不是某个业务流程临时计算出来的状态。如果字段值会随请求、时间、用户、环境或外部资源变化，它通常不应该放在枚举实例上。枚举是类型模型的一部分，不适合作为运行期上下文容器。

枚举还可以实现接口。这让有限集合中的每个常量以统一类型暴露能力，同时又不允许外部随意扩展常量集合。

```java
interface HasCode {
    String code();
}

enum ErrorLevel implements HasCode {
    INFO("I"),
    WARN("W"),
    ERROR("E");

    private final String code;

    ErrorLevel(String code) {
        this.code = code;
    }

    public String code() {
        return code;
    }
}
```

需要注意的是，枚举不能继承其他类，因为所有枚举都隐式继承 `java.lang.Enum`。但枚举可以实现一个或多个接口。这个限制也提醒我们：枚举更像受控的值域类型，而不是普通面向对象继承体系中的可扩展基类。

## 枚举与 java.lang.Enum

所有枚举类型都会隐式继承 `java.lang.Enum<E>`。源码中不能显式写 `extends Enum`，但编译器会生成相应继承关系。`Enum` 为所有枚举提供了一组基础方法，包括 `name()`、`ordinal()`、`compareTo()`、`equals()`、`hashCode()`、`toString()`、`getDeclaringClass()` 等。

`name()` 返回枚举常量在源码中声明的名字。它是一个稳定的内部标识，但只有在你承诺不重命名常量时才适合对外使用。很多代码为了方便直接把 `name()` 写入文件、数据库或网络消息，短期很省事，长期却可能把 Java 源码命名和外部协议绑定死。一旦常量从 `IN_PROGRESS` 重命名为 `RUNNING`，外部数据就会受到影响。

`ordinal()` 返回枚举常量的声明序号，从零开始。它主要服务于枚举自身和某些标准库实现，不应该作为业务编码持久化或传输。调整枚举声明顺序、在中间插入新常量，都会改变后续常量的 `ordinal`。如果外部系统曾保存这些数字，含义就会错乱，而且这种错乱通常不会在编译期暴露。

```java
enum ResultCode {
    SUCCESS("S"),
    FAILED("F");

    private final String code;

    ResultCode(String code) {
        this.code = code;
    }

    String code() {
        return code;
    }
}
```

如果需要稳定编码，应显式定义字段，如 `code`、`value`、`symbol`。字段值可以独立于 Java 常量名演进，也可以根据协议要求使用小写、数字、短字符串或其他格式。常量名服务于 Java 代码阅读，外部编码服务于系统边界，两者最好不要无意识地混在一起。

`toString()` 默认返回 `name()`，但枚举可以重写 `toString()`。重写后适合改善日志或界面展示，但更不应该把 `toString()` 当成稳定协议值。`toString()` 的语义偏向人类可读，未来可能因为展示需求变化而改变；协议编码应该使用明确命名的方法，例如 `code()`。

`compareTo()` 按 `ordinal` 比较，因此枚举天然具有声明顺序。这个顺序可用于表达非常明确的自然顺序，例如 `LOW`、`MEDIUM`、`HIGH`。但如果顺序只是代码书写上的排列，不应该依赖它进行业务排序。需要稳定排序规则时，可以定义显式权重字段，或用比较器清楚表达规则。

`getDeclaringClass()` 返回声明该枚举常量的枚举类型。大多数业务代码用不到它，但在泛型工具、反射处理和框架代码中有意义。相比普通对象，枚举的运行时元信息更规则，标准库也能利用这些信息提供专用集合和查找能力。

## 编译器生成的方法：values 与 valueOf

每个枚举类型都会由编译器生成两个常用静态方法：`values()` 和 `valueOf(String)`。

```java
for (Priority priority : Priority.values()) {
    System.out.println(priority.name() + ":" + priority.weight());
}

Priority high = Priority.valueOf("HIGH");
```

`values()` 返回当前枚举所有常量组成的数组，顺序与源码声明顺序一致。需要注意，返回的是数组而不是不可变列表；为了避免调用方修改内部结构，编译器生成的方法通常会返回内部数组的克隆。因此频繁在热路径中调用 `values()` 会产生数组分配。普通业务代码不必过度担心，但在性能敏感循环中，可以把结果缓存为私有静态不可变集合。

```java
enum Priority {
    LOW, NORMAL, HIGH;

    private static final List<Priority> ALL = List.of(values());

    static List<Priority> all() {
        return ALL;
    }
}
```

`valueOf(String)` 按常量名精确查找枚举值。它区分大小写，不会忽略空格，也不会理解外部编码。找不到常量时会抛出 `IllegalArgumentException`，参数为 `null` 时会抛出 `NullPointerException`。因此它适合解析可信的内部名字，不适合直接处理用户输入或不稳定外部字段。

更稳妥的做法是在枚举中提供显式解析方法，把大小写、空白、未知值、错误处理和外部编码集中在一个地方。

```java
enum Level {
    DEBUG("debug"),
    INFO("info"),
    WARN("warn"),
    ERROR("error");

    private static final Map<String, Level> BY_CODE =
            Arrays.stream(values()).collect(Collectors.toUnmodifiableMap(Level::code, level -> level));

    private final String code;

    Level(String code) {
        this.code = code;
    }

    String code() {
        return code;
    }

    static Optional<Level> fromCode(String code) {
        return Optional.ofNullable(BY_CODE.get(code));
    }
}
```

这种写法的好处是调用方不用知道 Java 常量名和外部编码之间的关系，也不用到处捕获 `IllegalArgumentException`。返回 `Optional` 表示外部值可能未知；如果业务希望未知值直接失败，也可以提供 `parseCode` 方法抛出带有上下文的异常。关键是由枚举自己统一定义解析契约，而不是让解析逻辑散落在调用方。

如果解析规则需要忽略大小写，应明确写出来；如果需要兼容旧编码，也应集中处理。不要指望 `valueOf` 替代协议解析。它的设计目标是根据枚举常量名查找枚举对象，而不是做宽容的输入清洗。

## 枚举中的字段和构造方法

枚举字段的核心原则是：只保存与常量身份强绑定、生命周期稳定、最好不可变的信息。字段让枚举从“名字列表”变成“带属性的值域”，但也容易诱导开发者把过多职责塞进枚举。

例如一个货币单位枚举可以保存字母编码和小数位数。

```java
enum CurrencyUnit {
    USD("USD", 2),
    JPY("JPY", 0),
    KWD("KWD", 3);

    private final String code;
    private final int fractionDigits;

    CurrencyUnit(String code, int fractionDigits) {
        this.code = code;
        this.fractionDigits = fractionDigits;
    }

    String code() {
        return code;
    }

    int fractionDigits() {
        return fractionDigits;
    }
}
```

这些字段是稳定属性：美元的编码、日元通常使用的小数位数，都是常量自身的描述。调用方通过枚举方法读取这些属性，语义比维护一个外部 `Map<CurrencyUnit, Integer>` 更直接。

但如果字段表示“当前汇率”“今日是否可交易”“当前用户是否允许使用”，就不适合放入枚举。它们依赖时间、市场、权限或外部数据，应该由服务、配置或上下文对象提供。枚举一旦持有这些状态，就会把全局共享实例变成可变业务容器，测试和并发都会变复杂。

构造方法中也应避免做复杂工作。枚举常量在类初始化时创建，构造过程如果读取文件、连接网络、访问数据库或依赖其他复杂静态初始化，可能导致类加载时机难以预测，甚至引发初始化循环。枚举构造方法最适合做简单赋值和轻量校验。

枚举字段还要注意可变对象引用。即使字段本身是 `final`，如果它引用的是可变对象，外部仍可能通过 getter 修改其内部状态。

```java
enum BadExample {
    INSTANCE(new ArrayList<>());

    private final List<String> values;

    BadExample(List<String> values) {
        this.values = values;
    }

    List<String> values() {
        return values;
    }
}
```

这种写法会暴露全局可变列表。更好的方式是只保存不可变集合，或者返回防御性副本，或者根本不要把变化数据放入枚举。枚举实例的共享性越强，越应该保持不可变。

## 枚举中的方法与常量特定类体

枚举可以定义普通方法，也可以定义抽象方法并让每个常量提供自己的实现。后者被称为常量特定类体，语法上是在某个枚举常量后写一段类体。

```java
enum Operator {
    PLUS {
        int apply(int a, int b) {
            return a + b;
        }
    },
    TIMES {
        int apply(int a, int b) {
            return a * b;
        }
    };

    abstract int apply(int a, int b);
}
```

这种写法可以把与枚举常量强绑定的行为放回枚举内部。调用方不需要写 `switch` 判断 `PLUS` 还是 `TIMES`，只需调用 `operator.apply(a, b)`。当新增一个运算符常量时，编译器会要求它实现抽象方法，从而减少遗漏分支的风险。

常量特定实现适合满足几个条件的场景：行为很小；行为确实由枚举常量本身决定；新增常量时必须新增对应行为；行为不依赖复杂外部资源；不同调用场景下规则基本一致。数学运算、简单格式化、固定状态流转、简单校验规则，往往可以考虑这种方式。

但它也有明显边界。第一，枚举文件会随着行为复杂度膨胀，最终变成很难阅读的大类。第二，行为如果依赖数据库、远程服务、配置、用户上下文或大型对象图，枚举内部很难优雅注入依赖。第三，同一个枚举在不同场景可能对应不同算法，把算法固定在枚举里会降低灵活性。

当行为变化方向不是“新增枚举值就新增行为”，而是“同一枚举值在不同上下文有不同策略”时，外部策略表通常更合适。

```java
Map<SortOrder, Comparator<String>> comparators = Map.of(
        SortOrder.ASC, Comparator.naturalOrder(),
        SortOrder.DESC, Comparator.reverseOrder()
);
```

这个设计让枚举只表达取值，策略由使用场景决定。它的可测试性和依赖管理通常更好。换句话说，枚举中的方法应当服务于枚举自身的稳定语义，而不是把枚举变成全局策略容器。

还有一种折中方式：枚举提供少量基础属性或标记，由外部服务根据这些属性组合行为。例如 `PermissionLevel` 可以知道自己的等级权重，但权限判定规则由权限服务完成。这样既避免散落魔法值，又不把复杂业务塞进枚举。

## 枚举与 switch 的设计取舍

枚举和 `switch` 经常一起出现。正确使用时，`switch` 能清楚表达“根据有限状态选择分支”；滥用时，它会让同一枚举的判断逻辑散落在系统各处，新增常量时需要到处找分支。

下面的代码非常直观。

```java
String label(Priority priority) {
    return switch (priority) {
        case LOW -> "low";
        case NORMAL -> "normal";
        case HIGH -> "high";
    };
}
```

如果标签只是枚举自身的稳定属性，更好的做法可能是把它放到字段中。

```java
enum Priority {
    LOW("low"),
    NORMAL("normal"),
    HIGH("high");

    private final String label;

    Priority(String label) {
        this.label = label;
    }

    String label() {
        return label;
    }
}
```

如果分支表达的是某个调用场景的流程选择，把 `switch` 保留在调用场景中反而更清晰。不要为了消灭所有 `switch` 而把上下文相关逻辑塞进枚举。评价标准不是“有没有 `switch`”，而是“分支知识属于枚举本身，还是属于使用枚举的那段流程”。

一个实用判断是看变化时的修改点。如果新增枚举常量时，某个分支一定需要同步处理，并且处理逻辑与常量身份直接绑定，可以考虑把行为移到枚举。反之，如果新增一个业务流程时，需要用同一个枚举做完全不同的分支，枚举内部方法就不应该承担这些流程差异。

`switch` 还涉及 `default` 的取舍。对内部枚举做穷尽判断时，不写 `default` 可以让新增常量更容易暴露编译错误或 IDE 警告。对外部兼容边界做防御时，`default` 可以作为保护网。两者没有绝对优劣，关键是让代码表达意图：这里是要求每个常量都被设计，还是允许未列出的情况进入通用处理。

对于库代码，还要考虑二进制兼容和调用方版本差异。一个库内部新增枚举常量后，旧调用方如果使用 `switch` 且有默认分支，可能会把新值归为旧默认行为；如果没有默认分支，可能在运行时暴露异常或在重新编译时暴露缺失分支。这不是枚举的缺陷，而是有限集合演进时必须面对的问题。设计公共枚举时，新增、删除、重命名常量都应被视为兼容性决策。

## EnumSet：专门服务枚举的集合

当需要表示“一组枚举值”时，优先考虑 `EnumSet`。它是 Java 标准库为枚举优化的 `Set` 实现，利用枚举常量数量有限且有声明序号的特性，通常比 `HashSet` 更紧凑，也更能表达意图。

```java
EnumSet<Direction> horizontal = EnumSet.of(Direction.EAST, Direction.WEST);

if (horizontal.contains(Direction.NORTH)) {
    // false
}
```

`EnumSet` 不允许存放 `null`，也不能混合不同枚举类型。它内部可以用位向量表达集合关系：某个枚举常量是否存在，可以映射到一个二进制位。这使得并集、交集、补集等操作非常高效。当然，业务代码不需要依赖具体实现细节，只需要知道它比通用集合更适合枚举值集合。

`EnumSet` 的常见创建方式包括 `noneOf`、`allOf`、`of`、`copyOf`、`complementOf`。

```java
EnumSet<Priority> empty = EnumSet.noneOf(Priority.class);
EnumSet<Priority> all = EnumSet.allOf(Priority.class);
EnumSet<Priority> important = EnumSet.of(Priority.NORMAL, Priority.HIGH);
EnumSet<Priority> notImportant = EnumSet.complementOf(important);
```

如果集合语义是权限、能力、标志位、可选模式组合，`EnumSet` 往往比用整数位掩码更可读。传统位掩码把多个布尔选项塞进一个数字，性能很好，但可读性和类型安全较弱。`EnumSet` 保留了位集合的紧凑优势，同时让每个选项仍然是具名枚举常量。

不过，`EnumSet` 是可变集合。把它作为字段暴露时仍要注意防御性复制或不可变包装。不要因为它是专用集合就忽略封装。如果一个对象内部保存 `EnumSet`，对外返回时可以使用 `EnumSet.copyOf` 创建副本，或返回不可变视图。

```java
final class FeatureConfig {
    private final EnumSet<Feature> enabled;

    FeatureConfig(Set<Feature> enabled) {
        this.enabled = enabled.isEmpty()
                ? EnumSet.noneOf(Feature.class)
                : EnumSet.copyOf(enabled);
    }

    Set<Feature> enabled() {
        return Collections.unmodifiableSet(enabled);
    }
}
```

还要注意空集合创建时需要枚举类型信息。`EnumSet.copyOf(emptySet)` 无法从空集合中推断枚举类型，可能抛出异常；此时应使用 `EnumSet.noneOf(Feature.class)`。这也是枚举集合与普通集合的一个小差异。

## EnumMap：专门服务枚举键的映射

当 `Map` 的键是枚举时，优先考虑 `EnumMap`。它根据枚举常量的声明顺序组织值，内部可以用数组存储，查找速度快，内存开销低，并且语义非常明确：键空间就是这个枚举的所有常量。

```java
EnumMap<Priority, String> labels = new EnumMap<>(Priority.class);
labels.put(Priority.LOW, "low");
labels.put(Priority.NORMAL, "normal");
labels.put(Priority.HIGH, "high");
```

`EnumMap` 不允许 `null` 键，但允许 `null` 值。它的迭代顺序是枚举常量的自然顺序，也就是声明顺序。这个顺序如果对调用方有意义，应在枚举声明处保持清晰；如果没有业务意义，就不要让外部逻辑依赖它。

`EnumMap` 适合表达“每个枚举值对应一个配置、处理器、标签、权重或转换规则”。相比 `HashMap<Enum, V>`，它更清楚地表达键的有限性，也避免了散列结构的额外开销。

```java
EnumMap<Operator, String> symbols = new EnumMap<>(Operator.class);
symbols.put(Operator.PLUS, "+");
symbols.put(Operator.TIMES, "*");
```

如果映射要求覆盖所有枚举常量，初始化后应主动校验完整性。`EnumMap` 本身不会强制每个枚举键都有值。可以比较 `map.keySet()` 与 `EnumSet.allOf(Type.class)`，或在构造逻辑中逐一填充。对于公共配置，缺失某个枚举值对应的处理器通常应该尽早失败，而不是等到运行到某个分支才得到 `null`。

在多线程环境下，`EnumMap` 和大多数普通集合一样不是线程安全的。如果构造完成后只读，可以把它封装为不可变映射或不再暴露可变引用；如果需要并发修改，应使用外部同步或其他并发结构。枚举键并不会自动解决并发问题。

## 枚举与外部值解析

枚举经常需要和外部文本、配置文件、命令行参数、数据库字段或网络协议互相转换。这里最容易出现的误区是直接使用 `name()` 和 `valueOf()` 作为边界协议。它们对于内部 Java 代码很方便，但外部世界通常有自己的命名规则、兼容要求和未知值处理方式。

更稳健的做法是为外部编码定义显式字段，并提供集中解析方法。

```java
enum SortOrder {
    ASCENDING("asc"),
    DESCENDING("desc");

    private static final Map<String, SortOrder> BY_CODE =
            Arrays.stream(values()).collect(Collectors.toUnmodifiableMap(SortOrder::code, order -> order));

    private final String code;

    SortOrder(String code) {
        this.code = code;
    }

    String code() {
        return code;
    }

    static Optional<SortOrder> fromCode(String code) {
        return Optional.ofNullable(BY_CODE.get(code));
    }
}
```

外部编码的设计要考虑几个问题。第一，编码是否大小写敏感。第二，未知值应该被拒绝、忽略、降级还是保留。第三，编码是否需要长期兼容。第四，旧编码是否会迁移到新编码。第五，解析失败时是否需要包含原始输入以便诊断。

对于内部配置，如果未知值代表配置错误，可以抛出异常并尽早失败。对于跨系统协议，如果对方未来可能发送新值，直接失败可能降低兼容性；此时可以返回 `Optional.empty()`，或者使用一个更开放的值对象保留原始字符串。

`UNKNOWN` 常量是常见设计，但它并不总是正确。它适合业务只需要“已知几类之外都按未知处理”的情况，例如日志统计或粗略展示。它不适合需要无损转发的协议字段，因为所有未来新值都会被压成同一个 `UNKNOWN`，原始信息丢失。若系统需要读取未知值后再写回，最好使用包含原始编码的对象，而不是纯枚举。

```java
record KnownOrRaw<T extends Enum<T>>(T known, String raw) {
    boolean isKnown() {
        return known != null;
    }
}
```

这类模型比单纯枚举复杂，但能表达更真实的边界：本系统认识一部分有限值，外部协议可能比本系统更新。通用 Java 程序在处理文件格式、插件协议、服务接口时都可能遇到这种问题。不要因为枚举方便，就把开放世界强行压成封闭集合。

反过来，如果取值集合本来就由本程序完全控制，没有外部扩展可能，纯枚举就很合适。设计的关键不是“枚举好不好”，而是识别边界是否开放。

## 枚举的序列化与持久化

Java 枚举有特殊的序列化规则。使用 Java 原生对象序列化时，枚举通常按常量名进行序列化，反序列化时返回已有的枚举常量实例，而不是创建新对象。这也是枚举单例能够天然保持实例唯一性的原因之一。

这种机制对语言内部很方便，但对长期数据兼容提出了要求：删除或重命名枚举常量可能导致旧数据无法反序列化；改变常量声明顺序通常不影响按名字的原生序列化，但会影响依赖 `ordinal` 的自定义持久化；修改字段值不会改变常量身份，却可能改变读取旧数据后的业务含义。

在数据库、文本文件、消息队列或公开接口中保存枚举时，优先使用显式稳定编码。不要保存 `ordinal`。是否保存 `name` 取决于常量名是否被视为协议的一部分。如果只是内部实现命名，仍然应该使用 `code` 字段。

```java
enum PaymentState {
    CREATED("created"),
    CONFIRMED("confirmed"),
    CANCELED("canceled");

    private final String code;

    PaymentState(String code) {
        this.code = code;
    }

    String code() {
        return code;
    }
}
```

持久化还要考虑版本演进。新增枚举常量一般比删除和重命名安全，但也不是完全无风险。旧版本程序读取新常量编码时可能不认识；报表、校验、分支处理、统计逻辑也可能遗漏新状态。删除常量则需要处理历史数据；重命名常量如果外部编码不变，影响可能较小，如果编码也变了，就需要迁移和兼容解析。

对于长期存储的数据，枚举演进最好遵循几个原则：外部编码一旦发布就尽量不变；废弃值先保留解析能力，再逐步停止产生；读取未知值时有明确策略；关键分支不要用宽泛默认值掩盖新状态；迁移脚本和兼容代码应与枚举变更一起设计。

还要区分序列化身份和业务属性。枚举字段不会让每个常量变成可独立序列化的普通对象；反序列化得到的是当前程序中声明的那个常量。也就是说，如果旧版本序列化了 `HIGH`，新版本中 `HIGH` 的字段权重从 `10` 改成 `100`，反序列化后看到的是新版本的权重。这通常符合“枚举代表当前代码定义”的直觉，但如果你期望保存当时的完整快照，枚举字段就不够，需要把字段值作为普通数据一起保存。

## 枚举单例

枚举可以用来实现单例。最典型写法是声明一个只包含 `INSTANCE` 的枚举。

```java
enum IdGenerator {
    INSTANCE;

    private final AtomicLong next = new AtomicLong();

    long nextId() {
        return next.incrementAndGet();
    }
}
```

枚举单例的优点是语义简单，实例唯一性由语言和运行时保证，并且天然处理 Java 原生序列化场景下的重复实例问题。相比手写双重检查锁、静态内部类或 `readResolve`，枚举单例更不容易写错。

但枚举单例不是所有单例需求的最佳答案。它不支持继承其他类，不能由外部传入构造参数，生命周期与类加载绑定，测试替换也不如依赖注入灵活。它适合无状态工具、轻量服务、进程内唯一对象、简单注册表等场景；不适合需要环境参数、资源释放、复杂初始化顺序或多实例测试的对象。

还要警惕单例本身带来的全局访问问题。枚举只是让单例实现更可靠，并不会让全局状态自动变得可维护。如果 `IdGenerator.INSTANCE` 内部持有可变计数器，调用方就共享同一份状态；如果单例内部依赖系统时间、文件或网络，测试仍然需要隔离这些依赖。不要因为枚举单例写法简洁，就把本应通过对象组合传递的依赖变成全局入口。

枚举单例最适合“确实只能有一个，并且这个唯一性是领域模型的一部分”的对象。如果只是为了调用方便而做单例，通常要重新考虑设计。很多时候，普通类加明确的生命周期管理，比全局枚举单例更容易测试和演进。

## 枚举与泛型、接口和多态

枚举可以与泛型和接口结合，形成类型安全的工具方法。由于所有枚举都继承 `Enum<E>`，泛型约束中可以写出“某个类型必须是枚举”。

```java
static <E extends Enum<E>> Optional<E> parseEnum(Class<E> type, String name) {
    try {
        return Optional.of(Enum.valueOf(type, name));
    } catch (IllegalArgumentException ex) {
        return Optional.empty();
    }
}
```

这个方法能处理任意枚举类型，但它仍然只按常量名解析。对于外部编码，通用工具不一定知道每个枚举的 `code` 方法。因此实践中可以通过接口表达“这个枚举有稳定编码”。

```java
interface CodedEnum {
    String code();
}

enum Mode implements CodedEnum {
    FAST("fast"),
    SAFE("safe");

    private final String code;

    Mode(String code) {
        this.code = code;
    }

    public String code() {
        return code;
    }
}
```

再配合泛型工具时，可以约束类型既是枚举又实现某个接口。

```java
static <E extends Enum<E> & CodedEnum> Optional<E> parseByCode(Class<E> type, String code) {
    for (E constant : type.getEnumConstants()) {
        if (constant.code().equals(code)) {
            return Optional.of(constant);
        }
    }
    return Optional.empty();
}
```

这里的交叉类型 `<E extends Enum<E> & CodedEnum>` 很有表达力：调用方不能传普通类，也不能传没有编码能力的枚举。它常用于框架工具、通用转换、校验器和配置解析。

需要注意，泛型工具不应该假设所有枚举都有无参构造、可变字段或某个固定方法。枚举的实例创建由语言控制，外部不能通过反射常规创建新常量。工具代码应通过 `Class<E>`、`getEnumConstants()`、接口方法和标准 API 读取枚举信息，而不是试图把枚举当成普通可实例化类。

枚举实现接口时，也可以让不同枚举类型在某些通用场景下被统一处理。例如多个枚举都实现 `CodedEnum`，它们可以共享日志打印、参数校验或编码解析逻辑。但不要为了复用而强行让无关枚举实现同一个大接口。接口应表达真实能力，而不是成为所有枚举的公共杂物箱。

## 枚举与反射

枚举在反射中也有专门支持。`Class` 提供了 `isEnum()` 和 `getEnumConstants()`，可以判断一个类型是否是枚举并获取常量数组。

```java
void printEnumConstants(Class<?> type) {
    if (!type.isEnum()) {
        return;
    }
    for (Object constant : type.getEnumConstants()) {
        System.out.println(constant);
    }
}
```

反射常用于通用框架、配置绑定、命令行解析和文档生成。通过枚举元信息，工具可以列出合法选项，给出错误提示，或自动生成帮助文本。

枚举常量本身也可以通过反射读取字段和注解。虽然这里不展开注解机制，但要知道枚举常量在源码上类似字段，可以作为元数据承载点。不过，反射会降低静态可见性和重构安全性，普通业务代码不应优先使用反射处理枚举。能用方法和字段直接表达的规则，就不要绕到反射层。

还有一个边界是：不要试图通过反射创建新的枚举常量。Java 语言设计上不允许在运行时扩展某个枚举的常量集合。即使有人通过非标准技巧修改运行时内部结构，也会破坏 `switch`、`EnumSet`、`EnumMap`、序列化和编译器假设，属于不可维护做法。如果取值集合确实需要运行期扩展，就应该使用普通类、注册表、配置模型或插件机制，而不是枚举。

反射读取枚举时还要注意常量特定类体。带有常量特定实现的枚举常量，运行时类可能是枚举类型的匿名子类；但 `getDeclaringClass()` 仍然能回到声明枚举。多数业务逻辑不需要关心这个细节，但框架代码如果直接比较 `constant.getClass()` 与枚举类型，可能遇到意外。更稳妥的方式是使用 `((Enum<?>) constant).getDeclaringClass()` 或已知的 `Class<E>`。

## 枚举的类加载与初始化

枚举常量在枚举类初始化时创建。类初始化通常发生在首次主动使用该类时，例如访问静态字段、调用静态方法、创建实例等。对枚举来说，访问某个常量、调用 `values()`、调用静态解析方法，都可能触发初始化。

初始化顺序大致是先创建枚举常量，再执行后续静态字段初始化和静态代码块。这一点对枚举内部的静态字段依赖很重要。枚举构造方法中不应该依赖后面才初始化的复杂静态状态，否则容易出现空值、初始化循环或难以理解的错误。

```java
enum Risky {
    A;

    private static final Map<String, Risky> BY_NAME = new HashMap<>();

    Risky() {
        // 不要在这里依赖 BY_NAME 进行复杂注册。
    }
}
```

更好的方式是在所有常量创建完成后，通过静态字段初始化构建查找表。

```java
enum SafeLevel {
    LOW("low"),
    HIGH("high");

    private static final Map<String, SafeLevel> BY_CODE =
            Arrays.stream(values()).collect(Collectors.toUnmodifiableMap(SafeLevel::code, level -> level));

    private final String code;

    SafeLevel(String code) {
        this.code = code;
    }

    String code() {
        return code;
    }
}
```

枚举类初始化由 JVM 保证线程安全，因此枚举单例不需要额外同步来保证唯一实例创建。但这不代表枚举内部的可变字段操作自动线程安全。初始化安全和运行期并发安全是两件事。`AtomicLong` 自身是线程安全的，普通 `long` 自增则不是；不可变字段天然安全，可变集合则需要额外保护。

类加载器也是枚举唯一性讨论中的一个边界。在同一个类加载器加载的同一个枚举类型中，每个常量唯一；但如果同名类被不同类加载器分别加载，它们在 JVM 看来是不同类型。普通应用很少直接遇到这个问题，但在插件化、容器、脚本引擎或复杂模块系统中，类型身份包含类加载器这一事实仍然成立。通用 Java 设计中，只要跨越模块边界传递枚举，就应确保双方使用的是同一份类型定义和兼容版本。

## 枚举的演进与兼容性

枚举看似只是增加或删除常量，但对使用方影响很大。因为枚举常量常常出现在 `switch`、持久化字段、配置文件、接口参数、日志统计、权限矩阵和测试用例中，任何变更都可能影响多个层面。

新增常量通常是最常见的演进方式。它在源码层面看似向后兼容，但使用方可能没有处理新值。对于内部代码，应重新检查所有 `switch`、`EnumMap` 初始化、`EnumSet` 白名单、解析逻辑和测试断言。对于公共 API，应在发布说明中明确新常量含义，并说明旧版本客户端遇到新值时的策略。

删除常量风险更高。旧数据、旧配置、旧日志、旧消息中可能仍然包含这个值。删除前通常需要先废弃，停止新数据产生，再保留读取兼容一段时间，最后才移除。即使源码中已经没有引用，也不代表外部世界没有历史值。

重命名常量也不只是代码风格调整。如果外部使用 `name()` 或 Java 原生序列化，重命名会改变可观察行为。如果只是在 Java 代码内部使用，并且外部编码由独立 `code` 字段保持不变，风险会小很多。重命名前要先确认常量名是否已经成为协议。

调整声明顺序则主要影响 `ordinal`、自然排序、`compareTo`、`EnumSet` 和 `EnumMap` 的迭代顺序，以及任何依赖声明顺序的展示逻辑。如果顺序有业务意义，调整就是行为变更；如果顺序没有业务意义，就不应让外部观察到它。

修改字段值也要谨慎。字段常常是外部编码、权重、展示文本或规则参数。改字段值可能比改常量名更隐蔽，因为编译器不会提示调用方。对于稳定编码字段，修改应视为协议变更；对于展示文本，修改可能影响测试快照或用户可见内容；对于权重，修改可能改变排序、优先级或阈值判断。

一个成熟的枚举演进流程通常包括：识别该枚举是否出现在外部边界；搜索所有 `switch`、`EnumMap`、`EnumSet` 和解析方法；检查持久化和序列化策略；补充新增常量对应测试；明确未知值处理；必要时保留废弃常量和兼容解析。枚举越接近公共 API，演进越要谨慎。

## 常见误区

第一个误区是把枚举当成字符串常量的薄包装。只声明常量、不利用类型约束、到处调用 `name()` 传输，确实能获得一些可读性，但没有充分发挥枚举价值。更好的做法是让方法签名使用枚举类型，让外部编码显式化，让解析逻辑集中化。

第二个误区是持久化 `ordinal()`。这几乎总是不好的选择。声明顺序是源码组织细节，不应该承担长期数据含义。哪怕当前团队承诺不调整顺序，未来插入新常量、合并分支或重构文件时也很容易破坏这个约定。

第三个误区是无条件使用 `valueOf()` 解析外部输入。`valueOf()` 只认识常量名，异常也偏底层。面对用户输入和外部协议，应该使用显式 `fromCode`、`parse` 或转换层，并定义未知值策略。

第四个误区是在枚举中放可变全局状态。枚举常量是共享实例，任何可变字段都可能被所有调用方看见。除非你明确要实现一个全局单例，并且已经处理并发、生命周期和测试问题，否则枚举字段应保持不可变。

第五个误区是把复杂业务流程塞进枚举方法。枚举可以有行为，但行为应与常量稳定语义强绑定。依赖外部资源、需要注入、经常变化、场景差异大的逻辑，应该放在服务、策略类或映射表中。

第六个误区是用 `UNKNOWN` 掩盖建模问题。`UNKNOWN` 有时很实用，但它会丢失原始值，也可能让调用方忽略本应处理的新状态。是否需要 `UNKNOWN`，取决于未知值是否还能被系统有意义地处理。

第七个误区是认为枚举天然不可变。枚举常量身份不可变，常量集合不可扩展，但字段引用的对象可能可变，枚举方法也可能修改内部状态。不可变需要设计出来，而不是由 `enum` 关键字自动保证。

第八个误区是认为新增枚举常量一定兼容。源码能编译不代表行为完整。新增常量后，所有穷尽判断、集合白名单、映射表、持久化解析、测试数据和外部文档都可能需要同步更新。

## 实践判断

设计枚举时，可以先明确它的语义边界。这个枚举到底表达一个领域概念，还是只是某个实现细节？它的取值集合由谁控制？未来是否可能由外部配置扩展？它是否会出现在文件、数据库、网络消息或公共 API 中？这些问题决定了枚举应该简单到只有常量名，还是需要稳定编码、解析方法、字段、兼容策略和测试。

如果枚举只在内部模块使用，并且取值集合非常稳定，可以保持简洁。常量名清楚、方法签名准确、必要时配合 `switch` 即可。不要为了显得完整而给每个枚举都加 `code`、`label`、`fromCode` 和复杂映射。

如果枚举跨越边界，就要更严谨。边界包括持久化、配置、命令行参数、公开库接口、跨进程消息、文本文件等。此时应避免把 `name()` 和 `ordinal()` 当作默认协议，优先设计显式编码；解析时要决定未知值策略；变更时要考虑旧数据和旧调用方。

如果枚举对应一组标志组合，使用 `EnumSet`；如果枚举是映射键，使用 `EnumMap`；如果每个枚举常量都有稳定属性，把属性放进字段；如果每个常量都有简单且稳定的行为，可以考虑常量特定实现；如果行为复杂或依赖上下文，让枚举只表达类型，把行为放到外部策略。

命名上，枚举类型名通常使用单数概念名，如 `Direction`、`Priority`、`TaskState`，常量名使用全大写加下划线。常量名应表达语义而不是编码细节。外部编码可以是小写、短横线、数字或其他协议格式，但应通过字段区分。

空值处理也要明确。一个方法参数如果不接受空值，就在入口处校验；如果空值有业务含义，考虑这个含义是否应该成为枚举常量。不要让 `null`、`UNKNOWN`、`NONE`、`UNSPECIFIED` 混用。`NONE` 往往表示明确没有；`UNKNOWN` 表示存在但不认识；`UNSPECIFIED` 表示未指定；这些语义不同，随意混用会让代码判断变得含糊。

测试上，枚举新增常量后可以增加覆盖所有常量的测试，尤其是对 `EnumMap`、外部编码和 `switch` 分支。对于公共枚举，测试还可以锁定稳定编码，防止误改。对于解析方法，应覆盖合法值、非法值、大小写、空值和旧编码兼容。

文档上，枚举常量如果承载领域状态，应说明每个常量的含义、状态流转、是否终态、是否允许外部输入。不要只让读者从名字猜测语义。枚举越接近业务核心，常量含义越需要明确；枚举越接近基础工具，类型名和字段方法越要清楚。

## 一个完整示例：用枚举表达任务状态

下面用一个简化的任务状态枚举串起前面的实践。假设任务有创建、运行、成功、失败、取消几个状态，需要稳定外部编码、判断是否终态、判断是否可以取消，并支持从外部编码解析。

```java
enum TaskStatus {
    CREATED("created", false, true),
    RUNNING("running", false, true),
    SUCCEEDED("succeeded", true, false),
    FAILED("failed", true, false),
    CANCELED("canceled", true, false);

    private static final Map<String, TaskStatus> BY_CODE =
            Arrays.stream(values()).collect(Collectors.toUnmodifiableMap(TaskStatus::code, status -> status));

    private final String code;
    private final boolean terminal;
    private final boolean cancelable;

    TaskStatus(String code, boolean terminal, boolean cancelable) {
        this.code = code;
        this.terminal = terminal;
        this.cancelable = cancelable;
    }

    String code() {
        return code;
    }

    boolean isTerminal() {
        return terminal;
    }

    boolean isCancelable() {
        return cancelable;
    }

    static Optional<TaskStatus> fromCode(String code) {
        return Optional.ofNullable(BY_CODE.get(code));
    }
}
```

这个枚举把几类知识放在了合适的位置。`code` 是外部稳定编码，避免暴露 Java 常量名；`terminal` 和 `cancelable` 是状态本身的稳定属性，放在字段中比在多个调用方写 `switch` 更集中；`fromCode` 统一了解析逻辑，让调用方明确处理未知编码。

同时，它也没有把所有任务流程都塞进枚举。比如“取消任务时要更新哪些资源”“失败后是否重试”“成功后如何通知下游”，这些行为依赖具体系统上下文，不适合放进 `TaskStatus`。枚举负责表达状态值和状态自身属性，流程服务负责执行操作。这个边界通常比“枚举越强大越好”更可维护。

如果未来新增 `PAUSED` 状态，需要同步考虑它的编码、是否终态、是否可取消、外部解析、状态流转和相关测试。由于属性在枚举构造参数中集中声明，新增常量时很容易看到需要补齐哪些信息。如果外部旧版本不认识 `paused`，边界层还要决定是拒绝、忽略、降级还是保留原始值。

这个例子也说明，好的枚举不是只有常量列表，也不是把业务都包进去。它的职责是用类型系统清楚表达有限取值，并收拢与这些取值稳定绑定的知识。

## 总结

Java 枚举是对有限取值集合的类型化建模。它比整数常量和字符串常量更安全，因为编译器能限制非法赋值；它比普通常量类更有表达力，因为常量本身可以携带字段和方法；它比开放类层次更受控，因为外部不能随意创建新常量。

使用枚举时最重要的不是记住 `values()`、`valueOf()`、`name()`、`ordinal()` 这些 API，而是理解它们的适用边界。`name()` 是源码常量名，不一定是外部协议；`ordinal()` 是声明顺序，不应持久化；`valueOf()` 是精确按名称查找，不适合直接解析不可信输入；`values()` 返回声明顺序下的所有常量，频繁使用时要留意数组分配和顺序依赖。

枚举适合保存不可变、稳定、与常量身份强绑定的属性；适合用 `EnumSet` 表达枚举集合，用 `EnumMap` 表达枚举键映射；适合在行为简单且稳定时使用常量特定实现；也适合实现少量确实需要全局唯一实例的单例。它不适合表达运行期动态扩展的值域，不适合承载复杂外部依赖，不适合保存随请求变化的状态，也不适合把开放协议中的未知未来值粗暴压平。

判断枚举设计是否健康，可以看三个信号：方法签名是否因枚举变得更准确；外部边界是否有稳定编码和未知值策略；枚举内部是否只包含属于常量自身的稳定知识。满足这三点时，枚举能显著提升 Java 代码的可读性、类型安全和可维护性。

# 3.1.3 异常机制

## 定位

异常机制是 Java 表达错误、传播失败和分离正常流程与异常流程的语言设施。它允许方法在无法完成承诺时抛出异常，让调用链上更合适的位置处理问题。异常不是普通分支的替代品，也不是隐藏错误的工具。好的异常设计应让失败原因明确、传播路径可控、资源释放可靠，并让调用方知道自己是否有能力恢复。

Java 异常体系建立在 `Throwable` 之上，主要分为 `Error` 和 `Exception`。`Error` 表示严重的虚拟机或系统级问题，普通应用代码通常不捕获。`Exception` 表示程序可处理或可声明的异常情况，其中 `RuntimeException` 及其子类属于非受检异常，其他 `Exception` 子类通常属于受检异常。

## 异常体系

`Throwable` 是所有可抛出对象的父类，包含消息、原因、栈轨迹和被抑制异常等信息。`Exception` 用于表示应用层可能处理的失败。`RuntimeException` 常见子类包括 `NullPointerException`、`IllegalArgumentException`、`IllegalStateException`、`IndexOutOfBoundsException` 等。受检异常常见于 I/O、解析、反射等调用方必须正视失败的 API。

受检异常和非受检异常的区别在于编译器是否强制处理或声明。方法可能抛出受检异常时，调用方必须捕获它，或在自己的方法签名中继续 `throws`。非受检异常不需要声明，但仍然会在运行时传播。

```java
String readFirstLine(Path path) throws IOException {
    try (BufferedReader reader = Files.newBufferedReader(path)) {
        return reader.readLine();
    }
}
```

这里 `IOException` 是受检异常，方法签名明确告诉调用方：读取可能失败，调用方必须决定如何处理。

## throw 与 throws

`throw` 用于在方法体内抛出一个异常对象，`throws` 用于在方法签名上声明方法可能向外传播的受检异常。两者位置和含义不同。

```java
void transfer(long amount) {
    if (amount <= 0) {
        throw new IllegalArgumentException("amount must be positive");
    }
}
```

当参数违反方法契约时，抛出 `IllegalArgumentException` 通常比返回特殊值更清晰。调用方传入非法参数属于编程错误，一般不应强制每次捕获。

异常消息应提供定位问题所需的信息，但不应泄露敏感数据。消息要说明哪个条件不满足，而不是只写 `"error"` 或 `"failed"`。如果捕获异常后重新抛出新异常，应保留原异常作为 cause，否则栈信息会断裂。

```java
try {
    return parse(raw);
} catch (NumberFormatException e) {
    throw new IllegalArgumentException("invalid number: " + raw, e);
}
```

## try-catch-finally

`try` 包含可能抛出异常的代码，`catch` 处理匹配的异常类型，`finally` 无论是否发生异常通常都会执行，常用于释放资源或清理状态。多个 `catch` 的顺序必须从具体到宽泛，否则较宽泛类型会吞掉后续分支。

```java
try {
    process(input);
} catch (InvalidInputException e) {
    reportBadInput(e);
} catch (ProcessingException e) {
    retryOrAbort(e);
} finally {
    cleanup();
}
```

`finally` 中应避免 `return`、`throw` 或覆盖原异常的操作。若 `try` 中已经抛出异常，而 `finally` 又抛出另一个异常，原异常可能被遮蔽，排查会变困难。`finally` 更适合做不会失败或失败影响可控的清理。

捕获异常必须有明确目的：恢复、转换、补充上下文、记录并终止当前流程。只捕获后空处理会隐藏错误，使程序进入未知状态。只为了打印日志再继续抛出也要谨慎，否则同一异常可能在多层重复记录。

## try-with-resources

`try-with-resources` 用于自动关闭实现了 `AutoCloseable` 或 `Closeable` 的资源。资源在 `try` 语句头中声明，代码块结束时按声明的逆序自动调用 `close`。它比手写 `finally` 更简洁，也能正确处理关闭期间产生的被抑制异常。

```java
try (InputStream in = Files.newInputStream(path);
     OutputStream out = Files.newOutputStream(target)) {
    in.transferTo(out);
}
```

如果主体代码抛出异常，关闭资源时又抛出异常，主体异常会作为主异常继续传播，关闭异常会被记录为 suppressed exception。可以通过 `getSuppressed()` 查看。这比传统 `finally` 更不容易丢失真正的失败原因。

自定义资源只要实现 `AutoCloseable` 就可以参与 try-with-resources。实现 `close` 时应让关闭操作幂等或至少可预测，避免重复关闭造成额外破坏。

## 受检异常与运行时异常的选择

异常类型选择应围绕“调用方能否合理恢复”和“是否属于方法契约的一部分”。如果失败是外部环境、输入数据或资源状态导致，且调用方可能采取替代方案，受检异常可以迫使调用方正视它。例如文件不存在、网络不可达、解析格式不符合预期等。

如果失败来自编程错误、非法参数、对象状态不满足调用条件，通常使用运行时异常。例如参数为负、对象尚未初始化、索引越界。这类错误应通过修正代码或调用顺序解决，而不是在每个调用点捕获。

受检异常的代价是会污染调用链签名。如果底层异常对上层没有恢复意义，可以在边界处转换成更符合当前抽象层的异常。转换时要保留 cause，并避免把底层实现细节暴露给不该知道的调用方。

```java
class ConfigLoadException extends Exception {
    ConfigLoadException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

自定义异常不宜过多。只有当调用方需要按类型区分处理，或需要表达清晰领域语义时，才值得定义新的异常类型。

## 异常传播与方法契约

方法签名、文档和异常类型共同构成契约。方法应说明哪些输入非法、哪些状态不允许调用、哪些外部失败会向外传播。受检异常在签名中表达，运行时异常则需要通过文档、校验和测试体现。

异常传播不是越早捕获越好。底层代码通常缺少上下文，不知道该重试、提示、降级还是终止。此时应让异常向上传播，直到到达拥有处理策略的边界。相反，如果当前层能补充关键上下文，应在当前层转换异常或追加信息。

不要用异常表达正常业务分支。例如查找某个元素不存在，如果这是常见结果，返回 `Optional`、空集合或明确状态通常比抛异常更合适。异常创建和栈轨迹采集有成本，更重要的是异常会让正常控制流不直观。

## 异常链与上下文保留

异常排查依赖完整上下文。底层异常通常知道具体失败点，例如哪个方法调用失败；上层代码知道业务语义，例如正在加载哪份配置、处理哪条记录。好的异常传播会把这两类信息连接起来，而不是互相覆盖。

```java
Config load(Path path) throws ConfigLoadException {
    try {
        String text = Files.readString(path);
        return parseConfig(text);
    } catch (IOException e) {
        throw new ConfigLoadException("failed to read config: " + path, e);
    } catch (ConfigParseException e) {
        throw new ConfigLoadException("invalid config: " + path, e);
    }
}
```

这里新异常表达当前抽象层的失败，cause 保留底层细节。调用方不需要知道文件读取和解析的所有实现细节，但排查时仍能沿 cause 链看到原始异常。

不要只复制异常消息。`throw new ConfigLoadException(e.getMessage())` 会丢失异常类型、栈轨迹和 cause。也不要把异常消息写得过度宽泛，例如 `"load failed"`。定位问题至少需要知道动作、对象和关键边界条件。

异常链也要控制层级。每一层都包装一次会产生很长的 cause 链，反而增加阅读成本。只有当当前层能提供新的抽象语义或关键上下文时，包装才有价值。否则直接传播更清晰。

## 自定义异常设计

自定义异常应服务于调用方处理，而不是服务于分类癖好。若调用方需要 catch 某类失败并采取不同策略，自定义类型有意义；若只是换个名字再抛出，可能没有必要。

自定义受检异常通常继承 `Exception`，自定义非受检异常通常继承 `RuntimeException`。构造方法至少应提供 message 和 cause 组合，方便保留上下文。

```java
class DuplicateKeyException extends RuntimeException {
    DuplicateKeyException(String message) {
        super(message);
    }

    DuplicateKeyException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

异常类应保持轻量。不要把复杂业务对象、可变集合或资源句柄塞进异常里。异常可能跨层传播、被日志系统记录或被序列化工具处理，携带过多状态会扩大耦合和泄露风险。需要结构化错误信息时，可以在边界层把异常转换成明确的错误结果对象。

命名上，异常类型应以失败语义为核心，例如 `InvalidFormatException`、`ConfigLoadException`、`DuplicateKeyException`。不要用 `CommonException`、`BaseException` 这类无法表达处理策略的名称作为业务失败的主要类型。

## 断言、校验与异常

Java 的 `assert` 用于开发和测试阶段检查内部假设，默认运行时可能不开启。公共参数校验和业务规则不能依赖 `assert`。如果输入来自调用方，应使用显式条件判断并抛出合适异常。

```java
void resize(int width, int height) {
    if (width <= 0 || height <= 0) {
        throw new IllegalArgumentException("size must be positive");
    }
}
```

参数校验应尽量靠近入口。越早失败，异常栈越接近问题来源，对象也越不容易进入半有效状态。构造方法尤其要维护不变量，一旦对象创建成功，调用方应能假设它处于可用状态。

内部状态校验与参数校验不同。参数不合法使用 `IllegalArgumentException`，对象状态不允许当前操作使用 `IllegalStateException`。例如已经关闭的对象再次执行写入，问题主要来自接收者状态，而不是某个参数值。

## 边界层处理策略

底层工具类、中间服务和程序边界对异常的处理方式不同。底层工具类通常不应决定重试、退出或打印提示，它只需准确抛出异常。中间层可以补充上下文、转换抽象层级。最外层边界才适合统一记录、转换为错误响应、展示提示或决定进程是否终止。

不要在每个方法里都 try-catch。到处捕获会打断正常阅读，也容易造成重复日志。只有当前层能做三件事之一时才捕获：恢复并继续；转换为当前层语义；执行必要清理。资源清理优先用 try-with-resources，而不是为了清理把业务逻辑包进大块 catch。

重试也属于异常处理策略。不是所有异常都适合重试。参数错误、格式错误、权限错误通常重试无效；临时资源不可用可能适合有限次数重试。重试代码应限制次数、记录上下文，并避免吞掉最终失败原因。

## 多异常捕获与精确重抛

Java 支持在一个 `catch` 中捕获多个异常类型，适合多个异常共享同一处理策略的场景。多个类型之间用 `|` 分隔，捕获变量在语义上是这些类型的共同父类型，不能被重新赋值。

```java
try {
    load();
} catch (IOException | ConfigParseException e) {
    throw new ConfigLoadException("config load failed", e);
}
```

多异常捕获的前提是处理方式确实相同。如果不同异常需要不同消息、不同恢复策略或不同指标统计，分开捕获更清晰。不要为了减少代码行数把语义不同的失败合并在一起。

精确重抛是编译器对异常流的分析能力。捕获较宽泛的异常后，如果没有把捕获变量改成别的异常对象，编译器可以推断重新抛出的具体受检异常类型。这让一些包装或清理代码既能写得集中，又不会把方法签名扩大成过宽的 `throws Exception`。不过在公共 API 中，仍应优先显式声明有意义的异常类型，而不是让调用方面对模糊的顶层异常。

## 异常与返回值的边界

异常和返回值都能表达失败，但适用场景不同。异常适合表示方法无法完成其承诺，尤其是失败路径相对少见、需要携带栈上下文或应由上层统一处理的情况。返回值适合表达正常结果空间的一部分，例如查询不到数据、校验未通过、条件不满足。

```java
Optional<User> findUser(String id) {
    if (id == null || id.isBlank()) {
        throw new IllegalArgumentException("id must not be blank");
    }
    return repository.findById(id);
}
```

这个例子中，空白 id 是调用方违反契约，使用异常；用户不存在是查询的正常结果，使用 `Optional`。把二者都做成异常会让调用方被迫捕获常见分支；把二者都做成返回值又会削弱参数契约。

有些 API 会使用结果对象同时携带成功值和错误信息。这适合批量处理、需要收集多个错误、或不希望单个失败中断全流程的场景。但结果对象也会让调用方每一步都显式检查状态。选择异常还是返回值，本质上是在传播便利、类型清晰和流程可读性之间取舍。

## 栈轨迹与排查成本

异常栈轨迹记录了异常创建和抛出路径，是排查问题的重要证据。频繁创建异常会有成本，因此异常不适合普通循环控制。更重要的是，异常路径会绕过正常返回路径，如果代码大量依赖异常驱动流程，读者很难沿着主流程理解程序。

栈轨迹越接近真实失败点，定位越容易。捕获后重新抛出新异常时保留 cause，就是为了不丢失原始失败点。相反，捕获后只抛出一个没有 cause 的新异常，会让排查者只能看到包装层，无法看到真正的文件、解析位置或调用点。

异常消息和栈轨迹互补。栈轨迹告诉“在哪里失败”，消息告诉“为什么这个输入或状态失败”。只靠栈轨迹可能缺少业务上下文，只靠消息可能缺少代码路径。高质量异常通常同时具备二者。

## 常见异常类型

`NullPointerException` 表示对空引用执行实例访问。它通常说明调用方传入了不允许为空的值，或对象初始化不完整。公共方法可以用 `Objects.requireNonNull` 在入口处快速失败。

`IllegalArgumentException` 表示参数值不满足要求。`IllegalStateException` 表示对象当前状态不允许执行该操作。两者的区别在于问题来自“传入的值”还是“接收者当前状态”。

`UnsupportedOperationException` 表示对象不支持某操作。它常见于不可变集合或只读视图。公开 API 若可能抛出该异常，应让调用方在文档或类型设计上提前知道。

`ClassCastException` 表示类型转换失败。大量显式强转通常说明类型设计不够明确，可以考虑泛型、接口或多态替代。

`ConcurrentModificationException` 表示迭代过程中检测到不允许的结构性修改。它是快速失败机制，不是并发安全保证。

## 常见误区与实践要点

不要捕获 `Exception` 后简单忽略。空 catch 会把错误变成沉默失败，后续代码可能基于错误状态继续运行。若确实允许失败，应明确写出降级策略或注释说明原因。

不要把所有异常都包装成同一个运行时异常。这样会丢失调用方按类型处理的能力。异常抽象应与当前层语义一致，而不是把所有失败都压平成“系统错误”。

不要在循环中用异常控制结束条件。解析、查找和边界判断应优先使用普通条件分支。异常适合少见且异常的路径。

不要在 `finally` 中返回值。它会覆盖 `try` 或 `catch` 中的返回，也可能吞掉异常。资源释放优先使用 try-with-resources。

记录异常时应记录异常对象本身，而不是只记录消息。只记录 `e.getMessage()` 会丢失栈轨迹和 cause 链。日志边界也要克制，避免同一异常在每一层都重复输出。

## 异常的执行语义

异常不是一种“带标签的返回值”，而是一条特殊的控制流。当代码执行到 `throw` 时，当前表达式和当前语句不会继续按普通路径完成，运行时会创建或接收一个 `Throwable` 对象，然后沿调用栈向外寻找第一个能够匹配该异常类型的 `catch`。如果当前方法没有匹配的处理器，当前方法会异常完成，调用者也会以同样方式继续向外传播。这个过程通常被称为栈展开。栈展开不仅改变了控制流，也会触发已经进入的 `finally` 块和 try-with-resources 资源关闭逻辑，因此异常传播与资源生命周期紧密相关。

匹配规则基于异常对象的实际类型和 `catch` 参数类型之间的赋值兼容关系。也就是说，`catch (IOException e)` 可以捕获 `IOException` 及其子类，但不能捕获同级的其他异常。多个 `catch` 从上到下匹配，一旦命中某个分支，后续分支不会再执行。正因为如此，较具体的异常类型必须写在前面，较宽泛的父类型必须写在后面。编译器会拒绝明显不可达的 `catch`，例如先捕获 `Exception` 再捕获 `IOException`，因为后者永远没有机会执行。

异常完成也会影响返回值语义。如果 `try` 块中已经计算出返回值，但在方法真正返回之前还需要执行 `finally`，那么 `finally` 仍然可以改变对象状态、抛出新的异常，甚至通过不推荐的 `return` 覆盖原返回路径。理解这一点有助于解释一些看似反直觉的行为：`finally` 不是“在方法结束之后”执行，而是在当前控制流离开 `try` 或 `catch` 之前执行。它参与控制流，而不是旁路通知。

异常对象本身也有生命周期。通常异常在失败点附近被创建，此时会采集栈轨迹。随后它可能被捕获、记录、包装、重新抛出，或者作为 cause 被挂到另一个异常对象上。重新抛出同一个异常对象不会重新生成一个新的原始失败点，栈轨迹仍主要反映它最初被创建或填充的位置。若人为复用异常对象，多个失败会共享同一份栈信息，诊断价值会明显下降。因此除极少数性能敏感且语义明确的场景外，不应把异常对象当作可复用常量。

还要区分“抛出异常”和“方法声明异常”。`throws` 只是编译期契约的一部分，它不负责抛出任何东西；真正改变运行路径的是 `throw`、被调用方法内部的异常完成，或虚拟机在运行某些字节码时抛出的运行时异常。一个方法可以声明某个受检异常但当前实现并不抛出它，这在接口演进、模板方法和多态实现中并不少见。反过来，方法即使没有声明运行时异常，也仍然可能因为空引用、非法状态、算术错误等原因异常完成。

## 编译期规则与多态边界

Java 对受检异常的强制处理是编译期规则，不是运行时规则。编译器会根据静态类型、方法签名和控制流分析，要求调用方捕获或声明可能传播的受检异常。运行时不会因为一个异常“没有写在 throws 中”就阻止它传播；对于运行时异常和错误，签名本来就不强制声明。这个设计把一部分失败契约提前到编译阶段，让 API 作者可以明确告诉调用方：这个操作的失败不是偶然细节，而是调用者必须纳入设计的可能结果。

重写方法时，异常声明受到里氏替换原则的约束。子类重写父类方法不能比父类方法声明更宽的受检异常，因为调用方可能只持有父类引用。如果父类方法声明 `throws IOException`，子类可以不声明受检异常，也可以声明 `IOException` 的子类，但不能新增一个无关的 `SQLException`。否则，使用父类类型编写的代码会在替换成子类实例后突然被迫处理新的受检异常，破坏多态调用的稳定性。运行时异常不受这个规则强制限制，但公开 API 仍应通过文档说明可能的契约违背。

接口方法同样遵守这个约束。一个实现类不能随意把底层实现细节的受检异常泄露到接口之外，除非接口本身就把这种失败建模为契约的一部分。这里体现了异常设计中的抽象边界：接口面向调用方描述“这个能力可能怎样失败”，实现类负责把数据库、文件、网络、解析等具体失败转换为接口语义允许的异常。若接口过早暴露底层异常，后续更换实现会变困难；若接口完全隐藏可恢复失败，调用方又可能失去合理处理机会。

构造方法也可以抛出异常。构造失败意味着对象没有成功建立不变量，调用方不能拿到一个半初始化对象继续使用。构造方法中抛出异常时，已经完成初始化的字段和已经创建的辅助对象可能需要清理，因此构造过程不应随意获取复杂资源；如果确实需要，应尽量在局部 try-with-resources 中完成，或者使用工厂方法把校验、资源申请和对象创建分离。一个好对象的边界应当清楚：要么构造成功并满足不变量，要么失败并给出明确原因。

静态初始化失败也属于异常机制的重要边界。类初始化代码如果抛出异常，运行时会把它包装成初始化相关的错误或异常状态，后续使用该类也可能继续失败。静态初始化适合做简单、确定、失败概率低的工作，不适合执行复杂 I/O、依赖外部环境或包含大量可恢复失败的逻辑。因为类初始化发生的时机可能由首次主动使用触发，调用方未必预期一个看似普通的类访问会引发复杂失败。

## finally 的细节与风险

`finally` 的价值在于把清理逻辑和成功失败路径解耦，但它不是无风险的万能收尾。最重要的风险是覆盖原始结果。若 `try` 抛出异常，`finally` 又抛出异常，最终向外传播的是 `finally` 的异常，原始异常可能丢失。若 `try` 准备返回一个值，`finally` 中又执行 `return`，最终返回值会被 `finally` 覆盖。若 `finally` 修改了即将返回的可变对象，调用方看到的是修改后的对象状态。这些行为都符合法律语义，却很容易制造排查成本。

因此，`finally` 中的代码应尽量短、稳定、可预测。释放锁、恢复标志位、关闭简单资源、清理临时状态是合适的；复杂业务逻辑、再次调用可能失败的远程操作、隐藏式返回值修正都不合适。清理动作如果本身可能失败，要考虑失败的优先级。通常主体异常比清理异常更重要，因为主体异常解释了操作为什么没有完成；清理异常解释的是善后过程又发生了什么。try-with-resources 对这个优先级做了更合理的默认处理：主体异常为主，关闭异常作为 suppressed 保留下来。

`finally` 也不能保证在所有场景下执行。进程被强制终止、虚拟机崩溃、底层环境直接中断运行时，普通 Java 语义没有机会继续展开栈。应用代码不应把关键数据持久化、跨进程事务确认等不可丢失动作完全寄托在 `finally` 上。`finally` 负责当前线程当前调用栈内的清理，而不是提供系统级可靠性保证。需要更强保证时，应使用事务、日志、幂等协议或外部一致性机制。

从可读性看，现代 Java 代码中手写 `finally` 的频率应低于早期代码。资源关闭优先用 try-with-resources；锁释放可以用结构化的 try-finally，但要让临界区尽量小；临时状态恢复可以考虑把状态改变封装成实现 `AutoCloseable` 的作用域对象。选择哪种写法的核心标准不是“少写几行”，而是异常路径是否自然、清理顺序是否明确、失败信息是否不会被覆盖。

## try-with-resources 的关闭顺序与 suppressed

try-with-resources 的一个细节是关闭顺序与声明顺序相反。先声明的资源通常依赖后声明资源之外的更大上下文，或者后续资源可能包装前面的资源。逆序关闭能够更接近栈式资源管理：最后打开的先关闭，最先打开的最后关闭。对嵌套包装流、连接和语句、临时作用域对象等场景，这种顺序更符合直觉，也更不容易在内层资源仍需外层资源时提前关闭外层。

被抑制异常是 try-with-resources 的关键改进。传统手写 `finally` 中，如果主体代码和关闭代码都失败，后发生的关闭异常很容易覆盖先发生的主体异常。try-with-resources 会把主体异常作为主异常传播，把关闭异常加入主异常的 suppressed 列表。这样调用方在日志中通常仍能看到真正导致操作失败的主体异常，同时也不会完全丢失关闭失败的信息。若主体代码没有异常而关闭失败，关闭异常会直接成为主异常。

自定义资源实现 `AutoCloseable` 时，`close` 方法的异常类型也会影响调用方体验。`AutoCloseable.close()` 声明的是 `throws Exception`，而 `Closeable.close()` 缩窄为 `throws IOException`。如果资源关闭只会出现某类具体失败，可以在实现中声明更窄的异常，避免迫使调用方处理过宽的 `Exception`。如果关闭失败对调用方没有恢复意义，也可以选择在内部记录并不抛出，但这种选择必须谨慎，因为它会隐藏资源释放问题。

资源对象还应考虑关闭幂等性。幂等关闭意味着多次调用 `close` 不会产生不同结果，也不会破坏已经释放的状态。标准库中许多资源尽量提供这种行为，但并非所有自定义资源都天然如此。若资源会被多个路径持有，或者可能在异常路径和普通路径中都触发关闭，幂等性能显著降低错误处理复杂度。即使无法完全幂等，也应明确关闭后的对象状态：后续操作是抛出 `IllegalStateException`，还是静默无效，不能让行为随机依赖内部残留状态。

try-with-resources 还支持在语句头使用已经声明且事实上不可变的资源变量。这个能力让资源创建和资源使用可以分开表达，但也要求读者清楚资源所有权：谁负责关闭，关闭后是否还会被其他代码使用。资源所有权不清时，自动关闭反而可能制造悬空状态。凡是一个方法接收外部传入的流、连接或缓冲对象，都要谨慎决定是否关闭它；很多时候“谁创建，谁关闭”是更简单的约定，除非 API 文档明确转移了所有权。

## 异常消息、cause 与诊断信息

异常消息应服务于定位，而不是服务于情绪表达。好的消息通常包含动作、对象、约束和关键值，例如“读取配置文件失败：路径不可访问”“参数 size 必须大于 0，实际为 -1”。坏消息常见于只写“error”“failed”“invalid”，或者把所有上下文都留给调用方猜。消息不是越长越好，尤其不能把敏感数据、完整凭证、大体量文本或用户隐私直接拼进异常。异常很可能进入日志、监控、错误报告和测试输出，消息内容必须假设会被更广泛地传播。

cause 链负责保留技术因果，消息负责补充当前层语义。包装异常时，如果新异常只是重复旧异常消息而不增加任何上下文，那么包装价值很低；如果新异常提供了当前操作的目标对象、抽象层级和业务含义，同时保留 cause，那么调用方既能看到高层失败，也能向下追溯低层原因。一个常见判断是：当前层是否能回答“我当时正在做什么”。能回答，就可以包装；不能回答，直接传播往往更好。

异常消息的边界还涉及国际化和用户展示。底层异常消息通常面向开发者和运维人员，不一定适合直接展示给最终用户。公共库和基础模块更应该产出稳定、准确、面向诊断的英文或项目约定语言消息；用户界面或交互边界可以再把异常转换成可理解的提示。把用户可见文案直接写入底层异常，会让异常类型承担展示职责，也会使调用方难以根据环境调整语言、语气和详细程度。

有时异常需要携带结构化信息，例如错误码、字段名、失败位置、可重试标记。可以通过自定义异常增加只读字段，但要克制。结构化字段应当是稳定契约的一部分，而不是把所有现场对象都塞进异常。字段越多，异常类型越像一个跨层数据传输对象，耦合也越强。若调用方主要通过结构化结果进行流程控制，可能应重新评估是否使用结果对象更合适。

## API 设计中的异常契约

设计公共方法时，应先想清楚方法承诺是什么。异常表达的是“方法无法完成承诺的原因”。如果方法承诺“根据 id 查找对象”，对象不存在也许是正常结果；如果方法承诺“必须取得存在的对象”，对象不存在就是异常。两个方法可以操作同一份数据，却因为承诺不同而使用不同的失败表达。异常设计不是从底层错误类型出发，而是从 API 语义出发。

公共 API 的异常契约应保持稳定。新增受检异常会迫使调用方修改代码，是明显的兼容性变化；新增运行时异常虽然不破坏编译，但可能改变调用方的运行时行为和错误处理预期。异常类型、抛出时机和消息中的关键结构都可能被调用方依赖。尤其是库代码，不应轻易把内部实现的异常类型直接暴露出去，否则一次实现替换就可能变成 API 破坏。

方法参数校验属于异常契约中最常见的一部分。对公开方法来说，非法参数应尽早失败，并指出违反了哪条约束。`NullPointerException`、`IllegalArgumentException`、`IllegalStateException` 的选择要保持一致：空值违反非空要求时，可以使用 `Objects.requireNonNull` 产生清晰的空值失败；普通值域错误用 `IllegalArgumentException`；对象生命周期或状态不允许当前操作时用 `IllegalStateException`。一致性比单个异常类型的偏好更重要，因为调用方会从一致模式中学习 API 的使用方式。

集合、迭代器和可选操作提供了另一个设计示例。一个只读视图如果收到修改请求，抛出 `UnsupportedOperationException` 比静默忽略更清楚，因为调用方的操作承诺没有被完成。一个迭代器没有下一个元素时，`next` 抛出 `NoSuchElementException`，而 `hasNext` 用布尔值表达普通状态。这里不是简单地“失败就异常”，而是区分检查式查询和承诺式动作：查询方法告诉你状态，动作方法在前置条件不满足时失败。

对于批量处理 API，异常契约需要额外斟酌。如果处理一千条记录时第三条失败，整个方法是否应立即失败，还是收集错误后继续处理？如果失败之间相互独立，结果对象可能比第一个异常更有用；如果任何一条失败都会破坏整体一致性，立即抛出异常更合适。异常机制天然偏向“失败即中断当前承诺”，而批量场景常常需要“尽量完成并报告所有问题”。两者不能混用得含糊不清。

## 分层捕获与转换

异常处理的层次感非常重要。最底层代码接近具体失败点，知道哪个系统调用、哪个解析步骤、哪个参数检查失败，却未必知道业务上应如何响应。上层边界知道当前操作的意图、用户请求或任务目标，却未必知道底层每个失败细节。好的异常流应让底层保留精确原因，让上层补充语义并决定策略。

转换异常时要避免两个极端。一个极端是完全不转换，把所有底层异常一路暴露给最高层，导致高层依赖低层实现细节。另一个极端是每层都转换，把一条简单失败包成十几层 cause 链，阅读者需要穿过大量同义包装才能看到重点。合适的转换点通常出现在抽象层边界：从文件格式进入配置语义，从底层通信进入仓储语义，从内部校验进入公共 API 语义。边界内的同层方法可以直接传播，边界处再统一转换。

捕获后记录也应有层次。一般来说，一个异常只应在负责最终处理的边界记录一次。中间层如果只是补充上下文并继续抛出，通常不需要记录；否则日志中会出现同一异常的多份栈轨迹，既增加噪声，又让排查者误以为发生了多次失败。中间层确实需要记录时，应说明它记录的是中间事件而不是最终错误，并避免使用过高严重级别。

恢复是捕获异常的另一个合理理由。恢复不是“吞掉异常继续执行”，而是有明确替代路径。例如读取可选配置失败后使用默认配置，解析某条独立记录失败后把它放入错误列表，临时资源不可用时切换到备用资源。恢复策略必须保证后续状态仍然有效。如果捕获后既没有恢复状态，也没有向外传播失败，只是让代码继续往下跑，那么异常被隐藏后通常会变成更难诊断的二次故障。

## 并发与异步边界中的异常

异常沿当前线程的调用栈传播。这个事实在并发和异步编程中尤其关键。在线程 A 中抛出的异常不会自动跳到线程 B 的调用栈中被捕获。如果一个任务被提交到执行器，任务内部抛出的异常通常会被执行框架捕获并保存在任务结果中，或者交给线程的未捕获异常处理器。调用方若只是提交任务而不检查结果，就可能错过任务失败。并发代码中的异常处理必须明确失败如何回到发起者、如何记录、如何影响其他任务。

`Future` 的异常会在获取结果时重新表达。任务内部抛出的异常通常被包装在 `ExecutionException` 中，调用方需要查看 cause 才能知道原始失败。这个包装不是多余的，因为 `get` 本身还可能因为等待线程被中断、任务取消等原因失败。不同失败来源需要不同策略：任务逻辑失败、等待被中断、任务被取消，并不是同一种情况。把它们都当成普通运行时异常重新抛出，会让并发控制语义丢失。

`CompletableFuture` 这样的异步组合 API 则把异常作为阶段结果的一部分传播。某个阶段异常完成后，依赖它的后续阶段可能被跳过，异常处理阶段可以恢复、转换或继续传播。这里的异常已经不再只是线程调用栈上的跳转，而是异步结果状态。编写这类代码时，要分清“转换成功值”的阶段和“处理失败”的阶段，避免在异常处理回调中返回一个看似成功但语义无效的默认值。

线程的未捕获异常处理器是最后防线，不是常规恢复机制。它适合统一记录那些逃出任务边界的失败，帮助定位遗漏的异常处理，但不适合承担业务补偿。若任务失败需要通知调用方、触发重试或更新任务状态，应在任务框架和结果协议中显式设计，而不是指望未捕获异常处理器从全局位置推断上下文。

并发代码还会放大异常安全问题。一个方法在持有锁、修改共享集合或更新多个相关字段时抛出异常，可能留下部分更新状态。异常安全的目标是：失败后对象仍保持可用不变量，锁一定释放，共享状态不会处于半提交状态。常见做法包括先在局部变量中完成可能失败的计算，再一次性发布结果；使用 try-finally 保证锁释放；把多步更新封装在能够回滚或重建状态的边界内。

## 性能、成本与异常滥用

异常的性能成本主要来自对象创建、栈轨迹填充和非局部控制流。现代运行时对普通路径有很多优化，但异常路径通常不是优化重点。偶尔失败时，这些成本完全可以接受，因为异常提供的诊断信息更重要；如果一个循环每次迭代都依赖异常跳出或判断状态，成本和可读性都会变差。性能不是反对使用异常的理由，滥用异常表达普通分支才是问题。

不要把异常作为解析控制流的默认工具。例如解析一批字符串时，如果格式错误是常见输入，就应先做轻量检查、使用能够返回失败状态的解析接口，或把每条记录的错误收集为结果，而不是对每个无效字段都抛出再捕获异常。相反，如果某个格式在契约上必须合法，偶发不合法代表上游严重错误，那么抛出异常能更快暴露问题。判断标准仍然是失败是否属于方法承诺之外的异常情况。

某些低层框架会为了性能创建不填充栈轨迹的异常，或者重写 `fillInStackTrace`。普通业务代码很少需要这样做。没有栈轨迹的异常会显著降低诊断价值，除非它是内部协议的一部分，并且不会越过清晰边界。对于绝大多数应用和库来说，清楚的异常类型、消息和 cause 比节省一次栈采集更重要。

性能优化还要避免过早捕获。捕获异常本身不是主要问题，问题是为了“防止抛出太远”而在小方法内反复捕获、包装、记录。这样不仅增加对象和日志成本，也破坏调用栈的清晰度。让异常自然传播到有处理策略的位置，通常既更快，也更容易诊断。

## 测试异常行为

异常是方法契约的一部分，因此也需要测试。测试不应只验证“抛出了某个异常”，还应验证关键消息、cause、状态不变量和资源清理。对于参数校验，测试应覆盖非法边界值，确认异常类型与 API 约定一致。对于包装异常，测试应确认原始异常没有丢失。对于恢复逻辑，测试应确认捕获异常后系统进入了预期的可用状态，而不是仅仅没有继续抛出。

测试异常消息时要把握粒度。完全匹配整段消息可能导致测试过脆，稍微调整措辞就失败；完全不检查消息又可能错过上下文缺失。更稳妥的做法是检查消息中稳定的关键片段，例如参数名、资源标识或错误类别。公共库若承诺错误码或结构化字段，则应优先测试这些稳定字段，而不是自然语言消息。

异常路径的测试还应关注副作用。一个写入方法在中途失败后，目标对象是否被部分修改？一个批量操作失败后，已经成功的项是否被保留，失败的项是否可追踪？一个持有资源的方法失败后，资源是否关闭？这些问题比异常类型本身更能体现异常安全。只测试 happy path 的代码，很容易在真实失败场景中暴露状态污染。

对并发和异步任务，测试要显式检查任务结果。提交任务后不等待、不读取结果，就无法断言任务内部异常。使用 `Future` 时应调用 `get` 并检查包装异常；使用 `CompletableFuture` 时应检查异常完成状态或相关处理阶段。否则测试可能在任务已经失败的情况下仍然通过，只因为异常发生在另一个线程而没有回到测试线程。

## 常见实践判断

当你准备抛出异常时，可以先问：当前方法承诺是否已经无法完成？如果答案是否定的，只是遇到了一个普通分支，那么返回值、状态对象或回调可能更合适。如果答案是肯定的，再问：调用方是否有能力恢复？如果调用方必须面对外部条件导致的失败，受检异常可以让契约更明确；如果这是调用方违反前置条件或对象状态错误，运行时异常更符合语义。

当你准备捕获异常时，可以问：当前层能做什么？能恢复，就恢复并保证状态有效；能补充抽象语义，就包装并保留 cause；能清理资源，就清理但尽量不覆盖主异常；什么都不能做，就不要捕获。很多坏的异常代码都来自“看到异常就 catch”的冲动。异常处理不是语法义务，而是策略选择。

当你准备自定义异常时，可以问：调用方是否会按这个类型做不同处理？这个类型是否表达了当前抽象层的失败语义？它是否比现有标准异常更清楚？如果只是为了让错误看起来“属于本项目”，自定义异常价值有限。标准异常已经覆盖了大量契约违背场景，领域异常应保留给真正需要领域语义和分层隔离的失败。

当你准备记录异常时，可以问：这里是不是最终处理边界？日志是否包含异常对象、关键上下文和唯一定位信息？是否会重复记录同一个 cause 链？是否泄露敏感数据？高质量日志不是越多越好，而是让排查者能从一处错误记录中看到足够上下文，并沿栈和 cause 找到真实失败点。

异常机制最终服务于代码的诚实性。一个方法不能完成承诺时，应诚实地失败；一个层次不知道如何处理时，应诚实地传播；一个边界决定降级时，应诚实地说明降级后的语义。隐藏失败、伪装成功、丢失 cause、把普通分支包装成异常，都会让系统在短期看起来更“顺”，长期却更难维护。

## 小结

Java 异常机制的核心是用类型化方式表达失败，并把处理责任交给最合适的层。受检异常强调调用方必须面对可恢复或外部条件相关的失败，运行时异常强调编程错误和契约违背；二者不是高低之分，而是契约表达方式不同。`throw` 改变运行时控制流，`throws` 描述编译期契约，`catch` 只有在当前层具备恢复、转换、清理或最终处理能力时才有意义。

可靠的异常代码应保留上下文和 cause，避免吞错，优先使用 try-with-resources 管理资源，并让异常类型与方法抽象层级一致。异常消息要能帮助定位，但不能泄露敏感数据；包装异常要增加新的语义，而不是重复旧消息；日志应集中在边界处记录，避免同一失败被多层重复输出。真正成熟的异常设计，不是让代码看起来没有错误，而是在错误发生时仍然能保持状态清楚、责任清楚、诊断路径清楚。

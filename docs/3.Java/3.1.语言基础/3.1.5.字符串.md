# 3.1.5 字符串

## 定位

字符串是 Java 中最常用的基础类型之一，用于表达文本、标识、协议字段、配置项和格式化结果。Java 的字符串模型包含不可变对象、字符编码、常量池、拼接优化、比较规则和可变构建器等知识点。理解字符串不仅能避免 `==`、乱码和性能误区，也能帮助写出边界清晰的文本处理代码。

`String` 是引用类型，不是基本类型。它表示一段不可变字符序列。不可变意味着字符串对象创建后内容不会改变，所有看似修改字符串的方法都会返回新字符串或复用已有字符串。不可变性让字符串可以安全共享、缓存哈希值、作为映射键，并参与字符串常量池机制。

## String 的不可变性

`String` 的常用方法如 `trim`、`replace`、`substring`、`toLowerCase` 都不会修改原对象，而是返回结果字符串。

```java
String text = " Java ";
text.trim();
System.out.println(text); // 仍然是 " Java "

String normalized = text.trim();
System.out.println(normalized); // "Java"
```

不可变对象的好处是容易推理。一个字符串传给多个方法后，调用方不用担心其中某个方法改变了它的内容。字符串作为 `HashMap` 键也很可靠，因为键的内容不会在放入后改变。

不可变不等于变量不可重新赋值。`String s = "a"; s = "b";` 中，变量 `s` 指向了另一个字符串对象，原来的 `"a"` 没有被修改。

## 创建方式与常量池

字符串字面量会进入字符串常量池。相同字面量通常复用同一个 `String` 实例。

```java
String a = "java";
String b = "java";
System.out.println(a == b); // true
```

使用 `new String("java")` 会创建新的字符串对象，变量指向的新对象与常量池中的字面量对象不是同一个引用。

```java
String a = "java";
String b = new String("java");
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true
```

`intern()` 会返回字符串在常量池中的规范引用。它适合少数需要引用规范化的场景，但不应作为普通字符串比较手段。业务代码比较文本内容应使用 `equals`。

编译期常量拼接可能被编译器合并。

```java
String a = "ja" + "va";
String b = "java";
System.out.println(a == b); // true
```

但如果拼接中包含运行期变量，结果通常是运行期创建的新字符串。

```java
String prefix = "ja";
String a = prefix + "va";
String b = "java";
System.out.println(a == b); // false
```

理解常量池有助于解释引用比较结果，但不应鼓励依赖引用身份判断内容。字符串内容比较始终优先 `equals`。

## 比较与判空

`==` 比较引用是否相同，`equals` 比较内容是否相同。忽略大小写比较可以使用 `equalsIgnoreCase`。排序或大小关系比较可以使用 `compareTo` 或 `compareToIgnoreCase`。

当变量可能为 `null` 时，可以把确定非空的常量放在左侧：

```java
if ("yes".equals(answer)) {
    System.out.println("accepted");
}
```

也可以先显式判空。选择哪种写法取决于团队风格和可读性。重要的是不要在可能为 `null` 的引用上直接调用方法。

空字符串 `""` 和 `null` 含义不同。`null` 表示没有字符串对象，空字符串表示有一个长度为 0 的字符串。空白字符串如 `"   "` 又不同于空字符串。Java 11 起 `isBlank()` 可以判断是否为空或只包含空白字符，`isEmpty()` 只判断长度是否为 0。

```java
"".isEmpty();       // true
" ".isEmpty();      // false
" ".isBlank();      // true
```

## 拼接与构建

少量字符串拼接可以直接使用 `+`，编译器会进行优化。在循环中反复拼接字符串可能创建大量中间对象，应使用 `StringBuilder`。

```java
StringBuilder builder = new StringBuilder();
for (String item : items) {
    if (builder.length() > 0) {
        builder.append(", ");
    }
    builder.append(item);
}
String result = builder.toString();
```

`StringBuilder` 是可变字符序列，非线程安全，但在局部变量中构建字符串通常没有并发问题。`StringBuffer` 与 `StringBuilder` 类似，但方法带同步，历史包袱较重。没有共享并发需求时优先 `StringBuilder`。

格式化字符串可以使用 `String.format`、`formatted` 或 `Formatter`。格式化适合可读性要求高、占位符清晰的场景，但大量高频调用要注意成本。简单拼接不一定比格式化差，关键是语义清晰。

```java
String message = "user %s has %d points".formatted(name, points);
```

多行文本可以使用文本块。文本块适合 SQL、JSON 模板、帮助文本等多行字面量，但仍应注意缩进和转义规则。

```java
String sql = """
        select id, name
        from users
        where active = true
        """;
```

## 字符、码点与 Unicode

`String.length()` 返回的是 UTF-16 代码单元数量，不一定等于用户感知的字符数量。Java 的 `char` 是 16 位，能够表示基本多文种平面内的字符，但有些 Unicode 字符需要两个 `char` 组成一个代理项对。

```java
String s = "𝄞";
System.out.println(s.length());          // 2
System.out.println(s.codePointCount(0, s.length())); // 1
```

如果处理的是普通英文、数字和常见符号，`charAt` 通常够用。若处理完整 Unicode 文本、表情符号或需要按码点遍历，应使用 `codePoints()` 或相关 API。更复杂的“用户感知字符”还涉及组合字符和字形簇，不能简单用 `char` 或码点数量判断显示长度。

大小写转换也与 Locale 有关。`toLowerCase()` 使用默认 Locale，某些语言环境下可能产生非预期结果。处理协议字段、内部标识和大小写无关的技术键时，应显式使用 `Locale.ROOT`。

```java
String key = raw.toLowerCase(Locale.ROOT);
```

## 编码与字节转换

字符串是字符序列，字节数组是编码后的二进制数据。二者转换必须指定字符集。若不指定，可能使用平台默认字符集，导致不同环境结果不一致。

```java
byte[] bytes = text.getBytes(StandardCharsets.UTF_8);
String decoded = new String(bytes, StandardCharsets.UTF_8);
```

文本文件、网络协议、数据库字段和外部系统交互都涉及编码。写入和读取必须使用同一编码约定。乱码通常不是字符串对象本身的问题，而是在字节与字符转换时使用了不一致或错误的字符集。

不要把任意二进制数据直接当字符串处理。若需要用文本形式承载二进制，应使用 Base64、十六进制等明确编码。

## 正则与分割

`String` 提供 `matches`、`replaceAll`、`split` 等基于正则表达式的方法。它们的参数不是普通字符串，而是正则模式。点号、竖线、反斜杠等字符在正则中有特殊含义，作为普通字符使用时需要转义。

```java
"a.b".split("\\."); // ["a", "b"]
```

高频正则匹配应考虑预编译 `Pattern`，避免重复解析模式。简单固定字符串替换可以使用 `replace`，不必使用 `replaceAll`。

`split` 默认会丢弃末尾空字符串。如果需要保留尾部空字段，可以传入负数 limit。

```java
"a,b,".split(",").length;     // 2
"a,b,".split(",", -1).length; // 3
```

## 子串、裁剪与规范化

`substring` 用于截取字符串片段，起始下标包含，结束下标不包含。下标基于 UTF-16 代码单元，不是用户感知字符。截取前应确认边界合法，否则会抛出 `StringIndexOutOfBoundsException`。

```java
String text = "abcdef";
String part = text.substring(1, 4); // "bcd"
```

`trim` 会移除首尾小于等于空格的传统空白字符，`strip` 基于 Unicode 空白定义，语义更完整。Java 11 起还提供 `stripLeading`、`stripTrailing`、`repeat`、`lines` 等方法。处理外部输入时，应先明确要移除的是 ASCII 空格、传统空白，还是 Unicode 空白。

规范化通常包括去除首尾空白、统一大小写、统一编码形式、处理空值和校验长度。规范化应尽量集中在输入边界完成，避免系统内部一会儿使用原始值、一会儿使用规范化值。对于标识符、键名和协议字段，使用 `Locale.ROOT` 做大小写转换更稳定。

```java
String normalizeKey(String raw) {
    if (raw == null) {
        throw new IllegalArgumentException("key required");
    }
    String key = raw.strip().toLowerCase(Locale.ROOT);
    if (key.isEmpty()) {
        throw new IllegalArgumentException("key must not be blank");
    }
    return key;
}
```

## StringJoiner、join 与收集

当目标是把多个片段按分隔符连接时，`String.join` 或 `StringJoiner` 比手写分隔符判断更清晰。集合流也可以使用 `Collectors.joining`。

```java
String csv = String.join(",", List.of("a", "b", "c"));

StringJoiner joiner = new StringJoiner(", ", "[", "]");
joiner.add("red");
joiner.add("green");
String result = joiner.toString(); // "[red, green]"
```

`String.join` 要求元素不能为 `null`，否则会抛出异常。若集合中可能有空值，应先过滤、替换或明确失败。分隔符连接常见于日志、展示和简单文本协议，但不应替代真正的结构化序列化格式。

在流式处理中，`Collectors.joining(delimiter, prefix, suffix)` 可以完成同类工作。需要注意的是，流让数据变换紧凑，但复杂逻辑不应为了链式写法牺牲可读性。

## 性能与内存边界

字符串性能问题通常来自大量临时对象、重复正则编译、无意义的格式化、过度拆分和大文本复制。少量字符串操作没有必要过早优化，但处于循环、批处理和解析器内层的代码应更加谨慎。

`StringBuilder` 可以设置初始容量，减少扩容复制。若大致知道结果长度，可以在构造时传入容量。

```java
StringBuilder builder = new StringBuilder(items.size() * 16);
```

`substring` 在现代 Java 中会创建新的字符串内容存储，不再长期共享原大字符串的底层数组。这样避免小子串意外保留大文本，但大量截取仍会产生对象和复制成本。解析大文本时，若性能敏感，可以考虑按索引扫描并延迟创建字符串。

正则 API 功能强，但不是所有文本处理都需要正则。固定前缀用 `startsWith`，固定后缀用 `endsWith`，包含判断用 `contains`，固定替换用 `replace`。这些方法语义更直接，也避免正则转义和编译成本。

## 安全与敏感文本

字符串不可变，因此敏感内容一旦放入 `String`，无法主动清除其内部内容，只能等待垃圾回收。对于密码、密钥等需要尽快擦除的内容，某些场景会使用 `char[]` 或专门的安全容器，并在使用后覆盖数组。不过在许多高层 API 中仍不可避免会接触字符串，关键是减少日志输出和长期保存。

日志中不要直接输出包含令牌、密码、身份证明或隐私信息的字符串。异常消息也应避免拼接敏感原文。字符串处理工具方法应区分“用于展示的脱敏值”和“用于计算的真实值”，不要把脱敏后的值再传入业务计算。

文本长度限制也是安全边界。外部输入字符串如果不限制长度，可能造成内存压力、正则灾难性回溯或日志膨胀。解析前先限制大小，正则模式避免高风险写法，是字符串处理的基础防护。

## 字符串与数值、日期转换

字符串经常作为外部输入格式，但程序内部不应长期用字符串保存已经有明确类型的数据。数字应尽早解析成数值类型，日期时间应解析成时间类型，布尔值应解析成布尔类型。这样编译器和 API 能帮助约束后续操作。

```java
int limit = Integer.parseInt(rawLimit);
BigDecimal amount = new BigDecimal(rawAmount);
boolean enabled = Boolean.parseBoolean(rawEnabled);
```

解析时要注意异常和精度。`Integer.parseInt` 遇到非法格式会抛出 `NumberFormatException`。金额和高精度小数不应先转成 `double` 再构造 `BigDecimal`，因为二进制浮点可能已经丢失十进制精度。应直接从字符串构造 `BigDecimal`，并在边界处校验格式、范围和小数位数。

日期时间字符串更需要明确格式、时区和区域设置。不要依赖隐式默认格式解析外部输入。内部逻辑应尽量使用明确的时间类型，只有在输入输出边界才与字符串互转。字符串表示适合传输和展示，不适合承担所有时间计算语义。

## 本地化与排序比较

字符串比较不总是简单的二进制或 Unicode 顺序。`compareTo` 按字典序比较 Unicode 值，适合内部键、协议字段和稳定技术排序。面向自然语言的排序可能需要 `Collator`，因为不同语言对重音、大小写和字符组合的排序规则不同。

大小写转换也有本地化差异。内部技术标识应使用 `Locale.ROOT`，自然语言展示文本则应根据目标语言环境处理。把这两类需求混在一起，会产生难以复现的区域相关问题。

相等比较也要区分需求。账号名、命令字、协议字段可能要求大小写不敏感；密码、令牌、哈希值通常要求逐字符精确匹配。不要把 `equalsIgnoreCase` 当成通用便利方法，它会改变安全和协议语义。

## 不可变带来的 API 设计影响

由于 `String` 不可变，方法接收字符串参数时通常不需要防御性复制。调用方无法在方法执行期间改变该字符串内容，这让字符串比可变字符数组更容易作为参数、键和缓存值使用。

但不可变也意味着每次“修改”都会产生新结果。链式调用虽然可读，但在大量文本处理时可能创建多个中间字符串。

```java
String normalized = raw.strip()
        .toLowerCase(Locale.ROOT)
        .replace("-", "_");
```

这段代码适合普通输入规范化。如果处理的是数百万行文本或大型内容，可能需要评估中间对象成本，改用一次扫描、`StringBuilder` 或流式处理。优化前应先确认热点，普通业务代码优先清晰。

字符串作为返回值时，应明确是否可能返回 `null`。很多 API 更倾向返回空字符串表示“有文本但为空”，用 `Optional<String>` 表示“可能没有文本”。无论选择哪种方式，都应保持方法契约一致，不要让调用方同时处理 `null`、空串和空白串三套语义。

## 从对象模型理解字符串

把字符串理解成“文本值”很方便，但写 Java 代码时还要意识到它首先是一个对象。变量里保存的是引用，引用指向堆中某个 `String` 实例；字符串字面量、运行期拼接结果、外部输入解析结果，都可能在不同时间以不同方式得到对象。这个事实解释了为什么 `==` 比较的是同一个对象而不是同一段文本，也解释了为什么不可变性对共享对象如此重要。

`String` 类本身是 `final` 的，不能被继承。这样做能避免子类改变相等性、哈希值、内部表示或不可变承诺。如果字符串可以被继承，一个看似普通的字符串引用就可能指向带有额外可变状态的子类对象，很多依赖字符串稳定性的类库都会变得难以保证正确性。不可继承、不可变、内容相等与稳定哈希值共同构成了字符串作为基础值对象的前提。

字符串的 `equals` 按内容比较，`hashCode` 也按内容计算。哈希算法的具体形式是 Java 平台长期稳定的一部分，因此相同内容的字符串会得到相同哈希值。实现通常会缓存计算过的哈希值，以便字符串作为哈希表键时减少重复计算。这个缓存不影响语义：程序不应该依赖缓存存在，只需要知道只要内容相同，`equals` 与 `hashCode` 就必须保持一致。

底层存储属于实现细节，不能写成业务假设。早期 Java 使用 `char[]` 表示字符串内容，后来的主流实现引入了更紧凑的内部表示，能够在文本只需要 Latin-1 范围时节省内存。无论底层是 `char[]`、`byte[]` 还是其他结构，`String` 对外暴露的仍然是按 UTF-16 代码单元索引的 API。也就是说，`length()`、`charAt()`、`substring()` 的语义不会因为内部压缩而改变。写业务代码时应依赖公开 API 的契约，而不是依赖某个版本的私有字段。

这种对象模型还影响内存判断。一个短字符串对象不只包含字符内容，还包含对象头、长度相关信息、内部数组或紧凑编码标记等开销；大量短字符串可能比想象中占用更多内存。反过来，对于重复出现的大量相同字面量，常量池和共享引用又能显著减少对象数量。分析字符串内存问题时，不能只看文本字符数，还要看对象个数、重复率、生命周期和集合结构。

## 字面量、类加载与运行期字符串

源代码中的字符串字面量在编译后会进入 class 文件常量池，运行时再由虚拟机把它们解析成字符串对象，并放入运行时的字符串常量池中。开发者通常不需要操作这个过程，但它解释了为什么同一个类或不同类中相同的字面量往往可以指向同一个规范对象。

字面量复用是一种平台机制，不是业务语义。下面两段代码都能得到内容为 `java` 的字符串，但对象身份不同：

```java
String literal = "java";
String runtime = new String(new char[] {'j', 'a', 'v', 'a'});
```

如果业务上要判断用户输入是否等于 `java`，只能使用内容比较。常量池可以减少内存、支持 `intern()` 这样的规范化工具，但它不能替代数据模型中的相等规则。把引用身份当作文本相等会让代码在简单单元测试中表现正常，却在运行期输入、反序列化、文件读取、数据库读取或字符串构造器结果上失败。

`intern()` 的真正用途是把一批内容相等但对象不同的字符串收敛到同一个规范引用。它可能用于编译器、解释器、解析器、符号表、协议关键字表等场景。使用前要确认两个条件：第一，字符串集合有较高重复率，规范化确实能减少对象或加速身份比较；第二，规范化集合的生命周期可控，不会把大量只出现一次的外部输入长期留在池中。普通业务代码为了“让 `==` 生效”而调用 `intern()`，通常是在绕开正确的比较方式。

字符串拼接也有编译期和运行期之分。全部由编译期常量组成的表达式可以直接折叠成一个字面量；只要其中包含运行期变量，就需要在运行期构造结果。现代 Java 对拼接做了很多优化，简单表达式中的 `+` 可读且合理。真正需要关注的是循环累计、深层嵌套拼接和高频日志消息构造，因为这些位置可能在短时间内产生大量临时对象。

## 文本边界：代码单元、码点、字形簇

字符串处理中最容易被低估的是“一个字符”到底指什么。Java 的许多基础 API 以 UTF-16 代码单元为索引，一个 `char` 就是一个代码单元。Unicode 码点是抽象字符编号，有些码点能用一个 `char` 表示，有些码点需要一对代理项表示。用户看到的一个字形还可能由多个码点组合而成，例如基础字母加组合重音符号，或者由连接符组合起来的复杂符号序列。

因此，在不同需求下要选择不同边界：

| 需求 | 更合适的口径 | 说明 |
| --- | --- | --- |
| 调用 `substring`、`charAt` | UTF-16 代码单元 | Java `String` 的索引契约就是这个口径 |
| 判断 Unicode 码点数量 | 码点 | 使用 `codePointCount` 或 `codePoints` |
| 限制内部协议字段长度 | 明确约定的单位 | 可以是字节数、代码单元数或码点数，但必须写清楚 |
| 面向用户的显示截断 | 字形簇或排版宽度 | 仅靠 `length()` 不可靠 |
| 编码后传输大小限制 | 字节数 | 必须先按目标字符集编码再计算 |

这个边界问题在截断字符串时尤其明显。如果只是为了限制日志长度，按代码单元截断通常可以接受，但要避免截断到代理项对中间，导致结果出现无效或替代字符。如果要截断面向用户的自然语言文本，除了代理项对，还要考虑组合字符和显示宽度。Java 标准库提供了码点级遍历能力，但完整的字形簇处理通常需要更高层的文本处理工具或明确的产品规则。

按码点遍历时，不要再把 `int` 强行塞回 `char`。`codePoints()` 返回的是 `IntStream`，每个元素代表一个 Unicode 码点。需要重新组装字符串时，可以用 `StringBuilder.appendCodePoint`：

```java
String source = "A𝄞B";
StringBuilder onlyLetters = new StringBuilder();
source.codePoints()
        .filter(Character::isLetter)
        .forEach(onlyLetters::appendCodePoint);
```

这里的重点不是所有代码都要改用码点，而是要让索引单位与需求一致。处理 ASCII 配置键时用 `charAt` 完全合理；处理任意 Unicode 输入时仍然按 `char` 逐个判断，就可能漏掉代理项、组合符和特殊大小写映射。

## Unicode 规范化与文本等价

两个字符串看起来相同，并不一定由相同的 Unicode 码点序列组成。例如带重音的字符可能有预组合形式，也可能由基础字符加组合符号构成。`equals` 比较的是代码单元序列，不会自动认为这些不同序列相等。对于文件名、用户输入、自然语言搜索、跨系统交换的文本，如果“视觉上相同”或“语言上等价”应被视为相同，就需要明确规范化策略。

Java 标准库提供 `java.text.Normalizer` 来做 Unicode 规范化。常见形式包括 NFC、NFD、NFKC、NFKD。NFC 倾向使用组合后的规范形式，常用于存储和比较前的统一；NFD 会拆分组合字符；带 K 的兼容规范化会把一些兼容字符也折叠，可能改变更强的语义。选择哪一种不能凭习惯，要看业务是否允许把兼容字符视为同一类文本。

```java
String normalized = Normalizer.normalize(input, Normalizer.Form.NFC);
```

规范化不是越多越好。对于密码、令牌、签名原文、哈希输入等要求精确字节或精确字符序列的内容，随意规范化可能破坏协议语义或安全属性。对于搜索关键字、标签、用户可见名称等自然语言场景，规范化又可能减少重复和匹配失败。关键在于把规范化放在清晰的边界上，并在方法契约中说明是否已经规范化。

大小写折叠也属于文本等价的一部分。`equalsIgnoreCase` 适合简单的大小写无关判断，但它不是完整的本地化比较，也不等于自然语言搜索。对于内部技术键，通常先使用 `Locale.ROOT` 转成统一大小写再比较；对于面向语言的排序和匹配，则应考虑 `Collator`、区域设置和搜索规则。混用这些策略会制造隐藏差异：同一个输入在开发机、服务器和不同用户区域设置下可能得到不同结果。

## 查找、匹配与替换的选择

字符串 API 中有很多看似相近的方法：`contains`、`indexOf`、`startsWith`、`endsWith`、`matches`、`replace`、`replaceAll`、`replaceFirst`。选择时应先问需求是不是正则。如果只是固定文本查找，`contains` 和 `indexOf` 更直接；如果只是固定文本替换，`replace` 更合适；如果需要模式匹配，才使用正则相关方法。

`matches` 还有一个常见细节：它要求整个字符串匹配正则，而不是只要包含某个片段。想判断是否包含某个模式，应使用 `Pattern` 和 `Matcher.find()`。很多校验代码误用 `matches("abc")` 后发现只有字符串正好等于 `abc` 才通过，就是因为没有区分全量匹配和局部查找。

```java
Pattern pattern = Pattern.compile("\\d+");
Matcher matcher = pattern.matcher("item-42");
System.out.println(matcher.find());    // true
System.out.println("item-42".matches("\\d+")); // false
```

替换时还要区分“正则模式”和“替换字符串”。`replaceAll` 的第一个参数是正则，第二个参数也不是完全普通的文本，`$1`、`\` 等字符在替换结果中有特殊意义。如果替换内容来自外部输入，应该使用 `Matcher.quoteReplacement` 避免被解释为分组引用或转义序列。模式来自外部输入并且只想按字面量匹配时，应该使用 `Pattern.quote`。

```java
String result = input.replaceAll(
        Pattern.quote(target),
        Matcher.quoteReplacement(replacement)
);
```

正则的表达力很强，但也会带来可维护性和性能风险。一个复杂正则如果没有命名、注释和测试，后续维护者很难判断它的边界。某些带有嵌套量词和回溯路径的模式，在特定输入下可能出现极慢匹配。对外部输入使用正则时，应尽量让模式具体、边界明确，避免在核心路径上反复编译同一个模式。

## 拆分、解析与结构化数据

`split` 很适合处理简单分隔符文本，但它不是通用解析器。CSV、URL 查询串、属性文件、JSON、XML、SQL、日志格式等都有各自的转义、引号、嵌套或编码规则，不能简单用逗号、等号或冒号拆开。看到一段字符串包含分隔符时，先确认格式是否真的简单；如果格式有成熟解析库，就不应临时手写拆分逻辑。

即便是简单分隔符，也要明确空字段、尾部字段和字段数量。例如配置项 `a,b,` 是否表示两个值还是三个值且最后一个为空？连续分隔符 `a,,b` 是否允许中间空值？字段前后空白是否要保留？这些规则如果不写清楚，调用方会在不同地方自行补救，最后形成一组互相矛盾的字符串处理习惯。

使用 `split` 时还要记住它接收正则表达式。分隔符如果是点号、竖线、问号、星号、反斜杠等正则特殊字符，就必须转义或使用 `Pattern.quote`。此外，`split` 会创建数组和多个子串；在解析超大文本或内层循环中，如果只需要逐段扫描，手写基于 `indexOf` 的迭代有时更节省对象。这里的判断仍然以场景为准：普通配置解析优先清晰，解析器内层才值得为分配成本做更细控制。

结构化数据转换还有一个原则：字符串只应该停留在边界。外部输入刚进入系统时是字符串，输出给外部系统时也常常变回字符串；但内部一旦知道它是数字、日期、枚举、路径、货币、标识符或布尔值，就应尽早转成对应类型。长期用字符串在内部流转，会让校验分散、错误推迟、语义丢失，也让重构变得困难。

## 文本块、转义与可读性

Java 的字符串字面量需要处理引号、反斜杠、换行、制表符等转义。普通单行字面量适合短文本，文本块适合多行文本。文本块不是简单地“把三引号之间的所有字符原样保存”，它有缩进处理、行终止符和转义规则。使用文本块时，应通过测试或打印确认实际内容，尤其是末尾换行、缩进空格和需要对齐的模板。

文本块能显著提升多行 SQL、示例 JSON、帮助文本和测试样本的可读性，但它不等于模板引擎。把大量变量用 `replace` 塞进文本块，很快会遇到转义、缺失字段、注入和格式错误问题。少量测试数据可以这样写，正式生成结构化文本时更应该使用对应的构建器、序列化库或格式化工具。

转义还影响正则和路径等内容。Java 字符串字面量中的反斜杠先由 Java 编译器处理，然后正则引擎再处理一次，所以正则里的一个字面量反斜杠经常要写成更多反斜杠。很多“正则写不对”的问题，本质是没有区分 Java 字符串转义层和正则语法层。调试时可以先把字符串打印出来，确认传给下游 API 的真实内容，再看下游规则是否正确。

## 格式化、插值与日志文本

`String.format`、`formatted` 和 `Formatter` 适合需要对齐、补零、小数位、日期格式片段或多参数模板的场景。它们的优势是表达意图集中，缺点是格式字符串与参数类型之间的错误通常在运行期才暴露，并且性能成本高于简单拼接。普通的两三个片段连接，用 `+` 往往更清晰；复杂展示文本，用格式化更容易维护。

格式化还要注意 Locale。数字的小数点、分组分隔符、日期名称、大小写转换都可能受区域设置影响。用于机器读取的文本应显式选择稳定区域，例如 `Locale.ROOT` 或协议规定的区域；用于用户展示的文本才应使用用户语言环境。把默认 Locale 隐式带入协议字符串，会让输出在不同运行环境中变化。

日志文本有自己的边界。很多日志框架支持占位符延迟格式化，这比先用字符串拼接构造完整消息更适合被日志级别过滤的路径。如果日志级别关闭，提前拼接仍然会执行对象构造和 `toString`。但日志框架的占位符也不是数据脱敏工具，敏感字段必须在进入日志前就被过滤或脱敏。不要为了调试方便把完整请求、完整令牌或完整个人信息直接拼进字符串。

## 设计字符串方法的契约

字符串工具方法最容易变成“什么都接收、什么都返回”的模糊 API。一个清晰的方法契约至少要回答几件事：是否允许 `null`？是否允许空字符串？是否把空白字符串视为有效？是否会裁剪首尾空白？是否区分大小写？是否执行 Unicode 规范化？返回值是否可能为 `null`？异常表示调用错误还是输入格式错误？

例如一个解析布尔配置的方法，可以有多种合理设计：只接受 `true` 和 `false`；额外接受 `yes`、`no`、`1`、`0`；忽略大小写；允许首尾空白；非法输入返回默认值；非法输入抛异常。这些选择本身没有绝对对错，但不能隐藏在实现细节里。隐藏规则会让调用方误以为某些输入被支持，或者把真正的配置错误静默吞掉。

更稳妥的做法是把边界处理放在方法入口，并让方法名或文档体现语义。`requireNonBlank` 表示空白会失败，`emptyToNull` 表示会把空串转换为缺失，`normalizeIdentifier` 表示会做标识符规范化。命名无法表达的规则，就用测试和简短注释固定下来。字符串规则越接近系统边界，越值得写得明确，因为这些规则往往会影响数据存储、兼容性和外部调用。

返回字符串时也要避免制造多义性。如果找不到值，返回 `null`、空字符串、`Optional<String>` 或抛异常都可能合理，但一个模块内应保持一致。尤其不要让同一个方法在某些分支返回 `null`，另一些分支返回空字符串来表达同一件事。调用方为了防御这种不一致，最终会在每处都写复杂判空，反而让真实语义更不清楚。

## 集合中的字符串

字符串经常作为 `Map` 的键、`Set` 的元素和排序列表的成员。用于集合键时，最重要的是在入集合前完成规范化。假设键名大小写不敏感，就应在放入和查询前使用同一套大小写规则；假设键名不允许首尾空白，就应在边界处裁剪或拒绝。不要在某些查询路径上临时 `trim`，在另一些路径上直接查原值，这会让集合里同时存在多个看似相同的键。

排序同样需要明确规则。`TreeMap<String, V>` 默认使用自然顺序，也就是 `String.compareTo`。如果使用大小写不敏感比较器，那么比较器认为相等的两个字符串不能同时作为不同键存在，即使它们的 `equals` 返回 false。比较器与相等性的关系会影响集合行为，尤其是在 `TreeSet`、`TreeMap` 中。使用自定义比较器时要确认它是否符合业务的“同一个键”定义。

集合中还要考虑空值策略。`List<String>` 里是否允许 `null`？`String.join` 遇到 `null` 会失败，流式处理中 `map(String::trim)` 遇到 `null` 也会失败。如果业务不允许空值，尽早校验比在每次使用前防御更好。如果业务允许空值，就应该在连接、排序、展示、序列化时写出明确处理规则。

## 大文本处理与流式思维

许多字符串 API 都适合中小文本：读取完整字符串、拆分成数组、替换后返回新字符串、一次性格式化结果。这些写法简单直观，但面对大文本时会放大内存和复制成本。一个 100MB 的文本如果被读成字符串，再 `split` 成数组，再对每行 `trim` 和 `replace`，峰值内存可能远大于原始文本大小。

处理大文本时，要考虑能否按流读取、按块扫描或逐行处理。Java I/O 和 NIO 提供了面向字符流的 API，可以避免把所有内容一次性放进内存。即使最终需要生成一个完整结果，也可以在过程中减少中间字符串数量，例如使用 `BufferedReader.lines()` 逐行处理，或者使用 `StringBuilder`/`Writer` 逐步输出。

性能优化不应只盯着 `StringBuilder`。很多字符串热点来自算法选择：反复从头查找、重复编译正则、对每条记录都创建格式化器、把大字符串拆成大量小对象、在集合中保存未规范化的重复值。定位这类问题应使用分析工具或基准测试确认热点，再选择优化手段。没有证据时，清晰的字符串代码通常比过早手写扫描器更可靠。

## 版本差异与适用范围

Java 字符串的核心语义长期稳定：不可变、按内容比较、按 UTF-16 代码单元索引、字面量可复用、字节转换需要字符集。这些规则可以作为通用 Java 知识使用。与此同时，某些 API 和实现细节与 Java 版本有关，写文章或代码时应区分语言契约、标准库 API 和具体实现优化。

例如，文本块是较新的语言特性；`isBlank`、`strip`、`lines`、`repeat` 等方法也不是所有旧版本都可用。`String` 的内部存储方式、拼接优化策略、字符串去重能力、哈希缓存细节等属于实现层面，即使在主流 OpenJDK 中有稳定表现，也不应该成为业务代码依赖。需要兼容旧运行环境时，应先确认目标 Java 版本，再决定能否使用这些 API。

还有一类差异来自默认环境：默认字符集、默认 Locale、默认时区、行分隔符。这些不是字符串类自身的语义变化，却会影响字符串与外部世界互转的结果。凡是用于持久化、网络传输、跨平台比较、构建缓存键或生成签名的字符串，都应避免依赖这些默认值，改为显式指定字符集、区域和格式。

## 判断一段字符串代码是否可靠

评审字符串代码时，可以从几个问题入手。第一，它比较的是引用还是内容？如果使用 `==`，是否真的在比较同一个对象身份？第二，它有没有明确处理 `null`、空串和空白串？第三，它是否在字节与字符之间显式指定了字符集？第四，它的索引、截断和长度限制是按代码单元、码点、字节还是显示宽度？第五，它使用正则时是否真的需要正则，特殊字符和替换字符串是否正确转义？

还要看数据边界。输入是否在进入系统时被规范化和校验？内部是否长期用字符串表示本该有类型的数据？输出是否可能泄露敏感内容？集合键是否有统一的大小写和空白规则？大文本处理是否会一次性创建过多中间对象？这些问题比“某个方法快不快”更重要，因为它们决定了字符串代码在真实环境中的正确性和可维护性。

可靠的字符串代码通常有一个共同特征：它不把文本当作模糊的字符堆，而是给每类文本赋予清楚语义。配置键、用户展示文本、协议字段、自然语言内容、日志消息、文件内容、二进制编码结果、数字和日期的文本形式，都有不同规则。把规则写在边界上，内部使用明确类型，必要时用测试覆盖典型边界，字符串处理就会从“到处都是小补丁”变成稳定的基础设施。

## 常见误区与实践要点

不要用 `==` 比较字符串内容。常量池可能让某些测试“碰巧通过”，但运行期构造的字符串会暴露问题。

不要在循环中大量使用 `+=` 拼接。局部少量拼接没问题，循环累计构建应使用 `StringBuilder` 或收集后 `String.join`。

不要混淆空、空白和 `null`。方法契约应明确是否允许 `null`，是否把空白视为无效输入。入口处统一规范化能减少后续分支。

不要在编码转换时依赖默认字符集。跨进程、跨文件或跨系统的文本都应显式指定 `StandardCharsets.UTF_8` 或协议要求的字符集。

不要用 `length()` 判断真实显示宽度或用户感知字符数。Unicode 文本处理需要明确按代码单元、码点还是字形簇处理。

不要把正则 API 当普通文本 API。固定文本替换用 `replace`，固定分隔符但含正则特殊字符时要正确转义或使用 `Pattern.quote`。

## 小结

Java 字符串的核心特征是不可变、引用类型、可常量池复用，并以 Unicode 字符序列形式存在。内容比较使用 `equals`，高频构建使用 `StringBuilder`，字节转换显式指定字符集，复杂文本处理要意识到 `char` 与 Unicode 码点的差异。掌握这些规则后，字符串代码才能在正确性、可读性和性能之间保持稳定。

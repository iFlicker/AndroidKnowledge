# 3.2.3.2 通配符

Java 泛型中的通配符用于描述“类型实参未知，但它满足某个边界”的参数化类型。它解决的不是如何声明一种新类型，而是如何在保留类型安全的前提下，让不同泛型实例之间建立可用的赋值关系。例如，方法只需要读取一组数字时，不应被迫只接收 `List<Number>`；它也应该能接收 `List<Integer>` 和 `List<Double>`。反过来，方法准备向集合写入整数时，也不应只接受 `List<Integer>`；`List<Number>` 和 `List<Object>` 同样能够容纳整数。

通配符的难点不在 `?`、`extends`、`super` 三个语法记号，而在于三个相互关联的问题：Java 泛型为什么默认不变；一个未知类型在不同边界下允许哪些读写操作；当方法需要在多个位置表达“同一个类型”时，为什么通配符有时必须转换为类型变量。理解这些问题之后，PECS、通配符捕获、类型推断和常见编译错误都能从类型安全约束中推导出来，而不必依赖孤立口诀。

## 泛型不变性是通配符出现的前提

Java 中普通引用类型具有子类型关系：如果 `Integer` 是 `Number` 的子类型，那么 `Integer` 引用可以赋给 `Number` 变量。但把类型放入泛型构造器后，这个关系默认不会自动传递。虽然 `Integer <: Number`，却不能推出 `List<Integer> <: List<Number>`。这种性质称为泛型的不变性。

不变性首先保护可变容器的写入安全。假设 `List<Integer>` 可以赋给 `List<Number>`，那么接收方就能通过后者加入 `Double`。原列表随后再按 `Integer` 读取时，类型契约已经被破坏：

```java
List<Integer> integers = new ArrayList<>();
// List<Number> numbers = integers; // 编译错误：泛型不变
// numbers.add(3.14);                // 若赋值成立，会污染 integers
```

这里不能把问题简单归结为“集合可变”。Java 的泛型类型声明本身没有声明型变方向，编译器不会根据某个对象当前是否只读来改变 `List<E>` 的子类型关系。即使某段代码承诺不调用 `add`，`List<Integer>` 仍然不是 `List<Number>`。Java 通过使用点型变，也就是在变量、参数或返回类型的使用位置写通配符，来表达当前视角允许的有限型变关系。

`List<? extends Number>` 不是 `List<Number>` 的另一种写法。前者表示“某个确定但未知的元素类型，该类型是 `Number` 或其子类型”；后者表示“元素类型恰好是 `Number`”。因此，下面的赋值关系成立：

```java
List<Integer> integers = List.of(1, 2, 3);
List<Double> doubles = List.of(1.5, 2.5);

List<? extends Number> first = integers;
List<? extends Number> second = doubles;
```

变量 `first` 背后仍然是 `List<Integer>`，变量 `second` 背后仍然是 `List<Double>`。通配符没有改变对象的真实泛型契约，只是给引用建立了一个能力受限的观察视图。正因为编译器不知道视图背后的精确元素类型，它才会限制某些写入操作。

## 无界通配符表示未知类型，而不是任意类型

`List<?>` 表示某个元素类型未知的列表。它可以引用 `List<String>`、`List<Integer>`、`List<Object>`，也可以引用其他合法的 `List<T>`。这里的“未知”与“任意”有重要区别：

- 未知类型表示实际类型已经确定，只是当前代码不知道它是什么。
- 任意类型容易让人误以为可以向集合放入任意对象，但这会破坏实际类型的约束。
- `List<Object>` 的元素类型明确是 `Object`，因此可以写入任意对象；`List<?>` 的元素类型不明确，因此不能写入任意非 `null` 对象。

对 `List<?>` 读取时，编译器只保证结果是 `Object`。无论实际元素类型是什么，它都一定是 `Object` 的子类型，所以这个读取是安全的：

```java
static int sizeAndInspect(List<?> values) {
    if (!values.isEmpty()) {
        Object first = values.get(0);
        System.out.println(first);
    }
    return values.size();
}
```

对 `List<?>` 写入具体值则不安全。实际对象可能是 `List<String>`，也可能是 `List<Integer>`，不存在一个非 `null` 的具体值能够被证明适合所有可能的元素类型。因此 `values.add("text")`、`values.add(1)` 和 `values.add(new Object())` 都不能通过编译。`null` 是例外，因为 `null` 可以转换为任何引用类型；但“语法上允许加入 `null`”不等于 API 语义上应该这样做，具体集合也可能因实现约束而拒绝 `null`。

通配符视图受限也不等于集合不可修改。元素类型未知只会阻止依赖具体元素类型的写入，不会阻止与元素类型无关的操作。例如 `clear()`、`remove(int)`、`remove(Object)`、迭代器的 `remove()` 仍可能修改集合。能否执行还取决于集合实现是否支持修改。因而把 `List<?>` 称为“只读列表”并不严谨，更准确的说法是：它不能安全接收任意非 `null` 的新元素，但仍可能支持删除、清空、重排或其他不要求构造元素值的操作。

当方法只关心容器结构或把元素当作 `Object` 观察时，无界通配符通常比类型参数更直接：

```java
static boolean isEmpty(Collection<?> collection) {
    return collection.isEmpty();
}
```

若写成 `<T> boolean isEmpty(Collection<T> collection)`，类型参数 `T` 只出现一次，方法内部也没有利用它建立任何关系，这个类型变量没有提供额外约束。使用 `Collection<?>` 能更准确地表达“元素类型与本操作无关”。

## 上界通配符建立安全的生产者视图

`? extends T` 是上界通配符，表示未知类型 `CAP` 满足 `CAP <: T`。例如 `List<? extends Number>` 可以引用 `List<Number>`、`List<Integer>` 或 `List<Double>`。由于任何可能的 `CAP` 都能向上转型为 `Number`，从列表读取的元素可以安全地按 `Number` 使用：

```java
static double sum(List<? extends Number> numbers) {
    double total = 0.0;
    for (Number number : numbers) {
        total += number.doubleValue();
    }
    return total;
}
```

这个方法只依赖 `Number` 提供的能力，不关心实际列表保存的是哪一种数字。相比参数为 `List<Number>`，上界通配符扩大了合法调用集合，却没有放松方法内部对读取结果的静态保证。

上界通配符通常不能接收具体的 `T`。对 `List<? extends Number>` 来说，实际类型可能是 `Integer`，也可能是 `Double`。即使准备写入的值声明为 `Number`，它也可能与实际元素类型不匹配：

```java
static void inspect(List<? extends Number> numbers) {
    Number first = numbers.get(0);

    // numbers.add(1);    // 编译错误：实际类型可能是 Double
    // numbers.add(1.0);  // 编译错误：实际类型可能是 Integer
    // numbers.add(first); // 仍然错误：first 的静态类型只是 Number
}
```

最后一行尤其容易误解。`first` 确实来自同一个列表，但一旦它被保存到 `Number` 变量中，类型信息就被扩大为 `Number`。编译器不能仅凭值的来源证明它仍然具有该列表捕获到的精确类型。即便直接写 `numbers.add(numbers.get(0))`，在某些涉及不同捕获的上下文中也可能需要辅助方法；编译器依赖的是静态类型关系，而不是运行时对象身份或开发者对值来源的直觉。

上界通配符不是让泛型整体变成协变。它只是声明了一个协变式的使用视图：`List<Integer>` 可以赋给 `List<? extends Number>`，但仍不能赋给 `List<Number>`。视图获得了更宽的可接收范围，同时失去了写入具体元素的能力，这是类型安全所要求的交换。

上界也可以用于嵌套位置。例如 `Comparator<? super T>`、`Function<? super T, ? extends R>` 中的 `extends` 都是在表达某个位置产生的结果至少可以当作目标上界使用。判断时应关注类型参数在操作中的流向，而不是看到 `extends` 就机械理解成类继承。

## 下界通配符建立安全的消费者视图

`? super T` 是下界通配符，表示未知类型 `CAP` 满足 `T <: CAP`。`List<? super Integer>` 可以引用 `List<Integer>`、`List<Number>` 或 `List<Object>`。无论实际是哪一种列表，写入 `Integer` 都安全，因为这些元素类型都能接收 `Integer`：

```java
static void addDefaults(List<? super Integer> destination) {
    destination.add(0);
    destination.add(1);
}
```

不仅可以写入 `Integer`，也可以写入它的子类实例；只是 `Integer` 是 `final` 类，没有实际子类。若下界是一个可继承类型，例如 `? super CharSequence`，则 `String` 和 `StringBuilder` 都能作为 `CharSequence` 的子类型写入。

下界通配符的读取能力较弱。`List<? super Integer>` 的实际元素类型可能是 `Object`，并且列表在传入方法前可能已经保存了字符串或其他对象。编译器只能保证读取结果是 `Object`：

```java
static void inspectDestination(List<? super Integer> destination) {
    destination.add(42);
    Object value = destination.get(0);
    System.out.println(value);

    // Integer number = destination.get(0);
    // 编译错误：已有元素不一定是 Integer
}
```

“能写入 `Integer`”不能推出“读取的一定是 `Integer`”。下界约束的是列表能够接收哪些值，并不承诺列表只包含这些值。只有调用方和方法之间另有更强的业务不变量，运行时才可能满足更具体的读取假设；但这种假设没有体现在 `? super Integer` 的静态类型中，不能用无检查强制转换代替类型证明。

下界通配符体现的是逆变式使用关系：如果方法需要消费 `Integer`，那么能够消费更宽类型的 `List<Number>` 和 `List<Object>` 也都合适。Java 没有声明点逆变语法，而是在每个使用位置通过 `? super T` 明确这种关系。

## 用读写能力推导三种通配符

判断通配符时，可以先暂时忘记 PECS，直接列出方法需要的能力：

| 类型 | 可安全读取为 | 可安全写入 | 典型含义 |
| --- | --- | --- | --- |
| `List<T>` | `T` | `T` 及其子类型 | 精确知道元素类型，需要完整读写能力 |
| `List<?>` | `Object` | 通常只有 `null` | 元素类型与操作无关 |
| `List<? extends T>` | `T` | 通常只有 `null` | 从结构中取得 `T` |
| `List<? super T>` | `Object` | `T` 及其子类型 | 向结构中放入 `T` |

表中的“读取”和“写入”指的是静态类型层面最强的普遍保证，并不等于对象所有方法的完整权限。例如 `List<? extends T>` 仍可能通过 `remove` 删除元素，`List<? super T>` 也能把元素读取成 `Object`。将其粗略称为“只读”和“只写”虽然便于记忆，却会掩盖这些合法操作。

另一个重要边界是：如果同一个参数既要精确读取为 `T`，又要写入 `T`，通常应该使用 `List<T>`，而不是强行选择某种通配符。通配符用于放宽调用方类型，同时主动收窄方法内部能力；当实现确实需要完整能力时，精确类型变量才是正确契约。

## PECS 的推导、成立条件与局限

PECS 是 “Producer Extends, Consumer Super” 的缩写：参数为方法生产 `T` 时使用 `? extends T`，参数从方法消费 `T` 时使用 `? super T`。例如复制操作中，源集合生产元素，目标集合消费元素：

```java
static <T> void copy(
        List<? extends T> source,
        List<? super T> destination) {
    for (T element : source) {
        destination.add(element);
    }
}
```

`source` 的实际元素类型可以比 `T` 更具体，`destination` 的实际元素类型可以比 `T` 更宽。调用 `copy(List<Integer>, List<Number>)` 时，编译器可以选择 `T` 为 `Integer` 或某个满足全部约束的类型，并证明从源读取后写入目标是安全的。

PECS 成立的前提是能够明确某个参数相对于类型 `T` 的主要数据流向。这里的 producer 和 consumer 是相对于当前方法而言，不是对象在整个系统中的永久身份。同一个 `List` 在一个方法中可能是生产者，在另一个方法中可能是消费者；`Comparator<T>` 虽然不保存 `T`，但它接收两个 `T` 参与比较，因此从类型流向看是 `T` 的消费者，常写成 `Comparator<? super T>`。

PECS 不是“所有参数都必须带通配符”的规则，至少有以下局限。

第一，如果参数同时需要精确生产和消费同一种类型，通配符会损失必要能力。例如原地交换列表元素需要从列表读取一个精确类型的值，再写回同一个列表。公开入口可接收 `List<?>`，但实现需要通过捕获转换把未知类型命名为某个 `T`；若方法还要接收一个外部 `T` 写入，则直接使用 `List<T>` 往往更清晰。

第二，PECS 只描述类型流向，不描述可变性、所有权和副作用。`List<? extends T>` 并不保证列表不可变，`List<? super T>` 也不保证方法一定写入。API 仍需单独说明是否复制、是否保留引用、是否修改传入对象。

第三，返回类型通常不应为了“生产者”口诀而机械写成 `List<? extends T>`。方法返回值确实向调用方生产元素，但返回通配符会迫使调用方继续处理未知捕获，降低可组合性。若实现能够承诺明确类型，优先返回 `List<T>`、具体类型参数或不可变的精确泛型类型。

第四，`extends` 与 `super` 不能修复缺失的类型关系。如果两个参数必须共享同一个未知类型，仅分别写成两个 `List<?>` 不足以表达相等关系；如果输入元素类型与返回类型有转换关系，也需要类型变量把多个位置连接起来。

第五，通配符的灵活性可能增加重载、类型推断和错误信息的复杂度。对私有方法或实现内部代码，如果调用类型已经精确且没有复用需求，使用通配符未必有收益。API 设计应以实际可接收范围和调用体验为依据，而不是以通配符数量为抽象程度指标。

## 类型变量与通配符解决不同问题

类型变量用于给一个未知类型命名，并在多个位置表达同一性或依赖关系；通配符用于接受某个边界内的任意未知类型，但通常不关心其名称。两者看起来都包含未知类型，却承担不同职责。

下面的方法需要保证返回值类型与输入元素类型一致，因此必须使用类型变量：

```java
static <T> T first(List<T> values) {
    if (values.isEmpty()) {
        throw new IllegalArgumentException("values must not be empty");
    }
    return values.get(0);
}
```

如果改成 `Object first(List<?> values)`，调用范围没有变宽，返回类型却退化为 `Object`。如果试图写 `? first(List<?> values)`，语法也不成立，因为通配符只能出现在类型实参位置，不能作为方法独立的返回类型名称。

相反，如果方法只打印元素，不需要把元素类型与其他位置关联，类型变量就显得多余：

```java
static void printAll(Iterable<?> values) {
    for (Object value : values) {
        System.out.println(value);
    }
}
```

一个实用判断是：若未知类型需要在参数、返回值、局部辅助关系或多个参数之间重复出现，就考虑类型变量；若只需要说明“接受某类参数化类型，但不关心精确类型”，就考虑通配符。不能只用“类型参数是否出现一次”机械判断，因为某些一次出现的类型变量仍可能通过边界参与推断，而复杂嵌套类型也可能让关系隐藏在函数式接口中。最终标准是签名是否表达了调用方需要的关系。

## 捕获转换：编译器为未知类型建立临时名称

编译器处理 `List<?>` 时，并不是把 `?` 当作 `Object`。它会在需要时执行捕获转换，为这个未知类型创建一个不可见的临时类型变量，编译错误中常显示为 `CAP#1`、`CAP#2`。可以把 `List<?>` 暂时理解为 `List<CAP#1>`，其中 `CAP#1 extends Object`；把 `List<? extends Number>` 理解为 `List<CAP#1>`，其中 `CAP#1 extends Number`。

捕获的关键是“同一次捕获中的未知类型是某个确定类型”。虽然调用者不知道它是什么，但辅助方法可以给它命名，并在读取与写回之间保持同一性。经典例子是交换列表元素：

```java
static void swap(List<?> list, int first, int second) {
    swapCaptured(list, first, second);
}

private static <T> void swapCaptured(
        List<T> list, int first, int second) {
    T value = list.get(first);
    list.set(first, list.get(second));
    list.set(second, value);
}
```

公开方法接收任意元素类型的列表，因为交换不需要知道元素是什么。辅助方法把该列表的通配符捕获为 `T`，于是从同一列表读出的 `T` 可以安全写回。这里没有运行时转换，也没有突破泛型约束；辅助方法只是让编译器能够表达原本隐含的“读取值和写入位置属于同一个元素类型”。

不是每个 `List<?>` 操作都需要手写辅助方法。编译器会自动在方法调用、赋值转换等场景中执行捕获转换。只有当当前表达式无法让它证明多个操作共享同一捕获时，才需要显式泛型辅助方法。

捕获也解释了为什么两个外观相同的 `List<?>` 不能互相写入。两个参数分别捕获时，可能得到 `CAP#1` 和 `CAP#2`，它们没有相等关系：

```java
static void moveFirst(List<?> source, List<?> destination) {
    Object value = source.get(0);
    // destination.add(value);
    // 编译错误：source 与 destination 的捕获类型可能不同
}
```

若业务要求二者元素类型相同，应把关系写进签名：

```java
static <T> void moveFirst(List<T> source, List<? super T> destination) {
    destination.add(source.get(0));
}
```

这个版本没有宣称两个列表类型完全相同，而是表达了真正需要的条件：目标能够消费源产生的 `T`。准确表达最小关系通常比要求两个参数都为 `List<T>` 更灵活。

## 捕获错误应如何阅读

泛型编译错误经常包含冗长的 “required: CAP#1, found: ...”。阅读时可按以下顺序还原：

1. 找到发生写入、赋值或方法调用的位置。
2. 找到 `CAP#n` 来自哪个通配符表达式。
3. 写出该捕获的上界和下界。
4. 判断当前值能否被证明属于同一个捕获类型。
5. 若业务上确实存在同一性，使用类型变量或捕获辅助方法表达；若不存在，则编译器拒绝是正确结果。

例如，对 `List<? extends Number>` 调用 `add(Integer)` 时，`CAP#1 extends Number` 只说明捕获类型不超过 `Number`，没有说明 `Integer <: CAP#1`。实际捕获可能是 `Double`，所以写入失败。对 `List<? super Integer>` 读取为 `Integer` 时，约束是 `Integer <: CAP#1`，没有说明 `CAP#1 <: Integer`；实际捕获可能是 `Object`，所以读取只能得到 `Object`。

这种上下界方向的检查比记忆“extends 不能写、super 不能读”更可靠。后一句口诀忽略了 `null`、`Object`、删除操作和捕获辅助方法，也无法解释复杂嵌套签名。

## 通配符与类型推断如何相互作用

泛型方法调用时，编译器会收集参数、目标类型和类型变量边界形成约束，然后求出满足约束的类型实参。通配符既可以提供约束，也可能让精确信息变得不可命名。

考虑以下合并方法：

```java
static <T> List<T> merge(
        List<? extends T> left,
        List<? extends T> right) {
    List<T> result = new ArrayList<>(left.size() + right.size());
    result.addAll(left);
    result.addAll(right);
    return result;
}
```

调用 `merge(List<Integer>, List<Double>)` 时，编译器需要选择一个同时是 `Integer` 和 `Double` 的上界的 `T`。根据调用上下文，它可能推断出 `Number`，也可能推断出更宽的交集相关类型。若赋给 `List<Number>`，目标类型会参与推断并引导 `T` 为 `Number`：

```java
List<Integer> integers = List.of(1, 2);
List<Double> doubles = List.of(3.0, 4.0);

List<Number> numbers = merge(integers, doubles);
```

目标类型推断说明，泛型方法的结果不能总从实参孤立判断。赋值目标、返回上下文、链式调用和重载候选都可能参与约束求解。因此，一个没有目标类型的 `var` 声明、一个明确的 `List<Number>` 声明和作为另一方法实参的调用，可能得到不同的可见静态类型。

下界通配符也会参与推断。例如：

```java
static <T> void addAll(
        Collection<? super T> destination,
        Collection<? extends T> source) {
    destination.addAll(source);
}
```

当目标为 `Collection<Number>`、源为 `Collection<Integer>` 时，`T` 至少要是 `Integer` 的某个上界，同时又必须能写入 `Number` 目标。`Integer` 本身满足要求，因此调用成立。若目标为 `Collection<String>`、源为 `Collection<Integer>`，不存在满足两边约束的合适 `T`，调用被拒绝。

类型推断失败时，常见修复方式包括：让变量或参数保留更精确的泛型类型；增加能够表达关系的类型变量；拆开过长的链式调用；提供明确的类型见证，例如 `ClassName.<Number>method(...)`；或者重新设计签名，避免把互不相关的捕获误写成必须相同。强制转换应是最后手段，因为它不会补充编译器缺失的逻辑关系，只会把验证责任转移给运行时。

## `var` 不会让通配符消失

局部变量类型推断 `var` 只省略类型书写，不会改变初始化表达式的静态类型，也不会把捕获类型变成可自由读写的具体类型：

```java
static List<? extends Number> loadNumbers() {
    return List.of(1, 2, 3);
}

static void useNumbers() {
    var numbers = loadNumbers();
    Number first = numbers.get(0);
    // numbers.add(4); // 仍然编译错误
}
```

`numbers` 的静态类型仍然带有上界通配符。`var` 有时会保留一个源代码中无法直接书写的捕获类型，这能让紧邻表达式的某些操作通过编译，但不应依赖晦涩的捕获细节设计公共逻辑。若后续代码需要明确契约，应声明可读的接口类型或调用一个命名良好的泛型方法。

## API 参数设计：尽量接受所需的最小能力

公开 API 的参数类型应同时考虑实现所需能力和调用方可接受范围。若方法只遍历数字，`Iterable<? extends Number>` 通常比 `List<Number>` 更准确：前者不要求随机访问、不要求元素恰好为 `Number`，也不暗示方法会修改集合。

```java
static double average(Iterable<? extends Number> values) {
    double sum = 0.0;
    int count = 0;
    for (Number value : values) {
        sum += value.doubleValue();
        count++;
    }
    if (count == 0) {
        throw new IllegalArgumentException("values must not be empty");
    }
    return sum / count;
}
```

若方法只把结果写入调用方提供的接收器，可用下界：

```java
static <T> void repeat(
        T value, int count, Collection<? super T> destination) {
    for (int i = 0; i < count; i++) {
        destination.add(value);
    }
}
```

这种签名允许把 `String` 重复写入 `Collection<String>`、`Collection<CharSequence>` 或 `Collection<Object>`。它没有承诺目标集合最初只含有 `T`，也没有承诺写入后可以从该参数精确读回 `T`。

如果参数同时承担输入和输出，应区分“从同一对象读写同一种元素”与“从一个对象读、向另一个对象写”。前者通常需要 `List<T>` 或捕获辅助方法，后者通常可以分别使用 `? extends T` 和 `? super T`。把数据流拆成两个参数常能得到更灵活、也更容易推断的 API。

通配符还应尽可能放在最靠近变化的位置。例如需要读取若干个 `List<Number>`，到底应写 `List<? extends Number>`、`List<List<? extends Number>>`，还是 `List<? extends List<? extends Number>>`，取决于外层列表和内层列表哪一层需要接受子类型。每一层通配符都代表一层独立的未知类型，不能省略后凭直觉认为编译器会自动传播。

## 返回类型设计：避免把未知类型负担转给调用方

参数使用通配符通常能扩大调用范围；返回值使用通配符却常常缩小调用方能力。假设方法返回 `List<? extends Number>`，调用方可以读取 `Number`，但无法增加任何具体数字，也无法把结果直接赋给 `List<Number>`。如果实现实际上构造并拥有一个 `List<Number>`，返回通配符没有增加安全性，只是隐藏了本可承诺的精确信息。

因此，公共方法通常应返回明确的类型变量或具体参数化类型：

```java
static <T> List<T> immutableCopy(
        Collection<? extends T> source) {
    return List.copyOf(source);
}
```

输入使用 `? extends T` 允许复制 `T` 的子类型来源，返回 `List<T>` 给调用方一个稳定、可组合的类型。这里返回列表不可修改是 `List.copyOf` 的运行时和 API 契约，而不是通配符带来的效果。

返回通配符并非一概错误。当 API 的本质就是隐藏某个实现选择，或者返回值来自无法承诺精确实参的异构来源时，通配符可能是诚实契约。例如只允许调用方观察一组实现内部类型的对象，可以返回 `List<? extends PublicBase>`。但应评估调用方是否需要把结果传给其他泛型 API、保存到字段、合并或转换；若常见操作都需要捕获辅助代码，说明抽象边界可能放错了位置。

还要区分通配符与接口抽象。为了阻止调用方修改，不应单靠返回 `List<?>`，因为它仍可能支持 `clear` 或 `remove`。应返回不可修改视图、不可变副本或只暴露读取操作的接口，并在文档中明确所有权与可变性。

## 函数式接口中的输入与输出方向

函数式接口把 PECS 的方向表现得更明显。以映射操作为例，转换函数接收输入元素，因此它消费 `T`；转换结果被方法读取，因此它生产 `R`：

```java
static <T, R> List<R> map(
        List<? extends T> source,
        Function<? super T, ? extends R> mapper) {
    List<R> result = new ArrayList<>(source.size());
    for (T value : source) {
        result.add(mapper.apply(value));
    }
    return result;
}
```

`Function<? super T, ? extends R>` 允许函数参数类型比 `T` 更宽，返回类型比 `R` 更具体。例如处理 `String` 源时，可以传入接收 `CharSequence` 的函数；需要 `Number` 结果时，可以传入返回 `Integer` 的函数。这种双向通配符提高了高阶 API 的复用性。

比较器同理。排序方法把 `T` 值交给比较器，比较器消费 `T`，因此常见签名为 `Comparator<? super T>`。若写成 `Comparator<T>`，接收 `Object` 或某个公共父类型的比较器将无法用于更具体的元素类型，尽管它完全具备比较能力。

但复杂函数式签名也会增加错误信息和推断负担。如果一个公共方法包含多层嵌套通配符、多个相互依赖的类型变量和重载，调用者可能必须添加显式参数类型才能让 lambda 通过编译。此时应评估是否拆分方法、引入命名接口、减少重载，或用中间变量让目标类型更清楚。类型系统能表达某个关系，不代表把全部关系压缩进一个签名就是最佳 API。

## 常见编译错误与根因

### 把 `List<子类>` 赋给 `List<父类>`

```java
List<Integer> integers = new ArrayList<>();
// List<Number> numbers = integers;
```

根因是泛型不变性。若接收方只读，改为 `List<? extends Number>`；若接收方写入 `Integer`，改为 `List<? super Integer>`；若必须完整读写 `Number`，则调用方必须提供真正的 `List<Number>`，不能通过通配符伪造能力。

### 向 `? extends T` 写入 `T`

```java
static void addNumber(List<? extends Number> values) {
    // values.add(1);
}
```

根因是实际元素类型可能比 `Number` 更窄且不是 `Integer`。修复取决于需求：只读则删除写入；需要写整数则使用 `List<? super Integer>`；需要对精确类型读写则引入 `<T extends Number>` 并让写入值也为同一个 `T`。

### 从 `? super T` 直接读取为 `T`

```java
static void readNumber(List<? super Integer> values) {
    // Integer value = values.get(0);
}
```

根因是列表可能是 `List<Object>`，已有元素不受“写入整数”这一能力约束。只能读取为 `Object`，或把 API 改成能承诺精确元素类型的 `List<Integer>` / `List<T>`。未经校验的强转会把潜在错误推迟到运行时。

### 两个 `List<?>` 之间复制元素

```java
static void copyUnknown(List<?> from, List<?> to) {
    // to.add(from.get(0));
}
```

根因是两个独立通配符对应两个可能不同的捕获类型。若它们必须同类型，使用 `<T> void copy(List<T>, List<? super T>)`；若业务无法提供这种关系，就不能安全复制。

### 通配符捕获导致 `set` 失败

```java
static void duplicateFirst(List<?> values) {
    // values.set(0, values.get(0));
}
```

某些表达式形态下编译器不能在调用点维持需要的捕获关系。使用 `<T>` 辅助方法读取并写回。不要把值先扩大成 `Object` 再强转，因为强转无法证明它属于目标捕获类型。

### 条件表达式或链式调用推断出过宽类型

不同分支、不同集合元素或多个重载可能迫使编译器寻找公共上界，最终类型比预期更宽。可以用明确的目标变量、显式类型见证、拆分表达式或调整方法签名解决。首先确认“预期类型”是否真的由所有输入共同保证，避免为了得到窄类型而加入不安全转换。

### 原始类型让错误延迟出现

```java
List<String> names = new ArrayList<>();
List raw = names;
raw.add(123);

// String name = names.get(0); // 运行时可能抛出 ClassCastException
```

原始类型不是通配符。`List<?>` 保留“存在某个未知元素类型”的约束，编译器仍会阻止不安全写入；原始 `List` 则关闭大部分泛型检查并产生 unchecked 警告。与旧 API 交互时，应把原始类型转换限制在边界，先验证元素，再返回带有明确类型的结构。

## 不应通过强制转换解决类型关系

面对复杂错误，直接把 `List<?>` 转成 `List<T>` 看似能让代码通过，但由于类型擦除，运行时通常无法验证集合中的每个元素是否真是 `T`。这种转换会产生 unchecked 警告，并可能形成堆污染：变量声明的参数化类型与对象中实际可出现的元素不一致。

安全的转换必须有独立证据。例如外部数据逐个经过 `Class<T>.isInstance` 检查后，复制到一个新建 `List<T>`，那么边界内部可以集中处理转换；或者私有实现维护了编译器无法表达但经过严格封装的不变量。即便如此，也应把 `@SuppressWarnings("unchecked")` 放在最小作用域，并用注释说明证明依据。仅凭“调用方目前都这样用”或“测试没有失败”不足以证明安全。

通配符错误往往是在揭示 API 没有表达真实关系：两个参数本应共享类型却写成独立 `?`；目标本应消费源元素却写成精确同型；返回值本可明确却暴露未知类型。优先修复签名，通常比压制警告更简单，也能让后续调用者获得同样的静态保证。

## 重载、可读性与二进制层面的限制

通配符不同不一定能形成合法重载。由于类型擦除，以下两个方法擦除后都可能成为相同的参数类型，不能同时声明：

```java
// static void process(List<? extends Number> values) {}
// static void process(List<? super Integer> values) {}
```

即使源代码中的类型边界不同，方法描述符无法仅靠这些泛型信息区分调用目标。设计重载时应使用不同的方法名、不同参数个数或擦除后不同的原始参数类型。

复杂通配符还会影响可读性。`Map<? extends K, ? extends List<? extends V>>` 可能准确，却要求调用者理解多层独立未知类型。若这个结构频繁跨越模块边界，可以考虑定义具有领域含义的接口或值对象，把复杂边界封装在少数方法中。封装不是为了躲避类型系统，而是让不变量有明确所有者。

另一方面，不应仅为“签名简单”而把所有参数都退化成 `Object` 或原始类型。好的 API 需要在静态保证、调用灵活性和可读性之间平衡：对常见调用提供自然推断，对不安全组合在编译期拒绝，对复杂关系使用命名类型或分步操作表达。

## 通配符与数组协变的对比

Java 数组是协变的：`Integer[]` 可以赋给 `Number[]`。为了防止通过 `Number[]` 写入 `Double` 污染实际的 `Integer[]`，数组在运行时保存组件类型并执行写入检查，错误表现为 `ArrayStoreException`。泛型集合选择不变性和通配符，在编译期限制不安全写入。

```java
Number[] numbers = new Integer[1];
// numbers[0] = 1.5; // 编译通过，但运行时抛出 ArrayStoreException
```

这个对比说明，不变性不是语言无端增加的限制，而是把错误从运行时提前到编译期。`List<? extends Number>` 提供类似协变的读取视图，但不允许可能破坏实际元素类型的写入，因此不需要依赖运行时组件类型检查。

数组与泛型的机制不同，也解释了为什么不能从数组规则直接推导泛型规则。看到 `Number[]` 能接收 `Integer[]`，不能据此期待 `List<Number>` 接收 `List<Integer>`；正确对应物是能力受限的 `List<? extends Number>`。

## 方案权衡：精确类型、通配符还是重新建模

选择签名时，可以从以下问题逐步判断。

首先，方法是否需要把某个未知类型在多个位置关联起来。如果需要，使用类型变量命名它；如果不需要，只表达边界即可使用通配符。

其次，参数相对于该类型主要是生产、消费，还是双向读写。生产考虑 `extends`，消费考虑 `super`，双向精确读写考虑 `T`。若只是结构操作且元素类型无关，考虑 `?`。

再次，放宽参数是否真的增加有价值的调用场景。公共、复用度高的集合算法通常受益明显；局部私有方法若只有一个精确调用者，复杂通配符可能只增加认知成本。

然后，返回值是否能够承诺更明确的类型。若能，避免返回通配符；若不能，确认未知类型确实是抽象的一部分，并为常见后续操作提供辅助 API。

最后，若签名需要多层通配符、多个捕获辅助方法和大量显式类型见证，问题可能不再是“如何写泛型”，而是数据模型缺少稳定抽象。可以考虑引入只读接口、消费者接口、转换对象或具有明确所有权的容器类型。重新建模有额外代码成本，但可能显著降低调用端复杂度并集中维护不变量。

这些选择没有脱离上下文的绝对优先级。精确类型提供最强能力但接受范围较窄；通配符扩大兼容范围但主动限制操作；命名抽象提高可读性和可维护性但增加类型数量。合理方案应让合法调用自然、非法调用尽早失败，并让方法体能够在不依赖 unchecked 转换的情况下实现。

## 实践检查清单

设计或审查包含通配符的 Java API 时，可以依次确认：

1. 是否因为泛型不变性而需要放宽某个参数，而不是误以为 `List<Sub>` 本来就应当是 `List<Super>`。
2. 每个 `?` 是否代表一个独立未知类型；需要相同类型的位置是否已用类型变量连接。
3. `? extends T` 参数是否主要提供 `T`，并且实现没有尝试写入具体 `T`。
4. `? super T` 参数是否主要接收 `T`，并且实现没有假定读取结果一定是 `T`。
5. 是否把 `List<?>` 错当成不可变集合，忽略了删除、清空等修改能力。
6. 是否机械套用 PECS，而没有考虑双向读写、返回值、所有权和可变性。
7. 捕获错误能否通过泛型辅助方法表达同一次捕获，而不是通过强转绕过。
8. 类型推断是否能服务常见调用；lambda、方法引用和链式调用是否需要过多显式类型。
9. 返回类型是否把本可明确的信息隐藏成通配符，给调用方造成不必要负担。
10. 是否存在原始类型、过宽的 `Object` 或 unchecked 转换形成的堆污染入口。
11. 多层嵌套通配符是否仍然可读；若关系复杂，是否应引入命名抽象。
12. 方法重载在类型擦除后是否仍具有不同签名。

## 总结

Java 泛型通配符的核心是“受边界约束的未知类型”。无界通配符表示元素类型与当前操作无关；上界通配符允许把不同子类型容器作为生产者观察；下界通配符允许把父类型容器作为某个具体类型的消费者使用。它们都没有改变泛型的不变性，而是在使用位置建立能力受限的型变视图。

读写规则来自边界方向：`CAP extends T` 能保证读取为 `T`，却不能保证写入某个具体 `T`；`T extends CAP` 能保证写入 `T`，却只能把既有元素读取为 `Object`。PECS 是这一推导的实用概括，但它只适用于能明确识别数据流向的参数，不能代替对双向操作、返回值、类型同一性和 API 所有权的分析。

当未知类型需要在多个位置保持一致时，类型变量和捕获转换负责给它命名。编译器显示的 `CAP#n` 不是神秘的运行时类型，而是静态分析中对某次通配符捕获的标记。读懂捕获的上下界，通常就能判断错误是签名缺少关系、操作方向不对，还是调用本身确实不安全。

最终，通配符设计的目标不是写出最复杂的泛型签名，而是准确分配能力：方法只获得实现所需的最小权限，调用者保留尽可能广的合法类型，类型推断能够覆盖常见用法，不安全组合在编译期被拒绝。做到这一点，通配符才真正从语法技巧变成可维护 API 的契约工具。

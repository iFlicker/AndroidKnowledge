# 3.2.2.1 CopyOnWriteArrayList

`CopyOnWriteArrayList` 是 `java.util.concurrent` 包提供的线程安全列表。它保留了 `List` 的顺序、按索引访问、允许重复元素等基本语义，却没有把每次读取都放进互斥区，而是把并发协调的主要成本集中到写操作：写线程先复制当前底层数组，在副本上完成修改，再一次性发布新数组；已经开始的读取和遍历继续使用旧数组，不必等待写线程。

这种设计常被概括为“写时复制”。名称虽然直观，却容易掩盖真正的选择条件。它不是一个在并发环境中普遍优于 `ArrayList` 的实现，也不是“读无锁”就必然高性能。它用写入的线性复制、写线程之间的串行化、额外分配和旧快照存活，换取单次读取路径短、遍历期间没有结构干扰、迭代器无需由调用方加锁等性质。只有当这些收益正好解决主要矛盾时，这笔交换才值得。

本文以 Java SE 的公开 API 契约为结论依据。涉及内部字段、锁对象和数组复制方式时，会明确说明它们属于特定 OpenJDK 版本的实现观察，而不是所有 Java 实现必须采用的结构。`CopyOnWriteArrayList` 自 Java 5 引入；较新的 `SequencedCollection` 相关方法则应按实际使用的 JDK API 文档判断，不能把新版本方法写进需要兼容旧版本的源码。

## 写时复制解决的不是“有没有锁”，而是“谁承担协调成本”

共享列表最直接的线程安全方案，是让读写都遵守同一个互斥协议。例如使用 `Collections.synchronizedList` 后，单个方法通过同步包装得到保护；遍历时，调用方还必须按文档要求在返回的列表对象上同步，否则一次遍历包含的多个方法调用无法被同一个临界区覆盖。这个方案的优点是修改通常在原结构上完成，不需要每次复制整个列表；缺点是高频读取和长时间遍历也会参与锁竞争，并可能阻塞写入。

`CopyOnWriteArrayList` 把策略反过来。读线程取得某个已经发布的数组引用后，只读取该数组；写线程不能原地修改这个数组，因为旧读者仍可能持有它。写线程必须构造一个新数组，修改新数组，然后把列表当前状态切换到新数组。于是，同一时刻可以存在多个逻辑版本：

- 列表字段指向最新数组；
- 某个较早创建的迭代器仍引用旧数组；
- 一个写线程正在锁内基于当前数组构造下一版本；
- 更早的数组只要仍被迭代器、分割迭代器或其他内部快照引用，就不能被垃圾回收。

关键不在于系统从此没有同步，而在于读取不再与写入争夺同一把锁。所有会改变列表结构或元素槽位的操作仍需要协调，否则两个写线程可能都基于同一旧数组生成不同结果，后发布者会覆盖先发布者的修改。写锁因此负责把“读取当前版本、计算新版本、发布新版本”串成不可分割的写入过程。

这种成本迁移适合以下组合，而不是一个孤立的“读多写少”标签：

1. 列表通常较小或中等，完整复制引用数组的成本有上界。
2. 遍历和读取远多于修改，而且遍历可能较长，不希望持有外部锁。
3. 允许正在进行的遍历看到创建迭代器时的旧成员集合，不要求立刻反映后续变更。
4. 修改延迟和分配量可以接受，写入不是吞吐核心。
5. 元素顺序有意义，或者调用方需要 `List` 而不只是集合成员关系。

“远多于”不能用固定比例替代测量。一个只有十几个元素、每秒修改数次而每秒读取数十万次的监听器列表，可能非常合适；一个包含几十万元素、每秒只写一次的列表，也可能因为每次复制的数据量、尾延迟和内存峰值而不可接受。读写比例必须与列表长度、写入批次、对象存活时间、延迟目标和垃圾收集行为一起评估。

## 公开语义与常见 OpenJDK 实现要分开理解

Java SE API 对外承诺的是线程安全的 `List`、写时复制的修改方式、快照式迭代器、迭代器不反映创建后的变更、迭代器修改方法不受支持，以及并发集合对应的内存一致性效果。调用方可以依赖这些公开语义。

以 OpenJDK 11、17 和 18 的源码为例，常见实现包含一个 `private transient volatile Object[] array` 字段，并用一个内部锁保护所有修改操作。这些版本中的锁字段是普通 `Object`，写方法通过内置监视器同步；更早的 OpenJDK 版本曾使用 `ReentrantLock`。因此，“`CopyOnWriteArrayList` 内部一定使用 `ReentrantLock`”“写锁一定公平”或“可以通过反射取得某种具体锁并参与同步”都不是可靠结论。锁类型是实现细节，应用代码不应依赖它。

同样，`volatile` 数组字段是上述 OpenJDK 版本用于发布新快照的实现方式，而不是说数组中的每个槽位都具有独立的 `volatile` 语义。数组对象本身在发布后不再被该列表原地修改，写线程把完整的新数组引用写入 `volatile` 字段，后续读线程读取该字段时获得一个已构造完成的版本。可以把它理解为“易变的是当前版本指针，版本内容在发布后保持结构不变”。

这里的“结构不变”只针对数组槽位和长度。槽位里存放的是元素引用，不是元素对象的深拷贝。元素内部字段仍然可以变化，后文会单独讨论这一边界。

OpenJDK 的典型读取路径会先把数组字段读取到局部变量，再在该数组上完成长度检查、索引访问或线性搜索。这样一次方法执行围绕一个数组版本展开，不会在搜索中途反复切换版本。典型写入路径则在锁内读取当前数组，创建新数组，复制引用，完成插入、替换或删除，再发布新数组。源码细节会随 JDK 演进，但这种快照读和串行写的总体机制与公开语义一致。

## `get` 很短，但多个读取不会自动组成同一快照

单次 `get(index)` 的核心成本接近普通数组索引：取得当前数组引用，检查索引，从对应槽位返回元素引用。它不需要获得写锁，因此不会因为另一个线程正在执行较慢的写操作而排队等待锁。不过，“单次读取使用某个稳定数组”不等于“连续多个方法调用都使用同一个数组”。

考虑以下代码：

```java
int size = list.size();
E last = list.get(size - 1);
```

`size()` 和 `get()` 是两个独立操作。第一个操作可能观察数组 A，随后另一个线程删除末尾元素并发布数组 B，第二个操作便可能在数组 B 上执行并抛出 `IndexOutOfBoundsException`。容器没有损坏，每个方法也都符合自己的契约；错误来自调用方把两个瞬时观察错误地拼成了一个原子事务。

类似地，先 `isEmpty()` 再 `get(0)`、先 `contains(x)` 再 `remove(x)`、先读取两个索引再假定它们来自同一列表版本，都需要重新审视。若业务需要固定成员快照，可以使用一次 `toArray()` 得到独立数组，或通过一次迭代器创建固定遍历视图；若业务需要“检查并修改”原子性，则应优先寻找类本身提供的原子组合方法，或者在更高层建立同步协议。仅仅因为单个方法线程安全，复合逻辑不会自动线程安全。

`contains`、`indexOf` 和 `lastIndexOf` 虽然不加写锁，却需要线性扫描当前数组，复杂度为 O(n)。读操作“无锁”也不等于所有读操作都是 O(1)。元素的 `equals` 还会进入搜索成本；如果 `equals` 昂贵、阻塞、修改共享状态或违反对称性与一致性，容器无法修复这些问题。

`size()` 返回某次读取到的数组长度。它不是近似计数，但返回之后列表可能立刻变化，因此这个准确值仍然只是调用时刻附近某个已发布版本的大小。并发程序应区分“返回值来自完整版本”与“返回值在后续步骤仍然有效”。

## 快照迭代器提供稳定成员序列，而不是实时视图

调用 `iterator()` 或 `listIterator()` 时，迭代器保存当时的数组引用。之后即使其他线程添加、删除或替换元素，迭代器仍按旧数组的顺序继续访问。旧数组不会被列表写线程修改，因此迭代过程不存在普通 `ArrayList` 那种结构性干扰，也不会因为列表并发修改而抛出 `ConcurrentModificationException`。

这不是弱一致遍历。许多并发容器的迭代器可能在遍历过程中观察到部分并发变更，但不保证形成某个单一时刻的完整快照；`CopyOnWriteArrayList` 的迭代器则固定在创建时取得的数组版本。两者都可能不反映最新状态，形成结果的方式却不同。

快照语义带来几个必须明确的结果：

- 迭代器创建后新增的元素不会出现；
- 创建后从列表删除的元素仍可能被这个迭代器访问；
- 创建后通过 `set` 替换的槽位不会改变旧迭代器保存的元素引用；
- 不同时间创建的两个迭代器可以遍历不同版本；
- 慢迭代器不会阻塞写线程，但会延长旧数组以及旧数组中元素引用的存活时间。

迭代器自身的 `remove`，以及列表迭代器的 `set`、`add`，都不受支持并抛出 `UnsupportedOperationException`。原因不只是实现偷懒：一个快照迭代器面对的索引属于旧数组，当前列表可能早已变化。若让迭代器修改当前列表，就必须定义旧索引如何映射到新版本、遇到重复元素删除哪一个、并发删除后是否重试等复杂语义。该类选择让迭代器纯读取，从根本上消除这类歧义。

增强 `for` 循环使用迭代器，因此也遵循快照语义。`forEach`、流和分割迭代器的具体实现应以所用 JDK 的 API 文档为准；在常见 JDK 中，它们同样基于稳定数组处理元素。不能因为一次遍历稳定，就把遍历中执行的业务动作理解为全局原子：回调可能持续很久，其他线程可以在此期间发布任意多个新列表版本。

下面的程序展示了“本轮通知成员固定，下一轮才看到注册变化”的效果：

```java
import java.util.concurrent.CopyOnWriteArrayList;
import java.util.function.Consumer;

public final class ListenerRegistry<T> {
    private final CopyOnWriteArrayList<Consumer<? super T>> listeners =
            new CopyOnWriteArrayList<>();

    public void register(Consumer<? super T> listener) {
        listeners.addIfAbsent(listener);
    }

    public void unregister(Consumer<? super T> listener) {
        listeners.remove(listener);
    }

    public void publish(T event) {
        for (Consumer<? super T> listener : listeners) {
            listener.accept(event);
        }
    }

    public int listenerCount() {
        return listeners.size();
    }
}
```

如果某个监听器在 `publish` 期间注销自己，它仍会完成当前迭代器已经确定的这一轮；后续新建迭代器时才不再出现。这种语义通常比“遍历到一半突然跳过成员”更容易推理。不过，容器只负责成员快照，不负责异常隔离。某个监听器抛出异常时，是否继续通知其余监听器，应由 `publish` 的业务策略明确处理。

## 每一种修改都要支付新数组成本

“修改”不只包括改变长度。`add`、`set`、`remove`、`clear`、批量添加、批量删除、条件删除、排序和整体替换等操作，只要改变列表可观察内容，通常都需要生成并发布一个新数组版本。写线程还要在同一内部锁上串行执行，所以写负载升高时同时出现复制成本和锁竞争。

### 尾部添加

在尾部执行 `add(e)` 时，典型实现创建长度为 n+1 的数组，把旧数组的 n 个引用复制过去，再写入新元素。时间复杂度和复制量都是 O(n)，空间上还需要一个新数组。与会预留容量、偶尔扩容的 `ArrayList` 不同，`CopyOnWriteArrayList` 不能简单保留一个可原地填充的多余容量区，因为旧快照必须保持不变，而每次发布的数组长度直接表示该版本大小。

连续执行一万次单元素 `add`，不是只复制最终一万个引用，而是依次复制长度 0、1、2 直到 9999 的数组，总复制量呈二次增长。若确实要一次加入很多元素，`addAll` 通常能在一次写锁和一次新数组构造中完成，比循环调用 `add` 更符合该结构的成本模型。

### 指定位置插入

`add(index, e)` 除了分配 n+1 的数组，还要复制插入点之前和之后的两个区间。它与尾部添加同为 O(n)，但中间插入包含额外的区间搬移。位置越靠前，需要后移的引用越多；从渐进复杂度看没有变化，从常数和内存带宽看仍有差异。

### 替换元素

`set(index, e)` 不改变长度，仍不能直接写入当前数组槽位，因为旧迭代器正在依赖那个数组。典型路径复制整个长度为 n 的数组，在副本对应位置放入新引用，再发布副本。因此 `set` 是 O(n)，不能按普通数组的 O(1) 替换来估算。

某些 OpenJDK 实现会对“新旧引用是同一个对象”的情况作细节优化或执行一次发布写，以保持特定内存语义；这类分支不属于应用层应依赖的性能契约。工程判断应保守地把 `set` 当成写操作，并避免用它高频更新每个元素。

### 按索引删除和按值删除

`remove(index)` 创建长度 n-1 的数组，复制删除点前后的元素。按值 `remove(o)` 还要先线性搜索匹配项，再复制保留区间。若搜索在锁外先做，写锁内仍需要确认列表是否已经变化；具体重试和校验方式属于实现细节。公开结果是删除第一个匹配元素，并保证单次方法调用的线程安全语义。

在包含大量重复元素时，按值删除只删除第一个相等元素。元素的相等性如果会随时间变化，搜索结果也会变化。尤其是元素在放入后修改了 `equals` 使用的字段，`contains`、`remove`、`addIfAbsent` 的行为可能与调用方基于旧值形成的直觉不一致。这是可变元素与值相等语义的问题，不是写时复制能解决的问题。

### 清空和批量修改

`clear()` 可以发布一个空数组，不必复制所有旧引用，但旧数组若仍被迭代器引用，仍然继续存活。清空列表只切断“最新版本”对元素的引用，并不保证所有快照立即释放元素。

`addAll`、`removeAll`、`retainAll`、`removeIf`、`replaceAll` 和 `sort` 等批量方法的实现会随 JDK 版本调整，但合理预期是：它们在一次受控写入过程中计算新内容并一次发布，而不是公开中间状态。调用方仍应注意传入集合、谓词、运算符和比较器的成本。若回调执行时间长，写锁持有时间可能随之增加；若回调重入修改同一个列表，具体行为、复杂度和可维护性都会恶化，应避免把有副作用的逻辑塞进批量操作。

批量方法的价值之一，是减少“每个元素都复制一次”的灾难性成本。例如先在局部 `ArrayList` 中准备完整批次，再执行一次 `addAll`，通常比在共享列表上循环 `add` 更合理。但这并不让批量写变成廉价操作：最终仍要构造包含旧元素和新元素的新数组，批次越大，瞬时分配越大。

## `addIfAbsent` 与 `addAllAbsent` 是成员去重操作，不是集合类型转换

`CopyOnWriteArrayList` 允许重复元素，也允许 `null`。普通 `add` 不做去重。`addIfAbsent(e)` 提供一个原子组合语义：当列表中没有与 `e` 相等的元素时，把它追加到末尾并返回 `true`；如果已经存在，则不添加并返回 `false`。它解决了下面这种“先检查、后执行”竞态：

```java
if (!list.contains(listener)) {
    list.add(listener);
}
```

两个线程可能同时发现元素不存在，再各自添加一次。把两步换成 `addIfAbsent` 后，检查与可能的追加由容器作为一个写操作协调。这里的“存在”依据列表的相等性规则，通常可理解为 `Objects.equals` 语义，不是引用身份。

原子不代表便宜。缺失时，方法需要确认不存在并复制数组；存在时，也至少需要搜索。竞争场景下，实现可能先在无锁快照中搜索，再进入写锁后根据当前版本重新校验，以避免错误地加入重复项。列表越大、候选越多、`equals` 越昂贵，去重成本越明显。

`addAllAbsent(c)` 会把输入集合中尚未出现在列表里的元素追加到末尾，并返回实际加入的数量。它还需要处理输入集合自身的重复项：某个值第一次被选中后，同批次后面的相等值不应再次加入。最终追加顺序遵循输入集合的迭代顺序，但只有首次判定为缺失的元素进入结果。

这个方法适合小规模注册表的一次批量补充，却不应被当作高性能集合并集。设当前列表有 n 个元素，输入有 m 个元素，朴素去重需要多次线性比较，成本可能接近 O(nm + m²)，具体优化依赖实现。若核心需求是高频并发成员去重而不关心索引和重复，基于 `ConcurrentHashMap` 的并发集合通常更贴合问题；若需要保序去重并批量发布，可以先在局部结构中完成构建，再发布不可变快照。

相等性还会影响监听器注册。方法引用、捕获 lambda 或包装对象每次创建时不一定与上一次对象相等，`addIfAbsent` 因而未必能阻止“逻辑上同一个回调”的重复注册。若去重依赖业务标识，应显式设计稳定键，而不是假设所有函数对象都有值相等语义。

## 内存一致性保证到“元素被访问”，不等于元素永远线程安全

Java SE 文档为该类声明了与其他并发集合相同的内存一致性效果：一个线程在把对象放入 `CopyOnWriteArrayList` 之前执行的动作，先行发生于另一个线程随后从列表中访问或移除该元素之后的动作。这个保证使列表可以承担元素引用的安全传递媒介。

例如，线程 A 先完整构造一个配置对象，再把它加入列表；线程 B 随后通过列表取得该对象时，可以观察到发布前的初始化结果，前提是程序没有在构造完成前泄漏对象引用，也没有其他数据竞争破坏对象状态。底层实现中的锁和新数组发布共同建立必要的可见性关系，但应用代码应依赖公开的内存一致性契约，而不是自行推断某条源码指令。

保证有明确边界：

1. 它不把元素变成不可变对象。
2. 它不让元素发布后的任意普通字段修改自动对其他线程可见。
3. 它不保护两个元素之间的业务不变量。
4. 它不使“先修改元素，再读取列表大小”之类无关组合自动建立事务。

假设列表中保存一个可变 `ArrayList`，多个线程从 `CopyOnWriteArrayList` 取出后同时修改内部列表，外层容器仍然完好，内部列表却可能发生数据竞争。又如元素包含 `int count`，线程取得元素后执行 `count++`，该复合读改写仍然不是原子的。解决方式是让元素不可变、使用原子字段或适当锁，或者用新元素替换旧元素并把状态更新纳入明确的发布协议。

写时复制是浅复制。新旧数组的相同位置通常引用同一个元素对象，因此旧迭代器和新列表版本可以同时看到同一个可变对象的内部变化。于是，“快照迭代器”准确含义是元素引用序列的快照，不是对象图的深度快照。若业务要求历史版本中的元素内容也冻结，元素本身必须不可变，或者创建深拷贝、持久化数据结构或专门的版本对象。

## `subList` 是受约束的视图，不是独立快照

`subList(from, to)` 返回的是原列表指定区间的视图，而不是复制出来的新 `CopyOnWriteArrayList`。视图中的修改会作用于父列表对应区间；父列表的变化也会影响视图是否仍可合法使用。把它当成独立列表跨线程长期保存，是常见风险。

`CopyOnWriteArrayList` 的子列表实现通常记录创建视图时父列表的预期数组和区间信息。子列表方法执行时会检查父列表当前数组是否仍与预期一致。如果父列表通过子列表之外的路径发生修改，视图的区间映射可能失效，后续操作可能抛出 `ConcurrentModificationException`。这里即使父列表本身是线程安全的，也不代表旧子列表能自动把边界重新映射到新数组。

这与父列表的快照迭代器形成鲜明对比：父列表迭代器刻意固定旧数组并继续工作；子列表则是可修改的范围视图，需要保持“第几个元素到第几个元素”与当前父列表之间的对应关系。父列表在区间前插入一个元素后，旧边界究竟应该跟随原元素还是保留原索引，并没有唯一答案，所以实现选择检测外部修改并拒绝继续。

因此：

- 临时在受控代码段中操作子区间，可以使用 `subList`；
- 需要独立、稳定的数据时，应复制子列表内容；
- 不要把子列表长期暴露给不知道父列表并发协议的调用方；
- 不要认为对子列表加外部锁就能约束绕过该锁直接修改父列表的代码；
- 对子列表返回的迭代器，也要按其具体视图和创建时快照语义理解，不能与父列表最新状态混为一谈。

下面这种返回方式把内部视图及其失效条件泄漏给了调用方：

```java
public List<Task> firstPage(int pageSize) {
    return tasks.subList(0, Math.min(pageSize, tasks.size()));
}
```

若需要返回稳定结果，可在一次明确的快照上计算范围。例如先 `toArray` 或用当前迭代器收集元素，再生成不可修改列表。这里仍要处理“读取大小”和“创建范围”之间的复合操作，不能简单把普通列表写法原样搬过来。

## 复合操作必须按业务原子性重新设计

线程安全容器只保证文档承诺的方法语义。跨多个方法的条件判断、跨多个列表的转移、列表与外部计数器的同步，都不自动成为一个原子操作。

一个典型错误是按索引读取后再按索引删除：

```java
E first = list.get(0);
E removed = list.remove(0);
```

两个调用之间可能有其他线程在头部修改列表，`first` 与 `removed` 不一定是同一对象。若目标是删除某个刚观察到的元素，可以使用按值删除并检查返回值，但在存在重复元素时仍要定义“删除哪一个”的业务含义。若目标是并发消费队首元素，列表本身就不是合适抽象，应使用并发队列。

另一个错误是用 `size()` 作为后续循环的固定边界并在每次循环中调用 `get(i)`。每次 `get` 都可能读取不同数组版本，长度缩短时会越界，长度增长时又不会自动纳入新元素。遍历应直接使用一次迭代器，从而固定成员序列：

```java
for (E element : list) {
    use(element);
}
```

需要条件添加时使用 `addIfAbsent`，需要条件批量添加时使用 `addAllAbsent`，是让类提供的单方法原子语义替代手写组合。对于类没有提供的事务，例如“列表 A 删除成功后必须向列表 B 添加，且任何线程不能看到中间状态”，应在更高层用共同锁、状态对象或单一不可变快照统一发布。两个线程安全列表不能自动组成线程安全事务。

不要尝试在 `CopyOnWriteArrayList` 实例上使用 `synchronized (list)` 来参与其内部写锁。内部锁不是公开监视器，外部同步与内部同步没有共同协议。外部锁只有在所有相关调用方都自觉遵守时，才能保护应用层复合操作；它并不会阻止不持有该外部锁的代码直接调用列表方法。

## 内存峰值不只是“旧数组加新数组”

一次写入至少要考虑当前数组和新数组同时存在。若当前数组包含 n 个引用，新数组包含约 n 个引用，复制期间数组存储的瞬时占用接近两份；插入或批量添加还会更大。数组复制完成后，列表字段不再引用旧数组，但旧数组是否立即可回收取决于其他引用。

快照迭代器是延长旧数组寿命的主要来源。一个线程创建迭代器后长时间阻塞、把迭代器保存到字段、在惰性流中迟迟不消费，都会让旧数组保持可达。旧数组又强引用其中每个元素，所以即使这些元素已经从最新列表删除，也不能被回收。若连续修改很多次且多个长生命周期迭代器分别持有不同版本，堆中可能同时存在多个完整数组版本。

这会形成三类压力：

- 分配速率升高：每次写都产生新数组，频繁写时产生大量短命对象；
- 存活集扩大：慢迭代器让旧数组晋升或长期存活；
- 引用保留：已删除的大对象仍被旧快照间接引用。

因此，列表元素本身很大时，复制的虽然只是引用而不是对象内容，旧数组仍可能延长大对象生命周期。评估内存不能只计算引用数组字节数，还要看这些引用维持的对象图。

批量写可以降低数组版本数量，却可能制造更大的单次分配和复制停顿。选择批次大小时需要在吞吐、尾延迟和内存峰值之间平衡。对于持续高频更新的大型数据集，写时复制通常不是靠调参数就能解决的问题，而是结构与负载不匹配。

## 性能退化通常从写路径、缓存与回收同时出现

在理想场景中，读取只需读取数组引用并索引，遍历连续数组也有良好的局部性。随着负载偏离假设，退化并不只表现为“写慢一点”。

首先，所有写线程在同一内部锁上排队。即使写入不同索引，它们也不能并行修改，因为每个新版本必须基于确定的前一版本。写线程数增加不会线性提升吞吐，反而可能增加上下文切换和锁竞争。

其次，每次写入都复制整个引用数组。列表越长，单次写延迟越高，内存带宽消耗越大。若写操作还执行线性搜索、谓词或比较器，总成本可能是扫描加复制。`addIfAbsent`、按值删除和去重批量添加尤其需要关注这一点。

再次，新数组发布会改变读线程访问的数组对象。虽然旧迭代器稳定，但频繁版本切换会增加缓存压力。具体硬件、虚拟机优化和垃圾收集器会显著影响结果，不能从“数组连续”直接推导出固定性能数字。

最后，高分配率会把问题转移给垃圾收集器。基准测试若只测极短时间、列表很小或没有保留迭代器，可能得出过于乐观的结论。可靠测试至少要覆盖真实列表长度、元素引用形态、读写比例、并发写线程数、批量大小、迭代持续时间和稳定运行后的分配率。吞吐之外还要观察写入分位延迟、暂停、堆占用和对象晋升。

以下信号通常说明应该重新选择结构：

- 写入从偶发配置变更演变为持续状态更新；
- 列表增长到每次复制都明显影响尾延迟；
- 大量线程同时调用 `add`、`remove` 或 `set`；
- 业务把列表当作队列，频繁从头部取走元素；
- 业务把列表当作映射，频繁按某个键线性查找；
- 长生命周期迭代器导致已删除对象迟迟不释放；
- 为弥补成本而堆叠复杂缓存、分片或反射优化。

最后一种尤其值得警惕。容器选择错误时，围绕它继续添加补丁通常比替换为匹配访问模式的结构更难维护。

## 监听器列表：最典型，但仍有未解决的问题

监听器注册表通常规模小、注册和注销少、事件分发多，而且希望分发时不持有一个会被用户回调拖住的锁。`CopyOnWriteArrayList` 正好让一次分发取得固定成员序列，注册变化在下一轮生效。

然而，容器没有解决以下问题：

- 回调抛出异常时是否中止后续通知；
- 回调执行缓慢时是否阻塞发布线程；
- 同一个监听器能否重复注册；
- 注销返回后，已经开始的分发是否仍允许调用它；
- 回调是否可以重入注册、注销或再次发布；
- 监听器对象的生命周期是否因注册表强引用而过长。

尤其是“注销返回后绝不再调用”这一强保证，与快照迭代语义冲突。注销只能阻止未来创建的快照包含该监听器，无法撤销已经创建的迭代器。若需要严格停止，需要额外状态，例如监听器包装器中的原子启用标记；分发前检查标记，注销先关闭标记再从列表移除。即使如此，检查与调用之间是否允许竞态，仍需按业务定义。

监听器回调可能修改注册表，但应清楚结果：当前分发快照不变，修改只影响后续分发。这个性质往往是优点，也应写进注册表自己的契约，而不是要求使用者猜测底层容器行为。

## 配置快照：比较“元素列表”与“整体状态”

另一类常见场景是共享规则、处理器链、路由条目或功能配置。读取线程频繁遍历，管理线程偶尔替换内容。`CopyOnWriteArrayList` 能支持逐项增删，但逐项修改会发布多个中间版本。例如把旧规则集合逐条清除再逐条加入新规则，读线程可能观察到空列表或半更新列表。

如果一次配置变更在业务上是整体事务，更合适的方式通常是先在局部构建完整不可变列表，再通过一个 `volatile` 字段或 `AtomicReference<List<T>>` 一次替换引用。这样发布单位就是整份配置，而不是一个个元素操作：

```java
import java.util.List;
import java.util.concurrent.atomic.AtomicReference;

public final class RuleSnapshot<R> {
    private final AtomicReference<List<R>> current =
            new AtomicReference<>(List.of());

    public List<R> current() {
        return current.get();
    }

    public void replaceAll(List<? extends R> rules) {
        current.set(List.copyOf(rules));
    }
}
```

该方案没有可变列表 API，调用方不能对共享状态逐项写入；一次替换直接表达配置版本边界。`List.copyOf` 不允许 `null`，而 `CopyOnWriteArrayList` 允许 `null`，迁移时必须确认元素约束。`List.copyOf` 也是浅层不可变，元素自身仍应满足相应线程安全要求。

若确实需要并发独立注册和注销，`CopyOnWriteArrayList` 的原子单项操作更方便；若主要操作是整体刷新，不可变快照通常语义更清晰、分配次数更可控。二者都使用“新版本替换旧版本”的思想，但更新粒度不同。

## 与同步列表、不可变快照和其他并发结构的权衡

### `Collections.synchronizedList`

同步列表适合读写都需要在同一互斥协议下进行，或者修改频率足以让全量复制不可接受的场景。它通常在原底层列表上修改，因此单次 `set`、尾部 `add` 的成本模型更接近被包装列表。代价是读取也要同步，遍历必须由调用方手动在列表上锁住整个过程。长遍历会阻塞修改，忘记锁住遍历又会破坏协议。

选择依据不是“一个有锁、一个没锁”，因为 `CopyOnWriteArrayList` 的写入同样有锁。真正区别是同步列表让读写共享互斥区，写时复制列表让读者持有不可变数组版本。

### `List.copyOf`、不可修改列表与原子引用

若数据构建后长期不变，直接安全发布不可变列表比保留可修改的并发列表更简单。若更新总是整体发生，原子替换不可变列表能让每次读取获得一个完整版本。它不提供并发逐项修改，但这可能正是避免错误 API 的优势。

要区分“不可修改视图”和“独立不可变副本”。`Collections.unmodifiableList(source)` 只是阻止通过视图修改，底层 `source` 变化仍会反映出来；`List.copyOf(source)` 通常得到不受后续源列表结构修改影响的不可修改结果。两者都不是元素深拷贝。

### 并发映射或并发集合

如果核心操作是按稳定键注册、查询、删除，`ConcurrentHashMap<K,V>` 比列表线性搜索更合适。若只关心成员关系，可使用基于并发映射的键集合视图。它们不提供与列表相同的索引和快照顺序语义，迭代一致性也不同，不能仅为性能替换而忽略契约变化。

监听器需要确定调用顺序时，列表优势明显；处理器只按 ID 查找时，映射通常更直接。既要按键定位又要稳定有序遍历，可能需要由单一状态对象同时维护索引和顺序，并整体发布，而不是让两个独立并发容器勉强保持一致。

### 并发队列与阻塞队列

若元素不断加入并被取走，业务表达的是生产消费，而不是共享快照列表。`ConcurrentLinkedQueue`、`LinkedBlockingQueue`、`ArrayBlockingQueue` 等结构提供更匹配的入队、出队、阻塞和容量语义。用 `CopyOnWriteArrayList` 反复删除头元素会在每次消费时复制剩余数组，规模上升后成本非常差。

### 读写锁保护的普通列表

`ReentrantReadWriteLock` 可以允许多个读者并行，并让写者在原列表上修改。它适合需要明确锁定多个读取步骤或需要在锁内完成复合操作的设计，但读锁获取、释放、升级限制、公平性和长读临界区都会增加复杂度。它的迭代不是自动快照；只要依赖列表稳定，就必须持有读锁。

对于短读、偶发写和小列表，`CopyOnWriteArrayList` 往往更容易使用；对于大列表、写入不能全量复制、又需要跨方法一致读取的场景，显式读写锁可能更合适。最终仍需基于真实负载验证。

## 元素设计决定快照是否真正容易推理

最理想的元素是不可变值对象：构造完成后字段不再变化，相等性稳定，读取不需要额外同步。这样旧数组和新数组共享元素引用不会引入内容漂移，快照的成员与成员状态都更接近一个可解释版本。

若元素可变，应至少回答：

- 哪个线程拥有修改权；
- 修改字段用什么可见性与原子性协议；
- `equals` 使用的字段是否会变化；
- 回调执行时元素能否被关闭、替换或失效；
- 旧迭代器访问已从最新列表删除的元素是否安全；
- 元素持有的资源何时释放。

资源对象尤其需要谨慎。把连接、文件句柄或可关闭服务放进列表后，删除并立即关闭，旧迭代器仍可能取得该对象并使用。容器的快照语义意味着“从成员表删除”和“没有任何进行中的使用者”不是同一个事件。需要引用计数、租约、停止标记、分阶段关闭或其他生命周期协议时，应在元素和注册表层显式设计。

`null` 是被允许的元素。虽然这符合公开契约，工程 API 仍可选择禁止 `null`，以免 `null` 同时表示合法成员、缺失值或初始化错误。禁止策略应在写入入口主动校验，而不是依赖该容器抛出异常，因为它不会替应用拒绝 `null`。

## 版本差异与源码阅读的正确边界

`CopyOnWriteArrayList` 的核心公开语义长期稳定，但接口随 Java 平台演进会增加能力。例如 Java 21 随 `SequencedCollection` 引入后，`List` 获得首尾访问、首尾修改和反向视图等相关方法；这些方法是否可用取决于编译和运行目标版本。讲解或示例若面向较早 JDK，应避免直接使用新方法，或者明确标注最低版本。

内部实现变化更不应成为调用方契约。OpenJDK 8 的源码可见 `ReentrantLock`，OpenJDK 11、17、18 的源码则使用内部监视器对象；未来版本仍可能调整字段、辅助方法、锁实现、数组复制优化、序列化代码和批量算法。能够稳定迁移的知识是：

- 读取围绕已发布数组快照进行；
- 修改通过复制形成新数组版本；
- 写入之间需要协调；
- 迭代器固定创建时快照且不支持元素修改；
- 元素是浅层引用；
- 公开内存一致性效果成立。

阅读源码时，可以用当前项目实际运行的 JDK 源码验证性能假设，却不应把私有字段名、锁类型、同步关键字位置或某个分支优化写进业务正确性条件。替换 JDK 供应商或升级版本后，这些细节可能变化，而公开 API 契约才是兼容边界。

## 构造、导出与克隆也体现浅快照边界

从数组或普通集合构造 `CopyOnWriteArrayList` 时，公开契约保证新列表取得自己的结构内容，后续修改输入数组或输入集合的结构不应被当作修改新列表的途径。这里复制的仍是元素引用：输入集合与新列表中的对应位置可以引用同一个可变元素。构造期间若输入集合被其他线程无协议地修改，结果取决于输入集合本身的并发契约；目标类不能为一个正在发生数据竞争的来源补造一致快照。

`toArray()` 返回一个调用方可以自由修改的独立数组，列表不会保留对该返回数组的引用。它适合作为明确的成员边界：调用方需要按索引执行多个步骤时，可以先取得一次数组，再围绕同一版本工作。代价是额外复制一次数组，而且数组中的元素仍是原引用。修改返回数组的槽位不会改变列表，修改槽位指向的可变对象则可能被列表其他读者观察到。

带目标数组参数的 `toArray(T[] a)` 还受运行时数组类型规则约束，可能复用传入数组，也可能创建新数组；若元素不能存入目标运行时类型，会抛出 `ArrayStoreException`。并发语义之外，这仍是普通集合与数组桥接 API 的类型边界。

`clone()` 返回列表的浅表副本。副本与原列表可以独立增删元素，不应共享同一个“当前数组版本指针”作为可变状态；但两个列表仍共享元素引用。克隆因此不能替代元素深拷贝，也不能建立两个对象图之间的隔离。如果目标只是向调用方暴露只读快照，通常返回不可修改列表或数组比暴露另一个可修改并发列表更能表达所有权。

序列化同样不应被理解为并发事务机制。它形成的是序列化过程所取得的列表内容表示，元素还必须满足相应序列化要求；应用若需要跨多个共享状态形成统一检查点，应在更高层先确定版本边界，而不是分别序列化若干线程安全容器后假定它们来自同一逻辑时刻。

## 选择前的定量判断

可以先用一个简单成本模型筛选。设列表平均长度为 n，单位时间读取次数为 R，遍历总访问元素数为 T，写入次数为 W。`get` 类读取大致与 R 成正比，遍历与 T 成正比，写入复制量大致与 W×n 成正比；按值搜索和去重还要附加线性比较成本。这个模型不提供精确时间，却能暴露数量级问题。

当 W 很小、n 有界、T 很大时，把协调成本放到 W 上通常合理。当 n 或 W 任一项持续增长，W×n 很快成为主导。若一个任务每秒刷新一百次、列表一万项，即使读取更多，也要认真评估每秒复制约百万引用带来的分配和带宽；若每次刷新通过一百次单项修改完成，实际复制量还会成倍放大。

测量时应将候选方案放在同一业务语义下比较。同步列表若遍历没有正确持锁，基准虽然更快，却比较了错误程序；不可变快照若每次读取都重新复制，也没有发挥整体发布优势；并发映射若丢失了所需顺序，则不是等价替代。

## 使用原则

`CopyOnWriteArrayList` 的优势可以归纳为稳定快照遍历、无需外部锁的读取路径和清晰的单次写操作发布。它的代价则是每次修改的数组复制、写线程串行、旧快照保留和浅复制边界。

实际使用时应坚持几条原则：

1. 先确认业务需要的是有序成员快照，而不是队列、映射或整体配置版本。
2. 用列表长度和写入频率共同判断成本，不用模糊的“读多写少”代替数据。
3. 遍历直接使用一次迭代器，不用 `size` 加多次 `get` 模拟稳定快照。
4. 用 `addIfAbsent` 和 `addAllAbsent` 表达容器支持的原子去重，不手写检查后添加。
5. 把元素可变性、资源生命周期和相等性视为独立并发问题。
6. 谨慎使用 `subList`，需要稳定独立结果时显式复制。
7. 尽量批量发布修改，避免循环执行单项写操作。
8. 观察分配率、旧数组存活和写入尾延迟，而不只看平均读取吞吐。
9. 依赖 Java SE 公开契约；使用源码做解释和调优时标注具体 JDK 前提。

当列表小、写入偶发、读取或分发极频繁，并且“本轮使用旧成员、下轮看到新成员”正是需要的语义时，`CopyOnWriteArrayList` 能把并发代码显著简化。反过来，当元素持续变化、列表很大、写入是核心路径或业务要求跨多个操作的实时一致性时，它昂贵而陈旧的快照不是附带缺点，而是与需求直接冲突的设计信号。

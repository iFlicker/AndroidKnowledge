# 3.2.2.2 ConcurrentHashMap

`ConcurrentHashMap<K, V>` 是 Java 并发集合中的哈希映射实现。它首先是一个有明确并发契约的 `ConcurrentMap`：多个线程可以并发读取和更新，单个公开操作不会破坏容器内部结构，若干“检查并更新”动作由专门的原子复合 API 表达。其次，在常见 OpenJDK JDK 8 及后续实现中，它通过无锁读取、CAS、桶级同步、协作扩容和分散计数降低全表互斥造成的竞争。

这两个层次必须分开理解。Java API 规范决定调用方可以依赖什么，例如禁止 `null`、单键更新的原子性、迭代器的弱一致性；OpenJDK 源码说明某个版本通常怎样实现这些承诺，例如桶首节点锁、`ForwardingNode`、红黑树桶和 `CounterCell`。应用代码应依赖前者，源码分析和性能判断才需要研究后者。除非特别说明，本文讨论 Java 8 引入并沿用至较新 OpenJDK 的常见实现框架，不把内部类名、阈值或字段布局当成所有 JDK 厂商和未来版本都必须保持的规范。

## 并发契约不是“所有操作形成一个事务”

`ConcurrentHashMap` 的线程安全首先表现为：多个线程并发调用 `get`、`put`、`remove`、`compute` 等方法时，容器不会因为数据竞争进入链断裂、节点丢失或数组结构损坏等非法状态。对于一个给定键，成功完成的更新具有原子效果，其他线程不会观察到该次更新写了一半的中间节点。

它还提供重要的内存可见性保证。规范说明，对某个键完成的更新操作，与随后报告该键更新后值的非空检索之间存在 happens-before 关系。直观地说，线程 A 构造一个对象并通过映射成功发布到键 `k`，线程 B 随后从 `k` 读取到该对象时，可以依赖这次容器发布建立的可见性，而不需要再为“引用从映射传递”额外加锁。

但这个保证不能被扩大：

- 它约束通过映射完成的发布和对应检索，不会让 value 此后的任意可变字段更新自动变成线程安全。
- 它保证单次方法及文档明确承诺的复合方法，不保证任意多个方法组合成事务。
- 它不维护多个键之间、多个容器之间或容器与外部资源之间的业务不变量。
- 它不提供全局时间点上的固定快照，遍历、统计和批量操作允许与并发更新交错。

例如，下面的两个调用各自线程安全，但组合并不原子：

```java
if (!map.containsKey(key)) {
    map.put(key, createValue());
}
```

两个线程可能都在 `containsKey` 阶段看到键不存在，然后各自创建对象并调用 `put`。容器最终仍然合法，但对象可能被重复创建，后写入者还会覆盖先写入者。如果要求“仅在缺失时安装一个值”，应使用 `putIfAbsent` 或在满足回调约束时使用 `computeIfAbsent`。

再如，将余额从键 `a` 转移到键 `b` 需要同时维护两个值。即使对两个键分别调用 `compute`，观察者仍可能看到只扣减尚未增加的中间状态。`ConcurrentHashMap` 的原子边界主要围绕单个键，不是通用事务系统。跨键不变量需要外部锁、不可变状态整体替换、串行状态机或真正的事务机制。

## 为什么禁止 null 键和值

`ConcurrentHashMap` 不允许 `null` 键，也不允许 `null` 值。构造映射本身不会失败，但向需要键或值的操作传入 `null`，通常会抛出 `NullPointerException`。这一限制不是单纯为了简化实现，而是为了让并发查询结果保持无歧义。

在允许空值的普通映射中，`get(key) == null` 可能表示两件事：键不存在，或者键存在但映射到 `null`。单线程代码可以随后调用 `containsKey` 区分，但在并发环境中，这两个调用之间状态可能变化，组合结果不再描述同一时刻。禁止空值后，`get` 返回 `null` 可以稳定解释为“本次检索没有找到映射”，也能作为原子复合 API 的控制信号。

这项约定贯穿多个方法：

- `computeIfAbsent` 的映射函数返回 `null`，表示不建立映射。
- `computeIfPresent`、`compute` 或 `merge` 的重映射函数返回 `null`，表示删除或不保留该映射。
- `search` 类批量操作使用非空结果表示已经找到目标。
- `getOrDefault` 可以为“当前没有映射”返回调用方给定的默认值，但默认值并没有被存入容器。

如果业务领域确实需要表达“已记录但没有实际值”，应使用显式对象，例如枚举、不可变占位对象或 `Optional`。把 `Optional` 作为长期存储值是否合适仍取决于领域设计，但它至少把“缺失映射”和“映射到空语义”区分开。不要用一个会与真实数据碰撞的字符串或数字哨兵替代，因为这会把并发歧义变成领域歧义。

禁止 `null` 也使读取的常见模式更直接：

```java
Value value = map.get(key);
if (value != null) {
    use(value);
}
```

这里的 `value` 是一次 `get` 的结果。若改写成先 `containsKey` 再 `get`，不仅多做一次查找，还会重新引入两个时刻之间的竞态。

## JDK 8+ 常见实现的结构前提

早期 JDK 7 的经典实现使用分段结构，常被概括为 `Segment` 加锁。JDK 8 的 OpenJDK 实现已经改变：主体是一个桶数组 `Node<K,V>[] table`，写入时结合 CAS 与桶级 `synchronized`，不再以固定数量的 `Segment` 作为主要并发控制单元。因此，“`ConcurrentHashMap` 有若干段，每段一把锁”只能用于讨论相应旧实现，不能用来解释 JDK 8+ 的常见源码。

在 JDK 8+ 常见实现中，一个桶位置可能处于几类状态：

1. `null`：桶为空，可以尝试通过 CAS 安装新节点。
2. 普通节点：桶内是链表，数组槽保存链表头节点。
3. `TreeBin`：桶已树化，由包装节点管理红黑树及相关并发协调。
4. `ForwardingNode`：当前表正在迁移，该桶的数据已转移或正在转移，节点指向新表。
5. 计算占位等特殊节点：用于协调部分计算型更新，普通查询和更新路径会识别其特殊哈希标记。

键的哈希值会先经过一次扰动，使高位信息参与低位索引计算，并清理实现保留的特殊标记位。桶索引通常利用数组长度为 2 的幂这一前提，通过位与得到。这里的关键不是背诵表达式，而是理解两个约束：数组索引由散列结果决定；实现保留负哈希等特殊值表示转发、树桶或计算状态。

节点的关键字段需要支持并发可见性。常见实现中，节点的 `val` 和 `next` 具有适合并发读取的可见性语义，数组槽访问由底层原子/易失方式完成。调用方不应依赖具体字段修饰符，因为这是实现细节；源码阅读时则要沿着“数组槽如何发布、节点链接如何发布、读取是否会获取这些写入”来验证无锁读取为何成立。

构造函数中的 `initialCapacity`、`loadFactor` 和旧式 `concurrencyLevel` 也容易被误解。在 JDK 8+ 常见实现中，`concurrencyLevel` 主要作为容量估算的兼容性提示，不等于创建多少把固定分段锁；`loadFactor` 参与初始容量估算，但运行期阈值和扩容控制由内部机制管理，并不意味着可以把普通哈希表的全部容量公式机械套过来。应用应把这些参数当作减少扩容与控制内存的提示，而不是稳定的并发度开关。

## 读取路径：通常无锁，但不是脱离同步语义

`get` 是理解 `ConcurrentHashMap` 性能特征的起点。常见实现的读取路径通常不获取互斥锁：

1. 对键计算扰动后的哈希值。
2. 读取当前桶数组和对应数组槽。
3. 若桶首节点的哈希与键匹配，则比较键并返回值。
4. 若桶首是普通链表，则沿 `next` 查找。
5. 若桶首是特殊节点，则委托给该节点的查找逻辑；树桶会树查找，转发节点会转到新表继续查找。

“无锁读取”并不等于“没有同步成本”或“可以读取任意未安全发布的数据”。数组槽和节点字段的发布依赖内存模型支持，CPU 仍可能承担屏障、缓存一致性和指针跳转成本。它只表示读取通常不用等待某个互斥锁持有者释放锁。

读取遇到扩容时也不需要停下等待整张表迁移完成。如果当前桶已经放置 `ForwardingNode`，查找会沿它指向的新表继续。这是扩容能够与查询、更新并行的重要基础。一次查询可能从旧表开始，在新表结束；这不构成快照，但仍应返回符合单键检索契约的结果。

键对象在放入映射后，参与 `hashCode` 和 `equals` 的状态必须保持稳定。`ConcurrentHashMap` 能保护自己的节点结构，却无法阻止调用方修改键。若键的哈希值变化，后续查询会从另一个桶开始，即使旧节点仍在映射里也可能无法找到。这个问题不是并发容器特有，但并发环境会让修改时序更难追踪，因此更应使用不可变键。

读取到 value 引用后，对 value 的使用仍需单独判断。如果 value 是不可变对象，发布后读取通常容易推理；如果 value 是普通可变对象，多个线程修改它的字段仍可能发生数据竞争。容器只保证引用映射关系，不会把每个 value 变成原子对象。例如 `ConcurrentHashMap<K, ArrayList<V>>` 不会让其中的 `ArrayList` 支持并发修改。

## 更新路径：CAS 快速路径与桶级同步

以 JDK 8+ 常见 `putVal` 思路为例，更新不是简单地“给整张表加锁”：

- 表尚未初始化时，线程竞争初始化控制状态，完成数组创建。
- 目标桶为空时，线程尝试 CAS 安装首节点；成功即可结束该桶的结构更新。
- 目标桶处于迁移状态时，当前线程可能先帮助扩容，再基于新表继续操作。
- 目标桶非空且不是迁移标记时，线程通常对桶首对象进入 `synchronized`，确认数组槽仍指向该桶首后，再在链表或树桶内查找并更新。

锁住桶首而不是整张表，使不同桶上的更新可以并行。锁内再次验证数组槽，是因为线程等待锁期间，桶可能已经被迁移或替换；没有这一步，线程可能在脱离当前表的旧结构上写入。链表中若找到相同键，更新其值；若未找到，追加节点，并在桶长度达到条件时考虑树化。树桶则使用自身的插入和替换逻辑。

这里的“桶级锁”也不能被理解为每个桶永久拥有独立锁对象。普通路径常复用桶首节点作为监视器，空桶安装使用 CAS，桶状态会随扩容和树化改变。实现的目标是缩小竞争范围，而不是建立一个与数组槽一一对应、永不变化的锁数组。

删除同样是更新。`remove(key)`、条件删除 `remove(key, value)`、计算函数返回 `null` 等路径需要在相应桶的协调下修改链或树。条件删除的价值在于把“值仍等于期望值”与删除合成一次原子判断，避免先 `get` 后 `remove` 删除了其他线程刚替换的新值。

```java
Value observed = map.get(key);
if (observed != null) {
    map.remove(key, observed);
}
```

上面的条件删除是安全模式：只有当前映射仍与 `observed` 相等时才删除。若写成 `map.remove(key)`，则从读取到删除之间其他线程写入的新值也会被无条件移除。

## 计数不是一把全局锁维护的精确字段

高并发更新时，如果所有线程都争用一个全局计数器来维护元素数量，计数本身会成为热点。JDK 8+ 常见实现采用类似 `LongAdder` 的分散计数思想：低竞争时更新基础计数，竞争出现时把增量分散到多个计数单元，读取总数时再汇总。

这种设计解释了两个现象。第一，写入不同桶虽然结构上互不冲突，仍需要更新全局规模信息，但分散计数可以减少它们在同一缓存行上的争用。第二，在并发更新期间汇总多个计数单元，结果可能已经落后于某些正在进行的变化，因此数量查询更适合监控和估算，而不是并发控制条件。

`mappingCount()` 返回 `long`，用于表达映射数量估计；`size()` 返回 `int`，当数量超过 `Integer.MAX_VALUE` 时受返回类型限制。两者在没有并发更新时可反映稳定数量，在并发更新时都不应被当作某个全局瞬间的事务快照。不能因为方法名是 `size` 就推导出实现会冻结所有桶后精确计数。

以下逻辑是典型错误：

```java
if (map.size() < limit) {
    map.put(key, value);
}
```

多个线程可能同时看到未达到限制，随后都插入，最终超过 `limit`。即使 `size()` 在每次返回时都精确，检查与写入也仍然是两个独立操作。若“最多允许 N 个条目”是硬约束，需要信号量、外部锁、串行准入器或其他把配额占用与插入绑定的协议。

同理，`isEmpty()` 适合查询提示，不适合在并发写入时证明“接下来不会有元素”。`clear()` 能执行清理，但与并发 `put` 交错时，不应把返回后的状态解释成跨线程的永久空表。需要停止写入并完成最终清理时，必须先通过更高层生命周期协议关闭生产者。

## 扩容：线程协作迁移而非整表长时间停顿

当映射规模达到内部阈值时，JDK 8+ 常见实现会创建更大的桶数组，并把旧桶迁移到新表。扩容需要处理所有桶，是一次总工作量较大的结构变化；但实现不会简单地让发起线程持有全表锁独自搬完，再允许其他线程继续。

内部控制字段同时编码初始化、阈值和扩容协调状态。一个线程发起扩容后，会设置新表和迁移索引；其他更新线程若发现目标桶是转发状态，可以进入帮助迁移逻辑，领取一段尚未处理的桶区间。多个线程处理不同区间，完成后共同推进扩容结束。并发度、CPU 数量、表大小和实现版本都会影响实际参与线程数，因此不能把“所有写线程必然一起扩容”当作契约。

对于旧表长度为 2 的幂、容量翻倍的常见情况，一个节点在新表中的位置只可能是原索引，或原索引加旧表长度。实现可根据哈希值新增的那一位把一个旧桶拆成低位链和高位链，而不必重新做通用取模。迁移完成后，旧桶数组槽被替换为 `ForwardingNode`，指向新表。

这个转发节点有三重作用：

1. 标记旧桶已经处理，防止重复迁移。
2. 引导并发查询从旧表进入新表。
3. 提示并发更新帮助迁移或在新表上重试。

因此，扩容期间可能同时存在旧表和新表，但不是任意两个互不关联的映射副本。转发标记和迁移区间协调把它们连接成一个过渡状态。理解扩容 bug 或性能尖峰时，应关注的不是“有没有全局锁”这一个问题，而是迁移总工作量、参与线程、热点桶、对象分配、缓存失效以及业务写入是否持续触发帮助迁移。

预估容量可以减少扩容频率，但不应盲目给出远大于实际需要的初始容量。过大的数组增加常驻内存和遍历扫描成本，过小则在增长阶段触发多轮迁移。合理做法是根据预期唯一键数量、增长速度和内存预算估算，并用目标 JDK、真实键分布和真实读写比例压测。

## 链表树化及其边界

当大量不同键落入同一个桶，链表查询成本会随桶内节点数线性增长。JDK 8+ 常见实现可在条件满足时把长链表转换为红黑树，以改善严重碰撞下的桶内查找上界。常见 OpenJDK 代码中的树化阈值是 8，退化回链表的阈值是 6；但只有表容量至少达到常见最小树化容量 64 时才倾向树化，否则通常优先扩容，希望通过更多桶分散碰撞。

这些数字是实现细节和源码分析前提，不是 `ConcurrentMap` API 契约。其他 JDK 实现或未来版本可以调整。工程设计不应依靠“构造恰好 8 个同哈希键就一定变成树”维持正确性。

树桶不是把桶首直接换成一个普通 `TreeNode` 就结束。常见实现使用 `TreeBin` 作为桶级包装，维护红黑树根、链式遍历关系和树桶内部的并发协调状态。更新树结构仍需要互斥协调；查询则尽量在允许的条件下进行树查找，在树结构正被写者调整时可以采用链式路径等策略避免读取不一致的树链接。

红黑树在键排序上还有一个常被忽略的前提。哈希相同而键不相等时，实现需要决定树中的相对位置。若键属于可比较的兼容类型，可以利用比较结果；否则会使用稳定的决胜逻辑建立结构顺序。这个顺序只服务桶内树结构，不代表 `ConcurrentHashMap` 获得了对外可见的排序迭代语义。

树化是碰撞防御和最坏路径优化，不是糟糕哈希设计的补偿许可证。若大量业务键故意返回相同哈希，所有更新仍集中在一个桶，桶级互斥成为热点；树结构只能降低查找比较次数，不能让同一桶的结构更新并行。良好的、稳定的哈希分布仍是吞吐的重要前提。

删除节点后，树桶规模降低到一定程度可能退化回链表。扩容拆分树桶时，低位和高位两组也可能分别保留树或退化成链表。由此可见，链表与树不是一次性、永久性的状态，而是会随规模和迁移变化。外部代码绝不能依赖某个键当前处于哪种桶结构。

## 原子复合 API 才是并发映射的核心能力

`ConcurrentHashMap` 的价值不只是允许多个线程调用 `put`，更重要的是提供把条件判断和更新放在同一原子边界内的方法。选择方法时，应先写清楚需要的状态转换。

### putIfAbsent：仅在缺失时安装

`putIfAbsent(key, value)` 在键没有非空映射时安装给定值，并返回之前的值；返回 `null` 表示本次成功安装。它适合值已经准备好、创建成本低或允许提前创建的场景。

```java
Value candidate = createValue();
Value existing = map.putIfAbsent(key, candidate);
Value effective = existing == null ? candidate : existing;
```

这里可能创建一个最终未采用的 `candidate`。如果构造很昂贵，可以考虑 `computeIfAbsent`，但后者对回调有更严格要求，并不只是“懒加载版 putIfAbsent”。

### replace 与条件 remove：比较并交换式更新

`replace(key, oldValue, newValue)` 只在当前值与 `oldValue` 相等时替换；`remove(key, value)` 只在当前映射匹配时删除。它们适合乐观状态转换：先读取一个不可变状态，根据它构造新状态，再尝试条件替换；失败说明期间有人更新，需要重新读取并决定是否重试。

```java
for (;;) {
    State oldState = map.get(key);
    if (oldState == null) {
        break;
    }
    State newState = oldState.next();
    if (map.replace(key, oldState, newState)) {
        break;
    }
}
```

若 `State` 不可变，这种写法让每次成功替换都成为清晰的单键状态跃迁。循环必须评估重试成本和终止条件；热点键下可能发生大量失败重试。

### computeIfAbsent：按键原子初始化

`computeIfAbsent` 在键当前没有映射时计算值，并在结果非空时建立映射。对该键而言，整个调用是原子的，常见实现会阻止其他线程同时完成冲突更新。但映射函数必须短小、确定、不会再次递归更新同一映射中的相关键。

```java
ConcurrentHashMap<String, LongAdder> counters = new ConcurrentHashMap<>();

counters.computeIfAbsent(category, ignored -> new LongAdder())
        .increment();
```

这个模式把计数器的首次安装交给映射，把后续高频增量交给 `LongAdder`。映射保证同一键最终关联一个被采用的计数器，计数器负责 value 内部的并发更新。

不要在映射函数中执行长时间网络调用、等待锁、阻塞 I/O 或不可控的用户代码。常见实现可能在占位或桶级协调状态下执行计算；慢回调会阻塞同桶或相关键的进展。回调抛异常时不会建立该映射，异常会向调用方传播。回调返回 `null` 时也不会建立映射，后续调用仍可能再次计算。

规范还禁止可检测的递归更新导致永不完成的计算，并可能抛出 `IllegalStateException`。即使某种跨键递归在某个版本偶尔可运行，也不应依赖它，因为键可能落入同桶，扩容和实现变化也会改变协调关系。

### compute、computeIfPresent 与 merge：按当前值转换

`compute` 无论键是否存在都把键和当前值传给重映射函数；返回非空结果就建立或替换映射，返回 `null` 则删除现有映射。`computeIfPresent` 只在已有非空映射时计算。`merge` 在键缺失时直接安装给定非空值，在键存在时用旧值和给定值计算新值；重映射结果为 `null` 时删除。

```java
frequencies.merge(word, 1L, Long::sum);
```

这比 `getOrDefault` 后再 `put` 正确，因为读取、求和和写回位于一个单键原子操作中：

```java
// 并发下可能丢失更新
frequencies.put(word, frequencies.getOrDefault(word, 0L) + 1L);
```

两个线程可能都读到旧值 5，然后都写入 6，实际两次增量只留下 1。`merge` 可避免这个丢失更新，但热点键上的所有更新仍需串行协调；若需求只是高频计数，`LongAdder` 作为 value 往往比反复替换装箱后的 `Long` 更适合。

所有计算型 API 的回调都应遵循相同原则：短、无阻塞、无外部副作用、尽量可预测。虽然方法对映射状态转换提供原子性，但无法回滚回调已经发送的消息、写入的文件或调用的远程接口。若回调执行外部副作用后更新失败或抛异常，就可能出现外部世界已变化而映射未变化的状态。

## “原子”不等于“回调恰好执行一次”

调用方常把 `computeIfAbsent` 理解成分布式意义上的 once-only 初始化，这是过度推断。它承诺的是相应映射更新的原子语义，不是对进程崩溃、对象生命周期、删除后重建或外部副作用的全局恰好一次。

一个键的映射函数在一次正常调用中不会被多个冲突更新同时完成，但以下情况都可能让计算在时间上再次发生：

- 映射函数返回 `null`，没有安装值。
- 映射函数抛出异常，调用失败。
- 映射后来被删除，后续调用重新初始化。
- 映射所在进程重启，内存状态丢失。
- 业务在多个 JVM 实例中各自拥有独立映射。

因此，不能仅靠 `computeIfAbsent` 实现跨进程唯一任务、支付去重或永久资源的恰好一次创建。对外部资源初始化，可以让回调只创建一个本地协调对象，再由该对象管理异步初始化状态；真正需要跨进程唯一性时，应使用具有相应一致性保证的外部系统。

也不要假设普通 `get` 与后续 value 操作形成原子整体。例如：

```java
List<Item> items = map.get(key);
if (items != null) {
    items.add(item);
}
```

即便引用读取安全，`ArrayList.add` 仍不是线程安全操作；同时另一个线程可能已经从映射移除或替换该列表。若每个键维护集合，可以使用不可变集合加条件替换、专用并发集合，或在 value 内封装明确的锁与生命周期。

## 弱一致遍历：允许前进，不承诺快照

`keySet`、`values`、`entrySet` 的迭代器和拆分迭代器是弱一致的。它们不会像普通非并发集合的 fail-fast 迭代器那样以 `ConcurrentModificationException` 作为主要并发修改反馈；它们可以与更新并发执行，并保证遍历过程能够继续，但不承诺反映某个单一时间点的完整映射。

一次遍历可能看到：

- 遍历开始前已完成且在访问相应桶时仍存在的映射。
- 遍历期间新加入并恰好位于尚未扫描区域的映射。
- 某些并发新增看不到。
- 某个键的旧值或新值之一，具体取决于读取时序。
- 已删除映射在遍历到该位置之前消失。

对单个键，遍历不会把一个节点呈现成结构损坏的半成品；对整个映射，观察结果却可能混合不同时刻。弱一致不是“随机错误”，而是一种有边界的活视图语义：优先保证并发进展和结构安全，放弃全局快照。

这适合缓存清理、近似监控、诊断输出、尽力而为的广播等场景。例如清理过期项时，漏掉本轮刚变化的条目通常可以在下一轮补上。但它不适合要求严格一致的财务结算、唯一清单导出、全量校验或必须与其他状态对齐的持久化快照。

需要稳定快照时，常见方案有三类：

1. 在允许弱一致输入的前提下复制到普通映射，得到“复制过程观察到的集合”，但这仍不是复制开始时的严格瞬时快照。
2. 通过外部读写锁或生命周期协议暂停更新，再复制。
3. 使用不可变状态加版本号，原子替换整个状态根，使读者固定读取某一版本。

具体选择取决于是否允许暂停写入、数据规模以及一致性要求。不能只写 `new HashMap<>(concurrentMap)` 就宣称获得了并发修改期间某一时刻的严格快照。

映射视图本身也是活视图，不是副本。通过 `keySet()` 或 `entrySet()` 删除元素会影响原映射；视图迭代观察的仍是映射的弱一致状态。`Map.Entry.setValue` 等能力还应以具体视图文档为准，不要从普通映射经验推断所有修改方式都受支持。

## size、mappingCount 与判空的正确语义

在映射静止时，`size()`、`mappingCount()` 和 `isEmpty()` 很直观；在并发更新时，它们只描述调用期间附近的观察。方法返回后，其他线程立即更新更是正常情况。关键问题不是“返回值是否每次逐项锁定后精确”，而是调用方能否把这个值用于需要原子判定的控制逻辑。

适合的用途包括：

- 监控面板展示大致规模。
- 决定是否触发一次非关键的维护尝试。
- 日志和诊断。
- 在已经由外部协议停止写入后做最终断言。

不适合的用途包括：

- 硬容量限制。
- 生成唯一编号。
- 判断所有工作都已经永久完成。
- 以“先判空再执行”为基础维护跨线程不变量。
- 依据数量精确分配与映射内容一一对应的外部资源。

若只关心是否存在某个键，应直接按键查询，而不是遍历或读取总数。若需要维护严格计数，可以把计数变化与业务状态变化放到同一更高层同步协议；单独增加一个 `AtomicInteger` 也不自动正确，因为映射更新成功而计数更新失败、重试或异常时仍会失配。

## 批量操作的并行语义

`ConcurrentHashMap` 提供 `forEach`、`search`、`reduce` 等批量操作，并有按键、按值、按条目或键值对处理的变体。带 `parallelismThreshold` 的版本会根据估计元素数量决定顺序执行还是拆分到 `ForkJoinPool.commonPool()` 并行执行。阈值设为 `Long.MAX_VALUE` 通常用于抑制并行，设为 `1` 则倾向最大化拆分，但最终调度仍受数据规模、公共池状态和实现策略影响。

这些操作同样不是快照。它们在执行期间允许并发更新，每个元素级读取可以利用映射的可见性保证，但最终聚合结果可能组合不同时间的状态。若规约要求全局一致输入，必须先建立外部快照或暂停更新。

批量函数应满足适合并行执行的性质：

- `forEach` 动作不应依赖访问顺序，因为映射没有稳定顺序。
- `search` 函数应允许在其他任务找到结果后不再处理剩余元素。
- `reduce` 的归约函数应尽量满足结合性；不同任务的分组和合并顺序不可依赖。
- 函数应避免长时间阻塞、共享可变副作用和对同一映射的复杂递归更新。

使用公共 `ForkJoinPool` 还有系统层面的影响。若回调阻塞 I/O 或等待外部锁，会占用公共池工作线程，干扰进程中其他依赖公共池的任务。对于小映射或轻量操作，任务拆分、调度和合并的成本可能高于并行收益。阈值应通过实际数据规模和运行环境基准测试确定，而不是因为“并行更快”就固定设为 1。

基本类型归约版本需要一个基准值，它应当是归约操作的单位元，例如求和用 0、求最大值可根据值域选择合适基准。错误的基准或非结合归约会让结果依赖拆分方式。即使函数在数学上正确，并发更新仍意味着结果不是稳定快照。

## 常见竞态及其修正

### 先检查后执行

```java
if (!map.containsKey(key)) {
    map.put(key, value);
}
```

问题是检查和写入可被其他线程穿插。使用 `putIfAbsent`；若值需要按需计算，使用满足回调约束的 `computeIfAbsent`。

### 读取后无条件覆盖

```java
Config oldConfig = map.get(key);
Config newConfig = derive(oldConfig);
map.put(key, newConfig);
```

另一个线程可能在 `get` 与 `put` 之间更新，随后被无条件覆盖。若转换依赖当前值，使用 `compute`；若转换逻辑适合在映射外执行，使用不可变值加条件 `replace` 重试。

### 读取后无条件删除

```java
if (expired(map.get(key))) {
    map.remove(key);
}
```

检查后，其他线程可能已替换成未过期的新值，仍被删除。保存观察值并调用 `remove(key, observed)`，或使用 `computeIfPresent` 在单键原子边界内判断。

### 把可变 value 当成自动线程安全

```java
map.computeIfAbsent(key, ignored -> new ArrayList<>()).add(item);
```

映射只原子安装列表，不保护后续 `add`。应根据语义选择并发集合、value 内部锁、不可变列表整体替换或其他聚合结构。

### 在计算回调中执行外部副作用

```java
map.computeIfAbsent(key, ignored -> remoteService.createResource(key));
```

远程调用可能很慢、失败或产生不可回滚副作用，还会延长相关更新的阻塞时间。更稳妥的设计通常是先原子安装一个表示初始化过程的本地对象，例如 `CompletableFuture`，再由明确的所有者执行外部工作，并设计失败后的移除、重试和取消语义。即便如此，也要处理回调发起者崩溃以及等待者如何超时。

### 用 size 做容量控制

读取数量后写入不是原子动作。若限制是软阈值，可接受短暂超限并异步清理；若限制是硬约束，应把许可获取、映射更新和失败补偿设计成一个协议。

### 依赖遍历顺序

桶分布、扩容、树化和并发更新都可能改变遍历次序。若需要按键有序的并发映射，应考虑 `ConcurrentSkipListMap`；若只需最终输出排序，可以复制后排序，但要明确复制时的一致性级别。

### 误以为 synchronized(map) 能与内部操作自动协作

调用方可以在外部对映射对象加锁，但普通 `get`、`put` 不会自动获取这把外部锁。除非所有相关访问都由应用代码严格遵守同一外部锁协议，否则它不能把已有并发调用变成跨方法事务。混用“部分代码外部加锁、部分代码直接调用”尤其容易产生虚假的安全感。

## 性能分析应围绕竞争分布

`ConcurrentHashMap` 的平均按键查找通常接近常数时间，严重碰撞且树化后桶内查找可接近对数时间。但并发性能不能只看复杂度。影响吞吐和尾延迟的因素包括：

- 键的哈希分布是否把更新集中到少数桶。
- 是否存在单个热点键，使所有原子更新在同一键上串行。
- 读写比例、删除比例和扩容频率。
- value 构造及计算回调是否昂贵或阻塞。
- 表和节点规模造成的内存占用、缓存未命中和垃圾回收。
- 线程数量是否远超可用 CPU，导致调度和竞争放大。
- 批量操作是否占用公共 ForkJoin 池。
- 目标 JDK 实现、硬件内存层次和运行参数。

“不同键”不一定意味着“无竞争”，因为不同键可能落在同一桶，扩容和计数也有共享协调点。反过来，高读取比例并不保证业务整体无锁，因为读取到的 value 可能有自己的锁或昂贵逻辑。

热点计数是典型例子。用 `merge(key, 1L, Long::sum)` 在语义上正确，但每次都更新同一映射值并产生装箱对象，热点键下协调成本明显。把 `LongAdder` 作为 value 可以把增量竞争分散到计数器内部：

```java
LongAdder counter = counters.computeIfAbsent(key, ignored -> new LongAdder());
counter.increment();
```

代价是读取 `LongAdder.sum()` 本身也更偏向并发统计语义，不适合需要与其他状态原子一致的精确事务计数。优化必须服从业务语义。

初始化容量值得关注，但不存在脱离场景的最佳数值。已知会快速装入大量唯一键时，合理预估能减少初始化和迁移；映射通常很小时，巨大初始容量只会浪费内存。基准测试应使用真实 `hashCode`、真实键分布、真实线程数和类似的 JVM 预热过程，并同时观察吞吐、P95/P99 延迟、分配率和 GC，而不是只测单线程平均耗时。

## 适用边界与替代方案

`ConcurrentHashMap` 适合以下核心需求同时成立的场景：

- 多线程共享、按键查询和更新。
- 不需要键的全局排序。
- 可以接受弱一致遍历和并发期间的近似统计。
- 主要不变量围绕单个键，或跨键协调由更高层处理。
- 键和值都非 `null`，键的哈希与相等性稳定。

典型用途包括注册表、按键缓存、共享索引、按维度统计器集合、连接或会话查找表。不过“缓存”还需要过期、容量、加载失败、淘汰、刷新和防击穿等策略；仅有 `ConcurrentHashMap` 不等于完整缓存。若需求复杂，经过验证的缓存库通常比自行把定时清理和计算回调拼在一起更可靠。

以下场景应考虑其他结构或更高层协议：

- 只在单线程或线程封闭范围使用：普通 `HashMap` 更简单，额外并发机制没有收益。
- 所有操作都必须在同一个全局锁内维护复杂不变量：普通映射加明确锁可能更容易证明正确。
- 需要按键排序、范围查询、最小或最大键：考虑 `ConcurrentSkipListMap`。
- 需要固定快照、版本历史或无锁读取整份配置：考虑不可变映射加原子根引用。
- 需要有界缓存和成熟淘汰策略：使用专用缓存实现。
- 需要跨进程一致性、持久化或事务：使用数据库、协调服务或其他外部系统。
- 工作负载几乎全部集中在一个键：映射无法消除该键本身的串行语义，应重新设计状态分片或聚合方式。

`Collections.synchronizedMap` 与 `ConcurrentHashMap` 也不是简单的“旧与新”。同步包装通常以单一互斥协议保护操作，遍历时调用方还需按文档对包装对象同步；它更容易与同一把锁下的额外状态组成事务，但并发度较低。`ConcurrentHashMap` 提供更细粒度并发和专门复合 API，却不提供任意跨键事务。选择取决于需要证明的语义，不是只比较基准吞吐。

## 源码阅读与验证路径

阅读具体 JDK 源码时，先记录版本、发行版和源码提交，再按状态转换追踪，不要只摘取常量。推荐顺序如下：

1. 从 `get` 确认普通节点、树桶和转发节点的查找分派。
2. 从 `putVal` 观察空桶 CAS、非空桶同步、相同键替换和树化触发。
3. 从初始化与容量控制字段理解表创建、阈值和扩容状态编码。
4. 从 `transfer` 观察任务区间领取、旧桶拆分和 `ForwardingNode` 安装。
5. 从计数更新路径观察基础计数、计数单元和扩容检查如何连接。
6. 从 `computeIfAbsent`、`compute`、`merge` 对照普通 `put`，确认回调执行位置、异常和 `null` 结果语义。
7. 从遍历器和批量任务确认迁移期间如何跨表前进，以及为何结果不是快照。

源码里的 `synchronized`、CAS 和 `volatile` 只能解释实现手段，最终仍要回到公开文档确认调用方可依赖的契约。反过来，只读 API 文档又不足以解释热点桶、扩容尖峰和回调阻塞为什么发生。规范与实现结合，才能同时回答“代码是否正确”和“为什么性能这样表现”。

并发测试也不能仅靠一次多线程循环没有报错。应把不变量写成可检查结果，并覆盖：

- 多线程对同键和不同键更新。
- `putIfAbsent`、条件 `replace`、条件 `remove` 的成功与失败分支。
- 计算函数返回 `null`、抛异常和发生递归更新的边界。
- 扩容期间并发读写和删除。
- 冲突键触发链表、树化及拆分的路径。
- 遍历与更新并发时只断言弱一致契约允许保证的性质。
- value 自身并发协议，而不是只检查映射引用存在。

压力测试可以提高竞态暴露概率，但不能证明不存在竞态。正确性应先来自 API 契约、内存模型和状态转换推理，再用测试验证实现假设和业务不变量。

## 使用原则总结

理解 `ConcurrentHashMap`，最重要的是同时守住三条边界。

第一，容器提供的是并发映射契约。单键操作和专门的复合 API 可以作为原子状态转换，任意多次调用、多个键、value 内部状态以及外部副作用不会自动获得事务性。

第二，JDK 8+ 常见 OpenJDK 实现通过无锁读取、空桶 CAS、桶级同步、树桶、分散计数和协作扩容实现高并发，但这些结构是解释性能与源码的前提，不是应用可以绑定的永久规范。旧版分段实现、其他 JDK 发行版和未来实现必须分别标注。

第三，高吞吐来自访问模式与设计假设匹配。均匀散列、较少热点、短小计算回调、清晰的 value 并发协议以及可接受的弱一致观察，才能发挥它的优势。若业务要求跨键事务、严格快照、排序范围、有界淘汰或跨进程一致性，应选择能直接表达这些要求的结构，而不是继续在 `ConcurrentHashMap` 外叠加隐含协议。

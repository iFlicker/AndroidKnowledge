# 3.2.2.5 ConcurrentSkipListMap

`ConcurrentSkipListMap<K, V>` 是 JDK 提供的线程安全有序映射。它同时实现 `ConcurrentMap` 与 `ConcurrentNavigableMap`：一方面支持并发映射需要的条件写入、条件删除和原子替换，另一方面按照键的自然顺序或构造时提供的 `Comparator` 维护全局有序关系，并提供邻近键查询、区间视图、首尾访问和升降序遍历。

它适合解决的不是“多个线程共享一个 Map”这么宽泛的问题，而是更具体的组合需求：数据需要长期保持按键有序，读写会并发发生，并且调用方经常执行范围扫描、前驱后继查询或首尾元素操作。如果只需要按键精确查找，`ConcurrentHashMap` 通常更直接；如果只需要一棵有序树且可以由外部锁保护，`TreeMap` 可能更节省结构成本；如果必须获得跨多个操作的严格快照或事务边界，仅依靠 `ConcurrentSkipListMap` 也不够。

本文讨论 Java 标准库公开契约，并结合常见 OpenJDK 实现解释其机制。接口语义是调用方可以依赖的兼容性边界；节点布局、索引层组织、CAS 工具和清理步骤属于实现细节，不同 JDK 版本可能调整，不能作为业务代码的依赖。

## 从 ConcurrentNavigableMap 契约理解它

普通 `Map` 描述键到值的映射，`SortedMap` 增加按键排序，`NavigableMap` 再增加“寻找小于、等于或大于给定键的最近条目”以及可导航范围视图。`ConcurrentMap` 则为并发场景补充若干不可分割的复合操作。`ConcurrentNavigableMap` 把后两组能力组合起来，因此 `ConcurrentSkipListMap` 的核心契约可以分为三层：

1. 映射层：一个按比较规则判定唯一性的键对应一个值。
2. 导航层：键始终有序，可以查询前驱、后继、首尾条目和区间。
3. 并发层：公开的单次访问和文档明确承诺的复合方法具有线程安全与原子语义，遍历和聚合观察则采用弱一致模型。

“有序”不是指插入顺序，也不是指值的顺序，而是指所有键在同一个比较关系下形成的顺序。假设键序列是 `10、20、40`，对键 `25` 调用导航方法时：

| 方法 | 含义 | 结果 |
| --- | --- | --- |
| `lowerKey(25)` | 严格小于目标的最大键 | `20` |
| `floorKey(25)` | 小于等于目标的最大键 | `20` |
| `ceilingKey(25)` | 大于等于目标的最小键 | `40` |
| `higherKey(25)` | 严格大于目标的最小键 | `40` |

当目标本身存在时，`floorKey` 和 `ceilingKey` 可以返回目标，`lowerKey` 和 `higherKey` 仍会越过目标。对应的 `lowerEntry`、`floorEntry`、`ceilingEntry`、`higherEntry` 返回键值条目而不是仅返回键。没有符合条件的元素时，这些查询返回 `null`。由于容器禁止空键和空值，`null` 可以无歧义地表示“没有结果”。

`firstKey`、`lastKey` 在映射为空时抛出 `NoSuchElementException`，而 `firstEntry`、`lastEntry` 在为空时返回 `null`。`pollFirstEntry` 与 `pollLastEntry` 不只是读取，还会原子地移除当时的首条目或尾条目。返回的条目是映射关系的快照表示，不能通过 `Entry.setValue` 把修改写回映射；需要更新时应调用映射自身的写方法。

这一组契约使它可以直接表达许多“从有序索引寻找边界”的操作，而不必先遍历整个集合。例如，查找不晚于某个时间点的最近版本可以使用 `floorEntry`，查找严格晚于游标的下一项可以使用 `higherEntry`，取出优先级最小的待处理项可以使用 `pollFirstEntry`。这些操作的价值不仅是代码短，还在于定位与读取或移除由容器作为一次操作完成，避免调用方自己拼接容易产生竞态的步骤。

## 跳表如何形成有序路径

`ConcurrentSkipListMap` 名称中的 Skip List 即跳表。跳表的底层是一条按键完整排序的链表，底层链表包含所有有效键；在它之上建立若干稀疏索引层，每层只抽取下一层的一部分节点。查找时先在较高层快速向右跳跃，遇到下一个键超过目标后向下一层下降，最终到达底层链表并确定目标是否存在。

假设底层有如下有序键：

```text
底层:  3 -> 8 -> 12 -> 17 -> 25 -> 31 -> 46 -> 60
一级:  3 ------> 12 -------> 25 -------> 46
二级:  3 ------------------> 25
```

查找 `31` 时，不必从底层的 `3` 开始逐个比较。可以先在二级索引跳到 `25`，发现下一跳不存在或已经越过目标，再下降到一级，随后下降到底层找到 `31`。索引节点不另外保存一份业务键值关系，而是为底层节点提供更快的到达路径。因此，底层链表决定真实映射内容，索引层主要影响查找效率。

跳表的层高通常通过随机化策略产生。插入一个新键时，它一定进入底层，但只有一定概率被提升到第一层，更小概率继续进入更高层。若每次提升概率近似固定，索引节点数量随层数几何递减，期望高度和查找路径长度都是 `O(log n)`。这里必须强调“期望”：跳表没有像严格平衡树那样通过旋转维护确定的高度上界，它依靠概率分布在大量节点上形成足够均匀的稀疏层。

随机分层并不改变键的排序语义。同一个键无论被提升到几层，在比较器中的位置都相同。层高也不是业务优先级，调用方无法通过键值以外的信息要求某个节点进入更高索引层。实现可以改变随机位的使用方式、索引创建时机和最高层维护方式，只要公开行为仍符合 `ConcurrentNavigableMap` 契约。

与平衡树相比，跳表对并发更新有一个重要结构优势：插入或删除通常只需要修改搜索路径附近的少量链接，不需要执行跨多个树节点的旋转来恢复全局平衡。局部链接更适合用 CAS 协调多个线程。不过，“适合非阻塞实现”不等于所有路径都没有竞争，也不等于操作一定在固定步数内完成。热点键、相邻热点区间、频繁失败的 CAS 和昂贵比较器仍会增加延迟。

## 比较器定义键身份

创建映射时可以显式传入 `Comparator<? super K>`；没有传入时，键必须能够按自然顺序相互比较，通常意味着实现 `Comparable`。比较器不仅决定遍历顺序，还决定映射眼中的键是否相同：当 `compare(a, b) == 0` 时，两个对象占据同一个键位置，即使 `a.equals(b)` 为 `false`。

下面的比较器只比较用户编号：

```java
import java.util.Comparator;
import java.util.concurrent.ConcurrentSkipListMap;

record UserKey(long id, String region) {}

Comparator<UserKey> byId =
        Comparator.comparingLong(UserKey::id);

ConcurrentSkipListMap<UserKey, String> users =
        new ConcurrentSkipListMap<>(byId);

users.put(new UserKey(7, "east"), "first");
users.put(new UserKey(7, "west"), "second");
```

映射最终只有一个比较位置，因为两个键的 `id` 都是 `7`。第二次 `put` 会替换该位置的值，而不是新增第二个条目。映射中实际保留哪个键对象属于具体操作和实现行为不应被过度依赖；业务上应该把比较结果为零理解为同一个逻辑键。

这与哈希映射的身份规则不同。`HashMap` 和 `ConcurrentHashMap` 主要通过 `hashCode` 与 `equals` 确认键，`ConcurrentSkipListMap` 通过排序比较确认键。若比较器与 `equals` 不一致，映射仍可工作，但作为一般 `Map` 使用时可能出现违反直觉的结果，例如一个与现有键不 `equals` 的对象却能取得同一值。最稳妥的设计是让比较相等与逻辑相等一致；如果故意不一致，应在类型和 API 文档中明确“唯一性由哪些字段组成”。

比较关系还必须满足稳定的全序要求。实践中至少应保证：

- 反对称性：`compare(a, b)` 与 `compare(b, a)` 的符号相反。
- 传递性：若 `a < b` 且 `b < c`，则应有 `a < c`。
- 比较相等的一致性：若 `compare(a, b) == 0`，则对其他键的相对顺序不应产生矛盾。
- 稳定性：键进入映射后，参与比较的状态不应改变。

违反这些条件会破坏搜索路径的推理基础。容器可能仍保持内存结构不崩溃，但查询、范围边界和删除可能找不到逻辑上已经存在的键。比较器也位于几乎所有核心路径上，应避免网络访问、锁等待、时间依赖、随机结果和明显副作用。一次查找会执行多次比较，并发重试可能再次比较同一对键，因此比较器必须允许重复调用。

自然顺序同样有类型边界。使用无比较器构造函数并不意味着任意 `K` 都能在运行时排序；如果键不可比较，或者不同运行时类型之间不能互相比较，首次需要排序的操作就可能抛出 `ClassCastException`。泛型声明不能独自证明所有传入对象满足自然顺序契约，API 设计应尽量用同一种稳定键类型。

## 为什么禁止 null

`ConcurrentSkipListMap` 不允许 `null` 键，也不允许 `null` 值。向映射写入空键或空值会抛出 `NullPointerException`，以空键执行需要比较的查询通常也会失败。

禁止空键首先来自排序要求：比较器或自然顺序未必定义 `null` 与普通键之间的位置。虽然调用方可以写出支持 `null` 的比较器，但并发映射接口还需要统一处理空值语义，因此标准实现直接排除空键。

禁止空值则让“没有映射”和“映射到空值”不再混淆。并发代码经常根据 `get(key) == null` 判断当前没有值，再决定是否安装新值；`putIfAbsent`、`computeIfAbsent`、条件删除以及弱一致观察也都需要清晰地区分缺失状态。如果允许真实值为 `null`，一次读取返回空值后，调用方无法判断条目本来不存在，还是存在但值为空，同时并发删除还可能发生在判断期间。

需要表达“存在但业务值为空”时，应使用明确的值对象、枚举状态或 `Optional` 风格的封装，而不是试图把 `null` 塞进容器。这样能把业务状态与容器缺失状态分离，也更便于原子更新。

## 常见 OpenJDK 实现中的节点与 CAS

以下内容描述 OpenJDK 常见版本中的总体思想，而不是 Java 规范要求的固定布局。OpenJDK 8、11、17、21 等版本在字段组织、辅助方法、原子访问工具和部分算法细节上存在变化。例如较早版本可见基于 `Unsafe` 的 CAS，较新实现可能使用 `VarHandle`；某些版本将值状态、删除标记和索引清理组织成不同的辅助步骤。阅读源码时应以正在运行的具体 JDK 源码为准。

总体上，映射维护一条保存全部映射关系的有序底层链，并在其上维护索引节点。节点的关键可变状态通常包括值或删除状态以及后继链接。线程先依据比较器搜索到候选前驱与当前节点，再通过 CAS 尝试把局部结构从“观察到的旧状态”改为“期望的新状态”。如果 CAS 失败，说明另一个线程已经改变了相关位置，当前线程重新搜索或重试，而不是假定原路径继续有效。

这种实现经常被称为非阻塞或无锁算法。更准确地说，常见核心更新路径不依赖覆盖整张表的互斥锁：某个线程暂停时，不应永久持有全局锁阻止所有其他线程推进。从系统整体看，竞争过程中总有操作能够取得进展，属于 lock-free 思路；但规范没有承诺每个调用都 wait-free，也没有承诺某个特定线程在有限重试次数内完成。线程可能因调度、CAS 竞争、垃圾回收或比较器成本而经历较长延迟。

CAS 只负责某个字段或链接的条件更新，完整操作往往包含多个阶段。例如插入通常先让新节点进入底层有序链，使映射关系对查询可见，再按随机层高补充上层索引。索引尚未全部建立时，底层条目已经可以是有效映射；其他线程最多走一条暂时不够短的搜索路径，而不应得到错误键值。这个次序体现了“底层链是真实数据，索引层是加速结构”。

删除也通常不是一次把所有相关引用同时从各层摘除。实现会先改变节点的逻辑状态，使普通查询不再把它视为有效映射，然后再断开底层链接并清理索引。并发线程在搜索时若遇到已逻辑删除的节点，可以帮助完成物理清理。帮助机制把清理责任从原删除线程扩散到经过该位置的其他线程，避免一个线程中断后留下永久阻塞搜索的半完成结构。

因此，源码中的短暂中间状态不能直接等同于公开语义。可能存在“底层节点已链接但索引未补齐”“节点已标记删除但链接尚未摘除”“索引仍指向一个正在被清理的底层节点”等瞬间。算法通过状态检查、CAS 和重试保证其他线程要么识别有效映射，要么协助清理后继续。调用方只应依据公开方法结果推理，不应通过反射观察内部字段并把它们当作业务状态。

## get 的搜索与可见性

`get(key)` 从高层索引向下定位到接近目标的底层位置，再检查候选节点的键和值状态。平均搜索成本为 `O(log n)`，最坏路径可能退化为较长的线性扫描。由于索引是概率结构，复杂度描述应使用期望值而不是严格保证。

在并发删除或插入发生时，搜索可能遇到失效索引、已删除节点或刚改变的后继链接。实现会跳过、重试或帮助清理。一次成功 `get` 返回的是该调用线性化点上与键关联的值；返回 `null` 表示在该操作的某个有效瞬间没有可见映射。它不表示调用返回之后键仍然缺失，另一个线程可以紧接着插入；同样，成功返回的值也可能在下一条语句执行前被替换。

并发容器保证的是引用的安全发布与映射操作的可见性，不会自动冻结值对象。线程 A 将一个已正确构造的值放入映射后，线程 B 通过容器取得该值，可以依赖并发集合提供的内存一致性效果观察到放入操作之前的构造结果。但如果之后多个线程继续修改值对象内部字段，仍需不可变设计、原子变量、锁或其他同步协议。线程安全的 Map 加上非线程安全的可变 value，并不会得到整体线程安全的业务对象。

## put 与同键竞争

`put(key, value)` 先定位键的位置。如果比较相等的有效节点存在，它原子地替换值并返回旧值；如果不存在，则构造新节点并尝试链接到底层链的正确位置。多个线程同时插入比较相等的键时，只有一个逻辑键位置成立，其他线程会观察竞争结果后转入替换或重试路径，不会形成两个同时有效的相等键。

底层插入成功后，映射关系已经建立。索引层可以随后补充，即使补充时其他线程删除了该节点，实现也必须识别状态变化并停止或清理无效索引。这个过程解释了为什么源码中一次 `put` 可能有多个 CAS，也解释了为什么不能把某个具体 CAS 简单认定为所有 JDK 版本共同的线性化点。线性化点取决于操作分支和实现版本，应从对应源码与规范共同判断。

`putIfAbsent` 与“先 `get` 再 `put`”不同。后者包含竞争窗口：

```java
if (map.get(key) == null) {
    map.put(key, value);
}
```

两个线程都可能在 `get` 时看到缺失，随后分别执行 `put`，后写入者覆盖先写入者。`putIfAbsent` 把“仅在当前缺失时安装值”作为一个原子条件更新：

```java
String existing = map.putIfAbsent(key, value);
if (existing == null) {
    // 当前调用成功安装了 value。
} else {
    // 其他映射已经占据这个比较位置。
}
```

这里的原子性限于一个键和一次方法调用。它不自动保证新值与另一个键、外部文件或数据库状态同步改变。

## remove 与逻辑删除

`remove(key)` 定位比较相等的节点并尝试使其映射失效，成功时返回旧值。常见 OpenJDK 算法会把删除拆成逻辑删除和结构清理：先通过 CAS 改变值或节点状态，让后续读取不再视其为有效条目；再处理底层后继链接以及上层索引。不同版本的删除标记表示方式不完全相同，但“先确立语义删除，再逐步解除结构引用”是理解并发删除的重要思路。

这样做避免要求一次原子指令同时修改前驱、当前节点、后继和多个索引层。若另一个线程遇到已删除节点，它可以帮助前驱越过该节点，或触发索引清理，然后继续自己的搜索。帮助清理既减少无效节点长期堆积，也让算法不依赖原删除线程必须运行到最后一步。

`remove(key, expectedValue)` 是条件删除：只有当前值与期望值相等时才删除。它适合“仍是我观察到的状态才撤销”的场景：

```java
boolean removed = map.remove(key, observedValue);
```

不过这里仍可能存在 ABA 风格的业务问题：值先从 `A` 变为 `B`，随后又变回与旧 `A` 相等的值，条件删除只按相等性无法识别中间变化。若业务需要检测版本演进，应把版本号纳入不可变值对象，并对完整版本值执行条件替换或删除。

删除完成并不意味着相关节点对象立即被垃圾回收。并发搜索线程、迭代器、范围视图的遍历状态或尚未清理的索引可能暂时持有引用。只要实现最终解除不可达结构且外部不再引用，垃圾回收器才能回收对象。不能把 `remove` 返回当成内存立刻下降的保证。

## 单次操作的线性化语义

线性化用于描述并发对象的单次操作：每个已完成操作看起来像在调用开始与返回之间的某个瞬间生效，并且这些瞬间可以组成一个符合顺序规范的历史。对 `get`、`put`、`putIfAbsent`、`remove`、`replace`、`pollFirstEntry` 等单次映射操作，可以按各自契约把结果理解为发生在某个线性化点。

但不能把这个结论无限扩大。下面几类观察通常不是整表线性化快照：

- `size()` 对整个映射进行计数或聚合观察。
- 迭代器跨越多个节点逐步读取。
- `forEach`、流操作和批量遍历。
- 先调用一个方法，再根据结果调用另一个方法的调用方组合。
- 同时读取多个键并要求它们满足业务不变量。

例如：

```java
if (!map.containsKey(a) && !map.containsKey(b)) {
    map.put(a, value);
}
```

即使每个方法本身线程安全，整个“两个键都不存在才写入”的规则仍不是原子的。另一个线程可以在任意两个调用之间改变 `a` 或 `b`。若业务不变量跨键，应使用外部锁、单线程所有权、版本化不可变状态、事务系统，或重新设计为能由单键原子更新表达的状态。

线性化也不意味着实时读取不会变化。`firstEntry()` 返回当时最小条目后，另一个线程可以立刻插入更小键；调用方接着处理返回值时，它可能已经不是当前最小值。需要“取出并独占当时最小项”应使用 `pollFirstEntry()`，需要“处理期间始终没有更小项”则还需要更高层的同步协议。

## 原子复合方法与回调边界

`ConcurrentSkipListMap` 继承 `ConcurrentMap` 和 `Map` 的复合更新方法，包括：

- `putIfAbsent`：缺失时写入。
- `remove(key, value)`：当前值匹配时删除。
- `replace(key, oldValue, newValue)`：当前值匹配时替换。
- `replace(key, value)`：键存在时替换。
- `computeIfAbsent`：缺失时根据键计算。
- `computeIfPresent`：存在时根据旧值计算。
- `compute`：无论存在与否都进行计算并决定保留或删除。
- `merge`：缺失时采用给定值，存在时合并旧值与给定值。

这些方法把“观察当前状态”和“按观察结果更新”放进容器定义的原子操作中，优于调用方自行组合 `get` 与 `put`。例如累加不可变计数值：

```java
map.merge(key, 1L, Long::sum);
```

这能保证针对该键的映射更新按方法契约原子完成，但不意味着回调只执行一次，也不意味着回调执行期间整个映射停止变化。Java 不同版本的默认方法与具体类实现可能采用不同重试和协调策略，编写映射函数时应遵守保守边界：

- 函数应短小，避免阻塞 I/O 和长时间计算。
- 不依赖只调用一次，不把不可撤销副作用直接放入函数。
- 不在函数中递归更新同一个逻辑键。
- 不把跨键不变量寄托在一次 `compute` 上。
- 能返回 `null` 的计算方法应明确理解其“保持缺失或删除映射”语义。

例如在 `computeIfAbsent` 中发送消息、扣减余额或写外部系统是不可靠的，因为异常、竞争或实现重试可能使外部副作用与最终映射状态不一致。更适合的做法是先原子安装一个明确的状态对象，再由独立流程依据状态执行副作用，并用版本或幂等标识协调。

同一个键上的原子更新也不保护值对象内部的原地修改：

```java
List<String> list = map.get(key);
list.add("x");
```

这里 `get` 是线程安全的，但 `List` 的修改是否安全完全取决于列表实现和外部同步。若希望通过映射原子替换整个值，可以使用不可变列表，并在 `compute` 中基于旧列表创建新列表；这样原子边界是“旧引用替换为新引用”，而不是对共享可变对象进行无保护修改。

## 弱一致迭代器

`ConcurrentSkipListMap` 的集合视图迭代器是弱一致的。它们不会像普通 `TreeMap` 的 fail-fast 迭代器那样，因为检测到并发结构修改就以 `ConcurrentModificationException` 作为主要反馈；也不会在创建迭代器时复制整张表形成固定快照。迭代可以与并发插入、删除和替换共同进行，并保持结构安全。

弱一致的关键含义是：

- 迭代器按其视图的排序方向推进。
- 创建迭代器时已经存在且遍历期间未被删除的元素通常可以被观察，但不应把非正式直觉提升为快照保证。
- 遍历期间新增或删除的元素可能被看到，也可能不被看到，取决于更新位置与迭代进度。
- 同一次遍历不代表映射在某个单独瞬间的完整状态。
- 不会返回内部损坏的节点，也不要求调用方为结构安全加锁。

例如升序迭代已经越过键 `20` 后，另一个线程插入键 `15`，当前迭代通常不会倒退去读取它；若插入键 `40` 而迭代尚未到达该区域，则有可能读到。删除和重新插入也可能让业务层观察与“固定名单”不同。代码不应通过一次弱一致迭代完成严格结算、全量一致性校验或必须恰好处理一次的任务分发。

若需要快照，应先定义快照边界。简单复制：

```java
var snapshot = new java.util.TreeMap<>(map);
```

只能保证复制完成后 `snapshot` 自身不随原映射继续变化；复制过程仍通过弱一致遍历读取原映射，因此在并发写入期间得到的内容未必对应原映射某个单一瞬间。要获得严格快照，必须在更高层冻结写入、使用统一锁、切换版本或采用支持多版本读取的数据结构。

升序迭代通常是结构的自然方向，降序视图需要沿不同路径定位前驱，标准文档也提示降序遍历可能比升序遍历更慢。具体差距取决于 JDK 实现、数据规模和访问模式，不能只凭这一点做性能结论，但在高频全量扫描中应分别基准测试。

## 范围视图不是副本

`subMap`、`headMap`、`tailMap` 和 `descendingMap` 返回的是受边界约束的并发视图，不是元素副本。视图与原映射共享底层数据：

- 通过原映射在范围内写入，视图可以观察到。
- 通过视图更新或删除范围内键，原映射会同步变化。
- 视图只能接受其边界允许的键，越界写入会抛出 `IllegalArgumentException`。
- 视图本身仍提供弱一致迭代和并发导航能力。

四参数 `subMap(fromKey, fromInclusive, toKey, toInclusive)` 能精确指定两端是否包含。两参数 `subMap(fromKey, toKey)` 继承 `SortedMap` 语义，通常是左闭右开。处理时间段、编号段或字典前缀时，应明确使用哪一种边界，避免把业务上的闭区间误写成右开区间。

```java
var window = map.subMap(start, true, end, false);
```

这个视图表示 `start <= key < end`。若调用方执行 `window.clear()`，删除的是原映射在该范围内的条目。若只是想把范围结果交给不受信任的调用方，应该复制成独立映射或暴露只读接口，不能把视图误当作隔离结果。

范围视图的边界比较仍使用原映射比较器。若比较器只比较键的一部分，边界也只按这部分字段生效。复合键设计必须保证范围端点能够准确表达业务区间，必要时使用专门的最小、最大哨兵字段，而不是依赖字符串拼接或不稳定的特殊值。

`descendingMap` 反转导航方向，其 `firstEntry` 对应原映射的最大键，`lower`、`higher` 等概念也按反向比较关系解释。多次取得的降序视图对象不要求具有对象身份上的唯一性，业务代码应依赖内容和契约，不依赖视图实例是否相同。

## 导航 API 的组合方式

导航方法适合以游标方式处理有序数据。例如逐页读取严格大于上一页末键的数据：

```java
static <K, V> java.util.List<java.util.Map.Entry<K, V>> nextPage(
        java.util.concurrent.ConcurrentNavigableMap<K, V> map,
        K cursor,
        int limit) {
    var result = new java.util.ArrayList<java.util.Map.Entry<K, V>>(limit);
    for (var entry : map.tailMap(cursor, false).entrySet()) {
        result.add(java.util.Map.entry(entry.getKey(), entry.getValue()));
        if (result.size() == limit) {
            break;
        }
    }
    return result;
}
```

`tailMap(cursor, false)` 排除游标本身，因此不会因上一页末键仍存在而重复首项。不过这仍是弱一致分页：并发插入到游标之前的条目不会出现在后续页，并发删除可能使总项数减少，并发写入到后续范围可能被当前或后续页看到。它适合实时浏览、监控窗口和允许变化的数据流，不适合严格导出。

若只需要一个邻近条目，直接调用 `higherEntry` 通常比创建视图再取首项更清晰。范围视图适合连续扫描或把边界作为后续多个操作的约束。选择 API 时应让代码直接表达业务语义，而不是把所有操作都转换成迭代。

`pollFirstEntry` 与 `pollLastEntry` 可用于多个线程争抢当前最小或最大键，但它们不是阻塞队列操作：映射为空时立即返回 `null`，不会等待新条目，也没有容量和背压协议。若任务系统需要等待、唤醒、超时、容量限制或优先级阻塞语义，应考虑 `BlockingQueue` 体系，而不是围绕有序映射自行轮询。

## size 的成本与含义

对链式并发有序结构而言，维护一个能在所有并发更新下精确、低成本读取的全局计数并不简单。JDK 文档明确提示 `size()` 不是常量时间操作；常见实现需要遍历并计数，成本可能达到 `O(n)`。在并发修改期间，返回值还可能只是遍历过程得到的瞬时观察，不能用作严格控制条件。

因此不要在热循环中反复调用：

```java
for (int i = 0; i < map.size(); i++) {
    // ...
}
```

这不仅可能把线性计数放大为平方级工作量，`size` 还会在循环期间变化，使索引式逻辑本身没有稳定意义。判断是否存在元素应优先使用 `isEmpty`、`firstEntry` 或具体键查询；处理元素应直接迭代；需要监控数量时可以接受近似值，或维护独立计数器并清楚它与映射更新之间不是天然事务。

即使某个 JDK 版本内部增加了计数辅助，也不能据此把 `size()` 当作严格快照。公开契约没有把“与所有并发操作组成整表瞬时快照”作为保证。业务若要求“当且仅当元素总数小于上限时插入”，先读 `size` 再 `put` 会产生竞争，多个线程可能同时通过检查。容量限制应使用具备容量契约的容器，或在外部同步下把计数判断与写入合并。

`mappingCount()` 是 `ConcurrentHashMap` 提供的长整型估计接口，不是 `ConcurrentSkipListMap` 的通用替代方案。不要因为两者都属于并发映射，就假设所有统计 API 和复杂度都相同。

## 可变键会破坏有序索引

键对象一旦进入映射，所有参与比较的字段都应保持不变。假设键按 `priority` 排序，插入后把 `priority` 从 `10` 改为 `100`，节点仍留在原来的链表位置，但比较器现在认为它应该位于另一处。之后 `get`、`remove`、范围查询和迭代边界都可能表现异常：遍历仍能在旧位置碰到对象，按新值搜索却沿另一条路径走开。

```java
final class MutableKey implements Comparable<MutableKey> {
    int priority;

    MutableKey(int priority) {
        this.priority = priority;
    }

    @Override
    public int compareTo(MutableKey other) {
        return Integer.compare(priority, other.priority);
    }
}
```

问题不只发生在直接修改字段时。比较器若读取外部可变配置、数据库状态、当前语言区域或随时间变化的权重，同一个键的相对顺序也会漂移。容器无法知道比较规则已经改变，更不会自动重新排序已有节点。

正确做法通常是使用不可变键。需要改变排序属性时，先以旧键删除，再以新键插入，并根据业务要求决定是否用外部同步保护这两个步骤。如果其他线程不能观察到中间缺失或同时存在的状态，那么“删除旧键再添加新键”也不够，需要更高层原子协议。另一种设计是让稳定标识作为哈希映射键，把可变排序属性维护在受控的独立索引中，但这会引入双索引一致性问题，必须明确谁负责协调。

值可以是可变的，但值的可变性不会改变键的搜索路径。即便如此，值对象自身的并发安全仍由调用方负责。若值的某个字段也参与外部范围判断，最好通过替换不可变值而不是原地修改，避免读者看到内部状态撕裂。

## 范围索引案例：按时间与序号定位事件

一个典型用途是维护可并发写入、按时间有序的事件索引。仅用毫秒时间戳作为键会发生冲突：同一毫秒内多个事件比较相等，后写入者覆盖前写入者。可以把时间与单调序号组成不可变复合键：

```java
record EventKey(long epochMillis, long sequence)
        implements Comparable<EventKey> {
    @Override
    public int compareTo(EventKey other) {
        int byTime = Long.compare(epochMillis, other.epochMillis);
        return byTime != 0
                ? byTime
                : Long.compare(sequence, other.sequence);
    }
}
```

映射声明如下：

```java
var events =
        new java.util.concurrent.ConcurrentSkipListMap<EventKey, Event>();
```

查询时间范围 `[startMillis, endMillis)` 时，需要构造能覆盖端点内所有序号的边界：

```java
EventKey from = new EventKey(startMillis, Long.MIN_VALUE);
EventKey to = new EventKey(endMillis, Long.MIN_VALUE);

var range = events.subMap(from, true, to, false);
```

由于复合键先按时间、再按序号排序，`from` 位于开始毫秒的所有正常序号之前，`to` 位于结束毫秒的所有正常序号之前，从而表达左闭右开的时间窗口。这个推导成立的前提是实际序号域与哨兵约定一致；如果业务允许 `Long.MIN_VALUE` 作为真实序号，应使用显式边界类型或重新设计键域，不能让哨兵与真实值冲突。

该结构支持并发追加、按时间扫描、查询某事件的前后邻居，也能通过 `headMap(cutoff, false).clear()` 删除某个边界之前的数据。但范围清理不是整批事务：清理期间并发写入可能与遍历删除交错。若生产者还能写入已过期时间段，清理结束后仍可能出现旧数据。工程上应定义写入水位线，例如禁止写入早于已确认水位的事件，或让清理流程按版本反复收敛。

范围视图还可能长期持有原映射引用。把很小的 `subMap` 缓存在生命周期很长的对象中，并不会复制或截断原映射，它只是一个指向原结构的窗口。内存所有权和修改权限都应按“共享视图”理解。

## 与 TreeMap 加锁比较

`TreeMap` 基于红黑树，非线程安全。要在并发环境使用，可以通过单一锁保护所有访问，或使用 `Collections.synchronizedSortedMap` 获得同步包装。这样的方案与 `ConcurrentSkipListMap` 都能提供有序映射，但同步模型不同。

外部锁保护的 `TreeMap` 优势在于原子边界可由调用方扩大。持有同一把锁时，可以连续读取多个键、修改多个条目并检查全局不变量，其他遵守协议的线程不会看到中间状态。需要严格快照时，也可以在锁内复制。树节点通常没有跳表多层索引的额外对象，单线程或低竞争下可能有较好常数表现。

代价是所有受保护操作互斥。慢比较器、长时间遍历或锁内业务逻辑会阻塞其他读写；遗漏某个未加锁访问则破坏整体协议。同步包装返回的迭代器也要求调用方在遍历期间手动持有包装对象的锁，不能只取得迭代器后在锁外使用。

`ConcurrentSkipListMap` 允许多个读写操作更细粒度地推进，弱一致遍历不会独占全表，适合持续在线的并发索引。但它不提供“锁住整张映射执行任意事务”的公开机制。即使调用方另建一把锁，所有直接访问映射的代码也必须共同遵守，容器本身不会自动配合这把锁。

选择依据不是简单的“并发容器一定更快”。数据量小、线程少、操作需要跨键事务时，加锁 `TreeMap` 可能更容易证明正确；线程多、范围读取频繁、允许弱一致观察且单键更新占主导时，`ConcurrentSkipListMap` 更符合访问模型。最终应在真实读写比例、键分布、范围长度和比较器成本下基准测试。

## 与 ConcurrentHashMap 比较

`ConcurrentHashMap` 以哈希定位为主，适合精确键查询和更新，通常期望接近常数时间；它不维护全局键顺序，不能高效回答“最近的小于键”“某区间全部键”或“当前最小键”。`ConcurrentSkipListMap` 为顺序和导航付出比较、指针跳转、索引节点与 `O(log n)` 期望访问成本。

如果业务每次只是 `get(id)`、`put(id, value)`，偶尔为了展示而排序一次，使用 `ConcurrentHashMap` 并在需要时复制排序可能更划算。如果范围查询是高频主路径，反复复制和排序会把成本推给每次查询，此时长期维护有序索引更合理。

热点键问题在两者中都存在，但表现不同。`ConcurrentHashMap` 的竞争集中在相同哈希桶或相同键的更新协调；跳表的相同键与相邻链接更新会发生 CAS 竞争，范围扫描还要穿过许多节点。不能只根据总吞吐判断，应同时观察尾延迟、写入分布和垃圾回收压力。

有时业务既需要按稳定 ID 精确查找，又需要按另一个属性范围查询，于是想同时维护 `ConcurrentHashMap<ID, Value>` 和 `ConcurrentSkipListMap<SortKey, ID>`。这形成二级索引，两个容器的单次操作各自原子，但跨容器更新不是原子。插入、删除、属性变化和失败恢复都可能留下不一致。需要通过单一状态所有者、锁、日志、版本号或可重建索引明确一致性策略，而不是因为两个容器都线程安全就认为组合也线程安全。

## 与其他索引方案的权衡

`ConcurrentSkipListSet` 与本类底层思想相近，适合只关心有序唯一元素、不需要独立值的场景。若代码把键和值存成同一个对象，使用 Map 可能制造重复身份；若需要按键取得附加信息，Map 更自然。

不可变排序数组适合写少读多且需要紧凑内存、二分查找和稳定快照的场景。每次更新可以复制并发布新数组，读取几乎没有同步协调，但写成本为 `O(n)` 且会分配新数组。数据规模和写频率较低时，这种简单方案可能优于通用并发跳表。

数据库 B 树或专用持久化索引适合数据超出进程内存、需要事务、恢复和多条件查询的场景。`ConcurrentSkipListMap` 是进程内对象容器，不提供持久化、崩溃恢复、跨进程一致性和查询优化器。不能因其支持范围查询就把它当作数据库索引替代品。

优先级队列适合反复取得最小或最大元素，但普通 `PriorityQueue` 不支持并发，`PriorityBlockingQueue` 支持阻塞取得优先元素却不擅长按任意键精确定位和范围视图。若核心操作是等待并消费最高优先级任务，阻塞优先队列的契约更准确；若既要按键更新、取消，又要导航和扫描，才需要评估有序映射以及更高层等待机制。

分段结构也是一种选择：按键前缀、时间桶或业务分区把数据放入多个较小容器，可以降低热点和缩短范围扫描，但跨分区查询、全局顺序和再分片会更复杂。只有在测量证明单个跳表成为瓶颈且分区边界稳定时，才值得引入这层设计。

## 性能判断不能只看 O(log n)

跳表查询、插入和删除的期望复杂度通常描述为 `O(log n)`，范围扫描通常是定位起点的 `O(log n)` 加上返回条目数量 `O(m)`。这些表达忽略了很多实际成本：

- 每次比较器调用的计算量。
- 节点和索引对象带来的内存占用与垃圾回收。
- 链式引用访问对 CPU 缓存局部性的影响。
- 同键或相邻位置竞争造成的 CAS 失败与重试。
- 范围视图方向、范围长度以及遍历中并发删除的清理成本。
- 值对象大小和后续业务处理时间。

随机层高还会让不同实例和不同数据分布的常数成本略有波动。基准测试应预热运行时，使用足够数据规模，区分升序与降序扫描，模拟真实键分布，并避免把数据生成、日志或锁外业务时间混进容器操作结果。

比较器尤其容易被低估。若比较复合字符串、进行区域敏感排序或临时分配对象，一次 `O(log n)` 查找会放大这些成本。可以把已经规范化的不可变排序字段放入键，或者预计算不会变化的排序分量，但预计算结果必须与逻辑相等和范围边界保持一致。

范围查询的成本还取决于实际返回量，而不只取决于起点定位。即使找到起点只需期望 `O(log n)`，扫描百万条结果仍需要访问百万个节点，并会长时间占用执行线程。弱一致遍历不会锁住写线程，却可能与写线程争用内存带宽和处理器缓存。对外提供范围接口时，应设置合理的页大小或结束边界，并通过稳定游标继续读取；不能因为底层是并发容器，就允许一次请求无上限地遍历全部数据。

延迟评估也应同时观察中位数和高分位。大部分操作可能很快，但个别调用会因调度暂停、连续 CAS 失败、清理已删除节点或遇到较长随机路径而变慢。若系统对最坏响应时间有严格上界，只有期望复杂度和平均吞吐不足以证明满足要求，还需要以目标运行时、硬件和负载进行压力测试，并为超时、降级或分片设计明确边界。

## 使用边界与设计检查

在采用 `ConcurrentSkipListMap` 前，应先回答以下问题：

1. 是否确实需要长期有序、邻近查询或范围扫描，而不是偶尔排序？
2. 键的比较规则是否构成稳定全序，比较相等是否就是期望的唯一性？
3. 键是否不可变，比较器是否纯粹、快速且可重复？
4. 是否接受弱一致遍历和非快照式统计？
5. 业务原子性是否主要局限于单个键或单次首尾移除？
6. 是否有人误把范围视图当成副本，或把 `size()` 当作容量门闩？
7. value 内部状态是否有独立的线程安全策略？
8. 是否需要等待、容量、事务、持久化或跨索引一致性，而这些能力本类并不提供？

API 边界也应尽量表达真正需要的能力。只要求并发导航时，可以依赖 `ConcurrentNavigableMap<K, V>` 接口，避免调用方绑定具体实现；只需要读取范围结果时，可以返回独立快照或受控迭代接口，避免泄露可写视图。若模块内部必须维护跨键不变量，应把映射封装在状态所有者中，不让任意调用方直接组合方法。

测试不仅要覆盖单线程排序结果，还应覆盖比较相等的不同对象、范围端点、空映射导航、条件更新失败、并发插入删除、弱一致遍历可接受范围和可变值的同步策略。并发测试应验证业务不变量，而不是断言某次调度必然产生固定遍历结果；弱一致行为本来就允许多个合法观察。

## 总结

`ConcurrentSkipListMap` 的本质是一个以比较顺序定义键身份、以跳表维护有序路径、以并发原子操作支持在线更新的 `ConcurrentNavigableMap`。底层完整链保存真实映射，上层概率索引缩短期望搜索路径；常见 OpenJDK 实现通过 CAS、逻辑删除、重试和协助清理避免整表互斥，但具体节点布局与线性化细节必须结合目标 JDK 版本理解。

它最有价值的能力是把并发映射与有序导航放在同一个契约中：单键条件更新、前驱后继、首尾移除和动态范围视图都可以直接表达。与此相伴的边界也必须接受：比较器决定唯一性，`null` 被禁止，键必须稳定，迭代与范围扫描弱一致，`size()` 成本较高且不是严格快照，多个方法或多个键之间的业务事务不会自动成立。

当访问模式以并发精确查询为主时，优先考虑 `ConcurrentHashMap`；当需要外部锁内的跨键事务和严格快照时，评估加锁 `TreeMap`；当高频需求确实是有序导航与范围索引，并且单次操作的并发语义足够表达业务规则时，`ConcurrentSkipListMap` 才是结构与契约同时匹配的选择。

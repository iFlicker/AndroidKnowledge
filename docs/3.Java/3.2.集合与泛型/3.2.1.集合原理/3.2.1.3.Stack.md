# 3.2.1.3 Stack

`java.util.Stack<E>` 是 Java 早期提供的后进先出容器。它的栈操作很少，真正值得理解的部分却不少：LIFO 抽象如何映射到动态数组，继承 `Vector` 为什么扩大了公开能力，方法级同步能保证什么、不能保证什么，迭代顺序为何容易与栈顶顺序混淆，以及新代码为什么通常应使用 `Deque`。

本文只讨论通用 Java 集合语境中的 `java.util.Stack`。重点不是重复数据结构教材中的栈定义，而是解释这个具体类的 API、实现约束、遗留兼容价值和迁移边界。

## 从 LIFO 抽象理解 Stack

栈的核心约束是 LIFO，即 Last In, First Out：最后进入的元素最先离开。假设依次压入 `A`、`B`、`C`，此时 `C` 位于栈顶；第一次弹出得到 `C`，第二次得到 `B`，最后才得到 `A`。

从操作语义看，一个最小栈通常只需要以下能力：

- 压入元素：把新元素放到栈顶。
- 查看栈顶：读取栈顶元素但不移除。
- 弹出元素：读取并移除栈顶元素。
- 判断空栈：确定是否还有元素。

`Stack` 分别用 `push`、`peek`、`pop`、`empty` 表达这些操作，另外提供 `search` 查询某元素距离栈顶的位置。只看这一组方法，`Stack` 的抽象相当直接。

问题在于，Java 类型的能力不只由类中新增的方法决定，还包括从父类继承的全部公开方法。`Stack<E>` 的声明是：

```java
public class Stack<E> extends Vector<E>
```

因此，一个 `Stack` 同时也是 `Vector`、`List`、`Collection` 和 `Iterable`。调用方不仅能操作栈顶，还能按下标读取、在中间插入、删除任意位置、排序、批量替换和取得子列表。这个类型在名称上承诺“栈”，在公开 API 上却暴露“同步动态数组”的完整能力。

这一区别决定了理解 `Stack` 时必须同时维护两个视角：

1. **栈视角**：只使用 `push`、`pop`、`peek` 等方法，序列遵循 LIFO。
2. **列表视角**：所有 `Vector` 和 `List` 操作都合法，容器可以被任意位置修改。

如果代码只遵守栈视角，调用者可以基于“后进先出”推理状态。如果混入列表视角，LIFO 就不再是类型系统强制的不变量，只剩团队约定。

## 为什么栈顶位于数组末尾

`Stack` 没有另建节点结构，也没有维护独立的栈顶指针。它直接复用 `Vector` 的动态数组存储：有效元素位于底层数组的前缀区间，逻辑长度由元素个数记录。栈底对应下标 `0`，栈顶对应下标 `size() - 1`。

把栈顶放在数组末尾是自然选择。尾部追加只需在当前有效区间之后写入一个引用；尾部删除只需减少逻辑长度并清理原末尾引用。若把数组头部作为栈顶，每次压入或弹出都可能移动其余元素，栈顶操作会从常数时间退化为线性时间。

例如，依次执行：

```java
Stack<String> stack = new Stack<>();
stack.push("A");
stack.push("B");
stack.push("C");
```

逻辑布局可以理解为：

| 数组下标 | `0` | `1` | `2` |
| --- | --- | --- | --- |
| 元素 | `A` | `B` | `C` |
| 栈位置 | 栈底 | 中间 | 栈顶 |

此时 `peek()` 和 `pop()` 都针对下标 `2`。弹出 `C` 后，`B` 成为新的栈顶。底层容量可能大于逻辑长度，多余槽位不是栈元素；删除末尾时实现会清除已经移出的引用，使对象在没有其他强引用时可以被垃圾回收。

动态数组意味着 `Stack` 具有容量与长度的区别。长度是当前元素数量，容量是底层数组可容纳的引用数量。压入元素时如果容量足够，写入成本很低；容量不足时，需要申请更大的数组并复制已有引用。因而 `push` 的单次最坏复杂度是 O(n)，长期连续压入通常按均摊 O(1) 分析。

`Stack` 只有无参构造器，不能像某些数组容器那样在构造时直接指定初始容量。不过它继承了 `Vector` 的 `ensureCapacity`，遗留代码确实可以预留容量：

```java
Stack<Integer> stack = new Stack<>();
stack.ensureCapacity(1_000);
```

这再次体现继承的双面性：复用父类能力很方便，但栈的使用者也因此必须知道 `Vector` 的容量 API。现代代码若需要明确容量，通常可以直接构造 `new ArrayDeque<>(expectedSize)`，意图更清楚。

## Vector 继承造成的设计后果

### 公开能力宽于栈抽象

继承最直接的后果是 `Stack` 无法限制元素只能从栈顶进入和离开。以下代码都能编译并正常运行：

```java
Stack<String> stack = new Stack<>();
stack.push("A");
stack.push("B");
stack.push("C");

stack.add(0, "X");
stack.remove(1);
stack.set(0, "Y");
stack.sort(String::compareTo);
```

这些操作未必破坏 `Vector` 的内部结构，却会破坏调用方对压栈顺序的推理。尤其是排序之后，“最后压入者先弹出”已经不成立，因为元素次序被整体重排。

因此，`Stack` 的问题不是功能不足，而是功能过多。一个只需要栈能力的方法若接收 `Stack<T>`，实际上也允许方法体使用随机访问和任意位置修改。接口没有精确表达最小需求。

### Stack 可以被当作 List 传播

由于子类型关系成立，`Stack` 可以传给任意接收 `List` 或 `Vector` 的 API：

```java
static void normalize(List<String> values) {
    values.sort(String::compareTo);
}

Stack<String> stack = new Stack<>();
stack.push("third");
stack.push("first");
stack.push("second");
normalize(stack);
```

调用 `normalize` 后，容器仍是同一个 `Stack` 实例，但其顺序已经由列表逻辑决定。问题不会在类型转换处暴露，后续 `pop()` 仍可执行，只是弹出次序可能不再符合原算法假设。

这说明“对象的运行时类型是 `Stack`”并不等于“对象一直按栈使用”。审查遗留代码时，不能只搜索 `push` 和 `pop`，还要检查该实例是否被赋值给 `List`、`Vector`、`Collection`，是否传给会修改顺序的通用方法，是否通过返回值泄露给外部调用方。

### 同步特征也由 Vector 带入

`Vector` 的许多操作使用实例监视器进行同步，`Stack` 的核心操作又调用或组合了这些能力。结果是 `Stack` 天生带有传统同步容器的语义：同一实例上的受同步保护操作会竞争同一对象锁。

这不是纯栈抽象必需的特征。局部算法中的栈通常只由一个线程访问，不需要每次压栈和弹栈都携带同步语义；真正的多线程工作栈则往往还需要阻塞、超时、关闭、任务认领或更细致的并发协议，单纯的方法同步也不够。

于是 `Stack` 同时处于两个不理想的位置：

- 在单线程场景，它的同步和宽接口通常是额外负担。
- 在复杂并发场景，它只保护单次容器操作，无法替代完整并发设计。

### 序列化与具体类型兼容

`Stack` 继承了 `Vector` 的可序列化能力，并具有自己的序列化版本标识。旧系统可能把它出现在对象字段、序列化数据、远程接口、反射配置或要求具体类型的方法签名中。此时它不只是一个可随意替换的内部实现，具体类名和对象形态可能已经成为兼容契约。

这也是“新代码不推荐使用”不等于“看到就机械替换”的原因。若 `Stack` 只存在于方法内部，替换通常简单；若它跨越模块边界或持久化边界，迁移必须先识别兼容要求。

## 五个栈方法的精确语义

`Stack` 自己新增的方法只有 `push`、`pop`、`peek`、`empty` 和 `search`。理解它们不难，但几个细节经常造成错误。

### push：压入并返回传入元素

`push(E item)` 把元素追加到有效序列末尾，并返回同一个元素引用。其核心效果相当于调用继承自 `Vector` 的尾部添加方法。

```java
Stack<String> stack = new Stack<>();
String value = new String("task");
String result = stack.push(value);

System.out.println(result == value); // true
System.out.println(stack.peek());    // task
```

返回元素主要是历史 API 风格，不意味着创建副本，也不意味着返回压入后的位置。多数代码并不需要使用该返回值。

`Stack` 允许压入 `null`，因为 `Vector` 允许保存 `null`：

```java
Stack<String> stack = new Stack<>();
stack.push(null);

System.out.println(stack.empty()); // false
System.out.println(stack.peek());  // null
```

这会让“查看结果为 `null`”与“没有元素”难以通过返回值区分，不过 `Stack.peek()` 在空栈时抛异常，并不会用 `null` 表示空栈。迁移到 `ArrayDeque` 时要特别处理这一差异，因为 `ArrayDeque` 禁止 `null` 元素。

### peek：读取但不移除栈顶

`peek()` 返回最后一个元素，容器长度不变。空栈调用会抛出 `EmptyStackException`。

```java
Stack<Integer> stack = new Stack<>();
stack.push(10);
stack.push(20);

int first = stack.peek();
int second = stack.peek();

System.out.println(first);        // 20
System.out.println(second);       // 20
System.out.println(stack.size()); // 2
```

“窥视”只保证容器结构没有因这次调用改变。若元素本身是可变对象，调用方拿到的是同一个对象引用，可以修改对象状态。容器同步不保护元素内部字段。

### pop：读取并移除栈顶

`pop()` 返回最后一个元素，同时把它从容器中删除。空栈调用同样抛出 `EmptyStackException`。

实现思路是先取得当前长度，再通过 `peek()` 读取最后一个元素，随后删除原最后位置。`pop()` 本身是同步方法，因此这几个内部步骤对使用同一个实例监视器的其他同步操作表现为一个临界区，而不是允许其他线程在读取和删除之间插入同一把锁保护的操作。

```java
Stack<Integer> stack = new Stack<>();
stack.push(10);
stack.push(20);

System.out.println(stack.pop());  // 20
System.out.println(stack.size()); // 1
System.out.println(stack.peek()); // 10
```

需要区分两类原子性：

- `pop()` 内部“读取栈顶并删除”是一次完整方法调用。
- 调用方自己组合的“判断、查看、再弹出”并不会自动成为一次原子动作。

### empty：判断当前时刻是否为空

`empty()` 本质上判断 `size() == 0`。它与继承来的 `isEmpty()` 在通常使用下表达相同结果，只是命名来自早期栈 API。

```java
Stack<String> stack = new Stack<>();
System.out.println(stack.empty());   // true
System.out.println(stack.isEmpty()); // true
```

在单线程局部代码中，两者主要是风格差异。在并发代码中，无论选择哪个，返回值都只描述检查发生时的状态。方法返回后，另一个线程可以立刻压入或弹出元素。

### search：返回从栈顶计算的 1 基距离

`search(Object target)` 从栈顶方向寻找与目标相等的元素。找到时返回它到栈顶的 1 基位置，栈顶返回 `1`；找不到返回 `-1`。它不是数组下标，也不是从栈底计算的位置。

```java
Stack<String> stack = new Stack<>();
stack.push("A");
stack.push("B");
stack.push("A");
stack.push("C");

System.out.println(stack.search("C")); // 1
System.out.println(stack.search("A")); // 2
System.out.println(stack.search("B")); // 3
System.out.println(stack.search("X")); // -1
```

存在重复元素时，返回最靠近栈顶的匹配项。底层可以理解为先从序列末尾寻找最后一个相等元素，再用 `size() - index` 转换为栈顶距离。

匹配依据是通常的相等性语义：目标为 `null` 时寻找 `null`，否则通过 `equals` 判断。`search` 是线性扫描，不应把它误解为具备索引结构的快速定位操作。

## Stack API 与继承 API 的语义冲突

同一个动作常有多种写法，这会进一步削弱代码的一致性。例如以下操作都能向末尾加入元素：

```java
stack.push(value);
stack.add(value);
stack.addElement(value);
stack.add(stack.size(), value);
```

对当前实现而言，它们都可能把元素放到栈顶位置，但表达意图不同。`push` 明确说“这是栈操作”；`add` 只说“这是集合或列表追加”；`addElement` 暴露 `Vector` 的历史命名；带下标的 `add` 更强调列表位置。

删除也有类似情况：

```java
stack.pop();
stack.remove(stack.size() - 1);
stack.removeElementAt(stack.size() - 1);
```

这些写法在非空时可能产生类似结构结果，但异常类型、返回值和代码意图并不完全相同。混用会让维护者无法确定代码究竟依赖栈契约还是列表契约。

一旦选择 `Stack` 表达 LIFO，较稳妥的约束是业务路径只使用栈方法，容量调整、序列化等历史兼容逻辑集中在边界层。若确实需要随机访问、排序或中间删除，就应承认数据模型已经超出纯栈，而不是继续用类名掩盖实际需求。

## 同步保证的准确边界

### 单次方法调用保护容器结构

`Stack` 的 `pop`、`peek`、`search` 声明为同步方法；`push` 虽没有在自身声明上写 `synchronized`，但它调用的 `Vector.addElement` 是同步方法。`empty` 调用同步的 `size`。从使用效果看，核心操作通过同一个 `Stack` 实例的监视器与相应 `Vector` 操作协调。

这类同步主要解决的是：两个线程同时修改底层动态数组时，不应把元素计数、数组内容和结构修改计数更新到互相矛盾的状态。它还建立相应锁释放与后续锁获取之间的可见性关系。

但“每个方法有同步路径”不能推导出“所有使用该对象的业务逻辑都线程安全”。容器只能理解一次方法调用，无法理解调用方想维护的跨调用条件。

### 检查后执行不是原子操作

最典型的错误是：

```java
if (!stack.empty()) {
    return stack.pop();
}
return null;
```

假设线程 T1 调用 `empty()` 得到 `false`，方法随后释放对象锁。在线程 T1 调用 `pop()` 之前，线程 T2 可以先弹出最后一个元素。T1 再执行 `pop()` 时仍会得到 `EmptyStackException`。

如果确实要求“为空则返回特殊结果，否则弹出”作为一个原子动作，必须把整个组合放在同一个锁范围内：

```java
static <E> E popOrNull(Stack<E> stack) {
    synchronized (stack) {
        return stack.empty() ? null : stack.pop();
    }
}
```

这段代码只适用于 `null` 不作为有效元素，或者调用方能接受“空栈”和“弹出 `null` 元素”返回同样结果。更严格的 API 可以使用 `Optional<E>`，但 `Optional.ofNullable` 仍无法区分栈中实际保存的 `null`。若历史数据允许 `null`，需要额外结果类型明确区分“没有元素”和“元素值为 null”。

外部同步还必须遵循同一锁协议。若一部分代码对 `stack` 加锁，另一部分代码对包装对象或无关锁加锁，二者不能形成互斥。若调用方绕过约定直接使用继承来的方法，也可能破坏业务层的不变量。

### 跨多个容器的条件需要更高层锁

假设一个操作要求从源栈弹出并压入目标栈，而且任何观察者都不能看到元素同时缺席或重复出现，那么分别调用 `source.pop()` 和 `target.push()` 不够。每个容器各自完整，不代表两个容器之间的转移是原子的。

```java
static <E> void moveTop(Stack<E> source, Stack<E> target) {
    E value = source.pop();
    target.push(value);
}
```

要维护跨容器不变量，需要统一的更高层同步策略，并考虑两个锁的固定获取顺序以避免死锁。若业务已经复杂到需要多对象事务、阻塞等待或高并发扩展，通常应重新设计所有权和通信方式，而不是继续依赖 `Stack` 的内置锁。

### 容器安全不等于元素安全

以下代码中的压栈和取栈可以受容器锁保护，但 `StringBuilder` 的内容修改不因此线程安全：

```java
Stack<StringBuilder> stack = new Stack<>();
stack.push(new StringBuilder());

StringBuilder builder = stack.peek();
builder.append("data");
```

`peek()` 返回引用后，对象锁已经释放。多个线程若同时修改同一个元素，仍需元素自身的同步协议、线程封闭或不可变设计。`Stack` 只管理引用序列，不会把元素变成线程安全对象。

## 遍历不是从栈顶到栈底

### Iterator 按列表顺序前进

`Stack` 继承 `Vector` 的 `Iterable` 能力。增强 `for`、`iterator()` 和 `listIterator()` 采用列表顺序，即从下标 `0` 到 `size() - 1`，也就是从栈底到栈顶。

```java
Stack<String> stack = new Stack<>();
stack.push("A");
stack.push("B");
stack.push("C");

for (String value : stack) {
    System.out.println(value);
}
```

输出顺序是 `A`、`B`、`C`，而连续 `pop()` 的顺序是 `C`、`B`、`A`。把增强 `for` 当作“按弹出顺序遍历”是常见误区。

若只想查看从栈顶到栈底的顺序，又不想修改原栈，可以使用继承来的反向列表迭代器：

```java
ListIterator<String> iterator = stack.listIterator(stack.size());
while (iterator.hasPrevious()) {
    System.out.println(iterator.previous());
}
```

这仍是列表 API，不是 `Stack` 专门提供的栈顶迭代器。它说明需求若经常涉及反向遍历，`Deque` 的迭代能力通常更贴近双端结构。

### Enumeration 也按底到顶观察

作为 `Vector` 子类，`Stack` 还继承 `elements()`，返回历史风格的 `Enumeration`。它同样从低下标向高下标读取，即从栈底到栈顶。

`Enumeration` 与现代 `Iterator` 的一个重要差异是传统实现不提供相同的 fail-fast 并发修改检测契约。不能因为它没有及时抛出 `ConcurrentModificationException`，就认为并发修改下的遍历结果稳定可靠。

### fail-fast 不是并发安全保证

`Iterator` 和 `ListIterator` 通常依据结构修改计数进行尽力而为的并发修改检测。如果创建迭代器后，通过迭代器之外的路径结构性修改容器，后续迭代可能抛出 `ConcurrentModificationException`。

```java
Stack<Integer> stack = new Stack<>();
stack.push(1);
stack.push(2);
stack.push(3);

for (Integer value : stack) {
    if (value == 2) {
        stack.pop();
    }
}
```

这类代码不能依赖“是否刚好抛异常”来证明正确性。fail-fast 的目标是尽早暴露部分错误用法，不是提供强一致快照，也不是可靠的线程间协调机制。

如果遍历期间需要删除当前元素，应使用迭代器自身允许的修改方法，并接受这是按列表顺序处理；如果要求遍历期间结构不变，则应在整个遍历期间持有同一个 `Stack` 实例锁，或先在锁内复制快照，再在锁外处理。

```java
static <E> List<E> snapshot(Stack<E> stack) {
    synchronized (stack) {
        return new ArrayList<>(stack);
    }
}
```

快照复制的是元素引用，不会深拷贝元素对象。它能固定容器成员和顺序，不能冻结元素内部状态。

### 顺序流继承的也是列表顺序

`stack.stream()` 的顺序与迭代器一致，也是从栈底到栈顶。若算法语义要求栈顶优先，直接对 `Stack` 调用顺序流会产生与 `pop` 次序相反的处理顺序。可以先复制后反转，也可以按下标逆序生成流，但这进一步说明 `Stack` 不是一个只暴露栈视角的类型。

## 异常与边界条件

### EmptyStackException

空栈调用 `peek()` 或 `pop()` 抛出 `EmptyStackException`。它是一个非受检异常，继承自 `RuntimeException`，调用方不需要在方法签名中声明。

```java
Stack<String> stack = new Stack<>();

try {
    stack.pop();
} catch (EmptyStackException exception) {
    System.out.println("stack is empty");
}
```

对于“空栈代表算法状态错误”的场景，异常式 API 能尽早暴露不变量破坏。例如括号解析完成前意外遇到右括号，直接弹出空栈通常说明输入非法或实现有误。

对于“空栈是正常控制流”的场景，反复依赖异常可能让逻辑不清晰。现代 `Deque` 同时提供抛异常和返回特殊值的成对方法，调用方可以显式选择。

### 继承方法可能抛出不同异常

若调用方不用 `peek()`，而是写 `get(size() - 1)`、`lastElement()` 或 `elementAt(size() - 1)`，空容器时可能观察到不同的下标类异常。结构结果相近不代表异常契约相同。

迁移、重构或封装时，应测试调用方是否依赖具体异常类型。尤其是公共方法若原样传播 `EmptyStackException`，替换内部实现后不应无意改成 `NoSuchElementException` 或 `IndexOutOfBoundsException`，除非公共契约允许变化。

### null 元素的歧义

`Stack` 允许 `null`，所以以下两个状态必须通过结构信息区分：

1. 栈为空，没有栈顶。
2. 栈非空，栈顶元素恰好是 `null`。

`peek()` 通过“空时抛异常、非空时返回元素”保留了区分能力。若自行包装成 `peekOrNull`，就会丢失区别。迁移到禁止 `null` 的实现时，这种历史用法必须先清理或编码为明确的哨兵对象、枚举状态、结果记录类型。

### 可变对象与 search

`search` 依赖 `equals`。如果元素的相等性与可变字段有关，入栈后修改这些字段会改变后续搜索结果。这不会破坏数组结构，却可能破坏业务认知。

此外，某些不规范的 `equals` 实现可能抛异常、违反对称性或在并发修改下产生不稳定结果。`Stack` 不会修正元素类型的相等性问题。对搜索结果有严格要求时，应优先使用不可变值对象，并保证 `equals` 契约正确。

## 复杂度与实际性能

在只使用栈顶操作的前提下，可以得到以下复杂度：

| 操作 | 时间复杂度 | 说明 |
| --- | --- | --- |
| `push` | 均摊 O(1)，扩容时 O(n) | 末尾追加；容量不足时复制数组 |
| `peek` | O(1) | 读取末尾引用 |
| `pop` | O(1) | 读取并删除末尾，不移动前部元素 |
| `empty` | O(1) | 判断逻辑长度 |
| `search` | O(n) | 从末尾向前线性查找 |
| 按下标访问 | O(1) | 来自 `Vector`，但不属于最小栈抽象 |
| 中间插入或删除 | O(n) | 来自 `Vector`，需要移动后续引用 |

复杂度只是第一层判断。与 `ArrayDeque` 相比，`Stack` 的实际成本还包括以下因素：

- 核心操作沿用 `Vector` 的同步机制，同一实例存在锁获取和竞争成本。
- 历史 API 较宽，更容易被非栈操作使用，增加维护和推理成本。
- 底层是连续引用数组，遍历局部性通常优于逐节点链式结构。
- 扩容会复制已有引用，峰值栈深很大且增长频繁时可能产生明显延迟。
- 弹出会清理原末尾引用，不会因为逻辑删除而永久保留该元素。

不能仅凭“有 `synchronized` 所以一定慢”下结论。现代 JVM 对无竞争锁有优化，具体差异受 JDK、运行环境、数据规模和竞争程度影响。更可靠的工程判断是：`Stack` 的同步语义通常没有为单线程局部算法提供价值，而 `ArrayDeque` 的 API 和实现更符合这一主流场景；若性能重要，应在真实工作负载上基准测试，不要用微小循环替代整体分析。

对于深度未知的搜索或解析算法，栈的内存占用与待处理状态数量有关。使用显式 `Stack` 可以避免递归调用栈深度限制，但并不会消除空间成本；它只是把状态从线程调用栈转移到堆上的集合对象中。若算法持续压入且无法收敛，任何可增长容器都可能耗尽内存。

## 遗留价值：何时保留 Stack

“推荐 `Deque`”是新代码选择原则，不是无条件重写命令。以下场景中，保留 `Stack` 可能是合理的阶段性决定：

### 公共签名要求具体 Stack

旧库可能声明：

```java
public void process(Stack<Token> tokens)
```

调用方必须提供 `Stack`，内部即使更偏好 `Deque` 也无法直接改变外部签名。若该 API 已被广泛使用，修改参数类型可能造成源码兼容或二进制兼容问题。可以先在边界层适配，再通过新重载、新版本接口或弃用流程迁移。

### 序列化数据包含 Stack

若对象字段以 Java 序列化或其他保留具体类型信息的形式持久化，直接把字段改为 `ArrayDeque` 可能导致旧数据无法读取、字段兼容逻辑变化或序列化形态改变。迁移应包含旧数据读取、转换和回归测试，而不是只修改声明。

### 代码依赖 null 元素

`Stack` 允许 `null`，`ArrayDeque` 不允许。直接替换会把原先成功的 `push(null)` 变成 `NullPointerException`。应先判断 `null` 是真实业务值、结束标记、未初始化状态，还是历史缺陷，再选择明确建模方式。

### 代码依赖 Vector/List 能力

如果现有代码同时使用 `push`、随机访问、`subList`、排序或任意位置修改，那么它实际上依赖的是“带栈方法的列表”，不是纯栈。替换为 `Deque` 前必须先重构数据模型或拆分职责。

### 外部同步以 Stack 实例作为锁

遗留代码可能存在大量 `synchronized (stack)`，并把该对象锁用于保护额外状态。把字段替换为另一个对象会改变锁身份和同步协议。迁移前要审查所有别名、锁获取位置和锁顺序。

保留 `Stack` 时，也应明确它是兼容选择而非新设计模板。可以把使用范围限制在适配层，通过私有方法封装栈操作，避免继续扩散具体类型。

## 现代替代：Deque 与 ArrayDeque

`Deque<E>` 表示双端队列，允许在两端插入和删除。它包含一组与栈语义直接对应的方法：

| 栈语义 | `Stack` | `Deque` |
| --- | --- | --- |
| 压入栈顶 | `push(e)` | `push(e)`，等价于 `addFirst(e)` |
| 弹出栈顶 | `pop()` | `pop()`，等价于 `removeFirst()` |
| 查看栈顶 | `peek()` | `peek()`，等价于 `peekFirst()` |
| 判断为空 | `empty()` | `isEmpty()` |

常见声明是：

```java
Deque<String> stack = new ArrayDeque<>();
stack.push("A");
stack.push("B");
stack.push("C");

System.out.println(stack.pop()); // C
System.out.println(stack.peek()); // B
```

这里的栈顶位于双端队列的头部。`push` 调用 `addFirst`，`pop` 调用 `removeFirst`。这与 `Stack` 把栈顶放在数组末尾的内部映射不同，但对只使用栈 API 的调用方，LIFO 结果一致。

### 为什么变量类型应优先写 Deque

声明为 `Deque<E>` 能表达“实现需要双端插入删除能力”，不会暴露 `List` 的随机访问、排序和中间位置修改。虽然 `Deque` 仍比最小栈接口更宽，因为它允许操作两端，但比 `Stack extends Vector` 更接近目标抽象。

Java 标准库没有独立的现代 `Stack` 接口。若领域确实要求严格限制为 `push`、`pop`、`peek`，可以在模块内部定义小接口或封装类；不过一般局部算法直接使用 `Deque` 已足够清晰。

### ArrayDeque 的关键差异

迁移时至少要审查以下差异：

1. `ArrayDeque` 不允许 `null`，`Stack` 允许。
2. `Stack.peek()` 空时抛 `EmptyStackException`；`Deque.peek()` 空时返回 `null`。
3. `Stack.pop()` 空时抛 `EmptyStackException`；`Deque.pop()` 空时通常抛 `NoSuchElementException`。
4. `Stack` 的增强 `for` 从栈底到栈顶；以 `push` 使用 `ArrayDeque` 时，普通迭代从栈顶到栈底。
5. `Stack` 方法带有传统同步语义；`ArrayDeque` 不是线程安全容器。
6. `Stack` 是 `List`，支持下标和中间修改；`ArrayDeque` 不是 `List`。
7. 两者的序列化形态和具体类型不同，不能假设跨边界透明替换。

第 4 点尤其容易漏掉。以下两段代码具有相同的弹出次序，却具有相反的普通迭代次序：

```java
Stack<String> legacy = new Stack<>();
legacy.push("A");
legacy.push("B");
legacy.push("C");

Deque<String> modern = new ArrayDeque<>();
modern.push("A");
modern.push("B");
modern.push("C");
```

`legacy` 的增强 `for` 输出 `A, B, C`，`modern` 的增强 `for` 输出 `C, B, A`。若原代码既弹栈又遍历，迁移必须为两种顺序分别写测试。

### Deque 的两组失败语义

`Deque` 为端点操作提供抛异常与返回特殊值的两组方法。例如：

- `removeFirst`、`pop`：为空时抛异常。
- `pollFirst`、`poll`：为空时返回 `null`。
- `getFirst`、`element`：为空时抛异常。
- `peekFirst`、`peek`：为空时返回 `null`。

由于 `ArrayDeque` 禁止 `null` 元素，`poll` 或 `peek` 返回 `null` 可以无歧义表示空容器。迁移时应根据原公共契约选择方法，而不是仅按名称机械替换。

### LinkedList 通常不是首选栈实现

`LinkedList` 也实现 `Deque`，能够作为栈使用，但每个元素需要链表节点，带来额外对象和引用开销，遍历局部性通常也不如数组结构。除非需求确实依赖其特定行为，否则普通 LIFO 工作负载通常优先考虑 `ArrayDeque`。

`LinkedList` 允许 `null`，但利用这一点维持旧代码中的空值设计，往往会继续保留歧义。若 `null` 不是不可变更的外部契约，迁移时更好的做法是先把状态建模清楚。

## 迁移方法与权衡

安全迁移不是把类名全文替换，而是列出旧代码实际依赖的语义。

### 第一步：确认所有使用方式

检查目标 `Stack` 是否只调用以下方法：

- `push`
- `pop`
- `peek`
- `empty` 或 `isEmpty`
- `size`

还要搜索它是否被作为 `List`、`Vector` 或 `Collection` 传递，是否调用 `get`、`set`、`add(index, e)`、`remove(index)`、`elements`、`listIterator`、`subList`、`sort`、`ensureCapacity`，以及是否被序列化或用作同步锁。

只看变量声明附近往往不够，因为别名会把具体类型能力传播到其他方法。

### 第二步：确定空值和空栈契约

需要回答：

- 是否允许压入 `null`？
- 空栈查看是异常还是返回特殊值？
- 空栈弹出是否被调用方捕获为 `EmptyStackException`？
- 某个返回 `null` 的包装方法，究竟表示空栈还是实际 `null` 元素？

这些问题决定能否直接使用 `ArrayDeque`，以及应选择 `pop`、`removeFirst`、`pollFirst` 中的哪一个。

### 第三步：固定顺序测试

至少测试三种顺序：

1. 连续压入后的连续弹出顺序。
2. 普通迭代顺序。
3. 搜索或快照逻辑依赖的顺序。

如果外部接口要求栈底优先列表，可以显式转换，而不是依赖容器默认迭代：

```java
static <E> List<E> bottomToTop(Deque<E> stack) {
    List<E> result = new ArrayList<>(stack);
    Collections.reverse(result);
    return result;
}
```

这里假设 `Deque` 按 `push` 使用，普通迭代为栈顶到栈底。转换函数把顺序要求写进名称和实现，避免调用方猜测。

### 第四步：重新设计并发边界

不能因为旧 `Stack` 有同步路径，就把它替换为共享的 `ArrayDeque` 后继续无锁访问。选择包括：

- 将栈限制在线程内部，不再共享。
- 使用外部锁保护所有访问和复合操作。
- 根据问题改用专用并发双端队列，例如需要并发端点操作时评估 `ConcurrentLinkedDeque`。
- 若需要阻塞生产消费语义，评估阻塞双端队列，而不是普通栈。

不同并发结构的原子操作、迭代一致性、容量和阻塞契约不同，不能只根据“都能 push/pop”判断等价。

### 第五步：隔离兼容层

若外部 API 暂时必须接收或返回 `Stack`，可以让核心算法使用 `Deque`，仅在边界转换：

```java
static <E> ArrayDeque<E> toDeque(Stack<E> source) {
    ArrayDeque<E> result = new ArrayDeque<>(source.size());
    for (E element : source) {
        if (element == null) {
            throw new IllegalArgumentException("null elements cannot be migrated");
        }
        result.addLast(element);
    }
    return result;
}
```

这段转换按 `Stack` 的底到顶迭代顺序，把元素依次加入 `ArrayDeque` 尾部。转换后若用 `removeLast` 作为栈顶操作，顺序与旧 `Stack` 一致；若核心代码坚持用 `Deque.push/pop` 操作头部，则应采用相反的构造顺序。关键不是某一种写法绝对正确，而是明确约定“哪一端是栈顶”，并让构造、迭代和弹出保持一致。

## 典型算法案例一：括号匹配

括号匹配适合展示栈的局部状态语义：遇到左括号时记录，遇到右括号时必须与最近尚未匹配的左括号配对。“最近尚未匹配”正是 LIFO。

以下示例使用 `Stack<Character>`，目的是说明该类的行为：

```java
import java.util.Stack;

public final class BracketValidator {
    private BracketValidator() {
    }

    public static boolean isBalanced(String text) {
        Stack<Character> openings = new Stack<>();

        for (int i = 0; i < text.length(); i++) {
            char current = text.charAt(i);
            if (current == '(' || current == '[' || current == '{') {
                openings.push(current);
                continue;
            }

            if (current == ')' || current == ']' || current == '}') {
                if (openings.empty()) {
                    return false;
                }
                char opening = openings.pop();
                if (!matches(opening, current)) {
                    return false;
                }
            }
        }

        return openings.empty();
    }

    private static boolean matches(char opening, char closing) {
        return (opening == '(' && closing == ')')
                || (opening == '[' && closing == ']')
                || (opening == '{' && closing == '}');
    }

    public static void main(String[] args) {
        System.out.println(isBalanced("{a + [b * (c - d)]}")); // true
        System.out.println(isBalanced("{a + [b * (c - d])}")); // false
    }
}
```

这个算法依赖三个不变量：

1. 栈中只保存尚未匹配的左括号。
2. 栈顶是最近遇到的尚未匹配左括号。
3. 扫描结束时栈必须为空，否则仍有未闭合括号。

这里不需要随机访问、排序、同步或 `search`。因此从现代 Java 设计看，`ArrayDeque<Character>` 更合适；把声明改为 `Deque<Character> openings = new ArrayDeque<>();` 后，核心 `push`、`pop`、`isEmpty` 逻辑保持直观。

需要注意，真实解析器可能还要忽略字符串字面量、转义字符和注释中的括号。那些属于词法或语法规则，不是 `Stack` 本身的能力。不能因为使用了栈，就假设任意文本中的括号扫描都能正确解析语言。

## 典型算法案例二：显式深度优先遍历

递归深度优先遍历依赖调用栈保存“之后还要处理的节点”。显式 `Stack` 可以把这部分状态搬到堆上的集合中，并让遍历顺序更可控。

```java
import java.util.ArrayList;
import java.util.List;
import java.util.Stack;

public final class DepthFirstTraversal {
    static final class Node {
        final String name;
        final List<Node> children = new ArrayList<>();

        Node(String name) {
            this.name = name;
        }

        Node add(Node child) {
            children.add(child);
            return this;
        }
    }

    public static List<String> preOrder(Node root) {
        List<String> result = new ArrayList<>();
        if (root == null) {
            return result;
        }

        Stack<Node> pending = new Stack<>();
        pending.push(root);

        while (!pending.empty()) {
            Node current = pending.pop();
            result.add(current.name);

            for (int i = current.children.size() - 1; i >= 0; i--) {
                pending.push(current.children.get(i));
            }
        }

        return result;
    }

    public static void main(String[] args) {
        Node root = new Node("A")
                .add(new Node("B").add(new Node("D")).add(new Node("E")))
                .add(new Node("C"));

        System.out.println(preOrder(root)); // [A, B, D, E, C]
    }
}
```

子节点按逆序压栈，是因为最后压入者最先弹出。若希望按列表中的正序访问孩子，就必须从末尾向前压入。这里的顺序推导直接依赖 LIFO，不能随意改成正序压栈。

显式栈避免了非常深递归可能造成的线程调用栈溢出，但它仍可能占用大量堆内存。若输入不是树而是含环图，还必须维护已访问集合，否则节点会被反复压入。栈只负责待处理顺序，不负责自动检测环。

## 典型算法案例三：逆波兰表达式求值

后缀表达式把操作符放在操作数之后。例如 `2 3 4 * +` 表示 `2 + 3 * 4`。扫描到数字时压栈，扫描到二元操作符时弹出两个操作数并把结果压回。

```java
import java.util.Stack;

public final class PostfixEvaluator {
    private PostfixEvaluator() {
    }

    public static int evaluate(String expression) {
        Stack<Integer> operands = new Stack<>();

        for (String token : expression.trim().split("\\s+")) {
            if (isOperator(token)) {
                int right = operands.pop();
                int left = operands.pop();
                operands.push(apply(left, right, token));
            } else {
                operands.push(Integer.parseInt(token));
            }
        }

        if (operands.size() != 1) {
            throw new IllegalArgumentException("invalid postfix expression");
        }
        return operands.pop();
    }

    private static boolean isOperator(String token) {
        return token.equals("+")
                || token.equals("-")
                || token.equals("*")
                || token.equals("/");
    }

    private static int apply(int left, int right, String operator) {
        switch (operator) {
            case "+":
                return left + right;
            case "-":
                return left - right;
            case "*":
                return left * right;
            case "/":
                return left / right;
            default:
                throw new IllegalArgumentException("unknown operator: " + operator);
        }
    }

    public static void main(String[] args) {
        System.out.println(evaluate("2 3 4 * +")); // 14
        System.out.println(evaluate("20 5 2 - /")); // 6
    }
}
```

弹出顺序不能写反。第一个弹出的是右操作数，第二个弹出的是左操作数。对加法和乘法，顺序错误可能被交换律掩盖；对减法和除法会直接产生错误结果。

这个简化示例还揭示了异常边界：操作数不足时，`pop()` 抛 `EmptyStackException`；数字格式错误时抛 `NumberFormatException`；除数为零时抛 `ArithmeticException`；最终剩余元素不为一个时主动判定表达式非法。生产级解析器通常应把这些底层异常转换为带输入位置的领域错误，而不是直接泄露容器异常。

## 典型算法案例四：带最小值查询的双栈结构

`Stack` 也可以作为更高层结构的组成部分。一个常见例子是同时维护数据栈和最小值栈，使压入、弹出和查询最小值都达到 O(1)。

```java
import java.util.EmptyStackException;
import java.util.Stack;

public final class MinStack {
    private final Stack<Integer> values = new Stack<>();
    private final Stack<Integer> minimums = new Stack<>();

    public void push(int value) {
        values.push(value);
        if (minimums.empty() || value <= minimums.peek()) {
            minimums.push(value);
        }
    }

    public int pop() {
        int value = values.pop();
        if (value == minimums.peek()) {
            minimums.pop();
        }
        return value;
    }

    public int min() {
        if (minimums.empty()) {
            throw new EmptyStackException();
        }
        return minimums.peek();
    }

    public boolean isEmpty() {
        return values.empty();
    }
}
```

最小值栈在遇到“小于或等于当前最小值”时压入，不能只在严格小于时压入。否则重复最小值中的一个被弹出后，辅助栈会过早丢失仍然存在的最小值。

这个实现还有一个并发边界：虽然两个字段分别是同步 `Stack`，`push` 和 `pop` 对两个栈的联合更新并不是自动原子。若 `MinStack` 会被多线程共享，应在更高层同步其公共方法，而不是依赖两个内部容器各自加锁。更进一步，既然高层已经统一加锁，内部通常没有必要继续使用 `Stack`，可以改为非线程安全的 `ArrayDeque`，把同步责任集中在 `MinStack`。

## 常见误区与纠正

### 误区一：Stack 是一个只允许栈操作的类型

实际情况：`Stack` 继承 `Vector`，公开完整列表能力。LIFO 只是建议使用方式，不是类型系统强制约束。

纠正方式：新代码优先把变量声明为 `Deque<E>`；遗留 `Stack` 应避免调用和传播列表方法。

### 误区二：Stack 线程安全，所以组合逻辑也是线程安全的

实际情况：单个操作的同步不能覆盖 `empty` 后 `pop`、跨两个栈转移、遍历期间保持稳定等复合条件。

纠正方式：识别需要原子维护的不变量，在统一外部锁或更高层对象中完成；复杂并发场景选择契约更匹配的并发结构。

### 误区三：增强 for 按弹栈顺序遍历

实际情况：`Stack` 按 `Vector` 的列表顺序从下标 `0` 向后遍历，即从栈底到栈顶。

纠正方式：需要栈顶优先时显式逆序遍历，或使用迭代顺序符合约定的 `Deque`，并为迁移前后顺序写测试。

### 误区四：search 返回数组下标

实际情况：`search` 返回从栈顶计算的 1 基距离；栈顶是 `1`，不存在是 `-1`。

纠正方式：不要将结果直接传给 `get`。若确实需要列表下标，可以在结果非负时计算 `size() - distance`，但频繁这样做说明调用方依赖列表视角。

### 误区五：push 不声明 synchronized，所以并发压栈不安全

实际情况：`push` 调用同步的 `Vector.addElement`，压入过程通过同一实例锁保护。判断线程安全不能只看最外层方法修饰符，还要看调用路径。

纠正方式：准确描述为“单次压入受底层同步方法保护”，同时不要把这一点扩大为复合操作原子。

### 误区六：Stack 不允许 null

实际情况：`Stack` 继承 `Vector` 的元素策略，允许 `null`。禁止 `null` 的是常用替代实现 `ArrayDeque`。

纠正方式：迁移前搜索并测试空值行为。尽量把“无值”建模为明确状态，而不是使用 `null` 栈元素。

### 误区七：把 Stack 换成 ArrayDeque 只改一行声明即可

实际情况：弹出顺序可能一致，但空值、异常类型、迭代顺序、同步语义、列表能力和序列化形态都可能变化。

纠正方式：先列语义清单，再迁移实现；尤其覆盖空栈、重复元素、迭代、并发和外部接口。

### 误区八：为了避免 EmptyStackException，任何地方都先 empty 再 pop

实际情况：在单线程中这可能只是冗长；在并发中仍存在竞态。若空栈是非法状态，提前检查还可能掩盖算法不变量问题。

纠正方式：根据契约选择策略。非法状态可以让 `pop` 直接失败；正常空结果可以使用 `Deque.poll`；并发原子检查需要统一同步或合适的并发 API。

### 误区九：Stack 的同步能保护迭代全过程

实际情况：获得迭代器、调用 `hasNext`、调用 `next` 是多个步骤，其他线程可在步骤之间修改容器。fail-fast 也不是同步机制。

纠正方式：在同一个 `Stack` 锁内完成遍历，或在锁内复制快照后遍历快照。

### 误区十：显式 Stack 一定比递归更省内存

实际情况：显式栈仍保存待处理状态，空间复杂度取决于算法。它主要改变状态存放位置和可控性，不会凭空消除状态。

纠正方式：分析最大待处理规模、元素对象大小和终止条件，而不是仅比较“递归”和“循环”的表面形式。

## API 设计建议

公共 API 应尽量表达调用方真正需要的最小能力。若一个方法只需要逐个弹出元素，直接接收可变 `Stack` 会带来两个问题：调用方失去所有权清晰度，方法也可以使用大量无关列表操作。

可以考虑以下设计：

- 方法内部自己创建 `ArrayDeque`，不把工作栈暴露出去。
- 输入使用不可变或只读的数据序列，方法内部转换为栈。
- 需要调用方提供双端结构时接收 `Deque<E>`，并文档化方法是否会消费其中元素。
- 领域上只允许栈操作且边界长期稳定时，定义窄接口并隐藏具体实现。

无论使用 `Stack` 还是 `Deque`，都应说明所有权：方法是否会弹出调用方传入的元素，是否保留引用，是否允许 `null`，是否可跨线程共享，迭代顺序按栈顶还是栈底开始。许多容器问题不是底层实现错误，而是所有权与顺序契约没有写清楚。

不建议在新公共接口中返回具体 `Stack`，除非兼容性要求如此。返回具体遗留类型会鼓励调用方依赖同步、下标、`Vector` 历史方法和序列化形态，使将来迁移更困难。

## 审查 Stack 代码的检查表

维护现有 `Stack` 代码时，可以按以下问题逐项确认：

1. 该变量是否只执行栈顶操作，还是也被当作 `List`、`Vector` 使用？
2. 哪一端被定义为栈顶，普通迭代顺序是否与业务顺序一致？
3. 是否允许 `null`，空栈和 `null` 元素如何区分？
4. 调用方是否依赖 `EmptyStackException`、具体类名或序列化形式？
5. 是否存在 `empty` 后 `pop`、`peek` 后 `pop` 等检查后执行竞态？
6. 是否在多个容器之间维护联合不变量？
7. 遍历期间是否可能结构性修改，是否需要锁内快照？
8. 元素对象是否可变，容器同步是否被误认为元素同步？
9. `search` 是否处于热点路径，线性扫描成本是否可接受？
10. 深度是否可能无界，扩容和内存占用是否有上限？
11. 替换为 `ArrayDeque` 后，迭代顺序和异常类型是否改变？
12. 若仍保留 `Stack`，能否把它限制在遗留适配层而不继续扩散？

这些问题共同决定 `Stack` 是“仍需兼容的具体类型”，还是“可以迁移的内部实现”。选择不应只依据类的年代，也不应只依据某个微基准结果。

## 总结

`java.util.Stack` 用很少的新增方法提供了 LIFO 操作，但它不是一个窄而纯粹的栈类型。由于继承 `Vector`，它同时暴露随机访问、任意位置修改、排序、历史遍历 API、容量管理、同步和序列化等能力。这个继承关系解释了它的大部分设计后果。

只看核心方法时，语义清晰：`push` 压入并返回元素，`peek` 查看栈顶，`pop` 查看并删除栈顶，`empty` 判断当前是否为空，`search` 返回从栈顶计算的 1 基距离。空栈上的 `peek` 和 `pop` 抛 `EmptyStackException`，`null` 可以作为有效元素。

线程安全必须限定在准确范围内：单次容器操作受同步机制保护，但检查后执行、遍历、跨容器更新和元素内部状态不因此自动安全。迭代又继承列表方向，从栈底到栈顶，与连续弹出的次序相反。

对新代码，通常使用 `Deque<E>` 配合 `ArrayDeque<E>`，因为它的能力边界、性能特征和栈式 API 更符合现代集合设计。对遗留代码，迁移前必须核对 `null`、异常、迭代顺序、同步、列表能力、锁身份和序列化兼容。理解这些边界，比简单记住“Stack 已过时，改用 Deque”更能支持可靠的工程决策。

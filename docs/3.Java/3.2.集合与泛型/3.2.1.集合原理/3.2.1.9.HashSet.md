# 3.2.1.9 HashSet

`HashSet` 是 Java 集合框架中最常用的通用集合实现之一。它实现 `Set` 接口，用元素相等性表达“成员是否已经存在”，适合去重、成员测试以及交集、并集、差集等集合运算。它不通过下标访问元素，不承诺迭代顺序，也不是线程安全容器。

理解 `HashSet` 不能只停留在“元素不重复”和“查询平均为 O(1)”。它的公开语义来自 `Set` 契约，常见实现则把元素存为 `HashMap` 的键。两层含义需要分开：前者决定调用者可以依赖什么，后者解释这些行为通常如何实现、成本来自哪里。容量、负载因子、哈希冲突、迭代成本、快速失败、克隆和序列化等问题，都建立在这个区分之上。

## Set 契约：集合中的“相同”意味着什么

`Set` 表示不包含重复元素的集合。这里的“不重复”不是引用地址不同，也不是某个业务字段看起来不同，而是由对象的相等性协议决定。对通常使用 `equals` 的集合实现而言，如果两个非空对象满足 `a.equals(b)`，集合便把它们视为同一个成员；`null` 与 `null` 也代表同一个成员。

因此，`HashSet.add(e)` 的返回值具有明确语义：

- 集合原本没有与 `e` 相等的元素，插入成功并返回 `true`。
- 集合已经包含与 `e` 相等的元素，集合保持不变并返回 `false`。

`Set` 不提供“按位置取得第几个元素”的操作，因为集合抽象关心成员关系而不是位置。即使某个实现恰好有可观察的遍历次序，这个次序也不自动成为 `HashSet` 的契约。把集合转换为数组或列表后可以按位置访问，但位置属于转换结果，不属于原集合的成员语义。

集合自身也定义了值语义。两个 `Set` 在包含相同元素时相等，而不要求具体实现相同，也不要求迭代顺序相同。一个 `HashSet` 可以与一个包含相同成员的 `TreeSet` 或 `LinkedHashSet` 相等。`Set.hashCode()` 则由各元素哈希值的总和定义，空元素按零处理。这使相等集合具有相同哈希码，也意味着集合实现不能随意把内部桶位置或迭代顺序掺入集合哈希值。

```java
Set<String> left = new HashSet<>(List.of("A", "B"));
Set<String> right = new LinkedHashSet<>(List.of("B", "A"));

System.out.println(left.equals(right)); // true
System.out.println(left.hashCode() == right.hashCode()); // true
```

`Set` 契约约束的是最终可观察行为，而不是具体存储方式。`HashSet` 选择哈希表实现成员关系；`TreeSet` 通过排序比较结果定义元素是否重复；`EnumSet` 使用枚举常量对应的位来表示成员。它们都属于集合，却有不同的顺序、空值、性能和并发边界。

## 常见实现：把元素存成 HashMap 的键

在常见 OpenJDK 实现中，`HashSet<E>` 内部维护一个 `HashMap<E, Object>`。集合元素作为映射的键，所有键都关联到同一个静态占位对象。集合只关心键是否存在，不关心与键关联的业务值，因此不需要为每个元素创建不同的 value。

概念上可以把核心字段和新增操作简化为：

```java
private transient HashMap<E, Object> map;
private static final Object PRESENT = new Object();

public boolean add(E element) {
    return map.put(element, PRESENT) == null;
}
```

这段代码用于说明实现关系，不应当替代对具体 JDK 源码的核对。内部字段、构造路径和序列化细节属于实现层面，可能随 JDK 版本变化；“不允许重复元素”“不保证顺序”“基本操作在哈希分布良好时具有常数级平均成本”等才是调用者更稳定的认识。

占位对象有三个作用。第一，它满足 `HashMap` 必须存储键值对的结构要求。第二，所有元素共享同一个对象，避免为每个成员额外创建一个独立值对象。第三，`put` 返回旧值，旧值是否为 `null` 可以直接转换为“此前是否存在该键”的布尔结果。由于 `HashSet` 自己控制内部映射，真实成员统一关联非空占位值，所以这种判断不会与“键存在但值为 null”的情况混淆。

委托给 `HashMap` 也意味着许多行为直接继承自键视角：

- 元素的哈希码决定候选桶位置。
- `equals` 用于在候选元素中确认是否相等。
- 容量、负载因子、扩容和冲突处理由底层映射负责。
- 迭代实际遍历底层映射的键。
- 底层映射允许一个 `null` 键，因此 `HashSet` 允许一个 `null` 元素。

“基于 `HashMap`”是解释常见实现的关键，但不能倒推出所有 `Set` 都允许 `null`、都有相同复杂度，或都使用 `equals` 加 `hashCode`。这些结论只适用于相应实现及其文档前提。

## add、contains 与 remove 的同一条定位路径

`HashSet` 最核心的三个操作是 `add`、`contains` 和 `remove`。它们表面上分别表示新增、查询和删除，底层却依赖同一套定位协议：先根据元素哈希值缩小候选范围，再用相等性确认目标。

### add：返回的是集合是否发生变化

新增元素时，底层映射尝试加入“元素 → 占位对象”这条映射。如果不存在相等键，就建立新节点、增加集合大小并返回 `true`；如果已有相等键，则保留集合中原来的键对象并返回 `false`。

“保留原来的键对象”是容易忽略的细节。`HashSet` 不是按相等性自动替换对象的容器。传入一个与已有元素相等、但并非同一引用的新对象，不会把旧对象换成新对象。如果业务需要“按标识找到并替换完整对象”，通常应使用 `Map<Id, Value>` 明确表达更新语义，而不是依赖 `Set.add`。

```java
final class User {
    private final long id;
    private final String name;

    User(long id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public boolean equals(Object other) {
        if (this == other) {
            return true;
        }
        if (!(other instanceof User)) {
            return false;
        }
        User that = (User) other;
        return id == that.id;
    }

    @Override
    public int hashCode() {
        return Long.hashCode(id);
    }
}

Set<User> users = new HashSet<>();
users.add(new User(1L, "old"));
boolean changed = users.add(new User(1L, "new"));

System.out.println(changed); // false
System.out.println(users);   // 仍然保存 name 为 old 的对象
```

### contains：查询的是相等成员，不是同一引用

`contains(candidate)` 会根据 `candidate` 的哈希码定位候选桶，再检查桶内是否存在与其相等的键。调用者传入的对象不必就是插入时的那一个引用，只要相等性和哈希协议一致，就能命中。

平均 O(1) 是基于哈希分布合理、容量管理正常等前提的期望成本，不是每次操作只做一步，也不是严格最坏上界。计算哈希码本身可能昂贵；冲突会增加桶内比较；恶意或低质量哈希输入会改变实际成本。部分现代 JDK 的 `HashMap` 会在满足条件时把冲突严重的桶树化，以限制长链上的查询成本，但这属于具体实现优化，不应成为编写错误哈希函数的理由。

### remove：删除同样依赖当前哈希与相等性

`remove(candidate)` 先定位相等键，命中后移除对应映射。返回 `true` 表示成员原本存在且已删除，返回 `false` 表示没有找到。删除不存在的成员不是异常，而是一个没有改变集合的操作。

因为删除也要重新定位，所以对象放入集合后若其哈希相关字段发生变化，`remove` 可能找不到仍然存在于内部表中的对象。此时遍历集合还能看到它，却无法用当前对象状态正常删除。这不是容器随机失效，而是元素破坏了哈希定位所要求的稳定性。

## equals 与 hashCode：正确性的共同协议

哈希集合要求相等对象具有相同哈希码。Java 对 `equals` 与 `hashCode` 的核心约束是：如果两个对象根据 `equals` 相等，那么它们的 `hashCode` 必须相等。反方向不成立，不相等对象可以具有相同哈希码，哈希冲突正是为这种情况准备的。

`HashSet` 不会只凭哈希码判定重复。哈希码用于快速找到候选区域，`equals` 才确认逻辑相等。因此：

- 相等对象哈希码不同，会被分到不同候选位置，集合可能错误地同时保存两个逻辑重复对象。
- 不相等对象哈希码相同，只会形成冲突；集合仍能通过 `equals` 区分它们，但性能可能下降。
- `equals` 不满足自反、对称、传递或一致性，会使成员关系本身变得不可推理。

覆盖 `equals` 时通常必须同步覆盖 `hashCode`。只覆盖其中一个是使用哈希容器时最常见的错误之一。由 IDE、记录类或经过审查的值对象模板生成两者，通常比手写零散规则可靠，但仍需要确认参与相等性的字段是否符合业务身份。

```java
final class Coordinate {
    private final int x;
    private final int y;

    Coordinate(int x, int y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public boolean equals(Object other) {
        if (this == other) {
            return true;
        }
        if (!(other instanceof Coordinate)) {
            return false;
        }
        Coordinate that = (Coordinate) other;
        return x == that.x && y == that.y;
    }

    @Override
    public int hashCode() {
        return 31 * x + y;
    }
}
```

默认继承 `Object.equals` 和 `Object.hashCode` 时，对象通常按引用身份比较。这样的对象也能安全放进 `HashSet`，只是集合表达的是“同一个实例是否存在”，不是“字段内容相同的值是否存在”。问题不在于必须进行值比较，而在于集合语义必须与类型的相等性定义一致。

继承层次中的相等性尤其需要谨慎。父类与子类若对可比较范围理解不同，容易破坏对称性；可变类把大量状态都纳入相等性，则容易破坏入集合后的稳定性。相等性不是为了迎合 `HashSet` 临时编写的工具方法，而是类型对外部世界声明的值或身份协议。

## 元素稳定性：对象仍在集合里，为什么却找不到

元素放入 `HashSet` 后，在其作为成员期间，参与 `equals` 或 `hashCode` 的状态应保持稳定。这里不是要求对象的所有字段都不可变，而是要求决定哈希位置和逻辑相等性的字段不变。日志字段、缓存字段等不参与相等性的状态可以变化，但仍需考虑线程安全和业务含义。

下面的类型把可变用户名用于相等性。对象加入集合后修改用户名，哈希码随之改变，而内部节点不会自动从旧桶迁移到新桶：

```java
final class Account {
    private String username;

    Account(String username) {
        this.username = username;
    }

    void rename(String username) {
        this.username = username;
    }

    @Override
    public boolean equals(Object other) {
        if (!(other instanceof Account)) {
            return false;
        }
        Account account = (Account) other;
        return Objects.equals(username, account.username);
    }

    @Override
    public int hashCode() {
        return Objects.hashCode(username);
    }
}

Account account = new Account("alice");
Set<Account> accounts = new HashSet<>();
accounts.add(account);

account.rename("bob");

System.out.println(accounts.contains(account)); // 结果不能再按正常成员语义理解
System.out.println(accounts.remove(account));   // 可能为 false
```

这种状态有时被称为“幽灵元素”：迭代可以看到，按当前状态查询却失败。更严重的是，随后可能再次插入当前状态相等的对象，使集合在业务视角上出现重复。集合内部结构未必损坏，损坏的是调用者与集合之间的定位协议。

可靠策略按优先级通常包括：

1. 把成员设计为不可变值对象，参与相等性的字段设为不可变。
2. 使用创建后稳定的标识字段定义相等性，例如不可变编号。
3. 修改关键字段前先从集合删除，修改后重新加入；删除必须发生在旧哈希状态仍有效时。
4. 如果业务本质是按标识更新实体，使用 `Map<Id, Entity>`，把键稳定性与实体可变状态分离。

不要把“重新调用 `add`”当作修复。对象已位于旧桶时，按新哈希新增可能产生新的内部条目或逻辑重复。也不要在集合遍历过程中直接修改影响相等性的字段后继续依赖查询结果；即使没有抛出异常，语义也已经不可靠。

集合本身也可以作为另一个哈希集合的元素，但风险更高。`Set.hashCode()` 依赖其成员，内层集合一旦变化，外层集合中的键哈希也随之变化。因此，作为哈希键或哈希集合元素的集合应当在存续期间保持不变，或在边界处保存不可变副本。

## null 元素：允许一个，不等于应该普遍使用

`HashSet` 允许一个 `null` 元素。重复添加 `null` 只有第一次返回 `true`，因为集合视角下两个 `null` 代表同一个成员。`contains(null)` 和 `remove(null)` 都是有效操作。

```java
Set<String> values = new HashSet<>();

System.out.println(values.add(null));    // true
System.out.println(values.add(null));    // false
System.out.println(values.contains(null)); // true
System.out.println(values.remove(null)); // true
```

允许 `null` 来自该实现的公开行为以及底层映射对空键的支持，但它不代表所有 `Set` 都接受 `null`。不同集合工厂、不可变集合、并发集合或专用集合可能拒绝空元素。以 `Set` 接口作为参数时，除非 API 明确约定，调用者不应假设任意实现都允许 `null`。

从 API 设计看，`null` 还会引入含义不清的问题：它可能表示未知、缺失、未初始化或一个真实成员。若集合元素会被直接调用方法，空元素还会把空值检查扩散到遍历路径。是否允许 `null` 应是接口契约，而不是因为 `HashSet` 恰好支持就默认开放。

## 容量、负载因子与扩容

`HashSet` 的容量和负载因子实质上控制底层哈希表。容量可理解为桶数组规模，负载因子决定元素数量达到什么程度时触发扩容。默认负载因子通常为 `0.75`，它在空间占用和冲突概率之间取得通用折中。

构造器中的 `initialCapacity` 不是“集合最多能放多少元素”，也不是“立即创建完全等大的存储”。它是底层哈希结构容量规划的输入；具体数组何时分配、容量如何取整属于实现细节。集合可以继续增长，超过阈值时通常会扩容并重新组织桶位置。

预估容量的目标是容纳预期元素数而尽量避免扩容，而不是简单把预期元素数原样作为初始容量。若希望在负载因子 `f` 下容纳 `n` 个元素，概念上需要为 `n / f` 留出容量，再考虑实现的容量取整和上限。现代 JDK 还可能提供按预期映射数计算容量的工厂能力；使用时应以目标 JDK API 为准。

```java
int expectedSize = 10_000;
float loadFactor = 0.75f;
int initialCapacity = (int) Math.ceil(expectedSize / loadFactor);

Set<String> ids = new HashSet<>(initialCapacity, loadFactor);
```

上例表达估算思路，不保证精确对应某个 JDK 最终桶数组长度。对普通小集合，默认参数通常足够。只有在元素规模可预测、构建频繁或扩容成本已通过测量成为问题时，预设容量才有明显价值。

负载因子过低会提前扩容，减少冲突却消耗更多桶空间，并增加遍历空桶的成本；负载因子过高可以提高桶利用率，却可能增加冲突和单次操作常数。调整负载因子属于性能工程，不应在没有数据分布和基准结果时仅凭经验随意修改。

扩容不是只分配一个更大数组。桶索引依赖容量，容量变化后节点需要重新分布或按实现规则迁移。扩容期间会产生明显的一次性工作和额外内存压力。批量构建大型集合时合理预估容量，可以减少这类阶段性成本；长期保留一个远大于实际元素数的集合，则可能让空间和迭代持续付费。

从集合删除大量元素通常不会自动把底层容量缩回与当前大小匹配。若一个曾经很大的集合长期只保留少量元素，而且遍历成本或内存占用重要，可以新建适当大小的集合并复制存活元素。是否值得重建需要结合生命周期和测量判断。

## 复杂度：平均常数时间不等于无条件常数

在哈希分布良好且容量管理合理时，`add`、`contains` 和 `remove` 具有常数级平均时间。`size`、`isEmpty` 等读取计数字段的操作通常也是常数时间。这里的“平均”包含多个前提：

- 元素哈希值能较均匀地分散到桶中。
- `hashCode` 和 `equals` 本身成本可控。
- 集合没有因容量规划失衡形成大量冲突。
- 没有把具体实现的对抗性最坏输入当作普通输入。

若所有元素都返回同一哈希码，哈希表首先会把它们集中到同一桶中。现代实现可能在桶足够大且表容量满足条件时使用树结构，但仍会执行更多比较、维护更多节点状态，也不能恢复到理想的均匀分布成本。正确做法是提供质量合理、计算稳定的哈希码，而不是依赖冲突补救机制。

哈希码也不应为了“绝对唯一”而引入昂贵计算。哈希函数允许碰撞，目标是在可接受成本下提供足够好的分布。包含数组、集合或深层对象图的自动哈希可能成本很高；如果对象频繁参与成员测试，这部分用户代码成本可能超过容器本身。

## 迭代成本与顺序不保证

`HashSet` 不保证迭代顺序。更准确地说，它不承诺顺序长期保持不变，也不承诺顺序与插入、自然排序或哈希码大小一致。同一批元素在某次运行中看起来顺序稳定，只是当前元素、容量、哈希结果和实现共同形成的观察结果。

以下变化都可能改变可观察顺序：

- 新增或删除元素。
- 扩容导致桶重新组织。
- 元素类型或其哈希实现变化。
- JDK 实现变化。
- 运行环境对某些元素哈希策略产生差异。

因此，不能把 `HashSet` 的 `toString()` 输出直接作为要求稳定顺序的协议，也不应让测试断言依赖某次遍历次序。需要展示稳定结果时，应先排序或选择具有所需顺序契约的集合。

```java
List<String> stable = new ArrayList<>(values);
stable.sort(Comparator.naturalOrder());
System.out.println(stable);
```

迭代时间也不是简单的 O(size)。`HashSet` 文档明确指出，遍历成本与集合元素数加底层实例容量之和相关。实现需要跨过桶数组中的空位置，再访问有效节点。因此，一个容量很大但只含少量元素的集合，成员查询可能仍然很快，完整遍历却会扫描大量空桶。

这一点影响容量调优：为避免一次扩容而给小集合设置极大初始容量，可能让之后每次遍历都承担额外成本。若工作负载以遍历为主，`LinkedHashSet` 维护的链式顺序可能让遍历更接近元素数量，但它会为每个条目付出额外链接和维护成本。选择应围绕真实操作比例，而不是只看新增和查询。

`spliterator` 和流同样不赋予 `HashSet` 稳定顺序。它的分割器通常报告 `DISTINCT`，因为元素不重复；但没有 `ORDERED` 特征。并行流可以利用可分割性处理元素，却不应被要求产生插入顺序结果。若最终结果有顺序要求，需要显式排序或从有顺序的数据源开始。

## fail-fast：错误侦测机制，不是并发保证

`HashSet` 的迭代器通常是 fail-fast 的：创建迭代器后，如果集合发生迭代器自身 `remove` 之外的结构性修改，迭代器可能抛出 `ConcurrentModificationException`。结构性修改指改变元素数量或内部结构的操作，例如成功新增、成功删除或清空；重复添加已有元素没有改变集合，通常不构成结构性修改。

```java
Set<String> names = new HashSet<>(List.of("A", "B", "C"));

for (String name : names) {
    if (name.equals("B")) {
        names.remove(name); // 通过集合直接修改，后续迭代可能抛异常
    }
}
```

在单线程遍历中删除当前元素，应使用迭代器的 `remove`：

```java
Iterator<String> iterator = names.iterator();
while (iterator.hasNext()) {
    String name = iterator.next();
    if (name.equals("B")) {
        iterator.remove();
    }
}
```

也可以在适合时使用 `removeIf`，由集合实现管理遍历和删除：

```java
names.removeIf(name -> name.startsWith("temp-"));
```

fail-fast 是尽力而为的错误检测，不能保证每次非法并发修改都抛异常，也不能用是否抛异常证明程序线程安全。修改计数检查没有替代锁、并发容器或不可变快照。程序正确性不能依赖 `ConcurrentModificationException` 必然出现。

异常名称中的 “Concurrent” 也不意味着问题只发生在多线程。单线程中一边使用增强 `for` 的迭代器，一边直接调用集合删除，同样可能触发。反过来，多线程读写即使暂时没有异常，也仍可能违反线程安全要求。

## clone：浅复制集合结构，不复制元素

`HashSet` 实现 `Cloneable`，其 `clone()` 返回集合的浅复制。新集合拥有独立的容器结构，对副本执行新增或删除不会直接改变原集合；但两个集合中的元素引用仍指向相同对象，元素内部状态没有被复制。

```java
HashSet<StringBuilder> original = new HashSet<>();
StringBuilder value = new StringBuilder("A");
original.add(value);

@SuppressWarnings("unchecked")
HashSet<StringBuilder> copied =
        (HashSet<StringBuilder>) original.clone();

copied.add(new StringBuilder("B")); // 不改变 original 的成员数量
value.append("!");                  // 两个集合都能观察到同一个元素对象变化
```

示例中的 `StringBuilder` 使用引用身份相等性，修改内容不会改变它继承的身份哈希语义；它只用于展示浅复制引用。若元素自身按可变内容实现相等性，修改后还会叠加前述哈希稳定性风险。

`clone` 的返回类型是 `Object`，调用者通常需要强制转换。对于公共 API，更明确的复制方式往往是构造器：

```java
Set<String> copied = new HashSet<>(original);
```

构造复制同样是元素层面的浅复制，但意图更清楚，也不要求调用方依赖具体类型的 `clone`。若需要深复制，必须根据元素类型定义如何复制；集合框架无法猜测对象图、共享引用、不可变对象和外部资源应如何处理。

## 序列化：保存逻辑成员，也要控制兼容与边界

`HashSet` 实现 `Serializable`。在常见实现中，内部映射字段是瞬态的，序列化逻辑会写出重建集合所需的容量、负载因子、元素数量和各元素，反序列化时再创建底层结构并逐个恢复成员。具体数据格式和校验逻辑属于 JDK 实现，应以目标版本源码和序列化规范为准。

集合可序列化不代表任意实例都能成功序列化。集合中的每个非空元素也必须可以被所用序列化机制处理；否则会在运行时失败。反序列化恢复的是对象状态，不应假设会保留序列化前某次观察到的迭代顺序，因为 `HashSet` 本来就没有顺序契约。

Java 原生序列化还涉及长期兼容、安全边界和资源消耗。把不可信字节直接反序列化为对象图会扩大攻击面；巨大元素数量或精心构造的数据也可能带来内存和 CPU 压力。跨服务、跨语言或长期存储通常更适合显式、可验证的数据格式。若必须使用原生序列化，应限制输入来源、配置过滤、控制对象规模并明确版本兼容策略。

`clone` 与序列化都不会修复元素相等性问题。一个在序列化前已经因为可变哈希字段而失去正常定位的集合，反序列化逐个重新插入后可能表现不同，因为节点会按当时状态重新定位。这种差异更说明原对象使用方式违反了集合前提，而不是一种可依赖的“修复方案”。

## 集合运算：以修改边界理解 API

`HashSet` 很适合执行并集、交集、差集和包含关系判断。Java 没有为这些运算提供专门的数学运算符，通常通过 `addAll`、`retainAll`、`removeAll` 和 `containsAll` 表达。

### 并集

复制左集合后加入右集合的全部元素，得到并集：

```java
Set<String> union = new HashSet<>(left);
union.addAll(right);
```

`addAll` 会修改接收者，并返回集合是否发生变化。它不会修改参数集合。先复制再运算可以保留两个输入集合，适合函数式边界或输入归调用方所有的场景。

### 交集

复制左集合后只保留右集合也包含的元素：

```java
Set<String> intersection = new HashSet<>(left);
intersection.retainAll(right);
```

### 差集

复制左集合后删除右集合中的元素：

```java
Set<String> difference = new HashSet<>(left);
difference.removeAll(right);
```

差集有方向，`left - right` 与 `right - left` 通常不同。对称差可以先求并集，再移除交集，或者分别计算两个方向差后合并。

### 子集关系

`left.containsAll(right)` 表示左集合是否包含右集合的每个成员，即 `right` 是否是 `left` 的子集。它只查询接收者，不修改任何集合。

批量方法的首要边界是“谁被修改”。直接执行 `left.retainAll(right)` 会破坏性地改变 `left`。如果 `left` 是缓存、字段、参数或多个调用者共享的集合，这种副作用可能远超局部代码。除非所有权明确且原地修改就是目标，否则应先复制接收者。

第二个边界是可修改性。`Set.of(...)`、某些不可修改包装或第三方只读集合不支持结构修改，调用修改方法可能抛出 `UnsupportedOperationException`。接口类型为 `Set` 并不承诺可变；可变性必须由创建方式和 API 文档说明。

第三个边界是自身操作。`set.addAll(set)` 不产生变化；`set.retainAll(set)` 不产生变化；`set.removeAll(set)` 会清空集合。虽然这些结果符合数学语义，但当参数别名意外指向同一实例时，清空结果可能是严重错误。公共方法接收多个集合参数时，应考虑别名关系。

第四个边界是元素相等性。集合运算沿用接收者实现的成员判断规则。对普通 `HashSet`，这意味着依赖 `equals` 与 `hashCode`；与其他 `Set` 实现混合运算时，性能和“相同元素”的判定路径可能受具体实现影响。特别是排序集合的比较器若与 `equals` 不一致，跨实现集合相等和批量运算会出现需要谨慎分析的边界。

## 暴露集合时的所有权与修改视图

类把内部 `HashSet` 直接返回给调用者，相当于把结构修改权一起暴露。调用者可以清空、加入非法业务值或在类执行关键逻辑期间修改集合。更稳妥的返回方式取决于所需语义：

- 需要固定快照：返回不可修改副本。
- 需要随原集合变化的只读观察：返回不可修改视图，并清楚说明它不是快照。
- 需要调用方独立修改：返回新的可变副本。
- 需要受控修改：只暴露领域方法，不直接暴露集合。

```java
final class Registry {
    private final Set<String> ids = new HashSet<>();

    public Set<String> snapshot() {
        return Set.copyOf(ids);
    }

    public boolean register(String id) {
        return ids.add(Objects.requireNonNull(id));
    }
}
```

`Set.copyOf` 的具体空值和重复处理遵循其自身契约，不应把它简单视为任意 `HashSet` 的原样包装。若集合可能包含 `null`，需要先确定 API 希望拒绝、过滤还是用其他表示替代。

参数所有权同样重要。构造器若直接保存调用方传入的可变 `Set` 引用，调用方之后的修改会改变对象内部状态。需要隔离时，应在边界处复制。复制解决的是容器结构别名，不是元素深复制；元素仍可能在外部被修改。

## 并发：HashSet 不提供线程安全访问

`HashSet` 不是线程安全集合。多个线程并发访问时，只要至少一个线程可能进行结构修改，就必须使用外部同步、线程封闭、不可变快照或合适的并发集合。未经同步的并发读写可能导致更新丢失、观察不一致、迭代异常或其他不可靠行为。

“先检查再执行”即使把单个方法换成同步包装，也需要在同一锁内完成：

```java
Set<String> set = Collections.synchronizedSet(new HashSet<>());

synchronized (set) {
    if (!set.contains("job")) {
        set.add("job");
    }
}
```

不过，对集合而言 `add` 本身已经原子表达“此前不存在则加入”，返回值常能消除显式的检查再执行：

```java
boolean firstRegistration = set.add("job");
```

`Collections.synchronizedSet` 用同步包装单次操作，但遍历时仍需要按其契约手动在返回的集合对象上同步，并在整个迭代期间持有锁：

```java
synchronized (set) {
    for (String value : set) {
        consume(value);
    }
}
```

如果主要需求是并发成员集合，可以使用 `ConcurrentHashMap.newKeySet()` 或 `ConcurrentHashMap` 的键集合相关能力。它提供适合并发访问的集合视图，但语义并非“完全等同于加锁的 HashSet”：它不允许 `null`，迭代通常是弱一致的，不抛出普通 fail-fast 迭代器所依赖的异常，也不提供全局快照一致性。

读多写少且集合规模不大时，`CopyOnWriteArraySet` 可能合适。它的读取和迭代基于快照，迭代期间不受后续修改影响；代价是每次写入复制底层数组，成员查询也通常需要线性扫描。它不适合大集合或频繁写入。

若集合在一个线程中构建完成，之后通过安全发布交给其他线程且不再修改，可以把它作为只读状态使用。安全发布可以通过 `final` 字段正确构造、同步、并发容器、易失性引用等方式建立可见性关系。仅仅“大家约定不改”不能弥补不安全发布。

并发容器只保护容器层面的成员结构，不保护元素对象内部状态。多个线程从线程安全集合取出同一个可变对象并修改其字段，仍然需要元素自身的同步策略。若字段还参与相等性，风险会同时扩展到成员定位。

## 可变元素的实践风险

可变元素问题不仅是“contains 可能失败”。它会影响整个集合生命周期：

- 去重失效：修改后再次加入逻辑相等元素，可能形成业务重复。
- 删除失效：清理代码找不到目标，导致对象继续被集合持有。
- 集合相等性变化：作为值对象使用的集合，其 `equals` 和 `hashCode` 会随成员状态变化。
- 外层哈希容器失效：一个可变 `HashSet` 若作为键或元素嵌套在其他哈希容器中，修改成员会改变它的哈希码。
- 并发推理困难：另一个线程可能在查询过程中改变元素状态，使哈希计算与相等性检查基于不同快照。

不可变元素是最简单的防线。普通类可以用 `final` 字段和受控构造表达不可变值；支持记录类的 Java 版本也可以使用记录类减少样板代码。但仍要注意，字段或记录组件若引用可变数组、列表或对象，外层对象并不会自动深度不可变。若这些组件参与相等性和哈希计算，外部修改组件仍可能改变对象的哈希表现。

实体对象常常需要更新状态，不宜为了放进集合而把所有字段冻结。此时应把稳定身份与可变属性分开：相等性只依赖不可变标识，或者改用 `Map<StableId, MutableEntity>`。后者还明确支持按标识替换和查找，比“在 Set 中寻找一个相等对象再修改”更贴合实体管理。

另一个常见错误是用 `hashCode` 本身作为唯一标识。哈希码允许碰撞，也可能因进程、实现或对象状态而变化，不能替代数据库键、协议 ID 或业务唯一编号。`HashSet` 能在碰撞后继续用 `equals` 区分对象，业务若只保存哈希码则丢失了这层确认。

## 与 LinkedHashSet、TreeSet 和 EnumSet 的选择

选择集合实现应从必需语义开始，而不是先选 `HashSet` 再补救。

### HashSet：只需要通用去重与快速成员测试

适合以下条件：

- 不要求迭代顺序。
- 相等性由 `equals` 与 `hashCode` 定义。
- 希望新增、删除、查询具有良好的平均性能。
- 可以接受哈希表容量和节点带来的空间成本。

它通常是通用 `Set` 的默认候选，但“默认”不等于无需检查顺序、并发和元素稳定性。

### LinkedHashSet：需要可预测的插入顺序

`LinkedHashSet` 在哈希表之外维护条目间的链接，通常按插入顺序迭代。重复添加已有元素不会把它移动到末尾。它保留哈希集合的成员判断方式，同时用额外链接和维护成本换取稳定顺序。

当输出、协议、测试或用户可见结果要求插入顺序时，应直接选择 `LinkedHashSet`，而不是依赖 `HashSet` 当前碰巧表现出的顺序。它的遍历通常与元素数量更直接相关，不像 `HashSet` 那样明显受底层容量中的空桶影响，但每个条目占用更多结构。

### TreeSet：需要排序、范围与相邻元素查询

`TreeSet` 基于排序结构，按自然顺序或 `Comparator` 组织元素。新增、删除和查询通常为 O(log n)，并支持 `first`、`last`、`floor`、`ceiling`、范围视图等有序集合操作。

它判定重复主要依据比较结果是否为零，而不是哈希码。比较器与 `equals` 不一致时，集合行为仍可按比较器工作，但可能不完全符合一般 `Set` 相等性直觉。`TreeSet` 对 `null` 的支持也不能按 `HashSet` 推断，通常应视比较规则和具体契约而定。

只是在最终输出时偶尔排序，不一定需要长期使用 `TreeSet`。可以在日常阶段用 `HashSet` 去重，输出时复制并排序。若频繁执行范围查询或始终需要有序视图，则 `TreeSet` 更自然。

### EnumSet：元素类型是单一枚举

元素全部来自同一个枚举类型时，`EnumSet` 通常是更精确的选择。它用紧凑位表示实现成员集合，按枚举常量声明顺序迭代，操作高效且类型语义明确。它不允许 `null`。

```java
enum Permission {
    READ, WRITE, EXECUTE
}

Set<Permission> permissions =
        EnumSet.of(Permission.READ, Permission.WRITE);
```

用 `HashSet<Permission>` 在功能上可以工作，却浪费了枚举取值范围固定、序号稳定这一专用条件。`EnumSet` 还提供 `allOf`、`noneOf`、`complementOf` 等适合标志集合的工厂方法。

### 选择对照

| 需求 | 更合适的候选 | 关键代价或边界 |
| --- | --- | --- |
| 通用去重、快速成员测试、无需顺序 | `HashSet` | 顺序不保证，迭代受容量影响 |
| 保持插入顺序 | `LinkedHashSet` | 额外链接与维护成本 |
| 排序、范围、最近元素查询 | `TreeSet` | O(log n)，依赖一致比较规则 |
| 单一枚举类型的成员集合 | `EnumSet` | 不允许 `null`，仅适用于枚举 |
| 高并发成员集合 | `ConcurrentHashMap.newKeySet()` | 不允许 `null`，弱一致迭代 |
| 极少写、多读且集合较小 | `CopyOnWriteArraySet` | 写入复制，查询通常线性 |

## 常见误区与排查路径

### “HashSet 会自动替换重复对象”

不会。新增相等对象只返回 `false`，已有成员通常保留。需要替换语义时使用以稳定标识为键的 `Map`，或显式先删除再新增。

### “只要 hashCode 不同，元素就一定不同”

不同哈希码确实意味着遵守协议的对象不应相等，但集合最终逻辑仍建立在完整的 `equals`/`hashCode` 契约上。更常见的问题是相等对象错误地产生不同哈希码，导致去重失败。

### “迭代顺序这次没变，以后就能依赖”

不能。未承诺的行为即使在大量测试中稳定，也不是接口保证。需要顺序就显式排序或选择有顺序实现。

### “抛出 ConcurrentModificationException 就说明有多线程”

不一定。单线程遍历时绕过迭代器修改集合也会触发。这个异常首先说明迭代期间检测到不受支持的结构变化。

### “没有抛 ConcurrentModificationException 就线程安全”

同样错误。fail-fast 不是同步机制，只是尽力检测。未同步并发访问的正确性不能由异常是否出现判断。

### “初始容量越大，性能越好”

过小可能频繁扩容，过大则浪费空间并拖慢迭代。容量应根据预计元素数、负载因子和操作分布规划。

### “集合里能看到对象，contains 就一定能找到”

只有对象自插入后保持哈希和相等性稳定时，这个推理才成立。遇到“遍历可见但查询失败”，首先检查参与 `equals`/`hashCode` 的字段是否被修改，其次检查这两个方法是否满足契约。

### “把 Set 包装成不可修改视图就彻底不可变”

不可修改视图只禁止通过该视图进行结构修改；原始集合仍可能变化，元素对象也可能变化。固定快照需要复制，深度不可变还要求元素及其可达状态满足相应约束。

## 如何验证一个 HashSet 使用方案

集合代码的测试不应只覆盖“添加三个字符串再查询”这一条正常路径。真正决定可靠性的，是相等性、修改边界和生命周期能否在边界输入下继续成立。

首先测试元素相等性。至少构造两个不同引用但逻辑相等的对象，确认第一次新增成功、第二次新增失败、集合大小仍为一，并且用任一等价对象都能执行 `contains` 和 `remove`。再构造哈希码相同但不相等的对象，确认冲突不会被误判为重复。若类型有继承层次，还应从父类和子类两个方向调用 `equals`，检查对称性。

其次测试可变状态。若设计声称参与相等性的字段不可变，应由类型结构阻止修改，而不只是依赖注释。若确实允许通过“先删除、再修改、后加入”的流程更新关键字段，则应测试删除发生在旧状态下，并测试异常路径不会让对象既不在旧位置、也没有成功进入新状态。更简单的设计通常是把稳定标识作为映射键。

再次测试批量运算的所有权。为并集、交集和差集准备有重叠、无重叠、完全相同、空集合以及参数别名等情况，同时断言输入集合是否应保持不变。测试只验证结果元素却不验证输入副作用，容易漏掉最危险的修改边界。若 API 接受任意 `Set`，还应覆盖不可修改集合、包含空元素的集合以及不同实现之间的组合。

迭代测试不应断言 `HashSet` 的具体顺序。若业务输出必须稳定，测试应针对“显式排序后的结果”或直接针对具有顺序契约的实现。遍历期间删除元素时，应分别验证迭代器 `remove` 或 `removeIf` 的合法路径；不要把一定抛出 `ConcurrentModificationException` 写成并发正确性的断言，因为快速失败只提供尽力检测。

容量与性能需要基准而不是普通单元测试。基准数据应接近真实元素数量、哈希分布、命中比例和读写比例，并区分构建、随机查询、删除和完整遍历。预热运行时、消费基准结果并使用成熟基准框架，可以减少即时编译、死代码消除和计时噪声造成的误判。比较候选实现时还要保持输入语义一致：不能让 `HashSet` 处理无序结果，却要求 `TreeSet` 额外承担排序后再把两者耗时直接比较。

性能排查可以遵循从证据到原因的顺序。查询慢时先测量 `hashCode`、冲突分布和元素规模；构建慢时检查扩容次数与对象分配；遍历慢时比较集合大小与底层容量；内存长期偏高时检查峰值容量、删除后的保留结构以及元素是否因删除失败而继续存活。直接更换集合实现可能掩盖问题，却不能修复错误的相等性或失控的对象生命周期。

## 设计与使用检查清单

使用 `HashSet` 前，可以沿以下问题确认设计是否闭环：

1. 成员“相同”究竟按引用身份、值内容还是稳定业务标识定义？
2. 类型是否同时正确实现 `equals` 与 `hashCode`？
3. 参与相等性和哈希的字段在元素入集合期间是否稳定？
4. 是否允许 `null`，这个决定是否写进 API 契约？
5. 是否真的不要求插入顺序、排序顺序或可重复输出顺序？
6. 预期规模是否足以需要容量规划，还是默认参数已经合适？
7. 工作负载是否包含大量完整遍历，过大容量会不会成为持续成本？
8. 批量集合运算能否修改输入，集合所有权是否清楚？
9. 是否会跨线程读写，应使用同步、不可变快照还是并发集合？
10. 返回集合时需要的是实时只读视图、不可变快照还是独立可变副本？
11. 是否把 `HashSet` 作为另一个哈希容器的键或元素，并确保其不再变化？
12. 是否通过真实数据和基准验证了性能判断，而不是只依赖平均 O(1) 的标签？

## 总结

`HashSet` 用 `Set` 契约表达唯一成员，用常见的 `HashMap` 键存储实现快速定位。占位值让映射结构只承担成员存在性；`add`、`contains` 和 `remove` 都依赖同一套哈希与相等性协议。其高效的前提不是“用了哈希表就自动成立”，而是元素哈希分布合理、`equals`/`hashCode` 正确且参与定位的状态保持稳定。

容量和负载因子决定空间、冲突、扩容与迭代之间的取舍。成员操作具有良好的平均成本，但完整遍历还会受底层容量影响；迭代顺序没有保证，fail-fast 只用于尽力发现非法结构修改。`clone` 复制容器结构而共享元素，序列化恢复逻辑成员但不提供顺序保证，也不应忽略安全和兼容边界。

在实践中，`HashSet` 最适合“不关心顺序，只关心某元素是否属于集合”的通用场景。需要插入顺序时选择 `LinkedHashSet`，需要排序和范围操作时选择 `TreeSet`，枚举成员集合优先考虑 `EnumSet`，并发场景则选择明确的同步策略或并发集合。容器选择只有同时匹配成员语义、修改边界、迭代要求和线程模型，才能长期保持正确且易于维护。

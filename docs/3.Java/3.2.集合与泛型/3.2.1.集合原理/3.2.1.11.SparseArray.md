# 3.2.1.11 SparseArray

## 本文讨论的对象与边界

本文把 `SparseArray` 作为一种通用数据结构名称，表示“以整数为键、只保存实际出现键值对的稀疏数组映射”。它不是 Java 标准库中的类型，文中的接口和代码只是为了讲解数据结构而设计的教学实现，不能据此推断任何现成类库的 API 契约。

本文只讨论抽象数据结构及其纯 Java 教学实现，不对应某个现成类库中的具体类型。这里关心的问题是：当键来自较大的整数空间，而实际出现的键较少时，如何用排序的 `int` 键数组和值数组实现一个紧凑映射；这种结构为什么能节省一部分对象与结构开销；二分查找、插入移动、删除标记、压缩和容量增长又怎样共同决定它的性能边界。

“稀疏”描述的是键空间与实际键集合之间的关系，而不是容器内部必须存在大量空槽。假设合法键位于 `0` 到 `1_000_000_000`，实际只出现 80 个键。若直接创建一个按键下标访问的数组，绝大部分位置都为空，空间浪费不可接受；稀疏映射只记录这 80 个键及对应值，存储成本主要随元素数增长，而不是随最大键值增长。

不过，仅凭“键很稀疏”还不足以选择本结构。哈希表同样只保存出现的键，并且通常提供平均常数时间查找。排序数组方案真正有吸引力的场景通常还具有以下特征：

- 键类型固定为 `int`，希望避免 `Integer` 装箱以及每条记录的节点对象。
- 元素数量处于小到中等规模，查找和遍历多，随机插入、删除相对少。
- 希望键始终有序，可以按升序遍历、做范围扫描或按索引读取。
- 数据可批量生成，或者写入阶段结束后进入长期只读阶段。
- 更看重紧凑内存、顺序访问和可预测布局，而不是把单次更新成本压到最低。

如果元素数量很大且更新频繁，或者业务要求高并发写入、稳定的对数级更新、平均常数时间查询，那么哈希表、平衡树或其他专用结构通常更合适。SparseArray 不是“整数键映射的普遍最优解”，而是一组访问模式与内存目标之间的具体折中。

## 从问题模型判断是否真的稀疏

### 键空间大小不等于元素规模

设整数键的理论取值范围大小为 \(U\)，实际存储的不同键数量为 \(n\)。可以用占用率

\[
\rho = \frac{n}{U}
\]

描述直接索引数组的密集程度。当 \(\rho\) 很低时，长度为 \(U\) 的直接索引数组会为大量从未使用的位置付费。稀疏映射将空间从与 \(U\) 相关，转为主要与 \(n\) 相关。

但这个公式只适合有限、明确且从零附近开始的键域。Java 的 `int` 包含负数，完整取值空间极大，不能真的据此分配直接索引数组。工程中更常见的判断方式是比较“最大键跨度”与“有效元素数”：

\[
span = maxKey - minKey + 1
\]

若 `span` 接近 `n`，可以通过偏移量 `key - minKey` 使用直接索引数组；若 `span` 远大于 `n`，紧凑稀疏映射更合理。例如键为 `10000、10001、10002、10003`，绝对值虽大，但跨度只有 4，偏移数组非常高效；键为 `3、9000、700000`，跨度巨大而只有 3 个元素，才是典型稀疏分布。

因此，“最大键很大”与“键稀疏”不是同一个判断。必须同时看最小键、最大键、元素数和键的聚集程度。若键集中在少数区段，还可以考虑分段数组、页表式结构或多个小型直接索引数组，而不必在单一 SparseArray 与哈希表之间二选一。

### 稀疏度没有通用阈值

不存在一个脱离环境的比例，例如“占用率低于 10% 就必须使用稀疏映射”。直接索引数组每个槽位可能是一个引用、一个基本类型值，或者“值加存在位”；元素值大小、引用宽度、对象头、对齐方式、运行时压缩引用策略都会改变盈亏点。另一方面，SparseArray 至少需要一份 `int[]` 键数组、一份值数组、数组对象本身以及预留容量。

真正的选择应从候选结构的总成本出发：

- 直接索引数组的成本约随键跨度增长，但访问最简单。
- SparseArray 的成本约随容量增长，容量通常略大于有效元素数。
- 哈希表除键和值外，还可能需要桶数组、装箱键、节点或开放寻址空槽。
- 树结构通常需要节点对象以及多个链接字段，但提供稳定有序操作。

当数据规模足以影响性能或内存时，应在目标运行时、真实键分布和真实操作比例下做基准测试。仅用大 O 或经验阈值无法代替测量。

### 访问模式比静态稀疏度更重要

两个同样保存 1000 个键的映射，可能需要完全不同的结构。第一个映射一次构建后执行千万次查询和顺序遍历；第二个映射持续随机插入和删除。前者能充分利用排序数组紧凑、局部性好的优势，后者会反复支付移动与压缩成本。

评估时至少应记录以下维度：

| 维度 | 更有利于 SparseArray 的特征 | 风险信号 |
| --- | --- | --- |
| 元素规模 | 小到中等，容量可控 | 持续增长且缺少上界 |
| 查询比例 | 查询、遍历显著多于更新 | 高频随机写 |
| 插入位置 | 多数键递增，可追加到尾部 | 多数插入发生在前部或中部 |
| 删除模式 | 删除可延迟整理或集中发生 | 删除后必须立即紧凑且频繁 |
| 顺序需求 | 需要键有序、范围扫描 | 完全不关心顺序 |
| 构建方式 | 可排序后批量构建 | 边接收边随机更新 |
| 并发方式 | 单线程拥有或构建后只读 | 多线程并发修改 |

这里的核心不是给结构贴上“快”或“慢”的标签，而是判断昂贵操作是否落在高频路径上。

## 排序键数组与值数组

### 并行数组表示

一个典型实现维护以下状态：

```text
keys   = [ 4,  9, 21, 38, ...]
values = [ A,  B,  C,  D, ...]
size   = 4
```

`keys[i]` 与 `values[i]` 共同组成一条映射记录。键数组的有效部分按严格升序排列：

\[
keys[0] < keys[1] < \dots < keys[size - 1]
\]

严格升序同时表达了两件事：键有序，并且不存在重复键。若写入一个已有键，只替换同一位置的值；若写入新键，则在保持顺序的位置插入。

使用并行数组而不是“条目对象数组”，可以省去每条记录对应的对象身份和对象头。键保存在 `int[]` 中，不需要把基本类型转换成 `Integer`。值仍然是引用时，`Object[]` 或泛型擦除后的引用数组保存的是对象引用，不会复制对象本身。

并行数组的代价是实现必须始终维护对齐不变量。插入、删除和压缩时，键和值必须以完全相同的区间和距离移动。任何一个数组少移动一格，都会让键对应到错误的值，而且错误可能直到后续查询才暴露。因此这类结构看似字段少，修改路径却需要严格测试。

### 容量与逻辑大小

数组长度是容量 `capacity`，有效映射数量是逻辑大小 `size`。两者满足：

\[
0 \le size \le capacity
\]

预留容量使新增元素不必每次都重新分配数组，但也意味着未使用尾部槽位占用空间。若实现采用删除标记，还要区分三个量：

- `capacity`：物理数组可容纳的槽位数。
- `usedSlots`：包含有效值或删除标记的已占用前缀长度。
- `liveSize`：当前真实可见的映射数量。

有的实现只保存 `size`，在需要报告真实数量时先压缩；有的实现同时维护已用槽位和有效元素计数。前者字段更少，但读取 `size()` 也可能触发线性整理；后者查询更直接，却增加每条修改路径维护计数的责任。不能只看方法名称推断其复杂度，必须看契约是否允许惰性工作在某个看似只读的方法中发生。

### 有序带来的额外能力

排序并不只是二分查找的前提。它还自然提供：

- 按键升序遍历，不需要额外排序。
- 找到第一个大于等于目标键的位置，即下界查询。
- 在两个边界键之间做连续范围扫描。
- 比较两个映射的键集合时，可以使用双指针线性合并。
- 批量构建时，可以检测重复键并顺序写入。

这些能力是哈希表通常不直接提供的。若业务确实需要它们，维护排序的写入成本可能是值得的；若业务完全不使用顺序，排序只是额外约束。

## 二分查找不仅返回“找到或没找到”

### 返回命中位置或插入点

对有序 `int[]` 查找目标键，最直接的结果有两类：

1. 找到，返回非负索引 `i`，满足 `keys[i] == key`。
2. 未找到，返回编码后的插入点，使调用方知道新键应放在哪里。

Java 标准库的若干二分查找方法采用 `-(insertionPoint + 1)` 编码未命中结果。加一是为了区分插入点 0 与命中索引 0；否则 `-0` 仍是 0。解码公式为：

```java
int insertionPoint = ~result;
```

这里利用按位取反与 `-(x + 1)` 等价。也可以写成 `-result - 1`，后者对不熟悉位运算的读者更直观。

```java
private static int binarySearch(int[] keys, int length, int target) {
    int low = 0;
    int high = length - 1;

    while (low <= high) {
        int mid = (low + high) >>> 1;
        int value = keys[mid];
        if (value < target) {
            low = mid + 1;
        } else if (value > target) {
            high = mid - 1;
        } else {
            return mid;
        }
    }
    return ~low;
}
```

循环结束时，`low` 是第一个大于目标键的位置；如果所有键都小于目标键，`low == length`，表示尾部追加。一次二分查找因此同时服务于读取、替换和插入，不需要未命中后再线性寻找位置。

### 复杂度成立的条件

二分查找需要 \(O(\log n)\) 次比较，但这个结论有明确前提：

- 有效键区间已经按同一整数顺序排序。
- 查找范围长度准确，没有把未初始化尾部当作有效键。
- 若存在删除标记，标记只出现在值数组，键数组仍保持原有顺序。
- 比较操作是常数成本；对 `int` 比较通常满足这一点。

如果每次查询前都先压缩删除标记，那么一次公开查询可能是 \(O(n)\)，尽管内部定位步骤仍是 \(O(\log n)\)。复杂度应描述完整方法路径，而不是只描述其中最醒目的算法。

### 下界与精确查找

精确查找关心“是否存在等于目标的键”，下界查找关心“第一个不小于目标的键在哪里”。在严格升序数组中，上面的未命中插入点就是下界位置。于是范围 `[fromKey, toKey)` 可以通过两次二分查找确定起止索引，再顺序遍历中间区间，总成本为：

\[
O(\log n + k)
\]

其中 \(k\) 是范围内返回的元素数。哈希表若没有额外索引，通常需要扫描全部元素才能回答同类范围问题。

## 插入：查找便宜，移动可能昂贵

### 替换与新增是两条路径

写入 `put(key, value)` 首先二分查找：

- 若命中已有键，直接替换 `values[index]`，键数组不变，通常为 \(O(\log n)\) 查找加 \(O(1)\) 替换。
- 若未命中，解码插入点，确保容量足够，再把插入点后的键和值整体右移一格，最后写入新记录。

中间插入移动的元素数量为 `size - insertionPoint`。最坏情况下，新键小于所有已有键，需要移动全部元素，时间复杂度为 \(O(n)\)。尾部追加不需要移动，若容量足够则定位后写入即可。

```text
插入前 keys: [3, 8, 20, 31, _]
插入键 12，插入点为 2

右移后 keys: [3, 8, _, 20, 31]
写入后 keys: [3, 8, 12, 20, 31]
```

值数组必须执行完全相同的右移。Java 中可以使用 `System.arraycopy` 完成重叠区间复制。其时间复杂度仍是线性的，但连续内存复制通常比手写逐项循环具有更低常数。不能因为底层复制经过优化，就把中间插入描述成常数时间。

### 递增键追加是重要特例

如果输入键严格递增，可以在尾部直接追加，避免每次二分与移动。追加方法应明确要求新键大于当前最大键；若不满足，可以退回普通 `put`，也可以抛出异常，具体取决于 API 契约。

递增追加把批量构建从“每次二分并可能移动”变为顺序写入。只要扩容采用几何增长，连续追加的摊还时间可接近 \(O(1)\) 每项，构建总成本为 \(O(n)\)。这是排序数组结构最值得利用的输入特征之一。

若键是非递减而可能重复，批量构建还需要定义重复策略：保留第一个值、保留最后一个值、拒绝重复，或通过合并函数生成新值。重复策略属于映射语义，不能由排序过程偶然决定。

### 插入点附近的删除槽位

采用惰性删除时，二分查找可能发现目标插入点或相邻位置已经有删除标记。实现可以复用该槽位，减少移动和压缩：

- 若查找命中相同键，但对应值已标记删除，可直接恢复该槽位。
- 若未命中且插入点处是删除槽位，可以在不破坏键序的条件下覆盖。
- 若相邻删除槽位不能容纳新键而保持顺序，仍需移动或先压缩。

复用逻辑能优化部分更新模式，但分支会显著增加。教学实现可以选择更容易证明正确的策略：命中已删除的同键时原位恢复；其他新增在空间不足或空洞较多时先统一压缩，再按普通插入处理。

## 删除标记与压缩

### 为什么不立即移动

从排序数组中删除索引 `i`，最直接的方法是把后续区间左移一格并清理尾部。这能始终保持紧凑，但每次删除最坏都要移动 \(O(n)\) 个元素。若短时间内连续删除多个位置，立即移动会反复复制相同区间。

惰性删除把 `values[i]` 替换为一个内部专用标记对象，键暂时保留。删除操作在已经找到索引后只做常数次赋值，并设置“存在垃圾槽位”的状态。后续在确有需要时一次性压缩所有标记：

```text
keys   = [ 2,  7, 13, 18, 25]
values = [ A,  X,  C,  X,  E]   X 表示删除标记

压缩后：
keys   = [ 2, 13, 25, _, _]
values = [ A,  C,  E, null, null]
```

若连续删除 `d` 项后只压缩一次，总移动量通常小于每次删除都立即移动。代价是压缩前物理槽位数大于有效元素数，某些操作必须跳过标记或先触发压缩。

### 删除标记必须与合法值区分

若映射允许保存 `null`，就不能用 `null` 表示已删除。实现需要一个调用方不可能获得其引用的私有哨兵对象：

```java
private static final Object DELETED = new Object();
```

判断删除状态必须使用引用比较 `value == DELETED`，不能调用 `equals`。哨兵对象只表达内部槽位状态，绝不能作为用户值返回。

若实现禁止 `null`，理论上可以用 `null` 作为空洞标记，但这会把数据结构内部便利变成公开 API 限制。是否允许 `null` 应先由语义需求决定，再选择内部编码；不应为了少一个哨兵对象而含糊地改变契约。

### 何时压缩

常见触发点包括：

- 插入时容量已满，但数组中存在删除槽位。先压缩可能腾出足够空间，从而避免扩容。
- 按索引访问或顺序遍历前，需要让逻辑索引与物理索引一致。
- `size()` 承诺返回有效元素数，而实现没有单独维护有效计数。
- 删除槽位比例超过阈值，继续保留会增加扫描和内存滞留。
- 显式调用 `compact()`，由使用者在批量删除后控制整理时机。

压缩本身是一次稳定过滤：从左到右扫描已用区间，把未删除记录写到下一个连续位置。相对键序不会改变。完成后应将尾部值引用清为 `null`，使不再使用的对象可以被垃圾回收；键数组尾部是否清零不影响对象回收，但清零有助于调试和避免误读。

### 惰性删除改变了复杂度语义

采用标记后，`remove(key)` 的定位仍需 \(O(\log n)\)，命中后的标记为 \(O(1)\)。然而压缩成本 \(O(n)\) 被推迟到了未来。不能据此宣称删除“就是 \(O(\log n)\)”而忽略债务。更准确的表述是：

- 单次删除通常只定位并标记，最坏不含压缩时为 \(O(\log n)\)。
- 某个后续操作可能承担 \(O(n)\) 压缩。
- 一组操作的摊还成本取决于压缩频率、删除分布和触发策略。

惰性工作提高了批处理效率，却可能造成延迟尖峰。如果系统要求每次读取延迟都稳定，自动在读取路径压缩未必合适；可以改成立即删除、显式压缩，或使用更新成本更稳定的结构。

## 容量增长与收缩

### 几何增长避免平方级复制

当紧凑后的 `size == capacity` 时，需要分配更大的键和值数组并复制已有内容。若容量每次只增加 1，连续插入 \(n\) 项会复制：

\[
1 + 2 + 3 + \dots + (n - 1) = O(n^2)
\]

因此动态数组通常采用几何增长，例如约 1.5 倍或 2 倍。这样扩容次数为对数级，连续尾部追加的总复制量为 \(O(n)\)，单次追加获得摊还 \(O(1)\) 的扩容成本。

增长因子越大，扩容次数和复制次数越少，但空闲容量更多；增长因子越小，空间更紧凑，但扩容更频繁。没有适用于所有负载的唯一比例。教学实现可以选简单、容易验证的策略，例如小容量至少增长到 4，之后增长为旧容量的 1.5 倍。

计算新容量时还要防止整数溢出。若请求容量已经接近数组允许的上限，`old + old / 2` 可能溢出为负数。健壮实现应检查最小所需容量、计算结果和运行时数组上限，并在无法满足时抛出清晰异常，而不是让负容量错误掩盖真正原因。

### 初始容量是时间与空间的交换

若能够估计最终元素数，预先设置接近该值的容量可避免多次扩容。估计过大则会长期保留空闲数组槽位。对于一次构建后长期存在的映射，可以先按增长策略构建，再通过 `trimToSize()` 压缩到精确容量；是否值得取决于对象数量、生命周期和内存压力。

初始容量不应根据最大键计算。SparseArray 的容量表达“预计保存多少条记录”，不是“最大键是多少”。把最大键误作容量会重新引入直接索引数组的空间问题。

### 自动收缩通常要谨慎

删除后立即缩小数组会造成额外复制，并可能在元素数量围绕阈值波动时产生扩容、收缩抖动。更常见的策略是不自动收缩，或者只有在空闲比例很高且容量超过某个下限时才收缩。显式 `trimToSize()` 能把时机交给调用方，适合“批量更新结束，接下来长期只读”的阶段转换。

删除标记压缩与容量收缩是两个不同动作：

- 压缩消除逻辑空洞，可以仍使用原数组。
- 收缩重新分配更短数组，减少预留空间。

把二者区分开，才能准确分析复制次数与内存收益。

## 按键查找与按索引访问不是一回事

### 两套坐标系统

按键查找使用业务键，例如 `get(38)`，需要二分查找；按索引访问使用排序后的序号，例如 `keyAt(3)`、`valueAt(3)`，直接读取第 4 条有效记录。二者的参数虽然都是整数，语义完全不同。

索引不是稳定标识。插入一个更小的键会让后续记录的索引全部加一；删除和压缩会让后续索引减一。调用方不能长期保存索引并把它当作键，也不能在结构修改后假设旧索引仍指向同一记录。

按索引访问通常要求数组已紧凑，否则物理索引可能落在删除标记上。实现可以在 `keyAt`、`valueAt` 前自动压缩，也可以维护从逻辑索引到物理索引的扫描，但后者会使随机索引访问变慢。自动压缩让访问语义简单，却意味着第一次索引读取可能是 \(O(n)\)。

### `indexOfKey` 与 `indexOfValue`

`indexOfKey(key)` 可以利用键排序二分查找，复杂度为 \(O(\log n)\)。`indexOfValue(value)` 没有按值建立索引，只能线性扫描，复杂度为 \(O(n)\)。名称都含 `indexOf`，并不意味着成本相同。

值比较还必须定义使用引用相等还是逻辑相等：

- 引用相等 `==` 查找同一个对象实例。
- 逻辑相等 `Objects.equals` 查找语义相等的值，并自然处理 `null`。

两种定义都可能合理，但必须写入契约。对可变值使用逻辑相等时，值对象在存储后改变其相等字段不会破坏键定位，因为定位只依赖整数键；它只会影响按值搜索结果。

### 遍历期间修改

一个最简单的遍历形式是先取得 `size()`，然后按递增索引读取。但如果循环中删除元素并触发压缩，索引会移动，容易跳过下一项。安全策略包括：

- 从尾到头按索引删除，使尚未访问的较小索引不受影响。
- 先收集待删除键，遍历结束后统一删除。
- 提供专门迭代器，由迭代器维护当前位置和修改检测。
- 明确禁止遍历期间结构修改。

教学实现若没有实现修改计数和 fail-fast 迭代器，就不应暗示普通索引循环具有并发修改检测能力。

## `null`、缺失与返回值契约

### 允许 `null` 时的二义性

若键 `7` 映射到 `null`，那么 `get(7)` 返回 `null`；若键 `7` 不存在，`get(7)` 也可能返回 `null`。单凭返回值无法区分“存在且值为空”和“缺失”。

常见解决方式有：

- 提供 `containsKey(key)`，调用方在需要时额外判断。
- 提供 `indexOfKey(key)`，非负结果表示存在。
- 提供带默认值的 `getOrDefault(key, defaultValue)`。
- 返回显式查询结果对象，包含 `found` 与 `value`。
- 禁止 `null` 值，让 `null` 专门表示缺失。

`Optional<V>` 不总能完整表达“允许 `null` 的已存在映射”，因为 `Optional` 本身通常把空值解释为缺失。若必须区分三态，应使用独立布尔标记或结果类型。

### `containsKey` 不应依赖值

当 `null` 是合法值时，以下实现是错误的：

```java
return get(key) != null;
```

正确判断应基于键二分查找和删除状态。键存在但值为 `null` 时仍应返回 `true`。这也是将“定位”和“取值”分开的原因。

### 默认值也可能是合法值

`getOrDefault` 方便但不能证明键存在，因为默认值可能恰好与已存值相同。它适合“不关心缺失与特定值的区别”的消费逻辑，不适合需要精确存在性判断的逻辑。API 应提供足够能力，让调用方不用依赖魔法值，例如用 `-1`、空字符串或特殊对象猜测缺失。

## 一个纯 Java 教学实现

下面的 `IntSparseMap<V>` 用于展示本文机制。它不是 Java 标准库 API，也没有覆盖序列化、迭代器、视图、并发控制、超大数组上限等生产级需求。实现允许 `null` 值，使用私有哨兵表示删除，按索引访问前会压缩。

```java
import java.util.Arrays;
import java.util.Objects;

public final class IntSparseMap<V> {
    private static final Object DELETED = new Object();

    private int[] keys;
    private Object[] values;
    private int usedSlots;
    private int liveSize;
    private boolean hasGarbage;

    public IntSparseMap() {
        this(4);
    }

    public IntSparseMap(int initialCapacity) {
        if (initialCapacity < 0) {
            throw new IllegalArgumentException("negative capacity");
        }
        keys = new int[initialCapacity];
        values = new Object[initialCapacity];
    }

    public int size() {
        return liveSize;
    }

    public boolean containsKey(int key) {
        int index = binarySearch(keys, usedSlots, key);
        return index >= 0 && values[index] != DELETED;
    }

    public V get(int key) {
        return getOrDefault(key, null);
    }

    @SuppressWarnings("unchecked")
    public V getOrDefault(int key, V defaultValue) {
        int index = binarySearch(keys, usedSlots, key);
        if (index < 0 || values[index] == DELETED) {
            return defaultValue;
        }
        return (V) values[index];
    }

    @SuppressWarnings("unchecked")
    public V put(int key, V value) {
        int result = binarySearch(keys, usedSlots, key);
        if (result >= 0) {
            Object previous = values[result];
            values[result] = value;
            if (previous == DELETED) {
                liveSize++;
                hasGarbage = containsDeletedSlot();
                return null;
            }
            return (V) previous;
        }

        int insertionPoint = ~result;
        if (usedSlots == keys.length && hasGarbage) {
            compact();
            result = binarySearch(keys, usedSlots, key);
            insertionPoint = ~result;
        }

        ensureCapacity(usedSlots + 1);
        int moved = usedSlots - insertionPoint;
        if (moved > 0) {
            System.arraycopy(keys, insertionPoint, keys,
                    insertionPoint + 1, moved);
            System.arraycopy(values, insertionPoint, values,
                    insertionPoint + 1, moved);
        }
        keys[insertionPoint] = key;
        values[insertionPoint] = value;
        usedSlots++;
        liveSize++;
        return null;
    }

    public void append(int key, V value) {
        if (usedSlots != 0 && key <= keys[usedSlots - 1]) {
            put(key, value);
            return;
        }
        ensureCapacity(usedSlots + 1);
        keys[usedSlots] = key;
        values[usedSlots] = value;
        usedSlots++;
        liveSize++;
    }

    public boolean remove(int key) {
        int index = binarySearch(keys, usedSlots, key);
        if (index < 0 || values[index] == DELETED) {
            return false;
        }
        values[index] = DELETED;
        liveSize--;
        hasGarbage = true;
        return true;
    }

    public int keyAt(int index) {
        checkLogicalIndex(index);
        if (hasGarbage) {
            compact();
        }
        return keys[index];
    }

    @SuppressWarnings("unchecked")
    public V valueAt(int index) {
        checkLogicalIndex(index);
        if (hasGarbage) {
            compact();
        }
        return (V) values[index];
    }

    public int indexOfKey(int key) {
        if (hasGarbage) {
            compact();
        }
        int index = binarySearch(keys, usedSlots, key);
        return index >= 0 ? index : -1;
    }

    public int indexOfValue(V value) {
        if (hasGarbage) {
            compact();
        }
        for (int i = 0; i < usedSlots; i++) {
            if (Objects.equals(values[i], value)) {
                return i;
            }
        }
        return -1;
    }

    public void clear() {
        Arrays.fill(values, 0, usedSlots, null);
        usedSlots = 0;
        liveSize = 0;
        hasGarbage = false;
    }

    public void compact() {
        if (!hasGarbage) {
            return;
        }
        int target = 0;
        for (int source = 0; source < usedSlots; source++) {
            Object value = values[source];
            if (value == DELETED) {
                continue;
            }
            if (source != target) {
                keys[target] = keys[source];
                values[target] = value;
            }
            target++;
        }
        Arrays.fill(values, target, usedSlots, null);
        usedSlots = target;
        liveSize = target;
        hasGarbage = false;
    }

    public void trimToSize() {
        compact();
        if (usedSlots != keys.length) {
            keys = Arrays.copyOf(keys, usedSlots);
            values = Arrays.copyOf(values, usedSlots);
        }
    }

    private void ensureCapacity(int minimumCapacity) {
        if (minimumCapacity <= keys.length) {
            return;
        }
        int oldCapacity = keys.length;
        int grown = oldCapacity + Math.max(1, oldCapacity >>> 1);
        int newCapacity = Math.max(minimumCapacity, Math.max(4, grown));
        keys = Arrays.copyOf(keys, newCapacity);
        values = Arrays.copyOf(values, newCapacity);
    }

    private void checkLogicalIndex(int index) {
        if (index < 0 || index >= liveSize) {
            throw new IndexOutOfBoundsException(
                    "index=" + index + ", size=" + liveSize);
        }
    }

    private boolean containsDeletedSlot() {
        for (int i = 0; i < usedSlots; i++) {
            if (values[i] == DELETED) {
                return true;
            }
        }
        return false;
    }

    private static int binarySearch(int[] array, int length, int key) {
        int low = 0;
        int high = length - 1;
        while (low <= high) {
            int middle = (low + high) >>> 1;
            int middleValue = array[middle];
            if (middleValue < key) {
                low = middle + 1;
            } else if (middleValue > key) {
                high = middle - 1;
            } else {
                return middle;
            }
        }
        return ~low;
    }
}
```

这个实现刻意展示了几个容易被忽略的契约：

第一，`size()` 使用独立的 `liveSize`，因此不会为了计数触发压缩。第二，按键读取可以在存在空洞时继续二分，因为删除只修改值，键顺序没有改变。第三，按索引访问先压缩，使逻辑索引等于物理索引。第四，`put` 返回 `null` 既可能表示新增，也可能表示旧值本来就是 `null`，若调用方需要区分，应在写入前使用 `containsKey`，或改造返回结果类型。

其中 `containsDeletedSlot()` 是为了在恢复已删除同键后重新计算垃圾状态，写法偏重可读性，最坏会线性扫描。生产实现可以维护删除计数，使该更新为常数时间。这个例子说明：增加一个状态字段可以降低某条路径成本，但也会扩大不变量和测试面。

`append` 在键不递增时退回 `put`，因此语义宽松。另一种设计是直接抛出 `IllegalArgumentException`，让批量构建错误尽早暴露。两者没有普遍优劣，关键是方法名和文档是否让调用方清楚知道违约行为。

### 实现需要验证的不变量

对这类容器，测试不应只验证几个返回值，还应在随机操作后检查内部不变量：

1. `0 <= liveSize <= usedSlots <= keys.length == values.length`。
2. `keys[0..usedSlots)` 严格升序。
3. `liveSize` 等于非 `DELETED` 值槽数量。
4. `hasGarbage == false` 时，有效区间不存在 `DELETED`。
5. 压缩前后所有可见键值对相同，且顺序相同。
6. 扩容前后所有可见键值对相同。
7. `null` 值与缺失键可以通过 `containsKey` 区分。

可以使用一个 `TreeMap<Integer,V>` 作为测试参考模型，随机执行写入、删除和查询，并在每一步比较键序与结果。参考模型只用于验证语义，不代表两者具有相同的性能或内部结构。

## 复杂度总表与隐藏条件

以下复杂度令 \(n\) 为物理已用槽位数，\(k\) 为输出范围内的元素数：

| 操作 | 常见成本 | 可能的隐藏成本 |
| --- | --- | --- |
| `get(key)` | \(O(\log n)\) | 若实现规定读取前压缩，可能为 \(O(n)\) |
| `containsKey(key)` | \(O(\log n)\) | 同样取决于是否压缩 |
| 替换已有值 | \(O(\log n)\) | 恢复删除槽位可能维护额外状态 |
| 插入新键 | \(O(\log n + n)\)，最坏 \(O(n)\) | 扩容复制或先压缩 |
| 递增尾部追加 | 容量足够时 \(O(1)\) | 几何扩容下摊还 \(O(1)\) |
| 惰性删除 | \(O(\log n)\) 定位后标记 | 压缩成本推迟到后续操作 |
| 立即删除 | \(O(n)\) 最坏移动 | 始终保持紧凑 |
| `keyAt(i)` / `valueAt(i)` | 紧凑时 \(O(1)\) | 首次访问可能触发 \(O(n)\) 压缩 |
| `indexOfKey` | \(O(\log n)\) | 若需逻辑索引，可能先压缩 |
| `indexOfValue` | \(O(n)\) | 值的 `equals` 成本也参与总成本 |
| 全量有序遍历 | \(O(n)\) | 空洞未压缩时需跳过标记 |
| 范围查找 | \(O(\log n + k)\) | 结果复制还需 \(O(k)\) 空间 |
| `compact()` | \(O(n)\) | 会写数组并清理尾部引用 |
| `trimToSize()` | \(O(n)\) | 通常包含压缩和重新分配 |

大 O 只说明规模趋势。小数组上的二分查找包含分支，哈希表查询包含哈希、寻址和可能的装箱，树查询包含节点跳转；实际快慢由规模、运行时优化和访问模式共同决定。

还应区分最坏复杂度、平均复杂度和摊还复杂度。几何扩容让一系列追加具有良好摊还成本，不代表每次追加都不会线性复制；惰性删除让多数删除很轻，不代表所有后续查询都保持对数成本。对延迟敏感系统，最坏单次停顿可能比长期平均吞吐更重要。

## 内存模型：节省了什么，又保留了什么

### 避免装箱和条目节点

使用 `int[]` 保存键，最直接的收益是每个键占一个基本类型槽位，不需要为键创建 `Integer` 对象。并行数组也不要求每个映射条目都是独立节点。相比典型节点式映射，它可能减少：

- 键装箱对象及其对象头和对齐空间。
- 每条记录的节点对象头。
- 节点中的哈希缓存、链接指针、父子指针或颜色标记等结构字段。
- 由大量小对象带来的分配和垃圾回收跟踪成本。

但“每个键一定节省多少字节”不能脱离具体 JVM、对象布局、引用压缩和实现方式给出固定答案。某些装箱值可能来自缓存，某些哈希表采用开放寻址而非节点，某些运行时引用宽度不同。本文只能说明结构差异，精确数字应通过对象布局工具和目标环境测量。

### 双数组也有固定成本

SparseArray 至少持有两个数组对象和容器对象。容量大于元素数时，尾部空闲槽位仍占空间；惰性删除期间，已删除键仍保留，值槽中的哨兵也占一个引用位置。小到只有一两个元素时，固定数组成本未必比其他方案低。

值对象没有被“压缩”进数组。若 `V` 是普通对象，值数组只保存引用，对象本身仍独立分配。结构优化主要针对键和条目组织方式，不会自动降低大型值对象的内存。

### 删除后必须清理引用

压缩或立即删除后，逻辑尾部可能仍残留旧值引用。如果不把这些槽位设为 `null`，即使元素从映射语义上已经删除，只要容器仍可达，垃圾回收器就可能认为旧值仍可达。这种问题常被称为过期引用滞留。

删除标记路径若将值替换成唯一哨兵，原值引用当场被解除；压缩时还需清理移动后尾部的重复引用。键是基本类型，不会保留对象，但清理键尾部可提高诊断可读性。

## CPU 局部性与真实运行成本

数组连续存放，按索引遍历时访问模式规则。处理器和运行时更容易预取后续位置，二分查找虽然会跳跃，但所有键仍集中在一块连续内存中。与需要追踪多个节点引用的树或链式结构相比，数组通常具有更好的空间局部性。

插入移动也利用连续布局。`System.arraycopy` 可以成块复制基本类型或引用范围，常数成本可能低于逐节点分配和链接。于是“小规模数组中移动几十个元素”不一定比理论上更新更便宜的节点结构慢。复杂度决定增长趋势，局部性和对象数量决定很大一部分常数。

不过，当数组很大时，二分查找的访问位置跨越多个缓存行，分支预测也未必理想；随机中间插入会写入长区间，并增加内存带宽压力。局部性优势不能消除线性移动。随着 \(n\) 增长，更新成本最终会成为主要限制。

评测时应避免只循环查询同一批极小数据，因为它们可能长期停留在高速缓存中，不能代表真实负载。基准数据应覆盖实际规模、命中率、键分布、插入位置和读写比例，并进行预热，避免把即时编译和类初始化成本误算为容器成本。

## 批量构建：利用已知顺序

### 已排序且唯一的输入

如果输入已经按键严格升序排列，最优构建路径是一次分配足够容量并顺序写入，时间 \(O(n)\)，不需要逐项二分和移动。若输入大小已知，甚至可以精确分配数组。

批量 API 应验证键序，至少在开发或测试模式中验证。错误地接受无序输入会破坏最核心的不变量，使后续二分查找返回不可预测结果。为了性能完全省略验证时，方法名应明确表达“调用方保证已排序且唯一”，并限制在可信内部边界使用。

### 未排序输入

未排序的批量输入通常有两条路线：

1. 逐项调用 `put`，实现简单，但随机顺序可能产生大量移动，最坏接近平方级。
2. 先将记录按键排序，再线性去重和写入，复杂度通常为 \(O(n \log n)\)。

当批量规模较大时，先排序通常更可预测。排序前若使用条目对象暂存，会增加对象分配；也可以并行排序键和值索引，或把键和值编码进适合的临时结构。具体实现应由数据类型和性能目标决定。

### 合并两个已排序映射

两个 SparseArray 都按键升序时，可以用类似归并排序的双指针算法合并：

- 较小键直接写入结果。
- 键相等时按冲突策略选择值。
- 任一输入耗尽后复制另一输入剩余部分。

总时间为 \(O(n + m)\)，结果天然有序。若对每个元素都向现有大映射执行普通 `put`，可能产生反复移动。批量场景应尽量使用结构已提供的顺序信息。

## 并发边界与安全发布

### 普通实现不是线程安全容器

一次写入可能依次更新键数组、值数组、`usedSlots`、`liveSize` 和垃圾状态；扩容还会替换两个数组引用。这些步骤不是一个不可分割动作。没有同步时，一个线程可能观察到新键却看到旧值数组，看到新计数却看不到对应元素，或者在另一个线程压缩期间读取移动中的位置。

即使某次并发测试没有失败，也不能证明这种访问可靠。Java 内存模型允许线程在缺乏 happens-before 关系时看到旧值或不一致组合。`volatile` 修饰单个字段不能自动把多字段结构修改变成原子事务。

### 可行的所有权策略

常见安全方式包括：

- 将可变实例限制在单一线程中，不跨线程共享。
- 在一个线程中完成构建，通过安全发布交给其他线程，并在发布后保持只读。
- 对所有读写统一使用同一把锁，不能只锁写不锁读。
- 使用读写锁，在读取远多于写入且测量证明有收益时允许并发读。
- 写时复制：修改者创建新实例，完成后原子替换只读快照。
- 改用专门的并发映射，而不是在此结构上堆叠复杂同步。

安全发布可以通过初始化后存入 `final` 字段、在同步块中交接、写入 `volatile` 引用或使用并发容器完成。发布只保证接收线程看到构造完成的状态；若发布后继续原地修改，仍需同步协议。

### 复合操作需要整体原子性

即便为每个公开方法加 `synchronized`，以下逻辑仍可能竞态：

```java
if (!map.containsKey(key)) {
    map.put(key, value);
}
```

两个线程可能都在检查后写入。若业务要求“仅缺失时写入”，需要在同一锁范围内执行检查与写入，或提供原子复合方法。线程安全不是“方法没有抛异常”，而是并发后仍维持业务不变量。

### 值对象的线程安全是另一层问题

容器同步只能保护键值关联和内部数组，不能让 `V` 指向的可变对象自动线程安全。读线程取得值后与写线程同时修改其字段，仍需要值对象自身的同步、不可变设计或线程约束。

## 与其他结构的权衡

### 与 `HashMap<Integer,V>` 比较

哈希表在哈希分布合理时通常提供平均 \(O(1)\) 的查询、插入和删除，更适合频繁随机更新和较大规模数据。SparseArray 的查询是 \(O(\log n)\)，中间插入和紧凑是 \(O(n)\)，但可避免整数键装箱和部分条目结构开销，并天然按键有序。

选择前应问：

- 是否真正需要有序遍历或范围查询？
- 键装箱和节点开销在总内存中是否显著？
- 更新比例是否足以让数组移动成为瓶颈？
- 数据规模是否仍处于数组方案的优势区间？

不要因为哈希表平均查询是常数时间就断言它在小数据上一定更快，也不要因为 SparseArray 更紧凑就断言它一定更省内存。实现与运行时差异必须测量。

### 与 `TreeMap<Integer,V>` 比较

平衡树同样保持键有序，并提供稳定 \(O(\log n)\) 的查询、插入和删除，以及丰富的范围视图。其更新不会移动长数组，但通常需要节点对象、链接字段和整数装箱，遍历时也要追踪节点。

当范围操作丰富、数据规模较大、插入删除频繁时，树的稳定更新成本更有价值。当数据较小、构建后以读取和顺序遍历为主时，并行数组更紧凑。两者都“有序”，但为有序付费的方式不同：树在每个元素上维护链接，数组在更新时移动连续区间。

### 与直接索引数组比较

直接索引数组用 `values[key - offset]` 定位，查询和写入都是简单的 \(O(1)\)，没有二分和键数组。如果键域紧凑且范围可控，它通常是更直接的选择。

直接索引结构还需要表达“槽位不存在”。值不允许 `null` 时可用 `null`，允许 `null` 时需要额外存在位，例如 `boolean[] present` 或位图。即便如此，在键跨度不大时，它仍可能比 SparseArray 简单且快速。

SparseArray 的优势出现在跨度远大于元素数、负键或大键使直接分配不合理时。不能把“整数键”自动等同于“应该用稀疏结构”。

### 与排序条目列表比较

也可以维护 `List<Entry<V>>` 并按键排序。它在概念上直观，但条目对象和装箱键可能增加内存与间接访问。若使用不可变记录类，代码可读性和通用集合 API 兼容性可能更好。SparseArray 用更专门的表示换取更紧凑布局，也承担更多自定义实现和测试成本。

如果性能与内存并非问题，优先使用标准、易懂、维护成本低的集合往往更合理。自定义数据结构只有在需求明确且收益经过验证时才值得进入生产代码。

### 选择矩阵

| 主要需求 | 更可能合适的结构 |
| --- | --- |
| 键域小且密集，追求最快直接访问 | 直接索引数组 |
| 随机更新频繁，通常不关心顺序 | 哈希表 |
| 需要有序范围能力且更新频繁 | 平衡树 |
| 整数键，小到中等规模，读多写少，关注紧凑布局 | SparseArray |
| 构建后不可变，查询极多 | 排序数组或专门不可变映射 |
| 多线程高频并发修改 | 专用并发映射或分片设计 |

矩阵只是起点。实际选择还要考虑团队熟悉度、API 兼容性、测试成本和性能证据。

## 常见误区

### “稀疏”意味着数组里保留最大键范围

这里的数组长度由元素容量决定，键值只是被排序保存，不作为物理下标。若按最大键分配数组，那是直接索引结构，不是本文讨论的紧凑稀疏映射。

### 查询是二分，所以所有操作都是对数复杂度

二分只解决定位。新键插入可能移动后半段，删除压缩可能扫描全部槽位，按值查找也是线性。分析必须覆盖完整操作路径。

### 惰性删除让删除成为没有代价的常数操作

删除前仍要定位，且空洞最终需要整理。惰性删除只是合并和推迟移动成本，不会消除成本，还可能把线性工作转移到不可预期的读取时刻。

### 有序索引可以长期缓存

索引是当前排序状态下的位置，不是稳定身份。任何较小键的插入、删除或压缩都可能改变它。需要稳定标识时应保存键。

### 返回 `null` 就能表示不存在

只有在明确禁止 `null` 值时才成立。允许 `null` 时必须提供独立存在性查询或结果类型。

### 自定义实现一定比标准集合节省

自定义实现可能减少某些结构开销，却增加代码、测试、缺陷和维护成本。若数据很小或生命周期很短，节省量可能没有意义；若值对象本身很大，键结构所占比例也可能很低。先测量，再决定是否值得承担专用结构。

### 只读并发不需要安全发布

对象构建完成后不再修改，确实适合多线程共享，但前提是引用通过建立 happens-before 关系的方式发布。把普通字段中的引用随意暴露出去，不应作为可靠并发协议。

## 设计与测试清单

设计一个整数键稀疏映射时，应先明确：

- 是否允许负键，是否对键范围有业务限制。
- 是否允许 `null` 值，如何区分缺失。
- 重复键写入是替换、拒绝还是合并。
- 删除是立即移动还是惰性标记，谁触发压缩。
- `size()`、按索引访问和遍历是否可能触发线性整理。
- 是否提供递增追加、批量构建和显式收缩。
- 索引的失效规则是否写入文档。
- 实例是线程封闭、外部加锁、不可变快照还是其他所有权模型。

测试至少覆盖：

- 空映射、单元素、首部/中部/尾部插入。
- 最小和最大 `int` 键、负键、零以及相邻键。
- 替换已有键，包括旧值或新值为 `null`。
- 删除不存在键、重复删除、删除后同键恢复。
- 多个空洞后压缩，验证键值对齐和尾部引用清理。
- 容量为 0 的首次插入、跨越多次扩容、压缩后避免不必要扩容。
- `keyAt`、`valueAt` 的边界异常和修改后的索引变化。
- 递增追加以及不满足递增条件时的既定行为。
- 批量输入的乱序、重复和冲突策略。
- 与参考映射进行随机差分测试。

性能测试要分别测量命中查询、未命中查询、尾部追加、中间随机插入、批量删除后首次读取、全量遍历和范围扫描。只给一个混合吞吐数字会隐藏延迟尖峰与操作差异。

## 总结

SparseArray 的核心不是某个类名，而是一种清晰的数据结构组合：用严格升序的 `int[]` 保存实际出现的键，用并行值数组保存映射结果，通过二分查找定位，通过数组移动维持顺序，通过删除标记合并整理成本，并用几何扩容控制连续构建的复制总量。

它把空间成本从巨大键跨度转移到实际元素数量，避免整数键装箱和逐条节点对象，并获得良好的顺序遍历局部性。相应代价是新键中间插入、压缩和收缩可能执行线性复制，索引会随结构修改而变化，允许 `null` 时必须明确缺失语义，普通可变实现也不能无同步地跨线程读写。

选择这种结构的可靠依据是访问模式：整数键、小到中等规模、键稀疏、读多写少、需要有序访问或可以批量构建。若键域密集，直接索引数组更简单；若随机更新频繁，哈希表通常更合适；若既要有序又要稳定更新，平衡树更有优势。把复杂度条件、内存布局、局部性和并发边界一起评估，才能判断 SparseArray 是否与问题真正匹配。

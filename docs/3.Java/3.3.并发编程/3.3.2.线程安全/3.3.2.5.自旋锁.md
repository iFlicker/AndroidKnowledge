# 3.3.2.5 自旋锁

自旋锁是一类“线程获取锁失败后不立即挂起，而是在用户态反复尝试获取锁”的同步策略。它的核心思想很直接：如果临界区极短，锁很快就会释放，那么让等待线程继续占用 CPU 做几次检查，可能比把线程阻塞、让操作系统调度出去、再在锁释放后唤醒回来更划算。自旋锁用持续尝试换取少一次线程挂起和唤醒的成本，因此它不是单纯的互斥工具，而是一种带有明显性能假设的等待策略。

理解自旋锁，不能只停留在“while 循环加 CAS”这个代码形态上。自旋锁同时涉及互斥、可见性、原子更新、CPU 消耗、调度成本、公平性、退避策略和失败边界。它在短临界区、低到中等竞争、线程数量不超过可用处理器太多的场景下可能很有效；在临界区较长、竞争激烈、锁持有线程被阻塞或 CPU 已经饱和时，则可能把系统推向更差的状态。自旋不是免费的等待，它只是把等待成本从“线程调度”转移为“持续消耗处理器时间”。

本文保持通用 Java 技术视角，讨论 Java 程序中自旋锁的概念、CAS 关系、内存语义、适用前提、退避设计、与阻塞锁的比较，以及在标准并发库和业务代码中的实践边界。自旋锁是一种值得理解的底层机制，但在多数业务代码中，它更适合作为分析并发工具和性能现象的知识，而不是默认手写的同步方案。

## 自旋锁到底是什么

锁的本质是保护共享资源的访问协议：同一时刻只有满足条件的线程可以进入临界区，其他线程必须等待。阻塞锁的典型做法是：线程获取锁失败后进入阻塞状态，等待锁释放时被唤醒；自旋锁的典型做法是：线程获取锁失败后仍保持运行态，不断检查锁状态，直到获取成功、超时、被取消，或者切换到其他等待方式。

从线程状态上看，自旋等待的线程通常不会因为锁失败而进入 `WAITING` 或 `BLOCKED` 状态，它仍然在 CPU 上执行循环逻辑。循环体可能只是一次原子比较，也可能包含 `Thread.onSpinWait()`、计数、退避、让步或短暂暂停。只要没有调用会阻塞的同步原语，它就没有把等待责任交给操作系统调度器，而是在当前线程中主动重试。

最小化的自旋锁可以用一个原子状态表示：

```java
import java.util.concurrent.atomic.AtomicBoolean;

final class SimpleSpinLock {
    private final AtomicBoolean locked = new AtomicBoolean(false);

    public void lock() {
        while (!locked.compareAndSet(false, true)) {
            Thread.onSpinWait();
        }
    }

    public void unlock() {
        locked.set(false);
    }
}
```

这段代码能说明自旋锁的基本形态：`false` 表示未持有，`true` 表示已持有；线程通过 CAS 把 `false` 改成 `true`，成功者进入临界区，失败者继续尝试。`unlock()` 再把状态写回 `false`，让其他线程有机会成功。

不过，这段代码只能作为概念示例，不能直接当成成熟锁使用。它不检查释放者是否为持有者，不支持重入，不支持中断，不支持超时，不提供公平队列，在高竞争下会制造大量 CAS 失败，也无法避免某个线程长期抢不到锁。真正的并发工具必须处理这些边界。理解自旋锁的第一步，就是区分“能表达互斥思想的示例”和“可以承载生产级同步语义的锁实现”。

自旋锁中的“锁”强调互斥，“自旋”强调等待方式。两者必须分开理解。互斥要求进入临界区前必须获得独占资格，离开临界区后必须发布释放动作；自旋只是竞争失败时的一种等待策略。一个同步器可以先自旋几次，再阻塞；也可以只在特定阶段自旋，之后进入队列；还可以在无锁算法中使用 CAS 重试，但并不把它称作锁。日常讨论中容易把“CAS 循环”“忙等”“自旋锁”混为一谈，实际设计时应看它是否保护临界区、是否有明确的持有与释放关系、是否允许多个线程同时推进。

## 为什么会选择自旋

线程阻塞不是一个纯 Java 层面的简单动作。线程获取阻塞锁失败后，通常要从运行态进入等待队列，调度器需要把它从 CPU 上移走，未来锁释放时再把它唤醒，让它重新参与调度。这个过程涉及运行时、操作系统和处理器之间的协作。虽然现代 JVM 和操作系统已经做了大量优化，但线程挂起与唤醒仍然不是零成本动作。

如果锁持有时间非常短，例如只更新一个计数器、交换一个引用、维护一个小型状态机，那么阻塞线程可能刚被挂起，锁就已经释放了。此时等待线程再被唤醒、重新调度回来，消耗的时间可能超过临界区本身。自旋的动机正来自这种不匹配：既然锁预计马上释放，就先在原地等一小段时间，避免过早进入昂贵的阻塞路径。

自旋有效的前提是“等待时间短且可预期”。这里的短不是抽象感觉，而是相对于线程阻塞和唤醒成本而言的短。若临界区只包含几个内存操作，自旋可能有收益；若临界区包含磁盘访问、网络调用、长循环、等待另一个任务、日志阻塞或复杂计算，自旋线程就可能在很长时间内空耗 CPU。自旋锁最怕的不是获取失败，而是锁持有线程无法快速释放。

还要注意，锁持有时间并不只由临界区代码行数决定。一个看起来很短的方法，可能因为缓存未命中、分支预测失败、对象分配、类加载、页错误、垃圾回收停顿、操作系统抢占或其他线程竞争而变慢。自旋策略在推理时必须留有余地，不能只根据“代码很短”就断定等待很短。

自旋还依赖 CPU 资源。等待线程自旋时仍会占用执行单元、缓存带宽和内存子系统资源。如果机器上可运行线程数量已经远大于可用处理器数量，自旋线程会和真正能释放锁的线程争抢 CPU，反而延迟锁释放。更糟的是，多个等待者同时自旋会不断读取或 CAS 同一个共享变量，让缓存行在处理器核心之间反复失效，造成总线和缓存一致性压力。

因此，自旋锁不是“比阻塞锁更快”的锁，而是“在特定等待形态下可能更快”的锁。它的收益来自避免调度成本，它的代价是消耗 CPU 和缓存资源。只讨论收益、不讨论代价，就会把自旋误用成通用优化。

## 自旋锁与 CAS 的关系

Java 中讲自旋锁通常会同时讲 CAS。CAS 是 Compare-And-Set 或 Compare-And-Swap 的简称，含义是：只有当某个内存位置当前值等于预期值时，才把它更新为新值；整个比较和更新过程以原子方式完成。自旋锁需要在多个线程之间竞争同一个状态位，普通读写无法保证“看到未加锁”和“设置为已加锁”这两个动作不可分割，因此常用 CAS 完成获取锁的原子竞争。

在 `AtomicBoolean.compareAndSet(false, true)` 中，线程表达的是：如果锁状态仍然是未持有，就把它改成已持有。多个线程同时执行时，最多只有一个线程能成功，因为第一个成功线程会改变状态，后续线程再比较时就会发现当前值不再是 `false`。这使得 CAS 成为轻量互斥的基础。

CAS 本身不是锁。CAS 只保证单个内存位置的一次条件更新原子，是否形成锁语义取决于外层协议。若 CAS 成功后线程进入临界区，退出时用合适的写入释放状态，其他线程再通过 CAS 获取，那么它构成一个简单自旋锁。若 CAS 失败后继续重试更新某个无锁数据结构的指针，它可能是无锁算法的一部分。若 CAS 只是更新一个计数器，就只是原子变量操作。把 CAS 等同于自旋锁，会模糊设计边界。

CAS 循环也不一定就是锁。下面这种写法是典型的原子累加重试，它没有持有锁的阶段，也没有独占临界区：

```java
import java.util.concurrent.atomic.AtomicInteger;

final class Counter {
    private final AtomicInteger value = new AtomicInteger();

    public int addAndGet(int delta) {
        for (;;) {
            int current = value.get();
            int next = current + delta;
            if (value.compareAndSet(current, next)) {
                return next;
            }
            Thread.onSpinWait();
        }
    }
}
```

这个循环失败后重试，是因为别的线程已经更新了同一个变量。它通过乐观方式完成单变量状态转换，不需要一个线程长期占有临界区。相比之下，自旋锁把竞争状态和临界区绑定在一起，CAS 成功者获得进入资格，失败者等待资格释放。

CAS 的优势是避免内核阻塞路径，失败时可以很快重试；缺点是高竞争下失败次数可能很多，且只能直接处理单个变量或一个可被封装成单个引用的状态。多个字段之间存在复杂不变式时，CAS 需要把状态设计成不可变快照并整体替换，或者退回锁保护。若只是把多个原子变量拼在一起，很容易出现每个变量单独原子、整体业务状态不一致的问题。

CAS 还常被提到 ABA 问题。所谓 ABA，是线程读取到值 A，准备 CAS；期间其他线程把值从 A 改成 B，又改回 A；第一个线程再 CAS 时发现当前值仍等于 A，于是误以为状态没有变化。对简单布尔自旋锁来说，ABA 通常不是主要问题，因为锁状态只表达是否持有，释放再获取本来就是合法状态变化；但对栈顶指针、链表节点、版本化资源等结构，ABA 可能破坏语义。此时需要版本戳、标记引用或其他协议辅助。

## 自旋锁的内存语义

自旋锁若只保证互斥而不保证可见性，就不是可靠的锁。一个线程在临界区内修改共享数据，释放锁后，另一个线程获取锁进入临界区，应该能够看到前一个线程释放锁之前的写入。这个要求依赖 Java 内存模型中的同步语义。

使用 `AtomicBoolean` 的 `compareAndSet` 和 `set`，通常可以获得 volatile 风格的读写语义。CAS 成功获取锁时包含对状态变量的原子读改写，释放时对状态变量的写入应与后续获取形成可见性配合。更通用地说，锁实现必须保证释放动作之前的普通写入，不会在另一个线程成功获取锁之后仍对它不可见。`synchronized` 和 `java.util.concurrent.locks.Lock` 在规范层面提供了清晰的释放与获取关系；手写自旋锁则必须谨慎选择原子变量、VarHandle 访问模式或 volatile 字段，不能用普通字段裸写。

下面这个错误示例展示了为什么普通字段不够：

```java
final class BrokenSpinLock {
    private boolean locked;

    public void lock() {
        while (locked) {
            // 错误：普通读可能被优化，且没有原子竞争。
        }
        locked = true;
    }

    public void unlock() {
        locked = false;
    }
}
```

这段代码有两个根本问题。第一，`while (locked)` 和 `locked = true` 不是原子操作，两个线程可能同时观察到 `false`，然后都进入临界区。第二，`locked` 不是 volatile，也没有其他同步动作，一个线程对它的写入不保证被另一个线程及时观察；编译器和处理器还可能在不破坏单线程语义的前提下做优化，使循环行为更不可控。它看起来像锁，实际既没有互斥保证，也没有可靠可见性。

即便用 `volatile boolean`，也只能解决可见性问题，不能解决原子竞争问题。多个线程仍可能都读到 `false`，然后分别写入 `true`。因此，自旋锁获取阶段需要 CAS 或等价原子操作；释放阶段需要具有发布语义的写入；进入临界区后，临界区保护的数据也必须只通过同一个锁协议访问。如果有的路径使用自旋锁，有的路径绕过锁直接读写，那么锁语义就被破坏。

内存语义还要求锁对象本身被安全发布。一个自旋锁实例如果在构造未完成时泄露给其他线程，或者保护的数据在没有同步的情况下发布，就可能让其他线程观察到未初始化或部分初始化状态。锁只能保护遵守协议之后的访问，不能自动修复创建和发布阶段的错误。

## 一个可讲清楚边界的简单实现

概念示例常用 `AtomicBoolean`，但稍微严谨一点的自旋锁至少应知道持有线程，避免非持有者释放锁。否则任何线程都能调用 `unlock()` 把状态改成未持有，正在临界区工作的线程就会失去互斥保护。

```java
import java.util.concurrent.atomic.AtomicReference;

final class OwnerCheckedSpinLock {
    private final AtomicReference<Thread> owner = new AtomicReference<>();

    public void lock() {
        Thread current = Thread.currentThread();
        while (!owner.compareAndSet(null, current)) {
            Thread.onSpinWait();
        }
    }

    public boolean tryLock() {
        return owner.compareAndSet(null, Thread.currentThread());
    }

    public void unlock() {
        Thread current = Thread.currentThread();
        if (!owner.compareAndSet(current, null)) {
            throw new IllegalMonitorStateException("current thread does not hold this lock");
        }
    }
}
```

这个实现仍然不完整，但比布尔状态更能暴露锁设计问题。它不支持重入：同一个线程再次调用 `lock()` 会自旋等待自己释放锁，造成自锁式死等。它不响应中断：等待线程即使被中断，也会继续自旋，除非外层另有退出条件。它没有超时：竞争严重时调用方无法控制等待上限。它不公平：后来的线程可能在锁刚释放时抢先成功，先来的线程可能长期失败。它也没有条件队列：无法像 `Condition` 或 `wait/notify` 那样在临界区内等待某个业务条件。

这正是手写自旋锁的价值和危险所在。它能帮助理解锁的基本组成：状态、原子获取、释放校验、临界区和内存语义；但它也会暴露成熟锁必须处理的复杂边界。实际业务代码若需要锁，优先考虑 `synchronized`、`ReentrantLock`、并发容器、原子类或更高层同步器。手写锁只有在非常明确的底层场景、性能目标和测试条件下才值得考虑。

如果确实要提供可取消或限时的自旋获取，接口语义必须提前设计清楚：

```java
import java.time.Duration;
import java.util.concurrent.atomic.AtomicReference;
import java.util.concurrent.locks.LockSupport;

final class TimedSpinLock {
    private final AtomicReference<Thread> owner = new AtomicReference<>();

    public boolean tryLock(Duration timeout) throws InterruptedException {
        long deadline = System.nanoTime() + timeout.toNanos();
        Thread current = Thread.currentThread();
        int failures = 0;
        while (!owner.compareAndSet(null, current)) {
            if (Thread.interrupted()) {
                throw new InterruptedException();
            }
            if (System.nanoTime() >= deadline) {
                return false;
            }
            if (++failures < 64) {
                Thread.onSpinWait();
            } else {
                LockSupport.parkNanos(1_000L);
            }
        }
        return true;
    }

    public void unlock() {
        Thread current = Thread.currentThread();
        if (!owner.compareAndSet(current, null)) {
            throw new IllegalMonitorStateException();
        }
    }
}
```

这段代码引入了超时、中断检查和轻微退避，但仍然不是通用锁。它只说明一个原则：自旋不能无限制地写成空循环。等待策略应有边界，至少要考虑中断、超时、退避和释放者校验。越接近可复用同步工具，越不能只靠一个 CAS 循环解决全部问题。

## 适用前提：临界区短、竞争可控、线程不会被阻塞

自旋锁最重要的适用前提是临界区极短。临界区中最好只包含少量内存读写、简单状态更新或轻量计算，不应包含可能长时间停顿的操作。尤其要避免在自旋锁保护范围内执行 I/O、等待其他线程、提交任务后同步等待结果、调用不可控回调、做复杂日志输出、进入可能阻塞的锁、执行大对象分配或触发耗时清理。只要锁持有线程可能长时间不释放，等待线程的自旋就会成为浪费。

第二个前提是竞争程度可控。低竞争时，自旋锁大多数时候可以一次 CAS 成功，失败重试很少；中等竞争且临界区短时，自旋可能仍有收益；高竞争时，大量线程反复 CAS 同一个变量，会造成缓存行争用和失败风暴。此时不是每个线程都“很努力”，系统就会更快；相反，它们会互相干扰，让真正的临界区执行变慢。

第三个前提是可运行线程数量不能显著超过处理器能力。如果有 4 个处理器核心，却有几十个线程在同一把锁上自旋，许多线程即使正在等待也会消耗调度机会。锁持有者一旦被抢占，其他线程继续自旋并不能释放锁，只会占用本该让持有者继续执行的 CPU 时间。自旋锁在这种情况下可能比阻塞锁更糟，因为阻塞锁至少能把等待者移出可运行队列。

第四个前提是等待时间应可预测。某些底层状态切换有稳定的短等待窗口，自旋几次可能很合理；而业务流程中的等待往往受外部系统、队列积压、垃圾回收、配置变化、数据规模和调用方行为影响，等待时间很难稳定。越难预测，越应该使用支持阻塞、超时和诊断的成熟同步工具。

第五个前提是代码必须能承受非公平性。简单自旋锁通常没有排队结构，锁释放瞬间谁先 CAS 成功由调度、缓存位置和时机决定。某个线程可能连续多次抢到锁，另一个线程长期失败。如果业务要求先到先服务、等待时间上界或饥饿控制，简单自旋锁并不合适。

因此，判断是否适合自旋锁，可以用一组问题逼近答案：临界区是否只有几十到几百纳秒级的轻量操作？锁持有者是否不会阻塞？竞争线程数是否接近可用处理器数量？等待失败是否允许重试？长期抢不到锁是否可接受？能否通过压测证明收益稳定？只要这些问题中有几个回答含糊，就应优先选择阻塞锁或更高层同步器。

## CPU 消耗与缓存一致性压力

自旋锁最直观的成本是 CPU。等待线程在循环中持续执行，哪怕循环体只有一条 `onSpinWait()`，它也不是睡眠。它会占用处理器执行周期，影响同一机器上其他线程获得 CPU 的机会。低竞争时这个成本几乎不可见；高竞争或锁长时间不释放时，这个成本会迅速放大。

比 CPU 时间更隐蔽的是缓存一致性压力。现代多核处理器通常为每个核心提供私有缓存，并通过缓存一致性协议维护共享内存视图。当多个核心反复读取同一个锁变量时，缓存系统需要维持它们对该缓存行的观察；当多个核心反复 CAS 同一个变量时，情况更严重，因为 CAS 往往需要获得该缓存行的独占修改权。即使 CAS 失败，也可能造成缓存行在核心之间来回转移。

这就是高竞争自旋锁容易塌陷的原因。每个等待线程都在争夺同一个锁状态缓存行，锁持有者释放时也需要写这个缓存行。等待者越多，竞争越激烈，释放与获取的成本越高，临界区外的同步成本可能超过临界区内真正的工作。此时继续增加线程数量不会提高吞吐，反而可能降低吞吐并增加延迟。

空循环还会影响处理器的执行资源和功耗。`Thread.onSpinWait()` 的作用是向运行时或处理器提示当前处于自旋等待，底层可能选择更合适的指令或策略，以降低对流水线和功耗的影响。它不是让线程阻塞，也不保证让出 CPU，更不是锁语义的一部分。它只能作为自旋循环中的友好提示，不能替代退避、超时或阻塞。

另一个常见误区是认为“自旋没有上下文切换，所以一定省资源”。上下文切换确实有成本，但忙等也有成本。阻塞锁把等待者从运行竞争中移走，代价是挂起和唤醒；自旋锁让等待者留在运行竞争中，代价是持续消耗处理器和缓存资源。两者没有绝对优劣，关键在等待时长、竞争强度和系统负载。

在性能分析中，如果看到 CPU 使用率很高但业务吞吐不升反降，且线程栈集中在 CAS 循环或自旋等待处，就要怀疑自旋过度。此时优化方向可能不是“再减少锁内代码一行”，而是降低共享热点、分片锁状态、减少竞争线程数、增加退避、改用阻塞锁，或者重新设计为队列化处理。

## 公平性、饥饿与锁顺序

简单自旋锁通常是非公平的。锁释放时，所有等待线程都可能同时观察到机会并竞争 CAS。谁成功不取决于谁等待更久，而取决于当时的调度、核心位置、缓存状态和执行时机。后来到达的线程如果刚好在锁释放时运行，可能比早已自旋很久的线程先成功。

非公平不一定是坏事。公平排队需要维护等待队列、处理入队出队和唤醒顺序，这些都会增加开销。非公平锁允许刚释放锁附近的线程或刚好运行中的线程快速再次获取锁，可能提升吞吐和缓存局部性。在吞吐优先、临界区短、等待时间不敏感的场景，非公平策略可能表现更好。

问题在于非公平会带来饥饿风险。饥饿不是死锁，系统整体仍在前进，只是某个或某些线程长期无法获得锁。简单自旋锁没有内置机制记录等待时间，也没有队列保证顺序，所以很难给出等待上界。如果调用方需要可预测延迟，或者某些线程承担重要任务，长期失败就不能接受。

公平自旋锁可以通过队列思想实现，例如票据锁或 CLH、MCS 一类队列锁思想。票据锁让每个线程获取一个递增票号，再等待服务号轮到自己；队列锁让每个等待者在自己的节点或前驱节点上自旋，减少所有线程争抢同一个全局状态的压力。这些设计能改善公平性和缓存争用，但实现复杂度更高，也未必适合直接用 Java 手写。

Java 标准库中的 `ReentrantLock` 提供公平模式选项，但它不是简单自旋锁。它基于更完整的同步队列机制，在竞争失败时可以让线程进入等待队列，并在释放时按策略唤醒。公平锁通常降低吞吐、提高等待顺序可预测性；非公平锁通常提高吞吐、降低顺序保证。自旋锁讨论公平性时，也应回到这个取舍：你想优化的是平均吞吐、尾延迟，还是先到先服务的语义。

锁顺序也是公平性之外的风险。自旋锁同样会发生死锁：线程 A 持有锁 1 后自旋等待锁 2，线程 B 持有锁 2 后自旋等待锁 1，二者都不会释放已有锁。阻塞锁死锁时线程可能表现为阻塞，自旋锁死锁时线程可能表现为 CPU 持续消耗。若多个锁组合使用，仍必须规定全局获取顺序，不能因为锁是自旋实现就忽略死锁条件。

## 退避策略：不要让所有线程以同一频率撞门

最朴素的自旋是失败后立刻重试：

```java
while (!state.compareAndSet(0, 1)) {
    Thread.onSpinWait();
}
```

这种写法在竞争很低时足够简单，但在竞争升高时会让所有等待者以很高频率访问同一个状态。锁一释放，所有线程几乎同时冲上来 CAS，只有一个成功，其余继续失败。失败者又以同样频率重试，造成持续争用。这就像所有人都不停撞同一扇门，门开的一瞬间也只能进一个，剩下的人制造了大量额外碰撞。

退避策略的目标是降低无效竞争。失败次数较少时可以快速自旋，因为锁可能马上释放；失败次数增加后，说明竞争较激烈或持有时间较长，就逐渐降低重试频率。退避可以是固定暂停、指数退避、随机退避、调用 `Thread.yield()`、短暂 `parkNanos()`，或者在若干次自旋后切换到阻塞等待。

一个简化的退避循环可能长这样：

```java
int failures = 0;
while (!locked.compareAndSet(false, true)) {
    if (failures < 32) {
        Thread.onSpinWait();
    } else if (failures < 128) {
        Thread.yield();
    } else {
        LockSupport.parkNanos(1_000L);
    }
    failures++;
}
```

这段代码不是推荐模板，而是展示等待层次。前几次失败仍假设锁很快释放；失败更多后让出调度机会；再失败则短暂暂停。实际参数不能拍脑袋决定，应结合临界区耗时、线程数量、处理器数量和压测数据调整。错误的退避可能两头不讨好：退避太弱，仍然争用严重；退避太强，锁已经释放却没人及时获取，增加延迟。

随机退避可以避免多个线程在相同节奏下同时重试。尤其在许多线程执行相同代码路径时，如果每个线程都按完全相同的失败次数和暂停时间重试，就可能形成同步碰撞。加入轻微随机性有助于打散竞争，但也会带来更难预测的延迟分布。

另一种重要策略是“自旋后阻塞”。成熟同步器常常不是纯自旋或纯阻塞，而是在短暂自旋失败后进入队列等待。短暂自旋尝试吃掉很快释放的锁，阻塞队列避免长期空耗 CPU。这个混合策略比简单二选一更接近真实系统需求。对业务代码来说，如果发现自己正在设计复杂退避和唤醒机制，通常意味着应该优先考虑标准库中已有的 `Lock`、队列或同步器。

## 与阻塞锁的比较

自旋锁和阻塞锁最大的差异不是“谁更高级”，而是等待成本不同。自旋锁等待时线程继续运行，消耗 CPU，避免或减少挂起唤醒；阻塞锁等待时线程暂停运行，节省 CPU，但需要调度器参与。选择哪一种，取决于等待时间、竞争强度、响应要求和系统负载。

| 维度 | 自旋锁 | 阻塞锁 |
| --- | --- | --- |
| 等待方式 | 失败后循环重试 | 失败后进入阻塞或等待队列 |
| 主要成本 | CPU 时间、缓存一致性压力、功耗 | 线程挂起、唤醒、上下文切换、队列维护 |
| 适合临界区 | 极短、不会阻塞、可预测 | 可短可长，尤其适合可能等待较久的临界区 |
| 高竞争表现 | 容易 CAS 风暴和 CPU 空转 | 等待者可被挂起，吞吐更稳定 |
| 公平性 | 简单实现通常不公平 | 可通过队列实现公平或近似公平 |
| 中断与超时 | 手写实现需额外设计 | 标准锁通常提供相关能力 |
| 诊断性 | 简单实现较差，线程可能一直 RUNNABLE | 线程 dump 更容易看到阻塞点 |

阻塞锁并不意味着慢。`synchronized` 和 `ReentrantLock` 在现代 JVM 中已经包含大量优化，并且在竞争激烈或临界区较长时，阻塞等待可以让 CPU 执行真正有进展的线程。自旋锁也不意味着轻量。一个高竞争自旋锁可能比阻塞锁造成更多 CPU 消耗和更差尾延迟。

比较二者时，还要看临界区是否会调用外部代码。如果临界区中会调用不可控方法，阻塞锁通常更安全，因为等待者至少不会一直烧 CPU；自旋锁则要求锁持有者尽快释放，外部代码的不确定性会破坏这个假设。反过来，如果临界区只是更新一个内部状态字段，且竞争很低，阻塞锁的完整队列和调度能力可能显得过重。

在 Java 业务开发中，默认选择通常不应是手写自旋锁，而是先使用表达语义更清楚的工具：互斥用 `synchronized` 或 `ReentrantLock`，单变量状态更新用原子类，生产消费用阻塞队列，阶段等待用 `CountDownLatch`、`CyclicBarrier` 或 `Phaser`，并发限流用 `Semaphore`。自旋锁适合作为这些工具底层机制的一部分，或在非常明确的热点路径中经过测量后使用。

## Java 中相关实践

Java 标准库没有提供一个直接名为 `SpinLock` 的通用锁类，这本身就是重要信号。通用锁要处理重入、所有者校验、等待队列、中断、超时、公平性、条件队列、监控和内存语义；纯自旋很难成为适合所有场景的默认抽象。因此，Java 并发库更多是在底层或局部策略中使用自旋思想，而不是鼓励业务代码直接手写自旋锁。

`AtomicInteger`、`AtomicReference`、`AtomicBoolean` 等原子类常通过 CAS 循环实现乐观更新。调用者看到的是原子操作接口，内部可能存在失败重试。这里的自旋是为了完成单变量状态转换，不一定构成锁。使用原子类时，要先确认问题是否能被单变量状态表达；如果业务不变式跨多个字段，盲目使用多个原子类通常会制造一致性漏洞。

`Thread.onSpinWait()` 是 Java 提供的自旋提示方法，适用于自旋循环内部。它告诉运行时当前线程处于忙等状态，运行时或处理器可以利用这个提示进行优化。它没有同步语义，不读取共享状态，不释放锁，不让线程进入等待队列，也不保证降低 CPU 使用率。正确用法是配合已有的原子或 volatile 条件，而不是把它当成并发控制工具。

`LockSupport.park()` 和 `parkNanos()` 则代表另一类等待方式：阻塞当前线程，直到许可、超时或中断等条件出现。很多同步器会结合短自旋和 `park`。自旋处理极短等待，`park` 处理较长等待。直接手写同步器时要非常谨慎，因为 `park/unpark` 有许可语义、时序问题和中断交互，稍有不慎就可能丢失唤醒或永久等待。

`synchronized` 的实现细节在不同 JVM 版本和运行配置中会变化，但从学习角度看，可以知道现代 JVM 会根据锁竞争情况进行优化，而不是一上来就走最重的阻塞路径。理解这一点的意义不是记住某个具体优化名称，而是避免把 `synchronized` 简化为“必然重量级”。实际性能应通过当前运行环境和负载测量，而不是根据过时印象判断。

`AbstractQueuedSynchronizer` 是许多 Java 同步器的基础。它围绕一个同步状态和一个等待队列构建获取、释放、排队、阻塞和唤醒协议。AQS 风格的同步器通常会在快速路径上尝试 CAS 获取状态，失败后进入队列，而不是让所有线程无限自旋。理解自旋锁有助于理解 AQS 为什么要区分快速尝试和排队等待：快速路径降低无竞争开销，队列路径控制高竞争下的 CPU 消耗和唤醒顺序。

`LongAdder` 这类高并发计数工具也体现了减少热点竞争的思路。它不靠所有线程反复争抢同一个计数位置，而是在竞争时把更新分散到多个槽位，读取时再汇总。它不是自旋锁，但它说明了一个关键原则：高竞争下，与其让所有线程围绕一个热点自旋，不如重新设计状态分布，减少共享热点。很多时候，消除竞争比优化锁更有效。

## 自旋与无锁算法的区别

自旋锁、CAS 循环和无锁算法经常一起出现，但它们不是同一个概念。自旋锁是一种互斥锁，某个线程成功获取后进入临界区，其他线程等待它释放。无锁算法通常不允许某个线程独占整个临界区，而是让多个线程通过原子步骤共同推进数据结构状态。一个线程失败并重试，不意味着系统停止；只要总有某个线程能完成进展，就可能满足无锁进展条件。

以原子计数器为例，多个线程同时递增，某个线程 CAS 失败后重试，是因为另一个线程已经成功推进了计数器状态。失败者没有等待一个持锁者离开临界区，而是在基于新状态重新计算。这种重试是乐观并发控制。自旋锁则不同：失败者必须等待锁状态从持有变为未持有，临界区内部只有持有者能执行。

这种区别影响故障表现。如果持有自旋锁的线程被阻塞、崩溃在未释放路径或进入长时间计算，所有等待者都无法进入临界区；而设计良好的无锁算法不依赖单个线程长期持有全局资格，其他线程仍可能继续推进。当然，无锁算法实现难度很高，也会遇到 ABA、内存回收、复杂状态一致性和活锁等问题，并不天然简单。

业务代码中常见的误用是：看到无锁算法使用 CAS 循环，就模仿写一个 CAS 自旋锁，以为这就是“无锁”。实际正相反，自旋锁仍然是锁，只是等待方式忙等。它不具备无锁进展保证，也不能避免死锁和饥饿。判断一个方案是不是无锁，不能只看有没有 `compareAndSet`，而要看线程失败时系统整体是否仍能由其他线程持续推进，是否存在持有者阻塞全部竞争者的情况。

## 可重入、所有者与异常释放

可重入是锁语义中的重要边界。`ReentrantLock` 和 `synchronized` 都允许同一线程在已经持有锁的情况下再次获取同一把锁，并通过持有计数在对应次数释放后真正解锁。简单自旋锁通常不支持重入。如果同一个线程在持有锁时再次调用 `lock()`，CAS 会一直失败，因为锁状态已经被自己占有，而它又在等待自己释放，结果就是无限自旋。

是否支持重入不是纯性能问题，而是接口契约问题。若锁保护的方法之间会相互调用，或者回调路径可能重新进入同一对象，就要明确锁是否可重入。不可重入锁可以减少状态复杂度，但要求调用结构严格避免重复获取；可重入锁更符合许多面向对象代码的调用习惯，但需要记录持有线程和重入次数，释放时也要校验次数。

所有者校验同样关键。没有所有者校验的布尔自旋锁，任何线程都能释放锁。这会让错误释放静默发生，破坏互斥后才在更远处暴露数据损坏。成熟锁通常会在非持有线程释放时抛出异常，尽早暴露协议错误。手写自旋锁如果不做这件事，就会让错误更难诊断。

异常释放是使用锁时的基本纪律。无论锁是自旋还是阻塞，只要临界区可能抛异常，释放动作就必须放在 `finally` 中：

```java
lock.lock();
try {
    updateSharedState();
} finally {
    lock.unlock();
}
```

自旋锁尤其不能遗漏释放。阻塞锁遗漏释放会让等待线程阻塞；自旋锁遗漏释放会让等待线程持续运行并消耗 CPU。两者都是严重错误，但自旋锁的错误更容易表现为 CPU 飙升和系统整体变慢。若线程在临界区中被异常打断且没有释放锁，等待者没有任何办法靠自旋解决问题。

## 中断、超时与取消

简单自旋锁的 `lock()` 循环通常不响应中断。线程即使被 `interrupt()`，也只是中断标志被设置，循环仍会继续执行，除非代码主动检查。对于需要可关闭、可取消或有服务等级要求的程序，不响应中断是一大缺陷。

中断不是强制停止线程，而是一种协作式取消信号。阻塞方法如某些等待、睡眠和队列操作会通过 `InterruptedException` 把信号交给调用方；自旋循环不会自动这样做。若自旋等待可能持续较久，就应该设计 `lockInterruptibly()` 或 `tryLock(timeout)` 之类接口，在循环中检查中断标志，并决定是抛出异常、返回失败，还是恢复中断后退出。

超时同样重要。没有超时的自旋等待会把“锁迟迟拿不到”变成无限 CPU 消耗。带超时的获取至少能让调用方记录现场、降级、放弃请求或触发告警。超时不是为了掩盖同步错误，而是为不可预期等待提供控制边界。

取消路径必须和释放路径配合。如果线程在等待锁时被取消，它不能错误释放自己没有持有的锁；如果线程在持有锁时接到中断，是否立即退出要由临界区语义决定，但退出前必须释放锁并维护共享状态一致。把中断检查随意塞进临界区，可能导致状态更新到一半就退出；完全不检查中断，又可能导致关闭流程无法完成。自旋锁只是等待策略，不能替代业务层对取消点和一致性的设计。

在标准库中，`Lock` 接口区分了 `lock()`、`lockInterruptibly()`、`tryLock()` 和带超时的 `tryLock()`，这正是因为获取锁的等待语义需要被明确表达。手写自旋锁若只提供一个无限等待的 `lock()`，使用范围就非常窄。

## 自旋锁中的常见错误

第一类错误是用普通变量实现锁。普通变量既不能保证原子竞争，也不能保证跨线程可见性。即使用 `volatile`，也只是让读写更可见，不能把“检查再设置”变成原子操作。自旋锁获取阶段必须使用 CAS 或等价原子机制。

第二类错误是在临界区中执行阻塞操作。自旋锁的基本假设是锁很快释放，阻塞操作会直接破坏这个假设。典型危险包括等待 Future 结果、阻塞队列操作、网络或文件 I/O、调用外部服务、等待另一个锁、长时间 sleep、复杂日志输出和不可控回调。只要临界区可能停住，等待者就可能空转。

第三类错误是认为自旋锁天然更轻。自旋锁少了挂起唤醒，不代表总体成本低。高竞争下它可能消耗更多 CPU、造成更严重缓存争用，并拉高尾延迟。轻量与否取决于场景，不取决于代码行数。

第四类错误是忽略公平性和饥饿。简单 CAS 自旋锁没有先来后到保证。若某些线程必须在有限时间内获得锁，或者请求顺序具有业务意义，就不应使用无队列的自旋锁。即使业务不要求严格公平，也应评估长期竞争下是否会出现某些线程等待过久。

第五类错误是把自旋锁当成可重入锁。不可重入自旋锁在递归调用或方法互调时很容易让同一线程等待自己。代码评审时要特别关注持锁方法是否调用了其他可能获取同一把锁的方法。

第六类错误是没有退避。失败后立即 CAS，在线程少时看不出问题，在线程多时会形成争用风暴。至少应考虑 `Thread.onSpinWait()`，更复杂场景还要考虑退避、超时或自旋后阻塞。

第七类错误是锁保护范围不一致。有的代码路径加锁，有的路径直接访问共享状态；有的读加锁，有的读不加锁；有的写通过原子变量，有的写通过普通字段。锁的正确性来自所有访问者遵守同一协议，单个方法写得再严谨，也无法保护绕过协议的访问。

第八类错误是用自旋等待业务条件。等待队列非空、配置加载完成、任务结果返回、资源可用等业务条件，通常更适合条件队列、阻塞队列、Future、Latch 或其他同步器。用 `while (!ready) {}` 等待业务条件会浪费 CPU，还容易遗漏中断、超时和异常传播。

## 诊断自旋锁问题

自旋锁问题常见表现是 CPU 使用率异常升高、吞吐下降、延迟抖动、某些请求长时间不完成，或者线程 dump 中大量线程处于 `RUNNABLE`，栈顶集中在 CAS 循环、`Thread.onSpinWait()` 或某个自定义 `lock()` 方法。与阻塞锁不同，等待自旋锁的线程可能看起来仍在运行，所以不能只搜索 `BLOCKED` 或 `WAITING`。

排查时先确认锁持有者在哪里。若实现记录了 owner，可以在诊断日志中输出持有线程、持有时长和获取失败次数。没有这些信息时，只能通过线程栈、日志时间线和代码审计推断。简单自旋锁可观察性差，这是它不适合作为通用业务锁的原因之一。

第二步看临界区是否真的短。检查锁内是否有 I/O、阻塞等待、外部调用、复杂循环、对象分配热点、日志或可能触发类加载的路径。许多“偶尔 CPU 飙升”的问题，根因是某个异常路径在持锁时做了慢操作，或者锁内调用的函数后来被扩展，破坏了最初的短临界区假设。

第三步看竞争强度。统计锁获取失败次数、平均自旋次数、最大自旋次数、持有时间分布和线程数量。如果失败次数远高于成功次数，说明大量 CPU 花在无效竞争上。此时可考虑降低共享频率、拆分锁、分片状态、批量更新、改用阻塞锁，或者让请求排队串行化处理。

第四步看是否存在遗漏释放。检查所有临界区是否使用 `try/finally`，释放者是否可能因为异常、提前返回或取消路径跳过 `unlock()`。对自旋锁来说，遗漏释放往往表现为所有等待者无限自旋。若 CPU 很高而没有任何线程继续完成临界区工作，就要优先怀疑锁泄漏或持有者卡住。

第五步看中断和关闭。服务停止时，如果等待线程仍在自旋而不检查中断，关闭流程可能长时间无法结束。线程池中的任务如果在自旋锁上无限等待，也会占住工作线程，让后续任务无法执行。等待策略必须和生命周期管理一起检查。

诊断工具上，线程 dump 能定位循环栈，性能采样能看到热点方法，日志能还原持有时长和失败路径，压力测试能放大竞争窗口。自旋锁问题不要只看单次运行结果，因为竞争问题高度依赖时序。需要在不同线程数、不同负载和不同临界区耗时下观察趋势。

## 如何决定是否使用自旋锁

做选择时，可以先把问题拆成三个层次。第一层是互斥语义：是否真的需要独占临界区，还是只需要单变量原子更新、消息传递、不可变快照或并发容器。若不需要独占，就不要引入锁。第二层是等待语义：竞争失败时是短暂等待、可中断等待、限时等待、按顺序等待，还是需要等待业务条件。第三层是性能语义：优化目标是平均吞吐、尾延迟、CPU 占用、顺序公平，还是简化代码。

如果只是计数、状态位切换、引用替换，优先考虑原子类或 VarHandle 表达单变量状态转换。如果需要保护一段普通临界区，优先考虑 `synchronized` 或 `ReentrantLock`。如果需要等待条件，优先考虑 `Condition`、阻塞队列或其他同步器。如果需要降低热点竞争，优先考虑分片、局部累积、批处理或改变数据流。自旋锁通常排在这些选择之后。

只有在下列条件同时满足时，手写或显式使用自旋锁才比较有讨论价值：临界区极短；锁持有者不会阻塞；竞争线程数有限；等待失败可以快速重试；不需要严格公平；能够接受不可重入或已明确处理重入；有中断、超时或退避边界；有压测数据证明它优于标准锁；有足够诊断信息定位问题。缺少其中任何几项，都应谨慎。

还要评估维护成本。自旋锁代码短，但语义复杂。团队成员是否都理解内存模型、CAS、退避和中断？未来维护者是否可能在临界区中加入慢操作？是否有测试覆盖高竞争和异常路径？如果答案不确定，使用标准同步工具往往更可靠。并发代码的最佳实践不是追求最底层，而是选择能准确表达需求、边界清晰、可诊断、可维护的工具。

## 与其他线程安全主题的关系

自旋锁和悲观锁、乐观锁、公平锁、可重入锁、CAS、AQS、volatile 等主题相互关联。悲观锁和乐观锁描述的是对冲突的预期和处理方式；自旋锁描述的是等待方式；公平锁描述的是竞争者顺序；可重入锁描述的是同一线程重复获取的语义；CAS 提供原子更新能力；volatile 提供可见性和有序性约束；AQS 提供同步队列框架。它们不是同一维度的分类，不能简单互相替代。

例如，自旋锁常用 CAS 获取，因而带有乐观尝试色彩，但它一旦获取成功，仍然是独占临界区的锁。`ReentrantLock` 可以是非公平的，也可以是公平的，但它不是纯自旋锁。`volatile` 可以让状态变化可见，却不能独立实现互斥。AQS 内部可能先尝试快速 CAS，再排队阻塞，这说明成熟同步器往往组合多种策略。

学习自旋锁的价值，正是在这些维度之间建立清晰边界。看到一个并发工具时，应问：它用什么状态表示占用？用什么原子操作改变状态？竞争失败时自旋、排队还是阻塞？释放时如何唤醒或发布？是否公平？是否可重入？是否响应中断？是否支持超时？是否保证可见性？这些问题比记住某个单一结论更重要。

## 小结

自旋锁用持续尝试换取避免线程挂起和唤醒，适合临界区极短、锁很快释放、竞争可控且 CPU 资源充足的场景。它通常依赖 CAS 完成原子竞争，并依赖正确的内存语义保证临界区写入对后续持有者可见。自旋等待可以降低短等待下的调度成本，但会消耗 CPU，制造缓存一致性压力，并在高竞争、长临界区或锁持有者被阻塞时迅速恶化。

自旋锁不是通用线程安全方案，也不是阻塞锁的天然替代品。它缺省往往不公平、不可重入、不响应中断、不支持超时，且诊断性较差。退避、自旋后阻塞、所有者校验、中断处理和超时控制能改善边界，但也会把实现推向成熟同步器的复杂度。对多数 Java 程序来说，优先选择标准并发库中语义明确的工具；只有在明确热点、明确约束并经过测量验证时，才考虑显式使用自旋锁。

真正掌握自旋锁，不是会写一个 `while (!compareAndSet(...))`，而是能解释它为什么可能快、为什么可能慢、它依赖哪些内存语义、它在哪些条件下成立、竞争失败时如何控制成本、异常和取消路径如何收尾，以及什么时候应该放弃自旋改用阻塞、排队或重新设计共享状态。

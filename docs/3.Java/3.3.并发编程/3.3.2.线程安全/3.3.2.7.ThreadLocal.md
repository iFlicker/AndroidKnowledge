# 3.3.2.7 ThreadLocal

`ThreadLocal<T>` 是 Java 提供的线程局部变量工具。它允许同一个 `ThreadLocal` 实例在不同线程中关联不同的值：线程 A 调用 `get()` 得到 A 自己绑定的值，线程 B 对同一实例调用 `get()` 得到 B 自己绑定的值，两者通常互不影响。它不是通过加锁协调多个线程访问同一个对象，而是把状态按线程分区，使每个线程只访问属于自己的那一份关联。

这种机制适合保存“与当前执行线程绑定、需要跨越多层调用、又不希望逐层显式传参”的上下文，例如一次同步调用链中的追踪标识、格式化器、事务边界或诊断信息。但它也会把依赖从方法参数隐藏到线程状态中，并把数据生命周期绑定到线程而不是任务。在线程池、异步任务和长期存活线程中，后一个特征尤其容易导致上下文污染、对象滞留和错误传播。

因此，理解 `ThreadLocal` 不能停留在“每个线程一份副本”这句话上。更准确的问题是：值实际存在哪里，何时创建，何时替换，何时清除；弱引用为什么不能自动解决泄漏；线程复用为何会破坏任务隔离；跨线程执行时上下文为何不会自然传播；以及什么时候直接传参、不可变上下文对象或其他作用域机制更合适。

## 从共享变量到线程局部状态

并发程序中的状态大致可以分为三类：

1. 方法局部状态只存在于一次调用中，由调用栈自然隔离。
2. 共享状态会被多个线程访问，需要不可变设计、互斥、原子操作或其他同步协议。
3. 线程局部状态在同一线程的多层调用之间共享，但不同线程分别持有自己的关联。

`ThreadLocal` 处理的是第三类。它没有把一个可变对象复制成若干份，也不会自动克隆传入的值。真正发生的是：每个 `Thread` 内部都有用于保存线程局部关联的表，同一个 `ThreadLocal` 对象充当查找键，各线程可以在自己的表中为这个键保存不同的 value。

设有一个静态字段 `CURRENT`，两个线程分别执行 `CURRENT.set("A")` 和 `CURRENT.set("B")`。静态字段意味着两个线程拿到的是同一个 `ThreadLocal` 对象，但 value 不保存在这个静态对象自身的普通实例字段里，而是分别进入两个线程各自的映射表。因此，`ThreadLocal` 实例可以安全地被多个线程用作键；各线程绑定的 value 是否安全，则取决于 value 自身是否又被跨线程共享。

“每个线程一份副本”只是方便理解的近似说法，容易产生两个误解：

- `ThreadLocal` 不会复制对象。两个线程完全可以把同一个可变对象都 `set` 进去，此时仍然存在共享状态和数据竞争。
- 隔离单位是线程，不是请求、任务、方法调用或业务会话。在线程池中，多个任务可以先后运行在同一工作线程上，因此会看到同一个线程槽位。

下面的示例展示最基本的隔离语义。两个线程共享同一个 `ThreadLocal<Integer>`，但各自维护自己的计数：

```java
public final class LocalCounterDemo {
    private static final ThreadLocal<Integer> COUNTER =
            ThreadLocal.withInitial(() -> 0);

    private static void incrementTwice() {
        COUNTER.set(COUNTER.get() + 1);
        COUNTER.set(COUNTER.get() + 1);
        System.out.println(Thread.currentThread().getName()
                + ": " + COUNTER.get());
    }

    public static void main(String[] args) throws InterruptedException {
        Thread first = new Thread(LocalCounterDemo::runTask, "first");
        Thread second = new Thread(LocalCounterDemo::runTask, "second");
        first.start();
        second.start();
        first.join();
        second.join();
    }

    private static void runTask() {
        try {
            incrementTwice();
        } finally {
            COUNTER.remove();
        }
    }
}
```

两个线程通常都会输出 `2`，而不是共同累加成 `4`。这里没有使用锁，因为计数值没有在线程之间共享。示例仍然在 `finally` 中调用 `remove()`，目的不是保证加法原子性，而是让值的生命周期在任务结束时闭合。

## 它解决什么问题

`ThreadLocal` 的主要价值不是“让任意代码线程安全”，而是降低特定线程封闭状态的传递成本。所谓线程封闭，是指可变对象只被一个线程访问。只要封闭边界不被破坏，该对象不需要为了与其他线程竞争而加锁。

### 跨层调用中的隐式上下文

一个入口方法可能经过服务、仓储、日志等多层调用。如果每层都需要读取同一份只读上下文，最清晰的方式通常是显式参数传递；但当调用层次很深、上下文属于横切关注点、并且大量中间方法只负责转交时，参数会扩散到许多接口。`ThreadLocal` 可以在入口处安装上下文，在深层代码中读取，在出口处清理。

这种便利有明确代价：方法签名不再完整表达依赖。一个看似只接收业务参数的方法，实际还要求当前线程预先安装某个值。测试、复用和异步改造时都必须知道这条隐藏约定。因此，`ThreadLocal` 更适合基础设施性质、作用域清楚的上下文，而不是普通业务数据的默认传递方式。

### 复用非线程安全但可重复使用的对象

某些对象创建成本较高且实例本身不支持并发访问，可以为每个线程保存一个实例，从而避免全局锁。例如旧式日期格式化器常被这样封装。这个策略只有在以下条件同时成立时才合理：

- 对象确实不能安全共享；
- 对象适合长期复用，内部状态能够在每次使用后恢复；
- 线程数量有边界，否则每个线程一份对象会放大内存占用；
- 线程长期存活时，保留该对象是有意为之，而不是遗漏清理。

如果对象创建便宜，局部创建通常更简单。如果已有不可变或线程安全实现，优先直接使用它。为了避免一次小对象创建而引入隐藏状态，往往得不偿失。

### 维护线程内的嵌套状态

有些基础设施需要知道当前线程是否已经进入某个作用域，例如重入深度、当前资源绑定或诊断标签。`ThreadLocal` 可以保存计数器、栈或上下文对象，使同一线程内的嵌套调用共享状态。

这种用法的关键不是 `set`，而是正确恢复先前状态。嵌套调用如果在退出时直接 `remove()`，可能把外层作用域安装的值一并删除；如果只覆盖不恢复，又会把内层值泄漏给外层。后文会给出保存旧值再恢复的作用域写法。

## 核心 API 与准确语义

`ThreadLocal` 的公开 API 很少，但几个方法的边界必须分清。

### `get()`

`get()` 返回当前线程与该 `ThreadLocal` 关联的值。如果当前线程还没有对应条目，`get()` 会调用初始化逻辑，随后把初始化结果保存到当前线程的映射中并返回。

默认的 `initialValue()` 返回 `null`。因此，`get()` 返回 `null` 可能有两种含义：尚未设置且默认值为 `null`，或者调用者明确执行过 `set(null)`。如果业务必须区分“未安装”和“已安装空值”，就不应只用 `null` 表达状态，可以使用专门的哨兵对象、包装类型，或设计禁止空值的上下文 API。

初始化是按线程发生的，不是整个 `ThreadLocal` 只执行一次。十个线程首次调用 `get()`，初始化函数最多会分别在这十个线程中执行。初始化函数在当前调用线程里同步执行；如果它耗时、阻塞或抛出异常，影响的也是本次 `get()` 调用。

### `initialValue()` 与 `withInitial()`

可以通过创建子类并覆盖 `initialValue()` 提供初始值，也可以使用 `ThreadLocal.withInitial(Supplier<? extends T>)`：

```java
public final class PerThreadBuffer {
    private static final ThreadLocal<StringBuilder> BUFFER =
            ThreadLocal.withInitial(() -> new StringBuilder(256));

    public static String buildLine(String prefix, int value) {
        StringBuilder builder = BUFFER.get();
        builder.setLength(0);
        builder.append(prefix).append('=').append(value);
        return builder.toString();
    }

    private PerThreadBuffer() {
    }
}
```

这里每个首次访问的线程创建自己的 `StringBuilder`。复用前必须重置内部状态，否则同一线程的后续调用会继承上次内容。这个例子还体现了长期缓存的权衡：如果某次构建使缓冲区扩容到很大容量，之后即使内容被清空，大数组仍可能随线程长期保留。对于大小波动显著的数据，局部创建或设置容量上限可能更稳妥。

`withInitial` 的 `Supplier` 不应依赖会变化的共享可变状态，除非调用方另有同步协议。它也不适合返回一个全局单例可变对象，否则不同线程虽然拥有不同表项，value 却仍指向同一个对象，隔离形同虚设。

### `set(T value)`

`set()` 将 value 与当前线程关联。当前线程已有同一个 key 的条目时会替换旧 value；没有时会创建新条目。操作只影响当前线程，不会修改其他线程的关联。

`set(null)` 与 `remove()` 不等价。`set(null)` 通常仍保留一个 key 存在、value 为 `null` 的条目；`remove()` 则移除当前线程表中的对应条目。需要结束生命周期时应调用 `remove()`，而不是用 `set(null)` 模拟清理。

### `remove()`

`remove()` 删除当前线程中与该 `ThreadLocal` 对应的条目。删除后再次调用 `get()`，初始化逻辑可以在当前线程中再次执行。

`remove()` 只清理调用它的线程。线程 A 不能通过调用 `remove()` 清理线程 B 的表项。因此，设置值的执行路径必须对同一线程上的清理负责。把清理提交到另一个线程，或者期待管理线程统一移除所有工作线程的值，都不符合 API 的作用边界。

截至 Java 25，`ThreadLocal` 的公共 API 没有提供“判断当前线程是否已有条目且不触发初始化”的方法。若业务必须区分“未安装”和“已安装空值”，应在自己的封装层中禁止空值、使用哨兵对象，或把存在状态与实际值一起封装；不要依赖内部 map 或反射探测。

## 值究竟存在哪里

理解内部结构是判断内存风险的基础。在 OpenJDK 的典型实现中，`Thread` 对象持有两个与线程局部变量相关的字段：一个服务于普通 `ThreadLocal`，另一个服务于 `InheritableThreadLocal`。普通 `ThreadLocal` 的关系可以概括为：

```text
Thread
  └── threadLocals: ThreadLocalMap
        └── table: Entry[]
              ├── Entry.key  --弱引用--> ThreadLocal
              └── Entry.value --强引用--> Object
```

这里有三个重要结论。

第一，map 属于线程，不属于 `ThreadLocal`。如果一个线程使用了二十个不同的 `ThreadLocal`，它们作为 key 共同存放在这个线程的一张表中。调用 `local.get()` 时，代码先取得当前线程，再从当前线程的 map 中查找 `local`。

第二，`ThreadLocalMap` 是为该场景定制的内部结构，不是一般用途的 `Map`。应用代码不能直接遍历或管理它。它采用数组和开放寻址处理冲突，没有把每个桶实现成链表或树。

第三，Entry 对 key 是弱引用，对 value 是强引用。key 变成不可达后可能被垃圾回收，Entry 的弱引用读取结果变为 `null`；但 value 不会因此自动变成弱引用或立刻从数组删除。key 为 `null` 而 value 仍存在的条目通常称为陈旧条目。

### 为什么 key 使用弱引用

假设业务创建了一个短生命周期的 `ThreadLocal` 对象，把它作为 key 存入长期线程的 map，随后业务丢失了对该 key 的所有强引用。如果 Entry 强引用 key，那么只要线程存活，map 就会让 key 一直存活；应用也无法再取得这个 key 来调用 `remove()`，关联会永久困在线程中。

弱引用至少允许 key 在外部强引用消失后被回收，并让 map 在后续操作中识别陈旧条目、清理其 value。它是一种降低风险并支持惰性清理的设计，不是完整的资源管理协议。

### 为什么 value 不能也简单改成弱引用

`ThreadLocal` 的契约要求：只要当前线程保留该关联，调用 `get()` 应能稳定取得 value。若 value 只被 map 弱引用，即使程序仍持有 `ThreadLocal` key，value 也可能在任意一次垃圾回收后消失，破坏存储语义。value 使用强引用是功能要求，生命周期必须由使用者通过作用域和 `remove()` 管理。

## 散列、冲突与惰性清理

每个 `ThreadLocal` 实例拥有一个内部散列码。新实例按固定增量生成散列值，使连续创建的 key 在长度为二次幂的表中尽量均匀分布。查找初始位置时，通常通过散列码与 `table.length - 1` 做按位与计算索引。

当目标位置已被其他 key 占用时，`ThreadLocalMap` 使用线性探测：沿数组向后寻找目标 key、空槽或陈旧条目，到达末尾后再从头继续。线性探测让数组结构紧凑，但删除不能简单把某个位置设空后结束，因为这可能截断同一探测簇，使后面的有效条目再也无法被找到。实现需要重新扫描和整理受影响区间。

`get()`、`set()` 和 `remove()` 的具体实现会在不同条件下触发陈旧条目清理。例如，查找过程中遇到 key 已经为 `null` 的 Entry，会清除其 value，并重新整理后续条目；插入时也可能替换陈旧槽位并执行启发式扫描。表达到阈值时，实现在扩容前还会尝试清理陈旧条目，只有清理后有效条目仍较多才扩容。

由此可以得到一个容易被忽略的边界：清理是惰性的，而且依赖该线程后续对 `ThreadLocalMap` 的操作。某个长期线程如果留下陈旧条目后很少再访问相关 map，value 就可能长时间保留。不能把“实现会顺便清理”当成确定的及时释放保证。

内部散列与清理算法属于具体 JDK 实现细节，不应被业务代码依赖。业务需要依赖的稳定原则只有两个：关联由线程持有；使用结束后在同一线程显式 `remove()`。

## 内存滞留与真正的风险链

讨论 `ThreadLocal` 时经常笼统地说“会内存泄漏”。更准确的说法是：如果线程比任务活得久，而任务安装的 value 没有移除，那么线程会继续强引用该 value，造成超出预期生命周期的对象滞留；如果线程被无限期保留，这种滞留就具有泄漏特征。

典型引用链为：

```text
GC Root
  -> 活跃线程或线程池
  -> Thread
  -> ThreadLocalMap
  -> Entry
  -> value
```

即使 Entry 的 key 已被回收为 `null`，从 `Thread` 到 value 的强引用链仍然存在。只要没有后续 map 操作触发清理、线程也没有终止，value 就无法回收。

不过，key 没有变成 `null` 也不意味着没有问题。实践中常把 `ThreadLocal` 声明为 `static final`，key 会随类长期存活，不会形成“key 消失”的陈旧条目，但 value 依然可能因为忘记 `remove()` 而一直绑定在线程上。此时风险不依赖弱引用机制，而是最直接的生命周期错误。

内存不是唯一后果。比堆占用更隐蔽的是上下文污染：线程池工作线程执行任务 A 时留下用户或追踪信息，稍后同一线程执行任务 B，B 在未初始化上下文的路径上读到了 A 的值。此时程序可能不报错，却产生错误日志、错误权限判断或错误路由。因而清理既是内存管理，也是正确性要求。

### 类加载器滞留

在支持动态部署或插件加载的 Java 进程中，value 的类可能由可卸载的类加载器定义。长期线程通过 `ThreadLocalMap` 持有 value，就会间接持有该类及其类加载器，使整批类元数据和相关静态状态无法卸载。此时堆转储中应沿 value 到 GC Root 的路径检查线程局部关联，而不能只统计 value 本身的大小。

### 短命线程与长期线程的差异

如果程序创建普通短命线程，在线程终止且 `Thread` 对象本身不再可达后，它持有的 map 和 value 都可以一起回收。忘记 `remove()` 的影响可能较短，但仍不应把线程终止当成清理策略，因为代码未来可能被迁移到线程池。

在线程池中，工作线程为复用而长期存活。任务结束不等于线程结束，所以任务安装的 value 也不会随栈帧返回而自然消失。`ThreadLocal` 的生命周期按线程计算，业务通常却希望按任务计算，这正是风险的根源。

## 正确建立生命周期

最稳妥的基本模式是：在明确的入口安装值，在覆盖所有退出路径的 `finally` 中移除。安装和清理必须在同一线程执行。

```java
public final class RequestContextHolder {
    private static final ThreadLocal<RequestContext> CURRENT =
            new ThreadLocal<>();

    public static RequestContext current() {
        RequestContext context = CURRENT.get();
        if (context == null) {
            throw new IllegalStateException("Request context is not installed");
        }
        return context;
    }

    public static void runWith(
            RequestContext context,
            Runnable action) {
        if (context == null || action == null) {
            throw new NullPointerException();
        }

        CURRENT.set(context);
        try {
            action.run();
        } finally {
            CURRENT.remove();
        }
    }

    public static final class RequestContext {
        private final String traceId;
        private final String operator;

        public RequestContext(String traceId, String operator) {
            this.traceId = traceId;
            this.operator = operator;
        }

        public String traceId() {
            return traceId;
        }

        public String operator() {
            return operator;
        }
    }

    private RequestContextHolder() {
    }
}
```

`runWith` 把安装、执行和清理封装在一个 API 内，调用方不容易忘记收尾。上下文使用不可变 `record`，避免深层代码修改同一线程内所有调用者看到的状态。

这个简单版本不支持嵌套作用域。若外层已安装上下文，内层再次调用 `runWith`，内层退出时执行 `remove()` 会删除整个关联，外层随后无法恢复自己的值。需要允许嵌套时，应保存旧状态并在退出时恢复：

```java
public final class NestedContext {
    private static final ThreadLocal<String> CURRENT = new ThreadLocal<>();

    public static void runWith(String value, Runnable action) {
        String previous = CURRENT.get();
        boolean hadPrevious = previous != null;

        CURRENT.set(value);
        try {
            action.run();
        } finally {
            if (hadPrevious) {
                CURRENT.set(previous);
            } else {
                CURRENT.remove();
            }
        }
    }

    public static String current() {
        return CURRENT.get();
    }

    private NestedContext() {
    }
}
```

如果合法旧值可以是 `null`，仅靠 `previous != null` 无法判断此前是否存在条目。由于公共 API 不提供不触发初始化的存在性判断，可以禁止空值、使用哨兵值，或把“是否安装”与实际 value 包装在一个非空对象中。

### 谁设置，谁清理

清理责任应由建立关联的作用域承担，而不是由任意读取者承担。深层方法如果只调用了 `get()`，通常不应擅自 `remove()`，否则会破坏上层作用域。相反，入口代码调用了 `set()`，就必须保证所有正常返回、异常和提前退出路径最终执行清理。

同一个方法多次覆盖值时，也要明确语义：这是重新定义当前作用域，还是进入嵌套作用域？前者可以最终移除，后者需要恢复旧值。没有统一规则时，最容易出现某层清掉别人的值或把临时值带出作用域。

### `AutoCloseable` 作用域

也可以用 `AutoCloseable` 表达安装与恢复，使调用点采用 try-with-resources。不过 `close()` 必须在创建作用域的同一线程执行，而且作用域对象通常只能关闭一次：

```java
public final class ContextScope implements AutoCloseable {
    private static final ThreadLocal<String> CURRENT = new ThreadLocal<>();

    private final String previous;
    private final boolean hadPrevious;
    private final Thread owner;
    private boolean closed;

    private ContextScope(String value) {
        this.previous = CURRENT.get();
        this.hadPrevious = previous != null;
        this.owner = Thread.currentThread();
        CURRENT.set(value);
    }

    public static ContextScope open(String value) {
        return new ContextScope(value);
    }

    public static String current() {
        return CURRENT.get();
    }

    @Override
    public void close() {
        if (Thread.currentThread() != owner) {
            throw new IllegalStateException("Scope closed by another thread");
        }
        if (closed) {
            return;
        }
        closed = true;

        if (hadPrevious) {
            CURRENT.set(previous);
        } else {
            CURRENT.remove();
        }
    }
}
```

调用方可以写成：

```java
try (ContextScope ignored = ContextScope.open("trace-42")) {
    doWork();
}
```

这种封装提升了结构化程度，但它仍是基于线程身份的动态作用域。若 `doWork()` 把工作提交到另一个线程，子任务不会因为作用域对象尚未关闭就自动获得值。

## 线程池复用：最常见的错误现场

下面的代码故意遗漏清理。固定大小为一的线程池确保两个任务复用同一个工作线程，因此第二个任务可能读到第一个任务遗留的值：

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public final class ContextLeakDemo {
    private static final ThreadLocal<String> USER = new ThreadLocal<>();

    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newFixedThreadPool(1);
        try {
            executor.submit(() -> USER.set("alice")).get();
            executor.submit(() ->
                    System.out.println("unexpected user: " + USER.get())
            ).get();
        } finally {
            executor.shutdown();
        }
    }
}
```

问题不在于 `ExecutorService` 的实现错误，而在于代码把任务级数据存进线程级容器，却没有在任务边界清除。正确做法是在第一个任务自身的 `finally` 中移除：

```java
executor.submit(() -> {
    USER.set("alice");
    try {
        process();
    } finally {
        USER.remove();
    }
});
```

不能只在调用 `submit()` 的线程中 `remove()`。提交线程和执行线程通常不是同一个线程，两边拥有不同的 `ThreadLocalMap`。同理，在 `Future.get()` 返回后由调用者移除，也只能移除调用者自己的关联。

### 在线程池钩子中统一清理是否足够

自定义 `ThreadPoolExecutor` 可以覆盖 `beforeExecute`、`afterExecute`，或统一包装 `Runnable`，在任务边界清理已知的 `ThreadLocal`。这种基础设施可以作为防御措施，但不能替代拥有者的正常清理，原因包括：

- 执行器只能清理它知道的变量，无法通过公共 API 枚举所有业务 `ThreadLocal`；
- 粗暴清理可能删除线程池框架或外层作用域有意保留的状态；
- 任务可能在另一个执行器、手工线程或直接调用路径中运行；
- 若只在任务开始前清理，可以避免前任务污染，却仍会让当前任务的 value 保留到下一任务开始；
- 若异常处理或包装层设计不完整，清理仍可能遗漏。

更可靠的分工是：业务作用域用 `try/finally` 管理自己的值；执行器边界可额外检查或清理明确约定的基础设施上下文。

## `InheritableThreadLocal`：创建线程时的快照式继承

普通 `ThreadLocal` 的值不会自动出现在另一个线程中。`InheritableThreadLocal<T>` 提供有限的父子线程继承能力：创建子线程时，运行时会根据父线程的 inheritable map 为子线程建立初始关联。默认 `childValue(parentValue)` 直接返回父值，也可以通过覆盖该方法进行转换或复制。

```java
public final class InheritedContextDemo {
    private static final InheritableThreadLocal<String> CONTEXT =
            new InheritableThreadLocal<>();

    public static void main(String[] args) throws InterruptedException {
        CONTEXT.set("parent-value");
        try {
            Thread child = new Thread(() ->
                    System.out.println(CONTEXT.get()));
            child.start();
            child.join();
        } finally {
            CONTEXT.remove();
        }
    }
}
```

这里的继承发生在线程创建阶段，不是每次任务提交时动态同步。父线程之后修改 value，已经创建的子线程不会自动跟随；子线程修改自己的映射，也不会自动回写父线程。

默认继承的往往只是同一个对象引用，不是深拷贝。如果 value 是可变对象，父子线程同时修改它仍然会发生数据竞争。若覆盖 `childValue()` 创建副本，需要定义清楚复制深度、成本和一致性，不能把“复制了最外层对象”误认为整个对象图都已隔离。

### 为什么它不适合普通线程池上下文传播

线程池工作线程通常在业务任务提交前就已创建。任务提交者与工作线程也不一定有父子关系。因此，提交任务时设置 `InheritableThreadLocal`，不能保证已有工作线程获得新值。更糟的是，工作线程可能保留创建它时继承的旧值，导致行为看似随机。

虚拟线程同样不改变 `ThreadLocal` 以线程为隔离单位这一事实。每个虚拟线程可以拥有自己的线程局部变量，但大量虚拟线程会放大“每线程保存一份状态”的总成本。对于是否继承 inheritable thread-local，`Thread.Builder` 还提供相关配置；代码不应依赖未经明确设计的默认传播。

## 跨线程和异步边界

只要执行可能切换线程，普通 `ThreadLocal` 上下文就不会自然跟随。常见边界包括：

- 向 `Executor` 或 `ExecutorService` 提交任务；
- `CompletableFuture` 的异步阶段；
- 并行流；
- 手工创建新线程；
- 定时任务；
- 由第三方框架决定执行线程的回调。

如果业务必须传播上下文，常见做法是在提交时捕获一个不可变快照，在执行线程中安装，执行结束后恢复原状态。下面是只传播一个已知上下文的简化包装器：

```java
import java.util.concurrent.Executor;

public final class ContextAwareExecutor implements Executor {
    private final Executor delegate;

    public ContextAwareExecutor(Executor delegate) {
        this.delegate = delegate;
    }

    @Override
    public void execute(Runnable command) {
        String captured = TraceContext.currentOrNull();
        delegate.execute(() -> {
            String previous = TraceContext.currentOrNull();
            try {
                TraceContext.setOrRemove(captured);
                command.run();
            } finally {
                TraceContext.setOrRemove(previous);
            }
        });
    }

    static final class TraceContext {
        private static final ThreadLocal<String> TRACE = new ThreadLocal<>();

        static String currentOrNull() {
            return TRACE.get();
        }

        static void setOrRemove(String value) {
            if (value == null) {
                TRACE.remove();
            } else {
                TRACE.set(value);
            }
        }
    }
}
```

这段代码强调“捕获、安装、执行、恢复”四个阶段。恢复而不是一律移除，是为了不破坏执行线程原先可能存在的合法状态。不过，它把 `null` 当作未安装标记，所以不支持“已安装但值为 null”的语义。

上下文传播不只是技术复制，还涉及业务选择。并非所有线程局部值都应该传播：安全凭证、事务资源、可变会话和大型缓存可能不适合进入后台任务。传播框架如果无差别复制所有值，会把隐藏依赖和生命周期问题扩大。更稳妥的策略是白名单化，只捕获明确设计为可传播且最好不可变的上下文字段。

对 `CompletableFuture` 而言，还必须关注每个阶段是否使用 `thenApply` 还是 `thenApplyAsync`、是否指定执行器。非异步阶段可能由完成前一阶段的线程执行，异步阶段可能使用公共池或指定执行器。依赖“碰巧仍在原线程”读取 `ThreadLocal`，会使行为随完成时机和调度方式变化。

## 与 Java 内存模型的关系

`ThreadLocal` 通过线程封闭减少共享，但它不是跨线程发布机制，也不会自动建立额外的 happens-before 关系。

如果线程 A 在自己的 `ThreadLocal` 中保存对象，线程 B 无法通过同一个 `ThreadLocal.get()` 取得 A 的条目。这避免了通过该入口共享，但对象仍可能通过静态字段、队列、回调或其他引用传给 B。一旦对象跨线程，仍需使用安全发布和同步协议。它曾经放在 `ThreadLocal` 中，不会赋予对象线程安全属性。

反过来，线程 A 捕获上下文对象并把包装任务提交到线程安全的执行器时，任务提交与执行器内部同步通常负责把任务及其捕获字段安全发布给工作线程。真正建立可见性的，是执行器和并发库的同步语义，不是 `ThreadLocal.set()`。工作线程随后把快照安装到自己的 map，只是建立该线程内的访问入口。

下面几种说法都不成立：

- “对象放进 `ThreadLocal` 后内部字段自动可见。”
- “两个线程使用同一个 `ThreadLocal`，所以它们可以交换数据。”
- “`ThreadLocal` 可以替代 `volatile`。”
- “不加锁是因为 `ThreadLocal` 内部实现了同步。”

不加锁的根本原因是各线程通常只修改自己的 map；一旦 value 本身被共享，锁、原子类、不可变对象等规则仍然适用。

## 典型设计案例：调用链追踪上下文

一个较合理的使用场景是同步调用链中的只读追踪上下文。入口生成或解析追踪标识，深层日志代码读取，出口清理。设计时应满足以下约束：

1. 上下文对象不可变，深层代码只能读取。
2. 未安装时快速失败或明确返回空值，不静默生成错误身份。
3. 安装 API 同时定义清理作用域。
4. 异步提交显式捕获需要传播的字段。
5. 不把大对象、连接、流或生命周期复杂的资源塞入上下文。

```java
public final class TraceContext {
    private static final ThreadLocal<State> CURRENT = new ThreadLocal<>();

    public static State requireCurrent() {
        State state = CURRENT.get();
        if (state == null) {
            throw new IllegalStateException("Trace context is missing");
        }
        return state;
    }

    public static void invoke(State state, Runnable action) {
        State previous = CURRENT.get();
        CURRENT.set(state);
        try {
            action.run();
        } finally {
            if (previous == null) {
                CURRENT.remove();
            } else {
                CURRENT.set(previous);
            }
        }
    }

    public static final class State {
        private final String traceId;
        private final String operation;

        public State(String traceId, String operation) {
            if (traceId == null || traceId.isBlank()) {
                throw new IllegalArgumentException("traceId is blank");
            }
            if (operation == null || operation.isBlank()) {
                throw new IllegalArgumentException("operation is blank");
            }
            this.traceId = traceId;
            this.operation = operation;
        }

        public String traceId() {
            return traceId;
        }

        public String operation() {
            return operation;
        }
    }

    private TraceContext() {
    }
}
```

这类 holder 应保持很小，只暴露受控的安装和读取方法。若直接公开 `ThreadLocal` 字段，任意代码都可以 `set`、`remove` 或保存可变值，所有权会迅速失控。

即使设计正确，也要问一个更基本的问题：追踪上下文是否真的不能显式传递？如果核心业务方法本来就需要知道调用身份，把上下文放进参数往往更清楚。`ThreadLocal` 更适合日志、诊断等横切读取，而不适合隐藏业务授权、计费归属等关键决策输入。

## 常见误区

### 误区一：`ThreadLocal` 让共享对象变得线程安全

如果每个线程 `set` 的是独立对象，线程封闭可以避免竞争；如果多个线程 `set` 的是同一个对象，则对象仍被共享。例如把全局 `ArrayList` 同时放入各线程的 `ThreadLocal`，不会让列表的并发修改变安全。判断标准是 value 的对象身份和可达路径，而不是外面包了什么容器。

### 误区二：key 是弱引用，所以不需要 `remove()`

弱引用只允许失去外部强引用的 key 被回收。value 仍被 Entry 强引用，清理又是惰性的。更常见的 `static final ThreadLocal` key 根本不会消失，但 value 仍会因线程长期存活而滞留。显式清理是生命周期协议，不能交给弱引用替代。

### 误区三：把 `ThreadLocal` 字段改成非静态就能避免泄漏

字段是否静态决定 key 的持有方式，不决定线程 map 是否强引用 value。短命 key 甚至更容易变成应用无法访问的陈旧条目。通常将 holder 设计成单一、长期可达的 `static final` key 更便于管理，但每次任务仍必须清理 value。

### 误区四：`set(null)` 等同于 `remove()`

`set(null)` 表示关联值为 `null`，通常仍占据表项；`remove()` 才删除关联。需要释放 value、减少表项并允许下次重新初始化时，应使用 `remove()`。

### 误区五：在调用线程清理工作线程的值

`remove()` 只作用于当前线程。哪个线程执行了 `set()`，就应在哪个线程的 `finally` 中清理。调用方在线程池外执行 `remove()`，不会触碰工作线程。

### 误区六：任务结束后值自然消失

方法局部变量会随栈帧退出失去引用，但 `ThreadLocal` value 存在线程对象持有的 map 中。线程池任务结束只归还工作线程，不销毁线程。两种生命周期不能混为一谈。

### 误区七：`InheritableThreadLocal` 能解决任意异步传播

它主要描述父线程创建子线程时的初始继承。线程池先创建线程后提交任务，不符合这个时序。即使新建子线程，默认继承可变对象引用也可能引入共享竞争。异步传播应显式定义捕获时点、复制内容、安装和恢复规则。

### 误区八：为减少创建开销，应把所有临时对象放进 `ThreadLocal`

每线程缓存会把对象数量乘以线程数量，并可能把偶发峰值容量保留到线程结束。现代 JVM 对短命小对象的分配和回收通常很高效，不应在没有测量的情况下用隐藏的长期状态交换微小分配成本。先评估对象创建是否真是瓶颈，再考虑有上限的复用策略。

### 误区九：把连接、文件句柄等资源长期绑定到线程

线程并不一定对应稳定的业务会话，线程池线程还会执行不同任务。把稀缺资源放进 `ThreadLocal` 容易模糊借用、归还、失效和异常关闭责任。资源通常应由明确的作用域管理，并在 `finally` 或 try-with-resources 中释放；除非框架已经建立了严格的线程绑定协议，否则不要自行延长资源生命周期。

## 何时应该使用，何时不该使用

一个适合 `ThreadLocal` 的场景通常同时具备这些特征：

- 数据只需要在同一线程的动态调用链中访问；
- 多层代码都需要读取，但逐层传参会显著污染基础设施接口；
- 上下文有清楚的安装和退出边界；
- value 较小、最好不可变；
- 跨线程时可以不传播，或有明确的捕获和恢复机制；
- 线程复用条件下能够保证清理。

下面几类需求通常应优先选择其他方式。

### 普通业务输入：显式参数

如果数据决定业务结果，方法签名应尽量表达这项依赖。显式参数便于测试、重构、异步执行和静态分析，也不会依赖当前线程身份。参数较多时可以聚合成不可变上下文对象，而不是直接转入 `ThreadLocal`。

### 跨线程共享状态：同步或消息传递

多个线程需要读写同一状态时，应使用不可变快照、并发容器、锁、原子类或消息队列等符合不变式的工具。`ThreadLocal` 只能让每个线程看到自己的关联，不能完成共享结果合并。

### 词法作用域数据：局部变量和对象字段

如果数据只在一个方法或对象生命周期内使用，局部变量、参数和封装良好的字段更直接。`ThreadLocal` 的动态作用域意味着任何深层方法都可能读取它，依赖边界更宽，也更难从代码结构看出。

### 结构化只读上下文：`ScopedValue`

较新的 Java 提供 `ScopedValue`，用于在线程执行的有界动态作用域内共享不可变数据。它强调单向绑定和结构化生命周期，特别适合只读上下文，并与虚拟线程、结构化并发的设计方向一致。`ScopedValue` 在 Java 21 至 Java 24 经历预览阶段，并在 Java 25 成为正式特性；项目使用时必须根据实际 JDK 版本和兼容要求确认 API。

`ScopedValue` 不是对所有 `ThreadLocal` 用法的直接替换。需要线程内可变缓存、反复 `set` 的状态或兼容较早 JDK 时，`ThreadLocal` 仍有适用空间。但对于“进入作用域绑定一个不可变上下文，退出后自动失效”的需求，结构化作用域通常更容易推理。

## 性能与容量权衡

`ThreadLocal.get()` 的常见路径是当前线程取 map，再按散列位置查找 Entry，通常开销不高；但不能据此认为它没有成本。成本来自多个层面：

- 每个使用该变量的线程都可能创建一个 value；
- 每个线程的 map 需要数组空间，条目增多时会扩容；
- 散列冲突和陈旧条目会增加探测与清理成本；
- 初始化函数可能昂贵；
- 大量线程或虚拟线程会把每线程状态的占用累加；
- 隐式依赖增加维护、测试和传播复杂度。

如果有 `N` 个并发存活线程，每个线程缓存大小约为 `S` 的对象，仅 value 的理论占用就接近 `N × S`，还未计算对象图、map 和线程本身。固定平台线程池中 `N` 通常可控；按任务创建大量虚拟线程时，哪怕单个 value 不大，总量也可能显著。

性能优化应基于测量。若使用 `ThreadLocal` 缓存可重用缓冲区，需要同时观察分配率、GC、峰值容量、线程数量和任务延迟，而不是只比较一次 `new` 与一次 `get()`。对于可能异常膨胀的缓存，可以在使用后根据容量决定清空复用还是 `remove()` 丢弃。

## 测试 ThreadLocal 代码

单线程成功路径不足以验证 `ThreadLocal` 设计。测试至少应覆盖以下维度。

### 线程隔离

让两个线程为同一个 key 设置不同值，通过 `CountDownLatch` 或其他同步器控制它们同时存在，验证彼此读取不串扰。不要依赖 `sleep` 猜测执行时机。

### 线程池复用

使用单线程执行器连续提交任务：第一个任务设置并清理，第二个任务确认没有读到旧值。这是检测任务边界是否闭合的最直接测试。

### 异常路径

在安装值后故意抛出运行时异常，随后复用同一线程确认值已清除。只有正常路径测试无法证明 `finally` 是否覆盖所有出口。

### 嵌套恢复

外层安装 A，内层安装 B；内层执行时应读到 B，内层退出后外层应重新读到 A，最外层退出后应无值。这个测试能发现“内层一律 remove 导致外层丢失”的错误。

### 异步传播

分别测试直接调用、普通 `execute`、包装后的上下文执行器和 `CompletableFuture` 异步阶段，明确哪些路径应传播、哪些不应传播。测试要指定执行器，避免公共池状态干扰结论。

### 值的可变性

若上下文包含集合或可变对象，应验证是否可能被子任务共享修改。很多所谓“ThreadLocal 失效”其实是不同线程拿到了指向同一 value 的引用。

内存滞留测试通常不能只靠一次 `System.gc()` 和断言完成，因为垃圾回收没有即时执行保证。更可靠的方式是结合长时间重复任务、堆占用趋势、堆转储和 GC Root 路径分析。

## 排查上下文污染与内存滞留

遇到疑似 `ThreadLocal` 问题时，可以沿以下顺序缩小范围。

第一，确认实际执行线程。日志中记录线程名和任务标识，判断设置、读取、清理是否发生在同一线程，以及不同任务是否复用了工作线程。

第二，搜索所有 `set()`、`withInitial()` 和覆盖 `initialValue()` 的位置。每个安装点都应有对应作用域，特别检查异常、提前返回和取消路径。只搜索 `remove()` 数量是否相等不够，因为安装与清理可能不在同一线程或同一所有权层次。

第三，检查 value 对象图。即使 value 本身很小，它也可能引用缓存、类加载器、连接包装、集合或整个任务对象。堆转储应查看 retained size 和到 GC Root 的完整路径。

第四，检查 key 是否仍然存活。key 为 `null` 表明形成陈旧 Entry；key 非空则可能是长期 `static final` key 下未清理的合法条目。两种情况都会滞留 value，修复重点都是生命周期。

第五，检查异步包装是否恢复旧状态。有些传播器只负责安装捕获值，却在执行后直接 `remove()`；这会清掉工作线程原有上下文。有些传播器又只在成功路径恢复，异常后留下新值。正确包装必须在 `finally` 中恢复进入前状态。

第六，检查第三方库边界。库可能使用自己的线程局部状态，也可能要求调用特定清理 API。不要反射遍历并粗暴清空线程的全部 map，这依赖内部实现、破坏封装，还可能删除运行库所需状态。应通过公开协议管理已知上下文。

## 设计检查清单

引入一个新的 `ThreadLocal` 前，可以逐项回答：

1. 这项数据为什么不能作为显式参数或不可变上下文对象传递？
2. 隔离单位真的是线程，还是请求、任务、事务或其他作用域？
3. value 是否不可变；若可变，是否严格线程封闭？
4. 谁负责设置，谁负责清理，异常时是否仍能清理？
5. 是否允许嵌套；内层退出后应移除还是恢复外层值？
6. 在线程池中是否会复用线程，第二个任务能否读到前一个任务的值？
7. 是否需要跨线程传播；传播哪些字段，在哪个时点捕获？
8. value 的大小乘以最大线程数后是否可接受？
9. 初始化失败、空值和未安装状态分别如何表达？
10. 项目是否能使用更结构化的作用域机制？

如果这些问题没有明确答案，暂时不引入 `ThreadLocal` 通常比先使用再补清理更稳妥。

## 小结

`ThreadLocal` 的本质是以当前线程为索引保存关联，而不是复制对象，也不是为共享对象加锁。它通过线程封闭减少竞争，适合在同一线程调用链中保存边界清楚的上下文或线程专属状态。

它的核心风险来自生命周期错位：值存在线程中，业务却常按任务理解生命周期。在线程池和长期线程里，遗漏 `remove()` 会造成对象滞留和任务间污染；弱引用 key 只能让失去外部引用的 key 被回收，不能替代显式清理。跨线程时，普通 `ThreadLocal` 不传播，`InheritableThreadLocal` 也只提供创建线程时的有限继承，异步场景必须明确捕获、安装和恢复协议。

可靠使用 `ThreadLocal` 的原则可以归纳为：只保存真正线程封闭且体积受控的状态；优先使用不可变 value；把安装和清理封装在同一作用域；支持嵌套时恢复旧值；线程池任务始终在 `finally` 中收尾；不要用它隐藏关键业务依赖；跨线程传播采用明确白名单。满足这些条件时，它是有价值的并发基础工具；缺少这些条件时，显式参数、局部变量、同步工具或结构化作用域通常更容易维护。

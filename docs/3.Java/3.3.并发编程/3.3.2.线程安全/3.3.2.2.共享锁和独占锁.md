# 3.3.2.2 共享锁和独占锁

## 定位
共享锁和独占锁是 Java 并发编程中的一个独立主题，它解决的问题是：区分可并发持有和必须排他持有的同步语义。学习这个主题时，不能只记一个类名或一个方法调用，而要把它放回并发程序的完整链路中理解：任务从哪里产生，在哪个线程中执行，是否访问共享状态，遇到等待和取消时如何响应，结果或异常又如何回到调用方。只有这些问题都清楚，代码才不仅是“能跑”，而是在压力、异常和关闭阶段仍然可解释。

阅读 Java 并发文档时要把三个层次分开：语言层面的语义、类库层面的工具、运行时和操作系统层面的调度。语言语义回答“哪些结果是被规范允许的”，类库工具回答“怎样表达等待、通知、排队、取消和结果传递”，运行时调度回答“什么时候真正获得 CPU、什么时候被阻塞、什么时候被唤醒”。很多并发错误来自把这三层混在一起，例如用时间先后推导可见性，用线程优先级推导执行顺序，或者用一个看起来没有报错的测试结果推导程序在高并发下也正确。

从知识体系上看，共享锁和独占锁既连接基础线程模型，也连接后续的线程安全、线程池、同步器和内存模型。它的学习目标不是背诵 API，而是建立判断能力：什么时候该使用它，什么时候应该换成更高层或更低层工具，什么时候一个看似简单的写法会破坏可见性、原子性或生命周期。本文保持通用 Java 技术视角，只讨论 Java 语言和标准并发库中的机制、边界与实践。

## 解决的问题
共享锁和独占锁面向的直接问题是区分可并发持有和必须排他持有的同步语义。在单线程程序中，控制流沿着调用栈顺序前进，数据读写通常由同一个线程完成；进入并发环境后，执行顺序、等待位置、异常传播和状态可见性都可能分离。这个主题的价值就在于给这种分离建立清晰规则，让开发者知道哪些行为由规范保证，哪些只是当前机器上的偶然结果。

读写锁的收益来自放开读读互斥。读多写少时有效，写多或读写切换频繁时成本可能高于普通互斥。

更具体地说，共享锁和独占锁会影响三个方面。第一是执行关系：某段代码由哪个线程执行，是否可能与其他代码并行，是否存在先后约束。第二是状态关系：共享变量在多个线程之间是否有一致的观察方式，复合操作是否会被打断，更新是否能被其他线程及时看见。第三是生命周期关系：任务开始、等待、取消、失败、完成和资源释放是否有明确路径。并发代码一旦缺少这些关系，就容易出现测试环境偶尔通过、生产压力下随机失败的问题。

## 核心 API 与术语
围绕共享锁和独占锁，最常见的 API 或术语包括：ReadWriteLock、ReentrantReadWriteLock、AQS 共享模式、AQS 独占模式。这些名称背后不是孤立功能点，而是一组协作约定。读源码或写业务代码时，应优先识别它们承担的是执行、排队、互斥、条件等待、结果传递、取消还是可见性职责。职责判断清楚后，API 的选择通常会变得直接。

读锁共享、写锁独占、锁降级、避免锁升级。这条流程描述了共享锁和独占锁在典型使用中的状态变化。流程中的每一步都可能失败或被中断，因此真实代码不能只写“理想路径”。例如等待可能超时，提交可能被拒绝，锁可能长期竞争，任务可能抛异常，关闭可能遇到不响应中断的工作线程。把失败路径提前纳入设计，是并发代码和普通顺序代码的重要差异。

下面的片段只用于说明核心用法，重点是观察调用位置和资源边界，而不是把它当成完整框架：

```java
lock.readLock().lock();
try { return map.get(key); } finally { lock.readLock().unlock(); }
```

这类代码最容易被忽略的是边界条件。调用前对象是否已经初始化完成，调用后是否需要等待或关闭，异常是否会被吞掉，当前线程被中断时是否应继续执行，都会影响最终正确性。写并发代码时，每个 API 调用都应回答“失败时怎么办”和“谁负责收尾”。

## 核心机制
共享锁和独占锁的核心机制可以从“状态、队列、唤醒、可见性”四个词理解。状态描述对象当前处于哪个阶段，队列描述竞争者或等待者如何排列，唤醒描述等待线程重新获得执行机会的条件，可见性描述一个线程的写入何时能被另一个线程可靠观察到。不是每个主题都会显式包含四个结构，但成熟的并发工具通常都能在内部找到这些要素。

以共享锁和独占锁为例，状态变化必须是可解释的。线程从未启动到可运行再到终止，锁从未持有到被某个线程持有再到释放，任务从已提交到执行中再到完成或取消，同步器从计数未满足到条件满足，这些变化都需要明确触发点。如果状态触发点不明确，调用者就会被迫用 sleep、轮询或经验时间等待，而这正是并发程序脆弱的常见来源。

队列和唤醒决定等待成本。阻塞等待可以节省 CPU，但涉及线程挂起和恢复；自旋等待可以避免上下文切换，但会消耗处理器时间；公平排队能降低饥饿概率，但可能牺牲吞吐；非公平竞争能提升局部效率，但等待时间更难预测。理解共享锁和独占锁时，不应只问“会不会等待”，还要问“等待者在哪里排队、由谁唤醒、唤醒后是否立即满足条件”。

可见性则决定结果是否可信。一个线程完成了写入，不代表另一个线程一定能立刻看到；一个线程按代码顺序先写 A 再写 B，也不代表另一个线程没有同步就能按同样顺序观察。Java 通过 synchronized、volatile、Lock、线程启动和终止、并发容器等同步动作建立 happens-before 关系。共享锁和独占锁如果涉及共享状态，就必须说明它依赖哪一种同步关系。

## 共享访问和独占访问的语义
共享锁和独占锁首先是访问语义，而不是某个具体类名。共享访问的含义是：多个线程可以同时进入同一段受保护区域，但这些线程的操作必须彼此兼容。最常见的兼容操作是只读操作，因为只读不会改变被保护对象的不变式；如果某个“读操作”会刷新缓存、移动游标、维护统计计数、延迟初始化字段、修改访问时间或触发回调，那么它就不再是纯读，不能简单放进共享锁保护的读路径。共享语义成立的前提不是方法名叫 read，也不是调用方主观认为它“只是查询”，而是这条路径确实不会破坏其他共享持有者正在依赖的状态。

独占访问的含义是：同一时刻最多只有一个线程可以进入受保护区域，并且在它持有锁期间，其他共享访问和独占访问都要被排除在外。独占访问通常用于写入、结构调整、状态转换和维护跨字段不变式。例如一个缓存结构同时维护键值映射、命中次数、过期队列和容量计数，写入时可能要同时更新多个位置；如果只保护其中一个字段，其他线程就可能观察到中间状态。独占锁的价值不是让代码“慢下来”，而是把一组必须整体成立的操作变成外部不可见的临界区。

共享和独占之间不是“读快写慢”这么简单，而是一套兼容矩阵。读锁与读锁兼容，读锁与写锁不兼容，写锁与写锁不兼容。兼容矩阵决定了线程能否同时进入，而不是决定临界区内部一定安全。临界区内部是否安全，还取决于访问对象是否只通过同一把锁保护、是否返回了可变内部引用、是否在持锁期间调用了外部代码、是否把受保护状态交给异步任务继续使用。很多读写锁误用不是因为锁实现有问题，而是因为程序只建立了表面上的兼容矩阵，没有建立完整的数据所有权边界。

理解共享锁时还要区分“并发读”和“一致读”。多个读线程同时执行，说明它们不会和写线程并发交错；但这不自动说明它们看到的是某个业务意义上的最新快照。读锁获取会与之前的写锁释放建立同步关系，因此读线程能看到前一个写线程在释放写锁前完成的写入；但是如果读线程在读取期间把内部对象暴露出去，或者读取多个没有同一保护协议的数据源，仍可能得到业务上不一致的组合。锁只能保证它保护范围内的内存访问关系，不能替代业务层对快照边界的定义。

独占锁也不等于全局互斥。只有竞争同一把锁的线程才会互斥；使用不同锁对象、不同实例、不同保护协议的代码不会自动排队。如果一个类的方法 A 使用 `this` 作为锁，方法 B 使用内部 `ReentrantReadWriteLock`，方法 C 完全不加锁，那么这三个入口之间没有统一同步关系。写并发组件时，应把“受哪把锁保护”作为字段级或不变式级约束写清楚，而不是只在某些方法上零散加锁。

| 访问组合 | 是否兼容 | 典型含义 | 需要额外注意的边界 |
| --- | --- | --- | --- |
| 读锁 + 读锁 | 兼容 | 多个线程同时观察稳定状态 | 读路径必须不修改受保护状态，也不能泄露内部可变对象 |
| 读锁 + 写锁 | 不兼容 | 写入需要等待已有读取结束，读取需要等待写入完成 | 长时间读会拖慢写入，读路径不能包含不可控阻塞 |
| 写锁 + 写锁 | 不兼容 | 写入之间串行维护不变式 | 写锁内调用外部代码可能放大临界区并引入死锁 |
| 不加锁 + 任意锁 | 不由锁约束 | 绕过保护协议 | 这是最常见的数据竞争来源之一 |

## 读写锁的抽象模型
读写锁可以看作“一个同步器，两种入口”。它不是两把互不相关的锁，而是一把锁同时暴露读入口和写入口。读入口以共享方式获取，写入口以独占方式获取，两者共同竞争同一份内部状态。这个模型很重要：如果把读锁和写锁理解成两个独立锁，就容易误以为读锁只管读、写锁只管写，实际上它们必须共同维护同一个对象的不变式。读锁释放和写锁释放都可能影响等待队列，写锁获取必须观察是否已有读者，读锁获取也要观察是否已有写者或等待中的写者。

读写锁适合读多写少，但“读多写少”不是只看方法调用次数。更准确的判断应包括临界区耗时、写入等待的业务代价、线程数量、读写切换频率以及读路径是否纯粹。一个系统中读调用可能占九成，但如果每次读都很短、写也很短，普通互斥锁可能更简单也更快；反过来，读调用占比没有特别夸张，但每次读需要遍历较大的不可变结构，而写入较少，读写锁就可能明显减少读读互斥成本。锁选择要看竞争窗口，不只是看调用计数。

读写锁的收益来自允许多个读者同时进入，因此它最怕两类情况。第一类是读路径太慢，导致写线程长时间等待。读锁本身是共享的，但共享不意味着无成本；大量长读会让写线程很难找到插入机会。第二类是写入太频繁，导致读者不断被写者打断，读写锁退化成复杂的互斥锁，还承担更多状态维护和队列管理开销。实践中，如果写入比例较高、临界区很短、读路径会阻塞外部资源，读写锁通常不是首选。

读写锁还要求读写双方采用同一份保护协议。假设一个配置对象用读写锁保护，读取方进入读锁后读取多个字段，写入方进入写锁后整体替换字段，这是一致的；如果某个维护线程为了省事直接修改字段，或者某个读取方绕过读锁读取其中一部分字段，读写锁提供的可见性和互斥关系就被破坏了。并发安全往往不是某个单点 API 的性质，而是所有入口共同遵守协议的结果。

在设计读写锁保护范围时，应尽量让临界区只覆盖必要的共享状态访问。可以在锁外准备新对象、计算新结果、执行耗时校验，然后在写锁内完成短小替换；可以在读锁内复制必要快照，然后在锁外做慢处理；可以把不共享的局部计算移出锁外。这样做不是机械追求“锁越短越好”，而是让锁保护的内容正好对应不变式，避免把无关阻塞带入所有读者和写者的等待路径。

## AQS 中的独占模式和共享模式
AbstractQueuedSynchronizer 通常简称 AQS，它是 Java 并发库中许多同步器的基础框架。AQS 的核心思想是用一个整数状态、一个等待队列和一组模板方法，把获取、释放、排队、阻塞、唤醒这些通用流程抽象出来。具体同步器只需要解释状态含义：独占模式下，状态通常表示某个线程是否独占持有以及重入次数；共享模式下，状态通常表示剩余许可、共享计数或读者数量。AQS 不知道业务上的“读”和“写”，它只提供独占获取和共享获取两套协议。

独占模式强调唯一持有者。线程调用独占获取时，如果同步器判断当前状态允许获取，就设置独占持有线程并返回；如果不允许，就进入等待队列。释放时，持有线程更新状态，如果完全释放，就唤醒后继节点继续竞争。`ReentrantLock` 的基本语义就是这种模式：同一时刻只有一个线程拥有锁，拥有者可以重入，释放次数必须与获取次数匹配。独占模式还支持条件队列，因此 `Condition` 通常和独占锁一起出现。

共享模式强调多个成功获取者可以同时存在。线程调用共享获取时，如果状态允许共享通过，就返回成功；如果不允许，就排队等待。共享释放后，如果状态变化可能让其他共享等待者通过，AQS 会传播唤醒，使多个共享节点有机会继续获取。`Semaphore`、`CountDownLatch` 和读写锁中的读锁都体现了共享模式，但它们对状态的解释不同：信号量关注许可数量，闭锁关注计数是否归零，读锁关注是否存在写者以及读计数是否可增加。

ReentrantReadWriteLock 同时使用 AQS 的独占和共享能力。写锁走独占模式，读锁走共享模式，两者共用一个同步状态。这个状态在实现上被拆成两部分：低位记录写锁重入次数，高位记录读锁持有数量。这样的设计让一个整数同时表达“有没有写者、写者重入了几次、当前有多少读者”。理解这一点可以解释很多行为：写线程重入写锁只是增加写计数；读线程获取读锁会增加读计数；只要读计数非零，其他线程的写锁获取通常不能成功。

AQS 的队列语义也提醒我们：锁释放以后，被唤醒的线程并不等于已经完成业务操作。唤醒只是让等待线程重新参与获取；它仍可能因为公平策略、竞争、取消、中断或状态变化而失败。并发代码不能把“调用 unlock”理解成“某个指定线程马上运行”，也不能把“队列里有等待者”理解成“等待者一定按业务希望的顺序完成”。AQS 保证的是同步器层面的排队与状态转换，业务层顺序仍要由更明确的协议表达。

## ReentrantReadWriteLock 的行为细节
ReentrantReadWriteLock 是 Java 标准库中最常用的读写锁实现。它提供一个读锁和一个写锁，读锁共享，写锁独占，并且支持可重入。所谓可重入，是指同一线程已经持有某种锁时，可以再次获取符合规则的同一锁，而不会被自己阻塞。写锁的可重入比较直观：持有写锁的线程再次获取写锁，只增加写锁持有计数；释放时递减计数，直到计数归零才真正释放。读锁也支持重入，同一线程可以多次进入读锁，释放次数也必须匹配。

读写锁的重入规则不等于任意升级和降级。持有写锁的线程可以再获取读锁，这为锁降级提供基础；但持有读锁的线程直接获取写锁通常会导致等待，甚至在多个读者同时尝试升级时形成永久等待。原因很简单：写锁必须等待所有读锁释放，而升级线程自己也持有读锁；如果它在不释放读锁的情况下等待写锁，就把自己也变成写锁等待条件的一部分。ReentrantReadWriteLock 没有提供自动安全升级，因为它无法替调用方决定释放读锁到获取写锁之间的状态变化如何处理。

ReentrantReadWriteLock 有公平和非公平两种构造方式。默认是非公平模式，线程获取锁时可能插队，这通常有利于吞吐，因为刚释放锁附近的线程可能更快继续执行，减少上下文切换和队列管理成本。但非公平模式下，等待时间更难预测，某些等待写锁的线程可能在大量读者涌入时被延迟。公平模式会尽量按照等待时间排序，减少饥饿概率，但会牺牲一部分吞吐，也不应被误解成严格的业务先来先服务。公平性是锁队列层面的策略，不保证任务提交顺序、请求优先级或端到端完成顺序。

读锁和写锁在条件变量支持上也不同。ReentrantReadWriteLock 的写锁可以创建 Condition，因为条件等待需要一个独占持有者在等待时释放锁、被唤醒后重新获取锁，并在独占保护下检查条件；读锁不支持 Condition，调用读锁的 `newCondition` 会抛出异常。这个限制符合语义：多个读者共享持有读锁，无法把条件等待和独占状态修改清晰绑定。需要条件协作时，通常应使用写锁、独立的 ReentrantLock，或者更高层的并发工具。

ReentrantReadWriteLock 还提供若干监控方法，例如查询读锁数量、写锁是否被持有、当前线程是否持有写锁、队列中是否有等待线程等。这些方法适合诊断和监控，不适合用来驱动核心并发逻辑。原因是这些值往往是瞬时观察，检查之后马上可能变化。如果业务代码写成“如果当前没有读者就执行某个无锁写入”，那就是典型的数据竞争。并发控制必须依赖获取锁这样的原子协议，而不是依赖锁状态的非原子快照。

使用 ReentrantReadWriteLock 时，最常见的形态是读方法获取读锁、写方法获取写锁，并在 finally 中释放。这个模式虽然简单，但要注意返回值。如果读方法返回内部可变集合、数组或对象引用，调用方可能在锁外修改它，破坏写锁本应保护的不变式。稳妥做法是返回不可变视图、 defensive copy，或者把修改操作也收回到受锁保护的方法中。读写锁保护的是对象内部状态，而不是返回引用之后的外部行为。

```java
final class Catalog {
    private final ReentrantReadWriteLock rw = new ReentrantReadWriteLock();
    private final Map<String, String> data = new HashMap<>();

    String get(String key) {
        rw.readLock().lock();
        try {
            return data.get(key);
        } finally {
            rw.readLock().unlock();
        }
    }

    void put(String key, String value) {
        rw.writeLock().lock();
        try {
            data.put(key, value);
        } finally {
            rw.writeLock().unlock();
        }
    }
}
```

这个例子只适合说明读写入口的基本结构。真实代码要继续追问：value 是否可变，返回后是否还能被外部修改；put 是否需要维护其他索引；get 是否会触发懒加载；是否允许在锁内调用回调；是否需要超时获取锁；写入失败时是否会留下半更新状态。只有这些问题都有答案，读写锁才真正对应了线程安全边界。

## StampedLock 的语义和适用场景
StampedLock 是另一种读写同步工具，它提供写锁、悲观读锁和乐观读三种访问方式。写锁是独占的，悲观读锁是共享的，乐观读不是传统意义上的锁。乐观读返回一个戳记，读取线程先按无锁方式读取字段，然后通过 validate 检查从获取戳记到校验期间是否发生过写入。如果没有写入，读取结果可以被接受；如果发生过写入，就要退回到悲观读锁或重新读取。它适合读操作非常频繁、写操作较少、读取字段数量有限且可以重试的场景。

StampedLock 和 ReentrantReadWriteLock 的差异很大，不能只把它当成“更快的读写锁”。首先，StampedLock 不可重入。同一线程已经持有写锁时再次获取写锁，或者在没有设计好的情况下重复进入同一模式，都可能把自己阻塞。其次，StampedLock 的解锁依赖戳记，错误的戳记或错误的模式会导致异常。再次，它不直接提供 Condition 语义，也不适合需要复杂条件等待的场景。最后，乐观读并不阻止写线程执行，它只是在事后验证读取期间是否发生写入。

乐观读的正确写法要求把受保护字段复制到局部变量，再校验戳记。如果校验失败，就进入悲观读路径重新读取。不能在 validate 之前把读到的中间对象发布出去，也不能在校验之后继续读取受保护字段并假设仍然有效。validate 检查的是某段读取窗口内是否有写入，窗口应当由“获取戳记、复制字段、校验”构成；一旦校验完成，后续如果还要使用共享字段，就已经进入新的时间窗口，需要重新建立同步关系。

```java
final class Point {
    private final StampedLock lock = new StampedLock();
    private double x;
    private double y;

    double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead();
        double currentX = x;
        double currentY = y;
        if (!lock.validate(stamp)) {
            stamp = lock.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }
        return Math.hypot(currentX, currentY);
    }

    void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp);
        }
    }
}
```

StampedLock 的转换方法可以在某些场景中减少释放再竞争的窗口。例如线程持有读锁后发现需要写入，可以尝试把读戳转换成写戳；如果转换成功，就可以继续在写模式下修改；如果失败，就要释放读锁，再获取写锁，并重新检查条件。这里的“重新检查”非常关键，因为释放读锁之后，其他写线程可能已经改变了状态。转换失败不是异常情况，而是并发竞争下的常规路径。

StampedLock 的优势通常出现在读取短小、写入稀少、冲突概率低的场景。它的代价是使用约束更强，代码更容易写错：不可重入、戳记必须正确保存和释放、乐观读必须校验、失败路径必须重读。对于需要简单可重入、条件等待、清晰锁所有权和更传统诊断方式的代码，ReentrantReadWriteLock 往往更容易维护。选择 StampedLock 前应先确认它解决的是实际读路径竞争，而不是为了追求一个看起来更高级的类名。

## 锁降级、锁升级和状态重查
锁降级指线程在持有写锁时获取读锁，然后释放写锁，从独占访问转为共享访问。降级的意义是：线程完成写入后，仍希望在一个稳定读视图下继续读取刚写好的状态，同时允许其他读者并发进入，但暂时不允许写者插入破坏这个读视图。ReentrantReadWriteLock 支持这种模式，因为写锁持有者可以获取读锁。正确顺序是先持有写锁，完成修改，再获取读锁，最后释放写锁；如果先释放写锁再获取读锁，中间就存在其他写者插入的窗口。

锁降级常用于缓存刷新。线程发现缓存失效后获取写锁，重新加载数据，更新缓存状态，然后获取读锁并释放写锁，最后在读锁保护下返回或使用新缓存。这里需要注意，耗时的数据加载不一定适合完全放在写锁内。如果加载过程不依赖当前共享状态，可以在锁外准备候选值，再进入写锁检查状态是否仍需更新；如果必须保证只有一个线程加载，则要权衡锁内耗时和重复加载成本。降级只解决写后继续读的同步窗口，不自动解决慢操作是否该持锁的问题。

锁升级指线程已经持有读锁时尝试获取写锁。直觉上，线程读到状态后发现需要修改，于是想从读模式升级到写模式；但在传统读写锁中，这通常是危险路径。多个读者可能同时发现需要升级，如果它们都持有读锁等待写锁，就没有人释放读锁，写锁条件永远不满足。即使只有一个读者升级，它也必须处理自己持有读锁导致写锁无法获取的问题。因此一般建议避免在持有读锁时阻塞式获取写锁。

安全的升级思路通常是释放读锁，再获取写锁，然后重新检查条件。重新检查不能省略，因为释放读锁到获取写锁之间，其他写者可能已经完成了修改，原先读到的判断依据已经过期。这个模式叫检查后重查，也可以理解为乐观地在读锁下发现候选状态，悲观地在写锁下确认并提交修改。它的缺点是代码更长，且存在重复判断；但它避免了读锁升级死等，也尊重了写锁必须排除所有读者的语义。

```java
Value getOrLoad(Key key) {
    rw.readLock().lock();
    try {
        Value cached = cache.get(key);
        if (cached != null) {
            return cached;
        }
    } finally {
        rw.readLock().unlock();
    }

    rw.writeLock().lock();
    try {
        Value cached = cache.get(key);
        if (cached == null) {
            cached = load(key);
            cache.put(key, cached);
        }
        return cached;
    } finally {
        rw.writeLock().unlock();
    }
}
```

这段代码展示的是释放读锁、获取写锁、重新检查条件的结构，而不是鼓励把所有加载都放进写锁。若 load 很慢，就要进一步拆分：先在写锁中登记加载状态，释放锁后加载，再用写锁提交结果，或者使用 Future、CompletableFuture 等方式表达进行中的加载。关键点是不要把“读到为空”直接等同于“自己一定负责写入”，因为这个结论只在读锁持有期间成立，一旦释放读锁，世界就可能变化。

## 使用方式
使用共享锁和独占锁时，第一步是识别任务性质。CPU 密集型任务、I/O 等待型任务、短小状态更新、长时间阻塞任务、阶段同步任务、资源限流任务，适合的工具不同。把所有需求都压到同一个 API 上，会让代码表面简单、实际边界混乱。例如需要结果时不应只用裸 Runnable，需要容量控制时不应只靠线程数量，需要条件协作时不应靠固定 sleep。

第二步是明确所有权。谁创建线程或同步器，谁提交任务，谁负责关闭，谁处理异常，谁记录拒绝，谁保证共享对象不被并发破坏，这些都属于所有权问题。所有权不清时，常见后果是线程池被局部代码关闭、共享锁被不同模块以不同顺序获取、ThreadLocal 被上一个任务污染、异常在后台线程中静默消失。

第三步是把正常路径和异常路径写在一起。只写成功路径的并发代码往往在演示中很好看，但在真实系统中会出现资源泄漏。锁释放要放在 finally，许可归还要放在 finally，计数器递减要放在 finally，ThreadLocal 清理要放在 finally，线程池关闭要处理 InterruptedException，等待结果要考虑 TimeoutException。并发代码的可靠性主要体现在这些收尾位置。

第四步是保留可替换空间。许多并发工具有相似目标但边界不同：wait/notify 可以表达条件等待，Condition 能拆分多个等待队列；CountDownLatch 适合一次性等待，CyclicBarrier 适合固定参与者循环阶段，Phaser 适合动态参与者；Atomic 类适合单变量乐观更新，Lock 适合复杂临界区。写文档和写代码都应说明为什么选择当前工具，而不是只说明它“可以用”。

## 状态流转
共享锁和独占锁的状态流转可以概括为：初始化阶段建立对象和依赖，运行阶段执行或等待，竞争阶段处理共享资源冲突，完成阶段发布结果或释放等待者，关闭阶段拒绝新工作并清理资源。这一描述虽然抽象，但适用于大多数 Java 并发工具。每个阶段都要避免“半初始化对象被发布”“等待条件丢失”“结果未安全发布”“关闭后仍提交任务”等问题。

初始化阶段最重要的是安全发布。线程启动前传入的不可变参数、构造完成后再发布的对象、通过 volatile 或锁保护发布的引用，都是可靠方式。反过来，如果 this 在构造过程中逃逸，或者共享集合在没有同步的情况下被多个线程看到，就可能让其他线程观察到未完成状态。共享锁和独占锁如果涉及对象共享，初始化边界必须清晰。

运行阶段最重要的是不要阻塞错位置。持锁执行慢 I/O、在线程池工作线程中等待同一个线程池的任务、在 ForkJoin 任务中长时间等待外部资源、在条件等待中忘记循环检查，都会制造隐蔽问题。等待不是错误，但等待必须有条件、超时、取消和诊断手段。

完成阶段最重要的是结果传播。任务返回值、异常、计数归零、状态标志变更、锁释放或 volatile 写入，都是向其他线程传播进展的方式。传播动作必须与读取动作配对：Future.get 读取任务结果，await 等待计数归零，lock 获取看到上一次 unlock 前的写入，volatile 读观察 volatile 写发布的状态。只有写端和读端使用同一套规则，状态流转才可靠。

## 公平性、饥饿与吞吐
共享锁和独占锁的调度策略必须在公平性和吞吐之间取舍。公平性关注等待线程是否能按照相对稳定的顺序获得机会，饥饿关注某类线程是否可能长期得不到机会，吞吐关注单位时间内能完成多少操作。三者不是同一个目标：更公平的排队通常会减少插队和局部性收益，吞吐可能下降；更激进的非公平竞争可能让刚运行过的线程更快再次获得锁，吞吐变好，但等待时间的尾部可能变长。读写锁尤其容易放大这种取舍，因为读者可以成批进入，而写者必须等待所有读者离开。

写线程饥饿是读写锁中最常被讨论的问题。假设读请求持续不断到来，非公平策略又允许新读者在某些时机直接进入，那么等待中的写线程可能很难等到读计数归零。写线程一旦长期得不到执行，读者看到的状态就会长期停留在旧版本，系统表面上读吞吐很高，实际更新延迟却不可接受。相反，如果策略过度偏向写者，读线程也可能在写入频繁时排队过久，读写锁的主要收益被削弱。因此判断读写锁是否合适时，要看读延迟、写延迟和状态新鲜度，而不仅是平均吞吐。

ReentrantReadWriteLock 的公平构造可以降低饥饿概率，但不能把并发系统变成严格的业务排队系统。公平锁关注的是同步器队列中的获取顺序；业务请求从提交到执行，还会经过线程调度、任务队列、超时取消、异常重试等多个环节。一个较早提交的请求可能较晚到达锁竞争点，一个较晚提交的请求可能先完成无锁准备工作。因此，公平锁适合改善锁竞争层面的等待分布，不适合承担全链路排序语义。需要严格顺序处理时，应使用单线程执行器、队列化状态机或显式序号协议，而不是只依赖读写锁公平参数。

StampedLock 的策略也不应被理解成稳定偏向某一类线程。它强调乐观读和戳记验证，适合减少无冲突读取成本，但在调度公平性、可重入和条件等待方面提供的约束更少。乐观读失败率一旦升高，读线程会频繁回退到悲观读或重试，代码路径复杂度和实际成本都会上升。此时继续使用 StampedLock 可能只是把锁竞争变成重试竞争，并没有真正解决写入冲突。观察乐观读成功率、写锁等待时间和重试次数，比单纯比较某次微基准结果更有意义。

处理饥饿问题不能只靠换锁。首先要缩短临界区，避免读锁内执行慢 I/O、复杂计算或调用不可控回调。其次要控制进入速率，必要时在更高层使用队列、限流或批量合并，避免无限读请求持续冲击写入窗口。再次要明确更新时效要求：有些配置更新允许稍有延迟，有些状态变更必须尽快被读者观察到，二者需要不同策略。最后要保留指标，例如读锁持有时间、写锁等待时间、队列长度、超时次数和重试次数。没有指标时，公平性和饥饿只会停留在主观感受。

## 线程安全边界
共享成立的前提是读路径不修改共享状态；一旦读路径会改变对象内部状态，它就必须按写路径重新评估保护方式。

线程安全边界必须落到共享状态上。只要一个对象、集合、计数器、状态标志或外部资源会被多个线程同时访问，就要明确谁负责保护它、保护范围有多大、读写之间是否需要原子性、写入后是否需要被其他线程及时看见。没有共享状态的任务通常只需要关注取消和资源释放；有共享状态的任务则必须同时关注互斥、可见性、有序性和生命周期。

对共享锁和独占锁来说，最小线程安全边界通常不是“某个类线程安全”这么粗，而是“哪些方法可以并发调用、哪些参数不能被调用方继续修改、哪些返回值不能暴露内部可变对象、哪些回调可能在持锁时发生”。同一个类中，有的方法可能线程安全，有的方法可能要求外部同步；同一个对象中，有的字段不可变，有的字段必须受锁保护。文档和代码注释应尽量表达这种精确边界。

还要区分结构安全和业务安全。并发容器可以保证自身结构不被破坏，但不能保证跨多个操作的业务不变式；AtomicInteger 可以保证单次递增原子，但不能保证“检查余额再扣减再记录日志”整体一致；线程池可以保证任务被工作线程执行，但不能保证任务内部访问的对象安全。共享锁和独占锁如果被误认为“用了就线程安全”，往往会掩盖更上层的不变式问题。

## 内存可见性与有序性
Java 并发中，正确性不仅是“没有两个线程同时写”。一个线程写入后另一个线程能否看到，另一个线程看到多个写入时是否能按预期顺序理解，同样关键。共享锁和独占锁只要牵涉共享变量，就要考虑 happens-before。synchronized 的释放和随后获取、volatile 写和随后读、线程 start 前后的关系、线程终止和 join 的关系、并发工具内部的同步动作，都是常见可见性来源。

有序性问题常被低估。编译器和处理器可以在不改变单线程结果的前提下重排序指令；缓存也可能让一个核心暂时看不到另一个核心的写入。JMM 允许这些优化，同时要求程序通过同步动作表达跨线程约束。因此，不能因为一段代码“按顺序写着”就认为另一个线程也会按顺序看到。对共享锁和独占锁的使用如果缺少同步边界，就可能在压力下暴露出旧值、默认值或部分构造对象。

可见性也不等于原子性。volatile 能让状态标志及时可见，但不能让自增、复合检查或多字段更新自动变成一个不可分割操作；锁能同时提供互斥和可见性，但锁的对象必须一致；CAS 能保证单个位置的比较更新原子，但不能自动维护多个变量之间的不变式。选择工具时，应先判断需要的是可见性、原子性、有序性，还是三者都需要。

## 实践要点
第一，优先使用表达力更明确的高层工具。能用 BlockingQueue 表达生产消费，就不要手写 wait/notify；能用 ExecutorService 管理任务，就不要散落创建 Thread；能用 CountDownLatch 表达一次性等待，就不要用循环 sleep；能用 AtomicInteger 表达单变量计数，就不要扩大成不必要的重量级锁。高层工具的价值在于把容易写错的等待、唤醒和内存语义封装起来。

第二，给等待设置边界。无界等待是并发故障扩大的常见原因。获取结果可以使用超时 get，条件等待可以使用带超时的 await，外部调用可以设置超时，线程池关闭可以设置 awaitTermination 时间。超时不是为了掩盖错误，而是为了让调用方有机会记录现场、释放资源或降级。

第三，保持中断语义。凡是调用会抛出 InterruptedException 的方法，都要认真决定当前层是否消费中断。如果当前层无法完成取消处理，通常应恢复中断标志并返回，让上层继续处理。静默吞掉中断会让关闭流程卡住，也会让调用者误以为任务已经遵守取消协议。

第四，减少共享可变状态。不可变对象、局部变量、消息传递、任务参数复制、并发容器和清晰的所有权，通常比到处加锁更容易维护。锁和原子类解决的是共享状态冲突，但最好的并发设计往往是先减少共享，再保护必须共享的部分。

第五，保持诊断信息。线程名应能反映用途，线程池应暴露队列长度和拒绝次数，长时间等待应能定位锁或条件，后台任务异常应记录。并发问题通常无法只靠阅读一行报错定位，足够的运行时信息能显著降低排查成本。

## 常见误区
不要在读锁中返回可变内部对象让外部修改。这是共享锁和独占锁中最需要避免的误区之一。它之所以危险，是因为代码在低并发、本机运行或单次测试时可能完全正常，但在调度时机变化、任务数量上升或关闭流程触发时暴露问题。并发程序的正确性不能建立在“通常来得及”“一般不会同时发生”“这个线程应该先执行”这类假设上。

另一个误区是把时间当同步。sleep 一段时间、等待“差不多执行完”、依赖日志先后顺序、用本机测试速度推断生产时序，都属于时间同步误区。正确做法是使用明确同步条件，例如 join、Future、Latch、Barrier、锁、条件队列或 volatile 状态。

还要避免把工具名当保证。使用线程池不代表任务不会丢，使用锁不代表锁对象选对了，使用 volatile 不代表复合操作原子，使用 ThreadLocal 不代表不会泄漏，使用公平锁不代表请求一定按业务顺序完成。每个工具都只保证它规范中定义的部分，超出边界的需求仍要由设计补上。

最后是忽略失败路径。任务抛异常后计数是否递减，锁是否释放，许可是否归还，线程池是否还能接收任务，Future 是否有人读取异常，中断是否被恢复，资源是否关闭，这些都必须在代码中体现。并发系统的许多严重问题不是正常路径错误，而是异常路径没有释放等待者或资源。

## 排查思路
排查共享锁和独占锁相关问题时，可以先问四个问题。第一，线程现在在哪里：运行、阻塞在锁、等待条件、等待 I/O、等待任务结果，还是已经终止。第二，共享状态由谁保护：锁对象是否一致，volatile 是否覆盖读写两端，原子操作是否足够表达不变式。第三，生命周期是否闭合：启动后是否可停止，提交后是否可取消，关闭后是否拒绝新任务，异常后是否释放等待方。第四，容量是否合理：线程数、队列长度、许可数量、任务耗时和上游流量是否匹配。

工具上可以结合线程 dump、日志时间线、线程池指标、锁竞争指标和重复压力测试。线程 dump 能看到 BLOCKED、WAITING、TIMED_WAITING 以及调用栈；线程池指标能看到任务是否堆积；日志可以确认取消和关闭是否发生；压力测试可以放大竞态窗口。排查时不要只看最终异常，要寻找第一个失去同步关系或容量边界的位置。

如果问题偶发，优先怀疑数据竞争、遗漏通知、异常路径未释放、线程池饱和、死锁或外部阻塞。共享锁和独占锁相关代码应尽量让这些情况可被观察，例如在拒绝策略中记录任务类型，在等待超时后输出关键状态，在关闭时记录未执行任务数量，在捕获异常时保留中断状态或根因。

## 深入理解与推导
理解共享锁和独占锁时，可以从一个更具体的推导过程开始：先假设程序只有一个线程，所有语句按照调用栈顺序执行，变量只被当前线程读写；再引入第二个线程，让同一份状态可能被两个执行流同时观察；最后再加入等待、取消、超时、异常和关闭。每增加一个因素，就需要增加一条明确规则。没有规则的地方，运行结果就只能依赖调度时机，而调度时机从来不是稳定契约。

第一条规则是所有权规则。一个并发对象或同步工具必须有清楚的创建者、使用者和释放者。创建者负责初始化完整状态，使用者负责遵守访问协议，释放者负责关闭或清理资源。共享锁和独占锁相关代码如果没有所有权规则，就会出现同一个对象被多个模块以不同方式访问：有的路径加锁，有的路径不加锁；有的路径处理取消，有的路径吞掉异常；有的路径认为可以关闭线程池，有的路径仍在提交任务。这类问题通常不是 API 本身难用，而是边界没有被写进设计。

第二条规则是条件规则。等待必须等待一个可验证的条件，而不是等待一段时间。条件可以是线程结束、任务完成、计数归零、队列非空、许可可用、状态标志改变或锁变为可获取。条件还必须由同一套同步机制保护：用锁保护的条件要在持锁时检查，用 volatile 发布的状态要由 volatile 读观察，用 Future 表示的结果要通过 get 获取，用同步器表示的阶段要通过对应 await 方法等待。条件和等待工具不匹配时，最容易出现遗漏通知、虚假成功或永久阻塞。

第三条规则是失败规则。共享锁和独占锁的正常路径通常很短，但失败路径决定系统是否可靠。提交失败时是否记录，等待超时时是否取消，任务异常时是否释放锁和许可，线程被中断时是否恢复中断标志，关闭时是否处理尚未执行的任务，这些问题都必须有答案。并发代码的失败路径尤其重要，因为一个线程的异常经常不会直接出现在另一个线程的调用栈中；如果没有显式传播和记录，调用方只能看到“没有结果”或“系统变慢”。

第四条规则是容量规则。任何等待队列、线程池、同步器许可、任务拆分粒度和重试循环都有容量含义。容量太小会导致吞吐不足，容量太大会隐藏过载并放大延迟，容量无界会把压力转化为内存占用或不可控等待。共享锁和独占锁如果用于高并发路径，就不能只讨论功能正确，还要讨论任务数量、平均耗时、峰值流量、等待上限和降级策略。容量规则不是性能优化阶段才考虑的问题，而是并发设计的一部分。

第五条规则是可见性规则。一个线程做完动作，不代表其他线程自然知道。必须通过锁释放与获取、volatile 写与读、线程启动与结束、并发容器操作、同步器状态变化等方式建立可见性。共享锁和独占锁的很多误用都可以归结为缺少可见性规则：写线程以为自己已经发布状态，读线程却可能看到旧值；任务以为已经初始化对象，其他线程却可能看到部分构造；关闭线程以为已经发出停止请求，工作线程却从不检查中断或状态标志。

把这五条规则连起来，就能形成一个检查清单：对象是否安全发布，访问是否遵守同一协议，等待是否有明确条件，失败是否释放资源，关闭是否能唤醒等待者，容量是否有上限，结果是否通过同步关系传播。共享锁和独占锁的学习价值正在于把这些问题具体化。只要清单中的某一项回答含糊，就应回到代码重新收紧边界，而不是依赖注释或经验说明“正常情况下不会出问题”。

## 阅读与复习建议
复习共享锁和独占锁时，建议不要只看结论，而要自己沿着一条并发路径推演：哪个线程先创建对象，哪个线程发布引用，哪个线程开始执行，哪个线程可能等待，哪个线程负责唤醒，异常会落在哪里，关闭时如何让等待者退出。能把这条路径讲清楚，通常比记住零散 API 更有用。

还可以把本文中的代码片段改写成几个小实验：增加线程数量，故意制造超时，主动触发中断，抛出运行时异常，重复执行上千次观察结果。实验不是为了依赖偶然输出，而是为了暴露自己对边界条件的理解是否完整。只要实验结果和预期不一致，就应回到线程安全边界、内存可见性、等待条件和生命周期四个角度重新检查。

## 小结
共享锁扩大只读并发，独占锁保护写入一致性，关键是读写路径都遵守同一边界。

掌握共享锁和独占锁，最终要形成两种能力：一是能写出边界清晰的代码，知道正常路径、失败路径和关闭路径分别如何运行；二是能解释运行时现象，知道线程为什么等待、为什么没有看到新值、为什么任务没有执行、为什么关闭没有结束。Java 并发不是把代码拆到多个线程中那么简单，而是给执行、状态和生命周期建立可验证的规则。

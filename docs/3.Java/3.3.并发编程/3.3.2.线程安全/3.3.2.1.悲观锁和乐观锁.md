# 3.3.2.1 悲观锁和乐观锁

## 定位

悲观锁和乐观锁不是两个具体类名，而是两种并发控制思想。它们讨论的核心问题是：当多个线程可能同时访问同一份共享状态时，程序应该先阻止冲突发生，还是允许并发执行、在提交时再检测冲突。前者通常被称为悲观策略，后者通常被称为乐观策略。Java 中的 `synchronized`、`ReentrantLock`、`ReadWriteLock` 等工具常用于实现悲观锁思路；`CAS`、`AtomicInteger`、`AtomicReference`、带版本号的比较更新等工具常用于实现乐观锁思路。但具体 API 只是载体，真正需要理解的是它们背后的冲突假设、等待方式、失败处理和线程安全边界。

并发程序中的共享状态有一个朴素但危险的事实：单线程下看似连续的操作，在多线程下可能被拆开、穿插、重复观察或观察不到。比如“读取余额、判断余额是否足够、扣减余额”在源代码中是三行连续语句，但只要中间没有同步约束，另一个线程就可能在任意一步插入自己的读写。悲观锁的做法是把这段复合操作包进一个互斥区域，让同一时刻只有一个线程进入；乐观锁的做法是先读取一个快照，基于快照计算新值，最后确认共享状态仍然等于刚才看到的版本，确认成功才提交更新，确认失败就放弃或重试。

这两种策略没有绝对优劣。悲观锁看起来保守，但它在高冲突、临界区复杂、失败代价高的场景中更容易给出确定的正确性边界。乐观锁看起来轻量，但它依赖冲突概率较低、重试成本可控、更新逻辑可以重复执行等前提；一旦冲突密集或重试体里混入不可重复副作用，乐观策略就可能从“高并发”变成“高消耗”。学习这个主题时，不能把结论简化为“悲观锁慢、乐观锁快”，而要问清楚：冲突是否频繁，冲突发生后能否重试，线程等待时是否占用 CPU，临界区是否包含阻塞调用，不变式是否跨多个变量，失败路径是否会泄漏状态。

本文保持通用 Java 技术视角，只讨论 Java 语言、Java 内存模型和标准并发库中的机制与实践。它是一篇叶子正文，重点不是索引 API，而是把“互斥阻塞”和“冲突检测重试”两种控制方式讲清楚，并能指导真实代码中的选择。

## 并发控制先解决什么问题

线程安全不是“多个线程运行起来”这么简单，而是多个线程共享同一份可变状态时，程序仍然满足预期不变式。这个不变式可以是计数器不能丢失递增，集合结构不能被破坏，缓存条目不能被部分初始化后发布，账户余额不能被扣成错误值，状态机不能从非法状态跳转。只要一个操作需要依赖“读取到的状态在提交前没有被其他线程改变”，就已经进入了并发控制的范围。

并发控制通常要同时处理三个层面的问题。第一是互斥，也就是某段代码或某个状态更新是否允许多个线程同时执行。第二是可见性，也就是一个线程的写入什么时候能被另一个线程可靠看见。第三是有序性，也就是跨线程观察到的事件顺序是否满足设计预期。悲观锁和乐观锁都必须回应这三个问题，只是回应方式不同：悲观锁通过进入和退出临界区建立互斥与同步关系；乐观锁通过原子比较更新、版本校验、`volatile` 语义或数据库式版本字段等方式建立提交点。

以计数器递增为例，`count++` 在 Java 源码中像是一个操作，实际可拆成读取、加一、写回。如果两个线程同时读取到 `10`，各自加一后都写回 `11`，就丢失了一次更新。悲观锁会让一个线程进入锁保护区完成整个递增，另一个线程等待锁释放后再递增，因此结果变成 `12`。乐观锁会让两个线程都先读取 `10`，但只有一个线程能通过“当前值仍然是 10 吗”的比较并写入 `11`，另一个线程发现比较失败后重新读取 `11`，再尝试写入 `12`。两者都能得到正确结果，但等待成本、失败成本、调度行为完全不同。

更复杂的共享状态往往不是单个整数，而是多个字段之间的关系。例如对象有 `state`、`owner`、`version` 三个字段，要求三者必须一起变化。此时单个 `AtomicInteger` 不能自动维护整体不变式；如果要走乐观路线，通常需要把相关字段封装成不可变快照，再用 `AtomicReference` 一次性替换；如果不适合快照替换，就需要用悲观锁包住多个字段的读写。并发控制的重点不是“用了哪个工具”，而是工具是否覆盖了真正的不变式边界。

## 两种冲突假设

悲观锁的名字来自它对冲突的假设：共享状态一旦允许并发访问，就很可能发生冲突，因此应先获得独占权，再执行读写。这个独占权可以由对象监视器、显式锁、读写锁的写锁、数据库行锁或其他互斥机制提供。线程没有拿到独占权时，不能进入临界区；它要么阻塞等待，要么限时等待，要么尝试失败后返回。悲观策略把冲突前置处理，换来的是临界区内部代码更接近单线程推理。

乐观锁的名字来自另一种假设：共享状态虽然可能被多个线程访问，但多数时候不会冲突，或者即使冲突，重新尝试也比排队阻塞便宜。因此线程可以先不加互斥锁，读取当前状态并计算新状态，最后通过某种校验确认“我计算时依赖的状态没有被别人改过”。校验成功就提交，校验失败就重读、重算、重试或放弃。乐观策略把冲突后置处理，换来的是无冲突路径更短、线程更少进入内核阻塞，但失败路径必须被认真设计。

这两个假设会直接影响代码结构。悲观锁代码通常呈现为“加锁、检查条件、修改状态、释放锁”，其中释放动作必须放在 `finally` 中。乐观锁代码通常呈现为“读取旧值、计算新值、比较提交、失败重试”，其中重试循环必须控制副作用、退避、终止条件和失败上报。悲观锁的危险点常在锁范围过大、锁顺序错误、等待条件错误、异常路径未释放；乐观锁的危险点常在无限自旋、ABA 问题、重试体副作用、版本号溢出、多个变量不变式没有一起校验。

冲突假设还会改变性能形态。悲观锁在竞争激烈时会让线程排队，吞吐可能下降，但 CPU 不一定被大量浪费，因为等待线程通常会阻塞或挂起。乐观锁在竞争轻微时吞吐很好，因为成功路径不需要排队；但在竞争激烈时，大量线程可能反复读取、计算、CAS 失败、再读取，消耗 CPU 却没有完成有效工作。所谓“锁竞争”不只是等待时间问题，也可能是处理器缓存一致性流量、上下文切换、线程调度和重试风暴共同造成的问题。

## 悲观锁的基本模型

悲观锁模型的核心是临界区。临界区是一段必须在互斥条件下执行的代码，线程进入前要获得锁，离开后要释放锁。锁保护的不是代码本身，而是某份共享状态的不变式；代码只是访问这份状态的路径。正确的悲观锁设计首先要回答“这把锁保护哪些字段或资源”，其次才是“使用 `synchronized` 还是 `ReentrantLock`”。如果多个字段共同构成一个不变式，它们通常应该受同一把锁保护；如果多个独立资源没有共同不变式，盲目共用一把大锁会降低并发度。

`synchronized` 是 Java 语言层面的互斥机制，进入同步方法或同步代码块时会获取对象监视器，退出时释放监视器。它既提供互斥，也提供可见性：一个线程释放某个监视器之前的写入，对随后获取同一监视器的线程可见。这里的“同一”非常关键。如果写线程锁的是 `lockA`，读线程锁的是 `lockB`，即使两个锁对象名字相似，也没有建立预期的同步关系。悲观锁的正确性往往败在锁对象不一致、锁对象逃逸、锁粒度含糊这些细节上。

`ReentrantLock` 是标准库中的显式锁。它与 `synchronized` 一样可以表达独占互斥和可重入，但额外提供了可中断获取、限时获取、公平模式、多个 `Condition` 等能力。显式锁的代价是必须手动释放，因此典型写法一定是 `lock.lock()` 后紧跟 `try`，在 `finally` 中 `unlock()`。如果中间代码抛出异常而没有释放锁，其他线程可能永久等待。显式锁不是“更高级所以默认更好”，它适合需要这些额外能力的场景；普通短临界区用 `synchronized` 往往更直接。

```java
final class PessimisticCounter {
    private int value;

    public synchronized int incrementAndGet() {
        value++;
        return value;
    }

    public synchronized int get() {
        return value;
    }
}
```

这个例子简单，却包含悲观锁的基本边界：`value` 的读写都通过同一个对象监视器保护，递增的读取、加一、写回不会与另一个递增交错，`get` 也能看见之前同步方法释放锁前的写入。若把 `get` 改成非同步方法，它就不再与写入端形成同一把锁上的可见性配对，代码可能在某些情况下读到旧值。悲观锁不是只给写操作加锁；只要读操作需要观察一致状态，也必须纳入同一同步协议。

## synchronized 的语义边界

`synchronized` 的优势是语义紧凑。同步代码块正常退出或异常退出时都会自动释放监视器，开发者不需要手写 `unlock`。同步方法的锁对象由方法类型决定：实例同步方法锁当前实例，静态同步方法锁对应的 `Class` 对象，同步代码块锁括号中的对象引用。理解这一点很重要，因为“看起来在同一个类里”不等于“锁的是同一把锁”。如果一个方法用 `synchronized (this)`，另一个方法用 `synchronized (new Object())`，后者每次都锁新对象，等于没有保护共享状态。

`synchronized` 是可重入的。线程已经持有某个对象监视器时，可以再次进入同一监视器保护的代码，退出次数与进入次数匹配后才真正释放。这让同步方法之间相互调用不会把自己锁死，但也可能掩盖临界区过大的问题。可重入不是鼓励在锁内随意调用复杂方法，而是保证必要的嵌套调用能够工作。持锁调用外部可重写方法、回调、监听器或可能阻塞的代码，仍然是需要谨慎避免的设计，因为调用方无法控制外部代码会不会再获取其他锁、等待 I/O 或反向调用当前对象。

`synchronized` 还提供等待与通知机制：`wait` 会释放当前监视器并进入等待，`notify` 或 `notifyAll` 会唤醒等待在同一监视器上的线程。但等待条件必须用循环检查，不能用 `if`。原因包括虚假唤醒、多个等待者竞争同一条件、被唤醒后条件又被其他线程改变等。虽然本文主题不是 wait/notify，但它体现了悲观锁的一条重要原则：锁只提供互斥，条件是否满足仍要由受锁保护的状态来判断。

```java
final class BoundedSlot<T> {
    private T item;

    public synchronized void put(T value) throws InterruptedException {
        while (item != null) {
            wait();
        }
        item = value;
        notifyAll();
    }

    public synchronized T take() throws InterruptedException {
        while (item == null) {
            wait();
        }
        T value = item;
        item = null;
        notifyAll();
        return value;
    }
}
```

这里的锁保护 `item` 和“槽位是否为空”这个条件。`put` 和 `take` 都在同一监视器内检查条件、修改状态、通知等待者。若用 `if` 替代 `while`，线程被唤醒后可能不重新确认条件，进而覆盖未取走的元素或取到空值。悲观锁的阻塞等待不是“睡一会儿再试”，而是围绕明确条件建立等待队列和唤醒规则。

## ReentrantLock 的能力与代价

`ReentrantLock` 适合需要更细控制的场景。`lockInterruptibly` 允许线程在等待锁时响应中断，这对需要可取消等待的任务很有价值。`tryLock` 可以立即尝试获取锁，失败后执行降级逻辑；带超时的 `tryLock` 可以避免无界等待。公平锁模式可以让等待时间更可预测，减少线程长期插队的概率。多个 `Condition` 可以把不同条件拆成不同等待队列，避免无关线程被频繁唤醒。

这些能力并不免费。显式锁使代码多了释放责任，也使锁的生命周期更容易跨方法、跨对象扩散。如果一个方法获取锁后调用另一个方法，而另一个方法又在复杂分支中释放锁，代码会很难推理。一般来说，`ReentrantLock` 应保持清晰的结构：谁获取谁释放，获取和释放尽量在同一层代码中出现，条件等待与状态修改都在锁内完成。只有当 `synchronized` 无法表达需求时，再使用显式锁的扩展能力。

```java
final class TimedUpdater {
    private final java.util.concurrent.locks.ReentrantLock lock =
            new java.util.concurrent.locks.ReentrantLock();
    private int value;

    public boolean updateWithin(long timeout, java.util.concurrent.TimeUnit unit)
            throws InterruptedException {
        if (!lock.tryLock(timeout, unit)) {
            return false;
        }
        try {
            value++;
            return true;
        } finally {
            lock.unlock();
        }
    }
}
```

这段代码表达了一个 `synchronized` 不便表达的语义：如果在限定时间内拿不到锁，调用方可以得到 `false` 并走其他路径，而不是无限等待。对服务端、任务调度、资源池等通用 Java 场景来说，限时等待常常比无限阻塞更容易诊断和恢复。但只要使用显式锁，`finally` 就不是风格问题，而是正确性要求。

公平模式需要单独理解。公平锁倾向于按等待顺序授予锁，可以降低饥饿风险，但通常会牺牲吞吐。非公平锁允许刚到达的线程在合适时机直接竞争锁，可能减少线程切换并提高吞吐，但某些线程的等待时间可能更不稳定。公平与非公平不是道德判断，而是吞吐和延迟分布之间的权衡：如果系统更关心整体吞吐且临界区短，非公平常见；如果系统更关心等待上限和请求顺序，公平性可能更重要。

## 乐观锁的基本模型

乐观锁模型的核心是比较提交。线程读取共享状态时不先排他，而是得到一个旧值或旧版本；随后基于旧值计算新值；提交时检查共享状态是否仍然等于旧值或旧版本；如果是，就原子地改成新值；如果不是，说明其他线程已经修改过，当前计算结果可能建立在过期前提上，必须失败、重试或重新合并。这个提交点必须是原子的，否则两个线程可能同时认为自己校验成功。

Java 标准库中的原子类主要通过 CAS 表达这一模型。CAS 的全称是 compare-and-swap 或 compare-and-set，含义是“如果当前位置的值等于期望值，就把它改成新值”。对调用者而言，它返回成功或失败；对并发控制而言，它把比较和写入合成了一个不可分割的硬件级或运行时支持的原子动作。`AtomicInteger.compareAndSet(expect, update)`、`AtomicReference.compareAndSet(expect, update)` 都是典型 API。

```java
final class OptimisticCounter {
    private final java.util.concurrent.atomic.AtomicInteger value =
            new java.util.concurrent.atomic.AtomicInteger();

    public int incrementAndGet() {
        while (true) {
            int current = value.get();
            int next = current + 1;
            if (value.compareAndSet(current, next)) {
                return next;
            }
        }
    }
}
```

这个例子与悲观计数器的结果相同，但过程不同。多个线程可以同时读取当前值并计算 `next`，只有一个线程能成功把旧值替换成新值，失败线程继续循环。无竞争时，整个更新不需要进入阻塞队列；有竞争时，失败线程不会被锁挂起，而是立即重新尝试。它适合更新很短、计算便宜、冲突相对有限的场景。如果把循环体中的计算换成昂贵操作或不可重复操作，乐观锁的优势就会迅速消失。

乐观锁不是“不加锁所以没有同步”。原子类的读写和 CAS 具有明确的内存语义，能保证对该变量的原子更新和必要的可见性。问题在于它通常只保护单个原子变量或一次引用替换。若共享不变式分散在多个普通字段中，只对其中一个字段 CAS 并不能让整体状态安全。正确的乐观设计常把多个字段放进不可变对象，再用 `AtomicReference` 替换整个快照。

## CAS、版本号与快照更新

CAS 适合“当前位置的旧值足以判断是否冲突”的情况。对整数计数器来说，旧值就是版本；对引用状态来说，旧引用可以代表一个快照。如果状态对象不可变，那么线程读到旧引用后，旧对象内部不会再变化；线程可以基于旧对象创建新对象，最后用 CAS 把引用从旧对象换成新对象。这样可以一次性维护多个字段之间的不变式。

```java
final class VersionedState {
    final long version;
    final String owner;
    final int permits;

    VersionedState(long version, String owner, int permits) {
        this.version = version;
        this.owner = owner;
        this.permits = permits;
    }
}

final class OptimisticStateHolder {
    private final java.util.concurrent.atomic.AtomicReference<VersionedState> state =
            new java.util.concurrent.atomic.AtomicReference<>(
                    new VersionedState(0L, "none", 0));

    public boolean addPermitFor(String owner) {
        while (true) {
            VersionedState current = state.get();
            VersionedState next = new VersionedState(
                    current.version + 1,
                    owner,
                    current.permits + 1);
            if (state.compareAndSet(current, next)) {
                return true;
            }
        }
    }
}
```

这类写法把 `version`、`owner`、`permits` 打包成一个不可变快照，避免了“一个字段更新成功，另一个字段还没更新就被读到”的中间状态。`version` 在这里不仅是数字，也是一种冲突识别方式：状态每变化一次，版本推进一次。很多系统中的乐观锁都使用版本号字段，例如读取记录时拿到 `version=7`，提交时要求“只有当前仍为 7 才能改成 8”。如果已经变成 8，说明有其他更新先完成，当前提交必须重新读取或告知调用方冲突。

版本号策略比直接比较业务值更能表达历史变化。假设一个值从 A 改到 B，又从 B 改回 A。只比较业务值时，线程可能误以为“仍然是 A，所以没有变过”；但版本号会从 1 变到 3，能够识别中间发生过变化。这就是乐观锁中的 ABA 问题。ABA 并不总是错误：如果业务只关心当前值，A 变 B 再变 A 可能无害；如果业务关心资源是否被释放后重新分配、节点是否被移除又插入、状态是否经历过某个阶段，ABA 就可能破坏正确性。Java 提供 `AtomicStampedReference`、`AtomicMarkableReference` 等工具来把引用与标记或版本一起比较，但是否需要它们取决于不变式是否关心历史变化。

版本号也有边界。版本字段可能溢出，虽然 `long` 溢出在多数业务周期内不现实，但理论上仍要理解；版本只在所有写路径都遵守同一协议时有效，如果某条路径绕过版本直接修改状态，乐观锁就失去意义；版本只能识别“发生过变化”，不能自动合并冲突，合并策略仍要由业务规则决定。乐观锁的难点通常不在 CAS 调用本身，而在失败后怎么处理冲突。

## 阻塞与重试

悲观锁和乐观锁最直观的差异是阻塞与重试。悲观锁在线程拿不到锁时通常会等待，等待线程可能进入 `BLOCKED`、`WAITING` 或 `TIMED_WAITING` 等状态，具体取决于使用的同步工具。等待期间线程不继续执行临界区代码，CPU 消耗较低，但可能发生上下文切换、调度延迟和锁队列排队。线程被唤醒后也不代表立即运行，它还要等待调度器分配 CPU，并可能再次竞争锁。

乐观锁失败时通常不阻塞，而是重试。重试的好处是避免挂起和恢复线程，在临界区极短、冲突很低时能获得很高吞吐。重试的代价是失败线程仍在占用 CPU，并且每次 CAS 失败都说明有一次计算结果作废。竞争越激烈，作废计算越多，处理器缓存行在多个核心之间来回失效，吞吐可能出现明显下降。所谓“无锁”并不等于“无等待”，更不等于“无成本”；它只是没有使用互斥锁把线程挂起。

设计乐观重试时，需要给循环加上现实边界。边界可以是最大重试次数、指数退避、让出 CPU、失败后降级到悲观锁、或者把冲突返回给调用方。无限自旋只适合非常短、预期很快成功的更新。如果更新逻辑包含较多计算，或者系统已经处于过载状态，持续自旋会让线程忙于失败重试，反而挤占真正能完成工作的线程。乐观策略需要认真对待失败路径，而不是把 `while (true)` 当作万能模板。

```java
public boolean incrementWithLimit(java.util.concurrent.atomic.AtomicInteger value,
                                  int limit,
                                  int maxRetries) {
    for (int retries = 0; retries < maxRetries; retries++) {
        int current = value.get();
        if (current >= limit) {
            return false;
        }
        if (value.compareAndSet(current, current + 1)) {
            return true;
        }
        Thread.onSpinWait();
    }
    return false;
}
```

这段代码展示了两个边界。第一，业务条件 `current >= limit` 必须在每次重读后重新判断，因为失败后旧判断已经过期。第二，重试次数有限，失败后把控制权交还调用方。`Thread.onSpinWait()` 只是给运行时和处理器一个提示，不是正确性保障，也不应该被理解成通用性能优化开关。是否使用退避，要根据竞争程度和延迟目标评估。

## 吞吐、延迟与竞争强度

吞吐关注单位时间完成多少操作，延迟关注单个操作从开始到结束花多久。悲观锁和乐观锁对这两个指标的影响不同。悲观锁在低竞争下成本可能很小，尤其是短临界区、锁对象稳定、JIT 能优化的场景；在高竞争下，线程排队会降低并行度，但等待者不会一直执行失败计算。乐观锁在低竞争下通常吞吐高、延迟低；在高竞争下，平均延迟和尾部延迟可能因为反复失败而变差。

冲突强度取决于多个因素。线程数量越多，访问同一热点状态的概率越高；临界区或计算窗口越长，其他线程插入修改的机会越大；状态越集中，热点越明显；重试越积极，竞争越可能自我放大。一个单独的全局计数器在几十个线程下可能成为热点；把计数分散到多个分片再汇总，可能比纠结悲观锁或 CAS 更有效。并发设计的第一步常常不是选择锁类型，而是减少共享热点。

悲观锁的延迟风险主要来自排队、持锁时间过长、锁顺序不当和锁内阻塞。如果一个线程持锁执行慢 I/O 或等待其他任务，所有需要同一把锁的线程都会被拖住。乐观锁的延迟风险主要来自重试风暴、缓存行争用、失败后没有退避、以及调用方无法感知冲突。两者都可能出现尾延迟问题，只是形态不同：悲观锁表现为等锁时间长，乐观锁表现为多次失败后才成功或最终失败。

评价性能时要避免只看单线程或低并发测试。很多 CAS 代码在两个线程下表现很好，在几十个线程同时打同一热点时反而不如一把锁稳定。很多锁代码在微基准中看起来慢，但真实业务中临界区本来包含不可拆分的不变式，使用锁反而减少重试和补偿逻辑。性能判断应基于冲突概率、临界区耗时、线程数量、失败代价、延迟目标和可观测指标，而不是基于“乐观”或“悲观”的名字。

## 饥饿、活锁与死锁

悲观锁常被讨论死锁，乐观锁常被讨论活锁，但两者都可能出现进展性问题。死锁是多个线程互相等待对方持有的资源，谁都无法继续。典型形式是线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1。解决思路包括固定锁顺序、缩小锁范围、避免持锁调用外部代码、使用限时 `tryLock`、在失败时释放已持有资源后重试。死锁不是锁工具本身的必然结果，而是资源获取顺序和等待条件形成了环。

饥饿是某些线程长期得不到执行机会或资源。非公平锁在高竞争下可能让新来的线程多次插队，等待队列中的某些线程长时间拿不到锁；线程优先级、任务队列策略、长时间持锁也可能造成饥饿。公平锁能降低锁层面的饥饿概率，但不能保证整个业务系统绝对公平，因为线程调度、上游流量和锁外工作仍会影响完成顺序。选择公平锁时要接受吞吐下降的可能。

活锁是线程没有阻塞，表面上都在运行，却因为彼此不断让步或重试而没有实质进展。乐观锁在高竞争下的无限 CAS 失败可以表现出类似活锁的症状：线程不断读、算、失败、再读、再算，CPU 很忙但成功提交很少。另一个例子是多个线程用 `tryLock` 获取多把锁，失败后同时释放并立即重试，如果没有退避和顺序控制，就可能反复撞在一起。活锁的处理思路通常是引入退避、随机化、优先级、队列化或降级到悲观路径。

乐观锁还可能出现单个线程饥饿。CAS 不保证每个线程最终成功，只保证某次成功意味着状态被某个线程推进了。在极端竞争下，一个线程可能连续失败很多次。对于计数器这类操作，这通常可以接受；对于用户可见请求或有超时要求的任务，必须考虑失败上限和返回策略。无锁算法常强调系统整体进展，但整体进展不等于每个线程公平进展。

## 适用场景

悲观锁适合冲突概率高、临界区包含多个相关状态、失败后难以重试、必须保持强一致不变式的场景。比如多个字段要一起检查和修改，操作中需要维护集合结构，更新过程包含不可重复的资源变更，或者调用方不能接受冲突后重复执行。悲观锁让进入临界区的线程拥有明确独占权，代码推理更直接。只要锁范围控制合理、锁顺序清楚、等待有边界，它是一种非常可靠的并发控制方式。

乐观锁适合冲突概率低、更新逻辑短小、状态可以用单个原子变量或不可变快照表达、失败后可以安全重试的场景。典型例子包括计数器、状态标志切换、无阻塞栈队列中的节点引用更新、缓存条目的版本化替换、读多写少且写冲突少的配置快照。乐观锁的理想路径是读取和提交之间窗口很短，重试体没有外部副作用，失败后重新计算成本很低。

读多写少场景需要细分。很多人会直接认为读多写少适合乐观锁，但这取决于读操作是否需要强一致快照、写操作是否昂贵、读写是否冲突。`ReadWriteLock` 也是悲观策略的一种扩展：多个读线程可共享读锁，写线程获取写锁时排斥其他读写线程。它适合读操作需要锁保护但读之间不冲突的场景。`StampedLock` 提供乐观读能力，但它的使用规则更复杂，尤其要注意验证戳和不可重入等边界。不能只因为名字里有 optimistic 就忽略校验要求。

对简单数值累加，`AtomicInteger` 或 `LongAdder` 常比显式锁更合适；对多个字段的一致更新，单个原子类通常不够；对复杂对象图和条件等待，锁往往更清晰；对只读快照和偶尔替换，`AtomicReference` 持有不可变对象可能很优雅。适用场景不是由 API 名称决定，而是由共享状态形状、冲突概率和失败处理能力决定。

## 边界：单变量、复合操作与业务不变式

并发控制最容易被低估的边界是“单次操作安全”与“复合逻辑安全”的区别。`ConcurrentHashMap` 的单次 `put`、`get` 是线程安全的，`AtomicInteger` 的单次 CAS 是原子的，锁保护的一段代码是互斥的。但业务逻辑通常由多个操作组成，例如“如果不存在则创建并初始化”“如果余额足够则扣减并记录”“如果状态为 NEW 则改为 RUNNING 并设置开始时间”。这些逻辑只有在整体不变式被保护时才安全。

悲观锁处理复合逻辑的方式比较直接：把检查和修改放进同一把锁保护的临界区。只要所有访问路径都遵守这把锁，读取到的条件在修改前不会被其他线程改变。乐观锁处理复合逻辑时要确保提交校验覆盖全部依赖。如果只校验其中一个字段，其他字段可能已经变化，计算结果就不再可靠。把相关字段封装为不可变快照，是乐观策略常用的边界收紧方式。

还有一种常见误区是“读不修改，所以不用同步”。如果读操作只读取单个 `volatile` 或原子变量，可能成立；如果读操作要观察多个字段之间的一致关系，就必须同步或读取一致快照。否则读线程可能看到字段 A 的新值和字段 B 的旧值，拼出一个从未真实存在过的状态。悲观锁用同一把锁保护读写，乐观快照用不可变对象一次性发布多个字段，都是为了解决这个问题。

业务不变式还可能跨越内存外资源，例如文件、网络连接、事务、外部服务调用。把这些操作放进锁内会增加持锁时间，放到锁外又可能破坏状态一致性。通用原则是：锁内只做必须与共享状态原子绑定的短操作，锁外做耗时或可失败的外部操作；如果必须跨外部资源保持一致，就需要更高层的事务、补偿或状态机设计，而不是简单扩大锁范围。悲观锁和乐观锁都不是分布式一致性或业务事务的替代品。

## 内存可见性与 happens-before

悲观锁和乐观锁的正确性都依赖 Java 内存模型。`synchronized` 的规则是：对同一个监视器的解锁 happens-before 后续对该监视器的加锁。这意味着线程 A 在同步块内写入共享变量并退出后，线程 B 进入同一监视器保护的同步块时，能看到这些写入。`ReentrantLock` 的加锁和解锁在内存语义上也提供类似效果。锁的互斥让复合操作不被交错，锁的同步语义让写入可靠发布给后续获取同一锁的线程。

原子类的操作也有内存语义。以常用原子变量为例，成功的 CAS 不只是改值，还会建立与该原子变量相关的可见性保证。`volatile` 读写、原子读写、CAS 等机制使线程之间能围绕某个变量建立发布和观察关系。但这不意味着所有普通字段都自动安全。若普通字段的写入发生在发布不可变快照之前，并且读线程通过原子引用读取到该快照，那么字段通过对象不可变性和安全发布得到保证；若普通字段被单独共享且没有同步，就仍可能发生数据竞争。

可见性不等于互斥。`volatile boolean stopped` 很适合表达停止标志，因为写线程设置为 `true` 后，读线程能及时看到；但 `volatile int count` 的 `count++` 仍然不是原子的，因为递增包含读、改、写。原子性也不等于完整业务安全。`AtomicInteger` 能保证单个数值更新不丢失，却不能自动保证另一个集合、日志或状态字段与这个数值同步变化。选择悲观锁还是乐观锁前，应先拆清楚需求是可见性、原子性、一致快照、条件等待，还是这些能力的组合。

有序性同样重要。编译器和处理器可以在不改变单线程语义的前提下重排序，Java 内存模型允许优化，但要求同步动作提供跨线程顺序约束。锁释放、锁获取、`volatile` 写读、线程启动、线程终止、并发工具内部同步，都是建立顺序的方式。没有这些同步关系时，不能用“代码写在前面”推断另一个线程一定先看到它。

## 锁粒度、锁分段与热点拆分

悲观锁的一个关键设计是锁粒度。锁粒度越粗，保护边界越清晰，出错概率越低，但并发度可能下降；锁粒度越细，并发度可能提高，但死锁、遗漏保护和不变式割裂的风险会上升。一个对象只有一把大锁时，所有操作串行化；多个字段分别加锁时，独立操作可以并发，但跨字段操作需要同时获取多把锁。是否拆锁，要看这些状态是否真的独立，以及跨状态不变式是否频繁出现。

锁分段是一种折中思路：把热点状态按某种规则分到多个段，每段有自己的锁。这样不同段上的操作可以并发，同一段上的操作仍受互斥保护。分段适合状态天然可分区的场景，例如按键的哈希分布保护多个桶。它不适合需要频繁全局一致视图的场景，因为全局操作可能要获取所有段锁，复杂度和死锁风险都会上升。分段不是免费优化，它把一个大问题拆成多个小问题，同时引入了全局协调成本。

乐观锁也有热点拆分问题。单个全局 CAS 变量在高并发下会成为缓存一致性热点。多个线程不断修改同一缓存行，CAS 失败率和总线流量都会上升。`LongAdder` 这类工具通过分散计数热点提升高并发累加吞吐，读取总和时再汇总多个分片。它牺牲的是某些时刻的精确即时视图和更复杂的内部结构，换来高竞争下更好的写入扩展性。这说明并发性能常常来自减少热点，而不只是替换锁类型。

锁粒度还影响可维护性。过粗的锁可能让性能瓶颈集中，过细的锁可能让开发者无法判断某个字段应该受哪把锁保护。实践中应优先让锁与不变式对应，而不是机械追求细粒度。只有当监控或压力测试证明锁竞争成为问题，并且状态边界可以清晰拆分时，再做分段、快照化或无锁化改造。

## 失败路径与副作用控制

悲观锁的失败路径主要包括获取锁被中断、限时获取失败、临界区抛异常、条件等待超时、关闭阶段仍有线程等待。处理原则是释放已经持有的资源，恢复必要的中断语义，向调用方报告明确结果。`synchronized` 会自动释放监视器，但锁内获得的其他资源仍需释放；`ReentrantLock` 必须在 `finally` 中释放；条件等待被中断后，应根据当前层职责决定是清理并抛出，还是恢复中断并返回。

乐观锁的失败路径是提交冲突。失败后不能简单地“再执行一遍”所有逻辑，必须区分纯计算和副作用。纯计算可以重复，例如根据旧数值算新数值；不可重复副作用不能放进重试体，例如发送消息、写文件、调用外部服务、扣减另一个资源、记录不可撤销日志。如果这些副作用在 CAS 前执行，CAS 失败后副作用已经发生但状态没有提交；如果重试多次，副作用可能发生多次。正确做法通常是先在本地纯计算，CAS 成功后再执行副作用，或者把副作用设计成幂等并能根据版本去重。

即使 CAS 成功后执行副作用，也要考虑副作用失败。共享状态已经更新，但后续动作失败，系统是否允许这种中间结果？如果不允许，就需要补偿、事务或状态机，而不是指望 CAS 解决所有问题。乐观锁特别适合保护内存状态更新，但它不自动解决跨系统、跨资源的一致性问题。悲观锁同样如此：锁只能约束同一进程内遵守协议的线程，不能让外部操作天然原子。

重试策略还要考虑调用方体验。对后台统计计数，失败后多重试几次通常没问题；对用户请求或有时限的任务，长时间自旋会扩大延迟；对批处理任务，冲突可以汇总后重新分片处理；对状态机转换，冲突可能意味着调用方拿到的前置条件过期，应返回明确错误。失败路径不是异常情况的附属品，而是乐观并发控制的正常组成部分。

## 典型误区

第一个误区是把悲观锁等同于低性能。现代 JVM 对锁有大量优化，短小、低竞争、对象不逃逸的同步代码不一定昂贵。更重要的是，性能不能脱离正确性讨论。若某个操作必须维护复杂不变式，使用一把清晰的锁可能比一组脆弱的原子变量更快也更可靠，因为后者在冲突时需要大量重试和补偿。锁的性能问题应通过测量确认，而不是通过名称预设。

第二个误区是把乐观锁等同于无锁高性能。CAS 循环在低竞争下很快，在高竞争下可能浪费大量 CPU。无锁算法通常需要更严格的数据结构设计和内存语义推理，难度不低。把普通业务逻辑包进无限 CAS 循环，往往只是把阻塞等待换成忙等失败。尤其当重试体包含对象分配、复杂计算或副作用时，乐观锁可能比悲观锁更差。

第三个误区是只给写加锁，不给读加锁。若读操作需要一致视图，它也必须使用同一同步协议。没有同步的读可能看到旧值、部分更新或跨字段不一致。只要读结果参与决策，读的线程安全就和写一样重要。正确做法可以是读也加同一把锁，或者使用不可变快照、`volatile` 引用、原子引用等方式发布一致状态。

第四个误区是锁对象选择随意。使用字符串字面量、可变字段、公开对象、装箱对象或每次新建的对象作为锁，都可能带来问题。公开锁对象可能被外部代码持有，导致不可控阻塞；可变锁字段可能让不同线程锁住不同对象；每次新建锁对象则根本不能互斥。锁对象应是私有、稳定、语义清晰的对象，除非有明确理由暴露同步协议。

第五个误区是在持锁期间调用外部代码。外部代码可能耗时、阻塞、抛异常、尝试获取其他锁，甚至回调当前对象。这样会扩大锁范围并引入死锁风险。更稳妥的做法是锁内完成状态检查和必要变更，复制需要的数据，释放锁后再执行外部调用。若外部调用结果必须影响状态，再以新的同步步骤提交结果。

第六个误区是忽略 ABA。对简单计数器，ABA 通常不是问题；对引用节点、资源槽位、状态阶段，ABA 可能非常严重。判断是否需要版本号的标准不是“有没有 CAS”，而是“状态从 A 离开又回到 A 时，当前线程是否仍可把它当作未变化”。如果答案是否定的，就需要版本、标记、不可变快照或更强的同步策略。

第七个误区是把并发容器当作业务事务。并发容器保证自身结构安全，并为单个操作提供线程安全，但不自动保证多个操作组成的业务流程安全。`map.get` 后再 `map.put` 的检查更新，如果没有使用 `compute`、`putIfAbsent` 或外部同步，就可能被其他线程插入。工具的保证边界必须和业务不变式边界对齐。

## 选择方法

选择悲观锁还是乐观锁，可以从几个问题开始。第一，共享状态能否用单个原子变量或不可变快照表达？如果不能，悲观锁通常更简单。第二，冲突概率是否低？如果大量线程同时修改同一热点，乐观重试可能很快退化。第三，失败后能否安全重试？如果重试会重复副作用或成本很高，应倾向悲观锁或显式冲突返回。第四，是否需要条件等待、可中断等待、限时等待、公平性或多个等待队列？如果需要，锁和同步器的表达力更强。第五，延迟目标是什么？如果不能接受无限等待或无限重试，两种策略都必须设置边界。

可以把选择过程概括为一句话：先按不变式选择正确性边界，再按竞争形态优化性能。若不变式跨多个字段，先用一把锁写清楚；若状态天然是单值更新，再考虑原子类；若读多写少且读需要一致性，考虑读写锁或不可变快照；若热点严重，考虑分段、分片、批量合并或减少共享，而不是只替换 API。并发设计不应从“我要用 CAS”或“我要用锁”开始，而应从“我要保护哪个状态关系”开始。

在代码评审中，也可以用同样的问题检查。锁保护的字段是否全部通过同一把锁访问？锁内是否有慢操作或外部调用？显式锁是否一定释放？条件等待是否用循环检查？CAS 重试体是否无副作用？CAS 失败是否有上限或退避？版本号是否覆盖所有写路径？读操作是否能看到一致状态？这些问题比争论“悲观还是乐观”更能发现实际缺陷。

## 小结

悲观锁和乐观锁是两种并发控制策略。悲观锁假设冲突值得提前阻止，通过互斥和阻塞等待让临界区内部更容易推理；乐观锁假设冲突较少或可重试，通过比较提交和版本校验提升无冲突路径的并发度。二者都能写出正确代码，也都能写出问题代码。真正的判断依据是共享状态形状、冲突概率、临界区成本、失败代价、可见性需求、进展性要求和诊断能力。

在 Java 中，`synchronized` 和 `ReentrantLock` 是理解悲观锁的核心入口。前者语义简洁，适合普通互斥；后者提供可中断、限时、公平和条件队列等扩展能力，但需要手动释放。`CAS`、原子类、不可变快照和版本号是理解乐观锁的核心入口。它们能把比较和提交合成原子动作，但不能自动解决多变量不变式、副作用、ABA 和冲突合并问题。

掌握这个主题后，应该能够解释一段并发代码为什么安全：哪份状态被保护，读写是否遵守同一协议，写入如何对其他线程可见，冲突发生时线程是阻塞还是重试，失败路径是否释放资源或返回明确结果，饥饿、活锁和死锁风险如何控制。只要这些问题能被具体回答，悲观锁和乐观锁就不再是概念标签，而是可以落到工程实践中的设计工具。

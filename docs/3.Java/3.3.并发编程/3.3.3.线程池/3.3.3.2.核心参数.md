# 3.3.3.2 核心参数

ThreadPoolExecutor 的核心参数不是几个可以随手填写的数字，而是一套容量控制模型。它们共同决定任务提交后是直接创建线程、进入队列、继续扩容，还是被拒绝；也决定线程什么时候保留、什么时候回收、异常线程如何补位、线程命名和上下文如何建立、系统过载时压力如何被暴露出来。理解线程池时，如果只记住“核心线程数、最大线程数、队列、拒绝策略”这些名词，很容易把配置写成经验值；真正可靠的配置要从任务来源、任务耗时、等待比例、资源上限、延迟目标和失败处理方式推导出来。

本文只讨论通用 Java 语境下 `java.util.concurrent.ThreadPoolExecutor` 的核心参数。它适用于普通 Java 服务端程序、命令行程序、桌面程序、批处理程序以及任何使用标准 JDK 并发库的场景；不讨论特定平台、框架或运行环境的额外封装。本文的重点也不是复述构造方法签名，而是解释六个关键参数 `corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`workQueue`、`threadFactory`、`handler` 如何协同，如何从需求推导配置，哪些边界必须提前说清楚，以及哪些常见误区会把线程池从“并发工具”变成“故障放大器”。

## 线程池参数解决的核心问题

线程池首先解决的是任务执行资源的复用问题。直接为每个任务创建一个线程看起来简单，但线程不是轻量到可以无限创建的对象。线程需要栈空间，需要调度成本，需要上下文切换，需要参与操作系统层面的资源竞争。任务数量一旦超过机器和外部资源能够承受的范围，直接创建线程会把压力扩散到内存、CPU 调度、锁竞争、连接池、文件句柄和下游系统，最终表现为整体延迟上升、超时增加，甚至进程无法继续创建线程。

线程池进一步解决的是容量边界问题。一个线程池至少要回答四个问题：同时最多执行多少任务，暂时不能执行的任务在哪里等待，等待空间满了以后怎么办，空闲线程是否需要保留。`ThreadPoolExecutor` 的核心参数就是围绕这四个问题组织的。`corePoolSize` 和 `maximumPoolSize` 控制可并发执行的线程数量，`workQueue` 控制尚未执行任务的等待空间，`keepAliveTime` 控制超出保留范围的空闲线程多久回收，`threadFactory` 控制工作线程如何创建，`handler` 控制线程池无法接收新任务时的处理方式。

线程池还承担背压表达的职责。背压不是某个具体 API，而是一种系统设计要求：当下游或执行资源处理不过来时，上游必须感知压力，不能无限制地继续投递工作。线程池的背压可以通过有界队列、有限最大线程数、拒绝策略、提交线程阻塞或调用方降级来表达。如果参数配置错误，压力不会消失，只会被转移。无界队列会把压力转移成内存占用和排队延迟；过大的最大线程数会把压力转移成调度开销和资源争抢；沉默的拒绝策略会把压力转移成数据丢失；不加诊断的自定义策略会让故障位置变得不可见。

因此，理解核心参数时不能只问“这段代码会不会运行”，而要问“任务增长时会在哪里排队，排队多久，队列满后谁知道，执行线程增长到哪里停止，线程空闲后是否回收，调用方如何知道任务没有被接收”。这些问题共同构成线程池的容量模型。

## 构造方法与参数位置

最常用的构造方法如下：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4,
        16,
        30L,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(200),
        namedThreadFactory,
        new ThreadPoolExecutor.AbortPolicy()
);
```

这七个构造参数中，前六个就是线程池容量模型和线程创建模型的主体，最后一个 `RejectedExecutionHandler` 决定容量用尽时的失败方式。虽然构造方法允许只传入部分参数并使用默认线程工厂和默认拒绝策略，但生产代码中更推荐显式写出 `threadFactory` 和 `handler`。原因很简单：线程池一旦发生问题，线程名、异常归属、拒绝行为和日志记录往往比“能不能创建线程”更影响排查效率。

这些参数的直观含义可以先用一张表概括：

| 参数 | 直接含义 | 主要影响 | 常见风险 |
| --- | --- | --- | --- |
| `corePoolSize` | 核心线程数 | 平稳状态下保留的执行能力 | 过小导致长期排队，过大导致空闲线程和资源占用 |
| `maximumPoolSize` | 最大线程数 | 队列满后允许扩张到的执行能力上限 | 过大造成调度和资源争抢，配无界队列时常被弱化 |
| `keepAliveTime` | 空闲线程存活时间 | 非核心线程回收速度，或允许核心线程超时时的回收速度 | 过短导致频繁创建销毁，过长导致峰值后资源迟迟不释放 |
| `workQueue` | 等待队列 | 未执行任务的缓冲、排序和背压方式 | 无界队列隐藏过载，有界队列需要拒绝策略配合 |
| `threadFactory` | 线程工厂 | 线程命名、优先级、守护属性、异常处理、上下文设置 | 默认命名不利于诊断，错误设置 daemon 会影响生命周期 |
| `handler` | 拒绝策略 | 线程池饱和或关闭后如何处理新任务 | 静默丢弃、错误重试或阻塞可能引入更隐蔽故障 |

表格只能帮助定位概念，不能替代对执行流程的理解。`ThreadPoolExecutor` 的行为不是先把所有线程创建到最大值，也不是永远先排队再执行，而是遵循一条明确的任务提交路径。

## execute 的提交流程

理解核心参数必须从 `execute` 的决策顺序开始。一个新任务提交到线程池时，大体流程可以概括为：如果当前工作线程数小于 `corePoolSize`，优先创建核心线程执行任务；否则尝试把任务放入 `workQueue`；如果入队失败，再尝试创建非核心线程，线程总数最多到 `maximumPoolSize`；如果仍然不能接收任务，就执行拒绝策略。真实实现还会处理线程池运行状态、并发竞争下的二次检查、工作线程退出后的补位等细节，但容量决策主线就是这四步。

这个顺序非常关键。它说明 `workQueue` 的选择会影响 `maximumPoolSize` 是否有机会发挥作用。如果使用无界队列，核心线程满了以后任务通常都能成功入队，线程池就很少继续创建超过核心数的线程；此时 `maximumPoolSize` 在大多数情况下不会参与容量扩张。相反，如果使用 `SynchronousQueue` 这种不存储任务的队列，核心线程满了以后入队通常失败，线程池会更积极地创建新线程直到最大线程数。使用有界队列时，线程池会先用核心线程执行，再用队列缓冲，队列满后才扩张到最大线程数。

可以把提交流程写成简化伪代码：

```java
void execute(Runnable task) {
    if (workerCount < corePoolSize) {
        if (addWorker(task, true)) {
            return;
        }
    }

    if (isRunning() && workQueue.offer(task)) {
        if (!isRunning() && remove(task)) {
            reject(task);
        } else if (workerCount == 0) {
            addWorker(null, false);
        }
        return;
    }

    if (!addWorker(task, false)) {
        reject(task);
    }
}
```

这段伪代码要表达的不是实现逐行细节，而是三个判断点：核心线程是否还有空间，队列是否接收任务，最大线程数是否还能扩张。许多配置误解都来自忽略这个顺序。例如把 `maximumPoolSize` 设置成很大，同时使用无界 `LinkedBlockingQueue`，然后期待线程池在高峰期自动扩到最大值，这个期待通常不会实现。又比如把队列设置得极小，同时把最大线程数设置得很高，会让线程池在短时间内创建大量线程，把压力转移到调度器和共享资源。

## corePoolSize：平稳执行能力的基线

`corePoolSize` 是线程池在正常运行中优先维持的工作线程数量。只要当前工作线程数少于核心线程数，新提交的任务通常会触发创建新线程，即使已有工作线程可能很快空闲。核心线程默认不会因为空闲而超时退出，除非显式调用 `allowCoreThreadTimeOut(true)`。因此，`corePoolSize` 更像是线程池的基础执行能力，而不是简单的“最小线程数”。线程池刚创建时不会自动创建全部核心线程，除非调用 `prestartCoreThread` 或 `prestartAllCoreThreads`，但一旦任务推动核心线程创建出来，它们通常会长期保留。

配置 `corePoolSize` 时，首先要看任务类型。CPU 密集型任务主要消耗处理器时间，线程数过多不会让 CPU 变多，只会增加上下文切换和缓存失效。对于这类任务，核心线程数通常接近可用处理器数量，或者略作调整以覆盖少量等待。等待型任务则不同，任务运行中有较多时间阻塞在 I/O、锁、条件等待、远程调用或其他外部资源上，此时单个线程并不会持续占用 CPU，可以配置更多线程来覆盖等待时间，但上限必须受外部资源容量约束。混合型任务需要拆分估算，不能只凭“接口是 I/O 型”或“代码里有计算”下结论。

常见的估算思路是先得到单个任务的计算时间和等待时间，再用等待比例推导线程数量。一个简化公式是：

```text
线程数 ≈ CPU 核数 × 目标 CPU 利用率 × (1 + 等待时间 / 计算时间)
```

这个公式只能作为起点，不能作为最终答案。因为真实程序还会受到锁竞争、垃圾回收、内存带宽、连接池大小、数据库吞吐、文件系统限制、网络延迟分布等因素影响。线程池配置完成后，应通过压测或线上指标观察 CPU 利用率、任务排队时间、队列长度、拒绝次数和外部资源耗时，再迭代调整。`corePoolSize` 的目标不是越大越好，而是在平稳流量下让任务尽快得到执行，同时不制造无意义的资源竞争。

`corePoolSize` 过小的表现通常是队列长期积压，任务等待时间明显高于执行时间，CPU 或外部资源并未达到合理利用率。此时增加核心线程数可能改善吞吐。但如果队列积压的根因是下游慢、锁竞争严重或任务内部串行化，盲目增加核心线程数只会让更多线程一起等待，甚至加重争用。`corePoolSize` 过大的表现则是线程长期空闲、上下文切换增多、内存占用提高，或者大量线程同时竞争有限的连接、锁和缓存。一个线程池的核心线程数应该服从整个系统的瓶颈，而不是只服从某个局部模块的提交速度。

还要注意核心线程并不等于永不退出的线程。调用 `allowCoreThreadTimeOut(true)` 后，核心线程也会在空闲超过 `keepAliveTime` 后被回收，但此时 `keepAliveTime` 必须大于零。这个选项适合任务具有明显潮汐特征、低谷期希望释放线程资源的场景。代价是下一个流量高峰到来时，线程池需要重新创建线程，可能带来冷启动延迟。如果任务对首批请求延迟敏感，可以考虑预启动核心线程；如果任务稀疏且对延迟不敏感，可以允许核心线程超时。

## maximumPoolSize：峰值扩张能力的上限

`maximumPoolSize` 是线程池允许存在的最大工作线程数。它并不表示线程池会主动创建这么多线程，也不表示任务一多就立即扩容。它只在核心线程已满、队列无法继续接收任务、线程池仍处于运行状态时参与决策。换句话说，`maximumPoolSize` 是队列无法吸收压力之后的最后一层执行扩张空间。

这个参数的最大误区是把它当成“高峰时的线程数”。高峰时线程数是否能接近最大值，取决于队列类型和队列容量。使用无界队列时，任务几乎总能排队，线程池通常不会扩展到核心线程数之外；使用小容量有界队列时，队列很快满，线程池会更频繁地扩张；使用 `SynchronousQueue` 时，提交任务必须直接交给工作线程，没有可见缓冲，因此 `maximumPoolSize` 会成为主要容量边界。只调大 `maximumPoolSize` 而不审视 `workQueue`，经常不会产生预期效果。

`maximumPoolSize` 的合理值要由资源上限反推。线程池中的任务如果要访问数据库连接池，而连接池最多只有 30 个连接，那么把最大线程数设到 200 可能只会制造 170 个等待连接的线程；如果任务要访问一个限速外部接口，过多线程会把限流错误和重试风暴放大；如果任务大量持锁，更多线程可能只是排在锁队列上。最大线程数不是为了“尽量并发”，而是为了在可承受资源范围内提高吞吐，并在超过范围时让压力通过队列长度或拒绝策略显现。

`maximumPoolSize` 也影响故障形态。最大值过小，短时波峰会更早进入拒绝策略；最大值过大，波峰可能被大量线程短暂吸收，但代价是调度成本、内存占用和下游压力上升。前者的故障更直接，后者的故障更隐蔽。工程上更可取的做法通常是设置一个有解释的最大值，再配合有界队列和可观测的拒绝策略，让过载在边界处被及时发现。无限制扩容看起来提高成功率，实际可能把一个局部过载变成整个进程不可用。

还要注意 `maximumPoolSize` 必须大于等于 `corePoolSize`。如果通过构造方法或 setter 设置了非法关系，会抛出异常。运行期调整线程池参数时也要注意顺序：如果要把核心线程数调到高于当前最大线程数，必须先调大最大线程数；如果要把最大线程数调到低于当前核心线程数，必须先调小核心线程数。动态调整不是简单赋值，它改变的是后续容量决策，不会粗暴杀掉正在执行任务的线程。

## keepAliveTime：空闲线程回收的时间边界

`keepAliveTime` 表示工作线程空闲后等待新任务的最长时间。默认情况下，它主要作用于超过 `corePoolSize` 的非核心线程：当这些线程空闲时间超过该值，就会退出，线程池规模回落到核心线程数附近。开启核心线程超时后，它也会作用于核心线程。这个参数控制的是线程池从峰值状态回落到平稳状态的速度。

`keepAliveTime` 过短，会让线程池在流量抖动时频繁创建和销毁线程。线程创建虽然比创建进程轻，但仍然有成本，包括分配线程栈、初始化线程对象、进入调度系统、执行线程工厂逻辑等。如果任务批次间隔略大于存活时间，线程池就会反复扩张和收缩，表现为延迟波动和额外 CPU 消耗。对于周期性小波峰，适当延长存活时间可以减少抖动。

`keepAliveTime` 过长，则会让峰值期间创建的非核心线程在低谷期长时间保留。线程即使空闲，也占用一定内存和运行时管理成本；如果线程工厂还设置了上下文类加载器、ThreadLocal 或其他线程局部资源，长期保留可能使资源释放延迟。特别是任务高峰很短、低谷很长的线程池，过长的存活时间会让峰值后的资源占用不必要地持续。

这个参数要结合队列和最大线程数理解。如果使用无界队列导致线程池几乎不创建非核心线程，那么 `keepAliveTime` 对平时行为影响有限；如果使用 `SynchronousQueue` 或很小的有界队列，非核心线程会频繁出现，`keepAliveTime` 就会明显影响峰值后的收缩速度。对于低延迟系统，保留一部分热线程可以减少创建成本；对于资源敏感的批处理系统，允许更快回收可能更合适。

`keepAliveTime` 不是任务超时时间。它不会中断正在运行的任务，也不会限制一个任务最多执行多久。一个任务如果卡在外部调用或死循环中，`keepAliveTime` 不会让它退出。任务执行超时需要由任务内部、Future 获取方、外部调用库或业务取消协议处理。把线程空闲回收时间误认为任务执行时间限制，是线程池配置中非常常见的概念混淆。

## workQueue：排队、缓冲和背压的核心

`workQueue` 是线程池中最容易被低估的参数。它不仅是一个存放 `Runnable` 的容器，还决定了任务提交路径、排队延迟、内存风险、公平性、排序语义以及最大线程数是否容易触发。不同队列会让同样的 `corePoolSize` 和 `maximumPoolSize` 表现出完全不同的行为。

`ArrayBlockingQueue` 是有界数组队列，容量固定，内存边界清晰，可以选择公平锁策略。它适合需要明确队列上限、希望过载及时暴露的场景。容量设置得太小会频繁触发扩容或拒绝，容量设置得太大则会增加排队延迟。它的价值在于让等待任务的数量可控，便于建立明确的背压模型。

`LinkedBlockingQueue` 可以有界，也可以无界。无参构造时容量接近整数最大值，在工程意义上通常等同于无界队列。无界队列会让任务提交看起来很顺畅，因为核心线程满后任务可以继续入队，但这也意味着过载不会立刻暴露，而是以队列长度、内存占用和等待时间的形式逐步积累。使用无界队列时，`maximumPoolSize` 往往很少发挥作用。若确实要使用链表队列，更推荐显式给出容量。

`SynchronousQueue` 不存储任务，每一次提交都需要和一个接收任务的工作线程直接配对。它适合任务不希望排队、希望把压力直接转化为线程扩张或拒绝的场景。使用它时，`maximumPoolSize` 是非常关键的安全边界；如果最大线程数设置过大，短时提交洪峰可能创建大量线程；如果最大线程数设置过小，拒绝会非常直接。它强调的是直接移交，而不是缓冲。

`PriorityBlockingQueue` 支持按优先级取任务，但默认无界。它可以表达任务优先级，却也引入饥饿和容量风险。低优先级任务在高优先级任务持续到来时可能长期得不到执行；无界容量会隐藏过载；优先级比较逻辑如果不稳定，还可能导致执行顺序难以解释。使用优先级队列时，必须明确低优先级任务的等待上限和监控方式。

队列容量的推导要围绕等待时间目标，而不是围绕“内存还能放多少任务”。如果任务平均耗时是 50 毫秒，核心线程数是 20，理想情况下每秒可以处理约 400 个任务。若上游短时每秒提交 800 个任务，队列每秒增长约 400 个。如果队列容量为 4000，十秒后就可能排满；而排在末尾的任务等待时间也可能接近十秒。队列容量越大，拒绝越晚发生，但尾部任务的延迟越高。对于有明确时效性的任务，过大的队列并不是缓冲，而是制造过期工作。

一个实用原则是：队列长度应该和任务的最大可接受等待时间相匹配。若任务超过两秒就没有意义，那么队列容量就不应允许在高峰下排出几十秒的等待。相比盲目扩大队列，更好的方式是限制队列、记录拒绝、让调用方降级或重试，并且把排队时间作为核心指标。线程池不是无限蓄水池，队列只是用来吸收合理抖动，不是用来掩盖持续过载。

## threadFactory：线程创建不是细节

`threadFactory` 决定线程池如何创建工作线程。很多示例直接使用默认线程工厂，这在演示代码中可以接受，但在实际工程中通常不够。线程工厂至少应该负责设置有意义的线程名。线程 dump、日志、监控和异常栈中如果只能看到 `pool-1-thread-3`，排查者很难知道这个线程池属于哪个模块、处理哪类任务、是否和当前故障有关。清晰的线程名是并发程序最便宜也最有效的诊断信息之一。

一个简单的命名线程工厂可以这样写：

```java
final class NamedThreadFactory implements ThreadFactory {
    private final AtomicInteger index = new AtomicInteger(1);
    private final String prefix;

    NamedThreadFactory(String prefix) {
        this.prefix = Objects.requireNonNull(prefix);
    }

    @Override
    public Thread newThread(Runnable r) {
        Thread thread = new Thread(r, prefix + "-" + index.getAndIncrement());
        thread.setDaemon(false);
        thread.setUncaughtExceptionHandler((t, e) ->
                System.err.println("Uncaught exception in " + t.getName() + ": " + e));
        return thread;
    }
}
```

线程工厂还可以设置是否为守护线程。守护线程不会阻止 JVM 退出，普通用户线程会阻止 JVM 退出。把线程池工作线程设置为 daemon 可能导致进程退出时任务还没完成就被终止；把本应随主流程结束的后台线程设置为非 daemon，则可能导致程序无法自然退出。是否使用守护线程必须由生命周期设计决定，不能为了“程序能退出”随手设置。

`UncaughtExceptionHandler` 需要特别说明。通过 `execute` 提交的 `Runnable` 如果抛出未捕获运行时异常，异常会从工作线程中冒出，可能被线程的未捕获异常处理器观察到，线程随后终止，线程池再按需补充工作线程。通过 `submit` 提交的任务则会被包装成 `FutureTask`，异常通常记录在 `Future` 中，只有调用 `get` 时才重新抛出。因此，线程工厂中的未捕获异常处理器不是后台任务异常处理的完整方案。任务本身仍应在合适层次捕获、记录和传播异常，调用方也应读取 `Future` 结果。

## handler：拒绝不是异常情况的附属品

`RejectedExecutionHandler` 处理两类情况：线程池已经关闭后仍有新任务提交，或者线程池仍在运行但工作线程和队列容量都已用尽。拒绝策略不是可有可无的末端细节，而是线程池背压模型的重要组成部分。只要线程池使用有界队列和有限最大线程数，就必须提前设计任务被拒绝时调用方应该看到什么。

标准库提供四种常见策略。`AbortPolicy` 是默认策略，会抛出 `RejectedExecutionException`。它的优点是失败明确，调用方能立即知道任务没有被接收；缺点是如果调用方没有处理异常，异常会沿提交路径传播。对于大多数不能静默丢任务的场景，`AbortPolicy` 是最容易保持正确性的选择。

`CallerRunsPolicy` 会让提交任务的线程自己执行该任务，前提是线程池未关闭。它的效果是降低提交速度，因为调用方线程被迫参与执行，从而形成一种同步背压。这个策略适合任务可以在调用方线程执行、执行时间可接受、不会破坏调用方线程语义的场景。但它也有明显风险：如果提交线程是关键调度线程、事件循环线程或持有重要锁的线程，直接执行任务可能造成延迟扩大、锁持有时间变长，甚至死锁。使用它之前必须确认任务在调用方线程执行是安全的。

`DiscardPolicy` 会静默丢弃新任务。它只适合任务天然可丢、且丢弃本身有外部可观测手段的场景。静默丢弃不能用于必须执行的业务任务，因为调用方会误以为提交成功。即使任务可丢，也建议自定义策略记录丢弃计数或日志，否则过载发生时系统缺少诊断信号。

`DiscardOldestPolicy` 会丢弃队列头部任务，然后重新尝试提交当前任务。它看起来适合“保留最新任务”的场景，但要注意队列头部通常是等待最久、最接近执行的任务；丢弃它可能破坏公平性和顺序语义。如果队列是优先级队列，所谓“最旧”还可能不是时间上最旧，而是队列排序下最先取出的任务。这个策略需要非常明确的业务语义支撑，不应作为通用过载处理方式。

自定义拒绝策略通常至少要做三件事：记录线程池名称、当前线程数、队列长度和任务类型；选择明确的失败方式，例如抛异常、返回失败结果、计数后丢弃或阻塞等待；避免在拒绝策略中执行复杂、耗时或可能再次提交同一线程池的逻辑。拒绝策略处在线程池已经饱和的路径上，此时做重操作很容易雪上加霜。特别要避免无边界重试，因为重试会继续向同一个饱和资源施压。

有些代码会在拒绝策略中调用 `workQueue.put(task)`，试图把拒绝变成阻塞提交。这种做法可以形成强背压，但风险很高：如果提交线程持有锁、占用连接、位于线程池工作线程内，阻塞可能引发死锁或线程池自我饥饿；如果线程池正在关闭，阻塞提交还可能破坏关闭语义。若确实需要阻塞式提交，应该把它设计成明确 API，并仔细处理超时、中断和关闭状态，而不是藏在拒绝策略中。

## 六个参数如何协同

线程池的协同关系可以从一个提交任务的时间线观察。任务到达时，`corePoolSize` 决定是否立即增加基础工作线程；核心线程满后，`workQueue` 决定任务能否等待；队列无法接收时，`maximumPoolSize` 决定是否临时扩张；扩张出的空闲线程由 `keepAliveTime` 决定何时退出；每个线程由 `threadFactory` 创建并携带诊断属性；所有容量都用尽或线程池关闭时，`handler` 决定任务失败形态。

这意味着任何一个参数都不应孤立配置。`corePoolSize = maximumPoolSize` 表示固定大小线程池，此时队列是主要缓冲边界，`keepAliveTime` 默认对普通核心线程没有明显影响。`corePoolSize` 较小、`maximumPoolSize` 较大、队列较小时，线程池更偏向在峰值时扩张线程。`corePoolSize` 较大、队列中等、最大线程数只略大于核心数时，线程池更偏向稳定执行和有限缓冲。使用 `SynchronousQueue` 时，线程池几乎没有排队缓冲，拒绝策略和最大线程数必须非常谨慎。

可以用几种典型组合理解参数协同：

| 组合 | 行为特征 | 适用倾向 | 主要风险 |
| --- | --- | --- | --- |
| 固定线程数 + 有界队列 | 并发数稳定，排队有上限 | 平稳服务、批处理消费者、资源受限任务 | 队列容量不合理会导致延迟过高或拒绝过早 |
| 小核心 + 大最大 + 小队列 | 波峰时快速扩张线程 | 短时峰值明显、任务耗时较短 | 线程暴增、下游资源争抢 |
| 核心线程 + 无界队列 | 提交不易失败，线程数接近核心值 | 对延迟不敏感且任务量可控的后台处理 | 过载隐藏、内存风险、最大线程数弱化 |
| 大最大 + SynchronousQueue | 直接移交任务，不排队 | 任务不应等待、允许快速失败或扩张 | 最大值过大时创建过多线程，过小时拒绝频繁 |
| 允许核心线程超时 + 有界队列 | 低谷释放资源，高峰重建线程 | 任务间歇明显、资源敏感 | 冷启动延迟、频繁创建销毁 |

协同关系还体现在任务内部行为上。如果线程池任务会等待同一个线程池中的其他任务完成，就可能出现线程池饥饿死锁。例如一个固定大小线程池有两个工作线程，两个外层任务各自提交一个内层任务到同一个线程池并同步等待结果；此时两个工作线程都被外层任务占住，内层任务排在队列里无法执行，外层任务又一直等待内层任务完成。这个问题不是简单把 `maximumPoolSize` 调大就能根治，而是需要重新设计任务依赖关系，避免工作线程阻塞等待同池任务。

```java
ExecutorService pool = Executors.newFixedThreadPool(2);

pool.submit(() -> {
    Future<Integer> inner = pool.submit(() -> 1);
    return inner.get();
});

pool.submit(() -> {
    Future<Integer> inner = pool.submit(() -> 2);
    return inner.get();
});
```

上面的代码片段表达的是一种风险模式：线程池内任务同步等待同一个线程池的后续任务。真实代码中这种关系可能隐藏在库调用、回调、批量拆分或异步封装里。配置参数时必须识别任务之间是否独立。如果任务存在依赖链，就不能只按“单个任务耗时”估算线程数量，还要考虑依赖任务是否有独立执行资源。

## 从需求推导配置

推导线程池参数时，第一步是定义任务边界。任务是 CPU 计算、阻塞 I/O、远程调用、文件处理、消息消费、定时清理，还是混合流程？任务是否有返回值，是否允许取消，是否允许重复执行，是否可以丢弃，是否有顺序要求？任务之间是否共享锁、连接、缓存或外部限流？如果这些问题没有答案，线程池参数只能靠猜。

第二步是估算流量和耗时。至少需要知道平均提交速率、峰值提交速率、平均执行时间、长尾执行时间，以及任务允许的最大排队时间。线程池容量要服务于延迟目标，而不仅是吞吐目标。一个系统每秒能处理一千个任务，并不代表每个任务都能在可接受时间内完成；如果其中大量任务排队等待，调用方看到的仍然是超时。

第三步是确定并发执行上限。这个上限要同时考虑 CPU、内存、连接池、文件句柄、远程服务限流、锁竞争和数据结构争用。对于 CPU 密集型任务，通常从处理器数量附近开始；对于等待型任务，可以按等待比例放大，但不能超过外部资源允许的并发。若任务需要数据库连接，线程数通常不应远大于连接池可用连接数；若任务会访问限流接口，线程数和重试策略必须尊重限流窗口。

第四步是确定队列容量。队列容量可以用“可接受等待时间 × 可持续处理速率”粗略推导。例如一个线程池稳定每秒处理 500 个任务，任务最多允许等待 2 秒，那么队列容量就不应轻易超过 1000；如果容量设置成 10000，就意味着极端情况下队尾任务可能等待远超目标。这个推导并不精确，但能防止把队列设置成没有语义的大数字。

第五步是选择拒绝策略。任务不能丢时，拒绝要显式抛出或返回失败；任务可降级时，拒绝策略要把信号传给调用方；任务可丢时，也要记录丢弃指标；任务适合由调用方执行时，才考虑 `CallerRunsPolicy`。拒绝策略和上游协议要匹配。线程池拒绝了任务，调用方却继续认为任务已提交成功，这是比抛异常更危险的错误。

第六步是设计观测指标。至少要能看到当前线程数、活跃线程数、核心线程数、最大线程数、队列长度、已完成任务数、提交失败或拒绝次数、任务执行耗时和排队耗时。没有这些指标，参数调优只能依赖事故后的猜测。`ThreadPoolExecutor` 提供了若干查询方法，但它们多是瞬时值，适合做监控参考，不应作为复杂业务同步条件。

一个比较稳健的推导过程可以总结为：先用任务性质估算核心线程数，再用资源上限约束最大线程数，然后用延迟目标约束队列容量，最后用拒绝策略定义过载行为。顺序不能反过来。如果先拍一个大队列，再把线程数调到“看起来够用”，通常会把延迟问题隐藏到队列里。

## 不同任务类型下的参数倾向

CPU 密集型任务强调处理器利用率和调度开销的平衡。核心线程数和最大线程数通常不宜远高于可用处理器数量。队列可以有界，用来吸收短时提交抖动；拒绝策略应明确告诉调用方当前执行能力不足。若任务可拆分，拆分粒度也要受控，过细的任务会让调度和队列开销超过计算收益。对这类任务，盲目增大线程数常常降低吞吐。

阻塞 I/O 或远程调用型任务的线程数通常可以高于处理器数量，因为线程大部分时间不在 CPU 上运行。但并发上限应由外部资源决定，例如连接池、限流规则、文件系统能力和远程服务吞吐。队列容量要结合调用超时。如果远程调用三秒超时，而任务在队列里已经等了十秒才开始执行，那么执行本身可能已经失去意义。对这类任务，排队时间和外部依赖耗时同样重要。

短任务高频提交的线程池要关注队列和上下文切换成本。任务本身很短时，线程池调度、对象分配、队列入队出队、锁竞争和指标记录可能成为主要开销。此时核心线程数不一定需要很大，队列实现和批处理策略反而更重要。若每个任务只做极少量操作，可以考虑是否需要合并任务、批量消费或使用更贴近需求的并发结构。

长任务线程池要关注隔离。长时间运行的任务会占用工作线程，导致短任务排队。如果同一个线程池混合执行长任务和短任务，短任务的延迟会被长任务拖垮。更合理的做法通常是按任务类型拆分线程池，让长任务池有自己的并发上限和队列，让短任务池保持较小排队延迟。线程池不是越共享越好，隔离可以让容量模型更可解释。

有顺序要求的任务不能只看线程数。线程池天然会让多个任务并发执行，执行完成顺序不一定等于提交顺序。即使使用单线程线程池，也要注意任务异常、拒绝和关闭对顺序的影响。如果任务需要按某个 key 保序，可以考虑分片队列、按 key 串行化或其他调度模型，而不是用一个全局大线程池解决所有问题。

可丢弃任务和不可丢弃任务也应分开。日志刷新、状态采样、缓存预热、后台统计等任务可能在某些情况下允许丢弃；订单处理、状态变更、持久化写入等任务通常不能丢。把两类任务放进同一个线程池，会导致拒绝策略无法表达清楚语义。线程池参数必须服务于任务语义，而不是让任务语义迁就某个公共线程池。

## Executors 工厂方法与隐藏参数

`Executors` 提供了若干便捷工厂方法，例如 `newFixedThreadPool`、`newCachedThreadPool`、`newSingleThreadExecutor` 和 `newScheduledThreadPool`。这些方法便于学习和快速编写示例，但它们隐藏了关键参数，实际使用时必须知道背后的配置。

`newFixedThreadPool(n)` 本质上创建核心线程数和最大线程数都为 `n` 的线程池，并使用无界 `LinkedBlockingQueue`。这意味着线程数固定，任务过多时主要进入无界队列，`maximumPoolSize` 不提供额外扩张空间。它适合任务量受控的固定并发执行，但如果上游可能持续超过处理能力，无界队列会带来内存和延迟风险。

`newSingleThreadExecutor()` 类似一个只有单个工作线程的执行器，并使用无界队列。它可以表达串行执行，但同样存在任务无限排队的风险。它还通过包装限制了重新配置能力，语义上比直接创建单线程 `ThreadPoolExecutor` 更强调“单线程执行器”。如果串行任务可能堆积，仍然需要考虑有界队列和拒绝策略。

`newCachedThreadPool()` 使用 `SynchronousQueue`，核心线程数为 0，最大线程数非常大，空闲线程默认保留 60 秒。它的特点是任务来了就尽量创建或复用线程，不做队列缓冲。对于大量短生命周期异步任务，它看起来很灵活；但在提交速度失控或任务阻塞时，它可能创建过多线程。使用这种模型时，必须确认任务来源有上限，或者外层已有可靠限流。

便捷工厂的问题不是“不能用”，而是默认参数经常和真实容量需求不匹配。更稳妥的做法是，当线程池承担重要工作时，直接使用 `ThreadPoolExecutor` 构造方法显式写出队列容量、线程工厂和拒绝策略。显式参数能迫使设计者回答容量边界，也能让代码评审和运行诊断更容易。

## 动态调整与生命周期边界

`ThreadPoolExecutor` 支持运行期调整核心线程数、最大线程数、保活时间和核心线程超时策略。这给调优留下了空间，但动态调整不能替代正确设计。调整线程数只影响后续任务调度和空闲线程回收，不会让正在执行的任务立刻中断，也不会自动清理已经排队的任务。队列容量通常也不是一个可以随意动态改变的属性，尤其是标准有界队列创建后容量固定。

运行期调大核心线程数可能触发新的工作线程创建，用于处理队列中的任务；调小核心线程数则会让多余空闲线程在之后退出。调大最大线程数会允许后续在队列满时创建更多线程；调小最大线程数不会中断已经运行的线程，而是限制未来规模并让空闲多余线程逐步退出。调整 `keepAliveTime` 会改变空闲线程等待新任务的时间边界。

关闭线程池也是核心参数的一部分，因为关闭状态会影响提交和拒绝。调用 `shutdown()` 后，线程池不再接收新任务，但会继续执行已经提交和排队的任务；调用 `shutdownNow()` 会尝试中断正在运行的任务，并返回尚未开始执行的任务列表。中断只是协作式信号，如果任务不响应中断，线程池无法强制安全终止它。线程池使用方必须明确谁负责关闭、何时关闭、关闭后是否等待完成、等待超时时如何处理。

一个基本的关闭模式如下：

```java
executor.shutdown();
try {
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        List<Runnable> dropped = executor.shutdownNow();
        System.err.println("Cancelled queued tasks: " + dropped.size());
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

这里的重点是保留中断语义。捕获 `InterruptedException` 后，如果当前层不能完全处理取消，应恢复中断标志，让上层知道当前线程曾被中断。线程池任务内部也应遵守类似原则：阻塞方法抛出中断异常时，不要无意义吞掉；循环任务应定期检查中断；资源清理应放在 `finally` 中。

生命周期还包括线程池所有权。线程池不应由随意的业务方法临时创建又不关闭，也不应把一个共享线程池交给任意调用方关闭。创建者、提交者和关闭者必须有清楚边界。很多线程池泄漏不是因为参数错，而是因为生命周期没有所有者：局部方法创建线程池后忘记关闭，或者某个模块关闭了其他模块仍在使用的共享线程池，最终导致拒绝异常或进程无法退出。

## 任务异常、Future 与参数的关系

线程池参数控制任务如何被接收和执行，但任务执行过程中的异常传播方式也会影响线程池表现。使用 `execute` 提交的任务抛出运行时异常时，异常会终止当前工作线程，线程池会根据需要创建新线程维持容量。使用 `submit` 提交的任务抛出异常时，异常会被封装进 `Future`，工作线程通常不会因为异常直接向外抛出而终止。调用方如果从不调用 `Future.get()`，异常就可能长期沉默。

这和核心参数的关系在于：线程池容量指标可能看起来正常，但任务已经持续失败。工作线程仍然存在，队列仍然消费，已完成任务数仍然增长，可实际业务结果可能全部失败。如果没有任务级异常记录和结果检查，线程池参数再合理也无法保证执行语义正确。线程池负责调度任务，不负责理解任务成功与否。

对于需要结果的任务，应使用 `Future`、`CompletableFuture` 或其他结果传递机制，并明确超时、取消和异常处理。对于不需要结果的后台任务，也至少应该在任务边界捕获异常并记录上下文。不要把 `threadFactory` 的未捕获异常处理器当作所有异常的兜底，因为不同提交方式对异常的包装不同。

```java
Future<String> future = executor.submit(() -> {
    // 任务中的异常会记录到 Future 中
    return query();
});

try {
    String result = future.get(500, TimeUnit.MILLISECONDS);
} catch (TimeoutException e) {
    future.cancel(true);
    throw e;
}
```

任务取消也不是强制停止。`future.cancel(true)` 会在任务已经开始时尝试中断执行线程，但任务是否停止取决于它是否响应中断。若任务阻塞在不响应中断的调用上，取消可能无法及时生效。线程池参数不能弥补任务取消协议的缺失；如果任务可能长时间阻塞，应选择支持超时和中断的 API，或在任务内部设置明确的退出条件。

## 参数配置中的常见误区

第一个误区是只调线程数，不调队列。很多人遇到任务积压时先把 `maximumPoolSize` 调大，却没有发现使用的是无界队列。由于任务总能入队，最大线程数几乎不触发，调大它没有实际效果。要让最大线程数参与峰值扩张，必须同时考虑队列类型和容量。

第二个误区是把队列设得很大，认为这样可以避免拒绝。拒绝减少了，但任务等待时间会增加，内存占用会增加，调用方可能在任务真正开始前就已经超时。过大的队列把过载从“显式失败”变成“慢性延迟”，对有时效要求的任务尤其危险。

第三个误区是把线程数设得很大，认为这样可以提高吞吐。线程数超过瓶颈资源后，吞吐不会线性增长，反而可能因为上下文切换、锁竞争、缓存失效和外部资源排队而下降。线程池不是把等待变成处理能力的魔法，它只能在资源允许的范围内组织并发。

第四个误区是使用默认线程工厂和匿名线程池。故障发生时，如果线程 dump 中全是类似 `pool-7-thread-12` 的名字，排查者需要额外推断线程池来源。线程池应有稳定、可识别的名称，拒绝日志和异常日志也应带上这个名称。

第五个误区是静默丢弃任务。`DiscardPolicy` 和某些自定义拒绝策略会让调用方误以为提交成功，实际任务已经丢失。除非任务明确可丢，并且系统有丢弃指标，否则不要静默丢弃。显式失败虽然看起来刺眼，但它能保护数据语义。

第六个误区是在同一个线程池中混放不同性质任务。长任务和短任务、可丢任务和不可丢任务、CPU 任务和阻塞任务、低优先级任务和高优先级任务混在一起，会让任何一组参数都无法同时满足所有需求。合理拆分线程池可以让容量边界更清晰。

第七个误区是在线程池任务中无限制地再次提交任务并等待。嵌套提交、同步等待和固定线程数结合时，很容易造成线程池饥饿。若任务存在父子关系，应考虑拆分线程池、使用异步组合、减少同步等待，或选择更适合任务拆分的并发框架。

第八个误区是把线程池当作资源隔离的全部。线程池只能限制执行线程数量，不能自动限制数据库连接、网络请求、文件句柄、内存分配或业务锁竞争。如果任务内部还会访问其他有限资源，就需要配合连接池、信号量、限流器、超时和熔断等机制。线程池边界只是系统容量边界的一部分。

第九个误区是忽略关闭。线程池创建后不关闭会导致线程泄漏和进程无法退出；过早关闭会导致后续提交被拒绝；关闭时不等待会导致任务未完成；关闭时吞掉中断会破坏上层取消流程。生命周期和参数一样重要。

第十个误区是把监控查询方法当作同步依据。`getActiveCount()`、`getQueue().size()`、`getPoolSize()` 等方法提供的是近似瞬时状态，适合监控和诊断，不适合编写依赖精确值的并发控制逻辑。线程池内部状态持续变化，外部读到的数值在下一瞬间就可能过期。

## 一个参数推导示例

假设有一类后台处理任务，单个任务平均执行 80 毫秒，其中约 20 毫秒为 CPU 计算，60 毫秒等待外部资源；任务峰值提交速率为每秒 300 个，平稳速率为每秒 120 个；外部资源最多允许 40 个并发请求；任务排队超过 2 秒就应该失败或降级。如何推导线程池参数？

首先看线程数。任务等待时间是计算时间的三倍，单纯按 CPU 估算可以配置高于处理器数量的线程。但外部资源并发上限是 40，因此线程池并发执行数不应明显超过 40，否则多出来的线程只是等待外部资源。可以把 `maximumPoolSize` 设置在 40 附近，或者略低于 40，为其他调用路径保留资源。核心线程数则可以根据平稳速率估算：若一个任务平均 80 毫秒，一个线程每秒理论处理约 12.5 个任务，120 个任务每秒大约需要 10 个线程；考虑长尾和调度损耗，可以从 16 或 20 个核心线程开始测试。

然后看队列。最大可接受排队时间是 2 秒，如果平稳处理能力按 20 个核心线程估算，每秒可处理约 250 个任务；若高峰每秒 300 个，短时间内队列增长并不夸张。队列容量可以从几百开始，而不是上万。若容量设为 500，在核心线程处理能力下已经对应约两秒级等待；如果扩张到最大线程数后处理能力更强，队列消化速度会提高。这个容量能吸收短时抖动，也能在持续过载时及时拒绝。

再看拒绝策略。如果任务排队超过 2 秒就没有意义，那么拒绝应该显式返回失败或触发降级，而不是静默丢弃，也不应该无限阻塞提交线程。可以使用 `AbortPolicy` 并在提交方捕获 `RejectedExecutionException`，转换成明确的失败结果；或者自定义策略记录指标后抛出业务异常。拒绝是设计的一部分，不是意外。

最后看线程工厂和保活时间。线程名应体现任务类别，例如 `report-worker-1`。若峰值明显而低谷较长，可以把 `keepAliveTime` 设置为几十秒，让非核心线程在高峰后释放；如果高峰频繁出现，可以适当延长，减少线程反复创建。核心线程是否允许超时要看低谷期资源要求和下一波任务延迟要求。

可能的初始配置如下：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        20,
        40,
        60L,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(500),
        new NamedThreadFactory("report-worker"),
        new ThreadPoolExecutor.AbortPolicy()
);
```

这个配置不是最终答案，而是一个有依据的起点。后续需要观察队列长度分布、任务排队时间、任务执行时间、拒绝次数、外部资源耗时和 CPU 利用率。如果队列经常接近 500 但外部资源未满，可以增加核心线程或最大线程；如果外部资源经常打满，应限制线程数或在任务内部增加限流；如果拒绝频繁且任务仍有价值，应检查上游流量或拆分处理链路；如果线程长期空闲，可以降低核心线程数或允许核心线程超时。

## 观测与排查

线程池参数是否合理，最终要靠运行数据验证。最基本的观测项包括 `getPoolSize()`、`getCorePoolSize()`、`getMaximumPoolSize()`、`getActiveCount()`、`getQueue().size()`、`getCompletedTaskCount()`、`getTaskCount()` 和拒绝次数。还应尽量记录任务从提交到开始执行的排队时间，以及从开始到结束的执行时间。只有把排队时间和执行时间分开，才能判断问题是线程池容量不足，还是任务本身变慢。

排查任务堆积时，可以按顺序问几个问题。当前活跃线程数是否接近线程池大小？队列是否持续增长？最大线程数是否因为队列类型没有触发？任务执行时间是否变长？工作线程是否阻塞在同一个锁、连接池或外部调用上？拒绝次数是否上升？线程池是否已经进入关闭状态？这些问题能把“线程池慢”拆成具体原因。

如果活跃线程数很低但队列很长，可能是线程池状态异常、工作线程创建失败、核心线程数过小且未补充、或者任务获取受队列语义影响。如果活跃线程数打满且队列增长，说明执行能力不足或任务阻塞。如果线程数已经扩到最大但 CPU 不高，通常要检查外部等待、锁竞争或连接池。如果 CPU 打满且上下文切换高，可能线程数过多或任务过碎。

线程 dump 是排查线程池问题的重要工具。通过线程名可以快速找到目标线程池，看工作线程处于 RUNNABLE、BLOCKED、WAITING 还是 TIMED_WAITING。若大量线程阻塞在同一把锁上，增加线程数通常没有意义；若大量线程等待连接，应该检查连接池和下游；若大量线程等待同池任务的 `Future.get()`，要警惕线程池饥饿；若线程都在正常运行但队列仍增长，可能确实需要提高处理能力或削减上游流量。

拒绝日志也要设计得可用。一个好的拒绝记录至少包含线程池名称、当前池大小、活跃线程数、队列大小、队列剩余容量、任务类别和提交来源。只记录一行“任务被拒绝”价值很低。过载时日志量也要受控，避免拒绝日志本身造成新的 I/O 压力，可以使用计数、采样或周期性汇总。

## 与同步、可见性和线程安全的边界

线程池保证任务会在线程池工作线程中执行，但不自动保证任务内部共享状态安全。多个任务并发访问同一个对象时，仍然需要使用锁、volatile、原子类、并发容器或不可变对象建立线程安全边界。`ThreadPoolExecutor` 内部有自己的同步机制来管理工作线程和队列，但这些机制不会延伸到任务对象的业务字段上。

任务提交和执行之间存在标准并发库提供的内存一致性效果。一般而言，提交任务之前线程中的操作，对任务开始执行时是可见的；任务执行完成后，通过 `Future.get()` 获取结果也会建立相应的可见性关系。但这不意味着任务内部对共享对象的任意写入会被其他线程无同步地安全观察。若多个任务写同一个普通对象字段，仍然可能发生数据竞争。

因此，线程池参数和线程安全设计要分开考虑。参数回答“有多少任务并发执行、多少任务排队、满了怎么办”；线程安全回答“这些并发执行的任务访问共享状态时是否正确”。把 `corePoolSize` 设置为 1 可以让某个线程池内的任务串行执行，但如果同一个共享对象还被其他线程池或外部线程访问，仍然不安全。把线程池设大也不会让非线程安全集合变安全。

ThreadLocal 也需要谨慎。线程池复用工作线程，意味着同一个线程会先后执行多个任务。如果任务向 ThreadLocal 写入数据但不清理，后续任务可能读到上一个任务留下的值，或者相关对象无法及时释放。线程池使用 ThreadLocal 时，应在任务边界使用 `try/finally` 清理。线程复用是线程池的优势，但也会放大线程局部状态泄漏。

## 小结

`ThreadPoolExecutor` 的核心参数共同描述了一套容量模型：`corePoolSize` 定义平稳执行能力，`maximumPoolSize` 定义峰值扩张上限，`keepAliveTime` 定义空闲线程回收边界，`workQueue` 定义等待和背压方式，`threadFactory` 定义线程创建和诊断属性，`handler` 定义容量耗尽或关闭后的失败语义。它们必须一起设计，不能孤立调参。

可靠的线程池配置通常从任务语义出发：任务是否 CPU 密集，是否阻塞，是否可丢，是否有顺序要求，是否访问有限外部资源，允许排队多久，过载时调用方应该看到什么。然后再把这些答案落到核心线程数、最大线程数、队列容量、保活时间、线程工厂和拒绝策略上。配置之后还要通过指标和压测验证，因为真实瓶颈经常出现在锁、连接池、外部服务、长尾耗时和任务依赖关系中。

最需要避免的是无解释的大数字和隐藏边界的默认值。无界队列会隐藏过载，过大线程数会放大争用，静默拒绝会丢失语义，匿名线程会降低可诊断性，忽略关闭会造成生命周期问题。线程池不是把并发风险消除的容器，而是把执行、排队、扩张、回收和拒绝这些决策集中到一组参数中。只有这些参数背后的取舍清楚，线程池才能在压力、异常和关闭阶段保持可解释。

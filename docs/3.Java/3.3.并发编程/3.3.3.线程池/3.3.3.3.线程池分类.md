# 3.3.3.3 线程池分类

## 为什么要讨论线程池分类

线程池分类讨论的不是“背几个工厂方法名称”，而是理解不同执行器模型背后的资源策略。Java 标准库提供了 `ThreadPoolExecutor`、`ScheduledThreadPoolExecutor`、`ForkJoinPool` 等基础实现，又通过 `Executors` 暴露了 Fixed、Cached、Single、Scheduled、WorkStealing 等常见工厂方法。它们看起来只是创建方式不同，本质上却代表了不同的线程数量策略、队列策略、任务调度策略和失败边界。

如果只从方法名理解线程池，很容易得到过度简化的结论：固定线程池就是固定数量，缓存线程池就是能复用线程，单线程池就是只有一个线程，调度线程池就是定时任务，工作窃取线程池就是并行执行。这样的描述没有错，但不足以指导工程选择。真正决定线程池行为的，是核心线程数、最大线程数、空闲线程存活时间、工作队列类型、线程工厂、拒绝策略、任务是否可拆分、任务是否会阻塞、任务是否需要顺序，以及任务失败后是否影响后续执行。

线程池的分类可以从两个层面理解。第一层是使用者视角的模型分类：固定并发、弹性扩张、单线程串行、延迟调度、周期调度、工作窃取、每任务一线程等。这个层面回答“我想让任务以什么方式执行”。第二层是实现者视角的参数分类：线程数量如何增长，队列是否有界，任务如何入队，空闲线程是否回收，饱和时如何处理，关闭时如何处理未执行任务。这个层面回答“这种模型在压力下会怎样失效”。学习线程池分类时，必须把这两个层面合在一起看。

Java 线程池的核心价值是把任务提交和线程管理解耦。调用方提交的是 `Runnable` 或 `Callable`，线程池负责决定何时创建线程、何时复用线程、任务排队还是拒绝、线程异常后是否补充、关闭时是否继续执行已提交任务。这个解耦让并发程序不必到处散落 `new Thread(...)`，但也带来一个重要后果：线程池本身成为容量边界。只要容量设计错误，任务就可能无限堆积、线程无限增长、延迟持续放大，或者在关闭阶段留下无法结束的后台线程。

本文保持通用 Java 技术视角，围绕线程池分类本身展开。重点不在于枚举 API，而是讲清楚各类线程池的底层参数差异、适用边界、常见误区，以及生产代码中如何从任务特征出发定制线程池。

## 线程池分类的共同基础

理解所有分类之前，要先回到 `ThreadPoolExecutor` 的执行规则。一个标准线程池提交任务时，大致遵循这样的顺序：如果当前工作线程数小于核心线程数，优先创建核心线程执行任务；否则尝试把任务放入工作队列；如果队列放不下，并且当前线程数小于最大线程数，再创建非核心线程执行任务；如果线程数已经达到最大值且队列也无法接收任务，就触发拒绝策略。这个顺序意味着线程池不是简单地“先把线程开满再排队”，也不是简单地“先排队再创建线程”，而是由核心线程数、队列类型和最大线程数共同决定。

最容易被忽略的是队列类型。`LinkedBlockingQueue` 如果不指定容量，默认接近无界；`ArrayBlockingQueue` 是固定容量数组队列；`SynchronousQueue` 不存储任务，提交方必须把任务直接交给工作线程；`DelayedWorkQueue` 按延迟时间排序，用于调度线程池；`ForkJoinPool` 内部则不是普通阻塞队列，而是每个工作线程维护双端队列，并通过工作窃取减少空闲。队列一变，线程池分类的实际表现就会完全不同。

线程数量参数也不能孤立理解。`corePoolSize` 表示线程池愿意长期保留的基础并发能力；`maximumPoolSize` 表示队列无法承接压力时允许扩张到的上限；`keepAliveTime` 表示超过核心线程数的空闲线程多久被回收。默认情况下，核心线程不会因为空闲而超时退出，除非调用 `allowCoreThreadTimeOut(true)`。因此，固定线程池的核心线程通常会长期存在，缓存线程池则依赖空闲超时回收多余线程。

拒绝策略是线程池分类中最能体现设计态度的一环。`AbortPolicy` 会抛出 `RejectedExecutionException`，让调用方显式感知提交失败；`CallerRunsPolicy` 会让提交任务的线程自己执行任务，从而形成一种反压；`DiscardPolicy` 静默丢弃新任务；`DiscardOldestPolicy` 丢弃队列头部任务再尝试提交新任务。拒绝策略不是异常处理细节，而是过载策略。任何没有明确拒绝策略的线程池，都只是在把过载问题留给默认行为。

线程工厂也属于分类边界的一部分。线程名、是否守护线程、优先级、未捕获异常处理器、上下文类加载器等都由 `ThreadFactory` 控制。标准 `Executors.defaultThreadFactory()` 创建的是非守护线程，优先级通常是普通优先级，线程名使用 `pool-N-thread-M` 形式。对长期运行的应用来说，可诊断的线程名非常重要，因为线程 dump、日志和监控中看到的都是线程名，而不是当初调用的工厂方法名。

下面这段代码展示了一个显式参数的线程池创建方式。它不是固定模板，而是为了说明分类背后真正需要关注的变量。

```java
ThreadFactory threadFactory = new ThreadFactory() {
    private final AtomicInteger index = new AtomicInteger(1);

    @Override
    public Thread newThread(Runnable task) {
        Thread thread = new Thread(task, "order-worker-" + index.getAndIncrement());
        thread.setUncaughtExceptionHandler((t, e) ->
                System.err.println("Uncaught in " + t.getName() + ": " + e));
        return thread;
    }
};

BlockingQueue<Runnable> queue = new ArrayBlockingQueue<>(1_000);

ThreadPoolExecutor executor = new ThreadPoolExecutor(
        8,
        16,
        30,
        TimeUnit.SECONDS,
        queue,
        threadFactory,
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

这段代码把线程数量、队列容量、线程命名和拒绝行为都写在明处。相比直接调用 `Executors.newFixedThreadPool(8)`，它更啰嗦，但容量边界更清楚。线程池分类的实用价值，也正是在这种“名称背后还有参数”的意识中体现出来。

## FixedThreadPool：固定并发模型

`FixedThreadPool` 通常通过 `Executors.newFixedThreadPool(nThreads)` 创建。它对应的底层模板是：核心线程数等于最大线程数，二者都为 `nThreads`；空闲线程存活时间为 0，因为没有超过核心线程数的非核心线程需要回收；工作队列使用无界 `LinkedBlockingQueue`；默认拒绝策略是 `AbortPolicy`，但只有在线程池关闭后或极端情况下才更容易触发，因为队列通常不会因为容量满而拒绝。

固定线程池表达的是“并发度固定，超出并发度的任务排队”。当任务数量超过线程数时，新任务不会推动线程池继续创建更多线程，而是进入队列等待。这个模型适合任务处理速度相对稳定、并发上限明确、希望保护下游资源不被过多线程同时访问的场景。例如一个批处理程序需要并发处理文件，但磁盘吞吐或外部服务只允许有限并发，那么固定线程池可以把同时运行的任务数限定在一个可解释范围内。

固定线程池的优点是行为直观。线程数不会随着瞬时流量暴涨，资源占用相对稳定，排查时也容易从线程数推断当前最大并发。它尤其适合 CPU 密集型任务或有明确外部并发限制的任务。CPU 密集型任务通常不应把线程数设置得远大于可用处理器数量，否则上下文切换会吞掉并行收益；外部资源受限任务则应按资源容量设置线程数，而不是按机器 CPU 数设置。

固定线程池的主要风险在于无界队列。因为 `newFixedThreadPool` 默认使用无界 `LinkedBlockingQueue`，当任务提交速度长期大于处理速度时，线程数仍然保持固定，任务却会持续堆积在内存中。调用方短时间内可能看不到拒绝异常，因为任务都“提交成功”了；真正的失败会以响应时间越来越长、内存占用持续上涨、最终内存溢出或长时间停顿的形式出现。这种风险在低压测试中很难暴露，因为低压下队列不会显著增长。

固定线程池还有一个容易误解的点：固定的是工作线程数量，不是任务完成时间，也不是业务吞吐。一个线程可以被慢任务长期占住；所有线程都被阻塞时，队列中的短任务也无法执行。如果任务会等待数据库、网络、文件锁或其他线程池的结果，那么固定线程池的有效吞吐取决于等待比例。此时只看线程数是不够的，还要看任务耗时分布、阻塞位置和队列长度。

使用固定线程池时，建议明确回答三个问题。第一，为什么并发度是这个数字，是 CPU 计算能力、外部连接数、资源限额，还是经验值。第二，队列是否允许无界增长，如果不允许，就不要直接使用 `Executors.newFixedThreadPool`，而应使用有界队列创建 `ThreadPoolExecutor`。第三，任务排队之后是否仍然有意义，如果任务具有时效性，队列过长时执行旧任务可能比拒绝新任务更糟。

一个更可控的固定并发线程池通常会写成下面这样：

```java
ThreadPoolExecutor fixed = new ThreadPoolExecutor(
        8,
        8,
        0L,
        TimeUnit.MILLISECONDS,
        new ArrayBlockingQueue<>(500),
        namedThreadFactory("report-fixed"),
        new ThreadPoolExecutor.AbortPolicy()
);
```

这仍然是固定并发模型，但它和 `Executors.newFixedThreadPool(8)` 有本质区别：队列容量是明确的，饱和后会让调用方知道提交失败。线程池分类不是反对 Fixed，而是反对只看到“固定线程数”这个名字，却忽略它默认搭配了无界队列。

## CachedThreadPool：弹性扩张模型

`CachedThreadPool` 通常通过 `Executors.newCachedThreadPool()` 创建。它对应的底层模板是：核心线程数为 0，最大线程数为 `Integer.MAX_VALUE`，空闲线程存活时间为 60 秒，工作队列使用 `SynchronousQueue`。`SynchronousQueue` 不保存任务，提交任务时必须找到一个空闲工作线程直接交接；如果没有空闲线程，而当前线程数还没达到最大值，线程池就会创建新线程。

缓存线程池表达的是“任务不排队，尽量用线程承接突发”。当大量短任务突然到来时，它可以快速创建线程，任务执行完后线程在空闲一段时间后被回收。这个模型适合大量短小、相互独立、提交峰值明显但持续时间不长的任务。它的核心假设是：任务平均耗时短，阻塞时间可控，突发结束后线程能回落。

这个模型的最大优势是延迟低。由于 `SynchronousQueue` 不让任务在队列里等待，只要系统还能创建线程，新提交的任务就倾向于尽快获得执行线程。对短任务来说，这能减少排队延迟，也能在突发时快速放大并发能力。它的另一个优势是空闲资源回收积极，因为核心线程数为 0，线程空闲超过 60 秒就会被回收。

但缓存线程池的风险也非常明显：最大线程数近似无界。只要任务提交速度超过任务完成速度，线程池就会不断创建新线程。线程本身占用栈空间和调度资源，线程过多会带来内存压力、上下文切换压力、锁竞争压力，甚至导致无法再创建新线程。更危险的是，如果任务会阻塞外部资源，比如等待慢网络、等待锁、等待另一个线程池结果，缓存线程池会把“等待”翻译成“继续创建线程”，最终可能把系统推向更严重的拥塞。

因此，`newCachedThreadPool` 不适合作为通用后台任务池，也不适合处理没有上游限流的高频请求。它看起来能“自动扩缩容”，但这种扩缩容缺少业务容量边界。真正可靠的弹性线程池通常会设置一个有意义的最大线程数，并配合有界队列或调用方限流。也就是说，工程上需要的是“受控弹性”，不是“无上限扩张”。

如果确实需要类似缓存线程池的低排队特性，可以使用 `SynchronousQueue` 加显式最大线程数：

```java
ThreadPoolExecutor elastic = new ThreadPoolExecutor(
        0,
        64,
        60L,
        TimeUnit.SECONDS,
        new SynchronousQueue<>(),
        namedThreadFactory("elastic-worker"),
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

这个线程池仍然倾向于直接交接任务，不把任务堆在队列里，但最多只扩张到 64 个线程。饱和后使用 `CallerRunsPolicy`，提交线程会被迫参与执行，从而降低继续提交的速度。这种设计不一定适合所有场景，却比默认缓存线程池更容易解释过载时的行为。

选择缓存线程池时，应特别警惕两个误区。第一个误区是把“可缓存”理解为“更节省资源”。它在空闲时确实能回收线程，但在压力下可能比固定线程池消耗更多资源。第二个误区是把“没有队列堆积”理解为“没有积压”。任务没有堆在 Java 队列里，不代表系统没有积压；它们可能积压成大量正在阻塞或等待调度的线程。

## SingleThreadExecutor：串行执行模型

`SingleThreadExecutor` 通常通过 `Executors.newSingleThreadExecutor()` 创建。它的底层思想是只有一个工作线程执行任务，任务按提交顺序进入队列，前一个任务完成后再执行后一个任务。典型参数可以理解为核心线程数和最大线程数都为 1，工作队列为无界 `LinkedBlockingQueue`。与直接创建一个单线程 `ThreadPoolExecutor` 不同，`Executors` 返回的是包装后的 `ExecutorService`，调用方不能把它强转回 `ThreadPoolExecutor` 修改线程数。

单线程执行器表达的是“把并发提交转化为串行执行”。它适合维护顺序性、避免共享状态并发访问、把一组相关任务收敛到同一执行流中。例如某些任务必须按提交顺序更新同一个内存状态，或者某个组件希望所有状态变更都在同一个线程中完成，那么单线程执行器可以降低锁的复杂度。它不是为了提升吞吐，而是为了提供稳定的执行顺序和更简单的状态模型。

单线程执行器有一个重要保证：如果工作线程因为任务抛出未捕获异常而终止，线程池会创建新的工作线程继续执行后续任务。也就是说，它保证的是“同一时间最多一个任务在执行”，不是“永远由同一个物理线程执行”。这一区别会影响 `ThreadLocal`、线程本地缓存、线程绑定资源等设计。如果代码把状态放在线程本地变量中，并假设永远由同一个线程保存，那么任务异常导致线程替换后，这些状态就可能丢失或变化。

单线程执行器的最大风险同样来自无界队列。由于只有一个线程执行任务，任何慢任务都会阻塞后续所有任务。提交方可能持续把任务放进队列，表面上没有并发冲突，实际上延迟已经被无限放大。如果队列中的任务具有时效性，单线程执行器可能会在很久之后执行已经过期的任务。串行并不等于安全，顺序性也不能替代容量控制。

使用单线程执行器时，最重要的是保证任务短小、非阻塞、边界清晰。一个长时间阻塞的任务会让整个执行器停止向前推进。更严重的是，如果运行在该执行器中的任务又提交另一个任务到同一个执行器，并同步等待后者完成，就会形成自我等待。因为唯一工作线程正在等待，而被等待的任务必须等当前任务结束后才有机会执行，这就是典型的线程池内饥饿死锁。

下面的代码展示了这种不应该出现的模式：

```java
ExecutorService single = Executors.newSingleThreadExecutor();

Future<String> outer = single.submit(() -> {
    Future<String> inner = single.submit(() -> "inner result");
    return inner.get(); // 唯一工作线程在这里等待，inner 没有机会运行
});
```

避免这类问题的方法不是简单地“换成更多线程”，而是识别任务之间的等待关系。串行执行器适合表达顺序，不适合在同一执行流内部阻塞等待后续任务。如果需要组合异步结果，应使用回调、`CompletableFuture` 的组合操作，或者把被等待任务提交到独立执行器，并且明确超时和取消策略。

单线程执行器也常被误用为“简化线程安全”的万能工具。它确实能让进入该执行器的任务串行执行，但它无法阻止其他线程绕过执行器直接访问同一状态。只有当共享状态的所有写入和相关读取都通过同一个串行边界进入时，单线程模型才成立。否则，单线程执行器只是局部顺序，不能保证整体线程安全。

## ScheduledThreadPool：延迟与周期调度模型

`ScheduledThreadPool` 通常通过 `Executors.newScheduledThreadPool(corePoolSize)` 创建，底层实现是 `ScheduledThreadPoolExecutor`。它的核心线程数由参数指定，最大线程数在实现中虽然可以设置，但由于使用的是内部延迟队列，通常不会像普通 `ThreadPoolExecutor` 那样通过队列满来扩张线程。任务队列按触发时间排序，只有到期任务才会被工作线程取出执行。

调度线程池表达的是“任务不是立即执行，而是在时间条件满足后执行”。它支持一次性延迟任务，也支持周期任务。一次性延迟任务通过 `schedule` 提交，在给定延迟后执行一次；固定频率周期任务通过 `scheduleAtFixedRate` 提交，按照初始延迟和周期计算下一次计划开始时间；固定延迟周期任务通过 `scheduleWithFixedDelay` 提交，以上一次执行结束时间为基准，再等待指定延迟后执行下一次。

固定频率和固定延迟的差异非常关键。`scheduleAtFixedRate` 关注计划节奏，适合希望按稳定频率触发的任务。如果某次执行耗时超过周期，下一次执行不会并发重入同一个周期任务，而是可能在上一次结束后尽快补上。`scheduleWithFixedDelay` 关注两次执行之间的休息间隔，适合任务耗时波动较大、希望避免连续追赶的场景。前者更像“按时钟节拍运行”，后者更像“做完一轮再休息一段时间”。

调度线程池的一个常见误区是把它当成精确计时器。Java 调度线程池只能保证任务不会早于给定延迟被启用，并不保证一定在那个时刻立即获得 CPU 执行。系统负载、线程数不足、长任务占用、垃圾回收、操作系统调度都会影响实际执行时间。因此，调度线程池适合一般延迟和周期调度，不适合需要硬实时保证的场景。

另一个重要边界是异常处理。周期任务如果在执行过程中抛出未捕获异常，后续执行通常会被取消。很多人以为周期任务会一直运行，结果某次偶发异常后任务静默停止。稳健的周期任务应在任务内部捕获并记录异常，确保一次失败不会意外终止整个周期调度。当然，是否吞掉异常要根据任务语义决定，但不能让异常无声地结束关键调度。

```java
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

ScheduledFuture<?> future = scheduler.scheduleWithFixedDelay(() -> {
    try {
        refreshCache();
    } catch (Exception ex) {
        logError(ex);
    }
}, 1, 10, TimeUnit.SECONDS);
```

调度线程池还需要关注关闭策略。`ScheduledThreadPoolExecutor` 提供了关于关闭后是否继续执行已有延迟任务、是否继续执行已有周期任务、取消任务是否从队列中移除等配置。默认行为并不总是符合业务预期。例如关闭时是否还要执行已经排队但未到期的任务，取决于任务是必须完成的收尾工作，还是已经没有意义的定时触发。对长期运行程序来说，取消后的任务如果长期留在队列中，也可能造成内存占用，因此可以根据需要启用取消即移除策略。

调度线程池适用的任务应尽量短小。调度线程池的线程既负责执行到期任务，也影响后续任务被取出的及时性。长时间阻塞任务会拖延同一调度池中的其他任务。如果任务到期后要执行大量耗时工作，常见做法是让调度线程池只负责触发，再把实际重任务提交到普通工作线程池。不过这会引入两个线程池之间的容量关系，必须确保工作线程池饱和时不会让调度线程不断制造积压。

## SingleThreadScheduledExecutor：串行调度模型

`Executors.newSingleThreadScheduledExecutor()` 可以看作调度线程池的单线程变体。它把延迟、周期和串行执行组合在一起：所有到期任务都由单个工作线程按调度顺序执行，同一时间只有一个任务运行。如果工作线程异常终止，执行器也会补充新线程继续处理后续任务。

这个模型适合需要顺序调度的轻量任务。例如多个周期性维护任务共享同一份状态，并且希望避免并发执行带来的锁复杂度，可以使用单线程调度执行器。但它的风险也非常直接：任何一个任务执行过久，都会影响其他到期任务。任务之间虽然不会并发冲突，却会互相拖延。

单线程调度执行器特别不适合混放不同优先级、不同耗时特征的任务。一个低优先级的慢任务可能阻塞高优先级的短周期任务；一个执行失败后不断重试的任务也可能占住唯一线程。分类上它看起来只是 Scheduled 和 Single 的组合，工程上却要同时承受二者的边界：无并发冲突，但吞吐受限；有时间调度能力，但时间精度受单线程执行影响。

如果使用这个模型，应尽量让任务具备三个特征：每次执行耗时短，异常被清楚记录，任务之间没有同步等待。如果某个任务可能执行很久，应拆到独立工作池；如果某个任务具有更高时效性，应单独使用调度池隔离。

## WorkStealingPool 与 ForkJoinPool：工作窃取模型

`WorkStealingPool` 通过 `Executors.newWorkStealingPool()` 或 `Executors.newWorkStealingPool(parallelism)` 创建，底层是 `ForkJoinPool`。与普通线程池不同，`ForkJoinPool` 面向可拆分任务和递归并行。每个工作线程通常维护自己的双端队列，自己执行任务时倾向于从本地队列一端取任务；当某个工作线程没有任务时，会尝试从其他工作线程的队列另一端窃取任务。这个机制减少了全局队列竞争，提高了细粒度任务的负载均衡能力。

工作窃取模型适合可以拆分为许多子任务、子任务之间相对独立、最终需要合并结果的计算。例如分治算法、树形遍历、数组分段计算、递归搜索等。它的典型使用方式不是提交大量彼此无关的长阻塞任务，而是把一个大任务拆成小任务，让空闲工作线程主动窃取未完成部分。`RecursiveTask` 和 `RecursiveAction` 正是为这种模型准备的。

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10_000;
    private final long[] values;
    private final int start;
    private final int end;

    SumTask(long[] values, int start, int end) {
        this.values = values;
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) {
                sum += values[i];
            }
            return sum;
        }
        int mid = (start + end) >>> 1;
        SumTask left = new SumTask(values, start, mid);
        SumTask right = new SumTask(values, mid, end);
        left.fork();
        long rightResult = right.compute();
        long leftResult = left.join();
        return leftResult + rightResult;
    }
}
```

工作窃取池的参数核心是并行度，而不是普通线程池中的核心线程数和最大线程数。并行度表示目标并行工作线程数量，默认通常与可用处理器数量相关。`ForkJoinPool` 可以在一定情况下补偿阻塞，但它的优势仍然建立在任务主要进行计算、拆分粒度合理、阻塞较少的前提上。如果把大量阻塞 I/O 任务提交给工作窃取池，工作线程会被等待占住，窃取机制也无法凭空产生吞吐。

使用工作窃取池时，要特别注意任务粒度。拆得太粗，空闲线程没有足够任务可窃取，并行度发挥不出来；拆得太细，任务对象创建、入队、窃取和合并的开销会超过计算收益。合理做法是设置阈值，让子任务在规模足够小时直接顺序计算。阈值没有固定答案，需要根据计算成本、数据规模和运行环境测试。

另一个边界是执行顺序。普通固定线程池虽然也不保证严格完成顺序，但任务通常来自一个共享队列；工作窃取池内部有多个队列，任务执行顺序更不适合被业务依赖。只要任务之间存在必须按顺序执行的副作用，就不应把工作窃取模型当作默认选择。它更适合纯计算或接近纯计算的任务，而不是依赖外部状态顺序更新的任务。

`Executors.newWorkStealingPool()` 返回的是 `ExecutorService`，它的线程通常是守护线程，并且任务可能在不同工作线程中执行。使用时不要依赖线程本地状态，也不要把它与需要长期存活的关键后台任务混在一起。对于并行流，Java 标准库也会使用公共 `ForkJoinPool`，这意味着某些看似无关的并行计算可能共享同一个公共池。公共池方便，但也容易造成互相影响；关键任务最好使用独立池。

## Scheduled 与普通线程池的本质差异

很多分类文章会把 Fixed、Cached、Single、Scheduled 平铺在一起，但 Scheduled 和前三者不只是参数不同。普通线程池解决的是“已有任务如何并发执行和排队”，调度线程池额外解决“任务何时变为可执行”。因此，它的队列不是按提交顺序简单排队，而是按触发时间组织。一个很早提交但延迟很长的任务，不应阻塞后面提交但更早到期的任务。

这种差异会影响容量判断。普通线程池的队列长度通常代表等待执行的任务数量；调度线程池的队列中可能包含大量尚未到期的任务，它们并不都构成当前执行压力，却构成未来触发压力和内存占用。周期任务还会在每次执行后重新计算下一次触发时间，因此一个周期任务不是简单的一次性队列元素，而是一个持续产生执行需求的调度实体。

调度线程池还涉及时间语义。延迟基于相对时间，而不是用来表达业务日历时间。标准调度执行器不会自动处理复杂日历规则，例如工作日、节假日、月末、夏令时切换等。若任务需要日历级调度，应在应用层明确计算下一次触发时间，或使用专门调度框架。线程池分类中把 Scheduled 理解成“定时器”可以入门，但工程上必须进一步区分相对延迟、固定频率、固定延迟和日历调度。

## Executors 工厂方法的便利与风险

`Executors` 的工厂方法降低了入门成本。用一行代码就能创建常见执行器，示例和小工具中非常方便。但在长期运行、流量不可控、任务耗时不稳定的程序中，直接使用这些工厂方法往往隐藏容量边界。所谓“Executors 工厂风险”，不是说这些方法本身错误，而是它们把关键参数折叠进默认模板，调用方容易忽略队列是否无界、线程数是否无界、拒绝策略是否符合预期。

`newFixedThreadPool` 和 `newSingleThreadExecutor` 的主要风险是无界队列。它们限制了并发线程数，却没有限制等待任务数。压力超过处理能力时，系统不会及时拒绝，而是把任务堆积起来。堆积越久，任务越过期，内存越紧张，排查越困难。很多系统在故障早期没有异常日志，因为提交确实成功；直到资源耗尽时，故障已经扩大。

`newCachedThreadPool` 的主要风险是最大线程数过大。它不让任务在队列中等待，而是倾向于创建线程承接任务。压力持续时，线程数会快速上升。线程数过多不仅占用内存，还会造成调度竞争，导致每个线程获得 CPU 的机会下降。更糟的是，线程数增长可能加剧下游资源竞争，使原本的慢任务更慢。

`newScheduledThreadPool` 的风险在于延迟队列和周期任务的长期积累。大量延迟任务如果没有取消策略，会占用队列；周期任务如果执行时间长于周期，会持续追赶；周期任务异常如果没有记录，可能悄悄停止。它不像普通线程池那样只看当前提交速度，还要看未来触发密度。

`newWorkStealingPool` 的风险在于模型不匹配。它适合可拆分计算，不适合作为所有异步任务的默认容器。把阻塞任务、顺序副作用任务、依赖线程本地上下文的任务都塞进工作窃取池，会削弱它的优势，还可能带来诊断困难。工作窃取池强调吞吐和负载均衡，不强调提交顺序。

因此，生产代码中常见建议是避免直接使用 `Executors` 创建默认线程池，而是显式构造 `ThreadPoolExecutor` 或 `ScheduledThreadPoolExecutor`。这条建议的重点不是“必须手写构造器才高级”，而是要求开发者把容量、命名、拒绝和关闭写清楚。对于小型命令行工具、一次性脚本、教学示例，工厂方法可以接受；对于服务端核心路径、长期运行任务、上游流量不可控路径，应优先显式配置。

下面的表格概括了常见工厂方法的底层模型和主要风险：

| 工厂方法 | 核心模型 | 典型队列 | 线程数量策略 | 主要风险 |
| --- | --- | --- | --- | --- |
| `newFixedThreadPool(n)` | 固定并发 | 无界 `LinkedBlockingQueue` | `core=max=n` | 任务无限堆积，延迟和内存压力被隐藏 |
| `newCachedThreadPool()` | 弹性扩张 | `SynchronousQueue` | `core=0,max` 近似无界 | 线程数量失控，阻塞任务放大资源竞争 |
| `newSingleThreadExecutor()` | 串行执行 | 无界 `LinkedBlockingQueue` | `core=max=1` | 慢任务阻塞后续任务，队列无限增长 |
| `newScheduledThreadPool(n)` | 延迟和周期调度 | 延迟队列 | 以核心线程执行到期任务 | 周期任务异常停止，长任务拖延调度 |
| `newWorkStealingPool()` | 工作窃取 | 多个工作队列 | 按并行度组织 | 不适合阻塞任务和顺序副作用任务 |

## 自定义线程池的选择思路

自定义线程池不是为了追求复杂，而是为了让资源边界和任务语义匹配。选择线程池时，应先描述任务，再描述参数。任务描述至少包括：任务是 CPU 密集还是 I/O 等待密集；任务平均耗时和尾部耗时如何；任务是否有时效性；任务是否允许丢弃；任务是否可以重试；任务是否访问共享状态；任务是否会提交子任务并等待；任务失败如何传播；关闭时未完成任务是否必须处理。

如果任务是 CPU 密集型，线程数通常接近可用处理器数量，或者略作调整。CPU 密集型任务的瓶颈是计算资源，把线程数设置得很大并不会让 CPU 变多，只会增加上下文切换和缓存失效。此类任务更适合固定并发或工作窃取模型。若任务可以分治拆分，`ForkJoinPool` 可能比普通固定线程池更合适；若任务不可拆分，只是独立计算任务，固定线程池通常更直观。

如果任务是 I/O 等待型，线程数可以大于处理器数量，因为线程等待外部资源时不会一直占用 CPU。但这不意味着线程数可以无限大。外部系统通常也有容量边界，例如连接数、请求频率、文件句柄、远端服务限流。I/O 线程池的大小应结合等待比例和下游容量估算，并通过有界队列、超时和拒绝策略防止积压。对于高并发 I/O，是否使用异步 I/O 或非阻塞模型也应纳入设计，而不是只靠增加线程数。

如果任务有严格顺序要求，可以考虑单线程执行器或按键分片的多单线程模型。单线程执行器简单，但吞吐受限；按业务键分片可以让同一个键串行，不同键并行。例如按账户、用户、订单等维度分派到不同队列，可以在保持局部顺序的同时提高整体并发。标准库没有直接提供完整的 keyed executor，但可以通过多个单线程池或自定义调度实现。关键是顺序边界必须明确，不能让同一份状态有时走串行池，有时被外部线程直接修改。

如果任务是周期性维护任务，应使用调度线程池，但要区分触发和执行。轻量维护任务可以直接在调度池中执行；耗时任务应由调度池触发后交给工作池。后一种设计要小心：如果工作池已经饱和，调度池仍持续触发，积压会转移到工作池队列中。必要时应在触发前检查上一次任务是否完成，或者使用显式状态避免重叠提交。

如果任务可以拆分并合并结果，应考虑工作窃取模型。使用前要确认任务主要是计算，而不是阻塞等待；确认子任务之间没有顺序副作用；确认拆分阈值合理；确认不会在公共池中执行可能长时间阻塞的操作。工作窃取池不是“更高级的线程池”，而是服务于特定计算结构的工具。

队列容量的选择同样重要。有界队列可以让过载尽早暴露，但容量太小会导致正常波峰下频繁拒绝；容量太大则会隐藏延迟，让任务在队列中等待很久。一个实用思路是按可接受排队时间反推容量：如果线程池每秒稳定处理 1,000 个任务，而业务最多接受 2 秒排队，那么队列容量就不应远大于 2,000。这个估算很粗，但比“随便设一个很大的数”更接近容量设计。

拒绝策略应与业务语义匹配。不能丢的任务应拒绝并让调用方重试或降级，不能静默丢弃；可以被新任务覆盖的刷新类任务，可以考虑丢弃旧任务或合并任务；提交方本身可以承受变慢时，`CallerRunsPolicy` 能形成简单反压；对低优先级统计任务，丢弃可能比拖垮主流程更合理。拒绝策略不是技术默认值，而是业务取舍。

## 关闭与生命周期分类

线程池分类还应包含生命周期视角。一个线程池从创建到关闭，至少经历运行、停止接收、等待已提交任务结束、强制取消、终止这几个阶段。`shutdown()` 表示不再接受新任务，但已提交任务会继续执行；`shutdownNow()` 会尝试中断正在执行的任务，并返回尚未开始执行的任务列表。这里的“尝试中断”非常重要，线程池无法强制杀死一个不响应中断的任务。

不同类型线程池的关闭风险不同。固定线程池和单线程池如果队列很长，`shutdown()` 后可能需要很久才能终止，因为队列中任务会继续执行。缓存线程池如果有大量正在阻塞的线程，`shutdownNow()` 也未必能迅速结束，因为任务可能吞掉中断或阻塞在不响应中断的位置。调度线程池关闭时还要决定延迟任务和周期任务是否继续。工作窃取池中如果任务递归等待不当，也可能导致关闭等待很久。

规范的关闭通常包含三个动作：先调用 `shutdown()`，再用 `awaitTermination` 等待一段时间，如果超时再调用 `shutdownNow()`，最后处理中断并记录未完成状态。不能只创建线程池而不关闭，尤其是默认非守护线程会阻止进程退出。也不能在共享线程池上由局部代码随意关闭，否则其他模块提交任务会收到拒绝异常。

```java
executor.shutdown();
try {
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        List<Runnable> dropped = executor.shutdownNow();
        logUnfinishedTasks(dropped);
    }
} catch (InterruptedException ex) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

生命周期所有权要在设计阶段确定。谁创建线程池，谁就通常负责关闭；如果线程池注入给多个组件使用，关闭应由容器或统一生命周期管理者完成；如果方法内部临时创建线程池，方法必须确保最终关闭；如果返回的是共享执行器，文档应说明调用方不能关闭。线程池分类如果只讨论运行期执行模型，而不讨论关闭模型，就会留下资源泄漏和任务丢失风险。

## 任务依赖与线程池饥饿

线程池分类的很多事故不是因为线程数太小，而是因为任务依赖关系与线程池模型冲突。最典型的是线程池饥饿死锁：线程池中的任务占住所有工作线程，然后同步等待同一个线程池中尚未执行的其他任务。因为没有空闲线程执行被等待任务，等待方也无法释放线程，系统就卡住。

固定线程池、单线程池和工作窃取池都可能遇到这个问题，只是表现不同。单线程池中，一个任务等待同池后续任务几乎必然卡住；固定线程池中，如果所有线程都在等待同池子任务，也会卡住；工作窃取池对 `fork/join` 模式做了特殊优化，但如果阻塞等待外部 `Future` 或普通同步器，仍可能破坏调度假设。

避免饥饿死锁的原则是：不要在有限线程池的工作线程中无界等待同一有限线程池的任务。如果确实需要任务组合，应使用同一模型支持的组合方式，例如 `ForkJoinTask.join()` 配合合理的 fork/join 结构，或 `CompletableFuture` 的非阻塞组合；如果必须阻塞等待，应使用独立线程池隔离依赖，并设置超时；如果任务有父子关系，应确认父任务等待时不会占住所有执行子任务的资源。

任务依赖还会影响队列选择。无界队列会让子任务顺利提交但迟迟不能执行；有界队列会更早暴露饱和，但也可能让工作线程在提交子任务时被拒绝；`CallerRunsPolicy` 可能让工作线程直接执行子任务，从而改变执行顺序。没有一种策略永远正确，关键是把依赖关系画清楚，再决定是否应该拆分线程池、改成异步组合或降低并发。

## ThreadLocal、异常与上下文传播

线程池复用线程，因此线程本地状态会跨任务存在。`ThreadLocal` 在普通新建线程模型中可能随着线程结束而消失，但在线程池中，工作线程执行完一个任务后不会立即销毁。若任务设置了 `ThreadLocal` 却没有清理，下一个复用同一工作线程的任务就可能看到旧值，或者至少让旧对象长期无法释放。这与 Fixed、Cached、Single 都有关，只是单线程池中更容易被误以为“本来就是同一个上下文”。

正确做法是在任务边界清理线程本地状态。设置 `ThreadLocal` 后应在 `finally` 中调用 `remove()`。如果需要跨线程传播上下文，不应假设线程池自动复制调用方线程的线程本地变量，而应显式捕获需要的值并作为任务参数传入，或使用经过审慎设计的上下文传播机制。线程池分类中的“线程复用”既是性能优化，也是上下文污染的来源。

异常处理也有分类差异。通过 `execute` 提交的 `Runnable` 抛出运行时异常时，异常会到达工作线程的未捕获异常处理逻辑，当前工作线程可能终止，线程池再补充新线程。通过 `submit` 提交的任务，异常会被封装进返回的 `Future`，只有调用 `get()` 时才会重新抛出。如果调用方从不检查 `Future`，异常就像被吞掉一样。周期调度任务的异常还可能导致后续周期取消。

因此，线程池使用中必须明确异常观察点。对于不关心返回值的后台任务，也应在任务内部捕获并记录异常；对于 `submit` 返回的 `Future`，应有地方调用 `get()` 或集中检查；对于 `CompletableFuture`，应配置异常处理阶段；对于调度任务，应防止一次偶发异常让周期任务无声消失。线程池只负责执行任务，不会自动把后台异常变成调用线程异常。

## 虚拟线程执行器的位置

从 Java 21 开始，虚拟线程成为标准平台特性，`Executors.newVirtualThreadPerTaskExecutor()` 提供了每个任务一个虚拟线程的执行器模型。它和传统线程池分类的关系需要谨慎理解：虚拟线程执行器不是通过复用少量平台线程来限制并发，而是让每个任务拥有一个轻量级虚拟线程，由运行时把大量虚拟线程调度到较少的平台线程上执行。

这个模型适合大量以阻塞风格编写的并发任务，尤其是任务大部分时间在等待 I/O 或同步结果，而代码又希望保持顺序写法。它能显著降低“一个任务一个平台线程”的成本，使代码不必为了扩展性过早改成复杂回调。但它并不取消容量设计。外部数据库连接、远端服务、文件句柄、内存、锁竞争仍然是有限资源。虚拟线程让线程本身不再是最紧张的资源，却不能让下游资源无限扩张。

虚拟线程执行器也不等同于 `CachedThreadPool`。缓存线程池创建的是平台线程，线程数量过多会带来沉重系统成本；虚拟线程执行器创建的是虚拟线程，创建和阻塞成本低得多。但二者都有一个相似风险：如果上游无限提交任务，而下游资源没有限流，系统仍然会积压。区别只是积压形态从平台线程爆炸，变成大量虚拟线程、等待对象、请求上下文和下游排队。

因此，在传统线程池分类之外，可以把虚拟线程看作“高并发阻塞任务的执行模型”，但不要把它当成“无需线程池设计”。使用虚拟线程时，常见做法是用信号量、连接池、限流器或有界队列保护真正稀缺的资源。也就是说，线程不再是主要限额，资源许可才是主要限额。

## 常见分类误区

第一个误区是认为线程池名称就是容量设计。Fixed 固定的是线程数，不固定队列长度；Cached 缓存的是空闲线程，不限制线程总数；Single 保证串行，不保证任务及时；Scheduled 支持延迟，不保证精确定时；WorkStealing 支持窃取，不适合所有异步任务。名称只是入口，参数和任务特征才决定行为。

第二个误区是认为线程数越大吞吐越高。线程数增加只有在存在可并行工作且资源未饱和时才可能提高吞吐。CPU 已满时增加线程会增加切换成本；下游服务已满时增加线程会增加等待和重试；锁竞争严重时增加线程会增加阻塞；内存紧张时增加线程会加速故障。线程池不是把压力消除，而是改变压力排队的位置。

第三个误区是只看平均耗时，不看尾部耗时。线程池容量常常被少量慢任务拖垮。平均 10 毫秒的任务，如果偶尔出现 5 秒阻塞，就可能占住工作线程并造成队列上涨。选择 Fixed 还是 Cached、队列设多大、超时设多长，都需要看耗时分布，而不是只看平均值。

第四个误区是多个业务共用一个线程池。共享线程池可以减少资源数量，但会引入互相影响。低优先级慢任务可能占满线程，导致高优先级任务延迟；大量周期任务可能挤压即时任务；阻塞任务可能拖垮计算任务。线程池隔离不是越多越好，但不同耗时、不同优先级、不同失败语义的任务应谨慎混放。

第五个误区是忽略队列中的等待时间。很多监控只看任务执行耗时，却不看从提交到开始执行之间的排队时间。用户感知的是总延迟，排队时间同样重要。一个任务执行只花 20 毫秒，但在队列里等了 5 秒，仍然是严重延迟。线程池分类中的队列策略，必须和排队时间指标一起看。

第六个误区是吞掉中断。线程池关闭、任务取消、超时控制都依赖中断协作。任务捕获 `InterruptedException` 后如果什么都不做，会让调用方误以为取消已经发出，实际任务仍可能继续运行。当前层无法完成取消处理时，应恢复中断标志，并尽快退出或把状态交给上层处理。

第七个误区是把拒绝当作异常事故，而不是正常容量信号。一个设计良好的有界线程池，在过载时拒绝任务是合理行为。拒绝说明系统保护了自身边界，调用方应降级、重试、返回错误或施加反压。真正危险的是没有拒绝，所有任务都被接收，直到内存或线程资源耗尽。

## 诊断线程池分类问题

排查线程池问题时，应先判断它属于哪一种失效形态：线程不够、队列过长、线程过多、任务阻塞、任务异常丢失、调度延迟、关闭不退出，还是上下文污染。不同分类对应的优先观察点不同。固定线程池优先看活跃线程数、队列长度和任务耗时；缓存线程池优先看线程数量增长和阻塞栈；单线程池优先看唯一工作线程卡在哪里；调度线程池优先看到期任务是否被长任务拖延；工作窃取池优先看任务拆分、阻塞和公共池争用。

线程 dump 是非常直接的工具。看到大量线程处于 `BLOCKED`，说明锁竞争或同步块等待严重；看到大量 `WAITING` 或 `TIMED_WAITING`，要结合栈判断是在等待队列、等待 Future、等待条件、等待 I/O，还是正常空闲；看到线程名都来自同一个线程池，可以迅速定位是哪类任务占用资源。没有可读线程名时，排查会困难很多。

线程池指标至少应包含当前线程数、活跃线程数、核心线程数、最大线程数、队列长度、已完成任务数、拒绝次数和任务耗时。更进一步，还应记录排队时间。对调度线程池，应记录实际执行时间与计划触发时间的偏差；对周期任务，应记录连续失败次数和最近一次成功时间；对工作窃取池，应关注并行度、窃取次数和阻塞情况。

诊断时不要只看最终报错。线程池故障经常是链式的：某个下游变慢，工作线程开始阻塞，队列开始增长，调用方等待超时，重试增加提交量，线程池进一步拥塞，最终出现拒绝或内存压力。最终异常只是结果，真正的问题在第一个容量边界失效的位置。把任务提交、开始执行、执行结束、异常、拒绝、取消和关闭都打点，才能还原时间线。

## 分类选择的实践清单

选择线程池时，可以按下面的顺序做判断。第一，任务是否需要延迟或周期触发；如果需要，优先考虑 `ScheduledThreadPoolExecutor`，再决定任务是在调度池内执行还是转交工作池。第二，任务是否必须串行；如果必须串行，考虑单线程执行器或按键分片执行器，并确认所有相关状态都经过同一串行入口。第三，任务是否适合分治计算；如果适合，考虑 `ForkJoinPool` 或工作窃取池。第四，任务是否是普通独立任务；如果是，再根据 CPU 密集或 I/O 等待密集选择固定并发、受控弹性或其他模型。

随后再确定参数。线程数来自资源容量，不来自随手填写；队列容量来自可接受排队时间和内存预算，不来自默认无界；拒绝策略来自业务语义，不来自默认习惯；线程名来自诊断需要，不应保留难以识别的默认名；关闭策略来自生命周期所有权，不应交给进程退出时碰运气。

最后做压力验证。线程池参数不能只靠静态推理。应该在接近真实任务耗时、真实提交速率和真实下游限制的环境中观察队列长度、拒绝次数、排队时间、线程数、CPU、内存和任务成功率。尤其要测试慢任务、突发流量、下游超时、任务异常和关闭过程。分类选择正确只是第一步，参数是否合理必须通过运行数据验证。

## 小结

线程池分类的核心，是把不同执行模型和不同容量边界对应起来。`FixedThreadPool` 适合固定并发，但默认无界队列会隐藏积压；`CachedThreadPool` 适合短任务突发，但默认最大线程数过大；`SingleThreadExecutor` 适合串行化状态变更，但慢任务会阻塞所有后续任务；`ScheduledThreadPool` 适合延迟和周期调度，但不能提供硬实时保证，周期任务异常和长任务拖延都需要处理；`WorkStealingPool` 适合可拆分计算，但不适合作为阻塞任务和顺序副作用任务的默认容器；虚拟线程执行器扩展了高并发阻塞任务的选择，但仍然需要保护真正稀缺的外部资源。

生产代码中，线程池不应只按名字选择，而应按任务特征、资源边界、排队策略、拒绝语义和生命周期所有权选择。`Executors` 工厂方法适合学习、示例和边界清楚的小场景；在长期运行和流量不可控的程序中，更推荐显式构造线程池，把线程数、队列、线程工厂、拒绝策略和关闭流程写清楚。线程池分类学到最后，应该形成一种判断能力：看到一个执行器名称，就能立刻追问它的队列是否有界、线程是否可能失控、任务是否会阻塞、异常是否能被观察、关闭是否能完成，以及过载时系统会在哪里承受压力。

# 3.3.3.6 正确关闭线程池

## 定位

正确关闭线程池是 Java 并发编程中非常容易被低估的主题。很多代码在创建线程池时会认真讨论核心线程数、最大线程数、队列类型、拒绝策略和线程工厂，却在关闭时只补上一行 `shutdown()`，甚至完全不关闭。在线程池短时间运行、进程很快退出、任务数量不多的演示代码里，这样做看起来没有明显问题；一旦线程池进入长期运行的服务、批处理程序、命令行工具、测试框架、插件式组件或可重复启动停止的模块，关闭阶段就会暴露出线程泄漏、任务丢失、进程无法退出、资源迟迟不释放、关闭过程卡死、调用方误以为任务完成等问题。

线程池关闭并不是“把线程杀掉”。`ExecutorService` 提供的是一个协作式生命周期协议：调用方停止提交新任务，线程池拒绝新的工作，已经提交的任务按照既定规则继续执行或被尝试取消，调用方可以等待终止，也可以在超时后升级为更强的关闭动作。这个协议能否生效，取决于线程池实现、任务代码、资源使用方式、异常处理方式和调用方等待策略共同配合。线程池本身无法神奇地让一个无视中断、永久阻塞或持有外部资源不释放的任务安全退出。

本文只从通用 Java 技术视角讨论线程池关闭。这里的“正确”不是指每个场景都套用同一段模板，而是指关闭逻辑能够回答几个关键问题：关闭由谁发起，关闭之后是否继续接收任务，已经在运行的任务如何处理，队列里尚未运行的任务是否允许执行，等待多久算超时，超时之后如何告警或升级，工作线程被中断时任务是否能尽快退出，任务持有的文件、网络连接、锁、临时对象或其他资源由谁释放，以及关闭失败时调用方能观察到什么。

## 为什么关闭线程池是生命周期问题

线程池隐藏了线程创建、复用和任务调度的细节，但没有消除生命周期。只要一个对象内部持有工作线程，它就不只是普通对象，而是一个有启动、运行、停止和终止阶段的执行器。线程池中的工作线程通常是非守护线程，非守护线程不结束，Java 进程就不会自然退出。即使某些场景使用守护线程，也不能把守护线程当成资源释放手段，因为守护线程可能在进程退出时被直接丢弃，任务的 finally 块、缓冲区刷新和外部资源关闭都未必有机会完整执行。

生命周期问题还体现在所有权上。谁创建线程池，谁通常就应该负责关闭线程池；谁把线程池作为共享依赖暴露出去，谁就应该定义调用者能不能关闭它。一个局部方法创建的短生命周期线程池，如果没有在 finally 中关闭，会让线程和队列对象留在进程中；一个全局共享线程池，如果被某个局部业务随手关闭，会导致其他仍在使用它的模块提交任务失败。关闭线程池的第一条原则不是选 `shutdown` 还是 `shutdownNow`，而是先确认线程池的所有权和生命周期边界。

线程池关闭还和任务语义绑定。线程池只知道任务是 `Runnable` 或 `Callable`，并不知道任务是在计算、等待锁、读取文件、等待队列、执行外部调用，还是在循环处理一批数据。对线程池来说，`shutdownNow()` 可以尝试中断正在运行的工作线程，并返回尚未开始执行的任务列表；对任务来说，中断只是一个请求。任务如果没有在阻塞点响应 `InterruptedException`，也没有在循环中检查中断状态，关闭线程池就可能长时间无法结束。

因此，正确关闭线程池至少包含两层协议：线程池生命周期协议和任务取消协议。前者由 `ExecutorService` 的方法表达，后者由任务代码自己实现。只写前者，不设计后者，关闭流程只能在简单任务上看似可靠；只设计任务取消，不停止线程池接收新任务，又会出现一边关闭一边继续提交的竞态。两层协议合在一起，才构成完整的关闭策略。

## ExecutorService 的关闭状态

理解关闭方法之前，需要先区分几个状态概念。`ExecutorService` 对外暴露了 `shutdown()`、`shutdownNow()`、`isShutdown()`、`isTerminated()`、`awaitTermination()` 等方法。它们描述的不是同一个问题。

`isShutdown()` 表示执行器是否已经开始关闭。只要调用过 `shutdown()` 或 `shutdownNow()`，通常就会进入 shutdown 状态。进入这个状态后的核心含义是：不再接收新任务。它不表示所有任务已经完成，也不表示工作线程已经退出。许多误用来自把 `isShutdown()` 当成“已经关完”，然后提前释放依赖资源或返回成功。

`isTerminated()` 表示关闭之后所有任务都已经完成，工作线程已经停止，线程池进入终止状态。只有先进入关闭流程，才可能最终 terminated。一个还在正常运行的线程池，即使当前没有任务，通常也不是 terminated；一个已经调用 `shutdown()` 的线程池，如果队列中还有任务或工作线程还在执行，也不是 terminated。

`awaitTermination(timeout, unit)` 表示当前线程等待线程池进入 terminated 状态，最多等待给定时间。它返回 `true` 说明在超时前已经终止，返回 `false` 说明等待到期但还没有终止。它还可能抛出 `InterruptedException`，表示等待线程自己被中断了。注意，`awaitTermination` 本身不会自动发起关闭；如果调用前没有执行 `shutdown()` 或 `shutdownNow()`，等待通常没有意义，因为线程池仍然处于可接收任务状态。

可以把线程池生命周期粗略理解为三个外部可观察阶段：运行中、正在关闭、已经终止。运行中可以提交任务；正在关闭会拒绝新任务，但已提交任务仍可能继续处理；已经终止表示没有任务和工作线程残留。真实实现内部可能有更细状态，例如 `ThreadPoolExecutor` 内部会区分运行、关闭、停止、整理和终止等阶段，但使用方最重要的是不要混淆“开始关闭”和“完成关闭”。

## shutdown 的语义

`shutdown()` 是温和关闭。它的基本语义是：线程池不再接受新提交的任务，但会继续执行已经提交的任务，包括正在运行的任务和阻塞队列中尚未开始的任务。它不会主动中断正在执行的任务。对调用方来说，`shutdown()` 是发出“停止接收新工作，处理完存量工作后退出”的请求。

这种语义适合任务必须尽量执行完的场景。例如已经提交的任务代表一批内存内计算、文件整理、结果汇总或状态转换，只要它们本身没有卡死风险，就应允许它们自然完成。温和关闭强调完整性：不接新活，但旧活做完。它的代价是关闭时间取决于存量任务数量和任务耗时；如果某个任务永久阻塞，线程池也可能无法终止。

调用 `shutdown()` 之后，再调用 `execute`、`submit`、`invokeAll` 等提交方法，任务会被拒绝。对于 `ThreadPoolExecutor`，拒绝行为由当前 `RejectedExecutionHandler` 决定；默认策略通常会抛出 `RejectedExecutionException`。这点很重要：关闭线程池不是让提交方法静默失效，而是让调用方能够发现生命周期已经结束。调用方如果把拒绝异常吞掉，就可能制造“任务好像提交了但实际没有执行”的假象。

`shutdown()` 不会等待任务完成。很多代码写成如下形式，以为方法返回时线程池已经关闭完毕：

```java
executor.shutdown();
releaseSharedResource();
```

这段代码只有在任务不再依赖 `releaseSharedResource()` 释放的资源时才安全。因为 `shutdown()` 返回后，队列中的任务和正在运行的任务仍可能继续访问共享资源。如果关闭线程池和释放资源之间有依赖关系，调用方必须先等待线程池真正终止，或者设计更精确的资源所有权，让任务自己在结束时释放自己持有的资源。

## shutdownNow 的语义

`shutdownNow()` 是强制意味更明显的关闭请求，但它也不是无条件杀死线程。它通常做两件事：第一，尝试停止所有正在执行的任务，常见方式是中断工作线程；第二，停止处理等待队列中的任务，并返回那些尚未开始执行的任务列表。返回列表让调用方有机会记录、补偿、重新调度或明确丢弃这些任务。

`shutdownNow()` 的关键限制在“尝试”二字。Java 标准库没有提供安全地强行终止任意线程的通用机制。线程被中断之后，只有在任务处于可中断阻塞方法中，或者任务主动检查中断状态并退出时，中断才会有效。任务如果在纯 CPU 死循环中不检查中断，或者阻塞在不响应中断的外部操作上，线程池可能仍然无法及时终止。

强制关闭适合几类情况：温和关闭等待超时，需要尽快让程序退出；任务结果已经不再需要，继续执行只会浪费资源；上层生命周期已经结束，再执行存量任务反而可能访问无效资源；系统进入异常状态，需要尽量中断后台工作并保留现场。它不适合被当成常规收尾的第一选择，因为它可能让队列任务不再执行，也可能让运行中任务在中断点提前退出。

`shutdownNow()` 返回的任务列表只包含尚未开始执行的任务，不包含已经开始运行但被中断的任务。正在运行的任务是否完成、是否回滚、是否留下部分结果，需要任务自己的取消和资源释放逻辑负责。如果使用 `submit` 提交任务，返回列表中的对象可能是内部包装后的 `FutureTask`，不一定直接等于原始 `Runnable`。因此，如果需要关闭后精确识别未执行任务，任务对象最好带有明确的业务标识，或在包装层保留可诊断信息。

还要注意，`shutdownNow()` 之后线程池仍不一定立即 `isTerminated()`。中断发出到任务实际退出之间有时间差；有些任务会在捕获 `InterruptedException` 后做清理；有些任务可能暂时持有锁或正在执行 finally。强制关闭之后仍然需要 `awaitTermination` 做有界等待，并在超时后记录告警。

## awaitTermination 的语义

`awaitTermination` 是关闭流程中的等待工具。它等待的不是“队列为空”，也不是“没有新任务”，而是线程池进入终止状态。终止意味着关闭已经发起，并且所有任务完成或停止，工作线程退出。正因为等待条件是终止状态，所以通常要在 `shutdown` 或 `shutdownNow` 之后调用。

这个方法有三个容易被忽略的点。第一，它返回布尔值，调用方必须检查。返回 `false` 不是小问题，而是表示线程池在指定时间内没有关完。正确代码应当至少记录日志、上报告警、尝试升级关闭或把失败传给上层。第二，它会抛出 `InterruptedException`，表示当前等待线程被要求停止等待。当前层如果无法完整处理取消，通常应恢复中断标志。第三，超时时间不是越长越好。过短会误判正常任务为关闭失败，过长会拖慢整体停止过程；应根据任务耗时、服务目标和资源重要性设定。

常见的两阶段关闭会这样使用：

```java
public static void shutdownGracefully(ExecutorService executor,
                                      long timeout,
                                      TimeUnit unit) {
    executor.shutdown();
    try {
        if (!executor.awaitTermination(timeout, unit)) {
            List<Runnable> dropped = executor.shutdownNow();
            // record dropped.size() and other diagnostic information
            if (!executor.awaitTermination(timeout, unit)) {
                // record that the executor did not terminate
            }
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}
```

这段代码表达了几个原则：先温和关闭，给已提交任务完成机会；等待有上限；超时后升级为中断和清空队列；等待线程被中断时也要触发关闭；最后恢复当前线程的中断状态。它不是所有场景的唯一模板，但它比只调用 `shutdown()` 更完整，因为它把正常路径、超时路径和中断路径放在同一个生命周期处理中。

## 拒绝新任务

线程池开始关闭后，拒绝新任务是正确行为。很多人把 `RejectedExecutionException` 只理解为线程池“满了”，其实线程池关闭后继续提交也会触发拒绝。拒绝并不总是错误，它是生命周期边界被触碰的信号。真正的问题是调用方是否理解这个信号，并能决定任务应该放弃、重试、转移到其他执行器，还是把失败返回给上层。

关闭期间仍然提交任务，通常说明系统存在所有权或时序问题。可能是生产者线程没有先停止，仍在源源不断提交；可能是某个回调在任务结束时又派生新任务；可能是关闭顺序错误，先关闭了下游执行器，再关闭上游调度器；也可能是共享线程池被某个模块提前关闭。排查这类问题时，不要只在提交处捕获异常，而要沿着生产链路找到谁在关闭后仍然认为线程池可用。

拒绝策略也会影响关闭语义。默认抛异常能让问题尽快暴露；`CallerRunsPolicy` 在普通饱和场景下可能用于反压，但在线程池关闭后通常不会执行任务；静默丢弃策略会让关闭期间的任务丢失更难发现；自定义拒绝策略可以记录任务类型、提交来源、线程池状态和队列长度。对需要可靠处理的任务，拒绝策略不应只是“不要报错”，而应清楚表达失败是否可接受。

如果组件对外暴露异步提交方法，关闭状态也应体现在组件接口中。比如组件进入 stopping 状态后，提交方法可以直接返回失败结果，而不是继续把任务推给已经关闭的线程池。这样调用方看到的是业务层面明确的“组件已关闭”，而不是底层抛出的 `RejectedExecutionException`。线程池拒绝是底层机制，组件生命周期应提供更清晰的语义。

## 处理中任务与队列任务

关闭线程池时，需要分别看待三类任务：尚未提交的新任务、已经提交但还在队列中的任务、已经被工作线程取出并正在运行的任务。`shutdown()` 拒绝第一类任务，但允许第二类和第三类继续完成。`shutdownNow()` 拒绝第一类任务，通常不再执行第二类任务，并尝试中断第三类任务。

这个分类能解释很多关闭阶段的现象。调用 `shutdown()` 后，日志里仍然看到任务继续执行，并不奇怪，因为队列任务会被继续处理。调用 `shutdownNow()` 后，某些任务仍然打印完成日志，也不奇怪，因为它们可能已经开始运行，并且在中断到达前或清理过程中完成了。调用 `shutdownNow()` 返回了若干 `Runnable`，也不代表这些任务已经“失败处理完毕”，只是说明它们没有开始执行。

如果队列任务具有业务意义，关闭策略必须说明这些任务如何处理。温和关闭表示尽量执行完队列任务；强制关闭表示返回未执行任务，由调用方决定是否记录、重试、持久化或丢弃。如果任务只是缓存刷新、预热、统计汇总等可丢弃工作，强制关闭时记录数量可能足够；如果任务代表必须完成的工作，就不应依赖普通内存队列承载关闭期间的可靠性，而应使用具备持久化、确认和重试语义的设计。

正在运行的任务更复杂。线程池只能通过中断表达取消请求，不能替任务决定一致性边界。一个任务可能已经完成一半：读到了输入，写了一部分输出，更新了局部状态，持有某个资源。任务被中断时，应尽量在明确的安全点退出，而不是在任意位置留下不一致状态。对复杂任务来说，循环边界、批次边界、事务边界、文件写入边界、锁释放边界，都可能是更合理的取消检查点。

## 中断响应是关闭能否成功的关键

Java 中断是一种协作式取消机制。调用 `Thread.interrupt()` 会设置目标线程的中断状态；如果目标线程正在执行某些可中断阻塞方法，例如 `BlockingQueue.take`、`Thread.sleep`、`Object.wait`、`LockSupport.park` 的相关封装、部分锁和同步器的可中断等待方法，这些方法可能抛出 `InterruptedException` 或提前返回。中断不会自动停止线程，也不会自动释放业务资源。

线程池调用 `shutdownNow()` 时，通常会中断工作线程。如果任务代码正确响应中断，关闭会比较顺畅。正确响应包括：捕获 `InterruptedException` 后结束当前任务或把取消信号传给上层；在无法继续抛出时恢复中断状态；在长循环中定期检查 `Thread.currentThread().isInterrupted()`；在 finally 中释放资源；避免捕获异常后继续无条件重试。

下面是一个较好的任务轮廓：

```java
class WorkerTask implements Runnable {
    @Override
    public void run() {
        try {
            while (!Thread.currentThread().isInterrupted()) {
                WorkItem item = nextItem();
                if (item == null) {
                    break;
                }
                process(item);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            cleanup();
        }
    }

    private WorkItem nextItem() throws InterruptedException {
        // for example: return queue.poll(1, TimeUnit.SECONDS);
        return null;
    }

    private void process(WorkItem item) {
        // process one bounded unit of work
    }

    private void cleanup() {
        // release resources owned by this task
    }
}
```

这段代码的重点不是具体业务，而是取消边界：循环会检查中断，阻塞获取工作时允许被中断，捕获中断后恢复状态，资源释放放在 finally 中。相反，最糟糕的写法是捕获 `InterruptedException` 后什么都不做，然后继续 while(true)；这会让关闭线程池的调用方误以为已经发送取消请求，实际任务却继续运行。

中断处理还有一个细节：抛出 `InterruptedException` 的方法在抛出异常前通常会清除当前线程的中断状态。因此，如果当前层不能把异常继续抛给上层，就应该调用 `Thread.currentThread().interrupt()` 恢复标志。恢复标志不是形式主义，而是让更外层的生命周期代码仍然能观察到取消请求。如果每一层都吞掉中断，最终关闭协议就会断裂。

## 两阶段关闭

实践中最常见的关闭策略是两阶段关闭：第一阶段温和关闭，第二阶段超时后强制关闭。它的核心思想是先尊重任务完整性，再保护停止过程的有界性。没有第一阶段，容易过早丢弃还可以正常完成的任务；没有第二阶段，任何一个卡住的任务都可能拖住整个关闭流程。

第一阶段调用 `shutdown()`，表示停止接收新任务并处理存量任务。随后调用 `awaitTermination` 等待一段合理时间。这个等待时间应根据任务类型确定。短任务组成的线程池可以设置较短等待；批处理任务可能需要更长等待；依赖外部资源的任务应结合外部调用超时；关键资源释放前的等待应谨慎设置，避免过早释放仍被任务使用的对象。

第二阶段在等待超时后调用 `shutdownNow()`，尝试中断正在运行的任务并取回队列中未执行的任务。此时应记录足够的诊断信息，例如线程池名称、超时时间、未执行任务数量、队列长度、活动线程数、调用关闭的线程、组件状态等。仅仅记录一句“shutdown timeout”通常不够，因为后续排查需要知道到底是任务太多、任务太慢、任务无视中断，还是上游仍在提交。

两阶段关闭还应处理等待线程被中断的情况。假设某个上层控制线程正在执行关闭流程，却在 `awaitTermination` 中被中断。此时当前层不能简单忽略异常，也不能只打印日志后继续等待。通常做法是立即调用 `shutdownNow()`，尽力推动线程池停止，然后恢复当前线程中断状态，让上层知道关闭流程也被要求取消。恢复中断后是否继续等待，要根据上层协议决定。

两阶段关闭不是越复杂越好。对于由更高层统一管理的共享线程池，局部代码不应擅自调用这套关闭流程；对于一次性局部线程池，最好用 try-finally 保证关闭；对于任务结果必须全部完成的场景，强制关闭后的未执行任务必须有补偿策略；对于可丢弃任务，记录数量和快速退出可能更重要。模板提供的是骨架，场景决定细节。

## 资源释放顺序

线程池关闭常常和资源释放同时发生。资源包括文件句柄、网络连接、数据库连接、缓冲区、锁、临时目录、计数器、监控注册项、回调引用、对象池等。正确顺序取决于资源由谁持有。如果任务仍可能访问某个共享资源，就不能在线程池真正终止前释放该资源；如果资源由任务自己创建并独占，任务应该在自己的 finally 中关闭；如果资源是线程池工作线程的上下文状态，则应在线程复用和线程退出时都考虑清理。

一个常见错误是先关闭依赖资源，再等待线程池结束：

```java
executor.shutdown();
connection.close();
executor.awaitTermination(30, TimeUnit.SECONDS);
```

如果队列里还有任务使用 `connection`，这段代码会让任务在关闭阶段失败。正确顺序通常是先停止新任务来源，再关闭线程池并等待存量任务结束，最后释放存量任务依赖的共享资源。当然，如果任务的目标就是监听某个资源，而关闭资源可以唤醒阻塞操作，则可能需要先关闭资源来打断阻塞，再等待线程池退出。关键不在固定顺序，而在明确资源和任务之间的依赖关系。

资源释放还要考虑异常路径。任务执行失败时是否关闭自己打开的资源，等待关闭超时时是否保留现场，强制关闭返回的未执行任务是否持有需要释放的对象，线程池未能终止时是否还能安全释放共享资源，这些问题都需要提前设计。不能把资源释放完全寄托在线程池关闭上，因为线程池只管理工作线程和任务队列，不知道任务内部打开了什么。

如果使用 `ThreadLocal`，关闭线程池时还要格外小心。线程池会复用工作线程，某个任务放入 `ThreadLocal` 的值如果没有清理，可能污染后续任务。线程池最终关闭时线程会退出，确实能释放线程持有的 ThreadLocalMap，但这不能替代每个任务执行后的清理。正确做法是在任务边界使用 try-finally 调用 `remove()`，让任务之间的隔离不依赖线程池何时终止。

## 服务生命周期中的关闭顺序

在线程池属于某个服务或组件时，关闭流程不应只从线程池本身出发，而应从整个服务生命周期出发。一个典型服务可能包含任务生产者、任务队列、执行线程池、结果处理器、外部资源和监控指标。关闭顺序如果反了，就会出现生产者还在提交而执行器已经关闭，执行器还在处理而资源已经释放，结果处理器已经停止而任务还在回调等问题。

比较稳妥的顺序通常是：先把组件状态切换为 stopping，使对外入口不再接受新的业务请求；再停止内部生产者或调度器，避免继续提交新任务；然后关闭执行线程池，等待已经提交的任务完成或取消；再关闭结果通道和共享资源；最后切换为 terminated 或 failed，并暴露关闭结果。这个顺序的本质是先关入口，再清存量，最后放资源。

组件状态最好显式表达，而不是只依赖线程池状态。例如一个服务可以维护 `NEW`、`RUNNING`、`STOPPING`、`TERMINATED`、`FAILED` 等状态。外部调用提交任务时先检查服务状态，只有 `RUNNING` 才允许提交；`STOPPING` 之后立即返回明确错误。这样可以避免把底层 `RejectedExecutionException` 暴露为唯一生命周期信号，也能避免关闭期间的并发调用产生含糊结果。

如果服务支持重复启动和停止，还要特别注意线程池是否可以复用。`ExecutorService` 一旦关闭，不能重新打开。停止后再次启动通常需要创建新的线程池实例，而不是试图复活旧实例。把已经关闭的线程池缓存在字段中，并在下一次启动时继续使用，会导致所有提交都被拒绝。重启式组件应在状态转换中清楚地区分旧执行器和新执行器，并保证旧执行器完全不再被提交。

## 超时、告警与诊断

关闭等待必须有超时。没有超时的 `awaitTermination` 会把关闭线程变成另一个潜在阻塞点。超时不是为了草率放弃，而是为了让系统在关闭异常时仍有机会输出诊断、触发告警、执行降级或交给上层决策。一个线程池迟迟无法关闭，通常意味着任务耗时超过预期、任务阻塞在外部资源、任务没有响应中断、队列堆积过多，或关闭顺序存在问题。

设置超时时需要结合任务性质。对短任务线程池，可以设置秒级等待；对大批量批处理，可以按批次大小和单任务耗时估计上限；对依赖外部调用的任务，应确保外部调用自身也有超时，否则线程池关闭超时只能发现问题，不能真正解除阻塞；对关键收尾任务，可能需要先停止入口，等待较长时间，再强制关闭并告警。

告警内容应可用于定位，而不是只说明失败。较好的关闭日志可以包含线程池名称、关闭阶段、温和等待时长、强制等待时长、是否被中断、未执行任务数量、活动线程数、已完成任务数、队列长度、最大线程数、任务类型摘要。对 `ThreadPoolExecutor`，这些指标可以通过公开方法获取一部分；对普通 `ExecutorService`，至少可以在封装层记录提交数量、完成数量、失败数量和拒绝数量。

如果线程池关闭超时，线程 dump 往往比单行日志更有价值。线程 dump 可以看到工作线程是 RUNNABLE、BLOCKED、WAITING 还是 TIMED_WAITING，栈顶停在锁竞争、队列等待、文件操作、外部调用还是业务循环。对于不响应中断的任务，线程 dump 能帮助判断是阻塞方法不可中断，还是任务吞掉了中断后继续执行。关闭超时的诊断目标不是立即“杀掉线程”，而是找到取消协议断裂的位置。

## 任务提交方式对关闭的影响

`execute` 和 `submit` 都能把任务交给线程池，但它们在异常观察和取消上有所不同。`execute` 提交的 `Runnable` 如果抛出未捕获的运行时异常，异常通常会到达工作线程的未捕获异常处理流程；`submit` 会把异常捕获到返回的 `Future` 中，调用方只有在 `Future.get()` 时才能看到。关闭线程池时，如果大量任务通过 `submit` 提交但从不读取 Future，任务失败可能被长期隐藏。

`Future.cancel(true)` 和线程池关闭也有关联。对单个任务来说，`cancel(true)` 表示如果任务尚未开始，尽量取消；如果已经运行，尝试中断执行线程。它和 `shutdownNow()` 一样依赖任务响应中断。区别在于 `cancel` 面向单个任务，`shutdownNow` 面向整个执行器。设计关闭策略时，可以先取消一批不再需要的 Future，再关闭线程池；也可以在线程池强制关闭后记录未执行任务。

`invokeAll` 和 `invokeAny` 这类批量提交方法也需要关注超时和关闭。如果调用方在工作线程中等待同一个线程池执行的其他任务，可能造成线程饥饿死锁：所有工作线程都在等待新任务完成，而新任务没有可用线程执行。关闭阶段遇到这种设计，温和关闭往往无法结束，因为任务之间互相等待。正确做法是避免在线程池任务中同步等待同一线程池的子任务，或者使用独立执行器、异步组合、足够的容量和有界等待。

任务提交方式还影响未执行任务的识别。通过 `submit` 提交时，队列中存放的通常是包装后的 `RunnableFuture`；通过 `execute` 提交时，队列中更可能是原始 `Runnable`。如果强制关闭后需要拿回未执行任务并做补偿，封装层应保留任务元数据，不能只依赖 `toString()`。一个实践方式是定义带任务 id、来源和创建时间的任务包装器，在执行、完成、拒绝和关闭返回时都能输出一致的诊断信息。

## 和 try-with-resources 的关系

较新的 Java 版本中，某些执行器相关类型可能提供更贴近资源关闭的用法，但通用原则仍然不变：关闭执行器要理解它的关闭语义、等待语义和任务取消语义。即使某个类型可以放进 try-with-resources，也不代表任务一定在代码块结束时都以业务期望的方式完成。语法糖只能减少忘记关闭的概率，不能替代生命周期设计。

在通用代码中，`ExecutorService` 最常见的安全写法仍然是 try-finally：

```java
ExecutorService executor = Executors.newFixedThreadPool(4);
try {
    submitTasks(executor);
} finally {
    shutdownGracefully(executor, 30, TimeUnit.SECONDS);
}
```

这里的重点是执行器由当前代码创建，生命周期也由当前代码负责。如果执行器作为参数传入，则当前方法通常不应关闭它，除非方法契约明确表示会接管所有权。把传入的共享执行器放进 finally 关闭，是非常危险的写法；调用方可能还需要继续使用它，其他模块也可能共用它。

## 常见误区

第一个误区是认为 `shutdown()` 会立即停止所有任务。事实上它只是停止接收新任务，并让已提交任务继续执行。调用它之后马上释放共享资源，或马上断言所有后台工作已经结束，都是错误的。需要确认任务完成时，应继续使用 `awaitTermination`、`Future.get`、同步器或其他明确等待条件。

第二个误区是认为 `shutdownNow()` 能可靠杀死线程。它只是尝试中断工作线程并返回未开始任务。任务不响应中断，线程就可能继续运行。依赖 `shutdownNow()` 强行修复不可取消任务，通常会把问题推迟到关闭阶段爆发。任务代码必须主动设计中断检查、阻塞超时和 finally 清理。

第三个误区是吞掉 `InterruptedException`。关闭流程中，当前线程被中断往往意味着上层也在取消或停止。如果捕获异常后只打印日志，不恢复中断状态，也不触发强制关闭，就会让调用链失去取消信号。正确做法要么继续抛出异常，要么完成本层必要清理后恢复中断。

第四个误区是关闭不属于自己的线程池。共享执行器、框架提供的执行器、调用方传入的执行器，都可能有独立生命周期。局部代码擅自关闭它，会让其他使用者提交失败。是否关闭一个执行器，应由所有权契约决定，而不是由“我用完了”决定。

第五个误区是用 `sleep` 等待关闭。睡眠只等待时间，不等待条件。睡了 1 秒后任务可能完成，也可能没有完成；机器慢、任务多或外部资源阻塞时，固定睡眠会变得不可靠。关闭线程池要等待 terminated 状态，等待任务结果要使用 Future 或同步器，等待条件变化要使用对应的并发工具。

第六个误区是忽略未执行任务。调用 `shutdownNow()` 返回的任务列表不是无用返回值。它代表还没有开始执行的存量任务。对可丢弃任务，至少应记录数量；对重要任务，应有补偿、重试或持久化策略。忽略这个列表，会让任务丢失变得无声无息。

第七个误区是把关闭写在任务内部。某个任务执行完后顺手关闭提交它的线程池，通常会破坏其他任务或提交者。线程池关闭应由生命周期管理者发起，而不是由普通工作任务自行决定。任务可以报告完成、失败或请求停止，但不应随意终结共享执行器。

第八个误区是没有停止生产者。只关闭执行器，不关闭上游定时器、监听器、生产线程或回调入口，会导致关闭期间不断出现拒绝异常。正确关闭应先停止任务来源，再关闭执行器。否则线程池虽然拒绝新任务，但系统仍在制造无效提交。

## 设计检查清单

写线程池关闭代码时，可以按下面的问题检查。

| 检查点 | 需要回答的问题 |
| --- | --- |
| 所有权 | 这个线程池由当前组件创建吗，当前组件是否有权关闭它？ |
| 入口控制 | 关闭开始后，外部和内部生产者是否都会停止提交新任务？ |
| 存量任务 | 队列中的任务应继续执行、取消、重新调度还是记录丢弃？ |
| 运行任务 | 正在运行的任务是否响应中断，是否有安全退出点？ |
| 等待边界 | 温和关闭等待多久，强制关闭后再等待多久？ |
| 中断处理 | 等待线程被中断时是否调用强制关闭并恢复中断状态？ |
| 资源释放 | 共享资源是否在线程池真正终止后释放，任务自有资源是否在 finally 中释放？ |
| 诊断信息 | 超时、拒绝、未执行任务和任务异常是否可观察？ |
| 重启语义 | 如果组件可重启，是否会创建新的线程池而不是复用已关闭实例？ |

这张表的价值在于把关闭从“调用哪个方法”提升为“生命周期是否闭合”。如果其中任意一项没有答案，关闭逻辑就可能只在正常路径上成立。真正稳定的并发代码，往往不是因为关闭语句多，而是因为入口、任务、资源和等待条件的关系清楚。

## 一个完整的关闭封装

实际项目中，可以把两阶段关闭封装成工具方法，但不要让工具方法掩盖调用方责任。工具方法负责执行通用协议，调用方仍然要决定线程池所有权、超时时间、日志内容和未执行任务处理方式。

```java
public final class ExecutorShutdown {
    private ExecutorShutdown() {
    }

    public static List<Runnable> shutdown(ExecutorService executor,
                                          long quietTimeout,
                                          long forcedTimeout,
                                          TimeUnit unit) {
        executor.shutdown();
        try {
            if (executor.awaitTermination(quietTimeout, unit)) {
                return List.of();
            }

            List<Runnable> notStarted = executor.shutdownNow();
            if (!executor.awaitTermination(forcedTimeout, unit)) {
                // record that some worker threads are still alive
            }
            return notStarted;
        } catch (InterruptedException e) {
            List<Runnable> notStarted = executor.shutdownNow();
            Thread.currentThread().interrupt();
            return notStarted;
        }
    }
}
```

这个封装刻意返回未开始任务列表，而不是在内部吞掉。调用方拿到列表后，可以根据任务性质处理。如果是可丢弃任务，可以记录数量和类型；如果是重要任务，可以转入补偿队列；如果是测试或命令行工具，可以把未执行任务数量作为失败信息的一部分。关闭工具不应假装所有任务都已经成功完成。

如果线程池是 `ThreadPoolExecutor`，还可以在关闭前后读取更多状态用于诊断，例如 `getActiveCount()`、`getQueue().size()`、`getCompletedTaskCount()`、`getTaskCount()`、`getPoolSize()` 等。这些数字在并发环境中是近似观察，不应作为严格业务判断，但足以帮助定位关闭超时的原因。诊断信息越靠近关闭现场，后续排查越容易。

## 小结

正确关闭线程池的核心不是背诵 `shutdown`、`shutdownNow` 和 `awaitTermination` 的名字，而是理解它们组成的生命周期协议。`shutdown()` 停止接收新任务并允许存量任务完成；`shutdownNow()` 尝试中断运行中任务并返回尚未开始的任务；`awaitTermination()` 有界等待线程池真正进入终止状态。三者配合使用，才能同时兼顾任务完整性和关闭有界性。

关闭是否真正可靠，还取决于任务是否响应中断、资源是否在正确边界释放、生产者是否先停止、拒绝新任务是否被正确处理、超时是否有告警和诊断、线程池所有权是否清晰。线程池是执行任务的基础设施，不是替任务承担取消和一致性责任的魔法容器。把关闭协议和任务取消协议一起设计，才是正确关闭线程池的关键。

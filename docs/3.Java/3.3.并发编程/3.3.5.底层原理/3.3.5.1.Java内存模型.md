# 3.3.5.1 Java内存模型

## 定位

Java 内存模型通常简称 JMM，它不是某一种缓存结构，也不是某个虚拟机实现里的内存区域划分，而是 Java 语言为并发程序定义的一组语义规则。它回答的问题可以概括为一句话：当多个线程读写同一份共享变量时，哪些写入必须被哪些读取看到，哪些执行顺序必须被保留，哪些看起来奇怪的结果仍然是规范允许的。只要程序进入多线程共享状态，单靠源码顺序和本机测试结果就不再足以证明正确性，必须回到 JMM 规定的同步关系和可见性规则。

单线程程序的理解方式比较直接。一个线程沿着调用栈执行语句，局部变量、对象字段和数组元素的读写都由同一个执行流观察，编译器和处理器即使做优化，也不能改变这个线程内部可感知的结果。多线程程序不同。线程 A 写入一个字段之后，线程 B 什么时候能看到？线程 A 先写 `data` 再写 `ready`，线程 B 看到 `ready == true` 时是否一定能看到新的 `data`？一个对象构造完成后把引用交给另一个线程，另一个线程是否可能看到对象字段的默认值？这些问题都不能靠“代码看起来先后如此”来回答。

JMM 的价值在于把并发正确性从经验推断变成规范推导。它一方面允许编译器、即时编译器和处理器在不破坏单线程语义的前提下进行优化，例如寄存器缓存、指令重排、公共子表达式消除、写缓冲合并；另一方面要求程序在需要跨线程通信时使用明确的同步动作，例如 `volatile` 读写、锁的释放与获取、线程启动和终止、并发工具内部的同步协议。程序员通过这些同步动作表达“这个写入应该对那个读取可见”“这段临界区不能并发进入”“这个对象已经构造完成，可以被其他线程使用”。规范再根据这些动作判断运行结果是否合法。

本文保持通用 Java 技术视角，只讨论 Java 语言、Java 虚拟机规范和标准并发工具中与内存语义直接相关的内容。这里的“内存模型”不等同于堆、栈、方法区等运行时数据区，也不等同于某个 CPU 的缓存一致性协议。运行时数据区描述 Java 程序在虚拟机内部如何组织内存，CPU 缓存协议描述硬件如何维护不同核心之间的数据一致性，而 JMM 描述 Java 程序在源代码和规范层面可以依赖什么并发语义。三者有关联，但不能混为一谈。

理解 JMM 的目标也不是背诵若干术语，而是建立判断能力。看到一段共享变量读写代码时，要能判断它是否存在数据竞争；看到 `volatile` 时，要能判断它解决的是可见性和特定有序性，而不是所有复合操作的原子性；看到 `synchronized` 或 `Lock` 时，要能判断互斥范围是否覆盖了所有相关状态；看到不可变对象和 `final` 字段时，要能判断构造期间是否发生了引用逃逸；看到线程启动、`join`、`Future.get`、并发容器操作时，要能意识到它们背后也承载着内存可见性约定。

## JMM 解决的问题

JMM 解决的第一类问题是可见性。一个线程对共享变量的写入，不保证会立刻被另一个线程读取到。原因并不只是硬件缓存，也包括编译器优化、寄存器暂存、处理器写缓冲以及运行时对代码的转换。只要程序没有建立同步关系，读线程就可能持续看到旧值，甚至在循环中因为优化而根本不再重新读取共享字段。JMM 并不承诺普通变量写入会在某个时间上限内自然传播给所有线程；它要求程序用同步动作表达传播边界。

第二类问题是有序性。源码中的语句顺序不等于跨线程观察顺序。编译器和处理器可以在不改变当前线程结果的前提下调整执行顺序。例如把两个彼此没有单线程依赖的写入交换，或者把一个普通读提前到条件判断之前。单线程看不到差异，但另一个线程如果同时读取这些变量，就可能观察到与源码顺序不一致的状态。JMM 的任务不是禁止所有重排序，而是定义哪些重排序不能越过哪些同步边界。

第三类问题是原子性。Java 对部分基本读写提供单次操作层面的原子性，例如引用读写和大多数基本类型读写不会读到半个值；但大量日常操作并不是单次读写，而是“读-改-写”的复合动作。`count++` 需要先读取旧值，再计算新值，再写回字段。多个线程同时执行时，即使每一次读和写本身都是原子的，整个递增过程也可能交错，导致更新丢失。JMM 会定义哪些操作天然原子，哪些操作需要锁、原子类或其他同步工具保护。

第四类问题是优化边界。现代 Java 程序运行在复杂的优化链路上。源码会被编译为字节码，字节码由解释器或即时编译器执行，热点代码可能被内联、消除、展开、重排，最终还要落到具体处理器的流水线和缓存层次。没有统一内存模型，Java 程序的并发行为就会过度依赖平台细节：某段代码可能在一种处理器上看似正确，在另一种处理器或另一种虚拟机优化级别下失败。JMM 提供的是跨平台语义底线，让符合规范的实现可以优化，也让正确同步的程序有可移植的行为。

第五类问题是正确性判断。很多并发错误不是每次都复现。某个数据竞争可能在本地运行一万次都没有暴露，但在更高并发、不同负载、不同处理器或不同编译阶段下出现。JMM 提供了推理方法：如果两个冲突访问之间没有 happens-before 关系，就不能要求读取方一定看到某个写入；如果程序没有数据竞争，就可以按更接近顺序一致性的方式理解；如果依赖普通变量进行跨线程通信，就必须承认结果可能超出直觉。它把“偶现”背后的不确定性具体化。

## 主内存与工作内存抽象

JMM 使用“主内存”和“工作内存”的抽象描述线程与共享变量之间的关系。主内存可以理解为所有线程共享变量的共同来源，工作内存可以理解为每个线程在执行过程中使用的私有副本或临时观察位置。线程不能直接操作主内存中的变量，而是把变量从主内存读入自己的工作内存，在工作内存中使用和修改，再在某些时机把修改写回主内存。这个抽象不是要求虚拟机真的按这种结构实现，而是用来解释为什么一个线程的修改不一定马上被另一个线程看到。

共享变量包括实例字段、静态字段和数组元素。它们位于对象或类的状态中，可能被多个线程通过引用访问。局部变量、方法参数和异常处理参数属于线程私有的栈帧内容，本身不会被其他线程直接读写，因此通常不是 JMM 中共享变量讨论的对象。需要注意的是，局部变量可以保存一个共享对象的引用。引用变量本身是线程私有的，但它指向的对象字段仍然可能被多个线程共享访问。很多误解来自把“引用在局部变量里”误认为“对象状态也线程私有”。

工作内存抽象可以帮助解释普通变量通信的不可靠性。线程 A 修改一个普通字段后，这个新值可能暂时只存在于它自己的执行上下文中；线程 B 读取同一字段时，可能仍然使用自己先前读取过的旧值。即使硬件最终会维护某种缓存一致性，JMM 也不会把“最终可能传播”提升为 Java 程序可以依赖的同步语义。没有同步动作时，读线程看到新值、旧值或某些交错状态，都可能是规范允许的结果。

同步动作会影响主内存与工作内存之间的关系。`volatile` 写要求把写入以特定方式发布出去，后续对同一 `volatile` 变量的读能够观察到这个发布，并且受到相应的有序性约束。锁释放会把临界区内对共享变量的写入发布给随后获取同一把锁的线程。线程启动会让启动线程在调用 `start` 之前的写入对新线程可见。线程终止与 `join`、`isAlive` 等观察动作之间也存在可见性关系。JMM 不是要求程序员手工刷新缓存，而是要求使用这些语言级或类库级同步边界。

把主内存和工作内存理解为“真实内存和线程缓存”是有帮助的直觉，但不能走得太远。真实实现可能使用寄存器、CPU cache、写缓冲、内存屏障、编译器约束和运行时协议组合达成 JMM 语义，并不需要存在一个可直接观察的“工作内存对象”。同样，主内存也不是只指堆中的某个物理位置。它是规范层面描述共享变量一致性的抽象。学习 JMM 时，应该把这个抽象用于推理可见性，而不是把它当成具体内存布局图。

## 可见性、有序性与原子性

可见性表示一个线程对共享变量的写入能否被另一个线程可靠观察到。可靠二字很重要：不是“某次运行看到了”，而是“根据规范必须能看到”。例如线程 A 在持有某把锁时写入 `value`，然后释放这把锁；线程 B 随后获取同一把锁并读取 `value`。这里锁释放 happens-before 随后的锁获取，所以 B 必须看到 A 在释放锁前对共享状态所做的写入，至少不能因为缓存或重排序无视这段同步关系。相反，如果 A 和 B 对 `value` 的访问没有共同锁、没有 `volatile`、没有线程启动终止等其他同步关系，那么 B 是否看到新值就不能由规范保证。

有序性表示程序中一组操作之间的先后约束在跨线程观察时是否成立。单线程内存在程序次序规则：一个线程中的每个动作 happens-before 后续动作。但这条规则只保证同一线程内部的语义推导，不意味着其他线程在缺少同步时也能按同样顺序观察所有普通变量。真正用于跨线程有序通信的是同步动作建立的 happens-before。比如写线程先写普通字段 `data`，再写 `volatile` 字段 `ready`；读线程先读到 `ready == true`，再读普通字段 `data`。这里 `volatile` 写和读之间建立了关系，写 `ready` 之前的普通写入也会通过传递性对读线程可见。

原子性表示一个操作是否不可分割。对一个 `int` 字段的单次读取或写入通常是原子的，线程不会读到半个 `int`。但 `int` 自增不是原子的，因为它包含读取旧值、计算新值和写回新值。`volatile int count` 也不能让 `count++` 变成原子操作；`volatile` 只能保证每次读写看到的是符合可见性规则的值，不能阻止两个线程同时读到相同旧值后分别写回相同新值。需要复合原子性时，应使用锁保护整个复合操作，或者使用 `AtomicInteger`、`LongAdder` 等专门工具。

三者经常一起出现，但不能相互替代。可见性解决“能不能看到写入”，有序性解决“能不能按同步约束理解前后关系”，原子性解决“操作会不会被并发交错破坏”。锁通常同时提供互斥、可见性和有序性，因为同一时刻只有一个线程进入临界区，并且 unlock 与后续 lock 之间有 happens-before。`volatile` 提供可见性和一定有序性，但不提供复合操作互斥。原子类提供单变量或特定结构上的原子更新，并通过内部的 volatile 语义和 CAS 操作提供相应可见性，但也不自动维护多个变量之间的不变式。

下面的示例展示了三者的差异：

```java
final class Counter {
    private volatile int value;

    int get() {
        return value;
    }

    void increment() {
        value++;
    }
}
```

`get` 每次读取的是 `volatile` 值，读取动作本身具备可见性语义；但 `increment` 仍然不是线程安全的复合递增。两个线程可能同时读取到 `10`，分别计算 `11`，再分别写回 `11`，最终只增加一次。把字段声明为 `volatile` 并没有改变 `value++` 的复合结构。如果业务要求递增不丢失，就应把递增放入同一把锁保护的临界区，或者改用原子类的 `incrementAndGet`。

## 重排序与 as-if-serial

重排序是理解 JMM 时最容易被误解的概念。它不是说虚拟机可以任意打乱代码，也不是说程序会随机执行，而是说在不改变单线程可观察结果的前提下，编译器、运行时和处理器可以调整某些操作的执行顺序。这个原则通常称为 as-if-serial：无论怎么优化，单线程执行结果必须像源码顺序执行一样。问题在于，多线程程序会让另一个线程观察中间状态，于是那些单线程无差异的调整可能变成并发语义差异。

例如，一个线程执行两次普通写入：

```java
data = 42;
ready = true;
```

在当前线程看来，只要后续自己读取时结果不变，这两个写入之间未必存在不可调整的依赖。另一个线程如果没有同步就循环读取 `ready`，在看到 `true` 后读取 `data`，并不能保证一定得到 `42`。原因可能是写入顺序被优化调整，也可能是 `ready` 的新值先对读线程可见，而 `data` 的新值尚未可见。JMM 不要求开发者区分具体是哪一种底层原因；只要缺少同步关系，就不能要求这种发布协议可靠。

重排序包括编译器层面的重排序和处理器层面的重排序。编译器可能把不影响单线程结果的读写移动到更合适的位置，处理器可能因为流水线、乱序执行、写缓冲和缓存协议而让不同核心观察到不同顺序。JMM 位于语言和实现之间，规定 Java 层面的同步动作必须产生足够的约束。实现可以通过内存屏障、禁止特定编译优化、插入特殊指令或利用平台已有保证来满足这些约束，但这些细节不是 Java 程序可移植推理的基础。

同步动作的本质之一就是限制重排序越界。`volatile` 写之前的普通读写不能被重排到这个 `volatile` 写之后，`volatile` 读之后的普通读写不能被重排到这个 `volatile` 读之前；锁释放和锁获取也会形成类似的发布与获取边界。它们不是禁止所有优化，而是在需要跨线程通信的位置画出边界，让边界前后的操作具有可推导的关系。正确的并发代码不是要求运行时停止优化，而是用同步机制告诉运行时哪些顺序不能破坏。

重排序也解释了双重检查锁定在早期写法中的问题。对象创建看似是一句 `instance = new Singleton()`，但概念上包含分配内存、初始化对象、把引用赋给变量等步骤。如果引用赋值对其他线程可见早于对象初始化完成，并且读取线程没有同步约束，就可能拿到一个非空但尚未完全初始化的对象。现代写法通常需要把实例字段声明为 `volatile`，或者使用静态内部类、枚举、类初始化等具备安全发布语义的方式。

## happens-before 是核心推理工具

happens-before 是 JMM 中判断可见性和有序性的核心关系。它不是简单的时间先后，而是一种规范定义的先行发生关系。如果动作 A happens-before 动作 B，那么 A 的结果对 B 可见，并且 A 的执行顺序在内存语义上排在 B 之前。反过来，如果两个动作之间没有 happens-before，不能仅凭时间上 A 似乎先发生就要求 B 看到 A 的写入。

最基础的规则是程序次序规则。在同一个线程中，按照程序顺序，前面的动作 happens-before 后面的动作。这使得单线程内部可以按源码顺序推导。但这条规则本身不能跨线程传播，除非它和其他同步规则连接起来。例如线程 A 中普通写 `data` 在 `volatile` 写 `ready` 之前，根据程序次序规则，写 `data` happens-before 写 `ready`；如果线程 B 的 `volatile` 读看到了 `ready` 的写入，根据 volatile 规则，写 `ready` happens-before 读 `ready`；再根据程序次序规则，读 `ready` happens-before 后续读 `data`。通过传递性，写 `data` happens-before 读 `data`，于是读线程能看到写线程发布的数据。

锁规则规定，对一个监视器锁的 unlock happens-before 后续对同一个监视器锁的 lock。这里必须是同一把锁。两个线程分别锁住不同对象，即使对象名字相似、代码位置相近，也不会建立锁规则。`synchronized` 方法锁住的是接收者对象或类对象，`synchronized` 代码块锁住的是括号中的对象。判断锁语义时，关键不是有没有写 `synchronized`，而是所有相关读写是否受同一把锁保护。

volatile 规则规定，对一个 `volatile` 变量的写 happens-before 后续对这个变量的读。这里同样要求是同一个 `volatile` 变量。`volatile` 常用于状态标志、一次性发布和轻量级配置刷新。写线程在写 volatile 标志之前完成普通字段写入，读线程在读到标志后读取普通字段，可以借助 happens-before 的传递性观察到发布前的状态。这个模式成立的前提是读线程确实通过同一个 volatile 变量观察到了发布动作。

线程启动规则规定，线程 A 调用线程 B 的 `start` 之前的动作 happens-before 线程 B 中的任何动作。这意味着在启动线程前准备好的参数、字段和对象状态，对新线程可见。线程终止规则规定，线程中的所有动作 happens-before 其他线程检测到该线程已经终止，例如从 `join` 成功返回或观察到 `isAlive` 为 `false`。因此，通过 `join` 等待工作线程结束后读取其写入结果，具备明确的可见性基础。

中断、默认初始化和对象终结也有相关规则。一个线程调用另一个线程的 `interrupt` happens-before 被中断线程检测到中断事件。对象的默认初始化 happens-before 任何其他动作，这保证对象字段至少有默认值基础。对于普通业务代码，最常用的仍然是程序次序、锁、volatile、线程启动、线程终止和传递性。掌握这些规则，已经可以推导绝大多数 Java 并发读写场景。

happens-before 的传递性非常重要。如果 A happens-before B，B happens-before C，那么 A happens-before C。很多同步工具就是利用传递性把普通变量写入“带过”同步边界。并发容器、阻塞队列、`Future`、`CountDownLatch` 等工具内部会使用 volatile、锁或其他同步原语，调用者通过它们的方法建立更高层的 happens-before。例如一个线程把元素放入阻塞队列，另一个线程取出该元素，队列实现会保证放入前对元素状态的正确发布语义，而调用者不需要手写内存屏障。

## 数据竞争与正确同步

当两个线程访问同一个共享变量，至少一个访问是写，并且这两个访问之间没有 happens-before 关系时，就存在数据竞争。数据竞争不是“结果一定错误”的同义词，而是“程序已经离开可靠推理区域”的信号。带有数据竞争的程序可能在某些运行中得到期望结果，也可能在另一些运行中看到旧值、丢失更新、观察到不一致状态，甚至因为优化导致循环无法退出。JMM 对这类程序保留了较大的实现空间。

正确同步的程序更容易推理。如果一个程序中所有顺序一致性冲突访问都通过 happens-before 排好序，通常可以把它理解为每个线程按某种交错顺序执行，且每次读取看到符合该顺序的写入。这个性质让并发程序回到人能理解的模型：虽然线程交错仍然复杂，但至少不会出现无同步读写带来的奇异可见性问题。正确同步不意味着没有业务 bug，但它移除了内存模型层面的不确定性。

数据竞争常见于状态标志、缓存字段、懒加载、统计计数和可变集合。比如一个线程写 `stopped = true`，另一个线程在 `while (!stopped)` 中循环，如果 `stopped` 是普通字段，就存在数据竞争。读线程可能一直使用旧值，循环无法按预期结束。把 `stopped` 声明为 `volatile` 可以解决这个单标志可见性问题。若停止过程还涉及多个字段、资源释放和状态迁移，仅靠一个 volatile 标志可能不够，需要更完整的同步协议。

可变集合的数据竞争尤其隐蔽。一个线程向普通 `ArrayList` 添加元素，另一个线程同时遍历或读取，不仅可能看不到新元素，还可能破坏集合内部结构，导致越界、死循环或异常。即使用 `Collections.synchronizedList` 或并发容器，也要区分单个方法调用的线程安全和复合业务操作的线程安全。检查后再插入、遍历期间依赖稳定快照、跨多个集合维护一致性，这些都可能需要额外锁或更高层设计。

避免数据竞争的基本方法有三类。第一，消除共享：把状态限制在线程内部，或者通过不可变对象传递结果。第二，使用互斥：用同一把锁保护所有相关读写，并把不变式维护在临界区内。第三，使用安全发布和无锁同步：用 `volatile`、原子类、并发容器或同步器表达特定通信模式。选择哪一种取决于状态复杂度。如果只是一个停止标志，volatile 足够直接；如果要同时维护多个字段之间的一致性，锁往往更清晰；如果是高频计数，原子类或累加器可能更合适。

## volatile 的语义与边界

`volatile` 是 JMM 中最常用也最容易被过度理解的关键字。它的第一层语义是可见性：对 volatile 变量的写会立即以规范要求的方式发布，后续线程读取同一 volatile 变量时能看到最近的符合规则的写入。它的第二层语义是有序性：volatile 写具有释放语义，volatile 读具有获取语义，编译器和处理器不能把相关普通读写随意移动到这些边界之外。它的第三层语义是单次读写原子性：对 volatile 变量本身的读写是一个独立动作，不会被缓存成线程私有的长期旧值。

典型用法是停止标志：

```java
final class Worker implements Runnable {
    private volatile boolean stopped;

    void stop() {
        stopped = true;
    }

    @Override
    public void run() {
        while (!stopped) {
            doWorkOnce();
        }
    }

    private void doWorkOnce() {
        // 执行一小段可中止的工作
    }
}
```

这里 `stopped` 只有一个写端语义：请求工作线程停止；读端只需要及时观察标志，不需要和其他字段组成复杂不变式。`volatile` 很适合这种模式。如果 `doWorkOnce` 内部可能长时间阻塞，volatile 本身不能唤醒阻塞线程，还需要中断、关闭资源或其他协作机制。这说明可见性和线程调度不是同一件事：看得见停止请求，不代表线程马上有机会检查它。

另一个典型用法是一次性发布：

```java
final class Holder {
    private int value;
    private volatile boolean ready;

    void publish(int newValue) {
        value = newValue;
        ready = true;
    }

    int readIfReady() {
        if (!ready) {
            return -1;
        }
        return value;
    }
}
```

写线程先写普通字段 `value`，再写 volatile 字段 `ready`。读线程先读取 `ready`，如果读到 `true`，再读取 `value`。根据 happens-before 和传递性，读线程能够看到发布前写入的 `value`。这个模式要求发布动作和观察动作围绕同一个 volatile 变量展开。如果读线程绕过 `ready` 直接读 `value`，或者写线程在写 `ready` 之后继续无同步地修改 `value`，语义就会改变。

volatile 的边界同样重要。它不能保证复合操作原子，不能自动维护多个字段之间的一致性，不能替代锁的条件等待，不能让阻塞调用退出，不能让一个普通集合变成线程安全集合，也不能保证“读到某个状态后直到使用期间状态不变”。如果读取 volatile 状态后需要基于它执行一组必须和其他线程互斥的操作，就应使用锁或原子更新协议。把 volatile 当成轻量级锁，是并发代码中很常见的误区。

volatile 也不是越多越好。滥用 volatile 会让状态协议分散，读者难以判断哪些字段一起构成不变式。对于多个相关字段，单独给每个字段加 volatile 可能只解决了单字段可见性，却没有解决跨字段一致性。比如 `lower` 和 `upper` 两个边界字段分别声明为 volatile，仍然不能保证另一个线程不会观察到 `lower > upper` 的中间状态。需要整体一致性时，用锁把检查和更新放在同一个临界区通常更容易推理。

## 锁的内存语义

锁在 JMM 中不仅提供互斥，也提供内存可见性。进入 `synchronized` 代码块或方法时，线程获取监视器锁；退出时释放监视器锁。对同一个监视器锁的释放 happens-before 后续对同一个锁的获取。因此，一个线程在临界区内对共享变量所做的写入，在它释放锁后，对随后获得同一把锁的线程可见。锁的这层发布-获取语义，是很多线程安全类能够可靠工作的基础。

下面的代码展示了同一把锁保护不变式：

```java
final class Range {
    private int lower;
    private int upper;

    synchronized void setLower(int value) {
        if (value > upper) {
            throw new IllegalArgumentException();
        }
        lower = value;
    }

    synchronized void setUpper(int value) {
        if (value < lower) {
            throw new IllegalArgumentException();
        }
        upper = value;
    }

    synchronized boolean contains(int value) {
        return value >= lower && value <= upper;
    }
}
```

这里 `lower` 和 `upper` 的所有读写都通过同一个对象监视器保护。锁不仅避免两个更新同时进入，还保证一个线程更新后的字段值对后续进入临界区的线程可见。更关键的是，检查和写入在同一个临界区内完成，不会出现一个线程检查通过后、另一个线程插入修改、再导致不变式被破坏的交错。

`ReentrantLock` 等显式锁也有类似内存语义。成功的 `unlock` happens-before 后续成功的 `lock`，前提仍然是同一个锁对象。显式锁提供可中断获取、定时获取、公平策略和多个条件队列等能力，但内存模型层面的核心仍然是释放与获取之间的关系。使用显式锁时，必须在 `finally` 中释放锁，否则异常路径会让其他线程永久等待。

锁的常见错误是锁对象不一致。一个方法使用 `synchronized (this)`，另一个方法使用 `synchronized (lock)`，第三个方法直接读字段，三者之间不会形成完整保护。还有一种错误是只给写操作加锁，读操作不加锁。没有同步的读操作可能看不到写线程在锁内发布的结果，因为锁规则需要读线程也获取同一把锁才能建立 happens-before。对共享状态的访问协议必须覆盖所有读写，而不是只覆盖看起来危险的写。

锁的另一个边界是临界区范围。临界区太小，可能没有覆盖完整不变式；临界区太大，可能把慢 I/O、回调或长时间计算放进锁内，导致竞争扩大甚至死锁。JMM 只保证正确使用同一锁时的可见性和互斥，不保证锁粒度天然合理。设计锁时要先明确被保护的状态和不变式，再决定哪些语句必须在锁内，哪些语句应该移出锁外。

## final 字段与不可变对象

`final` 字段在 JMM 中有特殊语义。如果一个对象构造过程没有让 `this` 逃逸，并且构造函数正常结束，那么其他线程在获得该对象引用后，对它的 final 字段可以看到构造函数中写入的值。这种保证比普通字段更强，支持不可变对象在多线程之间安全共享。不可变对象之所以适合并发，不只是因为没有后续修改，也因为 final 字段语义降低了发布时的可见性风险。

例如：

```java
final class Point {
    private final int x;
    private final int y;

    Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    int x() {
        return x;
    }

    int y() {
        return y;
    }
}
```

只要构造期间没有把 `this` 存入全局变量、注册到回调、启动线程或传给外部可见对象，`Point` 的 final 字段对其他线程具备可靠初始化语义。这个类没有 setter，也不暴露可变内部状态，因此对象一旦构造完成，就可以在多个线程之间共享，而不需要每次读取都加锁。

final 语义也有边界。第一，final 只保护字段引用本身或基本值，不自动保护引用指向对象的内部可变状态。`private final List<String> list` 只能保证 `list` 这个引用在构造后可见，不能保证 `ArrayList` 内部结构线程安全。若要构造真正不可变对象，需要复制输入、禁止后续修改，并避免暴露内部可变对象。第二，final 不能补救构造期间的 this 逃逸。如果构造函数中把当前对象注册给另一个线程，另一个线程可能在构造完成前观察对象状态，final 保证会被破坏。

构造期间 this 逃逸的形式很多。直接写 `Global.instance = this` 是明显逃逸；在构造函数中启动线程，线程的 `run` 方法访问当前对象，也是逃逸；把当前对象传给一个可能保存引用或立即回调的方法，同样可能逃逸；在构造函数中注册监听器、发布事件或把内部类实例交给外部，也可能让 `this` 暴露。安全做法是让构造函数只完成内部初始化，等对象完全构造后，再由外部工厂或初始化方法完成注册和启动。

不可变对象是减少 JMM 复杂度的强力手段。一个状态如果能设计成构造后不变，就不需要为每次读取设计同步协议。多个线程可以共享同一个值对象，或者通过 volatile 引用发布新的不可变快照。比如配置对象可以设计为不可变类，更新时构造新对象，再用 volatile 引用替换；读线程每次读取引用后使用该快照，不需要关心对象内部字段是否正在变化。这里 volatile 负责发布“引用替换”，final 和不可变设计负责保证“快照内部一致”。

## 安全发布

安全发布指的是：一个对象构造完成后，以一种能让其他线程可靠看到其初始化状态的方式，把对象引用交给其他线程。对象引用本身被其他线程拿到，并不等于对象内部状态已经对它可见。没有安全发布时，另一个线程可能看到对象字段的默认值、旧值或不一致状态，即使构造函数在源码上已经写完。

常见的安全发布方式包括：在静态初始化阶段发布对象；通过 volatile 字段发布引用；通过锁保护发布和读取；通过线程安全容器、阻塞队列、并发工具传递对象；在线程启动前完成对象准备并作为新线程可达状态；通过 final 字段构造不可变对象并避免 this 逃逸。它们背后都能找到 happens-before 关系或 final 字段特殊语义。

静态初始化是经常被低估的安全发布方式。类初始化由虚拟机保证串行化，一个线程完成类初始化后，其他线程使用该类时能看到初始化结果。因此，使用静态 final 字段保存不可变对象，或者使用静态内部类实现延迟初始化，通常能获得清晰可靠的发布语义。这种方式适合生命周期与类一致、初始化过程不依赖复杂外部状态的对象。

volatile 引用发布适合“整体替换”模式：

```java
final class ConfigCenter {
    private volatile Config current = new Config(10, "default");

    Config current() {
        return current;
    }

    void update(int limit, String mode) {
        current = new Config(limit, mode);
    }

    static final class Config {
        final int limit;
        final String mode;

        Config(int limit, String mode) {
            this.limit = limit;
            this.mode = mode;
        }
    }
}
```

`Config` 是不可变对象，`current` 是 volatile 引用。写线程构造新配置后用 volatile 写发布，读线程通过 volatile 读获得某个完整快照。读线程可能拿到旧快照或新快照，但不会看到一个正在被修改的半成品配置。这个模式比多个 volatile 字段分别更新更容易维护一致性。

锁保护发布适合对象后续仍会被修改的场景。发布引用和读取引用都必须在同一把锁内完成，并且对象的后续访问也应遵守同一协议。如果只在发布时加锁，发布后所有线程无锁访问内部可变字段，安全发布只能保证初始状态可见，不能保证后续并发修改安全。安全发布解决的是“对象如何第一次被看见”，线程安全解决的是“对象被看见之后如何被并发访问”。两者有关联，但不是同一个问题。

不安全发布最常见的形式是普通静态字段、普通实例字段或普通集合中保存对象引用，然后其他线程无同步读取。代码在低压力下可能长期正常，因为对象构造和引用赋值通常很快；但规范上并没有要求读线程一定看到完整初始化状态。对于生命周期长、共享范围广的对象，发布方式必须明确写进设计，而不能依赖“构造函数已经执行完”这种单线程直觉。

## 双重检查锁定与延迟初始化

延迟初始化是 JMM 讨论中的典型场景，因为它同时牵涉可见性、重排序、原子性和安全发布。最简单但可靠的方式是把整个访问方法声明为 synchronized：

```java
final class LazyService {
    private Service service;

    synchronized Service get() {
        if (service == null) {
            service = new Service();
        }
        return service;
    }
}
```

这段代码正确，因为所有读写都在同一把锁内。缺点是初始化完成后，每次读取仍然要进入同步方法。对于大多数程序，这个成本未必是问题；但在极高频访问场景中，开发者可能想减少锁竞争，于是出现双重检查锁定。

正确的双重检查锁定需要 volatile：

```java
final class LazyService {
    private volatile Service service;

    Service get() {
        Service result = service;
        if (result == null) {
            synchronized (this) {
                result = service;
                if (result == null) {
                    result = new Service();
                    service = result;
                }
            }
        }
        return result;
    }
}
```

外层读取避免初始化完成后的加锁，内层同步保证只有一个线程创建对象，volatile 字段保证引用发布不会被重排序到构造完成之前，并且后续读线程能看到完整初始化的对象。局部变量 `result` 不是语义必需，但可以减少 volatile 读取次数，也让返回值和检查值保持一致。

如果去掉 volatile，双重检查锁定就可能出错。一个线程在锁内创建对象并把引用赋给字段，另一个线程在外层看到字段非空后直接返回。由于外层读取没有获取锁，无法和写线程的 unlock 建立 happens-before；由于字段不是 volatile，也没有发布-获取语义。读线程可能看到尚未完全初始化的对象，或者看不到构造中写入的普通字段。这个问题不是“某个虚拟机实现慢一点”的问题，而是缺少规范保证。

还有一些替代方式更简单。静态内部类利用类初始化的线程安全语义实现延迟加载；枚举适合单例；显式在程序启动阶段完成初始化则避免延迟复杂度。选择哪种方式应先看需求：是否真的需要延迟，是否需要处理初始化异常，是否需要支持重载，是否需要传入参数。为了省掉一次锁而引入复杂发布协议，未必划算。

## 标准并发工具中的内存语义

很多开发者以为 JMM 只和 `volatile`、`synchronized` 有关，其实标准并发工具也依赖并暴露内存语义。理解这些语义，可以避免重复发明低层同步协议。`java.util.concurrent` 中的队列、锁、同步器、任务结果和原子类，通常都在文档中定义了类似“某个动作之前的操作 happens-before 另一个线程中对应动作之后的操作”的关系。

线程启动和任务提交之间要区分。直接调用 `Thread.start` 时，启动前的写入对新线程可见。使用 `ExecutorService` 提交任务时，提交任务之前的动作通常通过执行器内部队列、锁和 volatile 状态安全传递给工作线程。任务本身的结果再通过 `Future.get`、`CompletableFuture` 完成动作或其他同步机制发布给等待方。调用者不需要知道线程池内部用什么队列和锁，但需要通过正确的 API 读取结果，而不是无同步地窥探任务对象内部字段。

阻塞队列的内存语义也很实用。一个线程把元素放入队列之前对该元素所做的写入，对另一个线程从队列取出该元素之后的操作可见。这个规则让生产者-消费者模型可以安全传递可变对象的已初始化状态。需要注意的是，队列只保证交接时刻的发布语义。如果生产者在放入队列后继续无同步修改同一个对象，消费者同时读取，仍然可能产生数据竞争。传递对象之后所有权是否转移，必须由业务协议规定清楚。

`CountDownLatch`、`CyclicBarrier`、`Semaphore`、`Phaser` 等同步器也承载可见性。一个线程在 `countDown` 之前完成的动作，对另一个线程从对应 `await` 成功返回后的动作可见。释放许可之前的动作与获取许可之后的动作之间，也会有实现定义的同步效果。调用者使用这些工具时，不只是获得阻塞和唤醒能力，也是在建立内存发布边界。

原子类提供的是更细粒度的无锁更新。`AtomicInteger`、`AtomicReference` 等类的方法具有 volatile 读写语义，并通过 CAS 或类似机制完成条件更新。它们适合单变量状态、引用替换、计数器和简单状态机。对于多个变量之间的不变式，可以把多个字段封装成一个不可变状态对象，然后用 `AtomicReference` 原子替换整个状态；也可以退回锁。不要把若干原子变量拼在一起就假设整体操作原子。

## 顺序一致性与 JMM 的现实模型

顺序一致性是一种更强、更容易理解的并发模型：所有操作看起来按照某个全局顺序执行，并且每个线程内部操作顺序与程序顺序一致。如果 Java 对所有普通读写都强制顺序一致，程序员会更容易推理，但编译器和处理器优化空间会大幅下降，许多平台上的实现成本也会更高。JMM 选择的是折中：对正确同步的程序提供接近顺序一致性的推理体验，对未同步的数据竞争程序不提供过度保证。

这就是为什么“没有同步也许能运行”不是可靠设计。未同步程序可能在某些硬件上表现得像顺序一致，尤其是在强内存模型平台、低优化级别、低并发压力或偶然调度顺序下。但 Java 程序的正确性不能建立在这些环境特征上。只要规范允许不同结果，未来的虚拟机版本、即时编译阶段、处理器架构或代码微小变动都可能改变表现。

JMM 还有一个重要目标：保证类型安全和基本内存安全。即使程序存在数据竞争，也不应该因为读到“凭空出现”的引用而破坏 Java 的类型系统。规范对 out-of-thin-air 值等问题有复杂约束，普通开发者通常不需要直接推导这些极端情形。实践中更重要的是不要依赖数据竞争程序的奇怪结果，而是通过 happens-before 建立清晰同步关系。

顺序一致性思维仍然有价值，但要放在正确前提下。对于没有共享可变状态的程序，可以按单线程顺序理解；对于所有共享访问都正确同步的程序，可以按线程交错和同步边界推理；对于存在数据竞争的程序，顺序一致性直觉就可能失效。判断一段并发代码时，第一步不是猜测实际执行顺序，而是找出所有共享变量访问，再检查冲突访问之间是否存在 happens-before。

## 常见场景推导

第一个场景是普通标志位停止线程。写线程执行 `running = false`，工作线程执行 `while (running) {}`。如果 `running` 是普通字段，两个线程对它的访问存在数据竞争。工作线程可能持续读取旧值，也可能因为循环体没有可观察副作用而被优化成更难退出的形式。把 `running` 声明为 volatile 后，写线程的停止请求对读线程可见。若循环体包含阻塞调用，还要配合中断或其他唤醒机制。

第二个场景是结果发布。工作线程计算出 `result` 后写 `done = true`，主线程看到 `done` 后读取 `result`。如果两个字段都是普通字段，主线程看到 `done == true` 时不一定看到最新 `result`。把 `done` 声明为 volatile，并要求工作线程先写 `result` 再写 `done`，主线程先读 `done` 再读 `result`，就能通过 volatile 和传递性建立发布关系。另一种更高层做法是使用 `Future.get`，让任务结果和异常通过标准同步工具传递。

第三个场景是缓存懒加载。多个线程调用 `getValue`，如果缓存为空就计算并保存。如果计算过程昂贵且结果必须只计算一次，普通字段加空判断不够，因为多个线程可能同时看到空值并重复计算，也可能看到未安全发布的结果。可以使用 synchronized 包住检查和赋值，或者使用正确的 volatile 双重检查，或者使用并发 Map 的 `computeIfAbsent`。不同方式的差异在于是否允许重复计算、是否需要异常缓存、是否需要移除或刷新。

第四个场景是多字段状态。对象有 `state`、`version`、`message` 三个字段，写线程需要同时更新，读线程需要看到彼此匹配的一组值。如果三个字段分别是 volatile，读线程仍然可能看到新 `state` 搭配旧 `message` 的组合，因为每个字段的 volatile 读写只保证单字段顺序，不保证一次读取跨多个字段形成快照。可以用锁保护整组字段，也可以把三者封装为不可变对象，用一个 volatile 或 AtomicReference 替换整个快照。

第五个场景是对象发布后继续修改。生产者构造一个对象，放入阻塞队列；消费者取出对象并处理。放入队列之前的初始化对消费者可见，这是安全的。但如果生产者放入队列后继续修改对象字段，消费者同时读取，就又产生了新的并发访问。队列解决的是交接点的发布，不解决交接后的所有权协议。常见做法是传递不可变对象，或规定放入队列后生产者不再访问，或在对象内部使用同步保护可变状态。

## 边界和误区

第一个误区是把 JMM 当成内存区域图。堆、栈、方法区、程序计数器等是运行时数据区；主内存和工作内存是内存模型抽象；CPU cache 和寄存器是硬件实现细节。它们可以互相帮助理解，但不能互相替代。说“变量在堆上所以线程可见”是不完整的。对象字段位于共享对象中，只表示其他线程有机会通过引用访问它，不表示写入对其他线程立即可见。

第二个误区是把 `volatile` 当成锁。volatile 适合状态标志、发布引用、单变量可见性和特定顺序约束，不适合保护复杂临界区。`volatile List` 只保证列表引用本身的读写可见，不保证列表内部结构线程安全；`volatile int` 不保证自增不丢失；多个 volatile 字段不保证组合不变式。需要互斥时就使用锁，需要条件等待时就使用条件队列或同步器，需要单变量原子更新时使用原子类。

第三个误区是只在写方法加锁。锁规则要求释放和获取同一把锁才能建立 happens-before。如果写线程加锁，读线程不加锁，读线程不一定看到锁内写入，也可能观察到不完整状态。线程安全类的设计通常要求所有访问共享可变状态的方法都遵守同一协议，包括读方法、写方法、迭代方法、回调触发方法和返回内部对象的方法。

第四个误区是相信时间顺序。`Thread.sleep`、日志先后、调试器单步、低并发测试、机器性能差异，都不是同步关系。线程 A 先打印日志，线程 B 后打印日志，不代表 B 能看到 A 在打印前的所有普通写入。sleep 一秒后再读字段，也不建立 happens-before。时间可以用于等待外部条件或降低轮询频率，但不能替代锁、volatile、join、Future、Latch 等同步机制。

第五个误区是忽略构造期间 this 逃逸。很多对象看起来字段都是 final，却在构造函数中注册监听、启动线程或调用可覆写方法，导致外部代码可能在构造完成前访问当前对象。final 字段的初始化安全依赖构造过程没有逃逸。写不可变类时，构造函数应尽量只做字段校验、复制和赋值，不执行会把当前对象交给外部世界的动作。

第六个误区是把线程安全和安全发布混为一谈。一个对象可以被安全发布，但发布后仍然不是线程安全的；一个线程安全对象如果通过不安全方式发布，其他线程第一次看到它时仍可能有初始化可见性风险，尽管许多标准并发类通过自身机制降低了这种风险。严谨的设计要同时回答两个问题：引用如何发布？发布后状态如何访问？

第七个误区是用单次压力测试证明没有问题。数据竞争的表现依赖调度、优化、硬件、负载和时机。测试可以帮助暴露问题，但很难穷尽所有交错。并发代码的正确性最终要靠规范推理：共享变量在哪里，读写是否冲突，冲突访问之间的 happens-before 从哪里来，不变式由谁保护，发布边界在哪里。测试结果只能作为辅助证据，不能替代同步设计。

## 实践中的设计原则

优先减少共享可变状态。最简单的 JMM 问题是没有 JMM 问题：如果状态只在线程内部使用，或者对象构造后不可变，跨线程通信就会少很多。任务参数可以复制成不可变对象，结果可以通过 Future 返回，阶段之间可以用队列传递消息，配置可以用不可变快照整体替换。越少共享，越少需要在低层内存语义上推理。

为每个共享状态指定唯一访问协议。一个字段要么只在同一把锁下访问，要么通过 volatile 发布和读取，要么通过原子类更新，要么封装在线程安全容器或同步器内部。不要让同一个字段有时加锁、有时不加锁，有时通过 volatile 标志推导、有时直接读写。访问协议一旦分裂，happens-before 关系就会出现缺口。

把不变式放在同步边界内。并发正确性通常不是单个字段正确，而是一组字段之间的关系正确。余额不能小于零、上下界不能交叉、状态和版本必须匹配、队列和计数必须一致，这些都是不变式。只保护单个字段读写不够，检查、修改和发布必须形成一个不可分割的逻辑单元。锁适合表达复杂不变式，原子引用适合表达不可变快照状态机，多个分散 volatile 字段通常不适合表达强一致不变式。

用高层工具表达意图。生产消费用阻塞队列，任务结果用 Future 或 CompletableFuture，阶段等待用 CountDownLatch 或 Phaser，资源限流用 Semaphore，读多写少的快照用不可变对象加 volatile 引用，高竞争计数用 LongAdder 或原子类。高层工具不仅减少样板代码，也把唤醒、排队、异常传播和内存可见性封装在经过验证的实现中。低层 `wait/notify`、手写自旋和裸共享字段应当谨慎使用。

明确生命周期和失败路径。JMM 解决的是内存语义，但真实并发程序还要处理启动、停止、取消、异常和资源释放。一个 volatile 停止标志如果没有被工作线程定期检查，就不能及时停止；一个安全发布的任务如果异常被后台线程吞掉，调用方仍然不知道失败；一个锁保护的状态如果异常路径没有释放锁，会造成永久阻塞。内存模型正确只是基础，还要把线程生命周期设计完整。

在代码审查时按问题清单检查。这个字段是否共享？是否可变？所有读写是否有同一同步协议？发布引用的动作是否安全？构造期间是否逃逸？volatile 是否被用来做复合操作？锁对象是否一致？读方法是否也参与同步？多个字段是否需要快照一致性？等待是否依赖明确条件而不是时间？这些问题比单纯问“有没有加 synchronized”更接近 JMM 的本质。

## 排查 JMM 相关问题

排查可见性问题时，先找共享变量和发布路径。典型现象包括停止标志偶尔不生效、配置更新后部分线程仍使用旧值、懒加载对象偶尔为空或字段异常、统计值丢失、队列交接后对象状态不一致。不要先假设是虚拟机 bug，应先检查写端和读端之间有没有 happens-before。普通字段写入加普通字段读取，哪怕时间间隔很长，也不构成同步。

排查有序性问题时，关注“先写 A 再写 B，读到 B 后认为 A 必然完成”的模式。如果 B 不是 volatile、锁释放、同步器完成等发布动作，这个推理通常不成立。即使 B 是 volatile，也要确认读线程确实先读到了同一个 volatile 变量，再读取 A；还要确认写线程没有在发布之后继续无同步修改 A。很多发布协议出错，不是因为没有任何同步，而是同步边界放错位置。

排查原子性问题时，寻找复合操作。自增、自减、检查后写入、先查缓存再填充、读取大小后按索引访问、两个字段一起更新、跨集合迁移元素，都不是简单读写。volatile 和安全发布无法自动保护这些复合过程。需要确认复合过程是否在同一锁内，是否使用了原子类的单个条件更新，或者是否能接受并发交错导致的重复计算和最终一致。

工具上，线程 dump 可以帮助定位线程是否阻塞在锁、条件队列、join、Future 或 I/O 上；日志可以记录发布动作、状态转换和取消路径；并发测试可以放大竞态窗口；代码审查和静态分析可以发现未同步字段访问。需要注意的是，JMM 错误未必表现为线程阻塞，也可能表现为读到旧值或不一致值。线程 dump 看不到“某个字段为什么不可见”，但能帮助排除锁竞争、死锁和等待路径问题。

复现偶发问题时，可以增加循环次数、并发线程数、运行时间和不同运行参数，但不要把复现不了当成正确。对于疑似数据竞争的代码，最可靠的修复路径是建立同步关系，而不是继续寻找能稳定复现的调度。修复后也要重新推导：新的同步动作是否覆盖所有读写？是否引入死锁或锁顺序问题？是否改变性能和生命周期边界？

## 小结

Java 内存模型是 Java 并发正确性的语义基础。它不描述某个具体缓存如何工作，而是规定多线程读写共享变量时，哪些可见性、有序性和原子性保证可以被程序依赖。主内存和工作内存是推理抽象，happens-before 是核心工具，volatile、锁、final 字段、安全发布、线程启动终止和并发工具都是建立同步关系的常见方式。

掌握 JMM 的关键，是把“看起来先发生”“本机测试能通过”“字段在堆上所以共享”“加了 volatile 所以线程安全”这些直觉，替换为规范化问题：共享变量是什么，是否存在冲突访问，冲突访问之间是否有 happens-before，复合操作是否具备原子性，对象引用是否安全发布，不变式是否被同一同步边界保护。只要这些问题能被清楚回答，并发代码就可以从偶然正确走向可解释、可维护和可移植的正确。

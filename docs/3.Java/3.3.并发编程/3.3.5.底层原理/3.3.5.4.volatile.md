# 3.3.5.4 volatile

## 定位
volatile是 Java 并发编程中的一个独立主题，它解决的问题是：提供变量可见性和有序性约束。学习这个主题时，不能只记一个类名或一个方法调用，而要把它放回并发程序的完整链路中理解：任务从哪里产生，在哪个线程中执行，是否访问共享状态，遇到等待和取消时如何响应，结果或异常又如何回到调用方。只有这些问题都清楚，代码才不仅是“能跑”，而是在压力、异常和关闭阶段仍然可解释。

阅读 Java 并发文档时要把三个层次分开：语言层面的语义、类库层面的工具、运行时和操作系统层面的调度。语言语义回答“哪些结果是被规范允许的”，类库工具回答“怎样表达等待、通知、排队、取消和结果传递”，运行时调度回答“什么时候真正获得 CPU、什么时候被阻塞、什么时候被唤醒”。很多并发错误来自把这三层混在一起，例如用时间先后推导可见性，用线程优先级推导执行顺序，或者用一个看起来没有报错的测试结果推导程序在高并发下也正确。

从知识体系上看，volatile既连接基础线程模型，也连接后续的线程安全、线程池、同步器和内存模型。它的学习目标不是背诵 API，而是建立判断能力：什么时候该使用它，什么时候应该换成更高层或更低层工具，什么时候一个看似简单的写法会破坏可见性、原子性或生命周期。本文保持通用 Java 技术视角，只讨论 Java 语言和标准并发库中的机制、边界与实践。

## 解决的问题
volatile面向的直接问题是提供变量可见性和有序性约束。在单线程程序中，控制流沿着调用栈顺序前进，数据读写通常由同一个线程完成；进入并发环境后，执行顺序、等待位置、异常传播和状态可见性都可能分离。这个主题的价值就在于给这种分离建立清晰规则，让开发者知道哪些行为由规范保证，哪些只是当前机器上的偶然结果。

volatile 适合状态信号和发布引用，不适合多步骤更新。它是轻量同步，不是低成本锁。

更具体地说，volatile会影响三个方面。第一是执行关系：某段代码由哪个线程执行，是否可能与其他代码并行，是否存在先后约束。第二是状态关系：共享变量在多个线程之间是否有一致的观察方式，复合操作是否会被打断，更新是否能被其他线程及时看见。第三是生命周期关系：任务开始、等待、取消、失败、完成和资源释放是否有明确路径。并发代码一旦缺少这些关系，就容易出现测试环境偶尔通过、生产压力下随机失败的问题。

## 核心 API 与术语
围绕volatile，最常见的 API 或术语包括：volatile 读写、内存屏障、禁止特定重排序、状态标志、安全发布。这些名称背后不是孤立功能点，而是一组协作约定。读源码或写业务代码时，应优先识别它们承担的是执行、排队、互斥、条件等待、结果传递、取消还是可见性职责。职责判断清楚后，API 的选择通常会变得直接。

写入发布状态、读取观察状态、配合不可变对象引用。这条流程描述了volatile在典型使用中的状态变化。流程中的每一步都可能失败或被中断，因此真实代码不能只写“理想路径”。例如等待可能超时，提交可能被拒绝，锁可能长期竞争，任务可能抛异常，关闭可能遇到不响应中断的工作线程。把失败路径提前纳入设计，是并发代码和普通顺序代码的重要差异。

下面的片段只用于说明核心用法，重点是观察调用位置和资源边界，而不是把它当成完整框架：

```java
private volatile boolean stopped;
```

这类代码最容易被忽略的是边界条件。调用前对象是否已经初始化完成，调用后是否需要等待或关闭，异常是否会被吞掉，当前线程被中断时是否应继续执行，都会影响最终正确性。写并发代码时，每个 API 调用都应回答“失败时怎么办”和“谁负责收尾”。

## 核心机制
volatile的核心机制可以从“状态、队列、唤醒、可见性”四个词理解。状态描述对象当前处于哪个阶段，队列描述竞争者或等待者如何排列，唤醒描述等待线程重新获得执行机会的条件，可见性描述一个线程的写入何时能被另一个线程可靠观察到。不是每个主题都会显式包含四个结构，但成熟的并发工具通常都能在内部找到这些要素。

以volatile为例，状态变化必须是可解释的。线程从未启动到可运行再到终止，锁从未持有到被某个线程持有再到释放，任务从已提交到执行中再到完成或取消，同步器从计数未满足到条件满足，这些变化都需要明确触发点。如果状态触发点不明确，调用者就会被迫用 sleep、轮询或经验时间等待，而这正是并发程序脆弱的常见来源。

队列和唤醒决定等待成本。阻塞等待可以节省 CPU，但涉及线程挂起和恢复；自旋等待可以避免上下文切换，但会消耗处理器时间；公平排队能降低饥饿概率，但可能牺牲吞吐；非公平竞争能提升局部效率，但等待时间更难预测。理解volatile时，不应只问“会不会等待”，还要问“等待者在哪里排队、由谁唤醒、唤醒后是否立即满足条件”。

可见性则决定结果是否可信。一个线程完成了写入，不代表另一个线程一定能立刻看到；一个线程按代码顺序先写 A 再写 B，也不代表另一个线程没有同步就能按同样顺序观察。Java 通过 synchronized、volatile、Lock、线程启动和终止、并发容器等同步动作建立 happens-before 关系。volatile如果涉及共享状态，就必须说明它依赖哪一种同步关系。

## 使用方式
使用volatile时，第一步是识别任务性质。CPU 密集型任务、I/O 等待型任务、短小状态更新、长时间阻塞任务、阶段同步任务、资源限流任务，适合的工具不同。把所有需求都压到同一个 API 上，会让代码表面简单、实际边界混乱。例如需要结果时不应只用裸 Runnable，需要容量控制时不应只靠线程数量，需要条件协作时不应靠固定 sleep。

第二步是明确所有权。谁创建线程或同步器，谁提交任务，谁负责关闭，谁处理异常，谁记录拒绝，谁保证共享对象不被并发破坏，这些都属于所有权问题。所有权不清时，常见后果是线程池被局部代码关闭、共享锁被不同模块以不同顺序获取、ThreadLocal 被上一个任务污染、异常在后台线程中静默消失。

第三步是把正常路径和异常路径写在一起。只写成功路径的并发代码往往在演示中很好看，但在真实系统中会出现资源泄漏。锁释放要放在 finally，许可归还要放在 finally，计数器递减要放在 finally，ThreadLocal 清理要放在 finally，线程池关闭要处理 InterruptedException，等待结果要考虑 TimeoutException。并发代码的可靠性主要体现在这些收尾位置。

第四步是保留可替换空间。许多并发工具有相似目标但边界不同：wait/notify 可以表达条件等待，Condition 能拆分多个等待队列；CountDownLatch 适合一次性等待，CyclicBarrier 适合固定参与者循环阶段，Phaser 适合动态参与者；Atomic 类适合单变量乐观更新，Lock 适合复杂临界区。写文档和写代码都应说明为什么选择当前工具，而不是只说明它“可以用”。

## 状态流转
volatile的状态流转可以概括为：初始化阶段建立对象和依赖，运行阶段执行或等待，竞争阶段处理共享资源冲突，完成阶段发布结果或释放等待者，关闭阶段拒绝新工作并清理资源。这一描述虽然抽象，但适用于大多数 Java 并发工具。每个阶段都要避免“半初始化对象被发布”“等待条件丢失”“结果未安全发布”“关闭后仍提交任务”等问题。

初始化阶段最重要的是安全发布。线程启动前传入的不可变参数、构造完成后再发布的对象、通过 volatile 或锁保护发布的引用，都是可靠方式。反过来，如果 this 在构造过程中逃逸，或者共享集合在没有同步的情况下被多个线程看到，就可能让其他线程观察到未完成状态。volatile如果涉及对象共享，初始化边界必须清晰。

运行阶段最重要的是不要阻塞错位置。持锁执行慢 I/O、在线程池工作线程中等待同一个线程池的任务、在 ForkJoin 任务中长时间等待外部资源、在条件等待中忘记循环检查，都会制造隐蔽问题。等待不是错误，但等待必须有条件、超时、取消和诊断手段。

完成阶段最重要的是结果传播。任务返回值、异常、计数归零、状态标志变更、锁释放或 volatile 写入，都是向其他线程传播进展的方式。传播动作必须与读取动作配对：Future.get 读取任务结果，await 等待计数归零，lock 获取看到上一次 unlock 前的写入，volatile 读观察 volatile 写发布的状态。只有写端和读端使用同一套规则，状态流转才可靠。

## 线程安全边界
volatile 不提供复合操作互斥

线程安全边界必须落到共享状态上。只要一个对象、集合、计数器、状态标志或外部资源会被多个线程同时访问，就要明确谁负责保护它、保护范围有多大、读写之间是否需要原子性、写入后是否需要被其他线程及时看见。没有共享状态的任务通常只需要关注取消和资源释放；有共享状态的任务则必须同时关注互斥、可见性、有序性和生命周期。

对volatile来说，最小线程安全边界通常不是“某个类线程安全”这么粗，而是“哪些方法可以并发调用、哪些参数不能被调用方继续修改、哪些返回值不能暴露内部可变对象、哪些回调可能在持锁时发生”。同一个类中，有的方法可能线程安全，有的方法可能要求外部同步；同一个对象中，有的字段不可变，有的字段必须受锁保护。文档和代码注释应尽量表达这种精确边界。

还要区分结构安全和业务安全。并发容器可以保证自身结构不被破坏，但不能保证跨多个操作的业务不变式；AtomicInteger 可以保证单次递增原子，但不能保证“检查余额再扣减再记录日志”整体一致；线程池可以保证任务被工作线程执行，但不能保证任务内部访问的对象安全。volatile如果被误认为“用了就线程安全”，往往会掩盖更上层的不变式问题。

## 内存可见性与有序性
Java 并发中，正确性不仅是“没有两个线程同时写”。一个线程写入后另一个线程能否看到，另一个线程看到多个写入时是否能按预期顺序理解，同样关键。volatile只要牵涉共享变量，就要考虑 happens-before。synchronized 的释放和随后获取、volatile 写和随后读、线程 start 前后的关系、线程终止和 join 的关系、并发工具内部的同步动作，都是常见可见性来源。

有序性问题常被低估。编译器和处理器可以在不改变单线程结果的前提下重排序指令；缓存也可能让一个核心暂时看不到另一个核心的写入。JMM 允许这些优化，同时要求程序通过同步动作表达跨线程约束。因此，不能因为一段代码“按顺序写着”就认为另一个线程也会按顺序看到。对volatile的使用如果缺少同步边界，就可能在压力下暴露出旧值、默认值或部分构造对象。

可见性也不等于原子性。volatile 能让状态标志及时可见，但不能让自增、复合检查或多字段更新自动变成一个不可分割操作；锁能同时提供互斥和可见性，但锁的对象必须一致；CAS 能保证单个位置的比较更新原子，但不能自动维护多个变量之间的不变式。选择工具时，应先判断需要的是可见性、原子性、有序性，还是三者都需要。

## volatile 在 Java 内存模型中的准确定位
volatile 不是一个“让变量直接存到主内存”的简单开关，也不是一个“比锁轻很多所以可以替代锁”的通用方案。更准确地说，它是 Java 内存模型提供的一种特殊同步变量语义：对 volatile 变量的写入属于同步动作，对 volatile 变量的读取也属于同步动作；所有线程看到 volatile 读写时，要遵守 Java 内存模型对同步顺序和 happens-before 的规定。这个定位非常重要，因为它把 volatile 从硬件缓存细节中抽离出来，落在 Java 语言规范可以保证的行为上。

Java 程序运行时，编译器、即时编译器和处理器都可能为了性能做优化。只要不改变单线程语义，普通读写之间就可能被调整执行顺序，普通字段的值也可能暂时保存在寄存器、处理器缓存或编译器分析得到的局部假设中。单线程看不出问题，因为当前线程总能按照自己的执行路径得到合理结果；多线程就不同了，一个线程的普通写入如果没有通过同步关系发布给另一个线程，另一个线程是否能看到、何时能看到、能否同时看到相关字段的更新，都不能靠代码文本顺序推断。

volatile 的价值就在于提供一条轻量但明确的同步边界。写线程在写 volatile 变量之前做过的普通写入，在读线程读到这个 volatile 写入之后，应当对读线程可见；写线程不能把 volatile 写之前的普通写重排序到 volatile 写之后，读线程也不能把 volatile 读之后的普通读重排序到 volatile 读之前。这样，volatile 变量本身常常被用作“发布点”或“观察点”：写端先准备状态，再写 volatile；读端先读 volatile，再使用被发布的状态。

但是，这个语义不应该被理解为“volatile 变量每次都强制从某个抽象主内存读取，其他字段也自动跟着刷新”。Java 内存模型描述的是跨线程可观察行为，不要求开发者依赖具体处理器缓存实现。不同硬件平台、不同虚拟机实现、不同即时编译阶段可能采用不同屏障指令、缓存一致性协议和优化策略。对 Java 开发者来说，可靠的判断方式不是猜测缓存行如何移动，而是问：读写两端是否围绕同一个 volatile 变量建立了 happens-before？需要保护的数据是否在这条 happens-before 关系覆盖的范围内？是否还存在需要互斥或复合原子性的操作？

也要注意，volatile 的可见性是与同步关系配套的，不是“写入发生后所有线程立刻停止当前工作并刷新视图”。如果读线程长时间不读取这个 volatile 变量，它当然无法通过该变量感知更新；如果读线程读取的是另一个普通字段，也不能自动借到这个 volatile 字段的语义；如果写端只更新了普通字段，没有执行约定好的 volatile 写，读端也没有可靠的观察点。因此，volatile 适合表达状态标志、版本号、不可变快照引用、单写多读发布等清晰模式，不适合在一堆普通字段旁边随意加一个 volatile 期待“整体线程安全”。

## 可见性：从写入可见到状态可信
可见性讨论的是一个线程对共享变量的写入，另一个线程在什么条件下能够可靠观察到。volatile 对可见性的保证可以概括为：对同一个 volatile 变量的写入 happens-before 后续每一次对该变量的读取。这里的“后续”不是指墙上时钟时间，也不是指日志打印先后，而是指 Java 内存模型中的同步顺序。所有 volatile 读写都参与一个全局的同步顺序，读线程看到某次 volatile 写入之后，写线程在该 volatile 写之前已经完成的写入，也会通过 happens-before 传递给读线程。

以停止标志为例，工作线程在循环中读取 `volatile boolean stopped`，控制线程调用 `stopped = true`。如果工作线程随后读到了 `true`，它就必须能看到控制线程在写 `stopped` 之前通过同一条线程执行路径发布的状态。这个例子看起来很小，但它说明了 volatile 的基本使用姿势：读写两端必须访问同一个 volatile 字段，写端用它发布变化，读端用它观察变化。只把字段声明成 volatile 却从不在关键路径读取它，或者写端和读端围绕不同字段组织逻辑，都不能得到想要的结果。

可见性还要求区分“看到变量的新值”和“状态整体可信”。如果一个对象中只有一个 `volatile int state`，而业务不变式只依赖这个状态值，那么读到新状态通常就足够了。可是如果业务不变式由多个普通字段共同组成，例如 `min`、`max`、`version`、`payload` 必须彼此匹配，仅靠把其中一个字段声明成 volatile 并不能让多字段更新变成整体事务。正确做法通常是把相关字段封装进不可变对象，构造完成后通过一个 volatile 引用一次性发布；或者使用锁保护整个更新过程；或者使用原子引用配合不可变快照做 CAS 替换。

可见性也不是越多越好。把大量字段都声明成 volatile，往往说明数据模型缺少清晰所有权。每个 volatile 字段都是一个同步变量，每次读写都限制部分优化并引入内存顺序成本。更严重的是，多个 volatile 字段之间仍然没有自动形成复合不变式。线程 A 先写 `volatile a` 再写 `volatile b`，线程 B 分别读取 `b` 和 `a`，确实会受到 volatile 同步顺序约束，但这不等于 B 读到的两个值一定来自同一次业务更新。如果业务需要“一组字段同进同退”，应优先用不可变聚合对象、锁或更明确的状态机，而不是把字段逐个 volatile 化。

实际写代码时，可以用三个问题检查可见性是否成立。第一，写端是否在所有需要发布的普通写入之后执行了 volatile 写？第二，读端是否在使用这些普通数据之前读取了同一个 volatile 变量？第三，这些普通数据在发布之后是否还会被无同步地继续修改？前两个问题决定 happens-before 是否建立，第三个问题决定发布出去的状态是否还会被写端或其他线程破坏。很多“volatile 明明写了还是不安全”的案例，根源不在 volatile 失效，而是发布之后继续修改同一个可变对象。

## 有序性与禁止重排序
有序性讨论的是操作之间的先后约束。普通 Java 代码的文本顺序主要约束单线程可观察结果，并不自然约束其他线程看到的顺序。编译器可能把互不依赖的普通读写调整顺序，处理器也可能让不同内存操作以不同顺序对其他核心可见。Java 内存模型允许这些优化，但要求同步动作提供跨线程可观察的边界。volatile 就是其中一种边界。

对 volatile 写来说，它具有近似“释放”的语义：在这次 volatile 写之前的普通读写，不能被重排序到 volatile 写之后。写端可以先构造对象、填充普通字段、计算派生数据，最后写 volatile 标志或 volatile 引用。这个 volatile 写像一个发布动作，告诉其他线程：在我发布之前准备好的状态，现在可以按照内存模型规则被观察。对 volatile 读来说，它具有近似“获取”的语义：在这次 volatile 读之后的普通读写，不能被重排序到 volatile 读之前。读端先读取标志或引用，再读取被发布的数据，才能保证读取动作没有越过观察点。

禁止重排序并不表示 volatile 前后的所有代码都不能优化，也不表示虚拟机会把每条语句严格按源代码顺序逐条执行。它约束的是会破坏 volatile 语义的那部分重排序。比如写端不能把对普通字段的初始化移动到发布引用之后，因为那会让读线程拿到引用却看到默认值；读端不能先读取被发布对象的字段，再读取 volatile 引用或标志，因为那会绕过观察点。只要不破坏这些跨线程语义，编译器和处理器仍然可以做许多局部优化。

双重检查锁定是理解有序性的典型例子。懒加载单例常见写法是先检查引用是否为空，进入 synchronized 后再次检查并创建对象。如果保存实例的字段不是 volatile，那么创建对象可以在概念上拆成分配内存、执行构造逻辑、把引用赋给字段几个步骤；在缺少合适约束时，另一个线程可能看到非空引用，却观察到尚未完成初始化的对象。把实例引用声明为 volatile 后，对引用的写入成为发布动作，对引用的读取成为观察动作，构造阶段的写入不能越过发布边界，读线程也不能绕过读取边界使用对象。

不过，双重检查锁定也说明了 volatile 的适用边界。volatile 只负责安全发布引用和约束初始化相关重排序，它不负责让对象后续的可变方法线程安全。如果单例对象构造完成后内部状态仍会被多个线程修改，那么这些修改仍需要各自的同步策略。很多文章把“volatile 修复双重检查锁定”讲成“volatile 让对象线程安全”，这是错误的。安全发布只解决对象从不可见到可见的交接问题，不解决对象生命周期内所有并发访问问题。

## happens-before：volatile 语义的推理方式
happens-before 是 Java 并发中最重要的推理工具之一。它不是简单的执行时间先后，而是内存可见性和有序性的规范关系。如果 A happens-before B，那么 A 的结果必须对 B 可见，并且 A 的执行顺序在内存模型意义上排在 B 之前。volatile 的规则是：对一个 volatile 字段的写入 happens-before 后续每个对该字段的读取。这个规则可以与程序顺序规则、传递性、线程启动和终止规则、锁规则等组合，推出更长的可见性链条。

例如，线程 T1 先执行普通写 `data = value`，再执行 volatile 写 `ready = true`。根据程序顺序规则，`data = value` happens-before `ready = true`；根据 volatile 规则，`ready = true` happens-before T2 后续读到 `ready` 为 true；根据传递性，`data = value` happens-before T2 在读到 ready 之后执行的操作。因此，T2 在看到 ready 为 true 后读取 data，应该能看到 T1 发布的 value。这个推理过程比“volatile 会刷新缓存”更精确，因为它说明了哪个写、哪个读、哪条变量、哪条传递链共同构成保证。

相反，如果 T1 写 `data = value` 后没有写 volatile，T2 只是普通读取 data，那么缺少同步边界；如果 T1 写的是 volatile `ready`，T2 却从不读取 ready 而直接读 data，也缺少观察点；如果 T2 先读取 data 再读取 ready，那么读取 data 的动作没有被放在 volatile 读之后，推理链也不成立。happens-before 的价值就在于把“看起来应该可以”的直觉拆成可验证的规则。

还要理解 volatile 规则只针对同一个 volatile 字段建立直接关系。写 `volatile a` 并不会直接 happens-before 读 `volatile b`。当然，所有 volatile 操作在同步顺序中有全局排序，并且程序顺序与传递性可能让某些复杂关系成立，但实际设计中不应依赖难以阅读的交叉推理。可靠的代码应让发布变量和观察变量保持一致：用 `ready` 发布，就用 `ready` 观察；用 `configRef` 发布配置快照，就用 `configRef` 获取快照；用 `state` 表达生命周期，就围绕 `state` 设计状态转换。

happens-before 还提醒我们不要把日志、调试器显示、sleep、线程优先级当同步关系。日志先打印不等于写入对另一个线程可见；sleep 结束不等于其他线程已经完成某个动作；调试器单步可能改变时序；线程优先级也不提供内存语义。并发程序必须通过语言和类库承认的同步动作建立推理链。volatile 的好处是语义小而明确，坏处也是语义小而明确：只要需求超过这条推理链，它就不够用。

## 为什么 volatile 不能保证复合操作原子性
原子性讨论的是一个操作是否不可分割。volatile 读写本身具有单次访问的原子可见语义：读一个 volatile int 不会读到撕裂值，写一个 volatile 引用不会只写入一半。但是，复合操作不是单次读写。`count++` 看似一行，实际包含读取当前值、计算新值、写回新值。即使 count 是 volatile，多个线程也可能同时读取到相同旧值，各自计算相同新值，再先后写回，导致更新丢失。

这个问题不能通过“volatile 更及时可见”解决。假设 count 初始为 0，线程 A 读到 0，线程 B 也读到 0；A 计算 1 并写回，B 也计算 1 并写回。两个 volatile 写都可以被其他线程看到，写入顺序也可以被同步顺序排序，但最终值仍然是 1，而不是 2。volatile 保证的是每次读写围绕内存模型的可见性和有序性，不保证“读取旧值、基于旧值计算、写入新值”这组三步之间没有其他线程插入。

复合操作不仅包括自增，还包括“先检查再执行”“先查再改”“多字段同时更新”“读一个集合再根据结果写另一个集合”等模式。比如 `if (!initialized) { init(); initialized = true; }` 即使 initialized 是 volatile，也可能多个线程同时观察到 false 并重复执行 init，除非 init 本身幂等且重复执行被设计允许。再比如用 volatile 保存余额，然后执行“余额足够才扣减”，也不能保证两个线程不会同时通过检查并同时扣减。只要业务要求检查和修改之间不可被打断，就需要 synchronized、Lock、Atomic 类的 CAS 循环，或者把状态转换封装进更高层并发工具。

Atomic 类和 volatile 的关系也容易混淆。许多 Atomic 类内部会使用 volatile 字段保存值，并通过 CAS 或 VarHandle 等机制完成原子更新。也就是说，Atomic 通常在 volatile 可见性之上增加了条件更新的原子性。`AtomicInteger.incrementAndGet()` 不只是“volatile 自增的封装”，它会反复尝试把观察到的旧值更新为新值，只有当期间没有其他线程成功改变该值时才提交。失败时重新读取并重试，因此不会像 volatile 自增那样丢失更新。

但是 Atomic 也不是万能锁。单个 AtomicInteger 可以保证一个整数的原子更新，AtomicReference 可以保证一个引用的原子替换，但多个原子变量之间仍然可能缺少整体不变式。若业务状态由多个字段共同组成，常见做法是把它们合并为不可变状态对象，用 AtomicReference 一次性替换；或者用 synchronized 保护整个临界区；或者使用专门的数据结构。选择依据不是“哪个更高级”，而是要保护的不变式到底落在一个内存位置、一个对象快照，还是一段跨多个对象的业务流程上。

## 发布模式：用 volatile 发布不可变快照
volatile 很适合做发布，前提是被发布的数据在发布后不会被无同步修改。最常见的形式是 volatile 引用指向一个不可变对象。写线程先构造一个完整的新对象，把所有字段填好，并确保这些字段不会再改变；然后把该对象引用写入 volatile 字段。读线程读取这个 volatile 引用后，就可以使用对象内部状态。由于构造和填充发生在 volatile 写之前，读线程读取引用发生在 volatile 读之后，happens-before 链条可以把对象内部状态安全传递过去。

这种模式常用于配置快照、路由表快照、策略对象、缓存索引、只读数据视图等场景。它的优点是读路径不需要加锁，只需一次 volatile 读；写路径通过新建对象替换旧对象，避免读线程看到半更新状态。它的代价是每次更新需要构造新快照，快照对象应尽量避免暴露可变集合或可变数组。如果不可避免要包含集合，应在发布前复制并包装为不可修改视图，或者在设计上约定集合只读且不再被持有者修改。

发布引用时还要警惕“浅不可变”。一个对象的字段声明为 final 或不提供 setter，并不自动意味着对象图整体不可变。如果字段引用的是可变集合、可变数组或其他可变对象，而这些对象在发布后仍被写线程修改，读线程就可能看到变化中的内部状态。volatile 只能安全发布引用和发布前已经完成的写入，不能冻结整个对象图。真正可靠的快照要么深度不可变，要么内部所有可变部分也有独立同步策略。

发布模式还需要考虑版本一致性。假设有两个 volatile 引用分别保存 `rules` 和 `metadata`，写线程更新 rules 后再更新 metadata，读线程分别读取两个引用，就可能拿到新 rules 和旧 metadata 的组合。即使每个引用本身都安全发布，组合状态仍然不一致。更稳妥的方式是定义一个包含 rules 和 metadata 的不可变 `Snapshot`，通过单个 volatile `snapshot` 字段发布。这样读线程每次只读取一个引用，拿到的是同一版本的整体状态。

在读多写少的场景里，volatile 快照模式通常比锁更简单，性能也更可预测。读线程不会阻塞写线程，写线程也不会阻塞读线程；旧快照可以继续被已经拿到引用的读线程使用，新读线程则看到新快照。不过它不适合写入频繁、快照构造昂贵、对象图巨大且复制成本高的场景。此时需要考虑读写锁、分段数据结构、增量更新、并发容器或其他并发设计，而不是强行把所有状态替换都做成 volatile 快照。

## 停止标志与生命周期协作
volatile boolean 常见用途是停止标志。一个线程在循环中周期性检查标志，另一个线程把标志设置为 true，请求它退出。这个模式简单、可读、成本低，适合循环体本身不会长时间阻塞，或者循环体每次阻塞都有超时并能回到检查点的任务。停止标志表达的是协作式取消：请求方发出信号，被请求方在合适位置观察信号并自行收尾。

停止标志的关键不是字段声明，而是检查点设计。工作线程如果在一个永不返回的阻塞调用里等待，就算停止标志是 volatile，也没有机会读取它；如果循环体一次执行很久，停止响应就会延迟；如果收尾代码没有放在 finally 或等价路径中，线程退出时可能泄漏资源。因此，volatile 停止标志通常要与超时等待、中断、资源关闭或队列唤醒配合使用。标志负责表达“应该停止”，其他机制负责让线程从阻塞点回来并完成清理。

中断和 volatile 停止标志不是互斥关系。中断是线程级协作机制，许多阻塞方法会在收到中断后抛出 InterruptedException；volatile 标志是对象或任务级状态，能表达更具体的生命周期含义。一个健壮的循环经常同时检查 `!stopped` 和 `!Thread.currentThread().isInterrupted()`，在捕获 InterruptedException 后恢复中断标志，并进入收尾路径。这样既尊重调用方的线程取消协议，也保留对象自身的停止状态。

停止标志还要避免“只写不读”或“读得太少”。有些代码提供了 `stop()` 方法设置 volatile 字段，但工作线程只在启动前检查一次，之后不再检查；这样的标志没有实际取消能力。还有些代码在多个嵌套循环中只在外层检查，内层可能长时间运行，导致停止延迟。更好的做法是在循环边界、批处理批次之间、阻塞调用返回后、长计算任务的阶段边界都设置检查点。检查点越贴近可安全退出的位置，取消响应越稳定。

停止标志不应承担复杂状态机的全部职责。如果任务状态包含 NEW、RUNNING、STOPPING、STOPPED、FAILED 等多个阶段，并且多个线程可能同时触发启动、停止、失败记录和资源释放，那么一个 volatile boolean 就太弱了。可以使用 volatile enum 表达只读观察状态，但状态转换本身可能需要 synchronized、AtomicReference CAS 或专门的生命周期管理器来保证合法迁移。否则就会出现重复启动、重复关闭、失败覆盖成功、资源释放两次等问题。

## volatile 与 synchronized 的边界
synchronized 同时提供互斥、可见性和有序性。进入 synchronized 块需要获取监视器，退出 synchronized 块会释放监视器；对同一个监视器的释放 happens-before 后续获取。因此，锁保护的临界区不仅能让同一时刻只有一个线程执行，还能让后续持有同一把锁的线程看到前一个线程在释放锁前的写入。volatile 不提供互斥，只提供围绕单个 volatile 变量的可见性和有序性。

如果需求是“一个线程写状态，多个线程读取状态，状态更新本身是单次赋值或不可变快照替换”，volatile 往往足够。如果需求是“多个线程都可能修改状态，而且修改依赖旧值”，volatile 通常不够。如果需求是“多个字段必须同时满足不变式”，锁通常更直接。如果需求是“临界区中包含多个对象、多个集合或外部资源的协调”，锁或更高层同步器更容易表达边界。判断时不要先问性能，而要先问正确性需要互斥吗，需要保护多大范围的不变式吗。

synchronized 的成本并不只来自阻塞。现代虚拟机对无竞争或低竞争锁有很多优化，真实成本取决于竞争程度、临界区大小、锁对象逃逸情况和运行时状态。volatile 的成本也不是零，它会限制重排序并可能引入内存屏障和缓存一致性流量。把 volatile 当成“低成本 synchronized”容易导致错误选型。正确顺序是：需要互斥就用互斥工具，需要条件等待就用条件工具，需要单变量原子更新就用 Atomic，需要发布和观察状态才考虑 volatile。

锁和 volatile 混用时要保持协议一致。一个字段如果所有读写都在同一把锁内，其实通常不需要再声明 volatile，因为锁已经提供可见性。反过来，如果有的路径在锁内写，有的路径无锁读取，即使字段是 volatile，也要确认无锁读取是否会破坏锁保护的不变式。最危险的写法是“写的时候加锁，读的时候为了快不加锁”，但读操作其实依赖多个字段的一致性。这样的读可能看到锁内更新过程之外的中间组合，导致业务判断错误。

在少数设计中，volatile 与 synchronized 可以合理配合。例如双重检查锁定中，外层 volatile 读用于快速路径，内层 synchronized 用于互斥创建；又如某些对象用锁执行复杂更新，更新完成后写 volatile 版本号，读线程先读取版本判断是否需要进入较慢路径。但这种组合需要清晰注释和严格推理。只要读者无法一眼看出哪个字段由锁保护、哪个字段由 volatile 发布、哪个不变式由谁维护，就应简化设计。

## volatile 与 Atomic、LongAdder、并发容器的边界
Atomic 类适合单变量或单引用的原子更新。它们通常通过 CAS 表达“如果当前值仍然是我刚才看到的旧值，就替换成新值；否则重新尝试或返回失败”。这让自增、计数、状态从 A 到 B 的条件迁移、引用快照替换等操作可以在无显式锁的情况下完成。volatile 只能保证读取和写入可见，不能保证比较与更新之间没有别人插入。只要更新逻辑依赖旧值，就应优先考虑 Atomic 或锁。

LongAdder 和 LongAccumulator 面向高并发计数热点。它们通过分散更新降低多个线程竞争同一个内存位置的压力，适合统计类场景，例如请求次数、事件数量、累计值。它们不适合需要每次读取都与所有更新形成强一致顺序的业务判断。volatile int 或 AtomicLong 在低竞争下可能更简单，LongAdder 在高竞争计数下吞吐更好，但读取结果通常用于近似实时统计，而不是作为严格条件控制。

并发容器解决的是容器内部结构安全和单个容器操作的并发语义。例如 ConcurrentHashMap 可以支持并发 put、get、compute 等操作，不需要外部用 volatile 发布整个 Map 才能让读线程看到更新。但这不等于跨多个容器操作的业务流程天然原子。若需要“如果不存在则创建并注册多个结构”，应使用容器提供的原子方法、外部锁或把流程合并到单个同步边界。volatile 引用指向一个普通 HashMap，只适合发布只读快照；如果发布后还要原地修改 HashMap，那就回到了数据竞争。

选择这些工具时，可以按不变式所在位置判断：不变式只在一个布尔标志或引用发布上，volatile 可能足够；不变式在一个数值的递增递减上，Atomic 或 LongAdder 更合适；不变式在一个 Map 的单键更新上，ConcurrentHashMap 的原子方法更直接；不变式跨多个字段、多个集合或多个外部动作，锁、事务化设计或更高层协调机制更可靠。工具不是按“轻重”排序，而是按语义匹配程度排序。

## 性能成本与机械同情
volatile 经常被描述为轻量同步，这是相对于互斥阻塞而言的。它不会像锁竞争那样让线程进入阻塞队列，也不需要显式进入和退出临界区；但它仍然有成本。volatile 读写会限制编译器和处理器重排序，可能引入内存屏障，可能让缓存一致性协议在多个核心之间传播写入，可能阻止某些值长期保存在寄存器中。对于低频状态标志，这些成本通常可以忽略；对于高频热路径，成本可能非常明显。

写 volatile 通常比读 volatile 更容易产生跨核心影响，因为写入会让其他核心缓存中的同一缓存行失效或需要更新一致性状态。如果多个线程频繁写同一个 volatile 变量，就会形成写竞争热点，缓存行在核心之间来回迁移，吞吐下降。此时换成 Atomic 也未必能解决，因为 CAS 同样竞争同一个位置；LongAdder、分片计数、批量合并或减少共享写入频率可能更合适。

伪共享也是性能问题之一。处理器缓存以缓存行为单位维护一致性，而不是以 Java 字段为单位。如果两个彼此无关的 volatile 字段恰好落在同一个缓存行上，线程 A 高频写字段 x，线程 B 高频写字段 y，也可能互相干扰，因为它们竞争的是同一缓存行的一致性状态。Java 标准库和虚拟机内部会在某些高性能结构中通过填充或布局技巧降低伪共享风险，但普通业务代码不应过早手写复杂填充。只有在性能剖析明确显示缓存行竞争时，才值得针对性处理。

volatile 还可能影响编译优化。普通字段如果只在当前线程可见，编译器可以做更激进的值缓存、循环外提、冗余读消除等优化；volatile 读必须保留对同步变量的观察语义，不能随意把循环中的 volatile 读取提到循环外。停止标志正是利用这一点：工作线程在循环中反复读取 volatile 字段，不能被优化成只读取一次。这个约束保证了正确性，也意味着在极热循环中频繁读 volatile 会带来成本。

性能讨论必须服务于语义。为了省一点 volatile 读，把停止标志读缓存到局部变量并在长循环中不再刷新，会破坏取消响应；为了避免锁，把多字段不变式拆成多个 volatile 字段，会破坏正确性；为了降低 CAS 失败，把共享状态改成普通字段，会产生数据竞争。真正有效的性能优化通常来自减少共享、降低写频率、缩小同步范围、使用不可变快照、批量处理和选择合适数据结构，而不是把同步语义偷偷删掉。

## 适用场景与不适用场景
volatile 适用的第一类场景是状态标志。字段只有少量写入，多个线程读取，根据标志决定是否启动某个分支、是否继续循环、是否进入降级路径或是否认为配置已就绪。这里的状态变化通常是单向或简单多态的，读线程不需要修改该字段，也不需要把多个字段同时作为事务更新。典型例子包括 `running`、`closed`、`ready`、`initialized` 等，但字段名不能替代设计，仍要检查状态转换是否真的简单。

第二类场景是不可变对象引用发布。配置、策略、只读表、快照、路由规则等数据可以构造成不可变对象，再通过 volatile 引用替换。读线程每次读取引用后使用当前快照，不需要锁。这个模式的关键是“替换引用而不是原地修改对象”。只要发布后继续改对象内部字段，volatile 引用的安全发布语义就不足以保护后续变更。

第三类场景是单写多读的最新值观察。例如一个后台线程周期性计算某个统计结果，多个读取线程只关心最新结果，而不参与更新，也不要求读取每一次中间变化。volatile 可以让读线程低成本获得最新发布的引用或数值。若多个线程都要写，或者每次更新都不能丢，需求就变了，需要 Atomic、队列、锁或聚合器。

第四类场景是轻量状态机的只读观察。某个对象内部用锁或单线程维护状态转换，同时用 volatile 字段暴露当前阶段给其他线程快速读取。读取方只做诊断、展示或粗粒度分支，不承担严格业务决策。若读取结果会驱动关键写操作，则必须确认从读取到后续动作之间是否需要原子性，否则就可能出现检查通过后状态已变的竞态。

不适用场景同样清晰。多个线程执行自增、自减、累加、扣减，不能只用 volatile；先检查再修改的逻辑不能只用 volatile；多个字段组成的不变式不能拆成多个 volatile；集合、数组、可变对象发布后继续原地修改，不能靠 volatile 引用保护；需要等待条件满足并阻塞线程，volatile 也不是条件队列，通常要用 wait/notify、Condition、Latch、Semaphore、BlockingQueue 或 Future 等工具。

## 常见误读的逐条澄清
第一，volatile 不是“强制所有线程立刻同步主内存”。Java 内存模型没有要求开发者以这种方式思考。正确说法是 volatile 读写建立特定同步关系，读线程在读取同一个 volatile 变量后，可以看到写线程在 volatile 写之前发布的结果。这个保证是规范层面的，不应退化成对某种硬件缓存行为的口号。

第二，volatile 不是“变量值总是最新”。如果一个线程没有读取该 volatile 变量，它就没有观察动作；如果读取发生在写入之前，它当然读不到之后的值；如果多个写线程连续写入，读线程可能只看到其中某个较新的状态，而不是每一次变化。volatile 适合观察当前状态，不适合传递每一个事件。需要不丢事件时，应使用队列、日志、流式结构或带序号的协议。

第三，volatile 不是“线程安全”。线程安全要看共享状态和不变式。只有当共享需求刚好落在 volatile 能保证的可见性、有序性和单次读写上时，volatile 才能构成足够的线程安全策略。只要存在复合更新、跨字段一致性、检查后执行、资源生命周期竞争，就需要其他同步机制。

第四，volatile 不等于 final。final 字段关注对象构造完成后的初始化安全和不可重新赋值语义；volatile 字段关注运行期间跨线程读写的可见性和有序性。一个字段可以是 final 但不是 volatile，表示构造后不再改变；一个字段可以是 volatile 但不是 final，表示运行期间会被发布新值。不可变对象内部常用 final 字段，外部用 volatile 引用发布新快照，这两者经常配合，但语义完全不同。

第五，volatile 不等于中断。中断是线程协作取消和阻塞唤醒机制的一部分，许多阻塞方法会响应中断；volatile 标志是共享状态。一个长时间运行的任务可以通过 volatile 标志知道应该停止，但如果它正在可中断阻塞中，通常还需要调用 interrupt 让它尽快返回。反过来，只调用 interrupt 而任务从不检查中断状态或吞掉 InterruptedException，也无法形成可靠取消。

第六，volatile 不是调试手段。遇到并发问题时随手给字段加 volatile，可能掩盖真正的数据竞争，也可能只在某些路径上改善现象。正确做法是先画出共享状态、写入路径、读取路径、同步关系和不变式，再决定是否需要 volatile。如果无法说清楚哪次 volatile 写 happens-before 哪次 volatile 读，这个 volatile 很可能不是可靠设计的一部分。

## 设计检查清单
设计 volatile 字段时，可以先确认字段承担的角色。如果它是停止标志，工作线程在哪里检查，阻塞时如何返回，退出时谁释放资源？如果它是发布引用，被发布对象是否不可变，内部集合是否会继续被修改，多个相关数据是否被合并进同一个快照？如果它是状态码，状态迁移是否单调，多个线程是否可能同时迁移，非法迁移由谁阻止？如果它是最新值，是否允许跳过中间值，是否存在多个写线程，读取结果是否用于关键业务决策？

然后确认 happens-before 链条。写端有哪些普通写入需要发布？它们是否都发生在 volatile 写之前？读端是否在使用这些数据之前读取同一个 volatile 字段？读端读取后是否持有一个局部快照，避免同一方法中多次读取 volatile 引用而看到不同版本？如果需要同一版本的数据，是否应该先把 volatile 引用读到局部变量，再围绕该局部变量完成后续计算？

接着确认原子性需求。是否存在 `++`、`--`、累加、最大最小更新、检查后设置、检查后创建、余额扣减、容量判断、状态迁移等复合操作？如果有，volatile 是否只是被用来观察状态，而真正更新由锁或 Atomic 完成？如果没有其他机制，volatile 多半不够。这个检查能在编码前避免大量“可见但不正确”的竞态。

还要确认失败路径。发布前构造对象失败怎么办？停止标志设置后工作线程多久能退出？退出过程中如果抛异常，状态会停在 STOPPING 还是 FAILED？读线程拿到旧快照是否仍然有效？写线程连续发布多个快照时旧快照如何被回收？并发代码的异常路径必须与正常路径使用同样清晰的同步协议，否则 volatile 只能保证别人看到一个混乱状态。

最后确认性能假设。volatile 读写是否在极热循环中？是否有多个线程高频写同一变量？是否可以把每次写改成批量发布？是否可以用不可变快照降低锁竞争？是否需要用性能测试验证，而不是凭“volatile 很轻”做结论？正确的性能优化应在不削弱内存语义的前提下减少共享和竞争。

## 实践要点
第一，优先使用表达力更明确的高层工具。能用 BlockingQueue 表达生产消费，就不要手写 wait/notify；能用 ExecutorService 管理任务，就不要散落创建 Thread；能用 CountDownLatch 表达一次性等待，就不要用循环 sleep；能用 AtomicInteger 表达单变量计数，就不要扩大成不必要的重量级锁。高层工具的价值在于把容易写错的等待、唤醒和内存语义封装起来。

第二，给等待设置边界。无界等待是并发故障扩大的常见原因。获取结果可以使用超时 get，条件等待可以使用带超时的 await，外部调用可以设置超时，线程池关闭可以设置 awaitTermination 时间。超时不是为了掩盖错误，而是为了让调用方有机会记录现场、释放资源或降级。

第三，保持中断语义。凡是调用会抛出 InterruptedException 的方法，都要认真决定当前层是否消费中断。如果当前层无法完成取消处理，通常应恢复中断标志并返回，让上层继续处理。静默吞掉中断会让关闭流程卡住，也会让调用者误以为任务已经遵守取消协议。

第四，减少共享可变状态。不可变对象、局部变量、消息传递、任务参数复制、并发容器和清晰的所有权，通常比到处加锁更容易维护。锁和原子类解决的是共享状态冲突，但最好的并发设计往往是先减少共享，再保护必须共享的部分。

第五，保持诊断信息。线程名应能反映用途，线程池应暴露队列长度和拒绝次数，长时间等待应能定位锁或条件，后台任务异常应记录。并发问题通常无法只靠阅读一行报错定位，足够的运行时信息能显著降低排查成本。

## 常见误区
不要用 volatile 实现 i++ 的线程安全。这是volatile中最需要避免的误区之一。它之所以危险，是因为代码在低并发、本机运行或单次测试时可能完全正常，但在调度时机变化、任务数量上升或关闭流程触发时暴露问题。并发程序的正确性不能建立在“通常来得及”“一般不会同时发生”“这个线程应该先执行”这类假设上。

另一个误区是把时间当同步。sleep 一段时间、等待“差不多执行完”、依赖日志先后顺序、用本机测试速度推断生产时序，都属于时间同步误区。正确做法是使用明确同步条件，例如 join、Future、Latch、Barrier、锁、条件队列或 volatile 状态。

还要避免把工具名当保证。使用线程池不代表任务不会丢，使用锁不代表锁对象选对了，使用 volatile 不代表复合操作原子，使用 ThreadLocal 不代表不会泄漏，使用公平锁不代表请求一定按业务顺序完成。每个工具都只保证它规范中定义的部分，超出边界的需求仍要由设计补上。

最后是忽略失败路径。任务抛异常后计数是否递减，锁是否释放，许可是否归还，线程池是否还能接收任务，Future 是否有人读取异常，中断是否被恢复，资源是否关闭，这些都必须在代码中体现。并发系统的许多严重问题不是正常路径错误，而是异常路径没有释放等待者或资源。

## 排查思路
排查volatile相关问题时，可以先问四个问题。第一，线程现在在哪里：运行、阻塞在锁、等待条件、等待 I/O、等待任务结果，还是已经终止。第二，共享状态由谁保护：锁对象是否一致，volatile 是否覆盖读写两端，原子操作是否足够表达不变式。第三，生命周期是否闭合：启动后是否可停止，提交后是否可取消，关闭后是否拒绝新任务，异常后是否释放等待方。第四，容量是否合理：线程数、队列长度、许可数量、任务耗时和上游流量是否匹配。

工具上可以结合线程 dump、日志时间线、线程池指标、锁竞争指标和重复压力测试。线程 dump 能看到 BLOCKED、WAITING、TIMED_WAITING 以及调用栈；线程池指标能看到任务是否堆积；日志可以确认取消和关闭是否发生；压力测试可以放大竞态窗口。排查时不要只看最终异常，要寻找第一个失去同步关系或容量边界的位置。

如果问题偶发，优先怀疑数据竞争、遗漏通知、异常路径未释放、线程池饱和、死锁或外部阻塞。volatile相关代码应尽量让这些情况可被观察，例如在拒绝策略中记录任务类型，在等待超时后输出关键状态，在关闭时记录未执行任务数量，在捕获异常时保留中断状态或根因。

## 深入理解与推导
理解volatile时，可以从一个更具体的推导过程开始：先假设程序只有一个线程，所有语句按照调用栈顺序执行，变量只被当前线程读写；再引入第二个线程，让同一份状态可能被两个执行流同时观察；最后再加入等待、取消、超时、异常和关闭。每增加一个因素，就需要增加一条明确规则。没有规则的地方，运行结果就只能依赖调度时机，而调度时机从来不是稳定契约。

第一条规则是所有权规则。一个并发对象或同步工具必须有清楚的创建者、使用者和释放者。创建者负责初始化完整状态，使用者负责遵守访问协议，释放者负责关闭或清理资源。volatile相关代码如果没有所有权规则，就会出现同一个对象被多个模块以不同方式访问：有的路径加锁，有的路径不加锁；有的路径处理取消，有的路径吞掉异常；有的路径认为可以关闭线程池，有的路径仍在提交任务。这类问题通常不是 API 本身难用，而是边界没有被写进设计。

第二条规则是条件规则。等待必须等待一个可验证的条件，而不是等待一段时间。条件可以是线程结束、任务完成、计数归零、队列非空、许可可用、状态标志改变或锁变为可获取。条件还必须由同一套同步机制保护：用锁保护的条件要在持锁时检查，用 volatile 发布的状态要由 volatile 读观察，用 Future 表示的结果要通过 get 获取，用同步器表示的阶段要通过对应 await 方法等待。条件和等待工具不匹配时，最容易出现遗漏通知、虚假成功或永久阻塞。

第三条规则是失败规则。volatile的正常路径通常很短，但失败路径决定系统是否可靠。提交失败时是否记录，等待超时时是否取消，任务异常时是否释放锁和许可，线程被中断时是否恢复中断标志，关闭时是否处理尚未执行的任务，这些问题都必须有答案。并发代码的失败路径尤其重要，因为一个线程的异常经常不会直接出现在另一个线程的调用栈中；如果没有显式传播和记录，调用方只能看到“没有结果”或“系统变慢”。

第四条规则是容量规则。任何等待队列、线程池、同步器许可、任务拆分粒度和重试循环都有容量含义。容量太小会导致吞吐不足，容量太大会隐藏过载并放大延迟，容量无界会把压力转化为内存占用或不可控等待。volatile如果用于高并发路径，就不能只讨论功能正确，还要讨论任务数量、平均耗时、峰值流量、等待上限和降级策略。容量规则不是性能优化阶段才考虑的问题，而是并发设计的一部分。

第五条规则是可见性规则。一个线程做完动作，不代表其他线程自然知道。必须通过锁释放与获取、volatile 写与读、线程启动与结束、并发容器操作、同步器状态变化等方式建立可见性。volatile的很多误用都可以归结为缺少可见性规则：写线程以为自己已经发布状态，读线程却可能看到旧值；任务以为已经初始化对象，其他线程却可能看到部分构造；关闭线程以为已经发出停止请求，工作线程却从不检查中断或状态标志。

把这五条规则连起来，就能形成一个检查清单：对象是否安全发布，访问是否遵守同一协议，等待是否有明确条件，失败是否释放资源，关闭是否能唤醒等待者，容量是否有上限，结果是否通过同步关系传播。volatile的学习价值正在于把这些问题具体化。只要清单中的某一项回答含糊，就应回到代码重新收紧边界，而不是依赖注释或经验说明“正常情况下不会出问题”。

## 阅读与复习建议
复习volatile时，建议不要只看结论，而要自己沿着一条并发路径推演：哪个线程先创建对象，哪个线程发布引用，哪个线程开始执行，哪个线程可能等待，哪个线程负责唤醒，异常会落在哪里，关闭时如何让等待者退出。能把这条路径讲清楚，通常比记住零散 API 更有用。

还可以把本文中的代码片段改写成几个小实验：增加线程数量，故意制造超时，主动触发中断，抛出运行时异常，重复执行上千次观察结果。实验不是为了依赖偶然输出，而是为了暴露自己对边界条件的理解是否完整。只要实验结果和预期不一致，就应回到线程安全边界、内存可见性、等待条件和生命周期四个角度重新检查。

## 小结
volatile 解决可见性和有序性，不解决复合原子性。

掌握volatile，最终要形成两种能力：一是能写出边界清晰的代码，知道正常路径、失败路径和关闭路径分别如何运行；二是能解释运行时现象，知道线程为什么等待、为什么没有看到新值、为什么任务没有执行、为什么关闭没有结束。Java 并发不是把代码拆到多个线程中那么简单，而是给执行、状态和生命周期建立可验证的规则。

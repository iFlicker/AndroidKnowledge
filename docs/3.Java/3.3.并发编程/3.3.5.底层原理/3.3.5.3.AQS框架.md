# 3.3.5.3 AQS框架

## 定位

AQS 是 `AbstractQueuedSynchronizer` 的常用简称，位于 `java.util.concurrent.locks` 包中。它不是一个拿来直接加锁的业务类，而是 Java 并发库中用于构建锁和同步器的基础框架。`ReentrantLock`、`Semaphore`、`CountDownLatch`、`ReentrantReadWriteLock`、`ThreadPoolExecutor.Worker` 等实现都直接或间接依赖它。理解 AQS 的重点不是记住某个公开方法，而是理解它怎样把“同步状态、等待队列、线程阻塞、线程唤醒、中断响应、超时等待、独占和共享语义”组织成一套可复用的模板。

普通业务代码通常不需要继承 AQS。大多数场景应该直接使用 Java 标准库已经提供的锁、信号量、倒计时门闩、读写锁、阻塞队列、并发容器或原子类。AQS 的价值在于它把同步器中最难写对的通用部分抽象出来：获取失败时如何入队，队列中的线程何时挂起，释放成功后怎样唤醒后继节点，等待过程中如何处理中断和取消，条件等待如何从条件队列转移到同步队列。具体同步器只需要定义“什么叫获取成功、什么叫释放成功、同步状态如何解释”，而不必重复实现完整的排队阻塞协议。

本文保持通用 Java 技术视角，围绕 AQS 自身机制展开。学习它时要把三个层次分开。第一层是 Java 内存模型和线程调度基础：共享变量如何保证可见性，线程如何阻塞和唤醒，中断标志如何传播。第二层是 AQS 框架：`state`、同步队列、条件队列、模板方法和获取释放流程。第三层是具体同步器：`ReentrantLock` 把 `state` 解释为重入次数，`Semaphore` 把 `state` 解释为许可数量，`CountDownLatch` 把 `state` 解释为剩余计数。混淆这三层会导致常见误解，例如把 AQS 队列等同于某一种锁，把 `state` 等同于线程持有者，或者认为唤醒线程就代表线程已经获得锁。

## AQS 解决的核心问题

同步器的共同问题可以概括为两句话：共享状态什么时候允许被通过，不能通过的线程在哪里等待。前者是状态判定问题，后者是排队等待问题。如果只写状态判定，可以用一个 `volatile int` 或 CAS 循环实现简单互斥；但在竞争激烈时，失败线程会反复自旋消耗 CPU。如果只写阻塞等待，又必须处理排队顺序、丢失唤醒、中断、超时、取消节点清理、释放后唤醒谁等细节。AQS 把这两部分结合起来：用一个 `volatile int state` 表示同步状态，用一个变体 CLH 同步队列保存竞争失败的线程，用 `LockSupport.park/unpark` 完成阻塞和唤醒。

AQS 的设计目标不是让所有同步器语义完全相同，而是抽出共同骨架。一个可重入互斥锁需要记录持有线程和重入次数；一个信号量需要允许多个线程同时获取许可；一个倒计时门闩需要在计数归零后让所有等待者通过。这些语义差别很大，但都可以表达成“尝试获取、失败排队、释放状态、唤醒等待者”。AQS 负责流程，子类负责语义。

这种分工带来两个直接好处。第一，具体同步器可以复用经过长期验证的等待队列与阻塞唤醒逻辑，减少手写 `wait/notify` 的遗漏通知、错误唤醒和复杂取消问题。第二，开发者阅读并发库源码时可以用统一模型理解不同工具：先看 `state` 表示什么，再看 `tryAcquire` 或 `tryAcquireShared` 怎样判断成功，再看 `tryRelease` 或 `tryReleaseShared` 怎样改变状态，最后看它暴露给调用者的是锁、许可、门闩还是读写协调。

## state：同步状态的唯一整数入口

AQS 内部最重要的字段是 `state`。它是一个 `volatile int`，并通过 `getState()`、`setState(int)`、`compareAndSetState(int, int)` 这几个受保护方法供子类访问。AQS 本身不规定 `state` 的业务含义，只规定它是同步器状态变化的核心存储位置。具体同步器必须把自己的语义压缩到这个整数以及少量额外字段中。

在 `ReentrantLock` 中，`state == 0` 通常表示锁未被持有，`state > 0` 表示当前独占线程已经持有锁，数值表示重入次数。第一次获取锁时，线程通过 CAS 把 `state` 从 0 改成 1，并记录独占持有线程；同一线程再次进入时增加 `state`；释放时递减 `state`，只有减到 0 才真正释放并清空持有线程。

在 `Semaphore` 中，`state` 表示可用许可数量。获取一个许可就是尝试把 `state` 减一，只要剩余许可不为负即可成功；释放许可就是把 `state` 加一，并可能唤醒等待者。这里没有“某个线程独占持有”的概念，多个线程可以同时通过同步器。

在 `CountDownLatch` 中，`state` 表示还需要倒数的次数。调用 `await()` 的线程在 `state` 不为 0 时等待；调用 `countDown()` 的线程把 `state` 减一；当 `state` 变成 0 时，所有等待者都可以通过。它没有重置能力，因为 `state` 一旦归零，门闩语义就完成了。

把这些例子放在一起可以看到，`state` 不是“锁状态”的同义词，而是“同步器状态”的通用载体。它可以表示重入次数、许可数量、倒计时剩余值、读写锁的组合位段，也可以在自定义同步器中表示阶段、容量或开关状态。但它只有一个 `int`，因此设计自定义同步器时必须谨慎。需要多个复杂字段共同维护不变式时，强行塞进 AQS 往往会让实现变得脆弱。

`state` 是 `volatile`，这保证了对它的读写具备可见性；CAS 更新则保证单次比较并修改的原子性。需要注意的是，AQS 不会自动让子类的所有字段都线程安全。比如 `ReentrantLock` 的独占持有线程字段需要与 `state` 的转移配合使用；如果自定义同步器还有其他共享字段，就必须明确它们是否只在持有同步器时访问，是否需要额外 `volatile`，是否能由 `state` 的释放获取关系间接保护。

## CLH 同步队列：竞争失败后的排队位置

AQS 使用一个 FIFO 风格的同步队列保存获取失败的线程。这个队列常被称为 CLH 队列的变体。传统 CLH 锁是一种基于队列的自旋锁，每个线程关注前驱节点状态；AQS 借用了“节点排队、关注前驱”的思想，但不是纯自旋，而是在合适时机用 `LockSupport.park` 挂起线程。这样可以兼顾排队结构和阻塞等待，避免失败线程长时间空转。

同步队列由 `head` 和 `tail` 两个节点引用维护。队列中的节点大致包含线程引用、前驱节点、后继节点、等待状态以及模式标记。模式标记用于区分独占和共享；等待状态用于描述节点是否取消、是否需要唤醒后继、是否处于条件队列等。队列初始化通常是延迟的：第一次有线程获取失败并需要入队时，AQS 才创建一个哨兵头节点，然后把等待节点追加到尾部。

一个竞争失败的线程不会随意插入队列中间，而是通过 CAS 追加到 `tail` 后面。追加成功后，它会进入一个循环：检查自己的前驱是否是 `head`，如果是，就再次尝试获取；如果获取成功，就把自己设置为新的 `head`，从队列语义上出队；如果不能获取，就根据前驱节点状态决定是否可以安全 park。这个循环体现了 AQS 的核心思想：入队不代表立刻阻塞，唤醒也不代表立刻成功，线程必须反复检查同步状态和队列位置。

为什么只有前驱是 `head` 的节点才有资格再次尝试获取？因为这能维持同步队列的基本 FIFO 语义。队列头节点代表已经成功获取或正在释放后的占位节点，头节点的直接后继是下一个应该被唤醒和竞争的线程。如果队列中任意节点都频繁尝试获取，排队顺序就会被破坏，等待者也难以被准确唤醒。当然，非公平锁允许新来的线程在入队前先尝试抢占一次，这属于同步器策略；一旦线程进入队列，AQS 的队列协议仍然围绕前驱和头节点推进。

取消节点是同步队列中非常重要但容易被忽略的部分。等待线程可能被中断、超时或在获取过程中失败退出。取消后的节点不能永久挡在队列中，否则后继线程可能无法被唤醒。AQS 会在入队、唤醒和寻找后继的过程中跳过或清理取消节点。由于并发修改不可避免，队列中 `prev` 和 `next` 链接在某些瞬间可能并不完全对称，源码中才会有从尾部向前查找有效节点等保守逻辑。理解这一点后，就不会把 AQS 队列误认为普通单线程链表。

## 独占模式与共享模式

AQS 支持两类获取模式：独占模式和共享模式。独占模式表示同一时刻只有一个线程能够成功获取同步状态，典型代表是 `ReentrantLock`。共享模式表示多个线程可以同时通过同步器，典型代表是 `Semaphore` 和 `CountDownLatch`。两种模式共用同步队列，但获取、释放和传播唤醒的规则不同。

独占模式的模板方法是 `tryAcquire(int arg)` 和 `tryRelease(int arg)`。AQS 的 `acquire(int)` 会先调用 `tryAcquire`，成功就直接返回；失败则把当前线程包装成独占节点入队，并在队列中等待。释放时，`release(int)` 调用 `tryRelease`，如果子类报告同步状态已经完全释放，AQS 就唤醒头节点的有效后继。独占模式中，一个释放动作通常只需要唤醒一个后继，因为只有一个线程能获得锁。

共享模式的模板方法是 `tryAcquireShared(int arg)` 和 `tryReleaseShared(int arg)`。`tryAcquireShared` 返回值有特殊含义：负数表示获取失败，零表示获取成功但后续共享获取不一定还能继续传播，正数表示获取成功且后继共享节点可能也能继续获取。共享释放时，`releaseShared(int)` 在状态变化可能允许更多线程通过后，会触发共享传播。共享模式的难点不在于“唤醒一个线程”，而在于某个线程成功通过后，是否应该继续唤醒下一个共享等待者。

`CountDownLatch` 是理解共享模式的好例子。只要计数未归零，所有 `await()` 线程都失败并进入共享等待；当最后一次 `countDown()` 把 `state` 改成 0，释放共享状态，等待队列中的多个线程都应该陆续被唤醒并通过。与互斥锁相比，这不是一个线程出队后独占资源，而是一批等待者共同观察到“门已经打开”。

`Semaphore` 也使用共享模式，但语义不同。它不是一次性打开的门，而是有限许可池。如果释放一个许可，理论上只多出一个线程可通过；如果释放多个许可，可能有多个线程通过。是否公平取决于具体实现策略。公平信号量会检查队列中是否已有前驱等待者，非公平信号量则允许新来线程直接尝试减少许可。这里可以看到，共享模式并不等于无限放行，它只表示同步状态允许多个线程同时成功。

## acquire 流程：从快速尝试到排队阻塞

AQS 的获取流程可以分成快速路径和队列路径。快速路径先调用子类的尝试获取方法，成功时完全不需要入队。这一点非常重要，因为无竞争场景是锁和同步器的常见路径，避免创建节点和阻塞可以显著降低成本。只有尝试失败时，线程才进入同步队列。

独占获取的典型流程如下。当前线程调用 `acquire(arg)`；AQS 调用 `tryAcquire(arg)`；如果成功，方法返回；如果失败，当前线程被封装为独占节点追加到队尾；入队后线程进入循环；如果前驱是 `head`，就再次调用 `tryAcquire(arg)`；成功后把自己设为新头节点并清理线程引用；失败时判断是否应该 park；如果应该挂起，就调用 `LockSupport.park`；被唤醒、中断或虚假返回后继续循环检查。

这个流程里有几个关键点。第一，`tryAcquire` 可能被调用多次，子类实现必须允许重复尝试，不能依赖“只调用一次”的假设。第二，线程从 `park` 返回不代表获取成功，它必须重新检查条件。返回原因可能是前驱释放后的唤醒，可能是中断，也可能是 `park` 允许的虚假返回。第三，AQS 的不可中断获取方法即使等待期间发现中断，也会继续完成获取，只是在最终返回前恢复中断标志；可中断获取方法则会在等待中断时抛出 `InterruptedException` 并取消节点。

共享获取流程类似，但成功后的处理不同。共享节点成为头节点后，可能需要继续向后传播唤醒，让更多共享等待者尝试获取。AQS 中的共享传播逻辑会结合返回值、头节点状态和后继节点模式判断是否继续唤醒。阅读共享模式源码时，如果只套用独占锁“一次释放唤醒一个”的模型，就容易看不懂 `setHeadAndPropagate` 一类方法的意义。

超时获取和可中断获取是同一套队列协议的变体。可中断获取要在等待过程中检测中断并退出；超时获取要计算剩余纳秒数，在超时后取消节点并返回失败。它们都必须保证取消节点不会破坏队列，后继线程仍然能被正确唤醒。这也是 AQS 作为框架的价值：同步器作者只实现状态语义，不必为每个超时、中断组合重写队列处理。

## release 流程：状态释放与后继唤醒

释放流程看起来比获取流程短，但它决定等待者能否继续推进。独占模式中，调用方执行 `release(arg)`；AQS 调用子类的 `tryRelease(arg)`；如果返回 `false`，说明同步状态还没有完全释放，例如可重入锁只是减少了一层重入；如果返回 `true`，说明释放完成，AQS 会检查头节点并唤醒有效后继。被唤醒的线程回到获取循环，只有当它的前驱是头节点且再次获取成功时，才真正成为新的持有者。

这说明 `tryRelease` 的返回值不是“这次 release 方法有没有执行成功”这么简单，而是“释放后是否应该唤醒后继”。以可重入锁为例，线程重入三次后必须释放三次。前两次释放只把 `state` 从 3 改到 2、从 2 改到 1，锁仍然由当前线程持有，不应该唤醒其他等待者；最后一次把 `state` 改到 0，才需要清空持有线程并唤醒后继。

共享模式中，`releaseShared(arg)` 调用 `tryReleaseShared(arg)`。如果子类返回 `true`，说明状态变化可能让共享等待者通过，AQS 会执行共享释放传播。`CountDownLatch` 中，只有计数从 1 变成 0 的那次 `countDown()` 才真正打开门；此前从 5 到 4、从 4 到 3 的变化不会唤醒等待者。`Semaphore` 中，释放许可通常可能让等待者继续竞争，因此会触发共享释放。

释放动作同时承担内存语义。基于 AQS 的同步器通常通过 `volatile state` 和 CAS、`LockSupport` 以及并发库约定建立必要的可见性。对 `Lock` 来说，成功的 `unlock` 与随后成功的 `lock` 之间具有类似监视器释放和获取的内存效果：释放前在临界区中的写入，应当对后续获取同一锁的线程可见。使用者不能绕过同一个同步器去读写受保护状态，否则这条可见性链路就会断开。

## ConditionObject：条件队列不是同步队列

AQS 还提供了内部类 `ConditionObject`，用于支持 `Lock.newCondition()` 返回的条件变量。理解 Condition 的关键是区分两个队列：同步队列保存正在竞争锁或同步状态的线程，条件队列保存已经持有锁但因某个条件不满足而主动等待的线程。调用 `await()` 的线程不是简单地在同步队列中睡眠，而是先释放当前持有的独占锁，进入条件队列；等到其他线程调用 `signal()` 或 `signalAll()` 后，条件节点再被转移到同步队列，重新竞争锁。

这个设计解释了 Condition 使用上的几个硬性规则。第一，调用 `await`、`signal`、`signalAll` 前必须持有与该 Condition 关联的锁。因为条件谓词通常由这把锁保护，只有持锁检查和修改条件，才不会丢失状态变化。第二，`await` 返回后必须重新检查条件，通常写在 `while` 循环中，而不是 `if`。因为线程可能被中断、虚假唤醒，或者被 signal 后转入同步队列但重新获得锁时条件已经被其他线程改变。第三，`signal` 不等于把等待线程立即交给 CPU 执行业务逻辑，它只是把条件队列中的一个节点转移到同步队列；被转移线程还要等当前线程释放锁，并在同步队列中重新获取成功。

条件队列是单向链表结构，节点的等待状态与同步队列不同。`await` 时，当前线程会被加入条件队列，然后完全释放锁的持有计数。对可重入锁来说，如果当前线程重入了多层，`await` 必须保存重入次数，释放到 `state == 0`，被唤醒并重新获取后再恢复原来的重入层数。这个细节说明 Condition 不是一个普通等待列表，它必须和独占同步器的状态语义紧密配合。

`signal` 的职责是转移节点，而不是直接恢复条件。调用 `signal` 的线程仍然持有锁，只有它之后执行 `unlock`，同步队列里的后继才有机会继续获取。很多初学者会误以为 `signal` 之后等待线程马上执行，于是在 signal 后继续长时间持锁做耗时操作，导致等待线程虽然已经被转移，却迟迟无法恢复。正确理解是：修改条件、signal、尽快释放锁，这三个动作组成完整协作。

Condition 与 `Object.wait/notify` 有相似思想，但表达能力更强。一个 `synchronized` 监视器对象只有一个等待集合，多个不同条件往往要混在一起，用 `notifyAll` 保守唤醒；一个 `Lock` 可以创建多个 Condition，把“队列非空”“队列未满”“阶段完成”等不同条件拆成不同等待队列，从而减少无关唤醒。不过 Condition 仍然不能替代条件谓词本身，等待者必须在循环中检查由锁保护的真实状态。

## 模板方法：AQS 让子类只定义语义

AQS 采用模板方法模式。公开或 final 风格的获取释放方法提供完整流程，受保护的 `tryAcquire`、`tryRelease`、`tryAcquireShared`、`tryReleaseShared`、`isHeldExclusively` 由子类按需覆盖。默认实现通常抛出 `UnsupportedOperationException`，表示该同步器不支持对应模式。例如只支持独占的同步器不需要实现共享获取，只支持共享的同步器不需要实现独占获取。

这种模式要求子类遵守非常严格的边界。`tryAcquire` 不应阻塞，不应调用可能长时间等待的方法，也不应在失败时自己入队。它只负责在当前状态下做一次快速判断和必要的状态更新，成功返回 `true`，失败返回 `false`。排队、阻塞、唤醒、中断处理由 AQS 外层流程完成。如果子类在 `tryAcquire` 内部再手写等待，就会破坏 AQS 队列协议，造成死锁、丢失唤醒或难以清理取消节点。

`tryRelease` 也应只处理状态释放和合法性检查。以独占锁为例，释放线程必须是持有线程，否则应该抛出 `IllegalMonitorStateException`。释放到完全空闲时返回 `true`，未完全释放时返回 `false`。它不需要也不应该直接唤醒后继线程，因为 AQS 会根据返回值执行统一唤醒。

`tryAcquireShared` 的返回值比布尔值复杂，子类必须认真设计。负数表示失败，零表示成功但不承诺继续传播，正数表示成功并允许传播。`Semaphore`、`CountDownLatch`、读写锁中的共享读锁都依赖这个返回值表达不同程度的共享可通过性。写自定义共享同步器时，如果随意返回固定正数或固定零，可能造成唤醒传播过度或不足。

`isHeldExclusively` 主要服务 Condition。`ConditionObject` 需要判断当前线程是否独占持有同步器，以保证 `await` 和 `signal` 的调用合法。一个希望支持 Condition 的独占同步器必须正确实现它，否则条件等待的语义就不完整。共享同步器通常不能直接支持 AQS 的 ConditionObject，因为条件等待要求释放并重新获取独占锁。

## ReentrantLock 与 AQS 的关系

`ReentrantLock` 是理解 AQS 独占模式的最典型入口。它内部有一个继承 AQS 的同步器实现，并区分公平和非公平两种策略。锁对象本身对外暴露 `lock`、`unlock`、`tryLock`、`lockInterruptibly`、`newCondition` 等方法；真正的同步状态和队列控制在内部同步器中完成。

非公平锁的典型特点是新来的线程在调用 `lock()` 时可以先尝试直接 CAS 抢占 `state`。如果此时锁刚好空闲，即使同步队列里已有等待线程，新线程也可能先获取成功。这种策略减少了线程切换和队列唤醒成本，吞吐通常较好，但等待时间更不稳定。公平锁则会在尝试获取前检查同步队列中是否存在前驱等待者，只有没有前驱时才允许获取空闲锁，从而更接近先来先服务。

可重入性由 `state` 和独占持有线程共同表达。当锁未被持有时，线程把 `state` 从 0 改为 1 并设置 owner；当同一线程再次获取时，直接增加 `state`；当其他线程尝试获取时失败入队。释放时必须由 owner 执行，递减 `state`，直到归零才清空 owner 并唤醒后继。这里的 owner 不是 AQS 的通用机制，而是独占同步器为了实现重入和非法释放检查维护的语义字段。

`ReentrantLock` 的 Condition 来自 AQS 的 `ConditionObject`。每个 Condition 都有自己的条件队列，但它们共享同一把锁的同步队列。线程在某个 Condition 上 await 后，先进入该 Condition 的条件队列；被 signal 后转入锁的同步队列；重新获取锁成功后，才从 await 返回。这就是为什么一个 `ReentrantLock` 可以创建多个条件队列，而一个等待线程最终仍然必须回到同一把锁的竞争流程中。

## Semaphore 与 AQS 的关系

`Semaphore` 把 AQS 的共享模式包装成许可模型。构造时给定的许可数量通常写入 `state`；`acquire` 尝试减少许可；`release` 增加许可。只要剩余许可足够，多个线程可以同时获取成功，因此它不是互斥锁。信号量适合表达资源容量、并发数上限或某种令牌配额，但它不负责保护某个临界区中的复合不变式，除非调用者把许可协议设计成资源访问协议的一部分。

公平信号量和非公平信号量的差异也建立在 AQS 队列判断上。非公平实现会直接尝试扣减许可，成功就通过；公平实现会先检查是否有排队前驱，避免后来的线程插队。公平性影响等待顺序，不改变许可总数，也不保证业务完成顺序。线程拿到许可后执行多久、是否及时释放、是否被外部阻塞，都超出信号量自身保证。

`Semaphore` 的一个常见误用是忘记在失败路径释放许可。获取许可后执行的代码如果抛出异常，必须在 `finally` 中释放，否则 `state` 中的许可数量会越来越少，最终所有线程都可能永久等待。AQS 可以保证释放许可后唤醒等待者，但不能替调用者判断业务是否应该归还许可。容量型同步器的正确性经常取决于这种资源所有权边界。

还要注意，信号量的 release 不要求一定由 acquire 的同一线程调用。这与 `ReentrantLock` 不同。互斥锁强调所有权，非法线程释放会破坏临界区；信号量强调许可计数，释放动作本质上是向许可池归还或增加许可。这个差异来自具体同步器对 `state` 的解释，而不是 AQS 队列机制的差异。

## CountDownLatch 与 AQS 的关系

`CountDownLatch` 使用 AQS 共享模式实现一次性门闩。构造时的计数写入 `state`；等待线程调用 `await` 时，如果 `state == 0` 就直接通过，否则进入共享队列等待；工作线程调用 `countDown` 时把计数减一；最后一次减到 0 时触发共享释放，等待者被陆续唤醒。它的核心语义是“等到一组事件全部发生”，而不是互斥访问资源。

这个同步器的不可重置特性来自 `state` 语义。计数归零后，`await` 永远可以直接通过，标准 `CountDownLatch` 没有把 state 重新设置为初始值的公开协议。如果需要可重复阶段同步，应选择其他更合适的同步器，而不是试图复用已经归零的门闩。

`countDown` 不会阻塞调用线程。它只是尝试减少计数，并在归零时释放等待者。多余的 `countDown` 在计数已经为 0 后不会让计数变成负数，也不会重新关闭门闩。等待线程被唤醒后也不是因为某个锁资源空闲，而是因为共享条件“计数为零”成立。

从 AQS 角度看，`CountDownLatch` 是共享传播最清晰的例子：最后一次释放不是唤醒一个竞争者去独占状态，而是让所有等待者观察到同一个完成条件。理解这一点后，就能区分它与 `Semaphore` 的共享模式差异。两者都共享，但一个是一次性完成条件，一个是可增减的许可容量。

## 自定义同步器的边界

继承 AQS 可以写出非常精巧的同步器，但这不应该成为常规选择。自定义同步器适合在已有标准工具无法准确表达语义，并且该语义可以清楚映射到 AQS 的状态和队列协议时使用。比如需要一个一次性或阶段性的门控工具、一个特殊容量规则、一个与资源生命周期紧密绑定的同步状态，且标准库工具组合会造成更大复杂度时，才值得考虑。

判断是否适合 AQS，可以问几个问题。同步状态能否用一个 `int` 作为核心表示？获取失败时是否确实需要 FIFO 风格等待队列？等待线程是否应该由释放动作唤醒？独占和共享模式是否能准确覆盖需求？条件等待是否必须绑定独占锁？中断和超时语义是否可以沿用 AQS 的方法族？如果这些问题回答不清，使用更高层工具通常更稳妥。

自定义 AQS 子类时，要避免把业务逻辑写进模板方法。`tryAcquire`、`tryRelease` 等方法应该短小、无阻塞、只修改同步状态，不应执行 I/O、调用外部回调、申请不可控资源或持有其他锁后再做复杂操作。模板方法运行在并发竞争最激烈的位置，任何长时间操作都会放大队列等待和死锁风险。

还要谨慎处理额外字段。AQS 只帮你管理 `state` 的可见性和队列，不会自动保护对象中的普通字段。如果额外字段只在成功获取后访问，要确保所有访问都遵守同一个同步器协议；如果字段参与获取判断，要么纳入 `state`，要么使用合适的 volatile、CAS 或锁保护。多个字段之间存在不变式时，更要证明更新顺序和失败回滚不会被其他线程观察到不一致状态。

可测试性也是边界之一。一个同步器不仅要通过单线程用例，还要覆盖竞争、超时、中断、取消、重复释放、非法释放、并发释放、队列中节点取消、共享传播等场景。许多错误只在极窄调度窗口出现，普通功能测试很难发现。除非有足够强的语义必要性和测试投入，否则复用标准同步器通常是更工程化的选择。

## 内存语义与可见性

AQS 建立在 Java 内存模型之上。`state` 是 `volatile`，CAS 具有 volatile 读写效果，基于 AQS 的锁释放和获取也提供同步器规定的内存效果。对使用者来说，最重要的结论是：受同一个锁保护的临界区，前一个线程在 `unlock` 前的写入，对后一个线程成功 `lock` 后应当可见；使用 `CountDownLatch` 等同步器时，倒计时线程在 `countDown` 前的相关写入，可以通过等待线程从 `await` 返回后的同步关系被观察到；使用信号量等工具时，也应遵守其获取释放所建立的同步边界。

但这些语义不是无条件扩散到所有代码。若某个共享变量有时在锁内访问，有时绕过锁直接访问，绕过路径就不受锁的可见性保护。若一个对象在构造未完成时发布给其他线程，AQS 也无法修复这个安全发布问题。若多个变量共同构成业务不变式，只用一个 volatile 标志可能只能发布“状态已变”，不能保证读者以原子方式观察到整个不变式。

另一个容易混淆的问题是 `park/unpark` 与可见性。`LockSupport.unpark` 可以让被许可的线程从 park 返回，但“线程被唤醒”不等于“业务条件必然成立”。AQS 在唤醒前后还会结合 volatile 状态、队列节点状态和再次尝试获取来建立正确性。业务代码使用 AQS 同步器时，也不应该把唤醒当成条件本身。无论是 Condition 的 await，还是获取锁后的条件检查，都要回到受同步器保护的真实状态。

## 公平性、吞吐与饥饿

AQS 提供队列基础，但公平性由具体同步器决定。公平锁或公平信号量通常会在尝试获取时调用类似“是否存在排队前驱”的判断，避免后来线程绕过队列。非公平实现则允许新线程在某些时机直接尝试获取。两者没有绝对优劣，只有目标差异。

公平策略更容易推理等待顺序，能降低长期饥饿风险，适合强调请求先后关系或等待时间上界的场景。但公平策略也可能降低吞吐，因为刚被唤醒的线程需要从阻塞状态恢复，期间资源可能短暂空闲；如果允许正在运行的新线程直接获取，处理器缓存和调度局部性可能更好。非公平策略常见于高吞吐锁实现，但它可能让队列中的某些线程等待更久。

需要强调的是，公平性只针对同步器获取顺序，不等于业务完成顺序。线程按顺序获得锁后，临界区外的工作耗时可能不同；线程获得信号量许可后，资源使用时间可能不同；线程被唤醒后，操作系统调度也可能让它晚于其他线程运行。AQS 能控制的是同步状态获取协议，不是整个程序的所有执行先后。

饥饿也不只由公平性决定。持锁时间过长、释放许可遗漏、等待条件过宽、线程池容量不足、外部 I/O 阻塞、优先级设置不当，都可能让等待者长期无法推进。排查 AQS 相关等待问题时，应同时检查同步器策略和业务持有时间，而不是只看公平标志。

## 与 synchronized、wait/notify 的区别

AQS 和 `synchronized` 都能构建互斥与等待，但抽象层次不同。`synchronized` 是语言内置监视器机制，进入和退出代码块由 JVM 管理；AQS 是 Java 类库中的框架，基于 volatile、CAS 和 LockSupport 实现锁与同步器。`ReentrantLock` 等 AQS 工具提供了更丰富的方法族，例如可中断获取、限时获取、多个 Condition、公平策略选择、查询队列状态等。

这并不意味着 AQS 一定比 `synchronized` 更好。简单互斥场景下，`synchronized` 语义直接、写法简洁、异常时自动释放监视器，也能被运行时优化。需要显式尝试获取、超时等待、可中断等待或多个条件队列时，`Lock` 与 AQS 体系更合适。选择工具时应该先看需求表达，而不是把底层实现当成优劣标签。

`Condition` 与 `wait/notify` 的区别也类似。`wait` 必须在持有对象监视器时调用，并进入该对象的等待集合；`notify` 或 `notifyAll` 唤醒同一等待集合中的线程。Condition 必须在持有关联 Lock 时调用，并进入对应 Condition 的条件队列。一个 Lock 可以有多个 Condition，因此能把不同等待条件拆开。无论哪种方式，等待都应使用循环检查条件，通知也不应替代状态修改。

## 典型误区

第一个误区是把 AQS 当作普通锁来使用。AQS 是同步器构建框架，不是业务层直接操作的锁对象。业务代码应该使用 `ReentrantLock`、`Semaphore`、`CountDownLatch` 等成品工具。只有在设计自定义同步器时，才需要继承 AQS 并实现模板方法。

第二个误区是认为 `state` 的含义固定。`state == 1` 在互斥锁里可能表示被持有一次，在信号量里可能表示还剩一个许可，在门闩里可能表示还差一次倒计时。阅读源码时必须先问这个同步器怎样解释 `state`，再分析获取释放流程。

第三个误区是认为入队顺序等于一定执行顺序。AQS 同步队列提供获取竞争的排队基础，但非公平策略允许插队，线程唤醒后还要受调度影响，获取成功后业务执行时间也不受队列控制。队列顺序只能解释同步器内部竞争，不应扩大为整个程序的业务顺序保证。

第四个误区是认为 `signal` 后等待线程已经满足条件。`signal` 只是把条件队列节点转移到同步队列，等待线程必须等到发信号线程释放锁，并重新获取锁成功，然后再次检查条件。条件变量的正确写法永远围绕“修改状态、发出信号、循环检查状态”展开。

第五个误区是忽略中断和超时。AQS 提供不可中断、可中断、限时等不同方法族，调用者要选择符合生命周期要求的方法。关闭流程中如果使用不可中断等待，线程可能无法及时退出；如果捕获 `InterruptedException` 后直接吞掉，中断协议会被破坏；如果限时等待返回失败后不清理业务状态，也会留下资源泄漏。

第六个误区是自定义同步器过度设计。很多需求可以由现有工具组合表达，例如用 `Semaphore` 控制并发量，用 `CountDownLatch` 等待一次性完成，用 `BlockingQueue` 表达生产消费，用 `ReentrantLock` 加多个 Condition 表达复杂条件等待。继承 AQS 意味着要承担同步器级别的正确性证明和测试成本，不应只因为“看起来更底层”而选择。

## 阅读源码的建议路径

阅读 AQS 源码时，不建议从所有方法逐行展开。更有效的路径是先选一个具体同步器，再沿着公开 API 进入 AQS。比如从 `ReentrantLock.lock()` 进入非公平或公平同步器的 `lock` 方法，看它怎样尝试 CAS 设置 `state`，失败后怎样调用 `acquire`。接着看 `acquire` 怎样调用 `tryAcquire`，失败后怎样 `addWaiter`，节点在队列中怎样通过 `acquireQueued` 循环等待，释放时怎样通过 `release` 和 `unparkSuccessor` 唤醒后继。

第二条路径是从 `CountDownLatch.await()` 和 `countDown()` 看共享模式。观察 `tryAcquireShared` 为什么在 `state == 0` 时返回成功，`tryReleaseShared` 为什么只有减到 0 才返回 true，再看共享释放如何传播。这样能把独占锁的“一次一个”模型和共享同步器的“条件满足后一批通过”模型分开。

第三条路径是从 `ReentrantLock.newCondition()` 看条件队列。重点看 `await` 如何检查持锁、加入条件队列、完全释放锁、park 等待、被 signal 后转移到同步队列、重新获取锁并恢复中断语义。再看 `signal` 如何要求当前线程持锁，如何从条件队列取出节点并转移。只要这条路径清楚，Condition 的大多数使用规则都能自然推出。

读源码时要特别留意方法名中的语义差异：`acquire`、`acquireInterruptibly`、`tryAcquireNanos` 对中断和超时的处理不同；`release` 和 `releaseShared` 的传播方式不同；`hasQueuedPredecessors` 与公平策略相关；`isHeldExclusively` 与 Condition 合法性相关。不要把这些差异视为 API 数量多，而要把它们看成同步器生命周期的不同出口。

## 一个简化的自定义同步器示意

下面的代码只是说明模板方法分工，不建议在业务中重复实现标准工具。它表达一个一次性开关：关闭时获取失败，打开后所有共享获取成功。

```java
final class OneShotGate {
    private final Sync sync = new Sync();

    public void await() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);
    }

    public void open() {
        sync.releaseShared(1);
    }

    private static final class Sync extends AbstractQueuedSynchronizer {
        @Override
        protected int tryAcquireShared(int ignored) {
            return getState() == 1 ? 1 : -1;
        }

        @Override
        protected boolean tryReleaseShared(int ignored) {
            setState(1);
            return true;
        }
    }
}
```

这个示意中，`state == 0` 表示门未打开，`state == 1` 表示门已打开。等待线程调用 `acquireSharedInterruptibly`，失败后进入 AQS 共享队列；打开线程调用 `releaseShared`，设置状态并触发共享释放传播。代码之所以短，是因为排队、阻塞、中断、取消和唤醒都由 AQS 处理。真实项目中仍应优先考虑 `CountDownLatch` 等现成工具，因为它们的语义、测试和维护成本更清晰。

如果要把这个示意扩展成可重置门、带容量的门或多阶段门，复杂度会快速上升。可重置意味着要处理已经通过的线程、正在等待的线程和下一轮状态之间的竞态；容量意味着要处理许可增减和公平性；多阶段意味着要防止上一阶段唤醒错误地穿透到下一阶段。这些问题都说明，自定义同步器的难点不在于写出几行模板方法，而在于证明状态语义与队列协议始终一致。

## 排查 AQS 相关问题的思路

遇到 AQS 相关等待、卡住或吞吐异常时，可以先定位线程停在哪里。线程 dump 中如果看到 `LockSupport.park`，并不代表问题就在 park 本身；它只是 AQS 等待的阻塞点。要继续看调用栈上方是 `ReentrantLock.lock`、`Condition.await`、`Semaphore.acquire`、`CountDownLatch.await`，还是其他同步器方法。不同入口对应不同状态语义。

第二步是确认同步状态由谁改变。锁等待要找谁持有锁、持有多久、是否在临界区内执行慢操作；信号量等待要看许可是否被正确释放，是否有异常路径泄漏许可；门闩等待要看 `countDown` 是否一定执行，异常时是否遗漏；Condition 等待要看条件谓词是否在同一把锁下修改，signal 是否发生，等待者是否使用循环检查。

第三步是检查中断、超时和关闭协议。调用了不可中断等待的方法，线程就不会因为中断立刻退出；调用可中断方法却吞掉 `InterruptedException`，上层可能以为取消成功而实际没有；超时返回后如果不移除业务登记或不归还资源，后续状态可能越来越混乱。AQS 已经处理队列节点取消，但业务资源的取消仍然需要调用者自己完成。

第四步是区分同步器问题和容量问题。大量线程等待锁可能是锁实现错误，也可能只是临界区太大；信号量等待可能是许可数设置过小，也可能是外部资源变慢；倒计时门闩不释放可能是某个任务失败后没有执行 `countDown`；Condition 长期等待可能是条件根本没有被业务状态满足。AQS 能解释等待机制，但不能替代容量设计和失败路径设计。

## 实践原则

使用基于 AQS 的工具时，第一条原则是让同步器语义与业务语义对齐。互斥访问用锁，容量控制用信号量，一次性等待用倒计时门闩，条件协作用 Lock 加 Condition。不要用一个工具勉强模拟所有场景，例如用 sleep 等待门闩完成，用信号量冒充可重入锁，用 volatile 标志替代条件队列。

第二条原则是把释放写在确定路径上。锁的 `unlock`、信号量的 `release`、倒计时门闩的 `countDown`、条件等待前后的状态恢复，都应明确放在不会被异常跳过的位置。并发程序中，一个遗漏释放往往不会立刻报错，而是把后续线程拖入长期等待。

第三条原则是缩短持有同步状态的时间。持锁时只做必须受保护的状态检查和修改，不要在临界区内执行不可控 I/O、长时间计算或外部回调。持有信号量许可时要清楚许可代表的资源边界，不要让许可覆盖与资源无关的耗时操作。等待者越多，持有时间越长，AQS 队列中的唤醒和上下文切换成本就越明显。

第四条原则是选择合适的方法族。需要响应取消就使用可中断获取，需要防止无限等待就使用限时获取，只能在短路径尝试就使用非阻塞 `tryLock` 或 `tryAcquire` 类方法。方法名中的 interruptible、nanos、timeout 不是装饰，而是生命周期契约的一部分。

第五条原则是保留诊断能力。对锁竞争激烈的路径记录关键耗时，对许可泄漏风险高的路径记录获取和释放，对门闩等待设置合理超时并输出剩余计数，对条件等待在超时后输出条件相关状态。AQS 本身提供一些队列查询方法，但这些方法多用于监控和诊断，不应作为业务正确性的核心判断。

## 小结

AQS 的核心可以概括为：一个由子类解释的 `volatile int state`，一条保存竞争失败线程的 CLH 变体同步队列，两套独占与共享获取释放协议，一个支持条件等待的 Condition 队列，以及一组由子类覆盖的模板方法。AQS 负责通用的排队、阻塞、唤醒、中断、超时和取消处理；具体同步器负责定义 state 的含义和状态转移规则。

理解 AQS 后，再看 `ReentrantLock`、`Semaphore`、`CountDownLatch` 等工具，就不会只停留在 API 层面。它们的差异来自对同一个框架的不同解释：可重入锁把 state 当作独占重入次数，信号量把 state 当作许可数量，倒计时门闩把 state 当作剩余事件数。掌握这条主线，可以帮助我们更准确地选择并发工具、阅读源码、分析等待问题，也能提醒我们在必要时才自定义同步器，并为状态语义、失败路径和测试成本承担完整责任。

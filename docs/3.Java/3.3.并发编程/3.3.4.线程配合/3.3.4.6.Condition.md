# 3.3.4.6 Condition

`Condition` 是 `java.util.concurrent.locks` 包中的条件等待抽象。它与 `Lock` 配合，使线程能够在持有锁并检查共享状态后，因某个业务条件暂时不成立而释放锁、进入等待；另一个线程修改受同一把锁保护的状态后，再通知一个或全部等待者重新参与锁竞争。

它解决的不是普通的互斥问题，而是“已经获得互斥访问权，但当前仍不能继续执行”的线程协作问题。例如，有界缓冲区为空时，消费者即使拿到了锁也没有元素可取；缓冲区已满时，生产者即使拿到了锁也没有空间可写。锁只回答“现在是否允许独占访问共享状态”，条件变量进一步回答“共享状态是否已经满足某类操作的前置条件”。

`Condition` 最重要的价值，是一把显式锁可以创建多个条件等待队列。生产者可以等待“非满”，消费者可以等待“非空”，状态变化后只通知可能受益的那一类线程。与把所有等待者放在同一个监视器等待集合相比，这种拆分能够更准确地表达协议，也可以减少无意义唤醒。不过，`Condition` 是一种底层协调工具：使用者必须自己定义状态谓词、锁保护范围、通知时机、中断策略、超时语义和关闭协议。只要其中一项含糊，代码就可能出现遗漏通知、永久等待、活性下降或难以诊断的竞态。

本文只讨论通用 Java 语境中的 `Condition`，重点解释它是什么、为什么需要它、等待与通知如何工作、怎样建立完整协议，以及何时应改用更高层并发组件。

## 从互斥到条件等待

假设多个线程共享一个有界队列。队列状态必须满足两个基本不变式：

- 元素数量不能小于零。
- 元素数量不能超过容量上限。

使用 `Lock` 可以保证检查和修改队列的复合操作不会被其他线程穿插，但仅有互斥还不够。消费者拿到锁后可能发现队列为空，此时有三种直观但有问题的做法。

第一种做法是在锁内循环检查。这样会一直占有锁，生产者无法进入临界区放入元素，消费者等待的条件反而永远无法成立。

第二种做法是释放锁后使用 `sleep`，过一段时间再重试。这不再阻塞生产者，但等待时间只是猜测：时间过短会频繁抢锁和轮询，时间过长会增加响应延迟，而且无法表达“状态刚刚发生变化”的事实。

第三种做法是先释放锁，再把当前线程加入某个自建等待结构。这会遇到检查与入队之间的竞态：消费者确认队列为空后、真正进入等待前，生产者可能已经放入元素并发出通知。通知发生时没有可见等待者，随后消费者才开始休眠，形成典型的遗漏通知。

条件等待需要把三个动作组织成一个不可被协议参与者破坏的过程：

1. 在锁保护下检查条件谓词。
2. 条件不成立时，原子地释放锁并进入对应的条件等待队列。
3. 被通知、被中断、超时或发生虚假唤醒后，重新获得锁，再次检查谓词。

`Condition.await` 提供了这个过程所需的协调语义。调用者不需要自己在“释放锁”和“进入等待”之间搭桥，但仍要遵守持锁调用和循环检查等使用协议。

## Condition、Lock 与状态谓词

`Condition` 不能脱离创建它的 `Lock` 独立工作。通常由 `Lock.newCondition()` 创建：

```java
Lock lock = new ReentrantLock();
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();
```

这里形成了三个不同但相关的对象：

- `lock` 保护共享状态以及所有依赖该状态的复合操作。
- `notEmpty` 表示一组正在等待“队列非空”的线程。
- `notFull` 表示一组正在等待“队列非满”的线程。

条件名称不是状态本身。`notEmpty` 不会自动知道队列是否为空，也不会在队列变化后自动通知线程。真正的条件是由共享字段计算出的布尔表达式，也称为条件谓词。例如：

```java
count > 0
count < items.length
closed || count > 0
```

一个可靠的条件等待协议必须明确以下对应关系：

| 协议要素 | 有界队列中的含义 |
| --- | --- |
| 受保护状态 | 数组、读写下标、元素数量、关闭标记 |
| 保护状态的锁 | 同一个 `ReentrantLock` 实例 |
| 条件谓词 | `count > 0`、`count < capacity` 等 |
| 等待队列 | `notEmpty`、`notFull` |
| 使条件成立的状态变更 | 放入元素、取走元素、关闭队列 |
| 通知动作 | 对相应 `Condition` 调用 `signal` 或 `signalAll` |

关键不是创建几个 `Condition`，而是让“谓词、状态变更和通知对象”保持一致。若消费者等待 `notEmpty`，生产者成功增加元素数量后就必须考虑通知 `notEmpty`；若关闭状态也能让消费者停止等待，关闭操作同样必须通知相关条件。只修改状态而不通知，会让等待线程即使逻辑上已经可以继续，也可能一直沉睡；通知了错误的条件，则可能只唤醒无法取得进展的线程。

`Condition` 的方法通常要求当前线程持有与它关联的锁。以 `ReentrantLock` 创建的条件为例，不持锁调用 `await`、`signal` 或 `signalAll` 会抛出 `IllegalMonitorStateException`。这个要求不是形式限制，而是为了保证条件检查、状态修改、等待和通知都处于同一个同步协议中。

## 两类队列：条件队列与同步队列

理解 `Condition` 时，最容易混淆的是“被通知”和“继续执行”。对于基于 `AbstractQueuedSynchronizer` 的典型锁实现，可以用两类等待队列建立心智模型：

- 同步队列保存正在等待获取锁的线程。
- 每个 `Condition` 拥有自己的条件等待队列，保存已经释放锁并等待某个条件的线程。

线程调用 `await` 前正在持有锁。进入等待后，它不再持有锁，并位于对应的条件队列中。另一个线程持锁修改状态并调用 `signal` 时，被选中的等待节点并不会直接越过锁继续运行，而是从条件队列转移到锁的同步队列，等待重新获取锁。只有重新成功获得锁后，`await` 才能返回，线程才能继续检查和操作共享状态。

```mermaid
flowchart LR
    A["线程持有锁并检查谓词"] --> B{"谓词成立？"}
    B -- "是" --> C["执行受保护操作"]
    B -- "否" --> D["await：加入条件队列并释放锁"]
    D --> E["条件队列中等待"]
    F["其他线程持锁修改状态"] --> G["signal 或 signalAll"]
    G --> H["等待节点转移到同步队列"]
    H --> K["通知线程在 finally 中释放锁"]
    K --> I["等待者重新竞争并获得锁"]
    I --> A
    C --> J["finally 中释放锁"]
```

图中有两个不能省略的环节。第一，等待线程必须释放锁，否则改变条件的线程无法进入临界区。第二，收到通知后必须重新获取锁，因为共享状态只能在锁保护下可靠检查，而且从通知发生到线程真正拿到锁之间，状态可能再次被其他线程改变。

这种队列迁移也解释了几个常见现象：

1. `signal` 返回后，被通知线程通常不会立即执行。发出通知的线程仍然持有锁，至少要等它执行 `unlock`，等待者才有机会获取锁。
2. 被通知线程恢复的先后顺序不能简单等同于业务顺序。它还受锁的竞争策略、调度时机、取消和其他竞争者影响。
3. 一次通知只表示“状态可能发生了有利变化”，不等于把某个业务资源直接交付给特定线程。
4. 条件队列不是独立的执行器，也不保存业务事件；它只保存等待者。没有等待者时调用 `signal`，通知不会为未来线程积累成一个可消费凭证。

最后一点尤其重要。`Condition` 不是信号量，也没有“许可”记忆。正确性应建立在受锁保护的状态上，而不是建立在通知次数上。即使某次 `signal` 发生时没有等待线程，只要状态已经被正确修改，后续线程拿到锁后检查谓词，仍应能够直接继续；如果程序必须依赖“记住一次通知”，应把该事实编码进共享状态，或者使用能表达许可、任务或消息的更高层组件。

## await 的完整语义

`Condition` 提供多种等待形式。它们共享“释放锁、等待、重新获取锁、返回前仍持有锁”的核心过程，但在中断和超时方面有所不同。

### 可中断等待：await

`await()` 是最常用的等待方法，声明抛出 `InterruptedException`。典型写法是：

```java
lock.lockInterruptibly();
try {
    while (!conditionPredicate()) {
        condition.await();
    }
    performAction();
} finally {
    lock.unlock();
}
```

这里用 `lockInterruptibly()` 只是为了让“等待获取锁”和“条件等待”两个阶段都能够响应中断；也可以根据调用契约使用普通 `lock()`。无论如何，`await()` 本身应位于 `try/finally` 保护的持锁区域内。

如果线程在等待期间被中断，`await()` 可以抛出 `InterruptedException`。按 Java 中断约定，异常抛出时中断状态通常已经被清除。当前方法必须明确选择策略：向上抛出，让更高层决定取消；完成必要清理后返回；或者在无法抛出受检异常的边界恢复中断状态。不能无条件捕获后忽略，否则上层发出的取消或关闭请求会丢失。

还要注意：`await` 在抛出 `InterruptedException` 前会重新获取锁，因此 `finally` 中的 `unlock` 仍然成立。等待期间锁已经暂时释放，但方法恢复或抛异常时，调用线程重新成为锁持有者。这个保证让调用者能够在一致状态下处理等待结果或清理工作。

### 不响应中断退出：awaitUninterruptibly

`awaitUninterruptibly()` 不会因为中断而抛出 `InterruptedException`。线程被中断后仍继续等待，直到被通知并重新获得锁；返回时会保留中断状态。

它适合极少数必须完成某个不可分割协调阶段、当前层不能接受提前退出的协议。但“不抛异常”不等于“更稳妥”。如果系统依靠中断关闭工作线程，使用该方法可能显著延迟退出，甚至在没有后续通知时永久等待。默认应优先设计可中断等待；只有调用契约明确要求推迟中断处理，且能够证明最终一定有唤醒路径时，才考虑不可中断版本。

### 相对超时：awaitNanos 与 await(time, unit)

`await(long time, TimeUnit unit)` 返回布尔值，通常以 `false` 表示在条件通知生效前超时。`awaitNanos(long nanosTimeout)` 返回估算的剩余纳秒数，特别适合在循环中维护总超时预算：

```java
long remaining = unit.toNanos(timeout);
lock.lockInterruptibly();
try {
    while (!conditionPredicate()) {
        if (remaining <= 0L) {
            return false;
        }
        remaining = condition.awaitNanos(remaining);
    }
    performAction();
    return true;
} finally {
    lock.unlock();
}
```

超时等待仍然必须使用 `while`。一次等待可能因为通知、虚假唤醒或中断以外的实现原因提前返回，而返回时谓词未必成立。`awaitNanos` 返回剩余预算，使代码可以把多次等待限制在同一个总体期限内。若每轮都重新传入原始超时时间，虚假唤醒或无效通知会让实际总等待时间不断延长。

超时也不是严格的实时保证。线程可能在期限附近从等待队列转入同步队列，却因为迟迟拿不到锁而在更晚时刻才返回。调度暂停和锁竞争同样会影响实际耗时。因此，超时应被理解为取消等待的一项边界条件，而不是精确到纳秒的执行承诺。

### 绝对期限：awaitUntil

`awaitUntil(Date deadline)` 使用绝对时间点作为截止期限。它在需要与已有的墙上时钟期限协议对接时有用，但墙上时钟可能被系统时间调整影响。对单进程内部的相对等待预算，基于单调时间思想的 `awaitNanos` 通常更容易推理；对外部接口已经提供绝对截止时间的情况，则可以在明确时钟语义后使用 `awaitUntil`。

无论使用哪一种超时形式，超时返回都只表示当前调用不再继续等待。它不会自动撤销已经发生的其他状态变化，也不会替调用者完成业务回滚。调用者需要定义“超时后返回什么、是否重试、是否传播取消、是否记录诊断信息”。

## 为什么必须用 while 检查条件

条件等待的标准形式是：

```java
while (!predicate()) {
    condition.await();
}
```

而不是：

```java
if (!predicate()) {
    condition.await();
}
```

使用循环至少有四个相互独立的理由。

### 虚假唤醒

`Condition` 接口允许等待方法发生虚假唤醒，即没有对应业务通知时也可能返回。规范允许实现对底层平台等待机制做合理映射，调用者必须把等待放在循环中，不能假设每次返回都代表条件已经成立。

### 多个线程竞争同一份状态

假设两个消费者都等待队列非空，生产者放入一个元素并唤醒等待者。即使多个线程因 `signalAll` 醒来，最终也只有一个线程能够先拿到锁并取走元素。第二个线程随后获得锁时，队列可能已经再次为空。若它使用 `if`，就会越过检查继续取元素，破坏队列不变式；使用 `while` 则会重新等待。

### 通知与重新获锁之间存在时间窗口

被 `signal` 选中的线程只是转入同步队列。通知线程释放锁后，其他线程可能先获得锁并改变状态。等原等待者最终拿到锁时，当初促使通知发生的状态未必仍然有效。

### 一个条件可能包含多个退出分支

实际协议常把业务可用状态和关闭状态组合起来。例如消费者等待的不是单纯的“队列非空”，而是“队列非空或者队列已经关闭”。线程醒来后必须重新判断是应该取数据、返回结束标记，还是继续等待。循环让判断逻辑始终以当前受保护状态为准。

因此，`signal` 的语义应理解为“请某些等待者重新检查”，而不是“保证条件对等待者成立”。判断权永远在重新获得锁后的谓词检查中。

## signal 与 signalAll 的选择

`signal()` 唤醒条件队列中的一个等待者，`signalAll()` 使所有等待者转移到获取锁的竞争过程。两者都必须在持有相关锁时调用。

### 适合 signal 的条件

当一次状态变化最多只允许一个同类等待者取得进展，并且同一条件队列中的等待者具有等价谓词时，可以考虑 `signal`。有界队列中放入一个元素后通知一个等待“非空”的消费者，通常符合这个模型；取走一个元素后通知一个等待“非满”的生产者，也通常如此。

“等价谓词”是使用 `signal` 的重要前提。如果同一个 `Condition` 上的线程实际等待不同条件，随便选中一个线程可能无法取得进展，而真正满足条件的线程仍在沉睡。例如，多个线程分别等待库存达到 1、10 和 100，却共用一个条件队列；库存从 0 变成 10 时，若 `signal` 恰好选择等待 100 的线程，它重新检查后继续等待，而等待 1 和 10 的线程没有机会运行。此时应重新设计条件划分，或在状态变化时使用 `signalAll` 让每个线程自行检查。

### 适合 signalAll 的条件

以下情况更倾向于使用 `signalAll`：

- 一次状态变化可能允许多个等待者同时或依次取得进展。
- 同一条件队列上存在不同谓词，无法安全确定哪一个等待者应被选择。
- 状态进入终止、关闭或失败阶段，需要确保所有等待者都能退出。
- 正确性比减少竞争更重要，且尚未证明单个通知不会造成活性问题。

`signalAll` 的代价是“惊群”：大量线程被转移到同步队列，随后串行获取锁、检查条件，大部分可能再次等待。这会增加上下文切换和锁竞争。不过，不能为了性能在没有证明的情况下把 `signalAll` 替换为 `signal`。通知策略首先要保证活性和协议完整，再根据等待者规模、状态变化频率和实际测量优化。

### 通知放在状态变更之后

通常应先在锁内完成状态修改，再通知相关条件：

```java
lock.lock();
try {
    updateSharedState();
    condition.signal();
} finally {
    lock.unlock();
}
```

由于通知和状态修改都在同一锁保护下，等待线程即使已经被转移，也要等通知线程释放锁后才能检查到完成后的状态。把通知放在修改之前，会使协议意图混乱；更严重的是，如果状态修改随后失败，等待者可能被无效唤醒。虽然循环检查可以避免错误地继续执行，但多余通知会降低可读性和效率。

通知后通常不需要立刻 `unlock` 之外的特殊操作，但应避免在持锁状态继续执行耗时 I/O、未知回调或大规模计算。被通知线程只有在锁释放后才能进展，过长的临界区会把通知到实际响应之间的延迟放大。

## 完整案例：可关闭的有界缓冲区

下面的实现展示了两个条件队列、可中断等待、关闭协议以及必要的状态不变式。它不是为了替代标准库中的 `BlockingQueue`，而是用来呈现一个完整 `Condition` 协议应包含哪些部分。

```java
import java.util.ArrayDeque;
import java.util.Deque;
import java.util.Objects;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public final class CloseableBoundedBuffer<E> {
    private final int capacity;
    private final Deque<E> queue = new ArrayDeque<>();
    private final Lock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();

    private boolean closed;

    public CloseableBoundedBuffer(int capacity) {
        if (capacity <= 0) {
            throw new IllegalArgumentException("capacity must be positive");
        }
        this.capacity = capacity;
    }

    public void put(E element) throws InterruptedException {
        Objects.requireNonNull(element, "element");
        lock.lockInterruptibly();
        try {
            while (queue.size() == capacity && !closed) {
                notFull.await();
            }
            if (closed) {
                throw new IllegalStateException("buffer is closed");
            }

            queue.addLast(element);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public E take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (queue.isEmpty() && !closed) {
                notEmpty.await();
            }
            if (queue.isEmpty()) {
                return null;
            }

            E element = queue.removeFirst();
            notFull.signal();
            return element;
        } finally {
            lock.unlock();
        }
    }

    public void close() {
        lock.lock();
        try {
            if (closed) {
                return;
            }
            closed = true;
            notEmpty.signalAll();
            notFull.signalAll();
        } finally {
            lock.unlock();
        }
    }

    public int size() {
        lock.lock();
        try {
            return queue.size();
        } finally {
            lock.unlock();
        }
    }
}
```

这个例子中，`queue` 和 `closed` 都受同一把锁保护。`closed` 没有声明为 `volatile`，因为所有读写都在锁内；混用额外的可见性机制不仅没有必要，还容易让维护者误以为可以在锁外读取并据此做复合决策。

`put` 等待的谓词不是单纯的“队列未满”，而是“队列未满或者已经关闭”。关闭后即使队列仍然满，生产者也必须从等待中退出并失败，不能继续永久等待空间。因此 `close` 要通知 `notFull` 上的全部生产者。

`take` 允许关闭后排空已有元素。当队列为空且未关闭时等待；关闭后，如果仍有元素就继续取，如果已经为空则返回 `null` 表示流结束。这个约定只是示例选择，真实 API 也可以返回 `Optional<E>`、抛出特定异常或使用单独结果类型，但必须避免在允许存储 `null` 时用 `null` 同时表示结束。示例通过 `Objects.requireNonNull` 排除了这种歧义。

普通放入只增加一个元素，因此调用 `notEmpty.signal()`；普通取出只释放一个槽位，因此调用 `notFull.signal()`。关闭则改变所有等待者的退出条件，所以两个条件都使用 `signalAll()`。如果关闭时只通知消费者，等待在满队列上的生产者将无法观察 `closed`，形成关闭泄漏。

该实现也体现了“通知不是状态”。真正持久的事实是 `closed`、队列内容和容量关系。即使调用 `close` 时没有等待者，后来的 `put` 和 `take` 仍能通过检查状态得到正确结果。

在生产代码中，如果需求只是一个普通有界阻塞队列，应优先使用 `ArrayBlockingQueue`、`LinkedBlockingQueue` 等成熟实现。手写版本的价值主要在于需要额外协议，例如显式关闭、特殊批处理、多个业务状态协同，并且这些需求无法通过组合标准组件清晰表达时。

## 带总超时预算的获取

超时接口常见的错误是每次唤醒都重新等待完整时长。下面的方法在前述缓冲区语义基础上，展示怎样维护一次调用的总预算：

```java
public E poll(long timeout, TimeUnit unit) throws InterruptedException {
    Objects.requireNonNull(unit, "unit");
    long remaining = unit.toNanos(timeout);

    lock.lockInterruptibly();
    try {
        while (queue.isEmpty() && !closed) {
            if (remaining <= 0L) {
                return null;
            }
            remaining = notEmpty.awaitNanos(remaining);
        }
        if (queue.isEmpty()) {
            return null;
        }

        E element = queue.removeFirst();
        notFull.signal();
        return element;
    } finally {
        lock.unlock();
    }
}
```

要使这段方法位于可编译上下文中，需要把它加入前一个类，并补充 `java.util.concurrent.TimeUnit` 导入。它把“超时”和“关闭且已排空”都编码为 `null`，接口较简洁但无法区分原因。如果调用方需要区别超时、关闭和成功，应返回明确的结果类型，而不是再增加锁外查询，因为锁外查询会引入新的竞态。

`timeout <= 0` 时，方法仍会先获得锁并检查当前状态：若已有元素，可以立即返回元素；只有当前确实无元素时才返回超时。这符合“最多等待给定时长”而不是“非正值必定失败”的常见语义。具体 API 也可以选择不同契约，但应在文档和测试中明确。

## 多条件状态机案例：单资源的暂停、恢复与关闭

`Condition` 不只用于队列。只要多个操作依赖同一组锁保护状态，并且需要等待不同谓词，就可以把对象建模为状态机。例如一个串行任务入口允许正常运行、暂停接收以及永久关闭：

```java
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

public final class PausableGate {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition resumedOrClosed = lock.newCondition();

    private boolean paused;
    private boolean closed;

    public void awaitOpen() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (paused && !closed) {
                resumedOrClosed.await();
            }
            if (closed) {
                throw new IllegalStateException("gate is closed");
            }
        } finally {
            lock.unlock();
        }
    }

    public void pause() {
        lock.lock();
        try {
            if (!closed) {
                paused = true;
            }
        } finally {
            lock.unlock();
        }
    }

    public void resume() {
        lock.lock();
        try {
            if (paused && !closed) {
                paused = false;
                resumedOrClosed.signalAll();
            }
        } finally {
            lock.unlock();
        }
    }

    public void close() {
        lock.lock();
        try {
            if (!closed) {
                closed = true;
                resumedOrClosed.signalAll();
            }
        } finally {
            lock.unlock();
        }
    }
}
```

这里多个等待者都依赖同一个谓词 `!paused || closed`，恢复和关闭都可能让全部等待者结束等待，所以使用 `signalAll`。`pause` 不需要通知，因为它只会让继续执行的条件从真变成假；通知通常与“某些等待者可能从不能继续变为能够继续”相关。

这个例子也展示了条件命名应体现“等待什么变化”或“什么状态允许继续”。若把条件简单命名为 `condition`，代码审查时很难验证 `signal` 是否发给了正确等待队列。对于复杂状态机，条件名称、谓词表达式和状态变更方法应能互相对应。

但还要看清它的边界：`awaitOpen` 返回后，调用方不再持有门对象内部的锁。另一线程可以立刻再次调用 `pause`。因此该方法只保证“调用返回前曾在锁保护下观察到可通过状态”，不保证调用方后续任意操作期间门始终开启。如果业务需要把“检查可通过”和“执行资源操作”绑定成一个原子过程，就必须把资源操作也纳入同一锁协议，或返回一个代表持有权的令牌。条件等待不能自动延长状态有效期。

## 中断、取消与关闭必须统一设计

等待线程离开条件队列通常有四类原因：条件通知、中断、超时和对象生命周期结束。前三类由 API 直接参与，最后一类必须由业务状态和通知共同表达。

### 不要吞掉 InterruptedException

下面的写法破坏了中断协议：

```java
try {
    condition.await();
} catch (InterruptedException ignored) {
}
```

它既没有完成取消，也没有恢复中断状态，外层代码可能继续执行并误以为当前操作正常。更合理的策略取决于方法边界：

- 方法本身允许抛出 `InterruptedException` 时，通常直接向上传播。
- 当前层必须完成一致性清理时，在持锁状态完成清理后再传播。
- 当前接口不能声明受检异常且决定提前返回时，通常调用 `Thread.currentThread().interrupt()` 恢复中断，再返回或抛出领域异常。
- 只有协议明确规定忽略本次中断并继续等待时，才推迟处理中断，而且应保留中断事实。

中断不是强制终止线程，它只是协作式请求。`Condition` 能让阻塞等待响应请求，但调用栈仍要传播并尊重它。

### 关闭状态必须进入谓词

仅仅设置 `closed = true` 不足以释放已经等待的线程；仅仅 `signalAll` 而没有关闭状态，也不能让等待者知道为什么应该退出。完整关闭协议需要同时完成：

1. 在锁内修改持久的关闭状态。
2. 通知所有可能因关闭而改变决策的条件队列。
3. 等待者醒来后在循环或分支中检查关闭状态。
4. 对关闭前已经存在的数据、正在进行的操作和后续调用给出一致语义。

如果对象有多个条件，关闭时应逐个分析。某个条件队列是否有线程可能永久等待？关闭状态是否能使其退出？是否需要全部唤醒？遗漏任何一类等待者都会让线程泄漏或关闭流程卡住。

### 超时不是关闭的替代品

给 `await` 加一个超时可以避免单次无限等待，但不能替代对象生命周期协议。若等待者每次超时后无条件重试，它仍可能永远存在；若超时被当作正常关闭，调用方又无法区分暂时拥塞和永久终止。超时用于限制一次调用的等待成本，关闭用于定义对象不再接受或产生工作的永久状态，两者解决的问题不同。

## 内存语义与锁边界

`Condition` 的可见性并不来自“通知”这个词本身，而来自它与锁协议的组合。线程在锁内修改共享状态，随后解锁；等待线程从 `await` 返回前重新获取同一把锁。按照 `Lock` 的内存同步语义，前一个线程在释放锁之前的写入，对后续成功获取同一锁的线程可见。

因此，条件谓词涉及的字段通常都应由关联锁保护。不要把检查拆成锁外快照和锁内等待：

```java
if (!ready) {
    lock.lock();
    try {
        condition.await();
    } finally {
        lock.unlock();
    }
}
```

这段代码在锁外读取 `ready`，并在读取与等待之间留下窗口。状态可能已经变为 `true` 且通知已经发生，随后线程才进入等待。正确做法是先获得锁，再在循环内检查和等待。

同样，不要因为某个字段声明了 `volatile` 就在锁外完成复合检查。`volatile` 能提供单次读写的可见性和相应有序性，但不能把“检查谓词、决定等待、修改多个字段”组合成原子协议。若 `Condition` 依赖多个字段维持不变式，统一在同一把锁内访问更容易证明。

这也意味着所有能够使谓词成立的状态修改路径都必须遵守同一把锁。如果某个“快捷路径”绕过锁修改字段，即使字段值最终可见，也可能与等待入队、其他字段更新或通知时机失去原子关系。条件协议的安全边界是整组状态转换，不是某一个布尔标记。

## Condition 与 Object.wait/notify 的关系

`Condition` 与内置监视器的 `wait`、`notify`、`notifyAll` 有相似的基本模式：

- 调用等待前必须持有相关锁。
- 等待会释放锁。
- 返回前会重新获得锁。
- 等待必须放在循环中检查谓词。
- 通知不会让等待者立即越过锁执行。

两者最重要的差异是组织能力。每个 Java 对象监视器只有一个等待集合；一个 `Lock` 可以创建多个 `Condition`，把不同谓词的等待者分开。`Condition` 还提供可中断锁获取、多个超时等待形式，以及与显式锁实现策略配合的能力。

但 `Condition` 并不是 `wait/notify` 的无条件升级。若代码只需要简单的互斥和单一条件，`synchronized` 的词法作用域与自动释放通常更简洁，也不容易忘记在 `finally` 中解锁。选择显式锁和条件变量，应有明确需求，例如多个条件队列、可中断获取锁、定时获取锁，或与现有 `Lock` 协议集成。

两套机制不能交叉替代。对 `Condition` 调用 `await` 释放的是关联的 `Lock`，不是某个对象监视器；对对象调用 `wait` 需要持有该对象监视器，也不会释放另一个 `ReentrantLock`。混用会形成两个彼此无关的同步域，使谓词和通知难以建立正确关系。

## 与更高层并发工具的权衡

`Condition` 表达能力很强，但表达能力越强，调用者承担的证明责任越多。选择工具时应先寻找能够直接表达问题的标准组件。

### BlockingQueue

生产者与消费者、容量限制、阻塞放入和获取已经由 `BlockingQueue` 完整表达。标准实现处理了等待队列、中断、超时和并发可见性。除非需要标准接口无法表达的状态机或原子组合操作，否则不应为了练习之外的目的手写条件队列。

### Semaphore

如果问题是一定数量的可复用许可，例如最多允许若干线程同时访问资源，`Semaphore` 更直接。许可可以在没有等待者时积累，而 `Condition.signal` 不保存通知。用条件变量模拟许可需要自己维护计数、上下界和取消恢复，容易出错。

### CountDownLatch、CyclicBarrier 与 Phaser

阶段完成、参与者汇合和多阶段推进有专门同步器。它们把计数或阶段转换封装为稳定协议。`Condition` 可以手工实现类似行为，但还要处理重复使用、破损状态、参与者变化和异常传播，通常没有必要。

### Future、CompletableFuture

等待单个异步结果及其异常更适合使用结果抽象。条件变量只表示“重新检查共享状态”，不会自动保存返回值、异常或取消原因。若使用 `Condition` 传递结果，必须额外维护完成标记、结果字段和异常字段，本质上是在重新实现 Future 的一部分。

### LockSupport

`LockSupport.park/unpark` 是更底层的阻塞原语，许可语义与 `Condition` 不同，也不自动提供锁和状态谓词协议。实现同步器或定制调度结构时可能直接使用它；一般业务代码更适合使用 `Condition` 或更高层组件。不能把 `park/unpark` 当成去掉持锁要求的 `await/signal`。

判断是否需要 `Condition`，可以问三个问题：是否存在由同一组共享状态导出的一个或多个谓词？检查谓词与状态转换是否必须在同一互斥边界内完成？标准并发组件是否无法直接表达所需协议？只有前两项成立且第三项确实无法满足时，手写条件协作才通常值得。

## 公平性、顺序与性能

`Condition` 接口没有承诺所有实现都采用相同的等待顺序，也不应把业务正确性建立在某个线程必然先被唤醒上。具体实现可以提供不同保证；例如由 `ReentrantLock` 创建的条件具有其实现定义的队列行为，但线程从条件队列转移后仍需竞争锁。锁是否公平、同步队列中是否还有其他竞争者、线程是否被取消以及运行时调度，都会影响最终获得锁的顺序。

即使使用公平 `ReentrantLock`，也不等于获得完整的业务公平：

- 公平锁主要约束锁竞争次序，不保证线程何时被操作系统调度。
- 条件等待者在收到通知前没有资格竞争锁。
- 不同 `Condition` 之间没有天然的全局业务顺序。
- 非阻塞快速路径、超时和中断可能改变观察到的先后关系。

如果业务要求严格按请求序号、优先级或截止期限处理，应把顺序显式编码在队列和调度策略中，而不是从 `signal` 或公平锁推导。

性能方面，多条件队列的主要优势是减少无关唤醒。例如队列由满变为非满时，只通知生产者，不必唤醒消费者。实际收益取决于等待者数量、临界区长度、状态变化频率和竞争程度。过细拆分条件也有成本：状态转换必须正确通知更多队列，关闭逻辑更复杂，遗漏某个通知的风险上升。因此，条件划分应围绕真正不同的谓词，而不是为每个方法机械创建一个条件。

持锁时间往往比 `signal` 与 `signalAll` 的微小选择更值得优先关注。不要在锁内执行不可控回调、阻塞 I/O、长时间计算或可能递归调用当前对象的外部代码。`Condition` 只能协调等待，无法消除过大临界区带来的延迟、死锁和吞吐问题。

## 常见错误及其根因

### 未持锁调用 await 或 signal

这会破坏检查、入队、状态修改和通知之间的原子关系。典型实现会抛出 `IllegalMonitorStateException`。修复方式不是捕获异常，而是重新建立统一的锁协议。

### 用 if 代替 while

根因是把“收到通知”误解为“条件已经保证成立”。虚假唤醒、多线程竞争以及通知到重新获锁之间的状态变化都会使这个假设失效。

### 修改状态后忘记通知

条件已经成立但等待者仍在条件队列中。若没有其他状态变化带来后续通知，线程可能永久等待。应列出每个谓词从假变真的全部状态转换，并逐一检查通知。

### 通知错误的 Condition

多个条件虽然绑定同一把锁，但等待队列相互独立。生产者释放了空间却通知 `notEmpty`，不会唤醒等待空间的生产者。清晰命名和把通知紧邻状态转换有助于审查。

### 把 signal 当作持久事件

没有等待者时通知不会积累。如果未来线程必须观察某个事实，就要把事实写入受保护状态；如果需要积累许可或消息，则选择 `Semaphore` 或队列。

### 只在正常路径通知

异常、失败或关闭也可能改变等待者的决策。例如对象进入不可恢复错误状态后，等待者应被全部唤醒并看到失败原因。否则发起失败的线程已经退出，其他线程仍在等待一个永远不会成立的正常条件。

### 在多个锁下访问同一谓词

若写线程用锁 A 修改字段并通知，而等待线程用锁 B 检查字段，两个操作不属于同一互斥与可见性协议。即使两个锁都是 `ReentrantLock`，也不能互相替代。

### 捕获中断后继续假装成功

中断说明等待被取消或上层要求线程尽快退出。吞掉异常后执行依赖条件成立的代码，可能直接破坏不变式。中断路径也必须先重新检查或明确退出，不能把异常当成一次普通通知。

### 关闭时只设置标记

已经在 `await` 中的线程不会因为普通字段变化自动醒来。关闭状态与 `signalAll` 必须配对，并覆盖所有相关条件队列。

### 在持锁状态调用未知代码

回调可能耗时、阻塞或反向调用当前对象，导致响应延迟或锁顺序死锁。应在锁内只完成状态快照和必要转换，释放锁后再执行不需要原子性的外部逻辑。

## 如何审查一个 Condition 协议

面对一段条件等待代码，可以按状态而不是按方法逐行审查。

首先列出全部共享字段和不变式。例如容量范围、状态枚举之间的合法转换、关闭后是否允许排空、失败后结果字段是否仍可读取。若不变式无法清楚写出，条件谓词通常也不会可靠。

然后列出每个 `Condition` 对应的精确谓词。不要只写“等待数据”或“等待完成”，而应写出类似 `count > 0 || closed`、`state == COMPLETED || state == FAILED` 的逻辑。确认所有谓词涉及的字段都由同一把关联锁保护。

接着枚举每个可能使谓词从假变真的状态转换，检查它是否在持锁状态通知了正确条件。还要枚举终止转换：关闭、失败、取消是否需要唤醒多个条件上的全部线程。

再检查每个等待点：

- 调用前是否持有关联锁。
- 是否使用 `while`。
- 返回后是否重新根据当前状态分支。
- 中断是否传播或恢复。
- 超时是否维护总预算。
- `finally` 是否无条件解锁。

最后检查锁外行为。返回值是否暴露内部可变状态？等待方法返回后，调用方是否错误地假设谓词会持续成立？是否在持锁时调用未知代码？是否存在绕过锁的“优化读取”？这些问题往往不出现在 `await` 那一行，却决定整个协议是否成立。

## 排查永久等待与高竞争

当线程长时间不再推进时，先区分它是在条件队列等待，还是在同步队列等待获取锁。线程转储中，二者都可能表现为等待状态，但调用栈位置不同：停在 `Condition.await` 附近通常是在等待条件；停在锁获取路径通常表示条件可能已经收到通知，但线程尚未获得锁。

排查永久条件等待时，可沿以下顺序：

1. 写出线程等待的实际谓词，而不是只看条件变量名称。
2. 找到所有能使谓词成立或使等待终止的状态转换。
3. 检查这些转换是否都使用同一把锁。
4. 检查它们是否通知正确的条件，异常和关闭路径是否遗漏。
5. 检查等待者醒来后是否因其他线程消费状态而再次等待。
6. 检查通知线程是否长期持锁，导致等待者已经转移却无法重新获取锁。

若表现为 CPU 抖动和频繁上下文切换，则关注 `signalAll` 是否过度、多个不同谓词是否错误共用一个条件、临界区是否过长，以及线程数量是否远大于可取得进展的资源数量。优化前应先确认协议正确，再通过指标或压力测试判断瓶颈，不能根据单次日志顺序推断。

测试 `Condition` 代码时，不应依赖固定 `sleep` 期待线程“应该已经等待”。更稳定的测试会使用门闩、屏障或可观测状态建立测试步骤，并为最终等待设置超时，防止测试套件永久卡住。测试至少应覆盖：

- 条件初始成立时不等待。
- 条件不成立后由另一个线程使其成立。
- 多个同类等待者竞争有限资源。
- 等待期间中断。
- 超时前成立和超时后仍不成立。
- 关闭时各条件队列都有等待者。
- 状态变更或回调抛异常时锁仍被释放。

并发测试无法单独证明所有执行交错都正确，但它能验证协议中的关键路径。最终仍需要依据锁边界、谓词和状态转换进行推理。

## 设计准则

一个可维护的 `Condition` 设计通常满足以下准则：

1. **先定义状态，再定义条件。** 条件变量只是等待者分类，正确性来自清晰的不变式和状态机。
2. **一个谓词只有一个同步域。** 检查、修改、等待和通知都使用创建该条件的同一把锁。
3. **等待永远重新检查。** `await` 返回只意味着线程获得了再次判断的机会。
4. **通知跟随状态转换。** 在锁内先完成状态修改，再通知可能取得进展的等待者。
5. **关闭和失败也是状态。** 它们必须进入谓词，并唤醒所有需要退出的线程。
6. **中断和超时是接口契约。** 不吞中断，不把每轮超时重置为完整预算，不用超时冒充关闭。
7. **不用通知次数保存业务事实。** 需要持久化的事实放进状态，需要积累的许可或消息交给相应并发组件。
8. **优先使用高层工具。** 只有标准同步器不能清楚表达协议时，才承担手写条件协作的复杂度。
9. **不从公平锁推导业务公平。** 严格顺序必须由显式排队与调度规则保证。
10. **让条件名称可以审查。** `notEmpty`、`stateChanged`、`resumedOrClosed` 比无意义的 `condition1` 更能暴露通知错误。

## 小结

`Condition` 把“持锁但条件不成立”的线程从忙等和时间猜测中解放出来，并允许一把 `Lock` 管理多个精确等待队列。它的标准协议是：持锁检查谓词；不满足时通过 `await` 释放锁并等待；其他线程在同一锁内修改状态并通知；等待者重新获取锁后用循环再次检查。

理解其机制时，要区分条件队列和锁的同步队列。`signal` 只是把等待者送回锁竞争过程，不会立即执行线程，也不会保证谓词仍然成立，更不会保存为未来可消费的事件。正确性依赖持久的受保护状态、同一把锁以及完整的状态转换通知。

在实践中，最难的部分通常不是调用 API，而是定义边界：哪些字段组成不变式，哪些变化使哪个谓词成立，中断和超时如何传播，关闭和失败怎样释放所有等待者，`signal` 是否足够，方法返回后状态保证能持续多久。只有这些问题形成闭环，`Condition` 才是精确的线程协作工具；否则，它只是把竞态从忙等位置移动到了更隐蔽的等待队列中。

# 3.3.4.3 CountDownLatch

`CountDownLatch` 是 Java 标准并发库提供的一次性同步器。它维护一个只能递减、不能重置的计数：一个或多个线程可以调用 `await()` 等待，其他线程在某些事件完成后调用 `countDown()` 递减计数；当计数第一次变为零时，所有正在等待的线程都可以继续执行，此后新调用 `await()` 的线程也会立即通过。

这个工具解决的不是“让若干线程同时运行”，也不是“收集异步任务结果”，而是建立一个明确的阶段边界：在约定数量的完成信号到达以前，等待方不能越过边界；信号全部到达以后，边界永久开放。因此，它最适合表达“一批前置事件完成后再继续”的一次性协作，例如主线程等待多个初始化任务结束、测试线程等待若干并发操作抵达观察点，或者多个工作线程等待统一的开始信号。

理解 `CountDownLatch` 的关键不在三个方法的调用形式，而在四个问题：

1. 初始计数代表什么事件，谁有权递减它；
2. 每个逻辑事件是否恰好贡献一次 `countDown()`；
3. 等待有无超时、中断和失败传播策略；
4. 计数归零建立了什么内存可见性关系，又没有解决哪些共享状态问题。

## 一次性门闩模型

门闩可以看成一道只有“关闭”和“打开”两个阶段的门，但关闭阶段内部还保存着一个剩余计数。构造函数 `new CountDownLatch(n)` 将剩余计数设为 `n`。当 `n > 0` 时门闩关闭，`await()` 会使调用线程等待；每次 `countDown()` 都尝试让剩余计数减一；当最后一次有效递减把计数从一变为零时，门闩打开并释放等待者。

门闩一旦打开就不会再次关闭。计数到零以后：

- `await()` 立即返回；
- 带超时的 `await(timeout, unit)` 立即返回 `true`；
- 再调用 `countDown()` 没有效果，计数不会变成负数；
- `getCount()` 持续返回零；
- 对象不能设置新计数，也不能开始下一轮。

“一次性”是它与许多阶段同步工具最重要的区别。`CountDownLatch` 的状态变化方向只有：

`正数 -> 更小的正数 -> 0`

不存在 `0 -> 正数`，也不存在把计数加回去的操作。这个限制使模型很简单：等待方只关心一个单调条件“计数是否已经为零”，不需要处理阶段编号、重复使用或参与者动态变化。但代价也很明确：只要业务需要周期性地汇合、反复开关或者动态登记任务，`CountDownLatch` 就不再是合适的核心抽象。

门闩与锁也不是同一种东西。锁保护临界区，要求线程获得所有权后才能访问共享状态，并在使用结束后释放；`CountDownLatch` 没有所有者，不提供互斥，也不限制谁调用 `countDown()`。任意持有引用的线程都可以递减计数，调用者无须先“获得门闩”。所以门闩能够约束阶段顺序，却不能阻止两个工作线程同时修改同一个非线程安全对象。

## 计数到底应该表示什么

初始计数必须对应可枚举、可核对的逻辑事件。常见设计有两类：

- 按参与任务计数：提交三个任务，每个任务结束时递减一次，初始值为三；
- 按完成事件计数：某流程要求“配置加载完成”“缓存预热完成”“连接建立完成”三个事件，初始值也为三，即使这些事件不一定由三个不同线程完成。

两种设计都成立，但同一个门闩中不能含糊地混用“线程数”“任务数”“事件数”。线程只是执行载体，线程池中的一个线程可以顺序执行多个任务；一个任务也可能产生多个逻辑完成事件。若构造门闩时按线程池大小计数，执行时却按任务递减，计数就与真实完成条件脱节。

可以把计数协议写成一个守恒关系：

> 初始计数 = 运行期间应该发生的有效完成信号总数。

每个信号应恰好发生一次。少一次，门闩永远无法归零；多一次，额外递减虽然不会让计数变为负数，却可能掩盖协议错误。例如初始值为二，任务 A 错误地递减两次，任务 B 尚未完成时门闩已经打开。`CountDownLatch` 只保证递减操作本身线程安全，不会识别“这次递减属于哪个任务”，更不会验证业务事件是否真的完成。

因此，最稳妥的编码结构通常是让一次任务执行对应一个 `try-finally`：

```java
import java.util.concurrent.CountDownLatch;

public final class Worker implements Runnable {
    private final CountDownLatch done;

    public Worker(CountDownLatch done) {
        this.done = done;
    }

    @Override
    public void run() {
        try {
            performWork();
        } finally {
            done.countDown();
        }
    }

    private void performWork() {
        // 执行该任务负责的工作
    }
}
```

`finally` 保证的是“任务已经结束，因此交付完成信号”，而不必然表示“任务成功”。这一区分很重要。如果等待方的条件是“所有任务都已经终止，无论成功或失败”，那么把 `countDown()` 放在 `finally` 中是正确的；如果条件严格要求“所有任务都成功”，只用门闩就不够，因为门闩不保存成功状态和异常。此时仍应在 `finally` 中递减，避免等待永久卡住，同时通过 `Future`、并发集合或原子引用单独传播失败。

不要因为担心失败就只在成功分支调用 `countDown()`。那种写法把“失败”转换成“门闩永远不开放”，等待方通常只能看到超时，真正异常反而丢失。更清晰的协议是：门闩记录任务是否结束，另一个通道记录任务以何种结果结束。

## API 与状态语义

`CountDownLatch` 的公开 API 很小，但每个方法的边界需要准确理解。

### 构造函数

`CountDownLatch(int count)` 创建指定初始计数的门闩。负数会触发 `IllegalArgumentException`。零是合法值，表示门闩从创建时就已经打开，任何等待都不会阻塞。

初始值为零有时可用于统一空任务集合与非空任务集合的处理：当任务列表为空时，调用方仍可构造门闩并执行同一套等待逻辑。不过，如果零来自任务统计错误，它也会让等待失去作用，所以创建位置应清楚说明计数来源。

### `await()`

无参数 `await()` 在计数归零前等待。若调用时计数已经为零，它立即返回。等待过程是可中断的：线程在进入方法时已经带有中断状态，或者等待期间收到中断，方法都会抛出 `InterruptedException`，并按该异常的一般约定清除中断状态。

`await()` 的返回只说明门闩已经打开，不说明每个任务成功，不说明线程已经终止，也不说明异步资源已经关闭。门闩何时递减完全由应用协议决定；如果任务在真正收尾前过早调用 `countDown()`，等待方就可能在资源仍被使用时继续执行。

### `await(long timeout, TimeUnit unit)`

带超时的等待在计数归零时返回 `true`，在等待时间耗尽时返回 `false`，被中断时仍抛出 `InterruptedException`。调用方必须检查布尔返回值，不能把正常返回一概理解为门闩已打开。

超时小于等于零时，方法不会进行实质等待，而是立即检查当前计数是否为零。这一行为可用于非阻塞状态探测，但若只是为了观察剩余计数，`getCount()` 更直观；若要依据“是否已经开放”决定控制流，带零超时的 `await` 能直接表达该问题。

### `countDown()`

`countDown()` 不阻塞，也不抛出 `InterruptedException`。计数大于零时，它以线程安全方式递减；最后一次有效递减负责使等待者具备继续执行的条件。计数已经为零时，调用没有效果。

“没有效果”不等于重复调用是良好实践。额外调用不会损坏同步器内部状态，但常常说明上层计数协议已经出错。不要依赖计数不为负这一性质来容忍重复回调。

### `getCount()` 与 `toString()`

`getCount()` 返回调用瞬间观察到的剩余计数，主要适合诊断、日志和监控。它不是预留名额或执行复合决策的原子接口。下面这种逻辑存在检查与执行之间的竞态：

```java
if (latch.getCount() > 0) {
    latch.countDown();
}
```

多个线程可以同时看到正数并继续调用。虽然同步器本身不会出现负计数，但业务层仍无法借此实现“只有一个线程负责最后一次递减”之类的协议。若递减权需要唯一所有者，应由任务身份、原子状态或锁来保证，而不是先读 `getCount()` 再决定。

`toString()` 通常包含当前计数，可辅助日志定位，但其文本格式不应作为程序逻辑或稳定接口解析。

## 等待与递减不是对称操作

初学者容易把 `await()` 和 `countDown()` 想成类似 `lock()` 与 `unlock()` 的成对调用。实际上二者既不要求由同一线程调用，也不是一对一关系。

一个门闩可以有多个等待者。计数归零时，所有已经等待的线程都可以继续，之后到达的等待者也直接通过。等待者的数量不参与计数，增加一个等待者不需要增加初始值。

同样，一个线程可以贡献多个完成信号，只要业务确实把多个独立事件委托给它；多个线程也可以竞争着递减同一个门闩。计数描述的是尚未完成的事件数，而不是等待者数，也不是门闩的持有者数。

这种非对称性使 `CountDownLatch` 能表达两种方向相反的典型协作：

1. 一个线程等待多个工作线程完成，即“多对一汇合”；
2. 多个工作线程等待一个启动信号，即“一对多放行”。

第二种模式通常构造初始值为一的门闩。所有工作线程先调用 `await()`，协调线程准备好环境后调用一次 `countDown()`。门闩打开后所有工作线程都能通过。它能让开始时机更集中，但不保证工作线程在同一个 CPU 时刻真正开始，也不提供调度公平性。

## 内存可见性：门闩发布了什么

`CountDownLatch` 不只是避免忙等，它还提供 Java 内存模型层面的同步保证。其 API 规范给出的核心效果可以概括为：

> 在线程调用 `countDown()` 之前发生的动作，happens-before 另一个线程从对应的 `await()` 成功返回之后发生的动作。

这条关系允许工作线程先写入结果，再递减门闩；等待线程从 `await()` 返回后读取结果时，可以看到由同步关系发布的写入。典型结构如下：

```java
import java.util.ArrayList;
import java.util.List;
import java.util.concurrent.CountDownLatch;

public final class VisibilityExample {
    public static void main(String[] args) throws InterruptedException {
        int taskCount = 3;
        String[] results = new String[taskCount];
        CountDownLatch done = new CountDownLatch(taskCount);
        List<Thread> threads = new ArrayList<>();

        for (int i = 0; i < taskCount; i++) {
            int index = i;
            Thread thread = new Thread(() -> {
                try {
                    results[index] = "result-" + index;
                } finally {
                    done.countDown();
                }
            }, "worker-" + i);
            threads.add(thread);
            thread.start();
        }

        done.await();

        for (String result : results) {
            System.out.println(result);
        }
    }
}
```

每个工作线程只写自己的数组位置，避免工作线程之间对同一元素产生数据竞争；主线程在门闩打开后读取所有位置。工作线程对对应元素的写入发生在其 `countDown()` 之前，主线程的读取发生在 `await()` 返回之后，因此这些结果能够被安全观察。

这里需要避免把保证扩大化。

第一，门闩提供的是特定同步边界上的可见性与有序性，不提供任意共享操作的原子性。若两个工作线程同时执行 `counter++`，即使二者随后都调用 `countDown()`，等待方最终看到的也可能是丢失更新后的值。门闩可以保证看到某个已形成的结果，不能保证该结果在并发计算过程中没有被竞争破坏。

第二，写入必须发生在 `countDown()` 之前。若工作线程先递减，再写结果，等待方可能在写入发生以前就从 `await()` 返回。代码书写顺序和协议都必须把“发布前的工作”放在递减之前。

第三，等待方必须真正经过与该门闩对应的同步路径。另一个不调用 `await()`、只轮询普通字段的线程，不能凭借门闩存在就自动获得可见性保证。

第四，多次递减形成的是协作链条的一部分。等待线程只有在计数归零并成功返回后，才能把门闩当作整批前置动作的完成边界。在计数仍为正时读取共享结果，可能只看到部分任务的写入，且业务上本来也尚未满足“全部完成”条件。

第五，可见性不替代对象所有权。即使主线程在 `await()` 后看到了工作线程构造的可变对象，如果工作线程在递减后仍继续修改该对象，双方仍可能发生数据竞争。正确做法是把 `countDown()` 放在该任务对发布对象的最后一次修改之后，或者继续用锁、不可变对象、并发容器等机制管理后续共享。

## 底层机制应如何理解

从使用者角度，只需依赖公开 API 与内存语义，不应把代码绑定到某个 JDK 的私有实现。不过，理解常见 OpenJDK 实现有助于解释它为什么能够支持多等待者和原子递减。

`CountDownLatch` 通常基于 `AbstractQueuedSynchronizer`，简称 AQS，实现共享获取模式。AQS 中的同步状态保存剩余计数：

- `await()` 尝试以共享模式获取同步状态；计数为零时获取成功，否则线程进入同步等待流程；
- `countDown()` 使用比较并交换方式尝试把正计数减一；
- 当递减结果为零时，共享释放成功，等待队列中的线程会被传播唤醒；
- 计数为零以后，共享获取持续成功，所以门闩保持永久开放。

这里的“唤醒”不等于线程立即获得 CPU。同步器只是使等待线程从不具备运行条件转为可以参与调度，实际执行顺序仍由运行时和操作系统决定。因此，不能根据 `countDown()` 返回时刻推断所有等待线程已经执行完后续代码。

也不要把实现细节误解为公平性承诺。`CountDownLatch` 的用途是等待条件，不是按到达顺序授予独占资源。多个等待者被放行后的先后顺序没有适合业务依赖的稳定保证。

线程处于 `await()` 时，线程转储中通常表现为等待同步器条件。阻塞等待避免了应用层持续轮询计数，但挂起、唤醒和调度仍有成本。因此，门闩适合存在实际阶段边界的协作，不适合在极短热点路径中被大量创建，只为替代一次普通方法调用。

## 模式一：一个协调者等待一批任务完成

这是最常见的用法。协调线程先确定任务集合，为每个任务分配一个计数，再提交任务并等待全部结束。完整设计还要处理三类失败：任务内部失败、任务未能提交、协调线程等待被中断。

下面的例子将门闩用于“结束通知”，将并发队列用于错误传播：

```java
import java.util.ArrayList;
import java.util.List;
import java.util.Queue;
import java.util.concurrent.ConcurrentLinkedQueue;
import java.util.concurrent.CountDownLatch;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.TimeUnit;

public final class BatchRunner {
    public static void main(String[] args) throws InterruptedException {
        List<String> inputs = List.of("alpha", "beta", "gamma");
        ExecutorService pool = Executors.newFixedThreadPool(3);
        CountDownLatch done = new CountDownLatch(inputs.size());
        Queue<Throwable> failures = new ConcurrentLinkedQueue<>();

        try {
            for (String input : inputs) {
                pool.execute(() -> {
                    try {
                        process(input);
                    } catch (Throwable failure) {
                        failures.add(failure);
                    } finally {
                        done.countDown();
                    }
                });
            }

            boolean completed = done.await(5, TimeUnit.SECONDS);
            if (!completed) {
                throw new IllegalStateException(
                        "batch timed out, remaining=" + done.getCount());
            }
            if (!failures.isEmpty()) {
                IllegalStateException batchFailure =
                        new IllegalStateException("one or more tasks failed");
                failures.forEach(batchFailure::addSuppressed);
                throw batchFailure;
            }
        } finally {
            pool.shutdownNow();
        }
    }

    private static void process(String input) {
        if (input.isBlank()) {
            throw new IllegalArgumentException("blank input");
        }
        System.out.println(Thread.currentThread().getName() + ": " + input);
    }
}
```

这个示例的门闩只回答“所有已提交任务是否都离开了执行体”。错误队列回答“是否有任务失败”。两者职责分开后，失败不会悄悄变成永久等待。

不过，示例仍隐含一个重要前提：所有 `pool.execute()` 都成功。如果提交过程中发生 `RejectedExecutionException`，相应任务从未开始，自然也不会执行任务体内的 `finally`，门闩就缺少一次递减。实际系统必须把“未提交”也视为该任务已经结束于失败状态，并补偿计数，或者在全部提交成功以后再进入等待协议。

一种直接做法是在每次提交外层捕获拒绝：

```java
try {
    pool.execute(task);
} catch (RuntimeException submissionFailure) {
    failures.add(submissionFailure);
    done.countDown();
}
```

这里补偿递减的依据是：该计数代表“这个任务尚未结束”，而提交失败已经使任务确定不会执行。需要确保被拒绝的任务不会又由拒绝策略在调用线程中执行，否则外层补偿和任务内 `finally` 可能重复递减。也就是说，补偿方案必须与线程池拒绝策略一起设计，不能只看一段局部代码。

如果任务天然有返回值和异常，`invokeAll()`、`Future` 或 `CompletableFuture.allOf()` 往往比“门闩加错误队列”表达得更完整。选择 `CountDownLatch` 的合理场景通常是：任务由既有回调或线程触发，协调者只需要一个独立的完成边界，或者需要多个等待者共同观察同一批事件。

## 模式二：统一起跑与完成汇合

并发测试或演示中经常同时使用两个门闩：

- `startGate` 初始值为一，工作线程等待统一放行；
- `doneGate` 初始值为工作线程数量，协调线程等待全部结束。

```java
import java.util.concurrent.CountDownLatch;

public final class StartTogetherExample {
    public static void main(String[] args) throws InterruptedException {
        int workers = 4;
        CountDownLatch ready = new CountDownLatch(workers);
        CountDownLatch start = new CountDownLatch(1);
        CountDownLatch done = new CountDownLatch(workers);

        for (int i = 0; i < workers; i++) {
            Thread thread = new Thread(() -> {
                ready.countDown();
                try {
                    start.await();
                    runOperation();
                } catch (InterruptedException interrupted) {
                    Thread.currentThread().interrupt();
                } finally {
                    done.countDown();
                }
            }, "participant-" + i);
            thread.start();
        }

        ready.await();
        start.countDown();
        done.await();
    }

    private static void runOperation() {
        System.out.println(Thread.currentThread().getName());
    }
}
```

`ready` 避免协调线程过早打开 `start`，使每个工作线程至少已经执行到准备点；`start` 建立统一放行边界；`done` 负责结束汇合。三个门闩分别表示三个不同条件，不能因为都是 `CountDownLatch` 就合并成一个计数。

即便如此，这也不是真正的物理同时开始。`start.countDown()` 之后，线程要经历唤醒和调度，实际进入 `runOperation()` 的时间仍有差异。因此，这个模式适合扩大并发重叠概率、控制测试阶段，不适合用来做精确计时或证明不存在竞态。

在测试中还应给 `ready.await()` 和 `done.await()` 设置合理超时，否则被测代码的缺陷可能让整个测试进程无限挂起。测试框架的全局超时可以作为最后保护，但局部超时能提供“卡在准备阶段还是完成阶段”这样的诊断信息。

## 模式三：服务就绪门闩

某个组件可能需要等待若干一次性初始化步骤完成，然后才允许请求进入。例如读取静态配置、构建只读索引和建立必要连接可以并行执行，完成后共同打开就绪门闩。所有请求线程在同一个门闩上等待，门闩打开后不再增加后续请求的阻塞成本。

这种模式成立需要满足两个条件：

1. 初始化阶段确实只发生一次；
2. 失败状态有独立表示，不能把“所有初始化动作都结束”误认为“组件可用”。

若初始化允许重新加载、断线重连或健康状态反复变化，一次性门闩就无法表达“就绪 -> 不就绪 -> 再次就绪”。应改用显式状态机、条件变量、异步结果引用或其他支持多阶段变化的结构。

就绪门闩也不能代替资源本身的线程安全。门闩只保证请求线程在初始化写入之后通过；资源开放后若仍会被并发修改，就需要继续遵守资源自己的同步协议。

## 异常：门闩不会替工作线程传播失败

`countDown()` 没有“成功递减”和“失败递减”的区别，计数归零也没有携带结果。因此，工作线程抛出的异常不会经由 `await()` 自动抛给等待线程。

使用原生 `Thread` 时，未捕获异常通常交给线程的 `UncaughtExceptionHandler`，等待线程仍只看到门闩是否开放。使用 `ExecutorService.execute()` 时，异常处理取决于工作线程和执行器配置；使用 `submit()` 时，异常通常被保存在 `Future` 中，只有调用 `get()` 才会重新暴露。无论哪种方式，门闩本身都不是异常通道。

常用的失败传播设计包括：

- 每个任务返回 `Future`，等待门闩后逐个检查结果；
- 使用线程安全集合收集全部失败；
- 使用 `AtomicReference<Throwable>` 保存第一个失败；
- 在首个失败出现时触发取消，再让每个任务在退出时递减；
- 直接改用能够组合结果和异常的 `CompletableFuture`。

选择哪种方式取决于业务语义。批处理可能需要收集全部失败；快速失败流程只关心第一个错误；可部分成功的流程还需要记录每个任务的状态。不能用“门闩最终归零”代替这些决策。

捕获范围也要谨慎。为了确保 `countDown()` 执行，并不需要一律捕获 `Throwable`；`finally` 本身就能在普通异常和错误展开栈时执行。若要收集失败，通常捕获 `Exception` 或明确的业务异常更合适。捕获 `Throwable` 会连严重虚拟机错误也纳入普通失败流程，只应在确有隔离要求并理解后果时使用。无论是否捕获，递减都应保留在 `finally`。

## 超时不是完成，也不是取消

带超时的 `await` 让等待方拥有退出边界，但返回 `false` 只表示在给定时间内计数没有归零。它不会：

- 自动中断工作线程；
- 撤销已提交任务；
- 阻止后续任务继续调用 `countDown()`；
- 将门闩标记为失败；
- 恢复初始计数；
- 保证超时瞬间所有任务仍未完成。

最后一点来自并发时序：超时返回与最后一次递减可能非常接近。调用方得到 `false` 后，门闩可能立刻归零。因此，超时处理应设计成幂等操作，不能假定工作方已经停止。

超时后的策略通常包括记录剩余计数、取消对应 `Future`、发出中断、关闭执行器、返回部分结果或将任务转入后台。`CountDownLatch` 只提供检测点，具体取消协议必须由任务和执行框架共同实现。

若一个流程有总截止时间，并且要依次等待多个对象，不应给每一步都重复使用完整超时时长，否则总等待可能远超预算。应先计算统一截止时间，每次等待前再换算剩余时间。`System.nanoTime()` 适合计算经过时间，因为它是单调时间源，不受系统墙上时钟调整影响。

## 中断处理：等待方需要明确退出语义

`await()` 抛出 `InterruptedException` 表示当前线程被要求停止等待。调用层有三种常见选择：

1. 方法本身声明 `throws InterruptedException`，把取消决定交给上层；
2. 当前层完成必要清理后恢复中断标志，再以返回值或业务异常退出；
3. 当前层确实拥有完整的中断消费语义，处理后继续，但这种情况应有明确理由。

最常见的错误是捕获后什么也不做：

```java
try {
    latch.await();
} catch (InterruptedException ignored) {
}
```

这既丢失取消信号，又让后续代码像门闩正常开放一样继续执行。更安全的写法是向上传播；如果接口不允许抛出受检异常，则至少恢复中断：

```java
try {
    if (!latch.await(3, TimeUnit.SECONDS)) {
        throw new IllegalStateException("timed out");
    }
} catch (InterruptedException interrupted) {
    Thread.currentThread().interrupt();
    throw new IllegalStateException("interrupted while waiting", interrupted);
}
```

恢复中断标志是因为抛出 `InterruptedException` 时标志通常已被清除，上层若还需要观察取消状态，就必须重新设置。把异常包装成业务异常是否合适取决于接口约定；库层方法通常更适合直接声明并传播 `InterruptedException`。

工作线程自身被中断时，也应进入 `finally` 递减完成门闩。这里的“完成”仍指任务退出，而不是业务成功。若工作代码捕获中断后继续无限运行，协调线程即使发出取消也可能一直等不到计数归零。

## 创建、提交与生命周期边界

一个可靠的门闩协议必须覆盖任务开始前的失败。常见陷阱是先按任务总数创建门闩，再逐个提交；提交到一半时执行器关闭或拒绝任务，未提交的那些任务永远不会递减。

可以根据场景采用以下策略：

- 保证执行器在提交阶段不会拒绝，并把这个前提纳入所有权约束；
- 每个提交失败都由提交线程补偿一次递减并记录失败；
- 先构建可确认的参与者集合，再创建与成功登记数量一致的门闩；
- 使用 `Future` 或结构化的任务组合 API，让任务生命周期由更高层抽象管理。

另一个陷阱是工作线程在等待同一个受限线程池中的其他任务。例如固定大小为二的线程池先运行两个“协调任务”，二者各自提交子任务后调用 `await()`；由于两个工作线程都被等待占满，子任务没有线程可运行，也就无法递减门闩。这是线程资源饥饿导致的死锁式停滞，不是 `CountDownLatch` 内部故障。

避免这类问题需要分析执行依赖图：负责产生完成信号的任务，必须有机会获得执行资源。不要只数门闩计数，还要核对线程池容量、队列策略、阻塞位置和任务嵌套关系。

门闩本身不需要关闭，也不持有外部资源。真正需要关闭的是执行器、文件、连接或其他任务资源。不要因为门闩归零就假定这些资源已经释放，除非递减明确放在资源关闭之后。

## 线程安全边界

`CountDownLatch` 自身支持多个线程并发调用其公开同步方法，但“用了线程安全类”不等于整个协作流程线程安全。

门闩解决的是阶段条件和相应可见性。以下问题仍需其他机制：

- 多个任务同时修改 `ArrayList`，需要并发容器、外部锁或分区写入；
- 多个任务对普通整数执行自增，需要原子变量、归约或锁；
- 多字段必须保持一致，需要把复合不变式放进同一个同步边界；
- 任务完成后仍修改已发布对象，需要明确后续所有权；
- 等待者同时消费一次性资源，需要互斥或状态转移协议。

一个实用判断方法是分别问两个问题：

1. “什么时候允许读取结果？”可以由门闩回答；
2. “结果是怎样被并发构造而不损坏的？”通常不能只由门闩回答。

如果每个任务只写互不重叠的位置，并且等待方只在门闩打开后读取，门闩往往足以完成最终发布。如果工作线程之间要访问同一可变位置，则仍要建立它们彼此之间的同步；等待方最后执行 `await()` 并不会倒过来修复此前已经发生的数据竞争。

## 常见误区

### 把初始计数写成线程池大小

线程池大小表示最大并行执行能力，不等于任务数量。五个任务交给两个工作线程，完成门闩通常应计五，而不是计二。除非协议明确要求“两个长期工作线程各自结束一次”，否则按池大小计数会提前放行。

### 把计数当可动态增加的任务登记器

`CountDownLatch` 没有增加计数的方法。任务数量必须在构造时确定。若任务执行过程中会发现新任务，初始计数难以准确给出，应考虑 `Phaser`、任务队列加显式终止检测，或者由异步组合工具跟踪动态依赖。

### 认为计数归零表示任务成功

归零只表示约定次数的 `countDown()` 已经发生。若递减放在 `finally`，它通常表示任务都已退出；若递减位置写错，甚至连退出都不能代表。成功、失败和部分成功必须另行记录。

### 在工作开始前递减

若代码先 `countDown()` 再执行工作，等待方可在真实结果产生前通过，同时也失去“工作写入 happens-before 等待后读取”的目标顺序。完成信号应位于所代表事件真正完成之后。

### 用 `getCount()` 驱动并发控制

`getCount()` 是瞬时观察值。读取后状态可以马上变化，适合诊断，不适合“先检查再操作”的复合协议。尤其不要通过它推断某个具体任务是否完成。

### 忽略带超时等待的返回值

`await(timeout, unit)` 正常返回既可能是 `true`，也可能是 `false`。忽略返回值会把超时当成功，后续读取到部分结果。

### 吞掉 `InterruptedException`

中断是等待线程的取消通道。吞掉后继续执行会破坏上层关闭协议。应传播，或在转换异常前恢复中断标志。

### 用一个门闩承担多个阶段

门闩打开后无法复位。第一轮结束后第二轮的 `await()` 会直接通过。循环任务应每轮创建新门闩，或者选择可重复使用的阶段同步器。

### 认为门闩能保证“同时开始”

初始值为一的启动门闩只能建立统一放行点，不能控制操作系统调度，也不能消除线程从唤醒到执行之间的差异。

### 认为等待线程退出会自动取消工作

等待超时、被中断或主动返回都不会改变计数，也不会影响工作线程。取消必须通过 `Future.cancel`、中断标志、共享取消令牌或执行器生命周期显式传递。

### 在持有关键锁时等待门闩

如果负责递减的任务需要获得同一把锁，就会形成循环等待。即使没有直接循环，持锁长时间等待也会扩大阻塞范围。调用 `await()` 前应检查当前线程持有的资源是否是工作方前进所必需的。

## 与其他并发工具的权衡

### `Thread.join()`

`join()` 等待一个具体线程终止，并提供线程终止相关的可见性关系。若代码直接创建少量线程，而且条件就是“这些线程全部结束”，逐个 `join()` 更直接。

`CountDownLatch` 不绑定线程身份，可以等待线程池任务、回调事件或一个线程产生的多个事件；也允许多个等待者观察同一完成边界。但它依赖调用方正确递减，无法像 `join()` 那样从线程实际终止状态自动得出结论。

### `Future` 与 `ExecutorService`

`Future.get()` 同时表达完成、返回值和异常，且通常与任务取消结合。若协调者提交任务并需要每个结果，`Future` 往往优于额外维护门闩。

门闩的优势是一个等待动作即可观察一组异构事件，且多个线程可以等待同一边界。它也能与不返回 `Future` 的旧式回调协作。代价是结果、异常和取消都要另建通道。

### `CompletableFuture`

`CompletableFuture.allOf()` 适合组合异步阶段，能够继续串联后续操作，并传播异常完成状态。数据流本身是异步的、需要变换或组合时，它通常比阻塞等待门闩更自然。

`CountDownLatch` 更适合命令式线程协作、测试控制点或只需一次阻塞边界的场景。不要为了使用门闩而把本可组合的异步流程改成阻塞流程。

### `CyclicBarrier`

`CyclicBarrier` 让固定数量的参与线程彼此等待：每个参与者到达屏障后等待，最后一个到达时共同进入下一阶段，并且屏障在正常情况下可重复使用。它适合迭代算法中的分阶段汇合。

`CountDownLatch` 的等待者和递减者可以是不同集合，等待者不计入初始计数，而且门闩不可重用。一个协调者等待若干工作者完成时，门闩通常更符合语义；固定参与者每轮都要互相等待时，屏障更合适。

### `Phaser`

`Phaser` 支持多阶段推进、动态注册与注销参与者，能力比门闩和屏障更强。任务数量会变化、流程有多轮阶段时，它能更准确表达协议。

能力更强也意味着状态和使用规则更复杂。事件数量固定且只等待一次时，`CountDownLatch` 的单调状态更容易审查和排错。

### `Semaphore`

`Semaphore` 的许可可以获取和释放，主要用于限制同时访问某资源的数量，或者在特定协议中传递许可。许可数能够回升，通常会长期重复使用。

`CountDownLatch` 的计数只下降，调用 `countDown()` 不是归还资源，`await()` 也不会消耗计数。用门闩做并发限流，或用信号量冒充一次性完成计数，都会让语义变得含糊。

### `Condition`、`wait/notify`

条件变量能够表达可反复变化的任意谓词，但调用方必须在锁保护下循环检查条件，并正确处理通知。它适合状态多次变化且与某把锁保护的数据紧密关联的场景。

`CountDownLatch` 把“计数归零”这个单调条件封装好，避免遗漏通知和虚假唤醒检查等细节。若需求恰好是一次性归零条件，应优先使用门闩；若条件会来回变化或需要多个不同条件队列，则应使用条件变量或更合适的高层结构。

## 如何设计一个可审查的门闩协议

在代码评审中，可以按以下顺序检查：

第一，给计数命名。不要只写“线程数”，要说清它是待完成任务数、待到达事件数还是待就绪组件数。

第二，列出所有递减路径。正常完成、业务异常、中断退出、提交失败和提前返回是否都恰好贡献一次信号。

第三，检查递减位置。它是否真的位于所代表事件之后，资源关闭和结果写入是否已经完成。

第四，检查等待边界。生产代码是否允许无限等待；若超时，调用方如何记录、取消和返回；若中断，是否传播或恢复标志。

第五，检查失败通道。等待方如何区分全成功、部分失败和超时，异常是否会在后台线程中丢失。

第六，检查共享结果。工作线程之间是否存在数据竞争，等待后的读取是否确实位于同步边界之后。

第七，检查执行资源。负责递减的任务是否可能因线程池饥饿、锁循环或队列拒绝而永远无法执行。

第八，确认一次性语义。对象没有被缓存后用于下一轮，业务状态也不会重新关闭。

第九，把 `getCount()` 限制在观测用途。若它参与正确性决策，需要重新审视协议。

第十，确保门闩引用不会被无关代码获得。同步器没有调用者身份校验，缩小可见范围能减少意外递减。

## 排查等待不结束或过早结束

门闩长期不归零时，先记录初始计数、当前 `getCount()`、任务提交数和任务实际开始数。随后检查每一个负责递减的执行路径：

- 任务是否成功提交；
- 是否在进入 `try-finally` 前抛出异常；
- 是否阻塞在 I/O、锁、另一个门闩或同一线程池的子任务上；
- 是否捕获中断后继续运行；
- 是否存在遗漏 `countDown()` 的提前返回；
- 等待线程是否持有工作线程需要的锁；
- 计数是否误用了任务数之外的值。

线程转储能帮助定位等待者和工作者当前停在哪里。若所有线程都在等待而没有线程能执行递减，重点检查循环依赖和线程池容量。若仍有工作线程运行，重点检查外部阻塞和取消响应。

门闩过早开放时，检查方向相反：

- 是否有任务重复递减；
- 初始计数是否小于逻辑事件数；
- 提交失败补偿与任务实际执行是否同时发生；
- 递减是否放在真实工作之前；
- 一个回调是否可能被重复触发；
- 是否错误复用了已经归零的旧门闩。

由于计数不会变成负数，重复递减不一定从最终状态直接看出来。可以在开发和测试阶段为每个任务维护独立的原子完成标记，检测重复完成；生产代码是否保留这类校验，应根据开销和故障风险决定。

## 适用边界

适合使用 `CountDownLatch` 的条件通常是：

- 完成条件可以在开始前确定为固定数量；
- 同步只需发生一次；
- 等待者只需要知道阶段已经开放，结果和异常可由其他结构承载；
- 递减责任能够清楚分配；
- 阻塞等待符合调用线程的执行模型；
- 超时和取消策略能够在门闩之外完整定义。

出现以下特征时，应优先考虑其他工具：

- 参与任务会动态增加或退出；
- 阶段需要重复执行；
- 每个任务都有重要返回值和异常；
- 希望非阻塞地组合后续流程；
- 核心需求是限制并发量而不是等待完成；
- 条件可能从真变回假；
- 等待线程与工作线程共享受限执行资源，阻塞会造成饥饿。

工具选择的目标不是使用功能最多的类，而是让代码中的状态模型与业务事实一致。固定、一次性、单调归零的完成条件正是 `CountDownLatch` 最清晰的边界。

## 总结

`CountDownLatch` 是一个固定初值、只减不增、归零后永久开放的一次性同步器。它把“一批事件全部完成后再继续”表达为明确的门闩边界，并为 `countDown()` 之前的动作与 `await()` 成功返回之后的动作建立 happens-before 关系。

正确使用它需要把计数视为严格协议，而不是大概的线程数量：初始计数必须对应真实逻辑事件，每个事件恰好递减一次，递减位于事件完成之后，并覆盖异常和提前退出路径。门闩只传递完成信号，不传递返回值、异常或取消，也不保护共享状态的复合操作。

实际代码还应为等待设置时间边界，保留中断语义，处理任务提交失败，避免在线程资源或锁依赖上形成循环等待。需要结果时考虑 `Future` 或 `CompletableFuture`，需要循环阶段时考虑 `CyclicBarrier`，需要动态参与者时考虑 `Phaser`，需要限流时考虑 `Semaphore`。当问题确实是固定数量事件的一次性汇合时，`CountDownLatch` 的简单单调模型才会成为优势。

# 3.3.4.1 Future

## Future 的定位：异步计算结果的句柄

`Future` 是 Java 并发库中用来表示“一个异步计算结果”的接口。它本身不负责创建线程，也不直接规定任务应该放在哪个队列里执行，更不保证任务一定已经开始运行；它提供的是一个面向调用方的结果句柄。调用方拿到这个句柄之后，可以查询任务是否完成，可以阻塞等待结果，可以在等待时设置超时，也可以发出取消请求。换句话说，`Future` 把“提交一段工作”和“稍后取得这段工作的结果”拆成了两个动作，让调用线程不必在提交位置立即停下来。

在同步调用模型里，方法返回值和异常都沿着当前调用栈立即回到调用者手中。调用者执行 `int value = calculate();` 时，要么得到 `value`，要么当前线程直接看到 `calculate` 抛出的异常。异步任务改变了这个结构：`calculate` 可能在另一个线程执行，调用者提交后继续做别的事，结果在未来某个时间才出现。此时返回值、异常、取消状态、完成状态都不能再靠普通调用栈自然传递，`Future` 的价值就在于把这些信息集中保存起来，并通过一组受控方法交给调用者。

理解 `Future` 时，最重要的是不要把它看成“异步编程的全部”。它只表达单个任务的单个结果，能力边界相当清楚：它能等待一次结果，能表示一次失败，能尝试取消一次任务，但它不擅长表达多个任务之间的依赖、回调链、流水线转换、任意数量任务的动态编排，也不负责自动把结果推送到某个回调中。正因为边界朴素，`Future` 在很多基础并发场景里仍然有价值：它足够简单，语义稳定，能与 `ExecutorService`、`Callable`、`FutureTask` 等标准工具自然协作。

本文保持通用 Java 技术视角，讨论标准 Java 并发库中的 `Future` 语义、使用方式、阻塞等待、异常包装、超时、取消、`FutureTask`、与线程池协作方式，以及它和 `CompletableFuture` 的边界。阅读本文时可以先抓住一条主线：`Future` 不是“任务本身”，也不是“线程本身”，而是调用方观察异步任务结果的窗口。

## 为什么需要 Future

没有 `Future` 时，调用者如果想在另一个线程里执行带返回值的任务，通常要自己解决四个问题。第一，如何把任务交给另一个线程。第二，如何把结果从工作线程传回提交线程。第三，工作线程抛出的异常如何保存和重新暴露。第四，提交线程等待多久、能否取消、取消后如何知道任务状态。这些问题都可以手写，但手写实现很容易引入竞态、遗漏通知、异常丢失和资源泄漏。

例如，一个线程启动另一个线程计算结果后，如果结果放在共享字段里，调用者必须保证这个字段的写入对读取线程可见；如果使用 `wait/notify`，必须保证等待条件和通知在同一个锁保护下，并且要处理虚假唤醒；如果工作线程抛异常，异常不会自动跨线程传播到提交线程；如果调用者不想永久等待，还需要实现超时逻辑；如果调用者希望取消任务，又需要约定一个能被工作线程观察到的停止信号。每一项都不是难以理解，但叠在一起之后就容易变成脆弱的并发代码。

`Future` 把这些常见需求收束为一个协议。任务完成时，结果被保存；任务失败时，异常被保存；调用 `get` 时，调用者要么得到结果，要么看到包装后的异常，要么在当前线程被中断时退出等待；调用带超时的 `get` 时，等待不会无限延长；调用 `cancel` 时，调用者向任务执行方发出取消请求；调用 `isDone` 和 `isCancelled` 时，调用者可以观察任务状态。这个协议并不神秘，但它让异步结果传递不再依赖零散约定。

`Future` 还解决了一个设计层面的表达问题。一个方法返回 `Future<T>`，就清楚地告诉调用方：这里不会立刻给出 `T`，而是给出一个将来可能产生 `T` 的句柄。这个返回类型本身就是约束，调用方必须选择等待、轮询、取消或把句柄继续传递。相比把任务悄悄提交到后台线程再通过共享变量通知，`Future` 更容易让代码边界显形。

## Future 与 Callable、Runnable、ExecutorService 的关系

`Future` 最常见的来源是 `ExecutorService.submit`。`ExecutorService` 负责接收任务并把任务交给线程池中的工作线程执行，`submit` 负责把提交动作和结果句柄绑定起来。对于需要返回值的任务，通常提交 `Callable<T>`；对于没有返回值的任务，可以提交 `Runnable`，得到的 `Future<?>` 仍然可以用来等待完成、观察异常或尝试取消。

`Callable` 与 `Runnable` 的核心区别有两个。第一，`Callable` 的 `call` 方法可以返回一个值，而 `Runnable` 的 `run` 方法没有返回值。第二，`Callable.call` 可以声明抛出受检异常，而 `Runnable.run` 不能直接声明受检异常。这个差异使 `Callable` 更适合表示真正的异步计算，而 `Runnable` 更适合表示只关心执行完成与否的动作。`submit` 接收二者后都会返回 `Future`，只是 `Future.get` 的结果类型不同。

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

Future<Integer> future = executor.submit(new Callable<Integer>() {
    @Override
    public Integer call() throws Exception {
        return loadAndCalculate();
    }
});

Integer result = future.get();
```

在这个例子中，提交线程调用 `submit` 后得到 `Future<Integer>`，工作线程稍后执行 `call`，`call` 的返回值会成为 `future.get()` 的返回值。如果 `call` 抛出异常，异常不会直接出现在提交 `submit` 的位置，而是在调用 `get` 时通过 `ExecutionException` 暴露。这个时间差非常关键：提交成功只说明任务被执行器接受，不说明任务已经成功执行。

`execute` 和 `submit` 的差异也应该分清。`Executor.execute(Runnable)` 只提交任务，没有返回 `Future`，调用方无法通过返回值等待任务完成或取得异常。`ExecutorService.submit` 会返回 `Future`，因此调用方有了后续观察和控制任务的句柄。对只需要“丢进去执行”的后台动作，`execute` 足够；对需要等待、取消、获取异常或与关闭流程协调的任务，`submit` 更合适。

`submit(Runnable task, T result)` 是一个容易被忽略的重载。它允许提交一个无返回值的 `Runnable`，但在任务正常完成后让 `Future.get` 返回预先给定的 `result`。这个重载不是为了让 `Runnable` 神奇地产生计算结果，而是为了在一些批处理或标记型任务中保留统一的 `Future<T>` 类型。使用它时要注意，返回的 `result` 与任务内部实际动作没有自动一致性关系，任务只要正常结束，`get` 就返回这个固定对象。

## Future 的核心方法

`Future` 接口的方法数量很少，正因为少，必须准确理解每个方法的边界。它们可以分成三类：等待结果的 `get`，发出取消请求的 `cancel`，查询状态的 `isDone` 与 `isCancelled`。

| 方法 | 主要作用 | 关键边界 |
| --- | --- | --- |
| `V get()` | 阻塞等待任务完成并取得结果 | 可能无限等待；可能抛出 `InterruptedException`、`ExecutionException`、`CancellationException` |
| `V get(long timeout, TimeUnit unit)` | 在限定时间内等待结果 | 超时抛出 `TimeoutException`，超时本身不会自动取消任务 |
| `boolean cancel(boolean mayInterruptIfRunning)` | 尝试取消任务 | 返回值表示取消请求是否被接受，不保证运行中的任务立即停止 |
| `boolean isDone()` | 查询任务是否已经结束 | 正常完成、异常完成、取消完成都属于 done |
| `boolean isCancelled()` | 查询任务是否在正常完成前被取消 | 只表示取消状态，不等价于工作线程已经立刻退出所有清理逻辑 |

`isDone` 的语义经常被误读。它不是“任务正常成功”的意思，而是“任务已经到达终态”。任务正常返回、任务抛出异常、任务被取消，都会让 `isDone` 返回 `true`。如果调用方只用 `isDone` 判断是否可以安全取得业务结果，就会漏掉异常和取消分支。正确做法是：`isDone` 只用于判断是否结束，真正需要结果时仍然调用 `get`，并处理 `get` 的各种异常。

`isCancelled` 也有自己的边界。它表示任务在正常完成之前被取消。如果任务已经正常完成，再调用 `cancel` 通常不会让它变成取消状态。另一方面，`isCancelled` 为 `true` 说明 `Future` 的状态已经是取消，但不应该据此假设任务内部每一层资源都已经完成清理。取消是一个协作过程，特别是运行中的任务需要有机会观察中断或其他停止条件，并在 `finally` 中释放资源。

`cancel` 的返回值不是“任务已停止”的证明。它只表示取消请求是否成功改变了 `Future` 的状态。任务尚未开始时，取消通常可以阻止它被执行；任务已经完成时，取消会失败；任务正在运行时，是否能停止取决于实现以及任务代码是否响应中断。这个语义决定了调用方不能把 `cancel(true)` 当成强制杀死线程的按钮。

## get：阻塞等待与结果发布

`get` 是 `Future` 最核心的方法。它把异步任务重新接回同步控制流：调用 `get` 的线程会等待，直到任务正常完成、异常完成、被取消，或者当前线程在等待时被中断。正常完成时，`get` 返回任务结果；异常完成时，`get` 抛出 `ExecutionException`；取消完成时，`get` 抛出 `CancellationException`；等待线程被中断时，`get` 抛出 `InterruptedException`。

阻塞等待是 `get` 的优势，也是风险来源。优势在于它提供了清晰的同步点：调用方明确知道从这一行之后可以使用任务结果。风险在于调用线程会停在这里，如果任务迟迟不完成，调用线程也会迟迟不能继续执行。对于少量明确受控的后台计算，这种等待很自然；对于高并发入口、资源持有期间、线程池工作线程内部或关闭流程中，无界 `get` 可能放大故障。

`Future.get` 还有内存可见性含义。按 `Future` 的规范，异步计算中的动作 happens-before 另一个线程中成功从对应 `get` 返回之后的动作。通俗地说，如果任务线程在返回结果前写入了一些状态，调用线程成功 `get` 之后，再读取通过正确方式共享出来的结果时，可以依赖 `Future` 建立的完成发布关系。当然，这不意味着任务内部随意修改的所有共享对象都自动具备业务层面的线程安全；它保证的是完成前动作对 `get` 返回后的可见性，而不是替代对共享可变状态的完整保护。

```java
Future<List<String>> future = executor.submit(() -> {
    List<String> values = new ArrayList<>();
    values.add("a");
    values.add("b");
    return values;
});

List<String> values = future.get();
```

上面的代码中，调用方通过 `get` 得到任务构造好的列表，可以看到任务在线程内完成的初始化结果。但如果这个列表随后继续被多个线程并发修改，`Future` 不会继续替这些后续修改提供互斥保护。`Future` 解决的是“结果何时发布给等待方”，不是“结果对象以后如何被并发访问”。

使用 `get` 时还要避免在持有关键锁时等待。假设线程 A 持有某个锁后调用 `future.get()`，而任务线程 B 完成计算前也需要同一把锁，那么 A 等 B，B 等 A，就会形成死锁。即使没有死锁，持锁等待也会扩大临界区，让其他线程长期阻塞。通常应把任务提交、等待结果、更新共享状态拆开，尽量不要把不可控耗时的等待放在锁内。

## 带超时的 get：等待边界与降级点

`get(long timeout, TimeUnit unit)` 给等待设置了时间边界。如果任务在指定时间内完成，它的行为与普通 `get` 类似；如果时间耗尽仍未完成，方法抛出 `TimeoutException`。这个异常只说明“当前等待没有在期限内得到结果”，并不说明任务已经失败，也不会自动取消任务。任务可能仍在队列中等待执行，也可能正在运行，还可能下一毫秒就正常完成。

这点非常重要。很多代码捕获 `TimeoutException` 后误以为任务已经停止，实际任务可能继续占用线程、连接、文件句柄或其他资源。如果调用方在超时后不再需要结果，通常应明确决定是否调用 `cancel(true)`。如果调用方只是暂时不能继续等，可以保留 `Future`，稍后再次查询或等待。超时策略本质上是业务和资源策略，不是 `Future` 能替你自动决定的事情。

```java
try {
    Report report = future.get(500, TimeUnit.MILLISECONDS);
    use(report);
} catch (TimeoutException timeout) {
    future.cancel(true);
    recordTimeout(timeout);
}
```

这个片段展示的是一种常见策略：等待超过上限后发出取消请求并记录现场。但它仍然不是强制停止保证。任务代码如果正在执行不可中断的计算，或者在不响应中断的阻塞调用中停留，取消不会立刻让工作线程退出。调用方要把超时视为控制流分支，而不是把它视为已经完成清理。

带超时的 `get` 还能作为系统诊断点。无界等待一旦卡住，调用方很难知道卡了多久、卡在哪类任务、当时线程池队列有多长。带超时等待可以在超时分支记录任务标识、线程池状态、上游请求信息和关键参数，让偶发问题有迹可循。并发系统中的超时不是单纯为了“失败得快”，也是为了把未知等待变成可观测事件。

超时时间不应该随意拍脑袋。过短会导致正常波动下大量误判，过长会让资源长期占用，完全无界则可能让故障扩散。更合理的做法是结合任务性质、调用链预算、线程池容量、重试成本和上游等待能力来确定。对于纯内存计算，超时通常应该短且稳定；对于可能访问外部资源的任务，超时还要与外部资源自身的超时设置配合，避免外层已经放弃而内层仍长时间阻塞。

## 异常传播：ExecutionException 与根因

异步任务中的异常不会自动出现在提交线程的调用栈中。`submit` 只负责接受任务并返回 `Future`，真正执行任务的是工作线程。如果工作线程里的 `Callable.call` 抛出异常，`Future` 会记录这个异常；调用方稍后执行 `get` 时，`get` 会抛出 `ExecutionException`，原始异常可以通过 `getCause()` 取得。

```java
try {
    future.get();
} catch (ExecutionException e) {
    Throwable cause = e.getCause();
    handleTaskFailure(cause);
}
```

`ExecutionException` 的存在说明异常跨越了线程边界。它不是根因本身，而是一个包装层。处理时如果只打印 `ExecutionException` 的消息，可能丢失真正的错误类型、堆栈和上下文。更稳妥的做法是保留 `cause`，根据根因类型决定是否重试、降级、向上抛出或转成领域异常。对于基础库代码，常见做法是恢复更有意义的异常层次；对于应用层代码，至少应记录根因和任务标识。

`submit` 对异常的处理方式也容易被忽略。通过 `execute` 提交的 `Runnable` 如果抛出未捕获的运行时异常，异常通常会交给工作线程的未捕获异常处理机制，具体表现与执行器实现和线程工厂有关。通过 `submit` 提交的任务，异常会被捕获并存入 `Future`，如果调用方从不调用 `get`，异常就可能长期只停留在 `Future` 里，表现为后台任务“悄悄失败”。因此，当使用 `submit` 提交有副作用但不关心返回值的任务时，也要考虑如何观察异常。

一种常见误区是把 `Future<?>` 丢进集合后从不读取。这样做可以实现批量提交，但如果后续没有遍历调用 `get`，任务失败不会自然反馈给提交方。正确的批处理通常会在某个同步点集中收集所有 `Future`，逐个 `get`，把成功、失败、超时、取消分别归类。即使最终业务不需要返回值，也需要决定失败是否影响整体结果。

`CancellationException` 与 `ExecutionException` 也要区分。取消不是任务抛出的业务异常，因此 `get` 在取消状态下抛出的是未受检的 `CancellationException`，而不是 `ExecutionException`。如果代码只捕获 `ExecutionException`，取消分支会继续向外传播。这个设计是合理的：取消表示调用方或调度方选择不再等待正常结果，它不是 `Callable` 自己计算失败的根因。

## 中断语义：等待线程与工作线程是两回事

`Future` 相关代码里有两个不同的中断对象：调用 `get` 的等待线程，以及实际执行任务的工作线程。等待线程在 `get` 中被中断时，`get` 抛出 `InterruptedException`，表示当前等待应该结束。这个事件不等于任务被取消，也不等于工作线程被中断。它只说明等待者自己收到了中断信号。

```java
try {
    return future.get();
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    throw e;
}
```

如果当前方法不能完整处理取消流程，通常应恢复中断标志，让上层代码知道这个线程已经被请求中断。静默吞掉 `InterruptedException` 是并发代码中的高风险行为，因为上层关闭流程、超时控制或任务调度可能都依赖中断信号传递。一旦某一层吞掉中断，外层就可能误以为线程仍然可以继续正常工作。

工作线程的中断则通常来自 `cancel(true)`。当任务已经开始运行且实现支持中断时，执行器可以尝试中断运行该任务的线程。但中断不是强制异常注入，也不是立刻终止。它只设置线程的中断状态，并使某些可中断阻塞方法抛出 `InterruptedException`。任务代码需要主动检查中断状态，或者调用能够响应中断的阻塞方法，才能及时退出。

```java
Callable<Long> task = () -> {
    long sum = 0L;
    for (long i = 0; i < Long.MAX_VALUE; i++) {
        if (Thread.currentThread().isInterrupted()) {
            throw new InterruptedException("cancelled");
        }
        sum += i;
    }
    return sum;
};
```

这类 CPU 循环如果从不检查中断，`cancel(true)` 很难让它及时停止。相反，如果任务经常调用 `BlockingQueue.take`、`Thread.sleep`、`CountDownLatch.await` 等可中断方法，它更容易响应取消。设计可取消任务时，应主动在长循环、分段处理、阻塞等待和资源访问之间放置中断检查点，并在退出时释放资源。

需要注意的是，任务因为中断而抛出 `InterruptedException` 时，对于 `Future.get` 来说，这仍然是任务异常完成，通常会包装在 `ExecutionException` 中，除非 `Future` 已经成功进入取消状态。也就是说，等待线程看到 `InterruptedException` 和任务内部抛出 `InterruptedException` 是两条不同路径：前者表示等待者被中断，后者表示任务执行失败或被协作式终止。

## cancel：取消请求而不是强制终止

`cancel(boolean mayInterruptIfRunning)` 是 `Future` 中最容易被过度解读的方法。它的名字看起来像“取消任务”，但更准确地说，它是“尝试取消任务”。如果任务还在队列中尚未开始，取消通常可以让它不再运行；如果任务已经完成，取消失败；如果任务正在运行，`mayInterruptIfRunning` 决定是否允许通过中断工作线程来协助取消。

当参数为 `false` 时，执行器不会为了取消这个任务而中断正在运行它的线程。如果任务已经开始，取消通常无法阻止它继续执行；如果任务尚未开始，仍可能取消成功。这个模式适用于不希望打断正在运行逻辑，只想阻止尚未开始任务的场景。它表达的是“不要为了这个取消请求干扰当前执行中的工作”。

当参数为 `true` 时，如果任务已经开始，执行器可以尝试中断工作线程。这里的关键词是“尝试”。任务是否停下，取决于任务代码是否响应中断、当前阻塞点是否可中断、执行器是否能定位运行该任务的线程，以及任务是否在取消请求到达前已经完成。调用方不能假设 `cancel(true)` 返回后任务内部立刻停止在某一行。

```java
Future<?> future = executor.submit(() -> {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            doOneChunk();
        }
    } finally {
        cleanup();
    }
});

boolean accepted = future.cancel(true);
```

这段代码体现了更合理的取消协议：循环体定期检查中断，退出时在 `finally` 中清理。`Future` 只能发出取消请求，任务本身必须把取消作为正常生命周期的一部分来设计。对于持有锁、打开文件、占用连接、修改共享状态的任务，取消路径尤其要明确收尾动作，否则取消可能只是在 `Future` 层面结束等待，却留下资源或状态问题。

取消还涉及任务幂等性。取消请求可能在任务开始前到达，也可能在任务执行到一半时到达，还可能在任务刚完成后到达但状态尚未被调用方观察。任务如果有外部副作用，例如写文件、更新数据库、发送消息或修改共享集合，就不能简单认为取消意味着“什么都没发生”。更稳妥的设计是让任务的每个阶段具备可恢复或可识别状态，至少保证重复清理、重复取消和部分完成不会破坏不变式。

## FutureTask：既是任务，也是 Future

`FutureTask<V>` 是 `Future` 机制的重要实现类。它同时实现了 `Runnable` 和 `Future<V>`，因此既可以作为任务交给线程或执行器运行，也可以作为结果句柄供调用方等待。它把 `Callable` 或 `Runnable` 包装成一个可运行、可等待、可取消的对象，是 `ExecutorService.submit` 等机制背后的核心构件之一。

```java
Callable<Integer> callable = () -> loadAndCalculate();
FutureTask<Integer> task = new FutureTask<>(callable);

Thread thread = new Thread(task);
thread.start();

Integer value = task.get();
```

这个例子没有使用线程池，直接把 `FutureTask` 放进一个新线程执行。`thread.start()` 负责启动执行，`task.get()` 负责等待结果。它展示了 `FutureTask` 的双重身份：对执行方来说它是 `Runnable`，对等待方来说它是 `Future`。实际开发中通常更推荐使用 `ExecutorService` 管理线程，但理解 `FutureTask` 有助于看清 `Future` 与任务执行之间的连接方式。

`FutureTask` 的一个关键特性是一次性执行。正常情况下，`run` 只会让内部任务计算一次；完成后再次调用 `run` 不会重新计算结果。这个语义让它适合表示一次性的延迟计算、缓存初始化或由外部线程触发的计算。调用方可以从多个线程调用 `get` 等待同一个结果，任务完成后所有等待者都能得到同一结果或同一异常。

`FutureTask` 还有一个受保护的 `done` 方法，任务进入终态时会被调用。子类可以覆写它做轻量通知或清理，但不应在其中执行长时间阻塞逻辑。因为 `done` 属于任务完成路径的一部分，阻塞过久可能拖慢执行线程，甚至影响执行器吞吐。它适合放置状态转移后的简单钩子，不适合放置复杂业务流程。

在自定义并发组件时，`FutureTask` 比手写 `Future` 更可靠。手写 `Future` 需要正确处理状态机、等待队列、异常保存、取消竞争、中断和可见性，任何细节出错都会造成难以复现的问题。`FutureTask` 已经提供了成熟实现，除非确实需要特殊队列或特殊调度语义，否则应优先复用它，而不是重新发明一套等待和取消机制。

## 与线程池协作：提交、排队、执行、回收

`Future` 单独存在时只是句柄，真正决定任务何时运行的是执行器。与 `ThreadPoolExecutor` 协作时，任务通常经历提交、入队或创建线程、执行、完成、结果保存这几个阶段。`Future` 可以观察最终完成状态，但它不直接告诉你任务当前是在队列中等待，还是正在某个工作线程上运行。要理解运行时行为，必须把 `Future` 和线程池配置一起看。

线程池的核心线程数、最大线程数、工作队列、拒绝策略和线程工厂都会影响 `Future` 的表现。提交成功后拿到 `Future`，并不代表任务已经开始执行；如果队列很长，`get` 可能长时间等待排队；如果线程池饱和并触发拒绝策略，`submit` 可能直接抛出 `RejectedExecutionException`，调用方甚至拿不到 `Future`；如果工作线程被长时间阻塞，后续任务的 `Future` 都会迟迟不完成。

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
        4,
        8,
        30,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(100),
        new ThreadPoolExecutor.AbortPolicy()
);
```

使用 `Future` 时不能只看 `get` 一行，还要看提交到什么执行器。无界队列可能让提交几乎总是成功，却把压力积累成越来越长的等待时间和内存占用；过小队列可能频繁拒绝任务；过大线程数可能导致上下文切换和资源竞争；过小线程数可能让任务互相等待时形成饥饿。`Future` 暴露的是结果等待，线程池配置决定等待背后的容量模型。

一个典型误区是在同一个固定大小线程池的工作线程中等待同一个线程池里的其他任务。如果线程池大小为 N，N 个任务都占住工作线程后再提交子任务并调用 `get` 等待，而子任务也需要这个线程池的空闲线程执行，就可能出现线程饥饿死锁。表面看没有锁，实际是线程资源被等待关系耗尽。

```java
Callable<String> parent = () -> {
    Future<String> child = executor.submit(() -> "child");
    return child.get();
};
```

这个片段单独看没有问题，但如果大量 `parent` 同时占满同一个小线程池，就可能让所有父任务都等待子任务，而子任务没有线程可用。解决方向不是简单把 `get` 换成别的方法，而是重新设计执行结构：避免在同一受限池内阻塞等待子任务，拆分不同性质的线程池，使用批量提交和集中等待，或者改用更适合依赖编排的模型。

线程池关闭也和 `Future` 密切相关。调用 `shutdown` 后，线程池不再接受新任务，但已提交任务会继续执行；调用 `shutdownNow` 会尝试中断正在执行的任务，并返回尚未开始执行的任务列表。已经返回给调用方的 `Future` 需要结合关闭语义解释：有些会正常完成，有些会异常完成，有些可能因取消而结束，有些如果任务不响应中断，仍可能拖延关闭。

## 批量任务：invokeAll、invokeAny 与 Future 集合

`ExecutorService` 提供了与 `Future` 相关的批量方法。`invokeAll` 接收一组 `Callable`，返回一组 `Future`，并在所有任务完成后返回；带超时的 `invokeAll` 会在时间耗尽时取消尚未完成的任务。`invokeAny` 则等待一组任务中任意一个成功完成并返回其结果，其他未完成任务通常会被取消。

`invokeAll` 适合“全部任务都需要一个终态”的场景，例如把一批独立计算分发出去，然后集中收集每个任务的成功或失败。它返回的 `Future` 顺序与输入任务顺序对应，这使结果归位比较方便。但调用方仍然要逐个 `get`，因为某个 `Future` 可能是异常完成或取消完成。`invokeAll` 返回不等于所有任务都成功，只等于所有任务都有了终态。

```java
List<Future<Result>> futures = executor.invokeAll(tasks);
for (Future<Result> future : futures) {
    try {
        collect(future.get());
    } catch (ExecutionException e) {
        collectFailure(e.getCause());
    }
}
```

`invokeAny` 适合多个候选任务只需要一个成功结果的场景。它与手写循环提交再竞争第一个结果相比，语义更直接。不过它的“任意一个成功”不代表任意一个完成。如果某些任务失败，执行器会继续等待其他任务成功；只有全部失败时，调用方才会看到异常。使用它时要确认任务之间是否真的可以互相替代，且取消其他任务不会破坏外部副作用。

手动维护 `List<Future<?>>` 时，等待顺序也很重要。如果按提交顺序逐个 `get`，第一个慢任务会阻塞后续已经完成任务的结果处理。这种方式简单，但可能增加尾部延迟。如果希望谁先完成先处理，可以考虑 `ExecutorCompletionService`。它内部把已完成任务放入完成队列，调用方从队列中取完成的 `Future`，避免被提交顺序绑定。

`ExecutorCompletionService` 仍然返回 `Future`，但改变了结果收集方式。它适合大量独立任务、结果可以乱序处理的场景。这个工具提醒我们：`Future` 表示结果句柄，不规定调用方必须按什么顺序等待。合理的等待顺序可以显著影响吞吐和响应时间。

## Future 的状态模型

从调用方角度看，`Future` 的状态可以简化为未完成、正常完成、异常完成、取消完成。未完成包括尚未开始、排队中、正在运行等内部阶段；正常完成表示有结果可取；异常完成表示任务抛出了异常；取消完成表示取消请求在正常完成前生效。`isDone` 对后三种终态都返回 `true`。

这个状态模型有几个推论。第一，`get` 的返回值只存在于正常完成分支。第二，异常完成和取消完成都不是“没有完成”，而是“以非正常结果完成”。第三，状态一旦进入终态，通常不会再回到未完成，也不会从异常完成变成正常完成。第四，多个线程同时等待同一个 `Future` 时，看到的是同一个终态结果，只是各自被唤醒和调度的时间可能不同。

状态竞争集中发生在完成、失败和取消之间。任务可能正在准备设置结果时，另一个线程调用取消；任务可能刚抛出异常时，调用方超时后调用取消；调用方可能在任务完成前一瞬间调用 `isDone` 得到 `false`，下一瞬间任务已经完成。`Future` 实现必须正确处理这些竞争，调用方也不能用一次状态查询推导永久结论。

例如，`if (!future.isDone()) { future.cancel(true); }` 不能保证取消一定成功。`isDone` 返回 `false` 后到 `cancel` 执行前，任务可能已经正常完成。正确代码必须检查 `cancel` 的返回值，并准备处理取消失败后的状态。同样，`isDone` 返回 `true` 后调用 `get` 仍然可能抛出异常或取消异常，因为 done 不等于 success。

在文档和代码注释中，应尽量使用精确词汇。可以说“任务已完成”，但要进一步说明是“正常完成”“异常完成”还是“取消完成”。可以说“取消请求已接受”，但不要轻易说“任务已经被杀掉”。这种措辞看似琐碎，实际能减少大量并发语义误解。

## CompletableFuture 的边界：不是简单替代，而是更高层模型

`CompletableFuture` 实现了 `Future`，但它的定位比传统 `Future` 更宽。传统 `Future` 主要提供阻塞等待和取消查询；`CompletableFuture` 还提供完成、回调、转换、组合、异常恢复等能力。它适合表达“一个阶段完成后继续执行另一个阶段”“两个异步结果合并”“任意一个结果先到就继续”等依赖关系。

传统 `Future` 的短板在复杂编排中很明显。假设任务 A 完成后要启动任务 B，任务 B 和 C 都完成后要合并结果，如果只用 `Future`，调用方往往需要手写多个 `get`、多个线程池提交、多个异常分支和取消分支。代码很快变成阻塞式编排，线程可能花大量时间等待而不是执行。`CompletableFuture` 通过 `thenApply`、`thenCompose`、`thenCombine`、`exceptionally` 等方法，把这些依赖写成阶段链。

但是，`CompletableFuture` 不是在所有地方都“替代” `Future`。如果需求只是提交一个任务，稍后在明确同步点等待结果，传统 `Future` 更简单，语义也更少。`CompletableFuture` 的异步回调会引入更多执行位置问题：回调在哪个线程执行，是否使用公共池，异常在哪个阶段恢复，取消是否传播到上游，链条过长时如何诊断。这些问题不是缺点，而是更高表达力带来的额外边界。

还要注意取消语义差异。`CompletableFuture` 的取消更多是完成状态层面的取消，未必能像某些执行器返回的 `Future` 那样中断正在运行的底层任务。尤其当它表示的是一个手动完成的阶段，或者由多个阶段组合而来时，`cancel` 很难简单等同于中断某个确定线程。使用它时应阅读具体构造方式和执行器选择，而不是套用传统 `FutureTask` 的直觉。

因此，选择标准可以很朴素：如果你需要的是单个异步计算结果和明确等待点，`Future` 足够；如果你需要非阻塞的阶段转换、任务组合、异常恢复链和回调式编排，考虑 `CompletableFuture`；如果你需要流式背压、持续事件、多生产者多消费者协作或结构化生命周期，可能还要选择其他并发或响应式工具。工具选择的核心不是新旧，而是语义是否匹配。

## 内存可见性与共享状态边界

`Future` 与内存可见性的关系必须讲清楚。任务在线程中完成计算，等待线程通过 `get` 成功返回，这两者之间存在规范定义的 happens-before 关系。这意味着任务完成前的动作对 `get` 返回后的线程可见。这个保证是 `Future` 能可靠传递结果的基础，否则等待者即使拿到“完成”信号，也可能看不到任务写入的结果。

但这个保证有边界。`Future` 保证的是任务完成与 `get` 返回之间的发布关系，不是对所有共享对象的永久保护。如果任务返回一个可变对象，调用方拿到后又把它交给多个线程并发修改，后续访问仍然需要自己的同步策略。如果任务内部修改了某个外部共享集合，即使 `get` 之后提交线程能看到这次修改，其他没有同步的线程也不因此获得完整线程安全保证。

更进一步，`Future` 不会自动让任务内部的复合操作具备原子性。任务如果读取共享状态、检查条件、再写回另一个共享状态，这些步骤是否安全取决于共享状态本身的保护方式。`Future` 只能告诉调用方“这个任务作为一个执行单元已经结束”，不能证明任务内部对共享资源的操作符合业务不变式。

```java
Future<?> future = executor.submit(() -> {
    sharedList.add("value");
});

future.get();
```

如果 `sharedList` 是普通 `ArrayList`，并且同时有其他线程并发读写，那么上面的代码仍然可能有问题。`future.get()` 可以让提交线程在返回后看到任务之前的写入，但不能让 `ArrayList` 在并发修改下变成线程安全容器。正确性仍然需要通过线程封闭、不可变对象、并发容器、锁或其他同步策略来保证。

这也是学习 `Future` 时最容易混淆的地方：结果传递安全不等于业务状态安全。`Future` 把一个任务的返回值和异常安全地交给等待者，但任务内部访问的世界可能更复杂。判断代码是否正确时，应分别检查任务结果发布、共享对象访问、资源生命周期和取消路径，而不是看到 `Future` 就默认“并发问题已经解决”。

## 常见实践模式

第一种模式是“提交后做本地工作，再等待结果”。调用方把耗时计算提交到执行器，自己继续处理不依赖该结果的部分，最后在需要结果的同步点调用 `get`。这种模式能够利用并行性，同时保持控制流相对简单。关键是本地工作与异步任务之间不能有隐藏的数据竞争，等待点也要有超时或明确的生命周期边界。

```java
Future<Data> dataFuture = executor.submit(this::loadData);
Metadata metadata = loadMetadataLocally();
Data data = dataFuture.get(1, TimeUnit.SECONDS);
return merge(data, metadata);
```

第二种模式是“批量提交，集中收集”。适用于一批互相独立的任务。调用方保存每个 `Future`，随后统一等待并汇总结果。这里要特别注意失败策略：一个任务失败是否终止全部，是否取消其他任务，是否保留部分成功结果，是否重试。`Future` 只提供单个任务状态，批量语义必须由调用方设计。

第三种模式是“可取消的长任务”。任务按块处理，每块之间检查中断状态或取消标志，收到取消请求后尽快退出，并在 `finally` 中释放资源。调用方用 `cancel(true)` 发出请求，用 `get` 或状态查询观察终态。这种模式的重点不在 `cancel` 调用本身，而在任务是否把取消纳入执行协议。

第四种模式是“结果缓存”。`FutureTask` 可以用于一次性初始化，让多个线程等待同一个计算结果。第一个线程触发执行，其他线程调用 `get` 等待。这个模式可以避免重复计算，但要处理初始化失败后的策略：失败结果是否缓存，是否允许重试，取消后是否重新创建任务。不同需求下答案不同，不能只靠一个 `FutureTask` 自动决定。

第五种模式是“完成顺序消费”。大量任务提交后，调用方不按提交顺序等待，而是用 `ExecutorCompletionService` 获取已完成的 `Future`。这种模式适合先完成先处理，能降低慢任务对整体处理的阻塞影响。它仍然需要逐个处理 `get` 的异常和取消分支。

## 典型误区

最常见的误区是永久等待 `Future.get()`。无界等待在简单示例里最省事，但在真实并发环境里可能把一个任务的问题扩散成调用线程堆积、线程池耗尽、关闭流程卡死。不是所有 `get` 都必须带超时，例如程序启动阶段等待必要初始化时，无界等待可能是明确设计；但只要任务耗时受外部资源、队列长度、锁竞争或上游数据影响，就应认真考虑等待边界和诊断信息。

第二个误区是认为超时等于取消。`get` 超时只影响当前等待，不影响任务继续运行。超时后是否取消，要看调用方是否还需要结果、任务能否安全取消、取消是否会留下部分副作用。如果超时后立即重试但不取消旧任务，系统可能同时运行多个已经无人需要的旧任务，进一步挤占线程池。

第三个误区是认为 `cancel(true)` 可以强制停止任何任务。Java 的中断是协作机制，不是安全的强制终止。任务不检查中断、阻塞在不可中断操作中、或者把 `InterruptedException` 吞掉，都可能让取消请求失效。正确做法是让任务代码参与取消协议，而不是把全部希望放在调用方的 `cancel` 上。

第四个误区是忽略 `submit` 后的异常。通过 `submit` 提交的任务异常会进入 `Future`，如果没人调用 `get`，异常可能不被及时发现。对于必须记录失败的后台任务，可以在任务内部捕获并记录，也可以统一收集 `Future` 并检查结果，还可以使用自定义执行器钩子。关键是不能让失败只存在于无人读取的句柄里。

第五个误区是在同一个受限线程池中嵌套等待。工作线程提交子任务到同一个池并阻塞 `get`，在任务量较大时可能造成线程饥饿。类似问题也会出现在持有锁时等待、持有连接时等待、占用许可时等待。等待前应先问：我现在占着什么资源？被等待的任务是否需要这些资源才能完成？

第六个误区是把 `isDone` 当成功判断。`isDone` 为真时，任务可能成功、失败或取消。只有调用 `get` 并成功返回，才能说明拿到了正常结果。需要非阻塞地判断成功时，传统 `Future` 并没有直接提供“成功结果是否可用且不抛异常”的方法，通常仍要在 `isDone` 后调用 `get` 并处理异常。

第七个误区是把 `Future` 当作任务队列监控工具。`Future` 不告诉你任务排队位置、已经运行多久、在哪个线程执行、线程池是否饱和。它只是单个任务的结果句柄。需要诊断线程池时，应结合执行器指标、日志、线程名、队列长度、拒绝计数和线程 dump，而不是只盯着 `Future` 状态。

## 设计检查清单

设计使用 `Future` 的代码时，可以按几个问题逐项检查。第一个问题：任务是否真的需要返回值？如果只需要执行动作，但仍需要等待完成或观察异常，`Future<?>` 是合理的；如果连完成和异常都不关心，可能只需要 `execute`，但要确认异常记录路径。第二个问题：任务提交到哪个执行器？执行器容量、队列、拒绝策略、线程命名和关闭责任是否明确？

第三个问题：等待点在哪里？等待点是否可能位于锁内、事务内、连接占用期间或线程池工作线程内部？是否设置超时？超时后是取消、重试、降级还是继续保留句柄？第四个问题：异常如何处理？`ExecutionException` 的根因是否被保留？取消和超时是否与普通失败分开？等待线程被中断时是否恢复中断标志？

第五个问题：任务能否响应取消？长循环是否检查中断？阻塞调用是否可中断或有自己的超时？资源是否在 `finally` 中释放？部分副作用是否可识别、可回滚或可容忍？第六个问题：结果对象是否会被多个线程继续访问？如果会，是否不可变、线程安全或受外部同步保护？

第七个问题：是否存在嵌套等待或容量反转？任务是否提交子任务并等待？父子任务是否共用同一个有限线程池？等待任务是否需要当前线程持有的锁、许可或连接？第八个问题：批量任务的整体策略是什么？一个失败是否取消其他任务？是否按提交顺序等待导致快任务被慢任务阻塞？是否需要 `CompletionService`？

这些问题不是形式化流程，而是把 `Future` 的边界转成可执行的工程检查。很多并发问题并不来自 API 用错，而来自等待、容量、异常、取消和共享状态之间缺少统一设计。`Future` 提供的是结果句柄，最终可靠性仍然取决于调用方是否把这些边界写清楚。

## 一个完整示例：有限等待、异常分类与取消

下面的示例展示一个相对完整的等待结构。它提交一个 `Callable`，在有限时间内等待结果；如果超时，发出取消请求；如果任务失败，保留根因；如果等待线程被中断，恢复中断标志并尝试取消任务。示例的重点不是业务逻辑，而是展示 `Future` 的几个分支应该同时存在。

```java
public Result loadWithLimit(ExecutorService executor) throws Exception {
    Future<Result> future = executor.submit(() -> loadResult());
    try {
        return future.get(800, TimeUnit.MILLISECONDS);
    } catch (TimeoutException e) {
        future.cancel(true);
        throw new IllegalStateException("loading timed out", e);
    } catch (InterruptedException e) {
        future.cancel(true);
        Thread.currentThread().interrupt();
        throw e;
    } catch (ExecutionException e) {
        Throwable cause = e.getCause();
        if (cause instanceof Exception) {
            throw (Exception) cause;
        }
        throw new IllegalStateException("loading failed", cause);
    }
}
```

这个结构比直接 `return future.get();` 繁琐，但它把关键决策显式写出来了。超时不再无限等待，中断不会被吞掉，任务异常不会只留下包装层，取消请求有明确位置。真实代码还可以继续扩展，例如记录任务标识、统计超时次数、区分可重试异常和不可重试异常、在上层统一处理线程池关闭。

需要强调的是，这个示例不是唯一正确写法。有些场景中断后不应该取消任务，因为任务结果仍然对其他等待者有价值；有些场景超时后不应该立刻取消，因为任务具有不可中断的关键区；有些场景不应该把根因强行转型抛出，而应包装成稳定的接口异常。`Future` 的 API 很少，但每个异常分支背后都是设计选择。

## 与其他线程配合工具的关系

`Future` 是线程配合工具之一，但不是所有协作问题都应该用它解决。它适合一段任务产生一个结果，调用方在某个时刻观察这个结果。`CountDownLatch` 更适合一个或多个线程等待若干事件发生，不强调返回值；`Semaphore` 更适合限制并发许可；`BlockingQueue` 更适合生产者消费者之间传递一串数据；`CompletableFuture` 更适合异步阶段组合；锁和条件队列更适合保护共享状态并等待特定条件。

如果需求是“等待十个任务都结束并收集每个结果”，`List<Future<T>>` 或 `invokeAll` 很自然。如果需求是“等待某个初始化信号”，`CountDownLatch` 可能更直接。如果需求是“持续从后台线程接收很多条消息”，`BlockingQueue` 比一堆 `Future` 更贴近模型。如果需求是“任务 A 的结果转换成任务 B，再和任务 C 合并”，`CompletableFuture` 往往比嵌套 `get` 更清晰。

选择工具时要看协作形态，而不是看哪个类更熟悉。`Future` 的形态是一次性结果，等待方主动拉取；队列的形态是多次消息，消费者持续获取；同步器的形态是状态条件，等待方等条件满足；回调和阶段模型的形态是完成后自动触发下一步。把不同形态混用，通常会让代码出现额外线程、额外阻塞和额外状态转换。

这也是本文反复强调边界的原因。`Future` 很重要，但它不是通用并发胶水。它越适合表达单次异步计算结果，越不应该被硬塞进长期事件流、复杂依赖图或共享状态保护里。正确使用工具的第一步，是承认每个工具只解决一类问题。

## 小结

`Future` 的核心价值是为异步计算提供一个稳定的结果句柄。通过它，调用方可以把任务提交给执行器，让工作线程完成计算，再在需要时等待结果、观察异常、设置超时或尝试取消。它让返回值和异常跨越线程边界有了标准表达，也让任务生命周期从“后台悄悄运行”变成“可以被查询和协调”的对象。

掌握 `Future`，不能只记住 `get` 和 `cancel`。更重要的是理解几个边界：`get` 会阻塞，普通 `get` 可能无限等待；带超时的 `get` 只是停止当前等待，不会自动停止任务；任务异常会被 `ExecutionException` 包装，根因在 `getCause` 中；取消是协作请求，不是强制终止；`isDone` 表示终态，不表示成功；`Future` 建立结果发布关系，但不替代共享可变状态的线程安全设计；线程池容量和等待位置会直接影响 `Future` 的行为。

在实践中，`Future` 最适合表达单个异步任务的结果。遇到批量任务，要设计收集顺序和失败策略；遇到长任务，要设计取消检查和资源清理；遇到复杂依赖，要考虑 `CompletableFuture` 或其他更高层工具；遇到共享状态，要回到锁、并发容器、不可变对象和内存可见性规则。把这些边界讲清楚，`Future` 才不是一个会阻塞的返回值包装器，而是 Java 并发程序中可预测、可诊断、可维护的线程配合机制。

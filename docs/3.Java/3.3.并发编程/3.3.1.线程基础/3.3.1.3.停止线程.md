# 3.3.1.3 停止线程

停止线程讨论的不是“怎样把一个正在运行的线程立刻杀掉”，而是“怎样让一个任务在可控位置结束，并且结束后程序仍然保持一致”。在 Java 并发中，线程通常承载一段任务逻辑：读取输入、更新状态、等待队列、持有锁、调用外部资源、写入结果。停止线程如果处理得粗糙，就可能把这些动作截断在任意位置，留下未释放的锁、半更新的数据、未关闭的资源、没有通知的等待者，甚至让调用方以为任务已经结束而实际状态仍然不可预测。因此，可靠的停止线程不是暴力终止执行流，而是一套协作式取消协议。

“协作式”是理解本文的关键。发起停止的一方只能提出请求，真正退出的时机要由执行任务的线程在安全点决定。这个安全点通常位于循环边界、阻塞方法返回处、阶段性任务完成处、资源操作的 finally 块之后，或者某个明确检查取消状态的位置。这样做看起来不如强制终止“直接”，但它保留了程序维护不变式的机会：当前步骤可以完成或回滚，锁可以释放，文件、套接字、数据库连接这类资源可以关闭，结果或失败原因可以被记录，调用方也可以得到可解释的终止状态。

本文保持通用 Java 技术视角，围绕 Java 线程、`interrupt` 中断机制、`volatile` 停止标志、阻塞方法响应、资源清理、`Thread.stop` 风险、取消协议和线程池关闭关系展开。它不把停止线程简化成某一个 API，而是把停止行为放回任务生命周期中讨论：谁发起停止，谁观察停止，哪些位置可以停止，停止时如何处理共享状态，阻塞时如何被唤醒，退出后如何让其他线程知道任务已经结束。

## 停止线程到底要停止什么

日常表达里常说“停止线程”，但在设计并发程序时，更准确的说法通常是“取消任务”或“结束某个工作循环”。线程是执行载体，任务才是业务动作。一个 `Thread` 可能只运行一个短任务，也可能承载一个长期循环；一个线程池工作线程会连续执行许多任务；一个任务也可能拆成多个子任务提交给不同线程。把“线程”和“任务”混为一谈，会导致停止边界不清：有时我们想停止的是当前计算，有时想停止的是后台服务循环，有时只是想让等待者放弃等待，有时则要关闭整个执行器。

最理想的结束方式是自然返回。`run` 方法执行完毕，线程生命周期进入终止状态，持有的 Java 栈随之销毁，局部变量消失，等待 `join` 的线程可以继续执行。自然返回没有神秘机制，和普通方法返回一样清晰，只是发生在另一个线程中。可靠的停止协议本质上就是把“外部取消请求”转化为“任务在合适位置自然返回”。如果任务能够主动检查取消状态并从 `run`、`call` 或循环方法中返回，停止线程就仍然遵循普通控制流，异常和资源清理也更容易推理。

相反，强行在任意指令点终止线程是危险的。线程可能正在修改两个相关字段中的第一个，可能已经从队列取出元素但还没处理完，可能持有某个对象的监视器，可能执行到一段需要 finally 收尾的资源代码。外部无法知道此刻是否安全，也无法替任务补全业务不变式。Java 早期提供过一些强制控制线程的方法，但现代并发编程应把它们视为危险遗留接口，优先使用中断、状态标志、超时和执行器关闭等协作方式。

停止线程至少涉及四个角色。第一，取消请求者，负责表达“希望任务结束”；它可能是用户线程、管理线程、超时控制逻辑，也可能是应用关闭流程。第二，任务执行者，负责在合适位置观察取消请求并退出。第三，等待者，可能正在等待任务完成、等待结果、等待计数器归零或等待线程终止。第四，资源所有者，负责释放任务运行期间获得的资源。简单程序中这些角色可能混在一起，但设计时仍应分开思考，否则很容易出现“请求已经发出，但工作线程不知道”“工作线程退出了，但资源没有关闭”“资源关闭了，但等待者没有被通知”的断裂。

## 协作式停止的基本模型

协作式停止可以概括为三句话：外部发出取消信号，任务在安全点观察信号，任务执行清理后退出。取消信号可以是线程中断，也可以是一个受同步机制保护的状态变量；安全点可以是循环条件、阻塞方法返回、任务阶段边界或资源处理边界；清理动作通常放在 `finally` 或 try-with-resources 中。三者缺一不可：只有信号没有检查，任务不会停；只有检查没有唤醒，阻塞任务可能永远停不下来；只有退出没有清理，程序状态仍然不完整。

一个最小的协作式循环如下：

```java
public final class Worker implements Runnable {
    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            doOneUnitOfWork();
        }
    }

    private void doOneUnitOfWork() {
        // 执行一个足够小的工作单元
    }
}
```

这个例子只展示了结构。真正值得关注的是“工作单元”的大小。如果一个工作单元需要几毫秒，中断后任务能很快退出；如果一个工作单元可能运行几分钟，并且内部没有任何取消检查，那么循环条件再正确也只能在几分钟后生效。协作式取消要求任务把长期工作拆成可观察的阶段，在阶段之间检查停止信号。检查频率不是越高越好，而是要与任务可接受的取消延迟匹配：太低会导致关闭缓慢，太高会把普通计算切得过碎，增加分支和同步成本。

协作式停止还要求取消信号具备可见性。一个线程设置了“应该停止”，另一个线程必须可靠看到。普通布尔字段没有跨线程可见性保证，编译器和处理器也可能在不破坏单线程语义的前提下做优化，使工作线程长期看不到写入。因此，停止标志至少应使用 `volatile`，或者由锁、原子变量、并发工具内部同步关系保护。中断标志由 JVM 和类库维护，适合表达“这个线程应尽快停止当前等待或当前任务”；`volatile` 标志适合表达更业务化的生命周期状态，例如某个服务循环是否继续运行。

## 中断机制不是强制终止

`Thread.interrupt()` 是 Java 中停止线程最核心的协作信号。调用某个线程对象的 `interrupt` 方法，会设置该线程的中断状态；如果该线程正阻塞在某些可中断方法上，这些方法会以约定方式响应中断，通常是抛出 `InterruptedException` 并清除中断状态。需要强调的是，中断不会把线程从 CPU 上强制拿掉，也不会跳过当前代码直接结束 `run` 方法。它只是一个请求：请被中断线程在合适位置处理取消。

中断机制包含几个常用操作。`thread.interrupt()` 向目标线程发出中断请求；`thread.isInterrupted()` 查询目标线程的中断状态，不会清除标志；`Thread.currentThread().isInterrupted()` 查询当前线程状态，也不清除标志；静态方法 `Thread.interrupted()` 查询当前线程中断状态并清除它。最后一个方法容易误用，因为它有副作用。如果只是想判断当前线程是否被请求停止，通常更适合使用 `Thread.currentThread().isInterrupted()`，避免不小心吞掉中断。

可中断阻塞方法的行为尤其重要。`Thread.sleep`、`Object.wait`、`Thread.join`、许多 `java.util.concurrent` 阻塞队列方法、部分锁和同步器的可中断等待方法，在检测到中断时会抛出 `InterruptedException`。抛出这个异常意味着当前阻塞已经被取消，同时中断状态通常已被清除。于是捕获异常的代码必须作出决定：当前层能否完成取消处理并退出？如果能，就做清理后返回；如果不能，应该恢复中断状态，把取消请求交还给上层。

```java
try {
    queue.take();
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    return;
}
```

恢复中断状态不是形式主义。方法签名如果不能继续抛出 `InterruptedException`，但当前层又没有资格把中断视为已经处理完成，就应调用 `Thread.currentThread().interrupt()`。否则，上层代码再检查中断状态时会发现标志已经消失，关闭流程可能误以为任务没有收到取消请求。许多“线程池关闭很慢”“后台线程停不下来”的问题，本质上是某一层捕获并吞掉了 `InterruptedException`，导致取消信号在调用链中丢失。

并不是所有阻塞都响应中断。进入 `synchronized` 等待获取监视器锁的 BLOCKED 状态时，线程不会因为中断而立刻获得锁或抛出异常；某些传统 I/O 或外部库调用也可能不按 Java 中断语义返回；纯 CPU 计算如果从不检查中断，也不会自动停止。设计取消协议时，必须识别任务可能卡在哪些位置。如果某段调用不响应中断，就需要提供其他退出手段，例如关闭底层资源、设置超时、避免无界等待、把不可中断调用隔离到可管理的线程或进程边界中。

## 中断状态的传播原则

处理中断的第一原则是不要无声吞掉。`InterruptedException` 不只是普通异常，它携带的是线程间的控制信号。捕获它之后继续执行普通逻辑，等于告诉调用者“我已经处理了取消”，但实际上任务可能既没有退出，也没有恢复标志。正确做法通常只有三类：继续向外抛出 `InterruptedException`；完成必要清理后返回；在无法抛出的地方恢复中断标志并尽快结束当前工作。

```java
public void run() {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            doInterruptibleWork();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        cleanup();
    }
}
```

这个结构表达了一个清晰含义：循环正常执行时检查中断；阻塞方法抛出中断异常时恢复标志；无论如何都执行清理；`run` 方法随后结束。恢复标志之后不应再进入长时间工作，否则恢复的意义会被削弱。很多代码在 catch 中恢复中断后继续下一轮循环，下一轮又调用清除中断状态的方法，结果仍然可能拖延退出。恢复中断应与“当前层即将让出控制权”配合使用。

如果方法本身声明 `throws InterruptedException`，通常最清晰的选择是继续抛出，让调用方决定取消策略。低层工具方法不应擅自把中断转换成业务默认值，因为它不知道调用方正在执行的是普通等待、关闭流程还是超时控制。只有当当前方法明确拥有整个任务生命周期，且已经完成全部清理，才可以把中断视为已消费。例如一个工作线程的顶层 `run` 方法捕获中断、记录必要信息、释放资源并返回，就是合理的终点。

还要注意 `Future.get`、`BlockingQueue.take`、`CountDownLatch.await` 这类方法抛出的 `InterruptedException` 发生在调用线程，而不是被等待任务的线程。调用线程被中断，说明“等待动作”应取消；它不自动说明后台任务已经停止。如果等待者决定不再需要结果，通常还应结合 `Future.cancel(true)` 或其他取消信号通知任务本身。把“等待被中断”和“任务已取消”混为一谈，会让后台工作继续占用资源。

## volatile 停止标志

`volatile` 停止标志是另一种常见协作方式。它适合任务主要运行在循环中、没有大量可中断阻塞，或者需要表达比线程中断更具体的业务状态。`volatile` 写入对后续读取可见，能保证一个线程设置停止标志后，另一个线程在读该标志时能观察到更新。它不能让线程从阻塞方法中醒来，也不能让复合操作自动具备原子性，因此更适合作为“是否继续运行”的状态发布机制，而不是万能取消工具。

```java
public final class FlagWorker implements Runnable {
    private volatile boolean running = true;

    public void shutdown() {
        running = false;
    }

    @Override
    public void run() {
        while (running) {
            doOneUnitOfWork();
        }
        cleanup();
    }
}
```

这段代码只有在 `doOneUnitOfWork` 本身不会长期阻塞时才足够。如果工作线程正阻塞在 `queue.take()` 上，另一个线程把 `running` 设置为 `false` 并不会唤醒 `take`。工作线程只有在队列中来了一个元素、阻塞返回并重新检查循环条件后才会退出。此时更好的协议可能是中断工作线程、使用带超时的 `poll`、投递特殊结束标记，或者关闭使阻塞操作返回的底层资源。停止标志解决“可见”，不解决“唤醒”。

`volatile` 标志也不适合表达复杂状态机。比如一个服务有 NEW、RUNNING、STOPPING、TERMINATED、FAILED 多个状态，还要保证只能从 RUNNING 转到 STOPPING，失败原因只记录一次，停止后不能重新启动。这时单个 boolean 过于粗糙，可能需要 `AtomicReference`、锁保护的状态机，或者更高层的生命周期管理对象。停止协议越复杂，越要把状态转换写清楚，避免多个线程同时发起启动、停止和失败处理时互相覆盖。

中断和 `volatile` 并不是二选一。许多可靠实现会同时使用二者：`volatile` 或原子状态表示组件生命周期，中断用于唤醒正在阻塞的工作线程。外部调用 `shutdown` 时先把状态改为停止中，再中断工作线程；工作线程从阻塞中返回后检查状态，执行清理并进入终止状态。这样既有明确的业务状态，又有让阻塞点尽快返回的机制。只用中断可能缺少业务语义，只用标志可能无法唤醒阻塞。

## 阻塞方法如何响应停止

停止线程最困难的场景往往不是 CPU 循环，而是等待。线程可能在等待时间、等待锁、等待条件、等待队列元素、等待任务结果、等待外部资源返回。每一种等待都有不同的取消能力。设计停止协议时，应逐一列出任务可能阻塞的位置，并确认每个位置是否可中断、是否可超时、是否能通过关闭资源唤醒，以及唤醒后如何判断是正常结果还是取消。

`Thread.sleep` 会响应中断，抛出 `InterruptedException`。如果睡眠只是重试间隔，捕获中断后通常应恢复中断并退出重试循环，而不是继续下一次重试。`Object.wait` 也会响应中断，但它必须在持有对象监视器时调用，并且返回后要重新检查等待条件。因为等待可能被通知、超时或中断打断，也可能出现虚假唤醒。停止逻辑不能只靠“被唤醒了”判断成功，而要回到条件本身。

```java
synchronized (lock) {
    while (!ready && !cancelled) {
        lock.wait();
    }
    if (cancelled) {
        return;
    }
    useReadyState();
}
```

阻塞队列通常提供可中断方法和带超时方法。`take` 和 `put` 可中断，适合在线程池或工作线程关闭时快速返回；`poll(timeout, unit)` 和 `offer(timeout, unit)` 在中断之外还提供时间边界，适合需要周期性检查其他状态的循环。选择哪一个取决于取消协议。如果只靠 `take`，必须保证停止时会中断线程或投递结束标记；如果使用超时 `poll`，可以周期性检查 `volatile` 状态，但会引入最长一个超时时间的取消延迟。

获取锁也有差异。`synchronized` 的入口等待不可中断，线程在等待监视器时即使被中断，也要等获得锁后才能继续执行并观察中断状态。`ReentrantLock.lockInterruptibly()` 则允许等待锁时响应中断，适合可能长时间竞争并且需要取消的场景。普通 `lock()` 不响应等待期间的中断，但获得锁后中断状态仍然存在，可以由后续逻辑检查。对需要关闭及时性的代码，锁获取方式本身就是取消协议的一部分。

等待另一个任务结果时也要谨慎。`Thread.join` 和 `Future.get` 都会在当前等待线程被中断时抛出 `InterruptedException`。这只取消了等待动作，不必然取消被等待线程或任务。若调用方在被中断后决定放弃结果，应考虑是否还要通知被等待任务停止，否则任务可能继续运行。对于 `Future`，可以调用 `cancel(true)` 请求中断执行该任务的线程；但如果任务已经完成，取消无效；如果任务未开始，可能被标记为取消；如果任务正在运行，是否退出仍取决于任务是否响应中断。

## 资源清理与不变式维护

停止线程必须和资源清理放在一起讨论。一个任务只要获取了需要归还的东西，就必须考虑取消发生时如何归还。这里的资源不只包括文件句柄、网络连接、数据库连接，也包括锁、信号量许可、队列中的占位、临时文件、线程本地变量、注册到其他组件的监听器、计数器中的“正在运行”状态。取消请求可能在任意可中断等待处到来，如果清理逻辑不在 finally 中，资源就可能泄漏。

```java
Semaphore semaphore = new Semaphore(10);

public void process() throws InterruptedException {
    semaphore.acquire();
    try {
        doWork();
    } finally {
        semaphore.release();
    }
}
```

这个例子中，`acquire` 成功后必须确保 `release` 执行；如果 `acquire` 本身被中断并抛出异常，则没有获得许可，也不能释放。取消协议经常需要区分“资源是否已经获得”。类似地，锁只在获得后释放，文件只在打开后关闭，任务只在放入队列后才需要撤销或标记。粗糙的 finally 可能造成反向错误，例如释放未获得的许可、关闭不属于自己的资源、错误地通知等待者任务已完成。

资源清理还要维护业务不变式。假设任务需要先从输入队列取出一条记录，再更新两个相关结构，最后写入完成队列。取消可能发生在记录取出之后、结构更新之前，也可能发生在结构更新一半时。安全停止要求任务在这些边界上有明确策略：要么把记录重新放回或记录为失败，要么保证更新是原子的，要么把任务设计成幂等可重试，要么在阶段之间保存足够的恢复信息。单纯退出线程并不能自动恢复业务状态。

清理动作本身也应尽量短小、可靠、不可无限阻塞。finally 中如果执行长时间等待，停止流程仍然会卡住；如果清理时再次吞掉中断，上层也无法感知关闭受阻。需要关闭多个资源时，应尽量让每个资源关闭失败都能被记录，同时继续清理其他资源。try-with-resources 可以简化普通资源释放，但对需要状态转换、通知等待者、恢复中断标志的任务，仍要显式写出生命周期收尾逻辑。

线程本地变量是容易忽略的清理对象。在线程池中，工作线程会被复用，一个任务写入的 `ThreadLocal` 如果不清理，可能污染后续任务。停止线程时如果任务因为异常或中断提前结束，更要在 finally 中移除线程本地状态。否则即使线程没有泄漏，线程关联的数据也可能长期留在工作线程上，造成内存占用或行为串扰。这里停止的不是线程池工作线程，而是当前任务的上下文。

## 为什么不要使用 Thread.stop

`Thread.stop` 的问题在于它会异步地向目标线程抛出 `ThreadDeath`，让目标线程在不可预测的位置终止，并释放它持有的监视器锁。释放锁听起来像是避免死锁，实际却会破坏锁保护的不变式。锁的意义是保护一段临界区完整执行：进入临界区后对象可能暂时处于中间状态，直到临界区末尾才恢复一致。如果线程在中间被强行停止，监视器被释放，其他线程就可能进入并观察到损坏状态。

设想一个对象用同一把锁保护两个字段，要求二者始终满足某种关系。线程在持锁状态下先更新字段 A，再更新字段 B。正常情况下，其他线程只有在锁释放后才能看到 A 和 B 的一致组合；但如果 `stop` 在更新 A 之后、更新 B 之前发生，锁会被释放，其他线程可以获得锁并看到不一致对象。更糟的是，这种损坏未必立刻抛异常，可能在后续很远的位置表现为错误结果，使问题难以定位。

`stop` 还会绕过任务自己的取消协议。任务无法选择安全点，无法保证当前操作完成或回滚，无法决定哪些资源应关闭，无法把失败原因转换成可理解的状态。虽然 finally 块在异常传播时通常有机会执行，但异步异常可能出现在几乎任何位置，清理代码本身也可能处于复杂状态；而且对象不变式已经可能在进入 finally 之前被破坏。可靠系统不应把正确性建立在“强制异常刚好出现在安全位置”这种偶然上。

因此，现代 Java 代码应避免使用 `Thread.stop`、`suspend`、`resume` 这类强制控制线程的方法。它们的问题不是“风格过时”，而是与对象封装、锁语义和资源生命周期根本冲突。停止线程应通过任务可理解的信号完成：设置状态、发出中断、唤醒等待、关闭资源、等待终止。这样任务可以在自己维护不变式的位置退出，调用方也能通过明确的状态判断停止是否完成。

## 取消协议的设计

一个完整的取消协议至少要回答六个问题。第一，谁可以发起取消？如果多个线程都能调用 `shutdown`，该方法是否幂等？第二，取消信号是什么？使用中断、`volatile` 状态、原子状态机、队列结束标记，还是组合使用？第三，任务在哪里检查信号？是每轮循环、每个批次、每次阻塞返回，还是每个阶段边界？第四，阻塞点如何被唤醒？依靠中断、超时、通知、关闭资源，还是放入特殊消息？第五，取消后如何清理资源并发布终止状态？第六，调用方如何等待或确认停止完成？

下面是一个带生命周期状态的简化模型：

```java
public final class ServiceLoop implements Runnable {
    private final BlockingQueue<Runnable> queue = new LinkedBlockingQueue<>();
    private volatile boolean running = true;
    private Thread owner;

    public void start() {
        Thread thread = new Thread(this, "service-loop");
        owner = thread;
        thread.start();
    }

    public void shutdown() {
        running = false;
        Thread thread = owner;
        if (thread != null) {
            thread.interrupt();
        }
    }

    @Override
    public void run() {
        try {
            while (running || !queue.isEmpty()) {
                Runnable task = queue.poll(1, TimeUnit.SECONDS);
                if (task != null) {
                    task.run();
                }
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            cleanup();
        }
    }
}
```

这个例子展示的是设计思路，不是通用框架。`running` 表达是否继续接收长期循环，`interrupt` 用于唤醒 `poll`，循环条件决定是否排空已有任务，finally 负责清理。真实代码还需要处理任务提交时的状态检查、`owner` 的安全发布、任务异常隔离、停止后拒绝新任务、重复启动限制等问题。重点在于：取消协议不是一个孤立方法，而是一组状态转换和责任划分。

取消协议还要定义“取消后是否处理已接收任务”。有些场景要求温和关闭：不再接受新工作，但已经进入队列的任务继续处理完；有些场景要求立即关闭：尽快中断当前工作并丢弃未开始任务；还有些场景要求阶段性关闭：当前批次完成后退出。不同策略没有绝对优劣，但必须清楚表达。调用方如果以为 `shutdown` 会立即停止，而实现实际会排空队列，就可能误判关闭耗时；实现如果丢弃任务却没有返回未处理清单，则可能造成数据丢失。

取消协议应当具备幂等性。关闭方法可能被多个线程调用，也可能在失败处理和正常关闭路径中重复调用。幂等关闭意味着第一次调用触发状态转换和唤醒，后续调用不会破坏状态，也不会重复释放资源。用 `AtomicBoolean.compareAndSet` 或锁保护状态转换都可以实现这一点。没有幂等性时，常见问题包括重复关闭同一资源、重复发送终止消息、重复记录失败、或者关闭过程中又接受了新任务。

## 线程池中的停止关系

使用线程池后，停止线程更应理解为取消任务和关闭执行器，而不是直接操作工作线程。`ExecutorService` 把任务提交、排队、执行和关闭封装起来。调用方通常不拥有工作线程，因此不应随意保存并中断线程池内部线程；正确入口是 `Future.cancel`、`shutdown`、`shutdownNow`、`awaitTermination` 以及任务自身对中断的响应。

`shutdown()` 表示温和关闭：执行器不再接受新任务，但已提交任务会继续执行。它不会中断正在运行的任务。调用 `shutdown` 后如果还提交任务，通常会触发拒绝策略。调用方如果需要等待关闭完成，应再调用 `awaitTermination`，并处理等待期间的中断。`shutdown` 只是发起关闭，不等于所有任务已经结束。

`shutdownNow()` 表示尝试立即关闭：执行器会尝试中断正在执行的任务，并返回尚未开始执行的任务列表。这里的“尝试”非常重要。正在运行的任务是否结束，取决于任务是否响应中断；如果任务忽略中断、卡在不可中断阻塞或执行很长的 CPU 计算，线程池仍然可能无法及时终止。`shutdownNow` 不是强杀，它只是更积极地发出取消请求并清理队列中未开始的任务。

`Future.cancel(boolean mayInterruptIfRunning)` 用于取消单个任务。参数为 `true` 时，如果任务已经开始执行，执行器可以中断执行该任务的线程；参数为 `false` 时，只取消尚未开始的任务，不中断正在运行的任务。即使传入 `true`，也仍然依赖任务对中断的协作。被取消的 `Future` 后续调用 `get` 会得到 `CancellationException`，这表示结果不可用，不表示任务内部一定在传入取消请求的瞬间已经完成清理。需要强一致关闭时，仍要结合额外的完成确认。

一个常见的线程池关闭模板如下：

```java
public static void shutdownAndAwait(ExecutorService executor) {
    executor.shutdown();
    try {
        if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
            executor.shutdownNow();
            if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
                // 记录仍未终止的情况
            }
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}
```

这个模板表达了几层含义：先温和关闭，给已提交任务自然结束的机会；超时后尝试立即关闭；等待关闭的线程如果被中断，也请求立即关闭并恢复中断状态。它仍然不是万能保证，因为任务如果不响应中断，执行器可能无法终止。但它比单独调用 `shutdown` 或 `shutdownNow` 更完整，至少把关闭请求、等待边界、升级策略和中断传播放在同一段逻辑中。

在线程池任务内部，最重要的是不要破坏工作线程。任务捕获 `Throwable` 后不加区分地吞掉，可能掩盖严重错误；任务长时间阻塞且不响应中断，会拖住工作线程；任务修改线程名、上下文类加载器、`ThreadLocal` 后不恢复，会影响后续任务。线程池复用线程的特性要求任务把自己的上下文清理干净。停止一个任务时，不能假设线程会随任务结束而销毁。

## 中断、标志和结束标记的选择

中断适合表达“请这个线程停止当前等待或尽快结束当前任务”，尤其适合阻塞方法支持 `InterruptedException` 的场景。它的优点是标准、轻量、被并发类库广泛支持；缺点是语义较通用，无法直接表达业务状态，也不能唤醒所有阻塞。使用中断时要维护传播规则，不要随意清除或吞掉。

`volatile` 或原子标志适合表达组件生命周期，例如是否继续运行、是否接受新任务、当前状态是否为停止中。它的优点是语义明确，读取成本低，适合循环检查；缺点是不能唤醒阻塞，不能保证复合状态转换，不能替代锁保护复杂不变式。使用标志时要保证所有读写都走同一套同步关系，不要一部分读 volatile，一部分读普通缓存字段。

结束标记适合生产消费队列。生产者放入一个特殊对象表示“没有更多数据”，消费者取到后退出。它的优点是与数据流合并，不需要持有消费者线程对象；缺点是多消费者场景要投递足够数量的结束标记，队列满时结束标记本身可能放不进去，而且它不适合立即取消正在处理的长任务。结束标记更像“输入结束”，不是通用线程取消。

超时适合给等待设置上界。它可以避免永久等待，让调用方有机会记录、重试或升级取消。超时不是严格取消信号：一次等待超时只能说明在这段时间内条件没有满足，不能说明被等待任务已经失败。滥用很短的超时会造成忙轮询和日志噪声；完全没有超时则可能让关闭流程永久挂起。合理做法是把超时与状态检查、取消请求和诊断信息结合起来。

实际工程中常常组合使用这些手段。例如一个消费者线程用 `volatile running` 表示服务是否继续，使用 `BlockingQueue.poll(timeout)` 周期性醒来检查状态，关闭时设置 `running=false` 并调用 `interrupt` 加快退出；对于普通任务，提交方通过 `Future.cancel(true)` 发中断，任务内部捕获 `InterruptedException` 后清理并返回；对于数据流处理，生产者发送结束标记表示正常结束，异常关闭则中断消费者并关闭资源。选择依据不是 API 喜好，而是任务可能停在什么地方。

## 边界：不可中断、长计算和外部调用

有些代码无法只靠中断优雅停止。等待 `synchronized` 锁时不可中断，某些外部库调用不响应 Java 中断，某些系统调用或远程调用只有自己的超时和关闭机制，纯 CPU 计算如果没有检查点也不会停。面对这些边界，可靠设计不能假装 `interrupt` 万能，而要在任务结构上预留退出路径。

对于长时间 CPU 计算，应主动设置检查点。检查点可以放在外层循环、批次边界、递归展开边界或每处理固定数量元素之后。检查点既可以检查中断，也可以检查取消标志。检查频率要综合计算成本和取消延迟：如果每次迭代都极短，没必要每步都读取 volatile；如果单次迭代可能很长，就应在迭代内部拆分阶段。长计算的取消质量通常取决于算法结构，而不是线程 API 本身。

对于不可中断的锁等待，应减少持锁时间、避免在持锁期间执行慢操作，并在可能取消的场景使用可中断锁获取方式。`ReentrantLock.lockInterruptibly` 可以让等待锁的线程响应中断，但它要求代码用显式锁重写临界区，并用 finally 解锁。是否替换 `synchronized` 取决于需求：如果锁竞争短且无需取消，`synchronized` 简洁可靠；如果锁等待可能很长并且关闭及时性重要，可中断锁更合适。

对于外部调用，应优先使用外部资源自身的超时和关闭机制。网络请求、文件通道、数据库驱动、消息客户端等通常有自己的取消或超时模型，Java 线程中断未必能让它们立即返回。停止协议可以在关闭时中断线程，同时关闭底层资源或触发客户端取消。这样即使调用不检查中断，也可能因为资源关闭而返回错误。需要注意的是，关闭共享资源会影响其他使用者，因此资源所有权必须清楚。

对于无法可靠停止的第三方代码，最保守的做法是隔离。可以限制它运行在线程池中的专用线程，设置任务超时和容量上限，避免占满关键执行器；必要时用进程级隔离，把真正的强制终止放到进程边界。Java 线程内部的协作式取消无法保证任意不受控代码都按期退出，因此设计上要把不可控部分的影响范围收窄。

## 常见误区

第一个误区是把 `interrupt` 当成强制停止。调用 `interrupt` 后马上假设任务已经结束，是错误的。中断只是请求，任务可能还在执行清理，可能正在不可中断区域，可能压根没有检查中断。需要确认结束时，应使用 `join`、`Future.get`、`awaitTermination` 或自定义终止状态等待，并给等待设置边界。

第二个误区是吞掉 `InterruptedException`。很多代码写成 catch 后打印日志继续循环，或者转换成运行时异常但不恢复中断状态。这会让上层关闭逻辑失去信号。处理 `InterruptedException` 时要明确当前层是否拥有取消语义；没有拥有权就继续抛出或恢复标志。

第三个误区是用普通 boolean 做停止标志。普通字段在多线程之间没有及时可见性保证，工作线程可能长期看不到更新。即使某次测试能停，也不能证明程序正确。停止标志应使用 `volatile`、原子变量或锁保护，并确保读写两端使用同一机制。

第四个误区是只设置标志不唤醒阻塞。工作线程如果阻塞在 `take`、`wait`、`join` 或外部调用上，单纯写 `running=false` 可能没有任何即时效果。停止协议必须覆盖阻塞点：中断、通知、超时、结束标记或关闭资源，至少要有一种能让线程离开等待。

第五个误区是用 `Thread.stop` 追求立即见效。它可能在临界区中间破坏对象不变式，并释放监视器让其他线程观察损坏状态。强制终止带来的不是可靠停止，而是把错误推迟到更难解释的位置。除非是在极特殊的受控实验或遗留兼容语境中分析行为，正常代码不应使用它。

第六个误区是关闭线程池后不等待。`shutdown` 只是拒绝新任务，已提交任务仍会运行；`shutdownNow` 只是尝试中断，不能保证立即结束。需要完成关闭流程的代码应调用 `awaitTermination`，并处理超时和中断。否则方法返回时后台任务可能仍在修改状态或占用资源。

第七个误区是混淆取消和失败。取消通常表示调用方不再需要结果或系统正在关闭，不一定表示任务逻辑出错；失败表示任务尝试完成但遇到异常。二者都可能导致没有正常结果，但记录、重试、告警和资源处理策略不同。设计任务状态时，应区分正常完成、取消、失败和超时。

第八个误区是在 finally 中做不可控等待。很多人知道要在 finally 中清理，却忽略清理也可能阻塞。如果关闭资源、刷新数据或通知其他线程的动作可能长时间等待，就需要超时、降级或诊断。否则取消请求已经到达，线程却卡在清理阶段，调用方仍然无法完成关闭。

## 排查停止线程问题的思路

排查线程停不下来时，先确认线程实际停在哪里。线程 dump 可以看到它是 RUNNABLE、BLOCKED、WAITING 还是 TIMED_WAITING，并能显示调用栈。RUNNABLE 不一定表示正在消耗 CPU，可能在本地方法或 I/O；BLOCKED 通常表示等待进入 `synchronized`；WAITING 和 TIMED_WAITING 可能来自 `wait`、`park`、`sleep`、`join` 或并发工具。只有知道阻塞位置，才能判断中断、标志、超时或资源关闭哪个缺失。

第二步检查取消信号是否发出。调用 `interrupt` 的线程对象是不是正确的那个？线程池任务是否通过 `Future.cancel(true)` 或执行器关闭发出中断？`volatile` 标志是否真的被写为停止？写入是否发生在另一个工作线程能看到的对象实例上？很多停止失败不是机制问题，而是发错对象、保存了旧引用、关闭了错误的执行器，或者任务已经被包装后调用方拿不到实际执行边界。

第三步检查信号是否被吞掉。搜索 `catch (InterruptedException`，看是否继续抛出、恢复中断或返回。检查是否调用了 `Thread.interrupted()` 并无意清除了标志。检查循环条件是否在每次长工作之后重新读取状态。检查第三方库是否捕获中断后继续重试。中断丢失经常发生在工具方法中，而表现却出现在最外层关闭流程。

第四步检查资源清理是否卡住。线程可能已经离开主工作循环，但阻塞在 finally 的关闭动作、日志刷新、回调通知或等待其他任务完成上。此时继续发送中断未必有用，因为清理逻辑可能没有响应。需要分析清理动作是否持有锁、是否等待同一个线程池中的任务、是否关闭共享资源导致互相等待，是否在异常路径中遗漏了通知。

第五步检查线程池关闭语义。确认是否调用了 `shutdown` 后仍有任务不断提交，拒绝策略是否掩盖了问题；确认是否对 `awaitTermination` 设置了合理时间；确认 `shutdownNow` 返回的未执行任务是否被处理；确认工作任务是否响应中断。如果线程池无法终止，问题通常在任务内部，而不是执行器关闭 API 本身。

## 实践建议

设计可停止任务时，优先让任务拥有清晰的生命周期方法，例如 `start`、`shutdown`、`awaitTerminated`，而不是把线程对象暴露给任意调用方。暴露线程对象会让外部随意中断、设置优先级或等待，破坏封装。更好的方式是由组件内部管理线程和取消协议，外部只表达“开始、停止、等待停止”的意图。

任务循环应围绕工作单元组织。工作单元太大会导致取消延迟，太小会增加调度和同步成本。每个工作单元结束后检查取消，每个可能阻塞的位置要么可中断，要么有超时，要么能通过关闭资源唤醒。循环条件应表达业务状态，而不是散落多个不一致的布尔变量。

尽量把取消作为正常路径之一，而不是异常的附属品。关闭系统、超时放弃、用户取消、上游不再需要结果，都是正常并发场景。代码中应有清晰的取消分支、清理分支和完成通知。把取消都记录成错误日志会制造噪声；完全不记录取消又会让排查困难。记录级别和内容应反映取消是否符合预期。

共享状态更新应放在可维护不变式的位置。不要在更新一半时调用可能长时间阻塞或可中断的方法；不要持锁执行无法控制耗时的外部调用；不要让取消在多个字段更新之间留下不一致状态。如果必须跨多个步骤更新，考虑使用局部临时结果，全部准备完成后一次性发布，或用锁保护完整提交阶段。

等待关闭时要给出时间边界和诊断。无限期 `join` 或无限期 `awaitTermination` 会让关闭流程缺少退路。设置超时后，不应只是静默返回，而要记录未完成任务数量、线程状态或关键上下文。超时本身不解决问题，但它让系统有机会把问题暴露出来。

最后，测试停止逻辑不能只测试正常完成。应覆盖运行中取消、阻塞中取消、取消后重复调用关闭、任务抛异常时关闭、关闭后继续提交、等待关闭被中断、资源清理抛异常等路径。并发停止问题经常藏在异常路径和收尾阶段，只有这些路径被验证，停止协议才真正可信。

## 小结

停止线程的核心不是让线程“立刻消失”，而是让任务在安全点结束。Java 推荐的方式是协作式取消：用中断表达通用取消请求，用 `volatile` 或原子状态表达生命周期，用可中断阻塞、超时、通知或资源关闭让等待返回，用 finally 和 try-with-resources 完成清理，用 `join`、`Future` 或执行器终止等待确认结果。

`Thread.stop` 之所以危险，是因为它把终止点从任务自己选择的安全边界，变成外部不可预测的任意位置，可能破坏锁保护的不变式。相比之下，中断和停止标志看似“慢”，但它们保留了程序维护一致性的能力。可靠的并发程序宁愿把取消延迟控制在可解释范围内，也不应为了表面的立即停止牺牲状态完整性。

写停止线程相关代码时，可以反复检查几个问题：取消信号是否可见，阻塞点是否能醒来，长计算是否有检查点，异常路径是否释放资源，中断是否被正确传播，线程池关闭是否等待确认，取消、失败和正常完成是否被区分。只要这些问题都有明确答案，停止线程就不再是一个危险动作，而是任务生命周期中可设计、可验证、可维护的一部分。

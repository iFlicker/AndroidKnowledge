# 3.3.1.6 守护线程与优先级

## 定位
守护线程与优先级是 Java 并发编程中的一个独立主题，它解决的问题是：理解线程属性对 JVM 退出和调度倾向的影响。学习这个主题时，不能只记一个类名或一个方法调用，而要把它放回并发程序的完整链路中理解：任务从哪里产生，在哪个线程中执行，是否访问共享状态，遇到等待和取消时如何响应，结果或异常又如何回到调用方。只有这些问题都清楚，代码才不仅是“能跑”，而是在压力、异常和关闭阶段仍然可解释。

阅读 Java 并发文档时要把三个层次分开：语言层面的语义、类库层面的工具、运行时和操作系统层面的调度。语言语义回答“哪些结果是被规范允许的”，类库工具回答“怎样表达等待、通知、排队、取消和结果传递”，运行时调度回答“什么时候真正获得 CPU、什么时候被阻塞、什么时候被唤醒”。很多并发错误来自把这三层混在一起，例如用时间先后推导可见性，用线程优先级推导执行顺序，或者用一个看起来没有报错的测试结果推导程序在高并发下也正确。

从知识体系上看，守护线程与优先级既连接基础线程模型，也连接后续的线程安全、线程池、同步器和内存模型。它的学习目标不是背诵 API，而是建立判断能力：什么时候该使用它，什么时候应该换成更高层或更低层工具，什么时候一个看似简单的写法会破坏可见性、原子性或生命周期。本文保持通用 Java 技术视角，只讨论 Java 语言和标准并发库中的机制、边界与实践。

## 解决的问题
守护线程与优先级面向的直接问题是理解线程属性对 JVM 退出和调度倾向的影响。在单线程程序中，控制流沿着调用栈顺序前进，数据读写通常由同一个线程完成；进入并发环境后，执行顺序、等待位置、异常传播和状态可见性都可能分离。这个主题的价值就在于给这种分离建立清晰规则，让开发者知道哪些行为由规范保证，哪些只是当前机器上的偶然结果。

守护线程影响进程是否等待它结束，优先级只是调度提示。二者都不是并发正确性的同步工具。

更具体地说，守护线程与优先级会影响三个方面。第一是执行关系：某段代码由哪个线程执行，是否可能与其他代码并行，是否存在先后约束。第二是状态关系：共享变量在多个线程之间是否有一致的观察方式，复合操作是否会被打断，更新是否能被其他线程及时看见。第三是生命周期关系：任务开始、等待、取消、失败、完成和资源释放是否有明确路径。并发代码一旦缺少这些关系，就容易出现测试环境偶尔通过、生产压力下随机失败的问题。

## 核心 API 与术语
围绕守护线程与优先级，最常见的 API 或术语包括：setDaemon、setPriority、NORM_PRIORITY、未捕获异常处理器。这些名称背后不是孤立功能点，而是一组协作约定。阅读标准库实现或写业务代码时，应优先识别它们承担的是执行、排队、互斥、条件等待、结果传递、取消还是可见性职责。职责判断清楚后，API 的选择通常会变得直接。

启动前设置属性、区分辅助线程和关键线程、避免依赖优先级保证顺序。这条流程描述了守护线程与优先级在典型使用中的状态变化。流程中的每一步都可能失败或被中断，因此真实代码不能只写“理想路径”。例如等待可能超时，提交可能被拒绝，锁可能长期竞争，任务可能抛异常，关闭可能遇到不响应中断的工作线程。把失败路径提前纳入设计，是并发代码和普通顺序代码的重要差异。

下面的片段只用于说明核心用法，重点是观察调用位置和资源边界，而不是把它当成完整框架：

```java
Thread t = new Thread(task, "cleaner");
t.setDaemon(true);
t.start();
```

这类代码最容易被忽略的是边界条件。调用前对象是否已经初始化完成，调用后是否需要等待或关闭，异常是否会被吞掉，当前线程被中断时是否应继续执行，都会影响最终正确性。写并发代码时，每个 API 调用都应回答“失败时怎么办”和“谁负责收尾”。

## 守护线程的语义
守护线程的核心语义不是“后台线程”，也不是“低优先级线程”，而是“当 JVM 判断是否可以自然退出时，不把这个线程当作必须等待的线程”。Java 线程有两类：用户线程和守护线程。用户线程代表程序仍有需要完成的工作；守护线程代表依附于用户线程存在的辅助工作。当所有用户线程都已经结束，只剩守护线程还在运行时，JVM 可以结束进程。此时守护线程中的循环、阻塞等待、finally 代码块、缓冲区刷新、日志写入、锁释放之后的后续逻辑，都不再构成阻止进程退出的理由。

这个语义非常严格，也非常容易被误读。守护线程并不是“在关闭时会被礼貌通知并执行清理”的线程。它没有一个由 Java 语言保证的、类似“请收尾然后退出”的统一回调。只要 JVM 到达只剩守护线程的状态，进程退出可以直接发生，守护线程可能停在任何普通执行点。它可能刚进入 try 块，可能正在 sleep，可能正在 wait，可能正在执行一次循环中的中间步骤，也可能刚拿到某个外部资源但还没有写完。把守护线程当作可靠清理线程，是并发生命周期设计中的典型错误。

更准确地说，daemon 属性描述的是线程和 JVM 生命周期之间的依附关系，而不是线程内部任务的重要性。一个任务是否重要，由业务语义、资源一致性和关闭协议决定；daemon 只影响 JVM 是否等待它。某个线程即使被设置为守护线程，它仍然可以执行复杂计算、持有锁、访问文件、维护缓存或处理网络 I/O；某个线程即使是用户线程，也可能只做很轻量的辅助工作。不能根据工作内容自动推导 daemon 属性，也不能根据 daemon 属性反向推导任务可以随意丢弃。

因此，判断一个线程能否做成守护线程，应从“被中断、被提前终止或来不及收尾时，程序是否仍然保持可接受的一致性”出发。周期性清理过期内存缓存、输出临时诊断信息、维护非关键统计、执行可以重新计算的辅助任务，通常比较适合守护线程。提交交易、写入必须持久化的数据、释放必须严格归还的外部资源、通知其他系统已经完成的状态转换，则不适合作为只由守护线程保障的工作。关键区别不在于代码长短，而在于提前停止是否会破坏系统承诺。

## JVM 退出条件
普通 Java 程序的自然退出可以用一句话概括：所有非守护线程结束后，JVM 不再因为守护线程继续运行而保持存活。这里说的是自然退出条件，不包括显式调用 `System.exit`、底层进程被操作系统终止、虚拟机发生严重错误等非正常路径。自然退出时，主线程本身也只是一个普通用户线程；如果主线程结束但还有其他用户线程在运行，JVM 仍会继续存活；如果主线程结束后只剩守护线程，JVM 就可以退出。

这条规则带来一个重要推论：main 方法返回不等于进程一定结束。很多初学者会把 main 方法看作整个程序的生命周期边界，但在并发程序中，main 方法经常只是启动者。只要它创建的某些用户线程仍在运行，程序就还没有结束。反过来，如果 main 方法创建的是守护线程，并且没有其他用户线程维持存活，那么 main 方法一返回，守护线程可能还没来得及执行几行代码，进程就已经结束。依赖“启动一个守护线程后它总会至少运行一次”的代码是不可靠的。

还要区分 JVM 退出条件和任务完成条件。JVM 存活只说明至少还有一个用户线程未结束，不说明所有任务都正确完成；JVM 退出只说明没有需要等待的用户线程，不说明守护线程已经完成。正确的程序不应把“进程还活着”当成任务成功，也不应把“守护线程没报错”当成资源已释放。任务完成应通过明确的同步关系表达，例如 `join`、`Future.get`、`CountDownLatch.await`、线程池的有序关闭协议，或者由上层生命周期管理器统一等待。

守护线程也不会绕过正常的异常处理规则。守护线程中未捕获的运行时异常会导致该线程终止，默认情况下异常会交给线程的未捕获异常处理器处理。不同之处只是：如果这个守护线程终止后没有用户线程了，JVM 可能随即退出；如果异常发生在用户线程中，只要还有其他用户线程，JVM 仍可能继续运行。无论线程是否守护，都应为长期运行任务设置清晰的异常记录策略，避免后台失败只留下一个难以追踪的输出。

## setDaemon 的时机
`Thread.setDaemon(boolean)` 必须在线程启动前调用。线程一旦调用 `start` 进入启动流程，再修改 daemon 属性会抛出 `IllegalThreadStateException`。这个限制并不是形式主义，而是为了让线程生命周期属性在真正并发执行前固定下来。否则，一个线程已经开始运行后再改变它对 JVM 存活的影响，会让生命周期判断和关闭协议变得难以解释。

从工程习惯上看，daemon 属性应在创建线程的同一处完成配置，并且尽量和线程名、优先级、未捕获异常处理器一起设置。不要把创建线程、设置属性、启动线程分散到多个模块中。属性分散后，调用顺序很容易失控：某个分支先启动了线程，另一个分支再调用 `setDaemon`，测试时可能没有覆盖这个顺序，线上才抛异常。更糟的是，阅读代码的人无法在一个位置判断线程的生命周期归属。

还要注意 daemon 属性的继承。新线程创建时，如果没有显式设置 daemon 属性，通常会继承创建它的当前线程的 daemon 状态。也就是说，在守护线程中创建的新线程，默认也可能是守护线程；在用户线程中创建的新线程，默认通常是用户线程。这个继承行为让“谁负责创建线程”变得更重要。库代码如果在不透明的位置直接创建线程，调用方就很难判断新线程是否会阻止 JVM 退出。因此，面向复用的组件更适合通过 `ThreadFactory` 显式表达线程属性，而不是把属性隐含在调用线程上下文里。

下面的例子强调配置顺序和异常边界。它不是推荐到处手写线程，而是说明裸 `Thread` 使用时应把属性固定在启动前：

```java
Thread worker = new Thread(() -> {
    try {
        runBackgroundLoop();
    } catch (Throwable error) {
        reportFailure(error);
        throw error;
    }
}, "sample-cleaner");

worker.setDaemon(true);
worker.setPriority(Thread.NORM_PRIORITY);
worker.setUncaughtExceptionHandler((thread, error) -> reportFailure(error));
worker.start();
```

这里最值得关注的是，`setDaemon`、`setPriority` 和异常处理器都发生在 `start` 之前。代码仍然没有承诺守护线程一定执行完清理，也没有承诺优先级决定调度顺序；它只是把线程属性集中表达，减少生命周期含糊。真实项目中，如果线程数量不止一个，通常应把这类配置收敛到线程工厂或执行器构造处。

## finally 与资源释放风险
守护线程最危险的误区之一，是以为 `finally` 一定会在进程退出前执行。`finally` 能保证的是：当控制流以正常返回、异常传播等 Java 语义离开 try 块时，会先执行对应的 finally。它不能保证在 JVM 因只剩守护线程而退出时，守护线程里每一个正在执行的 try 块都有机会完成。换句话说，`finally` 是线程内部控制流的收尾工具，不是进程生命周期的持久化协议。

这个区别在资源释放场景中尤其关键。假设守护线程从队列中取出一批数据，先写入内存缓冲，再准备刷新到外部存储。如果所有用户线程结束，JVM 可以在刷新之前退出。即使代码写了 finally，finally 也可能没有机会执行到；即使 finally 开始执行，也不应假设其中的每一步都一定完成。对于必须落盘、必须归还、必须提交或必须通知的工作，应由用户线程或明确的关闭流程承担，不能只放在守护线程的尾部。

更稳妥的做法是把资源生命周期和线程 daemon 属性解耦。关键资源应有显式 `close`、`shutdown` 或 `stop` 方法，由拥有者在线程仍可被等待时调用。关闭过程通常包括：停止接收新任务，发出取消信号，唤醒阻塞线程，等待正在执行的用户线程或任务在超时内结束，记录未完成工作，最后释放资源。守护线程可以参与辅助清理，但不应是唯一负责提交最终状态的一方。

如果确实需要守护线程维护某种资源，也应把设计目标降级为“尽力而为”。例如定期清理临时对象、更新可丢弃缓存、采集近似指标，这些工作即使少执行一次或中途停止，程序仍能保持基本正确。相反，如果少执行一次就会造成数据丢失、重复提交、锁长期占用或外部系统状态不一致，就应把这项工作放入可等待、可取消、可观测的用户线程或线程池任务中。

## 核心机制
守护线程与优先级的核心机制可以从“状态、队列、唤醒、可见性”四个词理解。状态描述对象当前处于哪个阶段，队列描述竞争者或等待者如何排列，唤醒描述等待线程重新获得执行机会的条件，可见性描述一个线程的写入何时能被另一个线程可靠观察到。不是每个主题都会显式包含四个结构，但成熟的并发工具通常都能在内部找到这些要素。

以守护线程与优先级为例，状态变化必须是可解释的。线程从未启动到可运行再到终止，锁从未持有到被某个线程持有再到释放，任务从已提交到执行中再到完成或取消，同步器从计数未满足到条件满足，这些变化都需要明确触发点。如果状态触发点不明确，调用者就会被迫用 sleep、轮询或经验时间等待，而这正是并发程序脆弱的常见来源。

队列和唤醒决定等待成本。阻塞等待可以节省 CPU，但涉及线程挂起和恢复；自旋等待可以避免上下文切换，但会消耗处理器时间；公平排队能降低饥饿概率，但可能牺牲吞吐；非公平竞争能提升局部效率，但等待时间更难预测。理解守护线程与优先级时，不应只问“会不会等待”，还要问“等待者在哪里排队、由谁唤醒、唤醒后是否立即满足条件”。

可见性则决定结果是否可信。一个线程完成了写入，不代表另一个线程一定能立刻看到；一个线程按代码顺序先写 A 再写 B，也不代表另一个线程没有同步就能按同样顺序观察。Java 通过 synchronized、volatile、Lock、线程启动和终止、并发容器等同步动作建立 happens-before 关系。守护线程与优先级如果涉及共享状态，就必须说明它依赖哪一种同步关系。

## 使用方式
使用守护线程与优先级时，第一步是识别任务性质。CPU 密集型任务、I/O 等待型任务、短小状态更新、长时间阻塞任务、阶段同步任务、资源限流任务，适合的工具不同。把所有需求都压到同一个 API 上，会让代码表面简单、实际边界混乱。例如需要结果时不应只用裸 Runnable，需要容量控制时不应只靠线程数量，需要条件协作时不应靠固定 sleep。

第二步是明确所有权。谁创建线程或同步器，谁提交任务，谁负责关闭，谁处理异常，谁记录拒绝，谁保证共享对象不被并发破坏，这些都属于所有权问题。所有权不清时，常见后果是线程池被局部代码关闭、共享锁被不同模块以不同顺序获取、ThreadLocal 被上一个任务污染、异常在后台线程中静默消失。

第三步是把正常路径和异常路径写在一起。只写成功路径的并发代码往往在演示中很好看，但在真实系统中会出现资源泄漏。锁释放要放在 finally，许可归还要放在 finally，计数器递减要放在 finally，ThreadLocal 清理要放在 finally，线程池关闭要处理 InterruptedException，等待结果要考虑 TimeoutException。并发代码的可靠性主要体现在这些收尾位置。

第四步是保留可替换空间。许多并发工具有相似目标但边界不同：wait/notify 可以表达条件等待，Condition 能拆分多个等待队列；CountDownLatch 适合一次性等待，CyclicBarrier 适合固定参与者循环阶段，Phaser 适合动态参与者；Atomic 类适合单变量乐观更新，Lock 适合复杂临界区。写文档和写代码都应说明为什么选择当前工具，而不是只说明它“可以用”。

## 线程优先级的含义
Java 线程优先级通过整数表达，范围从 `Thread.MIN_PRIORITY` 到 `Thread.MAX_PRIORITY`，默认值是 `Thread.NORM_PRIORITY`。它的直观含义是告诉运行时和底层调度系统：在存在多个可运行线程时，某些线程可以被视为更值得获得调度机会。但这个含义必须加上一个限制：优先级只是调度提示，不是执行顺序保证，不是互斥规则，不是实时性承诺，也不是吞吐量必然提升的开关。

线程调度发生在 JVM、操作系统和硬件共同作用的环境中。Java 规范不会要求所有平台都以相同方式解释优先级，也不会要求优先级高的线程一定先运行、一定运行更多时间、一定不会被优先级低的线程阻塞。某些平台会把 Java 的 10 个优先级映射到底层较粗的几个等级，某些平台会受到进程权限、调度策略、负载状态和时间片机制影响，某些虚拟机实现还会为了可移植性弱化优先级差异。因此，依赖优先级来证明程序正确性，天然不具备跨平台稳定性。

优先级尤其不能替代同步。假设一个高优先级线程等待低优先级线程释放锁，那么低优先级线程如果得不到运行机会，高优先级线程反而会一直阻塞。这类问题说明：优先级改变不了锁、条件、I/O 和内存可见性的基本规则。一个线程优先级再高，只要它在 `BLOCKED`、`WAITING` 或 `TIMED_WAITING` 状态，就不是可运行线程；一个低优先级线程只要持有高优先级线程需要的锁，它就可能成为关键路径。正确的解决方式是缩短临界区、避免持锁阻塞、拆分锁粒度或调整任务设计，而不是盲目提高等待者优先级。

在 CPU 密集型任务中，优先级也不是简单的性能旋钮。提高某类线程优先级可能让它们获得更多运行机会，但也可能导致其他线程饥饿、响应延迟变大、后台维护任务无法及时执行，最终让整体系统表现更差。降低某些线程优先级也不一定让它们“只在空闲时运行”，因为操作系统仍可能在不同时间片中调度它们。优先级适合表达轻微倾向，不适合承担容量控制。需要限制资源时，应优先考虑线程池大小、队列容量、限流器、信号量、任务拆分和背压机制。

## 平台差异与可移植性
线程优先级的跨平台差异，是理解它为什么只能作为提示的关键。Java 提供统一 API，是为了让代码可以表达相对意图；但真正调度线程的是运行时和操作系统。不同操作系统的线程优先级范围不同、调度策略不同、权限模型不同，同一个 Java 优先级可能被映射成相同的底层优先级，也可能被压缩或调整。即使在同一个系统上，不同 JVM 实现、不同启动参数、不同进程负载，也可能让观察结果发生变化。

这意味着，用本机实验推导优先级行为时必须谨慎。一个循环打印的示例可能看起来“高优先级线程打印更多”，但这只是当前机器、当前负载、当前 JVM、当前输出设备共同作用的结果。加入 I/O、锁竞争、垃圾回收、安全点、线程创建开销、JIT 编译、终端刷新之后，输出顺序就会变化。并发文档和生产代码都不应把这种实验结果写成规范结论。

可移植的写法应遵守两个原则。第一，程序正确性不能依赖优先级。无论优先级是否生效，线程之间的数据一致性、任务完成、关闭流程和异常传播都应成立。第二，优先级调整必须可删除。也就是说，把 `setPriority` 去掉后，程序最多表现为吞吐、延迟或倾向变化，而不应出现死锁、数据丢失、顺序错误或关闭失败。如果删除一个优先级设置会让程序“不正确”，说明设计把调度提示当成了同步机制。

对于需要强实时保证的场景，普通 Java 线程优先级通常不是足够的抽象。实时性涉及操作系统调度策略、优先级反转控制、内存分配、停顿、I/O 抖动和资源隔离等一系列问题。标准 Java 并发中的优先级不能承诺截止时间，也不能保证某个线程在固定时间内得到 CPU。通用 Java 程序应把优先级理解为“有限、弱化、可被忽略的倾向表达”，而不是实时调度合同。

## 线程工厂配置
当线程由 `ExecutorService` 管理时，通常不应在任务代码里直接创建线程，也不应让每个调用点各自设置 daemon 和 priority。更好的做法是使用 `ThreadFactory` 集中配置线程属性。线程工厂的职责是把“线程如何命名、是否守护、优先级是多少、异常如何处理”收敛到一个地方，使执行器创建的每个工作线程都遵守同一套规则。

下面是一个只展示配置意图的简化线程工厂。真实代码可以按需要加入更严格的日志、指标和命名规则：

```java
import java.util.concurrent.ThreadFactory;
import java.util.concurrent.atomic.AtomicInteger;

final class NamedThreadFactory implements ThreadFactory {
    private final AtomicInteger sequence = new AtomicInteger(1);
    private final String prefix;
    private final boolean daemon;
    private final int priority;

    NamedThreadFactory(String prefix, boolean daemon, int priority) {
        if (priority < Thread.MIN_PRIORITY || priority > Thread.MAX_PRIORITY) {
            throw new IllegalArgumentException("invalid priority: " + priority);
        }
        this.prefix = prefix;
        this.daemon = daemon;
        this.priority = priority;
    }

    @Override
    public Thread newThread(Runnable task) {
        Thread thread = new Thread(task, prefix + "-" + sequence.getAndIncrement());
        thread.setDaemon(daemon);
        thread.setPriority(priority);
        thread.setUncaughtExceptionHandler((t, error) -> reportUncaught(t, error));
        return thread;
    }
}
```

线程工厂的好处不仅是复用代码。它让线程属性成为构造执行器时的显式决策，而不是散落在任务实现里的隐式副作用。任务只描述“要做什么”，线程工厂描述“由什么样的线程执行”。这种分离能降低误用概率：任务作者不需要关心工作线程是否守护，执行器拥有者则可以统一决定这组任务是否允许在 JVM 退出时被放弃。

不过，线程工厂也不能掩盖线程池关闭协议。把线程池里的工作线程设为守护线程，并不等于线程池不需要关闭。线程池仍然可能持有任务队列、工作线程、外部资源引用和未处理异常。对于长期运行的程序，应显式调用 `shutdown` 或 `shutdownNow`，再配合 `awaitTermination` 等待一段受控时间。daemon 属性只是最后一道生命周期属性，不是资源管理的替代品。

在线程池中使用守护线程要格外谨慎。适合的情况通常是执行可丢弃的辅助任务，且进程退出时不要求这些任务完成；不适合的情况包括：执行必须返回给调用方的计算、提交必须完成的写操作、维护全局一致性状态、作为唯一消费端处理不可重放队列。线程池隐藏了线程创建细节，也容易让人忘记“工作线程是否会阻止 JVM 退出”这个问题，因此线程工厂的命名和构造参数最好直接体现用途。

## 状态流转
守护线程与优先级的状态流转可以概括为：初始化阶段建立对象和依赖，运行阶段执行或等待，竞争阶段处理共享资源冲突，完成阶段发布结果或释放等待者，关闭阶段拒绝新工作并清理资源。这一描述虽然抽象，但适用于大多数 Java 并发工具。每个阶段都要避免“半初始化对象被发布”“等待条件丢失”“结果未安全发布”“关闭后仍提交任务”等问题。

初始化阶段最重要的是安全发布。线程启动前传入的不可变参数、构造完成后再发布的对象、通过 volatile 或锁保护发布的引用，都是可靠方式。反过来，如果 this 在构造过程中逃逸，或者共享集合在没有同步的情况下被多个线程看到，就可能让其他线程观察到未完成状态。守护线程与优先级如果涉及对象共享，初始化边界必须清晰。

运行阶段最重要的是不要阻塞错位置。持锁执行慢 I/O、在线程池工作线程中等待同一个线程池的任务、在 ForkJoin 任务中长时间等待外部资源、在条件等待中忘记循环检查，都会制造隐蔽问题。等待不是错误，但等待必须有条件、超时、取消和诊断手段。

完成阶段最重要的是结果传播。任务返回值、异常、计数归零、状态标志变更、锁释放或 volatile 写入，都是向其他线程传播进展的方式。传播动作必须与读取动作配对：Future.get 读取任务结果，await 等待计数归零，lock 获取看到上一次 unlock 前的写入，volatile 读观察 volatile 写发布的状态。只有写端和读端使用同一套规则，状态流转才可靠。

## 线程安全边界
守护线程不保证执行完 finally 或持久化动作

线程安全边界必须落到共享状态上。只要一个对象、集合、计数器、状态标志或外部资源会被多个线程同时访问，就要明确谁负责保护它、保护范围有多大、读写之间是否需要原子性、写入后是否需要被其他线程及时看见。没有共享状态的任务通常只需要关注取消和资源释放；有共享状态的任务则必须同时关注互斥、可见性、有序性和生命周期。

对守护线程与优先级来说，最小线程安全边界通常不是“某个类线程安全”这么粗，而是“哪些方法可以并发调用、哪些参数不能被调用方继续修改、哪些返回值不能暴露内部可变对象、哪些回调可能在持锁时发生”。同一个类中，有的方法可能线程安全，有的方法可能要求外部同步；同一个对象中，有的字段不可变，有的字段必须受锁保护。文档和代码注释应尽量表达这种精确边界。

还要区分结构安全和业务安全。并发容器可以保证自身结构不被破坏，但不能保证跨多个操作的业务不变式；AtomicInteger 可以保证单次递增原子，但不能保证“检查余额再扣减再记录日志”整体一致；线程池可以保证任务被工作线程执行，但不能保证任务内部访问的对象安全。守护线程与优先级如果被误认为“用了就线程安全”，往往会掩盖更上层的不变式问题。

## 内存可见性与有序性
Java 并发中，正确性不仅是“没有两个线程同时写”。一个线程写入后另一个线程能否看到，另一个线程看到多个写入时是否能按预期顺序理解，同样关键。守护线程与优先级只要牵涉共享变量，就要考虑 happens-before。synchronized 的释放和随后获取、volatile 写和随后读、线程 start 前后的关系、线程终止和 join 的关系、并发工具内部的同步动作，都是常见可见性来源。

有序性问题常被低估。编译器和处理器可以在不改变单线程结果的前提下重排序指令；缓存也可能让一个核心暂时看不到另一个核心的写入。JMM 允许这些优化，同时要求程序通过同步动作表达跨线程约束。因此，不能因为一段代码“按顺序写着”就认为另一个线程也会按顺序看到。对守护线程与优先级的使用如果缺少同步边界，就可能在压力下暴露出旧值、默认值或部分构造对象。

可见性也不等于原子性。volatile 能让状态标志及时可见，但不能让自增、复合检查或多字段更新自动变成一个不可分割操作；锁能同时提供互斥和可见性，但锁的对象必须一致；CAS 能保证单个位置的比较更新原子，但不能自动维护多个变量之间的不变式。选择工具时，应先判断需要的是可见性、原子性、有序性，还是三者都需要。

## 实践要点
第一，优先使用表达力更明确的高层工具。能用 BlockingQueue 表达生产消费，就不要手写 wait/notify；能用 ExecutorService 管理任务，就不要散落创建 Thread；能用 CountDownLatch 表达一次性等待，就不要用循环 sleep；能用 AtomicInteger 表达单变量计数，就不要扩大成不必要的重量级锁。高层工具的价值在于把容易写错的等待、唤醒和内存语义封装起来。

第二，给等待设置边界。无界等待是并发故障扩大的常见原因。获取结果可以使用超时 get，条件等待可以使用带超时的 await，外部调用可以设置超时，线程池关闭可以设置 awaitTermination 时间。超时不是为了掩盖错误，而是为了让调用方有机会记录现场、释放资源或降级。

第三，保持中断语义。凡是调用会抛出 InterruptedException 的方法，都要认真决定当前层是否消费中断。如果当前层无法完成取消处理，通常应恢复中断标志并返回，让上层继续处理。静默吞掉中断会让关闭流程卡住，也会让调用者误以为任务已经遵守取消协议。

第四，减少共享可变状态。不可变对象、局部变量、消息传递、任务参数复制、并发容器和清晰的所有权，通常比到处加锁更容易维护。锁和原子类解决的是共享状态冲突，但最好的并发设计往往是先减少共享，再保护必须共享的部分。

第五，保持诊断信息。线程名应能反映用途，线程池应暴露队列长度和拒绝次数，长时间等待应能定位锁或条件，后台任务异常应记录。并发问题通常无法只靠阅读一行报错定位，足够的运行时信息能显著降低排查成本。

## 适用场景与选择边界
守护线程适合依附性强、结果可丢弃、退出时不需要严格收尾的辅助工作。典型例子包括定期清理临时缓存、维护本地近似统计、刷新非关键内存视图、执行调试或监控辅助循环、在主工作线程存在时提供后台支持。这些任务有一个共同点：它们帮助程序运行得更整洁或更高效，但不是程序对外承诺的一部分。即使它们在进程退出时没有完成最后一次循环，系统也不会因此丢失必须保存的数据或破坏不可恢复的不变式。

不适合守护线程的任务也有清晰特征：任务完成本身就是对外承诺，或者任务中断会留下难以恢复的中间状态。比如把内存中的关键变更写入外部存储、发送必须到达的通知、消费不可重放的队列消息、执行需要成对提交和回滚的操作、释放必须精确归还的许可或租约。这些任务需要明确的成功、失败、重试、取消和补偿路径。将它们放在守护线程中，并不会自动让风险消失，只是让进程退出时更容易看不见这些风险。

选择 daemon 属性时，可以把问题拆成三层。第一层问“这个线程是否代表程序仍有未完成的关键工作”。如果答案是是，它通常应是用户线程，或者由用户线程负责等待其任务完成。第二层问“这个线程是否有明确关闭协议”。如果没有，即使它是用户线程，也可能导致程序无法退出；如果有，即使它是守护线程，也应在正常关闭流程中尽力执行协议。第三层问“提前停止会造成什么后果”。如果后果只是少一次清理或少一个近似指标，daemon 可以接受；如果后果是数据不一致，就不应依赖 daemon。

线程优先级的适用边界更窄。它可以用来表达“这个线程相对更希望及时得到运行机会”，例如让交互相关线程相对高于批处理线程，让低价值后台计算相对低于核心处理线程。但这种表达只能作为调度倾向，不能作为程序语义。更常见、更可靠的做法是把不同任务放入不同执行器，用线程池大小、队列容量、任务拆分粒度、限流和超时来控制资源。优先级可以作为辅助手段，但不应成为第一设计工具。

当一个问题看起来可以通过优先级解决时，通常应先问几个更具体的问题。是不是某个临界区太大，导致高价值任务长时间等待锁？是不是某个线程池混放了短任务和长任务，导致短任务排队？是不是阻塞 I/O 没有超时，导致工作线程被占满？是不是队列无界，把过载隐藏成延迟？这些问题的根因通常是结构设计，而不是线程优先级不够高。调整优先级只能改变症状出现的概率，不能消除容量和同步边界的问题。

## 守护线程、关闭钩子与显式关闭
讨论守护线程时，经常会同时提到关闭钩子和显式关闭，但三者解决的问题不同。守护线程决定 JVM 自然退出时是否等待该线程；关闭钩子是在 JVM 关闭阶段尝试执行的一段回调；显式关闭是程序在仍有正常控制流时主动释放资源、停止任务、等待完成。最可靠的是显式关闭，因为它发生在程序仍能清楚表达顺序、超时和错误处理的时候。关闭钩子可以作为补充，守护线程则只能表达生命周期依附。

关闭钩子本身也不应被滥用。它适合做短小、有限、尽力而为的清理，不适合承载复杂业务流程。关闭阶段的环境已经不稳定：其他线程可能正在终止，资源可能已经部分关闭，锁可能被某个不再前进的线程持有，新的任务提交也可能不再安全。如果把大量关键逻辑放入关闭钩子，再配合守护线程完成实际工作，就会得到一个很难测试、很难保证时序的关闭流程。

显式关闭的写法应尽量普通而清晰。拥有线程或线程池的对象提供 `close` 或 `shutdown` 方法；方法内部先改变状态，阻止新任务进入；再通过中断、条件通知、队列哨兵或执行器关闭唤醒工作线程；然后等待一段有上限的时间；最后记录未完成事项并释放外部资源。这个过程是否使用守护线程并不重要，重要的是调用方能够知道关闭何时开始、何时结束、失败如何暴露。

在实现 `AutoCloseable` 的组件中，daemon 属性更应该被视为兜底属性，而不是主要关闭机制。组件被正常使用时，调用方应调用 `close`；测试中也应验证 `close` 能让线程结束或线程池停止。只有在调用方忘记关闭、进程即将退出、且任务本身允许丢弃时，daemon 属性才避免辅助线程无限期拖住进程。这个定位能避免两个极端：一种是所有后台线程都设为用户线程导致程序无法退出，另一种是所有后台线程都设为守护线程导致关键工作丢失。

## 常见误区
不要把关键资源提交放在守护线程中。这是守护线程与优先级中最需要避免的误区之一。它之所以危险，是因为代码在低并发、本机运行或单次测试时可能完全正常，但在调度时机变化、任务数量上升或关闭流程触发时暴露问题。并发程序的正确性不能建立在“通常来得及”“一般不会同时发生”“这个线程应该先执行”这类假设上。

另一个误区是把时间当同步。sleep 一段时间、等待“差不多执行完”、依赖日志先后顺序、用本机测试速度推断生产时序，都属于时间同步误区。正确做法是使用明确同步条件，例如 join、Future、Latch、Barrier、锁、条件队列或 volatile 状态。

还要避免把工具名当保证。使用线程池不代表任务不会丢，使用锁不代表锁对象选对了，使用 volatile 不代表复合操作原子，使用 ThreadLocal 不代表不会泄漏，使用公平锁不代表请求一定按业务顺序完成。每个工具都只保证它规范中定义的部分，超出边界的需求仍要由设计补上。

最后是忽略失败路径。任务抛异常后计数是否递减，锁是否释放，许可是否归还，线程池是否还能接收任务，Future 是否有人读取异常，中断是否被恢复，资源是否关闭，这些都必须在代码中体现。并发系统的许多严重问题不是正常路径错误，而是异常路径没有释放等待者或资源。

还有一个常见误区是把 daemon 和 priority 绑定理解。守护线程可以有高优先级，用户线程也可以有低优先级；二者分别描述生命周期和调度倾向，没有必然关系。把守护线程理解成“低级线程”，容易让人低估它对共享状态、锁和资源的影响。一个高优先级守护线程如果持有锁或占用 CPU，仍可能影响用户线程；一个低优先级用户线程如果迟迟不结束，仍会让 JVM 保持存活。

也不要用优先级修补错误的等待关系。如果两个线程需要按顺序协作，应使用队列、锁、条件、同步器或 Future 表达依赖；如果某个线程必须等待另一个线程完成，应使用 `join` 或任务结果机制；如果某段状态必须可见，应使用同步动作发布。把生产者设成高优先级、消费者设成低优先级，或者反过来，都不能保证队列不丢数据、条件不遗漏、状态不乱序。优先级最多影响运行机会，不会创建 happens-before 关系。

最后，不能因为线程是守护线程就忽略异常。守护线程异常退出后，用户线程可能继续运行，只是某个辅助能力悄悄消失；也可能因为它是最后剩下的线程之一，进程很快结束，现场信息极少。长期运行的守护任务应有异常处理策略：哪些异常可以记录后继续，哪些异常应让线程退出，哪些异常应通知拥有者，哪些异常说明程序状态已经不可恢复。没有异常策略的后台线程，会让系统在最需要诊断时缺少证据。

## 排查思路
排查守护线程与优先级相关问题时，可以先问四个问题。第一，线程现在在哪里：运行、阻塞在锁、等待条件、等待 I/O、等待任务结果，还是已经终止。第二，共享状态由谁保护：锁对象是否一致，volatile 是否覆盖读写两端，原子操作是否足够表达不变式。第三，生命周期是否闭合：启动后是否可停止，提交后是否可取消，关闭后是否拒绝新任务，异常后是否释放等待方。第四，容量是否合理：线程数、队列长度、许可数量、任务耗时和上游流量是否匹配。

工具上可以结合线程 dump、日志时间线、线程池指标、锁竞争指标和重复压力测试。线程 dump 能看到 BLOCKED、WAITING、TIMED_WAITING 以及调用栈；线程池指标能看到任务是否堆积；日志可以确认取消和关闭是否发生；压力测试可以放大竞态窗口。排查时不要只看最终异常，要寻找第一个失去同步关系或容量边界的位置。

如果问题偶发，优先怀疑数据竞争、遗漏通知、异常路径未释放、线程池饱和、死锁或外部阻塞。守护线程与优先级相关代码应尽量让这些情况可被观察，例如在拒绝策略中记录任务类型，在等待超时后输出关键状态，在关闭时记录未执行任务数量，在捕获异常时保留中断状态或根因。

## 深入理解与推导
理解守护线程与优先级时，可以从一个更具体的推导过程开始：先假设程序只有一个线程，所有语句按照调用栈顺序执行，变量只被当前线程读写；再引入第二个线程，让同一份状态可能被两个执行流同时观察；最后再加入等待、取消、超时、异常和关闭。每增加一个因素，就需要增加一条明确规则。没有规则的地方，运行结果就只能依赖调度时机，而调度时机从来不是稳定契约。

第一条规则是所有权规则。一个并发对象或同步工具必须有清楚的创建者、使用者和释放者。创建者负责初始化完整状态，使用者负责遵守访问协议，释放者负责关闭或清理资源。守护线程与优先级相关代码如果没有所有权规则，就会出现同一个对象被多个模块以不同方式访问：有的路径加锁，有的路径不加锁；有的路径处理取消，有的路径吞掉异常；有的路径认为可以关闭线程池，有的路径仍在提交任务。这类问题通常不是 API 本身难用，而是边界没有被写进设计。

第二条规则是条件规则。等待必须等待一个可验证的条件，而不是等待一段时间。条件可以是线程结束、任务完成、计数归零、队列非空、许可可用、状态标志改变或锁变为可获取。条件还必须由同一套同步机制保护：用锁保护的条件要在持锁时检查，用 volatile 发布的状态要由 volatile 读观察，用 Future 表示的结果要通过 get 获取，用同步器表示的阶段要通过对应 await 方法等待。条件和等待工具不匹配时，最容易出现遗漏通知、虚假成功或永久阻塞。

第三条规则是失败规则。守护线程与优先级的正常路径通常很短，但失败路径决定系统是否可靠。提交失败时是否记录，等待超时时是否取消，任务异常时是否释放锁和许可，线程被中断时是否恢复中断标志，关闭时是否处理尚未执行的任务，这些问题都必须有答案。并发代码的失败路径尤其重要，因为一个线程的异常经常不会直接出现在另一个线程的调用栈中；如果没有显式传播和记录，调用方只能看到“没有结果”或“系统变慢”。

第四条规则是容量规则。任何等待队列、线程池、同步器许可、任务拆分粒度和重试循环都有容量含义。容量太小会导致吞吐不足，容量太大会隐藏过载并放大延迟，容量无界会把压力转化为内存占用或不可控等待。守护线程与优先级如果用于高并发路径，就不能只讨论功能正确，还要讨论任务数量、平均耗时、峰值流量、等待上限和降级策略。容量规则不是性能优化阶段才考虑的问题，而是并发设计的一部分。

第五条规则是可见性规则。一个线程做完动作，不代表其他线程自然知道。必须通过锁释放与获取、volatile 写与读、线程启动与结束、并发容器操作、同步器状态变化等方式建立可见性。守护线程与优先级的很多误用都可以归结为缺少可见性规则：写线程以为自己已经发布状态，读线程却可能看到旧值；任务以为已经初始化对象，其他线程却可能看到部分构造；关闭线程以为已经发出停止请求，工作线程却从不检查中断或状态标志。

把这五条规则连起来，就能形成一个检查清单：对象是否安全发布，访问是否遵守同一协议，等待是否有明确条件，失败是否释放资源，关闭是否能唤醒等待者，容量是否有上限，结果是否通过同步关系传播。守护线程与优先级的学习价值正在于把这些问题具体化。只要清单中的某一项回答含糊，就应回到代码重新收紧边界，而不是依赖注释或经验说明“正常情况下不会出问题”。

## 阅读与复习建议
复习守护线程与优先级时，建议不要只看结论，而要自己沿着一条并发路径推演：哪个线程先创建对象，哪个线程发布引用，哪个线程开始执行，哪个线程可能等待，哪个线程负责唤醒，异常会落在哪里，关闭时如何让等待者退出。能把这条路径讲清楚，通常比记住零散 API 更有用。

还可以把本文中的代码片段改写成几个小实验：增加线程数量，故意制造超时，主动触发中断，抛出运行时异常，重复执行上千次观察结果。实验不是为了依赖偶然输出，而是为了暴露自己对边界条件的理解是否完整。只要实验结果和预期不一致，就应回到线程安全边界、内存可见性、等待条件和生命周期四个角度重新检查。

## 小结
守护属性管理生命周期，优先级表达倾向，可靠程序仍要靠明确关闭协议。

掌握守护线程与优先级，最终要形成两种能力：一是能写出边界清晰的代码，知道正常路径、失败路径和关闭路径分别如何运行；二是能解释运行时现象，知道线程为什么等待、为什么没有看到新值、为什么任务没有执行、为什么关闭没有结束。Java 并发不是把代码拆到多个线程中那么简单，而是给执行、状态和生命周期建立可验证的规则。

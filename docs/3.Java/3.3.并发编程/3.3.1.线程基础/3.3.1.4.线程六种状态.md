# 3.3.1.4 线程六种状态

## 定位
线程六种状态指的是 Java 标准库中 `Thread.State` 枚举定义的六个生命周期观察状态：`NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 和 `TERMINATED`。它们不是线程调度器内部的完整状态机，也不是操作系统线程状态的一一映射，而是 Java 运行时提供给开发者的一组诊断视角：一个线程对象当前是否尚未启动、是否有资格执行、是否在争用监视器锁、是否在无限期等待、是否在限时等待，或者是否已经结束。

学习线程状态的价值不在于背出六个英文单词，而在于能解释真实并发程序中的现象。为什么某个线程一直不继续执行？为什么线程 dump 中大量线程显示 `RUNNABLE`，但 CPU 并没有被打满？为什么 `sleep` 后线程不是 `WAITING` 而是 `TIMED_WAITING`？为什么两个线程都在等待，却一个是 `BLOCKED`，另一个是 `WAITING`？为什么线程已经运行完毕，却不能再次 `start`？这些问题都要回到 Java 线程状态、同步机制、等待机制和调度机制之间的关系中理解。

本文保持通用 Java 技术视角，只讨论 Java 语言、Java 虚拟机和标准并发库中的概念与实践。线程状态本身常出现在故障诊断、锁竞争分析、线程池问题排查、死锁定位和等待链分析中，但它并不是业务逻辑的同步条件。换句话说，`getState()` 可以帮助观察和解释线程，却不应该成为控制业务流程的核心依据。可靠的并发程序应使用锁、条件队列、`volatile`、原子类、线程安全容器、`Future`、同步器等明确机制表达同步关系，而不是通过反复读取线程状态来猜测另一个线程“应该已经走到哪里”。

## `Thread.State` 的基本含义
`Thread.State` 是 Java 对线程生命周期的抽象枚举。一个线程在任意时刻最多处于其中一种状态，但这个状态是瞬时快照：你读取它时得到的是当时的观察结果，下一瞬间它可能已经变化。由于线程调度、锁释放、通知、超时、中断和任务结束都可能发生在极短时间内，状态快照只能作为诊断线索，不能单独证明完整因果关系。

六种状态可以先用一张表建立整体印象：

| 状态 | 核心含义 | 常见进入原因 | 常见离开原因 |
| --- | --- | --- | --- |
| `NEW` | 线程对象已创建，但还没有调用 `start()` | 执行 `new Thread(...)` 后尚未启动 | 调用 `start()` 后进入可运行流程 |
| `RUNNABLE` | Java 线程有资格运行，可能正在执行，也可能等待操作系统分配 CPU 或等待某些底层资源 | `start()` 后、从阻塞或等待中恢复后、获得锁后 | 争用 `synchronized` 失败、调用等待方法、执行结束 |
| `BLOCKED` | 正在等待进入一个 `synchronized` 监视器 | 进入同步方法或同步块时锁被其他线程持有 | 目标监视器被释放并成功竞争到锁 |
| `WAITING` | 正在无限期等待另一个线程的动作 | `Object.wait()`、无超时 `Thread.join()`、无超时 `LockSupport.park()` | 被通知、目标线程结束、被唤醒、被中断 |
| `TIMED_WAITING` | 正在带时间上限地等待 | `Thread.sleep()`、带超时 `Object.wait(long)`、带超时 `join`、带超时 `park` | 时间到、被通知、目标线程结束、被唤醒、被中断 |
| `TERMINATED` | 线程执行已经结束 | `run()` 正常返回或抛出未捕获异常 | 终态，不能回到其他状态 |

这张表只描述最常见的触发方式，真实运行时还要结合调用栈、锁信息和并发工具内部实现分析。比如很多并发工具最终可能使用 `LockSupport.park` 让线程进入等待；线程等待文件、网络或其他本地资源时，在 Java 层不一定呈现为 `WAITING` 或 `BLOCKED`；一个线程处于 `RUNNABLE` 也不代表它正在消耗 CPU，因为它可能只是处在 Java 定义的可运行范围内。

## `NEW`：已经创建但尚未启动
`NEW` 是线程对象的初始状态。执行 `new Thread(...)` 只是创建了一个普通 Java 对象，并没有创建一条正在运行的执行流，也没有让 `run()` 方法开始执行。只有调用 `start()`，运行时才会为这个线程安排实际执行，并在新的线程上下文中调用它的 `run()` 方法。直接调用 `run()` 只是在当前线程里执行一个普通方法，不会让目标线程从 `NEW` 进入后续生命周期。

理解 `NEW` 要特别区分“线程对象”和“线程执行流”。线程对象可以被引用、命名、设置优先级、设置是否为守护线程，也可以传入构造参数；但在 `start()` 之前，它还没有参与调度，也不会与其他线程并发执行自身任务。这个阶段适合完成启动前配置，例如设置线程名、设置守护属性、安装未捕获异常处理器、准备传入任务所需的不可变参数。启动之后再修改某些属性，要么没有意义，要么受到 API 限制，要么会让生命周期边界变得不清晰。

`NEW` 状态还提示一个重要约束：一个 `Thread` 实例只能启动一次。线程从 `NEW` 经过 `start()` 进入后续状态，最终到达 `TERMINATED` 后，不能再回到 `NEW`，也不能再次 `start()`。如果需要再次执行同样的任务，应创建新的线程对象，或者更常见地把任务提交给执行器，由线程池复用工作线程。复用的是线程池中的工作线程，不是同一个已经终止的 `Thread` 对象重新启动。

在设计层面，`NEW` 阶段的关键问题是安全发布。传给线程的任务对象、参数对象和共享依赖应在启动前构造完整，并通过清晰的方式传入。`Thread.start()` 本身建立了重要的内存语义：在调用 `start()` 之前发生的动作，对新线程中的动作可见。这个保证让“先准备参数，再启动线程”的模式成立。但如果对象在构造过程中让 `this` 逃逸，或者启动后又在没有同步的情况下继续修改共享配置，新线程仍可能观察到不一致状态。

## `RUNNABLE`：可运行不等于正在运行
`RUNNABLE` 是最容易被误解的状态。按照 Java 的抽象，一个处于 `RUNNABLE` 的线程正在 Java 虚拟机中执行，或者已经具备执行资格，等待底层系统安排。它不同于许多操作系统教材中“ready”和“running”的细分：Java 没有把“正在 CPU 上执行”和“已经就绪但正在排队等待 CPU”拆成两个 `Thread.State` 状态。于是，一个线程显示为 `RUNNABLE`，只能说明它没有处于 Java 层面的 `NEW`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 或 `TERMINATED`，不能直接说明它此刻一定占用处理器。

这一区别对诊断非常重要。线程 dump 中看到大量 `RUNNABLE` 时，第一反应不应是“这些线程都在疯狂运行”。它们可能确实在执行 CPU 密集型循环，也可能在进行本地方法调用、等待底层 I/O、等待操作系统资源，或者只是就绪但尚未获得时间片。要判断是否真的 CPU 忙，应结合 CPU 使用率、热点栈、采样分析和线程栈位置，而不是只看状态名称。

线程进入 `RUNNABLE` 的路径很多。`start()` 之后，新线程通常会进入可运行范围；从 `BLOCKED` 成功获得监视器锁后，会回到可运行范围；从 `WAITING` 或 `TIMED_WAITING` 被唤醒并重新满足继续执行条件后，也会回到可运行范围；从 `sleep` 时间到期后，也会重新竞争调度机会。注意“回到 `RUNNABLE`”不代表马上执行下一行代码，它仍要等待调度器安排，也可能马上再次进入锁竞争或等待。

`RUNNABLE` 还提醒我们不要用线程优先级推导执行顺序。Java 提供线程优先级设置，但优先级只是给调度器的提示，不是跨平台的严格顺序保证。不同操作系统、不同虚拟机实现、不同负载环境下，优先级的效果可能差异很大。并发程序如果需要先后关系，应使用明确同步机制；如果需要阶段推进，应使用 `join`、`Future`、`CountDownLatch`、`Phaser`、锁和条件等工具，而不是寄希望于某个线程“更可能先运行”。

从代码设计角度看，`RUNNABLE` 状态下最危险的问题通常不是状态本身，而是没有退出条件的忙等、没有同步的共享变量读取、长时间持锁执行复杂逻辑、以及把阻塞式操作混入不适合阻塞的执行环境。一个线程长期 `RUNNABLE` 且 CPU 占用很高，可能是死循环、自旋过度、重试没有退避、队列消费逻辑无法退出；一个线程长期 `RUNNABLE` 但 CPU 不高，则要结合栈看是否在本地 I/O、系统调用或底层库等待。

## `BLOCKED`：等待进入 `synchronized` 监视器
`BLOCKED` 表示线程正在等待一个监视器锁，也就是等待进入某个 `synchronized` 方法或 `synchronized` 代码块。只有与内置监视器相关的锁竞争才会在 `Thread.State` 中体现为 `BLOCKED`。如果使用 `ReentrantLock`、`Semaphore`、`CountDownLatch`、`BlockingQueue` 等并发工具，线程等待时通常不会显示为 `BLOCKED`，而更可能显示为 `WAITING` 或 `TIMED_WAITING`，因为这些工具常基于队列和 `LockSupport.park` 挂起线程。

这一区别是排查锁问题的基础。看到 `BLOCKED`，应重点查找线程栈中正在进入的同步方法或同步块，并查看 dump 中是否有 “waiting to lock” 之类的锁标识，以及哪个线程当前持有该监视器。持锁线程的栈比阻塞线程的栈更关键，因为真正决定等待时间的是持锁线程什么时候释放锁。只盯着被阻塞线程，往往只能看到它卡在入口处，却看不到谁让它进不去。

`BLOCKED` 与 `WAITING` 的差别不仅是名字不同。`BLOCKED` 是进入临界区之前的锁竞争，线程还没有拿到目标监视器；`Object.wait()` 造成的 `WAITING` 则要求线程已经持有该对象监视器，然后主动释放监视器并进入等待集合。前者是在门外排队，后者是进门后发现条件不满足，把锁交出去并等待通知。理解这个区别，才能解释为什么调用 `wait()` 必须先持有对应对象的监视器，也才能理解为什么被 `notify` 唤醒后还要重新竞争锁。

导致 `BLOCKED` 长时间存在的常见原因包括：临界区过大，持锁期间执行慢操作，锁对象选择过粗，多个无关资源共用一把锁，回调在持锁期间执行，锁顺序不一致引发死锁，或者某个持锁线程因为异常路径没有及时退出临界区。`synchronized` 在方法返回或异常抛出时会自动释放监视器，但这并不意味着所有持锁逻辑都安全。自动释放只能保证不会因为忘记写 `unlock` 而永久持有内置锁，不能保证临界区设计合理，也不能保证持锁期间调用的外部代码不会长期阻塞。

优化 `BLOCKED` 问题时，应先确认锁保护的共享不变式是什么，再决定是否缩小临界区、拆分锁、改用并发容器、改用读写锁或改用无锁结构。不要为了让 dump 中少出现 `BLOCKED` 就盲目换工具。`synchronized` 的语义简单、可见性可靠、异常释放清晰，在很多场景下是合适选择。真正要避免的是用一把锁保护过多无关状态，或者在持锁期间做不受控的耗时工作。

## `WAITING`：无限期等待另一个动作
`WAITING` 表示线程正在无限期等待某个其他动作发生。典型来源包括无超时的 `Object.wait()`、无超时的 `Thread.join()` 和无超时的 `LockSupport.park()`。所谓无限期，并不是说永远无法恢复，而是说等待本身没有内置时间上限；它需要另一个线程执行通知、目标线程结束、许可出现、线程被中断，或者发生其他唤醒条件。

`Object.wait()` 是理解 `WAITING` 的经典入口。线程调用某个对象的 `wait()` 前，必须持有该对象的监视器；调用后，它会释放这个监视器并进入该对象的等待集合。之后，其他线程可以在持有同一对象监视器时调用 `notify()` 或 `notifyAll()` 发出通知。被通知的线程不会立刻继续执行，它只是从等待集合转移出来，接下来还要重新竞争该对象监视器。只有重新获得锁后，`wait()` 才能返回。

这套机制带来一个必须遵守的写法原则：条件等待要放在循环中，而不是用一次 `if` 判断。原因有三个。第一，线程被唤醒时，条件可能已经被其他线程改变。第二，可能存在虚假唤醒，即线程没有对应业务条件也从等待中返回。第三，多个等待者共享同一条件时，通知只表示“条件可能变化”，不表示“当前线程一定可以继续”。因此，等待者应在持有同一把锁时反复检查条件，不满足就继续等待，满足才离开等待循环。

`Thread.join()` 造成的 `WAITING` 表示当前线程等待另一个线程终止。它适合表达明确的生命周期依赖：调用方需要等待被调用线程运行完，才能继续读取结果、释放资源或进入下一阶段。`join` 还带有重要内存语义：一个线程中的所有动作 happens-before 另一个线程从该线程的 `join` 成功返回。也就是说，等待线程结束不仅是时间上的等待，也是结果可见性的同步边界。相比用 `sleep` 猜测另一个线程是否执行完，`join` 是明确得多的工具。

`LockSupport.park()` 是许多高级同步器的底层构件。它使用“许可”模型：一个线程可以被 `park` 挂起，也可以被另一个线程 `unpark`。如果许可已经存在，后续 `park` 可能立即返回；如果没有许可，`park` 会等待。与 `wait/notify` 相比，`park/unpark` 不要求调用方持有某个对象监视器，也不绑定对象等待集合；但它更底层，直接使用时更容易写出难以维护的协议。通常应优先使用基于它构建的高层并发工具。

`WAITING` 状态的最大风险是没有退出路径。无限期等待必须能回答三个问题：谁会让条件成立，谁会发出通知或唤醒，如果对方失败了怎么办。没有这些答案，等待就可能在异常路径、任务丢失、通知遗漏、线程提前终止或关闭流程中永久挂起。可靠设计通常会给等待增加取消机制、关闭信号或外层超时，即使内部使用无超时等待，也要确保系统层面存在可控退出方式。

## `TIMED_WAITING`：带时间上限的等待
`TIMED_WAITING` 表示线程正在限时等待。常见来源包括 `Thread.sleep(long)`、带超时的 `Object.wait(long)`、带超时的 `Thread.join(long)`、带超时的 `LockSupport.parkNanos` 或 `parkUntil`。它与 `WAITING` 的核心差别是：即使没有其他线程通知、唤醒或结束，等待线程也会在时间条件满足后恢复到可竞争执行的状态。

`Thread.sleep()` 是最常见也最容易误用的入口。`sleep` 只让当前线程在指定时间内不参与执行，它不会释放已经持有的监视器锁。也就是说，如果线程在 `synchronized` 块中调用 `sleep`，其他需要同一把锁的线程仍然可能处于 `BLOCKED`。因此，`sleep` 不能用来“礼貌地让出锁”，也不能用来表达条件等待。真正需要等待条件变化时，应使用条件队列、同步器或其他明确协作机制。

带超时的 `wait` 与 `sleep` 有本质区别。`wait(timeout)` 必须在持有目标对象监视器时调用，调用后会释放该监视器，并在被通知、中断、超时或虚假唤醒后返回并重新获得监视器。它适合表达“条件不满足时等待一段时间再检查”。由于返回原因可能是通知、超时或虚假唤醒，调用方仍然必须在循环中检查条件，并根据剩余时间决定是否继续等待。

带超时的 `join` 适合在生命周期等待上设置上限。比如调用方希望等待工作线程结束，但不能无限期卡住，就可以使用限时等待，并在超时后记录现场、发出取消请求、尝试中断或进入降级路径。需要注意，`join(timeout)` 返回不代表目标线程一定已经结束；返回后必须再次检查目标线程是否仍然存活。把“等待方法返回”误解成“条件已经成立”，是限时等待中的常见错误。

`TIMED_WAITING` 对诊断也有双重含义。一方面，它可能是健康的节流、定时任务、重试退避或限时等待；另一方面，大量线程长期处于 `TIMED_WAITING` 也可能表示程序在用 `sleep` 掩盖同步缺失，或者大量请求都在等待慢资源。判断时要结合栈位置。如果栈显示在线程池空闲等待任务，通常是正常现象；如果栈显示业务代码反复 `sleep` 等待某个标志变化，就应考虑用更明确的通知机制替代。

超时不是同步正确性的替代品。给等待加超时可以防止故障无限扩大，也能提供诊断机会，但它不能自动修复条件竞争、通知遗漏或可见性问题。正确的限时等待应同时具备条件检查、时间预算、中断处理和失败路径。尤其是多次循环等待时，应使用剩余时间而不是每轮都重新等待完整超时时间，否则总等待时长可能远超预期。

## `TERMINATED`：执行流已经结束
`TERMINATED` 是线程生命周期的终态。线程的 `run()` 方法正常返回，或者抛出未捕获异常导致执行结束，线程都会进入该状态。进入 `TERMINATED` 后，这个 `Thread` 对象不能再次启动。线程对象本身仍可能被其他对象引用，也可能被读取名称、状态等信息，但它所代表的执行流已经结束。

终止并不等于结果已经被正确处理。裸线程的 `run()` 如果抛出未捕获异常，异常不会自动传播到创建线程的调用栈；它通常由线程的未捕获异常处理器处理，或者输出到标准错误。调用方如果需要获取结果和异常，通常应使用 `Callable`、`Future`、`CompletableFuture` 或执行器框架，而不是只依赖裸 `Thread`。线程状态只能告诉你它结束了，不能告诉你业务是否成功。

`TERMINATED` 与内存可见性也有关。一个线程中的动作 happens-before 另一个线程检测到该线程已经终止的某些同步动作，例如成功从 `join` 返回。也就是说，如果主线程需要读取工作线程写入的结果，合理方式是通过 `join`、`Future.get`、同步器或其他同步关系等待完成，而不是简单循环观察某个普通字段。只看到线程已经结束，不代表所有共享结果都以正确方式发布给读取方；必须使用规范定义的同步边界。

终止状态还需要和线程池复用区分。在线程池中，一个工作任务执行完，并不意味着承载它的工作线程进入 `TERMINATED`；工作线程可能回到池中继续等待下一个任务。只有线程池关闭、工作线程退出运行循环，那个实际 `Thread` 才会终止。排查线程池问题时，如果任务完成但线程仍处于等待任务的状态，这是线程池复用的正常表现，不应误判为线程泄漏。真正要关注的是线程池是否能按预期关闭、任务异常是否被记录、队列是否积压、工作线程是否长期卡在不可取消的操作上。

## Java 状态与操作系统调度状态的差异
Java 的六种线程状态是面向 Java 程序员的抽象，不是操作系统调度器状态的逐项转译。操作系统通常会区分就绪、运行、阻塞、睡眠、等待 I/O、僵尸等不同状态，不同系统命名和细节也不同。Java 为了提供跨平台模型，把这些底层差异压缩成 `Thread.State` 的六个枚举。因此，不能把 Java 状态表直接当成操作系统状态图。

最典型的差异是 `RUNNABLE`。在操作系统视角中，一个线程可能正在 CPU 上运行，也可能在就绪队列中等待时间片，还可能在某些系统调用或本地 I/O 上等待。Java 层面可能仍把它呈现为 `RUNNABLE`，因为它没有进入 Java 监视器阻塞或 Java 等待队列。于是，`RUNNABLE` 更像“没有被 Java 同步机制挂起”的状态集合，而不是“正在执行机器指令”的精确描述。

`BLOCKED` 也有特定含义。它不是所有阻塞的总称，而是等待进入 `synchronized` 监视器。线程等待数据库、文件、网络、队列、`Lock`、`Semaphore` 或 `Future` 时，不一定显示为 `BLOCKED`。如果把 `BLOCKED` 理解为“任何阻塞”，就会在排查时误判问题类型：看到没有 `BLOCKED` 就以为没有等待，或者看到 `WAITING` 就忽略锁竞争。正确做法是把状态和栈一起看，栈中的方法调用往往比状态名更能说明具体等待点。

`WAITING` 和 `TIMED_WAITING` 也不是操作系统睡眠状态的简单同义词。它们描述的是 Java 层等待语义：等待通知、等待线程结束、等待许可、等待时间到期等。底层实现可能依赖操作系统原语，也可能由虚拟机和运行时协作完成。对应用开发者而言，重要的不是底层用了哪个系统调用，而是 Java 规范和类库为这些等待提供了哪些唤醒、中断、可见性和异常语义。

因此，分析线程问题应建立三层视角。第一层是 Java 状态：它告诉你线程在 Java 抽象中属于哪类生命周期位置。第二层是 Java 栈：它告诉你线程停在什么方法、什么同步器、什么业务调用。第三层是系统指标：CPU、上下文切换、I/O、内存、锁竞争采样等指标告诉你底层资源是否成为瓶颈。三层结合，才能避免只凭一个状态名下结论。

## 状态转换的主线
线程状态转换可以从一条主线理解：对象创建后是 `NEW`；调用 `start()` 后进入可运行范围，也就是 `RUNNABLE`；执行过程中如果争用 `synchronized` 失败，进入 `BLOCKED`；如果主动等待其他动作，进入 `WAITING`；如果主动限时等待，进入 `TIMED_WAITING`；等待结束、获得锁或被唤醒后通常回到 `RUNNABLE`；`run()` 完成后进入 `TERMINATED`。

这条主线并不是一个严格的单向链表，因为线程可能在 `RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 之间多次往返。例如线程运行一段代码，进入同步块失败而 `BLOCKED`；拿到锁后继续 `RUNNABLE`；发现条件不满足调用 `wait` 进入 `WAITING`；被通知后重新竞争锁，可能短暂 `BLOCKED`；拿到锁后再次 `RUNNABLE`；之后调用 `sleep` 进入 `TIMED_WAITING`；时间到后继续 `RUNNABLE`；最后执行结束进入 `TERMINATED`。

转换点比状态名更重要。`NEW` 到 `RUNNABLE` 的关键触发是 `start()`，并且只能发生一次；`RUNNABLE` 到 `BLOCKED` 的关键触发是进入被占用的内置监视器；`RUNNABLE` 到 `WAITING` 的关键触发是无超时等待方法；`RUNNABLE` 到 `TIMED_WAITING` 的关键触发是带超时等待或睡眠；等待状态回到 `RUNNABLE` 的关键触发可能是通知、超时、中断、许可或目标线程结束；任意执行路径只要 `run()` 结束，最终都会进入 `TERMINATED`。

中断会影响许多等待转换，但中断不是强制杀死线程。对 `sleep`、`wait`、`join` 等会响应中断的方法，中断通常会让等待提前结束并抛出 `InterruptedException`，同时清除中断标志。对某些不抛出该异常的等待或计算，中断只是设置标志，线程是否退出取决于代码是否检查并响应。把 `interrupt()` 理解成“立即停止线程”是严重误区。它更准确地说是协作式取消信号。

状态转换还受到同步器实现影响。内置锁竞争显示为 `BLOCKED`，而 `ReentrantLock.lock()` 等待时通常表现为 `WAITING`，带超时或可中断获取可能表现为 `TIMED_WAITING` 或中断返回。`BlockingQueue.take()` 在队列为空时可能让线程进入 `WAITING`，`poll(timeout)` 可能进入 `TIMED_WAITING`。因此，阅读 dump 时要知道使用的是哪类同步工具，不能把所有等待都套进 `synchronized` 的模型。

## 锁竞争与状态观察
锁竞争是线程状态最常见的诊断场景之一。内置锁由每个对象的监视器提供，进入 `synchronized` 时如果监视器已被其他线程持有，当前线程就会等待，状态通常显示为 `BLOCKED`。等待线程不持有目标锁，也不能执行临界区代码；持锁线程继续运行，直到退出同步块或同步方法时释放监视器。释放后，等待线程并不按业务意义上的绝对公平顺序必然获得锁，具体竞争策略由运行时实现决定。

诊断 `BLOCKED` 时，应寻找三个信息：等待者卡在哪里，目标锁是什么，持有者在哪里。等待者的栈能定位入口，目标锁标识能把多个线程关联到同一把锁，持有者栈能解释为什么锁迟迟不释放。如果持有者正在执行慢 I/O、远程调用、复杂计算或等待另一个锁，问题根因通常在持有者，而不是在所有显示 `BLOCKED` 的线程上。

死锁是锁竞争的极端情况。两个或多个线程形成环形等待：线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1，双方都无法继续。线程 dump 往往能识别 Java 级别的监视器死锁，并给出 involved threads 和 locked monitors。解决死锁不能靠增加 `sleep` 或提高优先级，而要打破环：固定锁顺序，减少嵌套锁，使用超时锁获取，拆分资源所有权，或者重新设计状态转移。

使用显式锁时，状态观察会不同。`ReentrantLock` 不使用对象监视器表达竞争，等待线程通常停在抽象队列同步器相关栈上，状态可能是 `WAITING` 或 `TIMED_WAITING`。这不表示没有锁竞争，只表示竞争不是以 `BLOCKED` 形式呈现。显式锁的优势包括可中断获取、可超时获取、公平策略选项、多个条件队列等；代价是必须在 `finally` 中释放锁，并且诊断时要理解它的队列和条件机制。

锁竞争还和可见性相关。无论是 `synchronized` 还是 `Lock`，正确使用时都不仅提供互斥，也提供内存可见性。一个线程在释放锁之前对共享变量的写入，对随后获得同一把锁的线程可见。关键是“同一把锁”。如果写端用一把锁，读端不用锁或使用另一把锁，就无法获得这条可见性保证。许多状态相关误判来自只看到了互斥动作，却忽略了读写两端是否使用同一个同步协议。

## `wait`、`join`、`sleep`、`park` 的差异
`wait`、`join`、`sleep` 和 `park` 都可能让线程暂停推进，但它们表达的语义完全不同。混淆这些方法，是理解线程状态时最常见的障碍。

`sleep` 表达的是“当前线程暂停一段时间”。它不依赖某个条件，不释放已持有的监视器，不需要其他线程通知。时间到或被中断后，线程获得继续竞争执行的机会。它适合简单的节流、演示、退避等待等有限场景，不适合作为线程间协作的主要手段。用 `sleep` 等待共享变量变化，通常意味着代码缺少明确通知或可见性设计。

`wait` 表达的是“在某个对象监视器保护的条件不满足时，释放锁并等待条件可能变化”。它必须和 `synchronized`、共享条件、循环检查、`notify` 或 `notifyAll` 配合使用。`wait` 的重点不是暂停，而是条件协作：等待者释放锁，让修改条件的线程有机会进入临界区；通知者修改条件后发出信号；等待者醒来后重新拿锁并再次检查条件。少了任何一环，都可能出现遗漏通知、条件竞争或永久等待。

`join` 表达的是“等待另一个线程结束”。它是生命周期协作工具，不是一般条件变量。调用者关心的是目标线程是否终止，而不是某个任意业务条件是否满足。它常用于主线程等待工作线程、测试代码等待后台任务、分阶段执行中等待前一阶段完成。`join` 的可见性语义也让它适合在目标线程结束后读取其安全发布的结果，不过更复杂的结果和异常传播通常应交给 `Future` 等工具。

`park` 表达的是“基于许可挂起当前线程”。它比 `wait` 更底层，不要求持有监视器，也不自动绑定某个条件对象。`unpark` 可以先于 `park` 发生并留下一个许可，这避免了某些 `wait/notify` 中常见的先通知后等待问题；但它也要求调用方自己维护条件和协议。高级同步器使用 `park` 构建可靠队列，普通业务代码直接使用时应非常谨慎。

可以用几个问题区分它们：是否必须持有监视器？`wait` 必须，`sleep`、`join`、`park` 不需要调用方持有业务对象监视器。是否释放已持有的监视器？`wait` 会释放对应对象监视器，`sleep` 不会，`join` 的内部实现细节不应被当作业务锁释放机制，`park` 也不会自动释放任意锁。等待条件是什么？`sleep` 等时间，`wait` 等条件变化，`join` 等线程终止，`park` 等许可或中断等唤醒因素。状态表现是什么？无超时等待多为 `WAITING`，带超时等待或睡眠多为 `TIMED_WAITING`。

## 中断、超时与唤醒的边界
线程状态变化经常伴随中断、超时和唤醒，但这三个概念不能混为一谈。中断是一个线程向另一个线程发出的协作式取消信号；超时是等待方法自己设置的时间边界；唤醒是等待状态结束的一类事件，可能来自通知、许可、目标线程结束、中断、超时或虚假唤醒。等待方法返回，只说明等待结束，不一定说明业务条件成立。

处理中断时，最重要的是保持语义一致。调用会抛出 `InterruptedException` 的方法时，当前层要么完成必要清理并结束任务，要么恢复中断标志并把控制权交给上层。静默吞掉中断会破坏取消协议，使关闭流程卡住。特别是在循环等待中，如果捕获中断后继续下一轮等待，又不恢复标志、不退出、不记录，就等于告诉上层“取消请求不存在”。

超时处理也要有明确策略。等待超时后，可以重试、取消、降级、抛出异常或记录诊断，但不能假装成功。带超时的 `join` 返回后要检查目标线程是否仍然存活；带超时的条件等待返回后要检查条件是否满足；带超时的获取锁失败后要决定是否放弃本轮操作。超时只是把无限等待变成有限等待，它本身不是业务完成信号。

唤醒还存在“被唤醒但不能继续”的情况。`wait` 被 `notifyAll` 唤醒后，多个等待线程会重新竞争同一把锁；即使某个线程拿到锁，也可能发现条件已被其他线程消费，于是继续等待。`park` 返回后也应检查外部条件，因为返回可能由许可、中断或其他原因触发。成熟的并发代码通常把等待写成“检查条件，不满足则等待，返回后继续检查”的结构，而不是把某次唤醒当成绝对承诺。

## 线程 dump 中如何解读六种状态
线程 dump 是观察线程状态最常用的工具。它通常包含线程名、线程 id、优先级、守护属性、Java 状态、调用栈、锁持有和锁等待信息。分析 dump 时，状态只是入口，调用栈和锁关系才是证据链。一个高质量的结论至少要能说明：线程停在哪个方法，等待什么资源，由谁释放，是否有超时或中断路径，是否有大量相同模式的线程。

看到 `NEW` 的线程通常较少，因为未启动线程只有在对象被保留引用且恰好被观察到时才会出现。若系统中大量线程对象处于 `NEW`，应检查是否创建了线程却没有启动，或者把线程对象当作任务对象缓存起来。更常见的问题不是 `NEW` 本身，而是任务创建和执行管理混乱。

看到大量 `RUNNABLE` 时，要先看 CPU。如果 CPU 很高，再看栈是否集中在计算、循环、序列化、正则、加密、压缩、集合遍历等热点位置；如果 CPU 不高，再看是否集中在本地 I/O、网络读写、文件系统或底层库调用。不要只凭 `RUNNABLE` 判断线程“没有阻塞”。对 Java 来说，它可能没有在 Java 同步器上等待；对系统来说，它仍可能受底层资源限制。

看到大量 `BLOCKED` 时，要按锁聚合。多个线程等待同一个监视器，说明这把锁是瓶颈或某个持锁线程异常。应找到持有该锁的线程栈，确认它是在正常短临界区内，还是在慢调用、嵌套锁、等待外部资源或执行回调。如果多个锁之间形成环，要按死锁处理；如果只是单把锁竞争严重，则考虑缩小临界区、减少共享状态或调整并发结构。

看到 `WAITING` 时，要识别等待类型。栈在 `Object.wait` 附近，说明等待对象条件通知；栈在 `Thread.join` 附近，说明等待目标线程结束；栈在 `LockSupport.park` 或同步器队列中，说明等待某个并发工具条件。不同等待有不同唤醒者：`wait` 需要通知者，`join` 需要目标线程结束，`park` 需要许可或同步器状态变化。排查时要沿着唤醒者继续追踪。

看到 `TIMED_WAITING` 时，要区分正常定时等待和异常拖延。线程池空闲工作线程等待任务、定时调度线程等待下一次触发、重试逻辑进行退避，可能都正常；但业务线程大量 `sleep`、大量限时 `poll` 超时、或者所有请求都卡在同一个带超时外部调用上，就需要分析容量、延迟和失败路径。限时等待的存在不代表安全，超时后是否处理才是关键。

看到 `TERMINATED` 时，要关注引用和结果。如果线程已经终止却仍被长期持有，可能是对象引用没有释放；如果调用方只知道线程终止却不知道任务是否成功，说明结果传播机制不足；如果线程池工作线程大量终止，可能是线程工厂、异常处理或池关闭策略有问题。终止状态本身是正常生命周期的一部分，异常的是终止原因不可见或终止后资源仍被错误持有。

## 状态不是业务同步条件
`Thread.getState()` 的返回值是观察结果，不是同步协议。用它驱动业务流程通常是不可靠的。例如，一个线程反复检查另一个线程是否 `TERMINATED`，然后读取普通共享字段，这并不能替代 `join` 或其他同步机制；一个线程看到另一个线程是 `WAITING` 后再 `notify`，也不能保证不会发生竞态，因为状态可能在检查和通知之间变化；一个线程等待另一个线程变成 `BLOCKED` 再做操作，更是把程序正确性建立在调度时机上。

可靠同步应表达条件，而不是表达状态猜测。如果需要等待线程结束，用 `join` 或 `Future.get`；如果需要等待计数归零，用 `CountDownLatch`；如果需要等待队列有元素，用 `BlockingQueue.take`；如果需要保护共享变量，用锁、`volatile` 或原子类；如果需要阶段协作，用 `CyclicBarrier` 或 `Phaser`。这些工具的共同点是定义了条件、等待方式、唤醒方式和内存语义，而 `getState()` 只提供一个可能马上过期的标签。

状态也不应被用来判断线程安全。一个线程当前不是 `BLOCKED`，不代表没有数据竞争；一个线程处于 `WAITING`，不代表它释放了所有业务资源；一个线程处于 `RUNNABLE`，不代表它能看到其他线程刚写入的普通字段；一个线程已经 `TERMINATED`，也不代表结果已经通过正确同步发布。线程安全要回到共享状态的访问协议，而不是回到线程当前标签。

在测试中依赖状态尤其危险。很多单元测试会用 `sleep` 等待后台线程“差不多执行到某处”，或者轮询状态直到看到 `WAITING` 再继续断言。这类测试可能在本机稳定，在慢机器、快机器、不同虚拟机参数或高负载环境下随机失败。更稳妥的做法是用门闩、屏障、队列或可控钩子让测试和被测线程建立明确握手。

## 内存可见性与状态的关系
线程状态描述执行位置，但并不自动解决内存可见性。Java 内存模型规定了哪些操作之间存在 happens-before 关系，只有通过这些关系，一个线程的写入才能被另一个线程可靠观察。常见关系包括：线程调用 `start()` 前的动作 happens-before 新线程中的动作；一个线程中的动作 happens-before 另一个线程从该线程的 `join` 成功返回；监视器解锁 happens-before 后续对同一监视器的加锁；`volatile` 写 happens-before 后续对同一变量的读；并发工具也会通过自身规范提供相应同步效果。

这意味着，状态转换常常伴随可见性边界，但不能随意泛化。`start` 和 `join` 有明确语义，锁释放和获取也有明确语义；但简单读取 `getState()` 并不能让另一个线程的普通写入变得可见。比如工作线程写入一个非 `volatile` 字段后结束，主线程如果没有 `join`、锁、`Future.get` 或其他同步动作，只是碰巧看到线程状态变化，再读取该字段，就没有建立足够的可见性依据。

`wait/notify` 也必须和同一监视器下的条件读写配合。通知本身不是把所有数据神奇刷新给等待者；真正的可见性来自通知线程在持有同一监视器时修改条件并释放锁，等待线程从 `wait` 返回前重新获得同一监视器。若条件变量在锁外读写，或者通知使用的对象和保护条件的锁不是同一个，程序就可能出现看似被通知却看不到正确状态的问题。

`sleep` 更不提供可见性保证。一个线程 `sleep` 一段时间后醒来，只能说明时间过去了，不能说明其他线程的普通写入已经对它可见。用 `sleep` 等待标志位变化，如果标志位不是 `volatile`，也没有锁或其他同步关系，可能在某些优化和缓存条件下长期看不到变化。即使当前测试能看到，也只是实现和时机的结果，不是可依赖的规范保证。

## 常见边界与误区
第一个误区是把 Java 六种状态等同于操作系统完整状态。Java 的 `RUNNABLE` 覆盖范围更粗，`BLOCKED` 只指监视器竞争，`WAITING` 和 `TIMED_WAITING` 也只表达 Java 等待语义。诊断时必须结合栈和系统指标。

第二个误区是认为 `BLOCKED` 才叫阻塞。很多等待发生在显式锁、队列、同步器、I/O 或本地方法中，Java 状态可能不是 `BLOCKED`。如果只搜索 `BLOCKED`，会漏掉大量真实等待。

第三个误区是认为 `sleep` 会释放锁。`sleep` 不释放已经持有的监视器，也不释放显式锁。持锁 `sleep` 会延长其他线程等待时间，通常不是协作手段。

第四个误区是认为 `notify` 后等待线程马上执行。`notify` 只唤醒等待集合中的一个线程，使它有机会重新竞争锁；通知线程还要退出同步块释放监视器，被通知线程也要竞争成功后才能从 `wait` 返回。

第五个误区是用 `if` 包裹 `wait`。等待返回不代表条件成立，必须用循环检查条件。否则会被虚假唤醒、竞争消费或错误通知击穿。

第六个误区是认为 `interrupt` 会杀死线程。中断是协作信号，线程要在可中断等待中响应异常，或在运行中检查中断标志并退出。不可取消的阻塞或不检查标志的循环不会因为中断自动停止。

第七个误区是把 `TIMED_WAITING` 当成一定健康。带超时等待比无限等待更可控，但如果超时后没有失败处理，或者大量线程依赖固定 `sleep` 轮询，本质问题仍然存在。

第八个误区是认为线程终止后结果自然可见。读取结果需要同步边界。`join`、`Future.get`、锁、`volatile` 或同步器都可以建立关系；简单观察状态不够。

第九个误区是把状态当成性能指标。状态能提示等待类型，但不能替代 CPU 采样、锁分析、队列长度、吞吐量、延迟分布和系统资源指标。尤其是 `RUNNABLE` 数量与 CPU 消耗之间没有简单线性关系。

第十个误区是把线程状态作为长期设计接口。业务模块不应要求调用方等待某个线程进入某个 `Thread.State`，这会暴露实现细节并制造竞态。模块应暴露任务完成、结果可用、关闭完成、容量不足等业务层信号。

## 实践中的诊断步骤
排查线程状态相关问题时，可以按固定顺序缩小范围。第一步，确认现象：是 CPU 高、响应慢、任务不结束、线程数增长、死锁、关闭卡住，还是结果不可见。不同现象对应不同状态解读，不能一上来只看某个线程标签。

第二步，采集多次线程 dump。单次 dump 只能看到瞬间状态，多次 dump 可以观察线程是否停在同一栈、是否持续等待同一锁、是否在不同调用之间推进。对于偶发问题，间隔采集比单点采集更有价值。如果同一批线程在多次 dump 中保持相同等待栈，说明它们很可能真的卡住；如果栈持续变化，问题可能是慢而不是死。

第三步，按状态聚类。`BLOCKED` 聚合同一监视器，查持有者；`WAITING` 聚合同一同步器或等待对象，查唤醒者；`TIMED_WAITING` 聚合同一超时调用，查是否正常节流或外部依赖变慢；`RUNNABLE` 聚合同一热点栈，结合 CPU 判断是否忙等或底层阻塞。

第四步，沿等待链追踪。等待线程不是根因的概率很高，唤醒者、持锁者、目标线程或外部资源才是关键。等待 `join` 就找被等待线程；等待锁就找持锁线程；等待队列就看生产者是否还在投递；等待计数器就看负责递减的任务是否失败；等待 `Future` 就看执行器是否饱和、任务是否异常、是否发生线程池内互等。

第五步，检查退出路径。任何等待都要问：中断是否有效，超时是否存在，关闭时是否会唤醒，异常时是否释放资源，失败是否记录。很多线程问题不是正常路径不会唤醒，而是异常路径跳过了唤醒、计数递减或许可释放。

第六步，回到同步协议。确认共享条件由哪把锁或哪个原子变量保护，读写是否使用同一协议，结果是否安全发布，是否用 `sleep` 代替了通知，是否用普通字段承载跨线程取消。只有协议闭合，状态才有解释力。

## 一个完整的理解框架
理解线程六种状态，可以把每个线程看成在三条轴上移动。第一条轴是生命周期：未启动、可运行、结束。第二条轴是等待原因：等待 CPU、等待监视器、等待条件、等待时间、等待线程结束、等待许可或外部资源。第三条轴是同步关系：是否有锁、是否有可见性保证、是否有条件谓词、是否有取消和超时。

`NEW` 和 `TERMINATED` 位于生命周期两端，强调线程对象只能启动一次，以及结束后不能复活。`RUNNABLE` 位于执行中间态，强调 Java 层面没有进一步区分就绪和运行。`BLOCKED` 指向内置锁竞争，强调监视器入口等待。`WAITING` 指向无限期协作等待，强调需要其他动作。`TIMED_WAITING` 指向有限等待，强调时间边界但不保证条件成立。

当你看到某个状态时，应立刻补充三个问题。第一，它为什么进入这个状态？也就是触发方法、锁、同步器或调度原因是什么。第二，它怎样离开这个状态？也就是谁释放锁、谁通知、谁结束、哪个时间到期、哪个中断生效。第三，离开后它能否看到正确数据？也就是是否存在 happens-before 关系。只有这三个问题都能回答，状态分析才从“看标签”变成了“解释系统行为”。

这个框架也能指导写代码。创建线程前准备好不可变输入，启动后不要依赖调度顺序；保护共享状态时使用一致锁或明确并发工具；等待条件时循环检查并处理中断；设置超时时不要把返回当成功；结束线程时通过 `join`、`Future` 或同步器传播结果；诊断时用状态、栈、锁和指标共同判断。六种状态并不复杂，复杂的是它们背后的协作协议。

## 小结
Java 线程六种状态是观察线程生命周期和等待原因的基础语言。`NEW` 表示线程对象尚未启动，`RUNNABLE` 表示线程处于 Java 可运行范围，`BLOCKED` 表示等待进入 `synchronized` 监视器，`WAITING` 表示无限期等待其他动作，`TIMED_WAITING` 表示带时间边界的等待，`TERMINATED` 表示执行流已经结束。

掌握它们时，要始终记住三条边界。第一，Java 状态不是操作系统状态的一一映射，尤其是 `RUNNABLE` 和 `BLOCKED` 的含义不能按字面扩大。第二，状态是诊断快照，不是业务同步条件，不能用 `getState()` 替代锁、条件、`join`、`Future` 或同步器。第三，等待和唤醒必须结合内存可见性、条件检查、中断处理和失败路径理解，否则线程看似被唤醒，程序仍可能不正确。

真正有用的能力不是背诵状态转换图，而是在看到线程 dump 或并发故障时，能沿着“线程停在哪里、等待什么、谁能让它继续、继续后能否看到正确数据”这条链路推导。只要这条链路清晰，线程状态就不再是零散枚举，而是解释 Java 并发程序运行行为的一套可靠工具。

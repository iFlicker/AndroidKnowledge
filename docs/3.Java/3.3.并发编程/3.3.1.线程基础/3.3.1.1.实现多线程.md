# 3.3.1.1 实现多线程

## 定位
实现多线程是 Java 并发编程中的一个独立主题，它解决的问题是：把一段任务交给独立执行流运行。学习这个主题时，不能只记一个类名或一个方法调用，而要把它放回并发程序的完整链路中理解：任务从哪里产生，在哪个线程中执行，是否访问共享状态，遇到等待和取消时如何响应，结果或异常又如何回到调用方。只有这些问题都清楚，代码才不仅是“能跑”，而是在压力、异常和关闭阶段仍然可解释。

阅读 Java 并发文档时要把三个层次分开：语言层面的语义、类库层面的工具、运行时和操作系统层面的调度。语言语义回答“哪些结果是被规范允许的”，类库工具回答“怎样表达等待、通知、排队、取消和结果传递”，运行时调度回答“什么时候真正获得 CPU、什么时候被阻塞、什么时候被唤醒”。很多并发错误来自把这三层混在一起，例如用时间先后推导可见性，用线程优先级推导执行顺序，或者用一个看起来没有报错的测试结果推导程序在高并发下也正确。

从知识体系上看，实现多线程既连接基础线程模型，也连接后续的线程安全、线程池、同步器和内存模型。它的学习目标不是背诵 API，而是建立判断能力：什么时候该使用它，什么时候应该换成更高层或更低层工具，什么时候一个看似简单的写法会破坏可见性、原子性或生命周期。本文保持通用 Java 技术视角，只讨论 Java 语言和标准并发库中的机制、边界与实践。

## 解决的问题
实现多线程面向的直接问题是把一段任务交给独立执行流运行。在单线程程序中，控制流沿着调用栈顺序前进，数据读写通常由同一个线程完成；进入并发环境后，执行顺序、等待位置、异常传播和状态可见性都可能分离。这个主题的价值就在于给这种分离建立清晰规则，让开发者知道哪些行为由规范保证，哪些只是当前机器上的偶然结果。

Thread 适合演示和少量定制，Runnable 适合无返回任务，Callable 适合有返回值或异常传播的任务，线程池适合长期管理并发资源。

更具体地说，实现多线程会影响三个方面。第一是执行关系：某段代码由哪个线程执行，是否可能与其他代码并行，是否存在先后约束。第二是状态关系：共享变量在多个线程之间是否有一致的观察方式，复合操作是否会被打断，更新是否能被其他线程及时看见。第三是生命周期关系：任务开始、等待、取消、失败、完成和资源释放是否有明确路径。并发代码一旦缺少这些关系，就容易出现测试环境偶尔通过、生产压力下随机失败的问题。

## 核心 API 与术语
围绕实现多线程，最常见的 API 或术语包括：Thread、Runnable、Callable、ExecutorService。这些名称背后不是孤立功能点，而是一组协作约定。写代码时应先识别它们承担的是执行、排队、互斥、条件等待、结果传递、取消还是可见性职责。职责判断清楚后，API 的选择通常会变得直接。

继承 Thread、实现 Runnable、实现 Callable、提交到线程池。这条流程描述了实现多线程在典型使用中的状态变化。流程中的每一步都可能失败或被中断，因此真实代码不能只写“理想路径”。例如等待可能超时，提交可能被拒绝，锁可能长期竞争，任务可能抛异常，关闭可能遇到不响应中断的工作线程。把失败路径提前纳入设计，是并发代码和普通顺序代码的重要差异。

下面的片段只用于说明核心用法，重点是观察调用位置和资源边界，而不是把它当成完整框架：

```java
new Thread(() -> work()).start();
```

这类代码最容易被忽略的是边界条件。调用前对象是否已经初始化完成，调用后是否需要等待或关闭，异常是否会被吞掉，当前线程被中断时是否应继续执行，都会影响最终正确性。写并发代码时，每个 API 调用都应回答“失败时怎么办”和“谁负责收尾”。

## 从 Thread 开始理解多线程
Thread 是 Java 中表示线程的基础类型。一个 Thread 对象不是一段普通函数调用，而是对一个独立执行流的建模：它有自己的调用栈、线程名、优先级、中断标志、守护线程属性、生命周期状态和未捕获异常处理器。创建 Thread 对象只是创建了一个 Java 层面的线程描述；只有调用 start 方法，运行时才会为它安排真正的执行。start 和 run 的差异是学习 Thread 时的第一条边界：start 表示请求启动新线程，run 只是一个普通方法调用。如果直接调用 run，代码仍然在当前线程的调用栈中顺序执行，并没有形成新的并发执行流。

继承 Thread 可以实现多线程，但它通常不是最推荐的业务写法。继承 Thread 的直接优势是简单：把要执行的逻辑写进 run 方法，然后调用 start。它适合小实验、学习线程生命周期、需要定制线程名称或未捕获异常处理器的少量场景。问题在于，继承会把“任务逻辑”和“线程载体”绑在一起。任务本身只是要做的工作，而 Thread 是承载工作的执行资源；二者一旦混合，任务就难以复用，也难以交给线程池、定时执行器或其他调度机制管理。Java 只有单继承，业务任务继承 Thread 后还会占用继承位置，这也会让设计变窄。

Thread 还有一个容易被忽略的限制：同一个 Thread 实例只能启动一次。线程终止后不能再次 start。这个规则看似简单，却能帮助理解“线程”和“任务”的差别。任务可以被重新创建、重新提交、批量执行；线程是一次生命周期对象，从 NEW 到 RUNNABLE，再到终止状态后就结束。试图复用一个已经启动过的 Thread，说明代码把线程当成了可重复执行的任务容器。正确做法通常是重新创建任务并提交给合适的执行器，而不是保存 Thread 对象反复启动。

```java
class ReportThread extends Thread {
    @Override
    public void run() {
        generateReport();
    }
}

Thread thread = new ReportThread();
thread.setName("report-worker");
thread.setUncaughtExceptionHandler((t, error) ->
        System.err.println(t.getName() + " failed: " + error.getMessage()));
thread.start();
```

这段代码展示的是 Thread 的基本能力：自定义线程、命名、启动和处理未捕获异常。真正写生产级并发代码时，还要继续问几个问题：调用方是否需要等待报告生成完成，报告生成失败后是否要通知调用方，线程是否应该响应取消，报告任务数量变多时是否允许无限创建线程。只要这些问题出现，裸 Thread 的表达力就开始不够，需要引入 Runnable、Callable、Future 或 Executor。

## Runnable：把任务从线程中拆出来
Runnable 的核心价值不是“另一种写多线程的语法”，而是把要执行的任务从 Thread 中拆出来。Runnable 只有一个 run 方法，没有返回值，也不能在方法签名上声明受检异常。它表达的是“可以被执行的一段动作”。这段动作可以交给 Thread 执行，也可以交给 Executor 执行，还可以在测试中直接调用。任务和执行载体分离后，代码的复用性和可组合性都会提高。

```java
Runnable task = () -> {
    loadData();
    transformData();
    saveResult();
};

new Thread(task, "data-import-worker").start();
```

Runnable 适合描述不需要向调用方返回值的任务，例如记录日志、刷新缓存、发送通知、异步清理临时资源、执行某个独立动作等。这里的“不返回值”并不表示任务没有结果，而是结果不通过方法返回传递。任务可能通过共享队列、回调、外部存储、日志、状态对象或同步器表达结果。也正因为如此，Runnable 很容易把结果传播和状态共享写得含糊：任务内部修改某个外部变量，调用方又在另一个线程读取，如果没有同步关系，就可能读到旧值或中间状态。

Runnable 的另一个边界是异常传播。run 方法不能声明受检异常，运行时异常如果没有在任务内部捕获，会沿着执行它的线程向外抛出。对于裸 Thread，这个异常会触发该线程的未捕获异常处理器；对于线程池，异常通常会被执行框架捕获，并影响 Future 或线程池工作线程的后续处理。调用方不能指望像普通方法调用那样在 submit 或 start 的地方直接 catch 到任务内部的异常。start 返回时，任务可能还没开始；submit 返回时，任务只是被接受，异常要到执行阶段才会发生。

因此，Runnable 的正确使用重点是明确边界：它是否访问共享状态，异常由任务内部处理还是交给执行框架，调用方是否需要等待完成，是否需要取消，任务本身是否可以重复执行。一个没有共享状态、失败后只需记录日志、无需返回值的 Runnable 很简单；一个会修改外部对象、要求上游感知失败、需要超时取消的 Runnable，就需要配合 Future、同步器或更明确的结果对象。

## Callable 与 Future：让结果和异常回到调用方
Callable 和 Future 解决的是 Runnable 表达力不足的问题。Callable 表示有返回值的任务，call 方法可以返回一个结果，也可以抛出异常。Future 表示一个异步计算的占位符，它把“任务已经提交”和“任务已经完成”这两个时间点分开。提交任务后，调用方立即拿到 Future；需要结果时，再通过 get 等待并取得结果。这个模型让多线程程序不必把结果写到共享变量中，也不必用 sleep 猜测任务是否结束。

```java
Callable<Integer> task = () -> {
    int count = queryRecordCount();
    return count;
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(task);

try {
    Integer count = future.get();
    System.out.println("count = " + count);
} finally {
    executor.shutdown();
}
```

Future.get 的语义非常重要。它会等待任务完成；如果任务正常完成，返回 Callable 的结果；如果任务内部抛出异常，get 会抛出 ExecutionException，原始异常可以从 getCause 中取出；如果等待线程被中断，get 会抛出 InterruptedException；如果使用带超时的 get，超时后会抛出 TimeoutException。也就是说，Future 把异步任务的结果、失败、取消和等待中断都显式表达出来了。相比让后台线程悄悄失败，Future 至少给调用方留下了处理路径。

```java
try {
    Integer result = future.get(2, TimeUnit.SECONDS);
    use(result);
} catch (TimeoutException e) {
    future.cancel(true);
    throw new IllegalStateException("task timed out", e);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    future.cancel(true);
} catch (ExecutionException e) {
    Throwable cause = e.getCause();
    handleTaskFailure(cause);
}
```

这段代码体现了 Future 使用中的几个关键点。第一，等待最好有边界，尤其是上游线程不能无限阻塞时。第二，超时并不等于任务自动停止，TimeoutException 只说明调用方等待结果超时；如果希望尝试取消任务，需要显式调用 cancel。第三，cancel(true) 只是向正在执行任务的线程发出中断请求，它不能强行杀死线程；任务必须在内部响应中断，取消才可能及时生效。第四，捕获 InterruptedException 后通常要恢复中断标志，因为当前方法可能没有能力完成整个取消协议，恢复标志可以让更上层知道这个线程曾被请求中断。

Callable 和 Future 也有自己的边界。Future 本身只表示一个结果，不适合表达多个阶段、持续流式输出或复杂依赖编排。Future.get 是阻塞等待，如果大量工作线程互相 get 对方提交到同一个小线程池的任务，可能造成线程池饥饿甚至死锁。Future.cancel 的取消语义也是协作式的，不能替代任务内部的资源关闭、循环检查和中断处理。理解这些边界，才能避免把 Future 当成“保证任务一定能结束”的工具。

## Executor：把线程管理交给执行器
Executor 的核心思想是：调用方提交任务，执行器负责安排任务在哪里、何时、以什么资源执行。它进一步强化了线程与任务分离。任务只描述要做什么，Executor 描述如何执行。ExecutorService 在 Executor 的基础上增加了生命周期管理、提交 Callable、获取 Future、批量执行、关闭等待等能力。实际项目中，直接 new Thread 的代码越多，线程数量、命名、异常、关闭和资源上限越难统一；把任务交给受控的执行器，可以让这些策略集中管理。

线程池不是“让并发更快”的魔法，而是复用线程、限制资源、排队任务和集中治理生命周期的机制。创建线程有成本，过多线程会增加上下文切换和内存占用；任务到达速度超过处理速度时，需要有队列或拒绝策略表达背压；程序关闭时，需要停止接收新任务并等待已提交任务结束。ExecutorService 正是把这些问题放到一个对象上管理。

```java
ExecutorService executor = Executors.newFixedThreadPool(4);

List<Future<Result>> futures = new ArrayList<>();
for (Job job : jobs) {
    futures.add(executor.submit(() -> process(job)));
}

try {
    for (Future<Result> future : futures) {
        collect(future.get());
    }
} finally {
    executor.shutdown();
}
```

这段代码比手写多个 Thread 更清晰：任务数量可以很多，但同时执行的工作线程数量由线程池控制；每个任务都有 Future 表示结果；最后由调用方负责关闭执行器。不过它仍然有需要注意的地方。固定线程池背后通常有队列，任务提交速度如果长期高于处理速度，队列会堆积；collect 阶段按提交顺序 get，可能因为前面的慢任务阻塞后面已经完成的结果消费；任务内部如果再向同一个线程池提交任务并等待结果，也可能耗尽工作线程。线程池解决的是执行资源管理，不自动解决业务级依赖和容量设计。

使用 Executor 时，最重要的是避免把 `Executors` 工厂方法当成无脑默认值。它们很方便，但部分工厂方法隐藏了队列大小、线程数量或线程创建策略。例如缓存线程池可能在高峰时创建大量线程；单线程执行器会把所有任务串行化，一个慢任务会阻塞后续任务；固定线程池如果搭配无界队列，过载可能先表现为内存增长而不是提交失败。更严格的代码通常会显式构造 ThreadPoolExecutor，指定核心线程数、最大线程数、队列、线程工厂和拒绝策略。

```java
ThreadFactory factory = runnable -> {
    Thread thread = new Thread(runnable);
    thread.setName("import-worker-" + THREAD_ID.incrementAndGet());
    thread.setUncaughtExceptionHandler((t, e) ->
            System.err.println(t.getName() + " failed: " + e.getMessage()));
    return thread;
};

ExecutorService executor = new ThreadPoolExecutor(
        4,
        8,
        30,
        TimeUnit.SECONDS,
        new ArrayBlockingQueue<>(200),
        factory,
        new ThreadPoolExecutor.CallerRunsPolicy()
);
```

显式配置线程池的意义在于让容量规则可见。线程数决定同时执行的上限，队列容量决定短时突发可以吸收多少任务，拒绝策略决定过载时如何反馈，线程工厂决定命名、守护属性和异常处理。不同策略没有绝对优劣，只有是否符合当前任务性质。CPU 密集型任务通常不宜远超处理器能力地开线程；I/O 等待型任务可以有更多并发，但也要受外部资源限制；低延迟任务不适合排在无界队列后面；必须保序的任务可能需要单线程执行器或明确的分区串行化。

ExecutorService 的关闭同样是实现多线程的一部分。shutdown 表示不再接受新任务，但会继续执行已提交任务；shutdownNow 会尝试中断正在执行的任务，并返回尚未开始执行的任务列表；awaitTermination 用于等待线程池终止。关闭流程不能只写一行 shutdown 后结束，因为任务可能不响应中断，调用方可能在关闭期间被中断，尚未执行的任务可能需要记录或补偿。

```java
executor.shutdown();
try {
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
        List<Runnable> dropped = executor.shutdownNow();
        logUnstartedTasks(dropped);
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
    Thread.currentThread().interrupt();
}
```

这类关闭代码体现了一个原则：启动多线程任务的人，也应该能解释任务如何停止。不能停止的线程会阻止资源释放，不能取消的任务会拖慢关闭，不能观测的队列会隐藏过载。Executor 把线程管理集中起来，同时也要求开发者认真管理它的生命周期。

## 线程与任务分离
实现多线程最容易混淆的是“线程”和“任务”。线程是执行载体，任务是要执行的工作。一个线程可以执行很多任务，一个任务也可以被不同执行策略承载。把这二者分开后，很多设计选择会变得清楚：任务类不应该负责无限制创建线程；线程池不应该理解过多业务状态；调用方不应该通过共享可变字段偷偷拿结果；取消协议不应该只依赖外部强行停止线程。

线程与任务分离至少带来四个好处：任务可以在不同执行策略中复用，可以在测试中同步执行，线程数量、命名、异常处理和拒绝策略可以集中治理，任务也可以被包装成限流、计时、重试或 Future 形式，而不需要改写 Thread 子类。

分离之后，也要避免另一个极端：认为任务完全不需要关心线程。任务虽然不创建线程，但它必须知道自己运行在并发环境中。它要清楚是否可以阻塞，是否会访问共享状态，是否需要响应中断，是否能被重复执行，失败后是否幂等，是否持有外部资源。执行器提供执行上下文，任务仍然要遵守并发契约。

一个实用判断是：任务对象最好只保存完成工作所需的不可变输入，执行过程中产生的可变状态尽量限制在 run 或 call 方法内部；如果必须共享状态，就通过清晰的同步机制共享；如果必须返回结果，就通过返回值、Future、并发队列或受保护的结果容器传递。不要让任务对象携带一个没有保护的可变字段，再由多个线程同时读写，也不要让调用方在没有等待完成的情况下读取任务内部状态。

## 共享状态：实现多线程后的真正难点
创建多个线程并不难，难的是多个线程同时触碰同一份状态。共享状态包括普通字段、集合、计数器、缓存、状态标志、文件句柄、连接对象以及任何有可变内部状态的对象。没有共享状态时，多线程更多是任务调度问题；一旦有共享状态，就必须讨论原子性、可见性、有序性和不变式。

一个典型错误是把局部变量和共享字段混为一谈。方法内部新建、没有逃逸的局部对象通常只被当前线程访问，不需要同步。对象字段则可能随着对象引用被多个线程访问。即使字段是 private，也不代表线程安全；private 只限制访问语法，不限制同一个对象被多个线程同时调用。线程安全要看访问路径和同步协议，而不是看字段修饰符是否私有。

```java
class UnsafeCounter {
    private int value;

    void increment() {
        value++;
    }

    int get() {
        return value;
    }
}
```

value++ 不是一个不可分割动作，它通常包含读取、加一、写回三个步骤。多个线程同时执行时，可能都读到同一个旧值，然后分别写回相同的新值，导致更新丢失。把 value 改成 volatile 只能改善可见性，不能让复合操作变成原子操作。对于这种单变量计数，可以使用 synchronized、Lock 或 AtomicInteger。

```java
class AtomicCounter {
    private final AtomicInteger value = new AtomicInteger();

    int incrementAndGet() {
        return value.incrementAndGet();
    }

    int get() {
        return value.get();
    }
}
```

AtomicInteger 适合单变量的原子更新，但它不是所有共享状态的通用答案。如果业务规则涉及多个字段之间的一致性，例如“先判断余额是否足够，再扣减余额，再增加冻结金额”，单个 AtomicInteger 无法表达整体不变式。此时更适合使用同一把锁保护整个临界区，或者重新设计数据结构，让状态更新可以用不可变快照和原子引用表达。工具选择必须服务于不变式，而不是只追求“看起来无锁”。

```java
class Account {
    private int balance;
    private int frozen;

    synchronized boolean freeze(int amount) {
        if (amount <= 0 || balance < amount) {
            return false;
        }
        balance -= amount;
        frozen += amount;
        return true;
    }
}
```

这段代码的重点不是 synchronized 语法，而是同一把锁同时保护了检查和更新。临界区的边界刚好覆盖业务不变式：余额不能被扣成负数，balance 和 frozen 的变化要作为一个整体被其他线程观察。实现多线程时，很多 bug 都来自锁粒度和不变式不匹配：锁太小会保护不住复合操作，锁太大又可能降低并发度或引入长时间阻塞。正确粒度要由状态关系决定。

共享状态还包括集合。普通 ArrayList、HashMap 等集合在并发修改时不保证结构安全；并发容器能保护内部结构，但不能自动保护跨多步业务逻辑。例如 ConcurrentHashMap 的 putIfAbsent 能原子地表达“没有则放入”，比先 containsKey 再 put 更可靠；但如果业务需要同时更新多个 map 或 map 外部的计数器，仍然需要更上层的同步协议。

## 异常处理：不要让后台失败静默发生
在单线程程序中，异常通常沿调用栈向上传播，调用方能直接 catch。多线程程序中，任务的执行栈和提交任务的调用栈不是同一条。调用 start 或 submit 的线程通常不会直接收到任务内部抛出的异常。异常处理方式取决于任务如何执行。

对于裸 Thread，run 方法中未捕获的 RuntimeException 或 Error 会导致该线程终止，并交给 UncaughtExceptionHandler 处理。如果没有设置自定义处理器，默认处理通常只是输出到标准错误。调用 start 的线程不会因此抛异常。因此，只用裸 Thread 执行重要任务时，至少应设置线程名和未捕获异常处理器，否则后台失败很容易被忽略。

对于 ExecutorService，通过 execute 提交 Runnable 和通过 submit 提交任务的异常表现并不完全相同。execute 提交的任务如果抛出运行时异常，会从工作线程的执行过程中冒出，线程池可能替换工作线程，异常有机会进入未捕获异常处理器。submit 提交的任务会被包装成 FutureTask，异常会被捕获并存入 Future，只有调用 Future.get 时才会以 ExecutionException 的形式重新出现。如果提交方从不读取 Future，任务失败就可能在业务层面静默。

这个差异带来一个实践要求：重要任务不要只 submit 后丢弃 Future。可以集中收集 Future 并检查结果，也可以在任务内部捕获并记录异常，还可以扩展 ThreadPoolExecutor 的 afterExecute 做统一监控。无论采用哪种方式，目标都是让失败可见。多线程程序最危险的状态之一，是后台任务已经失败，但调用方只看到系统没有产生预期结果。

异常处理还要区分“任务失败”和“线程中断”。InterruptedException 不是普通业务失败，它通常表示当前线程被请求停止等待或协作取消。吞掉中断会破坏线程池关闭、上层超时和取消协议。当前方法如果能完成清理并终止，可以直接返回；如果不能完全处理，应调用 Thread.currentThread().interrupt() 恢复中断标志。不要把 InterruptedException 简单包装成运行时异常后继续执行很长逻辑，这会让调用方失去取消控制。

## 生命周期：从创建到结束都要可解释
线程生命周期不是只知道几个枚举状态，而是要能解释状态变化背后的原因。Thread.State 包括 NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。NEW 表示 Thread 对象已创建但尚未 start；RUNNABLE 表示可运行，可能正在运行，也可能等待操作系统调度；BLOCKED 表示等待进入 synchronized 临界区；WAITING 表示无限期等待某个条件，例如 join 或 wait；TIMED_WAITING 表示带时间限制的等待，例如 sleep、带超时的 wait 或 Future.get；TERMINATED 表示 run 方法结束或异常退出。

这些状态是诊断工具，不是业务协议。不能因为线程处于 RUNNABLE 就断定它正在消耗 CPU，也不能因为处于 WAITING 就断定它有问题。关键是知道它为什么等待、谁会唤醒它、等待条件是否还能满足。实现多线程时，生命周期设计应覆盖启动、执行、等待、取消、完成、失败和关闭。只关心启动，不关心结束，会留下难以回收的线程和资源。

join 是理解生命周期的基础工具。一个线程调用另一个线程的 join，表示等待后者终止。join 建立了重要的同步关系：被等待线程终止前的动作，对成功从 join 返回的线程可见。也就是说，如果工作线程在结束前写入某个结果，等待线程通过 join 确认它已经结束后，再读取结果，具备明确的可见性基础。但这并不意味着 join 可以替代所有同步。join 只能表达“线程已经结束”，不适合表达中间状态，也不适合让长期运行线程不断发布进展。

```java
class ResultHolder {
    int value;
}

ResultHolder holder = new ResultHolder();
Thread worker = new Thread(() -> holder.value = compute());
worker.start();
worker.join();
System.out.println(holder.value);
```

这段代码中，join 让主线程等待 worker 结束，并获得 worker 结束前写入的可见性。若去掉 join，主线程可能在线程尚未执行时就读取 value。若 worker 是长期服务线程，不会自然终止，那么 join 就不是合适的结果传递方式，应改用 Future、BlockingQueue、回调或其他同步结构。

取消是生命周期中最容易被误解的部分。Java 标准线程模型没有提供安全的强制杀死线程方式。stop 已经过时，因为它可能在任意位置终止线程，破坏对象不变式并释放锁。推荐做法是协作式取消：外部通过中断、volatile 标志、Future.cancel 或关闭资源发出停止请求，任务内部在合适位置检查并退出。协作式取消要求任务代码本身配合，例如循环中检查中断，阻塞调用能响应中断，finally 中释放资源。

```java
class PollingTask implements Runnable {
    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                pollOnce();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
        cleanup();
    }
}
```

这个例子表达的是取消边界：循环条件检查中断，阻塞方法抛出 InterruptedException 后恢复中断标志，循环结束后清理资源。真实代码还需要确保 cleanup 不会无限阻塞，并且 pollOnce 内部不会吞掉中断。多线程任务如果完全不检查取消信号，即使 Future.cancel(true) 或 shutdownNow 被调用，也可能继续运行很久。

守护线程也是生命周期的一部分。daemon 线程不会阻止 Java 进程退出，非 daemon 线程会阻止进程结束。守护线程适合执行辅助性后台工作，但不适合承载必须完成的数据写入、事务提交或关键清理。因为当所有非守护线程结束时，守护线程可能没有机会正常完成。设置 daemon 属性必须在 start 之前完成，且要清楚它对进程退出语义的影响。

## 选择标准：不同实现方式解决不同问题
实现多线程并不是在几个 API 之间做语法偏好选择，而是在任务语义、结果需求、资源治理和生命周期要求之间做权衡。下面的表格可以作为初步判断，但真实代码仍要结合容量、共享状态和异常处理一起看。

| 方式 | 适合场景 | 主要优势 | 主要边界 |
| --- | --- | --- | --- |
| 继承 Thread | 学习、实验、少量定制线程行为 | 直观，能直接接触线程属性 | 任务和线程耦合，不能复用同一线程实例，资源管理分散 |
| Runnable + Thread | 简单无返回任务，少量独立后台动作 | 任务从线程中拆出，写法轻量 | 没有返回值，异常不回到提交方，线程数量难统一治理 |
| Callable + Future | 需要结果、异常传播、等待和取消的异步任务 | 结果和失败可显式获取，支持超时等待 | get 会阻塞，取消是协作式，不适合复杂依赖编排 |
| ExecutorService | 多任务执行、线程复用、资源上限控制 | 集中管理线程、队列、关闭和拒绝策略 | 需要认真配置容量和生命周期，错误使用可能造成堆积或饥饿 |
| ThreadPoolExecutor | 对线程池参数有明确要求的场景 | 队列、线程数、工厂、拒绝策略可控 | 参数需要结合任务模型设计，配置不当会放大故障 |

如果任务只有一个、生命周期很短、只是为了说明并发执行，用 Thread 可以接受。如果任务逻辑要复用，优先把它写成 Runnable 或 Callable。如果调用方需要结果或失败原因，用 Callable/Future 比共享变量更清晰。如果任务会持续批量提交，就应使用 ExecutorService 管理线程，而不是每个任务 new Thread。如果系统对队列长度、拒绝策略、线程命名和关闭有要求，应显式配置 ThreadPoolExecutor。

还要根据任务性质选择并发度。CPU 密集型任务不宜让线程数远超处理器能力；I/O 等待型任务可以适度增加并发，但仍要受连接数、文件句柄、外部容量和内存限制；混合任务要避免把长阻塞任务和低延迟任务混放在同一线程池中。线程池不是越大越好，队列也不是越长越安全，过大的队列会把过载延迟到更难处理的位置。

任务之间存在依赖时，要特别警惕在线程池内部等待同一线程池的其他任务。假设一个固定大小为 2 的线程池中，两个任务都在执行中，并且各自提交子任务后调用 get 等待子任务完成；如果子任务也只能进入同一个线程池队列，就没有空闲线程执行它们，系统会卡住。这类问题不是 Future 的错，而是执行资源和依赖关系形成了闭环。解决思路包括拆分线程池、避免阻塞等待、使用完成队列、调整任务拆分方式或采用更适合依赖编排的工具。

## 边界和误区：实现多线程不等于线程安全
实现多线程只是让代码可以被多个执行流同时运行，它不自动带来线程安全。一个类在单线程下正确，不代表放到多个线程中仍然正确；一个任务能被线程池执行，不代表它访问的对象就是安全的；一个 Future 能返回结果，不代表任务中间状态可以随便共享。线程安全来自清晰的状态边界和同步协议，而不是来自“用了并发 API”这个事实。

第一个常见误区是认为启动顺序等于执行顺序。代码中先 start 线程 A，再 start 线程 B，只表示启动请求的发出顺序，不表示 A 的 run 一定先执行，也不表示 A 的某个写入一定先被 B 看到。调度由运行时和操作系统共同决定，程序不能依赖这种偶然顺序。需要顺序时，要用 join、锁、条件、Future、同步器或其他明确机制表达。

第二个误区是把 sleep 当成同步工具。sleep 只能让当前线程在一段时间内不主动运行，它不保证其他线程已经完成某个动作，也不建立可靠的可见性关系。用 sleep 等待任务完成，在本机可能“刚好够用”，在负载变化、机器变慢、任务变多或调度时机变化时就会失败。正确做法是等待具体条件：任务完成用 Future 或 join，队列非空用 BlockingQueue，阶段完成用 CountDownLatch，状态变化用锁条件或 volatile 标志。

第三个误区是认为 volatile 可以替代锁。volatile 适合状态标志、配置引用、安全发布等需要可见性和有序性的场景，但不适合多步复合操作。`volatile int count; count++;` 仍然会丢更新。锁不仅提供可见性，还提供互斥；原子类提供单变量原子更新；并发容器提供特定数据结构的并发访问。需求不同，工具不同。

第四个误区是忽略任务的重复执行问题。Runnable 和 Callable 对象可以被多次提交，但对象内部如果保存可变字段，就可能让多次执行互相污染。一个任务对象如果只封装不可变输入，重复提交通常更安全；如果它在字段中保存执行中间状态，就应该明确它是一次性对象，或者把中间状态改为方法局部变量。不要默认“任务对象”和“任务执行”是一一对应的，除非代码结构确实保证。

第五个误区是关闭责任不清。线程池通常应该由创建者关闭，而不是由随手拿到引用的任务关闭。库代码如果接收外部传入的 ExecutorService，一般不应擅自 shutdown，因为执行器可能由调用方共享管理。相反，如果某个组件自己创建了专用线程池，它就必须在组件关闭时释放它。所有权规则比 API 调用更重要。

## 一个完整实现路径的推演
可以用一个简单场景把上述内容串起来：调用方有一批独立任务，每个任务读取输入、计算结果、返回给调用方，任务可能失败，整体等待不能无限期，程序结束时要释放线程资源。这个需求不适合为每个任务继承 Thread，因为任务数量可能变化，线程创建无法集中管理；也不适合只用 Runnable 修改共享列表，因为结果和异常传播会变得混乱。更合适的建模是把单个工作写成 Callable，把批量执行交给 ExecutorService，把结果通过 Future 收集，把超时和关闭写进控制流程。

```java
ExecutorService executor = Executors.newFixedThreadPool(4);
List<Future<Result>> futures = new ArrayList<>();

for (Input input : inputs) {
    futures.add(executor.submit(() -> calculate(input)));
}

try {
    for (Future<Result> future : futures) {
        try {
            Result result = future.get(3, TimeUnit.SECONDS);
            save(result);
        } catch (TimeoutException e) {
            future.cancel(true);
            recordTimeout(e);
        } catch (ExecutionException e) {
            recordFailure(e.getCause());
        }
    }
} finally {
    executor.shutdownNow();
}
```

这段代码仍然不是万能模板，却展示了实现多线程的基本思路。任务逻辑由 Callable 表达，线程资源由 ExecutorService 管理，结果通过 Future 取回，失败通过 ExecutionException 暴露，等待通过超时设置边界，关闭通过 shutdownNow 尝试取消仍在运行的任务。接下来还要根据真实需求继续改进：是否应该保存所有 Future 后再统一等待，是否应该使用 CompletionService 按完成顺序消费，是否应该区分单个任务超时和整体批次超时，calculate 是否正确响应中断，save 是否需要同步或事务保护，线程池队列是否有上限。

这个推演说明，“实现多线程”不是选一个入口 API 后结束，而是围绕任务、执行器、结果、异常、共享状态和生命周期形成闭环。闭环中任何一处没有规则，问题都可能在压力、异常、取消或关闭阶段暴露。

## 核心机制
实现多线程的核心机制可以从“状态、队列、唤醒、可见性”四个词理解。状态描述对象当前处于哪个阶段，队列描述竞争者或等待者如何排列，唤醒描述等待线程重新获得执行机会的条件，可见性描述一个线程的写入何时能被另一个线程可靠观察到。不是每个主题都会显式包含四个结构，但成熟的并发工具通常都能在内部找到这些要素。

以实现多线程为例，状态变化必须是可解释的。线程从未启动到可运行再到终止，锁从未持有到被某个线程持有再到释放，任务从已提交到执行中再到完成或取消，同步器从计数未满足到条件满足，这些变化都需要明确触发点。如果状态触发点不明确，调用者就会被迫用 sleep、轮询或经验时间等待，而这正是并发程序脆弱的常见来源。

队列和唤醒决定等待成本。阻塞等待可以节省 CPU，但涉及线程挂起和恢复；自旋等待可以避免上下文切换，但会消耗处理器时间；公平排队能降低饥饿概率，但可能牺牲吞吐；非公平竞争能提升局部效率，但等待时间更难预测。理解实现多线程时，不应只问“会不会等待”，还要问“等待者在哪里排队、由谁唤醒、唤醒后是否立即满足条件”。

可见性则决定结果是否可信。一个线程完成了写入，不代表另一个线程一定能立刻看到；一个线程按代码顺序先写 A 再写 B，也不代表另一个线程没有同步就能按同样顺序观察。Java 通过 synchronized、volatile、Lock、线程启动和终止、并发容器等同步动作建立 happens-before 关系。实现多线程如果涉及共享状态，就必须说明它依赖哪一种同步关系。

## 使用方式
使用实现多线程时，第一步是识别任务性质。CPU 密集型任务、I/O 等待型任务、短小状态更新、长时间阻塞任务、阶段同步任务、资源限流任务，适合的工具不同。把所有需求都压到同一个 API 上，会让代码表面简单、实际边界混乱。例如需要结果时不应只用裸 Runnable，需要容量控制时不应只靠线程数量，需要条件协作时不应靠固定 sleep。

第二步是明确所有权。谁创建线程或同步器，谁提交任务，谁负责关闭，谁处理异常，谁记录拒绝，谁保证共享对象不被并发破坏，这些都属于所有权问题。所有权不清时，常见后果是线程池被局部代码关闭、共享锁被不同模块以不同顺序获取、ThreadLocal 被上一个任务污染、异常在后台线程中静默消失。

第三步是把正常路径和异常路径写在一起。只写成功路径的并发代码往往在演示中很好看，但在真实系统中会出现资源泄漏。锁释放要放在 finally，许可归还要放在 finally，计数器递减要放在 finally，ThreadLocal 清理要放在 finally，线程池关闭要处理 InterruptedException，等待结果要考虑 TimeoutException。并发代码的可靠性主要体现在这些收尾位置。

第四步是保留可替换空间。许多并发工具有相似目标但边界不同：wait/notify 可以表达条件等待，Condition 能拆分多个等待队列；CountDownLatch 适合一次性等待，CyclicBarrier 适合固定参与者循环阶段，Phaser 适合动态参与者；Atomic 类适合单变量乐观更新，Lock 适合复杂临界区。写文档和写代码都应说明为什么选择当前工具，而不是只说明它“可以用”。

## 状态流转
实现多线程的状态流转可以概括为：初始化阶段建立对象和依赖，运行阶段执行或等待，竞争阶段处理共享资源冲突，完成阶段发布结果或释放等待者，关闭阶段拒绝新工作并清理资源。这一描述虽然抽象，但适用于大多数 Java 并发工具。每个阶段都要避免“半初始化对象被发布”“等待条件丢失”“结果未安全发布”“关闭后仍提交任务”等问题。

初始化阶段最重要的是安全发布。线程启动前传入的不可变参数、构造完成后再发布的对象、通过 volatile 或锁保护发布的引用，都是可靠方式。反过来，如果 this 在构造过程中逃逸，或者共享集合在没有同步的情况下被多个线程看到，就可能让其他线程观察到未完成状态。实现多线程如果涉及对象共享，初始化边界必须清晰。

运行阶段最重要的是不要阻塞错位置。持锁执行慢 I/O、在线程池工作线程中等待同一个线程池的任务、在 ForkJoin 任务中长时间等待外部资源、在条件等待中忘记循环检查，都会制造隐蔽问题。等待不是错误，但等待必须有条件、超时、取消和诊断手段。

完成阶段最重要的是结果传播。任务返回值、异常、计数归零、状态标志变更、锁释放或 volatile 写入，都是向其他线程传播进展的方式。传播动作必须与读取动作配对：Future.get 读取任务结果，await 等待计数归零，lock 获取看到上一次 unlock 前的写入，volatile 读观察 volatile 写发布的状态。只有写端和读端使用同一套规则，状态流转才可靠。

## 线程安全边界
任务建模和线程生命周期分离

线程安全边界必须落到共享状态上。只要一个对象、集合、计数器、状态标志或外部资源会被多个线程同时访问，就要明确谁负责保护它、保护范围有多大、读写之间是否需要原子性、写入后是否需要被其他线程及时看见。没有共享状态的任务通常只需要关注取消和资源释放；有共享状态的任务则必须同时关注互斥、可见性、有序性和生命周期。

对实现多线程来说，最小线程安全边界通常不是“某个类线程安全”这么粗，而是“哪些方法可以并发调用、哪些参数不能被调用方继续修改、哪些返回值不能暴露内部可变对象、哪些回调可能在持锁时发生”。同一个类中，有的方法可能线程安全，有的方法可能要求外部同步；同一个对象中，有的字段不可变，有的字段必须受锁保护。文档和代码注释应尽量表达这种精确边界。

还要区分结构安全和业务安全。并发容器可以保证自身结构不被破坏，但不能保证跨多个操作的业务不变式；AtomicInteger 可以保证单次递增原子，但不能保证“检查余额再扣减再记录日志”整体一致；线程池可以保证任务被工作线程执行，但不能保证任务内部访问的对象安全。实现多线程如果被误认为“用了就线程安全”，往往会掩盖更上层的不变式问题。

## 内存可见性与有序性
Java 并发中，正确性不仅是“没有两个线程同时写”。一个线程写入后另一个线程能否看到，另一个线程看到多个写入时是否能按预期顺序理解，同样关键。实现多线程只要牵涉共享变量，就要考虑 happens-before。synchronized 的释放和随后获取、volatile 写和随后读、线程 start 前后的关系、线程终止和 join 的关系、并发工具内部的同步动作，都是常见可见性来源。

有序性问题常被低估。编译器和处理器可以在不改变单线程结果的前提下重排序指令；缓存也可能让一个核心暂时看不到另一个核心的写入。JMM 允许这些优化，同时要求程序通过同步动作表达跨线程约束。因此，不能因为一段代码“按顺序写着”就认为另一个线程也会按顺序看到。对实现多线程的使用如果缺少同步边界，就可能在压力下暴露出旧值、默认值或部分构造对象。

可见性也不等于原子性。volatile 能让状态标志及时可见，但不能让自增、复合检查或多字段更新自动变成一个不可分割操作；锁能同时提供互斥和可见性，但锁的对象必须一致；CAS 能保证单个位置的比较更新原子，但不能自动维护多个变量之间的不变式。选择工具时，应先判断需要的是可见性、原子性、有序性，还是三者都需要。

## 实践要点
第一，优先使用表达力更明确的高层工具。能用 BlockingQueue 表达生产消费，就不要手写 wait/notify；能用 ExecutorService 管理任务，就不要散落创建 Thread；能用 CountDownLatch 表达一次性等待，就不要用循环 sleep；能用 AtomicInteger 表达单变量计数，就不要扩大成不必要的重量级锁。高层工具的价值在于把容易写错的等待、唤醒和内存语义封装起来。

第二，给等待设置边界。无界等待是并发故障扩大的常见原因。获取结果可以使用超时 get，条件等待可以使用带超时的 await，外部调用可以设置超时，线程池关闭可以设置 awaitTermination 时间。超时不是为了掩盖错误，而是为了让调用方有机会记录现场、释放资源或降级。

第三，保持中断语义。凡是调用会抛出 InterruptedException 的方法，都要认真决定当前层是否消费中断。如果当前层无法完成取消处理，通常应恢复中断标志并返回，让上层继续处理。静默吞掉中断会让关闭流程卡住，也会让调用者误以为任务已经遵守取消协议。

第四，减少共享可变状态。不可变对象、局部变量、消息传递、任务参数复制、并发容器和清晰的所有权，通常比到处加锁更容易维护。锁和原子类解决的是共享状态冲突，但最好的并发设计往往是先减少共享，再保护必须共享的部分。

第五，保持诊断信息。线程名应能反映用途，线程池应暴露队列长度和拒绝次数，长时间等待应能定位锁或条件，后台任务异常应记录。并发问题通常无法只靠阅读一行报错定位，足够的运行时信息能显著降低排查成本。

## 常见误区
任务对象不应隐藏可变共享状态。这是实现多线程中最需要避免的误区之一。它之所以危险，是因为代码在低并发、本机运行或单次测试时可能完全正常，但在调度时机变化、任务数量上升或关闭流程触发时暴露问题。并发程序的正确性不能建立在“通常来得及”“一般不会同时发生”“这个线程应该先执行”这类假设上。

另一个误区是把时间当同步。sleep 一段时间、等待“差不多执行完”、依赖日志先后顺序、用本机测试速度推断生产时序，都属于时间同步误区。正确做法是使用明确同步条件，例如 join、Future、Latch、Barrier、锁、条件队列或 volatile 状态。

还要避免把工具名当保证。使用线程池不代表任务不会丢，使用锁不代表锁对象选对了，使用 volatile 不代表复合操作原子，使用 ThreadLocal 不代表不会泄漏，使用公平锁不代表请求一定按业务顺序完成。每个工具都只保证它规范中定义的部分，超出边界的需求仍要由设计补上。

最后是忽略失败路径。任务抛异常后计数是否递减，锁是否释放，许可是否归还，线程池是否还能接收任务，Future 是否有人读取异常，中断是否被恢复，资源是否关闭，这些都必须在代码中体现。并发系统的许多严重问题不是正常路径错误，而是异常路径没有释放等待者或资源。

## 排查思路
排查实现多线程相关问题时，可以先问四个问题。第一，线程现在在哪里：运行、阻塞在锁、等待条件、等待 I/O、等待任务结果，还是已经终止。第二，共享状态由谁保护：锁对象是否一致，volatile 是否覆盖读写两端，原子操作是否足够表达不变式。第三，生命周期是否闭合：启动后是否可停止，提交后是否可取消，关闭后是否拒绝新任务，异常后是否释放等待方。第四，容量是否合理：线程数、队列长度、许可数量、任务耗时和上游流量是否匹配。

工具上可以结合线程 dump、日志时间线、线程池指标、锁竞争指标和重复压力测试。线程 dump 能看到 BLOCKED、WAITING、TIMED_WAITING 以及调用栈；线程池指标能看到任务是否堆积；日志可以确认取消和关闭是否发生；压力测试可以放大竞态窗口。排查时不要只看最终异常，要寻找第一个失去同步关系或容量边界的位置。

如果问题偶发，优先怀疑数据竞争、遗漏通知、异常路径未释放、线程池饱和、死锁或外部阻塞。实现多线程相关代码应尽量让这些情况可被观察，例如在拒绝策略中记录任务类型，在等待超时后输出关键状态，在关闭时记录未执行任务数量，在捕获异常时保留中断状态或根因。

## 深入理解与推导
理解实现多线程时，可以从一个更具体的推导过程开始：先假设程序只有一个线程，所有语句按照调用栈顺序执行，变量只被当前线程读写；再引入第二个线程，让同一份状态可能被两个执行流同时观察；最后再加入等待、取消、超时、异常和关闭。每增加一个因素，就需要增加一条明确规则。没有规则的地方，运行结果就只能依赖调度时机，而调度时机从来不是稳定契约。

第一条规则是所有权规则。一个并发对象或同步工具必须有清楚的创建者、使用者和释放者。创建者负责初始化完整状态，使用者负责遵守访问协议，释放者负责关闭或清理资源。实现多线程相关代码如果没有所有权规则，就会出现同一个对象被多个模块以不同方式访问：有的路径加锁，有的路径不加锁；有的路径处理取消，有的路径吞掉异常；有的路径认为可以关闭线程池，有的路径仍在提交任务。这类问题通常不是 API 本身难用，而是边界没有被写进设计。

第二条规则是条件规则。等待必须等待一个可验证的条件，而不是等待一段时间。条件可以是线程结束、任务完成、计数归零、队列非空、许可可用、状态标志改变或锁变为可获取。条件还必须由同一套同步机制保护：用锁保护的条件要在持锁时检查，用 volatile 发布的状态要由 volatile 读观察，用 Future 表示的结果要通过 get 获取，用同步器表示的阶段要通过对应 await 方法等待。条件和等待工具不匹配时，最容易出现遗漏通知、虚假成功或永久阻塞。

第三条规则是失败规则。实现多线程的正常路径通常很短，但失败路径决定系统是否可靠。提交失败时是否记录，等待超时时是否取消，任务异常时是否释放锁和许可，线程被中断时是否恢复中断标志，关闭时是否处理尚未执行的任务，这些问题都必须有答案。并发代码的失败路径尤其重要，因为一个线程的异常经常不会直接出现在另一个线程的调用栈中；如果没有显式传播和记录，调用方只能看到“没有结果”或“系统变慢”。

第四条规则是容量规则。任何等待队列、线程池、同步器许可、任务拆分粒度和重试循环都有容量含义。容量太小会导致吞吐不足，容量太大会隐藏过载并放大延迟，容量无界会把压力转化为内存占用或不可控等待。实现多线程如果用于高并发路径，就不能只讨论功能正确，还要讨论任务数量、平均耗时、峰值流量、等待上限和降级策略。容量规则不是性能优化阶段才考虑的问题，而是并发设计的一部分。

第五条规则是可见性规则。一个线程做完动作，不代表其他线程自然知道。必须通过锁释放与获取、volatile 写与读、线程启动与结束、并发容器操作、同步器状态变化等方式建立可见性。实现多线程的很多误用都可以归结为缺少可见性规则：写线程以为自己已经发布状态，读线程却可能看到旧值；任务以为已经初始化对象，其他线程却可能看到部分构造；关闭线程以为已经发出停止请求，工作线程却从不检查中断或状态标志。

把这五条规则连起来，就能形成一个检查清单：对象是否安全发布，访问是否遵守同一协议，等待是否有明确条件，失败是否释放资源，关闭是否能唤醒等待者，容量是否有上限，结果是否通过同步关系传播。实现多线程的学习价值正在于把这些问题具体化。只要清单中的某一项回答含糊，就应回到代码重新收紧边界，而不是依赖注释或经验说明“正常情况下不会出问题”。

## 阅读与复习建议
复习实现多线程时，建议不要只看结论，而要自己沿着一条并发路径推演：哪个线程先创建对象，哪个线程发布引用，哪个线程开始执行，哪个线程可能等待，哪个线程负责唤醒，异常会落在哪里，关闭时如何让等待者退出。能把这条路径讲清楚，通常比记住零散 API 更有用。

还可以把本文中的代码片段改写成几个小实验：增加线程数量，故意制造超时，主动触发中断，抛出运行时异常，重复执行上千次观察结果。实验不是为了依赖偶然输出，而是为了暴露自己对边界条件的理解是否完整。只要实验结果和预期不一致，就应回到线程安全边界、内存可见性、等待条件和生命周期四个角度重新检查。

## 小结
实现多线程的重点不是 API 数量，而是把任务、执行载体、结果和生命周期拆开。

掌握实现多线程，最终要形成两种能力：一是能写出边界清晰的代码，知道正常路径、失败路径和关闭路径分别如何运行；二是能解释运行时现象，知道线程为什么等待、为什么没有看到新值、为什么任务没有执行、为什么关闭没有结束。Java 并发不是把代码拆到多个线程中那么简单，而是给执行、状态和生命周期建立可验证的规则。

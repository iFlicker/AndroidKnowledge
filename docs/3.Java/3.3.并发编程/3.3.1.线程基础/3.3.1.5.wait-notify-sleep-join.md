# 3.3.1.5 wait-notify-sleep-join

`wait`、`notify`、`notifyAll`、`sleep` 和 `join` 都可能让代码的执行表现为“一个线程停下来，之后再继续”，但它们解决的不是同一种问题。`wait/notify` 是基于对象监视器的条件协作机制；`sleep` 是当前线程主动进行限时暂停；`join` 是当前线程等待另一个线程终止。它们在调用资格、是否释放锁、线程状态、唤醒原因、中断行为和内存可见性方面都有明确差异。只凭“都会等待”把它们互换，往往会造成锁长期占用、遗漏通知、条件竞争、测试偶发失败或结果不可见。

本文保持通用 Java 视角，只讨论 Java 语言与标准类库中的语义。重点不是背诵方法签名，而是建立一套可推理的等待协议：线程究竟在等什么，等待前必须持有什么，等待时释放什么，谁负责改变条件，醒来后为什么还要检查条件，中断和超时如何退出，以及等待结束后共享数据为什么可见。

## 先建立整体区别

可以先用下表定位这些 API 的职责。表中的“释放锁”特指调用线程已经持有的 Java 监视器或显式锁，不是操作系统可能管理的其他资源。

| 方法 | 所属类 | 表达的等待原因 | 调用时是否要求持有特定监视器 | 等待时是否释放锁 | 常见 Java 线程状态 | 主要结束原因 |
| --- | --- | --- | --- | --- | --- | --- |
| `wait()` | `Object` | 某个受监视器保护的条件暂不成立 | 必须持有调用对象的监视器 | 释放调用对象的监视器；不释放其他锁 | `WAITING` | 通知、中断或虚假唤醒 |
| `wait(timeout)` | `Object` | 条件暂不成立，但等待有时间上限 | 必须持有调用对象的监视器 | 同上 | `TIMED_WAITING` | 通知、超时、中断或虚假唤醒 |
| `notify()` | `Object` | 告知一个等待者：受保护状态可能已变化 | 必须持有调用对象的监视器 | 方法本身不释放监视器 | 调用者通常仍为 `RUNNABLE` | 从等待集中选择一个线程 |
| `notifyAll()` | `Object` | 告知所有等待者：受保护状态可能已变化 | 必须持有调用对象的监视器 | 方法本身不释放监视器 | 调用者通常仍为 `RUNNABLE` | 使所有等待者进入重新竞争阶段 |
| `sleep(duration)` | `Thread` 静态方法 | 当前线程暂时不参与执行 | 不要求持有监视器 | 不释放任何已持有的监视器或显式锁 | `TIMED_WAITING` | 时间到或中断 |
| `join()` | `Thread` 实例方法 | 当前线程等待目标线程终止 | 调用方不需要持有业务监视器 | 不应把它当作释放业务锁的机制 | `WAITING` | 目标线程终止或当前线程被中断 |
| `join(timeout)` | `Thread` 实例方法 | 等待目标线程终止，但有时间上限 | 同上 | 同上 | `TIMED_WAITING` | 目标线程终止、超时或当前线程被中断 |

这里最重要的第一层结论是：`wait` 等条件，`sleep` 等时间，`join` 等线程结束；`notify/notifyAll` 只是通知条件可能变化，不负责替等待者判定条件，也不保证等待者立即执行。

第二层结论是：判断“会不会释放锁”不能笼统地看线程是否进入等待状态。`wait` 的设计目的之一就是在条件不成立时释放对应监视器，让其他线程有机会获得锁并改变条件；`sleep` 只暂停当前线程，不会释放它已经持有的锁。`join` 的公共语义是等待目标线程终止，调用者不能依赖某个 JDK 实现内部是否使用目标线程对象监视器，更不能据此推导自己持有的业务锁会被释放。

## 对象监视器、入口集合与等待集合

理解 `wait/notify` 前，需要先明确对象监视器的角色。Java 中每个对象都可以关联一个内置监视器。线程进入以某对象为锁的 `synchronized` 方法或代码块后，就持有该对象的监视器。在它退出同步区域之前，其他试图进入同一监视器保护区域的线程可能因竞争失败进入 `BLOCKED` 状态。

围绕一个监视器，可以从概念上区分两类等待者：

- 入口竞争者还没有获得监视器。它们试图进入 `synchronized` 区域，但锁正被其他线程持有，通常显示为 `BLOCKED`。
- 条件等待者曾经获得过监视器，在同步区域中发现条件不成立，于是调用该对象的 `wait`，释放该监视器并进入该对象的等待集合，通常显示为 `WAITING` 或 `TIMED_WAITING`。

这两类线程都没有继续执行业务代码，但原因不同。入口竞争者在等锁，条件等待者在等状态变化。`notify` 操作的是对象的等待集合，不是所有正在竞争该对象监视器的线程；释放监视器则会让入口竞争者和已经收到通知的等待者都有机会参与后续竞争。规范并不承诺一个可供业务依赖的公平顺序。

`wait` 必须由持有目标对象监视器的线程调用。例如调用 `lock.wait()` 时，当前线程必须位于 `synchronized (lock)` 或持有同一个 `lock` 监视器的同步方法语境中。否则会抛出 `IllegalMonitorStateException`。`notify` 和 `notifyAll` 具有同样要求。这个约束不是形式规定，而是为了让“检查条件、进入等待”和“修改条件、发出通知”都在同一个互斥协议内完成。

假设允许线程在锁外先检查条件，再单独进入等待，就会出现一个致命窗口：等待者看到条件为假，通知者随后把条件改为真并发出通知，而等待者此时还没有真正加入等待集合；通知不会被保存，等待者之后才开始等待，可能永远收不到下一次通知。这就是典型的遗漏通知。把条件检查和 `wait` 放在同一个同步区域中后，通知者无法在两者之间插入：等待者调用 `wait` 时会以一个受监视器协议约束的动作释放锁并进入等待；之后通知者才能获得锁、改变条件和通知。

因此，`wait/notify` 的正确性从来不只取决于两个方法调用，而取决于一整套协议：

1. 选择一个稳定且双方共享的监视器对象。
2. 定义由该监视器保护的条件谓词，例如“队列非空”或“组件已关闭”。
3. 等待者持锁检查谓词，不满足就在循环中调用同一对象的 `wait`。
4. 修改者持有同一监视器改变谓词涉及的状态。
5. 状态改变到可能使等待者继续的方向后，在同一监视器内调用 `notify` 或 `notifyAll`。
6. 等待者从 `wait` 返回并重新获得监视器后，再次检查谓词。

任何一步使用了不同的锁、在锁外访问条件、把通知当成状态本身，都会破坏这套协议。

## `wait` 的完整语义

线程调用 `wait` 时，已经位于目标对象的同步区域内。调用成功后，它会释放目标对象的监视器，暂停执行并加入该对象的等待集合。这里的“释放”有精确边界：只释放调用 `wait` 的那个对象监视器。如果线程还持有其他对象的监视器，其他锁不会随之释放。因此，在嵌套锁中调用 `wait` 仍可能让其他线程被外层或另一把锁阻塞，甚至形成死锁。

等待线程从 `wait` 返回前，必须重新获得同一个对象的监视器。这意味着收到通知不等于已经继续执行。通知线程调用 `notify` 后仍持有锁，它可以继续执行同步块中的代码；只有它退出同步区域或以其他正常方式释放监视器后，被通知线程才有机会和其他竞争者争锁。被通知线程竞争成功后，`wait` 才返回，随后代码仍在原同步区域中执行。

无参数 `wait()` 是无限期条件等待，在正常观察中对应 `Thread.State.WAITING`。`wait(long millis)` 和 `wait(long millis, int nanos)` 提供时间上限，通常对应 `TIMED_WAITING`。参数为零的毫秒等待仍表示无限期等待，而不是“完全不等待”。纳秒参数用于补充时间精度，但实际恢复时刻受系统时钟粒度、调度延迟和锁竞争影响，不能把它当作精确定时器。

`wait` 可能因为以下原因结束等待：

- 另一个持有同一对象监视器的线程调用了 `notify`，当前线程恰好被选中。
- 另一个线程调用了 `notifyAll`。
- 带超时的等待到达时间上限。
- 当前线程被其他线程中断，方法抛出 `InterruptedException`。
- 发生虚假唤醒，即没有对应通知、超时或业务条件成立，等待仍然返回。

即使确定收到了真实通知，也不能直接推导业务条件已经成立。通知和条件不是一回事。通知只表示共享状态可能发生了值得重新检查的变化。等待者收到通知后还要等待锁，在此期间，另一个线程可能先获得锁并消费掉资源；或者同一个等待集合中有多类条件，通知针对的是另一类等待者；又或者通知者只是触发了关闭状态，而不是生产了数据。

## 为什么条件等待必须使用 `while`

正确的条件等待形式是“在循环中检查谓词”，而不是“用 `if` 检查一次”：

```java
public final class OneSlotBuffer<T> {
    private T value;
    private boolean full;

    public synchronized void put(T newValue) throws InterruptedException {
        while (full) {
            wait();
        }
        value = newValue;
        full = true;
        notifyAll();
    }

    public synchronized T take() throws InterruptedException {
        while (!full) {
            wait();
        }
        T result = value;
        value = null;
        full = false;
        notifyAll();
        return result;
    }
}
```

这里的条件谓词分别是 `!full` 和 `full`。它们都由当前对象监视器保护，检查、状态修改和通知都在同步方法内完成。`put` 在槽位已满时等待，`take` 在槽位为空时等待；每次改变 `full` 后通知等待者。

如果把 `while` 改为 `if`，至少有三类问题。

第一类是虚假唤醒。Java 规范允许线程在没有明确通知的情况下从等待中返回。虚假唤醒不应被理解为“运行时经常无缘无故出错”，而应被理解为 API 契约要求调用者始终以条件为准。循环检查使代码不依赖唤醒原因：条件仍不满足就继续等待。

第二类是竞争消费。假设两个消费者都在等待一个空缓冲区。生产者放入一个元素并调用 `notifyAll`，两个消费者都从等待集合转向锁竞争。第一个消费者获得锁并取走元素，第二个消费者之后获得锁时缓冲区已经再次为空。若使用 `if`，第二个消费者会从 `wait` 后直接执行取值，破坏不变式；使用 `while` 时，它会发现 `!full` 仍然成立并再次等待。

第三类是多条件共用等待集合。一个对象只有一个与监视器关联的等待集合，但业务可能有“非空”“未满”“未关闭”等多个条件。`notifyAll` 会唤醒所有等待者，某个线程被唤醒不代表它等待的那一项条件成立。循环让每个线程根据自己的谓词作决定。

条件循环还有一个容易忽略的要求：谓词必须在持有同一把锁时读取和修改。只把 `wait` 放在循环里，但在锁外修改 `full`，仍然存在数据竞争和通知时序问题。循环解决的是返回后重新验证，监视器协议解决的是检查与等待之间的原子衔接以及共享状态的可见性，两者缺一不可。

## `notify` 与 `notifyAll` 的选择

`notify()` 从同一对象等待集合中选择一个线程，使其具备重新竞争监视器的资格。选择哪个线程由实现决定，调用者不能指定，也不能依赖先等待者先被选择。`notifyAll()` 则让等待集合中的所有线程都进入后续竞争阶段。二者都不会立即交出锁，也都不等价于线程已经完成条件动作。

当以下条件能够被严格证明时，`notify` 可能是正确且更节制的选择：

- 等待集合中所有线程等待的是同一个逻辑谓词。
- 一次状态变化至多允许一个等待者继续。
- 任意一个等待者被选择都能消费这次状态变化。
- 不存在关闭、取消或其他条件需要同时唤醒不同类别的等待者。

只要对象等待集合中混有不同条件，`notify` 就可能选中一个当前无法继续的线程。该线程拿到锁后重新检查条件，又回去等待；真正能够继续的线程却没有收到通知。如果之后没有新的状态变化和通知，系统可能停滞。前面的单槽缓冲区同时包含生产者和消费者两类等待者：槽位由满变空时应唤醒生产者，由空变满时应唤醒消费者，但 `notify` 无法指定类别，因此使用 `notifyAll` 更容易保证活性。

`notifyAll` 的代价是可能造成“惊群”：多个线程醒来并争夺同一把锁，最终只有少数线程满足条件，其余线程再次等待。线程很多、状态变化频繁时，这会增加调度和锁竞争成本。然而正确性优先于对唤醒数量的微观优化。在原生监视器无法拆分等待集合而又存在多个条件时，通常先选择 `notifyAll`。如果性能分析确认无效唤醒已经成为瓶颈，更合适的改进往往是改用 `ReentrantLock` 的多个 `Condition`，把“非空”和“未满”等条件分到不同等待队列，而不是贸然把 `notifyAll` 改成 `notify`。

通知的时机也要和状态修改绑定。通常应先在锁内完成状态修改，再通知：

```java
synchronized (lock) {
    ready = true;
    lock.notifyAll();
}
```

不能把通知当作状态存储。若通知发生时没有线程等待，这次通知不会像消息一样排队保存；以后开始等待的线程不会“补收”旧通知。正确协议之所以不怕先通知后等待，是因为等待者先检查持久化的条件状态：如果 `ready` 已经为真，它根本不会调用 `wait`。所以防止遗漏通知的核心不是记住每一次通知，而是用受锁保护的状态表达事实，用通知促使等待者重新检查事实。

## `sleep`：只等待时间，不参与条件协议

`Thread.sleep` 是静态方法，作用对象始终是当前正在执行的线程。即使通过某个线程变量写成 `other.sleep(...)`，被暂停的仍然是当前线程；这种写法会制造错误暗示，应始终使用 `Thread.sleep(...)`。

`sleep` 让当前线程在指定时间内进入 `TIMED_WAITING`，时间到后重新获得被调度的资格。它不能保证恰好在指定时刻继续：运行时可能晚于目标时间恢复，因为线程还要等待调度，恢复后也可能立即遇到锁竞争。`sleep` 提供的是“不早于某个等待时长主动继续”的近似时间约束，不是实时调度保证。

最关键的锁语义是：`sleep` 不释放当前线程已经持有的任何监视器，也不自动释放 `ReentrantLock` 等显式锁。下面的代码会让另一个需要 `lock` 的线程至少等待睡眠时间加上其余临界区时间：

```java
public void pauseWhileHoldingLock(Object lock) throws InterruptedException {
    synchronized (lock) {
        Thread.sleep(1_000);
    }
}
```

如果目的只是延迟下一次操作，应尽量把 `sleep` 放到临界区外。若目的是等待另一个线程改变受锁状态，使用 `sleep` 更是错误模型：持锁睡眠会阻止修改者获得锁；锁外睡眠则没有条件通知和可靠可见性，只能反复猜测状态是否已经变化。

`sleep` 合理的典型用途包括演示线程调度、实现简单的测试延迟、在明确设计的重试策略中进行退避，或在无需被其他状态提前唤醒的低精度周期循环中暂停。即使如此，生产级调度通常更适合 `ScheduledExecutorService`，重试通常需要最大次数、总体截止时间、抖动和错误分类。`sleep` 不适合等待任务完成、等待队列出现元素、等待某字段变为特定值，也不适合代替限流器和调度器。

用 `sleep` 修复竞态通常只会缩小或移动竞态窗口。例如主线程启动工作线程后睡眠一秒再读取结果，不能保证工作线程在一秒内完成；机器负载、运行时编译、外部资源和调度都会改变耗时。即使工作线程大多数时候完成，若结果字段没有正确同步，时间经过本身也不建立 happens-before。需要等待线程终止时应使用 `join`，需要等待任务结果时应使用 `Future.get`，需要等待条件时应使用条件队列或合适的同步器。

## `join`：等待目标线程终止

在 `current.join()` 这一表达中，容易混淆两个线程：调用 `join` 方法的是当前线程，被等待的是 `current` 所引用的目标线程。更直观地写：

```java
public final class JoinExample {
    private int result;

    public void run() throws InterruptedException {
        Thread worker = new Thread(() -> result = compute(), "worker");
        worker.start();

        worker.join();
        System.out.println(result);
    }

    private int compute() {
        return 42;
    }
}
```

调用 `worker.join()` 后，执行 `run()` 的线程等待 `worker` 终止。目标线程的 `run` 正常返回或因未捕获异常结束，都会进入终止状态并使等待者具备继续执行的条件。无超时 `join` 常使调用线程进入 `WAITING`，带超时 `join` 常使其进入 `TIMED_WAITING`。

`join` 等待的是线程生命周期，不是业务成功。目标线程即使因未捕获异常终止，`join` 也会正常完成，不会把目标线程的异常自动重新抛给等待者。若调用方需要结构化获取返回值和异常，`Callable`、`Future` 或 `CompletableFuture` 通常比裸 `Thread` 更适合。裸线程可以通过未捕获异常处理器记录失败，但这仍不同于将异常作为结果传播给等待方。

对一个尚未启动、处于 `NEW` 状态的线程调用 `join` 通常会立即返回，因为该线程并不存活。对已经 `TERMINATED` 的线程调用也会立即返回。对当前线程自身调用无超时 `join` 会等待自身终止，而当前线程又必须从 `join` 返回才能继续走向终止，因此会形成无法自行满足的等待，除非被其他线程中断。代码设计应避免自连接。

带超时 `join` 只保证等待最多受到给定时间上限约束，不保证返回时目标线程已经结束：

```java
worker.join(2_000);
if (worker.isAlive()) {
    worker.interrupt();
    throw new IllegalStateException("worker did not stop in time");
}
```

超时后必须检查 `isAlive()` 或其他明确完成状态。方法返回可能因为目标已结束，也可能仅仅因为时间到。并且 `interrupt()` 仍是协作式请求，不能保证目标线程立即终止；目标任务需要响应中断。真实关闭流程还应决定中断后是否再次等待、等待多久、最终如何报告未终止状态。

另一个重要边界是不要持有目标线程完成所需的锁再调用 `join`。例如主线程持有 `lock` 并等待工作线程结束，而工作线程结束前必须进入 `synchronized (lock)`，双方就形成等待环。`join` 不会替调用者释放任意业务监视器。任何阻塞等待都应检查“被等待方是否需要我当前持有的资源才能完成”。

## 线程状态不是唤醒保证

这些 API 和 Java 六种线程状态有典型对应关系，但状态只是某一时刻的诊断快照。

- 无超时 `wait` 和 `join` 通常表现为 `WAITING`。
- 带超时 `wait`、带超时 `join` 和 `sleep` 通常表现为 `TIMED_WAITING`。
- 等待者收到 `notify` 后，在重新获得监视器前可能表现为 `BLOCKED`，因为它此时正在争用监视器。
- 调用 `notify` 的线程不会因为通知动作自动进入等待；它仍然持锁并继续执行。
- 时间到、被通知或目标线程结束，只让线程具备继续推进的条件，不保证它立刻获得 CPU 或监视器。

不要通过轮询 `getState()` 决定何时通知，也不要把观察到 `WAITING` 当作业务握手完成。状态读取与后续动作之间存在竞态，线程可能已经变化。测试若需要确认工作线程确实到达某个阶段，应使用 `CountDownLatch`、`Phaser`、阻塞队列或专门的测试钩子建立明确同步，而不是 `sleep` 一段时间或等待状态标签。

## 中断：取消的是当前等待

`wait`、`sleep` 和 `join` 都声明抛出 `InterruptedException`。当线程在这些方法中等待时，另一个线程对它调用 `interrupt()`，等待会结束并抛出异常。抛出时线程的中断状态会被清除，因此捕获异常的一层必须明确决定：当前层是否已经完整处理取消请求。

若方法可以继续向上声明 `throws InterruptedException`，通常直接传播最清晰。若接口不允许抛出受检异常，而当前层也不拥有取消决策，常见做法是恢复中断标志并尽快退出：

```java
public void run() {
    try {
        performWork();
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return;
    } finally {
        cleanup();
    }
}
```

静默捕获后继续普通工作会吞掉取消信号。尤其是在等待循环中，下面这种处理会让上层以为已经请求停止，但任务继续等待：

```java
while (!ready) {
    try {
        lock.wait();
    } catch (InterruptedException ignored) {
        // 错误：取消请求在这里丢失
    }
}
```

中断也不等于业务条件成立。线程从 `wait` 因中断退出时，方法以异常方式结束，并且在抛出异常前会重新获得目标对象监视器，因此 catch 或 finally 所在的同步区域仍可按锁语义清理状态。线程从 `sleep` 或 `join` 因中断退出，只表示当前线程不再继续这次睡眠或终止等待；被 `join` 的目标线程不会因此自动被中断。若等待者决定取消目标任务，必须另外向目标线程发送中断或使用任务框架的取消机制。

“捕获后恢复中断”也不是无条件模板。如果当前层就是任务的最高层，并且已经把中断转换成明确的终止、完成所有清理且不会再把控制权交给依赖中断标志的上层，那么它可以消费中断并结束。关键不在于每个 catch 都机械恢复，而在于取消语义不能悄悄消失。

## 超时等待应使用剩余时间

带超时的 `wait` 和 `join` 给等待增加边界，但循环等待时不能每轮都重新使用完整超时时长。虚假唤醒、无关通知或竞争失败都可能使线程多次返回等待；若每次再等待完整的五秒，总耗时可能远超五秒。

更稳妥的方式是先计算截止时间，每轮根据剩余预算等待：

```java
public synchronized boolean awaitReady(long timeout, TimeUnit unit)
        throws InterruptedException {
    long remaining = unit.toNanos(timeout);
    long deadline = System.nanoTime() + remaining;

    while (!ready) {
        if (remaining <= 0L) {
            return false;
        }

        long millis = TimeUnit.NANOSECONDS.toMillis(remaining);
        int nanos = (int) (remaining
                - TimeUnit.MILLISECONDS.toNanos(millis));
        wait(millis, nanos);
        remaining = deadline - System.nanoTime();
    }
    return true;
}
```

这里使用 `System.nanoTime()` 计算经过时间，因为它适合测量相对时长，不受系统墙上时钟校准直接影响。`currentTimeMillis()` 更适合表示日历时间，不适合构造可靠的超时预算。示例仍需要处理极端大的超时导致加法溢出的工程边界；在普通受控参数范围内，它展示了核心思想：方法返回成功只取决于 `ready`，而不是取决于某次 `wait` 返回。

注意 `wait(long millis, int nanos)` 对参数有范围要求，而且当毫秒与纳秒都为零时表示无限等待。上面的代码在剩余时间大于零但不足一毫秒时，会传入 `0` 毫秒和正纳秒，因此仍是有限等待。调用方还应定义零超时和负超时的业务语义，通常将其视为立即检查而不等待。

超时不是正确同步的替代品。它只能限制故障扩散，并为调用方提供取消、降级和诊断机会。等待超时后需要返回明确结果或抛出明确异常，不能继续假装条件已经满足。对于 `join(timeout)`，超时后检查目标线程；对于条件等待，超时后在锁内最后检查一次谓词；对于多阶段操作，应让各阶段共享同一总体截止时间，而不是每一层都重新获得完整预算。

## happens-before 与数据可见性

等待和唤醒不只是调度问题，还涉及 Java 内存模型。正确使用监视器时，一个线程在退出 `synchronized` 区域、释放某监视器之前的写入，happens-before 另一个线程随后成功获得同一监视器后的动作。`wait` 会释放监视器，返回前又会重新获得监视器；通知者通常在同一监视器内修改状态并最终释放锁。因此等待者重新获得锁后，能够看到通知者在释放同一锁之前完成的写入。

需要准确表述的是：`notify` 调用本身不是一条可以脱离锁使用的“刷新内存”指令。可见性来自同一监视器的释放与后续获取，条件协议又要求状态读写在该监视器下完成。如果通知者修改普通字段时没有持有保护该字段的锁，或者等待者醒来后在锁外读取另一份未同步数据，就不能仅凭“发生过 notify”推出所有结果可见。

`sleep` 不建立任何专门的跨线程 happens-before 关系。一个线程睡了足够久，不代表它醒来后就必然看到另一个线程对普通字段的写入。时间流逝不是内存同步。若共享标志需要跨线程可见，应使用 `volatile`、锁、原子变量或并发工具，而不是定期睡眠后读取普通字段。

`join` 则具有明确的线程终止内存语义：目标线程中的所有动作 happens-before 另一个线程成功检测到它已经终止，例如从对该线程的 `join` 调用中返回。前面的 `JoinExample` 中，工作线程写入 `result`，调用线程从 `worker.join()` 成功返回后读取 `result`，这条读取受终止与连接关系保护，不要求 `result` 额外声明为 `volatile`。

对于带超时的 `join`，只有确认目标线程已经终止，才能依赖这条终止关系。若 `join(timeout)` 因超时返回且目标仍存活，调用方不能把这次返回当作结果发布边界。类似地，调用线程自身被中断导致 `join` 抛出异常时，也没有成功等待到目标终止。

还应区分可见性与业务原子性。监视器可以让锁内写入对后续持同一锁的线程可见，也可以让临界区互斥，但前提是所有相关访问都遵循同一协议。`join` 可以发布目标线程结束前的动作，却不能修复目标线程与第三方线程之间原本存在的数据竞争。内存语义只保证规范定义的边，不能为整个程序自动补齐同步。

## 一个支持关闭的条件队列案例

下面的有界队列示例把条件循环、两类条件、中断、关闭和超时放在一起。生产者等待“队列未满且未关闭”，消费者等待“队列非空或已关闭”。由于原生监视器只有一个等待集合，状态变化使用 `notifyAll`。

```java
public final class MonitorBoundedQueue<E> {
    private final Object[] elements;
    private int head;
    private int tail;
    private int size;
    private boolean closed;

    public MonitorBoundedQueue(int capacity) {
        if (capacity <= 0) {
            throw new IllegalArgumentException("capacity must be positive");
        }
        elements = new Object[capacity];
    }

    public synchronized void put(E element) throws InterruptedException {
        if (element == null) {
            throw new NullPointerException("element");
        }
        while (size == elements.length && !closed) {
            wait();
        }
        if (closed) {
            throw new IllegalStateException("queue is closed");
        }
        elements[tail] = element;
        tail = (tail + 1) % elements.length;
        size++;
        notifyAll();
    }

    public synchronized E take() throws InterruptedException {
        while (size == 0 && !closed) {
            wait();
        }
        if (size == 0) {
            return null;
        }

        @SuppressWarnings("unchecked")
        E element = (E) elements[head];
        elements[head] = null;
        head = (head + 1) % elements.length;
        size--;
        notifyAll();
        return element;
    }

    public synchronized void close() {
        if (!closed) {
            closed = true;
            notifyAll();
        }
    }
}
```

这个案例中的 `close` 必须通知所有等待者。若只修改 `closed` 而不通知，已经在 `wait` 中的生产者和消费者不会仅因字段变化自动醒来。若只 `notify` 一个线程，可能还有其他等待者永久停留，特别是在关闭后不会再有新的入队或出队动作触发通知。

`take` 在关闭后仍允许取走已有元素，只有“已关闭且为空”才返回 `null`。这说明条件谓词必须来自业务语义，不能简单写成“没数据就等”。`put` 则在关闭后拒绝新元素。所有字段都由同一个对象监视器保护，不需要再把 `closed` 声明为 `volatile`；若调用方提供不加锁的 `isClosed` 方法，则需要让它也同步或改用其他一致的发布方案。

这个实现用于解释监视器协议，而不是建议在实际项目中重复实现队列。标准库的 `ArrayBlockingQueue`、`LinkedBlockingQueue` 等已经处理了容量、条件队列、中断和并发细节，通常应直接使用。自行实现的价值主要在于理解：高层阻塞队列内部也必须解决条件、唤醒、互斥和可见性问题。

## 常见错误及其根因

### 在未持有监视器时调用 `wait` 或通知

错误代码常见形式是直接调用 `lock.wait()`，却没有 `synchronized (lock)`。结果不是“等待不生效”，而是抛出 `IllegalMonitorStateException`。即使当前线程持有另一把锁也无效；必须持有调用对象本身的监视器。

### 等待和通知使用不同对象

等待者在 `conditionLock` 上调用 `wait`，通知者却在 `this` 上调用 `notifyAll`，两个对象拥有不同监视器和等待集合，通知无法到达等待者。用于保护谓词、执行等待和发送通知的对象必须一致。

### 在锁外检查条件

先在锁外执行 `if (!ready)`，再进入同步块调用 `wait`，会留下检查与等待之间的遗漏通知窗口。检查和等待必须在同一个同步区域中，并通过 `while` 连续表达。

### 用 `if` 代替 `while`

这会忽略虚假唤醒、多等待者竞争和多条件共用等待集合。正确判断依据始终是条件谓词，不是“我被唤醒了”。

### 修改状态后忘记通知

共享字段即使在锁内更新，也不会自动把条件等待者从等待集合移出。状态从不满足变为可能满足时，需要发出匹配的通知。反过来，只通知而不修改持久状态也不可靠，因为通知不会保存。

### 认为 `notify` 会立即切换线程

通知线程仍持有监视器。若它在通知后继续执行长操作，被通知线程只能等待。应尽量让通知后的锁内工作保持必要且短小，但也不能为了“早点释放”把必须原子完成的状态更新移到锁外。

### 在持锁状态下 `sleep`

这不会让其他线程获得同一锁，反而延长临界区。若另一个线程必须拿到该锁才能产生当前线程等待的结果，程序只会平白停顿，严重时暴露锁顺序问题。

### 用 `sleep` 猜测线程完成

固定等待时间既可能过短导致读取未完成结果，也可能过长拖慢测试和程序。它还不提供结果可见性。等待裸线程结束使用 `join`，等待任务使用 `Future`，测试阶段协作使用门闩或屏障。

### 认为 `join(timeout)` 返回就是完成

限时连接可能只是超时。返回后要检查目标线程是否仍存活，并决定取消和后续等待策略。

### 捕获并吞掉 `InterruptedException`

这会截断取消传播，使关闭线程以为请求已经送达，但工作线程继续运行。当前层应传播、恢复中断后退出，或明确消费并完成整个取消流程。

### 持有被等待方所需资源再阻塞

持锁调用 `join`、持锁等待 `Future`、持锁等待另一个条件，都可能让被等待方无法获得完成所需的锁。分析阻塞点时必须画出资源依赖，而不能只看调用方法是否合法。

### 把线程状态当作同步协议

等待线程显示 `WAITING` 只是瞬时观察，不保证下一步仍在等待。用 `getState` 决定通知时机或读取结果会制造检查后变化竞态。业务同步应由持久状态与同步工具表达。

## 工具选择边界

原生 `wait/notify` 适合教学、维护遗留监视器代码，或实现边界很小、条件很少且对象所有权清晰的同步组件。它的优势是语言内建、与 `synchronized` 直接配合；局限是每个监视器只有一个等待集合，通知选择不可控，超时和取消协议需要手工组织，复杂后很难审查。

需要一把锁对应多个独立条件队列时，`ReentrantLock` 与 `Condition` 更合适。它能分别建立 `notEmpty`、`notFull` 等条件，定向 `signal`，减少无效唤醒，并提供可中断或限时锁获取。但它仍要求条件循环，并且必须在 `finally` 中释放锁；换工具不会取消条件谓词和中断处理这些基本规则。

生产者消费者、工作队列和容量控制通常直接选择 `BlockingQueue`。等待一次性事件完成可选择 `CountDownLatch`；等待一组线程在阶段点会合可选择 `CyclicBarrier` 或 `Phaser`；等待异步计算结果及异常可选择 `Future` 或 `CompletableFuture`；需要定时或周期执行应选择 `ScheduledExecutorService`。这些工具把等待条件和内存语义封装成更清晰的领域操作，通常比手写监视器协议更容易验证。

`join` 适合少量裸线程之间明确的终止依赖，例如启动两个专用线程后等待它们结束。任务由执行器管理时，不应为了调用 `join` 去获取线程池内部工作线程；应通过 `Future`、执行器终止等待或结构化的任务组合等待。`join` 只能表达“线程结束”，不能直接表达“任务成功并返回某值”。

`sleep` 的边界最窄：它是时间暂停，不是协作原语。需要可被新任务提前唤醒的调度、精确的周期管理、条件变化通知、结果等待或容量控制时，都应选择对应工具。判断是否误用 `sleep` 的一个实用问题是：“如果另一线程现在已经完成目标动作，我是否仍被迫睡满这段时间？”如果答案是肯定的，通常说明应该使用能响应事件的等待机制。

## 排查等待问题

排查线程卡住时，先获取多次线程 dump，而不是只看一次状态。对 `wait` 相关线程，确认它等待在哪个对象、条件字段由谁修改、是否存在通知者、通知者是否还能获得同一监视器。对 `BLOCKED` 线程，查找持有监视器的线程及其栈；被通知线程可能只是卡在重新获取锁，而不是没有收到通知。

对 `join` 相关线程，沿目标线程继续分析。目标线程是否仍运行，是否阻塞在调用者持有的锁上，是否陷入不可中断操作，是否因循环不响应中断而无法结束。对 `sleep` 线程，判断它是在合理退避，还是用固定延迟掩盖条件同步缺失。大量业务线程重复睡眠轮询普通字段，通常意味着模型需要重构。

日志应记录状态变化而非只记录通知动作。例如记录“队列由空变为非空”“组件进入关闭状态”“工作线程开始终止”，比单独记录“调用 notifyAll”更有诊断价值，因为真正决定等待者行为的是条件。等待超时时，应输出剩余关键状态、目标线程名和必要的资源标识，但不要在持锁区域执行耗时日志或调用不可控外部代码。

测试并发协议时，可以用门闩精确控制线程到达点，重复运行压力场景，并覆盖中断、超时、关闭、多个等待者竞争和异常退出。不能以“连续运行几次没有挂住”证明不存在遗漏通知；应从监视器、谓词和状态转换上证明每一种等待都有使其退出的路径。

## 结论

`wait/notify`、`sleep` 和 `join` 的共同点只有“当前线程可能暂时不继续执行”，它们的同步含义完全不同：

- `wait` 是在持有对象监视器时发现条件不成立，释放该监视器并进入条件等待；返回前必须重新获得监视器。
- `notify` 和 `notifyAll` 只表示条件可能变化，不保存通知、不立即交出锁，也不替等待者判断条件。
- 条件等待必须使用受同一监视器保护的持久状态，并在 `while` 中反复检查，以应对虚假唤醒、竞争消费和多条件通知。
- `sleep` 只等待时间，不释放已持有的锁，不提供跨线程可见性，也不能证明其他线程已经完成。
- `join` 等待目标线程终止，并在成功等待到终止后提供明确的 happens-before 关系；带超时返回时仍要确认目标是否存活。
- `wait`、`sleep` 和 `join` 都可以被中断，处理中断时必须保留取消语义；超时等待则要基于剩余时间并在返回后重新验证条件。

选择工具时，应先说清等待对象：等条件、等时间、等线程终止、等任务结果、等队列元素还是等阶段完成。等待原因一旦明确，API 选择、锁释放语义、退出路径和内存可见性才能形成闭环。

# 3.4.1 BIO

BIO 是 Java 传统 I/O 编程模型中最容易理解、也最容易被误用的一部分。这里的 BIO 通常指 blocking I/O，即调用线程在执行读、写、连接、接收连接等操作时，如果数据或条件暂时不满足，就会被阻塞住，直到操作完成、超时、发生异常或线程被中断。它不是某一个单独类的名字，而是一组以 `java.io` 流、`java.net.Socket`、`java.net.ServerSocket` 为代表的编程方式。BIO 的价值在于模型直接：一个调用对应一个清晰的等待点，一个连接通常交给一个线程处理，代码流程接近同步业务逻辑；它的局限也来自同一点：等待期间线程资源被占用，并发连接数上升后，线程数量、栈内存、上下文切换和调度开销都会迅速变成系统瓶颈。

理解 BIO 不能只停留在“阻塞所以落后”的标签上。大量命令行工具、内部管理端、低并发服务、批处理任务、文件处理流程以及测试辅助程序仍然适合使用 BIO。它的可读性、调试便利性、异常传播方式和资源生命周期都很直观。如果连接数量有限、每个连接的处理时间短、服务端部署资源充足，BIO 的工程复杂度往往低于事件驱动模型。相反，如果系统面对大量长连接、慢客户端、突发连接风暴或需要在少量线程中管理大量 I/O 事件，BIO 就需要非常谨慎，甚至应换用 NIO 或更高层网络框架。

## 阻塞 I/O 的基本语义

阻塞的本质是调用线程进入等待状态。以 `InputStream.read(byte[])` 为例，调用者希望从输入端读取若干字节。如果缓冲区里已有数据，方法可以立即返回；如果暂时没有数据，但流还没有结束，线程会等待；如果对端关闭输出，方法可能返回 `-1`；如果底层连接异常，方法抛出 `IOException`。类似地，`ServerSocket.accept()` 会等待新的连接到来，`Socket.connect()` 会等待连接建立或失败，`OutputStream.write(byte[])` 在发送缓冲区可用空间不足时也可能阻塞。

这种语义看似简单，但有几个边界必须清楚。第一，阻塞不等于一定占满 CPU。线程被阻塞时通常不会忙等消耗 CPU，而是由操作系统挂起，直到内核发现 I/O 条件变化再唤醒。第二，阻塞不等于无成本。每个 Java 线程都有栈空间和调度成本，线程越多，内存占用、上下文切换、锁竞争和故障排查难度越高。第三，阻塞不保证一次读写完成完整消息。TCP 是字节流，`read` 只表示当前读取到一段字节，不理解应用层消息边界；文件流也可能因为缓冲策略、文件长度和调用参数而返回少于请求长度的字节数。第四，阻塞点需要配合超时或关闭机制，否则网络异常、对端不响应、协议设计缺陷都可能让线程长期挂起。

BIO 中最常见的抽象是流。`InputStream` 和 `OutputStream` 处理字节，适合二进制协议、文件复制、压缩数据和网络报文；`Reader` 和 `Writer` 处理字符，适合文本协议和字符文件。字节与字符之间的桥梁是 `InputStreamReader`、`OutputStreamWriter`，它们必须指定字符集或明确接受平台默认字符集的风险。流是单向的，读写通常分别由不同对象负责；包装流通过装饰器模式增加缓冲、数据类型读写、对象序列化、打印格式化等能力。BIO 的 API 设计鼓励把底层资源包装成更适合业务语义的对象，但也要求开发者清楚关闭最外层流时会连带关闭底层资源。

## 典型线程模型

最基础的 BIO 服务端模型是“主线程 accept，工作线程处理连接”。主线程持有 `ServerSocket` 并循环调用 `accept()`，每接收到一个连接，就把 `Socket` 交给一个新线程或线程池。工作线程在自己的同步流程里读取请求、处理业务、写回响应，然后关闭连接或继续处理同一连接上的下一次请求。

```java
try (ServerSocket server = new ServerSocket(9000)) {
    while (true) {
        Socket socket = server.accept();
        Thread worker = new Thread(() -> handle(socket), "bio-worker");
        worker.start();
    }
}

static void handle(Socket socket) {
    try (Socket s = socket;
         BufferedReader reader = new BufferedReader(
                 new InputStreamReader(s.getInputStream(), StandardCharsets.UTF_8));
         BufferedWriter writer = new BufferedWriter(
                 new OutputStreamWriter(s.getOutputStream(), StandardCharsets.UTF_8))) {
        String line = reader.readLine();
        if (line != null) {
            writer.write("echo: " + line);
            writer.newLine();
            writer.flush();
        }
    } catch (IOException e) {
        // 真实服务通常记录连接标识、远端地址和异常类型。
    }
}
```

这个例子展示的是 BIO 的直观性，也展示了它的隐患。`accept()` 阻塞等待连接；`readLine()` 阻塞等待换行、流结束或异常；`write` 和 `flush` 也可能因为对端接收慢而阻塞。如果为每个连接都创建原生线程，短时间内大量连接会导致线程数量失控。使用线程池可以限制并发工作线程数量，但如果工作线程全部被慢连接占住，新的连接虽然能被 `accept` 接收，却可能在任务队列中排队，甚至被拒绝。线程池不是把 BIO 变成高并发模型的魔法，它只是把资源消耗变得可控。

更稳妥的 BIO 服务端通常使用有界线程池、有界队列、合理的拒绝策略和连接超时。线程池大小应根据业务处理时间、阻塞等待比例、机器 CPU 和内存确定，而不是简单设置成“越大越好”。对于纯 I/O 等待占比较高的场景，线程数可以大于 CPU 核心数；对于处理逻辑含大量 CPU 计算的场景，线程数过高只会增加调度开销。队列容量过大也会掩盖过载，让请求在内存里长时间排队，最终超时或造成雪崩。更好的策略是在系统接近承载上限时尽早拒绝、快速失败或进行限流。

## 流、缓冲与消息边界

BIO 编程中，缓冲是性能和语义之间的关键层。直接对 `SocketInputStream` 或 `FileInputStream` 进行小块读写会频繁进入内核，系统调用成本明显；用 `BufferedInputStream`、`BufferedOutputStream`、`BufferedReader`、`BufferedWriter` 包装后，应用层可以用较小的调用单位表达逻辑，底层按较大的块读写，提高吞吐。缓冲并不改变流的顺序语义，也不自动识别消息，它只是在用户态暂存数据。

文本协议常用 `BufferedReader.readLine()` 读取一行。它适合以换行作为消息边界的简单协议，但要注意三个问题。第一，返回的字符串不包含行分隔符，应用层如果需要保留原始换行必须自行处理。第二，如果对端一直不发送换行也不关闭连接，`readLine` 会一直等待，除非设置了读超时或连接被关闭。第三，文本读取需要字符集解码，如果发送端和接收端字符集不一致，中文、符号和多字节字符都可能出现乱码或解码异常。对于二进制协议，通常不能用字符流读取，应使用字节流并设计明确的帧格式。

二进制消息常见做法有三类：固定长度、分隔符、长度前缀。固定长度简单但浪费空间，不适合变长内容；分隔符便于调试，但需要处理转义和内容中出现分隔符的问题；长度前缀最常用，先读取固定字节数的长度字段，再读取指定长度的正文。BIO 下读取长度前缀不能假设一次 `read` 就拿到完整长度字段，必须循环读取直到满足期望长度或遇到 EOF。

```java
static void readFully(InputStream in, byte[] buffer, int offset, int length) throws IOException {
    int total = 0;
    while (total < length) {
        int n = in.read(buffer, offset + total, length - total);
        if (n == -1) {
            throw new EOFException("stream ended before reading expected bytes");
        }
        total += n;
    }
}
```

这个辅助方法体现了 BIO 的一个核心习惯：把底层流的“尽力返回”包装成上层协议需要的“完整读取”。`DataInputStream.readFully` 已经提供了类似能力，实际项目可以直接使用。写入侧也要理解 `flush` 的含义。对于带缓冲的输出流，`write` 可能只是写入用户态缓冲区，`flush` 才推动数据进入底层流；对于 `PrintWriter`，默认不会在每次输出后抛出异常，很多错误需要通过 `checkError` 才能发现，因此在严肃网络协议中更推荐显式使用 `BufferedWriter`、`OutputStream` 或 `DataOutputStream` 并处理 `IOException`。

## InputStream 与 OutputStream

`InputStream` 和 `OutputStream` 是 Java 字节 I/O 的根抽象。所谓字节 I/O，是指 API 只承诺处理一串 8 位字节，不关心这些字节代表文本、图片、压缩包、序列化对象还是自定义协议报文。字节流适合所有二进制内容，也适合在字符集尚未确定之前读取原始数据。网络协议、文件复制、摘要计算、压缩解压、加密解密等场景，都应优先从字节流理解数据路径，再在必要位置转换成更高层对象。

`InputStream.read()` 每次读取一个字节，返回 `0` 到 `255` 之间的整数，遇到流结束返回 `-1`。`read(byte[])` 和 `read(byte[], int, int)` 允许一次读取多个字节，但返回值只表示本次实际读取数量，不保证填满数组。`OutputStream.write(int)` 写出低 8 位，`write(byte[])` 和 `write(byte[], int, int)` 写出字节数组的一段内容。无论读还是写，方法名都很朴素，却不能把它们理解为“读取一个完整对象”或“写出一条完整消息”。流只负责按顺序搬运字节，消息完整性必须由调用方或包装层定义。

字节流体系中的包装类体现了装饰器思想。`BufferedInputStream` 和 `BufferedOutputStream` 提供缓冲；`DataInputStream` 和 `DataOutputStream` 按固定格式读写 Java 基本类型；`ObjectInputStream` 和 `ObjectOutputStream` 处理对象序列化；`DigestInputStream` 可以在读取时更新摘要；`CheckedInputStream` 可以在读取时计算校验值。包装越多，代码越接近业务语义，但也越需要明确每层的职责。例如在长度前缀协议中，底层可以是 socket 输入流，中间包一层缓冲，外层用 `DataInputStream` 读取 `int` 长度和正文。此时应用层仍然要检查长度上限，不能因为使用了 `readInt` 就默认输入可信。

输出流的一个细节是“写入成功”的含义。调用 `write` 返回通常只说明数据已经被交给当前流对象，若中间有缓冲，数据可能还没有进入底层文件描述符；即使进入操作系统发送缓冲区，也不代表对端应用已经读取并处理。`flush` 只能推动用户态缓冲向下游提交，不能提供端到端确认。文件场景中，`flush` 也不等于数据已经稳定落盘，底层文件系统和磁盘缓存仍可能延迟写入。需要强落盘语义时，应理解 `FileDescriptor.sync()`、`FileChannel.force()` 等更底层能力的成本和适用范围，而不是把普通 `flush` 当成事务提交。

另一个常见边界是零长度读写。`read(byte[], off, 0)` 可以直接返回 `0`，这不代表 EOF；`read` 返回 `0` 在某些流实现或零长度请求中可能出现，循环读取时要避免把 `0` 当成有效进展导致死循环。对 socket 阻塞读而言，正常读取正长度缓冲区时通常会等待至少一个字节、EOF 或异常，但编写通用工具方法时仍应对返回值做严谨处理。输出零长度数组通常没有效果，也不应被用作心跳或刷新信号；如果协议需要心跳，应该定义明确的心跳帧。

## Reader 与 Writer

`Reader` 和 `Writer` 处理字符，而不是原始字节。它们的存在是因为文本不是简单的字节数组：同一个字符在 UTF-8、UTF-16、GBK 等字符集中可能对应不同字节序列，一个字符也可能由多个字节组成。`InputStreamReader` 负责把字节解码成字符，`OutputStreamWriter` 负责把字符编码成字节。只要程序跨越字节与字符边界，就必须知道字符集在哪里确定、错误输入如何处理、换行和缓冲由哪一层负责。

字符流适合已经确定编码的文本内容，例如读取 UTF-8 配置文件、处理行文本、实现简单的文本协议。它不适合读取任意二进制内容，也不适合在协议头和协议体混合二进制字段时偷懒使用。把二进制数据强行转成字符串，可能破坏不可见字节、截断零字节、触发替换字符，或者因为解码器跨缓冲区处理多字节字符而产生难以定位的问题。一个稳妥原则是：协议边界、长度字段、校验字段、压缩和加密内容使用字节流；只有明确的文本字段才进入字符流。

`BufferedReader.readLine()` 是字符流中最常用的方法之一，但它隐含了“行”的协议假设。它识别常见行终止符，返回时去掉行分隔符，因此调用方无法从返回值判断原始行尾是 `\n`、`\r\n` 还是 `\r`。这对大多数文本处理无关紧要，但对需要保留原始格式、计算签名、转发协议报文的程序就很重要。并且 `readLine` 没有内建最大行长度限制，面对不发送换行的输入会持续积累字符并阻塞等待。服务端读取外部输入时，常常需要自己实现带长度上限的行读取，或者在字节层限制总输入大小。

`Writer` 的缓冲也容易被忽视。`BufferedWriter.write` 后如果没有 `flush`，对端可能迟迟收不到响应；但每写一个字段就 `flush` 又会降低吞吐。文本协议通常在写完一条完整响应后刷新，而不是在每个字符、每个片段后刷新。`PrintWriter` 提供 `println` 和格式化输出，写起来方便，但它会吞掉很多 `IOException`，只通过内部错误标记暴露状态。教学示例中它很常见，生产级通信代码应慎重使用，尤其是需要区分超时、连接重置、磁盘满、权限错误等故障时。

## 典型阻塞点

BIO 的“阻塞”并不只发生在 `read`。从连接建立到关闭，多个 API 都可能让当前线程等待。理解这些等待点，有助于设计超时、线程池和日志，也能在排查线程堆栈时快速判断系统卡在哪里。

| 操作 | 可能阻塞的位置 | 常见原因 | 常用边界控制 |
| --- | --- | --- | --- |
| `ServerSocket.accept()` | 等待新连接 | 没有客户端连接、监听队列暂时无连接可取 | `setSoTimeout`、关闭 `ServerSocket` |
| `Socket.connect()` | 等待连接建立 | 网络不可达、对端无响应、握手未完成 | `connect(address, timeout)` |
| `InputStream.read()` | 等待至少一个字节、EOF 或异常 | 对端未发送、文件设备慢、管道无数据 | 读超时、关闭流、协议大小限制 |
| `BufferedReader.readLine()` | 等待行结束、EOF 或异常 | 对端未发送换行、字符解码等待更多字节 | 读超时、自定义最大行长度 |
| `OutputStream.write()` | 等待下游可写 | socket 发送缓冲区满、磁盘慢、管道对端不读 | 写超时能力依实现而定、关闭连接、限速 |
| `flush()` | 等待缓冲数据提交到下游 | 下游阻塞、网络拥塞、磁盘压力 | 减少不必要 flush、设置整体请求超时 |
| `close()` | 等待关闭过程完成或触发异常 | 缓冲刷新、协议关闭、文件系统延迟 | try-with-resources、记录 suppressed 异常 |

写阻塞比读阻塞更容易被低估。很多人把 socket 写入想象成“内存复制”，但当对端读取很慢、网络拥塞或本机发送缓冲区耗尽时，写操作同样可能卡住工作线程。服务端如果在持有共享锁时向慢客户端写响应，就可能把其他连接也拖住。文件写入也类似，磁盘或网络文件系统压力高时，一个看似普通的日志写入、结果文件写入或临时文件关闭都可能成为延迟尖刺来源。

关闭也不是完全无成本的动作。关闭带缓冲的输出流通常会先刷新缓冲，刷新失败时会抛出异常；关闭 socket 可能影响对端观察到的连接状态；关闭 `ObjectOutputStream` 还涉及对象流自身的协议收尾。try-with-resources 能保证关闭顺序和异常保留，但不能替调用者决定业务语义。对于请求已经失败、响应不再可信的连接，关闭前是否继续刷新缓冲要结合协议要求判断；对于文件输出，关闭失败可能意味着写入结果不完整，应把它视为写操作失败的一部分。

## 连接生命周期与协议循环

一个 BIO 连接可以只处理一次请求，也可以在同一个连接上循环处理多次请求。一次请求后立即关闭的模型最简单：服务端读取请求、写出响应、关闭 socket。它天然释放资源，也避免空闲连接占用线程，但频繁建立连接会增加握手成本，并且不适合需要会话状态或连续交互的协议。长连接模型可以复用连接，但工作线程会在每次等待下一条请求时继续占用资源，因此必须设置空闲超时、最大请求数、最大连接时长或服务端主动回收策略。

长连接 BIO 的处理循环通常是“读一帧、处理一帧、写一帧、继续读”。这里的“一帧”必须是协议定义的完整请求，而不是一次 `read` 返回的字节。若协议允许流水线，即客户端连续发送多个请求而不等待响应，服务端还要考虑响应顺序、写缓冲和异常恢复。BIO 的顺序代码很适合严格一问一答的协议，但当协议要求同一连接上并发处理多个请求、乱序响应或持续推送时，一个线程处理一个连接的模型就会变得笨重，通常需要额外队列、写线程或换用事件驱动框架。

半关闭是连接生命周期中容易混淆的概念。`shutdownOutput()` 表示本端已经发送完数据，后续不能再写，但仍可继续读；对端读取到 EOF 后可以知道本端发送结束。`shutdownInput()` 表示本端不再读取输入。半关闭常见于“客户端发送完整请求体后等待服务端响应”的协议设计，但前提是双方都把 EOF 作为某个方向数据结束的合法信号。如果协议使用长度前缀或分隔符作为消息边界，就不一定需要半关闭；如果对端没有预期半关闭，可能把它当作连接结束或异常处理。

服务端优雅停机也要围绕生命周期设计。监听线程应能通过关闭 `ServerSocket` 或 accept 超时退出；线程池应停止接收新任务，并给已接收连接一定处理时间；超过宽限期的连接要关闭底层 socket，释放阻塞线程；日志中要区分正常停机导致的 socket 关闭和真实网络异常。客户端则应明确连接所有权：谁创建 socket，谁负责最终关闭；把 socket 交给上层封装后，就不要在多个地方重复关闭或一边关闭一边继续读写。

## 缓冲策略与内存占用

缓冲的第一目标是减少频繁小 I/O 的成本，但缓冲也会改变数据可见时机和内存占用。`BufferedInputStream` 默认缓冲区较小，适合普通场景；大文件顺序读取、批量复制或高吞吐网络传输可以通过压测调整缓冲区。缓冲区太小会增加系统调用，太大则会让每个连接或每个任务占用更多堆内存。BIO 服务端如果给每个连接都分配大缓冲，在连接数上升时很快会把内存消耗放大。

读缓冲还会影响协议组合。假设在同一个 socket 上先用 `BufferedReader` 读取文本头，再直接从原始 `InputStream` 读取二进制体，可能出错，因为 `BufferedReader` 已经提前从底层流读入了一部分字节并保存在自己的缓冲区里，原始流的位置已经越过这些字节。混合读取时应保持单一入口：要么全部在字节层读取并自己解析文本头，要么使用能同时管理缓冲和边界的协议解析器。多个包装对象共享同一个底层输入流时，最容易出现“数据明明发来了却读不到”的错觉。

写缓冲同样要避免多层失控。应用层可能有 `BufferedOutputStream`，字符层可能有 `BufferedWriter`，协议层可能有自己的 `ByteArrayOutputStream`，操作系统还有 socket 发送缓冲区。多层缓冲能提升吞吐，但也会让一次响应在多个地方排队。对于交互式协议，写完一条完整响应后及时刷新很重要；对于批量导出或文件复制，减少 flush 次数更重要。选择策略时要明确优先级：是降低延迟，还是提高吞吐，还是限制内存峰值。

`ByteArrayOutputStream` 常被用来先组装完整消息，再一次性写入底层流。这样便于计算长度前缀或校验值，但如果消息可能很大，就会在内存中复制和扩容，造成峰值压力。大消息更适合分块传输：先写头部，再按固定大小从文件或输入流搬运内容，同时限制总长度和传输时间。BIO 不反对大数据传输，但它要求调用线程在传输期间一直参与等待和搬运，所以更需要控制连接数和任务并发。

## 资源管理与异常安全

BIO 资源通常对应操作系统文件描述符、socket、管道或文件句柄。它们不是普通 Java 对象，不能只依赖垃圾回收来释放。对象不可达并不等于底层描述符马上关闭；即使最终清理发生，也可能已经超过进程允许打开文件数，导致新的连接或文件操作失败。因此，任何打开流、socket、server socket 的代码都应有明确关闭路径。try-with-resources 是默认写法，复杂生命周期可以封装成实现 `AutoCloseable` 的对象统一管理。

关闭顺序一般遵循“最外层包装先关闭”。因为最外层最了解还未刷新的缓冲和协议收尾，关闭它会级联关闭下游资源。手动分别关闭多层流时，容易先关闭底层导致外层刷新失败，也容易把同一个异常记录多次。try-with-resources 中声明多个资源时，会按声明的相反顺序关闭；如果业务代码抛出异常，关闭时又抛异常，后者会作为 suppressed exception 保存。排查文件写入失败或响应写出失败时，不应忽略 suppressed exception，它可能包含真正的关闭刷新错误。

资源所有权需要在接口层说明。一个方法接收 `InputStream` 时，到底是读取后关闭，还是只读取不关闭，应有约定。工具方法通常不应该擅自关闭调用者传入的流，除非方法名或文档明确表达“接管资源”。相反，创建资源的方法应尽量把关闭责任交给清晰的调用方。所有权混乱会导致两类问题：一种是泄漏，资源无人关闭；另一种是过早关闭，上层还想继续读写时发现流已关闭。

异常安全还包括“部分成功”的处理。写文件时，可能已经写出前半部分，后半部分失败；写网络响应时，可能状态行已发出，正文失败；读取请求时，可能已经修改了部分业务状态，随后连接断开。BIO API 只能报告 I/O 结果，不能自动回滚业务动作。重要操作应先完成输入校验，再执行不可逆副作用；文件更新应使用临时文件和原子移动；网络协议应定义失败响应和连接关闭语义；业务层应区分“请求未读取完整”“请求已处理但响应失败”“请求处理失败但连接仍可复用”等状态。

## 与 NIO 的边界比较

BIO 与 NIO 的差异不只是“旧 API 和新 API”。BIO 的核心表达是调用线程等待一个 I/O 操作完成，代码天然呈线性结构；NIO 的核心表达是通道、缓冲区、选择器和就绪事件，少量线程可以管理大量连接，但业务流程会变成状态机或回调链。BIO 消耗线程换取直观控制流，NIO 消耗模型复杂度换取连接规模和事件调度能力。两者都可以写出高质量程序，也都可能因为协议边界、缓冲、异常处理不当而出现问题。

判断是否需要从 BIO 转向 NIO，可以看瓶颈是否真的来自“阻塞线程太多”。如果服务只有几十或几百个并发连接，瓶颈在数据库、外部服务、锁竞争或 CPU 计算，改成 NIO 未必有收益，反而会增加排查难度。如果连接数很高但活跃请求很少，或者每个连接经常空闲等待，BIO 的线程占用就会很不经济。若系统需要统一管理读写超时、背压、连接注册、协议状态机和大量长连接，NIO 或成熟网络框架通常更合适。

也不要把线程池包装 BIO 误认为等价于 NIO。线程池可以限制最大工作线程数，使系统在过载时不至于无限创建线程；但被阻塞的任务仍然占着线程。连接数超过线程池容量后，后续任务只能排队或被拒绝。NIO 的选择器模型则允许线程在没有事件时不绑定某个连接，等连接可读可写时再处理。二者的资源表达方式不同，适合的容量规划也不同。

从学习路径看，BIO 是理解 I/O 的基础。只有先掌握流的单向性、部分读写、缓冲刷新、EOF、超时、关闭和协议边界，学习 NIO 时才能明白选择器只是改变等待方式，并没有消除消息解析、资源管理和异常恢复这些基本问题。很多 NIO bug 本质上仍是 BIO 时代的错误，只是从阻塞调用变成了事件处理中的状态遗漏。

## 排查与观测

BIO 问题通常可以从线程堆栈开始排查。线程 dump 中如果大量工作线程停在 socket 读方法附近，说明线程正在等待对端发送数据或等待 EOF；如果停在 socket 写方法或 flush 附近，说明下游接收慢或发送缓冲受阻；如果停在文件读写或同步方法附近，说明存储层可能是瓶颈；如果大量线程停在锁等待，I/O 模型未必是主要问题。堆栈只能说明当前瞬间状态，最好结合连接数、请求量、线程池活跃数、队列长度和超时日志一起判断。

日志应记录 I/O 阶段，而不只是记录“处理失败”。例如客户端可以分别记录连接开始、连接成功、写请求完成、读响应开始、读响应完成、关闭完成；服务端可以记录 accept、读到完整请求、业务处理完成、响应写出、连接关闭。这样当延迟升高时，才能看出时间花在连接建立、对端响应、业务计算、写回还是关闭阶段。日志字段要克制，避免在高并发下输出完整大报文，但关键的远端地址、请求标识、协议长度、耗时和异常类型应保留。

指标比单条日志更适合观察趋势。BIO 服务端至少应关注当前连接数、活跃工作线程数、线程池队列长度、拒绝任务数、读超时数、连接重置数、平均处理耗时、响应写失败数和打开文件描述符数量。文件处理程序则应关注吞吐、单文件耗时、失败路径、磁盘空间和临时文件残留。没有这些观测手段时，BIO 程序看上去只是“偶尔卡住”，很难判断该调超时、加线程、改协议还是限制客户端。

测试 BIO 程序时，应刻意模拟慢输入、慢输出、半包、粘包、提前关闭、超长消息、乱码、磁盘满和并发连接。只用本机快速客户端发送完整请求，往往会掩盖真实网络中的阻塞点。对于文本协议，要测试没有换行、只有半个多字节字符、不同换行符和超长行；对于长度前缀协议，要测试负数长度、超过上限长度、声明长度与实际内容不一致。BIO 代码越顺序，越容易让人误以为异常路径简单，测试反而要覆盖这些等待和部分完成状态。

## 设计检查清单

写 BIO 代码前，可以先问几个具体问题。输入从哪里结束，是 EOF、换行、固定长度、长度前缀还是分隔符？单条消息最大多大，超出后如何处理？连接读超时、连接建立超时和整体请求超时分别是多少？写响应失败后业务是否已经生效，是否需要补偿或记录？一个连接能处理几次请求，空闲多久回收？线程池多大，队列多长，拒绝后客户端会看到什么？这些问题的答案比选择某个流类更重要。

代码层面也有一些直接检查点。是否显式指定字符集；是否循环处理部分读取；是否避免用 `available()` 判断完整消息；是否在锁外执行阻塞 I/O；是否使用 try-with-resources 或清晰的关闭所有权；是否限制行长度和正文长度；是否记录关键 I/O 阶段耗时；是否有办法打断 `accept` 和工作线程阻塞；是否避免为每个请求无限制创建线程；是否对 `flush` 的时机有明确选择。能把这些问题处理清楚，BIO 程序在适合的场景中会非常稳定。

## 超时、关闭与中断

BIO 最大的工程风险之一是没有边界的等待。`Socket` 提供 `setSoTimeout(int timeout)` 设置读超时。超时后，阻塞在读操作上的线程会收到 `SocketTimeoutException`，连接本身不一定已经关闭，调用者可以选择重试、关闭或上报错误。`ServerSocket` 也可以设置 accept 超时，用于周期性检查服务状态或优雅退出标志。连接建立阶段可以使用 `socket.connect(address, timeout)` 控制连接超时，避免默认连接等待过长。

超时不是越短越好。过短会把正常网络抖动误判为故障，造成重试风暴；过长会让故障恢复变慢，占用线程。合理的超时应结合调用链预算、对端服务能力、网络环境和业务容忍度确定。服务端对读超时尤其敏感：如果客户端建立连接后不发送完整请求，工作线程就可能长期被占用，这类慢请求攻击或异常客户端会显著压低 BIO 服务能力。即使是内部系统，也应设置连接读超时、请求最大长度和空闲连接回收策略。

线程中断与 BIO 的关系容易产生误解。许多阻塞 I/O 调用不会因为 `Thread.interrupt()` 立即退出，特别是传统 `java.io` 流上的阻塞读写。可靠的取消方式通常是关闭底层 socket 或流，使阻塞调用抛出异常返回。对于服务端优雅停机，常见做法是关闭 `ServerSocket` 打断 `accept()`，停止接收新连接，然后关闭或等待已有连接处理完成。对工作线程而言，可以维护连接集合，在停机时关闭 socket；也可以让每个连接处理循环定期检查退出标志，但检查标志不能唤醒正在无限期阻塞的 `read`，仍需要超时或关闭配合。

资源关闭应使用 try-with-resources。关闭包装流通常会关闭底层流和 socket；关闭 socket 会关闭其输入输出流。需要半关闭连接时，可以使用 `shutdownOutput()` 表示本端不再发送数据，但仍可继续读取对端响应；使用 `shutdownInput()` 表示本端不再读取。半关闭适合某些请求体结束后等待响应的协议，但如果协议没有明确定义，随意半关闭会造成对端误判连接状态。无论如何，关闭流程都要考虑异常覆盖问题：业务异常和关闭异常都可能发生，try-with-resources 会把关闭异常作为 suppressed exception 附加到主异常上。

## BIO 与文件 I/O

虽然 BIO 常被放在网络编程语境下讨论，文件流同样是阻塞式的。`FileInputStream.read`、`FileOutputStream.write`、`RandomAccessFile.read` 都可能等待磁盘、文件系统缓存或底层设备完成操作。文件 I/O 的阻塞通常不像网络那样不可预测，但在机械磁盘、网络文件系统、大文件复制、频繁 fsync 或磁盘压力高时，同样会拖慢调用线程。

文件 BIO 的关注点包括缓冲大小、追加与覆盖、同步落盘、字符集、临时文件和异常恢复。使用 `BufferedInputStream` 可以减少小块读取的系统调用；使用 `Files.newBufferedReader`、`Files.newBufferedWriter` 可以明确字符集并简化资源管理。写配置文件或状态文件时，不应直接覆盖目标文件后再写入，因为进程崩溃可能留下半截文件。更稳妥的做法是写入临时文件，必要时 `fsync`，再原子移动到目标路径。BIO API 本身不保证业务级原子性，调用者要根据文件语义设计恢复策略。

## 性能特征与适用场景

BIO 的性能瓶颈主要来自线程数量和阻塞等待。对于短连接、低并发、请求处理简单的服务，BIO 可以达到足够好的吞吐，而且代码维护成本低。对于连接数远大于活跃请求数的场景，例如大量客户端保持连接但很少发送数据，BIO 会浪费大量线程等待。一个线程对应一个连接时，即使大多数线程处于休眠状态，栈内存和调度元数据也是真实成本。线程数达到几千甚至更多后，系统可能出现频繁上下文切换、内存压力、监控噪声和停机变慢等问题。

BIO 的吞吐还受到缓冲策略影响。每次读写一个字节通常是反模式；但缓冲区也不是越大越好，过大的缓冲区会增加内存占用，降低缓存局部性。文件复制可以使用 8KB、16KB、64KB 等大小进行基准测试，根据磁盘、文件系统和业务模式选择。网络协议则要平衡延迟和吞吐：小消息如果长时间停留在应用缓冲区中，会增加响应延迟；频繁 flush 又会增加系统调用和小包数量。TCP 还涉及 Nagle 算法、发送缓冲区和接收缓冲区，但 BIO 程序通常先从应用层缓冲和协议边界优化开始。

适合 BIO 的场景通常有明确特征：并发连接数有限，调用链以同步逻辑为主，团队希望降低事件驱动复杂度，服务对极端连接规模没有要求，或者程序是工具型、批处理型、管理型。需要谨慎使用 BIO 的场景包括：海量长连接、慢客户端不可控、连接空闲时间远大于活跃时间、单机连接数需要持续扩展、读写事件需要统一调度、协议解析需要高性能状态机。选择模型时不应只问“BIO 旧不旧”，而应问“阻塞线程是否是这个场景可接受的资源表达”。

## 异常与边界处理

BIO 程序的异常处理不应只写一个空的 `catch (IOException ignored)`。不同异常代表不同处置方式。`EOFException` 或 `read` 返回 `-1` 通常表示对端正常关闭或文件结束；`SocketTimeoutException` 表示超时，可按协议选择重试或断开；`ConnectException` 表示连接建立失败；`SocketException: Connection reset` 常见于对端异常关闭或发送 RST；`MalformedInputException` 可能是字符集解码失败；`FileNotFoundException` 可能是路径不存在、权限不足或目标是目录。日志中应记录操作类型、远端地址、文件路径、协议状态和请求标识，避免只留下异常堆栈。

边界处理还包括输入大小限制。网络服务不能无限制地读取一行或一个长度前缀声明的正文。攻击者或错误客户端可以发送超长行、负数长度、超大长度、永不结束的请求体。读取前应检查长度范围，读取过程中应累计字节数并在超过限制时关闭连接。文本协议也要考虑不同换行符、空行、尾部空白和编码错误。文件处理要考虑符号链接、相对路径、目录穿越、文件被并发修改、磁盘空间不足和权限变化。

多线程 BIO 还需要处理共享状态。每个连接一个线程并不意味着线程安全自动成立。如果多个连接共享计数器、缓存、连接注册表、输出文件或业务对象，仍然需要并发控制。不要在持有全局锁时进行阻塞 I/O，否则一个慢客户端可能拖住所有请求。更好的做法是把共享状态访问控制在短临界区内，阻塞读写放在锁外；或者通过队列把 I/O 结果交给单独的状态管理线程。BIO 的代码流程看似顺序，但只要引入多连接处理，就已经进入并发编程领域。

## 常见误区

第一个误区是认为 `available()` 可以判断一条消息是否完整。`InputStream.available()` 只表示在不阻塞的情况下大致可读取的字节数，不代表流的总长度，也不代表应用层消息边界。用它来决定是否读取完整请求会产生偶发截断和协议错误。

第二个误区是认为 `read(byte[])` 会填满整个数组。实际上它可能返回任意正数，只要读取到至少一个字节就可以返回。需要固定长度时必须循环读取，或者使用 `DataInputStream.readFully`。

第三个误区是忘记字符集。`new String(bytes)`、`FileReader`、`FileWriter` 等使用平台默认字符集的写法在跨机器、跨地区、跨部署环境时容易出问题。严肃代码应显式使用 `StandardCharsets.UTF_8` 或协议规定的字符集。

第四个误区是无限制创建线程。为每个连接 `new Thread` 在演示程序中清晰，在生产服务中很危险。即使使用线程池，也要限制队列、设置线程命名、监控活跃线程和拒绝数量。

第五个误区是只关闭外层业务对象而忘记底层连接。BIO 资源通常关联文件描述符，泄漏后会导致“too many open files”、端口耗尽或文件无法删除。try-with-resources 是默认选择，复杂生命周期则需要明确所有权。

第六个误区是把 BIO 代码中的阻塞等待当作业务处理时间。监控线程耗时、接口延迟和线程池占用时，需要区分 CPU 计算、锁等待、读等待、写等待、连接等待。只有知道时间花在哪里，才能决定是优化算法、增加线程、设置超时、拆分协议还是换用 I/O 模型。

## 实践建议

编写 BIO 程序时，可以遵循几个实用原则。先定义协议边界，再写读写代码；先设置超时和大小限制，再考虑性能优化；先用 try-with-resources 明确资源所有权，再处理复杂关闭；先用有界线程池控制并发，再做压测决定线程数。对于服务端，`accept` 循环要能退出，工作线程要有清晰命名，连接处理要记录关键上下文。对于客户端，连接、读、写分别设置合理超时，重试必须有次数、退避和幂等性判断，不能在网络抖动时无限重试。

当 BIO 程序出现性能问题时，不要立即重写成 NIO。先用线程 dump、连接数、线程池指标、GC 日志、文件描述符数量和系统调用统计定位瓶颈。如果线程大多阻塞在读操作，说明连接空闲或对端慢；如果阻塞在写操作，说明对端接收慢或网络拥塞；如果大量线程等待锁，问题可能是共享状态设计；如果 CPU 很高，瓶颈未必在 I/O 模型。只有确认线程等待模型本身成为主要限制时，切换到 NIO、异步 I/O 或成熟网络框架才是合理成本。

## 小结

BIO 的核心是用同步、阻塞、顺序的代码表达 I/O 流程。它的优势是简单、直观、容易调试，适合连接规模有限、协议简单、开发维护成本敏感的场景；它的劣势是线程与连接或任务容易绑定，面对大量慢连接和长连接时资源消耗明显。掌握 BIO 不是为了停留在传统模型，而是为了理解 Java I/O 的基础语义：流不保证消息边界，读写可能部分完成，缓冲影响性能和延迟，关闭与超时决定系统能否从异常中恢复。只有把这些基本问题处理好，再讨论 NIO 或更高级抽象，才不会把复杂模型建立在脆弱的 I/O 认知之上。

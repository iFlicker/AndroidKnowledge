# 3.4.2 NIO

NIO 是 Java 为解决传统阻塞 I/O 在可伸缩性、缓冲管理和通道抽象上的不足而提供的一组 API。这里的 NIO 通常指 `java.nio`、`java.nio.channels`、`java.nio.file` 等包中的能力，其中网络编程最常讨论的是 Buffer、Channel、Selector 和非阻塞 I/O。它不是简单地把“阻塞”改成“不阻塞”，而是把 I/O 操作从面向流的顺序读写，转向面向缓冲区、通道和事件就绪状态的编程模型。理解 NIO 的关键，是理解数据不再直接从流进入业务代码，而是先进入 Buffer；连接不再必须由一个线程独占等待，而是可以由 Selector 统一监听多个 Channel 的就绪事件。

NIO 的价值主要体现在大量连接和事件驱动处理上。传统 BIO 中，一个线程阻塞等待一个连接的数据；NIO 中，一个线程可以通过 Selector 观察许多通道，只有当某个通道可读、可写、可连接或可接收时，线程才处理对应事件。这样，线程资源不再随空闲连接线性增长，系统可以用较少线程管理大量连接。但 NIO 的代价也很明显：代码从顺序流程变为状态机，半包、粘包、写缓冲积压、兴趣事件切换、连接关闭和异常清理都要显式处理。NIO 适合需要可伸缩 I/O 的系统，不适合为了追求“看起来高级”而引入到简单程序中。

## Buffer：NIO 的数据容器

Buffer 是 NIO 最基础的抽象。无论是从文件读取、向 socket 写入，还是在应用层解析协议，数据都围绕 Buffer 流动。常用实现包括 `ByteBuffer`、`CharBuffer`、`IntBuffer` 等，其中 `ByteBuffer` 最重要，因为文件和网络底层都是字节。Buffer 有三个核心位置属性：`capacity` 表示容量，创建后通常不变；`position` 表示下一次读或写的位置；`limit` 表示当前可读或可写边界。理解这三个属性，是正确使用 NIO 的第一步。

Buffer 的常见生命周期是写入模式、翻转、读取模式、清理或压缩。刚分配的 Buffer 处于写入模式，`position` 为 0，`limit` 等于 `capacity`。从 Channel 读取数据到 Buffer 时，数据写入 `position` 之后的位置，`position` 随之增加。调用 `flip()` 后，`limit` 被设置为原来的 `position`，`position` 回到 0，此时可以从 Buffer 中读取刚写入的数据。读取完成后，如果数据全部消费，调用 `clear()` 回到写入模式；如果还有未消费数据，调用 `compact()` 把剩余数据移到开头，再继续写入。

很多 NIO Bug 都来自 Buffer 状态切换错误。忘记 `flip()` 会导致读取不到刚写入的数据；读取后忘记 `clear()` 或 `compact()` 会导致后续写入位置错误；把 `clear()` 理解成清零数据也不准确，它只是重置位置标记，旧数据仍可能留在内存里，只是不再被视为有效内容。解析协议时尤其要谨慎：Buffer 中可能只有半个消息，也可能有多个消息连在一起。应用层不能假设一次 `channel.read(buffer)` 对应一个完整请求。

`mark()`、`reset()`、`rewind()`、`duplicate()`、`slice()` 这些方法也经常出现在稍复杂的 NIO 代码中。`mark()` 会记录当前 `position`，后续 `reset()` 可以回到这个位置，适合在尝试解析一段可选字段失败后回退。`rewind()` 把 `position` 置回 0，但不改变 `limit`，适合对同一段有效数据重复读取。`duplicate()` 创建共享底层存储但拥有独立位置属性的新 Buffer，常用于把同一段数据交给不同组件读取而避免互相移动 `position`。`slice()` 从当前 `position` 到 `limit` 切出一个视图，新 Buffer 的 `position` 从 0 开始，容量等于剩余长度，也共享底层存储。共享底层存储意味着数据修改互相可见，不能把它们当作深拷贝使用。

`flip()` 与 `compact()` 的选择体现了 Buffer 管理的核心思路。读取通道后，Buffer 中 `[0, position)` 是新写入的数据，`[position, limit)` 是可继续写入的空位；调用 `flip()` 后，`[0, limit)` 变成可读数据。业务解析器消费一部分后，如果 `position == limit`，说明本轮数据完全消费，`clear()` 即可；如果 `position < limit`，说明尾部还有未完成消息，此时 `compact()` 会把 `[position, limit)` 这段剩余数据搬到开头，随后 `position` 指向搬移后数据的末尾，`limit` 恢复为 `capacity`，下一次通道读取会接在剩余数据后面。这个动作不只是“清理空间”，更是在维护字节流连续性。

Buffer 的绝对读写和相对读写也要区分。`get()`、`put(byte)` 会移动 `position`，属于相对操作；`get(index)`、`put(index, value)` 不移动 `position`，属于绝对操作。协议解析时，绝对读取可以用来预检长度字段，例如先检查 `buffer.remaining() >= 4`，再用 `getInt(buffer.position())` 观察长度，但暂不推进 `position`；确认整包可用后再正式移动位置。这样可以避免“读了一半发现不够，只能手动回退”的混乱。相反，如果所有解析都依赖相对读取，就要更严格地管理 `mark/reset` 或自定义状态变量。

还有一个容易忽略的边界是 Buffer 的容量策略。容量太小会导致频繁 `compact()`、扩容或临时累积，容量太大又会浪费内存，尤其在大量连接场景中，连接数乘以每连接读缓冲大小会快速放大。固定 8KB 或 16KB 读缓冲只是一种常见起点，不是普适答案。文本协议、二进制协议、请求大小分布、最大帧长度、连接空闲比例都会影响缓冲策略。工程上常见做法是为每个连接保留一个适中的读缓冲，再为大消息使用临时聚合缓冲或分块处理，同时设置最大帧长度，避免被异常输入拖垮内存。

```java
ByteBuffer buffer = ByteBuffer.allocate(8192);
int n = channel.read(buffer);
if (n == -1) {
    channel.close();
    return;
}
buffer.flip();
while (buffer.hasRemaining()) {
    byte b = buffer.get();
    // 根据协议状态消费字节
}
buffer.clear();
```

上面的代码只适合展示 Buffer 读写切换，并不适合直接作为完整协议解析器。真实解析器要在 Buffer 中保留未完成消息，通常会维护连接级上下文，把 ByteBuffer、解码状态、待写队列和业务状态绑定在一起。对于长度前缀协议，解析器需要先确认 Buffer 中是否有足够的长度字段，再确认是否有完整正文；如果不够，就 `compact()` 保留剩余字节，等待下一次可读事件。

## 堆内缓冲区与直接缓冲区

`ByteBuffer.allocate` 创建堆内缓冲区，内存由 JVM 堆管理，分配和回收成本低，能被普通 Java 数组访问。`ByteBuffer.allocateDirect` 创建直接缓冲区，内存通常位于堆外，I/O 操作可以减少一次从堆到本地内存的复制，在高吞吐场景可能更有优势。直接缓冲区不是免费的：分配和释放成本更高，生命周期受 Cleaner 和 GC 触发影响，不恰当的大量分配会造成堆外内存压力，并且排查起来比堆内对象更复杂。

选择缓冲区类型要结合使用方式。短生命周期、小块、业务层频繁访问字节数组的数据，堆内 Buffer 通常更简单。长生命周期、复用频繁、面向 socket 或文件的大块读写，可以考虑直接 Buffer。很多高性能网络框架会实现自己的缓冲池，避免频繁分配和释放。手写 NIO 程序时，即使不用复杂池化，也应避免每次读写事件都新建 ByteBuffer；连接级读缓冲可以复用，写缓冲可以按消息生命周期管理。

还要注意字节序。`ByteBuffer` 默认使用大端序，可以通过 `order(ByteOrder.LITTLE_ENDIAN)` 修改。网络协议常规定网络字节序，即大端；文件格式和某些二进制协议可能使用小端。读取 `int`、`long` 等多字节类型前必须明确协议字节序，否则跨语言、跨平台通信会出现难以排查的数据错误。

## Channel：双向 I/O 通道

Channel 是 NIO 中对 I/O 端点的抽象。与传统流不同，很多 Channel 是双向的，例如 `SocketChannel` 既可以读也可以写；`FileChannel` 可以在指定位置读写文件；`ServerSocketChannel` 用于接收连接；`DatagramChannel` 用于 UDP 通信。Channel 的读写方法以 Buffer 为参数，数据从 Channel 读入 Buffer，或从 Buffer 写入 Channel。

网络 Channel 可以配置为阻塞或非阻塞。使用 Selector 时，Channel 必须处于非阻塞模式。`SocketChannel.configureBlocking(false)` 之后，`read` 在没有数据时可能返回 0，`write` 在暂时无法写入更多字节时也可能返回 0，`connect` 可能立即返回 false，需要等待连接完成事件后调用 `finishConnect()`。这与 BIO 的阻塞语义完全不同：非阻塞调用返回并不代表操作完成，调用者必须用状态和事件管理后续进度。

`FileChannel` 与网络 Channel 不同，它不能直接注册到 Selector。文件通常被认为总是可读或可写，操作系统对普通文件的就绪通知意义有限。NIO 文件 I/O 的价值更多在于位置读写、批量传输、内存映射和与 `Path`/`Files` API 配合，而不是 Selector 多路复用。网络 NIO 与文件 NIO 虽然同属 NIO 包，但适用机制不同，不能混为一谈。

Channel 还提供一些高效传输能力。`FileChannel.transferTo` 和 `transferFrom` 可以在文件和通道之间传输数据，某些平台上可以利用零拷贝路径减少用户态和内核态之间的数据复制。它适合静态文件发送、大文件复制等场景，但具体行为受操作系统、文件系统、通道类型和 JDK 实现影响。使用时仍要处理部分传输：一次 `transferTo` 不保证传完整个范围，需要循环直到传输字节数达到预期或遇到异常。

Channel 的线程安全边界也需要谨慎对待。许多 Channel 实现允许多个线程并发调用某些方法，但这不等于业务层可以随意从多个线程同时读写同一个连接。非阻塞网络程序通常把一个 `SocketChannel` 绑定到一个 I/O 线程，在这个线程里完成读写和兴趣事件修改；其他线程如果产生响应，只投递任务或写入线程安全队列，再唤醒所属 Selector。这样可以避免读缓冲、写队列、关闭状态和 `SelectionKey` 被多个线程交错修改。对通道本身的并发保护，不能替代连接上下文的并发设计。

`ServerSocketChannel` 与 `SocketChannel` 的职责也不能混淆。前者只代表监听端点，关注 `OP_ACCEPT`，每次 `accept()` 得到一个新的 `SocketChannel`；后者代表具体连接，关注读、写或连接完成事件。`accept()` 在非阻塞模式下也可能返回 `null`，因为就绪只是提示“可能可以接收连接”，并不承诺每次调用都一定成功。处理 `OP_ACCEPT` 时通常循环 `accept()`，直到返回 `null`，这样可以一次性取尽内核已经准备好的连接，减少下一轮 select 的额外唤醒。

`DatagramChannel` 又是另一种语义。UDP 没有连接字节流，接收端调用 `receive(buffer)` 得到一个数据报及其来源地址，发送端调用 `send(buffer, address)` 发出一个数据报。数据报有消息边界，不存在 TCP 那种粘包半包问题，但存在丢包、乱序、重复和超过路径 MTU 后被分片或丢弃的风险。使用 Selector 监听 `DatagramChannel` 时，事件模型仍然是就绪通知，但应用层协议设计不能套用 TCP 字节流思维。

文件通道的定位能力常被低估。`FileChannel.read(buffer, position)` 和 `write(buffer, position)` 可以在不改变通道当前 position 的情况下访问指定文件偏移，适合并发读取文件不同区域或实现简单的随机访问结构。`map()` 可以创建 `MappedByteBuffer`，把文件区域映射到内存地址空间，应用像访问内存一样读写文件内容。内存映射适合大文件随机读取、索引文件、日志段读取等场景，但它也带来页缓存、强制刷盘、映射释放时机和文件截断并发访问等问题。不能因为它看起来“零拷贝”就默认比普通读写更适合所有文件操作。

## Selector：多路复用的核心

Selector 允许一个线程监控多个 Channel 的就绪状态。可注册的事件包括 `OP_ACCEPT`、`OP_CONNECT`、`OP_READ`、`OP_WRITE`。注册后会得到 `SelectionKey`，它代表 Channel 与 Selector 的关系，并保存兴趣事件、就绪事件和可附加对象。附加对象通常用来存放连接上下文，例如读缓冲、写队列、协议解析器和会话状态。

典型事件循环大致如下：创建 Selector；创建并配置非阻塞 ServerSocketChannel；注册 accept 事件；循环调用 `selector.select()` 等待事件；遍历 selected keys；根据事件类型处理 accept、read、write、connect；处理完后从 selected 集合中移除 key。移除这一步非常重要，因为 selected keys 集合不会自动清空，忘记移除会导致同一个事件被重复处理。

```java
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);
server.bind(new InetSocketAddress(9000));
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();
    Iterator<SelectionKey> it = selector.selectedKeys().iterator();
    while (it.hasNext()) {
        SelectionKey key = it.next();
        it.remove();
        if (!key.isValid()) {
            continue;
        }
        if (key.isAcceptable()) {
            accept(selector, key);
        } else if (key.isReadable()) {
            read(key);
        } else if (key.isWritable()) {
            write(key);
        }
    }
}
```

这个循环隐藏了许多工程细节。处理事件时不能让业务逻辑长时间占用 selector 线程，否则其他连接的 I/O 事件会被延迟。常见做法是 selector 线程只负责读写和轻量协议编解码，耗时业务提交到工作线程池；业务完成后把响应放入连接写队列，再唤醒 selector 线程注册写事件。跨线程修改 `SelectionKey` 的兴趣事件时，要注意并发可见性和 `selector.wakeup()`，否则 selector 可能继续阻塞，不能及时发现新任务。

Selector 的“兴趣事件”和“就绪事件”是两个不同集合。兴趣事件由程序通过 `key.interestOps(...)` 设置，表示希望 Selector 关注哪些状态；就绪事件由 Selector 在 select 返回后写入，表示本轮观察到哪些状态可能已经发生。程序不能把 `readyOps` 当作长期状态保存，因为它只反映一次选择结果；也不能忘记在写队列为空后取消 `OP_WRITE`，因为兴趣集合会影响之后每一次 select。一个连接当前可以同时对读和写感兴趣，也可能在连接建立阶段只关注 `OP_CONNECT`，在关闭阶段取消所有兴趣并释放资源。

`SelectionKey` 的有效性也有明确生命周期。注册成功后 key 有效；调用 `cancel()`、通道关闭或 Selector 关闭后，key 会失效。处理事件前检查 `key.isValid()` 是必要的，但这只是防御性动作，不应掩盖关闭流程混乱。更好的做法是把关闭收敛到统一函数，由它负责设置连接关闭标志、取消 key、关闭 channel、清理附件和队列。否则一个连接可能同时被读异常、写异常、业务超时、外部关闭请求触发关闭，最后出现重复释放、空指针或事件循环中访问已关闭通道。

`select()`、`select(timeout)`、`selectNow()` 的语义也不同。`select()` 会阻塞直到至少一个通道就绪、Selector 被唤醒或线程被中断；`select(timeout)` 最多等待指定时间，适合在事件循环中周期性执行超时扫描；`selectNow()` 立即返回，适合事件循环被外部任务唤醒后快速检查，但如果在没有节制的循环中调用，会形成忙轮询。高质量的事件循环通常不是简单地永远 `select()`，而是结合任务队列、定时任务、超时扫描和唤醒机制，明确每次循环除了 I/O 就绪还要处理哪些本地任务。

`wakeup()` 是跨线程协作的关键。一个线程正在 `select()` 阻塞时，另一个线程把响应加入某个连接的写队列，并希望尽快注册 `OP_WRITE`，就需要唤醒 selector 线程。常见模式是外部线程只把一个任务放入 selector 线程的任务队列，然后调用 `selector.wakeup()`；selector 线程醒来后先执行任务，修改 key 的兴趣事件，再进入下一次 select。这样可以把 key 修改集中在 owner 线程，降低并发风险。直接在业务线程里改 `interestOps` 虽然有时可行，但更容易遇到可见性、取消竞争和唤醒时机问题。

就绪事件本身也不是“操作一定不会阻塞或一定成功”的保证。它更像一个提示：在刚才观察的时刻，内核认为该通道有机会执行某类操作。等 Java 代码真正调用 `read`、`write`、`accept` 或 `finishConnect` 时，状态可能已经变化，调用仍可能返回 0、返回 null、抛出异常或只完成一部分。因此事件处理代码必须以实际调用结果为准，而不是把 `isReadable()`、`isWritable()` 当作完成承诺。

## 读事件、写事件与背压

NIO 中读事件比较容易理解：Channel 可读时，循环读取直到 `read` 返回 0 或 -1。返回正数表示读到了字节；返回 0 表示当前没有更多数据；返回 -1 表示对端关闭输出，通常需要关闭连接或进入协议定义的半关闭状态。读取后将 Buffer 翻转给解析器，解析器可能产出完整请求，也可能保留半包等待下一次读事件。

写事件更容易被误用。很多初学者会长期注册 `OP_WRITE`，结果 selector 几乎一直返回写就绪，因为大多数空闲 socket 在发送缓冲区未满时都是可写的。这会造成事件循环空转，占用 CPU。正确做法通常是：只有当连接有待发送数据，并且上一次写入没有全部完成时，才关注写事件；写队列清空后，取消 `OP_WRITE` 兴趣，只保留读事件或其他必要事件。

非阻塞写必须处理部分写。`SocketChannel.write(buffer)` 可能只写出 Buffer 的一部分，甚至返回 0。未写完的 Buffer 必须保留在连接上下文中，等待下一次可写事件继续写。不能在 selector 线程中用 while 死循环等待全部写完，因为这会把非阻塞模型退化成忙等。也不能无限制堆积待写消息，否则慢客户端会占用越来越多内存。背压是 NIO 服务端必须面对的问题：当对端接收慢或网络拥塞时，服务端要限制每个连接的写队列大小，必要时暂停读取、丢弃低优先级消息或关闭连接。

读侧也有背压。如果业务线程池处理不过来，selector 线程继续无限读取并投递任务，会把压力转移到内存队列。可以按连接、按全局队列或按业务类型设置阈值。当积压超过阈值时，暂时取消读兴趣，等业务处理追上后再恢复。这样能把压力停留在 TCP 接收窗口和对端发送速率上，而不是让本进程无限吸收数据。

背压策略需要和协议语义一致。请求响应式协议可以在某个连接的未完成请求数超过阈值时暂停读；流式协议可以按窗口大小推进；消息推送系统可以为不同优先级消息设置丢弃或降级策略；文件传输服务可以限制每连接和全局的待发送字节数。没有一种“永远正确”的背压实现。关键是让内存占用、处理能力和对端速率之间有明确边界，而不是让每个层级都假设下一层能无限承载。

写侧背压还会影响公平性。如果事件循环每次遇到一个可写连接就把它的写队列尽可能写空，某个高速连接可能长期占用 I/O 线程，其他连接的读写延迟被放大。实践中可以限制单轮每个连接最多写出的字节数或消息数，写到上限后即使队列未空，也把机会让给其他 key，下一轮再继续。这个策略牺牲一点单连接吞吐，换取整体延迟和公平性。类似地，读事件也可以限制单轮读取次数，避免一个活跃连接一直占据循环。

半关闭是另一个容易被忽略的边界。TCP 允许一端关闭输出方向但仍接收数据，Java 中可以通过相关 socket API 表达这种状态。大多数请求响应服务会在 `read` 返回 -1 时直接关闭连接，因为对端已经不再发送请求；但某些协议可能允许对端结束上传后继续等待响应。是否支持半关闭不是 NIO 自动决定的，而是应用协议决定的。无论选择哪种策略，都要在连接上下文中明确状态，否则容易出现读侧已关闭但写侧队列仍在增长的资源泄漏。

异常和返回值的处理也应区分“正常结束”和“异常失败”。`read` 返回 -1 表示流结束，属于连接生命周期的一部分；`IOException` 可能表示连接重置、网络错误或本地关闭竞争；`write` 返回 0 通常不是错误，只是暂时不能继续写。把这些情况都记录成同一种错误日志，会让运行时观察变得混乱。更合理的做法是对协议错误、远端关闭、超时关闭、本地限流关闭和异常关闭分别计数，便于判断系统是在正常回收连接，还是遇到了真实故障。

## 协议解析与状态机

NIO 的事件驱动模型要求协议解析器显式保存状态。在 BIO 中，一个线程可以写成“读长度、读正文、处理、写响应”的顺序形式；在 NIO 中，读长度时可能只收到 2 个字节，读正文时可能收到半个包，写响应时可能只写出前 100 字节。连接上下文必须记住当前解析到哪一步、还差多少字节、已经累积了哪些数据。

以长度前缀协议为例，解析器可以维护两个阶段：先读取 4 字节长度，再读取指定长度正文。每次读事件到来后，把新字节追加到连接缓冲中，然后循环尝试解析完整消息。如果长度字段不完整，停止；如果长度非法，关闭连接；如果正文不完整，停止；如果完整，切出一条消息交给业务处理，并继续检查 Buffer 中是否还有下一条消息。这种循环处理可以同时解决半包和多个包连在一起的问题。

字符协议还要处理解码边界。UTF-8 等变长编码中，一个字符可能跨越两次网络读取。如果直接把每次读取到的字节转成字符串，可能出现乱码或替换字符。更稳妥的做法是使用 `CharsetDecoder`，让解码器维护跨 Buffer 的状态，并正确处理 malformed input、unmappable character 和 end-of-input。对于行协议，还要限制单行最大长度，避免对端发送永不换行的大量字节。

协议解析器最好被设计成“喂入字节，产出零个或多个消息”的组件，而不是直接依赖 Channel。这样做有两个好处：第一，解析逻辑可以用普通字节数组或 ByteBuffer 单元测试，不需要真实网络连接；第二，解析器不关心数据是一次到达还是多次到达，只关心当前缓冲中是否已经满足协议条件。NIO 程序的复杂度很大一部分来自边界切分不可预测，解析器越独立，越容易覆盖这些边界。

长度字段必须有防御性校验。长度为负、长度超过最大帧、长度与协议头版本不匹配、压缩后大小异常、声明长度和实际内容校验失败，都应该尽早拒绝。没有最大长度限制的 NIO 服务非常危险，因为攻击者只需要发送一个巨大的长度前缀，再慢慢发送正文，就可以让服务端为未完成消息保留大量内存。即使不是恶意攻击，客户端 Bug 也可能产生类似效果。最大帧长度不是细节参数，而是协议资源模型的一部分。

如果协议支持流水线，即同一连接在前一个响应返回前继续发送后续请求，连接上下文还要记录未完成请求队列。解析器产出的消息进入业务线程池后，响应可能乱序完成；如果协议要求按请求顺序返回，就需要在连接维度缓存已完成但暂不能发送的响应。如果协议允许乱序，则每个请求必须有请求 ID，响应携带对应 ID。NIO 本身不提供顺序保证之外的应用语义，字节流顺序只保证对端发送的字节按序到达，不保证业务处理后的响应天然满足协议要求。

解析器还要处理关闭时的残留数据。连接关闭时，如果 Buffer 中还有未完成帧，是把它视为协议错误、静默丢弃，还是在某些协议中视为合法结尾，要根据协议定义决定。字符流协议可能允许最后一行没有换行符，二进制长度前缀协议通常不允许正文长度不足。把这些规则写清楚，比在代码里靠偶然行为处理更可靠。

## 线程模型与任务分发

NIO 并不等同于单线程。简单服务可以用一个 selector 线程完成所有 I/O 和业务，但只要业务处理可能耗时，就需要分离 I/O 线程和工作线程。常见模型包括单 Reactor 单线程、单 Reactor 多线程、主从 Reactor 多线程。单 Reactor 单线程结构简单，但任何耗时任务都会阻塞所有连接；单 Reactor 多线程由一个 selector 线程负责所有 I/O，业务交给线程池；主从 Reactor 通常由一个 acceptor 接收连接，再把连接分配给多个 selector 线程，每个 selector 管理一部分连接。

多 selector 线程可以提高多核利用率，也能减少单个 selector 管理的连接数量。但它引入连接分配、跨线程任务投递、key 修改和关闭协调等复杂度。连接通常在 accept 后选择一个 worker selector 注册，后续该连接的 I/O 尽量固定在同一线程，避免并发读写 Channel。业务线程不要直接操作 Channel 写数据，而是把响应放入连接的线程安全队列，再通知所属 selector 线程处理。这样可以减少锁和竞态，让 Channel 操作保持线程亲和性。

任务分发要注意顺序语义。如果同一连接上的请求需要按顺序响应，业务并行处理可能导致后完成的先写回。可以为每个连接维护序号和有序输出队列，或者在协议层允许乱序响应并携带请求 ID。没有明确策略时，NIO 的并发处理很容易破坏应用层协议假设。

单 Reactor 多线程是很多手写示例的自然升级，但它也有瓶颈：所有连接的读写仍集中在一个 selector 线程上，业务线程池只能减轻计算或阻塞业务的压力，不能提高 I/O 事件分发本身的并行度。当连接数、活跃连接比例或单连接吞吐继续上升时，需要考虑多个 I/O 线程。主从 Reactor 模型把接收连接和连接读写分离，主 Reactor 只处理监听 socket 的 accept，再把新连接分配给某个从 Reactor。分配策略可以是轮询、按当前连接数、按负载指标或按连接来源哈希，但无论哪种策略，都应尽量让连接后续固定在同一个 I/O 线程。

工作线程池的大小也不能简单等于 CPU 核数或连接数。若业务主要是 CPU 计算，线程数接近可用核心数更容易稳定；若业务会等待数据库、文件、远程服务或锁，线程数可能需要更高，但同时要设置队列上限和超时。NIO 解决的是 I/O 等待线程占用问题，不会自动解决下游阻塞。如果 selector 线程把请求无限提交给一个无界线程池或无界队列，系统只是把阻塞风险从连接线程转移到了任务队列。

I/O 线程和业务线程之间的数据交接也要控制对象生命周期。读缓冲通常属于连接和 I/O 线程，不能在解析后直接把还会被 `clear()` 或 `compact()` 的 ByteBuffer 交给业务线程长期持有。常见做法是在解析出完整消息后复制必要字节、构造不可变消息对象，或使用引用计数缓冲并严格规定释放责任。手写 NIO 时如果没有成熟缓冲池，优先选择简单明确的复制，等测量证明复制成为瓶颈后再引入更复杂的零拷贝消息对象。

线程模型还影响停机流程。优雅停机通常包括停止 accept 新连接、继续处理已有连接一段时间、拒绝或取消新的业务任务、把写队列中允许发送的响应刷出、关闭超时未完成连接、最后关闭 selector 和线程池。直接关闭 Selector 会让阻塞在 select 上的线程退出，但如果没有先处理连接上下文，可能丢失待写响应或留下不清晰的错误日志。停机也是连接生命周期的一部分，应当像正常读写一样被设计。

## 异常、关闭与资源清理

NIO 的关闭路径比 BIO 更复杂。一个连接可能在读事件中发现 -1，可能在写事件中发生 `IOException`，可能因为协议错误被主动关闭，也可能因为业务超时被管理线程关闭。关闭时应取消 SelectionKey、关闭 Channel、释放或回收 Buffer、清空待写队列、通知业务层连接结束。`key.cancel()` 只是取消注册关系，不等于立即关闭底层通道；通常需要显式 `channel.close()`。

异常处理应区分连接级异常和事件循环级异常。单个连接解析失败、远端重置、写入失败，不应让整个 selector 线程退出。事件处理代码应捕获连接级异常并关闭对应连接。selector 本身打开失败、循环中出现无法恢复的错误、线程被要求停机，则属于更高层生命周期问题。事件循环中还要避免因为某个 key 的处理异常而跳过 `it.remove()`，造成重复处理。

空闲连接管理也是 NIO 服务端的重要部分。Selector 只告诉你 I/O 就绪，不会自动告诉你连接是否“业务上超时”。连接上下文应记录最后读写时间、最后请求时间或心跳时间，定期扫描并关闭超时连接。扫描可以在 selector 线程中周期性执行，也可以由外部定时器投递任务到 selector 线程。不要在外部线程直接关闭正在被 selector 线程处理的连接而不做协调，否则容易出现竞态和难以复现的异常。

## NIO 与 NIO.2 的关系

Java 7 引入的 NIO.2 扩展了文件系统 API 和异步通道能力。`Path`、`Files`、`FileSystem`、`WatchService` 让文件操作比旧的 `File` 更强；`AsynchronousSocketChannel`、`AsynchronousServerSocketChannel`、`AsynchronousFileChannel` 提供异步 I/O 风格，操作完成后通过 Future 或 CompletionHandler 获取结果。NIO.2 并不替代 Selector 模型，而是提供另一类抽象。不同平台上异步通道的底层实现可能不同，性能特征也需要实测。

在日常 Java 网络服务中，直接手写 Selector 的机会并不总是很多，因为成熟框架已经封装了缓冲池、事件循环、协议拆包、写队列、背压和线程模型。但理解 NIO 仍然很重要，因为这些框架的设计语言来自 NIO。遇到 CPU 空转、写队列积压、半包解析错误、连接泄漏、selector 线程被业务阻塞等问题时，只有理解底层模型，才能看懂线程 dump 和框架指标。

需要注意，NIO.2 的异步通道并不等同于“永远由操作系统完成真正异步 I/O”。不同平台、不同通道类型和不同 JDK 实现可能采用不同底层机制，有的操作可能由系统异步通知完成，有的可能由线程池模拟异步效果。使用 `AsynchronousChannelGroup` 时，线程池配置、回调执行时间和异常处理同样重要。如果完成回调中执行耗时业务，异步通道的完成通知线程也会被阻塞。换句话说，NIO.2 改变的是 API 风格和部分底层能力，不会取消背压、缓冲管理和业务隔离这些基本问题。

`Path` 和 `Files` 所代表的文件系统 API 与 Selector 网络模型也应分开理解。`Files.readAllBytes`、`Files.lines`、`Files.copy` 等方法让文件操作更简洁，但它们不是网络多路复用。`WatchService` 可以监听目录变化，但不同平台的事件合并、丢失、递归监听能力和延迟特征不同，需要按使用场景验证。把所有 `java.nio` 包里的能力都笼统称作“非阻塞 NIO”是不准确的：NIO 是一组围绕缓冲、通道、文件系统和多路复用的 API 家族，其中只有一部分用于网络非阻塞事件循环。

## NIO 与 BIO 的边界

BIO 的编程模型是阻塞、顺序和线程栈保存状态。一个连接一个处理线程时，代码可以自然写成读取请求、执行业务、写出响应，局部变量就是协议状态，异常沿调用栈传播，关闭资源通常放在 finally 中。这个模型的优点是直观、易调试、易与同步业务代码组合。缺点是连接等待会占用线程，连接数很大且大部分时间空闲时，线程栈、调度和上下文切换成本会迅速上升。

NIO 的编程模型是非阻塞、事件驱动和显式状态机。连接没有独占线程，协议状态必须放在连接上下文中；读写不保证一次完成，必须处理半包和部分写；业务不能随便阻塞 I/O 线程，必须通过任务队列和工作线程隔离。它的优势是用较少线程承载大量连接，并能更细粒度控制读写、积压和超时。代价是代码复杂度显著增加，调试时要同时观察事件循环、连接状态、缓冲位置和任务队列。

因此，选择 BIO 或 NIO 应该从问题出发。连接数少、请求短、业务同步、团队更重视开发速度时，BIO 或基于线程池的阻塞 I/O 可能足够好。连接数多、长连接多、空闲等待多、需要主动推送或需要严格控制每连接内存和写入积压时，NIO 更有价值。高吞吐文件复制、静态文件发送、代理转发、消息网关、长连接服务都是 NIO 常见适用范围。但如果业务瓶颈在数据库、锁竞争或复杂计算，单纯把网络层改成 NIO 可能几乎没有收益。

还要区分“阻塞模式的 Channel”和“传统 BIO”。NIO 的 `SocketChannel` 可以配置为阻塞模式，这时它的读写语义接近传统 socket 流，但仍使用 Buffer 和 Channel API；传统 `InputStream`/`OutputStream` 也可以通过 `Channels` 工具类与 Channel 互相包装。这说明 API 形态、阻塞语义和线程模型是三个维度，不能简单地把所有 Channel 都视为非阻塞，也不能把所有 Stream 都视为低性能。真正决定系统行为的是通道是否非阻塞、是否使用 Selector、是否有清晰的缓冲和线程模型。

从维护角度看，BIO 的风险通常集中在连接线程数量、超时设置和阻塞调用堆积；NIO 的风险则集中在状态机错误、事件兴趣管理、内存积压和 selector 线程被阻塞。两者都需要超时、限流、异常清理和监控。NIO 不是 BIO 的全面替代，而是在特定连接模型下提供更强控制力的工具。

## 一个连接在 NIO 中的生命周期

一个 TCP 服务端连接在 NIO 中通常从 `OP_ACCEPT` 开始。监听通道就绪后，服务端接受连接，得到 `SocketChannel`，设置为非阻塞，创建连接上下文，分配读缓冲和写队列，然后把它注册到某个 Selector，初始兴趣一般是 `OP_READ`。从这一刻开始，连接的状态不再由某个线程调用栈保存，而是由 `SelectionKey` 的附件和应用维护的数据结构保存。

当读事件到来，I/O 线程从通道读取字节到读缓冲，处理返回值，翻转 Buffer，交给解析器尽可能产出完整消息。产出的消息如果可以立即处理，就在 I/O 线程内完成轻量逻辑；如果需要耗时业务，就提交给工作线程，并记录未完成请求。解析结束后，若有剩余半包则 `compact()`，否则 `clear()`。如果读到 -1 或协议错误，进入关闭流程。如果业务积压过多，可能暂时取消 `OP_READ`，把压力反馈给对端。

业务线程完成处理后，不直接写 Channel，而是把响应编码成一个或多个待写 Buffer，加入连接写队列，并投递任务给连接所属 I/O 线程。I/O 线程发现写队列非空后注册 `OP_WRITE`。写事件到来时，它从队列头部取 Buffer 调用 `write`，写完一个移除一个；遇到部分写则保留当前 Buffer，等待下一次可写；队列清空后取消 `OP_WRITE`。如果写队列超过上限，可以触发限流、丢弃或关闭，这取决于协议和业务策略。

连接可能因为多种原因结束：远端关闭、协议错误、空闲超时、业务超时、服务停机、写失败或本地限流。无论原因如何，最终都应走同一条清理路径。清理路径需要保证 key 不再被处理，channel 被关闭，附件从管理结构中移除，缓冲和队列不再被引用，相关计数和日志能够反映真实原因。NIO 程序的稳定性很大程度取决于这些边缘路径，而不是只取决于正常读写路径。

## 可观察性与排障

NIO 程序如果没有指标，很难判断问题发生在哪一层。至少应观察连接数、活跃连接数、每轮 select 返回 key 数、selector 循环耗时、读写字节数、读写事件次数、写队列长度、待处理任务数、业务线程池队列长度、连接关闭原因、协议解析错误数和超时数。单看 QPS 或平均延迟不足以定位 NIO 问题，因为一个写事件空转就可能让 CPU 飙高，而业务吞吐并没有提高。

CPU 空转常见原因是长期注册 `OP_WRITE`、`selectNow()` 忙轮询、异常 key 没有取消、事件处理异常后没有移除 selected key、任务队列唤醒策略错误等。排查时可以记录每轮循环处理的事件类型和数量，观察是否大量出现写就绪但没有实际写出字节，或者同一个 key 反复抛出异常。线程 dump 中如果 selector 线程长期不在 select 上，而是在业务方法、日志输出、锁等待或远程调用中，就说明 I/O 线程职责已经被破坏。

内存上涨常见原因是慢客户端导致写队列积压、协议半包没有最大长度限制、业务队列无界、每次事件分配新 Buffer、连接关闭后附件仍被全局结构引用。排查时要把“连接数乘以每连接基础内存”估算清楚，再观察大对象来自读缓冲、写缓冲、业务消息还是任务队列。NIO 的低线程数并不代表低内存；如果每个连接都保留过大的 Buffer 或未限制的待写队列，大量空闲连接也会消耗可观内存。

延迟抖动则可能来自 selector 线程被单个连接占用、单轮读写没有公平上限、业务线程池排队、GC、日志同步刷盘或下游服务波动。NIO 只能让网络等待更可控，不能消除系统其他部分的排队。排障时应把延迟拆成读取等待、解析、业务排队、业务执行、响应排队、写出等待几个阶段，分别记录时间，避免把所有问题都归因于 Selector。

## 常见误区

第一个误区是认为 NIO 一定比 BIO 快。NIO 的优势在于连接可伸缩性和事件调度，不保证单个请求延迟更低。低并发、短连接、简单同步处理场景中，BIO 可能更容易达到目标性能。

第二个误区是把非阻塞读写写成忙等循环。`read` 或 `write` 返回 0 时，应回到 selector 等待下一次就绪，而不是继续空转消耗 CPU。

第三个误区是长期注册写事件。写事件通常在 socket 发送缓冲区有空间时就就绪，长期关注会导致 selector 不断醒来。只有存在未写完数据时才应关注写事件。

第四个误区是忽视 Buffer 的半包问题。一次 read 不是一次消息，一次 write 也不是一次完整发送。协议解析和写队列必须处理部分读写。

第五个误区是让 selector 线程执行业务逻辑、访问数据库或等待远程调用。这样会阻塞所有连接的 I/O 处理，使 NIO 失去意义。

第六个误区是无限制缓存待写数据。慢客户端会把服务端内存拖垮，必须有连接级和全局级的积压限制。

第七个误区是把 `ByteBuffer.clear()` 当成数据擦除。它只重置 `position`、`limit` 和标记，不会把底层内存清零。如果缓冲区复用后被错误地按旧边界读取，仍可能读到旧内容。安全敏感数据还要额外考虑覆盖或避免长时间保留。

第八个误区是认为 Selector 事件等于完整消息。Selector 只管理通道就绪，不理解应用协议。消息边界、最大长度、解码状态、请求顺序和响应顺序都属于应用层职责。

第九个误区是过早追求零拷贝和直接内存。直接 Buffer、内存映射、`transferTo` 都有适用场景，但也会增加生命周期和平台差异复杂度。没有测量之前，清晰的缓冲复用和明确的背压通常比复杂优化更重要。

第十个误区是忽略测试中对网络切分的模拟。只用一次完整写入和一次完整读取测试协议解析器，无法覆盖 NIO 最容易出错的地方。测试应覆盖长度字段被拆开、正文被拆开、多个消息粘在一起、非法长度、连接中途关闭、部分写和写队列积压等情况。

## 实践建议

手写 NIO 程序时，先把连接上下文设计清楚：读缓冲、解析状态、写队列、最后活跃时间、所属 selector、关闭标志都应有明确位置。然后定义协议帧格式和最大长度，确保解析器能够在任意切分的字节流上工作。事件循环中保持处理短小，耗时任务交给工作线程，跨线程结果通过队列投递回 I/O 线程。写事件按需注册，写完立即取消。所有关闭路径收敛到一个方法，避免遗漏 key、channel 或缓冲资源。

性能调优应从测量开始。观察 selector 线程 CPU、selected key 数量、每轮循环耗时、读写字节数、写队列长度、业务线程池队列、连接数和超时关闭数。高 CPU 不一定是吞吐高，可能是写事件空转或异常循环；连接数高不一定有压力，关键看活跃读写和积压；延迟高也不一定是 NIO 问题，可能是业务线程池、锁、GC 或下游服务。NIO 给了程序更细的控制权，也要求程序承担更细的责任。

## 小结

NIO 的核心不是“新 I/O”这个名字，而是 Buffer、Channel、Selector 组合出的事件驱动模型。Buffer 让数据位置和边界显式化，Channel 让 I/O 端点统一化，Selector 让少量线程可以管理大量连接。它解决了 BIO 在线程等待上的可伸缩性问题，同时把协议状态、部分读写、写入背压、跨线程协调和资源清理暴露给开发者。掌握 NIO 的重点，是接受 I/O 不再是线性脚本，而是一组随事件推进的状态机；只有把状态机、缓冲区和资源生命周期设计清楚，NIO 才能成为可维护的高并发基础，而不是难以排查的复杂代码。

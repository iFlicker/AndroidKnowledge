# 3.4.5 Socket编程

Socket 编程是 Java 网络编程的基础。它把应用程序与传输层连接起来，使程序可以通过 TCP 或 UDP 在不同进程之间交换字节。Java 标准库提供了 `Socket`、`ServerSocket`、`DatagramSocket`、`SocketChannel`、`ServerSocketChannel`、`DatagramChannel` 等 API，既能编写同步阻塞程序，也能编写非阻塞事件驱动程序。无论使用哪一种 API，核心问题都不是“怎样调用 read 和 write”，而是如何理解连接、字节流、报文边界、超时、关闭、异常、缓冲、并发和协议设计。

Socket API 的抽象层次较低。TCP 只提供可靠、有序、面向连接的字节流，不理解请求、响应、消息、对象或文本行；UDP 提供无连接的数据报，不保证送达、不保证顺序、不保证去重。应用层必须自己定义消息格式、长度限制、重试策略、心跳、错误处理和安全边界。很多网络程序的问题并不来自 Java 语法，而是来自把传输层能力误认为应用层语义：以为一次 write 对应一次 read，以为连接没断就代表对端健康，以为超时只是网络慢，以为关闭 socket 就不会丢数据。这些假设都需要被拆开。

## TCP Socket 的连接模型

TCP 通信通常由服务端监听端口，客户端发起连接。服务端使用 `ServerSocket` 绑定本地地址和端口，调用 `accept()` 等待连接；客户端使用 `Socket` 连接服务端地址。连接建立后，双方都可以通过输入流读取字节、通过输出流写入字节。TCP 保证字节顺序和可靠传输，但不保证应用层消息边界。发送端写入两次，接收端可能一次读到合并后的数据；发送端一次写入大量数据，接收端可能多次读到片段。这就是常说的粘包和半包问题，更准确地说，是字节流协议本来就没有消息边界。

```java
try (Socket socket = new Socket()) {
    socket.connect(new InetSocketAddress("example.com", 80), 3000);
    socket.setSoTimeout(5000);
    OutputStream out = socket.getOutputStream();
    InputStream in = socket.getInputStream();
    out.write("ping\n".getBytes(StandardCharsets.UTF_8));
    out.flush();
    byte[] buffer = new byte[1024];
    int n = in.read(buffer);
    if (n != -1) {
        String response = new String(buffer, 0, n, StandardCharsets.UTF_8);
        System.out.println(response);
    }
}
```

这个例子只展示基本流程，不代表完整协议实现。真实程序需要循环读取，直到满足协议定义的结束条件；需要处理 `read` 返回 -1；需要限制响应大小；需要区分连接超时、读超时和连接重置。`connect` 超时控制连接建立阶段，`setSoTimeout` 控制读取阻塞等待时间，写超时在传统 `Socket` API 中没有同样直接的设置方式，写操作可能因为对端接收慢或网络拥塞而阻塞。需要严格写超时时，可以使用非阻塞通道、异步任务加关闭 socket，或更高层网络库。

服务端最简单的模型是 accept 循环加工作线程。它适合低并发和教学场景，但生产代码要限制线程数、连接数、请求大小和空闲时间。服务端 `ServerSocket` 构造或 bind 时可以设置 backlog，它表示已完成连接等待应用 accept 的队列长度提示；实际行为受操作系统限制。backlog 不是无限连接缓冲，也不能替代应用层限流。连接建立后，如果应用处理慢，客户端仍可能在后续读写中超时。

## Java Socket API 的角色与生命周期

Java 传统阻塞 Socket API 主要围绕 `ServerSocket`、`Socket`、`InputStream` 和 `OutputStream` 展开。`ServerSocket` 代表一个监听端点，它负责绑定本地地址、进入监听状态、接受连接；`Socket` 代表一条已经建立或正在建立的双向通信连接；输入输出流则是对 TCP 字节流的 Java I/O 包装。理解这些对象的角色边界很重要：`ServerSocket` 本身不承载某个客户端的数据，`accept()` 返回的每个 `Socket` 才对应一个具体连接；`Socket` 不是消息队列，不能天然区分请求和响应；流对象关闭时通常会连带关闭底层 socket，因此包装流的关闭顺序也会影响连接生命周期。

客户端创建 `Socket` 有两种常见方式。一种是直接使用带远端地址的构造方法，它会在构造期间发起连接；另一种是先创建未连接的 `Socket`，再调用 `connect(SocketAddress endpoint, int timeout)`。后者更适合真实程序，因为连接超时可以被明确设置，也可以在连接前配置一些 socket 选项。连接成功后，`getInputStream()` 和 `getOutputStream()` 返回的流可以反复使用，不应每次收发消息都重新获取或重新包装出多个互相竞争的读写器。尤其是同时混用 `BufferedReader`、`DataInputStream`、`ObjectInputStream` 等多个包装器读取同一个输入流时，很容易因为某个包装器预读了字节，导致另一个包装器看不到预期数据。

服务端生命周期通常分成监听 socket 和连接 socket 两层。监听 socket 在进程启动时创建，关闭时意味着服务不再接受新连接；连接 socket 在每次 `accept()` 成功后创建，关闭时只影响该客户端连接。服务端停机时，常见做法是先关闭监听 socket，使阻塞在 `accept()` 的线程退出，再通知连接处理线程停止接收新请求，最后按协议关闭已有连接。仅仅设置一个停止标志而不关闭监听 socket，可能让 accept 线程长期阻塞；只关闭监听 socket 而不管理已接受连接，又可能让进程看似停止接入却仍被旧连接占用资源。

`Socket` 的几个状态查询方法也要谨慎理解。`isConnected()` 表示这个 socket 曾经连接成功，并不表示当前连接仍然健康；`isClosed()` 只表示本地是否调用过关闭；`isInputShutdown()` 和 `isOutputShutdown()` 只反映本端是否关闭了对应方向。TCP 对端异常断开常常要等到本端读写时才暴露出来，因此不能用这些状态方法替代心跳、读写超时或协议层健康检查。稳定的连接管理应当基于最近一次成功读写时间、连续失败次数、协议状态和关闭原因，而不是只看某个布尔方法。

一个典型阻塞服务端会把 accept 循环和连接处理分开，并用线程池限制并发。示例代码可以很短，但它背后隐含几个边界：线程池满时如何处理新连接，连接处理超时时如何释放资源，服务停止时如何打断 accept，异常日志如何记录远端地址。这些问题如果不在基础结构里统一处理，后续业务处理代码会不断重复 try-catch-finally，并且容易遗漏关闭路径。

```java
ExecutorService workers = Executors.newFixedThreadPool(64);

try (ServerSocket server = new ServerSocket()) {
    server.setReuseAddress(true);
    server.bind(new InetSocketAddress("0.0.0.0", 9000), 128);

    while (!Thread.currentThread().isInterrupted()) {
        Socket client = server.accept();
        client.setSoTimeout(10_000);
        workers.execute(() -> {
            try (Socket socket = client) {
                handleConnection(socket);
            } catch (IOException e) {
                // 真实程序应记录远端地址、连接阶段和关闭原因。
            }
        });
    }
} finally {
    workers.shutdown();
}
```

这个模型适合作为理解阻塞 I/O 的起点，但不是万能模板。`newFixedThreadPool` 的任务队列默认可能无界，如果 accept 速度远高于处理速度，任务会在内存中堆积；如果改成有界队列，又要定义拒绝策略和新连接关闭方式。服务端还要考虑半开连接、慢请求、响应写不出去、客户端频繁重连等情况。Socket API 本身只提供通信原语，资源治理必须由应用补上。

## 阻塞读写的真实语义

很多 Socket bug 来自对 `read` 和 `write` 的语义想得过于理想。`InputStream.read(byte[])` 的返回值表示本次实际读到的字节数，只要读到至少一个字节就可以返回，不会为了填满缓冲区才返回。返回 `-1` 表示对端已经有序关闭输出方向，本端再也读不到更多数据；返回 `0` 对传统阻塞 `SocketInputStream` 不是常规数据读取结果，但在某些包装 API 或零长度数组读取中可能出现。读取固定长度报文时，必须循环直到累计长度满足要求，或者遇到 EOF、超时、长度超限等异常条件。

`OutputStream.write(byte[])` 返回类型是 `void`，这容易让人误以为写入已经被对端应用读取。实际上，写成功通常只表示数据已经交给本机内核发送缓冲，至多说明本端没有在该时刻发现连接错误。对端是否收到、是否解码成功、是否执行业务逻辑，都需要应用层响应来确认。即使写操作抛出异常，也未必能判断对端究竟收到了多少字节；因此重试策略不能简单建立在“写失败就一定没发送”之上。

`flush()` 的含义也取决于流包装层。对 `Socket` 原始输出流来说，写入已经直接进入底层实现，`flush()` 通常没有像文件或缓冲流那样显著的刷新效果；但如果外面包了 `BufferedOutputStream`、`BufferedWriter`、`PrintWriter` 或对象流，`flush()` 就可能决定数据是否离开 Java 层缓冲。文本协议中使用 `PrintWriter` 还要注意它默认吞掉部分 I/O 异常，需要检查 `checkError()`，否则写失败可能被静默隐藏。对于网络协议，推荐显式使用会抛出 `IOException` 的流 API，并把编码、换行、刷新时机写清楚。

读写阻塞还与缓冲、窗口和对端行为有关。读阻塞通常表示本端等待数据或等待 EOF；写阻塞则常见于本端发送速度超过对端接收速度，或者网络路径拥塞导致发送缓冲无法继续容纳更多数据。传统阻塞 `Socket` 只有读超时设置，没有对单次写调用完全对称的写超时。若业务要求“响应必须在指定时间内写完”，需要把写操作放到可取消的任务中并在超时后关闭 socket，或者采用 NIO、异步 I/O、成熟网络框架来管理待写队列和超时。关闭 socket 是打断阻塞读写的常见手段，但这也意味着连接整体终止，不能把它当作普通的局部取消。

固定长度读取可以用 `DataInputStream.readFully`，但它并不替你做协议保护。它会努力读满指定长度，如果对端提前关闭会抛出 `EOFException`，如果设置了读超时则可能抛出 `SocketTimeoutException`。真正安全的长度前缀读取应先读取长度字段，检查范围，再读取负载。对于大报文，还要考虑流式处理，不能所有消息都一次性读入堆内存。下面的代码展示的是最小边界，不是完整框架。

```java
static byte[] readFrame(DataInputStream in, int maxFrameSize) throws IOException {
    int length = in.readInt();
    if (length < 0 || length > maxFrameSize) {
        throw new ProtocolException("invalid frame length: " + length);
    }
    byte[] payload = new byte[length];
    in.readFully(payload);
    return payload;
}

static void writeFrame(DataOutputStream out, byte[] payload, int maxFrameSize) throws IOException {
    if (payload.length > maxFrameSize) {
        throw new ProtocolException("frame too large: " + payload.length);
    }
    out.writeInt(payload.length);
    out.write(payload);
    out.flush();
}
```

如果协议允许连续请求，解码器还要维护“已经读到但尚未组成完整消息”的缓冲。不能在一次 `read` 没拿到完整消息时丢弃已有字节，也不能在一次 `read` 拿到多个消息时只处理第一个。阻塞流写法通常把这个状态隐含在循环和局部变量里；NIO 写法会把它显式放在连接上下文中。无论哪种模型，核心原则都一样：输入是任意切分的字节片段，输出必须按协议帧边界恢复成完整消息。

## 应用层协议设计

Socket 传输的是字节，应用层协议负责解释字节。一个协议至少要定义消息边界、编码方式、字段顺序、错误表示、最大长度和连接生命周期。常见边界方案包括固定长度、分隔符、长度前缀和自描述格式。固定长度适合字段大小稳定的二进制协议；分隔符适合简单文本命令，但要处理转义和超长行；长度前缀适合大多数二进制或混合协议；自描述格式如 JSON 行协议可读性好，但仍需要边界，例如每行一个 JSON 或先写长度再写 JSON。

长度前缀协议要防止恶意或错误长度。读取长度字段后，必须检查是否为负数、是否超过最大允许值、是否与当前认证或业务状态匹配。不要根据对端声明的长度直接分配巨大数组。可以设置全局最大消息大小，也可以按消息类型设置限制。文本行协议要限制单行长度，防止对端一直发送非换行字符导致内存增长。协议错误应有明确处置：返回错误后关闭连接，还是丢弃当前消息继续处理，取决于协议能否恢复同步。

编码方式也要明确。文本协议应指定 UTF-8 或其他固定字符集，不应依赖平台默认编码。二进制协议要指定字节序、整数宽度、有符号还是无符号、浮点格式和时间表示。时间戳应说明单位和时区语义。布尔值、枚举、可选字段、空值、数组长度都要有明确表示。协议越是被多个版本或多个语言实现使用，越需要把这些细节写成稳定契约。

协议还要定义交互时序。最简单的是短连接请求-响应：客户端连接，发送一个请求，读取一个响应，然后关闭连接。它的状态少，排障容易，但连接建立成本高，吞吐和延迟可能受限。长连接顺序请求-响应可以复用连接，但必须规定同一连接上请求是否必须一个接一个完成，服务端是否可以主动推送，空闲多久关闭，关闭前是否发送告别帧。若允许请求并发复用，就必须引入请求 ID、响应 ID、乱序响应处理和流量控制，否则客户端无法判断某个响应属于哪个请求。

协议错误处理要尽量让双方重新回到明确状态。长度字段非法、消息类型未知、字段缺失、校验失败、状态不允许等错误，不应都被笼统处理成“关闭连接”。有些错误可以返回错误帧后继续处理下一帧，有些错误会破坏字节流同步，只能关闭连接。长度前缀协议如果长度字段已经可信但负载内容非法，通常还能跳过该帧继续；分隔符协议如果遇到超长行且还没看到分隔符，可能很难判断下一字节是否仍属于错误消息，此时关闭连接更简单可靠。协议文档应把这些恢复能力写出来，否则不同实现会产生不兼容行为。

版本演进也是协议设计的一部分。只要协议可能长期运行或被多个程序实现，就需要考虑字段扩展、能力协商和兼容策略。常见做法是在连接建立后先交换版本和能力位，或者在每个消息头中放入版本号、消息类型和保留字段。新增字段应尽量做到旧端可忽略，新端可判断缺省值；删除字段和改变字段语义要非常谨慎。不要把“当前只有一个客户端和一个服务端”当作不写版本的理由，因为 Socket 协议一旦落到文件、配置、第三方实现或长期部署环境中，升级顺序就很难完全同步。

文本协议和二进制协议各有代价。文本协议便于手工调试、日志记录和跨语言实现，但解析成本更高，转义规则和长度限制容易被忽略。二进制协议紧凑、类型明确、解析高效，但抓包阅读困难，调试工具和兼容策略需要额外建设。选择协议格式时，不应只比较“文本可读”或“二进制快”，还要看团队是否能稳定维护编码器、解码器、测试样例、兼容性文档和错误诊断工具。

## 阻塞 Socket 的资源管理

阻塞 Socket 编程中，每个连接通常由一个线程处理。线程在 `read` 上等待请求，在业务逻辑中处理，在 `write` 上发送响应。代码结构清晰，但资源边界要严格。每个连接都占用文件描述符、内核 socket 缓冲区、Java 对象和可能的线程。连接泄漏会导致文件描述符耗尽；线程泄漏会导致内存和调度压力；未关闭的输出流可能让对端一直等待响应结束。

try-with-resources 是管理客户端连接和短生命周期服务端连接的默认方式。对于长连接，不能简单在每次请求后关闭 socket，而要定义连接循环和退出条件。退出条件包括读到 EOF、协议要求关闭、空闲超时、最大请求数、服务停机、异常或对端重置。无论哪种路径，都应收敛到统一关闭逻辑，确保 socket、输入输出包装流、连接注册表和业务状态都被清理。

`setSoTimeout` 是阻塞读的基本保护。没有读超时的服务端容易被慢客户端占满线程。超时后是否关闭连接取决于协议：对于请求-响应短连接，通常关闭；对于长连接，可能允许若干次空闲超时或依赖心跳。还可以设置 `setKeepAlive` 开启 TCP keepalive，但它是操作系统级空闲探测，默认间隔通常较长，不适合作为业务实时心跳。业务层心跳应由协议定义，包含发送周期、超时次数和误判处理。

资源管理还包括包装流和辅助对象的所有权。通常让 `Socket` 成为连接生命周期的根资源即可，输入流、输出流和各种 reader/writer 只在连接处理范围内使用。不要把同一个 socket 的输入流交给多个组件长期持有，也不要在业务层随意关闭传入的流，除非接口契约明确说明调用者转移了所有权。更稳妥的做法是封装一个连接上下文，对外提供 `send`、`close`、`remoteAddress`、`lastReadTime` 等受控操作，避免业务代码直接操作底层流。

关闭逻辑要能处理重复调用和并发调用。一个线程可能因为读到 EOF 准备关闭连接，另一个线程可能因为业务超时也准备关闭连接，第三个线程可能正在发送响应。关闭方法应当是幂等的：第一次执行真实关闭和状态清理，后续调用只返回。连接状态可以用锁、原子变量或事件循环串行化维护，但必须避免“关闭时还在写，写失败又再次关闭，关闭回调再次写日志或发消息”的循环。日志也要控制粒度，正常 EOF、业务空闲关闭和异常连接重置不应都按高严重级别刷屏。

服务端还要防止资源被慢客户端长期占用。慢客户端可能很久才发完请求头，也可能读响应很慢导致服务端写缓冲堆积。读超时解决的是“等输入太久”，不解决“输出无人消费”。在阻塞线程模型里，写慢会占住工作线程；在事件驱动模型里，写慢会扩大待写队列。无论模型如何，都应设置最大请求读取时间、最大响应排队大小、最大空闲时间和每连接最大未完成请求数。限制不是为了拒绝正常用户，而是让异常连接无法拖垮整个进程。

## 关闭、半关闭与 TIME_WAIT

Socket 关闭比“调用 close”更细。TCP 是全双工连接，一端可以关闭输出方向，表示不再发送数据，但仍可读取对端数据。Java `Socket.shutdownOutput()` 对应这种半关闭，常用于发送完请求体后通知对端 EOF，再继续等待响应。`shutdownInput()` 表示本端不再读取输入。半关闭必须由协议支持，否则对端可能把它理解为连接结束或错误。

正常关闭通常经历 FIN 交换；异常关闭可能发送 RST。应用层如果在还有未读数据时直接关闭，或设置了特殊 linger 行为，可能导致对端看到连接重置。`setSoLinger` 可以影响 close 行为，但不应随意使用。许多程序试图用 linger 解决“立即释放端口”或“确保数据发出”的问题，结果引入阻塞 close、RST 或平台差异。一般情况下，保持默认关闭语义，设计好协议结束条件和超时，比调节 linger 更可靠。

`TIME_WAIT` 是 TCP 正常连接关闭后可能出现的状态，常见于主动关闭的一方。它用于确保旧连接的延迟报文不会影响新连接，并允许最后的 ACK 丢失时重传。大量短连接可能导致许多 TIME_WAIT 连接，占用本地端口和内核资源。解决思路不是粗暴消灭 TIME_WAIT，而是减少不必要短连接、使用连接复用、合理设置客户端并发和超时、理解哪一方主动关闭。端口复用选项如 `SO_REUSEADDR` 有特定语义，不能把它当作所有端口占用问题的通用解法。

## TCP 选项与缓冲

Java `Socket` 暴露了一些常见 TCP 选项。`setTcpNoDelay(true)` 可以禁用 Nagle 算法，让小包尽快发送，适合低延迟交互协议；默认 Nagle 会尝试合并小包，提高网络利用率，但可能与延迟确认叠加造成小消息延迟。是否禁用要根据协议特征和实测决定。频繁发送很小消息的程序还应考虑应用层批量合并，而不是只依赖 TCP 选项。

`setSendBufferSize` 和 `setReceiveBufferSize` 提示内核发送和接收缓冲区大小。较大缓冲区有利于高带宽高延迟链路吞吐，但会增加内存占用；较小缓冲区可能降低延迟感知和内存占用，但容易限制吞吐。操作系统可能调整或限制实际大小。应用层缓冲、JVM 堆内存、内核缓冲和网络拥塞窗口共同影响传输表现，不能只看一个参数。

`setReuseAddress` 在服务端重启绑定端口时常被提到，但其行为依赖平台和 socket 状态。它不意味着可以任意多个服务同时监听同一地址端口，也不应作为解决旧进程未退出的手段。`setTrafficClass`、`setOOBInline` 等选项更少使用，涉及网络设备和协议细节，普通应用应谨慎。

## UDP Socket 编程

UDP 通过 `DatagramSocket` 或 `DatagramChannel` 发送数据报。每个数据报有独立边界，接收端一次 `receive` 得到一个报文；如果接收缓冲区太小，超出部分可能被截断。UDP 不建立连接，不保证送达、顺序或唯一性，也没有 TCP 那样的流控和拥塞控制保障。应用如果需要可靠性，必须自己设计序号、确认、重传、去重、超时和拥塞控制；如果不需要可靠性，也要能容忍丢包和乱序。

```java
try (DatagramSocket socket = new DatagramSocket()) {
    byte[] payload = "hello".getBytes(StandardCharsets.UTF_8);
    DatagramPacket packet = new DatagramPacket(
            payload, payload.length, InetAddress.getByName("127.0.0.1"), 9000);
    socket.send(packet);
}
```

UDP 适合对实时性要求高、能容忍部分丢失或由应用层控制可靠性的场景，例如某些探测、广播发现、实时状态上报和自定义协议。它不适合简单替代 TCP 来“提高性能”。没有连接建立成本不代表总是更快；一旦应用层重新实现可靠传输、拥塞控制和分片重组，复杂度会迅速上升。UDP 报文大小还受到 MTU 影响，过大报文会分片，分片丢失会导致整个报文不可用。实践中应控制单个 UDP 载荷大小，并设计丢包容忍策略。

`DatagramSocket.connect` 并不会建立 TCP 式连接，它只是固定远端地址，之后只能与该地址通信，并可能让本地更早收到某些 ICMP 错误。接收 UDP 时，应检查来源地址和端口，尤其是服务端可能收到任意来源的数据报。不要把 UDP 来源伪造问题忽略在协议之外，涉及控制指令或敏感数据时需要认证和完整性校验。

## DNS、地址与端口

Socket 编程离不开地址解析。`InetAddress.getByName` 可能触发 DNS 查询，查询耗时、缓存和失败都会影响连接建立。一个主机名可能解析出多个 IPv4 或 IPv6 地址，连接策略会影响可用性和延迟。Java 和操作系统都有 DNS 缓存策略，域名变更不一定立刻反映到长时间运行的进程。高可用客户端通常需要处理多地址重试、解析失败退避和连接池中旧地址淘汰。

端口分为服务端监听端口和客户端临时端口。客户端主动连接时，如果不显式绑定本地端口，系统会分配临时端口。大量短连接可能耗尽临时端口范围，尤其是连接目标相同且 TIME_WAIT 较多时。服务端也要区分监听地址：绑定 `0.0.0.0` 表示监听所有 IPv4 地址，绑定 `127.0.0.1` 只接受本机回环访问，绑定具体网卡地址则只接受对应路径。地址绑定是安全边界的一部分，不能随意使用所有地址监听。

## 并发模型与连接管理

Socket 服务端的并发模型决定资源消耗和复杂度。每连接一线程最直观，但面对大量空闲连接不经济；线程池可以限制工作线程，但不能消除阻塞等待；NIO selector 可以用少量线程管理大量连接，但要求事件驱动状态机；异步通道和网络框架进一步封装了事件循环和回调。选择模型要看连接数量、消息频率、业务耗时、团队维护能力和延迟目标。

连接池是客户端常见优化。建立 TCP 连接需要握手和资源分配，频繁短连接会增加延迟和系统开销。连接复用可以提高效率，但也带来空闲检测、最大连接数、连接健康、并发使用限制和服务端关闭处理。一个连接是否可以被多个请求并发复用，取决于应用层协议是否支持多路复用或请求 ID。普通顺序请求-响应协议如果多个线程同时写同一 socket，很容易造成字节交错和响应归属混乱。

服务端连接管理应维护当前连接数、每个远端的连接数、空闲时间、待写字节数、认证状态和关闭原因。过载时要有策略：拒绝新连接、缩短空闲超时、暂停读取、返回错误、限速或关闭低优先级连接。没有边界的连接管理会让异常客户端或流量峰值把内存和线程耗尽。

## 异常与故障诊断

Socket 异常往往带有环境特征。`ConnectException: Connection refused` 表示目标地址可达但端口没有服务或被拒绝；`SocketTimeoutException` 可能出现在连接或读取阶段，含义取决于调用点；`UnknownHostException` 表示域名无法解析；`NoRouteToHostException` 表示路由不可达；`SocketException: Connection reset` 常见于对端异常关闭或 RST；`Broken pipe` 常见于向已关闭连接写入。诊断时要记录本地地址、远端地址、连接阶段、超时配置、请求标识和重试次数。

网络故障具有不确定性。连接可能在任何时刻断开，写成功不代表对端应用已经处理，读超时不代表对端永远不可用，重试可能造成重复请求。应用协议需要幂等性设计、请求 ID、去重或事务边界。对于非幂等操作，简单超时重试可能产生重复提交。客户端库应区分连接建立失败、请求未发出、请求已发出但未收到响应等状态，并把不确定性暴露给上层决策。

排查网络问题时，可以结合应用日志、线程 dump、连接状态、抓包和系统指标。线程阻塞在 `SocketInputStream.read` 表示等待对端数据；阻塞在写入可能表示对端接收慢或网络拥塞；大量 `CLOSE_WAIT` 通常说明本端收到对端关闭后没有及时 close；大量 `SYN_SENT` 可能是连接目标不可达或被防火墙丢弃；大量 `TIME_WAIT` 多见于短连接主动关闭。应用层日志与系统状态结合，才能避免只从异常文本猜测。

## 常见误区

第一个误区是认为 TCP 是消息协议。TCP 是字节流，应用层必须处理半包、粘包和消息长度。

第二个误区是没有超时。连接超时、读超时、空闲超时和业务超时分别解决不同问题，不能只设置一个或完全不设置。

第三个误区是多个线程直接写同一个 socket。除非协议和写入队列明确保证顺序与帧边界，否则会造成数据交错。

第四个误区是把 TCP keepalive 当成业务心跳。系统级 keepalive 间隔通常较长，不能替代协议层健康检测。

第五个误区是无限制读取对端声明的数据长度。必须设置最大消息大小和异常关闭策略。

第六个误区是把 UDP 当成更快的 TCP。UDP 省掉的是传输层可靠性和连接管理，不是省掉复杂性；可靠性需求会回到应用层。

## 实践建议

编写 Socket 程序时，先写协议文档，再写代码。明确消息边界、最大长度、编码、错误码、连接关闭规则、心跳和重试语义。客户端要设置连接和读超时，重试要有上限、退避和幂等判断。服务端要限制线程、连接数、请求大小、空闲时间和待写队列。所有连接关闭路径要统一，日志要记录远端地址和关闭原因。阻塞模型适合低并发或简单程序，高连接数和慢客户端场景应考虑 NIO 或成熟网络框架。

性能调优要用数据验证。小消息延迟高时，可以评估 `TCP_NODELAY`、应用层批量策略和 flush 时机；吞吐不足时，观察缓冲区、窗口、CPU 和业务处理耗时；连接异常多时，检查超时、负载均衡、服务端 backlog、端口耗尽和防火墙策略。Socket 编程跨越应用、JVM、操作系统和网络设备，单一参数很少能解释所有问题。

协议测试应覆盖比“正常请求正常响应”更多的情况。客户端提前断开、服务端提前关闭、请求只发送一半、长度字段声明过大、连续发送多个请求、响应被拆成多段、读超时后重试、同一连接上发生协议错误，这些都应该被纳入测试。对于 TCP，测试要刻意模拟半包和多包合并，因为本机快速通信很容易让错误实现看起来正常。可以把编码器和解码器拆成纯字节层组件，用任意切分的输入片段喂给解码器，确认它只在完整消息到达时产出结果，并能保留未完成数据。

上线后的可观测性也应围绕连接生命周期设计。只统计请求成功率不够，还要知道连接建立失败、读超时、写失败、远端关闭、协议错误、心跳超时、拒绝连接、当前连接数、空闲连接数、待写队列长度和每类关闭原因。日志中最好能区分本端主动关闭和对端关闭，区分业务决定关闭和底层异常关闭。没有这些信息时，Socket 问题往往会被笼统归为“网络不稳定”，实际可能是协议长度错误、线程池耗尽、写队列积压或服务端主动限流。

安全层面也不能忽略。明文 Socket 只能提供传输能力，不提供机密性、完整性和身份认证。需要保护内容时，应使用 TLS 或在协议层加入经过审查的认证与加密设计，而不是自制简单混淆。即使在受控网络中，也应校验输入长度、字段范围和状态转换，避免把内部网络等同于可信输入。Socket 程序越接近基础设施，越需要把异常输入当作常态处理。

## 小结

Socket 编程的核心是用传输层能力构建应用层协议。TCP 提供可靠有序字节流，但不提供消息边界和业务成功语义；UDP 提供数据报边界和低开销发送，但不保证可靠性。Java 的 Socket API 足够直接，也足够底层，要求开发者明确超时、关闭、缓冲、异常、并发和安全边界。写出稳定网络程序的关键，不是记住某个构造方法，而是承认网络不可靠、读写会部分完成、连接状态会变化、对端可能很慢或异常，并把这些情况设计进协议和资源管理之中。

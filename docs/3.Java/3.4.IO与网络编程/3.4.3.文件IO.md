# 3.4.3 文件IO

文件 I/O 是 Java 程序与持久化数据、配置、日志、缓存、导入导出文件交互的基础能力。它看起来只是“读文件、写文件”，实际包含路径表达、文件系统语义、字节与字符转换、缓冲、随机访问、原子更新、异常恢复、权限和并发访问等一系列问题。Java 提供了多代文件 I/O API：早期的 `FileInputStream`、`FileOutputStream`、`Reader`、`Writer`、`RandomAccessFile` 和 `java.io.File`；NIO 中的 `FileChannel`、`ByteBuffer`；Java 7 之后的 `Path`、`Files`、`FileSystem`、`FileVisitor` 等。它们并不是简单的新旧替代关系，而是在不同层次上解决不同问题。

学习文件 I/O 的关键不是记住所有类名，而是建立几个边界意识：文件内容本质上是字节，文本只是按字符集解释字节后的结果；路径不是普通字符串，它受文件系统、工作目录、符号链接和权限影响；一次读写不一定完成预期数量，写入成功也不一定表示数据已经安全落盘；关闭资源不是形式主义，而是释放文件描述符、刷新缓冲和解除文件占用的必要动作；文件更新如果需要可靠性，必须考虑崩溃发生在任意一步时的状态。

## 路径与文件对象

早期 Java 使用 `java.io.File` 表示文件或目录路径。`File` 这个名字容易误导，它并不代表已经打开的文件，也不保证路径存在；它只是一个路径名的抽象。调用 `exists()`、`isFile()`、`isDirectory()`、`length()` 等方法时，才会向文件系统查询状态。由于文件系统状态可能被其他进程或线程同时改变，查询结果只能作为某一时刻的观察，不能作为后续操作一定成功的保证。例如先判断文件存在再打开，中间文件可能被删除；先判断目录不存在再创建，中间可能已有其他进程创建。

现代代码更推荐使用 `Path` 和 `Files`。`Path` 表示路径，提供 `resolve`、`normalize`、`relativize`、`toAbsolutePath` 等路径操作；`Files` 提供读写、复制、移动、删除、属性查询、目录遍历等静态方法。与 `File` 相比，`Path` 更贴近不同文件系统，异常信息也更具体。比如 `Files.delete(path)` 在文件不存在时抛出 `NoSuchFileException`，权限不足时抛出 `AccessDeniedException`，目标是非空目录时抛出 `DirectoryNotEmptyException`，比简单返回布尔值更利于诊断。

路径处理不要依赖字符串拼接。不同系统的分隔符、根路径、相对路径规则和符号链接都会让字符串拼接变脆弱。应使用 `base.resolve(child)` 构造子路径，必要时用 `normalize()` 消除 `.` 和 `..`。如果程序接收外部输入作为文件名，还要防止路径穿越：不能只检查字符串是否包含 `..`，更稳妥的做法是把目标路径解析为规范化的绝对路径，再确认它仍位于允许的根目录下。符号链接会让这个问题更复杂，涉及安全边界时还要决定是否跟随链接。

`Path` 的几个常用转换也要分清语义。`toAbsolutePath()` 只是把相对路径按当前工作目录展开，不要求路径真实存在，也不会解析符号链接；`normalize()` 只在路径文本层面折叠 `.` 和 `..`；`toRealPath()` 会访问文件系统，要求路径存在，并按选项决定是否解析符号链接。也就是说，`normalize()` 得到的是“更规整的路径表达”，`toRealPath()` 得到的是“文件系统当前确认过的真实路径”。如果程序要判断某个用户给出的路径是否在工作目录内，通常不能只做字符串前缀判断，而应先把根目录和目标路径转成合适的绝对路径或真实路径，再用 `startsWith` 判断路径层级关系。即便如此，检查与打开之间仍可能发生竞态，所以安全敏感的文件访问还需要限制可操作的目录、文件名规则和打开方式。

`Path` 也不局限于默认磁盘文件系统。Java 的 NIO 抽象允许不同 `FileSystem` 实现，例如压缩包文件系统、自定义内存文件系统或测试中使用的虚拟文件系统。多数业务代码只接触默认文件系统，但使用 `Path` 和 `Files` 而不是字符串，能让代码更容易被测试、更容易迁移到非默认文件系统，也更容易表达“路径计算”和“文件系统访问”的边界。一个方法如果只需要拼接路径、计算相对关系，就可以只接收 `Path` 而不立即访问文件系统；只有真正读取、写入、创建或查询属性时，才需要处理 `IOException`。

`java.io.File` 与 `Path` 之间可以互相转换：`file.toPath()` 和 `path.toFile()`。但并不是所有 `Path` 都能转成 `File`，只有默认文件系统中的路径才一定适合 `toFile()`。新代码如果没有兼容旧 API 的压力，优先把 `Path` 作为内部模型；遇到旧库只接受 `File` 时，再在边界处转换。这样做能避免把旧 API 的布尔返回值、弱异常信息和字符串式路径操作扩散到整段文件处理逻辑中。

## 字节流与字符流

文件内容在磁盘上是字节。`InputStream`、`OutputStream`、`FileInputStream`、`FileOutputStream` 面向字节，适合图片、压缩包、二进制协议、加密数据和未知编码文件。`Reader`、`Writer` 面向字符，适合文本文件。字符流并不是直接从磁盘读取字符，而是在字节流之上使用字符集解码或编码。这个转换过程必须明确字符集，否则程序会依赖平台默认编码，导致跨环境乱码。

```java
Path path = Path.of("config.txt");
try (BufferedReader reader = Files.newBufferedReader(path, StandardCharsets.UTF_8)) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
```

这段代码的重点不在打印，而在三个习惯：使用 `Path` 表示路径，显式指定 UTF-8，用 try-with-resources 关闭资源。`readLine()` 会去掉行分隔符，如果需要保留原始换行或区分 `\n`、`\r\n`，就不能只依赖它。读取文本还要考虑 BOM、非法字节序列、超长行和大文件内存占用。`Files.readString`、`Files.readAllLines` 很方便，但会把内容整体读入内存，适合小文件，不适合未知大小或大文件。

写文本时同样要明确字符集和打开选项。`Files.newBufferedWriter(path, charset, options...)` 可以指定 `CREATE`、`TRUNCATE_EXISTING`、`APPEND`、`CREATE_NEW` 等选项。追加日志和覆盖配置文件的语义完全不同。`APPEND` 表示写到文件末尾，但多线程或多进程同时追加时，单次写入的原子性受系统和写入大小影响，不能天然保证复杂记录不会交错。覆盖写入更要小心，直接 `TRUNCATE_EXISTING` 后如果程序崩溃，原文件内容可能已经丢失而新内容尚未完整写完。

字符集错误处理也值得单独注意。简单使用 `Files.newBufferedReader(path, StandardCharsets.UTF_8)` 时，底层解码器通常会在遇到非法字节序列时抛出异常或替换字符，具体行为取决于 API 包装方式和解码器配置。对于配置文件、协议文件、数据导入文件，静默替换非法字符可能掩盖数据损坏；对于用户上传的普通文本，过于严格又可能降低容错性。需要精确控制时，可以使用 `CharsetDecoder` 设置 `CodingErrorAction.REPORT`、`REPLACE` 或 `IGNORE`，并明确记录处理策略。不要把“能读出字符串”误认为“文本是正确的”，因为损坏字节、错误编码和意外 BOM 都可能在后续解析阶段才暴露问题。

文本文件还涉及行分隔符和换行语义。Java 字符串中的 `\n`、系统属性 `line.separator`、`BufferedWriter.newLine()` 和已有文件中的 `\r\n`、`\n` 并不总是同一回事。跨平台生成文本时，如果文件格式规定了换行，应按格式写入；如果只是给本机用户阅读，使用 `newLine()` 或系统行分隔符更自然。逐行读取时，`readLine()` 不返回行分隔符，因此它适合处理“按行解释”的内容，不适合做保留字节级格式的转换。需要原样复制文本文件时，应按字节复制；需要规范化换行时，应把这是一次格式转换讲清楚，而不是把它隐藏在读取和写入之间。

字节流与字符流之间的桥梁是 `InputStreamReader` 和 `OutputStreamWriter`。这两个类看似只是适配器，实际承担着编码和解码责任。一个常见问题是混用层次：先用字节流读取一部分字节，再把同一个流交给 `Reader`，如果前面读取破坏了字符边界，后续解码可能出错；或者对同一个输出目标既写字节又写字符，导致文件格式不稳定。除非文件格式本身就是“头部二进制、后部文本”这类明确设计，否则应在一个文件处理流程内保持同一抽象层次：要么按字节处理，要么按字符处理。

## 打开选项与写入语义

文件写入的行为很大程度由打开选项决定。`StandardOpenOption.CREATE` 表示不存在就创建，存在则继续按其他选项处理；`CREATE_NEW` 表示必须新建，如果文件已存在就失败，适合避免覆盖；`TRUNCATE_EXISTING` 会在打开时截断已有内容；`APPEND` 会让每次写入追加到末尾；`WRITE`、`READ` 表示访问方向；`SYNC` 和 `DSYNC` 要求更强的同步写入语义。很多 bug 并不是写入代码本身错了，而是打开选项表达了错误的业务语义。

例如生成导出文件时，通常不希望悄悄覆盖同名文件，可以使用 `CREATE_NEW`；更新缓存文件时，可以接受覆盖，但要能在失败后重建；更新关键状态文件时，直接截断写入风险较高，应采用临时文件替换；追加审计记录时，要考虑每条记录是否由一次底层写入完成，以及是否需要同步落盘。API 的默认行为并不等于业务上正确，文件 I/O 代码应把“允许覆盖吗”“允许不存在吗”“失败后旧文件是否必须保留”“写入成功是否要求落盘”这些问题显式化。

读取也有语义差异。`Files.readAllBytes`、`Files.readString`、`Files.readAllLines` 把整个文件读入内存，代码短但隐含了“文件足够小”的前提。`Files.newInputStream`、`newBufferedReader` 和 `FileChannel` 则允许流式处理。一个库方法如果接收任意路径，不应轻易用整体读取；如果业务规则明确限制文件大小，可以先检查大小并在超限时拒绝，再整体读取。检查大小不能作为绝对保证，因为读取前后文件可能变化，但它仍然是保护内存的有效前置条件。真正需要强约束时，应在读取过程中累计已读字节数，超过上限立即停止。

写入时还要区分“覆盖文件内容”和“替换文件对象”。`TRUNCATE_EXISTING` 是在原路径对应的文件上改内容；临时文件加 `move` 是让路径最终指向另一个已经写好的文件。前者可能保留部分文件属性、权限和硬链接关系，也可能在崩溃时留下半成品；后者更利于原子替换，但可能改变文件的创建时间、部分属性或链接关系。哪一种更合适取决于业务：修改一个普通配置文件通常偏向替换；维护一个被其他组件持有打开句柄的大文件时，替换路径未必影响已有句柄，反而可能不是想要的效果。

文件删除也不是“把路径从世界上抹掉”这么简单。`Files.delete` 删除路径项，若文件被其他进程打开，不同平台的表现可能不同：有的平台允许目录项删除但已打开句柄仍能访问内容，有的平台会因为占用而失败。`deleteIfExists` 只是在文件不存在时不抛异常，并不表示其他失败会被忽略。删除用于清理临时文件时，可以容忍失败并记录；删除用户数据或关键状态时，则需要明确错误处理和重试策略。

## 缓冲与批量读写

缓冲是文件 I/O 性能的基本手段。没有缓冲时，程序如果每次读写一个字节或一个字符，会产生大量方法调用和系统调用，性能很差。`BufferedInputStream`、`BufferedOutputStream`、`BufferedReader`、`BufferedWriter` 在用户态维护一段缓冲区，把小块操作合并为较大块。`Files.newBufferedReader` 和 `Files.newBufferedWriter` 已经返回带缓冲的字符流；字节复制则可以显式使用缓冲数组。

```java
static void copy(Path source, Path target) throws IOException {
    try (InputStream in = new BufferedInputStream(Files.newInputStream(source));
         OutputStream out = new BufferedOutputStream(Files.newOutputStream(target))) {
        byte[] buffer = new byte[8192];
        int n;
        while ((n = in.read(buffer)) != -1) {
            out.write(buffer, 0, n);
        }
    }
}
```

这里必须使用 `out.write(buffer, 0, n)`，不能直接写整个数组，因为最后一次读取通常不会填满缓冲区。缓冲区大小没有唯一最优值。8KB 是常见起点，较大的 32KB、64KB 在大文件顺序读写时可能更好，但过大缓冲会增加内存占用，尤其是大量并发文件操作时。性能优化应通过实际基准测试决定，而不是凭直觉把缓冲区调到很大。

`flush()` 表示把用户态缓冲推送到底层流，但不等价于数据已经持久化到物理介质。操作系统还会有页缓存，磁盘控制器也可能有缓存。如果业务需要崩溃后尽量保证数据落盘，需要使用 `FileChannel.force(true)` 或相关同步选项，并理解这会显著影响性能。日志、数据库、状态文件和普通临时文件对落盘保证的要求不同，不能用同一种策略处理所有写入。

`InputStream.read(byte[])` 的返回值也经常被误解。它最多读取数组长度那么多字节，但可以少读；返回 `-1` 才表示流结束。对于文件流，顺序读取通常会尽量填满缓冲，但 API 合约并不保证每次都填满。网络流、管道流和某些包装流更容易出现短读。写文件时，`OutputStream.write(byte[], off, len)` 通常会尝试写完整个区间，失败则抛异常；而 `FileChannel.write(ByteBuffer)` 明确可能只写入一部分，需要根据 `ByteBuffer.hasRemaining()` 循环。理解“部分读写”是写出健壮 I/O 代码的基础。

缓冲层还有一个所有权问题。包装流关闭时会关闭被包装的底层流，`BufferedOutputStream.close()` 会先刷新再关闭底层输出流。方法内部如果自己创建了底层文件流，就应该自己关闭最外层包装对象；如果底层流由调用方传入，方法一般只负责写入和刷新，不应该擅自关闭，除非 API 文档明确转移所有权。这个边界在工具方法中尤其重要，否则一个看似方便的写入函数可能让调用方后续无法继续使用同一个压缩流、加密流或网络流。

缓冲不是越多越好。`BufferedReader` 上再套一个 `BufferedReader` 没有意义；对已经内部缓冲的组件重复包装，可能只增加内存和延迟。相反，完全没有缓冲的小块读写会放大系统调用成本。比较稳妥的做法是：在文件边界处建立一层明确的缓冲；在解析层、编码层和业务层之间传递合适的抽象；避免每一层都为了“保险”再套一层缓冲。性能问题出现后，再通过采样、指标和基准测试确认瓶颈是在系统调用、编码解码、磁盘、锁竞争还是内存分配。

## 随机访问与 FileChannel

顺序流适合从头到尾读写。需要跳到指定位置读写时，可以使用 `RandomAccessFile` 或 `FileChannel`。`RandomAccessFile` 支持 `seek`、`read`、`write`，可以打开为只读或读写模式。它常用于固定格式文件、索引文件、断点续传、局部更新等场景。要注意随机写可能覆盖已有内容，不会自动插入字节；写入超过文件末尾会扩展文件，中间空洞在某些文件系统上可能表现为稀疏文件。

`FileChannel` 提供更现代的能力：按位置读写、获取文件大小、截断文件、锁定文件区域、在通道之间传输数据、内存映射等。`FileChannel.read(buffer, position)` 可以在不改变通道当前位置的情况下读取指定位置，这对并发读取和索引访问很有用。`transferTo`、`transferFrom` 适合大文件复制或文件到网络通道传输，但一次调用不保证完成全部传输，必须循环处理返回值。

内存映射文件通过 `FileChannel.map` 把文件区域映射到内存，返回 `MappedByteBuffer`。程序可以像访问内存一样访问文件内容，操作系统按页加载和写回。它适合大文件随机读取、索引结构和需要减少显式 read/write 调用的场景。但内存映射也有边界：映射区域大小受地址空间限制；文件被截断后访问映射区域可能出现异常；解除映射的时机不如普通流关闭那样直观；写回仍然涉及操作系统策略，必要时要调用 `force`。因此内存映射不是普通文件读写的无脑替代。

`FileChannel` 与流之间可以互相配合，但要注意当前位置。通过 `FileInputStream.getChannel()` 获得的通道与原始流共享文件位置：流读了一部分，通道的位置也会变化；通道改变位置，流后续读取也会受影响。混用时如果没有严格设计，很容易出现跳读、重复读或覆盖写。独立的位置读写方法 `read(buffer, position)` 和 `write(buffer, position)` 不改变通道当前位置，适合并发读取固定区域，但并不自动解决多个线程更新同一区域时的数据竞争。只要存在共享可变文件内容，就仍然需要上层协议、锁或单写模型。

`ByteBuffer` 本身也有使用门槛。写入缓冲区后要 `flip()` 才能让通道从缓冲区读出；通道读入缓冲区后要根据 `position`、`limit`、`remaining` 判断还有多少数据；处理完部分数据后可能需要 `compact()` 保留未处理字节。很多 NIO 代码出错不是因为文件 API，而是因为缓冲区状态切换不清楚。对于普通文件复制和文本处理，传统流 API 更直接；只有当位置读写、通道传输、内存映射或与其他 NIO 通道协作能带来清晰收益时，才需要主动引入 `FileChannel`。

## 文件属性、权限与链接

文件不仅有内容，还有属性。常见属性包括大小、最后修改时间、是否目录、是否普通文件、是否符号链接、权限、所有者以及文件系统相关的扩展属性。`Files.readAttributes` 可以一次读取一组属性，减少多次查询；`BasicFileAttributes` 提供跨平台基础信息；在支持的系统上还可以使用 `PosixFileAttributes`、`DosFileAttributes`、`AclFileAttributeView` 等视图。属性 API 的设计说明了一点：不同文件系统的能力并不相同，跨平台代码只能依赖最小公共语义，平台特定能力要做能力检测和降级处理。

`Files.isRegularFile`、`isDirectory`、`isSymbolicLink` 等方法默认会按各自规则处理链接。判断符号链接时要特别谨慎，因为链接本身和链接指向的目标是两个层次。删除符号链接通常删除链接项，不删除目标；复制符号链接可以复制链接本身，也可以复制它指向的内容，取决于选项；遍历目录时跟随链接可能走出原目录树。安全边界内的文件操作最好明确是否允许符号链接，而不是依赖默认行为。对外部可写目录执行清理、打包或加载配置时，符号链接是必须考虑的边界。

文件时间戳也不能过度解读。最后修改时间的精度受文件系统影响，可能是纳秒、毫秒或更粗；复制文件时是否保留时间取决于选项和权限；只比较时间戳判断文件是否变化可能漏掉快速连续修改，也可能因为时钟、挂载方式或远程文件系统而误判。需要可靠判断内容变化时，可以结合文件大小、修改时间、版本号、校验和或内容摘要。时间戳适合做缓存优化的线索，不适合做强一致判断的唯一依据。

权限模型同样有平台差异。`Files.isReadable`、`isWritable`、`isExecutable` 是便捷检查，但检查结果只代表当前时刻和当前进程视角，后续打开仍可能失败。先检查再操作容易产生竞态，实际代码通常应直接执行目标操作并处理异常。创建文件时，如果需要限制权限，应尽量在创建时指定属性，而不是先创建宽权限文件再修改；后者在并发或安全敏感环境下会出现短暂暴露窗口。跨平台库如果要设置权限，应把“无法设置精确权限”作为可预期情况处理，而不是假设所有系统都支持同一套权限位。

`Files.copy` 和 `Files.move` 对属性的处理也要看选项。`COPY_ATTRIBUTES` 请求复制属性，但不保证所有属性都能被复制；`REPLACE_EXISTING` 允许覆盖目标；`ATOMIC_MOVE` 请求原子移动；`NOFOLLOW_LINKS` 影响链接处理。复制一个普通文件看似只有内容，实际可能涉及权限、时间戳、所有者、ACL、扩展属性和符号链接语义。文档型工具、备份工具和构建工具通常要比普通业务程序更重视这些细节，因为它们处理的是文件对象本身，而不仅是文件中的业务数据。

## 原子更新与崩溃恢复

许多文件写入看似简单，真正困难的是崩溃恢复。假设程序要更新一个配置文件，直接打开目标文件并覆盖写入，如果写到一半进程退出，目标文件会变成半截内容。更常见的可靠做法是：先在同一目录写入临时文件；写完后刷新并尽可能强制落盘；再使用原子移动替换目标文件。临时文件与目标文件放在同一目录，是为了让移动操作在同一文件系统内完成，增加原子替换的可能性。

```java
static void writeAtomically(Path target, String content) throws IOException {
    Path dir = target.toAbsolutePath().getParent();
    Path tmp = Files.createTempFile(dir, target.getFileName().toString(), ".tmp");
    try {
        Files.writeString(tmp, content, StandardCharsets.UTF_8,
                StandardOpenOption.TRUNCATE_EXISTING);
        try (FileChannel channel = FileChannel.open(tmp, StandardOpenOption.WRITE)) {
            channel.force(true);
        }
        Files.move(tmp, target,
                StandardCopyOption.REPLACE_EXISTING,
                StandardCopyOption.ATOMIC_MOVE);
    } catch (IOException e) {
        try {
            Files.deleteIfExists(tmp);
        } catch (IOException suppressed) {
            e.addSuppressed(suppressed);
        }
        throw e;
    }
}
```

`ATOMIC_MOVE` 如果文件系统不支持，会抛出 `AtomicMoveNotSupportedException`。是否降级为普通移动取决于业务要求：如果文件必须始终保持旧版本或新版本之一，就不应静默降级；如果只是缓存文件，可以接受普通替换。即使执行了文件 `force`，目录项本身是否持久化还与平台有关，极端可靠性要求下还要考虑目录 fsync。普通应用未必需要做到数据库级保证，但必须知道“写方法返回成功”和“断电后一定存在完整新文件”不是同一个承诺。

临时文件策略也影响可靠性和安全性。`Files.createTempFile` 会创建一个实际文件，避免单纯“生成一个临时文件名”带来的竞争；临时文件最好放在目标文件同一目录，既便于同文件系统原子移动，也能继承相近的权限和空间限制。临时文件命名不要暴露敏感信息，清理失败要可接受并可追踪。程序异常退出后遗留的临时文件不一定是错误，但应有可识别的后缀、版本或清理策略，避免长期堆积。

原子替换还涉及读者协议。写者用临时文件加移动可以避免读者看到半截目标文件，但前提是读者只读取正式文件名，不主动扫描并读取临时文件；读者如果需要判断文件是否完整，可以使用校验和、长度字段、版本字段或结束标记。对于多文件状态，例如一个目录中有数据文件和索引文件，单个文件的原子替换不等于整个状态的原子更新。常见做法是引入清单文件或版本目录：先写完整的新版本目录，最后原子切换一个指针文件或目录名，让读者始终看到某个完整版本。

崩溃恢复要从任意一步失败来推演。临时文件创建成功但内容未写完，重启后应删除或忽略；内容写完但移动前崩溃，旧文件仍在，临时文件可清理；移动成功但清理旧辅助文件失败，应能继续使用新文件；移动不支持原子语义时，应根据业务选择失败、降级或采用更高层事务机制。只有把这些状态列清楚，文件写入才算有恢复策略。否则代码在正常路径上看起来简洁，真正出问题时却无法判断磁盘上留下的文件属于哪一个阶段。

## 文件遍历、过滤与大目录

目录遍历常用 `Files.list`、`Files.walk`、`DirectoryStream` 和 `Files.walkFileTree`。`Files.list` 只列出一层，返回惰性 Stream，需要关闭；`Files.walk` 递归遍历，也需要关闭；`DirectoryStream` 适合大目录逐项处理；`walkFileTree` 通过 `FileVisitor` 回调处理进入目录、访问文件、访问失败和离开目录等事件，适合复杂递归复制、删除、统计。

处理大目录时，不要把所有路径一次性收集到列表再处理，除非确认规模可控。目录可能包含几十万甚至更多文件，一次性加载会占用大量内存。递归遍历还要考虑符号链接循环、权限不足、文件在遍历过程中被删除、路径过长等问题。`FileVisitOption.FOLLOW_LINKS` 应谨慎使用，因为跟随符号链接可能走出预期目录或形成循环。访问失败不一定要终止整个遍历，可以根据业务记录错误后继续。

删除目录树不是一个原子操作。删除过程中可能有新文件创建，可能遇到权限不足，可能部分成功部分失败。因此删除工具要明确失败后的状态和重试策略。复制目录树同样如此，复制文件内容、复制属性、创建目录、处理覆盖冲突都可能失败。不要把复杂文件树操作包装成一个简单布尔值返回，否则排障时很难知道失败发生在哪个路径。

`Files.walk` 返回的是惰性 Stream，这既是优点也是风险。优点是可以边遍历边处理，不必一次性把所有路径放进内存；风险是遍历过程中异常可能在终端操作时才出现，且 Stream 未关闭会持有目录资源。使用时应写成 try-with-resources，并在终端操作中处理受检异常带来的不便。很多人为了让 lambda 简洁，会把 `IOException` 包装成运行时异常，这可以接受，但要确保最终日志保留原始路径和异常原因。

递归复制和递归删除通常更适合 `walkFileTree`，因为它把目录进入、文件访问、失败处理和目录离开拆成明确回调。删除目录树时需要先删除子文件，再在 `postVisitDirectory` 删除目录本身；复制目录树时要在进入目录时创建目标目录，再访问文件时复制内容。遇到访问失败时，工具应决定是立即终止、记录后继续，还是按错误类型区分处理。这个决策不应藏在默认异常传播里，因为不同用途的容忍度不同：备份工具可能希望尽量复制可读文件，部署工具可能要求任何失败都终止。

大目录还会暴露排序问题。文件系统返回目录项的顺序通常没有业务含义，不同平台、不同文件系统甚至同一目录变化后都可能不同。如果处理结果需要稳定顺序，例如生成清单、打包、计算目录摘要，就应该显式排序。但排序意味着需要把路径收集到内存，和大目录流式处理存在冲突。实际设计中要在稳定性、内存占用和性能之间取舍，必要时按分批、外部排序或限制目录规模处理。

## 文件锁与并发访问

`FileChannel.lock` 和 `tryLock` 提供文件锁能力，可用于进程间协调。锁可以是独占锁或共享锁，可以覆盖整个文件或文件区域。文件锁的行为受操作系统影响，有的平台是强制锁，有的平台更接近建议锁；同一 JVM 内多个通道对同一文件加锁也可能出现 `OverlappingFileLockException`。因此文件锁适合做协作式约束，不适合当作跨平台强一致并发控制的唯一基础。

同一进程内多线程写同一个文件时，仍需要 Java 层同步或单写线程模型。`BufferedWriter`、`OutputStream` 等对象通常不是为并发写入设计的。即使底层追加操作某些情况下具备原子性，多条逻辑记录也可能因为多次 write 被交错。日志系统通常使用专门的异步队列、编码器和滚动策略，而不是让多个业务线程直接操作文件流。

读写并发还涉及可见性。一个线程写文件后，另一个线程何时能读到新内容，取决于缓冲刷新、操作系统缓存和打开方式。对于同一 JVM 内的共享数据，如果需要强一致，文件往往不是最合适的通信机制；如果必须用文件通信，就要定义写入完成标志、临时文件命名、原子重命名或校验和，避免读者读到半成品。

文件锁还容易被误用为线程锁。线程锁保护的是进程内对象状态，文件锁协调的是文件系统对象上的访问意图，两者层次不同。一个进程内如果多个线程共享同一个 `BufferedWriter`，即使对文件加了锁，也不能保证 Java 对象内部缓冲的并发访问安全；反过来，只有 Java `synchronized` 也不能阻止另一个进程写同一个文件。需要同时面对多线程和多进程时，应分别处理进程内同步和进程间协调，并尽量把写入集中到一个明确的组件。

轮询文件变化也是常见并发模式。Java 提供 `WatchService` 监听目录事件，但它不是可靠消息队列：事件可能合并、丢失或以平台相关方式呈现，监听的是目录而不是任意深度的完整树，文件写入完成也不一定对应一个简单事件。适合用它触发“可能有变化，重新扫描或校验”的流程，而不是把每个事件当作不可丢失的业务事实。对于重要输入文件，仍应结合临时文件命名、完成后重命名、大小稳定检测或校验文件来判断可消费状态。

## 异常处理与资源管理

文件 I/O 的异常信息是诊断依据。`NoSuchFileException`、`FileAlreadyExistsException`、`AccessDeniedException`、`DirectoryNotEmptyException`、`NotDirectoryException`、`FileSystemLoopException` 等异常比笼统的 `IOException` 更有意义。捕获异常时应记录路径、操作、打开选项和上下文，不要只记录“读写失败”。如果路径来自用户输入，还要注意日志中是否需要脱敏。

资源管理的默认方式是 try-with-resources。文件流、目录 Stream、DirectoryStream、FileChannel 都应及时关闭。文件描述符泄漏会导致打开文件失败，表现为 “too many open files” 或平台相关错误。泄漏还可能让文件删除、重命名或覆盖行为异常。对于返回 Stream 的 `Files.lines`、`Files.list`、`Files.walk`，很多人误以为 Stream 只是内存对象而忘记关闭，实际它背后持有目录或文件资源。

关闭顺序一般由包装层处理：关闭最外层流会关闭底层流。不要在多个包装对象共享同一个底层流时随意关闭其中一个，除非明确所有权。比如一个方法接收调用方传入的 OutputStream，通常不应该擅自关闭它，因为所有权属于调用方；但方法自己打开的文件流应由自己关闭。公共 API 设计中要明确“谁打开，谁关闭”或在文档中说明调用方责任。

异常处理不要把所有失败都压扁成 `false` 或空结果。文件不存在、权限不足、路径是目录、磁盘空间不足、文件名非法、符号链接循环、目标已存在，这些失败的处理方式往往不同。比如读取可选配置文件时，不存在可以使用默认值；存在但无法读取应当报错；内容损坏应当提示修复；权限不足可能需要运维处理。把这些情况都吞掉，会让程序以错误配置继续运行，后续问题更难定位。

也不要在 finally 或关闭阶段覆盖原始异常。try-with-resources 会把关闭时的异常作为 suppressed exception 挂到主异常上，这是它优于手写 finally 的重要原因。手写关闭逻辑时，如果在 catch 中抛出新异常，容易丢失真正的读写失败原因。需要清理临时文件时，也应把清理失败作为附加信息，而不是覆盖原始写入失败。文件 I/O 的排障高度依赖异常链，保留上下文比把异常包装成统一业务异常更重要；如果必须包装，也应把原异常作为 cause。

路径日志要兼顾诊断和安全。记录完整绝对路径能帮助定位问题，但路径中可能包含用户名、项目名、租户标识或业务文件名。通用库可以记录传入路径和操作类型，由调用方决定日志级别；面向终端用户的程序则应把底层异常转换成可理解的信息，同时保留调试日志。不要只输出“文件操作失败”，也不要在所有场景下无脑输出敏感路径。

## 性能与实践建议

文件 I/O 性能受多层因素影响：应用缓冲、JVM 分配、系统调用、页缓存、文件系统、磁盘类型、同步策略、并发程度和数据访问模式。顺序读写通常比随机读写快；批量操作通常比小块频繁操作快；复用缓冲通常比分配大量临时数组稳定；异步写入可以降低调用线程延迟，但会引入积压和丢失风险。优化前应先测量吞吐、延迟、CPU、磁盘利用率、I/O 等待和 GC。

实践中可以遵循几条原则。小文本配置可以用 `Files.readString` 和 `Files.writeString`，但要显式字符集并考虑原子更新。大文件逐行处理用 `Files.newBufferedReader` 或 `Files.lines`，但关闭 Stream。二进制大文件复制优先使用 `Files.copy` 或带缓冲的流，性能敏感时评估 `FileChannel.transferTo`。频繁写入日志不要自己拼一个全局 `FileWriter`，应使用成熟日志组件或至少用单写线程和有界队列。缓存文件要能容忍损坏，读取时校验，失败时重建。持久状态文件要考虑临时文件、校验和、版本字段和回滚策略。

选择 API 时可以按问题层次判断。只需要一次性读写小文本，`Files.readString`、`writeString` 最清晰；需要按行处理，使用 `BufferedReader` 或 `Files.lines`；需要二进制流式复制，使用 `Files.copy` 或缓冲字节流；需要随机位置读写，使用 `RandomAccessFile` 或 `FileChannel`；需要大文件随机访问且能接受映射边界，评估 `MappedByteBuffer`；需要递归处理目录树，使用 `walkFileTree`。不要为了“现代”而把简单文本读取写成复杂 NIO，也不要因为熟悉旧流 API 就在需要原子移动、属性视图和目录遍历时放弃 `Files`。

测试文件 I/O 代码时，应覆盖的不只是正常读写。至少要考虑：文件不存在、目标已存在、父目录不存在、路径指向目录、空文件、超大文件、非法编码、权限不足、磁盘空间不足的模拟、并发读写、临时文件残留、符号链接、嵌套目录和清理失败。并不是每个项目都能在自动化测试中真实模拟所有文件系统错误，但关键路径应把异常分支设计成可测试的结构，例如把路径选择、内容编码、写入策略和错误处理拆开。临时目录测试要确保结束后清理，同时不能因为清理失败掩盖测试本身的失败。

文件 I/O 还需要关注可观测性。批量导入、导出、备份、清理和索引这类任务，最好记录处理了多少文件、多少字节、耗时、失败路径、跳过原因和重试次数。单个文件操作失败时，日志应包含操作类型、路径、打开选项、文件大小或预期大小等上下文。没有这些信息，线上问题常常只能复现或猜测。可观测性不是额外装饰，而是文件系统这种外部状态参与时的基本工程能力。

## 常见误区

第一个误区是把 `File` 当成打开的文件句柄。`File` 或 `Path` 只是路径抽象，不占用文件描述符；真正需要关闭的是流、通道和目录遍历对象。

第二个误区是使用默认字符集。默认字符集会随运行环境变化，文本文件应显式指定编码。

第三个误区是用 `readAllBytes` 处理未知大小文件。它会把内容全部读入内存，可能造成内存压力甚至 OOM。

第四个误区是认为 `flush` 等于安全落盘。`flush` 只处理应用缓冲到底层流，持久化保证需要更强的同步手段，并且要接受性能成本。

第五个误区是直接覆盖重要文件。可靠更新应使用临时文件加替换策略，并明确文件系统不支持原子移动时如何处理。

第六个误区是忽视目录遍历资源。`Files.walk` 返回的 Stream 必须关闭，否则可能泄漏文件句柄。

第七个误区是把路径规范化当作安全校验。`normalize()` 只能清理路径文本，不能证明目标没有通过符号链接跳出边界，也不能消除检查与使用之间的竞态。

第八个误区是认为文件锁可以自动解决所有并发问题。文件锁不能替代进程内同步，也不能跨平台提供完全一致的强制约束；它只应作为协作协议的一部分。

第九个误区是忽略文件属性。复制或替换文件时，权限、时间戳、所有者、链接语义可能影响后续行为。业务只关心内容时可以简化，但工具类代码和持久状态代码不能默认属性无关。

第十个误区是把临时文件当作普通中间变量。临时文件可能残留、可能被其他扫描逻辑误读、可能不在同一文件系统、可能权限不合适。可靠写入时，临时文件的位置、命名、清理和替换策略都要纳入设计。

## 小结

Java 文件 I/O 的表层是 API 选择，深层是文件系统语义和可靠性边界。字节流、字符流、缓冲、通道、路径、目录遍历、文件锁和原子移动分别解决不同问题。写出可靠文件代码，需要明确字符集、控制内存占用、正确关闭资源、处理部分读写和具体异常，并根据业务价值决定是否需要同步落盘、临时文件替换和恢复策略。文件是最常见的持久化媒介之一，也正因为常见，才容易被低估；把这些边界处理清楚，简单的读写代码才能在真实环境中长期稳定运行。

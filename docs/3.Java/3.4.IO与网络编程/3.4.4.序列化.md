# 3.4.4 序列化

序列化是把内存中的对象状态转换成可存储、可传输表示的过程；反序列化则是把这种表示重新构造成对象。Java 标准库中最具代表性的机制是 `Serializable` 与 `ObjectOutputStream`、`ObjectInputStream`，另一个较少使用但更可控的接口是 `Externalizable`。除此之外，实际开发中还会使用 JSON、XML、Protocol Buffers、Avro、Kryo 等格式或框架。本文聚焦通用 Java 视角下的对象序列化机制、兼容性、资源与安全边界，而不是把序列化简单理解成“对象转字节数组”。

序列化要解决的问题包括持久化对象快照、跨进程或跨网络传递数据、缓存复杂结构、复制对象图等。它也会引入明显风险：对象结构和字节格式耦合，类演化可能破坏兼容；反序列化会绕过普通构造流程，破坏不变量；不可信数据反序列化可能触发安全漏洞；对象图过大可能造成内存压力；隐式序列化字段可能泄露敏感信息。因此，序列化不是一个“能用就行”的工具，而是一份长期数据契约。凡是数据会跨版本、跨系统、跨信任边界保存或传输，就必须认真设计格式和兼容策略。

## Serializable 的基本模型

`Serializable` 是一个标记接口，不包含方法。一个类实现它后，`ObjectOutputStream` 可以按 Java 对象序列化协议写出该对象的非 transient、非 static 字段；`ObjectInputStream` 可以读取字节流并重建对象。序列化过程会遍历对象图：如果对象字段引用了其他对象，被引用对象也需要可序列化，否则会抛出 `NotSerializableException`。同一个对象在对象图中被多次引用时，序列化流会保留引用关系，反序列化后仍能恢复共享引用，而不是简单复制成多个独立对象。

```java
final class User implements Serializable {
    private static final long serialVersionUID = 1L;

    private final String name;
    private final int age;
    private transient String cachedDisplayName;

    User(String name, int age) {
        this.name = Objects.requireNonNull(name);
        this.age = age;
        this.cachedDisplayName = name + "(" + age + ")";
    }
}
```

`serialVersionUID` 是序列化兼容性的核心标识。反序列化时，流中的类描述与本地类的 `serialVersionUID` 必须匹配，否则会抛出 `InvalidClassException`。如果没有显式声明，JVM 会根据类结构计算一个值；类名、字段、方法、访问修饰符等变化都可能影响计算结果，使得轻微改动导致旧数据无法读取。因此，可序列化类应显式声明 `serialVersionUID`，并把是否变更它作为兼容性决策，而不是交给编译器和运行时偶然决定。

`transient` 字段不会被默认序列化，常用于缓存、派生值、临时资源、密码明文、连接句柄等不应进入持久格式的字段。`static` 字段属于类，不属于对象实例，也不会作为对象状态写出。反序列化后，transient 字段会得到默认值，必要时需要在 `readObject` 中重建。对于 final 字段，默认反序列化可以设置其值，但这进一步说明反序列化不是普通构造过程。

## 对象图、引用与构造过程

Java 原生序列化保存的是对象图，而不是单个扁平对象。它会记录类描述、字段值、数组、字符串、枚举、对象引用句柄等信息。对象图中如果存在循环引用，例如 A 引用 B、B 又引用 A，原生序列化可以处理；如果同一对象被两个字段引用，反序列化后这两个字段仍指向同一个实例。这是原生序列化相对简单文本格式的一个能力，但也意味着序列化结果可能包含远超预期的对象范围。

对象图遍历会把所有可达的非 transient 对象纳入序列化。一个看似小的对象如果引用了大型缓存、集合、父对象、服务上下文或异常堆栈，可能写出大量数据，甚至因为某个深层对象不可序列化而失败。因此，可序列化类应谨慎设计字段，不要把运行期服务对象、线程池、文件句柄、数据库连接等放进默认序列化路径。必要时使用 DTO，把需要传输或持久化的数据与运行期对象分离。

反序列化不会调用可序列化类的普通构造器。对于实现 `Serializable` 的类，反序列化会分配对象并填充字段；其第一个不可序列化父类的无参构造器会被调用。这个机制带来两个后果。第一，构造器中的校验、归一化和派生字段初始化不会自动执行；第二，对象可能以普通构造器永远不会产生的状态出现。类如果依赖不变量，应通过自定义 `readObject` 校验反序列化结果，发现非法状态时抛出 `InvalidObjectException`。

## 自定义 writeObject 与 readObject

`Serializable` 支持私有的 `writeObject` 和 `readObject` 方法，用于控制默认序列化前后的额外逻辑。它们不是接口方法，而是由序列化机制通过反射识别。常见用途包括写入派生数据、压缩字段、加密敏感字段、重建 transient 缓存、校验对象不变量、兼容旧字段等。

```java
private void writeObject(ObjectOutputStream out) throws IOException {
    out.defaultWriteObject();
}

private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
    in.defaultReadObject();
    if (age < 0) {
        throw new InvalidObjectException("age must not be negative");
    }
    cachedDisplayName = name + "(" + age + ")";
}
```

`defaultWriteObject` 和 `defaultReadObject` 处理默认字段；如果完全手写字段，就要承担顺序、类型和兼容性责任。`readObject` 中的校验非常重要，因为反序列化输入可能来自旧版本、损坏文件或恶意构造。即使数据来自本系统，也可能因为部署回滚、磁盘损坏、手工修改或程序 Bug 变得不可信。

还有两个特殊方法 `writeReplace` 和 `readResolve`。`writeReplace` 可以在序列化前用另一个对象替换当前对象，常用于序列化代理模式；`readResolve` 可以在反序列化后返回替代对象，常用于单例、枚举式实例或规范化对象。对于单例类，如果不实现 `readResolve`，反序列化可能创建出新的实例，破坏单例语义。枚举类型的序列化由语言和运行时特殊处理，通常比手写单例更安全。

## Externalizable 与显式格式

`Externalizable` 继承自 `Serializable`，要求类实现 `writeExternal` 和 `readExternal`。与默认序列化不同，它把字段写入和读取完全交给类自己控制。反序列化 `Externalizable` 对象时，会先调用 public 无参构造器，再调用 `readExternal` 填充状态。因此类必须提供 public 无参构造器，这会影响封装设计。

`Externalizable` 的优势是格式显式、可减少默认序列化的类描述开销、可控制字段顺序和兼容逻辑。缺点是责任重：读写顺序必须严格一致，类型转换和版本判断必须自己处理，遗漏字段不会有默认机制兜底。对于长期持久化或跨语言传输，很多时候更推荐定义独立数据格式，而不是依赖 Java 原生序列化或 `Externalizable`。如果只是 Java 内部短期缓存，`Externalizable` 可以作为性能和控制权的折中，但要配套测试覆盖版本演化。

显式格式的一个常见做法是在流开头写入魔数和版本号。魔数用于识别文件类型和字节序，版本号用于决定后续字段如何读取。新增字段时可以在高版本写入，低版本读取时忽略或使用默认值；删除字段时可以保留读取逻辑但不再使用；字段语义变化时要非常谨慎，必要时引入新字段而不是复用旧字段名。格式设计越显式，兼容策略越可控。

## 序列化流中的类描述、字段与句柄

理解 Java 原生序列化时，不能只看 `ObjectOutputStream.writeObject` 这一行调用。对象流内部有自己的协议结构：流头标识这是一个对象序列化流；随后写入对象、类描述、字段名、字段类型、数组长度、字符串内容、枚举名称、块数据和对象引用句柄。类描述记录的是类名、`serialVersionUID`、序列化标志、可序列化字段列表以及父类描述。字段值本身通常按类型分组写入：基本类型直接写值，引用类型写对象、字符串、数组、枚举或已有句柄。

句柄是对象图语义的关键。对象第一次写入时会分配一个句柄，后续再遇到同一对象引用时，流中只写“引用前面那个句柄”。这就是为什么共享引用和循环引用能够恢复。假设一个订单对象的 `buyer` 和 `payer` 字段都指向同一个用户实例，反序列化后两个字段仍指向同一个对象，而不是两个字段值相等但身份不同的用户对象。这个特性对某些图结构很有用，但对“只想传输数据值”的场景也可能让格式变得难以推理，因为对象身份也被纳入了协议。

句柄表还会影响长连接或批量写入。一个 `ObjectOutputStream` 写出对象后会记住已经写过的对象。如果同一个对象实例被修改后再次写入，默认情况下流可能只写出引用句柄，读取端看到的是第一次反序列化得到的那个对象，而不是修改后的字段快照。调用 `writeUnshared` 可以要求当前对象本身按非共享方式写入，调用 `reset` 可以清空流的句柄表和类描述缓存，使后续对象按新的对象图写出。它们不是性能开关，而是改变流语义的工具，使用前要明确读取端期望的是对象身份共享还是每次写入的独立快照。

对象流也不是简单的“一个对象一段字节”。`ObjectOutputStream` 构造时会写入流头，`ObjectInputStream` 构造时会读取流头。如果在同一个文件或同一个 socket 上反复创建新的 `ObjectOutputStream`，中间会插入多个流头，读取端如果只用一个 `ObjectInputStream` 连续读取对象，可能在第二个流头处报 `StreamCorruptedException`。相反，如果一直复用同一个对象流，又要管理句柄表、flush 时机和消息边界。工程上更稳妥的方式是明确外层协议：要么一个文件只对应一个对象流，要么在更外层定义长度、数量、版本和结束条件，不要把对象流当成自解释的网络消息协议。

默认序列化字段也有固定规则。实例字段中，非 `static`、非 `transient` 的字段会进入默认字段集合；字段名和字段类型是兼容性判断的一部分；字段声明顺序不应被当成业务语义依赖。对象流能读取旧流中存在但当前类已经删除的字段，也能给旧流中不存在的新字段填默认值，这是 Java 原生序列化提供的基础演化能力。但“能读”不代表“语义正确”，因为默认值可能是非法状态，删除字段也可能意味着丢失了构造当前对象所需的信息。

少数情况下，可以用 `serialPersistentFields` 显式声明序列化字段，而不是让默认机制直接使用类的真实字段集合。它是一个名为 `serialPersistentFields` 的 `private static final ObjectStreamField[]` 字段，序列化机制会把它视为持久字段定义。配合 `ObjectOutputStream.PutField` 和 `ObjectInputStream.GetField`，类可以把内部实现字段和外部持久格式解耦。例如类内部把 `firstName`、`lastName` 改成了一个 `Name` 值对象，但持久格式仍然维持旧的 `firstName`、`lastName` 字段，以便读取历史数据。这个机制很强，但也会增加维护复杂度，因此适合已经明确需要长期兼容原生序列化格式的类。

```java
private static final ObjectStreamField[] serialPersistentFields = {
        new ObjectStreamField("firstName", String.class),
        new ObjectStreamField("lastName", String.class)
};

private void writeObject(ObjectOutputStream out) throws IOException {
    ObjectOutputStream.PutField fields = out.putFields();
    fields.put("firstName", name.first());
    fields.put("lastName", name.last());
    out.writeFields();
}

private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
    ObjectInputStream.GetField fields = in.readFields();
    String firstName = (String) fields.get("firstName", "");
    String lastName = (String) fields.get("lastName", "");
    this.name = new Name(firstName, lastName);
}
```

`serialPersistentFields` 不是用来隐藏字段的安全机制。它控制的是默认持久字段集合，不能替代加密、访问控制和输入校验。如果字段里有敏感信息，正确做法仍然是不要写入、写入前加密或使用专门的安全存储。它真正适合解决的是“类内部结构已经重构，但历史字节流仍要按旧字段名读取”的兼容问题。

## 类演化与兼容性

序列化兼容性不是“只要 serialVersionUID 不变就没问题”。保持相同 UID 只是允许反序列化继续进行，字段变化仍会影响对象状态。一般来说，新增非 transient 字段时，旧数据中没有该字段，反序列化后得到默认值；删除字段时，新类会忽略旧数据中的该字段；修改字段类型通常不兼容；修改类层次结构、字段访问方式、枚举常量、内部类结构都可能产生复杂影响。对于长期存储的数据，应把兼容性作为格式设计问题，而不是依赖默认规则。

默认值也可能破坏业务语义。新增 `int retryLimit` 字段后，旧数据反序列化得到 0，可能表示不重试，也可能与预期默认值 3 冲突。此时需要在 `readObject` 中识别旧数据并填充合理默认值。但默认序列化并不会自动告诉你对象来自哪个版本，除非你自己保存版本字段。把 `private int version = 2;` 作为普通字段写入，可以在反序列化后判断并迁移状态，但新增版本字段本身在旧数据中也是默认值，需要设计初始约定。

可序列化类的包名和类名也是格式的一部分。移动类、重命名类、改变嵌套类结构，都会让旧数据无法按原类加载。可以通过自定义 `ObjectInputStream.resolveClass` 做映射，但这会增加维护成本。对于需要长期保存的数据，使用与 Java 类名解耦的格式通常更稳妥。原生序列化更适合短生命周期、同一系统内部、版本同步升级的对象传递或缓存。

兼容性判断还要区分“二进制能否读出”和“业务是否仍然成立”。新增字段、删除字段、调整 `private` 方法，可能在序列化协议层面仍然兼容；但如果新增字段代表新的必需约束，旧数据默认值就可能让对象处于半初始化状态。反过来，某些协议层面不兼容的变化，也可以通过保留旧字段、引入代理类或自定义读取逻辑来迁移。实际设计时，应先确定数据生命周期：如果数据只是短期缓存，升级时删除旧缓存可能比维护复杂兼容更合理；如果数据是用户文件、审计记录或跨系统交换格式，就必须把每一次字段变化视为协议演进。

`serialVersionUID` 的变更策略也要清晰。保持不变表示“我承诺当前类仍能理解旧流，并会在读取后修正对象状态”；修改它表示“我拒绝读取旧流”。这不是简单的版本号递增规则。很多小改动不需要改 UID，但需要补充迁移逻辑；某些看似小的语义改变反而应该改变 UID，避免旧数据被误读成新含义。团队协作中，可序列化类的字段变更应像数据库 schema 变更一样被审查：新增字段的默认值是什么，删除字段是否仍需读取，类型改变是否可迁移，旧样本是否仍能通过测试。

对于默认值迁移，一个常见模式是在类中保存明确的格式版本字段，而不是依赖 `serialVersionUID` 区分所有业务版本。`serialVersionUID` 解决的是类描述匹配问题，格式版本解决的是字段语义演进问题。版本字段可以参与默认序列化，也可以由自定义 `writeObject` 写入。在 `readObject` 中读取后，根据版本补齐新增字段、转换旧语义、拒绝未知高版本。未知高版本尤其要谨慎：当前程序可能不知道未来版本新增了什么约束，盲目读取可能比直接失败更危险。

```java
private static final long serialVersionUID = 3L;

private int formatVersion = 2;
private int retryLimit = 3;

private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
    in.defaultReadObject();
    if (formatVersion == 0) {
        retryLimit = 3;
        formatVersion = 1;
    }
    if (retryLimit < 0) {
        throw new InvalidObjectException("retryLimit must not be negative");
    }
}
```

这个例子里 `formatVersion == 0` 可代表早期流中没有版本字段，因为 `int` 缺失时会得到默认值 0。真实项目中不要只靠单个数字完成所有迁移，还要让迁移逻辑可测试、可回滚、可观察。特别是批量读取旧文件或旧缓存时，应能区分“格式不支持”“数据损坏”“类路径缺失”“业务校验失败”几类错误，否则排查时只能看到笼统的反序列化异常。

## 继承、不可序列化父类与不变量

继承结构会让序列化更复杂。一个类实现 `Serializable` 后，它的可序列化父类字段会按各自的类描述写入；遇到第一个没有实现 `Serializable` 的父类时，该父类的字段不会从流中恢复，而是在反序列化时通过调用它的无参构造器初始化。因此，不可序列化父类必须有可访问的无参构造器，否则子类反序列化会失败。这里的“可访问”不是 public 的唯一含义，而是对反序列化机制和子类来说可调用；但从设计角度看，依赖这种细节本身就说明继承层次需要审视。

这个规则容易制造隐藏状态丢失。假设父类 `BaseAccount` 没有实现 `Serializable`，其中维护 `createdAt`、`status` 或校验标志；子类 `PremiumAccount` 实现了 `Serializable`。反序列化 `PremiumAccount` 时，父类状态来自父类无参构造器，而不是序列化流。若父类无参构造器只设置了占位默认值，子类对象就可能带着不符合业务含义的父类状态继续运行。解决方式不一定是让父类也实现 `Serializable`，更可能是避免在继承层次上混合持久状态和运行期状态，或者让子类在 `readObject` 中显式恢复和校验必要字段。

构造器不执行的问题不仅影响字段默认值，也影响类不变量。普通构造器可能会检查参数范围、复制可变集合、排序字段、注册监听器、计算哈希缓存或建立双向关系；反序列化默认只是填字段。任何依赖构造器建立的不变量，都应在 `readObject` 中重新验证。对于不可变类，反序列化尤其需要防御可变对象引用：即使字段是 `final`，如果字段引用的是可变集合、数组或日期对象，也要确认反序列化后不会暴露可变内部状态。必要时在 `readObject` 后复制集合、包装为不可修改视图或使用序列化代理重建对象。

还有一个细节是 `readObject` 方法本身不应调用可被子类覆盖的方法。反序列化过程中对象尚未完全恢复，调用可覆盖方法可能让子类看到半初始化状态，产生空指针、错误缓存或安全绕过。这与构造器中避免调用可覆盖方法的原则类似。`readObject` 更适合做局部字段读取、基础校验和私有辅助方法调用。如果类不是为了继承设计的，可序列化类通常应声明为 `final`，或者谨慎限定可覆盖行为，减少子类把序列化过程变成扩展点的风险。

## transient、敏感数据与派生状态

`transient` 的作用是把字段排除在默认序列化之外，但它不是“这个字段不重要”的标记。很多 transient 字段恰恰对对象运行很重要，只是不适合直接写入流。缓存、懒加载结果、线程安全工具、文件句柄、数据库连接、锁对象、随机数生成器、校验摘要、加密器、日志器等都可能是 transient。反序列化后这些字段会变成默认值，如果类方法没有考虑恢复过程，就会在某个后续调用中才暴露问题。

恢复 transient 字段有几种方式。最直接是在 `readObject` 里调用私有初始化方法；如果字段可以懒加载，则在访问方法中发现为空时重建；如果字段代表外部资源，则不应自动重连，而应让调用方显式注入或重新打开。选择哪种方式取决于字段语义。派生缓存可以自动重建，外部资源不应悄悄恢复，因为反序列化一个对象不应隐式打开文件、建立网络连接或启动线程。一个好的可序列化类会区分“对象状态”和“运行环境”，前者可以进入流，后者应通过清晰的生命周期管理恢复。

敏感数据处理也不能只靠 `transient`。把密码字段标记为 transient 可以避免默认写入，但如果 `writeObject` 手动写出了它，或者敏感值被另一个非 transient 对象间接引用，仍然会泄露。更常见的问题是把认证上下文、配置对象或异常对象序列化，里面间接包含令牌、路径、主机名、用户标识或堆栈参数。序列化对象图是可达性驱动的，敏感信息不一定出现在当前类的字段名中。设计前应从对象图角度审查：哪些对象可达，哪些字段会进入流，哪些内容即使是 private 也会被写出。

有些字段看起来可以序列化，实际上不应序列化。例如 `Lock`、`ThreadLocal`、`ExecutorService`、`InputStream`、`OutputStream`、`Socket`、`ClassLoader` 等对象代表运行期资源或环境能力。把它们放入默认对象图，轻则抛出 `NotSerializableException`，重则把不稳定的内部实现细节写入持久格式。更好的做法是把数据对象与执行对象拆开：可序列化类只表达必要状态，运行期能力由服务层、工厂、上下文或依赖注入在对象恢复后重新连接。

## 安全边界

反序列化不可信数据是 Java 安全领域长期高风险点。`ObjectInputStream.readObject` 不只是解析数据，它会根据流中的类描述加载类、分配对象、调用 `readObject`、`readResolve` 等方法，并可能触发复杂对象图中的代码路径。如果类路径中存在危险的 gadget 链，恶意序列化数据可能导致远程代码执行、文件操作、网络访问或资源耗尽。即使没有代码执行，攻击者也可以构造巨大对象图、深层嵌套、重复引用或哈希冲突数据，造成内存和 CPU 压力。

基本原则是不要对不可信输入使用 Java 原生反序列化。如果必须读取可能不完全可信的数据，应使用 JDK 的对象输入过滤机制，例如 `ObjectInputFilter`，限制允许的类、数组长度、对象图深度、引用数量和字节数。过滤策略应尽量使用白名单，而不是宽泛黑名单。反序列化后仍要进行业务校验，不能因为类成功构造就认为数据可信。

敏感字段也要注意。默认序列化会写出所有非 transient、非 static 字段，包括 private 字段。访问修饰符保护的是 Java 代码访问，不保护序列化内容。密码、密钥、令牌、个人敏感信息、连接凭据等不应默认进入序列化流；如果确实需要保存，应使用专门的加密、密钥管理和访问控制机制。不要把“字段是 private”当成数据不会泄露的理由。

`ObjectInputFilter` 的价值在于把“先读出来再判断”前移到“读取过程中限制”。过滤器可以根据类名、数组长度、引用数量、对象图深度、字节数等指标决定允许、拒绝或未决定。实践中应尽量把允许类型收窄到业务 DTO、基础 JDK 值类型和必要集合类型，避免使用宽泛的 `java.*` 或整个应用包名前缀作为通行证。集合类型也不能只允许集合类本身，还要限制元素类型、深度和数量，否则攻击者仍可能构造巨大的嵌套集合。

```java
ObjectInputFilter filter = ObjectInputFilter.Config.createFilter(
        "maxdepth=20;maxrefs=10000;maxbytes=1048576;" +
        "com.example.dto.*;java.base/java.lang.String;java.base/java.util.ArrayList;!*");

try (ObjectInputStream in = new ObjectInputStream(inputStream)) {
    in.setObjectInputFilter(filter);
    Object object = in.readObject();
}
```

过滤器不是完整的安全模型。它不能替代传输层认证、文件权限、签名校验、数据完整性校验和业务字段校验。它也不能保证被允许类的 `readObject` 没有副作用。因此，真正的安全边界应组合多层措施：不从不可信来源接收原生序列化流；必须接收时使用白名单过滤；读取前验证来源和完整性；读取后校验业务约束；把反序列化入口隔离在很小的模块内；避免让任意插件、脚本或第三方库类型进入可反序列化白名单。

安全还包括拒绝服务风险。即使所有类都是“安全类”，输入也可能消耗大量内存和 CPU。例如一个极深的链表会导致深度压力，一个大数组会直接申请大量内存，一个带大量元素的哈希结构会触发大量对象创建和哈希计算。过滤器中的 `maxdepth`、`maxrefs`、`maxarray`、`maxbytes` 等限制应按业务数据规模设置，而不是随意给一个很大的数字。对于文件导入类场景，还应在外层限制文件大小、读取超时和并发数量。

## readObject、writeObject 与序列化代理模式

自定义 `writeObject` 和 `readObject` 既可以增强默认序列化，也可能破坏默认机制。它们的方法签名必须准确，并且通常声明为 `private`，因为序列化机制按固定签名查找，不是通过重写接口方法调用。写错访问修饰符、参数或异常声明时，代码仍能编译，但方法不会被序列化机制调用，这类 Bug 很隐蔽。对关键可序列化类，可以通过单元测试验证自定义逻辑确实执行，例如检查 transient 字段是否恢复、非法流是否被拒绝、旧样本是否按预期迁移。

在 `readObject` 中，`defaultReadObject` 一般应尽早调用，然后立即校验字段。不要把反序列化得到的对象提前发布给其他线程、注册到全局容器或放入静态集合，也不要在校验完成前调用复杂业务逻辑。反序列化期间的对象处于构建中状态，过早发布会让其他代码看到不完整对象。对于安全敏感类型，如果发现字段非法，应抛出 `InvalidObjectException`，让读取过程失败，而不是吞掉异常后构造一个“尽量可用”的对象。

序列化代理模式适合不可变值对象、内部表示复杂但外部语义稳定的对象，以及需要强不变量保护的类型。真实对象的 `writeReplace` 返回一个简单代理；真实对象的 `readObject` 可以直接抛出异常，禁止攻击者绕过代理格式构造真实对象；代理类的 `readResolve` 再调用真实对象的构造器或工厂方法。这样反序列化恢复对象时仍经过普通构造路径，不变量由同一套代码维护。

```java
final class Period implements Serializable {
    private static final long serialVersionUID = 1L;

    private final LocalDate start;
    private final LocalDate end;

    Period(LocalDate start, LocalDate end) {
        if (start.isAfter(end)) {
            throw new IllegalArgumentException("start must not be after end");
        }
        this.start = start;
        this.end = end;
    }

    private Object writeReplace() {
        return new SerializationProxy(this);
    }

    private void readObject(ObjectInputStream in) throws InvalidObjectException {
        throw new InvalidObjectException("proxy required");
    }

    private static final class SerializationProxy implements Serializable {
        private static final long serialVersionUID = 1L;

        private final LocalDate start;
        private final LocalDate end;

        SerializationProxy(Period period) {
            this.start = period.start;
            this.end = period.end;
        }

        private Object readResolve() {
            return new Period(start, end);
        }
    }
}
```

代理模式的意义不是让代码显得高级，而是把持久格式收敛到小而稳定的代理对象上。真实对象可以重构内部字段，只要代理格式不变，旧数据仍能恢复。代理也能避免反序列化直接写入真实对象私有字段，从而减少无效状态出现的机会。不过它并不适合所有类型：可被外部继承的类、对象图中必须保留复杂共享引用身份的类型，以及需要特殊性能优化的大型结构，都要评估代理带来的语义和性能成本。

## 线程可见性、不可变对象与防御性复制

序列化经常被用来保存对象快照，但“快照”并不会自动与并发修改隔离。如果一个线程正在修改对象图，另一个线程同时序列化它，写出的结果可能是某个中间状态：集合大小和元素不一致，两个相关字段只更新了一个，或者对象图中一部分来自旧状态、一部分来自新状态。`ObjectOutputStream` 不会替应用层加锁，也不了解业务不变量。需要一致快照时，应在更高层使用不可变对象、复制数据、读写锁或单线程所有权，确保序列化时对象图稳定。

不可变对象对序列化更友好，但也不是完全无风险。不可变类的字段通常是 `final`，构造器负责校验不变量；反序列化绕过构造器后，仍需通过 `readObject` 或代理模式恢复这些保障。若不可变类持有数组或可变集合，必须进行防御性复制或包装，否则攻击者可以通过构造特殊流让内部字段引用可变对象，进而破坏对象不可变性。序列化代理模式之所以适合不可变类，就是因为它让恢复过程重新经过构造器。

并发工具类本身通常不应作为长期序列化格式。某些 JDK 集合和并发集合实现了 `Serializable`，是为了在特定场景下保存容器内容或支持框架能力，不意味着应用应把内部并发结构作为外部协议。把 `ConcurrentHashMap` 直接序列化，会把“当前实现和当前键值状态”一起写出，却不会保存线程间正在发生的操作语义。对外部格式来说，更稳定的做法是提取普通键值数据，序列化成明确 DTO 或 schema。

## 异常、内部类、记录类和枚举的特殊性

异常对象通常实现了序列化能力，但把异常作为长期数据保存要谨慎。异常中可能包含堆栈、原因链、suppressed 异常和自定义字段，这些内容既可能很大，也可能泄露路径、参数和环境信息。跨进程传递错误信息时，更好的做法是定义错误码、消息、上下文摘要和可公开的诊断字段，而不是直接序列化异常对象。异常对象适合在同一 Java 体系内部做短期诊断，不适合作为稳定外部协议。

非静态内部类、匿名类和 lambda 表达式也不适合作为稳定序列化类型。非静态内部类隐式持有外部类引用，可能把外部对象图一起写出；匿名类和 lambda 的生成类名、捕获字段和实现细节不稳定，编译器、构建方式和代码结构变化都可能改变它们的序列化形态。即使某个 lambda 通过交叉类型或特定函数式接口被声明为可序列化，也不应把它当成可长期保存的行为对象。行为应通过明确的策略名称、枚举、配置或脚本标识表达，而不是把运行时代码对象写入流。

枚举的序列化由 Java 运行时特殊处理，主要按枚举类型和常量名称恢复，不按普通字段集合恢复。枚举适合表达有限、稳定的取值集合，但枚举常量删除或重命名会破坏旧数据读取；新增常量通常对旧数据读取没有问题，却可能让旧程序无法理解新数据。枚举字段如果参与外部协议，要把常量名称视为协议值，不要轻易改名。若外部语义需要更稳定的编码，可以给枚举显式定义 code，并在自定义格式中写 code，而不是依赖枚举常量名。

记录类可以实现 `Serializable`，但记录类的序列化语义也需要谨慎理解。记录类强调透明的数据载体，组件就是主要状态；这使它适合简单 DTO，但不自动解决版本演化和安全问题。记录组件新增、删除、改名或改类型，仍会影响序列化兼容。记录类的紧凑构造器用于普通构造路径的校验，而反序列化恢复时的具体行为受 Java 版本和序列化规范约束；为了长期稳定，仍应用样本测试验证不变量和兼容性，而不是假设“record 天然安全”。

## 与文本和二进制数据格式的取舍

Java 原生序列化的优点是使用方便、能保存对象图和引用关系、对 Java 类型支持丰富。缺点是格式与 Java 类强耦合，跨语言困难，可读性差，安全风险高，版本演化复杂，序列化结果通常不够紧凑。JSON 可读性好、生态广、适合接口和配置，但类型表达有限，引用关系和二进制数据需要额外约定。Protocol Buffers、Avro 等二进制格式强调 schema、兼容演化和跨语言，适合服务接口和长期数据契约，但需要定义 schema 和生成或编写映射代码。

选择格式时要问几个问题：数据是否跨语言？是否跨版本长期保存？是否来自不可信边界？是否需要人类可读？是否需要保存对象引用关系？性能瓶颈在 CPU、网络还是存储？如果只是进程内深拷贝，序列化可能不是最佳方式，手写复制或不可变对象更清晰。如果是服务间通信，应优先选择明确 schema 的格式。如果是本地短期缓存，原生序列化可以考虑，但要能容忍版本升级后缓存失效。

## 性能与资源管理

原生序列化使用反射和对象图遍历，性能通常不是最优。它会写入类描述和句柄信息，首次写入某类对象时开销更大。`ObjectOutputStream` 对已经写过的对象会维护句柄表，重复写同一个对象时可能只写引用；这有利于保留共享引用，但也可能在长时间复用同一个流时导致内存增长，并且再次写入同一对象即使字段已变化，也可能只写出旧引用语义。可以调用 `reset()` 清空句柄表，让后续对象按新对象写出，但这也会增加流大小。

对象图过大是常见问题。序列化一个集合时，集合中的每个元素都会被处理；序列化一个异常对象时，堆栈、cause、suppressed 异常也可能进入对象图；序列化内部类实例时，可能隐式持有外部类引用，导致把外部对象也写出。可序列化 DTO 应尽量简单、扁平、明确，不要直接序列化复杂领域对象或运行时对象。

流的关闭同样重要。`ObjectOutputStream` 包装在底层 `OutputStream` 之上，关闭它会刷新并关闭底层流。网络场景中，如果需要在同一连接上继续使用底层流，不能随意关闭包装流；文件场景中则应及时关闭以释放文件描述符。读取对象时要处理 `ClassNotFoundException`，因为本地类路径可能没有流中声明的类。还要处理 `InvalidClassException`、`StreamCorruptedException`、`OptionalDataException` 等具体异常，它们能帮助定位是版本不兼容、流损坏还是读写顺序错误。

## 常见误区

第一个误区是认为实现 `Serializable` 就完成了序列化设计。真正的设计包括 UID、字段选择、transient、版本演化、安全过滤、对象校验和测试。

第二个误区是依赖自动生成的 `serialVersionUID`。自动值对类结构敏感，容易让无害改动破坏旧数据读取。可序列化类应显式声明。

第三个误区是反序列化后不校验对象不变量。构造器不会按普通方式执行，非法字段组合可能进入系统。

第四个误区是对外部输入使用原生反序列化。除非有严格白名单过滤和安全边界，否则应避免。

第五个误区是序列化领域对象或服务对象。长期数据契约应使用明确 DTO 或 schema，避免把内部实现细节变成持久格式。

第六个误区是忽视 transient 字段恢复。缓存、派生字段和运行期状态反序列化后需要重建，否则对象可能在后续方法中出现空指针或错误结果。

## 实践建议

如果一个类确实需要 Java 原生序列化，应保持它的数据职责单一，显式声明 `serialVersionUID`，将不应持久化的字段标记为 transient，在 `readObject` 中恢复派生状态并校验不变量。对于长期保存的数据，加入版本字段并编写迁移测试。对于敏感或不可信边界，优先选择明确 schema 的格式，并拒绝原生反序列化。对于缓存，可以把格式版本纳入文件名或头部，升级后直接丢弃旧缓存，避免为低价值数据维护复杂兼容。

测试也要覆盖序列化兼容性。可以保存一份旧版本生成的样本字节或样本文件，在新版本测试中读取，确认默认值、字段迁移和不变量正确。也要测试损坏数据、未知版本、缺失字段、超大输入和非法字段组合。序列化 Bug 往往在部署升级或恢复数据时才暴露，越早用样本固定契约，越容易控制风险。

还应避免把序列化能力随意暴露在公共父类上。一个父类实现了 `Serializable`，所有子类都会被推入“可能被序列化”的设计空间，后续字段添加、构造器约束和安全审查都要承担额外负担。如果父类只是为了某个局部场景方便而实现接口，可能让大量本不应该持久化的子类也具备序列化入口。更稳妥的做法是让真正的数据载体实现序列化，服务类、管理器、上下文对象和持有外部资源的对象保持不可序列化。对于不可序列化但又可能被误用的类，可以在文档、测试或架构约束中明确禁止。

内部类和匿名类尤其不适合作为稳定序列化类型。非静态内部类会隐式持有外部类引用，默认序列化可能把外部对象一起写出；匿名类和 lambda 的生成形式、名称和捕获字段也不适合作为长期数据格式。即使某些函数式接口对象在特定条件下可以被序列化，也不应把它们保存为长期契约。稳定格式应使用命名清晰、字段明确、职责单一的静态类或普通顶层类。

集合字段也需要设计约束。默认序列化会保存集合的具体实现，而不是只保存“列表”或“映射”的抽象含义。把 `ArrayList` 改成 `LinkedList`、把 `HashMap` 改成有序映射，可能改变序列化形态或反序列化后的行为。更重要的是，集合元素的类型、数量和顺序应有业务约束：是否允许 null，是否允许重复，映射键是否区分大小写，顺序是否有意义，都应在反序列化后校验。对于哈希集合和映射，还要注意对象的 `equals`、`hashCode` 与字段演化关系；如果反序列化后参与哈希计算的字段为空或被迁移，集合内部语义可能变得异常。

序列化代理模式是提高安全性和兼容性的一种方式。核心思路不是直接序列化复杂对象，而是在 `writeReplace` 中返回一个简单、稳定、可校验的代理对象；代理对象反序列化后通过 `readResolve` 调用公开构造或工厂方法重建真实对象。这样可以让真实对象的不变量仍由构造逻辑维护，并避免暴露内部字段结构。它的成本是需要额外代理类和更明确的格式设计，但对于不可变对象、值对象和安全敏感对象很有价值。

如果序列化结果需要写入文件或通过网络发送，还要把它放回 I/O 语境中考虑。对象流有头部，多个 `ObjectOutputStream` 连续写到同一个底层流会写入多个流头，读取端如果只按一个对象流解析可能失败；长期复用同一个对象流又会受到句柄表影响。网络上发送对象时，仍然需要消息边界，不能只把对象写入 socket 后期待接收端自然知道对象结束位置。文件中保存多个对象时，应定义数量、结束标记或外层容器，并处理读到 EOF 时是正常结束还是文件损坏。

## 小结

序列化把对象状态变成字节，但它真正影响的是系统边界和长期兼容性。Java 原生序列化提供了便利的对象图保存能力，也带来了类耦合、安全、性能和演化成本。使用它时，应明确数据是否可信、是否长期保存、是否跨版本、是否需要对象引用语义，并通过 `serialVersionUID`、`transient`、自定义 `readObject`、过滤器和兼容测试管理风险。对于许多接口和持久化场景，显式 schema 或文本/二进制数据格式更适合作为稳定契约。把序列化当成数据协议来设计，而不是当成对象的附属功能，才能避免后续版本和安全问题。

# 3.5.2 注解

## 定位

注解是 Java 对程序元素附加结构化元数据的机制。它可以标在类、接口、字段、方法、构造方法、参数、类型使用位置、包和模块等元素上，用来表达“这个元素具有某种额外语义”。注解本身不直接改变代码执行逻辑，它只是把信息写入源码、class 文件或运行时元数据中，真正产生效果的是编译器、注解处理器、静态分析工具、运行时框架或测试工具对这些信息的解释。

注解的价值在于把散落在外部配置、命名约定或手写注册代码中的信息，贴近被描述的程序元素。比如一个方法是否允许被某个框架调用，一个字段应该使用哪个外部名称，一个类型是否属于某个扩展点，一个参数是否不能为空，这些信息如果只放在文档或 XML 中，很容易和代码演进脱节。注解把约定放回代码附近，并且可以被编译器和工具读取。

注解不是万能配置语言。它适合表达稳定、局部、与程序结构紧密相关的元数据，不适合承载大量运行期频繁变化的业务配置。注解值必须是编译期常量、枚举、类字面量、注解或这些类型的数组，不能直接引用运行时对象。这个限制促使注解保持描述性，而不是把复杂逻辑塞进元数据里。

在通用 Java 技术视角下，注解通常有三类使用方式：编译器内建注解，例如 `@Override`、`@Deprecated`、`@SuppressWarnings`；编译期处理注解，通过 JSR 269 注解处理器生成代码、校验约束或输出资源；运行时注解，通过反射读取并驱动框架行为。理解注解要同时关注声明模型、保留策略、目标位置、读取方式和处理时机。

## 注解类型的声明模型

注解类型使用 `@interface` 声明。它看起来像接口，但内部声明的是注解元素。注解元素类似无参方法，可以有返回类型和默认值。允许的返回类型包括基本类型、`String`、`Class`、枚举、注解以及这些类型的一维数组。

```java
public @interface Column {
    String name();
    boolean nullable() default true;
    Class<?> converter() default Void.class;
}
```

使用注解时，需要为没有默认值的元素提供值。元素名为 `value` 且只有一个需要显式赋值的元素时，可以省略元素名。

```java
@Column(name = "user_name", nullable = false)
private String name;

@Tag("fast")
class FastCase {
}
```

注解元素不能返回任意对象，也不能有参数。这是因为注解要能被编译进 class 文件，并被编译器、工具和运行时以稳定格式读取。注解的表达能力受限，恰好也是它的优势：它天然更像声明式元数据，而不是隐藏的执行逻辑。

注解类型本质上会被编译成继承 `java.lang.annotation.Annotation` 的接口。运行时读取到的注解实例通常由 JDK 生成的代理对象表示，调用 `annotation.name()` 会返回注解元素值。注解实例的 `equals`、`hashCode`、`toString` 有专门规范，因此不要用普通对象的直觉理解它们。

注解元素的默认值不是复制到每个使用点，而是保存在注解类型本身的元数据中。读取某个注解时，如果使用点没有显式指定该元素，反射 API 会返回当前注解类型声明的默认值。这意味着修改注解类型的默认值可能影响所有未显式赋值的使用点，即使那些使用点的 class 文件没有变化。

注解元素还有几个容易被忽略的细节。第一，元素不能声明 `throws`，不能有参数，不能带类型参数，也不能使用递归类型作为返回值。第二，元素值必须能被编译器写进 class 文件的常量结构中，因此 `String name() default System.getProperty("user.name")` 这样的写法不成立，数组默认值也必须用注解语法给出常量数组。第三，数组元素在使用时可以省略一层大括号，但这只是语法便利，不改变元素类型。第四，注解元素虽然写成方法形式，但使用者不能把它理解成会执行任意逻辑的方法调用；反射拿到的注解实例只是在读取已记录的元数据。

```java
public @interface Checks {
    String[] value() default {};
    Level level() default Level.ERROR;
}

@Checks("name")
class SingleCheck {
}

@Checks(value = {"name", "email"}, level = Level.WARNING)
class MultipleChecks {
}
```

注解类型也可以嵌套使用。嵌套注解适合表达小型、固定、不会独立复用的结构，例如一个规则包含名称、严重级别和少量参数。如果嵌套层级过深，或者某个嵌套结构在多处具有独立生命周期，就应考虑把它提升为普通类型、配置对象或外部配置，而不是继续堆叠注解。注解的可读性很大程度来自“就近表达”，一旦读者需要在多层大括号和数组之间寻找真实含义，这个优势就会减弱。

```java
public @interface RuleSet {
    Rule[] value();

    @interface Rule {
        String name();
        Level level() default Level.ERROR;
    }
}
```

从 class 文件角度看，注解值记录的是名称到值的映射。使用点只保存显式写出的元素值，未写出的元素值由注解类型的默认值补齐。因此“新增一个有默认值的元素”通常对已有使用点兼容，而“新增一个没有默认值的元素”会让所有旧使用点在重新编译时失败。删除元素或改变元素返回类型也很危险：旧 class 文件里可能仍然带着原元素名，运行时读取时会遇到注解类型和使用点元数据不匹配的问题。对于已经发布给外部使用的注解，演进策略要像对待公开方法签名一样谨慎。

## 元注解：描述注解的注解

元注解用于描述注解类型自身。最常用的是 `@Retention`、`@Target`、`@Documented`、`@Inherited` 和 `@Repeatable`。它们决定注解能标在哪里、会被保留到哪个阶段、文档工具是否展示、类继承查询是否可见，以及同一位置能否重复标注。

`@Retention` 决定注解保留策略。`RetentionPolicy.SOURCE` 只保留在源码中，编译后丢弃，适合编译器检查和源码级工具。`RetentionPolicy.CLASS` 会写入 class 文件，但运行时反射通常不可见，适合字节码工具或编译期后续处理。`RetentionPolicy.RUNTIME` 会写入 class 文件并可被反射读取，适合运行时框架。

```java
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.TYPE, ElementType.METHOD})
public @interface Audited {
}
```

`@Target` 限定注解可出现的位置。常见目标包括 `TYPE`、`FIELD`、`METHOD`、`CONSTRUCTOR`、`PARAMETER`、`ANNOTATION_TYPE`、`PACKAGE`、`TYPE_PARAMETER`、`TYPE_USE` 等。没有声明 `@Target` 的注解可以用于多种传统声明位置，但明确声明目标更利于工具检查，也能避免误用。

`TYPE_USE` 是容易被忽视的目标，它允许注解出现在类型使用位置，例如 `List<@NonEmpty String>`、`@Readonly String[]`、`throws @Critical IOException`。这类注解常用于静态分析和类型检查扩展。它和 `FIELD`、`PARAMETER` 等声明位置不同：前者描述“类型的这次使用”，后者描述“这个字段或参数声明”。

`@Inherited` 的名字容易造成误解。它只影响类级注解通过 `Class.getAnnotation` 沿父类链查询的行为，不影响接口，不影响方法和字段，也不意味着子类 class 文件真的声明了该注解。若框架需要处理方法级注解继承，必须自己定义查找规则，例如是否沿重写链、接口方法和桥接方法查找。

`@Repeatable` 允许同一元素上重复使用同一种注解。它需要一个容器注解类型来承载多个值。反射读取时应优先使用 `getAnnotationsByType` 或 `getDeclaredAnnotationsByType`，它们会处理可重复注解和容器注解之间的展开关系。

```java
@Repeatable(Roles.class)
public @interface Role {
    String value();
}

public @interface Roles {
    Role[] value();
}
```

`@Documented` 的语义相对简单：被它标注的注解类型，在被使用到公开 API 上时，可以被文档工具纳入生成文档。它不影响编译、class 文件保留策略或运行时读取结果。公共库中用于约束使用者行为的注解，通常应考虑是否加上 `@Documented`，因为使用者阅读 API 文档时需要看到这些额外契约；内部实现细节注解则未必需要出现在文档中。

元注解之间也存在组合关系。例如一个可重复注解的容器注解通常应具有与被包含注解一致或更宽的 `@Retention` 和 `@Target`，否则反射展开和工具处理会出现不一致。一个运行时可重复注解如果容器不是运行时可见，运行时就无法通过容器恢复多个值。类似地，注解若既想标在类上又想标在其他注解类型上，就需要同时包含 `TYPE` 和 `ANNOTATION_TYPE`，后者是组合注解和元注解设计中常见的目标。

可以把常见元注解理解为几组问题：`@Target` 回答“能写在哪里”，`@Retention` 回答“能活到什么时候”，`@Inherited` 回答“类层级查询时是否沿父类找”，`@Repeatable` 回答“同一个位置能否出现多次”，`@Documented` 回答“是否作为公开文档的一部分”。设计注解时如果不能清晰回答这些问题，通常说明注解语义还没有收敛。

## 保留策略与处理时机

注解的处理时机决定了设计方向。源码级注解适合做编译器提示、错误抑制和源码生成输入。它们不进入运行时，无法通过反射读取，但能在编译阶段提供反馈。`@Override` 就是典型例子：它帮助编译器确认方法确实覆盖了父类型方法，运行时并不需要知道这个事实。

class 级注解会进入 class 文件，但不以运行时反射可见的方式保留。字节码分析工具、构建工具或编译后的处理步骤可以读取它们。选择 CLASS 保留策略通常意味着注解是给工具链看的，而不是给业务运行时看的。

运行时注解适合让框架在程序启动或执行过程中读取。例如测试框架发现测试方法，序列化框架读取字段别名，依赖注入容器识别构造方法和生命周期方法。运行时注解更灵活，但也引入反射扫描、访问控制、启动成本和运行时失败问题。

设计注解时不要无脑使用 RUNTIME。运行时可见会扩大元数据体积和反射可依赖面，也会让原本只应在编译期暴露的约定变成运行时兼容负担。如果注解只用于生成代码或静态校验，应优先选择 SOURCE 或 CLASS；只有确实需要反射读取时，才使用 RUNTIME。

同一个语义也可能同时有编译期和运行时版本。例如某个框架既可以在编译期生成注册表，也可以在运行时扫描注解。此时要明确哪一层是权威来源，避免生成代码和运行时扫描结果不一致。常见做法是编译期处理负责生成显式元数据，运行时只加载生成物，而不是重复全量扫描。

保留策略和目标位置共同决定注解能被谁看到。一个 `SOURCE` 注解即使标在方法上，也不会进入 class 文件；一个 `RUNTIME` 注解如果只允许标在 `TYPE_PARAMETER` 上，也不能用普通的类声明注解 API 读取到字段声明上的信息。很多“注解读取不到”的问题不是反射 API 本身失效，而是声明时没有把保留策略、目标位置和读取 API 对齐。

保留策略还会影响库的兼容承诺。`SOURCE` 注解通常只影响重新编译时的行为，旧二进制产物不会携带它；`CLASS` 注解会影响依赖 class 文件的工具；`RUNTIME` 注解会进入运行时反射契约，外部框架可能在不重新编译的情况下依赖它。保留阶段越靠后，注解越可能成为二进制兼容和运行时兼容的一部分。一个库若把注解从 RUNTIME 改成 CLASS，使用者代码未必编译失败，但运行时框架可能突然无法发现元数据，这类变化应视为破坏性变更。

在多模块工程中，还要区分“注解类型所在模块”和“使用注解的业务模块”。使用者编译时必须能访问注解类型，否则源码无法通过；运行时是否需要带上注解模块，则取决于保留策略和读取方式。若运行时反射会解析注解实例，注解类型本身也必须在运行时 classpath 或 module path 中可见。若注解只在 SOURCE 阶段被处理并生成普通代码，运行时未必需要携带处理器和注解定义，但具体取决于生成代码是否引用该注解类型。

## 标准注解与语言内建语义

Java 标准库提供了一组常见注解，它们展示了注解并不总是服务于运行时框架。`@Override` 是源码级检查，帮助编译器发现误以为覆盖、实际只是重载的方法。它的价值不在运行时，而在编译期把命名错误、参数列表错误、父类型变更等问题提前暴露出来。即使没有它，真正覆盖的方法仍然会覆盖；加上它，是为了让“我期望这里覆盖父类型方法”这个意图被编译器验证。

`@Deprecated` 表示被标注的 API 不再推荐使用。它可以标在类型、方法、字段、构造方法等元素上，并且在较新的 Java 版本中带有 `since` 和 `forRemoval` 元素，用于表达从哪个版本开始废弃、是否计划移除。废弃注解本身不阻止调用，通常只产生编译告警；真正的迁移质量取决于文档是否说明替代方案、废弃原因和兼容周期。如果只标注 `@Deprecated` 而没有解释，使用者仍然难以判断应该迁到哪里。

`@SuppressWarnings` 用于在局部范围抑制编译器或工具告警。它的危险在于范围过大时会遮蔽真实问题。实践中应尽量把它放在最小可行范围，例如单个局部变量、单个方法或单个类型，而不是整个包或大类；同时让抑制原因可以从上下文看出来。告警抑制不是修复问题，它只是告诉工具“这里的风险已经被人工判断过”。

`@SafeVarargs` 针对泛型可变参数带来的堆污染告警。泛型数组和可变参数组合时，编译器无法完全验证类型安全；当方法实现确实不会向可变参数数组写入不安全元素，也不会把数组暴露给不可信代码时，可以用该注解表达安全承诺。它本质上是作者对实现细节的声明，使用不当会把真实的类型安全问题隐藏起来。

`@FunctionalInterface` 用于声明某个接口被设计为函数式接口。它要求接口只有一个抽象方法，默认方法、静态方法以及 `Object` 上的公共方法不计入这个限制。这个注解不是函数式接口成立的必要条件：只要结构满足要求，接口就可以作为 lambda 或方法引用的目标类型。加上它的意义在于锁定设计意图，防止后续维护者无意中新增第二个抽象方法。

还有一类标准注解用于工具和语言模型协作，例如 `@Generated` 用于标记生成代码来源，`@Native` 用于提示常量可能被本地代码引用。理解这些标准注解时，不应只问“它运行时有没有效果”，而要问“哪个工具会读取它”。Java 注解体系的核心正是统一承载元数据，而不是把所有行为都压到运行时。

## 运行时读取与反射规则

运行时注解通常通过反射读取。`AnnotatedElement` 是类、方法、字段、构造方法、参数等元素共同实现的注解读取接口。常用方法包括 `isAnnotationPresent`、`getAnnotation`、`getAnnotations`、`getDeclaredAnnotation`、`getDeclaredAnnotations`、`getAnnotationsByType`。

```java
Retention retention = Audited.class.getAnnotation(Retention.class);
Audited audited = serviceClass.getAnnotation(Audited.class);
boolean present = method.isAnnotationPresent(Deprecated.class);
```

`getAnnotation` 和 `getDeclaredAnnotation` 的差异在于是否考虑继承可见性。对类级注解来说，`getAnnotation` 可能受到 `@Inherited` 影响；`getDeclaredAnnotation` 只看当前元素直接声明。方法、字段、构造方法等元素没有 `@Inherited` 的自动规则，因此框架若要支持“子类方法继承父类方法注解”，必须自行遍历父类和接口。

注解可以标在方法参数上，但参数名和参数注解是两回事。参数注解可以通过 `Method.getParameterAnnotations()` 或 `Parameter.getAnnotations()` 获取；参数名是否可读，取决于编译时是否保留参数元数据。框架不要默认所有 class 都带有可靠参数名，除非它控制了编译参数或提供了替代绑定方式。

类型使用注解需要通过 `AnnotatedType` 相关 API 读取，例如 `Field.getAnnotatedType()`、`Method.getAnnotatedReturnType()`、`Parameter.getAnnotatedType()`。如果只调用 `getAnnotations()`，拿到的是声明元素上的注解，而不是泛型参数、数组组件或通配符边界上的类型使用注解。

```java
Field field = Repository.class.getDeclaredField("names");
AnnotatedType annotatedType = field.getAnnotatedType();
```

读取注解时还要注意代理和合成元素。运行时生成的代理类、编译器生成的桥接方法、内部类产生的合成构造参数，都可能影响扫描结果。框架通常需要过滤 `isSynthetic()`、识别 `isBridge()`，并决定是否把桥接方法上的注解映射回用户声明的方法。

反射读取还有“直接声明”和“间接呈现”的区别。`getDeclaredAnnotations()` 只关心当前元素直接声明的注解；`getAnnotations()` 在类元素上可能包含从父类继承而来的注解；`getAnnotationsByType()` 会把可重复注解的容器展开。若框架把这几个 API 混用，很容易在可重复注解、继承注解和组合注解场景下得到不一致结果。一个稳妥的做法是先定义自己的注解查找语义，再封装成统一工具方法，而不是在各处临时调用反射 API。

注解反射还受到访问控制和模块边界影响。读取 public 元素上的运行时注解通常比较直接，但扫描非 public 成员、访问私有构造方法或跨模块深度反射时，可能受到 Java 语言访问规则和模块系统封装规则限制。注解本身可见不等于被标注的成员就可以被调用；读取到某个私有方法上的注解，也不代表可以无条件执行该方法。框架要把“读取元数据”和“访问目标成员”分成两个步骤处理。

注解实例是不可变视角下的元数据代理。反射返回的注解对象通常实现注解接口，并按照注解规范实现 `equals` 和 `hashCode`：同一注解类型、所有元素值相等，才认为两个注解相等；数组元素按内容比较而不是按数组引用比较。这个规则使注解可以安全地作为缓存键的一部分，但仍应注意 class loader 边界：同名注解类型如果由不同类加载器加载，在 JVM 看来不是同一个类型。

运行时扫描还要考虑成本。全 classpath 扫描不是 Java 反射天然提供的能力，通常需要框架自己枚举 classpath、读取目录或 jar，再逐个加载或解析 class 文件。加载类可能触发静态初始化之外的其他副作用风险，解析字节码则需要额外工具。成熟框架常通过索引文件、编译期生成物、限定包名、缓存和延迟解析来降低成本。对于普通业务代码，若只需要读取少量已知类上的注解，直接反射足够；若要发现大量未知类型，设计时就应把发现机制和性能成本纳入考虑。

## 类型使用注解

类型使用注解是 Java 注解体系从“声明元素元数据”扩展到“类型位置元数据”的关键能力。传统注解更多标在类、字段、方法、参数这些声明上；类型使用注解则标在类型出现的位置上。一个字段声明 `List<@NonEmpty String> names` 中，字段本身是 `names`，字段类型是 `List<String>`，而 `@NonEmpty` 描述的是泛型实参 `String` 这个类型使用位置。它表达的不是“字段 names 有某个属性”，而是“列表中的元素类型带有某个约束”。

这种区别在数组、泛型和通配符中尤其重要。`@A String[]`、`String @A []` 和 `String[] @A []` 可能描述不同层级的类型使用位置；`List<? extends @Readonly Document>` 描述的是通配符上界中的 `Document` 使用；`Map<@Key String, @Value Object>` 可以分别描述键和值的位置。静态分析工具可以利用这些细粒度信息做更准确的推断，普通声明注解则很难表达同样的结构。

类型使用注解不必都是运行时注解。很多类型使用注解服务于编译器插件、静态分析器或 IDE 检查，保留到 SOURCE 或 CLASS 就足够。若确实需要运行时读取，必须同时满足 `@Target(TYPE_USE)` 和 `@Retention(RUNTIME)`，并使用 `AnnotatedType` 家族 API。字段、返回值、参数、父类、接口、异常类型等位置都有相应的 annotated 访问入口，泛型实参和数组组件还需要继续向下解析。

```java
class Repository {
    private List<@NonEmpty String> names;
}

Field field = Repository.class.getDeclaredField("names");
AnnotatedParameterizedType listType =
        (AnnotatedParameterizedType) field.getAnnotatedType();
AnnotatedType elementType = listType.getAnnotatedActualTypeArguments()[0];
NonEmpty marker = elementType.getAnnotation(NonEmpty.class);
```

类型使用注解的误区是把它当成运行时强校验。标注 `List<@NonEmpty String>` 并不会自动检查列表里每个字符串是否非空。它只是提供了更精确的元数据，后续仍然需要静态分析器、编译器扩展、运行时验证器或框架读取并执行规则。另一个误区是忽略泛型擦除：运行时虽然可以通过反射读取声明处的泛型签名和类型使用注解，但对象实例本身并不会携带每个元素的泛型实参信息。注解标在声明上，不是给每个运行时对象单独附加标签。

选择 `TYPE_USE` 时也要避免过宽。许多注解实际上只适合声明位置，例如“这个方法是入口”“这个字段映射到外部名称”。把它们也允许用于类型使用位置，会让使用者写出语义不清的代码。反过来，可空性、取值范围、资源限定、文本格式等更贴近类型语义的约束，才更适合考虑 `TYPE_USE`。目标位置越精确，工具越容易给出明确错误，读者也越容易理解注解到底修饰谁。

## 编译期注解处理

注解处理器是 Java 编译器暴露的标准扩展点，通常基于 JSR 269。处理器在编译阶段读取源码和 classpath 上的程序元素，检查注解并生成新的源码、class 或资源。它不能像运行时反射那样执行目标程序，也不能依赖目标类已经完成加载；它面对的是编译器的元素模型，例如 `TypeElement`、`ExecutableElement`、`VariableElement`、`AnnotationMirror`。

编译期处理的优势是错误可以提前暴露，生成代码可以被普通编译器检查，运行时无需扫描大量类型。典型用途包括根据注解生成工厂、注册表、元模型、序列化适配器、查询 DSL 或配置索引。它也可以做静态校验，例如要求某个注解只能标在 public 类上，要求构造方法参数类型满足约束，要求多个注解组合不能冲突。

编译期处理也有边界。处理器只能看到当前编译轮次可见的源码和依赖符号，无法知道运行时动态加载的类；它生成的代码会参与后续编译轮次，但不能随意修改已有源码；它的性能会影响构建速度；如果处理器没有正确声明支持的注解、选项和增量属性，构建系统可能无法高效缓存。

注解设计若面向编译期处理，应尽量让信息在编译期可解析。使用 `Class<?>` 元素时，处理器读取 class 值可能遇到 `MirroredTypeException`，更通用的方式是通过 `AnnotationMirror` 和 `TypeMirror` 解析类型。注解值不要要求处理器执行用户代码，因为编译期模型并不是运行时对象模型。

注解处理器的基本入口是 `Processor` 接口，实际开发中常继承 `AbstractProcessor`。处理器声明自己支持哪些注解、支持哪个源码版本、接受哪些处理选项。编译器会把当前轮次中匹配的元素交给处理器，处理器可以通过 `Messager` 报告错误或警告，通过 `Filer` 生成源码和资源，通过 `Elements`、`Types` 查询语言模型。处理器报错应尽量绑定到具体元素，这样编译输出才能定位到使用者写错的位置。

```java
@SupportedAnnotationTypes("example.GenerateFactory")
@SupportedSourceVersion(SourceVersion.RELEASE_17)
public final class FactoryProcessor extends AbstractProcessor {
    @Override
    public boolean process(Set<? extends TypeElement> annotations,
                           RoundEnvironment roundEnv) {
        for (Element element : roundEnv.getElementsAnnotatedWith(GenerateFactory.class)) {
            // validate element and generate source
        }
        return true;
    }
}
```

编译期处理是分轮次的。某一轮处理器生成的新源码，会在后续轮次中继续进入编译器模型；当没有新的源码产生时，编译器会进入最后轮次，处理器可以做收尾校验。分轮次模型要求处理器避免假设“一开始就能看到所有生成类型”。如果处理器 A 生成了处理器 B 需要读取的类型，B 可能要等到后续轮次才能看到完整信息。处理器之间若没有清晰边界，构建结果会变得依赖执行顺序。

生成代码时，处理器不应覆盖同一轮或前一轮已经生成的同名文件。`Filer` 会阻止重复创建同一路径，避免多个处理器互相覆盖。生成代码也不应依赖不可重复的时间戳、随机数或本机绝对路径，否则会破坏构建缓存和可复现性。好的处理器输出应主要由源码、注解值、处理器选项和依赖类型共同决定。

注解处理器不能修改已有源码，只能生成新文件或报告诊断信息。这一点和某些源码转换工具不同。若希望改变用户代码行为，通常要生成辅助类、注册表、代理类或元数据文件，再让普通代码显式调用或让运行时框架加载这些生成物。把“校验”和“生成”分开思考会更清晰：校验负责阻止非法用法，生成负责把注解描述转换为普通 Java 代码或资源。

处理器还应关注增量构建。一个处理器如果读取了整个 classpath、扫描了不相关文件或把所有输出聚合到一个文件，构建工具就很难判断某个源文件变更会影响哪些输出。虽然不同构建系统对增量注解处理的声明方式不同，但通用原则是一致的：输入范围越精确，输出归属越清楚，缓存命中率越高；处理器越像全局扫描器，增量构建越困难。

编译期处理和运行时反射不是互斥关系，而是不同阶段的工具。编译期处理适合做确定性检查和生成，运行时反射适合处理动态发现、插件扩展和运行时环境相关决策。若同一框架同时提供两种路径，应明确优先级，例如“有生成索引时使用索引，否则退回反射扫描”。否则使用者很难解释为什么同一个注解在开发环境、测试环境和打包产物中行为不同。

## 注解与配置的边界

注解常被用作配置，但它和外部配置文件有不同边界。注解随代码一起编译和发布，适合描述代码结构自身的稳定属性；外部配置可以在不重新编译的情况下变化，适合环境差异、部署参数和业务开关。如果某个值经常按环境变化，把它写进注解通常会导致重新发布成本。

注解也不适合表达复杂流程。若一个注解拥有大量元素、嵌套数组和互斥规则，使用者很难理解，处理器或框架也难以给出清晰错误。此时应考虑把复杂配置移到类型安全的构建器、配置对象或外部 DSL 中，注解只保留入口标记或少量关键参数。

命名约定同样重要。注解名应表达语义而不是实现方式，例如 `@Transactional` 表达事务边界，`@Generated` 表达生成来源，`@Nullable` 表达空值约束。元素名应避免含糊的 `flag`、`type`、`value2`，默认值应代表最常见且安全的行为。布尔元素尤其要谨慎，多个布尔值组合容易形成难以解释的状态空间，枚举往往更清晰。

注解一旦被外部使用，就成为 API 的一部分。删除元素、修改默认值、改变保留策略、收窄目标位置、重命名注解类型，都可能破坏使用者代码或运行时行为。对于公共库，注解演进需要像类和方法一样考虑兼容性。新增可选元素通常较安全，新增必填元素则会让所有旧使用点编译失败。

还有一种边界是“局部结构元数据”和“全局装配规则”的边界。注解适合描述某个类、方法或参数自身的局部性质，但当规则需要跨多个模块、多个环境、多个部署单元协调时，单个注解往往不足以表达全局决策。例如优先级排序、条件启用、资源配额、路由规则、权限矩阵等，如果全部塞进注解，会让局部代码背负全局配置压力。更稳妥的做法是让注解表达候选能力或静态属性，再由外部配置、注册表或组合规则决定最终装配。

注解也不应该成为隐藏依赖的唯一入口。如果一个类只有在被某个扫描器发现时才会注册，而代码中没有任何显式引用，删除、重命名、模块化或打包裁剪时都可能出现难以发现的问题。编译期生成索引、单元测试覆盖扫描结果、启动阶段输出明确诊断，都可以降低这种隐式依赖风险。注解提高了声明的便利性，但也把一部分控制流转移给工具链，工程上需要相应的可观察性。

字符串元素是注解配置中最需要克制的部分。字符串可以表达名称、路径、表达式、查询片段、格式模板，但编译器无法理解它的内部结构。若字符串实际上只允许几个固定值，应改用枚举；若字符串引用 Java 类型，应优先使用 `Class<?>`；若字符串引用另一个程序元素，应考虑是否能通过类型、方法引用风格的生成代码或编译期校验来约束。无法避免字符串时，应提供清晰格式、转义规则和错误报告。

## 常见误区

第一个误区是认为标了注解就会自动生效。注解只是元数据，必须有编译器、处理器、框架或工具读取它。一个自定义 `@Log` 注解如果没有任何处理逻辑，就不会让方法自动打印日志。判断注解行为时，要问“谁在什么时候读取它，读取后做了什么”。

第二个误区是忽略保留策略。很多人定义注解后运行时读取不到，是因为没有设置 `@Retention(RetentionPolicy.RUNTIME)`。反过来，也有人把只用于编译期的注解设为 RUNTIME，造成不必要的运行时暴露。保留策略应该由处理时机决定，而不是固定模板。

第三个误区是误解 `@Inherited`。它不会让方法注解继承，也不会让接口注解自动出现在实现类上。依赖注解继承的框架必须清楚写出自己的查找规则，否则使用者会在继承、重写和接口默认方法场景中得到意外结果。

第四个误区是把注解当成依赖注入或执行逻辑本身。注解可以声明“需要某种行为”，但行为应由清晰的框架流程、代理、生成代码或调用约定实现。如果大量业务分支隐藏在注解组合中，代码会变得难以追踪和测试。

第五个误区是忽略类型使用注解和声明注解的区别。`@NotNull String name` 里的注解可能是修饰参数声明，也可能是修饰类型使用，取决于注解目标。静态分析工具和运行时框架读取的 API 不同，设计时必须明确希望表达的是“这个参数”还是“这个类型位置”。

第六个误区是把注解默认值当成“写死在使用点”。默认值属于注解类型，不属于每个使用点。修改默认值可能改变所有未显式赋值位置的读取结果，尤其是运行时框架读取注解时，旧业务 class 文件也可能受到新注解类型默认值影响。因此默认值应当稳定，公共注解的默认值变更要按兼容性变更处理。

第七个误区是认为注解可以可靠表达所有继承语义。Java 语言中的方法重写、接口实现、桥接方法、泛型擦除和默认方法会让“从哪里继承注解”变成框架级策略，而不是语言统一规则。即使两个框架都使用同一个注解，也可能在接口方法和父类方法上采取不同查找顺序。使用注解定义公共契约时，应把继承规则写进文档或处理器校验中。

第八个误区是忽略注解处理器的构建成本。处理器运行在编译链路中，任何全局扫描、低效 I/O、不可缓存输出都会直接拉长反馈周期。一个注解看起来只是一行标记，但它背后可能触发大量符号解析和代码生成。设计编译期注解时，除了语义正确，还要关注处理器是否可增量、错误信息是否清晰、生成代码是否稳定。

## 实践建议

设计注解前先确定处理者和处理时机。若处理者是编译器或注解处理器，优先使用 SOURCE 或 CLASS；若处理者是运行时框架，再使用 RUNTIME。为注解写清楚目标元素、默认值语义、是否可重复、是否参与继承查找，以及错误使用时处理器或框架会如何报告。

注解元素保持少而稳定。优先使用枚举表达有限模式，使用 `Class<?>` 表达类型引用，使用嵌套注解表达小型结构，避免让字符串承担过多隐式语义。对字符串值要定义格式、大小写、空值含义和冲突规则。默认值要可预测，不要让省略参数产生危险行为。

运行时框架读取注解时，应集中封装扫描逻辑，缓存解析结果，并在启动阶段校验明显错误。对继承、接口、桥接方法、可重复注解和类型使用注解，要有一致规则。编译期处理器读取注解时，应使用编译器元素模型，不要强行加载用户类，也不要依赖运行时反射。

测试注解行为时，不只测试“标了注解能生效”，还要测试未标注、默认值、重复标注、非法组合、继承重写、错误目标和版本兼容。注解越接近公共 API，越需要用测试锁定处理规则。

对公共注解，建议把文档写在注解类型本身，而不是只写在框架说明页。文档至少说明：这个注解修饰的对象是什么，保留策略为何如此，目标位置为何如此，谁会读取它，默认值的语义是什么，与相关注解同时出现时如何合并或冲突，非法使用会在编译期还是运行期报错。注解越声明式，越需要明确解释声明和实际行为之间的连接。

对运行时注解扫描，建议把扫描范围做成显式输入。让使用者传入根包名、候选类集合、配置类或生成索引，比默认扫描整个 classpath 更可控。扫描结果应在启动阶段尽早校验，错误信息要指向具体类、成员和注解值。若扫描结果会缓存，缓存键应考虑 class loader、配置选项和相关注解版本，避免在动态加载或测试隔离场景下复用错误结果。

对编译期处理器，建议把错误尽量前置到编译失败，而不是生成一段之后必然运行失败的代码。处理器应避免吞掉异常后继续生成不完整文件；也应避免只输出普通日志而不报告编译诊断。使用 `ERROR` 级别阻止非法用法，用 `WARNING` 级别提示可疑但仍可编译的用法，用 `NOTE` 级别输出少量调试信息。诊断信息本身也是注解 API 体验的一部分。

## 组合注解与语义分层

在较大的代码库中，一个元素上可能同时出现多个注解。它们有的来自语言或标准库，有的来自静态分析工具，有的来自框架，有的来自项目自身。设计注解体系时，需要避免让多个注解在同一语义层面互相竞争。例如一个注解声明生命周期，一个注解声明可见范围，一个注解声明外部名称，这三者职责清晰；如果两个注解都试图声明同一个开关，只是默认值不同，就会制造优先级问题。

组合注解是一种常见做法：在一个自定义注解上再标注其他元注解或框架注解，让使用者用一个更贴近业务语义的注解表达一组固定配置。组合注解的好处是减少重复、统一命名、隐藏底层配置细节；风险是语义来源变得间接。框架若支持组合注解，需要定义搜索深度、冲突处理、别名机制和循环引用保护。否则同一个元素直接标注和通过组合标注时，结果可能不一致。

注解语义还应分层。底层注解描述技术机制，例如“这个方法需要被某个处理器扫描”；上层注解描述领域含义，例如“这是一个公开命令”。如果所有代码都直接使用底层技术注解，未来替换处理器或调整框架时会影响大量使用点。通过上层语义注解隔离技术细节，可以让注解成为稳定契约，而不是实现细节泄漏。

但组合和分层不应过度。注解链过深会让读者难以判断实际生效规则，也会增加扫描成本和错误提示难度。一个实用原则是：使用者看到注解名时，应能大致理解它带来的约束；处理器报错时，应能定位到用户直接写下的注解，而不是只指出内部组合注解。

组合注解还涉及别名问题。一个上层注解可能希望把自己的 `name` 元素映射到底层注解的 `value` 元素，或者把多个底层元素固定为某组值。Java 语言本身不提供注解元素别名机制，这类能力必须由框架或处理器定义。如果框架没有明确支持，单纯把一个注解标在另一个注解上，并不会自动让底层注解的元素值从上层注解传递过去。使用组合注解前，要确认处理者是否真的会递归解析元注解，以及如何处理元素覆盖。

冲突合并也要提前定义。假设一个方法直接标注了 `@Role("admin")`，同时又通过组合注解间接带有 `@Role("user")`，最终是两个角色都生效，直接标注覆盖间接标注，还是视为冲突报错？不同语义会导向不同实现。安全相关、事务相关、资源生命周期相关的注解尤其不适合默默合并，因为使用者可能误以为某个更严格约束已经覆盖了宽松约束。清晰的冲突规则比“尽量猜测使用者意图”更可靠。

## 与类型系统和文档的关系

注解经常被用来补充类型系统无法直接表达的信息，例如可空性、线程安全约束、资源所有权、外部协议字段名、序列化格式等。它们能增强工具理解代码的能力，但不能替代类型系统本身。能用类型表达的约束，通常优先用类型表达；只有类型系统表达不了、表达成本过高或需要工具跨切面识别时，再考虑注解。

例如一个参数只能接受有限模式时，枚举类型比字符串加注解更稳；一个对象必须先构造完成才能使用时，构造方法和不可变字段比 `@Initialized` 注解更可靠；一个方法可能失败时，清晰的返回类型或异常模型比只标注 `@MayFail` 更可执行。注解适合让工具理解额外语义，但运行时正确性不应完全依赖读者记得遵守注解说明。

注解也承担文档角色。`@Deprecated` 不只是给编译器看的标记，还应配合说明替代方案、移除计划和适用范围。公共注解类型本身应有文档，解释它能标在哪里、谁会读取它、默认值是什么意思、与其他注解冲突时如何处理。缺少文档的注解会变成隐式协议，使用者只能通过阅读处理器或框架源码猜测行为。

当注解用于静态分析时，还要区分“强约束”和“提示”。强约束违反后应让编译或检查失败；提示只帮助工具推断，不一定能覆盖所有路径。把提示性注解当成强安全边界，会造成错误安全感；把强约束注解只当文档，又会让规则无法落地。

## 小结

注解是 Java 元数据系统的核心。它把结构化说明附着在程序元素上，让编译器、工具和运行时框架可以基于统一模型进行检查、生成和执行。正确使用注解的关键不是会写 `@interface`，而是明确它描述什么、保留到哪个阶段、由谁读取、如何处理错误、如何随 API 演进。把注解用于稳定的声明式语义，避免把复杂可变逻辑塞进注解，才能让它成为清晰的契约，而不是隐藏耦合的来源。

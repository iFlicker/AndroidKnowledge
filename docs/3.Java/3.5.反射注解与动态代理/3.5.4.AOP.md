# 3.5.4 AOP

## 定位

AOP 是 Aspect-Oriented Programming，即面向切面编程。它关注的是一类横跨多个模块、多个对象、多个方法的逻辑如何组织。日志、权限、事务、缓存、重试、审计、指标采集、限流、资源关闭等逻辑通常不属于某个单一业务对象，却又需要围绕大量业务方法执行。如果把它们直接写进每个方法，业务代码会被重复样板淹没；如果完全依赖调用者记得手动执行，又容易遗漏。AOP 提供了一套把横切逻辑从主业务流程中分离出来，并在特定连接点织入执行的思路。

AOP 不是某个特定框架的同义词，也不等于动态代理。动态代理是实现 AOP 的一种常见手段，编译期或类加载期字节码织入也可以实现 AOP，甚至手写模板方法和装饰器也能表达部分 AOP 思想。理解 AOP 的重点是概念模型和边界：哪些点可以被拦截，拦截逻辑如何声明，何时织入，执行顺序如何确定，失败时如何传播。

在通用 Java 语境中，AOP 最常见的实现方式有三类：基于代理的运行时 AOP，基于编译期或后编译期的字节码织入，基于类加载或代理库的运行时增强。代理 AOP 简单、标准、易于按对象边界启用，但通常只能拦截通过代理对象发起的调用。字节码织入能力更强，可以改变目标类自身的字节码，但工具链复杂度和调试成本也更高。

AOP 的目标不是让所有逻辑都“切面化”。业务规则、领域流程和核心状态变化仍应清晰地写在业务模型中。适合切面的逻辑通常具备几个特征：与业务意图正交，重复出现在多个位置，需要围绕方法执行前后生效，执行策略可以用统一规则描述，并且即使抽离出去也不会让业务语义变得难以理解。

## 核心概念

连接点是程序执行过程中可以被增强的位置。不同 AOP 实现支持的连接点范围不同。代理型 AOP 通常支持方法调用或方法执行；字节码织入可能支持字段访问、构造方法、异常处理块、静态初始化等更细粒度位置。讨论 AOP 时必须明确连接点范围，否则“为什么这个地方没有生效”会变成长期困惑。

切点是对连接点的筛选规则。它描述哪些类、哪些方法、哪些注解、哪些包、哪些参数模式应该被切面处理。切点可以基于名称表达式，也可以基于注解、类型层次或显式注册。切点越宽，越容易误伤不该增强的方法；切点越窄，越容易遗漏。公共切点应追求可读和稳定，而不是炫技式表达。

通知是切面在连接点附近执行的动作。常见通知类型包括前置通知、后置返回通知、异常通知、最终通知和环绕通知。环绕通知最强，可以决定是否调用原方法、调用几次、何时调用、如何处理返回值和异常。能力越强，越需要约束；很多场景只需要前置或最终通知，不应滥用环绕通知改变业务语义。

切面是切点和通知的组合。它定义“在哪些地方执行什么横切逻辑”。织入是把切面应用到目标程序的过程。织入时机可以是编译期、后编译期、类加载期或运行期。引介则是给目标类型增加新的接口、方法或状态的能力，并非所有 AOP 实现都支持。

这些概念的意义在于让横切逻辑可以被讨论和管理。没有这些边界，所谓 AOP 很容易退化成“某个框架帮我在方法前后做点事”，一旦多个切面叠加、异常传播、顺序冲突或性能问题出现，就很难分析。

## 基于代理的 AOP

基于代理的 AOP 最容易理解。框架创建一个代理对象，调用者依赖代理对象而不是直接依赖目标对象。代理在方法入口处匹配切点，按顺序执行拦截器链，最终调用目标方法。JDK 动态代理可以代理接口，子类代理可以代理可继承的类。两者都属于“调用必须经过代理对象”这一类模型。

一个简化的拦截器链可以这样理解：

```java
interface MethodInterceptor {
    Object invoke(MethodInvocation invocation) throws Throwable;
}

interface MethodInvocation {
    Method method();
    Object[] arguments();
    Object proceed() throws Throwable;
}
```

每个拦截器拿到调用上下文，执行自己的逻辑，然后选择是否调用 `proceed()` 进入下一个拦截器或目标方法。计时拦截器会在 `proceed` 前记录开始时间，在 finally 中记录耗时；权限拦截器可能在检查失败时直接抛出异常，不再调用后续逻辑；缓存拦截器可能命中缓存后直接返回。

代理 AOP 的优点是实现简单、按对象组合、易于在运行时决定是否启用。它不需要修改目标类字节码，通常也更容易与依赖注入容器结合。缺点是边界清楚但容易被忽略：目标对象内部的自调用不会经过代理；final 方法、private 方法、静态方法、构造方法通常不能通过普通代理拦截；如果调用者绕过容器直接 new 目标对象，切面也不会生效。

代理 AOP 的另一个问题是对象身份。调用者拿到的是代理对象，不是原始目标。`getClass()`、注解读取、泛型信息、`equals` 和序列化行为都可能受到影响。框架需要决定对外暴露代理类型还是目标类型信息，并提供稳定的工具方法处理代理拆包或目标类识别。

## 字节码织入

字节码织入通过修改或生成 class 字节码，把切面逻辑直接插入目标类。织入可以发生在编译期，也可以发生在编译后处理阶段，还可以发生在类加载期间。与代理不同，织入后的目标类本身就包含增强逻辑，因此自调用、构造方法或非接口方法等场景可以获得更强支持，具体能力取决于织入工具。

编译期织入的优势是运行时路径更直接，很多错误能在构建阶段暴露，部署时不需要额外代理对象。缺点是构建链路更复杂，生成或修改后的字节码与源码之间存在距离，调试时行号、调用栈和工具识别可能更难理解。后编译期织入还要处理增量构建、缓存和依赖 class 的一致性。

类加载期织入在类被加载进 JVM 前修改字节码，灵活性更高，可以按运行环境决定增强策略。但它依赖类加载机制和代理参数，排查问题时需要确认类是否已经被加载、是否被同一个类加载器处理、转换器顺序如何、多个工具是否同时修改同一类。

字节码织入不意味着可以无成本地增强任何位置。修改字节码必须保持验证通过，不能破坏栈帧、异常表、泛型签名、调试信息和访问规则。插入代码还可能改变初始化时机、异常路径和性能特征。织入越底层，越需要严格测试和可观测性。

## 调用流程与通知顺序

多个切面同时作用于同一个连接点时，顺序是核心问题。以环绕通知为例，外层通知先进入，内层通知后进入，目标方法执行后再按相反顺序退出。资源管理、事务、缓存和异常转换的相对顺序会直接改变语义。

例如事务和重试同时存在时，重试在事务外层意味着每次重试都开启新事务；事务在重试外层意味着多个尝试可能落在同一个事务边界内，通常不是预期。权限检查应在昂贵操作前执行；指标记录通常应包住尽可能完整的调用；异常转换的位置会影响上层看到的异常类型。AOP 框架必须提供明确的排序机制，使用者也必须把顺序当成设计的一部分。

通知类型也影响异常路径。前置通知抛出异常时，目标方法不会执行；后置返回通知只在正常返回后执行；异常通知只在抛出异常时执行；最终通知无论成功失败都会执行；环绕通知如果吞掉异常，外层通知和调用者就看不到原始失败。切面代码不应随意吞异常，除非它的职责就是容错、降级或异常转换。

返回值处理同样需要谨慎。环绕通知可以替换返回值，但必须满足方法返回类型约束。对基本类型返回值，返回 `null` 会导致拆箱异常。对泛型返回值，运行时擦除可能掩盖部分类型问题，直到调用者使用结果时才暴露。切面若修改参数或返回值，应有明确契约。

## 注解驱动的切点

注解常被用于声明切点。相比复杂的名称表达式，注解更贴近代码语义。例如在方法上标注某个注解表示需要审计，在类上标注某个注解表示该类所有 public 方法进入监控。注解驱动的切点易读，也利于编译期检查目标位置。

但注解切点同样有边界。注解保留策略必须能被处理时机读取：运行时代理需要 RUNTIME 注解，编译期织入可以使用 SOURCE 或 CLASS 注解。注解放在接口方法还是实现方法上，也会影响代理能否读取到。JDK 动态代理面对的是接口方法，子类代理面对的是目标类方法；框架若想同时支持两者，就需要定义合并规则。

继承和重写是另一个复杂点。子类重写方法后是否继承父类方法注解？接口方法注解是否作用于实现方法？类级注解是否作用于所有方法？多个注解同时存在时哪个优先？这些规则不能依赖使用者猜测，应在框架层固定下来。

注解不应承载过多切面逻辑。一个注解声明“需要缓存”是合理的；把缓存键表达式、异常降级脚本、线程模型、序列化策略和重试流程全部塞进一个注解，往往会变成难以维护的小型语言。复杂策略更适合放在配置对象或代码扩展点中，注解只作为入口标记。

## 性能与可观测性

AOP 的性能成本来自匹配切点、创建调用上下文、执行拦截器链、反射或方法句柄调用、异常包装以及可能的代理层级。多数业务系统中，单次方法拦截成本不是最大瓶颈，但大量细粒度方法被切面包裹时，成本会累积。特别是集合循环中的小方法、低延迟计算路径和序列化热点路径，应避免无差别切入。

优化的第一原则是缩小切点。不要用过宽规则匹配整个包下所有方法，再在运行时用 if 排除大部分方法。应在代理创建或织入阶段就确定哪些方法需要增强。第二是缓存元数据，注解解析、方法签名分析、表达式编译都应尽量在启动阶段完成。第三是控制拦截器数量和顺序，让常见快速失败逻辑靠前。

可观测性比单纯性能更重要。AOP 会改变调用栈，切面异常可能包裹业务异常，多个代理层可能让日志难以阅读。因此框架应提供清晰的切面启用日志、匹配结果诊断、执行耗时指标和异常链保留。使用者也应避免在切面里写模糊日志，例如只打印“执行失败”，却不包含方法签名、切面名和原始异常。

调试 AOP 时，第一步应确认调用是否真的经过代理或织入后的类；第二步确认切点是否匹配；第三步确认通知顺序；第四步确认目标方法是否执行；第五步确认异常是否被转换或吞掉。按这个流程排查，比盲目修改切点表达式更可靠。

## 设计取舍

AOP 带来的主要收益是关注点分离和一致性。横切逻辑集中后，可以统一修改策略，减少重复代码，也更容易审计。例如所有需要鉴权的方法都经过同一套权限切面，所有需要计时的服务都输出统一指标。

主要代价是隐式性。阅读一个业务方法时，看不到所有会在它前后执行的逻辑。过度 AOP 会让代码行为必须通过框架配置、注解、代理链和织入报告拼起来才能理解。越接近领域核心的逻辑，越应显式；越是技术性、重复性、边界性的逻辑，越适合切面。

另一个取舍是局部灵活性和全局一致性。切面集中管理后，个别方法想要特殊处理可能需要额外注解或排除规则。排除规则太多说明切点设计可能过宽，或者这个横切逻辑并不适合统一抽象。

实现方式也要按团队和系统成熟度选择。代理 AOP 足以覆盖大多数服务边界方法，调试和部署成本较低。字节码织入适合确实需要拦截更细粒度连接点或避免代理边界的场景，但应配套构建、测试、诊断和回滚方案。

## 与相邻技术的边界

AOP 经常和装饰器、拦截器、过滤器、模板方法、事件监听混在一起讨论。它们确实都能把一段逻辑放在主流程之外，但抽象重点不同。装饰器强调对象组合：调用者明确拿到一个包装后的对象，包装层对单个对象或一组接口增加行为。拦截器强调调用链：请求进入某个框架边界后，按顺序经过多个处理节点。过滤器强调输入输出流或请求响应的前后处理。模板方法强调父类固定流程，子类填充步骤。事件监听强调在状态变化后广播事实，让监听者异步或同步响应。

AOP 的独特之处在于“按连接点声明匹配规则”。开发者不一定显式组合每一个目标对象，也不一定在每个调用入口手动注册拦截链，而是通过切点表达“符合这些条件的执行点都要被增强”。这种声明式匹配带来了批量治理能力，也带来了行为不够显眼的问题。判断是否需要 AOP 时，可以问一个简单问题：这段逻辑是否真正横跨多个稳定边界，并且它的启用规则能被清楚声明？如果答案是否定的，普通组合或显式调用往往更容易理解。

装饰器比 AOP 更显式。阅读构造代码时可以看到对象被哪些包装层包住，单元测试也可以直接替换某个包装层。缺点是当同一种增强要应用到大量对象时，装配代码会膨胀，且不同对象可能出现包装顺序不一致。AOP 把装配集中到切点和通知中，适合统一策略，但阅读单个类时看不到完整调用路径。

模板方法适合稳定流程内部的步骤复用，例如“打开资源、执行操作、关闭资源”这类由父类控制的结构。它要求目标代码主动继承或调用模板，因此侵入性更强，但语义也更直接。AOP 则适合目标代码不应感知技术性增强的场景，例如统计耗时、输出审计记录、做统一异常转换。若某段逻辑是业务流程不可分割的一步，就不应为了复用而切成通知；它应该留在领域方法、应用服务或模板流程里。

事件监听适合“某件事已经发生”的扩展点。监听者通常不应阻止原操作成功，也不应依赖监听顺序改变主流程语义。AOP 可以在目标方法执行前阻断调用，也可以在环绕通知中改变返回值或异常，因此权力更大。权力越大，越需要明确约束。很多审计、消息投递、缓存刷新任务看似可以用 AOP，也可以用事件；如果它们必须决定目标方法能否执行，AOP 更合适；如果它们只是对结果做后续响应，事件往往更松耦合。

## 代理式 AOP 与织入式 AOP 的系统差异

代理式 AOP 和字节码织入都能实现“在方法前后执行横切逻辑”，但它们解决问题的方式不同。代理式 AOP 改变的是对象引用：调用者实际调用代理对象，代理对象再决定是否转发到目标对象。织入式 AOP 改变的是类定义：目标类加载或执行前已经被插入增强逻辑。前者像在对象外部套一层调用入口，后者像把增强语句写进目标类的字节码。

| 维度 | 代理式 AOP | 字节码织入 |
| --- | --- | --- |
| 生效前提 | 调用必须经过代理对象 | 目标类字节码已被织入或加载期转换 |
| 常见连接点 | public 方法、接口方法、可代理类方法 | 方法执行、构造、字段访问等，取决于工具能力 |
| 自调用 | 通常不生效，因为没有经过代理 | 通常可以生效，因为类自身已被增强 |
| 构建复杂度 | 较低，多在运行时创建代理 | 较高，需要编译期、后编译期或类加载期工具链 |
| 调试体验 | 需要识别代理层和拦截器链 | 需要理解织入后的字节码、行号和调用栈 |
| 适用范围 | 服务边界、接口边界、对象协作边界 | 需要更细粒度连接点或无法接受代理边界的场景 |
| 主要风险 | 绕过代理、自调用失效、代理类型差异 | 字节码兼容、构建缓存、类加载顺序、工具冲突 |

代理式 AOP 的“窄”并不一定是缺点。很多系统真正需要增强的是应用服务、仓储接口、远程调用客户端、任务处理器这类对象边界。把切面限制在这些边界上，反而能让行为更可控。若一个设计要求切面深入到对象内部的 private 方法或每一次字段读取，应该重新审视是否真的需要 AOP，还是领域模型的边界没有划清。

织入式 AOP 的“强”也不自动意味着更好。它可以覆盖更多连接点，但这会扩大问题空间。被织入的类在源码中看不到完整逻辑，排查时要同时考虑源码、织入规则、构建产物和运行时加载路径。它适合少数对连接点粒度有硬需求的基础设施能力，而不适合把所有复用逻辑都搬到底层织入。对于普通业务系统，优先使用代理式 AOP，只有在代理边界无法满足明确需求时再考虑织入，通常是更稳妥的顺序。

选择实现方式时还要看失败后的恢复成本。代理配置错误通常可以通过调整容器配置、切点表达式或对象注入方式修复；织入错误可能导致类加载失败、验证失败或难以定位的运行时异常。代理式方案的灰度和回滚也更简单，因为它常常可以按对象或配置开关控制。织入式方案一旦进入构建产物，回滚通常需要重新构建和发布。

## 可测试性设计

AOP 的可测试性不能只依赖“业务方法单测通过”。切面改变的是调用路径，测试也必须覆盖调用路径。一个目标类直接 new 出来调用时，代理式切面不会生效；如果单测只测这个目标类本身，就无法证明切面配置正确。对关键切面，应至少有一层集成测试通过真实装配方式拿到代理对象或织入后的类，再触发实际方法调用。

切面测试可以分成三层。第一层是切面内部逻辑测试，把通知中的纯逻辑抽成普通类或函数，直接验证输入输出。例如缓存键生成、异常分类、指标标签构造、权限决策都可以不依赖 AOP 框架测试。第二层是拦截器链测试，用一个可控的 `MethodInvocation` 或测试目标方法验证 `proceed()` 是否被调用、调用几次、异常如何传播。第三层是装配测试，验证切点是否匹配预期方法、排除规则是否生效、多个切面顺序是否符合设计。

这种分层能避免两种极端。第一种极端是所有测试都启动完整容器，导致测试慢、定位困难、反馈不清晰。第二种极端是只测纯逻辑，不测实际织入路径，最后上线才发现调用根本没有经过代理。切面越基础，越应该把纯逻辑和装配路径都测到；切面越接近观察性能力，越要确认自身失败不会污染业务结果。

测试异常路径尤其重要。很多切面在成功路径上很简单，真正复杂的是失败时如何处理：前置检查失败是否阻止目标方法，目标方法抛异常后异常通知是否保留原始 cause，最终通知清理失败是否覆盖主异常，环绕通知是否在某些分支忘记调用 `proceed()`。这些行为都应通过小而明确的测试命名表达出来。测试名最好说明语义，而不是只写“testAop”。

顺序测试也不可省略。多个切面叠加时，仅验证每个切面单独工作并不够。可以用记录列表、计数器或测试事件收集器记录进入和退出顺序，断言外层、内层、目标方法、异常处理和 finally 的相对位置。不要依赖日志文本作为唯一断言，因为日志格式经常调整；更好的方式是暴露测试专用的观察点或使用可替换的记录接口。

对织入式方案，还应把构建产物纳入测试。源码编译通过不代表织入结果正确，单独测试未织入类也不能覆盖真实行为。可以在构建后运行一组针对织入产物的测试，确认目标类能加载、目标方法能执行、行号和异常栈仍可用于排障。若织入规则由配置生成，还要测试配置变更对匹配范围的影响。

## 失败模式与排障路径

AOP 问题常见的困难不是“代码一定很复杂”，而是失败现象和原因之间隔着一层间接性。方法没有增强，可能是切点没匹配，也可能是对象没有被代理，还可能是代理类型不对、方法不可代理、注解读取位置不对、类已经提前加载、织入工具没有执行。方法被增强了但结果不对，可能是通知顺序错误、异常被转换、返回值被替换、上下文泄漏或某个环绕通知调用了多次 `proceed()`。

排障时应先确认“有没有进入切面”，再确认“进入的是哪个切面”。如果完全没有进入，优先检查对象来源和切点匹配；如果进入了错误切面，检查切点过宽和排序规则；如果进入顺序错误，检查显式优先级而不是猜测框架默认；如果目标方法没执行，检查环绕通知和前置检查；如果异常不符合预期，检查异常通知、finally 清理和外层包装。

代理式 AOP 的第一类失败是绕过代理。目标对象被手动创建、从错误工厂取得，或在构造方法中把 `this` 暴露出去，都会让后续调用没有经过代理。第二类失败是方法形态不支持代理，例如 private 方法、final 方法、静态方法或不可继承类型。第三类失败是自调用：同一个对象内部从一个方法调用另一个方法，本质上是 `this` 调用，不会重新进入代理层。解决这些问题不能只靠改切点表达式，而要调整对象边界或调用路径。

织入式 AOP 的失败更偏工具链。类是否经过织入、织入发生在编译期还是加载期、类加载器是否一致、多个转换器顺序如何、增量构建是否复用了旧产物，都会影响结果。排查时要拿到可验证的证据，例如构建日志、织入报告、加载期转换日志或反编译后的目标片段。没有证据时，容易把构建问题误判成运行时逻辑问题。

还有一类失败来自上下文和线程。切面设置了线程局部状态，却没有在异常路径清理；异步任务读取不到调用线程中的上下文；并发调用共享了切面实例字段；重试切面在多线程环境下复用了可变参数对象。这些问题与 AOP 框架本身无关，而是横切逻辑扩大了共享状态的影响范围。切面代码应默认按多线程环境设计，不把调用级数据放进共享字段。

## 团队治理与文档化

AOP 一旦成为公共机制，就不只是代码技巧，而是团队协作规则。新增一个切面，意味着未来阅读任何匹配方法时都要把这个切面纳入心智模型。修改一个切点，可能影响几十个甚至几百个方法。因为影响面大，切面配置、切点表达式和注解语义都应该像公共接口一样接受评审。

文档化的重点不是罗列框架用法，而是说明行为契约。每个切面至少应回答：它匹配哪些连接点，默认是否启用，是否允许跳过目标方法，是否会修改参数、返回值或异常，顺序要求是什么，失败时如何处理，如何排除某个方法，如何在测试中验证。对于会改变执行次数的切面，还要说明目标方法需要满足什么前提，例如幂等性、可重入性或补偿能力。

切面命名也要克制。名字应表达职责，而不是表达实现手段。`AuditAspect` 比 `AnnotationAroundAspect` 更有信息量，`RetryInterceptor` 比 `CommonAspect` 更可维护。切点命名同样重要，一个名为 `serviceMethods` 的切点如果实际上匹配了工具类和配置类，会持续制造误解。切点表达式越复杂，越需要拆成具名片段，并为每个片段写清楚意图。

团队还应建立变更策略。新增观察性切面可以先在开发或测试环境打开匹配报告，确认范围后再进入生产。新增会改变语义的切面应采用显式 opt-in，不应默认扫整个包。修改已有切面顺序时，要列出受影响的其他切面，并用测试固定关键顺序。删除或收窄切点时，要确认是否有方法依赖旧行为，否则看似删除无用增强，实际可能移除了必要的资源边界。

最后，AOP 不应成为逃避模块设计的工具。如果很多逻辑都只能靠切面补进去，说明对象之间的显式协作可能不够清楚。如果一个方法必须依赖多个不可见切面才能保持正确，说明它的前置条件和后置条件没有被清楚表达。优秀的 AOP 使用方式，是让技术性横切逻辑从业务代码中退场，同时让系统行为仍然可以被定位、解释和测试。

## 常见横切职责的差异

不同横切职责看起来都可以写成“方法前后做点事”，但语义差异很大。日志切面通常只观察调用，不应该改变返回值和异常；指标切面需要覆盖成功、失败和耗时，并尽量保证自身失败不影响业务；权限切面在目标方法前执行，失败时应阻止目标方法；缓存切面可能跳过目标方法，并且需要定义键、过期、穿透和异常策略；重试切面可能多次调用目标方法，因此目标方法必须能承受重复执行；事务切面需要管理资源边界，并和异常传播规则绑定。

这些差异决定了切面不能只按技术形态分类。一个环绕通知可以实现缓存，也可以实现重试，还可以实现异常转换，但它们对目标方法的要求完全不同。缓存要求方法结果可由参数决定，重试要求方法具备幂等性或有补偿策略，事务要求资源参与同一一致性边界。切面文档应说明它对被增强方法的前提假设，而不是只说明如何开启。

以重试和缓存为例，如果一个方法会修改外部状态，简单重试可能导致重复写入；如果一个方法返回与当前时间、随机数或调用者身份有关的结果，简单缓存可能返回错误数据。AOP 让横切逻辑更容易批量启用，也让错误策略更容易批量扩散。因此越是能改变执行次数、执行时机或返回值的切面，越需要保守的切点和明确的 opt-in 机制。

异常转换切面同样需要边界。把底层异常统一转换成领域异常有助于上层处理，但如果转换时丢失原始 cause、错误码和上下文，就会损害排障能力。最终通知适合释放资源，但如果 finally 中抛出新异常，可能覆盖目标方法原始异常。切面代码本身也要像业务代码一样接受审查和测试，不能因为它是“基础设施”就默认可靠。

## 切点表达与可读性

切点表达式是 AOP 中最容易走向复杂化的部分。表达式可以按包名、类名、方法名、参数、返回类型、注解和类型层次匹配，但可表达不等于可维护。一个包含多层通配符、否定条件和组合逻辑的切点，短期内减少了标注工作，长期会让团队难以判断某个方法是否被增强。

可读的切点通常有明确业务边界。例如“实现某个服务接口的 public 方法”“标注某个注解的方法”“某个模块暴露给外部的命令处理器”。脆弱的切点通常依赖命名巧合，例如“所有名字以 Manager 结尾且不在 internal 包下的方法”。命名规则可以作为辅助，但不应成为关键安全或一致性边界。

注解切点的可读性较好，但也要避免注解泛滥。如果每个方法都要标注多个技术注解才能正常工作，说明系统边界可能没有设计好。可以考虑在类级别声明默认规则，在方法级别只做少量覆盖。覆盖规则必须清楚：方法注解是追加、替换还是禁用类级规则？多个注解组合冲突时谁优先？

切点还需要可诊断。框架可以在启动时输出被匹配的方法数量、关键切面匹配清单或调试报告，帮助使用者发现过宽或过窄的规则。生产环境不一定要打印完整清单，但至少应能通过调试开关获得。没有诊断能力的 AOP 系统，在行为不符合预期时只能靠猜。

## 线程安全与上下文传播

切面经常需要访问上下文，例如当前调用 ID、用户身份、事务状态、指标标签或资源句柄。上下文传播方式会影响线程安全。把调用级状态保存在切面实例字段中通常是错误的，因为切面对象往往被多个线程共享。调用级状态应放在方法栈、显式上下文对象或受控的线程局部变量中，并确保请求结束后清理。

线程局部变量能让深层调用读取上下文，但也有泄漏风险。线程池复用线程时，如果切面在异常路径没有清理上下文，后续无关任务可能读到旧值。异步调用还会带来上下文丢失：在一个线程设置的上下文不会自动出现在另一个线程。AOP 框架若支持异步边界，需要明确上下文是否复制、如何清理、是否允许修改。

资源管理切面尤其要注意并发。事务、锁、连接、缓冲区等资源应严格绑定调用边界，不能被多个调用共享，除非资源本身设计为线程安全且共享语义明确。环绕通知中的 try-finally 是常见结构，但 finally 中也要处理清理失败，避免掩盖主异常或留下半清理状态。

上下文还会影响测试。依赖全局上下文的切面如果没有测试清理，会让测试之间互相污染。实践中应为切面提供显式的上下文入口和清理工具，测试每个用例后恢复状态。横切逻辑越底层，越不能依赖隐式全局状态维持正确性。

## 版本演进与兼容性

切面规则一旦在项目中大规模使用，就会形成兼容性约束。修改切点、改变默认顺序、调整异常转换规则、让某个注解从 opt-in 变成默认启用，都可能改变大量方法行为。AOP 配置的演进应像公共 API 一样谨慎，尤其是基础库被多个模块复用时。

一个稳妥做法是为切面能力定义版本化策略。新增切面先以显式注解或配置开启；观察稳定后再考虑扩大默认范围。修改已有切面的行为时，先增加新选项或新注解，让旧行为继续可用一段时间。移除旧规则前，提供诊断工具列出受影响的方法。

兼容性还包括生成或增强后的字节码。织入式方案升级工具版本后，可能改变生成结构、栈帧、调试信息或与其他工具的交互。构建流水线应把织入结果纳入测试，而不是只测试源码编译。代理型方案升级后，也要测试代理类选择、接口识别、方法匹配和异常包装是否变化。

对于公共框架，AOP 的默认行为应保守。默认切点越宽、默认通知能力越强，越容易在使用者不知情时改变行为。更好的体验是：默认只做可观察、不改变语义的增强；会改变执行、返回或异常的增强必须显式开启。

## 常见误区

第一个误区是把 AOP 当成隐藏业务逻辑的工具。切面适合技术性横切逻辑，不适合把核心业务分支放到方法外部偷偷执行。否则业务行为会分散在注解、配置和切面中，阅读和测试都变得困难。

第二个误区是忽略自调用。代理型 AOP 只能拦截经由代理对象的调用，目标对象内部直接调用自身方法不会进入代理。若某个切面必须覆盖内部调用，需要调整调用结构或选择织入式方案。

第三个误区是切点过宽。匹配整个命名空间再通过复杂排除规则修正，容易误伤基础类、工具类或高频方法。切点应表达业务边界或明确注解，而不是依赖脆弱命名。

第四个误区是顺序未定义。多个切面叠加时，如果顺序靠框架默认或类加载顺序决定，系统行为会不稳定。事务、重试、缓存、异常转换、指标之间都需要明确顺序。

第五个误区是吞掉异常。切面中捕获异常后只记录日志不重新抛出，会让调用者以为操作成功。只有明确的降级或容错切面才应改变异常语义，并且要返回符合契约的结果。

## 实践建议

引入 AOP 前，先列出横切逻辑的共同特征和不适用范围。能用普通函数、装饰器或模板方法清楚表达的，不必急着引入复杂切面。需要 AOP 时，优先选择可读的切点，例如明确注解或稳定接口边界；避免依赖过于复杂的名称表达式。

为每个切面写清楚职责：它是否允许跳过目标方法，是否修改参数或返回值，处理哪些异常，顺序要求是什么，是否线程安全，是否可重复执行。切面内部保持单一职责，不要把权限、日志、缓存和异常转换写进同一个拦截器。

在代理型 AOP 中，确保调用方拿到的是代理对象，避免手动创建目标对象绕过容器。对自调用边界要在设计中显式处理。对注解切点，要定义接口方法、实现方法、父类方法和类级注解的合并规则。

测试 AOP 不应只测业务方法结果，还应验证切面是否生效、顺序是否正确、异常是否按预期传播、禁用或排除规则是否有效。对关键切面，可以写小型集成测试覆盖代理创建和实际调用路径。

落地时还应为切面建立最小可观测闭环。至少能回答四个问题：某个方法为什么被匹配，哪些切面按什么顺序执行，目标方法是否被真正调用，最终异常或返回值是否被某个切面改写。没有这些信息，AOP 问题往往会被误判成业务代码问题。对基础切面，可以在开发或测试环境输出匹配报告；对生产环境，可以保留低成本指标和采样日志，避免排障时只能临时改代码。

切面配置也应接受代码评审。因为一次切点变更可能影响大量方法，它的风险不低于修改公共基类或公共工具方法。评审时应关注匹配范围、排除规则、顺序、异常语义和回滚方式。对于会改变执行结果的切面，建议先在少量显式标注的方法上验证，再逐步扩大范围。

## 小结

AOP 是组织横切逻辑的编程思想，核心在于连接点、切点、通知、切面和织入。它能让重复的技术性逻辑集中管理，但也会增加行为隐式性。通用 Java 中常见实现包括代理和字节码织入：前者简单清晰但受代理边界限制，后者能力更强但工具链复杂。使用 AOP 时应保持切点可读、顺序明确、异常透明、测试充分，并把它用于真正横切的技术职责，而不是隐藏领域逻辑。

# 3.5.5 APT

## 定位

APT 通常指 Annotation Processing Tool，也就是 Java 注解处理机制。现代 Java 中，注解处理主要通过 JSR 269 规范集成在 `javac` 编译流程里。它允许处理器在编译期读取源码和依赖中的程序元素，分析注解，报告错误或警告，并生成新的源码、class 文件或资源文件。APT 的核心价值是把一部分原本需要运行时反射、手写注册或外部配置完成的工作，提前到编译期完成。

APT 不是运行时反射。反射面对的是已经加载的 `Class`、`Method`、`Field`；APT 面对的是编译器中的元素和类型模型，例如 `Element`、`TypeMirror`、`TypeElement`、`ExecutableElement`。处理器不能假设目标类已经能被加载或执行，也不应该通过反射调用用户代码。它观察的是“正在编译的程序结构”，并在编译过程中产生额外文件。

APT 也不等同于注解本身。注解只是输入元数据，处理器才是执行逻辑。一个注解如果没有处理器或运行时框架读取，就不会产生行为。APT 的典型应用包括生成工厂、服务注册表、序列化适配器、元模型类、类型安全访问器、配置索引，以及在编译期校验注解使用是否合法。

与运行时扫描相比，APT 的优势是错误更早暴露，生成代码可被 IDE 和编译器检查，运行时启动成本更低，封装边界更清晰。代价是构建流程更复杂，处理器必须理解编译器模型，增量编译和多轮生成需要谨慎设计，生成代码也会成为公共 API 或至少成为可调试的工程产物。

理解 APT 时，最重要的是把它放回“编译器扩展点”这个位置。它不是给业务代码增加一段神秘的自动执行逻辑，而是在编译器分析源码、建立符号表、准备生成字节码的过程中，允许第三方处理器读取一部分编译器已经掌握的信息，并把这些信息转化为新的编译输入或诊断信息。处理器能看见包、类、接口、枚举、注解类型、字段、方法、构造方法、参数、类型参数、注解镜像和类型关系；但它不能像运行时对象那样调用目标代码，也不应该依赖目标代码的静态初始化、构造方法或业务行为。

这一区分会影响很多设计判断。比如，一个注解 `@Route("/users")` 在源码上只是一个字符串元数据。运行时框架可以扫描所有类，读取注解并建立路由表；APT 则可以在编译期读取所有带注解的元素，生成一个明确的 `RouteRegistry` 类。前者的灵活性来自运行时类路径，后者的稳定性来自编译期可见结构。再比如，一个注解 `@JsonName("user_name")` 可以被反射框架在序列化时读取，也可以由处理器生成字段到名称的映射代码。两种方案都合理，区别不在于注解本身，而在于元数据被谁读取、在什么时候读取、错误在什么时候暴露、成本在什么时候支付。

## APT 解决的问题

APT 最适合解决“结构已知、规则明确、运行时不需要再猜”的问题。许多 Java 框架都需要在类型和配置之间建立映射：接口映射到实现类，字段映射到外部名称，方法映射到调用入口，声明式配置映射到可执行代码。没有 APT 时，这些映射通常依赖手写注册、XML 或 properties 配置、运行时反射扫描、约定式命名，或者在运行时动态拼装。它们并非不能工作，但常见缺点是错误暴露晚、IDE 难以检查、重构不安全、启动阶段成本高、配置与代码容易漂移。

编译期处理把这类映射提前到了构建阶段。处理器可以在用户写错注解目标、缺少必要构造方法、泛型边界不满足、多个元素声明冲突、配置值无法解析时直接让编译失败。生成代码随后作为普通 Java 源码参与编译：如果它引用了不存在的方法、使用了错误类型或违反了访问控制，编译器会继续给出明确错误。这样，原本可能在某个运行分支才出现的问题，被压缩到一次构建反馈里。

APT 还解决了“重复样板代码”的问题，但这不是它的全部价值。生成 getter、setter、构造器、适配器、代理类或注册表只是表象，更深层的价值是让结构化规则保持单一来源。开发者只声明稳定事实，例如“这个类型可以被工厂创建”“这个字段对应外部名称”“这个接口需要生成实现”“这个方法是命令入口”。处理器负责把这些事实转换为机械、重复、容易出错的代码。只要注解语义清晰，生成代码就比手写样板更可维护。

不过，APT 不应该被当成“万能代码生成器”。如果生成逻辑依赖大量业务条件、运行时环境、用户输入、数据库内容或网络数据，那么它很可能不适合放在注解处理器里。编译器期望处理器是确定、快速、可重复的构建步骤。一个处理器如果每次编译都访问外部服务、读取机器特有路径、根据当前时间生成不同内容，构建就会变得不可预测。APT 的边界可以概括为：它适合处理源码结构和编译期常量，不适合处理运行时状态和外部可变世界。

从工程组织上看，APT 还常用于分离“注解 API”和“处理器实现”。使用者编译主代码时只需要依赖注解类型和生成代码所需的运行时 API，而处理器本身通常只放在编译器的 processor path 中。这样可以避免把处理器实现、编译器工具类、源码生成库等构建期依赖带到运行时类路径。公共库设计尤其要注意这一点：注解包、处理器包、运行时支持包的职责越清楚，使用者越容易理解依赖边界。

## JSR 269 的处理器模型

一个注解处理器通常继承 `AbstractProcessor`，并通过 `@SupportedAnnotationTypes`、`@SupportedSourceVersion` 或覆写方法声明自己支持哪些注解和源码版本。编译器发现处理器后，会在编译过程中调用它的 `process` 方法。

```java
@SupportedAnnotationTypes("example.AutoFactory")
@SupportedSourceVersion(SourceVersion.RELEASE_17)
public final class AutoFactoryProcessor extends AbstractProcessor {
    @Override
    public boolean process(Set<? extends TypeElement> annotations, RoundEnvironment roundEnv) {
        for (Element element : roundEnv.getElementsAnnotatedWith(AutoFactory.class)) {
            // inspect element and generate files
        }
        return false;
    }
}
```

`process` 的返回值表示这些注解是否已经被当前处理器“认领”。返回 `true` 表示后续处理器不再处理这些注解；返回 `false` 表示其他处理器仍可处理。公共处理器通常要谨慎选择返回值，避免无意中阻止其他工具读取同一注解。

处理器通过 `processingEnv` 访问编译环境。`Messager` 用于输出错误、警告和提示；`Filer` 用于生成源码、class 或资源；`Elements` 提供元素工具方法；`Types` 提供类型关系和类型构造工具；`Options` 可以读取编译器传入的处理器选项。

```java
Messager messager = processingEnv.getMessager();
Types types = processingEnv.getTypeUtils();
Elements elements = processingEnv.getElementUtils();
Filer filer = processingEnv.getFiler();
```

APT 的输入输出都受编译器控制。处理器不应随意写工程目录中的任意文件，而应通过 `Filer` 创建文件。这样编译器和构建工具才能追踪生成文件来源，避免重复创建、缓存失效和并发写入问题。

`AbstractProcessor` 只是便利基类，不是处理器唯一形式。JSR 269 的核心接口是 `javax.annotation.processing.Processor`。直接实现该接口可以获得更细粒度控制，但通常没有必要，因为 `AbstractProcessor` 已经处理了初始化、支持注解类型、支持选项、源码版本等模板逻辑。真正需要关注的是 `process` 的幂等性、输入过滤、错误报告和生成文件策略。

处理器的生命周期也值得单独理解。编译器创建处理器实例后，先调用 `init(ProcessingEnvironment)`，随后在若干轮中反复调用 `process`。同一个处理器实例会跨轮复用，所以实例字段可以保存“已经生成过的文件名”“已经见过的元素键”“跨轮收集的聚合信息”等状态。但这些状态只属于一次编译任务，不应该被写成静态全局缓存。静态缓存可能跨编译任务残留，在 IDE、持续构建或复用编译进程时制造非常隐蔽的问题。

处理器声明支持的注解类型时，既可以写具体全限定名，也可以使用通配符 `"*"`。通配符表示处理器愿意被所有注解触发，适合做全局检查或特殊工具，但公共处理器应尽量避免滥用。支持范围越大，编译器越可能在更多场景调用处理器，性能和交互复杂度也越高。多数业务型处理器只需要声明自己的注解类型，并在 `process` 中继续防御性检查元素类型。

`@SupportedSourceVersion` 也不只是形式。源码版本会影响编译器接受的语言结构，例如记录类、密封类、模块声明、局部变量类型推断等不同 Java 版本中的语法和模型细节。处理器如果只在旧版本语言结构上测试过，就不应轻易声明过高版本支持。相反，如果处理器逻辑只依赖稳定的元素和类型 API，也可以通过覆写 `getSupportedSourceVersion()` 返回 `SourceVersion.latestSupported()`，但仍需要测试它在目标 Java 版本上的行为。

处理器选项通过编译参数传入，常见形式是 `-Akey=value`。处理器需要通过 `getSupportedOptions()` 或 `@SupportedOptions` 声明自己识别哪些 key。未声明的选项可能被编译器或构建工具提示为未识别。选项适合控制生成代码包名、是否开启严格校验、是否输出调试信息、是否启用某些兼容模式等构建期行为。选项不适合承载大量业务配置，因为它们脱离源码结构，难以被 IDE 重构和定位。

处理器发现通常依赖 Java 标准服务加载机制。处理器 jar 中可以包含 `META-INF/services/javax.annotation.processing.Processor` 文件，文件内容是一行或多行处理器全限定类名。`javac` 在 processor path 上发现这些服务声明后加载处理器。也可以通过编译参数显式指定处理器类。无论采用哪种方式，处理器都应该假设自己可能和其他处理器同时运行，并避免修改全局状态、抢占无关注解或对处理器执行顺序做脆弱假设。

```text
META-INF/services/javax.annotation.processing.Processor

example.processor.AutoFactoryProcessor
```

处理器返回值经常被误解。`process` 返回 `true` 表示当前处理器认领本轮传入的注解类型，后续处理器不会再收到这些注解类型；返回 `false` 表示没有独占，其他处理器仍可处理。返回值不是“处理成功或失败”的布尔结果。处理失败应通过 `Messager` 报告错误，必要时跳过生成。对于生态中可能被多个工具共同读取的注解，返回 `false` 往往更稳妥；对于内部私有注解，返回 `true` 可以减少重复处理。

## Element 与 TypeMirror

编译期模型中，`Element` 表示程序元素，`TypeMirror` 表示类型。两者容易混淆。类、方法、字段、参数、包都是元素；`String`、`List<String>`、`T`、`? extends Number`、数组类型则是类型。元素有名称、修饰符、封闭元素、包含元素和注解；类型有种类、类型参数、上界、可赋值关系等。

常见元素子类型包括 `TypeElement`、`ExecutableElement`、`VariableElement`、`PackageElement`、`TypeParameterElement`。类和接口对应 `TypeElement`，方法和构造方法对应 `ExecutableElement`，字段和参数对应 `VariableElement`。判断元素种类应使用 `getKind()`，判断修饰符使用 `getModifiers()`。

```java
if (element.getKind() != ElementKind.CLASS) {
    processingEnv.getMessager().printMessage(
            Diagnostic.Kind.ERROR,
            "@AutoFactory can only be used on classes",
            element);
}
```

`TypeMirror` 的子类或种类包括声明类型、数组类型、基本类型、类型变量、通配符、可执行类型等。比较类型时，不要只比较字符串名称。应使用 `Types.isSameType`、`Types.isAssignable`、`Types.isSubtype` 等工具方法。字符串适合生成代码和错误消息，不适合作为类型系统判断的唯一依据。

```java
TypeElement serializable = elements.getTypeElement("java.io.Serializable");
boolean ok = types.isAssignable(element.asType(), serializable.asType());
```

泛型处理是 APT 的重要能力。`TypeElement.getTypeParameters()` 可以读取类型参数，`DeclaredType.getTypeArguments()` 可以读取参数化类型实参，`ExecutableElement.getReturnType()` 和 `getParameters()` 可以读取方法签名。生成代码时要保留必要的泛型信息，否则生成类可能产生 unchecked 警告或类型不安全。

元素模型强调“声明位置”，类型模型强调“使用后的类型”。一个泛型类 `Box<T>` 的 `TypeElement` 描述的是类声明本身；而源码中的 `Box<String>`、`Box<Integer>`、`Box<? extends Number>` 都是不同的 `DeclaredType`。如果处理器只关心类上是否存在某个注解，`TypeElement` 通常足够；如果处理器要判断字段类型、方法返回值、参数类型是否满足某个可赋值关系，就必须进入 `TypeMirror` 世界。

`Element` 的封闭关系也很重要。字段的 `getEnclosingElement()` 是所属类型，方法参数的封闭元素是所属方法，内部类的封闭元素是外部类型，包元素则位于更上层。生成代码时经常需要从任意被注解元素向上找到最近的 `TypeElement`，以确定包名、外部类名和访问边界。不要简单假设注解只会出现在类上；即使注解的 `@Target` 限制了位置，处理器也应在错误输入或不完整编译状态下保持防御。

名称处理同样不能粗糙。`getSimpleName()` 返回简单名，`getQualifiedName()` 返回全限定名，但只有部分元素拥有全限定名。方法、字段、参数没有像类型那样的全限定名；内部类的二进制名、规范名和源码书写名也可能不同。生成类名时如果把 `Outer.Inner` 直接拼成文件名或包名，容易产生非法或冲突名称。更稳妥的做法是明确区分包名、顶层类名、嵌套类路径和生成类后缀，并为内部类设计稳定的转义规则。

类型比较要尊重擦除和泛型。`List<String>` 与 `List<Integer>` 不是同一个完整类型，但擦除后都是 `List`。有些处理器只关心原始容器类型，有些处理器必须保留实参。`Types.erasure` 可以取得擦除类型，`Types.asMemberOf` 可以在某个声明类型上下文中解析成员类型。例如父类中声明 `T get()`，子类绑定为 `String` 后，直接看父类方法返回类型只能得到 `T`，放到子类上下文中解析才能得到更具体的返回类型。这类细节决定了生成代码是否真正类型安全。

可访问性判断也不能只看 `public`。同包访问、`protected`、嵌套类访问、构造方法访问都会影响生成代码能否调用目标成员。处理器生成的类放在哪个包、是否作为嵌套类、是否继承目标类型，都会改变访问规则。最简单的策略是把生成类放在与目标类型相同的包中，并只调用明确可访问的 public、protected 或 package-private 成员；但如果目标类型位于不同模块或有强封装要求，还需要结合模块导出关系和实际编译配置判断。

在错误恢复场景下，`Element` 和 `TypeMirror` 可能包含错误类型。用户源码还没有完全编译通过时，编译器仍可能调用处理器，此时某些依赖类无法解析，类型种类可能是 `ERROR`。健壮的处理器应识别这种状态，避免把错误类型当成正常类型生成代码。通常做法是延迟处理相关元素，或者报告更贴近注解使用点的错误；如果错误来自用户代码中其他位置，则应减少二次噪音，让编译器原始错误先暴露。

| 关注点 | 常用 API | 典型用途 |
| --- | --- | --- |
| 元素种类 | `Element.getKind()` | 判断注解是否放在类、方法、字段或参数上 |
| 修饰符 | `Element.getModifiers()` | 检查是否 `public`、`abstract`、`static`、`final` |
| 类型关系 | `Types.isAssignable`、`Types.isSubtype` | 判断字段、参数、返回值是否满足约束 |
| 泛型实参 | `DeclaredType.getTypeArguments()` | 读取 `List<String>` 中的 `String` |
| 包和名称 | `Elements.getPackageOf`、`TypeElement.getQualifiedName()` | 生成包名、类名和错误消息 |
| 注解镜像 | `Element.getAnnotationMirrors()` | 读取复杂注解值并避免加载用户类 |

## 编译轮次

JSR 269 处理器按轮次运行。第一轮处理原始源码中的注解；如果某个处理器生成了新的源码文件，编译器会把这些新文件纳入后续轮次，其他处理器也可以继续处理新文件中的注解。直到某一轮没有新的源码生成，最后会进入 `processingOver()` 为 true 的收尾轮次。

轮次机制解决了“生成的代码还可能带有注解”的问题，也引入了状态管理问题。处理器不能假设所有被注解的元素都在第一轮出现，也不能在每一轮都重复生成同一个文件。常见做法是在处理器实例中维护已经处理或已经生成的类型集合，并在每轮只处理新增元素。

```java
private final Set<String> generated = new HashSet<>();

private void generateOnce(TypeElement type) throws IOException {
    String generatedName = type.getQualifiedName() + "Factory";
    if (!generated.add(generatedName)) {
        return;
    }
    JavaFileObject file = processingEnv.getFiler().createSourceFile(generatedName, type);
    try (Writer writer = file.openWriter()) {
        writer.write("// generated source");
    }
}
```

`Filer` 不允许同一轮或不同轮重复创建同一个文件，否则会抛出 `FilerException`。这通常说明处理器没有正确去重，或多个处理器争抢同一个生成文件。生成文件命名应稳定，并能从输入元素推导出来。

收尾轮次不适合再生成依赖其他生成结果的新源码。此时 `roundEnv.processingOver()` 为 true，处理器更适合做最终校验或释放资源。如果前面轮次已经出现错误，`roundEnv.errorRaised()` 会提示已有错误发生，处理器可以减少后续噪音。

多轮处理的一个关键点是“本轮可见”不等于“全部可见”。如果处理器要为每个被注解类型生成一个独立文件，它可以在每轮处理本轮出现的元素并立即生成。但如果处理器要生成一个全局索引，就需要在前几轮收集所有元素，等确定不会再出现新源文件后再生成聚合输出。这个策略看似自然，却也带来风险：如果在最后一轮才生成新的源码，该源码不会再被后续注解处理轮充分处理；如果聚合文件本身需要被其他处理器读取，就可能错过时机。因此，聚合生成必须设计清楚它是否需要参与后续处理。

处理器之间也会通过轮次间接协作。处理器 A 生成了带有注解的源码，处理器 B 可以在下一轮读取这些注解。反过来，如果 B 假设 A 的生成物在第一轮就存在，就会失败。公共处理器不应该依赖未声明的执行顺序。若两个处理器确实需要协作，更可靠的方式是让上游生成明确的源码或资源，下游在后续轮次发现它们；或者把协作逻辑收敛在同一个处理器中。

跨轮状态要以稳定键保存。直接把 `Element` 对象放进长期集合有时可以工作，但更稳妥的是保存全限定名、生成文件名或自定义符号键。不同轮次中，编译器可能为新源码创建新的元素对象；同一个逻辑类型在不同上下文下也可能对应不同模型实例。保存稳定字符串可以降低去重逻辑对编译器实现细节的依赖。

另一个常见问题是“延迟元素”。当某个被注解元素依赖的类型当前还是错误类型时，处理器如果立即生成代码，生成结果可能不完整；如果立即报错，又可能掩盖真正的上游编译错误。很多处理器会把这类元素暂存到下一轮重试。如果后续轮次仍无法解析，再根据情况报告错误。延迟策略需要有终止条件，避免在最后一轮悄悄丢弃输入。

## 读取注解值

在处理器中读取注解值有两种方式。简单场景可以调用 `element.getAnnotation(MyAnnotation.class)`，得到类似运行时注解的代理对象。对于基本类型、字符串、枚举和简单数组，这种方式很方便。但遇到 `Class<?>` 注解元素时，直接调用可能抛出 `MirroredTypeException` 或 `MirroredTypesException`，因为被引用的类型不一定能以运行时 `Class` 形式加载。

更通用的方式是使用 `AnnotationMirror`、`AnnotationValue` 和编译器模型解析注解。`element.getAnnotationMirrors()` 返回元素上的注解镜像，处理器可以根据注解类型匹配目标注解，再读取显式元素值和默认值。这种方式更繁琐，但不会强制加载用户类，也更适合处理类型值、嵌套注解和复杂数组。

```java
for (AnnotationMirror mirror : element.getAnnotationMirrors()) {
    Element annotationElement = mirror.getAnnotationType().asElement();
    if (((TypeElement) annotationElement).getQualifiedName().contentEquals("example.AutoFactory")) {
        Map<? extends ExecutableElement, ? extends AnnotationValue> values =
                elements.getElementValuesWithDefaults(mirror);
    }
}
```

处理器应避免把注解值解析成脆弱字符串后到处传递。对于类型值，优先保留 `TypeMirror`；对于枚举值，保留变量元素或明确的枚举常量名；对于嵌套注解，保留结构化对象。只有在生成源码时，才把它们转换为合法 Java 代码文本。

默认值也要正确处理。`AnnotationMirror.getElementValues()` 只返回显式写出的元素值，不包含默认值。若需要完整配置，应使用 `Elements.getElementValuesWithDefaults`。这能避免“用户省略了默认元素，处理器却当成没有配置”的问题。

注解设计本身会影响处理器复杂度。注解元素只能使用 Java 注解允许的类型：基本类型、`String`、`Class`、枚举、注解、这些类型的一维数组。它们必须是编译期常量或类型字面量，不能引用任意对象。这个限制并不是缺陷，而是让注解适合作为稳定元数据。若某个配置需要表达复杂对象图，通常不应该硬塞进注解数组，而应考虑使用普通代码、配置文件或显式注册 API。

`RetentionPolicy` 也需要按用途选择。APT 读取源码和编译模型中的注解，注解不一定要保留到运行时。只服务于注解处理器的注解可以使用 `SOURCE`，这样不会进入 class 文件；如果其他编译后工具或字节码分析需要读取，可以使用 `CLASS`；只有运行时反射也需要读取时才使用 `RUNTIME`。把所有注解都设成 `RUNTIME` 看似省事，但会扩大运行时元数据暴露范围，也会让使用者误以为该注解在运行时自动生效。

`@Target` 则是第一层防线。处理器可以检查元素种类，但注解自身应该先声明允许的位置。只能标在类型上的注解不应允许字段和方法；只能标在构造参数上的注解不应允许类型。清晰的 `@Target` 能让编译器在处理器运行前拦截一部分错误，也能让 IDE 给出更准确的补全和提示。处理器中的校验负责补充语义规则，而不是替代注解类型的基本约束。

读取枚举注解值时，`AnnotationValue.getValue()` 通常会给出表示枚举常量的 `VariableElement`，而不是简单字符串。读取 `Class` 值时，底层通常是 `TypeMirror`。读取嵌套注解时，则会得到 `AnnotationMirror`。处理器若把这些值统一转成字符串，短期能减少代码量，长期会失去类型信息和定位能力。更好的做法是把注解解析成处理器内部的结构化模型，例如 `RouteSpec`、`FactorySpec`、`FieldBinding`，模型字段保留类型、元素和值的语义，再由后续生成阶段负责格式化输出。

数组注解值也要处理默认空数组和单元素数组。Java 注解语法允许在只有一个名为 `value` 的元素时省略元素名，也允许数组在单元素时省略大括号。处理器看到的是规整后的模型，但错误消息最好贴近用户写法。例如配置重复、空字符串、不合法路径、类类型不满足约束时，应指出具体注解元素，而不是只指向整个类。

## 生成源码与资源

APT 最常见的输出是 Java 源码。处理器通过 `Filer.createSourceFile` 创建 `JavaFileObject`，写入合法 Java 代码。生成代码应包含包名、导入、类声明、泛型、修饰符和必要注释。不要依赖格式随意的字符串拼接来处理复杂源码；项目复杂后可以使用源码生成库，或至少集中封装代码输出逻辑。

```java
JavaFileObject file = filer.createSourceFile("example.UserFactory", originatingElement);
try (Writer writer = file.openWriter()) {
    writer.write("package example;\n");
    writer.write("public final class UserFactory {\n");
    writer.write("  public User create() { return new User(); }\n");
    writer.write("}\n");
}
```

`originatingElement` 很重要。它告诉编译器生成文件来自哪些源元素，构建工具可用它做增量编译和依赖追踪。生成文件如果聚合多个输入，也应尽可能传入相关元素。忽略来源信息可能让增量构建无法精确判断哪些生成文件需要重建。

除了源码，处理器还可以生成资源文件，例如服务注册文件、索引文件、配置描述等。资源生成同样应通过 `Filer.createResource` 完成，并注意多轮去重和聚合写入问题。多个处理器或多个模块同时写同一路径时容易冲突，公共格式要设计合并策略。

生成代码应尽量可读。虽然生成物不是手写代码，但开发者会在调试、报错和代码跳转时阅读它。清晰的类名、方法名、缩进和来源注释能显著降低排查成本。生成代码也应避免过度依赖处理器运行时库，除非这是明确设计；否则使用者可能在运行时缺少处理器依赖。

生成源码时，最容易被低估的是合法 Java 语法的细节。包名可能为空，类名可能与关键字冲突，内部类名称需要展开，泛型边界可能包含多个上界，字符串字面量需要转义，注解值可能包含换行和反斜杠。简单字符串拼接适合极小示例，一旦生成逻辑涉及泛型、方法重载、注解、异常声明和复杂表达式，就应考虑使用专门的源码生成库，或至少建立一套可靠的输出工具函数。生成代码的 bug 本质上也是编译器输入 bug，排查成本往往比普通手写代码更高。

导入管理也要谨慎。生成器可以选择全部使用全限定名，避免导入冲突；也可以生成导入以提升可读性。使用导入时必须处理同名类型冲突，例如 `java.util.Date` 与 `java.sql.Date`。若生成代码位于目标类相同包内，同包类型无需导入，但嵌套类、静态成员和通配符导入仍可能产生歧义。公共处理器通常更偏向稳定正确，而不是追求最短代码。

生成文件命名应体现来源和职责。为每个类型生成辅助类时，可以使用目标类型名加稳定后缀，例如 `UserFactory`、`UserJsonAdapter`、`UserMetaModel`。为方法或字段生成辅助结构时，需要避免重载和同名字段冲突，可以把封闭类型、成员名、参数签名摘要纳入名称。名称中不应包含随机数或构建时间。稳定名称不仅帮助增量编译，也让开发者能根据报错快速找到生成物。

资源文件比源码更容易出现合并问题。源码文件通常由单个处理器拥有，重复创建会立刻失败；资源文件却常常承载全局索引或服务声明，多个输入都想向同一个文件追加内容。`Filer` 不提供“打开已有资源并追加”的通用安全语义，因此处理器应在内存中聚合内容后一次性写出，或为每个输入生成独立资源，再由明确的后续步骤合并。直接尝试读取旧资源、追加再写回，容易在并行构建和增量构建中出错。

生成代码是否依赖运行时库，需要提前设计。如果生成类只使用 JDK API 和用户类型，使用者运行时不需要额外依赖处理器库；如果生成类继承了处理器项目中的基类、调用了运行时辅助方法或实现了某个框架接口，那么这些运行时类型必须作为普通依赖存在。处理器依赖和运行时依赖混在一个 jar 中会让使用者难以区分，常见做法是拆成 `annotations`、`processor`、`runtime` 等模块。

生成物还应避免泄露机器环境。绝对路径、当前用户名、本地临时目录、当前时间戳、构建机信息通常都不属于语义输出。它们会让缓存命中率下降，也可能暴露隐私。若确实需要来源注释，可以写“Generated by X. Do not edit.”，并记录源元素名称，而不是记录本地文件系统路径。

## 错误报告与校验

APT 的一项重要价值是把错误提前到编译期。处理器可以用 `Messager.printMessage` 报告错误、警告和提示。传入相关 `Element`、`AnnotationMirror` 或 `AnnotationValue` 后，编译器和 IDE 能把错误定位到具体代码位置。

```java
messager.printMessage(
        Diagnostic.Kind.ERROR,
        "Class annotated with @AutoFactory must have an accessible constructor",
        typeElement);
```

错误消息应说明违反了什么规则、当前元素是什么、如何修复。不要只输出“invalid annotation”。如果注解处理器面对大量元素，错误也要尽量一次性报告多个独立问题，而不是遇到第一个就中止，除非继续处理会产生误导性错误。

校验规则要与生成逻辑一致。例如生成工厂要求目标类非 abstract、有可访问构造方法、构造参数可解析，那么这些条件都应在生成前检查。不要等生成出的代码编译失败后让使用者从二次错误中猜原因。生成代码编译失败通常说明处理器没有尽到校验责任或输出代码不合法。

警告适合提示不推荐但仍可工作的用法，例如注解放在即将废弃的模式上、配置值未使用、生成代码采用降级路径。错误适合阻止必然不正确或无法生成的用法。过多警告会让使用者忽视真正问题，处理器应克制。

好的错误消息应该站在注解使用者的视角，而不是处理器作者的视角。使用者需要知道哪一条规则被违反、当前写法为什么不满足、最小修复方式是什么。比如“cannot process element”没有价值；“`@AutoFactory` can only be applied to non-abstract classes because the generated factory must instantiate the type”就明确得多。若错误与某个注解元素值有关，应尽量把诊断绑定到 `AnnotationValue`，让 IDE 可以定位到具体参数。

处理器还要区分“输入非法”和“处理器内部失败”。输入非法应该报告 `Diagnostic.Kind.ERROR`，并尽量继续检查其他独立元素，让使用者一次看到多个问题。内部失败，例如生成器状态不一致、I/O 异常、未覆盖的类型种类，则应输出清晰的处理器错误，并保留必要上下文。不要吞掉异常后继续生成半成品，也不要把完整堆栈无条件塞进普通错误消息；调试选项可以控制更详细输出。

有些校验涉及跨元素关系，例如名称唯一、路径不冲突、一个接口只能有一个默认实现、多个字段不能映射到同一个外部名称。这类规则通常需要聚合视图。处理器可以先收集所有候选，再统一检查冲突。冲突错误最好同时指向至少一个具体元素，并在消息中列出另一个冲突元素的可读名称。若编译器 API 难以同时绑定多个位置，也应在文字中提供足够线索。

校验的边界也要克制。处理器应该验证与自身生成逻辑直接相关的条件，而不是对用户代码做泛化风格检查。比如工厂处理器需要关心构造方法是否可调用、目标类是否可实例化、参数是否可解析；它不需要评价类名是否优雅、字段是否过多、业务方法是否复杂。把无关规则塞进处理器会让注解使用成本变高，也会让处理器成为难以关闭的编译阻塞点。

## 增量编译与处理器类型

注解处理器会影响构建性能。一个处理器如果每次都扫描所有类型、生成聚合文件或无法声明输入输出关系，构建工具可能不得不放弃部分增量编译。大型工程中，处理器性能和增量能力会直接影响开发体验。

从行为上看，处理器大致可分为隔离型和聚合型。隔离型处理器中，每个生成文件只依赖一个或少数明确输入元素，例如为每个 `@AutoFactory` 类生成一个对应工厂。某个输入变化时，只需要重建对应输出。聚合型处理器会把许多输入合并成一个输出，例如生成全局注册表。任意输入变化都可能影响聚合输出，因此增量粒度更粗。

设计处理器时，应优先采用隔离生成。只有确实需要全局视图时才生成聚合文件。即便需要聚合，也可以考虑分层：先为每个元素生成局部元数据，再由运行时或后续步骤合并，减少单个处理器对全局输入的依赖。

性能方面，处理器应避免读取无关 classpath、避免在每轮重复全量扫描、避免进行网络或外部进程调用、避免写入不稳定内容导致缓存失效。生成文件中不要包含当前时间、随机数或绝对路径，除非这些信息确实属于输出语义；否则每次构建都会产生不同内容。

隔离型和聚合型不是道德评价，而是依赖形态。隔离型处理器更容易增量化，因为输入和输出之间存在局部对应关系；聚合型处理器有时不可避免，例如全局服务索引、全局路由表、全局 schema 汇总。真正的问题不是“能否完全避免聚合”，而是“聚合是否确实由需求决定，是否可以缩小影响面，是否能让构建工具理解输入输出关系”。如果全局输出只是为了方便运行时查找，可能存在替代设计；如果全局输出是语义要求，就应明确接受它的构建成本。

处理器性能瓶颈常来自不必要的重复工作。每一轮都遍历所有已知元素、每个字段都重复解析同一个常用类型、每次生成都重新扫描 classpath、每个注解都读取大型资源文件，都会在大工程中放大。处理器可以缓存 `TypeElement`、常用 `TypeMirror`、解析后的注解模型和已校验结果，但缓存范围应限于当前编译任务，并且要考虑错误类型和多轮新增元素。

另一个性能细节是输出稳定性。即使处理器速度很快，只要它每次都生成字节不同的文件，后续编译任务仍会被迫重做。稳定输出要求元素遍历顺序可预测，集合写出前排序，资源文件换行一致，生成代码格式稳定，不包含非语义信息。很多编译器模型返回的集合顺序不应该被当成长期契约；处理器若生成聚合文件，最好显式按全限定名或声明顺序排序。

构建工具对注解处理器的增量支持各有差异，但处理器自身可以做的基本原则相通：通过 `Filer` 生成文件，传入来源元素；不要在 processor path 之外写文件；不要读取不可追踪的输入；不要让输出依赖环境随机状态；明确文档化处理器是隔离型还是聚合型；提供关闭调试输出和额外扫描的选项。处理器越像一个纯函数，构建系统越容易缓存和并行化它。

## APT 与运行时反射的取舍

APT 和运行时反射经常解决相似问题，但取舍不同。运行时反射灵活，能处理运行时才出现的类型，部署后也能根据类路径变化发现新实现；缺点是启动扫描和访问检查成本高，错误晚暴露。APT 静态，编译期就生成明确代码，运行时路径快且类型安全；缺点是只能处理编译期可见元素，构建配置更复杂。

如果框架要求零配置发现所有类路径上的实现，运行时扫描可能更自然。如果框架可以要求使用者标注注解并在编译期生成注册表，APT 能显著降低运行时成本。很多成熟方案会混合使用：APT 生成快速路径，运行时反射作为兼容或调试路径。但混合方案必须明确优先级和一致性，避免两套发现机制结果不同。

APT 生成代码还改善了可调试性。调用栈会出现具体生成类，IDE 可以跳转，编译器可以检查方法签名。反射失败常表现为运行时异常，而生成代码失败通常在编译期暴露。对于公共库，提前报错往往比运行时灵活性更有价值。

但 APT 不适合所有动态需求。用户在运行时提供类名、插件在部署后加入、脚本动态定义行为、配置按环境频繁变化，这些都超出纯编译期处理范围。不要为了追求“无反射”而把本应动态的需求硬塞进编译期。

两者还存在维护体验上的差异。反射方案的核心逻辑集中在运行时框架中，使用者写的代码较少，但框架内部需要处理各种访问控制、类型转换和异常路径。APT 方案把一部分复杂度转移到构建阶段，使用者会在工程中看到生成类，编译错误也可能来自生成代码。公共库如果选择 APT，就需要为生成物命名、位置、可读性、调试方式和版本兼容性负责。

选择 APT 还是反射，可以从几个问题判断。第一，目标类型是否在编译期可见，并且变化必须经过重新编译？如果是，APT 有基础条件。第二，错误是否越早暴露越好？如果注解配置错误会导致严重运行时失败，APT 更有优势。第三，运行时扫描成本是否显著？如果启动或首次调用路径对性能敏感，生成代码更合适。第四，是否需要部署后动态扩展？如果需要，纯 APT 可能过于静态。第五，团队是否能接受构建复杂度和生成代码调试？如果不能，简单反射或显式注册可能更务实。

显式手写注册也是一个常被忽略的选项。对于规模很小、变化很少、性能不敏感的场景，手写一张注册表可能比引入处理器更清楚。APT 的收益来自规则重复和规模增长；如果只有两三个类型，处理器带来的构建配置、错误诊断和维护成本未必划算。工程实践中，最好的方案不一定是技术上最自动的方案，而是让错误、成本和复杂度出现在最合适位置的方案。

## 常见误区

第一个误区是用反射思维写处理器。处理器中面对的是 `Element` 和 `TypeMirror`，不是 `Class`。强行加载用户类可能失败，也可能触发不该发生的依赖问题。读取类型注解值时应使用编译器模型，而不是依赖 `Class<?>`。

第二个误区是重复生成文件。多轮处理会多次调用 `process`，如果不记录已生成名称，就容易触发 `FilerException`。处理器必须理解轮次，并保证同一文件只生成一次。

第三个误区是错误定位粗糙。生成代码编译失败后，使用者看到的是生成文件中的错误，而不是原始注解使用点。处理器应在生成前校验输入，并把错误绑定到原始元素。

第四个误区是生成代码不可读。不可读的生成代码会让调试和排查变得痛苦。即使生成物不需要手工维护，也应保持命名清晰、结构稳定、来源可追溯。

第五个误区是忽视增量编译。处理器在小项目中看似没有成本，在大项目中可能成为构建瓶颈。全局扫描、聚合输出、不稳定文件内容和缺失来源元素都会削弱增量能力。

第六个误区是把注解当成配置语言。注解适合表达紧贴源码声明的少量结构化元数据，不适合承载复杂流程、环境差异和大段文本配置。注解元素越复杂，处理器越难给出清晰错误，使用者也越难重构。若配置本身具有独立生命周期，应考虑普通配置文件或显式 API。

第七个误区是让处理器依赖执行顺序。多个处理器同时存在时，除非通过生成文件和轮次自然衔接，否则不应假设谁先谁后。依赖顺序的处理器在简单命令行编译中可能正常，在 IDE、不同构建工具或增量编译中就可能失败。

第八个误区是把 processor 依赖放进运行时路径。处理器通常只应在编译期使用。若使用者必须在运行时依赖处理器 jar，多半说明模块拆分不清，或者生成代码错误地引用了处理器内部类型。正确的依赖边界能减少包体积、类路径冲突和运行时安全风险。

## 实践建议

设计 APT 方案时，先明确输入注解、生成文件、错误规则和运行时使用方式。注解应表达稳定的编译期事实，生成代码应尽量少依赖运行时反射。处理器应使用 `Elements` 和 `Types` 做类型判断，使用 `Filer` 输出文件，使用 `Messager` 报告可定位错误。

生成代码前充分校验。检查目标元素种类、修饰符、构造方法、泛型边界、注解组合和可访问性。错误消息要面向使用者，告诉他改哪里，而不是暴露处理器内部状态。生成文件命名要稳定，包名要可预测，避免与用户手写类冲突。

处理多轮时维护状态，避免重复生成。尽量采用隔离型输出，并传入来源元素。不要在生成文件中写入时间戳和绝对路径。处理器自身也要有测试：可以用编译测试工具构造源码输入，断言编译成功或失败、错误消息、生成文件内容和多轮行为。

对于公共处理器，文档应说明支持的 Java 源码版本、处理器选项、注解保留策略、生成代码位置、是否支持增量、是否生成运行时依赖以及常见错误。APT 是构建期工具，使用者最需要的是可预测和可诊断。

从实现顺序看，一个可靠处理器通常先建立内部模型，再生成代码。第一步扫描注解元素，只做轻量过滤。第二步把 `Element`、`TypeMirror`、注解值解析成自己的领域模型，并在这个阶段完成大部分校验。第三步根据模型生成源码或资源。这样可以避免在生成字符串的过程中夹杂大量编译器 API 调用，也便于单元测试生成逻辑。模型层越清楚，处理器越不容易变成难以维护的巨型 `process` 方法。

测试处理器不能只测试普通 Java 方法。更重要的是编译测试：给定一段源码输入，运行真实编译器和处理器，断言编译成功、失败诊断、生成文件内容、多轮行为和错误定位。至少应覆盖合法最小输入、非法注解目标、缺少必需成员、泛型类型、内部类、默认注解值、重复配置、增量相关的来源元素，以及处理器选项。处理器本身是编译期工具，只有放进编译流程里测试，才能发现模型和轮次问题。

版本兼容也要提前考虑。Java 语言不断增加新结构，处理器至少要知道自己遇到未知 `ElementKind`、未知 `TypeKind` 或新修饰符时如何处理。保守策略是只支持明确验证过的元素种类，对未知结构给出可理解错误；开放策略是对不相关结构忽略，只在生成逻辑必须理解时才失败。无论哪种策略，都应避免因为 switch 没有 default 或空指针而让处理器崩溃。

最后，APT 的设计应保留退出机制。使用者可能因为构建性能、兼容性、调试需要或特殊部署方式，不想使用生成代码。公共库可以提供手写注册、显式工厂、禁用某些处理规则或输出调试信息的方式。一个好处理器不仅能在理想路径上自动生成代码，也能在问题发生时让使用者知道如何绕开、定位和逐步恢复。

## 小结

APT 把注解驱动的分析和代码生成放到 Java 编译期。它基于 JSR 269 的元素和类型模型，通过多轮处理读取注解、校验程序结构、生成源码或资源。它适合把稳定的结构性工作提前完成，减少运行时反射和扫描成本，并把错误暴露在编译阶段。正确使用 APT 的关键是尊重编译期模型、理解轮次、通过 `Filer` 生成稳定文件、用 `Messager` 报告清晰错误，并在增量编译和可调试性上保持克制设计。

# AndroidKnowledge

AndroidKnowledge 是一个面向 Android 开发者的知识体系仓库，目标是把计算机基础、JVM、Java、Kotlin、Android 应用开发、Framework、性能、安全、工程化和问答题组织成可持续扩写的文档网络。

## 顶层文件和目录
- [RootIndex.md](RootIndex.md)：整个文档的阅读入口，索引 `./docs/*` 下所有一级、二级、三级目录。
- [AndroidVersionChangeLog.md](AndroidVersionChangeLog.md)：Android 版本变化的独立入口，后续用于记录每个版本的机制变化、API 变化、兼容变化和迁移建议。
- [docs/](docs)：主文档目录，按知识体系分层组织内容。
- [reference/](reference)：外部参考资料入口，只做延伸阅读和来源链接，不作为主线正文。

## 主文档目录
- `1.计算机基础`：操作系统、计算机网络、数据结构与算法、数据库。
- `2.JVM`：Java 虚拟机与 Android 虚拟机。
- `3.Java`：语言基础、集合、并发、I/O、反射注解与动态代理。
- `4.Kotlin`：语言基础、协程与 Flow、泛型函数式、工程实践。
- `5.Android`：基础、进阶、三方库、性能优化、架构工程化、Framework、安全、跨端。
- `6.问答`：面试题、场景题、排障案例和知识卡片。

## 目录约定
- 非叶子目录保留一个同名 Markdown 文件作为该目录索引，但索引文件放在与目录同级的位置，例如 `docs/5.Android.md` 对应 `docs/5.Android/`。
- 二级及更深目录也遵循同样规则，例如 `docs/5.Android/5.2.进阶.md` 对应 `docs/5.Android/5.2.进阶/`。
- 最后一级主题不再额外创建同名目录，直接在父目录下平铺 Markdown 文件，例如 `docs/1.计算机基础/1.2.计算机网络/1.2.2.应用层/1.2.2.1.HTTP协议.md`。
- 目录名前缀使用数字层级，便于阅读顺序稳定，例如 `5.4.性能优化`、`5.4.1.启动优化`。
- 非叶子索引文件名与目录名保持一致，并包含其直接子目录索引和直接正文文档索引。
- 叶子正文文件名直接使用完整编号和主题名。

## 后续扩写原则
- 先写目录索引，再写正文：新增正文前先确认它属于哪个目录、是否需要新增更细层级。
- 避免纯搬运：外部资料放在 `reference` 或延伸阅读中，主文档应沉淀自己的结构化理解。
- 建立反链：正文提到版本变化时链接到 `AndroidVersionChangeLog.md`，问答内容应链接回主文档。

## 正文编写约束
- 正文只写在叶子节点 Markdown 文件中。非叶子目录索引文件只承担导航、定位和索引职责，不扩写成长篇正文。
- 每篇正文必须以文件标题为核心，写成一篇完整、详细、内容丰富的技术文章，而不是提纲、速记、问答摘抄或资料拼贴。
- 每篇正文都应围绕自己的标题展开。不要套用统一模板生硬填充，不要让不同主题的文章呈现千篇一律的结构和叙述方式。
- 正文应按主题自然拆分子标题。子标题数量和层级按内容需要决定，不设固定套路；但不能整篇文章一屏铺到底。
- 每篇正文不少于 8000 字，最多不超过 20000 字。篇幅不足时优先补充背景、概念边界、机制细节、推导过程、典型问题和实践案例；篇幅过长时应拆分为更细主题。
- 只有 `docs/5.Android/` 和 `docs/2.JVM/2.2.Android虚拟机/` 路径下的正文可以写 Android 相关内容。其他路径只写该主题自身的通用知识，不主动引入 Android 场景、Android 源码、Android API 或 Android 面试关联。
- `docs/1.计算机基础/`、`docs/2.JVM/2.1.Java虚拟机/`、`docs/3.Java/`、`docs/4.Kotlin/` 下的正文应保持通用技术视角。即使这些知识服务于 Android 学习，也不要在正文中写 Android 迁移、Android 示例或 Android 机制映射。
- `docs/6.问答/` 下的正文可以作为复习、面试、排障或知识卡片入口，但仍应链接回主文档，不应替代主文档正文的系统讲解。
- 正文应优先讲清楚“是什么、为什么、解决什么问题、核心机制是什么、边界在哪里、常见误区是什么”。是否加入代码、图示、流程、对比表、案例和延伸阅读，按主题需要决定。
- 正文中的流程图、结构图、时序图和状态图可以使用 Mermaid 表达。
- 涉及版本、标准、API 行为、兼容性或事实性结论时，应标明适用范围；如果与 Android 版本变化有关，只能出现在允许写 Android 内容的路径下。
- 禁止直接复制外部文章或把外部资料简单拼接成正文。外部资料应作为理解后的参考来源沉淀到正文中，必要时放在文末延伸阅读或 `reference/` 中。
- 标题、子标题和正文措辞要与当前文件主题一致。不要为了凑篇幅引入无关历史、泛泛介绍、重复定义或与标题弱相关的内容。
